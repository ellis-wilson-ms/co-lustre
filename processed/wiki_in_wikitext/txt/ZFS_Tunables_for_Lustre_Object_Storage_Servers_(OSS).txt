This page describes ZFS and Lustre ZFS OSD module parameters relevant to tuning Lustre Object Storage Servers (OSS). OSS workloads are bulk-data-intensive — dominated by large sequential reads and writes — and benefit from aggressive prefetch, write aggregation, and I/O concurrency tuning.

== ZFS Module Parameters ==

These parameters are set via the ZFS kernel module (zfs.ko). They can be changed at runtime by writing to /sys/module/zfs/parameters/<param> or made persistent via /etc/modprobe.d/zfs.conf.

=== zfs_arc_max ===

Maximum size of the Adaptive Replacement Cache (ARC) in bytes. The ARC caches both file data and metadata blocks. On a dedicated OSS node where Lustre is the primary consumer of memory, increasing this allows more data to stay cached and reduces backend I/O.

Default: 50% of RAM
Suggested for OSS: 75% of RAM

Example (set to 96 GiB on a 128 GiB node):
  echo 103079215104 > /sys/module/zfs/parameters/zfs_arc_max

Or persistently in /etc/modprobe.d/zfs.conf:
  options zfs zfs_arc_max=103079215104

=== zfs_dirty_data_max ===

Maximum amount of dirty (uncommitted) data system-wide, in bytes. Increasing this allows the server to absorb more workload variation before ZFS begins throttling writers. The value should ideally match the backend storage capability, allowing 2–3 seconds of dirty data to be aggregated for write merging and more efficient I/O ordering.

The code uses 10% of system memory as the default, capped by zfs_dirty_data_max_max (default 25% of RAM or 4 GiB, whichever is less). Setting zfs_dirty_data_max explicitly bypasses the zfs_dirty_data_max_max cap.

Default: 10% of RAM (max 4 GiB)
Suggested for OSS: 10% of RAM, or 2–3 seconds of full-bandwidth writes (whichever is larger)

Note (ZFS 2.x change): In ZFS 2.0+, the default cap zfs_dirty_data_max_max was changed from a hard 4 GiB to 25% of RAM. On systems with large memory, the auto-computed default is now more generous than in ZFS 0.7.x. You may still want to set an explicit value for predictable behavior.

Example (set to 8 GiB):
  echo 8589934592 > /sys/module/zfs/parameters/zfs_dirty_data_max

=== zfetch_max_distance ===

Maximum number of bytes to prefetch per stream. Lustre OSS workloads frequently involve large sequential reads, so a higher prefetch distance keeps the read pipeline full and reduces I/O stalls.

Default: 8 MiB (ZFS 0.7.x); 64 MiB (ZFS 2.1+)
Suggested for OSS: 64 MiB (67108864 bytes)

Note (ZFS 2.x change): The default was increased from 8 MiB to 64 MiB in ZFS 2.1. If running ZFS 2.1+, the default already matches the suggested value. On older ZFS versions, set this explicitly.

Example:
  echo 67108864 > /sys/module/zfs/parameters/zfetch_max_distance

=== metaslab_debug_unload ===

When set to 1, prevents ZFS from unloading metaslab spacemaps from memory after they are loaded. This avoids repeated I/O to re-read spacemaps during allocation, at the cost of increased memory usage. On OSS nodes with sufficient RAM, keeping spacemaps cached reduces allocation latency for writes.

Default: 0
Suggested for OSS: 1

Example:
  echo 1 > /sys/module/zfs/parameters/metaslab_debug_unload

=== I/O Scheduling (formerly zfs_vdev_scheduler) ===

Note: The zfs_vdev_scheduler parameter was removed in ZFS 0.8.0. In prior versions, it allowed ZFS to override the Linux I/O scheduler for its VDEVs (commonly set to "deadline"). Starting with ZFS 0.8+, I/O scheduling is handled entirely by the Linux block layer via blk-mq.

For OSS workloads on modern kernels, configure the I/O scheduler at the block device level:

  echo mq-deadline > /sys/block/sda/queue/scheduler

Or persistently via a udev rule:
  # /etc/udev/rules.d/60-lustre-oss-scheduler.rules
  ACTION=="add|change", KERNEL=="sd*", ATTR{queue/scheduler}="mq-deadline"

For NVMe devices, the "none" scheduler is typically best since the device has its own internal scheduling:
  ACTION=="add|change", KERNEL=="nvme*", ATTR{queue/scheduler}="none"

=== VDEV I/O Concurrency ===

The following parameters control how many concurrent I/Os ZFS will issue to each VDEV. The defaults are conservative and designed for single-spindle configurations. OSS workloads benefit from higher concurrency, especially with multi-disk VDEV groups (mirrors, raidz) or SSDs.

==== zfs_vdev_async_read_max_active ====

Maximum asynchronous read I/Os active to each device.

Default: 3
Suggested for OSS: 16

==== zfs_vdev_async_write_min_active ====

Minimum number of asynchronous write I/Os active to each device. ZFS uses this as the floor when dirty data is below the zfs_vdev_async_write_active_min_dirty_percent threshold.

Default: 2 (ZFS 0.7.x); 1 (ZFS 2.x)
Suggested for OSS: 5

Note (ZFS 2.x change): The default was lowered from 2 to 1 in ZFS 2.0. Setting this explicitly to 5 is recommended for OSS workloads to avoid write stalls when dirty data is low.

==== zfs_vdev_async_write_max_active ====

Maximum number of asynchronous write I/Os active to each device.

Default: 10
Suggested for OSS: 10

==== zfs_vdev_async_write_active_min_dirty_percent ====

Threshold (as percentage of zfs_dirty_data_max) below which the I/O scheduler limits concurrent write operations to zfs_vdev_async_write_min_active. Above this threshold, concurrent write operations increase linearly up to zfs_vdev_async_write_max_active.

Default: 30
Suggested for OSS: 20

Lowering this causes ZFS to ramp up write concurrency sooner, which helps prevent dirty data accumulation on busy OSS nodes.

==== zfs_vdev_sync_read_min_active ====

Minimum number of synchronous read I/Os active to each device.

Default: 10
Suggested for OSS: 16

==== zfs_vdev_sync_read_max_active ====

Maximum number of synchronous read I/Os active to each device.

Default: 10
Suggested for OSS: 16

==== zfs_vdev_aggregation_limit ====

Maximum amount of data to aggregate into a single I/O request. Larger values allow ZFS to merge adjacent writes into fewer, larger I/Os, which improves throughput on bulk-data OSS workloads.

Default: 1 MiB (ZFS 2.x); 128 KiB (ZFS 0.7.x)
Suggested for OSS: Match the ZFS recordsize (typically 1–4 MiB)

Note (ZFS 2.x change): The default was increased from 128 KiB to 1 MiB in ZFS 2.0. If your recordsize is 1 MiB and you are running ZFS 2.x, the default may already be sufficient.

Example (set to 4 MiB):
  echo 4194304 > /sys/module/zfs/parameters/zfs_vdev_aggregation_limit

=== Write Throttling Behavior ===

When dirty data is less than zfs_vdev_async_write_active_min_dirty_percent of zfs_dirty_data_max, ZFS keeps only zfs_vdev_async_write_min_active outstanding writes per VDEV. Dirty data will build up more quickly below this threshold, and because the default is only 1–2 outstanding writes per disk, ZFS can start to delay or even halt writers.

By lowering zfs_vdev_async_write_active_min_dirty_percent (from 30 to 20) and raising zfs_vdev_async_write_min_active (from 1–2 to 5), the write pipeline stays active at lower dirty data levels, avoiding unnecessary throttling on busy OSS nodes.

== Lustre ZFS OSD Module Parameters ==

These parameters are specific to the Lustre ZFS OSD (osd-zfs.ko). They control how Lustre interacts with the underlying ZFS pool. They can be viewed and set via lctl or through /sys/module/osd_zfs/parameters/.

=== osd_txg_sync_delay_us (module param) ===

Delay in microseconds before syncing a ZFS transaction group. When set to zero or a positive value, Lustre inserts this delay instead of issuing an immediate TXG sync. Increasing this allows more operations to be batched into a single TXG, which can improve throughput at the cost of slightly increased commit latency.

When set to -1 (the default), normal TXG sync behavior is used with no added delay.

Default: -1 (disabled)
Set via /etc/modprobe.d/osd_zfs.conf:
  options osd_zfs osd_txg_sync_delay_us=0

=== osd_object_sync_delay_us (module param) ===

Delay in microseconds for individual object sync operations. When set to zero or a positive value, Lustre inserts this delay instead of issuing an immediate object sync. This can reduce sync overhead on busy OSS nodes at the cost of slightly increased per-object sync latency.

When set to -1 (the default), normal object sync behavior is used with no added delay.

Default: -1 (disabled)
Set via /etc/modprobe.d/osd_zfs.conf:
  options osd_zfs osd_object_sync_delay_us=0

=== Per-target tunables (via lctl) ===

These are set per OST device, e.g., for an OST named "lustre-OST0000":

  lctl set_param osd-zfs.lustre-OST0000.<param>=<value>

readcache_max_filesize (RW)
  Maximum file size (in bytes) eligible for the read cache. Files larger than this value bypass the OSD read cache. On OSTs handling very large files, lowering this can free memory for other uses. The default is effectively unlimited (all files are eligible).

  Default: 18446744073709551615 (OBD_OBJECT_EOF — unlimited)

  Example (limit read caching to files ≤ 64 MiB):
  lctl set_param osd-zfs.lustre-OST0000.readcache_max_filesize=67108864

auto_scrub (RW)
  Enables or disables automatic OI scrubbing on the OST.

  lctl set_param osd-zfs.lustre-OST0000.auto_scrub=1

nonrotational (RW)
  Marks the underlying storage as non-rotational (SSD/NVMe). This hints to Lustre that seek-avoidance optimizations are unnecessary.

  lctl set_param osd-zfs.lustre-OST0000.nonrotational=1

force_sync (WO)
  Write-only parameter that forces an immediate sync of the OST. Useful for diagnostics or before administrative operations.

  lctl set_param osd-zfs.lustre-OST0000.force_sync=1

== Recommended OSS Tuning Summary ==

For a dedicated OSS node with 128 GiB RAM and SAS/SATA disk-based ZFS pool:

  # ZFS module parameters (/etc/modprobe.d/zfs.conf)
  options zfs zfs_arc_max=103079215104
  options zfs zfs_dirty_data_max=8589934592
  options zfs zfetch_max_distance=67108864
  options zfs metaslab_debug_unload=1
  options zfs zfs_vdev_async_read_max_active=16
  options zfs zfs_vdev_async_write_min_active=5
  options zfs zfs_vdev_async_write_max_active=10
  options zfs zfs_vdev_async_write_active_min_dirty_percent=20
  options zfs zfs_vdev_sync_read_min_active=16
  options zfs zfs_vdev_sync_read_max_active=16
  options zfs zfs_vdev_aggregation_limit=4194304

  # Linux I/O scheduler (via udev rule)
  ACTION=="add|change", KERNEL=="sd*", ATTR{queue/scheduler}="mq-deadline"

  # Lustre ZFS OSD module parameters (/etc/modprobe.d/osd_zfs.conf)
  options osd_zfs osd_txg_sync_delay_us=0
  options osd_zfs osd_object_sync_delay_us=0

In addition to the kernel module parameters, it is recommended that ZFS compression is enabled when creating ZFS datasets for OSTs. See Creating Lustre Object Storage Services (OSS) for examples of the commands to create OSTs with compression enabled.
