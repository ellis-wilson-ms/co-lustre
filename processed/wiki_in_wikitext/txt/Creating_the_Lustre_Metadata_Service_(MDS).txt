== Syntax Overview ==

The syntax for creating an MDT is:

<pre style="overflow-x:auto;">
mkfs.lustre --mdt \
  [--reformat] \
  --fsname <name> \
  --index <n> \
  --mgsnode <MGS NID> [--mgsnode <MGS NID> …] \
  [ --servicenode <NID> [--servicenode <NID> …]] \
  [ --failnode <NID> [--failnode <NID> …]] \
  [ --backfstype=ldiskfs|zfs ] \
  [ --mkfsoptions <options> ] \
  <device path> | <pool name>/<dataset> <zpool specification>
</pre>

The command line syntax for formatting an MDT incorporates several additional parameters when compared to that of the much simpler [[Creating the Lustre Management Service (MGS)|MGT]]. First, the metadata target requires a record of the [[Lustre Networking (LNET) Overview|NIDs]] that can provide the [[Lustre Management Service (MGS)|Lustre management service (MGS)]]. The MGS NIDs are supplied using the <code>--mgsnode</code> flag. If there is more than one potential location for the MGS (i.e., it is part of a high availability failover cluster configuration), then the option is repeated for as many failover nodes as are configured (usually there are two).

Ordering of the MGS nodes in the command line is significant: the first <code>--mgsnode</code> flag must reference the NID of the current active or primary MGS server. If this is not the case, then the first time that the MDS tries to join the cluster, it will fail. The first time mount of a storage target does not currently check the failover locations when trying to establish a connection with the MGS. When adding new storage targets to Lustre, the MGS must be running on its primary NID.

The MDT must also be supplied with the name (<code>--fsname</code>) of the Lustre file system (maximum 8 characters), and a unique index number (<code>--index</code>) for the file system. There must always be an MDT with <code>index=0</code> (zero) for each file system, representing the root of the file system tree. For many Lustre file systems, a single MDT (referred to as <code>MDT0</code>) is sufficient.

The list of service nodes (<code>--servicenode</code>) or failover nodes (<code>--failnode</code>) must be specified for any high availability configuration. Although there are more compact declarations for defining the nodes, for simplicity, list the NID of each server that can mount the storage as a separate <code>--servicenode</code> entry.

The next example uses the <code>--servicenode</code> syntax to create an MDT that can be run on two servers as an HA failover resource:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo \
>   --index 0 \
>   --mgsnode 192.168.227.11@tcp1 \
>   --mgsnode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.11@tcp1 \
>   /dev/dm-2
</pre>

The command line formats a new MDT that will be used by the MDS for storage. The MDT will provide metadata for a file system called <code>demo</code>, and has index number <code>0</code> (zero). There are two NIDs defined as the nodes able to host the MDS service, denoted by the <code>--servicenode</code> options, and two NIDs supplied for the MGS that the MDS will register with.

The <code>--failnode</code> syntax is similar, but is used to define only a failover target for the storage service. For example:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo \
>   --index 0 \
>   --mgsnode 192.168.227.11@tcp1 \
>   --mgsnode 192.168.227.12@tcp1 \
>   --failnode 192.168.227.11@tcp1 \
>   /dev/dm-2
</pre>

Here, the failover host is identified as <code>192.168.227.11@tcp1</code>, one server in an HA pair (and which, for the purpose of this example, has the hostname <code>rh7z-mds1</code>). The <code>mkfs.lustre</code> command was executed on <code>rh7z-mds2</code> (NID: <code>192.168.227.12@tcp1</code>), and the <code>mount</code> command must also be run from this host when the service starts for the very first time.

== MDT Formatted as an LDISKFS OSD ==

The syntax for creating an LDISKFS-based MDT is:

<pre style="overflow-x:auto;">
mkfs.lustre --mdt \
  [--reformat] \
  --fsname <name> \
  --index <n> \
  --mgsnode <MGS NID> [--mgsnode <MGS NID> …] \
  [ --servicenode <NID> [--servicenode <NID> …]] \
  [ --failnode <NID> [--failnode <NID> …]] \
  [ --backfstype=ldiskfs ] \
  [ --mkfsoptions <options> ] \
  <device path>
</pre>

The following example uses the <code>--servicenode</code> syntax to create an MDT that can be run on two servers as an HA failover resource:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo \
>   --index 0 \
>   --mgsnode 192.168.227.11@tcp1 \
>   --mgsnode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.11@tcp1 \
>   --backfstype=ldiskfs \
>   /dev/dm-2
</pre>

The <code>--failnode</code> syntax is similar, but is used to define only a failover target for the storage service. For example:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo --index 0 \
>   --mgsnode 192.168.227.11@tcp1 --mgsnode 192.168.227.12@tcp1 \
>   --failnode 192.168.227.11@tcp1 \
>   --backfstype=ldiskfs \
>   /dev/dm-2
</pre>

The above examples are repeated from the main introduction to the syntax, but are included here to maintain symmetry with the rest of the text. However, note that the <code>--backfstype</code> flag has been set to <code>ldiskfs</code>, which tells <code>mkfs.lustre</code> to format the device as an LDISKFS OSD.

== MDT Formatted as a ZFS OSD ==

=== Formatting an MDT using only the mkfs.lustre command ===

'''Note:''' For the greatest flexibility and control when creating ZFS-based Lustre storage targets, do not use this approach – instead, create the zpool separately from formatting the Lustre OSD. See [[#Formatting an MDT using zpool and mkfs.lustre|Formatting an MDT using zpool and mkfs.lustre]].

The syntax for creating a ZFS-based MDT using only the <code>mkfs.lustre</code> command is:

<pre style="overflow-x:auto;">
mkfs.lustre --mdt \
  [--reformat] \
  --fsname <name> \
  --index <n> \
  --mgsnode <MGS NID> [--mgsnode <MGS NID> …] \
  [ --servicenode <NID> [--servicenode <NID> …]] \
  [ --failnode <NID> [--failnode <NID> …]] \
  --backfstype=zfs \
  [ --mkfsoptions <options> ] \
  <pool name>/<dataset> \
  <zpool specification>
</pre>

The next example uses the <code>--servicenode</code> syntax to create an MDT that can be run on two servers as an HA failover resource:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo \
>   --index 0 \
>   --mgsnode 192.168.227.11@tcp1 --mgsnode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.12@tcp1 \
>   --servicenode 192.168.227.11@tcp1 \
>   --backfstype=zfs \
>   demo-mdt0pool/mdt0 \
>   mirror sdb sdd
</pre>

In addition to defining the parameters of the MDT service itself, the command defines a mirrored ZFS zpool called <code>demo-mdt0pool</code> consisting of two devices, and creates a ZFS dataset called <code>mdt0</code>. Normally, it is expected that the MDT will be created from a larger pool of storage, to maximize performance and meet requirements for capacity; the above example is provided for the purposes of outlining the command line syntax.

The <code>--failnode</code> syntax is similar, but is used to define only a failover target for the storage service. For example:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 system]# mkfs.lustre --mdt \
>   --fsname demo --index 0 \
>   --mgsnode 192.168.227.11@tcp1 --mgsnode 192.168.227.12@tcp1 \
>   --failnode 192.168.227.11@tcp1 \
>   --backfstype=zfs \
>   demo-mdt0pool/mdt0 mirror sdb sdd
</pre>

'''Note:''': When creating a ZFS-based OSD using only the <code>mkfs.lustre</code> command, it is not possible to set or change some properties of the zpool or its vdevs, such as the <code>multihost</code> and <code>ashift</code> properties. For this reason, it is highly recommended that the zpools be created independently of the <code>mkfs.lustre</code> command, as shown in the next section.

=== Formatting an MDT using zpool and mkfs.lustre ===

To create a ZFS-based MDT, create a zpool to contain the MDT file system dataset, then use <code>mkfs.lustre</code> to create the actual file system dataset inside the zpool:

<pre style="overflow-x:auto;">
zpool create [-f] -O canmount=off -o multihost=on \
  [-o ashift=<n>] \
  -o cachefile=/etc/zfs/<zpool name>.spec | -o cachefile=none \
  <zpool name> \
  <zpool specification>

mkfs.lustre --mdt \
  [--reformat] \
  --fsname <name> \
  --index <n> \
  --mgsnode <MGS NID> [--mgsnode <MGS NID> …] \
  [ --servicenode <NID> [--servicenode <NID> …]] \
  [ --failnode <NID> [--failnode <NID> …]] \
  --backfstype=zfs \
  [ --mkfsoptions <options> ] \
  <pool name>/<dataset>
</pre>

For example:

<pre style="overflow-x:auto;">
# Create the zpool
# Pool will comprise 3 mirrors each with 2 devices.
# Mirrors will be concatenated (striped).
zpool create -O canmount=off -o multihost=on \
  -o cachefile=none \
  demo-mdt0pool \
  mirror sdd sde mirror sdf sdg mirror sdh sdi

# Format MDT0 for Lustre file system "demo"
mkfs.lustre --mdt \
  --fsname demo \
  --index 0 \
  --mgsnode 192.168.227.11@tcp1 --mgsnode 192.168.227.12@tcp1 \
  --servicenode 192.168.227.12@tcp1 \
  --servicenode 192.168.227.11@tcp1 \
  --backfstype=zfs \
  demo-mdt0pool/mdt0
</pre>

The output from the above example will look something like this:

== Starting the MDS Service ==

The <code>mount</code> command is used to start all Lustre storage services, including the MDS. The syntax is:

<pre style="overflow-x:auto;">
mount -t lustre [-o <options>] \
  <ldiskfs blockdev>|<zpool>/<dataset> <mount point>
</pre>

The <code>mount</code> command syntax is very similar for both LDISKFS and ZFS storage targets. The main difference is the format of the path to the storage. For LDISKFS, the path will resolve to a block device, such as <code>/dev/sda</code> or <code>/dev/mapper/mpatha</code>, whereas for ZFS, the path resolves to a dataset in a zpool, e.g. <code>demo-mdt0pool/mdt0</code>.

The mount point directory must exist before the mount command is executed. The recommended convention for the mount point of the MDT storage is <code>/lustre/<fsname>/mdt<n></code>, where <code>&lt;fsname&gt;</code> is the name of the file system and <code>&lt;n&gt;</code> is the index number of the MDT.

The following example starts a ZFS-based MDS:

<pre style="overflow-x:auto;">
# Ignore MOUNTPOINT column in output: not used by Lustre
[root@rh7z-mds2 ~]# zfs list
NAME                 USED  AVAIL  REFER  MOUNTPOINT
demo-mdt0pool       2.87M  9.62G    19K  /demo-mdt0pool
demo-mdt0pool/mdt0  2.79M  9.62G  2.79M  /demo-mdt0pool/mdt0

[root@rh7z-mds2 ~]# mkdir -p /lustre/demo/mdt0

[root@rh7z-mds2 ~]# mount -t lustre demo-mdt0pool/mdt0 /lustre/demo/mdt0

[root@rh7z-mds2 ~]# df -ht lustre
File system          Size  Used Avail Use% Mounted on
demo-mdt0pool/mdt0  9.7G  2.8M  9.7G   1% /lustre/demo/mdt0
</pre>

'''Note:''' The default output for <code>zfs list</code> shows mount points for the <code>demo-mdt0pool</code> Pool and <code>mdt0</code> dataset in the <code>MOUNTPOINT</code> column. Just as for all ZFS-formatted OSDs, the content in this column can be ignored.

To reduce confusion, the ZFS file system <code>mountpoint</code> property can be set equal to <code>none</code>. For example:

<pre style="overflow-x:auto;">
zfs set mountpoint=none demo-mdt0pool
zfs set mountpoint=none demo-mdt0pool/mdt0
</pre>

'''Note:''' Only the <code>mount -t lustre</code> command can start Lustre services. Mounting storage as type <code>ldiskfs</code> or <code>zfs</code> will mount a storage target on the host, but it will not trigger the startup of the requisite Lustre kernel processes.

== Stopping the MDS Service ==

To stop a Lustre service, umount the corresponding target:

<pre style="overflow-x:auto;">
umount <mount point>
</pre>

The mount point must correspond to the mount point used with the <code>mount -t lustre</code> command. For example:

<pre style="overflow-x:auto;">
[root@rh7z-mds2 ~]# df -ht lustre
File system          Size  Used Avail Use% Mounted on
demo-mdt0pool/mdt0  9.7G  2.8M  9.7G   1% /lustre/demo/mdt0
[root@rh7z-mds2 ~]# umount /lustre/demo/mdt0
[root@rh7z-mds2 ~]# df -ht lustre
df: no file systems processed
[root@rh7z-mds2 ~]# lctl dl
[root@rh7z-mds2 ~]#
</pre>

Using the regular <code>umount</code> command is the correct way to stop a given Lustre service and unmount the associated storage, for both LDISKFS and ZFS-based Lustre storage volumes.

Do not use the <code>zfs unmount</code> command to stop a Lustre service. Attempting to use <code>zfs</code> commands to unmount a storage target that is mounted as part of an active Lustre service will return an error.
