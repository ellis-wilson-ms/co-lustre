== Overview ==

Lustre supports multi-tenancy — isolating multiple user groups or organizations on a single filesystem — through two complementary mechanisms:

* '''Nodemaps''' — map client UIDs/GIDs to server-side values, apply per-group security policies (root squash, read-only, audit, encryption).
* '''Filesets''' (subdirectory mounts) — restrict clients to see only a subdirectory of the filesystem as their mount root.

When combined, nodemaps and filesets provide strong tenant isolation: each tenant has its own UID/GID space, its own root squash policy, and can only access its own portion of the filesystem.

For detailed nodemap reference, see [[UID GID Mapping]]. For general security setup, see [[Setting up Lustre Security]].

== Nodemaps ==

Nodemaps group clients by network address and apply identity mapping and security policies to each group.

=== Creating a Nodemap ===

 lctl nodemap_add tenantA

This creates a new nodemap named <code>tenantA</code>.

=== Adding Client Ranges ===

Associate client NID ranges with the nodemap:

 lctl nodemap_add_range --name tenantA --range 10.0.1.[0-255]@tcp

Multiple ranges can be added:

 lctl nodemap_add_range --name tenantA --range 10.0.1.[0-255]@tcp
 lctl nodemap_add_range --name tenantA --range 10.0.2.[0-255]@tcp

Clients connecting from these NIDs are automatically assigned to the <code>tenantA</code> nodemap.

=== Configuring Security Properties ===

Each nodemap has several configurable properties:

{| class="wikitable"
! Property
! Description
! Example
|-
| <code>admin</code>
| Allow client root to have root privileges on the server (0 = squash root, 1 = allow)
| <code>lctl nodemap_modify --name tenantA --property admin --value 0</code>
|-
| <code>trusted</code>
| Trust client-supplied UID/GID without mapping (0 = map, 1 = trust)
| <code>lctl nodemap_modify --name tenantA --property trusted --value 0</code>
|-
| <code>squash_uid</code>
| UID to map root (uid 0) to when admin=0
| <code>lctl nodemap_modify --name tenantA --property squash_uid --value 99</code>
|-
| <code>squash_gid</code>
| GID to map root (gid 0) to when admin=0
| <code>lctl nodemap_modify --name tenantA --property squash_gid --value 99</code>
|-
| <code>readonly_mount</code>
| Force clients to mount read-only
| <code>lctl nodemap_modify --name tenantA --property readonly_mount --value 1</code>
|-
| <code>audit_mode</code>
| Enable or disable audit logging for clients in this nodemap
| <code>lctl nodemap_modify --name tenantA --property audit_mode --value 1</code>
|-
| <code>forbid_encryption</code>
| Prevent clients from using client-side encryption
| <code>lctl nodemap_modify --name tenantA --property forbid_encryption --value 0</code>
|}

A typical multi-tenant configuration disables admin and trusted:

 lctl nodemap_modify --name tenantA --property admin --value 0
 lctl nodemap_modify --name tenantA --property trusted --value 0
 lctl nodemap_modify --name tenantA --property squash_uid --value 99
 lctl nodemap_modify --name tenantA --property squash_gid --value 99

=== UID/GID Mapping ===

Nodemaps can translate UIDs and GIDs between client-side and server-side values. This allows different tenants to use overlapping UID spaces without conflict:

 # Map client UID 500 to server UID 1000
 lctl nodemap_add_idmap --name tenantA --idtype uid --idmap 500:1000
 
 # Map client GID 500 to server GID 1000
 lctl nodemap_add_idmap --name tenantA --idtype gid --idmap 500:1000

Range mappings apply an offset to a block of IDs:

 # Map client UIDs 1000-1999 to server UIDs 10000-10999
 lctl nodemap_add_idmap --name tenantA --idtype uid --idmap 1000:10000

To remove a mapping:

 lctl nodemap_del_idmap --name tenantA --idtype uid --idmap 500:1000

=== Viewing Nodemap Configuration ===

 # List all nodemaps
 lctl nodemap_info
 
 # Show details for a specific nodemap
 lctl get_param nodemap.tenantA.*

=== Activating Nodemaps ===

Nodemaps must be activated to take effect:

 lctl set_param nodemap.active=1

Check activation status:

 lctl get_param nodemap.active

== Filesets (Subdirectory Mounts) ==

Filesets allow clients to mount a subdirectory of the filesystem as their root. The client cannot see or access anything above the mounted subdirectory.

=== Mounting a Fileset ===

 mount -t lustre mgs@tcp:/fsname/tenantA_data /mnt/tenant

The client sees <code>/mnt/tenant</code> as the filesystem root. Paths like <code>../../</code> cannot escape the fileset boundary.

=== Creating the Subdirectory ===

The subdirectory must exist before clients can mount it:

 # On a client with full filesystem access
 mkdir -p /mnt/lustre/tenantA_data
 chown root:root /mnt/lustre/tenantA_data
 chmod 755 /mnt/lustre/tenantA_data

=== Fileset with Nodemap ===

For stronger isolation, configure the nodemap to enforce a specific fileset:

 lctl nodemap_set_fileset --name tenantA --fileset /tenantA_data

When this is set, clients in the <code>tenantA</code> nodemap automatically mount the specified subdirectory, even if they attempt to mount the full filesystem:

 # Client in tenantA nodemap mounts:
 mount -t lustre mgs@tcp:/fsname /mnt/tenant
 # Effective mount point is /fsname/tenantA_data

This prevents tenants from bypassing the fileset restriction.

== Combining Nodemaps and Filesets ==

A complete multi-tenant setup:

 # 1. Create nodemaps for each tenant
 lctl nodemap_add tenantA
 lctl nodemap_add tenantB
 
 # 2. Assign network ranges
 lctl nodemap_add_range --name tenantA --range 10.0.1.[0-255]@tcp
 lctl nodemap_add_range --name tenantB --range 10.0.2.[0-255]@tcp
 
 # 3. Disable admin/trusted for both
 lctl nodemap_modify --name tenantA --property admin --value 0
 lctl nodemap_modify --name tenantA --property trusted --value 0
 lctl nodemap_modify --name tenantB --property admin --value 0
 lctl nodemap_modify --name tenantB --property trusted --value 0
 
 # 4. Set root squash
 lctl nodemap_modify --name tenantA --property squash_uid --value 99
 lctl nodemap_modify --name tenantA --property squash_gid --value 99
 lctl nodemap_modify --name tenantB --property squash_uid --value 99
 lctl nodemap_modify --name tenantB --property squash_gid --value 99
 
 # 5. Map UIDs to non-overlapping server ranges
 lctl nodemap_add_idmap --name tenantA --idtype uid --idmap 1000:10000
 lctl nodemap_add_idmap --name tenantB --idtype uid --idmap 1000:20000
 
 # 6. Assign filesets
 lctl nodemap_set_fileset --name tenantA --fileset /tenantA_data
 lctl nodemap_set_fileset --name tenantB --fileset /tenantB_data
 
 # 7. Activate nodemaps
 lctl set_param nodemap.active=1

With this configuration:
* TenantA clients (10.0.1.x) can only see <code>/tenantA_data</code> and their UID 1000 maps to server UID 10000.
* TenantB clients (10.0.2.x) can only see <code>/tenantB_data</code> and their UID 1000 maps to server UID 20000.
* Neither tenant's root user has admin privileges.

== Multiple Filesets per Nodemap (Lustre 2.17+) ==

Starting with Lustre 2.17, a single nodemap can be associated with multiple filesets. This allows a tenant to have access to several isolated directory trees:

 lctl nodemap_set_fileset --name tenantA --fileset /tenantA_data
 lctl nodemap_set_fileset --name tenantA --fileset /tenantA_scratch

Clients can then mount either fileset:

 mount -t lustre mgs@tcp:/fsname/tenantA_data /mnt/data
 mount -t lustre mgs@tcp:/fsname/tenantA_scratch /mnt/scratch

== Making Configuration Persistent ==

Nodemap configuration is stored on the MGS and persists across server restarts. However, ensure the MGS is updated:

 # Verify persistent configuration
 lctl get_param nodemap.tenantA.*
 
 # Changes made with lctl nodemap_* commands are automatically
 # persisted to the MGS configuration log.

To verify on restart:

 lctl get_param nodemap.active
