== Overview ==

Starting with '''Lustre 2.14''', client-side encryption is supported using the Linux '''fscrypt''' framework. Files are encrypted and decrypted transparently on the client — data stored on OSTs and metadata stored on MDTs is encrypted at rest. Servers never see plaintext data.

This feature provides data confidentiality without requiring any changes to the Lustre server infrastructure.

'''Note:''' Client-side encryption is a relatively recent feature. Always refer to the Lustre Operations Manual and release notes for your specific version for authoritative guidance.

For broader security topics, see [[Setting up Lustre Security]].

== How It Works ==

* Encryption and decryption happen entirely on the '''client''' side.
* Each encrypted directory has an '''encryption policy''' that specifies the encryption algorithm and a reference to the encryption key.
* Files within an encrypted directory inherit the policy — all file data and filenames are encrypted.
* Servers (MDS and OSS) store only ciphertext. They do not need access to encryption keys.
* Clients without the correct key can see that files exist but see '''ciphertext filenames''' (encoded, unreadable names) and cannot read file contents.

| <code>CONFIG_FS_ENCRYPTION=y</code> (or <code>=m</code>) in the client kernel
|-
| fscrypt userspace tool
| <code>fscrypt</code> CLI (v0.3.0+) or <code>fscryptctl</code>
|-
| Server support
| Lustre 2.14+ servers (they store ciphertext transparently)
|-
| Filesystem feature flag
| <code>encrypt</code> feature must be enabled on MDT and OSTs (enabled by default on 2.14+ with ldiskfs or zfs)
|}

Verify kernel support:

 grep -i encryption /boot/config-$(uname -r)

Check that the Lustre filesystem supports encryption:

 lctl get_param mdt.*.enable_encryption
 # Should return 1

== Key Management ==

Encryption keys are managed through the Linux kernel keyring subsystem. Two common approaches:

=== Using the fscrypt Tool ===

The <code>fscrypt</code> tool (https://github.com/google/fscrypt) provides a user-friendly interface:

 # Initialize fscrypt metadata on the filesystem (one-time setup)
 fscrypt setup /mnt/lustre
 
 # Create a protector (passphrase-based)
 fscrypt encrypt /mnt/lustre/encrypted_dir

This will prompt for a passphrase and set up the encryption policy on the directory.

=== Using fscryptctl ===

The lower-level <code>fscryptctl</code> tool provides more direct control:

 # Generate a new key
 fscryptctl add_key /mnt/lustre
 
 # Set encryption policy on a directory (must be empty)
 fscryptctl set_policy <key_descriptor> /mnt/lustre/encrypted_dir

=== Using the Kernel Keyring Directly ===

For automated or scripted workflows:

 # Add a key to the filesystem keyring
 keyctl add logon fscrypt:<descriptor> <raw_key> @s
 
 # Or use the kernel's add_key system call programmatically

== Setting Up an Encrypted Directory ==

Step-by-step process:

 # 1. Create the directory (must be empty to set a policy)
 mkdir /mnt/lustre/secure_data
 
 # 2. Set up encryption using fscrypt
 fscrypt encrypt /mnt/lustre/secure_data
 # Follow the prompts to choose a protector (passphrase, PAM, etc.)
 
 # 3. Files created in this directory are now automatically encrypted
 echo "sensitive data" > /mnt/lustre/secure_data/secret.txt
 
 # 4. Verify encryption is active
 fscrypt status /mnt/lustre/secure_data

== Access Semantics ==

=== With the Key ===

Clients that have the encryption key loaded in their kernel keyring see files normally:

 $ ls /mnt/lustre/secure_data/
 secret.txt
 $ cat /mnt/lustre/secure_data/secret.txt
 sensitive data

=== Without the Key ===

Clients without the key (or after the key is removed) see encrypted filenames:

 $ ls /mnt/lustre/secure_data/
 ,cVx3kR2mE8p4nQ7...    # ciphertext filename
 $ cat /mnt/lustre/secure_data/,cVx3kR2mE8p4nQ7...
 # Returns -ENOKEY

Servers also see only ciphertext:

 # On the MDS
 debugfs /dev/sda1 -R "ls /ROOT/secure_data"
 # Shows encrypted filename entries

=== Locking the Directory ===

To remove the key and lock the directory on a client:

 fscrypt lock /mnt/lustre/secure_data

After locking, the directory reverts to showing ciphertext filenames until the key is provided again.

== Encryption with Lustre Striping ==

Encryption works transparently with Lustre striping. When a file is striped across multiple OSTs:

* Each stripe is independently encrypted with the same key.
* The encryption is applied at the client level before data is sent to OSTs.
* Stripe size, stripe count, and OST pool settings are unaffected.

 # Setting striping on an encrypted directory works normally
 lfs setstripe -c 4 -S 4M /mnt/lustre/secure_data/

== Limitations and Considerations ==

* '''Filename length''' — encrypted filenames are longer than plaintext names due to encoding. This may affect the maximum filename length (typically 255 bytes). Very long filenames may fail with <code>-ENAMETOOLONG</code>.
* '''CPU overhead''' — encryption and decryption consume CPU cycles on the client. AES-NI hardware acceleration (available on modern x86 CPUs) significantly reduces this overhead.
* '''No server-side search''' — since filenames and data are encrypted, server-side operations like <code>lfs find</code> by name will not match plaintext names.
* '''Backup considerations''' — backups of encrypted data capture ciphertext. Restoring requires the same encryption key. Ensure keys are backed up securely and independently of the data.
* '''Cross-client access''' — all clients that need to read encrypted files must have access to the same encryption key. Key distribution is the administrator's responsibility.
* '''Immutable policy''' — once an encryption policy is set on a directory, it cannot be removed. Files must be copied out, the directory removed, and recreated without encryption.
* '''Empty directory requirement''' — the encryption policy can only be set on an empty directory.

== Checking Encryption Status ==

 # Check if a directory is encrypted
 fscrypt status /mnt/lustre/secure_data
 
 # Check filesystem-level encryption support
 fscrypt status /mnt/lustre
 
 # Check server-side feature flag
 lctl get_param mdt.*.enable_encryption
