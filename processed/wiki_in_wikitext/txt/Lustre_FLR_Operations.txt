== Overview ==

File Level Redundancy (FLR) allows individual Lustre files to have multiple ''mirrors'' — independent copies of the file data stored on different OSTs. FLR provides:

* '''Application-level redundancy''' without requiring RAID at the storage layer.
* '''Cross-pool replication''' — place mirrors on different OST pools (e.g., flash vs. spinning disk).
* '''Cross-site replication''' — mirrors can reside on OSTs at different locations.
* '''Read performance''' — reads can be served from any up-to-date mirror.

FLR uses a ''delayed write'' model: during writes, only the preferred mirror is updated directly. Other mirrors are marked stale and must be explicitly resynchronized.

== Creating Mirrored Files ==

Use <code>lfs mirror create</code> to create a new file with multiple mirrors. Each <code>-N</code> flag defines a mirror with its own layout:

 lfs mirror create -N -S 4M -c 2 -p flash -N -c -1 -p archive /mnt/lustre/file

This creates a file with two mirrors:
* '''Mirror 1:''' 4 MB stripe size, 2 stripes, on the <code>flash</code> pool.
* '''Mirror 2:''' full stripe count (<code>-1</code>), on the <code>archive</code> pool.

Additional options per mirror:

{| class="wikitable"
! Option !! Description
|-
| <code>-N [count]</code> || Define a new mirror (count specifies number of identical mirrors)
|-
| <code>-S size</code> || Stripe size for this mirror
|-
| <code>-c count</code> || Stripe count for this mirror (-1 = all OSTs in pool)
|-
| <code>-p pool</code> || OST pool for this mirror
|-
| <code>-o ost_indices</code> || Specific OST indices for this mirror
|}

== Mirror Flags ==

Each mirror has a set of state flags:

{| class="wikitable"
! Flag !! Meaning
|-
| <code>init</code> || Mirror is initialized and contains valid data
|-
| <code>stale</code> || Mirror is out of date (not in sync with the preferred mirror)
|-
| <code>prefer</code> || Mirror is preferred for writes and reads
|}

View mirror status with <code>lfs getstripe</code>:

 lfs getstripe /mnt/lustre/file

The output shows each mirror's ID, layout, and flags.

== Extending (Adding Mirrors) ==

Add a new mirror to an existing file:

 lfs mirror extend -N -c 2 -p nearline /mnt/lustre/file

The new mirror is initially stale and must be resynchronized.

=== Merging a Victim File ===

You can add an existing file as a mirror of another file using <code>--victim</code> (or <code>-f</code>):

 lfs mirror extend --victim /mnt/lustre/src_file /mnt/lustre/dest_file

The source file (<code>src_file</code>) is consumed and becomes a mirror of <code>dest_file</code>. The source file is removed after the merge. Both files must be on the same file system.

== Splitting Mirrors ==

Remove a mirror from a mirrored file:

 lfs mirror split --mirror-id 2 /mnt/lustre/file

By default, the split mirror is destroyed (<code>-d</code>). To preserve it as a separate file:

 lfs mirror split --mirror-id 2 -f /mnt/lustre/split_copy /mnt/lustre/file

Options:

{| class="wikitable"
! Option !! Description
|-
| <code>--mirror-id N</code> || The ID of the mirror to split off
|-
| <code>-d</code> || Destroy the split mirror (default)
|-
| <code>-f newfile</code> || Save the split mirror as a new independent file
|}

A file must retain at least one mirror after splitting.

== Resynchronizing Mirrors ==

After writes, stale mirrors must be explicitly resynchronized:

 lfs mirror resync /mnt/lustre/file

To resynchronize only a specific mirror:

 lfs mirror resync --only 2 /mnt/lustre/file

Resync copies data from an up-to-date mirror to all stale mirrors (or the specified mirror). After successful resync, the stale flag is cleared.

'''Best practice:''' Run <code>lfs mirror resync</code> periodically or after write operations complete. Consider using a cron job or monitoring tool to detect and resync stale mirrors.

== Verifying Mirrors ==

Verify that all mirrors contain identical data:

 lfs mirror verify /mnt/lustre/file

This reads and compares the data across all mirrors byte-by-byte. It is an I/O-intensive operation and should be used selectively.

A non-zero exit code indicates a mismatch. Stale mirrors are expected to differ and are skipped during verification.

== The Delayed-Write Model ==

FLR uses a '''delayed-write''' approach:

# During a write, only the '''preferred mirror''' (or the first non-stale mirror) receives the data.
# All other mirrors are marked '''stale'''.
# The user or administrator must explicitly run <code>lfs mirror resync</code> to bring stale mirrors up to date.

This design avoids the write amplification and latency penalty of synchronous mirroring. However, it means that between writes and resync, the file effectively has reduced redundancy.

'''Important:''' If the only up-to-date mirror's OST fails before resync, data for the dirty regions is lost. Plan resync frequency according to your durability requirements.

== Setting the Preferred Mirror ==

Mark a specific mirror as preferred for reads and writes:

 lfs setstripe --comp-set -I <mirror_id> --comp-flags=prefer /mnt/lustre/file

== When to Use FLR vs. RAID ==

{| class="wikitable"
! Criterion !! FLR !! Hardware/Software RAID
|-
| Granularity || Per-file || Per-device or per-volume
|-
| Use case || Selective replication of critical files || All data on the volume
|-
| Cross-pool/site || Yes — mirrors on different pools or sites || No
|-
| Write model || Delayed write, explicit resync || Synchronous (RAID-1) or parity (RAID-5/6)
|-
| Storage overhead || Only for mirrored files || Entire volume
|-
| Failure handling || Manual or scripted resync || Automatic rebuild
|-
| Read performance || Parallel read from multiple mirrors || Single read path (RAID-1 may use both)
|}

FLR is best suited for:
* Critical checkpoint files in HPC jobs.
* Cross-tier placement (fast SSD mirror + capacity HDD mirror).
* Cross-site disaster recovery.
* Input decks or shared libraries read by many clients.

For full-volume redundancy, use RAID at the storage layer or ZFS mirroring.

== Operational Examples ==

=== Create a 3-way mirrored file ===

 lfs mirror create -N3 -c 4 /mnt/lustre/critical_data

=== Check mirror status ===

 lfs getstripe --mirror-count /mnt/lustre/critical_data
 lfs getstripe -v /mnt/lustre/critical_data

=== Find all files with stale mirrors ===

 lfs find /mnt/lustre --mirror-state=sp /mnt/lustre

=== Resync all stale files in a directory ===

 lfs find /mnt/lustre/project --mirror-state=sp -print0 | xargs -0 -n1 lfs mirror resync
