=Introduction =
This design provides implementation details for the [[Layout Enhancement Solution Architecture]].

In the Lustre file system the data of a file is striped over one or more objects each residing on an OST. The ''layout'' of a file is an attribute of the file which describes the mapping of file data ranges to object data ranges. The layout is stored on the MDT as a trusted extended attribute (<code>trusted.lov</code>) of the file and are sent to clients as needed. The layout of a file is often simply referred to as its ''striping'', since in the current (2.5) implementation of the Lustre file system only non-redundant striped (RAID0) layouts are permitted. This project will design enhancements to the representation and handling of layouts to support features such as Data On MDT, File-Level Replication, [[Progressive File Layouts]], live data migration (via File-Level Replication), and RAID1/5/6 or erasure coding. The Layout Enhancement (LE) project is therefore a prerequisite for the Data on MDT and File-Level Replication projects, described in separate documents.

In this document we present several new layout types: composite layouts to support replication and layouts extents (sections 2.1 and 2.2), RAID layouts (section 3), compact layouts for widely striped files (section 4), and large layouts (section 5).

= 2.1. Composite Layouts =

-----

In order to support file-level replication and extent based layouts we define a new ''composite layout'' type which comprises several ''plain'' (non-composite) layouts designating (partial) mirrors of the file data. This type generalizes RAID0+1 to allow heterogeneous stripe sizes and counts among mirrors and to allow for the possibility of partial mirroring schemes, potentially with each replica on different OST storage pools with different performance and capacity characteristics.

Composite layouts are described by <code>struct lov_comp_md_v1</code> which is defined below.

A composite layout begins with a <code>struct lov_comp_md_v1</code> followed by <code>lcm_entry_count</code> entries (instances of <code>struct lov_comp_md_entry_v1</code>), followed by <code>lcm_entry_count</code> plain layouts (<code>struct lov_mds_md_v</code>3, or any other non composite layouts).

{|
|width="100%"|This design does not support nested composite layouts (i.e. components which are themselves composites) to avoid complexity and recursion in the implementation of layout handling. We believe that non-nested layouts provide sufficient flexibility for current projects and anticipated future uses.
|}

Each entry has an identifier <code>(lcme_id</code>) which is unique within the lifetime of the composite layout. This is the only way to identify an entry or component within a composite layout. In contrast to existing layout types, composite are intended to updated as components (file replicas) are added and removed. Therefore the particular index on an entry in <code>lcm_entries</code> is not a suitable identifier. Entry identifiers are selected by the server side layout handling code as new components are added to a composite layout.

The offset and size members of the entry <code>(lcme_offset</code> and <code>lcme_size</code>) describe the location of the contained plain layout. We do not assume that the component offsets of the entries are in any particular order nor do we assume any relationship among the component sizes. The offsets are chosen to be multiples of 8 so that components will be suitably aligned in memory. Assume <code>lcm</code> points to an in-memory instance of <code>struct lov_comp_md_v1</code> which contains three entries (<code>lcm-&gt;lcm_entry_count == 3</code>).

Recall that extent based layouts permit different layouts to be used in different extents of a file. They may be used to set progressively wider striping as a file grows in size, to prevent inconsistent out of space errors as individual OSTs become full, and to enable incremental migration, replication, and HSM restore. The extent member (<code>lcme_extent</code>) of a composite layout entry describes the range of the file to which the component layout applies. Note that in the simplest case of replication each component would have the same extent of [0, ∞). For a more interesting use of component extents, consider a file ''F'' which cannot grow in size because one of its objects belongs to an OST with no free space. To handle this, the layout ''L<sub>0</sub>'' of ''F'' is converted to a composite layout ''C'' which contains ''L''<sub>0</sub> as a component with extent [0, ''s'') where ''s'' is the size of ''F'' rounded down to the nearest multiple of <code>lmm_stripe_size</code>. Then a new plain layout ''L''<sub>1</sub> is allocated (with objects on other OSTs) and added to ''C'' but with extent [''s'', ∞). Now data appended to ''F'' goes to objects belonging to ''L''<sub>1.</sub>

{|
|width="100%"|In the design we normally assume that the component extents in a composite layout have the same starting offset at byte 0 of the file. The extents may each form a non-overlapping a subset of [0, ∞), or the may all start at file offset 0, or there may be some other overlap. However, we should try not to use the component extent start as an offset when accessing the component objects. That is, if a component has a single object ''O'' and extent [''s'', ∞) then the byte at logical file offset ''p'' should be found at offset ''p'' of ''O'' and not at ''p'' - ''s''. In this way an extent with non-zero start can be converted to one which starts at 0 without moving any data. Similarly, assume that file data is safely mirrored to another component, a component whose extent starts at zero can be figuratively ''punched'' to have some positive start without remapping the objects, followed by a punch of corresponding objects' data.
|}

= 2.2. RPCs for Composite Layouts =


-----

{|
|width="100%"|The use of dedicated RPCs for managing composite layouts has been superseded in the [https://wiki.hpdd.intel.com/display/PFL/PFL+2%3A+High+Level+Design PFL 2: High Level Design] by the use of virtual xattrs that can address individual components of the file.
|}

 

In order to construct and modify composite layouts a set of new RPCs are defined. These are sent using a single high level RPC with opcode <code>MDS_LOV_COMP_MD_OP</code> and format <code>RQF_MDS_LOV_COMP_MD_OP</code> based on the existing MDS_SWAP_LAYOUTS RPC. Depending on the operation one or two FIDs are packed in an MDT body along with capabilities and LDLM data. To this a <code>struct lov_comp_md_op</code> is added which specifies the exact operation to be performed and any operation specific data.

{|
|width="100%"|<pre>struct req_msg_field RMF_LOV_COMP_MD_OP =
        DEFINE_MSGF(&quot;lov_comp_md_op&quot;, 0, sizeof(struct lov_comp_md_op),

static const struct req_msg_field *mdt_lov_comp_md_op[] = {
        &amp;RMF_PTLRPC_BODY,
        &amp;RMF_MDT_BODY,
        &amp;RMF_MDT_LOV_COMP_MD_OP,
        &amp;RMF_CAPA1,
        &amp;RMF_CAPA2,
        &amp;RMF_DLM_REQ,
};

struct req_format RQF_MDS_LOV_COMP_MD_OP =
        DEFINE_REQ_FMT0(&quot;MDS_LOV_COMP_MD_OP&quot;,
                        mdt_lov_comp_md_op, empty);</pre>
|}

<code>struct lov_comp_md_op</code> is defined below.

{|
|width="100%"|<pre>enum lov_comp_md_opc {
        LCMO_MOVE_ENTRY         = 0,
        LCMO_UPDATE_ENTRY       = 1,
        LCMO_SET_PRIMARY        = 2,
};

enum lov_comp_md_op_flags {
        LCMO_FL_ID_NEQ          = 1 &lt;&lt; 0, /* negate sense of id matching */
};

struct lov_comp_md_op {
        __u32   lcmo_opc;
        __u32   lcmo_flags;
        __u32   lcmo_layout_gen[2];
        __u32   lcmo_entry_id;
        __u32   lcmo_entry_flags;
        __u32   lcmo_entry_flags_mask;
        __u32   lcmo_padding;
        struct lu_extent lcmo_entry_extent;
};</pre>
|}

The file(s) to be operated on are identified by <code>fid1</code> (and <code>fid2</code>) of the MDT body. Files must reside on the MDT receiving the RPC; otherwise no operation is performed and <code>-EPROTO</code> is returned. For quota maintenance purposes they must also have the same ownership (UID and GID) or <code>-EPERM</code> is returned. If <code>lcmo_layout_gen[0]</code> is not -1 then is must be equal to the layout generation of the file identified by <code>fid1</code>; otherwise no operation is performed and <code>-EXXX</code> is returned. Similarly for <code>lcmo_layout_gen[1]</code> and the file identified by <code>fid2</code> for operations involving two files.

In each operation <code>lcmo_entry_id</code> designates one or more entries in a composite layout by using a known id or by using a virtual id from <code>enum lcme_id</code>. Setting <code>LCMO_FL_ID_NEQ</code> negates the sense of matching and may only be used with a specific id, <code>LCME_ID_PRIMARY</code>, or <code>LCME_ID_STALE</code>; otherwise no operation is performed and <code>-EINVAL</code> is returned. If either file has a non-composite layout then <code>LCME_ID_ANY</code> should be used to specify their single-component. If the designated entries do not exist then no operation is performed and <code>-EBADSLT</code> is returned.

<code>lcmo_extry_extent</code> must be a valid non-empty extent, have <code>e_start</code> and <code>e_end</code> both 0 (empty), or have <code>e_begin</code> and <code>e_end</code> both -1 (special, see below); otherwise no operation is performed and <code>-EINVAL</code> is returned.

* <code>LCMO_MOVE_ENTRY</code> - move one or more components between the files (Fd and Fs) identified by <code>fid1</code> and <code>fid2</code>. Fd is the destination and Fs the source. The newly created entries in Fd will have flags and extents equal to <code>lcmo_entry_flags</code> and <code>lcmo_entry_extent</code>.
* <code>LCMO_UPDATE_ENTRY</code> - update the flags and/or extent of the entries in the file (F0) specified by <code>fid1</code> of the MDT body. If O is the supplied struct lov_comp_md_op then for easch such entry E we set <code>E.lcme_flags = (E.lcme_flags &amp; ~O.lcmo_entry_flags_mask) | (O.lcmo_entry_flags &amp; O.lcmo_entry_flags_mask)</code>. If <code>O.lcmo_entry_extent</code> is not { <code>-1, -1</code> } then we set <code>E.lcme_extent = O.lcmo_entry_extent</code> as well.
* <code>LCMO_SET_PRIMARY</code> - the designated entry in the file (F0) identified by <code>fid1</code> is set primary.

Clients can delete a given set of components from a composite layout by opening a volatile file and moving the set to the volatile file.

= 3. RAID Layouts =


-----

{|
|width="100%"|While RAID-1 and RAID-10 layouts offer degrees of replication they are not to be confused with the layouts for file-level replication described above. They are fixed layouts similar to the existing RAID-0 layouts currently used by a Lustre file system. Compared to composite layouts they are simpler and more compact but they are also less expressive and less flexible. RAID1 and RAID-10 layouts may be converted to composite layouts. Composite layouts whose components all have the same stripe size and stripe count may be expressed as RAID1 and RAID10 layouts. Full read/write support for files with RAID-4/5/6 layouts is beyond the scope of this design.
|}

RAID layouts are specified using <code>struct lov_raid_md_v1</code> defined below. This format borrows from the [http://www.snia.org/sites/default/files/SNIA_DDF_Technical_Position_v2.0.pdf SNIA Common RAID Disk Data Format Specification] and allows us to specify common simple (0, 1, 4, 5, 6) and nested RAID levels (10, 40, 50, 60).

{|
|width="100%"|<pre>enum {
        LOV_MAGIC_RAID_V1 = 0x0A1D0A1D,
};

enum lrm_pattern {
        /* Primary RAID Levels */
        LRM_PRL_0               = 0, /* RAID-0, striped objects */
        LRM_PRL_1               = 1, /* RAID-1, mirrored objects */
        LRM_PRL_4               = 4, /* RAID-4, 0 + parity object */
        LRM_PRL_5               = 5, /* RAID-5, 0 + distributed parity */
        LRM_PRL_6               = 6, /* RAID-6, 0 + 2 distributed parity */
        LRM_PRL_MASK            = 0xff,
        /* RAID Level Qualifiers */
        LRM_56Q_RP0             = 0x000, /* Rotating Parity 0 */
        LRM_56Q_RPN             = 0x100, /* Rotating Parity N */
        LRM_56Q_DR              = 0x000, /* Data Restarts on first object */
        LRM_56Q_DC              = 0x200, /* Data Continues below parity */
        LRM_RLQ_MASK            = 0xf00,

        /* Secondary (Nested) RAID Levels */
        LRM_SRL_0               = 0x0000, /* striped primary groups */
        LRM_SRL_1               = 0x1000, /* mirrored primary groups */
        LRM_SRL_MASK            = 0xf000,

        LRM_RAID_0      = LRM_PRL_0,
        LRM_RAID_1      = LRM_PRL_1,
        LRM_RAID_4      = LRM_PRL_4,
        LRM_RAID_5      = LRM_PRL_5,
        LRM_RAID_6      = LRM_PRL_6 | LRM_56Q_RP0 | LRM_56Q_DR,

        LRM_RAID_10     = LRM_PRL_1 | LRM_SRL_0,
        LRM_RAID_40     = LRM_PRL_4 | LRM_SRL_0,
        LRM_RAID_50     = LRM_PRL_5 | LRM_56Q_RP0 | LRM_56Q_DR | LRM_SRL_0,
        LRM_RAID_60     = LRM_PRL_6 | LRM_56Q_RP0 | LRM_56Q_DR | LRM_SRL_0,

        LRM_FL_RELEASED = 0x8000000, /* LOV_PATTERN_F_RELEASED */
};

struct lov_raid_md_v1 {
        __u32           lrm_magic;
        __u32           lrm_pattern;
        __u32           lrm_layout_gen;         /* u32! */
        __u32           lrm_stripe_size;
        __u16           lrm_count[2];
        __u32           lrm_padding;
        char            lrm_pool_name[16];      /* 16 == LOV_MAXPOOLNAME */
        struct lu_fid   lrm_objects[0];
};

/* sizeof(struct lov_raid_md_v1) == 40 */</pre>
|}

Non-nested RAID (levels 0, 1, 4, 5, and 6) layouts have a <code>lrm_count[1]</code> value of 1. Nested RAID (levels 10, 40, 50, and 60) have a<code> lrm_count[1]</code> value greater than 1. In the nested case, the file objects are arranged in to <code>lrm_group_count</code> ''primary groups'' each of which is a RAID set for the primary RAID level (0, 1, 4, 5, 6) with <code>lrm_count[0]</code> objects. Then depending on the secondary RAID level these primary groups are mirrored or striped to create the file. For levels 5 and 6 <code>lrm_count[0]</code> includes the parity objects (and hence must be at least 3 for RAID-5 and 4 for RAID-6. The number of objects in is RAID layout is always equal to the product <code>lrm_count[0] * lrm_count[1]</code>.

= 4. Compact Layouts for Widely Striped Files =


-----

Existing RAID-0 layout formats <code>(struct lov_mds_md_v1</code> and <code>struct lov_mds_md_v3) </code>use an explicit array of object identifiers to map each stripe index to a specific OST object. When using FIDs alone to identify objects this approach requires 16 bytes per stripe. The current implementation packs an OST index together with the object identifier and needs 24 bytes per stripe. The allocation of memory buffers to transmit, receive, and handle these layouts for very widely striped files (over 160 stripes) can be costly. To avoid this cost we define a compact layout (<code>struct lov_wide_md_v1</code>) based on a bitmask of OST indices which reduces memory consumption for widely stripe files by a factor of 192 for the current maximally-striped layout of 2000 stripes.

{|
|width="100%"|To make use of this layout type we need to assume that a given MDT can construct a vector <code>seq</code> of FID sequences together with a positive integer <code>stride</code> such that

* for each valid OST index <code>i</code>, either <code>seq[i] == 0</code> or <code>seq[i]</code> belongs to OST <code>i</code> and is allocated to the MDT,
* for each invalid OST index <code>i</code>, we have <code>seq[i] == 0</code>,
* for all non-negative <code>integers i</code> and <code>j</code>, if <code>seq[i] != 0</code> and <code>seq[j] != 0</code> then <code>seq[i] - seq[j] = (i - j) * stride</code>.

To be useful the vector <code>seq</code> must have many non zero slots. The exact means by which such a vector may be constructed are beyond the scope of this design.
|}

<code>struct lov_wide_md_v1</code> is defined below.

{|
|width="100%"|<pre>enum {
        LOV_MAGIC_WIDE_V1 = 0x0B160B16,
};

struct lov_wide_md_v1 {
        __u32 lwm_magic;
        __u32 lwm_layout_gen;
        __u32 lwm_stripe_size;
        __u32 lwm_stripe_count;
        char  lwm_pool_name[16];      /* 16 == LOV_MAXPOOLNAME */
        __u64 lwm_obj_seq_begin;      /* used with stride (below) and OST index to
        __u64 lwm_obj_seq_stride;      * compute FID sequence (f_seq) of objects */
        __u32 lwm_obj_oid;            /* common FID OID (f_oid) of all objects */
        __u32 lwm_obj_ver;            /* common FID version (f_ver) of all objects */
        __u32 lwm_pattern;
        __u32 lwm_ost_idx_begin;
        __u32 lwm_ost_idx_rotate;
        __u32 lwm_ost_idx_mask_width; /* mask width in bits */
        __u64 lwm_ost_idx_mask[];
};

/* sizeof(struct lov_wide_md_v1) == 72 */</pre>
|}

Every lov_wide_md_v1 may be converted into an equivalent lov_mds_md_v1 as sketched below. There are three ideas at work:

* The <code>lwm_ost_idx_*</code> members describe the set of OSTs which contain objects for the file
* The <code>lwm_obj_*</code> specify the FID of an object in the file given its OST index.
* <code>lwm_ost_idx_rotate</code> is used to rotate the array of used OST indexes to ensure that the placement of initial stripes is not biased toward lower index OSTs.

{|
|width="100%"|<pre>int lmm_v1_from_lwm_v1(struct lov_user_md_v1 *m, const struct lov_wide_md_v1 *w)
{
        unsigned int stripe_index;
        unsigned int i;
        int rc;

        if (w-&gt;lwm_ost_seq_begin == 0 ||
            w-&gt;lwm_ost_seq_stride == 0 ||
            w-&gt;lwm_ost_idx_mask_width == 0)
                return -EINVAL;

        /* The stripe count must equal the number of bits set in the ost index mask. */
        if (w-&gt;lwm_stripe_count != bit_set_count(w-&gt;lwm_ost_idx_mask, w-&gt;lwm_ost_idx_mask_width))
                return -EINVAL;

        /* Ensure m is properly sized for stripe_count objects. */
        rc = lmm_v1_resize_magically(m, LOV_MAGIC_V3, stripe_count);
        if (rc &lt; 0)
                return rc;

        /* m-&gt;lmm_oi must be set by caller. */
        m-&gt;lmm_magic = LOV_MAGIC_V3;
        m-&gt;lmm_pattern = w-&gt;lwm_pattern;
        m-&gt;lmm_stripe_size = w-&gt;lwm_stripe_size;
        m-&gt;lmm_stripe_count = w-&gt;lwm_stripe_count;
        memcpy(m-&gt;lmm_pool_name, w-&gt;lwm-&gt;pool_name, sizeof(m-&gt;lmm_pool_name));

        stripe_index = 0;

        for (i = 0; i &lt; w-&gt;lwm_ost_idx_mask_width; i++) {
                const struct lov_ost_data_v1 *o;
                struct lu_fid fid;
                unsigned int ost_idx;

                if (!bit_is_set(w-&gt;lwm_ost_idx_mask, i))
                        continue;

                ost_idx = w-&gt;lwm_ost_idx_begin + (w-&gt;lwm_ost_idx_rotate + i) %
                        w-&gt;lwm_ost_idx_mask_width;

                fid.f_seq = w-&gt;lwm_obj_seq_begin + ost_idx * w-&gt;lwm_obj_seq_stride;
                fid.f_oid = w-&gt;lwm_obj_oid;
                fid.f_ver = w-&gt;lwm_obj_ver;

                o = &amp;m-&gt;lmm_objects[stripe_index];
                fid_to_ostid(&amp;o-&gt;l_ost_oi, &amp;fid);
                o-&gt;l_ost_idx = ost_idx;

                stripe_index++;
        }

        return 0;
}</pre>
|}

= Handling of Large Layouts =


-----

{|
|width="100%"|This layout type and handling strategy described below depends on future work related to the Data on MDT project.
|}

There are (at least) two issues with handing large layouts:

* Storing the layout as an extended attribute on an MDT inode limits its size to 64 KB.
* Retrieval of the layout in the initial open RPC requires a large request/reply buffer allocation.
* Layout sizes are constrained by single transfer size limits from lower layers (LNet).

To address these issues we define an indirect layout type which specifies that the file layout is stored as the data of a second ''layout'' file.

{|
|width="100%"|<pre>enum {
        LOV_MAGIC_INDIR_V1 = 0x0F1DF1D,
};

struct lov_ind_md_v1 {
        __u32 lim_magic;       /* LOV_MAGIC_INDIR_V1 */
        __u32 lim_layout_size;
        struct lu_fid lim_fid; /* FID of the MDT file containing layout */
};</pre>
|}

Clients encountering this layout type may send OST_READ requests to the MDT using the supplied FID (<code>lim_fid</code>) to retrieve the true file layout data. The layout file must have a name (derived from the FID of the original file) in a directory (<code>LAYOUTS</code> or similar) on the MDT but this name will not exist in the client visible namespace.
