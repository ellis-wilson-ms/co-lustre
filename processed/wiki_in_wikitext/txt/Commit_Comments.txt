= Commit message content =

Writing good commit comments is critical to ensuring that changes are easily understood, even years after they were originally written. The commit comment should contain enough information about the change to allow the reader to understand the motivation for the change, what parts of the code it is affecting, and any interesting, unusual, or complex parts of the change to draw attention to.

The reason for a change may be manyfold: bug, enhancement, feature, code style, etc. so providing information about this sets the stage for understanding the change. If it is a bug, include information about what usage triggers the bug and how it manifests (error messages, LBUG, etc). If it is a feature, include information about what improvement is being made, and how it will affect usage.

Providing some high-level information about the code path that is being modified is useful for the reader, since the files and patch fragments are not necessarily going to be listed in a sensible order in the patch. Including the important functions being modified provides a starting point for the reader to follow the logic of the change, and makes it easier to search for such changes in the future.

If the patch is based on some earlier patch, then including a '''Fixes:''' label that contains the short git commit hash and summary of the original patch is useful for tracking the chain of dependencies. This can be very useful if a patch is landed separately to different maintenance branches, if it is fixing a problem in a previously landed patch, or if it is being imported from an upstream kernel commit.

Having long commit comments that describe the change well is a good thing. The commit comments will be tightly associated with the code for a long time into the future, even many of the original commit comments from years earlier are still available through changes of the source code repository. In contrast, bug tracking systems come and go, and cannot be relied upon to track information about a change for extended periods of time.

=Commit message format=

Unlike the content of the commit message, the format is relatively easy to verify for correctness. Having the same standard format allows Git tools like <code>git shortlog</code> to extract information from the patches more easily.

The first line of the commit comment is the commit summary of the change. Changes submitted to the <code>fs/lustre-release</code> branch require a [https://jira.whamcloud.com/projects/LU/issues Lustre JIRA ticket] number at the beginning of the commit summary. A Lustre JIRA ticket is one that begins with LU and is therefore part of the Lustre project within JIRA.  For patches to other projects, such as documentation, a different JIRA project should be used (e.g. <code>LUDOC</code>). If the patch is submitted for the <code>fs/lustre-release</code> repository without a Lustre JIRA ID in the first line, then it will automatically receive a <code>Review: -2</code> which will prevent the patch from being submitted to a release branch. You would then need to fix the summary line and resubmit the patch.

The commit summary should also have a component: tag immediately following the Jira ticket number that indicates which Lustre subsystem that the commit is related to. Example Lustre subsystems relate to modules like: <code>llite</code>, <code>lov</code>, <code>osc</code>, <code>mdc</code>, <code>lmv</code>, <code>ldlm</code>, <code>ptlrpc</code>, <code>mds</code>, <code>oss</code>, <code>mdd</code>, <code>osd-ldiskfs</code>, <code>osd-zfs</code>, <code>ldiskfs</code>, <code>lnet</code>, <code>libcfs</code>, <code>socklnd</code>, <code>o2iblnd</code>; functional components like <code>recovery</code>, <code>quota</code>, <code>grant</code>, <code>nrs</code>, <code>utils</code>; or auxiliary components like <code>build</code>, <code>tests</code>, <code>iokit</code>, <code>docs</code>. This subsystem list is not exhaustive, but provides a good guideline for consistency.

The commit summary line must be 62 characters or less, including the Jira ticket number and component tag, so that <code>git shortlog</code> and <code>git format-patch</code> can fit the summary onto a single line. The summary must be followed by a blank like. The rest of the comments should be wrapped to 70 columns or less. This allows for the first line to be used a subject in emails, and also for the entire body to be displayed using tools like <code>git log</code> or <code>git shortlog</code> in an 80 column window.

    LU-nnnnn component: short change description under 62 columns
    
    The "component:" should be a lower-case single-word subsystem of the
    Lustre code that best encompasses the change being made.  Examples of
    components include modules: llite, lov, lmv, osc, mdc, ldlm, mds, oss,
    ptlrpc, osd-ldiskfs, osd-zfs, ldiskfs, lnet, socklnd, o2iblnd, libcfs;
    functional subsystems: recovery, quota, grant; and auxiliary areas:
    build, tests, docs.  This list is not exhaustive, but is a guideline.
    
    The commit comment should contain a detailed explanation of changes
    being made.  This can be as long as you'd like.  Please give details
    of what problem was solved (including error messages or problems that
    were seen), a good high-level description of how it was solved, and
    which parts of the code were changed (including important functions
    that were changed, if this is useful to understand the patch, and
    for easier searching).  Wrap lines to fit within 70 columns.
    
    Signed-off-by: Your Real Name <your_email@domain.name>
    Change-Id: Ixxxx(added automatically if missing)xxxx

===The Signed-off-by: line===

The <code>[[Signed-off-by]]:</code> line asserts that you have permission to contribute the code to the project according to the [[Signed-off-by|Developer's Certificate of Origin]].

==The Change-Id: line and Code Style==

Gerrit needs to automatically identify updates to existing patches and update the existing change instead of creating a new one for each patch submitted. This preserves the history of patch comments, and allows comparing old and new versions of a patch for more efficient code reviews. For this to work properly the changes you create locally need to have a unique commit id in them. The same <code>Change-Id:</code> line should be used for all versions of a patch, even if it is ported to other branches.

Please make sure the <code>Change-Id:</code> line and <code>Signed-off-by:</code> lines are at the bottom of the comment. This is required by the <code>fs/lustre-release</code> and other projects in order for the patch to be approved. If your comments are missing either of these lines the patch will be automatically be rejected at submission time.

The <code>Change-Id:</code> field is inserted automatically into the commit message by installing the <code>contrib/git-hooks/commit-msg</code> hook into each Git repository's <code>.git/hooks/</code> directory. The <code>contrib/git-hooks/prepare-commit-msg</code> script should be installed as well. From the top-level Lustre tree checkout:

    $ ln -sf ../../contrib/git-hooks/commit-msg .git/hooks/
    $ ln -sf ../../contrib/git-hooks/prepare-commit-msg .git/hooks/

At <code>git commit</code> time this will run the <code>.git/hooks/prepare-commit-msg</code> script from the currently Lustre tree to check the code changes for style errors using <code>contrib/checkpatch.pl</code>, and add comments at the end of the commit message with any warnings or errors. After the commit message has been saved, the <code>.git/hooks/commit-msg</code> script will verify that the commit message format matches the format specified above, and insert the <code>Change-Id:</code> field into the commit message, if one isn't already present.  Using a symlink instead of copying the hooks will ensure that the hooks are updated appropriately between branches and as changes are made to the scripts.

If a patch needs to be resubmitted for some reason (e.g. original patch was landed, then reverted), the original <code>Change-Id:</code> should be referenced as <code>Was-Change-Id:</code> so that it is possible to correlate these patches together in the future.

==Fixes tag==
The <code>Fixes:</code> tag is very useful for identifying a fix made to a previously-landed commit.  This is important when the earlier patch is being cherry-picked to a different branch, it is easier to find any later patches that fix it so that they can be included in a single patch to avoid landing the patch with a regression.  For example, to find patches related to LU-13468, run "<code>git log --grep LU-13468</code>" to find both the original patch and the patch with the <code>Fixes:</code> tag.  Generating the <code>Fixes:</code> line can be simplified by adding the following lines to your <code>$HOME/.gitconfig</code> file:

    [core]
        abbrev=10
    [pretty]
        fixes=Fixes: %h (\"%s\")

and then running any git logging command (including with <code>-S</code> or <code>--grep</code> to search for patches):

    $ git log -1 --pretty=fixes b1acf734f3

or directly from your editor (vim, of course) to insert the line into the commit message:

    :.!git log --pretty=fixes --grep LU-13468

This will produce a line that is suitable for inclusion directly into the commit message, for example:

    Fixes: b1acf734f3 ("LU-13468 fld: repeat rpc in fld_client_rpc after EAGAIN")

==Test-Parameters: tag==
The <code>Test-Parameters:</code> tag can improve test coverage of patches significantly, if used correctly.  This can be used to instruct the test system to run specific tests with particular Linux distributions, CPU architectures, numbers of clients, MDTs, OSTs, etc., or to minimize testing for patches that do not affect code functionality (e.g. man pages, comments, function or file renaming, etc.).

'''NOTE''' : most <code>Test-Parameters</code> are run *in addition to* the regular set of tests for each patch.  The only exceptions are <code>trivial</code>, <code>ignore</code>,
 <code>fortestonly</code>, <code>forjanitoronly</code>.

There is a page dedicated to describe the <code>Test-Parameters:</code>, see [https://wiki.whamcloud.com/display/PUB/Changing+Test+Parameters+with+Gerrit+Commit+Messages Changing Test Parameters with Gerrit Commit Messages], but some examples of common parameters include:

; <code>Test-Parmeters: trivial</code> : patch does not affect code functionality, run only build and smoke test
; <code>Test-Paramters: testlist=sanity env=ONLY=102,ONLY_REPEAT=100</code> : run <code>sanity.sh test_102</code> 100 times in a loop, *in addition to regular tests*
; <code>Test-Paramters: testlist=sanity clientarch=aarch64 clientdistro=rocky9.5</code> : run <code>sanity.sh</code> on a rocky9.5 aarch64 client, *in addition to regular tests*

==Additional commit tags==

A number of additional commit tags can be used to further explain who has contributed to the patch, and for tracking purposes. These tags are commonly used with Linux kernel patches. These tags should appear before the <code>Signed-off-by:</code> tag.

    Acked-by: User Name <user@domain.com>
    Tested-by: User Name <user@domain.com>
    Reported-by: User Name <user@domain.com>
    Reviewed-by: User Name <user@domain.com>
    Test-Parameters: additional testing parameters
    CC: User Name <user@domain.com>
    CoverityID: nnnnnn ("type of issue")
    {Organization}-bug-id: arbitrary bug tracking identifier
    Lustre-change: https://jira.whamcloud.com/browse/LU-nnnnn
    Lustre-commit: {git commit hash of master patch}
    Linux-commit: {git commit hash of upstream kernel patch}

The <code>{Organization}-bug-id:</code> tag (e.g. <code>Whamcloud-bug-id: LDEV-123</code>, <code>HPE-bug-id: MRP-123</code>, or <code>Oracle-bug-id: b=12345</code>) can be used to track this patch in other bug databases.

The <code>Lustre-commit:</code> and <code>Lustre-change:</code> tags are used to reference the original <code>master</code> version of a patch that is being ported to another branch.

Patches pushed to the upstream kernel should include <code>Lustre-bug-id:</code> to reference the JIRA URL for the LU ticket, in the form <code>https://jira.whamcloud.com/browse/LU-nnnnn</code>. Patches should also include the <code>Reviewed-on:</code> permalink URL for the Gerrit patch, in the form <code>https://review.whamcloud.com/nnnnn</code>. The upstream maintainers do not want to have commit hashes for non-kernel commits, so upstream patches should not include the <code>Lustre-commit:</code> tag.

Patches ported from the upstream kernel, the patch should keep the original commit summary line, with slight modification to conform to Lustre standards: include the JIRA LU ticket number, and replace "staging" and other pathnames in the summary with a subsystem. The <code>Linux-commit:</code> tag is used to reference the upstream kernel commit hash when it is merged into the master branch.  The original author should be kept as the author of the patch using the <code>git --author</code> option, as well as the original <code>Signed-off-by:</code> tag(s).

=Patch porting examples=

For patches backported from master to a Lustre maintenance branch (e.g. b2_10) there are some conventions to follow so that the changes/fixes can more easily be tracked across branches. The simples method for porting a patch from one branch to another is to use the "Cherry Pick" button on the patch directly in Gerrit, which can be used for patches that apply cleanly to the specified target branch as long as it is in the same Git repository.  Alternately, if this is not possible, use "<code>git cherry-pick {commit_hash|branch}</code>" from the command line to pull the patch onto the (current) branch where you want the patch to land, and then using the normal patch submission process to push the patch to Gerrit or submit it to the upstream kernel.  This will apply the whole patch (as best as is able, and show conflicts where needed), copy the commit message, preserve the original patch author.  With luck, there will not be any patch conflicts and no further work is needed.  If necessary, the patch conflicts need to be resolved before committing the patch.  For the commit message:

* the entire commit message, including the summary line, should be copied from the source patch without any changes
* the original Author of the patch should be kept.  This should be done automatically when using git cherry-pick but is lost when applying the patch manually, so "<code>git commit --author='Original Author <author@email.com>'</code>" should be used
* the <code>Signed-off-by:</code> line of the original author should also be kept
* the person who commits the ported change should add a <code>Signed-off-by:</code> line with their name and email following the original one
* the '''original''' <code>Change-Id:</code> line
* the '''original''' <code>Reviewed-by:</code> lines can be kept, and Gerrit will automatically add them as reviewers to the new patch
* the <code>Tested-by: Maloo</code> and <code>Tested-by: Jenkins</code> lines should be removed from the new commit message, though any <code>Tested-by:</code> lines from real people can be kept
* the <code>Reviewed-on: http://review.whamcloud.com/nnnnn</code> line should be changed to <code>Lustre-change: http://review.whamcloud.com/nnnnn</code> (please use the "permalink" Gerrit URL format as shown)
* the <code>cherry picked from commit ca09fda138b6d72588f40e4cf79c5f2de832d2dd</code> line should be changed to <code>Lustre-commit: ca09fda138b6d72588f40e4cf79c5f2de832d2dd</code>
* any non-trivial changes to the original patch should be described after the original <code>Signed-off-by:</code> and <code>Lustre-commit:</code> lines before your own <code>Signed-off-by:</code> line
* if two or more patches are being combined into a single patch (normally only when there are bugs in the original patch that were fixed in later commits) the full commit message from each patch should be kept, along with the <code>Signed-off-by:</code> and <code>Lustre-commit:</code> and <code>Lustre-change:</code> lines of each commit

==Patches ported from master to upstream kernel==

These are treated similarly as patches ported to maintenance branches (keep all comments and <code>[[Signed-off-by]]:</code> lines from the original patch) add new <code>Signed-off-by:</code> and comments afterward, but replace the <code>Lustre-commit:</code> line (which doesn't mean anything in the upstream kernel git) with <code>Lustre-bug-id:</code> (or equivalent internal tracking information {jira_URL} so that the original bug can still be identified.  When submitting patches upstream, please also follow the <code>Documentation/process/submitting-patches.rst</code> instructions in the kernel tree, and use <code>contrib/scripts/get_maintainer.pl</code> to generate the CC list for the patch.  If in doubt, submit the patch only to <code>lustre-devel</code> first to get feedback from the Lustre maintainers.

    lustre/llite: simplify dentry revalidate
    
    Lustre client dentry validation is protected by LDLM lock, so
    any time a dentry is found, it's valid and no need to revalidate
    from MDS, and even it does, there is race that it may be
    invalidated after revalidation is finished.
    
    Reviewed-on: http://review.whamcloud.com/7475
    Lustre-bug-id: https://jira.whamcloud.com/browse/LU-3544
    Change-Id: I7700cbaddc4ec08e12c9f7d8021783a6135dd35a
    Signed-off-by: Lai Siyao <lai.siyao@intel.com>
    Signed-off-by: Oleg Drokin <oleg.drokin@intel.com>
