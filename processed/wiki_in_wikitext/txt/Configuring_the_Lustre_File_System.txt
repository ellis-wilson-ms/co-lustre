--- Content ---
This page describes how to configure a Lustre file system, including creating a combined MGS/MDT, OSTs, and mounting clients. Both ldiskfs and ZFS backend examples are provided.

This guide applies to Lustre 2.15 and later. For the authoritative reference on all configuration options, see the Lustre Operations Manual.

Note: Use dotted-quad (dot-decimal) notation for IP addresses (IPv4) rather than host names. This aids in reading debug logs and helps when debugging configurations with multiple interfaces.

== Prerequisites ==

Before configuring the Lustre file system, ensure the following are complete:

* Hardware is set up and meets Lustre requirements (dedicated servers for MDS and OSS, 64-bit CPUs, adequate memory).
* Lustre server and client packages are installed.
* Block devices for MDTs and OSTs are prepared. If using hardware or software RAID, configure it before formatting Lustre targets. For ZFS backends, install ZFS packages and load the zfs kernel module.
* (Optional) Network interface bonding is configured on Ethernet interfaces. See [[Setting_Up_Bonding|Setting Up Bonding]].
* (Optional) Run the sgpdd-survey benchmark to establish baseline storage performance before formatting OSTs.

== Step 1: Configure LNet ==

LNet (Lustre Networking) determines which network interfaces Lustre uses. If your system has a single TCP/IP interface and that is sufficient, LNet configuration is optional — LNet will auto-detect and use it.

LNet configuration is required if you are using InfiniBand (o2ib), multiple Ethernet interfaces, or need to define routes between networks.

=== Modern method: lnetctl and YAML (recommended) ===

Since Lustre 2.7, the preferred method is to use the '''lnetctl''' utility with a YAML configuration file stored at '''/etc/lnet.conf'''. This file is loaded at boot by the lnet systemd service.

Example '''/etc/lnet.conf''' for a single TCP network on eth0:

  net:
    - net: tcp
      interfaces:
          0: eth0

Example for InfiniBand (o2ib) on ib0:

  net:
    - net: o2ib
      interfaces:
          0: ib0

Example with both TCP and InfiniBand:

  net:
    - net: tcp
      interfaces:
          0: eth0
    - net: o2ib
      interfaces:
          0: ib0

To apply the configuration manually (or test it):

  modprobe lnet
  lnetctl lnet configure
  lnetctl import /etc/lnet.conf

To verify:

  lnetctl net show

To export the current running configuration to a YAML file:

  lnetctl export /etc/lnet.conf

=== Legacy method: module parameters ===

The older method uses kernel module parameters in '''/etc/modprobe.d/lustre.conf'''. This is still supported but lnetctl/YAML is preferred for new deployments.

  options lnet networks="tcp0(eth0)"

Or for InfiniBand:

  options lnet networks="o2ib0(ib0)"

For details on the legacy method, see [[Static_LNet_Configuration|Static LNet Configuration]].

== Step 2: Create the MGS/MDT ==

The Management Server (MGS) stores configuration for all Lustre file systems it manages. The MGS is commonly combined with the first Metadata Target (MDT) on a single block device for simple deployments.

=== ldiskfs backend ===

On the MDS node, format a combined MGS/MDT:

  mkfs.lustre --fsname=<fsname> --mgs --mdt --index=0 /dev/<block_device>

Example:

  mkfs.lustre --fsname=temp --mgs --mdt --index=0 /dev/sdb

=== ZFS backend ===

With ZFS, the device argument is a ZFS pool/dataset name. You must also specify '''--backfstype=zfs'''. If using files or specific disks, pass them as additional arguments after the pool/dataset name (mkfs.lustre will create the zpool for you):

  mkfs.lustre --fsname=<fsname> --mgs --mdt --index=0 --backfstype=zfs <pool>/<dataset> [vdevs...]

Example using a real disk:

  mkfs.lustre --fsname=temp --mgs --mdt --index=0 --backfstype=zfs mdt-pool/mdt0 /dev/sdb

Example using a mirrored ZFS pool (recommended for MDT reliability):

  mkfs.lustre --fsname=temp --mgs --mdt --index=0 --backfstype=zfs mdt-pool/mdt0 mirror /dev/sdb /dev/sdc

=== Separate MGS ===

For multi-filesystem environments, create the MGS on its own dedicated device:

  mkfs.lustre --mgs /dev/sda1

Then create the MDT separately, pointing at the MGS:

  mkfs.lustre --fsname=<fsname> --mdt --index=0 --mgsnode=<MGS_NID> /dev/sdb

=== Notes ===

* The '''--fsname''' is limited to 8 characters. The default is "lustre".
* '''--index=0''' explicitly sets the MDT index. This is strongly recommended.
* Up to 4096 MDTs can exist per file system (Lustre 2.8+ DNE feature). Additional MDTs are created with --mdt --index=1, --index=2, etc., specifying '''--mgsnode''' instead of '''--mgs'''.

== Step 3: Mount the MGS/MDT ==

On the MDS node:

  mount -t lustre /dev/<block_device> /mnt/mdt

For ZFS:

  mount -t lustre mdt-pool/mdt0 /mnt/mdt

If the MGS and MDT are on separate devices, mount both.

Verify the target started by checking system logs:

  dmesg | tail
  # Look for: "Server temp-MDT0000 on device /dev/sdb has started"

== Step 4: Create OSTs ==

On each OSS node, format one or more OSTs. Each OST should use a dedicated block device (no partitioning).

=== ldiskfs backend ===

  mkfs.lustre --fsname=<fsname> --mgsnode=<MGS_NID> --ost --index=<N> /dev/<block_device>

Example (first OST on oss0):

  mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp --ost --index=0 /dev/sdc

Example (second OST on oss1):

  mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp --ost --index=1 /dev/sdd

=== ZFS backend ===

  mkfs.lustre --fsname=<fsname> --mgsnode=<MGS_NID> --ost --index=<N> --backfstype=zfs <pool>/<dataset> [vdevs...]

Example with RAIDZ2:

  mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp --ost --index=0 \
      --backfstype=zfs ost-pool0/ost0 raidz2 /dev/sde /dev/sdf /dev/sdg \
      /dev/sdh /dev/sdi /dev/sdj

=== Notes ===

* Always specify '''--index''' at format time. This makes it easy to map OST numbers in error messages back to physical devices.
* You can have as many OSTs per OSS as the hardware allows. Typical deployments use 2-8 OSTs per OSS.
* If block devices are accessible from multiple OSS nodes (shared storage), mount each OST from only one OSS at a time. Enable multiple-mount protection to prevent data corruption.

== Step 5: Mount OSTs ==

On the OSS node where each OST was created:

  mount -t lustre /dev/<block_device> /mnt/ost0

For ZFS:

  mount -t lustre ost-pool0/ost0 /mnt/ost0

Repeat for each additional OST. After mounting, the OST connects to the MGS and registers with the MDS. You should see log messages like:

  Lustre: temp-OST0000: received MDS connection from 10.2.0.1@tcp
  Lustre: MDS temp-MDT0000: temp-OST0000_UUID now active, resetting orphans

== Step 6: Mount the Client ==

On each client node:

  mount -t lustre <MGS_NID>:/<fsname> <mount_point>

Example:

  mount -t lustre 10.2.0.1@tcp:/temp /lustre

Repeat on each client node that needs access.

=== Persistent client mounts ===

Add an entry to '''/etc/fstab''' for automatic mounting at boot:

  10.2.0.1@tcp:/temp    /lustre    lustre    defaults,_netdev    0 0

=== Troubleshooting mount failures ===

* Check system logs on both client and servers: ''dmesg'' and ''/var/log/messages'' (or ''journalctl'').
* Ensure firewall rules allow incoming connections on TCP port 988 (for socklnd). This is a common issue with new installations.
* Verify LNet connectivity: '''lctl ping <MGS_NID>'''

== Step 7: Verify the File System ==

Run the following commands on a client node to confirm the file system is operational:

  # Show space usage per OST and MDT
  lfs df -h

  # Show inode usage per OST and MDT
  lfs df -ih

  # Test write
  dd if=/dev/zero of=/lustre/testfile bs=4M count=2

  # Verify file was created
  ls -lsah /lustre/

  # Clean up
  rm /lustre/testfile

Expected output from '''lfs df -h''':

  UUID                bytes    Used    Available  Use%  Mounted on
  temp-MDT0000_UUID    8.0G   400.0M      7.6G     0%  /lustre[MDT:0]
  temp-OST0000_UUID  800.0G   400.0M    799.6G     0%  /lustre[OST:0]
  temp-OST0001_UUID  800.0G   400.0M    799.6G     0%  /lustre[OST:1]
  filesystem_summary:  1.6T   800.0M      1.6T     0%  /lustre

== Complete Example (ldiskfs) ==

This example creates a file system called '''temp''' with a combined MGS/MDT, two OSTs on separate OSS nodes, and one client.

{| class="wikitable"
|-
! Parameter !! Value
|-
| File system name || temp
|-
| Network type || TCP
|-
| MGS/MDS node || 10.2.0.1@tcp (mds)
|-
| MGS/MDS block device || /dev/sdb
|-
| MGS/MDS mount point || /mnt/mdt
|-
| OSS node 1 || oss0
|-
| OST 0 block device || /dev/sdc
|-
| OST 0 mount point || /mnt/ost0
|-
| OSS node 2 || oss1
|-
| OST 1 block device || /dev/sdd
|-
| OST 1 mount point || /mnt/ost1
|-
| Client node || client1
|-
| Client mount point || /lustre
|}

'''1. Configure LNet''' (on all nodes):

Create '''/etc/lnet.conf''':

  net:
    - net: tcp
      interfaces:
          0: eth0

Start LNet:

  modprobe lnet
  lnetctl lnet configure
  lnetctl import /etc/lnet.conf
  lnetctl net show

'''2. Create combined MGS/MDT''' (on mds):

  [root@mds ~]# mkfs.lustre --fsname=temp --mgs --mdt --index=0 /dev/sdb

'''3. Mount MGS/MDT''' (on mds):

  [root@mds ~]# mkdir -p /mnt/mdt
  [root@mds ~]# mount -t lustre /dev/sdb /mnt/mdt

'''4. Create and mount OST 0''' (on oss0):

  [root@oss0 ~]# mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp \
                     --ost --index=0 /dev/sdc
  [root@oss0 ~]# mkdir -p /mnt/ost0
  [root@oss0 ~]# mount -t lustre /dev/sdc /mnt/ost0

'''5. Create and mount OST 1''' (on oss1):

  [root@oss1 ~]# mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp \
                     --ost --index=1 /dev/sdd
  [root@oss1 ~]# mkdir -p /mnt/ost1
  [root@oss1 ~]# mount -t lustre /dev/sdd /mnt/ost1

'''6. Mount the client''' (on client1):

  [root@client1 ~]# mkdir -p /lustre
  [root@client1 ~]# mount -t lustre 10.2.0.1@tcp:/temp /lustre

'''7. Verify''':

  [root@client1 ~]# lfs df -h
  [root@client1 ~]# dd if=/dev/zero of=/lustre/testfile bs=4M count=2
  [root@client1 ~]# ls -lsah /lustre/

== Complete Example (ZFS) ==

This example creates the same '''temp''' file system using ZFS backends. The ZFS pool and dataset are created automatically by mkfs.lustre.

'''1. Configure LNet''' (same as above on all nodes).

'''2. Load ZFS''' (on all server nodes):

  modprobe zfs

'''3. Create combined MGS/MDT''' (on mds):

  [root@mds ~]# mkfs.lustre --fsname=temp --mgs --mdt --index=0 \
                    --backfstype=zfs mdt-pool/mdt0 mirror /dev/sdb /dev/sdc

'''4. Mount MGS/MDT''' (on mds):

  [root@mds ~]# mkdir -p /mnt/mdt
  [root@mds ~]# mount -t lustre mdt-pool/mdt0 /mnt/mdt

'''5. Create and mount OST 0''' (on oss0):

  [root@oss0 ~]# mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp \
                     --ost --index=0 --backfstype=zfs ost-pool0/ost0 /dev/sdd
  [root@oss0 ~]# mkdir -p /mnt/ost0
  [root@oss0 ~]# mount -t lustre ost-pool0/ost0 /mnt/ost0

'''6. Create and mount OST 1''' (on oss1):

  [root@oss1 ~]# mkfs.lustre --fsname=temp --mgsnode=10.2.0.1@tcp \
                     --ost --index=1 --backfstype=zfs ost-pool1/ost1 /dev/sde
  [root@oss1 ~]# mkdir -p /mnt/ost1
  [root@oss1 ~]# mount -t lustre ost-pool1/ost1 /mnt/ost1

'''7. Mount and verify client''' (same as ldiskfs example):

  [root@client1 ~]# mkdir -p /lustre
  [root@client1 ~]# mount -t lustre 10.2.0.1@tcp:/temp /lustre
  [root@client1 ~]# lfs df -h

== Additional Configuration ==

=== Adding more OSTs ===

Repeat the mkfs.lustre --ost and mount steps with the next --index value. No restart of existing services is required.

=== Adding more MDTs (DNE) ===

Lustre 2.4+ supports Distributed Namespace (DNE) with multiple MDTs. Create additional MDTs:

  mkfs.lustre --fsname=temp --mdt --index=1 --mgsnode=10.2.0.1@tcp /dev/sdX

Then assign subdirectories to the new MDT:

  lfs mkdir -i 1 /lustre/project_data

For striped directories across multiple MDTs (Lustre 2.8+):

  lfs mkdir -c -1 /lustre/large_shared_dir

=== Changing stripe defaults ===

The default stripe configuration is: stripe_size=1MB, stripe_count=1. Modify with:

  lfs setstripe -S 2M -c 4 /lustre/mydir

=== Configuration utilities ===

* '''mkfs.lustre''' — Format a disk for a Lustre service.
* '''tunefs.lustre''' — Modify configuration on an existing Lustre target disk.
* '''lctl''' — Control Lustre features, parameters, and debugging.
* '''mount.lustre''' — Start a Lustre client or target service.
* '''lfs''' — Configure and query file layout, quotas, and other options.
