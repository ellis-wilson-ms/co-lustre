The <code>zdb</code> (ZFS Debug) command is a useful tool for examining the low-level structure and metadata of a ZFS pool or dataset, and can read the information from the on-disk data structures of exported pools as well as from the ZFS pool cache file. The output is subject to change over time as ZFS is further developed, which means that the exact content contained in the output may vary depending on which version of ZFS is being used.

This guide will not go into the details of the <code>zdb</code> command but will highlight one useful feature for examining ZFS metadata.

To read the configuration of a <code>zpool</code> that is imported on a host, use the following command:

<pre style="overflow-x:auto">
zdb -C[C] <poolname>
</pre>

If the <code>-C</code> option is used, the output will display the cached content from the zpool configuration cache. When <code>-C</code> is referenced more than once, <code>zdb</code> will also display on-disk configuration (called the Meta Object Set, or MOS).

For example:

<pre style="overflow-x:auto">
[root@rh7z-mds1 ~]# zpool list mgspool
NAME      SIZE  ALLOC   FREE  EXPANDSZ   FRAG    CAP  DEDUP  HEALTH  ALTROOT
mgspool  1008M  2.23M  1006M         -     1%     0%  1.00x  ONLINE  -
[root@rh7z-mds1 ~]# zdb -C mgspool

If the pool has not been imported to the current host, or if there is no zpool cache file on the host for a currently imported pool, use the following command:

<pre style="overflow-x:auto;">
zdb -e <poolname>
</pre>

This will provide a very large amount of output. To get the MOS configuration, use the following:

<pre style="overflow-x:auto;">
zdb -eC[C] <poolname>
</pre>

'''Note:''' The MOS includes the <code>hostid</code> of the last host to import the zpool. If the <code>hostid</code> (as presented by the SPL) of the node that imported the zpool is 0 (zero), then this field will not be presented in the <code>zdb</code> MOS output. If there is no <code>hostid</code> in the MOS, then this strongly indicates that the hosts are not configured with proper <code>hostids</code>, and the ZFS volume will not be adequately protected from multiple concurrent accesses from multiple hosts. 

Please see the article [[Protecting File System Volumes from Concurrent Access]] for details on the importance of correctly setting the <code>hostid</code> when working with high availability failover storage configurations.
