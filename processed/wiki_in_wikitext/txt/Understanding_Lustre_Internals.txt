== Lustre Architecture ==

=== What is Lustre? ===

Lustre is a GNU General Public licensed, open-source distributed
parallel file system developed and maintained by DataDirect Networks
(DDN). Due to the extremely scalable architecture of the Lustre file
system, Lustre deployments are popular in scientific supercomputing, as
well as in the oil and gas, manufacturing, rich media, and finance
sectors. Lustre presents a POSIX interface to its clients with parallel
access capabilities to the shared file objects. As of this writing,
Lustre is the most widely used file system on the top 500 fastest
computers in the world. Lustre is the file system of choice on 7 out of
the top 10 fastest computers in the world today, over 70% of the top
100, and also for over 60% of the top 500 <ref> Whamcloud. Whamcloud community wiki. https://wiki.whamcloud.com/, 2021. </ref>.

=== Lustre Features ===

Lustre is designed for scalability and performance. The aggregate storage capacity and file system bandwidth can be scaled up by adding more servers to the file system, and performance for parallel applications can often be increased by utilizing more Lustre clients. Some practical limits are shown in [[#Lustre Numbers|Table 1]] along with values from known production file systems.

Lustre has several features that enhance performance, usability, and
stability. Some of these features include:

*  POSIX Compliance: With few exceptions, Lustre passes the full POSIX test suite. Most operations are atomic to ensure that clients do not see stale data or metadata. Lustre also supports <code>mmap()</code> file IO. 

*  Online file system checking: Lustre provides a file system checker (LFSCK) to detect and correct file system inconsistencies. LFSCK can be run while the file system in online and in production, minimizing potential downtime.

*  Controlled file layouts: The file layouts that determine how data is placed across the Lustre servers can be customized on a per-file basis. This allows users to optimize the layout to best fit their specific use case.

*  Support for multiple backend file systems: When formatting a Lustre file system, the underlying storage can be formatted as either ldiskfs (a performance-enhanced version of ext4) or ZFS.

*  Support for high-performance and heterogeneous networks: Lustre can utilize RDMA over low latency networks such as Infiniband or Intel OmniPath in addition to supporting TCP over commodity networks. The Lustre networking layer provides the ability to route traffic between multiple networks making it feasible to run a single site-wide Lustre file system.

*  High-availability: Lustre supports active/active failover of storage resources and multiple mount protection (MMP) to guard against errors that may results from mounting the storage simultaneously on multiple servers. High availability software such as Pacemaker/Corosync can be used to provide automatic failover capabilities.

*  Security features: Lustre follows the normal UNIX file system security model enhanced with POSIX ACLs. The root squash feature limits the ability of Lustre clients to perform privileged operations. Lustre also supports the configuration of Shared-Secret Key (SSK) security.

*  Capacity growth: File system capacity can be increased by adding additional storage for data and metadata while the file system in online.

=== Lustre Components ===

Lustre is an object-based file system that consists of several components:

[[File:Lustre_components.png|500px|thumb|Figure 1. Lustre file system components in a basic cluster]]

*  Management Server (MGS) - Provides configuration information for the file system. When mounting the file system, the Lustre clients will contact the MGS to retrieve details on how the file system is configured (what servers are part of the file system, failover information, etc.). The MGS can also proactively notify clients about changes in the file system configuration and plays a role in the Lustre recovery process.

*  Management Target (MGT) - Block device used by the MGS to persistently store Lustre file system configuration information. It typically only requires a relatively small amount of space (on the order to 100 MB).

*  Metadata Server (MDS) - Manages the file system namespace and provides metadata services to clients such as filename lookup, directory information, file layouts, and access permissions. The file system will contain at least one MDS but may contain more.

*  Metadata Target (MDT) - Block device used by an MDS to store metadata information. A Lustre file system will contain at least one MDT which holds the root of the file system, but it may contain multiple MDTs. Common configurations will use one MDT per MDS server, but it is possible for an MDS to host multiple MDTs. MDTs can be shared among multiple MDSs to support failover, but each MDT can only be mounted by one MDS at any given time.

*  Object Storage Server (OSS) - Stores file data objects and makes the file contents available to Lustre clients. A file system will typically have many OSS nodes to provide a higher aggregate capacity and network bandwidth.

*  Object Storage Target (OST) - Block device used by an OSS node to store the contents of user files. An OSS node will often host several OSTs. These OSTs may be shared among multiple hosts, but just like MDTs, each OST can only be mounted on a single OSS at any given time. The total capacity of the file system is the sum of all the individual OST capacities.

*  Lustre Client - Mounts the Lustre file system and makes the contents of the namespace visible to the users. There may be hundreds or even thousands of clients accessing a single Lustre file sysyem. Each client can also mount more than one Lustre file system at a time.

*  Lustre Networking (LNet) - Network protocol used for communication between Lustre clients and servers. Supports RDMA on low-latency networks and routing between heterogeneous networks.

The collection of MGS, MDS, and OSS nodes are sometimes referred to as
the “frontend”. The individual OSTs and MDTs must be formatted with a
local file system in order for Lustre to store data and metadata on
those block devices. Currently, only ldiskfs (a modified version of
ext4) and ZFS are supported for this purpose. The choice of ldiskfs or
ZFS if often referred to as the “backend file system”. Lustre provides
an abstraction layer for these backend file systems to allow for the
possibility of including other types of backend file systems in the
future.

[[:Media:Lustre_components.png|Figure 1]] shows a simplified version of the Lustre file system components in a basic cluster. In this figure, the MGS server is distinct from the MDS servers, but for small file systems, the MGS and MDS may be combined into a single server and the MGT may coexist on the same block device as the primary MDT.

=== Lustre File Layouts ===

[[File:File_striping.png|500px|thumb|Figure 2. Normal RAID0 file striping in Lustre]]

Lustre stores file data by splitting the file contents into chunks and
then storing those chunks across the storage targets. By spreading the
file across multiple targets, the file size can exceed the capacity of
any one storage target. It also allows clients to access parts of the
file from multiple Lustre servers simultaneously, effectively scaling up
the bandwidth of the file system. Users have the ability to control many
aspects of the file’s layout by means of the <code>lfs setstripe</code> command,
and they can query the layout for an existing file using the
<code>lfs getstripe</code> command.

File layouts fall into one of two categories:

# Normal / RAID0 - File data is striped across multiple OSTs in a round-robin manner.
# Composite - Complex layouts that involve several components with potentially different striping patterns.

==== Normal (RAID0) Layouts ====

A normal layout is characterized by a stripe count and a stripe size.
The stripe count determines how many OSTs will be used to store the file
data, while the stripe size determines how much data will be written to
an OST before moving to the next OST in the layout. As an example,
consider the file layouts shown in [[:Media:File striping.png|Figure 2]] for a simple file system with 3 OSTs
residing on 3 different OSS nodes <ref> Ignore the LOV and OSC components for the moment. These are client-related pieces that will not be discussed at this point. </ref>. Note that Lustre indexes the OSTs starting at zero.

File A has a stripe count of three, so it will utilize all OSTs in the
file system. We will assume that it uses the default Lustre stripe size
of 1MB. When File A is written, the first 1MB chunk gets written to
OST0. Lustre then writes the second 1MB chunk of the file to OST1 and
the third chunk to OST2. When the file exceeds 3 MB in size, Lustre will
round-robin back to the first allocated OST and write the fourth 1MB
chunk to OST0, followed by OST1, etc. This illustrates how Lustre writes
data in a RAID0 manner for a file. It should be noted that although File
A has three chunks of data on OST0 (chunks #1, #4, and #7), all these
chunks reside in a single object on the backend file system. From
Lustre’s point of view, File A consists of three objects, one per OST.
Files B and C show layouts with the default Lustre stripe count of one,
but only File B uses the default stripe size of 1MB. The layout for File
C has been modified to use a larger stripe size of 2MB. If both File B
and File C are 2MB in size, File B will be treated as two consecutive
chunks written to the same OST whereas File C will be treated as a
single chunk. However, this difference is mostly irrelevant since both
files will still consist of a single 2MB object on their respective
OSTs.

==== Composite Layouts ====

A composite layout consists of one or more components each with their
own specific layout. The most basic composite layout is a Progressive
File Layout (PFL). Using PFL, a user can specify the same parameters
used for a normal RAID0 layout but additionally specify a start and end
point for that RAID0 layout. A PFL can be viewed as an array of normal
layouts each of which covers a consecutive non-overlapping region of the
file. PFL allows the data placement to change as the file increases in
size, and because Lustre uses delayed instantiation, storage for
subsequent components is allocated only when needed. This is
particularly useful for increasing the stripe count of a file as the
file grows in size.

The concept of a PFL has been extended to include two other layouts: Data on MDT (DoM) and Self Extending Layout (SEL). A DoM layout is specified just like a PFL except that the first component of the file resides on the same MDT as the file’s metadata. This it typically used to store small amounts of data for quick access. A SEL is just like a PFL with the addition that an extent size can be supplied for one or more of the components. When a component is instantiated, Lustre only instantiates part of the component to cover the extent size. When this limit is exceeded, Lustre examines the OSTs assigned to the component to determine if any of them are running low on space. If not, the component is extended by the extent size. However, if an OST does run low on space, Lustre can dynamically shorten the current component and choose a different set of OSTs to use for the next component of the layout. This can safeguard against full OSTs that might generate a <code>ENOSPC</code> error when a user attempts to append data to a file.

Lustre has a feature called File Level Redundancy (FLR) that allows a user to create one or more mirrors of a file, each with its own specific layout (either normal or composite). When the file layout is inspected using <code>lfs getstripe</code>, it appears like any other composite layout. However, the <code>lcme_mirror_id</code> field is used to identify which mirror each component belongs to.

==== Distributed Namespace ====

[[File:layoutEA.png|300px|thumb|Figure 3. Using FID to access a file’s Layout EA and corresponding OST objects]]

The metatdata for the root of the Lustre file system resides on the
primary MDT. By default, the metadata for newly created files and
directories will reside on the same MDT as that of the parent directory,
so without any configuration changes, the metadata for the entire file
system would reside on a single MDT. In recent versions, a featured
called Distributed Namespace (DNE) was added to allow Lustre to utilize
multiple MDTs and thus scale up metadata operations. DNE was implemented
in multiple phases, and DNE Phase 1 is referred to as Remote
Directories. Remote Directories allow a Lustre administrator to assign a
new subdirectory to a different MDT if its parent directory resides on
MDT0. Any files or directories created in the remote directory also
reside on the same MDT as the remote directory. This creates a static
fan-out of directories from the primary MDT to other MDTs in the file
system. While this does allow Lustre to spread overall metadata
operations across mutliple servers, operations with any single directory
are still constrained by the performance of a single MDS node. The
static nature also prevents any sort of dynamic load balancing across
MDTs.

DNE Phase 2, also known as Striped Directories, removed some of these
limitations. For a striped directory, the metadata for all files and
subdirectories contained in that directory are spread across multiple
MDTs. Similar to how a file layout contains a stripe count, a striped
directory also has a stripe count. This determines how many MDTs will be
used to spread out the metadata. However, unlike file layouts which
spread data across OSTs in a round-robin manner, a striped directory
uses a hash function to calculate the MDT where the metadata should be
placed. The upcoming DNE Phase 3 expands upon the ideas in DNE Phase 2
to support the creation of auto-striped directories. An auto-striped
directory will start with a stripe count of 1 and then dynamically
increase the stripe count as the number of files/subdirectories in that
directory grows. Users can then utilize striped directories without
knowing a priori how big the directory might become or having to worry
about choosing a directory stripe count that is too low or too high.

==== File Identifiers and Layout Attributes ====

[[File:lustre_sw_stack.png|500px|thumb|Figure 4. Basic view of Lustre software stack]]

Lustre identifies all objects in the file system through the use of File
Identifiers (FIDs). A FID is a 128-bit opaque identifier used to
uniquely reference an object in the file system in much the same way
that ext4 uses inodes or ZFS uses dnodes. When a user accesses a file,
the filename is used to lookup the correct directory entry which in turn
provides the FID for the MDT object corresponding to that file. The MDT
object contains a set of extended attributes, one of which is called the
Layout Extended Attribute (or Layout EA). This Layout EA acts as a map
for the client to determine where the file data is actually stored, and
it contains a list of the OSTs as well as the FIDs for the objects on
those OSTs that hold the actual file data. [[:Media:LayoutEA.png|Figure 3]] shows an example of accessing a file with
a normal layout of stripe count 3.

=== Lustre Software Stack ===

[[File:Lustre_IO.png|300px|thumb|Figure 5. Lustre I/O operation: Lustre client requesting file data]]

The Lustre software stack is composed of several different layered components. To provide context for more detailed discussions later, a basic diagram of these components is illustrated in [[:Media:Lustre sw stack.png|Figure 4]] <ref> There are more interactions between components than are shown in the diagram, but this simplified view is presented in order to illustrate some of the key interactions.</ref>. The arrows in this diagram represent the flow of a request from a client to the Lustre servers. System calls for operations like read and write go through the Linux Virtual File System&nbsp;(VFS) layer to the Lustre LLITE layer which implements the necessary VFS operations. If the request requires metadata access, it is routed to the Logical Metadata Volume (LMV) that acts as an abstraction layer for the Metadata Client (MDC) components. There is a MDC component for each MDT target in the file system. Similarly, requests for data are routed to the Logical Object Volume (LOV) which acts as an abstraction layer for all of the Object Storage Client (OSC) components. There is an OSC component for each OST target in the file system. Finally, the requests are sent to the Lustre servers by first going through the Portal RPC (PTL-RPC) subsystem and then over the wire via the Lustre Networking (LNet) subsystem. 

Requests arriving at the Lustre servers follow the reverse path from the
LNet subsystem up through the PTL-RPC layer, finally arriving at either
the OSS component (for data requests) or the MDS component (for metadata
requests). Both the OSS and MDS components are multi-threaded and can
handle requests for multiple storage targets (OSTs or MDTs) on the same
server. Any locking requests are passed to the Lustre Distributed Lock
Manager (LDLM). Data requests are passed to the OBD Filter Device (OFD)
and then to the Object Storage Device (OSD). Metadata requests go from
the MDS straight to the OSD. In both cases, the OSD is responsible for
interfacing with the backend file system&nbsp;(either ldiskfs or ZFS) through
the Linux VFS layer.

[[:Media:Lustre IO.png|Figure 5]] provides a simple
illustration of the interactions in the Lustre software stack for a
client requesting file data. The Portal RPC and LNet layers are
represented by the arrows showing communications between the client and
the servers. The client begins by sending a request through the MDC to
the MDS to open the file. The MDS server responds with the Layout EA for
the file. Using this information, the client can determine which OST
objects hold the file data and send requests through the LOV/OSC layer
to the OSS servers to access the data.

== TESTS ==

This Section describes various tests and testing frameworks used to test Lustre functionality and performance.

=== Lustre Test Suites ===

Lustre Test Suites (LTS) is the largest collection of tests used to test Lustre file system. LTS consists of over 1600 tests, organized by their purpose and function. It is mainly composed of bash scripts, C programs and external applications. LTS provides various utilities to create, start, stop and execute tests. LTS can be used to execute test process automatically or in discrete steps. Using LTS the test process can be run as a group of tests or individual tests. LTS also allows to experiment with configurations and features such as ldiskfs, ZFS, DNE and HSM (Hierarchical Storage Manager). Tests in LTS are located in <code>/lustre/tests</code> directory in Lustre source tree and the major components in the test suite are given in [[#Lustre Test Suite components|Table 2]].

=== Terminology ===

In this Section, we describe relevant terminology related to Lustre Test Suites. All scripts and applications packaged as part of the <code>lustre-tests-*.rpm</code> and <code>lustre-iokit-*.rpm</code> are termed as Lustre Test Suites. The individual suites of tests contained in <code>/usr/lib64/lustre/tests</code> directory are termed as Test Suite. An example test suite is <code>sanity.sh</code>. A test suite is composed of Individual Tests. An example of an individual test is <code>test 4</code> from <code>large-lun.sh</code> test suite. Test suites can be bundled into a group for back-to-back execution (e.g. <code>regression</code>). Some of the LTS test examples include - Regression (<code>sanity, sanityn</code>), Feature Specific (<code>sanity-hsm, sanity-lfsck, ost-pools</code>), Configuration (<code>conf-sanity</code>), Recovery and Failures (<code>recovery-small, replay-ost-single</code>) and so on. Some of the active Lustre unit, feature and regression tests and their short description are given in [[#Lustre Tests|Table 3]].

=== Testing Lustre Code ===

When programming with Lustre, the best practices for testing are test often and test early in the development cycle. Before submitting the code changes to Lustre source tree, developer must ensure that the code passes acceptance-small test suite. To create a new test case, first find the bug that reproduces an issue, fix the bug and then verify the fixed code passes the existing tests. This means that the newly found bug/defect is not covered by the test cases from the test suite. After making sure that any of the existing test cases do not cover the new defect, a new test case can be introduced to exclusively test the bug.

==== Bypassing Failures ====

While testing Lustre, if one or more test cases are failing due to an issue not related to the bug that is currently being fixed, bypass option is available for the failing tests. For e.g., to skip <code>sanity.sh</code> sub-tests 36g and 65 and all of <code>insanity.sh</code> set the environment as,

<pre>    export SANITY_EXCEPT=&quot;36g 65&quot;
    export INSANITY=no</pre>
A single line command can also be used to skip these tests when running <code>acceptance-small</code> test, as shown below.

<pre>    SANITY_EXCEPT=&quot;36g 65&quot; INSANITY=no ./acceptance-small.sh</pre>

==== Test Framework Options ====

The examples below show how to run a full test or sub-tests from the <code>acceptance-small</code> test suite.

<ul>
<li><p>Run all tests in a test suite with the default setup.</p>
<pre>        cd lustre/tests
        sh ./acceptance-small.sh</pre></li>
<li><p>Run only <code>recover-small</code> and <code>conf-sanity</code> tests from <code>acceptance-small</code> test suite.</p>
<pre>        ACC_SM_ONLY=&quot;recovery-small conf-sanity&quot; sh ./acceptance-small.sh</pre></li>
<li><p>Run only tests 1, 3 and 4 from <code>sanity.sh</code>.</p>
<pre>        ONLY=&quot;1 3 4&quot; sh ./sanity.sh</pre></li>
<li><p>skip tests 1 to 30 and run remaining tests in <code>sanity.sh</code>.</p>
<pre>        EXCEPT=&quot;`seq 1 30`&quot; sh sanity.sh</pre></li></ul>

Lustre provides flexibility to easily add new tests to any of its test scripts.

== UTILS ==

=== Introduction ===

The administrative utilities provided with Lustre software allow to set up Lustre file system in different configurations. Lustre utilities provide a wide range of configuration options for creating a file system on a block device, scaling Lustre file system by adding additional OSTs or clients, changing stripe layout for data etc. Examples of some Lustre utilities include,

* <code>mkfs.lustre</code> - This utility is used to format a disk for a Lustre service.
* <code>tunefs.lustre</code> - This is used to modify configuration information on a Lustre target disk. 
* <code>lctl</code> - <code>lctl</code> is used to control Lustre features via ioctl interfaces, including various configuration, maintenance and debugging features.
* <code>mount.lustre</code> - This is used to start Lustre client or server on a target.
* <code>lfs</code> - <code>lfs</code> is used for configuring and querying options related to files.

In the following Sections we describe various user utilities and system configuration utilities in detail.

==== lfs _migrate ====

The <code>lfs_migrate</code> utility is used to migrate file data between Lustre OSTs. This utility does the migration in two steps. It first copies the specified files to set of temporary files. This can be performed using <code>lfs setstripe</code> options, if specified. It can also optionally verify if the file contents have changed or not. The second step is to then swap the layout between the temporary file and the original file&nbsp;(or even renaming the temporary file to the original filename). <code>lfs_migrate</code> is a tool that helps users to balance or manage space usage among Lustre OSTs.

==== lctl ====

The <code>lctl</code> utility is used for controlling and configuring Lustre file system. <code>lctl</code> allows the following capabilities - control Lustre via an ioctl interface, setup Lustre in various configurations, and access debugging features of Lustre. Issuing <code>lctl</code> command on Lustre client gives a prompt that allows to execute <code>lctl</code> sub-commands. Some of the common commands associated with <code>lctl</code> are <code>dl, device, network up/down, list_nids, ping nid, and conn_list</code>.

To get help with <code>lctl</code> commands <code>lctl help &lt;COMMAND&gt;</code> or <code>lctl --list-commands</code> can be used.

Another important use of <code>lctl</code> command is accessing Lustre parameters. <code>lctl get, set_param</code> provides a platform-independent interface to the Lustre tunables. When the file system is running, <code>lctl set_param</code> command can be used to set parameters temporarily on the affected nodes. The syntax of this command is,

<pre>    lctl set_param [-P] [-d] obdtype.obdname.property=value   </pre>
In this command, <code>-P</code> option is used to set parameters permanently, <code>-d</code> deletes permanent parameters. To obtain current Lustre parameter settings, <code>lctl get_param command</code> can be used on the desired node. For example,

<pre>    lctl get_param [-n] obdtype.obdname.parameter</pre>

==== llog_reader ====

The <code>llog_reader</code> utility translates a Lustre configuration log into human-readable form. The syntax of this utility is,

<pre>    llog_reader filename</pre>
<code>llog_reader</code> reads and parses the binary formatted Lustre’s on-disk configuration logs. To examine a log file on a stopped Lustre server, mount its backing file system as <code>ldiskfs</code> or <code>zfs</code>, then use <code>llog_reader</code> to dump the log file’s contents. For example,

<pre>    mount -t ldiskfs /dev/sda /mnt/mgs
    llog_reader /mnt/mgsCONFIGS/tfs-client</pre>
This utility can also be used to examine the log files when Lustre server is running. The ldiskfs-enabled <code>debugfs</code> utility can be used to extract the log file, for example,

<pre>    debugfs -c -R 'dump CONFIGS/tfs-client /tmp/tfs-client' /dev/sds
    llog_reader /tmp/tfs-client</pre>

==== mkfs.lustre ====

The <code>mkfs.lustre</code> utility is used to format a Lustre target disk. The syntax of this utility is,

<pre>    mkfs.lustre target_type [options] device</pre>
where <code>target_type</code> can be OST, MDT, networks to which to restrict this OST/MDT and MGS. After formatting, the disk using <code>mkfs.lustre</code>, it can be mounted to start the Lustre service. Two important options that can be specified along with this command are <code>--backfstype=fstype</code> and <code>--fsname=filesystem_name</code>. The former forces a particular format for the backing file system such as <code>ldiskfs</code>&nbsp;(default) or <code>zfs</code> and the later specifies the Lustre file system name of which the disk is part of (default name for file system is <code>lustre</code>).

=== mount.lustre ===

The <code>mount.lustre</code> utility is used to mount the Lustre file system on a target or client. The syntax of this utility is,

<pre>    mount -t lustre [-o options] device mountpoint</pre>
After mounting users can use the Lustre file system to create files/directories and execute several other Lustre utilities on the file system. To unmount a mounted file system the <code>umount</code> command can be used as shown below.

<pre>    umount device mountpoint</pre>
Some of the important options used along with this utility are discussed below with the help of examples.

<ul>
<li><p>The following <code>mount</code> command mounts Lustre on a client at the mount point <code>/mnt/lustre</code> with MGS running on a node with nid <code>10.1.0.1@tcp</code>.</p>
<pre>        mount -t lustre 10.1.0.1@tcp:/lustre /mnt/lustre </pre></li>
<li><p>To start the Lustre metadata service from <code>/dev/sda</code> on a mount point <code>/mnt/mdt</code> the following command can be used.</p>
<pre>        mount -t lustre /dev/sda /mnt/mdt</pre></li></ul>

=== tunefs.lustre ===

The <code>tunefs.lustre</code> utility is used to modify configuration information on a Lustre target disk. The syntax of this utility is,

<pre>    tunefs.lustre [options] /dev/device</pre>
However the tunefs utility does not reformat the disk or erase the contents on the disk. The parameters specified using tunefs are set in addition to the old parameters by default. To erase old parameters and use newly specified parameters, use the following options with tunefs.

<pre>    tunefs.lustre --erase-params --param=new_parameters</pre>

== MGC ==

=== Introduction ===

The Lustre client software involves primarily three components, management client (MGC), a metadata client (MDC), and multiple object storage clients (OSCs), one corresponding to each OST in the file system. Among this, the management client acts as an interface between Lustre virtual file system layer and Lustre management server (MGS). MGS stores and provides information about all Lustre file systems in a cluster. Lustre targets register with MGS to provide information to MGS while Lustre clients contact MGS to retrieve information from it.

The major functionalities of MGC are Lustre log handling, Lustre distributed lock management and file system setup. MGC is the first obd device created in Lustre obd device life cycle. An obd device in Lustre provides a level of abstraction on Lustre components such that generic operations can be applied without knowing the specific devices you are dealing with. The remaining Sections describe MGC module initialization, various MGC obd operations and log handling in detail. In the following Sections we will be using the terms clients and servers to represent service clients and servers created to communicate between various components in Lustre. Whereas the physical nodes representing Lustre’s clients and servers will be explicitly mentioned as ‘Lustre clients’ and ‘Lustre servers’.

=== MGC Module Initialization ===

When the MGC module initializes, it registers MGC as an obd device type with Lustre using <code>class_register_type()</code> as shown in [[#code:class_register_type|Source Code 1]]. Obd device data and metadata operations are defined using the <code>obd_ops</code> and <code>md_ops</code> structures respectively. Since MGC deals with metadata in Lustre, it has only <code>obd_ops</code> operations defined. However the metadata client (MDC) has both metadata and data operations defined since the data operations are used to implement Data on Metadata (DoM) functionality in Lustre. The <code>class_register_type()</code> function passes <code>&amp;mgc_obd_ops</code>, <code>NULL</code>, <code>false</code>, <code>LUSTRE_MGC _NAME</code>, and <code>NULL</code> as its arguments. <code>LUSTRE_MGC_NAME</code> is defined as <code>“mgc”</code> in <code>include/obd.h</code>.

=== MGC obd Operations ===

[[File:mgc_llite_comm_1.png|thumb|576x384px|Figure 6. Communication between llite and mgc through obdclass.]]

MGC obd operations are defined by <code>mgc_obd_ops</code> structure as shown in [[#code:mgc_obd_ops|Source Code 2]]. Note that all MGC obd operations are defined as function pointers. This type of programming style avoids complex switch cases and provides a level of abstraction on Lustre components such that the generic operations can be applied without knowing the details of specific obd devices.

<span id="code:mgc_obd_ops" label="code:mgc_obd_ops"></span>'''Source Code 2: mgc_obd_ops structure defined in mgc/mgc_request.c'''</span>

<syntaxhighlight lang="c">static const struct obd_ops mgc_obd_ops = {
        .o_owner        = THIS_MODULE,
        .o_setup        = mgc_setup,
        .o_precleanup   = mgc_precleanup,

In Lustre one of the ways two subsystems share data is with the help of <code>obd_ops</code> structure. To understand how the communication between two subsystems work let us take an example of <code>mgc_get_info()</code> from the <code>mgc_obd_ops</code> structure. The subsystem llite makes a call to <code>mgc_get_info()</code>&nbsp;(in <code>llite/llite_lib.c</code>) by passing a key (<code>KEY_CONN_DATA</code>) as an argument. But notice that llite invokes <code>obd_get_info()</code> instead of <code>mgc_get_info()</code>. <code>obd_get_info()</code> is defined in <code>include/obd_class.h</code> as shown in [[:Media:Mgc llite comm 1.png|Figure 6]]. We can see that this function invokes an <code>OBP</code> macro by passing an <code>obd_export</code> device structure and a <code>get_info</code> operation. The definition of this macro concatenates <code>o</code> with <code>op</code>&nbsp;(operation) so that the resulting function call becomes <code>o_get_info()</code>.

[[File:mgc_llite_comm_2.png|thumb|576x384px|Figure 7. Data structures involved in the communication between mgc and llite subsystems.]]

So how does llite make sure that this operation is directed specifically towards MGC obd device? <code>obd_get_info()</code> from <code>llite/llite_lib.c</code> has an argument called <code>sbi-&gt;ll_md_exp</code>. The <code>sbi</code> structure is a type of <code>ll_sb_info</code> defined in <code>llite/llite_internal.h</code>&nbsp;(refer [[:Media:Mgc llite comm 2.png|Figure 7]]). And the <code>ll_md_exp</code> field from <code>ll_sb_info</code> is a type of <code>obd_export</code> structure defined in <code>include/lustre_export.h</code>. <code>obd_export</code> structure has a field <code>*exp_obd</code> which is an <code>obd_device</code> structure&nbsp;(defined in <code>include/obd.h</code>). Another MGC obd operation <code>obd_connect()</code> retrieves export using the <code>obd_device</code> structure. Two functions involved in this process are <code>class_name2obd()</code> and <code>class_num2obd()</code> defined in <code>obdclass/genops.c</code>.

In the following Sections we describe some of the important MGC obd operations in detail.

=== mgc_setup() ===

<code>mgc_setup()</code> is the initial routine that gets executed to start and setup the MGC obd device. In Lustre MGC is the first obd device that is being setup as part of the obd device life cycle process. To understand when <code>mgc_setup()</code> gets invoked in the obd device life cycle, let us explore the workflow from the Lustre module initialization.

[[File:mgc_setup.png|576x384px|thumb|Figure 8. mgc_setup() call graph starting from Lustre file system mounting]]

The Lustre module initialization begins from the <code>lustre_init()</code> routine defined in <code>llite/super25.c</code>&nbsp;(shown in [[:Media:Mgc setup.png|Figure 8]]). This routine is invoked when the <code>‘lustre’</code> module gets loaded. <code>lustre_init()</code> invokes <code>register_filesystem(&amp;lustre_fs_type)</code> which registers <code>‘lustre’</code> as the file system and adds it to the list of file systems the kernel is aware of for mount and other syscalls. <code>lustre_fs_type</code> structure is defined in the same file as shown in [[#code:lustre_fs_type|Source Code 3]].

When a user mounts Lustre, the <code>lustre_mount()</code> gets invoked as evident from this structure. <code>lustre_mount()</code> is defined in the same file and which in turn calls <code>mount_nodev()</code> routine. The <code>mount_nodev()</code> invokes its call back function <code>lustre_fill_super()</code> which is also defined in <code>llite/super25.c</code>. <code>lustre_fill_super()</code> is the entry point for the mount call from the Lustre client into Lustre.

<span id="code:lustre_fs_type" label="code:lustre_fs_type"></span>'''Source code 3: lustre_fs_type structure defined in llite/super25.c'''

<syntaxhighlight lang="c">static struct file_system_type lustre_fs_type = {
        .owner          = THIS_MODULE,
        .name           = "lustre",
        .mount          = lustre_mount,
        .kill_sb        = lustre_kill_super,
        .fs_flags       = FS_RENAME_DOES_D_MOVE,
};</syntaxhighlight>

<code>lustre_fill_super()</code> invokes <code>lustre_start_mgc()</code> defined in <code>obdclass/obd_mount.c</code>. This sets up the MGC obd device to start processing startup logs. The <code>lustre_start_simple()</code> routine called here starts the MGC obd device&nbsp;(defined in <code>obdclass/obd_mount.c</code>). <code>lustre_start_simple()</code> eventually leads to the invocation of obdclass specific routines <code>class_attach()</code> and <code>class_setup()</code>&nbsp;(described in detail in the [[Understanding_Lustre_Internals#OBDCLASS|Section 5]]) with the help of a <code>do_lcfg()</code> routine that takes obd device name and a lustre configuration command (<code>lcfg_command</code>) as arguments. Various lustre configuration commands are <code>LCFG_ATTACH, LCFG_DETACH, LCFG_SETUP, LCFG_CLEANUP</code> and so on. These are defined in <code>include/uapi/linux/lustre/lustre_cfg.h</code> as shown in [[#code:lcfg_command_type|Source Code 4]].

<span id="code:lcfg_command_type" label="code:lcfg_command_type"></span>'''Source code 4: Lustre configuration commands defined in include/uapi/linux/lustre/lustre_cfg.h'''</span>

<syntaxhighlight lang="c">enum lcfg_command_type {
        LCFG_ATTACH               = 0x00cf001, /**< create a new obd instance */
        LCFG_DETACH               = 0x00cf002, /**< destroy obd instance */
        LCFG_SETUP                = 0x00cf003, /**< call type-specific setup */
        LCFG_CLEANUP              = 0x00cf004, /**< call type-specific cleanup
                                                 */
        LCFG_ADD_UUID             = 0x00cf005, /**< add a nid to a niduuid */
        . . . . .

The first <code>lcfg_command</code> that is being passed to <code>do_lcfg()</code> routine is <code>LCFG_ATTACH</code> which will result in the invocation of obdclass function <code>class_attach()</code>. We will describe <code>class_attach()</code> in detail in [[Understanding_Lustre_Internals#OBDCLASS|Section 5]]. The second <code>lcfg_command</code> passed to <code>do_lcfg()</code> function is <code>LCFG_SETUP</code> which will result in the invocation of <code>mgc_setup()</code> eventually. <code>do_lcfg()</code> calls <code>class_process_config()</code>&nbsp;(defined in <code>obdclass/obd_config.c</code>) and passes the <code>lcfg_command</code> that it received. In case of <code>LCFG_SETUP</code> command the <code>class_setup()</code> routine gets invoked. This is defined in the same file and its primary duty is to create hashes and self export and call obd device specific setup. The device specific setup call is in turn invoked through another routine called <code>obd_setup()</code>. <code>obd_setup()</code> is defined in <code>include/obd_class.h</code> as an inline function in the same way <code>obd_get_info()</code> is defined. <code>obd_setup()</code> calls the device specific setup routine with the help of the <code>OBP</code> macro (refer [[Understanding_Lustre_Internals#MGC_obd_Operations|Section 4.3]] and [[:Media:Mgc llite comm 1.png |Figure 6]]). Here, in case of MGC obd device <code>mgc_setup()</code> defined as part of the <code>mgc_obd_ops</code> structure (shown in [[#code:mgc_obd_ops|Source Code 2]]) gets invoked by the <code>obd_setup()</code> routine. Note that the yellow colored blocks in [[:Media:Mgc setup.png|Figure 8]] will be referenced again in [[Understanding_Lustre_Internals#OBDCLASS|Section 5]] to illustrate the lifecycle of the MGC obd device.

==== Operation ====

<code>mgc_setup()</code> first adds a reference to the underlying Lustre PTL-RPC layer. Then it sets up an RPC client for the obd device using <code>client_obd_setup()</code>&nbsp;(defined in <code>ldlm/ldlm_lib.c</code>). Next <code>mgc_llog_init()</code> initializes Lustre logs which will be processed by MGC at the MGS server. These logs are also sent to the Lustre client and the client side MGC mirrors these logs to process the data. The tunable parameters persistently set at MGS are sent to MGC and Lustre logs processed at the MGC initializes these parameters. In Lustre the tunables have to be set before Lustre logs are processed and <code>mgc_tunables_init()</code> helps to initialize these tunables. Few examples of the tunables set by this function are <code>conn_uuid</code>, <code>uuid</code>, <code>ping</code> and <code>dynamic_nids</code> and can be viewed in <code>/sys/fs/lustre/mgc</code> directory by logging into any Lustre client. <code>kthread_run()</code> starts an <code>mgc_requeue_thread</code> which keeps reading the lustre logs as the entries come in. A flowchart showing the <code>mgc_setup()</code> workflow is shown in [[:Media:Mgc setup vs cleanup.png|Figure 10]].

=== Lustre Log Handling ===

Lustre extensively makes use of logging for recovery and distributed transaction commits. The logs associated with Lustre are called <code>‘llogs’</code> and config logs, startup logs and change logs correspond to various kinds of <code>llogs</code>. As described in [[Understanding_Lustre_Internals#llog_reader|Section 3.2.4]], the <code>llog_reader</code> utility can be used to read these Lustre logs. When a Lustre target registers with MGS, the MGS constructs a log for the target. Similarly, a <code>lustre-client</code> log is created for the Lustre client when it is mounted. When a user mounts the Lustre client, it triggers to download the Lustre config logs on the client. As described earlier MGC subsystem is responsible for reading and processing the logs and sending them to Lustre clients and Lustre servers.

==== Log Processing in MGC ====

[[File:mgc_process_config.png|576x384px|thumb|Figure 9. mgc_process_config() call graph]]

The <code>lustre_fill_super()</code> routine described in [[Understanding_Lustre_Internals#mgc_setup.28.29|Section 4.4]] makes a call to <code>ll_fill_super()</code> function defined in <code>llite/llite_lib.c</code>. This function initializes a config log instance specific to the super block passed from <code>lustre_fill_super()</code>. Since the same MGC may be used to follow multiple config logs (e.g. ost1, ost2, Lustre client), the config log instance is used to keep the state for a specific log. Afterwards <code>lustre_fill_super()</code> invokes <code>lustre_process_log()</code> which gets a config log from MGS and starts processing it. <code>lustre_process_log()</code> gets called for both Lustre clients and Lustre servers and it continues to process new statements appended to the logs. It first resets and allocates <code>lustre_cfg_bufs</code>&nbsp;(which temporarily store log data) and calls <code>obd_process_config()</code> which eventually invokes the obd device specific <code>mgc_process_config()</code>&nbsp;(as shown in [[:Media:Mgc process config.png|Figure 9]]) with the help of <code>OBP</code> macro. The <code>lcfg_command</code> passed to <code>mgc_process_config()</code> is <code>LCFG_LOG_START</code> which gets the config log from MGC, starts processing it and adds the log to list of logs to follow. <code>config_log_add()</code> defined in the same file accomplishes the task of adding the log to the list of active logs watched for updates by MGC. Few other important log processing functions in MGC are - <code>mgc_process_log</code>&nbsp;(that gets a configuration log from the MGS and processes it), <code>mgc_process_recover_nodemap_log</code>&nbsp;(called if the Lustre client was notified for target restarting by the MGS), and <code>mgc_apply_recover_logs</code>&nbsp;(applies the logs after recovery).

=== mgc_precleanup() and mgc_cleanup() ===

[[File:mgc_setup_vs_cleanup.png|576x384px|thumb|Figure 10. mgc_setup() vs. mgc_cleanup()]]

Cleanup functions are important in Lustre in case of file system unmounting or any unexpected errors during file system setup. The <code>class_cleanup()</code> routine defined in <code>obdclass/obd_config.c</code> starts the process of shutting down an obd device. This invokes <code>mgc_precleanup()</code>&nbsp;(through <code>obd_precleanup()</code>) which makes sure that all the exports are destroyed before shutting down the obd device. <code>mgc_precleanup()</code> first decrements the <code>mgc_count</code> that was incremented during <code>mgc_setup()</code>. The <code>mgc_count</code> keeps the count of the running MGC threads and makes sure not to shut down any threads prematurely. Next it waits for any requeue thread to gets completed and calls <code>obd_cleanup_client_import()</code>. <code>obd_cleanup_client_import()</code> destroys client side import interface of the obd device. Finally <code>mgc_precleanup()</code> invokes <code>mgc_llog_fini()</code> which cleans up the lustre logs associated with the MGC. The log cleaning is accomplished by <code>llog_cleanup()</code> routine defined in <code>obdclass/llog_obd.c</code>.

<code>mgc_cleanup()</code> function deletes the profiles for the last MGC obd using <code>class_del_profiles()</code> defined in <code>obdclass/obd_config.c</code>. When MGS sends a buffer of data to MGC, the lustre profiles helps to identify the intended recipients of the data. Next the <code>lprocfs_obd_cleanup()</code> routine&nbsp;(defined in <code>obdclass/lprocfs_status.c</code>) removes <code>sysfs</code> and <code>debugfs</code> entries for the obd device. It then decrements the reference to PTL-RPC layer and finally calls <code>client_obd_cleanup()</code>. This function&nbsp;(defined in <code>ldlm/ldlm_lib.c</code>) makes the obd namespace point to NULL, destroys the client side import interface and finally frees up the obd device using <code>OBD_FREE</code> macro. [[:Media:Mgc setup vs cleanup.png|Figure 10]] shows the workflows for both setup and cleanup routines in MGC parallely. The <code>class_cleanup()</code> routine defined in <code>obd_config.c</code> starts the MGC shut down process. Note that after the <code>obd_precleanup()</code>, <code>uuid-export</code> and <code>nid-export</code> hashtables are freed up and destroyed. <code>uuid-export</code> HT stores uuids for different obd devices where as <code>nid-export</code> HT stores ptl-rpc network connection information.

=== mgc_import_event() ===

The <code>mgc_import_event()</code> function handles the events reported at the MGC import interface. The type of import events identified by MGC are listed in <code>obd_import_event</code> enum defined in <code>include/lustre_import.h</code> as shown in [[#code:obd_import_event|Source Code 5]]. Client side imports are used by the clients to communicate with the exports on the server (for instance if MDS wants to communicate with MGS, MDS will be using its client import to communicate with MGS’ server side export). More detailed description of import and export interfaces on obd device is given in [[Understanding_Lustre_Internals#OBDCLASS|Section 5]].

== OBDCLASS ==

=== Introduction ===

The obdclass subsystem in Lustre provides an abstraction layer that allows generic operations to be applied on Lustre components without having the knowledge of specific components. MGC, MDC, OSC, LOV, LMV are examples of obd devices in Lustre that make use of the obdclass generic abstraction layer. The obd devices can be connected in different ways to form client-server pairs for internal communication and data exchange in Lustre. Note that the client and server referred here are service clients and servers roles temporarily assumed by the obd devices but not physical nodes representing Lustre clients and Lustre servers.

Obd devices in Lustre are stored internally in an array defined in <code>obdclass/genops.c</code> as shown in [[#code:obd_devs|Source Code 6]]. The maximum number of obd devices in Lustre per node is limited by <code>MAX_OBD_DEVICES</code> defined in <code>include/obd.h</code>&nbsp;(shown in [[#code:max_obd_devices|Source Code 7]]). The obd devices in the <code>obd_devs</code> array are indexed using an <code>obd_minor</code> number (see [[#code:obd_device|Source Code 8]]). An obd device can be identified using its minor number, name or uuid. A uuid is a unique identifier that Lustre assigns for obd devices. <code>lctl dl</code> utility (described in [[Understanding_Lustre_Internals#lctl|Section 3.2.3]]) can be used to view all local obd devices and their uuids on Lustre clients and Lustre servers.

The first field in this structure is <code>obd_type</code> as shown in [[#code:obd_type|Source Code 11]] that defines the type of the obd device - a metadata or bulk data device or both. <code>obd_magic</code> is used to identify data corruption with an obd device. Some of those are <code>obd_attached</code> - means completed attach, <code>obd_set_up</code> - finished setup, <code>abort_recovery</code> - recovery expired, <code>obd_stopping</code> - started cleanup, <code>obd_starting</code> - started setup and so on. <code>obd_uuid_hash</code> and <code>obd_nid_hash</code> are <code>uuid-export</code> and <code>nid-export</code> hash tables for the obd device respectively. An obd device is also associated with several linked lists pointing to <code>obd_nid_stats</code>, <code>obd_exports</code>, <code>obd_unlinked_exports</code> and <code>obd_delayed_exports</code>. Some of the remaining relevant fields of this structure are <code>obd_exports</code>, kset and kobject device model abstractions, timeouts for recovery, proc entries, directory entry, procfs and debugfs variables.

=== MGC Life Cycle ===

As described in [[Understanding_Lustre_Internals#MGC|Section 4]] MGC is the first obd device setup and started by Lustre in the obd device life cycle. To understand the lifecycle of MGC obd device let us start from the generic file system mount function <code>vfs_mount()</code>. <code>vfs_mount()</code> is directly invoked by the <code>mount</code> system call from the user and handles the generic portion of mounting a file system. It then invokes file system specific mount function, that is <code>lustre_mount()</code> in case of Lustre. The <code>lustre_mount()</code> defined in <code>llite/llite_lib.c</code> invokes the kernel function <code>mount_nodev()</code> as shown in [[#code:lustre_mount|Source Code 9]] which invokes <code>lustre_fill_super()</code> as its call back function.

<span id="code:lustre_mount" label="code:lustre_mount"></span>'''Source code 9: lustre_mount() function defined in llite/llite_lib.c'''

<syntaxhighlight lang="c">
static struct dentry *lustre_mount(struct file_system_type *fs_type, int flags,
                                   const char *devname, void *data)
{
        return mount_nodev(fs_type, flags, data, lustre_fill_super);
}</syntaxhighlight>

<code>lustre_fill_super()</code> function is the entry point for the mount call into Lustre. This function initializes lustre superblock, which is used by the MGC to write a local copy of config log. The <code>lustre_fill_super()</code> routine calls <code>ll_fill_super()</code> which initializes a config log instance specific for the superblock. The <code>config_llog_instance</code> structure is defined in <code>include/obd_class.h</code> as shown in [[#code:config_llog_instance|Source Code 10]]. The <code>cfg_instance</code> field in this structure is unique to this superblock. This unique <code>cfg_instance</code> is obtained using <code>ll_get_cfg_instance()</code> function defined in <code>include/obd_class.h</code>. The <code>config_llog_instance</code> structure also has a uuid (obtained from <code>obd_uuid</code> field of <code>ll_sb_info</code> structure defined in <code>llite/llite_internal.h</code>) and a callback handler defined by the function <code>class_config_llog_handler()</code>. We will come back to this callback handler later in the MGC life cycle process. The color coded blocks in [[:Media:Mgc lifecycle.png|Figure 11]] were also part of <code>mgc_setup()</code> call graph shown in [[:Media:Mgc setup.png|Figure 8]] in [[Understanding_Lustre_Internals#MGC|Section 4]].

<syntaxhighlight lang="c">struct config_llog_instance {

        struct super_block      *cfg_sb;
        struct obd_uuid          cfg_uuid;
        llog_cb_t                cfg_callback;

[[File:mgc_lifecycle.png|576x384px|thumb|Figure 11. Obd device life cycle workflow for MGC]]

The file system name field (<code>ll_fsinfo</code>) of <code>ll_sb_info</code> structure is populated by copying the profile name obtained using the <code>get_profile_name()</code> function. <code>get_profile_name()</code> defined in <code>include/lustre_disk.h</code> obtains a profile name corresponding to the mount command issued from the user from the <code>lustre_mount_data</code> structure.

Then <code>ll_fill_super()</code> then invokes the <code>lustre_process_log()</code> function (see [[:Media:Mgc lifecycle.png|Figure 11]]) which gets the config logs from MGS and starts processing them. This function is called from both Lustre clients and Lustre servers and it will continue to process new statements appended to the logs. <code>lustre_process_log()</code> is defined in <code>obdclass/obd_mount.c</code>. The three parameters passed to this function are superblock, logname and config log instance. The config instance is unique to the super block which is used by the MGC to write to the local copy of the config log and the logname is the name of the llog to be replicated from the MGS. The config log instance is used to keep the state for the specific config log (can be from ost1, ost2, Lustre client etc.) and is added to the MGC’s list of logs to follow. <code>lustre_process_log()</code> then calls <code>obd_process_config()</code> that uses the <code>OBP</code> macro (refer [[Understanding_Lustre_Internals#MGC_obd_Operations|Section 4.3]]) to call MGC specific <code>mgc_process_config()</code> function. <code>mgc_process_config()</code> gets the config log from the MGS and processes it to start any services. Logs are also added to the list of logs to watch.

We now describe the detailed workflow of <code>mgc_process_config()</code> by describing the functionalities of each sub-function that it invokes. The <code>config_log_add()</code> function categorizes the data in config log based on if the data is related to - ptl-rpc layer, configuration parameters, nodemaps and barriers. The log data related to each of these categories is then copied to memory using the function <code>config_log_find_or_add()</code>. <code>mgc_process_config()</code> next calls <code>mgc_process_log()</code> and it gets a config log from MGS and processes it. This function is called for both Lustre clients and Lustre servers to process the configuration log from the MGS. The MGC enqueues a DLM lock on the log from the MGS and if the lock gets revoked, MGC will be notified by the lock cancellation callback that the config log has changed, and will enqueue another MGS lock on it, and then continue processing the new additions to the end of the log. Lustre prevents the updation of the same log by multiple processes at the same time. The <code>mgc_process_log()</code> then calls <code>mgc_process_cfg_log()</code> function which reads the log and creates a local copy of the log on the Lustre client or Lustre server. This function first initializes an environment and a context using <code>lu_env_init()</code> and <code>llog_get_context()</code> respectively. The <code>mgc_llog_local_copy()</code> routine is used to create a local copy of the log with the environment and context previously initialized. Real time changes in the log are parsed using the function <code>class_config_parse_llog()</code>. Under read only mode, there will be no local copy or local copy will be incomplete, so Lustre will try to use remote llog first.

The <code>class_config_parse_llog()</code> function is defined in <code>obdclass/obd_config.c</code>. The arguments passed to this function are the environment, context and config log instance initialized in <code>mgc_process_cfg_log()</code> function and the config log name. The first log that is being parsed by the <code>class_config_parse_llog()</code> function is <code>start_log</code>. <code>start_log</code> contains configuration information for various Lustre file system components, obd devices and file system mounting process. <code>class_config_parse_llog()</code> first acquires a lock on the log to be parsed using a handler function (<code>llog_init_handle()</code>). It then continues the processing of the log from where it last stopped till the end of the log. To process the logs two entities are used by this function - 1. an index to parse through the data in the log, and 2. a callback function that processes and interprets the data. The call back function can be a generic handler function like <code>class_config_llog_handler()</code> or it can be customized. Note that this is the call back handler initialized by the <code>config_llog_instance</code> structure as previously mentioned in [[#code:config_llog_instance|Source Code 10]]. Additionally, the callback function provides a config marker functionality that allows to inject special flags for selective processing of data in the log. The callback handler also initializes <code>lustre_cfg_bufs</code> to temporarily store the log data. Afterwards the following actions take place in this function: translate log names to obd device names, append uuid with obd device name for each Lustre client mount and finally attach the obd device.

Each obd device then sets up a key to communicate with other devices through secure ptl-rpc layer. The rules for creating this key are stored in the config log. The obd device then creates a connection for communication. Note that the start log contains all state information for all configuration devices and the lustre configuration buffer (<code>lustre_cfg_bufs</code>) stores this information temporarily. The obd device then use this buffer to consume log data. The start log resembles to a virtual log file and it is never stored on the disk. After creating a connection, the handler performs data mining on the logs to extract information (uuid, nid etc.) required to form Lustre <code>config_logs</code>. The parameter <code>llog_rec_hdr</code> passed with <code>class_config_llog_handler()</code> function decides what type of information should be parsed from the logs. For instance <code>OBD_CFG_REC</code> indicates the handler to scan obd device configuration information and <code>CHANGELOG_REC</code> asks to parse for changelog records. Using the extracted nid and uuid information about the obd device, the handler now invokes <code>class_process_config()</code> routine. This function repeats the cycle of obd device creation for other obd devices. Notice that the only obd device exists in Lustre at this point in the life cycle is MGC. The <code>class_process_config()</code> function calls the generic obd class functions such as <code>class_attach()</code>, <code>class_add_uuid()</code>, <code>class_setup()</code> depending upon the <code>lcfg_command</code> that it receives for a specific obd device.

=== Obd Device Life Cycle ===

In this Section we describe the work flow of various obd device life cycle functions such as <code>class_attach()</code>, <code>class_setup()</code>, <code>class_precleanup()</code>, <code>class_cleanup()</code>, and <code>class_detach()</code>.

==== class_attach() ====

The first method that is called in the life cycle of an obd device is <code>class_attach()</code> and the corresponding lustre config command is <code>LCFG_ATTACH</code>. The <code>class_attach()</code> method is defined in <code>obdclass/obd_config.c</code>. It registers and adds the obd device to the list of obd devices. The list of obd devices is defined in <code>obdclass/genops.c</code> using <code>*obd_devs[MAX_OBD_DEVICES]</code>. The attach function first checks if the obd device type being passed is valid. The <code>obd_type</code> structure is defined in <code>include/obd.h</code>&nbsp;(as shown in [[#code:obd_type|Source Code 11]]). Two types of operations defined in this structure are <code>obd_ops</code>&nbsp;(i.e., data operations) and <code>md_ops</code>&nbsp;(i.e., metadata operations). These operations determine if the obd device is destined to perform data or metadata operations or both.

The <code>lu_device_type</code> field of <code>obd_type</code> structure makes sense only for real block devices such as <code>zfs</code> and <code>ldiskfs</code> osd devices. Furthermore the <code>lu_device_type</code> differentiates metadata and data devices using the tags <code>LU_DEVICE_MD</code> and <code>LU_DEVICE_DT</code> respectively. An example of an <code>lu_device_type</code> structure defined for <code>ldiskfs osd_device_type</code> is shown in [[#code:osd_device_type|Source Code 12]].

The <code>class_attach()</code> then calls a <code>class_newdev()</code> function which creates, allocates a new obd device and initializes it. All obd device loaded has metadata or data operations (or both) defined for them. For instance the LMV obd device has its <code>md_ops</code> and <code>obd_ops</code> defined in structures <code>lmv_md_ops</code> and <code>lmv_obd_ops</code> respectively. These structures and the associated operations can be seen in <code>lmv/lmv_obd.c</code> file. The <code>obd_minor</code> initialized here is the index of the obd device in <code>obd_devs</code> array.

The obd device then creates a self export using the function <code>class_new_export_self()</code>. The <code>class_new_export_self()</code> function invokes a <code>__class_new_export()</code> function which creates a new export, adds it to the hash table of exports and returns a pointer to it. Note that a self export is created only for a client obd device. The reference count for this export when created is 2, one for the hash table reference and the other for the pointer returned by this function itself. This function populates the <code>obd_export</code> structure defined in <code>include/lustre_export.h</code>&nbsp;(shown in [[#code:obd_export|Source Code 13]]). Various fields associated with this structure are explained in the next Section. Two functions that are used to increment and decrement the reference count for obd devices are <code>class_export_get()</code> and <code>class_export_put()</code> respectively. The last part of <code>class_attach()</code> is registering/listing the obd device in the <code>obd_devs</code> array which is done through <code>class_register_device()</code> function. This functions assigns a minor number to the obd device that can be used to lookup the device in the array.

==== obd_export Structure ====

This Section describes some of the relevant fields of the <code>obd_export</code> structure&nbsp;(shown in [[#code:obd_export|Source Code 13]]) that represents a target side export connection (using ptlrpc layer) for obd devices in Lustre. This is also used to connect between layers on the same node when there is no network connection between the nodes. For every connected client there exists an export structure on the server attached to the same obd device. Various fields of this structure are described below.

* <code>exp_handle</code> - On connection establishment, the export handle id is provided to client and the subsequent client RPCs contain this handle id to identify which export they are talking to.
* Set of counters described below is used to track where export references are kept. <code>exp_rpc_count</code> is the number of RPC references, <code>exp_cb_count</code> counts commit callback references, <code>exp_replay_count</code> is the number of queued replay requests to be processed and <code>exp_locks_count</code> keeps track of the number of lock references.

==== class_setup() ====

[[File:class_setup.png|576x480px|thumb|Figure 13. Workflow of class_setup() function in obd device lifecycle]]

The primary duties of <code>class_setup()</code> routine are create hashtables and self-export, and invoke the obd type specific setup() function. As an initial step this function obtains the obd device from <code>obd_devs</code> array using <code>obd_minor</code> number and asserts the <code>obd_magic</code> number to make sure data integrity. Then it sets the <code>obd_starting</code> flag to indicate that the set up of this obd device has started (refer [[#code:obd_device|Source Code 8]]). Next the <code>uuid-export</code> and <code>nid-export</code> hashtables are setup using Linux kernel builtin functions <code>rhashtable_init()</code> and <code>rhltable_init()</code>. For the <code>nid-stats</code> hashtable Lustre uses its custom implementation of hashtable namely <code>cfs_hash</code>.

A generic device setup function <code>obd_setup()</code> defined in <code>include/obd_class.h</code> is then invoked by <code>class_setup()</code> by passing the <code>odb_device</code> structure populated and the corresponding lcfg command (<code>LCFG_SETUP</code>). This leads to the invocation of device specific setup routines from various subsystems such as <code>mgc_setup()</code>, <code>lwp_setup()</code>, <code>osc_setup_common()</code> and so on. All of these setup routines invoke a <code>client_obd_setup()</code> routine that acts as a pre-setup stage before the creation of imports for the clients as shown in [[:Media:Class setup.png|Figure 13]]. The <code>client_obd_setup()</code> defined in <code>ldlm/ldlm_lib.c</code> function populates <code>client_obd</code> structure defined in <code>include/obd.h</code> as shown in [[#code:client_obd|Source Code 14]]. Note that the <code>client_obd_setup()</code> routine is called only in case of client obd devices like osp, lwp, mgc, osc, and mdc.

A new import connection for the obd device is created using the function <code>class_new_import()</code>. Finally <code>client_obd_setup()</code> creates a new ldlm namespace for the obd device that it just set up using the <code>ldlm_namespace_new()</code> routine. This completes the setup phase in the obd device lifecycle and the newly setup obd device can now be used for communications between subsystems in Lustre.

==== class_precleanup() and class_cleanup() ====

[[File:class_cleanup_1.png|672x499px|thumb|Figure 14. Lustre unmounting and initiation of class_cleanup() in obd device lifecycle.]]
[[File:class_cleanup_2.png|672x499px|thumb|Figure 15. class_cleanup() workflow in obd device lifecycle.]]

Lustre unmount process begins from the <code>ll_umount_begin()</code> function defined as part of the <code>lustre_super_operations</code> structure (shown in [[#code:lustre_super_operations|Source Code 16]]). The <code>ll_umount_begin()</code> function accepts a <code>super_block</code> from which the metadata and data exports for the <code>obd_device</code> are extracted using the <code>class_exp2obd()</code> routine. The <code>obd_force</code> flag from <code>obd_device</code> structure is set to indicate that cleanup will be performed even though the obd reference count is greater than zero. Then it periodically checks and waits to finish until there are no outstanding requests from vfs layer.

<span id="code:lustre_super_operations" label="code:lustre_super_operations">'''Source code 16: lustre_super_operations structure defined in llite/super25.c'''</span>

<syntaxhighlight lang="c">const struct super_operations lustre_super_operations =

The cleanup cycle then invokes the <code>ll_put_super()</code> routine defined in <code>llite/llite_lib.c</code>. This function obtains the <code>cfg_instance</code> and profile name corresponding to the <code>super_block</code> using <code>ll_ get_ cfg_instance()</code> and <code>get_profile_name()</code> functions respectively. Next it invokes <code>lustre_end_log()</code> routine by passing the super block, profile name and a config llog instance initialized here. The <code>lustre_end_log()</code> function defined in <code>obdclass/obd_mount.c</code> ensures to stop following updates for the config log corresponding to the config llog instance passed. <code>lustre_end_log()</code> resets lustre config buffers and calls <code>obd_process_config()</code> by passing the lcfg command <code>LCFG_LOG_END</code> and MGC as obd device. This results in the invocation of <code>mgc_process_config()</code> which calls <code>config_log_end()</code> method when <code>LCFG_LOG_END</code> is passed. The <code>config_log_end()</code> finds the config log and stops watching updates for the log.

Further <code>ll_put_super()</code> invokes <code>class_devices_in_group()</code> method which iterates through the obd devices with same group uuid and sets the <code>obd_force</code> flag for all the devices. Afterwards it calls <code>class_manual_cleanup()</code> routine which invokes obdclass functions <code>class_cleanup()</code> and <code>class_detach()</code> in the order. The <code>class_cleanup()</code> is invoked through <code>class_process_config()</code> by passing the <code>LCFG_CLEANUP</code> command.

<code>class_cleanup()</code> starts the shut down process of the obd device. This first sets the <code>obd_stopping</code> flag to indicate that cleanup has started and then waits for any already arrived connection requests to complete. Once all the requests are completed it disconnects all the exports using <code>class_disconnect_exports()</code> function (shown in [[:Media:Class cleanup 1.png|Figure 14]]). It then invokes obd generic function <code>obd_precleanup()</code> that ensures that all exports get destroyed. <code>obd_precleanup()</code> calls device specific precleanup function (e.g. <code>mgc_precleanup()</code>). <code>class_cleanup()</code> then destroys the <code>uuid-export</code>, <code>nid-export</code>, and <code>nid-stats</code> hashtables and invokes <code>class_decref()</code> function. <code>class_decref()</code> function asserts that all exports are destroyed.

<code>class_manual_cleanup()</code> then invokes <code>class_detach()</code> function by passing the <code>LCFG_DETACH</code> command. <code>class_detach()</code>&nbsp;(defined in <code>obdclass/obd_config.c</code>) makes the <code>obd_attached</code> flag to zero and unregisters the device (frees the slot in <code>obd_devs</code> array) using <code>class_unregister_device()</code> function. Next it invokes the <code>class_decref()</code> routine that destroys the last export (self export) by calling <code>class_unlink_export()</code> method. <code>class_unlink_export()</code> calls <code>class_export_put()</code> that frees the obd device using <code>class_free_dev()</code> function. <code>class_free_dev()</code> calls device specific cleanup through <code>obd_cleanup()</code> and finally invokes <code>class_put_type()</code> routine that unloads the module. This is the end of the life cycle for the obd device. An end to end workflow of <code>class_cleanup()</code> routine is illustrated in [[:Media:Class cleanup 2.png|Figure 15]].

=== Imports and Exports ===

[[File:import_export.png|576x384px|thumb|Figure 16. Import and export pair in Lustre]]

[[File:ost_mdt_comm.png|576x288px|thumb|Figure 17. Communication between ost and mdt server obd devices in Lustre]]

Obd devices in Lustre are components including lmv, lod, lov, mdc, mdd, mdt, mds, mgc, mgs, obdecho, ofd, osc, osd-ldsikfs, osd-zfs, osp, lwp, ost, and qmt. Among these mdc, mgc, osc, osp, and lwp are client obd devices meaning two server odb device components such as mdt and ost need one client device to establish communication between them. This is also applicable in case of a Lustre client communicating with Lustre servers. Client side obd devices consist of self export and import whereas server side obd devices consist of exports and reverse imports. A client obd device sends requests to the server using its import and the server receives requests using its export as illustrated in [[:Media:Import export.png|Figure 16]]. The imports on server obd devices are called reverse imports because they are used to send requests to the client obd devices. These requests are mostly callback requests sent by the server to clients infrequently. And the client uses it’s self export to receive these callback requests from the server.

For any two obd devices to communicate with each other, they need an import and export pair <ref> Feiyi Wang, H Sarp Oral, Galen M Shipman, Oleg Drokin, Di Wang, and He Huang. Understanding lustre internals. 4 2009.</ref>. For instance, let us consider the case of communication between ost and mdt obd devices. Logging into an OSS node and doing <code>lctl dl</code> shows the obd devices on the node and associated details (obd device status, type, name, uuid etc.). Examining <code>/sys/fs/lustre</code> directory can also show the obd devices corresponding to various device types. An example of the name of an obd device created for the data exchange between OST5 and MDT2 will be <code>MDT2-lwp-OST5</code>. This means that the client obd device that enables the communication here is lwp. A conceptual view of the communication between ost and mdt through import and export connections is shown in [[:Media:Ost mdt comm.png|Figure 17]]. LWP (Light Weight Proxy) obd device manages connections established from ost to mdt, and mdts to mdt0. An lwp device is used in Lustre to send quota and FLD query requests (see [[Understanding_Lustre_Internals#File_Identifiers.2C_FID_Location_Database.2C_and_Object_Index|Section 7]]). [[:Media:Ost mdt comm.png|Figure 17]] also shows the communication between mdt and ost through osp client obd device.

The obdfilter directory from <code>/proc/fs/lustre</code> lists osts present on the OSS node. All of these osts have their export connections listed in the nid format in their respective <code>exports</code> directory. The export connection information is stored in a file called <code>export</code> in each of the export connections directory. Viewing the <code>export</code> file corresponding to MDT2 shows the following fields.

* <code>name</code>: Shows the name of the ost device.
* <code>client</code>: The nid of the client export connection. (nid of MDT2 in this example.)
* <code>connect_flags</code>: Flags representing various configurations for the lnet and ptl-rpc connections between the obd devices.
* <code>connect_data</code>: Includes fields such as <code>flags, instance, target_version, mdt_index and target_index</code>.
* <code>export_flags</code>: Configuration flags for export connection.
* <code>grant</code>: Represents target specific export data.

== LIBCFS ==

=== Introduction ===

Libcfs provides APIs comprising of fundamental primitives for process management and debugging support in Lustre. Libcfs is used throughout LNet, Lustre, and associated utilities. Its APIs define a portable run time environment that is implemented consistently on all supported build targets <ref> Lustre Wiki. Lustre wiki - Subsystem Map. https://wiki.lustre.org/Subsystem_Map, 2020. </ref>. Besides debugging support libcfs provides APIs for failure injection, Linux kernel compatibility, encryption for data, Linux 64 bit time addition, log collection using tracefile, string parsing support and capabilities for querying and manipulating CPU partition tables. Libcfs is the first module that Lustre loads. The module loading function can be found in <code>tests/test-framework.sh</code> script as shown in [[#code:libcfs_loading|Source Code 17]]. When Lustre is mounted, <code>mount_facet()</code> function gets invoked and it calls <code>load_modules()</code> function. <code>load_modules()</code> invokes <code>load_modules_local()</code> that loads Lustre modules libcfs, lnet, obdclass, ptl-rpc, fld, fid, and lmv in the same order.

In the following Sections we describe libcfs APIs and functionalities in detail.

=== Data Encryption Support in Libcfs ===

Lustre implements two types of encryption capabilities - data on the wire and data at rest. Encryption over the wire protects data transfers between the physical nodes from Man-in-the-middle attacks. Whereas the objective of encrypting data at rest is protection against storage theft and network snooping. Lustre 2.14+ releases provides encryption for data at rest. Data is encrypted on Lustre client before being sent to servers and decrypted upon reception from the servers. That way applications running on Lustre client see clear text and servers see only encrypted text. Hence access to encryption keys is limited to Lustre clients.

=== CPU Partition Table Management ===

Libcfs includes APIs and data structures that help with CPU partition table management in Lustre. A CPU partition is a virtual processing unit and can have 1-N cores or 1-N NUMA nodes. Therefore a CPU partition is also viewed as a pool of processors.

A CPU partition table (CPT) consists of a set of CPU partitions. CPTs can have two modes of operation, NUMA and SMP denoted by <code>CFS_CPU_MODE_NUMA</code> and <code>CFS_CPU_MODE_SMP</code> respectively. Users can specify total number of CPU partitions while creating a CPT and ID of a CPU partition always starts from 0. For example: if there are 8 cores in the system, creating a CPT:

<pre>    with cpu_npartitions=4:
        core[0, 1] = partition[0], core[2, 3] = partition[1]
        core[4, 5] = partition[2], core[6, 7] = partition[3]
    cpu_npartitions=1:
        core[0, 1, ... 7] = partition[0]</pre>
Users can also specify CPU partitions by a string pattern.

<pre>    cpu_partitions=&quot;0[0,1], 1[2,3]&quot;
    cpu_partitions=&quot;N 0[0-3], 1[4-8]&quot;</pre>
The first character “N” means following numbers are NUMA IDs. By default, Lustre modules should refer to the global <code>cfs_cpt_tab</code>, instead of accessing hardware CPUs directly, so concurrency of Lustre can be configured by <code>cpu_npartitions</code> of the global <code>cfs_cpt_tab</code>.

[[#code:cfs_cpu_partition|Source Code 21]] and [[#code:cfs_cpt_table|Source Code 22]] show data structures that define CPU partition and CPT. A CPU partition consists of fields representing CPU mask (<code>cpt_cpumask</code>) and node mask (<code>*cpt_nodemask</code>) for the partition, NUMA distance between CPTs (<code>*cpt_distance</code>), spread rotor for NUMA allocator (<code>cpt_spread_rotor</code>) and NUMA node if <code>cpt_nodemask</code> is empty (<code>cpt_node</code>). Number of CPU partitions, structure representing partition tables and masks to represent all CPUs and nodes are the significant fields in the <code>cfs_cpt_table</code> structure.

== File Identifiers, FID Location Database, and Object Index ==

=== File Identifier (FID) ===

Lustre refers to all the data that it stores as objects. This includes not only the individual components of a striped file but also such things as directory entries, internal configuration files, etc. To identify an object, Lustre assigns a File IDentifier (FID) to the object that is unique across the file system.  <ref>There is one exception to FID uniqueness. Certain internal files (like accounting logs) may use the same FID on multiple storage targets since that information is never directly accessed from outside of the target.</ref> A FID is a 128-bit number that consists of three components: a 64-bit sequence number, a 32-bit object ID, and a 32-bit version number. The data structure for a FID is shown in [[#code:struct_lu_fid|Source Code 24]]. As noted in the code, the version number is not currently used but is reserved for future purposes.

<span id="code:struct_lu_fid" label="code:struct_lu_fid"></span>
'''Source Code 24: FID structure (include/uapi/linux/lustre/lustre_user.h)'''
<syntaxhighlight lang="c">struct lu_fid {
       /**
        * FID sequence. Sequence is a unit of migration: all files (objects)
        * with FIDs from a given sequence are stored on the same server.
        * Lustre should support 2^64 objects, so even if each sequence
        * has only a single object we can still enumerate 2^64 objects.
        **/
        __u64 f_seq;
        /* FID number within sequence. */
        __u32 f_oid;
        /**
         * FID version, used to distinguish different versions (in the sense
         * of snapshots, etc.) of the same file system object. Not currently
         * used.
         **/
        __u32 f_ver;
} __attribute__((packed));</syntaxhighlight>

Sequence numbers are controlled by the Lustre file system and allocated to clients. The entire space of sequence numbers is overseen by the sequence controller that runs on MDT0, and every storage target (MDTs and OSTs) runs a sequence manager. As the file system is started and the storage targets are brought online, each sequence manager contacts the sequence controller to obtain a unique range of sequence numbers (known as a super sequence). Every client that establishes a connection to a storage target will be granted a unique sequence number by the target’s sequence manager. This ensures that no two clients share a sequence number and that the same sequence number will always map to the same storage target.

When a client creates a new object on a storage target, the client allocates a new FID to use for the object. The FID is created by using the sequence number granted to the client by the storage target and adding a unique object ID chosen by the client. The client maintains a counter for each sequence number and increments that counter when a new object ID is needed. This combination of target-specific sequence number and client-chosen object ID (along with a version number of zero) is used to populate the <code>lu_fid</code> structure for the new object. It should be noted that FIDs are never reused within the same Lustre file system (with a few exceptions for special internal-only objects). If a client exhausts a sequence number and cannot create more FIDs, the client will contact the target and request a new sequence number.

It is important to understand that the use of the term “client” in this context does not just refer to Lustre file system clients that present the POSIX file system interface to end-users. A FID client is any node that is responsible for creating new objects, and this can include other Lustre servers. When a Lustre file system client uses the POSIX interface to create a new file, it will use a sequence number granted by an MDT target to construct a FID for the new file. This FID will be used to identify the object on the MDT that corresponds to this new file. However, the MDS server hosting the MDT will use the layout configuration for this new file to allocate objects on one or more OSTs that will contain the actual file data. In this scenario, the MDS is acting as a FID client to the OST targets. The MDS server will have been granted sequence numbers by the OST targets and use these sequence numbers to generate the FIDs that identify all the OST objects associated with the file layout.

==== Reserved Sequence Numbers and Object IDs ====

The sequence controller does not allocate certain sequence numbers to the sequence managers. These sequence numbers are reserved for special uses such as testing or compatibility with older Lustre versions. Information about these reserved sequence numbers can be found in <code>include/lustrefid.h</code>. Below is a list of sequence ranges used by Lustre:

* IGIF (Inode and Generation In FID): sequence range = [12, 2^{32} - 1]<br /> This range is reserved for  compatibility with older Lustre versions that previously identified MDT objects using the ext3 inode number on the backend file system. Since those inode values were only 32-bit integers, a FID can be generated for these older objects by simply using the inode number as  the sequence number. Since ext3 reserves inodes 0-11 for internal purposed, these sequence numbers will be used for other internal      purposes by Lustre.

* IDIF (object ID In FID): sequence range = [2^{32}, 2^{33} - 1]<br /> This range is used for compatibility to distinguish OST objects allocated from MDT0000 with sequence 0.  Bit 33 of the FID Sequence is set to 1, and the OST index along with the high 16 bits of the object number are encoded into the lower 32 bits of the sequence number.  The low 32 bits of the OST Object ID is stored in the FID OID.

* OST_MDT0: sequence = 0<br /> Used to identify existing objects allocated by MDT0000 on old formatted OSTs before the introduction of FID-on-OST.

* LLOG: sequence = 1<br /> Used internally for Lustre Log objects.

* ECHO: sequence = 2<br /> Used for testing OST IO performance to avoid conflicting with any "real" data objects.

* OST_MDT1 ... OSTMAX: sequence range = [3-9]<br /> Used for testing file systems with multiple MDTs prior to the release of DNE.  These have never been used in production.

* Normal Sequences: sequence range = [2^{33}, 2^{64} - 1] This is the sequence range used in normal production and allocated to the sequence manager and clients.  NOTE: The first 1024 sequence numbers in this range are reserved for system use.

The header file also contains some predefined object IDs that are used for local files such as user/group/project accounting logs, LFSCK checkpoints, etc.  These are part of the <code>local_oid</code> enumeration, a portion of which is shown in [[#code:enum_local_oid|Source Code 25]].

==== <code>fid</code> Kernel Module  ====

FID-related functions are built into the <code>fid</code> kernel module. The source code for this module is located in the <code>lustre/fid</code> directory.  For Lustre clients, two files are used to build this module: <code>lproc_fid.c</code> and <code>fid_request.c</code>. The <code>lproc_fid.c</code> file just contains functions needed to support debugfs and won’t be discussed in detail here.

The <code>fid_request.c</code> file contains the core functions needed to support the FID client functionality. The module entry/exit points are <code>fid_init()</code> and <code>fid_exit()</code>, but these functions just call <code>debugfs_create_dir(...)</code> and <code>debugfs_remove_recursive(...)</code> to add/remove the necessary debugfs entries. The real initialization starts in the <code>client_fid_init</code> function. This function is registered as part of the OBD operations (<code>struct obd_ops</code>) to be invoked by the MDC and OSP subsystems.  The function’s main responsibility is to allocate memory for a <code>lu_client_seq</code> structure which is then passed to <code>seq_client_init(..)</code> (an abbreviated version of which is shown in [[#code:func_seq_client_init|Source Code 26]]) where the structure is initialized. The cleanup routine starts in <code>client_fid_fini(..)</code> which then calls <code>seq_client_fini(..)</code>. These two functions decrement the appropriate reference counts on other structures and free up the memory allocated to the <code>lu_client_seq</code> structure.

There are only two other functions exported by the <code>fid</code> module: <code>seq_client_get_seq</code> and                         <code>seq_client_alloc_fid</code>. The <code>seq_client_get_seq</code> function is mainly used by the OSP subsystem when requesting a new sequence number that will be used for precreating objects. The <code>seq_client_alloc_fid</code> function is used to request a new  FID from the client’s currently allocated sequence. If the FID values in the current sequence are exhausted, a call is made to <code>seq_client_alloc_seq</code> to request a new sequence number.

The <code>fid_lib.c</code> file defines some special sequence ranges and reserved FIDs. The <code>fid_store.c</code> file contains functions used to persist sequence information to backend storage. The functions defined in this file are not exported by the module and are just used internally by the code in <code>fid_handler.c</code>. The <code>fid_handler.c</code> file contains the functions used by FID servers to handle requests from FID  clients for sequence number allocations. The <code>fid</code> module entry/exit points (<code>fid_init()</code> and <code>fid_exit()</code>) make calls to <code>fid_server_mod_init()</code> and <code>fid_server_mod_exit()</code> to handle server-specific initialization and cleanup. Server requests are handled by the functions <code>seq_server_check_and_alloc_super()</code> and <code>seq_server_alloc_meta()</code>.

=== FID Location Database (FLD) ===

Since no two storage targets ever share the same sequence numbers, a client can determine the location of an object based on the sequence number in the object's FID. To do this, a lookup table that maps sequence numbers to storage targets must be maintained. This lookup table is called the FID Location Database. The full FLD is stored on MDT0, but each Lustre server will maintain a subset of the FLD for the sequences assigned to it. Clients can send queries to a server to request a FID lookup in the server FLD. The response is then added to the client’s local FLD cache to speed future lookups.

The code related to FLD is contained in the <code>lustre/fld</code> directory. The <code>fld</code> kernel module is built from the following source files:

* <code>fld_internal.h</code> - Contains structure definitions for FLD caching (as well as the function declarations used in the other  <code>*.c</code> files.
* <code>lproc_fld.c</code> - Defines functions for debugfs support.
* <code>fld_cache.c</code> - Defines functions for caching results of FLD lookups on clients. These functions are only for internal use and are not exported by the module.
* <code>fld_request.c</code> - Defines module entry/exit points (<code>fld_init()</code> and <code>fld_exit()</code>) as well as the    function used for looking up FLD entries (<code>fld_client_lookup()</code>).
* <code>fld_index.c</code> - Included only by FLD servers. Defines functions for managing the FLD database itself. Only the <code>fld_insert_entry()</code> function is exported for external use. All the other functions in this file are for internal use by the <code>fld</code> module.
* <code>fld_handler.c</code> - Included only by FLD servers. Defines the functions needed to handle FLD queries from clients.

=== Object Index (OI) ===

The Object Storage Device (OSD) layer acts as an abstraction between the MDT/OST layers and the underlying backend file system (ldiskfs or ZFS) used to store the actual objects. Although Lustre may use FIDs to reference all objects, the backend file system does not. It   is the responsibility of the OSD abstraction layer to convert a Lustre FID into a storage cookie that can be used by the backend file   system to locate the desired object. The term “storage cookie” is used to represent some identifier that is specific to the type of backend file system being used. In the case of ldiskfs, the storage cookie consists of the file system inode and generation number and  is encapsulated in the <code>osd_inode_id</code> structure shown in [[#code:struct_osd_inode_id|Source Code 27]].

<span id="code:struct_osd_inode_id" label="code:struct_osd_inode_id">
'''Source Code 27: osd_inode_id structure (osd-ldiskfs/osd_oi.h)'''
</span>
<syntaxhighlight lang="c">
/*
* Storage cookie. Datum uniquely identifying inode on the underlying file
* system.
*
* osd_inode_id is the internal ldiskfs identifier for an object. It should
* not be visible outside of the osd-ldiskfs. Other OSDs may have different
* identifiers, so this cannot form any part of the OSD API.
*/

        __u32 oii_ino; /* inode number */

};

The functions themselves are defined in <code>lustre/osd-ldiskfs/osd_oi.c</code>. The OI is implemented using the Index Access Module (IAM) functions defined in the <code>lustre/osd-ldiskfs/osd_iam.*</code> source files.
