--- Description ---
It would be very useful for debugging to have a parameter tree under ldlm.namespaces..resources.*.* (and also linking the  under osc|mdc.*.ldlm.resources.* or similar) that allowed dumping all of the LDLM lock resources, along with the current locks on each resource. Essentially there should be a ...resources. file for each resource (where  is in the form of a FID ::.) and "lctl list_param" could be used on the client or server to list all of the resources with active locks. The content of each . file would be the list of locks on that resource (one lock per line in YAML format), and the lock parameters, like (shown wrapped for convenience, but should be printed one line per lock):

ldlm.namespaces.testfs-MDT0000-mdc-ffff.resources.0x200000007:0x1:0x0.0=
       - { lockh: 0x2e1ae7d1e49694a8, remote: 0x2e1ae7d1e496949a,
           type: IBT, req: PW, grant: PR, flags: 0x40210000000000,
           nid: 192.168.10.35@tcp, bits: 0x13, try_bits: 0x0 }
ldlm.namespaces.testfs-OST0000-osc-ffff.resources.0x10001000:0x2:0x0.0=
       - { lockh; 0x2e1ae84397894a4, remote: 0x2e1ae7d17364748c,
           type: EXT, req: PW, grant: PW, flags: 0x40210000014000,
           nid: 192.168.10.36@tcp, start: 0, end: 1048575, gid: 0 }
       - { lockh; 0x2e1ae84397794b4, remote: 0x2e1ae7d173688430,
           type: EXT, req: PW, grant: PW, flags: 0x40210000014000,
           nid: 192.168.10.36@tcp, start: 10485760, end: 11534335, gid: 0 }

The output should have common fields at the start, and type-specific fields at the end, to simplify parsing by scripts (eg.awk can find fields by position).

It would be useful to show both granted and blocked locks for each resource, so it is possible to see how much contention there is on each resource.  The main intention would be to allow debugging why a client or server thread is blocked on a DLM lock by dumping the current holder(s) and waiter(s) on a particular lock resource, and then following the trail to the node(s) and thread PIDs holding a lock on the resource to see why it is not releasing the lock.

This would also be useful on the MDS for dumping all of the flock locks in the system for debugging/monitoring.

Writing "clear" or "0" into one of the . files would either cancel all locks on the resource (if on the client and no references are held), or send a blocking callback to clients locking that resource (if on the server). We might refine this to allow cancelling a specific local or remote lock handle on that resource by writing the specific local or peer  into ..

--- Comment (adilger, Thu, 8 Aug 2024 20:07:36 +0000) ---
James, the point of kernfs is that it doesn't require creation additional inodes/directories, etc. but is rather using the existing data structures (ldlm_resource, ldlm_lock) to represent directories/files and the filesystem interface is exposing this to tools and user/application access with an ASCII read/write interface.  The kernfs interface exists going back to el7.9 so it isn't any issue to use this in any released version.

While I understand you like Netlink, and that (AFAICS) is great for mass parameter scraping, I don't that is usable by the majority of tools and humans, unlike a filesystem interface.  If I'm debugging a Lustre issue, I don't want to dump all 5M locks and have to look through them, I want to do an "ls" or "list_param" with a regexp or grep to see which FIDs are available, use list_param/grep to process the contents of individual FIDs/locks, repeat as I understand the problem further.

--- Comment (adilger, Wed, 7 Aug 2024 17:48:20 +0000) ---
For implementation, the reason that I suggest kernfs instead of sysfs is that (AFAIK) it allows using the data structures themselves (struct ldlm_resource) for the tree, rather than having to allocate an inode/dentry for every resource in the tree like sysfs does.  That would potentially waste GBs of memory for millions of locks held by the server just to have this tree.

However, using kernfs instead of sysfs may have the problem that we need to explicitly mount a separate "ldlm_debugfs" filesystem for this.  If that is required, then it makes sense to mount and unmount it automatically somewhere within the existing param namespace (e.g. at /sys/kernel/debug/lustre/ldlm/resources and then symlink from /sys/kernel/debug/lustre/ldlm/namespaces/NAMESPACE/resources to ../../resources/NAMESPACE so that the locks of each namespace are visible under the namespace itself.
