Question & Quick Answer

Q: Can dentry->d_inode become NULL between dget(dentry) and dput(dentry)?

A: Yes, it can. dget() only guarantees the dentry object itself won't be freed; it does not guarantee the stability of its association with an inode. 

What dget() Actually Protects

The function dget(dentry) performs a single crucial task: it increments the dentry's reference count (d_count), preventing the memory of the dentry object itself from being deallocated until a corresponding dput() is called .

It explicitly does NOT protect :

•   The dentry->d_inode pointer from becoming NULL.

•   The inode pointed to by d_inode from being disassociated or freed.

•   The dentry's state (e.g., from "positive" to "negative").

The lifetime of the dentry object and its association with an inode are managed independently.

Scenario: How d_inode Can Become NULL

A typical race condition involves two threads operating on the same path :

Thread A (Your Code):

	1. Holds a pointer to a dentry (e.g., from a lookup).

	2. Calls dget(dentry) to grab a reference.

	3. Intends to access dentry->d_inode later.

Thread B (Concurrent Operation):

	1. Performs a delete operation (e.g., unlink) on the same file.

	2. This triggers a call to d_delete(dentry) in the kernel.

	3. If certain conditions are met (e.g., the dentry's reference count is 1 at the start of the delete operation), the kernel calls dentry_unlink_inode(dentry). The key function here is __d_clear_type_and_inode(dentry), which directly sets dentry->d_inode = NULL .

Result for Thread A:

    When Thread A checks dentry->d_inode after this sequence, it will find it is NULL, even though the dentry object itself remains valid. The dentry has transitioned from a "positive" state to a "negative" state .

Key Takeaway

This behavior is a normal and intentional part of the VFS design. You must always assume that dentry->d_inode can become NULL after dget(). The dget() function protects the dentry's existence, not its connection to an inode.

Safe Usage Pattern

To safely access the inode, you should not rely solely on dget(). A common safe practice involves:

1.  Using locks (like dentry->d_lock) to briefly check the d_inode pointer.

2.  If the pointer is valid, obtaining a separate reference to the inode using ihold() or igrab().

3.  Then releasing the dentry lock.

This pattern ensures that once you have a reference to the inode, it won't be freed prematurely.

Example pseudo-code:

struct inode *inode = NULL;

spin_lock(&dentry->d_lock);
if (dentry->d_inode) {
    inode = dentry->d_inode;
    ihold(inode); // Take a reference on the inode
}
spin_unlock(&dentry->d_lock);

if (inode) {
    // ... Safely use the inode ...
    iput(inode); // Release the inode reference
} else {
    // Handle negative dentry (file doesn't exist)
}

Using higher-level helper functions like d_inode() (which may include safety checks) is also recommended .

When is it Stable?

The association is stable only under broader synchronization mechanisms, such as when holding a parent directory's lock (e.g., i_rwsem), which prevents concurrent renames or deletions within that directory. This stability comes from the directory lock, not from dget(). 

The main panic is caused by the case that dentry->d_inode becomes NULL even after get() before final dput().
