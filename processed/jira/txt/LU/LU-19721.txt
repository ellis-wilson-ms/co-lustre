--- Description ---
While running IOR on a Lustre 2.17.0 client (lustre-2.17.0-RC3), we observed read failures.

The workload is SSF (Single Shared File) with Lustre striping enabled.

Below is a simplified reproducer using 32 clients, 4 OSS, and 8 OSTs.

# Server: 2.17.0-RC3
# Client: 2.17.0-RC3

mkdir /lustre/stripe
lfs setstripe -c -1 -S 16M /lustre/stripe/

# run IOR with 1MB SSF 
salloc -N 32 -p src --ntasks-per-node=16 mpirun -mca btl_openib_if_include mlx5_0:1 -x UCX_NET_DEVICES=mlx5_0:1 -np 512 --allow-run-as-root /work/tools/bin/ior -i 10 -w -r -b 16g -t 1m -C -Q 17 -e -vv -o /lustre/stirpe/file 

In many cases, the IOR read phase fails.

--- Comment (qian_wc, Wed, 24 Dec 2025 02:39:43 +0000) ---
First,  we need to acquire write semaphore (on @invalidate_lock) to avoid reading the stale page that has already discarded by lock BAST.

This semaphore is on an inode which may have multiple stripes.

Second, 

Please see the commit message in "LU-19427 llite: fix invalidate_lock deadlock for mixed BIO/DIO", we need to use try lock to avoid possible deadlock especially for the workload mixed with BIO and DIO, thus we introduce a buffered_read_check_cbt CB to fire the pending buffered read OSC extents into wire to avoid the deadlock when trylock failed to acquire the lock.

And this patch make BASTs are not serialized and can execute in parallel. But it may cause the starvation of the share read lock acquiring on @invalidate_lock for the normal reads as the BASTs are always with high priority than the normal read I/O call during acquiring @invaldiate_lock. But I think it is acceptable. And in LU-19427 we are not busy-waiting and will relax CPU when trylock failed.

--- Comment (adilger, Tue, 23 Dec 2025 21:20:24 +0000) ---
Comments from Patrick in chat about this issue, before patch https://review.whamcloud.com/63148 ("Revert LU-19427 llite: fix invalidate_lock deadlock for mixed BIO/DIO") was landed:

LU-16651 serializes all cbASTs to the same file. 

The proposed fix is to make them parallel again, so the implication (not written anywhere) is that the eviction cause is extreme slowness

LU-19427 spins on trying to get a lock - using trylock - which I realize now could be very bad for performance.

It adds a while(trylock()) on a semaphore, which is going to busy-wait on the atomic and A) waste a lot of time, and B) hammer the atomic causing the swap time to rise exponentially.

So yeah, revert is go.  We'll still have a regression in parallel bl callbacks but less serious

--- Comment (qian_wc, Mon, 22 Dec 2025 07:47:15 +0000) ---
The above patch allows multiple lock blocking callbacks from multiple threads and OSC objects can enter into the critical section of invalidate_lock to discard pages. This means one thread from the context of lock blocking callbacks held the invalidate_lock, all other threads can enter the critical section at the same time and discard pages in parallel.

This can avoid lock blocking callbacks execute serially due to the write invalidate_lock which may cause the possible timeout for lock blocking callbacks.
