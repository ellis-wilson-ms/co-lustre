--- Description ---
wake_up(&osc_lru_waitq) is called unconditionally from

osc_lru_unreserve() on every page release. When the VFS

fadvise(POSIX_FADV_DONTNEED) path triggers page invalidation

via ll_releasepage -> osc_page_delete, this results in one

wake_up call per page, each taking the waitqueue spinlock.

With many cores calling fadvise in parallel from many

threads, as rclone does, this causes catastrophic spinlock

contention. Observed on a 256-core customer system running

rclone with 64 parallel transfers copying 2TB of 10GB

files over Lustre:

	69% of all CPU cycles spent in

  native_queued_spin_lock_slowpath, almost entirely from

  __wake_up_common_lock in the fadvise -> osc_page_delete

  path

	rclone consuming 6500% CPU (65 cores), nearly all in

  kernel spinlock contention

	29% system CPU across 256 cores (~74 cores in kernel)

	Effective throughput ~4.8 GB/s read + 4.8 GB/s write,

  well below what the hardware (40+ SSD OSTs over IB)

  can deliver

The I/O pattern itself is reasonable: 100% full-size 1MB

RPCs (256 pages), sequential access, healthy RPC latencies

(1-20ms). The bottleneck is entirely CPU-side.

The rclone copy pattern is: read 1MB -> write 1MB ->

fadvise(DONTNEED) on source. The fadvise invalidates

pages one at a time, each calling wake_up on the global

osc_lru_waitq. With ~89 threads hitting this path

concurrently on 256 cores, the single waitqueue spinlock

becomes a serialization point.

Calling fadvise on every block is wasteful - it would be

much better to let the file system handle cache eviction -

and unfortunately the kernel fadvise machinery is

single-page, which is very inefficient for Lustre's

per-page teardown path.

Fix: guard wake_up with waitqueue_active() so the

spinlock is only taken when there are actual waiters,

and consolidate the wake_up into osc_lru_unreserve()

so there is a single call site.
