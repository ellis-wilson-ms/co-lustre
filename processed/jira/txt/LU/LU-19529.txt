--- Description ---
Lustre RAFT implementation needs to handle RAFT storage failures: RAFT log write, 'vote'/'term' update and log application. The RAFT node that fails to write RAFT data should be removed from the RAFT cluster, and a non-voting RAFT node should be promoted to voting mode for system health.

One thing to consider: should this change be permanent? Since the RAFT nodes are chosen manually, and voting mode is also set manually, how to resolve the conflict between the change and setting?

--- Comment (laisiyao, Wed, 5 Nov 2025 07:13:43 +0000) ---
Mmm, this can be done after RAFT snapshot support, otherwise RAFT log replication and application may take long time.

And MGS0 should start RAFT service, while upon other MDT startup, it needs to check whether it's located on the primary/failover node of MGS0, if not, it can start RAFT service.

--- Comment (adilger, Wed, 5 Nov 2025 06:44:30 +0000) ---
I think the RAFT node selection should be mostly automatic. For example, always start one RAFT on the MGS node. Whenever ann MDT is mounted it would check if an RAFT service is anlready running on the MDS, or start one if not. For every separate MDS node until 5 are started. Each MDS would try to start a RAFT service, and if it finds an existing cluster with 5 writers then it would become a read-only RAFT replica (to keep local config files uptodate after any write has updated the config across the main servers).
