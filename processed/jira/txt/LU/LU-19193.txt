--- Description ---
Our test:

Test pool contains 106 HDDs in two draid2:11d:1s:53c vdevs with two NVMe special devices.

Run obdfilter-survey on the OST with rszlo=128k and rszhi=4M

During the run, monitor ZFS IOs with `zpool iostat -r 10`

We observed the overwhelming majority of IOs to disk were size 4k, for all record sizes obdfilter-survey tested. Write rates were much worse than we observed in the past on the same system.

--- Comment (behlendorf, Fri, 25 Jul 2025 22:14:29 +0000) ---
Alex, Andreas I've finally had a chance to add some debugging, reproduce this, and get to a root cause.

The short version is, if only RPCs with a non-zero offset (off != 0) are seen by osd_grow_blocksize() when first writing to the file, then obj->oo_next_blocksize is never set, and osd_grow_blocksize() then fails to picks a new block size, which means everything gets written out as 4K blocks. 4K is the default value regular file objects are set to when they're first initialized.

I was able to reproduce this with obdfilter-survey.  By using enough objects (16) and threads (32) it was likely at least one of the objects would only be initially written to at a non-zero offset which would lock in the 4K block size.  Here's the debugging I added to osd_choose_next_blocksize() and an example log snippet:

 

        /*
         * client sends data from own writeback cache after local
         * aggregation. there is a chance this is a "unit of write"
         * so blocksize.
         */
        if (off != 0) {
                CDEBUG(D_INODE, "object "DFID": offset %u len %u\n",
                    PFID(lu_object_fid(&obj->oo_dt.do_lu)),
                    (unsigned)off, (unsigned)len);
                return;
        }

 

OBDFILTER_SURVEY: tests_str="write" nobjlo=16 nobjhi=16 thrlo=32 thrhi=32 rszlo=4096 rszhi=4096 size=16384 obdfilter-survey

 

00080000:00000002:14.0:1753473892.489435:0:2247510:0:(osd_io.c:948:osd_choose_next_blocksize()) object [0x2:0x22f9:0x0]: offset 4194304 len 4194304
00080000:00000002:26.0:1753473892.492049:0:2247520:0:(osd_io.c:948:osd_choose_next_blocksize()) object [0x2:0x22fe:0x0]: offset 4194304 len 4194304
00080000:00000002:12.0:1753473892.493550:0:2247516:0:(osd_io.c:948:osd_choose_next_blocksize()) object [0x2:0x22fc:0x0]: offset 4194304 len 4194304

Removing this (off != 0) early return does make sure we pick a blocksize based on the observed RPC length.  That's better, but I think we also want to grow the block size as much as possible when we see a bunch of small sequential write RPCs come in.  I'm going to mull it over a bit more, but I wanted to post some analysis.

Related to this I think we should still make sure to initialize new regular file objects based on the pools worst case minimum allocation size.

--- Comment (behlendorf, Thu, 24 Jul 2025 00:20:45 +0000) ---
Alex, I think trying to intelligently set the recordsize makes a lot of sense, but we should put in place a few safeguards.  For ZFS volumes we have the same basic problem where we want to avoid picking a recordsize we know will be inefficient.  There's some logic in the zfs command to make sure a reasonable default recordsize gets picked for the volume.  We could provide a new ZFS function the OSD could use to grab this default size, and then it can be used as a floor for the recordsize.  You'd never want to go smaller than this since it won't buy you any performance.  Lustre could do the calculation itself but this seems like something ZFS should provide to avoid future headaches.

> any blocksize we choose at this moment is final because blkid > 0, AFAIU

Almost, we do have a tiny bit of wiggle room.  The recordsize is final once the open txg containing the first writes for blocks >0 is closed and we begin syncing it out.  At that point it's locked in.  It looks to me like osd_grow_blocksize() and osd_choose_next_blocksize() are already exploiting the small window, so in practice I don't think that helps us.

Reading though the gerrit review it wasn't clear to me if this change was needed to help with some specific workload.  Or maybe to fix some slow tests in the CI?  Do you recall?

--- Comment (behlendorf, Tue, 22 Jul 2025 00:36:45 +0000) ---
The change isn't required to support newer ZFS versions.  It's solely intended to be a performance optimization.  There was some discussion in the original review regarding the new heuristic and how often a non-optimal block size would get selected.  At the time I don't think we had much real world data for this.

What I'm still not very clear on in why we're frequently seeing Lustre clients send small RPCs for what should be nice sequential write workloads generated by the computes.  Or if there's potentially some other subtle bug on the servers causing us to select a small record size due to the patch.

The other non-obvious issue this exposed is that internally ZFS imposes a minimum allocation size based on the pool layout.  This is primarily a concern for dRAID which needs to always pad the record size out to a full RAID stripe width (e.g for 4K disks and 8d+2p the minimum allocation size in 32K).  Unfortunately, there's not currently any public interface Lustre can call to get this size.  We probably need to add one for Lustre to use as a floor for the minimum reasonable record size.
