--- Comment (isaac, Tue, 25 Mar 2014 22:02:53 +0000) ---
Some of the test failures (e.g. zio_wait" class="issue-link" data-issue-key="LU-4716">LU-4716) we've seen could be the known ZFS IO starvation issue with the Linux cfq scheduler. Although ZFS automatically set IO scheduler to noop on whole disks, the host OS could still be using cfq for the disks behind the guess OS disks. It seemed that ZFS pools on OSS were set up:

zpool import -f -o cachefile=none -d /dev/lvm-Role_OSS lustre-ost3

It was not clear to me how devices under /dev/lvm-Role_OSS were setup and used. I think it makes sense to make sure that our test system:

1. Use whole disks for zfs pools on guest VMs.

2. Use noop IO scheduler for corresponding disks on host OS.
