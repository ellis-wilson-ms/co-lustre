J_ASSERT_JH(jh, handle->h_buffer_credits > 0);

This bug kills an OSS, then the OSS hits it on startup after that.  If we back off to a version without  25600) " class="issue-link" data-issue-key="LU-4611">LU-4611 and start the OSS, it works fine.  Then we can go back to a version with  25600) " class="issue-link" data-issue-key="LU-4611">LU-4611 and start successfully.  I assume the OSS is attempting to re-do the problematic operation each time, which is why starting once with an old version clears things up.

	/* optimistic optimization: LMA is set first and usually fit inode */
