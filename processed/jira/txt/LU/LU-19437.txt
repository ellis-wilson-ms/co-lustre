I see several problems with setting LNET_PEER_CONFIGURED though:

	while having this flag peer can't be changed by LNet, e.g. lnet_peer_set_primary_nid(), lnet_peer_add_nid() will not have effect on such peer

	peer merging and 'parallel discovery' stop working in LNetAddPeer() because of above

So consider in the following code the first NID is added to peer with LNET_PEER_CONFIGURED (and has LNET_PEER_LOCK_PRIMARY usually)

        if (LNET_NID_IS_ANY(&pnid)) {
            ...
            rc = lnet_add_peer_ni(&pnid, &LNET_ANY_NID, discovery,
                          flags | LNET_PEER_CONFIGURED);
        ...  
        } else if (!nid_same(&pnid, &nids[i])) {
            rc = lnet_add_peer_ni(&nids[i], &LNET_ANY_NID, true, 0); lp_merge_primary_nid = pnid;

                    }
                }
                lnet_discover_peer_nid(&nids[i]);

the the following NIDs are added as new peers for parallel discovery and later be merged because have lp_merge_primary_nid showing their preferred primary NID. Consider two scenarious:

	flags to use for parallel discovery peer remains as 0 - then these peers can be deleted by discovery like this ticket describes, so we lose NID info

	flags to use are set to LNET_PEER_CONFIGURED to prevent deletion - but these peer will never be merged then

both cases are not satisfying. Also having LNET_PEER_CONFIGURED in first created peer (primary one) prevents also its changing during merge.

Hence we can't use LNET_PEER_CONFIGURED without breaking current behavior.

From conditions for peer deletion it looks like LNET_PEER_NO_DISCOVERY can prevent deletion as well, as I can see that might work because initial discovery is done despite that flag, and after that flag will be either kept or reset according with remote discovery state

Probably it is simpler to add just new flag which would prevent peer from deletion, e.g. LNET_PEER_NO_DELETION and don't mark peer for deletion while that flag is set

The part of related code:

lnet_discovery_event_reply() {
...
		/* Detect whether this peer has toggled discovery from on to
		 * off and whether we can delete and re-create the peer. Peers
		 * that were manually configured cannot be deleted by discovery.
		 * We need to delete this peer and re-create it if the peer was
		 * not configured manually, is currently considered DD capable,
		 * and either:
		 * 1. We've already discovered the peer (the peer has toggled
		 *    the discovery feature from on to off), or
		 * 2. The peer is considered MR, but it was not user configured
		 *    (this was a "temporary" peer created via the kernel APIs
		 *     that we're discovering for the first time)
		 */
		if (!(lp->lp_state & (LNET_PEER_CONFIGURED |
				      LNET_PEER_NO_DISCOVERY)) &&
		    (lp->lp_state & (LNET_PEER_DISCOVERED |
				     LNET_PEER_MULTI_RAIL))) {
			CDEBUG(D_NET, "Marking %s:0x%x for deletion\n",
			       libcfs_nidstr(&lp->lp_primary_nid),
			       lp->lp_state);
			lp->lp_state |= LNET_PEER_MARK_DELETION;
		}
}

So LNet considers that peer as not added manually but as one of created internally and mark it for deletion. It seems that peers added by LNetAddPeer() should have flag LNET_PEER_CONFIGURED to mark them as 'manually' added and prevent their silent removal.

I noticed also that description is not the same as action there - peer is just deleted and not re-created, if it would re-created we would be able to find it by NID I assume.
