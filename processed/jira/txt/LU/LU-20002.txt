--- Description ---
Currently Lustre has no way to distinguish cached read (page cache hit) bytes from total read bytes. Both fast reads and slow reads are tallied into the same read_bytes counter.

Adding a separate cached_read_bytes stat with BYTES_FULL type allows users to see what fraction of their read I/O is being served from the page cache vs going through the full cl_io/DLM path. This enables computing a cache hit ratio (cached_read_bytes / read_bytes) which is valuable for performance analysis and tuning.

Implementation: add LPROC_LL_CACHED_READ to the llite stats enum, register it in the opcode table as cached_read_bytes with LPROCFS_TYPE_BYTES_FULL, and tally it in ll_do_fast_read() when the fast read succeeds.

--- Comment (paf0186, Tue, 17 Mar 2026 14:14:55 +0000) ---
That's incorrect - it's all cached reads.  You haven't seen my gerrit comment yet, so I'll copy it here (especially for anyone reading this later):

"

No, fast reads are all cached reads unless fast read has been disabled; it's a little confusing. (And you're thinking of tiny writes, which are limited to inside a single page.)

When the kernel calls us for a read, we pass the entire iovec into fast reads (ll_do_fast_read), and it tries to naively read from the page cache.

If they're already there and uptodate, great, off we go. This is the path for all reads after the first one for any pages. So all cached reads.

You're probably also thinking of what the kernel does when pages are not uptodate yet - which is how we mark them when readahead pulls them in until they've been accessed.

So if the pages are there but not uptodate (readahead has pulled them in but they've not been accessed yet), readpage() is called to bring them uptodate.

If the pages are NOT in cache, fast read fails with ENODATA and it continues on to 'slow' read and spins up a whole clio, etc.

Prior to fast reads/without fast reads, all reads used the full/slow path, even re-reads from cache."

This cached-vs-uncached IO accounting is already possible by comparing the llite..stats vs. osc..stats counters.

For example, reading a 2MB file from a client twice, shows that the llite..stats::read_bytes,read counters show 4MB of IO, but the osc..stats::read_bytes,ost_read counters show 2MB of IO:

# dd if=/myth/tmp/2M of=/dev/null bs=1M
2+0 records in
2+0 records out
2097152 bytes (2.1 MB, 2.0 MiB) copied, 0.00345304 s, 607 MB/s
# lctl get_param llite.*.stats | grep read
read_bytes                4 samples [bytes] 1048576 1048576 4194304 4398046511104
read                      4 samples [usecs] 295 118215 219479 23803260995
# lfs getstripe -i /myth/tmp/2M
4
# lctl get_param osc.myth-OST0004-osc-ffff89102e88f000.stats | grep read
read_bytes                2 samples [bytes] 1048576 1048576 2097152 2199023255552

IMHO, what is really missing here is a useful tool to display aggregate stats on a client (and server) from all of the per-device stats.  There are llstat and llobdstat, but they only print stats from a single device at a time.  It would be useful to rewrite them in Python and give them the ability to show the aggregated values from all OSC and MDC devices, similar to lljobstat does for job_stats on a server.

At that point, being able to display derived values like cached.read_bytes = llite.read_bytes.sum - osc.read_bytes.sum is fairly straight forward, and the average bandwidth for uncached IO could be computed from the osc.read_bytes.sum - osc.ost_read.sum, and cached IO bandwidth would be (llite.read_bytes.sum - osc.read_bytes.sum) / (llite.read.sum - osc.ost_read.sum).

There are now also per-OSC-per-IOsize latency stats available in osc.*.io_latency_stats

io_latency_by_size:

elapsed_time: 602663.959906696
rd_1024K: { 65536us: 2, }

but there doesn't appear to be a corresponding top-level llite.*.io_latency_stats parameter file that could be used to compute the deltas for cached vs. uncached IO.
