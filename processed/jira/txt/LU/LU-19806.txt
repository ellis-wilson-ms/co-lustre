--- Comment (adilger, Wed, 21 Jan 2026 20:07:25 +0000) ---
The most common source of failure that I've seen is in test-framework.sh due to the use of ${fsname}_CREATED_POOLS causing:

test-framework.sh: line 9303: 56af3883_CREATED_POOLS: bad substitution

because bash does not allow variable names to start with a digit.  This can be easily fixed in the test-frameworks.h code, but is not retroactive to all old versions/branches, so it would be better to constrain Autotest to set a filesystem name that does not start with a digit.

Something like FSNAME=fRND7HEX or maybe FSNAME=fsRND6HX to ensure the whole name cannot be translated as a hex number (not sure if that makes a difference or not, but not harmful).  With 16^6=2^24=16M filesystem names that should still be enough to avoid collisions for concurrent filesystem names (about 2^12 = 1/4096 chance of two random 24-bit names colliding).  Given that only ~300-400 test sessions are running at one time, maybe double that overlapping at any one time, the risk is low.  Autotest could "allocate" unique filesystem names instead of purely random names to avoid name re-use for a long period of time to entirely avoid the chance of collision.
