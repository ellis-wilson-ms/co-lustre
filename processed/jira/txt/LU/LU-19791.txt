--- Comment (paf0186, Tue, 20 Jan 2026 20:19:28 +0000) ---
With the reproducer, Augment/Opus 4.5 was able to figure out the bug and fix.  It looks good to me.  I have it working on a sanity-quota test and then it will push against this ticket.

The Bug

The bug occurs when project quota is enabled with root_prj_enable=1 and an async (buffered) write is attempted after the quota is exceeded on another client.

Root Cause

In osd_declare_inode_qid() (both ldiskfs and zfs backends), there's code that handles the QUOTA_FL_ROOT_PRJQUOTA flag. This flag is set when project quota is enabled for root users via the root_prj_enable parameter.

The original code looked like this:

if (tmp_flags & QUOTA_FL_ROOT_PRJQUOTA) {
    if (qi->lqi_is_blk)
        force = th->th_ignore_quota;
    else
        force = 0;
} 

The problem: This code unconditionally resets the force variable when QUOTA_FL_ROOT_PRJQUOTA is set. But the force variable was already set at the top of the function based on OSD_QID_FORCE:

bool force = !!(osd_qid_declare_flags & OSD_QID_FORCE) || th->th_ignore_quota; 

 

 

The OSD_QID_FORCE flag is set for async writes in osd_declare_write_commit():

if (!(lnb[i].lnb_flags & OBD_BRW_SYNC))
    declare_flags |= OSD_QID_FORCE; 

 

 

So for async writes:

	OSD_QID_FORCE is set → force = true

	But then QUOTA_FL_ROOT_PRJQUOTA check resets force = th->th_ignore_quota (which is false for normal writes)

	Quota enforcement is applied → -EDQUOT is returned

	The write silently fails during writeback, causing data loss

The Fix

Preserve OSD_QID_FORCE by only resetting force when that flag is NOT set:

if (tmp_flags & QUOTA_FL_ROOT_PRJQUOTA) {
    /* Disable th_ignore_quota for project quota when
     * root_prj_enable is set. However, preserve
     * OSD_QID_FORCE which is set for async writes -
     * those should always bypass quota enforcement
     * to avoid silent data loss.
     */
    if (!(osd_qid_declare_flags & OSD_QID_FORCE)) {
        if (qi->lqi_is_blk)
            force = th->th_ignore_quota;
        else
            force = 0;
    }
} 

Why This Is Correct

The root_prj_enable feature is meant to make project quota apply to root users (who normally bypass quota via CAP_SYS_RESOURCE). The original code disabled th->th_ignore_quota for this case.

However, OSD_QID_FORCE is a different mechanism - it's set for async writes to prevent silent data loss. When a user does a buffered write(), the data is cached and the syscall returns success. The actual write happens later during writeback. If that writeback fails with -EDQUOT, the user loses data without knowing.

The design intent is:

	Sync writes (OBD_BRW_SYNC): Enforce quota, return error immediately

	Async writes (OBD_BRW_ASYNC): Bypass quota enforcement because the write() already returned success

The bug was that QUOTA_FL_ROOT_PRJQUOTA handling incorrectly overrode the async write protection.

--- Comment (sergey, Tue, 20 Jan 2026 15:28:47 +0000) ---
Normally when we add a page to the cache(osc_page_cache_add->osc_queue_async_io) we check that current UID/GID/PRJ is not OVER QUOTA(osc_quota_chkdq). If the client hasn't got -EDQUOT yet for this ID, it doesn't know that this page can't be added to the cache. This will be known after sending BRW with this page to the OSD. Anyway we are not going to drop this page because "everything that came to the cache must be written". At some point pages added to the cache will be written and the OSD will return -EDQUOT telling our client that some ID is over quota(osc_brw_fini_request). Trying to add a new async page to the cache OSC checks the quotas(osc_quota_chkdq) and if there is no quotas, it switches to SYNC write:

        /* out of quota, try sync write */

        if ((rc == -EDQUOT && !cl_io_is_mkwrite(io)) || prio > IO_PRIO_NORMAL) {

                rc = vvp_io_commit_sync(env, io, queue,

                                        vio->u.readwrite.vui_from,

                                        vio->u.readwrite.vui_to);

                ....

That's why we don't set OSD_QID_FORCE for OBD_BRW_SYNC. It is a way to check that we still have some quotas and able to write. If there is still no quotas(QSD will even ask QMT), QSD returns -EDQUOT or proceeds handling BRW_SYNC pages in opposite case.
