The issue is that the bit values passed in to cfs_create_hash() was never updated to reflect this change in behavior. I don't know if its a simple (32 - hs_cur_bits) as a solution.

It looks like the answer is even simpler than that.  The caller in cfs_hash_bd_from_key() was properly changed to pass "bits" instead of "mask" to the various hash functions, but one critical hash function ldlm_res_hop_hash() was missed during this conversion, because the API was changed from "mask" to "bits" but the function signature was kept the same so there was no compiler warning generated that this function was still expecting a large "mask" argument and was returning only a few different buckets based on the small "bits" argument.  The following additional patch looks like it should fix this issue:

It would have been safer to change the function signature (e.g. replace "unsigned mask" with "const unsigned char bits", since 8 bits is large enough to store the maximum possible 63 bits value), and this would have allowed the compiler to complain about any unconverted functions.  However, given the current goal is to remove cfs_hash_* completely, I'm not sure it makes sense to introduce an API change as that would just slow down the other patches in flight that are replacing this code.

--- Comment (simmonsja, Wed, 27 Mar 2024 01:38:24 +0000) ---
The patch for LU-17174 fixed  a real bug. I see itsa difference of about 1/2 in performance. I think I understand what happened. We went from a bit mask which is pass in with cfs_create_hash() to using that value in hash_u32() which does a __hash_32(val) >> (32 - bits) since the idea was to use the higher bits that will have more randomness. The issue is that the bit values passed in to cfs_create_hash() was never updated to reflect this change in behavior. I don't know if its a simple 32 - hs_cur_bits as a solution.
