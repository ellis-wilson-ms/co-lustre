--- Description ---
This document presents the design and implementation for hole-aware read optimization in the Lustre file system. The design enables clients to cache file hole information locally and serve zero-filled pages for hole regions without long client I/O path, network round-trips and bulk data transfer, significantly improving performance for sparse file read operations while maintaining data consistency through Lustre's existing DLM (Distributed Lock Manager) infrastructure.

Hole-Aware Read Optimization Architecture for Lustre File System

Executive Summary

This document presents a high-level design for implementing hole-aware read optimization in the Lustre file system. The design enables clients to cache file hole information locally and serve zero-filled pages for hole regions without network round-trips and bulk data transfer, significantly improving performance for sparse file read operations while maintaining data consistency through Lustre's existing DLM (Distributed Lock Manager) infrastructure.

1. Introduction

1.1 Problem Statement

Current Lustre implementations treat hole regions in sparse files as regular data, requiring expensive RPC calls to OSTs (Object Storage Targets) that ultimately return zero-filled pages. This results in:

	Unnecessary network traffic for hole regions

	Increased latency for sparse file reads

	Suboptimal resource utilization on both clients and servers

	Performance degradation for applications working with large sparse files

1.2 Design Objectives

	Performance Optimization: Eliminate RPC calls for reading hole regions by returning zero-filled pages locally

	Shorten I/O path: Avoid the long I/O path of Lustre client I/O engine stack

	Consistency Guarantee: Maintain data consistency across multiple clients using DLM extent locks

	Scalability: Leverage existing Lustre infrastructure to minimize implementation complexity

	Transparency: Provide seamless operation without changing user-space APIs

2. Architecture Overview

2.1 Core Components

2.2 Key Design Principles

	Reuse Existing Infrastructure: Leverage OSC extent red-black trees and DLM extent locks

	Consistency First: Ensure hole information remains consistent across all clients through DLM lock protection

	Lazy Evaluation: Detect and cache holes only when beneficial

	Threshold-Based Optimization: Cache only holes above a configurable extent size threshold

	Layered Architecture: Implement optimizations at appropriate layers for maximum efficiency

3. Detailed Design

3.1 Hole Extent Management

3.1.1 Data Structure Integration

Hole extents are managed using the existing osc_extent infrastructure with the following enhancements:

	Hole Marker: Add a flag (e.g., oe_hole) to distinguish hole extents from regular data extents

	Red-Black Tree Integration: Store hole extents in the same osc_object->oo_root tree alongside regular extents

	State Management: Define specific states for hole extent lifecycle (e.g., OES_CACHE with oe_hole=1)

	Lock Association: Maintain association between hole extents and protecting DLM locks

3.1.2 Hole Extent Lifecycle

Creation → Validation → Caching → Invalidation → Cleanup
    ↓          ↓          ↓           ↓           ↓
[Detect]   [Lock]    [Store]     [Write]     [Remove]

Creation Sources:

	Client-side extended truncate operations (extending file beyond current size)

	Client-side punch hole operations (creating holes in existing data)

	Server-side hole detection during lock enqueue (for read locks)

	Server-side hole detection during read RPC processing (for adjacent regions)

Validation: All hole information must be protected by valid DLM extent locks granted to a client

Caching: Store hole extents in OSC object's red-black tree with appropriate metadata (start, end, state, hole flag)

Invalidation: Remove hole extents when conflicting write operations occur or locks are revoked

3.2 Server-Side Hole Detection

3.2.1 Backend Integration: ext4 extent_status

For OSD-ldiskfs (ext4 variant) backend:

	extent_status Infrastructure: Leverage ext4's extent_status red-black tree for efficient hole detection

	Hole Types: Detect both actual holes (unallocated regions) and unwritten extents (allocated but not written)

	Efficient Scanning: Use extent_status tree for O(log n) lookup complexity

	Accurate Boundaries: Provide precise hole start and end positions

3.2.2 Lock Enqueue Enhancement

During DLM extent lock enqueue requests for read operations:

	Hole Scanning: OST examines the requested extent range using ext4's extent_status infrastructure

	Hole Identification: Identify hole regions within the range of the extent lock to be granted to the client

	Threshold Filtering: Only report holes larger than the configured threshold (e.g., 64 KiB)

	Lock Response: Include hole information in the lock grant response (hole ranges, boundaries)

Lock Grant Message Enhancement:

	Extend lock grant reply to include hole extent information

	Provide hole range list or bitmap for efficient transmission

	Include hole metadata (size, alignment, characteristics)

3.2.3 Read RPC Processing

During read RPC handling on OST:

	Adjacent Hole Detection: Scan for holes in regions adjacent to the read request for subsequent reads on the client

	Lock Overlap Check: Verify hole regions overlap with client's granted lock extent

	Threshold Application: Apply size threshold to filter insignificant holes

	Response Augmentation: Include hole information in read RPC response when applicable

Read RPC Response Enhancement:

	Add hole information section to read reply

	Provide hole ranges within or adjacent to read range

	Enable client to cache holes for future reads

3.3 Client-Side Hole Processing

3.3.1 Hole Information Reception

Lock Grant Processing:

	Parse hole information from lock grant responses

	Create hole extents in OSC object's red-black tree

	Associate hole extents with the granted lock (optional, for efficient cleanup)

	Validate hole ranges against lock boundaries

Read RPC Response Processing:

	Extract hole information from read responses

	Update or create hole extents as appropriate

	Merge with existing cached holes when possible

	Maintain consistency with existing cached holes

3.3.2 Read Path Optimization

Layer 1: VFS/LLITE Layer:

	Early Detection: Check for hole extents before entering complex I/O stack

	Zero Filling: Directly zero-fill pages for hole regions and mark pages as uptodate at llite layer. This can avoid the long Lustre client I/O stack.

	Partial Hole Handling: Handle reads that span both hole and data regions by splitting the request

	Fast Path: Provide immediate response for pure hole reads without lower layer involvement

Layer 2: LOV Layer:

	Stripe Coordination: Coordinate hole information across multiple stripes for striped files

	Hole Range mapping: Map the hole range from OSC object to file logic range

Layer 3: OSC Layer:

	Extent Lookup: Query red-black tree for hole extents covering read range

	RPC Avoidance: Skip RPC generation for pure hole reads

	Hybrid Requests: Split requests spanning holes and data appropriately

	Cache Management: Maintain hole extent cache consistency

3.3.3 Caching hole extent list on Lock resources (optional)

On both client and server lock namespace, the hole extents can be caching on the lock resources (OSC objects) which can accerlate the hole lookup.

3.4 Consistency Management

3.4.1 DLM Integration

Lock Protection:

	All hole extents must be covered by valid DLM extent locks

	Hole extent validity tied to lock lifetime

	Automatic cleanup when locks are revoked or expired

	Lock-hole association for efficient invalidation

Lock Blocking Callbacks:

	Identify conflicting hole extents during lock revocation

	Remove invalidated hole extents from red-black tree

	Ensure no stale hole information persists after lock release

	Handle partial lock revocation scenarios

Lock Namespace Separation:

	Client Side: Client DLM lock namespace manages local lock state and hole associations

	Server Side: Server DLM lock service coordinates locks across clients and distributes hole information

3.4.2 Write Invalidation

Hole Extent Removal:

	Detect write operations overlapping with cached hole extents

	Remove conflicting hole extents during write processing

	Update hole boundaries for partial overlaps

	Maintain red-black tree consistency during removal

Conflict Resolution:

	Handle concurrent access scenarios appropriately

	Ensure write operations take precedence over cached hole information

	Provide proper synchronization mechanisms (object locks, extent locks)

	Atomic hole extent removal during write setup

3.5 Memory Optimization

3.5.1 Threshold-Based Caching

Size Threshold:

	Define configurable minimum hole size for caching (e.g., 64 KiB default)

	Skip caching for holes smaller than threshold to reduce memory overhead

	Balance memory usage against performance benefits

	Prevent cache pollution with insignificant holes

Adaptive Thresholds:

	Consider dynamic threshold adjustment based on system load

	Account for client memory pressure in threshold decisions

	Provide tunable parameters for different workload patterns

	Monitor cache effectiveness and adjust accordingly

3.5.2 Extent Coalescing

Hole Merging:

	Combine adjacent hole extents when possible

	Reduce red-black tree overhead through coalescing

	Maintain efficient lookup performance with balanced trees

	Implement lazy coalescing to minimize processing overhead

Cache Size Management:

	Implement maximum cached hole extents per object limit

	Integrate with memory shrinker infrastructure for pressure response

	Provide LRU or other eviction policies for cache management

	Monitor and report cache memory usage

4. Implementation Considerations

4.1 Performance Characteristics

Benefits:

	Eliminate RPC calls and network data transfers for reading hole regions

	Hole detection at VFS/LLITE layer before entering complex Lustre I/O stack

	Direct page zero-filling without RPC setup or network involvement

	Bypass complex Lustre I/O processing for holes

	Reduced server load and network bandwidth consumption

Trade-offs:

	Additional memory usage for hole extent caching

	Increased complexity in lock callback processing

	Potential for false positives requiring cache invalidation

	Server-side overhead for hole detection and reporting

4.2 Compatibility and Migration

Backward Compatibility:

	Design must work with existing Lustre deployments

	Graceful degradation when servers don't support hole detection

	No changes required to user-space applications

	Protocol versioning for feature negotiation

Incremental Deployment:

	Support mixed environments with hole-aware and traditional clients

	Allow server-side hole detection to be enabled independently

	Provide configuration options for gradual rollout

	Enable per-client and per-OST configuration

5. Configuration and Tuning

5.1 Tunable Parameters

Client-Side Configuration:

	Hole caching enable/disable flag (llite..hole_cache_enable*)

	Minimum hole size threshold (osc..hole_cache_threshold_kb*, default: 64 KiB)

	Maximum number of cached hole extents per object (osc..max_hole_extents*, combined with shrinker)

	Hole cache timeout values (osc..hole_cache_timeout_seconds*)

Server-Side Configuration:

	Hole detection enable/disable flag (osd-ldiskfs..hole_detection_enable*)

	Hole reporting threshold (osd-ldiskfs..hole_report_threshold_kb*, default: 64 KiB)

	Maximum holes reported per lock/RPC (filter..max_holes_per_response*)

	Performance impact limits (filter..hole_detection_cpu_limit_percent*)

5.2 Monitoring and Observability

Performance Metrics:

	RPC calls avoided due to hole detection

	Network bandwidth saved (bytes not transferred)

	Memory usage for hole extent caching

	Hit/miss ratios for hole cache lookups

	Lock callback processing overhead

	Server-side hole detection overhead

Debugging Support:

	Hole extent tree visualization tools

	Lock and hole extent correlation tracking

	Performance impact measurement tools

	Detailed logging for hole operations

	Statistics per OSC/OST for analysis

6. Future Enhancements

6.1 Advanced Optimizations

Predictive Hole Detection:

	Proactive hole scanning based on access patterns

	Machine learning-based hole prediction

	Adaptive caching strategies based on workload analysis

Cross-Client Coordination:

	Shared hole information across multiple clients

	Disable hole detection and cache mechanisms for high conflict contended objects

6.2 Extended Functionality

LNet Sparse Read Optimization Integration:

	Coordinate with LNet-level sparse read optimization

	Avoid transmitting zero-filled pages at network layer

	Bulk descriptor remapping for hole regions

	Threshold-based Sparse Read Optimization activation

Compression Integration:

	Coordinate with compression features

	Optimize hole handling in compressed files

	Unified sparse file and compression strategies

	Hole-aware compression algorithms

7. Conclusion

The hole-aware read design provides significant performance improvements for sparse file read operations in Lustre while maintaining the system's consistency guarantees and scalability characteristics. By leveraging existing infrastructure (OSC extent red-black trees, DLM extent locks) and implementing careful consistency management through lock-based protection, this design offers a practical path to optimize one of the most common performance bottlenecks in sparse file workloads.

The layered architecture ensures optimizations are applied at appropriate levels:

	VFS/LLITE Layer: Early detection and zero-filling for maximum efficiency

	LOV Layer: Stripe coordination and object hole mapping management

	OSC Layer: Hole extent caching and RPC avoidance

	DLM Layer: Consistency management through lock protection

	OST Server: Hole detection using backend filesystem capabilities

The design's modular nature allows for incremental implementation and deployment, making it suitable for production environments where gradual rollout and backward compatibility are essential requirements. The threshold-based approach ensures that optimizations are only applied when beneficial, preventing unnecessary overhead for small holes or memory-constrained environments.
