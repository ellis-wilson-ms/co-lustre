--- Description ---
It would be useful for diagnostic purposes to track statistics for average file size/blocks, component count, stripe count for each file opened on the MDS so that it is possible to collect this during debugging to get an idea of the active file and directory layout in use, independent of the filesystem default layout that may not be in use in different parts of the filesystem.

--- Comment (adilger, Thu, 13 Mar 2025 22:59:26 +0000) ---
Hi sanyam14091, sorry for the delayed reply, I was traveling last week and have been catching up.

It probably makes sense to add new "mdt.*.file_stats" parameter that tracks these different stats. This should be done when the file is being closed, since the client(s) will send the file size/blocks to the MDS at that time. I don't know if it is better to do this once per close (to reflect higher usage even if many clients are accessing the same file), or only on the last close (to reflect the files that are actually stored in the filesystem)? It is probably easiest to account the stats on every close, and would give more information than what can be obtained just from scanning the filesystem.

The stats themselves should be held in mdt_thread_info->mdt_device like mdt_rename_stats and the stats parameter files can be handled like the rename stats as well. The actual stats histogram could be printed similar to the IO size histogram in obdfilter.*.job_stats (shown here with line breaks for clarity, but it would be a single line in the output for parsing):

size:      { samples: 31, unit: kbytes, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1
blocks:    { samples: 31, unit: kbytes, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 1: 1, 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1
atime_age: { samples: 31, unit: seconds, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 1: 1, 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1 
btime_age: { samples: 31, unit: seconds, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 1: 1, 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1 
ctime_age: { samples: 31, unit: seconds, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 1: 1, 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1 
mtime_age: { samples: 31, unit: seconds, min: 0, max: 102400, sum: 294912, sumsq: 13388218368,
        hist: { 1: 1, 4K: 25, 8K: 1, 16K: 1, 32K: 2, 128K: 1 

Where blocks is from la_blocks but converted to kbytes (I think bytes will overflow the sumsq: field) and atime/btime/ctime/mtime_age is relative to the current time, so that it is possible to determine the fraction of files used that are more than, say 1d/1w/1m/1y old.

That stats would be collected in mdt_mfd_close() based on the valid attributes in ma->ma_attr.la_* using .la_valid to determine what stats are avialable (size, blocks, maybe even atime/mtime/ctime?). The file the layout size in ma->ma_lmm_size would be useful to collect (if set).

The file layout should be available via ma->ma_lmm for files, though you might defer collecting the stats from the layout to a later patch (e.g. component count, mirror count, stripe count, etc.) since that involves decoding the layout and is non-trivial (it could be lov_mds_md_v1/v3 or lov_comp_md_v1, may need iteration, etc).

Directory stats could also be done in a separate patch and a separate parameter file (mdt.*.dir_stats), since it isn't clear the stats collection would be very similar to regular files.

--- Comment (adilger, Mon, 24 Feb 2025 05:07:53 +0000) ---
Hi sanyam14091, thanks for working on this.

There is already a mechanism in Lustre to account statistics and generate a histogram of data.  The stats recording is lprocfs_oh_tally_pcpu() and lprocfs_oh_tally_log2_pcpu() for log2 buckets, and you can look at how "brw_stats" is accounted in lustre/osd-ldiskfs.

	struct obd_hist_pcpu

	lprocfs_init_brw_stats()

	lprocfs_oh_tally_log2_pcpu()

	lprocfs_oh_clear_pcpu()

	lprocfs_fini_brw_stats()

However, I don't want the histogram to be printed in the same ad-hoc format as brw_stats, but rather in a YAML format that is still human readable, but also machine parseable.  We can work out the format.
