--- Description ---
There are a number of subtests that take excessively long to complete for the value that they bring.  This ticket is for general fixes related to speeding up subtests and optimizing the test-framework.sh infrastructure to be more efficient.

--- Comment (adilger, Fri, 3 Apr 2026 16:11:14 +0000) ---
To start with, the same should be done for sanity as I proposed for conf-sanity.sh in LU-14772, namely that the tests are moved over to a separate sanity-something.sh, but that script is still run from sanity.sh until Autotest and other test runners are made aware of the new scripts and run them directly. That will avoid all of the moved scripts being skipped, while it would be possible to run the sanity-something.sh scripts separately as needed.

--- Comment (adilger, Fri, 3 Apr 2026 16:01:46 +0000) ---
For sanity.sh, I would propose to split out test cases that are unmounting clients or servers, test cases that are setting fail_loc on the server, or are otherwise affecting the operation of the filesystem. Not only does that slow down testing significantly, it makes it hard/impossible to run sanity on a client that does not have passwordless SSH access to the server.  

There have been requests in the past to have a "sanity" test that can be run on a client to check correct operation without affecting the global filesystem.  Also, the ability to run subtests in parallel (which also requires that they do not disrupt global operations) could also speed up testing significantly.
