--- Description ---
During LFSCK layout reconstruction, it may rebuild a corrupted LOV EA on the MDT based only on the trusted.fid information stored on each OST object.  The client will need to pass a flag to the OST objects (e.g. during EC writes) to mark them as EC parity and not regular data.

LFSCK in turn needs to check trusted.fid for this flag on the OST objects, and so that the rebuilt MDS layout rebuild process marks the reconstructed component/EC mirror with LOV_PATTERN_PARITY and not use it as a regular data mirror after LFSCK rebuilds the layout.

--- Comment (adilger, Fri, 7 Nov 2025 09:56:16 +0000) ---
I think it is fine if the unwritten OST objects are not restored, at least they will not provide random garbage on read.  It is OK to set the PARITY flag on these objects when they are first written.

While it is true that FLR mirrors have the same behavior, the vast majority of usage creates the mirrors only with "lfs mirror extend" so they are not attached to the file until after the data has been sync'd.  I think it is pretty rare to use a default layout that has multiple mirrors defined, and even more rare for them to become corrupted and need LFSCK to rebuild the layout.

For LFSCK layout rebuild, this issue could be fixed by marking the components for unused objects (with zero size and 07666 permission) as STALE.  It might be possible to use the layout version on the objects, or the smaller mirror (presumably with less data written to it), to determine if a mirror with written data is stale.  Or if they are all the same size then arbitrarily pick one and mark the others STALE, so they will be resync'd afterward.

--- Comment (paf0186, Thu, 6 Nov 2025 22:36:03 +0000) ---
Ah, this is interesting.

Filter fid on the OST is populated on first write, but we don't write to parity components immediately.  So lfsck doesn't restore them in this case.

But after investigating it, this is exactly what happens with mirrored files.  If you have two mirrors and you never write to the second one (ie, never call resync), it is NOT restored.  The file loses the mirror.

So this is the same as how parity mirrors work.  Is that acceptable - lfsck won't restore the parity components unless parity has been written to.  There's not a good way around it and it's the same behavior as FLR mirrors, so I think it's probably fine...?
