--- Description ---
I'd like to share few thoughs on MDS-OST the synchronization process we have now.

As you know we use llog which is supposed to work like a circular buffer. There is a problem in the current implementation though. Originally the idea was that generic Lustre recovery takes care and a failed RPC would get through at some point or we just ignore an error as is, keep corresponding llog record and hope that a next reboot would re-process this lost record.

There are few of problems with this approach:

	the llog can overflow before «next» reboot

	lost (due to an RPC error) record leads to llog overflow at some point

	re-processing old record may apply data outdated by another change long ago

So it's probably time to re-think how to deal with all this:

	Define transient errors (like EIO, EINVAL, what else?) and block processing new records
	
		probably define how many retries are allowed, then drop RPC and record?

		Limit number of saved failed RPCs so they don't consume too much RAM

	
	

	Define «good» errors (like ENOENT) and just cancel llog records immediately

This way we're still able to hit llog overflow problem if processing is being blocked for long enough. The only workaround would be to drop the oldest records to keep going.

comments, thoughts are welcome.
