And a possible other reason to follow this code-path, is that we play with namespaces during our Slurm jobs.

Doing some more investigations/testing we have found that credentials of ptlrpcd_0 threads do have their *session_keyring initialized upon first need, so may be a possible solution could be to do it during the thread start-up ?!

After joint brainstorming with Aurelien, because both still unhappy with current/first fix direction (populate the session_keyring of each ptlrpcd threads at init) that we both consider more as a work-around than a real fix, we have found that in construct_get_dest_keyring() Lustre version, lookup_user_key() should not be called because if current thread has no session_keyring set, then an attempt to populate it will cause commit_creds() to be called which will trigger this BUG() because we are already running with overrided cred. 

So I have pushed https://review.whamcloud.com/c/fs/lustre-release/+/64923 where we call key_get() instead which is conform to what is being done in Kernel's version of construct_get_dest_keyring() and which was already done before e2156e69cc ("LU-18145 gss: fix key unlink for regular user") has landed.

This reverts a part of "e2156e69cc LU-18145 gss: fix key unlink for regular user" (https://review.whamcloud.com/c/fs/lustre-release/+/56053) that do not seem to be intended to fix the original problem there ...
