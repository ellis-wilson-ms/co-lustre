I am reopening this ticket as I have found an issue with the patch "LU-19580 gss: refresh context vs LBUG upon gss error for LDLM" https://review.whamcloud.com/62467 that you made.

 

With this patch, there are situations where the client can loop indefinitely on sending SEC_CTX_INIT RPCs, instead of returning -EPERM. The simple reproducer I have is the following:

	activate skn security flavor for cli2mdt

	mount client

	on MDS, revoke the keyring key corresponding to the SSK being used by the client (keyctl revoke xxx)

	on client, flush sec contexts with 'lfs flushctx /mount_point'

I guess one could find an equivalent reproducer with Kerberos.

 

Without patch "LU-19580 gss: refresh context vs LBUG upon gss error for LDLM", the client returns -EPERM because the server is unable to authenticate it (as the SSK key is revoked).

But with the patch, the client enters an endless loop of SEC_CTX_INIT RPCs, and the syscall never returns.

 

I have identified the problem is with this added code:

https://review.whamcloud.com/c/fs/lustre-release/+/62467/5/lustre/ptlrpc/sec.c#772

if (timeout == MAX_SCHEDULE_TIMEOUT &&
   (GSS_ROUTINE_ERROR(ctx2gctx(ctx)->gc_gss_err) ==
    GSS_S_NO_CONTEXT || rc2 == -ENODATA)) {
/* Context is in error, but if MAX_SCHEDULE_TIMEOUT
* this is very likely for a LDLM req and being
* transient during a failover/failback on server side,
* so try to refresh it !
*/
CDEBUG(D_SEC,
      "ctx is in error (%p, fl %lx), trying to refresh it\n",
      ctx, ctx->cc_flags);
clear_bit(PTLRPC_CTX_ERROR_BIT, &ctx->cc_flags);
sptlrpc_sec_put(sec);
goto again;
}

With the reproducer explained above, we are in the context of an LDLM request, so timeout is MAX_SCHEDULE_TIMEOUT. And the call to sptlrpc_req_replace_dead_ctx(req, sec) returns -ENODATA. So we enter this condition and jump to 'again', over and over.

 

I am wondering about the 'rc2 == -ENODATA' part of the check. Is it really required for the bug you wanted to fix? If I remove this part of the check, then no more infinite loop with my reproducer, the syscall returns -EPERM to the application (and FYI in this case the gc_gss_err value is 0xd0000, not 0x80000/GSS_S_NO_CONTEXT).

 

Another interrogation I have is about the way that was chosen to address the bug you reported. Patch "LU-19580 gss: refresh context vs LBUG upon gss error for LDLM" basically implements another retry mechanism inside sptlrpc_req_refresh_ctx, but if the bug we need to fix is due to a target failover, then the normal Lustre HA mechanism would do the retries. And the fix should only focus on avoiding the LBUG, by handling the error case better in the upper layers.
