--- Comment (adilger, Sun, 19 Oct 2025 07:57:17 +0000) ---
Bobijam has a patch https://review.whamcloud.com/50622 ("LU-15834 lfs: mirror extend take current OSTs into account") to address this issue, but it needs to be refreshed and inspected.

Also, Ronnie has a patch series ending with https://review.whamcloud.com/60718 ("LU-19066 lod: take failure domains into account in lod_ost_alloc_rr") to assign OSTs to ensure that each stripe of an EC file is in a different failure domain. At a minimum this would mean each data+EC stripe is on a different OST, but likely also a different OSS and possibly a different controller. It would also benefit from reviews, as I have only looked at it in passing. 

That said, even for a mirrored file, it is possible to have redundancy with two stripes on the same OST, as long as the layout of both mirrors is the same (stripe count, stripe size), and the same OST is not in the same relative offset within the layout. In the above example, since OST0000 is in stripe 0 of the first mirror, and stripe 1 of the second mirror. It would not cause data loss if OST0000 failed, but if either of the other two OSTs failed then data would be lost (66% chance of data loss with 2 failures).  

If the file is striped + mirrored over 4 OSTs, then the chance of data loss with a double failure is only 33%.
