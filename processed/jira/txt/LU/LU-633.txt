--- Comment (mdiep, Tue, 22 Nov 2011 21:20:21 +0000) ---
Requirement from Di

1. mds_survey should be a script similar as obdfilter_survey, so the user can used that to evaluate lustre MD performance directly on their MDS device and you can put it to lustre/io_kit/mds_survey .

2. Current mds_survey code (i.e. those lctl utils and lustre core support for mds_survey) are here http://review.whamcloud.com/#change,1429 , which is based on current master.

3. Use cases

   a. pre-requirement: the user should at least start up MDS before running the script. If the user needs to create the real objects on OST, the OSS should be setup as well.

   b. The script will check MDS in the script and setup the MD echo client for the MDS, and you can refer to obdfilter-survey.

   c. After md echo client is setup, it can be run with these parameters.

            lothr=min hithr=max count=xxx Ops="op1 op2 op3" dir_count=xxx sh mds_survey

            lothr/hithr means minum and maxim threads count the user want to run with mds_survey, for example lothr=256 hithr=1024 means it will run with 256,512,1024 (increased by mod 2)

            count means the file_count per thread will create/destroy/getattr or sth.

            Ops="op1 op2 op3" means the script should be run with these operations, currently we support create, mkdir, destory, rmdir, lookup, getxattr, setxattr, you can see the document in the attachment for details.

             dir_count=xxx means how many directories you want to run for the test,  so threads / dir_count is how many threads will run on a single directory, for example if threads = 1024, dir_count=256, it means 1024/256 = 4 threads will operate per directory.

    d. We suppose to support DNE in this script as well, but I did not try this on DNE branch yet. So please do this for single MDT in your first step. It probably need add a few lines for DNE.
