--- Description ---
Lustre has a number of tunable parameters, but they are not very well documented. 

It would be useful to start adding man pages in section 4 ("kernel interfaces/special files") for each of the tunable parameters and stats files. This will obviously need a bunch of separate patches, but needs to start somewhere.

--- Comment (adilger, Fri, 20 Feb 2026 08:02:14 +0000) ---

It contains separate man pages for each of different places where at_min, at_max, and at_history may appear.  It looks like this just needs to be documented clearly here and in the Git tree.

--- Comment (adilger, Fri, 20 Feb 2026 07:42:42 +0000) ---
fdilger, one question that needs to be answered here is whether parameters that are specific to device types (e.g. mdt.*.enable_remote_dir) should include the device name into the man page path (e.g. mdt.enable_remote_dir), and parameters that are shared between device types (e.g. osc.*.max_rpcs_in_flight and mdc.*.max_rpcs_in_flight) should pick one of those names (e.g. osc.max_rpcs_in_flight.4) to hold the content and then reference that page from the other device names (e.g. mdc.max_rpcs_in_flight.4 contains ".so osc.max_rpcs_in_flight.4"), or should the filenames just be the "base" name (e.g. max_rpcs_in_flight)?

Most of the parameters that have been documented so far are "global" parameters that are located at the top-level of the parameter hierarchy and not instantiated on a per-device basis.  Having the per-device parameters use a single "global" name seems OK at first glance and reduces some overhead, but I worry about device-specific parameters that might already have the same name, but actually have different contents/meanings.  It is easy to pick the type-specific filenames today, even if there may be several instances of these files (e.g. kbytes or state or pool), but it would be hard to disambiguate them in the future if there are conflicts.

It would be great to edit the initial description in this ticket to explain the attributes of an ideal parameter man page, so that it can be used for reference in the future.  Possibly also in a Documentation/man4/README file for reference in the Git repo.

--- Comment (adilger, Thu, 10 Oct 2024 22:00:40 +0000) ---
There should be a consistent naming format used for the man pages related to the parameter name, so that the man page names could be generated automatically by e.g. "lfs get_param".  One proposal (possibly a bit lengthy) would be "lctl-param-TARGET.-.NAME" as a general template, and specifically something like "lctl-param-osc.-.max_dirty_mb.4" or similar.  I don't think it is possible or practical to embed "*" into the filename, so "-" is one option that could be used to replace it.

Each file would have the parameter name, a one-line description, sample output, and provide a description of the output.  This should include if the parameter is global or per-device, and whether it is only present with certain configurations.  If the parameter is readable and/or writeable, and whether writing to the parameter will reset it.
