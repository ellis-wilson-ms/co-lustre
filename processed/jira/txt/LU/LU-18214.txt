--- Comment (green, Sun, 10 Nov 2024 01:43:54 +0000) ---
I obtained a crashdump of this issue and it's confirming my suspicions. the crashing call to ldlm_flock_deadlock() is here:

                         /* Add lock to blocking list before deadlock
                         * check to prevent race
                         */
                        ldlm_flock_blocking_link(req, lock);
                        if (ldlm_flock_deadlock(req, lock)) {
                                ldlm_flock_blocking_unlink(req);
                                ldlm_flock_destroy(req, mode, *flags);
                                *err = -EDEADLK;
                                RETURN(LDLM_ITER_STOP);
                        }

we can also obviously see in the lock state that it's not really added to the resource (we can see the resource addition happens after the above statement)

the lock IS in the hash that we look for conflicts in (we just added it before the call to ldlm_flock_deadlock after all):

     {
      l_exp_flock_hash = {
        next = 0x0,
        pprev = 0xff229d874f34fe70
      },

 

Looking at the arguments, the req lock is asking for PW. the conflicting lock found is a granted PR lock, since it's granted obviously nothing is blocking it. We still try to see if anything is blocked by it and I guess something is, which we'll search and eventually come full circle to ourselves?

Which I think makes sense, we are essentially reproducing the classic deadlock scenario:

client 1: get lock1 (granted), ask for lock2

client 2: get lock2 (granted), ask for lock1

In this case the complication is probably that there's a gazillion of those granted read locks, but then there's a competing number of PW requesters equal to number of clients and they need to wade through those PR locks which with some probability would lead them back to themselves?

I think previous patch iteration was noted as causing the test to fail because flock enqueue failed, but I think this is not a test bug as the test was written to be run from a single client. Add more clients with multiple conflictign lock attempts and returning EDEADLOCK makes sense I think.

--- Comment (ys, Fri, 8 Nov 2024 08:35:54 +0000) ---
I want to give a comment for the deadlock detection code. 

The test_5c in performance-sanity hit a assert in lustre flock detection code. It did a sample test:

type   start  end
read   0      4
read   5      9
read   10     14 
.....
read   200000 200004
write  0      99999998

 It intents to measure the performance for flock merging. But if we run it on multiple clients parallel, we got a LBUG. The  case can be simplified as below:

client1          client2

open file1       open file1
read(0,10)-lk1 

 

We should determine such case as a deadlock. Otherwise those two clients will block forever. You can run below test on any filesystem to verify it.

t5() {
	echo -e "R$((i * 10)), 5\nP20\nW0,99999999"  | ./flocks_test 6 tfile
}

touch tfile
t5&
t5 

 

 So the right solution is return a EDEADLK on client2 in such case.
