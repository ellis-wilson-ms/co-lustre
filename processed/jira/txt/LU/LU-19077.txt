--- Description ---
One of the important intersections of QOS together with nodemaps is that the UID/GID/PRJID sent in from the client in ptlrpc_body_v2/v3 (pb_uid, pb_gid, pb_prjid) also need to be mapped from the tenant IDs to the global IDs.  Otherwise, the e.g. [0-99999] IDs sent in the RPCs will be the same for all tenants and QOS will likely not be doing what they want because the same limits will be applied on the sum of all tenant RPCs instead of separately for each tenant.

While this is related to nodemaps, it is not exactly the same as LU-17902 because that is putting a TBF limit on the RPCs sent by exports on that nodemap, rather than on the IDs associated with the nodemap.

--- Comment (mrasobarnett, Thu, 26 Feb 2026 10:37:07 +0000) ---
Just wanted to add a positive confirmation that this is the case currently with Lustre 2.17.0, and is indeed not the expected behaviour, in my opinion, when id-mapping is used with nodemaps.

In my testcase, I am using a nodemap configuration to squash all projids within a tenant to a single tenant-level projID, but using nodemap ID offsets to ensure this is unique for each tenant on the storage. For example:

 [root@server1 ~]# lctl get_param -R nodemap.tenant1
nodemap.tenant1.admin_nodemap=1
nodemap.tenant1.audit_mode=1
nodemap.tenant1.banlist=[]
nodemap.tenant1.child_raise_privileges=none
nodemap.tenant1.deny_mount=0
nodemap.tenant1.deny_unknown=1
nodemap.tenant1.enable_cap_mask=off
nodemap.tenant1.fileset=
[
 { primary:     /tenants/tenant1 }
]
nodemap.tenant1.forbid_encryption=0
nodemap.tenant1.id=5
nodemap.tenant1.idmap=[]
nodemap.tenant1.map_mode=projid
nodemap.tenant1.offset=
{
 start_uid: 100000,
 limit_uid: 100000,
 start_gid: 100000,
 limit_gid: 100000,
 start_projid: 100000,
 limit_projid: 100000
}
nodemap.tenant1.parent=
nodemap.tenant1.ranges=
[
 { id: 24, start_nid: 10.100.10.121@tcp1, end_nid: 10.100.10.121@tcp1 },
 { id: 23, start_nid: 10.100.10.21@tcp1, end_nid: 10.100.10.21@tcp1 }
]
nodemap.tenant1.rbac=file_perms,quota_ops,fscrypt_admin,local_admin
nodemap.tenant1.readonly_mount=0
nodemap.tenant1.sepol=
nodemap.tenant1.squash_gid=65534
nodemap.tenant1.squash_projid=0
nodemap.tenant1.squash_uid=65534
nodemap.tenant1.trusted_nodemap=0

and the tenant fileset directory has it's projid set to:

[root@server1 ~]# lfs project -d /mnt/lustre/tenants/tenant1
100000 P /mnt/lustre/tenants/tenant1

So canonical projid is 100000, however it is squashed to '0' locally on the clients within the nodemap.

I then set a TBF rule on the canonical projid as so:

 [root@server1 ~]# lctl get_param ost.OSS.ost_io.nrs_policies | grep tbf
  - name: tbf projid
  - name: tbf projid
[root@server1 ~]# lctl get_param ost.*.ost_io.nrs_tbf_rule
ost.OSS.ost_io.nrs_tbf_rule=
regular_requests:
CPT 0:
tenant1_iops projid={100000} 5, ref 0
default * 10000, ref 1
high_priority_requests:
CPT 0:
tenant1_iops projid={100000} 5, ref 0
default * 10000, ref 0

and we see that the rule is not matching ('ref 0'), and the default rule is matching.

If I set a TBF rule on the local projid '0':

[root@server1 ~]# lctl get_param ost.*.ost_io.nrs_tbf_rule
ost.OSS.ost_io.nrs_tbf_rule=
regular_requests:
CPT 0:
tenant1_projid projid={0} 2, ref 1
tenant1_iops projid={100000} 5, ref 0
default * 10000, ref 0
high_priority_requests:
CPT 0:
tenant1_projid projid={0} 2, ref 0
tenant1_iops projid={100000} 5, ref 0
default * 10000, ref 0

we now see the rule is matching ('ref 1').

Ultimately, I think what I'm configuring here would be better handled with LU-17902, however I just wanted to provide a positive confirmation report that this behaviour exists as Andreas described.
