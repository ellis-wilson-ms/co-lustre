--- Description ---
Currently, PCC requires that the administrator track and assign the archive IDs for each client with PCC configured.  This may be OK for traditional HSM archives (of which there may only be 2-3 for a filesystem), but with PCC this is prone to error, and duplicate archive IDs may result in data loss if the MDS tries to retrieve the archived files from the wrong client.

It would be useful if the MDT tracked archive ID assignment for the PCC archives, so that admins could configure PCC without having to track this themselves.  It can probably just be an integer that is incremented, since we now allow up to 2^32 archive IDs in a single filesystem.

--- Comment (adilger, Thu, 7 Aug 2025 00:33:21 +0000) ---
rread, there should probably be two separate functions - one to request a new unique ID, and one to assign the ID to the directory (which doesn't exist today).

To implement llapi_assign_archive(const char *path, __u32 id) likely needs several components:

	a mechanism to track assigned IDs persistently on MDT0000 (or on other MDTs in the future)

	an interface for clients to request the new ID from the service (e.g. GET_INFO or SET_INFO RPC?)

	new kernel API to assign the HSM archive ID to a directory and store it on the MDT

For clients that are in a nodemap, the assigned IDs should be mapped into the nodemap ID space (LU-18775), and be unique within the client nodemap, and the filesystem ID should be unique across the whole filesystem.
