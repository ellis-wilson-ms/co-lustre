--- Comment (eaujames, Wed, 4 Mar 2026 12:02:50 +0000) ---
adilger, I try to reformulate what I have understood from LU-11446:

	Global distributed link count should be stored in linkea leh_padding -> leh_linkcount

	The ldiskfs/zfs inode should store only local links

	agent deletion/insertion can be trigger by computing the remote link count:  leh_linkcount - (i_nlink - local_agent_ref)

Did I miss something?

I think that would have the advantage to keep e2fsck unchanged: it repairs only the local references.

But we should address the following issues:

	Agent local ref: if all the links are remote, we should maintain i_nlink = 1 for the local agent (otherwise e2fsck will complain). This means we count an additional ref for the local agent (that is not a Lustre link).

	Backward/Forward compatibility: existing Lustre FS rely on i_nlink to get the global Lustre link count. So we still have to manage the transition between old FS to new FS. E.g flag in LMA/inode can be added to determine if the inode is storing the global refs or the local one (reccount is not sufficient for an overflow linkea).

	This would e2fsck version incompatible, the current version fix i_nlink via linkea ("LU-11446 e2fsck: check trusted.link when fixing nlink"). We need to revert this if we implemented that i_nlink stores only the local refs.

	The code changes could be significant (linkea, lfsck, osd link refs, mdd link refs, attr get, agent ref, e2fsck...) and so the bug occurrences.

I think we could achieve the same with remote link count in linkea (except for e2fsck, it needs to trust linkea for counting nlinks). e2fsck is able to recompute the global link count for the overflow case by summing the number of local ref counted, and the remote link count saved in linkea (minus one for the agent ref counted).

  * and its name entry may reside on different MDTs. Under such case, we will  
  * create an agent entry on the MDT where the object resides. The agent entry 
  * references the object locally, that makes the object to be visible to the  
  * userspace when mounted as 'ldiskfs' directly. Then the userspace tools,    
  * such as 'tar' can handle the object properly.                              
  *                                                                            
  * We handle the agent entry during set linkEA that is the common interface   
  * for both regular file and directroy, can handle kinds of cases, such as    
  * create/link/unlink/rename, and so on.                                      
  *                                                                            
  * NOTE: we can NOT do that when ea_{insert,delete} that is only for directory
  *                                                                            
  * XXX: There are two known issues:                                           
  * 1. For one object, we will create at most one agent entry even if there    
  *    may be more than one cross-MDTs hard links on the object. So the local  
  *    e2fsck may claim that the object's nlink is larger than the name entries
  *    that reference such inode. And in further, the e2fsck will fix the nlink

  *    object is visible to userspace when mounted as 'ldiskfs'. That is enough
  */
