--- Description ---
Similar to compression, erasure coding needs careful handling of feature introduction. In particular, we need to handle the complex mix Of interactions between clients and servers as EC is introduced - handling old clients and new servers, new clients and old servers, and protecting users from using partially implemented erasure coding.

--- Comment (paf0186, Mon, 3 Nov 2025 18:50:39 +0000) ---
Here's a question, do we want the server to block clients which do not have the EC connect flag from creating EC'd files?

I think the answer is "no", since they'll handle them correctly.  The only thing that's weird is they could try to create an EC'd file that has an incorrect format, but the server checks should catch that, and the only way to set the parity flag from an old client is writing some code against LLAPI (and even open coding some stuff).

So I'm thinking the server doesn't have to restrict old clients - the pattern difference should restrict them in the ways we need.  There's some possible weirdness with an incomplete server implementation - prior to the patch which restricts EC/parity mirrors from being write targets, that could get weird.  But I feel like we can ignore an obscure case where a partially patched server is used with an old client and an EC file.

Let me know if any of this seems off to you.

Attn bobijam as well

--- Comment (paf0186, Mon, 3 Nov 2025 18:43:47 +0000) ---
Right, this ticket - admittedly not super clearly phrased - is to track actually enabling that flag and the various gyrations related to it.

 

My plan is to add the flag to both client and server at the start of the past series, but imitate the way we handled compression and put an enable disable setting in both the client and in LFS.  Technically, existing servers will allow you to create components with the parity flag.

Old clients accessing newer servers should be OK, as you noted since they'll just use the components they can and not use others.

But as we land support for erasure coding in series, we need that enable/disable support in both lfs and the kernel.  When the implementation is complete, we'll remove the setstripe env check and set the llite check to default enable.

Still thinking about some of the details here.

--- Comment (adilger, Mon, 3 Nov 2025 16:57:29 +0000) ---
Note that there is already a connection flag for this feature landed in patch https://review.whamcloud.com/58843 ("LU-10911 idl: add OBD_CONNECT2_FLR_EC flag"):

#define OBD_CONNECT2_FLR_EC            0x10000000000ULL /* parity support */    

There is also LOV_PATTERN_PARITY for the LOV EA lmm_pattern field to prevent old clients (and servers?) from trying to access the EC components as data.  They should be ignored in FLR files since patch https://review.whamcloud.com/51060 ("LU-16837 llite: handle unknown layout component"), and only mirror components with "known" patterns should be accessed.
