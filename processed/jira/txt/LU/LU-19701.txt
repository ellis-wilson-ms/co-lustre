--- Description ---
The immediate write mirroring flag is a file level layout flag, like the FLR state flags (eg, LCM_FL_RDONLY).  Currently it is not possible to set these flags from userspace, but we want to be able to add them when we update a file.

Because this flag change affects ongoing operations, it will have to be a layout intent, with handling similar to lfs mirror resync.

Specifically, we'll need to take the layout to SYNC_PENDING, flush all writes, then make the change conditional on the layout being in SYNC_PENDING.  This will be really very similar to that case.

--- Comment (paf0186, Mon, 15 Dec 2025 18:53:14 +0000) ---
Yes, I think that's what I'm suggesting.  In order to add the immediate flag (whether it is file level or mirror level):

We have an existing file in either RDONLY or WRITE_PENDING state, we have to take the layout to SYNC_PENDING - the same as for a resync operation.  Once the layout is to set to SYNC_PENDING, we do the flush - exactly the same as for resync - then use an intent to tell the MDS to update the layout atomically to include the new flag, IF the layout is still in SYNC_PENDING.  Exactly like a resync.  (Because we have to prevent clients from doing IO with the old layout without the flag.)

RE: flag location, we already have LCME_FL_NOSYNC, which is a replica level flag to skip syncing them.  That is what made me think _IMMEDIATE should be a file level flag not a component level flag.  (There are no 'mirror' level flags, sadly - the solution for EC is we're just placing LCME_FL_PARITY on all the components of the parity mirror.)

But yeah, I felt like the 'right' way to go is "IMMEDIATE" marked on the file level, then "NOSYNC" marked on mirrors we don't want to sync immediately.
