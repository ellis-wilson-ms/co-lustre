--- Description ---
We ran into a strange issue when the sticky bit is set on a directory and the directory is owned by root.

Symlinks owned by non root users in that directory give permission denied when trying to open the symlink.

Steps to reproduce:

1)

mkdir /lustre_mnt/testdir

chmod 1777 /lustre_mnt/testdir

chown root:root /lustre_mnt/testdir (if the mkdir wasn't done as root)

2) Now as a non root user

cd /lustre_mnt/testdir

touch testfile

ln -s testfile testlink

cat testlink - this works as the user 

3) As root, try to cat testlink:

> cat testlink

cat: testlink: Permission denied

> cat testfile 

> 

Cat of testfile worked, but not the symlink.

4) Change permissions on /lustre_mnt/testdir to 777 

chmod 777 /lustre_mnt/testdir

Now the cat of testlink works as root

5) Change permissions on /lustre_mnt/testdir to 1777

chmod 1777 /lustre_mnt/testdir

Now the cat of testlink gives permission denied

 

It seems like the directory must be owned by root for this to happen.   We were confused by this behavior and weren't sure if it's a bug?

--- Comment (JIRAUSER18805, Wed, 26 Mar 2025 17:04:50 +0000) ---
The problem was the fs.protected_symlinks sysctl setting... Closing report.
