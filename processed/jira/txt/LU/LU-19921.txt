--- Comment (JIRAUSER19464, Wed, 25 Feb 2026 05:44:15 +0000) ---
Tim, Thanks for reviewing. Yes, this feature is designed to integrate Lustre mount directory access control with GCP IAM.

We initially considered using nodemaps, but that requires each client to identify its group affiliation via its NIC (or something similar) so the server can locate the correct nodemap to enforce directory access control during mount. However, determining group affiliation is difficult in a cloud environment, because

	Multiple containers on the same VM host, belonging to different groups, share the same IP address.

	It is almost impossible to determine group affiliation (project, org, etc.) based purely on the user, especially since a user might hold multiple roles across different groups.

Therefore, instead of relying on nodemaps, this feature integrates Google Cloud IAM directly into the Lustre mount process. During the mount, it will make an upcall to the GCP IAM server to authenticate and authorize access to the specific mount subdirectory.

The implementation leverages the existing GSS framework. An init context RPC (for IAM security) handles the authentication and authorization via the GCP IAM server. The workflow is as follows:

Client-Side Mechanics

	Upcall Cache: The client uses an upcall cache (iam_upcall_cache) to retrieve IAM tokens for a given principal. When a user accesses the mount, the kernel calls out to a userspace helper script to fetch the user's IAM token. This token is passed back to the kernel, which then packs the token and the target mount directory into the SEC_CTX_INIT payload.

Server-Side Mechanics (MDT / OFD)

	Token Validation: When the server receives the SEC_CTX_INIT request, it triggers a server-side upcall to validate the token and mount directory against the GCP IAM server. The IAM server returns the resulting permissions (e.g., rw, deny, expired).

	Security Context: Upon successful validation, the server generates a security context, assigns a random 64-bit handle to it, and replies to the client with this context.

Client Connection

	Connection Verification: The client then sends a connect RPC to the server that includes this 64-bit handle. The server uses the handle to locate the corresponding security context and attaches it to the export. All subsequent RPCs will be verified against this security context.
