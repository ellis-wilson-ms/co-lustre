--- Description ---
This need come from the field. The CEA backup MDT images between for each maintenance (in case of disaster recovery). They use e2image for this to backup raw images on another Lustre FS. This is useful because e2image copy only the allocate blocks in a sparse file.

But e2image is slow : it takes around 1h30 to backup a 15TB MDT with only several hundreds of GB allocated. The admins have tight schedules for a maintenance, so this is problematic.

This can be explained with those points:

	Some MDTs have a lot of files (e.g: genomic use cases), so a single threaded e2image take time for the inode scanning all the inodes to get the blocks to copy.

	e2image uses 4K (blocksize) buffer IOs to copy the blocks.

I think the performances can be improved:

 1. by using multiple threads to scan the target inodes like e2fsck does (it scans different part of the FS in parallel).

 2. trying to do larger IO (read/write).

 3. parallelizing the block copy

 4. using direct IO to avoid spending time to allocate/free pages.
