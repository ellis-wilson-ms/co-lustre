--- Comment (ashehata, Mon, 1 Dec 2014 22:51:39 +0000) ---
Some more details:

mdc_setup() calls client_obd_setup() calls client_import_add_conn() calls import_set_conn calls() ptlrpc_uuid_to_connection() calls LNetDist()

mdc_setup() is only triggered on startup.  This essentially picks the src_nid and sticks with that through out.  So the addition of a new "closer NI" doesn't retrigger updating the connection hash maintained in ptlrpc.

Another option (but not as part of this bug) when a network is added connections should be evaluated and updated if there exists an NI which creates a more preferred path to the destination.

This will allow LNet to take advantage of going directly over the shortest path when updates occur.  

Ideally, however, it seems that NIDs shouldn't be visible outside of LNet.  I realize however, that this would be a major change.

--- Comment (ashehata, Mon, 24 Nov 2014 22:01:54 +0000) ---
The issue here is that both the client and the servers are on ib0 with the addition of the network dynamically.  this makes the configuration invalid due to the presence of the route that bridges the ib0 and tcp.

There are a couple of options to fix this:

1. reject the addition of a network which makes the configuration inalid

2. remove the route

Currently investigating the best solution.
