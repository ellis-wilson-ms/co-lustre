--- Description ---
From the document "The grace period applies to all users. The user block soft limit is for all users who are using a blocks quota.

Soft limit – The grace timer is started once the soft limit is exceeded. At this point, the user/group/project

can still allocate block/inode. When the grace time expires and if the user is still above the soft limit, the

soft limit becomes a hard limit and the user/group/project can't allocate any new block/inode any more.

The user/group/project should then delete files to be under the soft limit. The soft limit MUST be smaller

than the hard limit. If the soft limit is not needed, it should be set to zero (0).

Hard limit – Block or inode allocation will fail with EDQUOT(i.e. quota exceeded) when the hard limit

is reached. The hard limit is the absolute limit. When a grace period is set, one can exceed the soft limit

within the grace period if under the hard limit.

"

--- Comment (adilger, Tue, 21 Oct 2025 20:26:24 +0000) ---
The problem with cached writes is that if the error is not generated at the instant the write() call is complete, then the application will often ignore it when there is an opportunity to return an error from fsync() (rarely called) or close() (ignored, because it is too late to do anything about it).  So, rather than failing data writes that exceed quota and losing user data, Lustre prefers to allow those cached writes to succeed over quota, and only prevent future writes.

On the one hand, "lazy quota" bugs administrators who want users to only use X GB/TB of storage (honestly, considering MB units of quota is almost meaningless today when storage is cents per GB), but on the other hand it is the user data that is most important in the end, so it is better to write out that cached data and use up some small amount of free space in the filesystem rather than having to re-run hours of compute workload.  With many clients and large caches, the cached data might still be GB, but the filesystem will be PB in size at that point so it is still a reasonable tradeoff.

If there is something lacking, it is probably notification to users of soft-quota exceeded.  This was at one time implemented with a D_TTY debug message that hooked into the tty and wrote directly to the stdout of the application, but I wasn't a fan of that at the time.  Maybe it is time to change my mind, if this would make it better for systems to have a soft quota limit that users could actually be notified about.  Alternately, if the Lustre quota could somehow be hooked into the normal kernel quota tools so that they print a message when the user logs in, that might also be a usable solution.

--- Comment (adilger, Tue, 21 Oct 2025 17:04:15 +0000) ---
Rajeev, I can't comment about the specific details about the issue here, but in general testing with quota limits of only a few MB is going to be difficult to get accurate behavior, since this still falls into the "grey area" of client-side writeback cache for the files.  I would suggest to test this with larger files and quotas, and also reduce the osc.*.max_dirty_mb on the client and/or use "oflag=direct" to avoid client side cache.
