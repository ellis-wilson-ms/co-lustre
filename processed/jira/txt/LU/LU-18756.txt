--- Description ---
When migrating files off an OST, whether due to end of life hardware replacement, in advance of lengthy maintenance actions, etc. it is possible to directly scan the OST filesystem for objects that need to be migrated, and use the parent (MDT) FID in the "trusted.fid" xattr to identify which MDT file needs to be migrated and have the layout updated to reference the replacement object on another OST.

However, for OST objects that have never been modified by a client (e.g. a 4-stripe file that only had 2 MB written) would leave "unclaimed" zero-length OST objects with no identifying information to do the reverse mapping to the MDT inode.  This may result in an OST not reporting any objects in use when scanned, but if the OST is removed then access to the file referencing these objects (e.g. stat()) will return an error (-ENOENT).

In such cases, it would be useful for the client to send an OST_SETATTR or similar RPC for objects that were never modified that contains the parent FID and other identifiers, as is done with OST_WRITE and OST_SETATTR RPCs generated by chown, chgrp, lfs project, etc.

In order to retroactively update OST objects that are missing the parent FID, it would be possible to detect this during OST_GETATTR, OST_READ, OST_PUNCH and similar RPCs to update the trusted.fid with the parent FID information, if the OST is not mounted read-only.

This update could potentially schedule a workqueue task (outside of the current RPC processing) to avoid blocking the RPC processing for read-only RPC threads.  While the overhead will mostly be only the first time this functionality is deployed when these objects are accessed the first time, or in rare cases where the objects are allocated to a new file but no IO is done to them, any kind of journaled write operation on these threads could lead to significant RPC processing delays, which is undesirable.

--- Comment (JIRAUSER19288, Tue, 2 Sep 2025 10:28:58 +0000) ---
adilger under normal usage, this is not an issue. Nevertheless, there is still a narrow use case (outside normal operation) where the async object ownership updates can cause temporary inaccessibility of the object. I've verified this experimentally and will explain it for reference below:

	Under "normal usage", i.e., "chown" is used to change file ownership to users that are mapped on the same nodemap/tenant, this is not an issue because the ID check works in the context of the nodemap - not the user. Since all IDs are valid in this case (old UID/GID and new UID/GID), the ID check does not interfere even if there is an ID mismatch between MDT and OSTs.

	The problematic use case is when file ownership is changed for users between nodemaps/tenants. Since a user is tied to one nodemap, the ID check for chown operations between nodemaps would only succeed on one or the other. Note that this kind of "chown" can only be done from a "trusted,admin" client, so I think there is no use case where a cross-nodemap "chown" is immediately followed by I/O from the same user (since they can't break out of their nodemap). In summary, there are two potential cases where the ID check can fail because of an ID mismatch between OSTs and MDT. Consider both from the perspective of a user of nodemap A:
	
		A file is "chown"ed from nodemap A to nodemap B. For users from nodemap A, the ID check already fails on the MDT side at "open()" time. Therefore, denying access is instant, although the OST objects are not updated yet - unless the ID check is not enabled on the MDT side.

		A file is "chown"ed from nodemap B to nodemap A. For users from nodemap A, and in the short time frame where there is an ID mismatch, the ID check initially succeeds on the MDT side, but fails on the OST side later when accessing the data. This appears to be the reason sanity-sec/75a can fail - albeit very rarely since there is already some delay due to the setup and tests themselves.

	
	

For case 2, I think your suggested automatic RPC retry/delay is reasonable to improve usability. This would also work for case 1 if the MDT ID check is not enabled (for whatever reason), and it solves the rare failures on sanity-sec/75a.

--- Comment (adilger, Fri, 29 Aug 2025 17:32:10 +0000) ---
mvef, do you think there will be an issue in normal usage with chown on objects causing the files to be inaccessible due to async object ownership updates on the OST?  We've never been very strict about the OST object UID/GID/PROJID updates because this might only be a few seconds delay with updating the quotas.  However, with a strict ID check on the OSTs this could become problematic if there is a version mismatch.

I'm wondering if there needs to be some kind of automatic RPC delay/retry (e.g. return -EINPROGRESS to client for some seconds) in the rare case where the OST ID doesn't match.  Under normal circumstances this would be a no-op, but it is better to delay RPC completion for a few seconds instead of returning an error to the application if there is a race between chown on the MDS and when the OST ID is updated.

This could be implemented by storing a timestamp in the OST object in the case where an OST ID mismatch is detected, and then -EINPROGRESS is returned until the timer expires, or the timestamp is zeroed if a setattr arrives from the MDS or client to update the UID/GID/PROJID or an access with the correct ID(s) is received.
