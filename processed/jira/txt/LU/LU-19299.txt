--- Description ---
I noticed that the Trash Can is not fully cleaned between test executions. I often need to run the same test multiple times to verify my ideas, and leftover stubs remain after each run.

To address this, I added a cleanup on exit:

 

@@ -7902,6 +7904,7 @@ test_117q() {
trash_setup
mkdir $dir || error "failed to mkdir $dir"
stack_trap "$LFS trash clean -e -v -v $dir"
$LFS mkdir -i 0 -c 2 $striped_dir ||
error "failed to mkdir $striped_dir"

 

With this change, the logs show that Trash Can cleanup is attempted and files are removed, but the directory stubs are still left behind:

 

Phase 0/PR: remove dir 1:0x240000402:0x7f:0x0 from Trash Can 
Phase 2/PO: remove dir 1:0x240000402:0x7f:0x0 from Trash Can 
Skip stub dir 1:0x240000402:0x7f:0x0 
...
Phase 1/EN: remove file 2:f117p.sanityn from Trash Can
 

During the next test session, these stubs remain in the Trash Can even though the files are cleaned. This happens because Trash Can is disabled after each test case, and files or directories under $tdir and $tfile are removed directly, bypassing proper cleanup.

To avoid generating so many orphan stub pFIDs, the removal of $tdir and $tfile should be handled within the trash cleanup script. There is a plan to make a patch for this issue once full Trash Can support for striped directories is implemented.
