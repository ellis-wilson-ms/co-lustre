--- Description ---
The current Shared Secret Key implementation only allows a single key to be set for any nodemap.  However, this makes it complex to update the secret key and expire the old key because the key on all the clients and servers need to be updated at the same time.

It would be better to allow multiple keys to be active on a single nodemap at one time, so that a new key can be added on the servers, the key can be updated on the clients for their next remount/reconnect, and then the old key can be removed when all clients have been reconnected.

--- Comment (sebastien, Wed, 19 Mar 2025 08:56:48 +0000) ---
Indeed, we could modify lgss_sk in userspace so that it tweaks the descriptor of the key it loads (lgss_sk -l) if it detects a key with the same descriptor is already loaded. Using '.' makes sense as it is a forbidden char in nodemap names.

lgss_sk -l:
   description = "lustre:fsname:nodemap"
   key = keyctl_search(KEY_SPEC_USER_KEYRING, "user", description)
   idx = 1
   while (key exists) and (payload is not identical) do
      description = "description".idx
      key = keyctl_search(KEY_SPEC_USER_KEYRING, "user", description)
      idx++
   done
   if (key not exists) or (payload is not identical) then
      add_key("user", description, KEY_SPEC_USER_KEYRING)

Unfortunately, it is not possible to record the number of keys for a given nodemap in the nodemap struct, as the keys are loaded independently on each server. The MGS has dedicated keys if communications with it are using SSK, and OSS/MDS have a different set of keys that are loaded only if communications with ost or mdt targets are configured to use SSK. Moreover, the nodemap struct is in the kernel, whereas all the gss context negotiation happens in userland.

The lsvcgssd daemon, responsible for carrying out SSK authentication, should try with all the keys in the kernel keyring whose description is "lustre:fsname:nodemap" or starts with "lustre:fsname:nodemap." I think just searching for "lustre:fsname:nodemap.$idx" keys in a loop would be fragile to gaps in the sequence, depending on how admins inserted keys with lgss_sk -l.

lsvcgssd:
   description = "lustre:fsname:nodemap"
   for each key in KEY_SPEC_USER_KEYRING with desc starting with description do
      verify hmac with key
      if success then break
   done

--- Comment (sebastien, Tue, 18 Mar 2025 17:25:09 +0000) ---
Unfortunately, I now realize by doing more tests that it is not possible for multiple keys to be active on a single nodemap at one time. It is true that this is not the nodemap that references the key, but the key that references the nodemap, allowing several keys to be generated for the same nodemap at one time. But it is not possible to have several keys for the same nodemap loaded into the kernel keyring at the same time. Indeed, the key descriptor for our SSK keys has the following syntax:

lustre::

Consequently, 2 SSK keys generated for the same nodemap would have the same descriptor. As the key descriptor is a unique key identifier in the kernel keyring, used to search for keys, only one key with a given descriptor can be loaded at one time.

This means servers cannot load, and cannot support multiple SSK keys for the same nodemap simultaneously. The kernel keyring search mechanism implemented for Lustre is also a problem, as it calls keyctl_search with a key descriptor built with the nodemap name. Moreover keyctl_search can only return one key exactly matching the descriptor.

So today this is not possible to easily expire old keys and roll out new keys, unless clients and servers keys are updated at the same time. We would have to change the syntax for key descriptors used by Lustre, e.g. by adding something like a time stamp or a hash. And we would also have to change the SSK key retrieval mechanism, as searching for an exact descriptor match would not possible. For instance we would cycle through the keys in the KEY_SPEC_USER_KEYRING, and do the partial key descriptor matching by ourselves.
