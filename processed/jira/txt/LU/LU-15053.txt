Initially it is cancelling 4 lru locks. But during cancelling 2 of them caused to enqueue new locks to QMT. It happens because qsd_id_blocking_ast handling LDLM_BC_CANCELLING set qunit to 0 and edquot to false. If earlier qunit and edquot were set it might require an adjustment and sending new an update to QMT. To send these quota requests new ldlm locks are to be created and added to LRU list. Thus in some circumstances  removing 4 locks from LRU we have to insert 2 new ones.

This could be simply reproduced if set block quota limit and write some data right before the test:

@@ -2551,6 +2551,11 @@ test_13(){
        set_ost_qtype "u" || error "enable ost quota failed"
        quota_show_check b u $TSTUSR
 
+       lfs setquota -u quota_2usr -B30M /mnt/lustre
+       ./runas -u quota_2usr dd if=/dev/urandom of=$DIR/$tdir/f33 bs=1M count=15
+       ./runas -u quota_2usr dd if=/dev/urandom of=$DIR/$tdir/f334 bs=1M count=15
+       lfs quota -uv quota_2usr /mnt/lustre
+  

As could be seen from above logs the reason of failure is ID:500. Normally before each testcase in sanity-quota it is checked that there is no limits and files/data on the system. As I see from the test suite log uid 500 is a sanityusr UID:

Quota settings for sanityusr : 
Disk quotas for usr sanityusr (uid 500):  

I guess this happens because ENABLE_QUOTA is set up. It causes to set quota limits for different users - not only TSTUSR and TSTUSR2 used in sanity-quota. That's the reason of failure. The test case needs to be reworked to support lru_size > 0, i.e. to support users with quota limits and stored data in the same test sytem.
