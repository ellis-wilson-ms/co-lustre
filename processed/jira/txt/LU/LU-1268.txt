--- Comment (niu, Thu, 29 Mar 2012 09:55:24 +0000) ---
Hi, Chris

Indeed, that's the problem of mixed OSC name for same OST I mentioned in LU-1257. The root cause is that the registration code and conf_param code are using different naming style, registration is using new name, conf_param is able to detect it's an old system, and use old name. (I think this is a defect, registration code should use old naming style as well)

The OST0174 is registered twice, the second time (you changed it to LDD_F_VIRGIN manually), mgs found that this OST has already registered, so it turn to update the failover nids, that's why only OST0174 has an extra 'add failnid' entry. (in first time registeration, the failover nid is stored along with 'add osc', so no extra 'add failnid').

I don't know if there is any way to purge out the 'add failnid' entry only, but we can try writeconf for MDT and OST0174, and let them reigster again, thus, there will be no 'add failnid' in the MDT log.

However, the long term solution should be purging all the old config logs, and regenerate all of them in 2.0, otherwise, when you use 'lctl conf_param' on the new OST, the mixed OSC name problem will show up again.
