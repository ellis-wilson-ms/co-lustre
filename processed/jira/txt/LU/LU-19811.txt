--- Description ---
The `lnetctl net add --net  --if ` command may incorrectly select IP addresses when generating NIDs, leading to mismatched IP types.

 

Scenario 1: Interface with both IPv4 and IPv6 addresses

When an interface has both IPv4 and IPv6 addresses configured:

	`lnetctl net add --net tcp --if ens38` correctly adds the proper IPv4 NID

	However, `lnetctl net add --net tcp6 --if ens38` generates an incorrect IPv6 NID (should generate a valid IPv6 NID)

# ip addr show dev ens38
4: ens38:  mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 00:0c:29:dc:8c:00 brd ff:ff:ff:ff:ff:ff
    altname enp2s6
    inet 10.0.6.233/24 scope global ens38
       valid_lft forever preferred_lft forever
    inet6 2001::211:2233:4455:203/64 scope global
       valid_lft forever preferred_lft forever# lnetctl net add --net tcp6 --if ens38
# lnetctl net show
net:
-     net type: lo
      local NI(s):
      -     nid: 0@lo
            status: up
-     net type: tcp6
      local NI(s):
      -     nid: 10.0.6.233@tcp6
            status: up
            interfaces:
                  0: ens38 

The command generates `10.0.6.233@tcp6` (IPv4 address with tcp6 type) instead of a proper IPv6 NID like `2001::211:2233:4455:203@tcp6`.

 

Scenario 2: Interface with IPv4 address only

When an interface has only IPv4 address configured:

	`lnetctl net add --net tcp --if ens38` correctly adds the proper IPv4 NID

	However, `lnetctl net add --net tcp6 --if ens38` generates an incorrect IPv6 NID (should return an error)

# ip addr show dev ens38
4: ens38:  mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 00:0c:29:dc:8c:00 brd ff:ff:ff:ff:ff:ff
    altname enp2s6
    inet 10.0.6.233/24 scope global ens38
       valid_lft forever preferred_lft forever# lnetctl net add --net tcp6 --if ens38
# lnetctl net show
net:
-     net type: lo
      local NI(s):
      -     nid: 0@lo
            status: up
-     net type: tcp6
      local NI(s):
      -     nid: 10.0.6.233@tcp6
            status: up
            interfaces:
                  0: ens38 

The command generates `10.0.6.233@tcp6` instead of failing with an error (since no IPv6 address exists on the interface).

 

Scenario 3: Interface with IPv6 address only

When an interface has only IPv6 address configured:

	`lnetctl net add --net tcp6 --if ens38` correctly adds the proper IPv6 NID

	However, `lnetctl net add --net tcp --if ens38` generates an incorrect IPv4 NID (should return an error)

# ip addr show dev ens38
4: ens38:  mtu 1500 qdisc fq_codel state UP group default qlen 1000
    link/ether 00:0c:29:dc:8c:00 brd ff:ff:ff:ff:ff:ff
    altname enp2s6
    inet6 2001::211:2233:4455:203/64 scope global tentative
       valid_lft forever preferred_lft forever# lnetctl net add --net tcp4 --if ens38
# lnetctl net show
net:
-     net type: lo
      local NI(s):
      -     nid: 0@lo
            status: up
-     net type: tcp4
      local NI(s):
      -     nid: 2001::211:2233:4455:203@tcp4
            status: up
            interfaces:
                  0: ens38 

The command generates `2001::211:2233:4455:203@tcp4` instead of failing with an error (since no IPv4 address exists on the interface).

 

Expected Behavior

	Commands should select IP addresses matching the specified network type (IPv4 for tcp/tcp4, IPv6 for tcp6)

	Commands should fail with appropriate errors when no matching IP address type exists on the interface

--- Comment (JIRAUSER20209, Fri, 23 Jan 2026 07:46:03 +0000) ---
Apologies, but the described behavior is not actually an issue. There is no mapping relationship between IP version and net number. The command `lnetctl net add --net  --if ` will select the first IP address that matches the interface, regardless of IP version. Therefore, I will be closing this ticket.
