--- Comment (adilger, Wed, 26 Mar 2025 10:48:14 +0000) ---
It isn't clear that we would know if sumsq would overflow or not?  It would just be a random 64-bit value at that point.

I think printing the values as positive is likely correct, since I don't think there are any stats which are negative values.  However, the real question is whether some write RPC actually took 2 days to complete, or if there is some other bug?

It probably also makes sense to put "overflow detection" on these counters, like:

        new = old + incr;
        if (unlikely(new new = LONG_MAX;

or similar, so that it doesn't overflow.

--- Comment (adilger, Wed, 19 Mar 2025 18:38:54 +0000) ---
It looks like the Maximum completion time for some of the write operations are reporting an unreasonably large value (149845092510 usec = 1.7 days), so either there is an overflow with the counter for a single operation, or there was some kind of problem that was causing the thread to be blocked for this length of time before it could finish.  It would be worthwhile to check the server console to see if it reported stuck server threads and dumped stack traces. 

Doing the math 186442470 writes * max 149845092510 usec/write = -8955998982476203532 approximately matches the reported sum.
