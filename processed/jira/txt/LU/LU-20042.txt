look at llog_handle lh=ffff9b37d0772c00, while it is being used by process 6789, another process 107684 was doing garbage collection and delete this plain llog.  That is root cause of issue.

I tend to think that whole usage of chd_current_log taken via llog_cat_current_log() is flawed, because it is not taking reference for loghandle, so as result there can be several processes with llog_handle pointers which can point to freed handle potentially. It was assumed that chd_current_log() is always alive while there are records in it and is destroyed only when all gone. But this issue shows that no records doesn't mean there are no users of loghandle. Moreover, there is other places in code where chd_current_log() is cleared while llog is active, e.g.:

            up_write(&loghandle->lgh_lock);
            /* When ENOSPC happened no need to drop loghandle
             * a new one would be allocated anyway for next llog_add
             * so better to stay with the old.
             */
            if (rc != -ENOSPC) {
                /* nobody should be trying to use this llog */
                down_write(&cathandle->lgh_lock);
                if (cathandle->u.chd.chd_current_log ==
                    loghandle)
                    cathandle->u.chd.chd_current_log = NULL;
                list_del_init(&loghandle->u.phd.phd_entry);
                up_write(&cathandle->lgh_lock);
                llog_close(env, loghandle);
            }

which, I believe, is also not safe here.

Meanwhile, the only way for loghandle to be freed is last refcount put, so that is natural to get reference each time we get pointer for llog and it costs us nothing

--- Comment (tappro, Wed, 1 Apr 2026 17:21:26 +0000) ---
there is long standing flaw in llog_cat_add_rec() leading to error like below:

llog_write_rec()) lustre-MDD0000: loghandle ffff8fd249e47000 with no header

and shortly after to various kind of bad outcomes

Here we go:

llog_cat_add_rec():
    if (!llog_exist(loghandle)) {
        rc = llog_cat_new_log(env, cathandle, loghandle, th);
        if (rc lgh_hdr != NULL) {
        /* If llog object is remote and creation is failed, lgh_hdr
         * might be left over here, free it first */
        LASSERT(!llog_exist(loghandle));
        OBD_FREE_LARGE(loghandle->lgh_hdr, loghandle->lgh_hdr_size);
        loghandle->lgh_hdr = NULL;
    }
    ...
    rc = llog_create(env, loghandle, th);
    /* if llog is already created, no need to initialize it */
    if (rc == -EEXIST) {
        GOTO(out, rc = 0)
    ...

so if llog_create() returns -EEXIST then loghandle is left without lgh_hdr initialized and caller will hit this discrepancy sooner or later

I think llog_cat_new_log() should return -EEXIST to caller so it would close loghandle and get the real current one or probably that can be done inside llog_cat_new_log() function somehow
