--- Comment (adilger, Mon, 19 Jan 2026 17:44:05 +0000) ---
One option would be sending something similar to OBD_BRW_SOFT_SYNC to the MDS so that it will start a journal commit if the client is getting low on memory (e.g. 3/4 of maximum) due to many uncommitted open RPCs.  However, it isn't clear if that is the biggest issue if the MDS open request buffer is kept in memory for a long time after the open is committed.

The client should not be saving the full open RPC reply in memory until close, since that is wasting a lot of memory (and likely directly contributing to this OOM issue).  After the open commits, it is enough to save only the basic open RPC after the open commit, enough for it can be replayed for refcounting, but there is no value to save the full reply data for DoM and Readdir-on-Open (ROO?).  It would be even better to free the open replay RPC buffers completely and regenerate them at replay time, see Simplified Interoperability MDS_OPEN replay and LU-5703.

--- Comment (shadow, Mon, 19 Jan 2026 07:09:18 +0000) ---
I spent some time to understand a situation. It looks, problem isn't related to the patch itself but it's common for this ticket and DoM. Both operations uses an open reply to store a data for a page cache, but open request live in memory until committed or (after Andriy S. patch) until close committed to avoid problems with recovery. it mean DoM files/large directory might to hold an up 256k extra memory for the open request 

So we might needs to disable this feature for systems with low memory (just as arm with 4G ram where 0.4-0.5G eats kdump, where kdump isn't needs for kvm setup and kvm able to collect crash dump itself).

or introduce a shrink buffer after use for small systems once it needs a data copy with reply buffer realloc.

--- Comment (adilger, Thu, 21 Aug 2025 20:33:07 +0000) ---
Reopen this to finish off the development of this feature.

The landed patch doesn't have any test case to confirm that the feature is actually working.  Something simple would be good, at least to check that no RPC is sent for a small directory:

        (( MDS1_VERSION >= $(version_code v2_16_52-123-g4a14a51a8e) )) ||
                skip "need MDS >= 2.16.52.123 for dir_read_on_open"
        local old=($($LCTL get_param llite.*.dir_read_on_open))
        (( $old > 0 )) || {
                $LCTL set_param llite.*.dir_read_on_open=1
                stack_trap "$LCTL set_param llite.*.dir_read_on_open=$old"
        }
        test_mkdir $DIR/$tdir
        touch $DIR/$tdir/$tfile.{1..10}
        cancel_lru_locks mdc
        local before=$($LCTL get_param mdc.*.stats |
                        calc_stats mdc.*.stats mds_readdir)
        ls $DIR/$tdir
        local after=$($LCTL get_param mdc.*.stats |
                        calc_stats mdc.*.stats mds_readdir)
        (( before == after )) || error "readdir RPC for small directory read"

It also hard codes the number of pages returned in the MDS code, ignoring the `lm_bufsize` parameter supplied by the client.  It should be possible to dynamically tune the size of directory data returned to the client, in case this is needed in production for some reason.

Several options exist, and more than one might make sense:

	the (badly named) mdc.*.mdc_dom_min_repsize=SIZE parameter, which is also shared with DoM read-on-open, but is currently ignored by the MDS

	extend llite.*.dir_read_on_open=SIZE to accept the page count and/or size (like mdc.*.max_pages_per_rpc does) instead of just 0/1, so it can be tuned separately from mdc_dom_min_repsize and has a more obvious name to link it to reading the directory on open

	add a new mdt.*.dir_read_on_open=SIZE to limit clients reply size instead of hard coding PAGE_SIZE as the limit
