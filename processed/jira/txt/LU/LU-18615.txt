--- Description ---
Allow files/directories to be marked with a "hidden" attribute, so that it does not appear in directory listings, even when using "ls -al". Currently, ".lustre" is one such super-hidden directory. 

Use Case:

Super-users may need to create directories/files for maintenance purposes. If these are visible to regular users, it could potentially break job scripts that depend on specific directory contents or name entries. The super-hide feature allows administrators to create necessary files/directories without affecting user environments.

--- Comment (JIRAUSER19279, Thu, 9 Jan 2025 18:30:48 +0000) ---
We've explored the option of not adding a new extended attribute and we are able to use/add an 1-bit _LMAC_SUPER_HIDDEN flag in the lma_compact of lustre_mdt_attrs as you suggested. In more details we are considering the following approach.

	Reserve one bit (LMAC_SUPER_HIDDEN=0x00000080) in lma_compat of lustre_mdt_attrs (trusted.lma).

	Use a new req_format: struct req_format RQF_MDS_SUPER_HIDE to send req from client to MDS. This needs to reserve (1) a new opcode "MDS_SUPER_HIDE=63" in "enum mds_cmd", and (2) "#define OBD_FAIL_MDS_SUPER_HIDE_NET 0x174"

	We also considered re-using the existing req_format for MDS_REINT_SETXATTR and the handler mdt_reint_setxattr, in which case we'll still need to reserve {{ sx_padding_8}} flag (or instead, one bit flag of sx_valid) in struct mdt_rec_setxattr, in order that the handler knows whether a req is from lfs super_hide_path or from a regular setxattr call. We think this approach is less ideal because (1) the REINT_SETXATTR req_format is larger in bytes than the one we create above. and (2) this approach needs to modify the mdt_reint_setxattr function to allow changes to trusted.lma if the flag is set. We are not sure the risk of doing this.

	We'll also need to reserve "#define OBD_IOC_SUPEPR_HIDE _IOWR('f', 262, struct obd_ioctl_data)" for ioctl command.

As for the name ("super_hide" vs "hide"), we thought "hidden" somewhat refers to a file/dir with a name prefixed with "." and "super_hidde" hides from directory listing. However, we are open to change the name. Some choices include:

	lfs hide_path [path]

	lfs hide [path]

	lfs super_hide_path [path]

	lfs super_hide [path]

Currently in our draft impl such files/dirs are not viewable (in terms of ls -a) by either root or a regular users. Not sure the impl path if we want the root to view them.

--- Comment (adilger, Tue, 7 Jan 2025 23:50:42 +0000) ---
Adding a whole xattr on disk (at least 40 bytes of inode space) and 8 bits in the protocol for essentially a single bit of information is wasteful.

There are already a number of bit flags that could potentially be used for this.  The on-disk ext4 and ZFS have attribute flags that can be set/checked/listed with chattr, lsattr and might be usable (EXT4_*FL or FS*FL, though I don't see a "*_HIDDEN_FL" defined).  Alternately, there is a Lustre-specific LMAC_FL* flags word in the lustre_mdt_attrs stored on each inode that can store a 1-bit flag without a full xattr just for this attribute.

The drawback of such an attribute is that it exposes a vector for viruses or malicious users to hide files from view.  Also, if files have UID/GID/PROJID of a regular user but are hidden from view, then they will appear in quota usage (and consume space in general) and users are perennially complaining if "du -sk does not match lfs quota -u" in some directory tree because they have hidden/open files, or quota IDs are being used in different directories, and this would only make it worse.

What are the intended uses for "super hidden files"?  Whether it is worthwhile to implement such a feature depends on the benefits/needs.

--- Comment (gerrit, Tue, 7 Jan 2025 22:18:42 +0000) ---
We propose adding an "lfs super_hide_path [path]" to extend the "super-hidden" feature. This will allow super-users to hide arbitrary paths from directory listings.

Key Points:

	The "lfs super_hide_path [path]" command will add and set "trusted.super_hidden" xattr to "1" for specified path.

	Modify "mdd_dir_page_build" function to check "trusted.super_hidden" before returning the dir page.

	For performance, only check xattr for hidden files/dirs (names prefixed with ".")

	Restrict modifications to "trusted.super_hidden" to the new command only. e.g. A direct "setfattr" system call to "trusted.super_hidden" should be ignored to prevent accidental modifications.
	
		This requires both the "mdc_xattr_common" function (in "lustre/mdc/mdc_request.c") and the "mdt_reint_setxattr" function (in "lustre/mdt/mdt_xattr.c") know whether the request is from "lfs super_hide_path".

		For the above purpose, we would like to reserve the currently-unused "__u32 sx_padding_8" integer in "struct mdt_rec_setxattr" and rename it to "sx_skip_super_hidden_check" as a new flag. The two aforementioned functions know the source of the req by checking the flag.

		"mdt_reint_setxattr" ignores xattr change reqs unless the flag is set to false.

	
	

// pseudo code
>if (!sx_skip_super_hidden_check) // flag is false. i.e. req is from setfattr not lfs super_hide_path
    if (xattrname=="trusted.super_hidden")
        ignore_this_change_xattr_req();

 

Plan:

We will work on the following two patches:

	A minimal patch to reserve the "sx_padding_8" int of "struct mdt_rec_setxattr" and rename it to "sx_skip_super_hidden_check".

	A second patch that implements "lfs super_hide_path".
