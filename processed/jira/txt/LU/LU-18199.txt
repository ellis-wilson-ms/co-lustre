--- Comment (ssmirnov, Fri, 6 Sep 2024 18:09:15 +0000) ---
pjones yes, we can do that if we are able to say that if going across subnets is involved then it is the user's responsibility to set up routing.

I don't think there's much harm in doing so because EXA installations, for example, already disable ksocklnd-config and set up own routing.

Another option is to change the default for skip_mr_route_setup to 1 (arguably that should have been the default from the start). This would make the choice to use ksocklnd-config  conscious on the user's part so that the user can enable ksocklnd-config and keep it if it works them. The disadvantage of this decision is that what I believe to be the majority of users with simpler routing setup for whom the script works may need to go through additional steps when installing lustre.

It is important to make sure the traffic flows as LNet intends, but in practice trying to do in an automated way keeps stumbling into special cases which need special handling.

That said I'd still like to understand and fix the issue Sebastian is seeing with the script.

--- Comment (ssmirnov, Thu, 5 Sep 2024 21:11:21 +0000) ---
I think there are two different problems. 

1) The script flushes the routing tables for the interface. In theory, this can cause loss of connectivity (although I haven't seen it with 3.x and 4.x kernel VMs in my environment, yet to try 5.x). It makes sense that if settings are applied correctly, then the connectivity can be restored. It is not clear to me why it would require a reboot or an interface restart though. Either way, flushing appears to be redundant because the script is trying to add routes only if there aren't any routes defined for the interface already.

2) In Cyril's case, there's also an overlap between "10.0.0.0/16 dev eth1" and "10.0.2.0/24 dev eth0", so when selecting which gateway to use, the script makes the mistake of picking up the eth0 gateway for eth1. Although I'd probably not recommend overlapping configurations, the script still should be able to avoid making the mistake by checking the "dev" parameter of the default route.

I attached an updated ksocklnd-config script which is supposed to have both 1 and 2 fixed. Could you please give it a try in your environment?

Thanks,

Serguei.
