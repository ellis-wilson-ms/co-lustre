--- Description ---
Use "lfs mirror resync" command to calculate and write parity code asynchronously into parity component(s) after reading the data of the corresponding data components.

We also should add a verify command:

lfs ec verify — reads back and validates parity data against the source data.

Both commands will use the designated EC read/write functionality to perform I/O to the parity components.

We need to consider the relationship between erasure coding and mirrors here.  Probably, the resync operation can annotate which components it actually resync'ed and that should take care of it.  Since a write to a mirror + EC file will mark all but one mirror STALE, then "lfs mirror resync" (synonymous with "lfs ec resync", if we have such a command) should resync all STALE mirror and EC components in the file.  There are already options to only resync specific components that can be used in unusual circumstances.

--- Comment (paf0186, Tue, 4 Nov 2025 21:04:40 +0000) ---
Right, I'm not so concerned with the verify operation itself, I'm actually concerned with being able to test our ability to regenerate data for individual stripes (and individual parity data stripes).  It might even be we want to do this with a dedicated test utility rather than shoehorn it into lfs, in which case no new options are needed.

--- Comment (adilger, Tue, 4 Nov 2025 19:31:40 +0000) ---
Do we need a separate "verify_ec_stripe" command?  I think this would be better as new options to "lfs mirror verify", if needed.  This already allows verifying a specific mirror, if needed.  From a user POV, they want to just verify the redundancy of a file is intact, and not know the low-levell details of the layout.

Note that it is possible in real life that there is a sector error on one of the OST data stripes, so if there is a EC mismatch when regenerating the parity, it would be worthwhile to try and regenerate each stripe's data chunk (one at a time) with both parities.  If they both generate the same data that does not match the stored data, then it is the data OST that is incorrect.

It may be that in the future we could leverage the compression header to store a data checksum for each chunk, to help detect the chunk errors independently to simplify the EC/data reconstruction, but it should be possible to do this independently (with some extra computation).

--- Comment (paf0186, Tue, 4 Nov 2025 18:55:09 +0000) ---
I've been thinking about how mirror verify works and what the equivalent is for verify of an erasure-coded layout. Obviously, the mirror verify simply checks that all non-stale mirrors contain the same data. I think the only obvious corresponding functionality for erasure code verify is calculating the parity again and confirming that it matches the existing parity.

Because fundamentally the purpose is to verify the integrity of the existing setup, not to let us test our ability to regenerate missing data.

That said, we clearly need a way to test the regeneration of missing data, and I would like to be able to do it in user space because:

	It lets us give this code a workout before kernel-side degraded reads are ready.

	It can serve as example code for implementing the calculation part of degraded reads in the kernel client.

So I'm thinking something like a special command which allows us to do verify for a single stripe of a data component.

It's a little clunky, but it would let us test things.

So it’d be something like you would specify a component ID — precisely identifying a data component — and then you would specify a stripe by ordinal, e.g., the 3rd stripe or 19th stripe in that component. Then the utility would regenerate just that stripe of that component and compare it to the data for that stripe read in the normal manner.

Something like:

lfs mirror verify_ec_stripe -c [COMPONENT_ID] -i [STRIPE_INDEX_IN_COMPONENT]

It would use the binding between the data component and the parity component, where the parity component stores the mirror ID of the data component, to find the parity component (since we want to test, in this case, a single specific EC).

This would be purely for testing.

Thoughts?

--- Comment (adilger, Fri, 30 May 2025 07:18:59 +0000) ---
As part of the resync and/or recovery, it is would be useful to be able to recover the size of the file based only on the data and parity stripes, even if the OST stripe holding the highest offset byte is lost. With FLR, the SOM xattr will cache the file size and blocks on the MDT inode, but it would add a layer of redundancy to also encode the file size into the EC stripes in some manner.

If the EC component is sized exactly with the number of parity bytes written (not rounded to the next stripe boundary), then it would be possible to use this to compute the file size within M bytes of the actual size (rounded up). It would be further possible to mostly determine the file size by subtracting up to M-1 zero bytes in the last stripe, but this is not 100% robust if the file actually has zeroes at the end of the file.  Having multiple parity stripes doesn't really help, since the number of parity bytes written to each stripe would be the same.
