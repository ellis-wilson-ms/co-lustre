--- Comment (adilger, Wed, 23 Oct 2024 03:09:46 +0000) ---
This was a test failure triggered by an early version of patch https://review.whamcloud.com/55311 which had a bug, but it is also triggered easily by other patches that are changing the LDLM LRU behavior, and exposes a weakness in the test and/or the statahead code.

It is checking that the number of cached inodes matched by statahead is large enough, but if the lru_size is constrained for some reason then the client cannot cache all of the inodes fetched by statahead before they are evicted by the LRU.

It probably makes sense to stop aggressive statahead if the fetched attributes are being dropped before they can be used.  If that is already happening, then possibly the test should check if the number of DLM locks is enough to cache the number of files being accessed.

I'm adding a call to lru_resize_enable() at the start of test_123a_base() to ensure that dynamic LRU size is enabled, which was causing it to fail on earlier versions of patch https://review.whamcloud.com/56642 "LU-18288 tests: lru_resize_disable sets lru_max_age" but possibly more needs to be done if it continues to fail in the future.
