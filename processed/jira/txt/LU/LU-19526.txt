--- Description ---
reading iov_iter_is_aligned() I noticed the following code:

static inline bool iov_iter_is_aligned(const struct iov_iter *i,
				       unsigned addr_mask, unsigned len_mask)
{
	if (likely(iter_is_ubuf(i))) {
		if (i->count & len_mask)
			return false;
		if ((unsigned long)(iter_iov(i) + i->iov_offset) & addr_mask)
			return false;

notice iter_iov returns struct iovec *, not address of the data.

--- Comment (stancheff, Tue, 28 Oct 2025 03:34:28 +0000) ---
Ah .. it passed the compiler but is kind of non-sense.

So the 'iter_is_ubuf()' and the iov_iter member ubuf arrived in the same commit:

   v5.19-10287-gfcb14cb1bdac 

And 'iov: introduce iov_iter_aligned()' arrived around the same time:

   v5.19-rc4-8-gcfa320f72882

So we have probably never seen a case where iter_is_ubuf() arrived without iov_iter_aligned() and the code in question has never been executed.

I think the best fix is provide an #define accessor for iter->ubuf() to satisfy the compiler so the code is sensible.

So it should have been:

	if (likely(iter_is_ubuf(i))) {
		if (i->count & len_mask)
			return false;
		if ((unsigned long)(i->ubuf + i->iov_offset) & addr_mask)
			return false;
		return true;
	}

But ubuf does not exist so the compiler chokes, so instead:

#ifdef HAVE_USER_BACKED_ITER
#define iter_ubuf(iter)			((iter)->ubuf)
#else
#define iter_ubuf(iter)			(0ul) /* unused */
#endif /* HAVE_USER_BACKED_ITER */

	if (likely(iter_is_ubuf(i))) {
		if (i->count & len_mask)
			return false;
		if ((unsigned long)(iter_ubuf(i) + i->iov_offset) & addr_mask)
			return false;
		return true;
	}

Using ((iter)>__iov>iov_base) would be the closest approximation to ubuf here but there really isn't any need since ubuf and iter_is_ubuf() are synonymous here.
