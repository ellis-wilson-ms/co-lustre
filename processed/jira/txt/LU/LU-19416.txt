--- Comment (adilger, Thu, 25 Sep 2025 17:53:19 +0000) ---
I'm of mixed opinion on this.

One alternative would be to make the name of the virtual .Trash directory configurable (e.g. tunable llite.*.trash_can_name parameter), to avoid this potential conflict, but I'm not sure if that is needed yet.

MacOS is using ".Trash" for this functionality (one directory per user), and in theory if Lustre was exported to a Mac (NFS or SMB) then MacOS might start trying to rename files into the .Trash directory when deleted via Finder.  We could potentially allow that, and treat it the same way as unlink of the file (i.e. set UNRM flag, create trusted.unrm xattr to hold original UID/GID/PROJID and set dtime).

It would be more tricky if a whole directory is being "deleted" in this way, but potentially a thread on the MDS could recursively process the whole directory, or we return -EPERM in that case?  Returning -EXDEV would cause rename() to try and create a new target directory tree under .Trash and then move the files individually into there.  Maybe good, maybe bad to allow mkdir() in the .Trash directory?
