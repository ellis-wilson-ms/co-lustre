--- Description ---
If "lnetctl net add" is used to create an lnet with the "ip2net" option, then tunables settings in the same command are ignored:

lnetctl net add --ip2net "tcp(eth0) 192.168.122.60" --credits=100

# lnetctl net show -v 4
net:
...
-     net type: tcp
      local NI(s):
      -     nid: 192.168.122.60@tcp
            status: up
            interfaces:
                  0: eth0
...
            tunables:
                  peer_timeout: 180
                  peer_credits: 8
                  peer_buffer_credits: 0
                  credits: 256
            lnd tunables:
                  conns_per_peer: 1
                  timeout: 49
                  tos: -1
            dev cpt: -1
            CPT: "[0,1]"
 

If "net" and "if" options are used instead, tunable settings get applied properly:

lnetctl net add --net tcp --if eth0 --credits 100

# lnetctl net show -v 4
net:
...
-     net type: tcp
      local NI(s):
      -     nid: 192.168.122.60@tcp
            status: up
            interfaces:
                  0: eth0
...
            tunables:
                  peer_timeout: 180
                  peer_credits: 8
                  peer_buffer_credits: 0
                  credits: 100
            lnd tunables:
                  conns_per_peer: 1
                  timeout: 49
                  tos: -1
            dev cpt: -1
            CPT: "[0,1]

--- Comment (ssmirnov, Tue, 18 Jun 2024 17:06:23 +0000) ---
The issue can manifest itself in two ways and there are two separate causes:

	If Netlink is used for communication between the user-space and the kernel, then it appears that the design prevents tunables from being added to the message

	If Netlink is detected to be not suppored the "old API" is used, and in that case there's an alignment problem caused by wrong padding in "struct lnet_ioctl_config_ni"
