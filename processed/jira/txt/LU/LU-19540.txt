--- Description ---
For distributed load balance of client-side LMV operations such as df/statfs() (lmv_select_statfs_mdt()) and DNE3 MDT mkdir balancing (lmv_qos_rr_index), the clients currently use the hash/modulo of their first NID to provide a decentralized mechanism to distribute clients across all configured MDTs.

In general this mechanism works reasonably well, but in some cases can produce pathalogical results (e.g. if each client has 8 network interfaces and the interface assigned a Lustre NID are all multiples of 8).

To combat such aliasing issues, one option would be to assign each new client connection a sequential integer from MDT0000 to provide a more uniformly distributed index number that can be used to distribute the client RPCs across MDTs (or other items).  While this may suffer from other types of aliasing (e.g. if the first node from each rack is assigned to a job), it is likely to provide more uniform distribution than purely the NID value.

--- Comment (adilger, Wed, 29 Oct 2025 23:24:38 +0000) ---
I'm not proposing to use the "client index number" with a hash, but rather as a sequence with modulo (number of targets), so that in common use cases there is a good uniform round-robin distribution of clients across targets.  That still won't be perfect (due to clients restarting and getting new index numbers), but statistically it should still be more uniform vs. a "random" value since every N client additions will cover a range of N different target numbers.

Since the "client index number" will not be "handed out" modulo the target count, just a sequential integer, then it should maintain it uniform distribution across MDTs (or whatever) even if the MDT count is changed.

For mkdir, the lmv_qos_rr_index is incremented on each use, so even if there are fewer clients than MDTs it would still advance across all MDTs in use.  This might cause some short-term imbalance if clients are used unevenly, but should still maintain long-term even distribution (regardless of what is used to determine the starting index).

For statfs, the lmv_statfs_start remains static, so it may be that only num_clients MDTs would be used for statfs(), but that would always be true no matter which subset of MDTs are selected, and statfs() leaves no lasting impact on a server.
