--- Description ---
A number of sites using containers with Lustre have requested the ability to dynamically configure nodemaps on a regular basis to handle environments where they are being created and removed on a e.g. a per-job basis to contain only the nodes in the job for a specific runtime environment.

Storing all of the nodemaps in the config llog with frequent updates would quickly consume all of the space in the config log, and be slow to process during mounting.

Instead, there should be some option to allow using "lctl set_param nodemap.NODEMAP.PARAM" or an option to "lctl nodemap" to directly create and manage nodemaps without using the config llog to distribute the parameters to each server.  This would require some external orchestration to set the nodemap parameters consistently across servers (e.g. "clush -a lctl set_param ...").

It might be desirable to have these dynamic nodemaps inherit settings from an existing persistent nodemap that would otherwise catch their NID range.  This would have a number of benefits:

	safety in case of server restart so that missing dynamic nodemaps do not result in clients in the default nodemap

	avoid the need to set every parameter for the new nodemap, only change the subdirectory and NIDs and possibly a few other parameters

	(optionally?) make the dynamic nodemaps "safe by default" in that they can only add further restrictions rather than weaken them (e.g. longer subdirectory path, only disable access permissions, only remove UID mappings from the parent).  Not totally sure about this, but seems interesting.

For identifying at mount time which nodemap is used, this could be hierarchical.  Find the top-level NID range as today and map it to a nodemap, and consider this the "parent" nodemap.  Then check if there is a second-level NID range below the parent and re-scan for matching NIDs and add extra restrictions from the child nodemap.  Possibly repeat (max 10 levels for safety?).

That avoids making the initial NID searching complex (broad NID ranges can be used for the whole cluster) and then become more finegrained only if needed.

--- Comment (adilger, Thu, 24 Oct 2024 08:00:21 +0000) ---
Sebastien,

this may be a dumb question, but I'm wondering whether dynamic (non-persistent) nodemaps even necessary?  I had previously thought that the nodemap code was using a configuration llog to store the nodemap information, which would use fixed-size records.

However, after working on the nodemap offset patch I now understand that the nodemap configuration records are stored in an OSD index object (IAM on ldiskfs, ZAP on ZFS, like the OI files on the MDT), and these are dynamically sized/managed and can grow quite large without running out of space.  Also, the OSD index objects have the added benefit that deleted entries in the index free up space that can be re-used when new entries are added to the index.  While this is not necessarily perfect, since it depends on the hash distribution and how the leaf buckets are split, the index can definitely withstand a large number of insert/delete cycles.  The same OSD index objects are used to store hundreds of millions of FID->inode mappings.

If there are not similar limitations on the OSD index as there are in the llog files for continually adding and removing nodemap config records, then it isn't clear if we the dynamic nodemaps should only be stored in memory, but should instead always be stored on disk?  That avoids one of the major issues with dynamic nodemaps - namely that they are lost if the MDS crashes.

I did a quick experiment and added 1000 nodemaps to a filesystem, deleted them all, and then added 1000 different nodemaps to the filesystem.  For both cases, the CONFIGS/nodemap file was about 64KB (slightly larger in the second case), so I think this lends support to the idea that the persistent nodemap config is not as limited as we thought.

At this point, the main scaling limitation appears to be from saving the pathname for the fileset into the "params" config llog used by "lctl set_param -P".  That definitely does have cycle limitations, though we could wonder whether the fileset name would change very frequently, or mostly only the NIDs assigned to that nodemap?

--- Comment (sebastien, Thu, 21 Mar 2024 17:32:04 +0000) ---
I have pushed first patches to implement dynamic nodemaps. I have divided into multiple small patches to make reviews easy. For now it does not handle the hierarchical nodemaps, it just allows to create and define properties directly on MDS and OSS nodes, and take these dynamic nodemaps into account when clients are connecting.

One limitation I have identified so far is that dynamic nodemaps are lost when any nodemap change or command is done on the MGS. This is because when a nodemap is modified on MGS side, it forces the other servers to refresh the entire nodemap config. After this refresh, it seems complicated to keep the dynamic nodemaps (i.e. automatically without re-running all the 'lctl nodemap_' commands on MDS/OSS side). By definition, the dynamic nodemaps are sitting on top of the legacy nodemaps, so if they change the dynamic nodemaps need to be rebuilt.

I now need to figure out how to implement the NID range hierarchy for the dynamic nodemaps. The NID ranges are currently managed as a red-black tree, so we cannot really tamper with that. But there is a single tree today, with a common root, so maybe it would be possible to implement the sub nid ranges as new r-b trees, attached to the nid range they are a subset of (their parent range).
