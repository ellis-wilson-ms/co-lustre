--- Comment (adilger, Wed, 17 Sep 2025 22:29:19 +0000) ---
Tim, it is clear from looking at the patch again that this "peer metadata" functionality is broken for IPv6 NIDs needs at a minimum some code changes to handle this properly, if not protocol change to work with large NIDs.

My understanding was that the "fetch metadata" functionality in LU-18808 was supposed to be generic enough that other LNDs can use it, and sanity-lnet test_270 was added with that expectation that this is working properly for TCP.  This isn't just a question of whether the current EFA LND does not need the large NID case, but whether EFA or some other LND would use this with large NIDs and it will be broken because the larger NID is clobbering some of the other data in the reply.

I now see in the patch that this network protocol message was added with a small NID at the start instead of reserving the few extra bytes necessary to allow a large NID:

struct lnet_nid_md_entry {
	lnet_nid_t			nid;
	__u8				buffer[LNET_MD_BUFFER_SZ];
} __attribute__((packed));

This seems like a terrible shortcoming at this point to be adding something that only allows IPv4 NIDs when we are working to enable IPv6 for upstream acceptance.

IFF you are already using this feature in production with the current message format, then it makes the most sense sense to change nid to be a union between struct lnet_nid and lnet_nid_t and then interpret struct lnet_nid.nid_size to dynamically determine the actual NID size and accesses the extra metadata in buffer with the correct offset.

What I don't understand is why the large NID is affecting num_nid_mappings instead of cutting into buffer in each of the struct lnet_nid_md_entry entries?

NB: please add lnme_ prefixes for these struct fields.

--- Comment (JIRAUSER18433, Tue, 16 Sep 2025 21:24:30 +0000) ---
This is an expected failure. Currently, there's only a need to attach metadata to small NIDs (for EFA small NID -> GID resolution - see https://review.whamcloud.com/c/fs/lustre-release/+/58407/comment/44f55e24_c944dbb3/). It's not difficult to attach metadata to large NIDs - but there isn't any use case for it right now - so there's no point.

 

The test is failure specifically because we are attempting to fetch metadata from a peer that doesn't have any (since it only has large NIDs). In the real world, this would never happen. The EFA LND will only call lnet_discover_nid_metadata() on peer that it definitely knows has metadata (i.e. it has a small EFA NID).

 

I can write a patch to skip this test in IPv6-only environment.
