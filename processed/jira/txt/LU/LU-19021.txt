--- Description ---
Under certain conditions, lctl get_param nodemap..exports reports multiple entries for seemingly the same export and UUID:

nodemap.tenant.exports=
[
 { nid: 192.155.155.50@tcp, uuid: 81d25e02-14c0-46fa-9054-590b7949ecb8 },
 { nid: 192.155.155.50@tcp, uuid: 81d25e02-14c0-46fa-9054-590b7949ecb8 },
 { nid: 192.155.155.50@tcp, uuid: 81d25e02-14c0-46fa-9054-590b7949ecb8 },
 { nid: 192.155.155.50@tcp, uuid: 81d25e02-14c0-46fa-9054-590b7949ecb8 }
]

This can occur for multiple MDTs or multiple OSTs because each *obd_connect() (for MDT and OST) adds the export as a member to the corresponding nodemap. This is expected, but there is no duplication check when printing the exports of anodemap. Therefore, several exports with the same UUID can be printed when listing them with lctl get_param.

--- Comment (JIRAUSER19288, Mon, 26 May 2025 16:38:22 +0000) ---
adilger It is the former case: All client connections share the same nodemap structure with multiple exports from the same connected client. 

Specifically, the nodemap struct uses the nm_member_list field where each entry is exp->exp_target_data.ted_nodemap_member. Also, the export's exp->exp_target_data.ted_nodemap is assigned this nodemap so that the nodemap can be retrieved later from the export through nodemap_get_from_exp(). This is why the issue in this ticket appeared, because lctl get_param would simply iterate over that list. But with the output, it wasn't clear that the connections were actually different.

However, the main challenge to limiting client connections in the nodemap is that idle clients disconnect after some time. I have seen this for client-OST connections, but not yet for client-MDT connections. This means the number of connected clients is likely inaccurate at any given time because an active client mount is not necessarily tracked in the nodemap. 

Since limiting clients can be done on the MDS side (assuming the connections are not transient for client-MDT), there are a few options. In principle, having a dataset with unique exports (based on the UUID) would be enough to be able to limit client connects as they can just be refused in nodemap_add_member(). 

Some options on how this can be done:

	Use a separate data structure in the nodemap structure to count unique exports (based on the UUID), preferably with O(1) lookup time since this needs to be done for each client connection.

	Use some form of associative array/hash table or similar instead of a nm_member_list where each entry is a list of connections for the same client.

--- Comment (adilger, Sat, 24 May 2025 20:52:55 +0000) ---
Related to this, a question came up about limiting the number of concurrently connected clients. I was wondering if client connections to a nodemap are sharing a single nodemap structure per server (possibly with multiple exports from the same client connected at the same time), or if the nodemap is separate (but largely identical) structure for each target?

Inthe first case (multiple exports sharing common nodemap) it would be difficult to limit connected clients, but if the nodemap is per target then it would be relatively straight forward to put an upper limit on the connection count. Presumably server exports would go to a separate nodemap (unless only the default nodemap is in use), so the nodemap exports would only be from clients.
