Investigation findings:

All lustre-master pipeline failures are interop

Every failing session on lustre-master is interop: master el10 client + b2_16 server (SLES 15 SP6 or Ubuntu 24.04). There are no master-on-master failures for this bug on the lustre-master pipeline. The lustre-reviews failures are from broken "custom" test sessions (116 failures across the board) and are likely environmental.

Two distinct failure modes

1. changelog_mask=+hsm set_param fails (ENOENT, rc=2)

Tests: sanity-sec test_35, sanity test_64e

Error: lustre-MDT0000: changelog_mask=+hsm failed: 2

This is a do_facet operation (runs locally on the b2_16 MDS via SSH). lctl set_param mdd.lustre-MDT0000.changelog_mask=+hsm returns ENOENT. This suggests the mdd changelog parameter path doesn't exist on the b2_16 server — possibly a b2_16-specific initialization issue on SLES 15 SP6/Ubuntu 24.04.

2. Changelog llog access fails

Tests: sanity test_60i ("can't read changelog"), sanity-hsm test_220A/220a/221/222a-d ("expected changelog was not emitted"), lustre-rsync-test test_1A/1a/2a/2c/3a

This is the llog client→server RPC access issue covered by LU-19750.

Root cause: LU-19750

The core issue is mdt_changelog_allow() in lustre/mdt/mdt_internal.h. On current master (2.17.50), since LUSTRE_VERSION_CODE >= OBD_OCD_VERSION(2, 17, 3, 0), the function returns false when mti_body is NULL — denying changelog access.

Timeline:

	Jan 7: 815dfd3451 merged (client fix: send mdt_body in llog open for nameless llogs)

	Feb 5: 1c0d9f16e4 merged (server fix: relax version check)

	Feb 5: 183eb9af3f merged (REVERT of server fix — caused 100% LBUG crash in sanity-sec test_35)

Current state: the client-side fix is applied (master clients send mdt_body), but the server-side fix is reverted. This means:

	Master client → master server: works (client sends mdt_body, server accepts it)

	Master client → b2_16 server: the new request format from the client may not be handled correctly by older servers

	b2_16 client → master server: client doesn't send mdt_body, master server denies access

This ticket may be a sub-issue of LU-19750. The changelog_mask set_param ENOENT on b2_16 servers needs separate investigation with access to a b2_16 test environment.
