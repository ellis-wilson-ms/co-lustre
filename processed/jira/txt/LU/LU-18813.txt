--- Description ---
Implement an memory-based OSD device with writeback support for Lustre.

It borrows lots of design from memory-based file systems such as tmpfs/ramfs.

The data is frist written into memory-based file system (called MemFS in short).

And then, the data can be flushed to the persistent storage in a delayed write-back manner.

Thanks for your review and testing.

There are many shared codes between MetaWBC, OSD-wbcfs and tmpfs/ramfs.

However, there are some places different.

For example, MetaWBC use Lustre super block/inode/dentry for MemFS, OSD-wbcfs has its own suerblock/inode/dentry and we have extend the @dir_context to get FID of a file during readdir().

All these need to be abstracted.

--- Comment (qian_wc, Fri, 14 Mar 2025 15:22:27 +0000) ---
In LU-17995, the in-mem OSD does not use any FS concept, the files and data are managed by some in-memory data structure such as list, rhashtable, and page array (for data), etc, al.

 

However, the design of memory FS based OSD is borrowed lots of idea from memory-based file systems such as ramfs/tmpfs. And lots of VFS code in Linux kernel can be reused in this design, such as:

	Page caching management;

	dcache

	icache

	Linux kernel write back codes
