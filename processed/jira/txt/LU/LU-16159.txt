--- Description ---
Once recovery is aborted, the existing update logs should be removed, because they are used for recovery only, and if it's corrupt, or inaccessible, if they are kept there after recovery abort, the next recovery will meet with the same issue again, besides, the log file size may become large, retrieving them in recovery may lead to recovery timeout.

Besides this, in LOD device initialization, it should do some sanity check for all update logs on it, if the log file is not accessible (e.g. OI mapping mismatch, which may lead to recovery timeout too), delete the log file FID from update catalog (do not delete this file at the moment because it's inaccessible) so that the log file is not visible to others.

--- Comment (adilger, Tue, 17 Feb 2026 21:48:14 +0000) ---
If the Lustre code is deleting the llog name/reference, but not the llog file itself, then this is a llog/osd-ldiskfs issue and not an e2fsck issue.  I do not expect that an llog file should have a link xattr, because that is for namespace back references to the parent directory.

--- Comment (nangelinas, Tue, 17 Feb 2026 18:16:43 +0000) ---
We have seen an issue with e2fck reporting update logs as having unattached inodes and @peggy suggested that this might be due to this patch. Since it removes the names from update logs when recovery is aborted, if e2fsck is ran before the inodes are removed (iirc, they are removed when they are emptied, or if 30 days pass), it will consider them as unattached and move them to lost+found.

We'll try to verify this, but is this expected behavior? If not, I'm not sure if the fix should be in Lustre or in e2fsck, I need to look into this a bit more, but does anyone have any suggestions?
