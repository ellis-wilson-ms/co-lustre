--- Comment (ssmirnov, Tue, 24 Sep 2024 15:40:14 +0000) ---
xiyanthis probably should have been covered under LU-17689. 

Here's what I'm thinking:

It looks like connections in INIT state can ignore RDMA_CM_EVENT_REJECTED - because they haven't "asked" to connect yet. If these do successfully change to either ACTIVE_CONNECT or PASSIVE_WAIT and get rejected again, they can handle it then. 

It looks like ESTABLISHED connections can ignore RDMA_CM_EVENT_REJECTED because they are already established. If something goes wrong, they will timeout on an actual tx and get closed and cleaned up because of that. 

It also looks like CLOSING connections can ignore this event because they are already in the process of being closed and DISCONNECTED ones because they already have been disconnected.

This covers all conn states, so if the above is correct, then RDMA_CM_EVENT_REJECTED can be ignored rather than treated as an LBUG. So we could just print a warning which would state what happened (which unexpected event received in what conn state) and allow the flow to continue.
