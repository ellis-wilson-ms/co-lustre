--- Comment (adilger, Wed, 4 Sep 2024 09:11:56 +0000) ---
On the one hand, if an EA inode is accessed directly then it means some kind of filesystem corruption is present.  On the other hand, one bad inode should not trigger the filesystem to go read-only, especially not in a loop that makes the whole server unusable.  While this is unlikely to happen (just a bit flip, or e2fsck linking an orphan inode there?), I'd rather make the code more robust rather than try to avoid this situation.

Some other options might include:

	unlinking the orphan EA inode from lost+found if it is found there, since it should really be attached to the inode.  might be tricky, especially if e2fsck just links it back, unless e2fsck is finding the original inode and relinking it there, but it may have already tried this...

	pass some "side" flag around lookup_one_len() to ldiskfs to indicate that this is OI Scrub running and it shouldn't complain about this case (which might also simplify the original patch https://review.whamcloud.com/55896 "LU-18080 ldiskfs: add NO_CHECKS flags for ext4_iget()".  I thought we already did something similar for OI Scrub in the past because of a similar patch, but I don't recall what the workaround was.

--- Comment (dongyang, Wed, 4 Sep 2024 05:59:41 +0000) ---
passing in LDISKFS_IGET_NO_CHECKS is not really doable, we need to change lookup_one_len()>__lookup_slow()>ldiskfs_lookup() that's too much changes in vfs.

The more important issue is ext4 will just remount ro when it's not expecting the EA_INODE flag, which seems

excessive but also makes sense since we manually linked the ea inode to lost+found using debugfs.

I think either we should figure out how to do test_61b without making ea_inode directly accessible, since we are really just handling a special case created by linking ea inode to lost+found, or relax the ext4 do not remount ro, then fix the looping in osd_initial_OI_scrub.
