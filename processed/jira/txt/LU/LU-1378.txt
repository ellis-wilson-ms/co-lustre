Di,

any ideas about the very high SEQ usage?  It isn't a critical problem for 2.1-2.3, since we have 64 bits of SEQ numbers, and all of the MDT0 SEQ numbers will be contiguous and should be kept in a single entry in the FLDB.

However, once we get to DNE with multiple MDTs and OSTs consuming super sequence ranges, it has the possibility to explode the number of FLDB entries and cause unnecessary overhead on the servers and all of the clients.

though to be honest I have no idea what this code is doing.  This is the sequence server, I'm not familiar enough with the code to be able to immediately jump to where this should be swabbed.

Of other interest is the actual FID value here.  The starting FID_SEQ_NORMAL is 0x200000400.  It would appear  that you have consumed almost 200 billion sequences already?  There should be on the order of 2 sequences per client node per mount, so is it realistic you've had 100B client mounts of this filesystem already, or is there some potential issue here that we should be looking into?  Even with 100k clients, this would be almost 2M mounts, which doesn't seem right.

Alternately, it is possible that the MDS is not tracking its own super-sequence correctly, and is getting a new 1B sequence on every mount, and has mounted 0x28 = 40 times?  While not an immediate and fatal problem, the code is currently implemented to have a relatively small number of super-sequences in use at one time, so if there is one allocated for each mount it may potentially cause problems down the line with DNE.
