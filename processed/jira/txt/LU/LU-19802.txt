--- Description ---
I installed a el9.5 cluster a while ago and must have built and installed e2fsprogs from sources at some point.

Leading to libext2fs.so.2.4 being installed in /lib.

I recently upgrade to 1.47.3-wc2.el9 which contains /usr/lib64/libext2fs.so.2.4

same so version but installs in a different location.

This newer version of libext2fs.so.2.4 has an incompatible API change and a symbol, ext2fs_mkdir has been replaced with ext2fs_mkdir2.

The problem is that the API changed but the library version remained the same. The debugfs binary was built and linked against the new API libext2fs.so.2.4 but executing it the linker/loader picked the "old" version of libext2fs.so.2.4 so the binary failed to link/load due to missing symbols.

While debugging, trying to run a test, I encountered a related issue.

Other utilities, such as mkfs.lustre calls out to the debugfs binary to check for whether "-O quota" is supported or not. This check failed because the debugfs binary could not execute but mkfs.lustre prints a misleading error message:

lt-mkfs.lustre FATAL: "-O quota" must be supported by e2fsprogs, please upgrade your e2fsprogs.

--- Comment (JIRAUSER19283, Tue, 20 Jan 2026 00:50:36 +0000) ---
Yes. That is I think exactly the issue. New binary is not compatible with the old library.

In semantic versioning, this is the kind of change that would be:

version a.b.c => a.b+1.0

To signify that an old application linked against a.b.c will also work against a.b+1.0

but a new application linked against a.b+1.0 will not work against an older a.b.x library

It is probably well outside of anything we can do about it other than letting upstream know.

rsahlberg@whamcloud.com, can you provide the actual error messages that you were getting?

must have built and installed e2fsprogs ... Leading to libext2fs.so.2.4 being installed in /lib.

:

upgrade to 1.47.3-wc2.el9 which contains /usr/lib64/libext2fs.so.4.2 ... installs in a different location.

Presumably you mean libext2fs.so.2.4.  I suspect this is the core of the problem.  Using "make install" from the local build resulted in two different libraries installed at the same time, and the older one is found first. it seems possible that a binary compiled against the new library was accessing the old library that was missing ext2fs_mkdir2()?

The patch that adds ext2fs_mkdir2() provides a wrapper function ext2fs_mkdir() that calls ext2fs_mkdir2() internally.  That should provide backward ABI/API compatibility for applications compiled against the older library, but the the reverse is not true for new binaries referencing ext2fs_mkdir2().  The library .so version numbering is outside our control (or we would potentially break other applications), but in general libext2fs is very good at maintaining backward compatibility, so there is rarely/never a reason to have different versions installed at the same time.

The "mkfs.lustre calls external debugfs binary" issue is potentially because you are using plain e2fsprogs instead of the Lustre-patched e2fsprogs?  The packages should Require "ldiskfsprogs" to ensure there is Lustre feature support, which is Provided by the Lustre-patched version.  If it is just a pathname issue for finding the debugfs binary, then this would be resolved by installing the Lustre packages on the system?
