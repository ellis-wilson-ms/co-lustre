--- Comment (adilger, Wed, 23 Oct 2024 22:03:53 +0000) ---
The 0x16 opcode is OST_FALLOCATE, and I can confirm that this opcode exists on b2_14.

However, I see in the message that this is MDT0000 so it is trying to send the fallocate to the MDS service?  The sanity test_413a/b are doing DoM+fallocate, which wasn't added until commit v2_14_53-10-g163870abfb patch https://review.whamcloud.com/41418 "LU-14382 mdt: implement fallocate in MDC/MDT", so this looks like a straight interop bug.

The check_fallocate_supported() function looks like it is properly being passed mdsN for the facet name, so it should be avoiding the use of fallocate.  However, I suspect that the check for fallocate_zero_blocks is always true for osd-ldiskfs, even for MDTs, because the same OSD module is used for OSTs as well.  Unfortunately, it looks like there is not an OBD_CONNECT_FALLOCATE flag that could be checked on the client to determine whether the target supports this feature, my bad.  However, that is now water under the bridge...

This can just add a version check in check_fallocate_supported() to see if the MDS facet is new enough:

        local fa_mode="osd-ldiskfs.$(facet_svc $facet).fallocate_zero_blocks"
        local mode=$(do_facet $facet $LCTL get_param -n $fa_mode 2>/dev/null |
                     head -n 1)
        [[ "$facet" !~ "mds" ]] || # older MDS doesn't support fallocate
                (( MDS1_VERSION >= $(version_code v2_14_53-10-g163870abfb) )) ||
                mode=""

I don't think this is a real functional regression, since DoM+fallocate didn't work in the past (and in general fallocate() is not guaranteed to work), it is just a test script issue.  The test script was changed to use fallocate in commit v2_15_54-53-g09fc9ccb1a ("LU-14824 test: sanity 413a/b unlink timeout v2") so this should be used for the Fixes: line.  So in  that respect, the test failing interop with b2_14 is a regression in 2.16.
