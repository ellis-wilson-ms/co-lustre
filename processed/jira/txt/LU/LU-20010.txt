--- Description ---
Lustre directories on ZFS always use the FatZAP format because each directory entry (luz_direntry) stores both the ZFS dnode ID and a Lustre FID, which exceeds the MicroZAP single uint64 value limitation. The FatZAP leaf block size was hardcoded to 14 (16K blocks) in Lustre, due to which even a single empty directory takes a lot of space.

On dRAID or RAIDZ pools, this results in 90-110K dsize per empty directory due to stripe alignment and parity overhead. For a typical MDT with millions of directories, this wastes significant pool space.

I did a minimal test by introducing osd_fzap_blockshift as a module parameter, replacing the hardcoded 14. The parameter is exposed via /sys/module/osd_zfs/parameters/osd_fzap_blockshift. 

Before:

# mkdir testdir1 && touch testfile1
# du --si test*
100k    testdir1
1.1k    testfile1

After:

# mkdir testdir1 && touch testfile1
# du --si test*
67k     testdir1
1.1k    testfile1

Testing on the dRAID2:9d:12c:1s pool with different leaf_blockshift values showed the following dsize per empty directory:

  blockshift=14 (16K): dsize=~100K (Currentdefault)

Vary FatZAP leaf block size:

  blockshift=12 (4K):  dsize=~67K

  blockshift=13 (8K):  dsize=~67K 

  blockshift=14 (16K): dsize=~100K

  blockshift=15 (32K): dsize=~100K

I think blockshift values from 12 to 15 make a lot of sense. The limits are already in place on ZFS, so I'm not explicitly adding any.

Future Work: 

The real fix would be to store luz_direntry (dnode + FID), allowing small directories to avoid FatZAP entirely. This TinyZAP implementation would require changes on both the ZFS and Lustre sides. This would reduce empty directory dsize to a very low value by storing entries in the existing dnode bonus buffer with no additional block allocation.

IMHO, one reason to avoid "does parameter exist" as the version check is that the test can suddenly stop working if something happens to the parameter (e.g. it is accidentally removed or becomes inaccessible or does not return a valid value).  In that case, rather than also testing that the parameter is working, it results in the tests not being run at all and we don't notice the regression.

My preference would be that (if necessary) there is an $OST1_VERSION check for the commit that added the fzap_blockshift parameter (v2_17_51-27-gab735e9962).  However, I think that always checking for size +7k does not significantly impact the validity of the test, and will pass for both old and new MDS versions.  The test could create more files so that the directory is always 8KiB in size, but I think that slows down the test without improving it in any way.

In either case, there is still the problem that older clients will fail this test when run against a new MDS, so this test should be skipped for older clients by adding an entry into lustre/tests/except/sanity.56aa.ex like:

--- Comment (adilger, Wed, 1 Apr 2026 22:49:05 +0000) ---
Adding comments here from the discussion in Gerrit so that they are not lost:

The best outcome would be for osd-zfs to query the VDEV/zpool directly to determine what the best choice for directory blockshift should be. It would also be reasonable to take a hint from striped directory creation to assume that the directory will be large and use a larger blockshift.

The truth is that 99% of admins won't even know that this parameter exists (even if there is a man page), let alone have some idea of what the optimum value is to use. My strong preference recently is that parameters should tune themselves, and only exist for when the auto-tuning got it wrong (or the 0.01% of users that know better).

Should the examples be changed so that OSTs get a larger value (15 or 16) and MDTs get a smaller one (12 or 13)? Are those reasonable values to use for production?

It still seems like the best outcome would be for the blockshift to be selected automatically for each ZAP object based on what it is used for:

	regular MDT directories use 12 or 13 (typically small and very random IOPS avoids write amplification)

	striped MDT directories could be larger if that improves performance (write inflation vs. wider tree for huge directories)

	OST directories typically larger (14, 15, 16?) due to RAID-Z/dRAID usage

This should probably be done to match the "typical use case" even without being able to probe the underlying VDEV geometry.
