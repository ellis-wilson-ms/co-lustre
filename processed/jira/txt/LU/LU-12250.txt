btw this is old but if you have selinux enabled (even in permissive mode, given the number of xattr requests it probably is) this actually could be LU-9193 - the selinux xattr request has different locking requirements so will cause many cancels when opening files in the same directory.

The fix only made its way to 2.13 (the 2.12 backport got reverted immediately) so that'll only help after both servers and clients get upgraded though. I'm told atos backported it (with LU-12212 that fixed the original patch) in their 2.12.5 ; it's a shame it didn't make it for the community version.

--- Comment (green, Wed, 1 May 2019 01:31:43 +0000) ---
the reason these are per MDS is because thread pools are shared on every node. For the same reason if you only have a single slow OST on an OSS node, it wold quickly clog all io threads on the entire node and all OSTs would feel slow.

So just keep in mind a single target could affect entire node performance if it's making threads to run for a long time. There were some discussions to allow NRS to create per-target limits on a node to avoid this, but I don't remember seeing any patches.

If you are graphing this stuff, definitely do it for all thread pools (there are multiple on MDS) + add the OSS counterparts too, I guess.

--- Comment (green, Tue, 30 Apr 2019 18:13:13 +0000) ---
it's possible that these things are just related.

When you open a lot of files - that creates a lot of locks therefore setting in motion mechanism to reduce lock pressure and cancel some of hte other outstanding locks - hence the lots of locks cancels.

But the other facet of this is when you do open with DoM if DoM code decides to return any data back - it first needs to read the said data from the server. This might be introducing slowdowns into RPC handling leadign to formation of incoming RPC queue as all the threads are busy trying to do opens and so everything else stuck in the queue behind them.

You can test this theory by looking into server side stats like this when the problem is occurring:

lustre/mds/MDS/mdt/stats and check the req_qdepth and perhaps req_waittime values.
