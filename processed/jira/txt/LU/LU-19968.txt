--- Comment (adilger, Tue, 24 Mar 2026 21:00:05 +0000) ---
I think force-trivial would be the very most that should be done, but if it is clear that a patch is only affecting, say Documentation/*,  LICENSES, lustre/ChangeLog, contrib/ then it doesn't really need any testing since those files don't affect any testable functionality.  A build would still be useful to verify that it doesn't break the packaging (e.g. extra files installed that are not packaged).  If we wanted to do some testing, then a single x86 session with runtests would be enough to at least verify the packages could be installed.

Having a "review-smoke" session that only ran runtests takes about 5 minutes (excluding node-provisioning and lustre-initialization) and would be a good "first session" for every patch that gates whether the rest of the sessions should be scheduled, if there is high load on the test system.
