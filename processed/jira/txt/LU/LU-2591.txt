--- Description ---
I've found a race between mount/umount and lov_notify().

The race is caused by the following scenario

1) mount/umount: mount runs and fails to communicate with some OSTs. Then import objects are registered to a pinger list.

2) pinger      : pinger succeeds to communicate with one OST

3) ptrprc_rcv  : ptlrpc_rcv calls lov_set_osc_active() and holds lov_refcount. 

4) mount/umount: For some reason, possibly mount fails or umount runs, ll_put_super() is called.

5) mount/umount: lov_disconnect() calls lov_del_target() and set all target->ltd_reap to actually disconnect OSCs.

6) ptlrpc_rcv  : puts lov_refcount which ptlrpc_rcv has gotten at (3). If lov_refcount becomes 0 here, ptlrpc_rcv tries to disconnect all OSCs at __lov_del_target()

7) ptlrpc_rcv  : some OSC's imports have still been LUSTRE_IMP_CONNECTING state, so ptlrpc_rcv thread have to wait for these imports' state to be changed to non-recovery state at ptlrpc_disconnect_import().

Now that ptlrpc_rcv thread is waiting for the import state to be changed to non-recovery state but ptlrpc_rcv is the one who is supposed to change a recovery state to a non-recovery state. So ptlrpc_rcv must hung. And that's why this node is not able to use ptlrpc_rcv thread anymore.
