The objdump above and call stack points to the ldlm_lock being dereferenced, but I'm not seeing a spot in the code where a lock is dereferenced without holding the spinlock we hold in the call stack having the general protection fault. We've seen cases in the past where some optimizer side effect or async multi-threaded handling with interrupts is at play to make the reported call stack not exactly correct. However, I do see a code path for l_resource - if one thread calls ldlm_lock_change_resource ** while another thread is in the call stack above, there could be a situation where a lock in the liq_waiting list has its l_resource changed right after it is accessed in ldlm_process_inodebits_lock and the old l_resource is freed leading to this GPF. 

LU-4801 removed a spinlock protecting l_resource and changed to RCU implementation instead. I am proposing a patch to ldlm_process_inodebits_lock that protects l_resource pointer by defining an RCU read critical section and safely accessing this pointer using rcu_dereference(). I would be curious to get other's opinions on this, as there are a lot of places in the Lustre code where l_resource is accessed directly instead of using RCU.

The issue here does not seem to be linked to dereference the l_resource. The "lock" pointer seems to be bad before the l_resource is dereferenced:

3a17c: 48 89 fd mov %rdi,%rbp
3a18a: 4c 8b 77 38 mov 0x38(%rdi),%r14 

The lock pointer is set in "%rbp" and then in "%rdi" (0x38 is the offset of l_resource in ldlm_lock). The kernel stack indicates that those are set to 0x5a5a5a5a5a5a5a5a at the crash time:

Oct 18 15:26:37 ip-198-19-88-236 kernel: RDX: 0000000000000000 RSI: ffffa2d7c14cbbc0 RDI: 5a5a5a5a5a5a5a5a
Oct 18 15:26:37 ip-198-19-88-236 kernel: RBP: 5a5a5a5a5a5a5a5a R08: ffffa2d7c14cbbd8 R09: ffffa2d7c14cbbf8
