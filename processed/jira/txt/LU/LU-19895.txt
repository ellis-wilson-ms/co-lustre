--- Description ---
This ticket is to track the semantics of how the large folios are implemented.

Do they impose a minimum "chunk" size for read-modify-write?  Does the stripe_size of a file affect the folio size?  Or are the folios opportunistically sized (e.g. application read/write size up to the stripe size).   Should there be a llite.*.large_folio=N[KMG] tunable that allows specify the minimum folio size for the file, or possibly in the near future the compression chunk size should set this internally, so that the page cache handles the complexity of aggregating multiple pages into a chunk to avoid issues with mmap.

It makes sense to allow large folio size for IOs up to stripe size, preferring to have larger folios to reduce the overhead of managing the pages.

--- Comment (stancheff, Fri, 20 Feb 2026 04:35:08 +0000) ---
folio order is determined in ll_write_begin() so len is limited to the stripe maximum.

With mapping_set_folio_order_range() and mapping_set_large_folios() enabled for the mapping __filemap_get_folio() may return a folio up to the size requested in fgf_flags which is typically:

    FGP_WRITEBEGIN | fgf_set_order(len)

where len is additionally limited to cli->cl_max_pages_per_rpc

The returned folio may be any size up to the maximum requested, the minimum is order-0 and, of course, -ENOMEM if cache is exhausted.

As memory gets more fragmented over time the only certainty is there will be some order 0 folios regardless of the size requested.
