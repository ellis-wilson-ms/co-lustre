--- Description ---
Implement a simple in-memory OSD device for Lustre. This enables Lustre servers to be run without any backing filesystem, such as ldiskfs or openZFS.

The main goal of this OSD is to make it easier to test Lustre on kernels which don't readily ldiskfs or openZFS. It will also make it simpler to set up a new Lustre dev environment, since many changes to Lustre do not depend on a particular filesystem.

--- Comment (JIRAUSER18433, Tue, 2 Jul 2024 16:07:09 +0000) ---
"interesting idea.. but you should remember that Lustre tests do stop and restart OSD many times during tests, so you still need some storage even for such a virtual OSD."

It should be possible to persist data in memory after a umount. The current implementation just destroys everything if the server is stopped.

 

"In your patch, I saw you are using in-memory data structs to manage the inodes and data (cache pages), could we borrow some implementation of ramfs or tmpfs to manage file data (using the kernel address mapping to mange the pages of a file) and directories?"

I thought about using ramfs/tmpfs. But I'm not sure if ramfs/tmpfs provides enough value-add to warrant the added complexity. The manual page management allows for the OSD to be very simple - which I think is very valuable. I'm open to reconsider - if there is something big we're missing out on by not using ramfs/tmpfs.

 

"how is it better than say ldiskfs running on an image in tmpfs ?"

You don't have to deal with ldiskfs patch series - if you test against a lot of different kernel versions, that's a big help.
