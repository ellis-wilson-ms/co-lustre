--- Comment (niu, Wed, 24 Aug 2011 23:14:01 +0000) ---
Actually, the ATTR_FROM_OPEN logic looks wrong to me.

On the server side, we don't acquire DLM lock for the setattr from open

/* In the raw-setattr case, we lock the child inode.
 * In the write-back case or if being called from open,
 *               the client holds a lock already.
 * We use the ATTR_FROM_OPEN (translated into MRF_SETATTR_LOCKED by
 * mdt_setattr_unpack()) flag to tell these cases apart. */

Howerver, the truth is that the client deosn't hold any lock at all, which means sometimes we could probably set attr to a file without DLM lock!

I think this part of ancient code was introduced to resolve some kind of deadlock/livelock problem, and with code changes, the deadlock/livelock problem doesn't exist anymore, but this part of code was not removed accordingly.

According to the current Lustre implementation, we don't gurantee the atomicity of open(O_TRUNC) (no DLM lock is holding through open+trunc), the open(O_TRUNC) can be regard as an open followed by a immediate truncate, so it looks safe to remove the ATTR_FROM_OPEN and SETATTR portal stuff.

There might be some slight impact on open cache when removing the ATTR_FROM_OPEN stuff, the truncate from open(O_TRUNC) could cause it's open lock being revoked, this looks not so efficient, but I think it's necessary for correctness.
