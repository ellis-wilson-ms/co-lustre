--- Description ---
when ost is mounted the conf_param is not read from the msg. Say msg enabled quota and 

ost is mounted after enabling the quota conf_param the new ost mount will not inherit the conf_param and reason for this is the ost does not read the conf_param from mgs

if you follow steps on two different host you can see this issue 

 ssh 10.103.18.154 'sudo mkfs.lustre --mgs --mdt --reformat --fsname=lustre --index=0 /dev/vdb1'

ssh 10.103.18.154 'sudo mkdir -p /mnt/mdt1'

ssh 10.103.18.154 'sudo mount -t lustre /dev/vdb1 /mnt/mdt1'

 ssh 10.103.18.154 'sudo mkfs.lustre --mdt --reformat --fsname=lustre --mgsnode=10.103.18.154@tcp --index=1 /dev/vdb2'

ssh 10.103.18.154 'sudo mount -t lustre /dev/vdb2 /mnt/mdt2'

ssh 10.103.18.154 'sudo lctl conf_param lustre.quota.ost=ugp'

 ssh 10.103.18.154 'sudo lctl conf_param lustre.quota.mdt=ugp'

ssh 10.103.18.26 'sudo mkfs.lustre --ost --reformat --fsname=lustre --mgsnode=10.103.18.154@tcp --index=0 /dev/vdb3'

ssh 10.103.18.26 'sudo mkdir -p /mnt/ost1'

 sudo mount -t lustre /dev/vdb3 /mnt/ost1

The new mounted ost will not have quota enabled

--- Comment (JIRAUSER18604, Tue, 20 Jan 2026 19:57:59 +0000) ---
The conf_param settings are not automatically propagated to a newly mounted OST, whereas set_param values are applied as expected. Based on this behavior, my current thought is to document that conf_param must be explicitly re-applied after mounting a new OST by re-running the conf_param command.

If required, we can later enhance the mount path to read and apply conf_param values automatically.

Below is the output demonstrating the behavior before and after propagating conf_param to the newly mounted OST.

//[rocky@ost-srv ~]$ ssh 10.103.18.154 \
  'sudo lctl get_param '\''osd-ldiskfs.*.quota_slave_*.enabled'\'''
osd-ldiskfs.lustre-MDT0000.quota_slave_dt.enabled=ugp
osd-ldiskfs.lustre-MDT0000.quota_slave_md.enabled=ugp[rocky@ost-srv ~]$ ssh 10.103.18.26 \
  'sudo lctl get_param '\''osd-ldiskfs.*.quota_slave_*.enabled'\'''
osd-ldiskfs.lustre-OST0000.quota_slave_dt.enabled=none
 

After re-applying conf_param:

 

[rocky@ost-srv ~]$ ssh 10.103.18.154 \
  'sudo lctl conf_param lustre.quota.ost=ugp'
[rocky@ost-srv ~]$ ssh 10.103.18.26 \
  'sudo lctl get_param '\''osd-ldiskfs.*.quota_slave_*.enabled'\'''
osd-ldiskfs.lustre-OST0000.quota_slave_dt.enabled=ugp

This confirms that re-running conf_param correctly propagates the quota settings to the newly mounted OST.

The issue is that during OST mount, the log lustre-MDT0000—where the conf_param entries reside—is not opened.

When we inspect the MDT log:

sudo lctl llog_print lustre-MDT0000- { index: 2,  event: attach,       device: lustre-MDT0000-mdtlov, type: lov, UUID: lustre-MDT0000-mdtlov_UUID }
- { index: 3,  event: setup,        device: lustre-MDT0000-mdtlov, UUID: }
- { index: 6,  event: attach,       device: lustre-MDT0000, type: mdt, UUID: lustre-MDT0000_UUID }
- { index: 7,  event: new_profile,  name: lustre-MDT0000, lov: lustre-MDT0000-mdtlov }
- { index: 8,  event: setup,        device: lustre-MDT0000, UUID: lustre-MDT0000_UUID, node: 0,
                                    options: lustre-MDT0000-mdtlov, failout: f }
- { index: 11, event: conf_param,   device: lustre, parameter: quota.ost=ugp }   

However, during OST mount, following  logs are opened—none of which contain the conf_param entries. As a result, a newly mounted OST does not have quota enabled.

From the OST side, only the following logs are processed:

// 10000000:00000040:0.0:1768596901.895563:0:3062:0:

(mgc_request_server.c:804:mgc_process_server_cfg_log())
Process leaving (rc=0 : 0 : 0)
 

Since lustre-MDT0000 is never opened by the OST, the conf_param entries are missed, and quota remains disabled on the OST.

how should this be fixed?

Ensuring that the OST also opens lustre-MDT0000 and applies the relevant conf_param entries, orWriting the required conf_param entries into a configuration log that the OST already opens during mount.
