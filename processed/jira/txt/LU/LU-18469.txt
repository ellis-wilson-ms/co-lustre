--- Description ---
It could be desirable to be able to de-activate a nodemap and later re-activate it.

De-activating would mean the nodemap definition is kept, but new client mount attempts would fail.

To implement this, it is possible to add a new nodemap property such as 'inactive'. A storage administrator can then set  'nodemap.nm0.inactive=1' to deactivate a nodemap, after which all new mount attempts for that nodemap will fail.

Similarly 'inactive=0' would reactivate the nodemap.

--- Comment (JIRAUSER19288, Mon, 25 Nov 2024 15:29:28 +0000) ---
sebastien as this topic came up in Gerrit that "inactive" may not be an appropriate property name because intuitively it could be understood as temporarily disabling all individual nodemap settings to defaults (similar as "lctl nodemap_activate 0" would work for global). Instead only new mounting attempts are denied by this setting.

Perhaps the property name "deny_mount" or "deny_all" would be more suitable (which would be consistent to the existing "deny_unknown" property)?

(edit) Since "deny_unknown" denies access even on existing clients, "deny_all" only working during mount would be inconsistent.
