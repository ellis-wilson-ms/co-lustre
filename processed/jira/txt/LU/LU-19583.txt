--- Description ---
Repeated deletion of files with the same name can fail in certain scenarios. Please refer to the following test case:

1. Create the following directory structure  

 

   [root@ai400-vm00 trash_1]# tree  
   .  
   ├── file1  
   └── subdir  
       └── file1  

2. Delete all files in the test directory  

   [root@ai400-vm00 trash_1]# rm -rf ./*  

3. Restore `.Trash/subdir/file1` back to `./file1`  

   [root@ai400-vm00 trash_1]# lfs trash mv .Trash/subdir/file1 ./file1  
   [root@ai400-vm00 trash_1]# cat file1  
   sub_file1  

   

4. Delete the `file1` that was just restored  

   [root@ai400-vm00 trash_1]# rm file1  
   rm: remove regular file 'file1'? yes  
   rm: cannot remove 'file1': File exists  

   

At this point, `rm` reports the error: cannot remove 'file1': File exists  

Moreover, the restored `file1` has been permanently deleted and did NOT go into .Trash

--- Comment (JIRAUSER17305, Tue, 11 Nov 2025 12:09:51 +0000) ---
Regarding this issue, we found another phenomenon:

After deleting a file in the current directory, if a file with the same name is moved (via mv) from another directory (any directory within the namespace that is not the current directory) and then deleted, an error occurs, and the deleted file does not enter .Trash.

Repeatedly creating and deleting files with the same name in the current directory behaves as expected: deleted files appear in .Trash with timestamp suffixes.

root@cn02:/aifs/trash_test/trash_test_3# echo 'file1' > ./file1 ; rm ./file1
root@cn02:/aifs/trash_test/trash_test_3# echo 'file2' > ./file1 ; rm ./file1
root@cn02:/aifs/trash_test/trash_test_3# echo 'file3' > ./file1 ; rm ./file1
root@cn02:/aifs/trash_test/trash_test_3# ls .Trash
file1  file1.2025-11-11T11:49:09  file1.2025-11-11T11:49:15

Deleting a file in the current directory, then moving a same-named file from another directory into it and deleting it, results in an error, and the deleted file does not enter .Trash.

root@cn02:/aifs/trash_test/trash_test_3# echo 'file3' > ../file1 ; mv ../file1 ./file1 ; rm ./file1
rm: cannot remove './file1': File exists
root@cn02:/aifs/trash_test/trash_test_3# echo 'file3' > /aifs/file1 ; mv /aifs/file1 ./file1 ; rm ./file1
rm: cannot remove './file1': File exists

root@cn02:/aifs/trash_test/trash_test_3# ls .Trash
file1  file1.2025-11-11T11:49:09  file1.2025-11-11T11:49:15

Correspondingly, if similar operations are performed using cp instead, everything works as expected.

root@cn02:/aifs/trash_test/trash_test_3# echo 'file3' > /aifs/file1 ; cp /aifs/file1 ./file1 ; rm ./file1
root@cn02:/aifs/trash_test/trash_test_3# echo 'file3' > ../file1 ; cp ../file1 ./file1 ; rm ./file1
root@cn02:/aifs/trash_test/trash_test_3# ls .Trash
file1  file1.2025-11-11T11:49:09  file1.2025-11-11T11:49:15  file1.2025-11-11T12:01:58  file1.2025-11-11T12:02:34

This seems to indicate that whether repeated deletion of a file with the same name fails is related to whether the file was renamed (moved) from another directory into the current directory.
