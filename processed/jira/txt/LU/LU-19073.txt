--- Description ---
For ease of use, it was proposed that the SSK keys could be ASCII-encoded (e.g. base64 or gss_base64url_encode()) so that they can be more easily processed by user tools (e.g. copy-paste from EMF SSK generator web interface or similar).  lgss_sk and other tools should be able to transparently detect the ASCII-encoded key format (e.g. by some string at the beginning, like "Lustre SSK v1.0\n" ) and decode it into the current binary struct format in memory (after skipping the descriptive string).

I think generating the ASCII-encoded SSK key could be done via a new -a|--ascii option to lgss_sk:

https://doc.lustre.org/lustre_manual.xhtml#table.lgss_sk_params

Having a header such as "Lustre SSK v1.0\n" would be fine, and allow lgss_sk and the internal loading function sk_load_keyfile() to discriminate between a binary SSK key and an ASCII-encoded SSK key.
