--- Description ---
If lctl get_param  has a file named  in the current working directory, it returns that instead of the expected values.

To reproduce:

# /usr/lib64/lustre/tests/llmount.sh
# cd /tmp
# lctl get_param version
version=
lustre: 2.1.51
kernel: patchless_client
build:  2.1.51-ga83a510-CHANGED-2.6.32-131.6.1.el6_lustre.g65156ed.x86_64
# echo "All your Lustre are belong to us" > ./version
# lctl get_param version
=
All your Lustre are belong to us
#

--- Comment (adilger, Fri, 2 Dec 2011 16:53:34 +0000) ---
Good catch. 

If anyone is going to do more than trivial changes to get_param, it would be good to move it into an llapi_get_param() helper that can be used in other parts of the code. In quite a number of places in lctl, lfs, mount.lustre, erx.

--- Comment (rhenwood, Fri, 2 Dec 2011 16:37:50 +0000) ---
John identified a more sinister potential issue with the behavior of get_param. One can unwittingly corrupt a local file if it has the same name as the :

# lctl set_param version="All your Nyancat'"
# lctl get_param version
=
All your Nyancat're belong to us

A better fix for this perilous behavior is to not have lctl {get|set}_param lookup the local path at all.

--- Comment (rhenwood, Fri, 2 Dec 2011 13:36:17 +0000) ---
The code appears to first check if a file called  is present and can open 0_RDONLY. If a file is not present, the code falls back to look for /proc/{fs,sys}/{lnet,lustre}/.

I suggest to fix the issue of get_param being easily confused, the path /proc/{fs,sys}/{lnet,lustre}/ should be checked first.
