--- Description ---
For multi-tenancy configuration on a Lustre filesystem, it would be preferable to allow multiple Tenants using a different user authentication service that is owned by the Tenant, eg: an LDAP per tenant.

In this scenario, Tenants may have many tens or hundreds of users and groups defined, and these will be highly dynamic, with users and groups being added and removed. These UIDs, GIDs, and PROJIDs may overlap across tenants, and we do not have the possibility of dictating what UID/GID/PROJID ranges each tenant may allocate from.

To support such an environment, the storage administrator could create an individual mapping for each and every ID used by a Tenant, however this would be extremely onerous on the Storage administrator to maintain and keep up to date, and it the Storage Administrator may not have any access or knowledge about the Tenant IDs, working more as a Cloud Service Provider, separate from the team managing the Tenant). As well, having hundreds of thousands or millions of different IDs to map at the server level will put strain on the server resources (memory, CPU).

Instead, could we add a new configuration option to a nodemap to specify an ID mapping offset start and end, so that automatically, every Tenant local ID would be mapped to a new value "UID+OFFSET" on the storage, without any explicit mapping being set.

Then a storage administrator could choose to allocate 1M canonical UIDs to a Tenant, by specifying this property:

Tenant1: canonical UID = UID+100,000

Tenant2: canonical UID = UID+200,000

Tenant3: ...

This way each tenant, will have distinct canonical UID/GID/PROJID range on the storage, without any effort to maintain the mapping rules as new users are created/deleted within the Tenants.

As discussed with adilger and sebastien all nodemaps will be able to have their UID/GID/PROJID offset set individually, but by default will all be set the same. The offset values will also only be applied to the individual FSID values when they are being loaded, and the original mapped values are what are saved. For ease of use, after the offset is declared users will not need to add it to every mapping (i.e. the admin will specify CLID:FSID as if offset didn't exist), and the offset will be automatically added in the back end when the nodemap handled.

mdilger, I was writing a long description here about how nodemaps store their configuration parameters on disk, but then the more I looked into it, the more confused I became, and I'll need some guidance/discussion with sebastien to clarify the situation. Unfortunately this is a all bit complex due to historical compatibility baggage with how the nodemap config records are stored on disk.  The nodemap records are stored in an index file ("IAM" key-value store), and every record needs to have a unique indentifier/key (struct nodemap_key) so that it can be found and updated as needed.  That means it isn't possible to have two records that are mapping the same client ID to a different filesystem ID for a given nodemap, or the same .  The key is made up of the nodemap number (nk_nodemap_id, which contains enum nodemap_idx_type in the top 4 bits to identify different record types for that nodemap) and the record ID (one of nk_cluster_subid, nk_range_id, nk_id_client, or nk_unused depending on the NODEMAP_*_IDX type of the key).  The combination of the nodemap ID, type, and range/client/cluster IDs gives each record a unique value.

There can be at most 15 different record types.  Each config record can hold exactly 32 bytes of config data.  Once the record type is found, the value of the 32-byte record can be decoded from union nodemap_rec depending on the record type.  This gives us one of struct nodemap_range_rec, nodemap_range2_rec (both used for configuring the IP address/NID range that the nodemap is handling, struct nodemap_id_rec (which is configuring the client ID-> filesystem ID mapping), struct nodemap_global_rec (which is configuring the global enabled/disabled state of the nodemap), and struct nodemap_cluster_roles_rec (which holds bits that enable/disable individual administrative roles on the nodemap).

--- Comment (adilger, Wed, 7 Aug 2024 20:25:56 +0000) ---
From an implementation POV, it is probably better to load the idmap values into the rbtree using their original values, and then apply the offset (and limit check?) as a separate step after any individual idmap rbtree mapping is done.  This would simplify the code flow and allow dumping the original values from the idmap later, and avoids issues if the offset is specified (or potentially, at great risk, modified) after the nodemap is created.

While adding or modifying a nodemap on an existing filesystem is potentially risky (every existing file would have the wrong UID/GID/PROJID on disk), this could potentially be needed due to administrative or organizational changes, and can be fixed by the storage admin running something like:

for ID in {0..limit}; do
      old_id=$((ID+orig_offset))
      new_id=$((ID+new_offset))
      lfs find SUBDIR -uid $old_id -print0 | xargs -0 chown $new_id
      lfs find SUBDIR -gid $old_id -print0 | xargs -0 chgrp $new_id
      lfs find SUBDIR -projid $old_id -print0 | xargs -0 lfs project -p $new_id
done

on the entire directory tree in question (preferably with a smaller ID range than 0..99999).  It might also be worthwhile to change "lfs find -uid/-gid/-projd" to allow finding a range of IDs instead of just a single value each time, so that the filesystem doesn't need to be walked 100k times.

It would be best to have a find like "lfs find SUBDIR -uid $orig_offset-$((orig_offset)) -o -gid $orig_offset-$((orig_offset)) -o -projid $orig_offset-$((orig_offset))" to find all of the affected files in a single pass, and then have a command to change the UID/GID/PROJID with ((new_offset-old_offset)) in a single operation (even if they are different values) so that the filesystem only needs to be walked a single time.

However, "lfs find" does not have "-o" support to combine multiple checks with OR, only an implicit AND of all conditions specified.
