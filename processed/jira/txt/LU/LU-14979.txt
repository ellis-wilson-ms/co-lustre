--- Description ---
Currently implemented recovery ping mechanism increases the next scheduled recovery ping attempt timeout exponentially (base 2) and limits the timeout at 900 seconds. This hard-coded value appears to be too high in many cases. Introduce a tunable parameter that can be used to limit the recovery ping timeout and come up with a reasonable default.

--- Comment (ssmirnov, Wed, 1 Sep 2021 21:24:20 +0000) ---
Chris,

I set up a test for LU-14978: Node A with one NI, Node B with two NIs, all on the same net. I was using lnetctl ping to create traffic from A to B. Then I executed "ifdown" on the interface corresponding to one of the B's NIs. (This was to simulate a hardware failure on node B.) Some lnetctl pings failed and the "failed" peer NI's health got decremented as seen by A. I left it alone for a few minutes, then brought the "failed" interface on B back up. A didn't realize that B had both NIs healthy until it got around to sending the next recovery ping. In my opinion, the delay was too long. Unless I initiate a ping from B to A and it uses the recovered interface, A has no idea that B has both NIs back until 900 second timeout expires.
