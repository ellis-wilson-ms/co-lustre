--- Comment (adilger, Fri, 14 Jun 2024 00:01:04 +0000) ---
I think this test is not very robust.  Firstly, it is using [ $((old_iused - new_iused)) -gt 400 ] for the numeric comparison instead of (( old_iused - new_iused > 400 ] )), so it looks like it fails most of the time because the number of objects decreased (i.e. negative difference) since the start of the test:

https://testing.whamcloud.com/test_sets/426720d0-057c-4837-9d06-ff94edc60903

waited 120 sec, old Iused 5084, new Iused 4748
waited 120 sec, old Iused 5551, new Iused 5183
waited 120 sec, old Iused 6617, new Iused 6249
waited 120 sec, old Iused 4007, new Iused 3639

This should be fixed with a simple patch and avoid most of the failures.

However, it also isn't clear if the test is totally robust, because it is creating files before max_create_count=0 is set, but the MDS may start creating new objects before all of the precreated OST objects are consumed.

Sometimes it is reporting a failure for unknown reasons, when it should be within the 400-object limit:

https://testing.whamcloud.com/sub_tests/cfb9000f-df2c-4a1f-bf8a-7bbd29604460

waited 120 sec, old Iused 2025, new Iused 2105
waited 120 sec, old Iused 3877, new Iused 4149

and other times it appears that the "old_iused" value returned zero, so "df" wasn't working or something (b2_14 servers in interop?):

https://testing.whamcloud.com/test_sets/1fd4e85b-a5bc-467f-ba14-6b12a1832eb6

https://testing.whamcloud.com/sub_tests/44fe3a1c-1540-4e1a-a970-ef591f7db166
