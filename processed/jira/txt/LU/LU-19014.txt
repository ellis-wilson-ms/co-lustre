--- Description ---
Two nodes at least append write a shared file in Lustre with memcg enabled.

It will hang in the following call traces:

[root@ouzo2 margu]$ cat /proc/2593160/stack

[] do_file_write_iter+0x3e9/0x5d0 [lustre]
[] vfs_write+0x2cb/0x410
[] ksys_write+0x5f/0xe0
[] do_syscall_64+0x5c/0xf0
[] entry_SYSCALL_64_after_hwframe+0x78/0x80 

After analyze the core dump of the hang system, we found that the @bdi_writeback data structure (wb) corresponded to the memcg has dirty I/O (in state 5: WB_registered | WB_has_dirty_io) with dirty pages, but can not write-out the dirty pages, and loop forever in balance_dirty_pages().

 

This is a bug in Lustre memcg code. In OSC, it stops the write back dirty pages once found that the unstable pages for an OSC is zero. This is incorrect, we should still write back the dirty pages to update the wb stat accounting, thus the write process can continue instead of looping forever.

--- Comment (qian_wc, Mon, 14 Jul 2025 01:51:21 +0000) ---
Do you have any testing results about the performance regression?

 

I think the problem is that when a memcg on a client is under memory pressure, it will trigger the fsync() on the file.

But other memcg may have lots of memory available and no need to sync data.

 

Fast commit in ext4 is a journaling optimization introduced in Linux 5.10 that aims to reduce commit latency, particularly in data=ordered mode. It works by storing a minimal delta of metadata changes in a shared fast commit space with the JBD2 journal. This allows for faster commits compared to traditional full commits, especially when many fsync() operations are involved.  

However, the fsync() on the OST (ldiskfs-osd) is actual a sync() operation and will trigger the synchronous journal commit and all pending transactions will need to commit the the persistent storage (not just the one issues the fsync call). Lots of this kind of transaction journal commit may have the negative impact on the performance.

Higher fsync() frequency on JBD2
- Reduced application throughput
- Higher journal overhead & interference
- Lower availability of disk resources for useful workloads 

 

The solution may be using soft sync when a certain memcg on a client is under memory pressure. In the meanwhile, using the Fast Commit (FC) in Ext4 or ZFS to just commit the data for a specific OST object not the whole transaction group.

FYI, FC: https://www.usenix.org/system/files/atc24_slides-shirwadkar.pdf
