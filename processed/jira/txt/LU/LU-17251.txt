--- Comment (adilger, Thu, 7 Nov 2024 04:48:50 +0000) ---
The main reason for test failures is that it looks like the SEQ_WIDTH is set at 16k and this is causing some OSTs to create far fewer objects if it rolls over during the test.  With the output from patch https://review.whamcloud.com/56853 ("LU-17251 tests: try to fix test_rr_alloc again") it is pretty clear that this is the case.

For example, there are 3 OST-MDT pairs are close to exceeding the 16KB SEQ_WIDTH limit (OST0004, OST0006, OST0007 with MDT0002), and then those are the OSTs that have fewer objects created on them:

https://testing.whamcloud.com/test_sets/f50ef9df-e885-45cc-a839-b95a455289cf

The patch has added "ost_set_temp_seq_width_all $DATA_SEQ_MAX_WIDTH" before starting to precreate the objects, and this has mostly fixed the test failures, while setting "lod.*.create_count=$create_count" repeatedly on those OSPs not make any difference if the sequence is close to running out.

It looks like it needs some more dramatic changes like forcing the sequence to roll over. However, at least that will not take very long because the number of objects remaining to wrap the sequence is not very large (e.g. a few hundred).
