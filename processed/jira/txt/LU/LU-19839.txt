--- Description ---
There would be benefit for some application profiles to allow for optional bypass of the Hybrid I/O mechanism.  Most likely to benefit would be applications with large request sizes that Hybrid would force down the direct path leaving large amounts of unused client memory and also increasing loads on the OSS, particularly when the active files would fit in the ever increasing memory of client nodes.

Steve, there is a patch - possibly posted just before your comment.

The problem here is that we need this setting to be persistent.  The goal is that it can be set to help programs which are unaware and unmodifiable, eg, a possible pain point of databases, which could conceivably generate patterns that would not interact well with hybrid IO but be difficult to detect programmatically.

The other issue with fadvise as an interface is WILLNEED isn't even persistent for an fd - it actually just queues the system to do a prefetch of that range.  Nothing is retained logically - a prefetch is done, and when it's done, that's it.  No persistent state or attempt to retain it in cache.  This doesn't map to your desired behavior.

I'm definitely open to other interfaces if we can figure out how they would work, though, so suggestions welcome.

Implementation Proposal: Persistent Hybrid I/O Bypass Flag

After discussion with Patrick Farrell, the following approach is proposed:

Approach: "Impossible Combination" of Ext4 Inode Flags

Since we cannot freely add new inode flags, we will use an impossible/contradictory combination of existing ext4 inode flags:

LUSTRE_HYBRID_BYPASS_FL = (LUSTRE_COMPR_FL | LUSTRE_NOCOMPR_FL)

These flags mean "compress file" and "don't compress file" respectively - a contradictory combination that has no meaning for regular files (Lustre doesn't implement compression anyway). This combination will now mean "bypass hybrid I/O for this file."

Key Properties:

	Persistent: Stored in ldiskfs inode flags on the MDT

	Backward compatible: Older clients see COMPR+NOCOMPR but ignore them (no compression support), so they just use normal hybrid I/O behavior

	Inode-level: Applies to all opens of the file, not per-fd

	Set via ladvise: Uses existing lfs ladvise infrastructure with a new advice type

Implementation Tasks:

1. Define flag combination in lustre_idl.h:

#define LUSTRE_HYBRID_BYPASS_FL (LUSTRE_COMPR_FL | LUSTRE_NOCOMPR_FL)

2. Add new ladvise type in lustre_user.h:

LU_LADVISE_HYBRID_BYPASS = 6,

3. Handle in ll_file_ioctl(): When LU_LADVISE_HYBRID_BYPASS is received, use setattr to set/clear the COMPR+NOCOMPR flags on the inode. Use LF_UNSET flag to clear.

4. Modify ll_hybrid_bio_dio_switch_check(): Check if inode has the bypass flag set; if so, return false (skip hybrid switching, stay with buffered I/O).

5. Add lfs ladvise -a hybrid_bypass command support.

6. Add tests for flag persistence and hybrid bypass behavior.

Usage:

# Enable hybrid bypass (force buffered I/O)
lfs ladvise -a hybrid_bypass /mnt/lustre/myfile

# Disable hybrid bypass (restore normal hybrid behavior)  
lfs ladvise -a hybrid_bypass --unset /mnt/lustre/myfile

Will proceed with implementation.

--- Comment (paf0186, Fri, 30 Jan 2026 01:41:23 +0000) ---
While I understand some desirability for the component level feature, I think we will definitely have to keep it inode level for usability.  There's also not a very good place to put PFL level flags.

More generally, part of the goal of hybrid IO is to not use page cache, and files which oscillate between large and small IOs are relatively rare.  I think optimizing caching by file component is a pretty narrow use case.  Also note that the lock cancellations are only within a PFL component already (in fact, just within a stripe - locking is all divided by stripe).
