--- Description ---
When using 'fallocate --posix' on a Lustre filesystem with ZFS backend

1. The standard 'fallocate' is attempted first

2. Due to ZFS not supporting standard 'fallocate', it falls back to POSIX writes for space pre-allocation

3. A subsequent truncate operation results in an unexpected file size

4. The issue can be resolved by using 'lfs migrate' on the affected file, but not LFSCK,

 

Steps to reproduce:

# Setup test environment
MOUNT_POINT=/mnt/lustre
TEST_DIR=${MOUNT_POINT}/test_dir
FILE=$TEST_DIR/temp_test_file
mkdir -p ${TEST_DIR}
rm -f $FILE

# Configure progressive file layout (1MB with 1 stripe, remainder with 2 stripes)
lfs setstripe -E 1M -c 1 -E -1 -c 2 $TEST_DIR

# Create test file
dd if=/dev/urandom of=$FILE bs=1M count=1
echo "MD5 original: $(md5sum $FILE | awk '{print $1}')"

# this returns 0
fallocate -l 2M --posix $FILE
echo $?

# metadata is already "corrupted" so truncate will behave incorrectly
truncate -s 1M $FILE
echo "MD5 after truncate: $(md5sum $FILE | awk '{print $1}')"
# expect to see 1M but we get 2M
du -b $FILE

# Attempt repair with LFSCK (will not fix fallocate corruption)
sudo lctl lfsck_start -A -r -t all -C -c -o -d > /dev/null
sudo lctl lfsck_query -w > /dev/null
echo "MD5 after lfsck: $(md5sum $FILE | awk '{print $1}')"

# Repair with lfs migrate (effective solution)
sudo lfs migrate $FILE
echo "MD5 after migrate: $(md5sum $FILE | awk '{print $1}')"

Tested using Lustre 2.15.6
