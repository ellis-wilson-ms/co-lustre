these llogs are kept in sync with MGS0 on all MDTs/OSTs.

--- Comment (adilger, Thu, 26 Feb 2026 19:59:41 +0000) ---
So even as we implement MGS upon RAFT, the MGS0 is still not fault-tolerant, the final result is the same as this implementation.

That would only affect the initial target registration?  It should be enough that the targets specify multiple MGS NIDs on startup, so that at least one of them is running when they try to connect.  One possibility to simplify this is to use DNS or Rendezvous protocol to handle MGS hostname lookup to get the MGS0 NID so that it could be running on any server node (LU-19412).  However, that is a low priority compared to having a redundant MGS to handle the client configuration, since client mounting is happening all the time.

--- Comment (laisiyao, Thu, 26 Feb 2026 05:11:42 +0000) ---
One possible way to implement fault tolerant MGS0 is as below:

	start MGS0 and targets like before

	when RAFT cluster is up, and leader elected, MGS0 notify all targets to switch to RAFT leader as MGS

But even with this, it's better to sync the configuration files and IR data via MGC like this patch set because it's more lightweight.  And this involves a few changes:

	MGS0 needs to get the nids of RAFT leader

	MGC on targets need to be stopped and restart with RAFT leader nids as connection nids, and targets need to register again to ensure target configuration files are created

	the config change commands like 'lctl set_param -p' that run on mgs host need to be forwarded to RAFT leader

--- Comment (laisiyao, Thu, 26 Feb 2026 02:10:23 +0000) ---
IMHO FLDB and Quota still need RAFT to be fault tolerant, since it's the standard way to implement replicated services.  As for MGS, it's different from other services: the MGS is started before RAFT cluster is setup, and MDTs/OSTs mkfs with the MGS nid set, and they need the configuration from MGS to fully setup (possibly not if they have backup in local storage).  So even as we implement MGS upon RAFT, the MGS0 is still not fault-tolerant, the final result is the same as this implementation.

--- Comment (adilger, Thu, 26 Feb 2026 00:59:38 +0000) ---

This covers information like quota limits (QMT), FLDB, flock, etc.  While those databases also do not change rapidly, they cannot be "offline" for a long time if a server has a hardware problem. --- Comment (laisiyao, Tue, 24 Feb 2026 07:48:51 +0000) ---
Yes, this implementation is much lightweight, and it can also support standalone MGS0. BTW the MGS replica will be started on OST nodes as well.

The backup MGS won't take over MGS0 as primary, but wait for MGS0 to come back. If MGS0 node can not get back, the administrator has two options:

	if MGS0 is standalone, recreate MGS0 and copy configuration files from other nodes to this new MGS0

	specify one server node as MSG0, and restart all server nodes
