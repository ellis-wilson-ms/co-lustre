--- Description ---
TBF assigns one token per RPC regardless of the RPC's resource footprint. A 4 MiB bulk write and a 4 KiB read consume identical tokens despite a 1000x difference in bandwidth. This makes rate limits unpredictable in terms of actual resource consumption.

Add page-weighted token accounting: for I/O RPCs (OST_READ, OST_WRITE), token cost equals the number of pages in the request (sum of rnb_len / PAGE_SIZE across all niobuf_remote entries). For non-I/O RPCs, cost remains 1.

The cost model is inferred from the rate format Bandwidth suffixes on rate= and minrate= (e.g. rate=4G for 4 GiB/s) select page-weighted accounting; the kernel converts to pages/s by dividing by PAGE_SIZE. Plain integers select the default rpcs model. Mixed formats (one with suffix, one without) are rejected.

--- Comment (hornc, Wed, 15 Apr 2026 15:42:10 +0000) ---
Fair points, and it is an interesting idea to have a cost model that reflects the underlying storage media. What I wanted to do with this patch was provide an easy way to set bandwidth limits for TBF classes. So an admin can easily say "this UID/NID/ProjID/JobID gets X bandwidth". A 64 KiB RPC consumes exactly 16x the bandwidth of a 4 KiB RPC, so charging 16x the tokens is the accurate model for this use-case.

That said, it should be fairly easy to add additional cost models to serve other use-cases.

--- Comment (adilger, Tue, 14 Apr 2026 10:34:50 +0000) ---
One reason that I didn't worry very much about implementing "bandwidth" for tokens is that this ignores the effects of IOPS on non-flash storage.  Consider an HDD OST (20 HDD in DCR/GRID RAID 8+2) might handle 2GiB/s of sequential 4 MiB x 512 IOPS of reads or writes (equivalent to 524288 x 4KiB writes) but only handle 200 random read/write IOPS (== 200 x 4 KiB writes, different by a factor of 2621x).  For HDDs the cost of reading/writing 64KiB is equivalent to 4KiB, and reads/writes are always IOPS limited in the end (peak bandwidth == IO size * IOPS limit).

Even for flash OSTs there is an extra cost to handling 4KiB IOPS vs. 64KiB IOPS, though it is pretty linear above that.  That makes me think the defaults here need to be tweaked whether an OST is rotational or not, and have a logarithmic cost vs. RPC size.
