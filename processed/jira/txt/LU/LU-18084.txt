--- Comment (adilger, Thu, 8 Aug 2024 02:19:14 +0000) ---
The "lfs: failed for '/oak': No such file or directory" error printed at the end is just the saved (first) error code that "lfs find" returns upon completion.  The expected behavior of all multi-file commands is that if an error is hit they continue to process all of the specified files, and return the error at the end to alert the user/script that something failed along the way.  Otherwise, the original error message may be long gone from the screen, and it may not be possible to figure out which file had the problem since processing all of the previous files may now also return an error (e.g. if rename/removal/etc. doesn't work on the same filename twice).

/oak/stanford/groups is on MDT0000 (like their parent dirs) while /oak/stanford/groups/msalerno is located on MDT0005

are all of the subdirectories under .../msalerno on MDT0005?

Clearly, something is going wrong, but except at the transition between groups and msalerno I can't see the MDT number making a difference, especially so far down in the tree.  Obviously with such a large directory traversal it isn't practical to collect debug logs.

Are you able to move the problematic directory trees to a different subdirectory on MDT0005 to see if that is the actual cause of the problem?  If moving the broken directories is not possible (linking to the original Jira issue would refresh my memory on this) then move their their parents, after moving the "good" content to parallel temporary directory to rename back in their place.

I guess one option for collecting debug logs might be something like having "lfs find" running on the MDT0005 MDS while it is also running "while sleep 1; do lctl dk /oak/debug/debug.$((date +%s.$RANDOM)); done" to capture the kernel debug logs, with debug_mb large enough that the log is not overflowing and enough debugging enabled to identify why that file is not found.

Typically writing debug logs into Lustre is not smart, but this could be on a different MDT, and is bulk IO so not going to generate a large amount of logs.  If the volume of logs is an issue during the scan, it would be possible to have one or more processes running on other client(s) that are grepping those logs for interesting pathnames and FIDs from perfuse/Stress/Sli-04-Rep-31.dcm and otherwise deleting the log files that.

That would at least tell you if that directory and file are being processed during the scan or something is causing them to be skipped entirely. 

Alternately, could that specific directory be moved to a new directory tree that is just as deep, but in a different parent directory?  Like:

# mkdir /oak/stanford/groups/msalerno/Archive_Rivanna_Data/QuantitativePerfusion/Results/Pig/010_BL/MOCO/perfuse/Stress.new
# rsync -a /oak/stanford/groups/msalerno/Archive_Rivanna_Data/QuantitativePerfusion/Results/Pig/010_BL/MOCO/perfuse/{Stress,Stress.new}/
# lfs mkdir -i 5 /oak/stanford/test
# mkdir -p /oak/stanford/test/msalerno/Archive_Rivanna_Data/QuantitativePerfusion/Results/Pig/010_BL/MOCO/perfuse/
# mv /oak/stanford/{groups,test}/msalerno/Archive_Rivanna_Data/QuantitativePerfusion/Results/Pig/010_BL/MOCO/perfuse/Stress
# mv /oak/stanford/groups/msalerno/Archive_Rivanna_Data/QuantitativePerfusion/Results/Pig/010_BL/MOCO/perfuse/{Stress.new,Stress}

and then run "lfs find -O 96 /oak/stanford/test".  That should have the same number of directory levels and remote MDT, and exactly the same files in that subdirectory, but far fewer files/directories to traverse/debug and would help isolate whether the issue is with the file itself, or with the traversal process, or something else.  The Stress directory could be moved back to its original location at Stress.old if that avenue of exploration doesn't hit the issue, and presumably restore the problem to its (mostly) original state.
