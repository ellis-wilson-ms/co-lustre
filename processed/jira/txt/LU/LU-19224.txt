--- Description ---
There have been several instances of hard-to-debug issues in the network where clients have mismatched TCP/RoCE MTU.  This still allows "lctl ping" to work, if the ping packet is smaller than the MTU of any peer, but larger messages can fail to transmit and it causes intermittent issues in the system.

It would be good for LNet to automatically detect and report an error if there is an MTU mismatch between peers in the same LNet network.

I think this would be the easiest to verify and report a clear message to the user, like "local interface 192.168.10.215@tcp0 (enp0s3) has MTU=9000 but peer 192.168.10.3@tcp0 has MTU=1500" or similar, and will cover 95% of the cases.

Sure, but I suspect this is a less common case than some random client trying to mount with a badly configured interface.  This could be tested with a "jumbo ping" and/or bulk read+write to test the connections (one for each conn_per_peer?) after the endpoint MTU is verified.

--- Comment (ssmirnov, Fri, 1 Aug 2025 20:09:56 +0000) ---
There's lnd-level negotiation which allows the peers to settle on the value of certain lnd parameters for the connection based on what is advertised by each peer.

I'm not sure if MTU size falls into the same category because it is something which is configured on the system, externally in relation to LNet. This makes sense, because generally LNet resides only on a subset of network nodes which take part in delivering LNet traffic - besides Lustre server, client and LNet router, there also likely are switches in the network, at least, and local LNet configuration has no effect on non-LNet network components, so MTU size can be set differently on the switch to the same effect.

So we can pretend that MTU is a "read-only" lnd parameter and fail connection attempts on mismatch, but if both endpoints advertise jumbo MTU size and any one of the switches on the path has a smaller MTU configured, there's going to be the same problem.
