--- Description ---
I'll repeat the comment I left in LU-5834:

While it's ironic that it was a member of the HYD team that originally requested this change, I wonder if it's really the behaviour we (the royal we, not the HYD we) desire.

It seems like a good idea on the face of it, but it has undesirable side-effects in certain use cases.

But this really does not feel like the element of least surprise.

A lesser surprise to me would be for the installation of the client to just fail with an error that it conflicts with the server such that the admin realises he needs to remove the server first and then install the client.

--- Comment (adilger, Fri, 14 Jul 2017 20:44:35 +0000) ---
Would it be enough to have a quick fix to change lustre.spec.in:

-Obsoletes: lustre-client 

so that the current lustre RPMs do not obsolete current lustre-client RPMs, but they obsolete older versions?  However, if you are configuring both YUM repos then you may have problems installing packages that appear in both repos, but if that worked previously then it should continue to work.  Also, we should strongly consider having only a single set of RPMs for userspace tools, tests, sources, etc.

I'd be OK with the above as a quick fix, but my preference is that we move toward our longer-term packaging goals, which is to go to "patched" and "patchless" kernel builds, which include both the client and server components.  Currently, the main difference would be that "patched" builds would include support for ldiskfs project quotas.  Would that be workable for IML?  I don't think it is installing clients in most cases (except maybe HSM Agent nodes, or possibly NFS gateways) so installing the server modules should be fine.
