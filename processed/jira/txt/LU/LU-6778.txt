--- Description ---
John Hammond pointed out that the seq_show functions that are used for the nodemaps don't iterate, and so they won't work if the output data is more than the seq_file limit. They need to be redone with large nodemaps in mind.

--- Comment (JIRAUSER19288, Tue, 18 Mar 2025 11:47:31 +0000) ---
I think the seq_file limit is its "size" field. "seq_printf" may return -1 in case "count" (the printed information) is larger than the "size" of the seq_file (see here). 

I believe this ticket may have come up from the way nids are printed. In this case, there is indeed a check in its "seq_show" to not overflow the seq_file's size field, see in this function of the patch. Nevertheless, this does not use seq_printf() to print the same way nodemap functions do. 

I have also run a few tests with a large output (> 1.4 million characters) from the same seq_show function. Depending on the amount printed, the seq_file's size field grows dynamically (doubling each time, starting from 4096 bytes and, in this test case, 2 MiB). I'm not sure if there is a limit before running out of memory, but given these observations and if this is enough headroom, this may not be an issue.

--- Comment (sebastien, Tue, 18 Mar 2025 09:28:23 +0000) ---
As far as I can see, there were no changes in nodemap_lproc.c in this direction since 2015, and seq_show is still in use to dump nodemap information. Although I am not sure what the seq_file limit is, it seems the seq_file interface reads data in chunks defined by the page size. 

Among all the nodemap properties, only id mappings and NID ranges can be very large, the other properties being single values or paths limited to PATH_MAX.

In my testing, I managed to add more than 1 million id mappings, and more than 1 million NID ranges to a nodemap. Then I was able to dump the corresponding values, without being limited or truncated.

So I was not able to hit any limitation in that area.
