--- Description ---
lfsck should be updated and take EC mirrors into account so that they are set correctly. For instance, this can include checking for (and more):

	Bidirectional link between data and parity mirror is correctly set.

	Parity and data component extents/stripe sizes match.

	OST uniqueness between paired data/parity components (if applicable).

	Parity stripe count and number of actual parity stripes are consistent with `lcme_dstripe_count` `lcme_cstripe_count`, and data stripes.

Note that lfsck should not check that parities are correct or recompute them, since this should be done via lfs mirror verify and lfs mirror resync, respectively.

--- Comment (adilger, Thu, 2 Apr 2026 05:12:27 +0000) ---
I think the most common case where EC will be needed is if the MDT file layout is corrupted (or more likely lost completely), and LFSCK will try to reconstruct the layout based on the information in the trusted.fid xattrs on each of the OST objects.  The OST objects will get the component ID stored in filter_fid.ff_layout.ol_comp_id, so the data and parity stripes will be put into separate components, but there is no way (AFAIK) that the OST object knows whether the component is for parity or data, or what the dstripe_count and cstripe_count are.  It might be possible to extrapolate this based on the stripe counts of the data and parity components (e.g. comp1 stripe_count = 48, comp2 stripe_count=12 gives 8+2), but this would be unreliable.  It would be better to send dstripe+cstripe to the OSTs with the write RPCs to store in filter_fid on the first write.

It might be possible for LFSCK to determine that a trusted.lov layout is corrupt/inconsistent, but I don't think would be easy for LFSCK to repair the file layout by itself because it couldn't know what parts of the layout are correct and what parts are wrong (unless egregiously so).  However, it could use the layout fragments stored on the OST objects to check layout correctness and possibly repair the MDT layout in this manner.
