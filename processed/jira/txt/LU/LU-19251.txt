--- Description ---
In order to make policy decisions on directories in an HSM-archived directory tree, there should be some state/flag stored on the directory itself that it was created as part of an HSM restore operation, or is part of an archived directory tree.

For example, if a directory is created because an archived bucket is restored and using "/" or "\" separators in the object names to create a directory hierarchy, then it would be useful to add a marker (e.g. flag or trusted.hsm xattr) on the directory. 

Since this HSM state on the directory needs to be understood by the MDS when a file or directory therein is being accessed/modified, it cannot be an arbitrary xattr stored on the file without adding overhead to each operation.

--- Comment (adilger, Thu, 11 Sep 2025 17:11:37 +0000) ---
As a starting point, I think we should only look at setting HS_EXISTS on directories that are part of an HSM imported tree (set via the Data Mover/copytool during directory tree creation, or manually afterward for existing filesystems), and not use HS_ARCHIVED or other HSM states to indicate anything about the state of the files in that tree.  This is enough to determine that the directory is part of an archive, and allow directory renames to be blocked.

This will almost certainly require some Lustre code changes to allow calling "lfs hsm_set/state/clear DIRECTORY" to save the state on the directory instead of only on a regular file.

--- Comment (adilger, Mon, 11 Aug 2025 17:51:57 +0000) ---
Note that this "part of HSM archive directory tree" state should not be confused with "this directory tree is archived/stubbed" (LU-13024), since the directory and its subdirectories would not need to be released for this state to be useful.

However, it may be that this state (in trusted.hsm?) could be used as the initial part of a directory stub, but the state could only be HS_EXISTS and/or HS_ARCHIVED (if all of the files in that directory are known to be archived) and never HS_RELEASED (because we don't yet support the HSM coordinator to automatically create the stubs in a directory when it is accessed, though this would definitely be useful).
