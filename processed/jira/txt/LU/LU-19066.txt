--- Description ---
To improve OST object allocation and minimize data and mirrors and EC parity sharing the same fault domain (as much as possible) it would be desirable to identify independent fault domains for Lustre OSTs that the MDS can use during layout creation or extension.

The MDS will always avoid allocating file stripes on the same OST, unless layout overstriping is used. The MDS can already identify whether OSTs are on the same OSS by comparing the (current) peer NID(s), and should avoid allocating mirror and EC stripes on OSTs sharing the same OSS nodes as the data stripes.  The MDS can additionally identify whether OSTs are in the same HA failover domain (controller pair) by comparing the OST's failover NIDs, and should prefer not to allocate mirror and EC stripes in the same HA group, when possible.

To maximize file-level availability (e.g. rack-level fault tolerance) and data redundancy it is necessary to provide additional fault domain information to the MDS that Lustre cannot determine for itself.  One option would be to add a "domain" tunable parameter (possibly a rack unit with the same PDU/switch, or something else), which is a single 32-bit integer that can be set on an OST and MDT (and passed to the MDS/client with the connection data) that indicates which fault domain an OST is in (and presumably cannot escape from due to hardware-level constraints).

The domain integer should primarily indicate whether an OST (or MDT) is in the same fault domain as another OST/MDT, or if it is in a different fault domain, and no particular semantic should be required beyond this.  However, a more sophisticated domain handling could encode information such as building.row.rack.pdu into 8-bit components of the integer and then use the Hamming Distance to separate the data and parity of stripes in a RAID set to reduce locality and single points of failure without having to actually understand the details of how these values are encoded. 

Then, when the MDS is allocating OST objects for an FLR-EC file, it should preferentially select OSTs with different domain values for each data and EC/mirror stripe (within the other constraints of pool, stripe_count, space, etc).  Similarly, when extending a RAID-0 file with an EC/mirror component, the MDS should prefer to add stripes on OSTs that do not overlap with the same domain as the data stripes as much as possible.

In some cases, pre-existing RAID-0 files may not have stripes on different fault domains, nor even on different OSS nodes (which may have been selected depending on availability and capacity at the time of allocation), so with smaller OST counts it may not be possible to fully isolate the data and parity stripes into different top-level fault domains, but the MDS object allocator should strive to minimize the overlap of the domains, HA failover groups, and OSS nodes, within the constraints of OST pools (e.g. NVMe vs. HDD).  In such cases, it is better in the long term to restripe such files to separate the data objects into different fault domains so that the parity can be added to make the data resilient, but that is an administrator action and the EC code should do the best it can with the resources available.  

There may also be some benefit even from EC parity that overlaps the same OSTs as the data stripes, if no other options exist.  It would still be possible to achieve resilience against any single-OST failure with 2-parity EC.  For example d=[0,1,2,3,4,5,6,7] and p=[0,4] would allow any single OST to be unavailable, or any two OSTs other than [0,4], while still maintaining data accessibility and reconstruction.  Using 4-parity EC in this example (e.g. "p=[0,2,4,6]") would allow any 2 OSTs to be unavailable, or any one of p=[0,2,4,6] plus any three of d=[1,3,5,7] at 50% data overhead vs. 100% overhead from mirroring, though this is still only half as efficient than having the data and parity stripes on disjoint OSTs/domains.

For FLR-EC with widely-striped files, the EC parity stripes only need to be on unique OSTs within a single RAID-Set (e.g. 8+2) and not necessarily across all OSTs in the filesystem.  This would allow a 40-stripe file on a 40-OST filesystem (e.g. 5 controllers x (4 OSS x 2 OSTs/OSS)/controller) to still add robust EC by ensuring that the 2 EC parity stripes do not overlap with the 8 data stripes in that RAID-Set, even though the EC OSTs may overlap with other data ECs within that file.

--- Comment (JIRAUSER19283, Thu, 14 Aug 2025 01:15:50 +0000) ---
Pushed 8 patch series to review, starting with https://review.whamcloud.com/c/fs/lustre-release/+/60711

Need to add automated tests. And when a final series lands need to add documentation for this to the wiki.

 

This series introduces a failure domain field that can be set through :

lctl set_param osp.lustrewt-OST0000-osc-MDT0000.failure_domain=0x55

It is stored persistently.

If not set, or if set to 0, it means no failure_domain is set for this OST and the old behavior for stripe selection during file creation is kept.

IF during stripe selection an OST with failure_domain set is selected, then future comps for this allocation will mark not just this OST to be avoided

but all other OSTs with the same failure domain setting too.

Unlike other considerations during stripe selection, failure domain is about integrity and not about performance or fair distribution of allocated space across OSTs, so allocating onto a conflicting OST is not allowed.

If not sufficient different failure domains are available, the allocation will fail and this the file creation will fail with an error.

 

Example:  10 OSTs configured across 3 different failure domains cause a 4-mirror file creation to fail like:

$ rm /mnt/l/sahlberg/test; lfs mirror create -N4  /mnt/l/sahlberg/test

lfs mirror mirror: error setting 352-byte layout on '/mnt/l/sahlberg/test': No space left on device (28)

lfs mirror mirror: lfs create: cannot create composite file '/mnt/l/sahlberg/test': No space left on device (28)

 

Right now it allows OSTs with failure_domain==0 to be mixed with OSTs with failure_domain!=0 and that is probably good enough for a first iteration.

We would need to think about what does it mean if some, but not all, OSTs have been configured with a failure_domain.

My thoughts that this would be a configuration error. Either all OSTs should have failure_domain configured or none. If only some have failure_domain configured I currently think in the lines of configuration mistake, stripe allocation result is undefined.

It feels like the natural place for this would be a setting on the individual OSTs as I think the initial text suggests. But conceptually an individual OST could belong to multiple different failure domains, for example you have failure domains in front of the OSS such as rack or power dependencies but also potentially failure domains that come from properties behind the OST such as a single disk array hosting volumes for multiple OSS.

For initial implementation and keep things simple a single integer that is configured by the administrator is likely sufficient.

But with that, a single integer in OST.domain we would still have I think two different failure domains with two different priorities to account for when allocating blocks to a file.

There would be an implicit failure domain, like today, based on OST to try to avoid allocating redundancy blocks from the same OST if possible.

But then there would be the explitic (optional) failure domain that is configured by the admin for each OST that would also be taken into account when deciding on where to allocate blocks.

And when allocating blocks it would have to take into account both these two failure domains. Additionally it feels like the impiicit failure domain would have higher priority than the explicit (it is more important to keep the blocks on different OSS than on different racks).
