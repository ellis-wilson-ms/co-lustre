--- Description ---
There were several reports of Lustre reexport via nfsv4 having a stale metadata problem that initially was determined to be lack of nanosecond timestamps support, but upon further research it looks like starting from about Linux kernel v5.5 the new i_version inode field is supposed to help even with coarse timestamp resolution.

 

Here's wha thte comment in linux/iversion.h says:

 * The inode->i_version field:
 * ---------------------------
 * The change attribute (i_version) is mandated by NFSv4 and is mostly for
 * knfsd, but is also used for other purposes (e.g. IMA). The i_version must
 * appear larger to observers if there was an explicit change to the inode's
 * data or metadata since it was last queried.
 *
 * An explicit change is one that would ordinarily result in a change to the
 * inode status change time (aka ctime). i_version must appear to change, even
 * if the ctime does not (since the whole point is to avoid missing updates due
 * to timestamp granularity). If POSIX or other relevant spec mandates that the
 * ctime must change due to an operation, then the i_version counter must be
 * incremented as well.

 * Not all filesystems properly implement the i_version counter. Subsystems that
 * want to use i_version field on an inode should first check whether the
 * filesystem sets the SB_I_VERSION flag (usually via the IS_I_VERSION macro).
 *
 * Those that set SB_I_VERSION will automatically have their i_version counter
 * incremented on writes to normal files. If the SB_I_VERSION is not set, then
 * the VFS will not touch it on writes, and the filesystem can use it how it
 * wishes. Note that the filesystem is always responsible for updating the
 * i_version on namespace changes in directories (mkdir, rmdir, unlink, etc.).
 * We consider these sorts of filesystems to have a kernel-managed i_version.
 *
 * It may be impractical for filesystems to keep i_version updates atomic with
 * respect to the changes that cause them.  They should, however, guarantee
 * that i_version updates are never visible before the changes that caused
 * them.  Also, i_version updates should never be delayed longer than it takes
 * the original change to reach disk.
 * 

 

With this in mind the simpliest way here is probably to set SB_I_VERSION for Lustre and then update the blocking AST to handler in llite to increase i_version for directories when an update (or update and some other bits) lock loss. Since inode normally does not go away, a re-lookup and re-obtaining the lock would find the inode and reuse it with the now increased i_version.

It's unclear how actual monotonically increasing i_version is important in practice.

 

Alternatively we can store i_version on disk (We already do for ldiskfs backend) and then transfer it to client, but that constitutes a protocol change for a not very clear benefit.

Further testing and investigation indicated that updating inode i_version alone is not enough to achieve ns v4 properly working because it's not propagated anywhere.

The reason is in fs/stat.c::vfs_getattr_nosec():

        if ((request_mask & STATX_CHANGE_COOKIE) && IS_I_VERSION(inode)) {
                stat->result_mask |= STATX_CHANGE_COOKIE;
                stat->change_cookie = inode_query_iversion(inode);
        }

so if we don't set SB_I_VERSION the kernel would not propagate it. ext4 sets the flag.

But in fact that same function would also call inode's i_op->getattr method and that provides us an opportunity to do the same thing.

In fact ceph does it like this in their getattr:

        if (request_mask & STATX_CHANGE_COOKIE) {
                stat->change_cookie = inode_peek_iversion_raw(inode);
                valid_mask |= STATX_CHANGE_COOKIE;
        }

in a way i_version is not very important I think (if we don't do clustered export which nfs v4 does not allow due to statefulness anyway).

Imagine nfs cached something about an inode, then the dlm lock was dropped on nfs server/lustre client. (this might not might not be due to server side changes - could be lru or whatnot), we increase the in memory i_version anyway and as long as that inode does not disappear from cache we'll find it later and even if there were multiple changes - just the different value is enough to invalidate the cache on the nfs client inquiring about this inode - things still work normally.

Overall as an nfs kludge that seems to be an ok workaround IMO, vs creating all sorts of compatibility nightmares with protocol changes to properly track i_version (note that zfs backend totally does not export it either and would need changes)

--- Comment (adilger, Tue, 5 Aug 2025 22:37:46 +0000) ---
the simpliest way here is probably to set SB_I_VERSION for Lustre and then update the blocking AST to handler in llite to increase i_version for directories when an update (or update and some other bits) lock loss.

The osd-ldiskfs inode->i_version field stores the transno for the last modification to that object.  This allows VBR to determine if the inode is in the same as it was when a replayed RPC was originally sent, so we cannot change the version that is stored on the OSD side.

The client does get the transno back in case of any modifying RPC sent to a file, which should match the inode version at the time of the modification, but I'm not sure if a non-modifying RPC will return the current transno of a file.

I'm not very keen on SB_I_VERSION updating the client i_version to some random values on every inode change, since it would get outdated compared to the MDT inode version, and at the same time not take into account the updates to OST objects that might happen on other nodes.

I think the blocking AST handler would be a minimal way to notify the client that the inode has changed, but if the DLM lock expires due to LRU age and then is changed the client will not know about it.  It would be best if there was some way to update the client version when the client gets a new DLM lock, to detect if the file was changed while it didn't have any lock.
