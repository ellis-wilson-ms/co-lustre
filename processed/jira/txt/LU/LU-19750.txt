--- Description ---
mdt_changelog_allow() has a check preventing "new" clients (after 2.17+) from accessing changelog if no correct credentials are provided in form of mdt_body structure.

which is not provided because llog_client_open() is used with nameless changelogs and the client can't access mdt_body buffer set after undefined name buffer:

[   39.512638] LustreError: 8576:0:(llog_cat.c:395:llog_cat_id2handle()) lustre-MDT0000-mdc-ffff8d05dc1a4000: error opening log id [0x1:0xa:0x0]: rc = -13
[   39.513001] Lustre: 8576:0:(llog_cat.c:839:llog_cat_process_common()) lustre-MDT0000-mdc-ffff8d05dc1a4000: can't find llog handle [0x1:0xa:0x0]: rc = -13
[   39.513395] LustreError: 8576:0:(mdc_changelog.c:326:chlg_load()) lustre-MDT0000-mdc-ffff8d05dc1a4000: fail to process llog: rc = -13

This causes sanity test_160a to fail:

lfs changelog: cannot access changelog: Permission denied
 sanity test_160a: @@@@@@ FAIL: MKDIR changelog mask count 0 != 1

--- Comment (sebastien, Fri, 9 Jan 2026 14:04:15 +0000) ---
Obviously the intent of the original patch ("LU-13064 sec: check permissions for changelogs access") landed as commit v2_13_53-103-g4e8fcee92d 6 years ago was to have this compatibility issue resolved after 2.17, but that didn't happen and the compatibility needs to be pushed out again. I also see in this comment on the original patch that we decided against adding an OBD_CONNECT2_CHLG_PERMS connect flag, and patch https://review.whamcloud.com/36989 ("LU-13064 sec: reserve flag for changelogs permissions") was abandoned at that time.

But the discussion was not exactly about the same part of the code. At that time we were considering the use of the connection flag on the client that calls llog_client_open(). Now we would use it on server side to decide if the fact that the client did not send a body with the request is acceptable or not.

--- Comment (adilger, Thu, 8 Jan 2026 21:52:52 +0000) ---
I pushed the second patch with the OBD_CONNECT_LLOG_UCRED flag to give us an alternative to allow better error handling in an earlier release (i.e. 2.18 or possibly even 2.17) for this case, but I'm not sure if it is noticeably better than the other patch which only bumps the version check.  

In some respects, this just adds another compatibility hurdle with the CONNECT flag, but it allows the MDS to return an error earlier for clients that should be able to fill in mdt_body, but do not for some reason.  I don't know if that is actually useful in practice.  It could be further improved to have a tunable to disallow older clients to access llogs without the mdt_body ucred info.
