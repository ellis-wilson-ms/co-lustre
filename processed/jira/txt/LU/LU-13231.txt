--- Comment (tappro, Sun, 20 Oct 2024 13:01:37 +0000) ---
Note, there is different cases when 'mgc lock leak' - note 'mgc' here as client locks - and 'mgs lock leak' which is about locks on server. This ticket was originally about 'mgs' but contains all test_901 failures linked after all.

First of all this doesn't hit master branch clients since 2022, all failures for 'mgs lock leak' in Maloo are about b2_14 client only. So I'd say this is b2_14 issue at the moment.

Exascale clients are failing with 'mgc lock leak 2 != 4' and that issue looks false because client has always at least 4 MGC locks - 'fsname-sptlrpc', 'fsname-client', 'fsname-cliir' and 'params', so if it reports oldcount as 2 - that is already false report.

dropping all unused locks from namespace MGC10.240.41.137@tcp 

so all 4 config locks were canceled upon umount and then all 4 were occurred after mount again. The problem is that two lock were canceled and not yet re-queued when test checks locks count, so it finds only 2 locks though there would be 4 in normal conditions.

This is test issue probably related to exascale mount process which differs from master, anyway test can't prove amount of locks count is stable when it takes it. Technically test should make sure none of these config locks are being re-queued at the moment. Probably just pause after mount would be enough, to make sure all configs were processed and all is settled.

So at the moment this bug, as it described, exists only in b2_14. Though test itself in master and exascale might need update to make sure lock count is stable
