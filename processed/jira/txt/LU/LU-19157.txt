--- Description ---
Administrators might want to immediately prevent access to Lustre for a particular client, based on criteria such as client activity or generated load on the file system.

In addition to the ability for administrators to shut off this client, it would also be useful to be able to deny this client access to data administratively. This would provide a stronger ability to protect access to Lustre, especially in cloud environments where Lustre admins may not have the ability to interact with or shut off client nodes.

In order to address this requirement, we could add a new 'banlist' nodemap property. So once a storage admin has identified clients from a given nodemap that must be blocked, they are able to add theirs NIDs to this ban list. The banned clients would still be considered part of that nodemap, but all file system accesses from those clients would be blocked. This means not only new mounts from the banned clients would be blocked, but also existing mounts would not be permitted to access Lustre anymore. Later on, as the clients are removed from the ban list, they would be able to mount Lustre again, and get back access to Lustre.

--- Comment (adilger, Mon, 7 Jul 2025 23:15:42 +0000) ---
I would suggest that "redeemed" (while a good descriptive word) is not closely associated to "banned", and an unfamiliar reader may not associate those terms together.  While not proper English, "un-banned" would be easier to understand?

Also, CWARN() has the traditional "debugging" output (timestamps, function name, filename, line number), but since this is more of an administrative/expected result, then LCONSOLE_WARN() might be more appropriate/pretty?

As for returning the "banned" status to the client, there is pb_flags (and possibly pb_op_flags) that could be used.  pb_flags holds "PTLRPC_MSG_*" and has lots of space.  While pb_op_flags applies to a specific RPC type (mostly used for *_CONNECT) I think this is generic for all RPC types (not just connect) so should go in pb_flags like:

#define PTLRPC_MSG_CLIENT_BANNED 0x0100

Obviously we don't want the client to print an error continuously when it is banned, but it might make sense to print something periodically (e.g. every 6h) if so that this appears in the logs if someone is wondering why their client doesn't work.  For old clients that don't understand the PTLRPC_MSG_CLIENT_BANNED they would just get -EPERM errors and not print anything, but this would be printed on the server console.
