--- Description ---
The OST_PUNCH RPC (as generated by "truncate -s" commands) does not set the UID/GID/PROJID stored on an OST object:

[adilger@mookie ~]$ > /myth/tmp/adilger/ost4/fallocate2
[adilger@mookie ~]$ truncate -s 1M /myth/tmp/adilger/ost4/fallocate2
# debugfs -c -R "stat O/340000400/d$((0x194e6 % 32))/194e6" /dev/vgmyth/lvmythost4 
debugfs 1.47.1-wc1 (18-May-2024)
Inode: 114772   Type: regular    Mode:  07666   Flags: 0x80000
Generation: 2361069134    Version: 0x00000069:0000007f
User:     0   Group:     0   Project:     0   Size: 1048576
:
Extended attributes:
  lma: fid=[0x340000400:0x194e6:0x0] compat=8 incompat=0

I haven't yet looked if this is because the client is not sending the UID/GID/PROJIDs in the OST_PUNCH RPC, if it isn't setting the appropriate OBD_MD_FLUID, ... flags in the obdo.o_valid field, or if the OST_PUNCH handling on the OST is not processing these fields to update the filter.fid xattr on the object.

--- Comment (adilger, Wed, 26 Feb 2025 22:15:15 +0000) ---
The ticket LU-16265 also describes automatically repairing OST objects that are missing a valid PROJID.

--- Comment (adilger, Wed, 26 Feb 2025 20:54:23 +0000) ---
To retroactively fix this on existing OST objects, it would be best if all single-object OST RPCs (OST_GETATTR, OST_READ, OST_SYNC (with FID), OST_LADVISE, OST_SEEK) sent the UID/GID/PROJID of the file to the OST in every RPC and the OSS used these to set the object inode IDs if the S_ISUID, S_ISGID, and S_ISVTX (with #define S_ISPRJID S_ISVTX?) flags are not set (see ofd_attr_handle_id() for details).

Since these are read-mostly RPCs, we probably do not want to be blocking these threads with journal transactions to update the inode, so it probably makes sense to push these items to a separate work queue that can do the updates.  While there may be a flurry of activity for an existing filesystem that was just upgraded to a new version that adds the ability to fix these objects, under normal operating conditions there should be relatively few such updates needed.

--- Comment (adilger, Wed, 26 Feb 2025 20:44:13 +0000) ---
There may be other RPCs that are not properly associating the UID/GID/PROJID with a file, and that should be investigated, but I think that OST_PUNCH is the main offender since it is allocating space.  I have verified that OST_WRITE is handling this properly:

Note the Mode: 07666 change to Mode: 0666 that indicates the UID/GID/PROJID have been set.

Calling chown, chgrp or "lfs project" to change the object ownership via OST_SETATTR also properly sets these IDs and clears the corresponding bit in the mode to indicate proper ID assignment.
