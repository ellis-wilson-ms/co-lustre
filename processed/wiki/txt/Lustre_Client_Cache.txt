In order to improve performance, the Lustre client kernel module implements some caching mechanisms.

Memory caching
The Lustre client kernel module uses dynamically allocated memory buffers when reading and writing files. They are configured and monitored using appropriate lctl parameters.

The Lustre cache is totally coherent across clients, as long as the writes make it into Lustre (not using buffered fwrite() in glibc).

Read cache
Possible values
By default, the client read cache size depends on the available system memory. It is dynamically expanded or reduced when the system load changes. This is calculated with the below formula:
New cache size = (Available memory + Current cache size)/2

The read cache size will not go below 64 MB. You can set any value under the system’s full amount of memory.

Configuration
The cache information and usage statistics are available through lctl:

$ lctl get_param llite.*.max_cached_mb
llite.testfs-ffff9558aaa84000.max_cached_mb=
users: 7
max_cached_mb: 31688
used_mb: 0
unused_mb: 31688
reclaim_count: 0
max_read_ahead_mb: 2048
used_read_ahead_mb: 0
The cache can also be set to a static value using this same parameter path:

lctl set_param llite.*.max_cached_mb=12G

Write cache
The write cache will store data written to files as to not saturate the network with little I/O. Dirty data will be written as soon as a full RPC is available for a file (4MB), or when another client tries to read the file, or the data gets too old (30s, managed by the kernel VM), or any of the cache limits are hit.

Possible values
The write cache size is static and can be set from 0 to 2047 (2GB). This is on a per-OST basis, and the memory footprint is proportional to the OST count.

Configuration
The size of the cache is accessed and set through the OSC parameters:

$ lctl get_param osc.*.max_dirty_mb
osc.testfs-OST0000-osc-ffff9558aaa84000.max_dirty_mb=2047
osc.testfs-OST0001-osc-ffff9558aaa84000.max_dirty_mb=2047
osc.testfs-OST0002-osc-ffff9558aaa84000.max_dirty_mb=2047
osc.testfs-OST0003-osc-ffff9558aaa84000.max_dirty_mb=2047
$ lctl set_param osc.*.max_dirty_mb=1024
osc.testfs-OST0000-osc-ffff9558aaa84000.max_dirty_mb=1024
osc.testfs-OST0001-osc-ffff9558aaa84000.max_dirty_mb=1024
osc.testfs-OST0002-osc-ffff9558aaa84000.max_dirty_mb=1024
osc.testfs-OST0003-osc-ffff9558aaa84000.max_dirty_mb=1024

Flushing the cache manually
You can flush the in-memory cache on the client by running as root:

lctl set_param ldlm.namespaces.*.lru_size=clear
This can also be achieved for a single file by calling fsync() on it after writing.

Disabling the cache
Lustre memory caching can be bypassed altogether by using the O_DIRECT flag when opening files.

Persistent Client Cache
The (PCC) mechanism allows clients equipped with internal SSDs to deliver additional performance for both read and write intensive applications that have node-local I/O patterns without losing the benefits of the global Lustre namespace. PCC is combined with Lustre HSM and layout lock mechanisms to provide persistent caching services using the local SSD storage, while allowing migration of individual files between local and shared storage. This enables I/O intensive applications to read and write data on client nodes without losing the benefits of the global Lustre namespace.
