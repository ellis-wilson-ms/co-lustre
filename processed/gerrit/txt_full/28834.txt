LU-10193 osd-zfs: backup index object with plain format

Lustre uses ZAP to implement index object. When tar the index
object via backend ZPL for backup, it is explained as regular
file, then when untar it, it is not ZAP formatted again, then
the Lustre cannot recognize the 'bad' formatted index object.

On the other hand, each backend FS has its own special format
for index object. Then we cannot migrate the index files from
one backend to another directly.

To resolve such issue, the patch will backup the index object
with plain format to the local '/index_backup' directory with
the name of source index's FID string and ".lbx" postfix when
umount the device.

The format of the backup is as following:
1) header: 512 bytes, including:
   magic:       4 bytes
   count:       4 bytes
   keysize:     4 bytes
   recsize:     4 bytes
   owner_fid:   16 bytes
   padding:     480 bytes

2) body: after the header, <key, rec> pairs one by one.

The backup will be done when server umount. The backup behavior
is controlled via new OSD lproc interface "index_backup". It is
off by default. You can turn it on to enable backup when server
umount via writing non-zero value to such lproc interface.

Patch Set 3: Code-Review-1

It isn't clear to me how this feature can be used?  There doesn't appear to be a user interface for creating these backups, and the comment indicates that this will happen every time the OSD is unmounted?  That isn't practical for filesystems that may have billions of objects in the OI, since there would be multi-GB backup files being created on every unmount, which would consume a large amount of space as well as take a long time to complete (if the admin didn't reboot the node in the middle because they thought the unmount was hung).  In that case, it is possible that the index backup will be truncated, and restoring from that backup will result in corruption.

If the index is backed up on every mount, how/when is it being restored?

Also, how does an existing filesystem know where the index files are located?  From what I can see from reading the patch, it appears they are kept in a list when accessed?  What if the number of index files grows very large?  Is this limited to specific internal index files, or all index files?  Does this include directory index files?

For backup/restore of an ldiskfs IAM index, the existing IAM regular file works OK, but the content of the OI file is no longer valid after the backup/restore.  I expect the same is true for ldiskfs->ZFS migration - that the OI file will not have any valid content anymore, so it would be better to just run OI Scrub to rebuild the OI file after the restore.  Are there other index files that need to be backed up and restored that are not handled by OI Scrub?  It looks like the FLDB is an index, but it isn't clear if there are others?

In the ZFS case, would it be enough for osd-zfs to change the object type of the restored index file (by checking the LMA flag, and verifying the ZAP header) so that the ZFS code would access it as a ZAP again?  For the FLDB there should be no filesystem-specific information in the ZAP that would make it invalid after a backup/restore.  ZFS should really be backed up and restored using zfs send/recv, so it wouldn't need to change the object type very often, only after tar backup/restore of a ZFS filesystem.  The OI ZAPs would also be invalid after a tar backup/restore, so they should be rebuilt after such an operation anyway.

Another option for doing this would be to have osd-zfs be able to import IAM-format files directly, since we already have the IAM code in osd-ldiskfs that could be shared, and there would be no need to "backup" these files for ldiskfs.

Another alternative would be to have the FLDB file be rebuilt from the FIDs assigned to each OSD, as well as their current reservations in the local "seq_srv" files, during LFSCK?  This would also help in the case when the FLDB file is corrupted, independent of whether it was part of a backup/restore or not. 

It would be good if you would discuss design decisions for significant changes like this before spending time to implement them.  Getting feedback during the design stage would avoid time spent implementing some feature and then getting feedback that causes the implementation to change significantly.

Thank Andreas. I will make more detailed explanations about the index backup.

1. Why we need to backup index.
It is mainly used for restoring ZFS backend from file-level backup, the source may be ldiskfs backup, it also can be ZFS backend. When the ZFS is restored via ZPL, the index file will be re-generated as regular file, not ZAP. Then osd-zfs cannot use it. So we need some way to re-generate the ZAP formatted index file. There may be some other way for that, such as you mentioned, we can make the zfs-osd to understand the IAM formatted index directly. But it is not general solution, because it means that the osd-zfs needs to understand other backend file format. If we support new backend in the future, we need more work to update osd-zfs. That is not reasonable. So I prefer the solution that, every backend, in spite of what the backend type is, they all backup their index files to the common formatted regular files. Then the backend only needs to understand such comment format, and then it can be restored from any file-level backup without caring about the source type. On the other hand, when migrate from ZFS to ldiskfs, we need similar work.

2. Which index files to be backup
It is NOT all the index files need to be backup with common format. The oi.xxx needs NOT to do that. The OI scrub will rebuild oi.xxx automatically. Only the up layer visible index files (NOT including dir ZAP, nor the LFSCK related index) need to be backup with common format. Current, there are very limited index files to be backup, mainly including the FLDB, and some quota files. Usually, their size will not be huge.

3. When the backup will be done.
The backup will happen when the server umount. But it is NOT for every time umount. There is a lproc interface to control that. By default, it is disabled, you can enable it when you want to backup. So if you do NOT want to backup, it will not affect your normal system usage, including mount and umount.

Anyway, I will add more detailed comment to make things more easy to be understood.
