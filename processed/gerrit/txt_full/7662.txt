--- Message (Andreas Dilger, 2013-09-23 15:18:21.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

(9 inline comments)

Some major problems with this patch:
- it checks for the DESTROY_PENDING flag on the object, but never sets this flag anywhere
- too many retries
- why is there IO happening on the object when it is being destroyed?  The destroy should only be sent after all clients have closed the file and it is unlinked on the MDS. When the destroy is handled on the OST it should also cause all of the extent locks to be cancelled with a special flag in the callback to force clients to discard their data. This ensures there are not continuous writes happening on a destroyed object.  Please verify if this is actually working correctly.
