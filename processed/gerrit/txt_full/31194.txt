--- Commit Message ---
LU-10406 mdt: invalidate cache upon LDLM lock

given new components (like LFSCK) can access storage w/o LDLM
locking, it's possible to find cache stale. so MDT should
invalidate local cache explicitly once LDLM lock is granted.

also, OSP should invdalite any state requested before invalidation
request as LDLM lock and request can be handled in different
order on the target and source MDTs. for example, get_attr was
sent first, the target MDT serviced it first, but the reply got
delayed for a reason. in the meantime LDLM lock was requested,
granted and received by the source MDT before reply for get_attr.
IOW, we get a state with old (potentially stale) state and LDLM
lock granted.

--- Message (Alex Zhuravlev, 2018-02-07 15:37:05.000000000) ---
Patch Set 1:

(3 comments)

> 1) I do NOT understand why its logic is clear/better than allowing the dt_xattr_get() caller to decide whether cache EA or not?

because one component can decide on locking (MDT) while another components have no idea on locking at all. for example, migration - MDT grabs lock on the sources, doesn't need to grab locks on the targets as they aren't exposed till some last transaction. or writeback cache where a whole disconnected subtree can be protected by a single lock, while cache is still used. why all other components need to be aware of that caching at all? they just work in the provided environment with _storage_. if you bind locks to object, you lose this. what would happen to prefetch mechanism (osp_declare_attr_get(), etc) if we check for locks in osp_attr_get() ? 

> Honestly, I also want to add some new mechanism or new use case to Lustre. I have ever did, but been denied for most of cases. So it is quite necessary to make some documentation to well define and describe our LU stack, API, cache, lock, and so on. That will make Lustre more open and maintainable.

there is Documentation/osd-api.txt. it would be great if more people can contribute to that document in some way, even questions help.

> But from logic view, it looks weird, more like hack. 

not sure why you consider this as a hack: the API is very simple, still rich enough to express what we need and not overloaded with lots of arguments, the components are simple as well. and flexibility is not lost.

>  but here we do that when enqueue lock.
because of lockless LFSCK which can populate cache at any time. so both modes can work well on its own and together: LFSCK can use prefetching, cache and MDT does too - with essentially very few lines of the code in MDT.
