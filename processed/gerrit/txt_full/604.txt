--- Commit Message ---
LU-5 readdir read multiple pages per rpc

add support for readdir to read multiple pages per rpc:
* because client has no idea how many directory pages it can read, it
  tries to read maximum pages each time, but will only store pages
  read from mds into page cache.
* add a flag LDF_COLLIDE to mark a dir page hash collides with the
  next page, and client will remove this page from page cache after
  processing.
* upon readpage bulk io failure, client won't be evicted, and client
  will resend the bulk request.
* support large page size on client: MDS_READPAGE RPC will fill reply
  with page size LU_PAGE_SIZE(4k), and if client page is bigger than
  LU_PAGE_SIZE, several pages will be integrated into one dir page with
  CFS_PAGE_SIZE.

One additional note - now that the MDS is sending multiple pages to the client in the bulk RPC, the chance of network error is greatly increased.  This will cause the client to be evicted needlessly if the bulk RPC fails.

On the OSS code this was fixed long ago, so that if a client fails during a bulk read it is not evicted, and simply retries the operation.  It looks like mdt_sendpage() should also be changed to ignore bulk failures like ost_brw_read() does.

Patch Set 2:

Oleg, I see we need to support large pages, but current 2.x seems lack of this.

In mdt_readpage(), start and end hash is stored in each directory page, if client page is bigger than server page, the hash value read by client should not be able to cover the range the client page really holds. eg, if client page is 8k, twice the size of server page, and this dir happens has 8k bytes of data, so on server side, it's two pages, and each page stores the start and end hash, but client treat them as one page, and will assume the hash range of the first server page as its, finally this should lead to problem.

To fix it, server should know the page size of client, and package directory entries into reply buffers according to client page size.

Please correct me if I misunderstood.

--- Message (Andreas Dilger, 2011-06-06 19:29:44.000000000) ---
Patch Set 2:

In the 1.x code for a readdir with a client request > server PAGE_SIZE the MDS would fill in the rest of the request (beyond i_size) with fake directory entries, one for each block needed to fill the rest of the request.

        bh = NULL;
        if (*off < i_size_read(inode)) {
                int err = 0;
                  
                bh = ext3_bread(NULL, inode, *off>>blkbits,
                                0, &err);
                :
                :
        }
        if (!bh) {
                struct ext3_dir_entry_2 *fake = (void *)buf;
                memset(fake, 0, sizeof(*fake));
                fake->rec_len = cpu_to_le16(blksize);
        }

This is fine for a few pages, but is very inefficient if the RPC size is 1MB and the directory size is only 4kB (which is very typical), since there will always be 1MB of data sent over the wire.

I would hope that for 2.x we can handle this at the client, either by filling in the lu_dirpage structures correctly to address the full hash range that fits into the client-side PAGE_SIZE, or by having the clients/servers agree to pack MDS_READPAGE requests in multiples of 4kB over the wire, with one lu_dirpage stored every 4kB in the client page cache, and the client needs to check (and pad, if necessary) the larger pages.

It may be that for compatibility reasons we need to assume servers are always filling in readdir requests with an lu_page every 4kB, and clients will need to handle this locally by doing lookups within a page (for PAGE_SIZE > 4kB) for hash values higher than the hash range in the first lu_dirpage.

Oleg, the reply dir entries are packed with page size LU_PAGE_SIZE, so the only requirement here is both server and client CFS_PAGE_SIZE bigger than LU_PAGE_SIZE (this should always be true):
* on server size, only LU_PAGE_SIZE will be used to store lu_dirpage inside a *page, and this lu_dirpage will be packed into reply buffer.
* on client, if necessary several lu_dirpage in reply buffer will be combined into one dir page.
