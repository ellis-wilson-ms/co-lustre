To give a bit more insight to anyone interested in reviewing the patch: test_160c was introduced by a patch (https://review.whamcloud.com/#/c/31723/) that aims at giving RESTORE requests priority over other kinds of request (mainly ARCHIVEs).

One of the coordinator's job is to periodically scan an llog of HSM requests and either:
 - send those requests to copytools
 - update the requests' status
 - delete them on completion

A whole scan of an llog takes time, and at some point, it became a bottleneck for HSM. It was then decided that as often as possible, the coordinator would run only a partial scan of the llog, just looking for requests to send. The clean up part (updating requests' status and deleting them) would still be performed regularly (once every `hsm.loop_period', a tunable), but less often. Whether the coordinator does a full or partial scan is controlled by the boolean flag `housekeeping' stored in the `struct coordinator'. Thus, we started calling the longer runs 'housekeeping runs' (at least I did).

An HSM request is stored in a `struct hsm_action_item' (hai) and multiple hais are packed in a `struct hsm_action_list' (hal). At anytime, hals should only contain compatible requests -- requests that can all be sent to the same copytool (they must all share the same archive-id).

The coordinator stores the requests it is about to send in an array of hals. In order to give priority to RESTORE requests, it sometimes has to make space in that array of hals. With luck, there is a compatible hal, in which case, even if the hal is full, it is quite easy to remove a few requests from it and replace them with the RESTORE. But in the event none of the hals are compatible, the coordinator must take a decision: dismiss a whole hal (potentially delaying thousands of requests), and build a new one, from scratch compatible with the RESTORE; or give up on scheduling that RESTORE, at least for now.

When I implemented the feature of RESTORE prioritization, I chose a trade-off: the coordinator would dismiss a whole hal, and pay for it, but only during 'housekeeping runs'. That can be summarized as "if it's cheap, let's do it, if it's not, let's do it... less often".

That's about it for context =)

Now, what is the current patch about? Well, test_260[abc] each check  RESTORE request prioritization in a different context:
 a) it's cheap, let's do it;
 b) it's not cheap, but we might as well do it;
 c) it's not cheap, and it's not the right time do it.

The issue with test_260c is that it is hard to ensure that "it's not the right time to do it" (translation: the coordinator is not doing a 'housekeeping run'). The previous implementation was racy. The new one uses the fact that when the coordinator runs for the first time (eg. after a restart), it always does a 'housekeeping run'. Consequently, it is safe to assume that the coordinator will not do another 'housekeeping run' for `loop_period' seconds (10s in the current implementation).

While this still assumes time doesn't drift too much from one test server to another and that execution time is negligible, the new implementation should yield much more deterministic results.

One might ask why I do not increase `loop_period' to something like 1000 seconds... And it turns out this would be a very valid question to ask and there is really no reason not to do it.
Will do in the next patch set.
