LU-5304: do not memset after LIBCFS_ALLOC

LIBCFS_ALLOC already zero out the memory allocated, so there is no
need to zero out the memory again.

--- Message (Andreas Dilger, 2014-07-08 19:52:05.000000000) ---
Patch Set 2: Code-Review-1

(1 comment)

Sorry, I don't think this is correct.  LIBCFS_ALLOC*() itself doesn't set __GFP_ZERO that I can see.  There is OBD_ALLOC*() that is used in lustre/ code that sets __GFP_ZERO, but the LIBCFS_ALLOC_*() calls do not expect the memory to be zeroed.

That said, it would be better to change these LIBCFS_ALLOC() calls to LIBCFS_ALLOC_GFP(..., GFP_NOFS | __GFP_ZERO) instead of calling memset() afterward.

--- Message (Patrick Farrell, 2014-07-08 19:58:36.000000000) ---
Patch Set 2: -Code-Review

Andreas,

Take a look at #define LIBCFS_ALLOC_POST(ptr, size)                                        \

I may be reading this incorrectly, but as I understand it:
LIBCFS_ALLOC becomes:
        LIBCFS_ALLOC_GFP(ptr, size, GFP_NOFS) or LIBCFS_ALLOC_GFP(ptr, size, 0)

And LIBCFS_ALLOC_GFP uses LIBCFS_ALLOC_POST, which zeroes the allocated memory.

The only exception is the !__KERNEL__ version of LIBCFS_ALLOC_GFP, which I believe is effectively deprecated.
