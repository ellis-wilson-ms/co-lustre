LU-12915 llite: Add discard_list of fids/jobids to debugfs

Add /sys/kernel/debug/lustre/llite/*/discard_list file. The
discard_list shows the fid, jobid, and file name for each file
that has had dirty pages discarded due to evictions.

A discard_list is maintained for each mounted file system. The
maximum number of items in the discard_list is 128. If more files
than that report dirty pages being discarded then the oldest items
in the list are discarded and replaced with the latest occurences.
A discard_list can be cleared by writing any value to the file.

Since the discard_list reports filenames, it is no longer necessary
to include the filename in the "dirty page discarded" warning
message. This solves a deadlock that sometimes occurs in
ll_dirty_page_discard_warn (see LUS-5719). Note: if the file has
been deleted, its name will not be shown in the discard_list.

(7 comments)

I'm not a huge fan of this patch, as it seems to add a lot of complexity for only marginal benefit as it doesn't really fix the need to do fid2path later on (on a FID which may have already been deleted).

My expectation is that this kind of problem is likely going to be investigated a long time after it occurs, and the relevant information in discard_list is unlikely to be available, only what is written to the logs since those are usually sent off-node immediately.

At that point, we may as well just live with printing only the FID/JobID in the console logs and be done with it?  After your explanation about the filenames not actually being printed, and confirmation in the (modern) logs I could find, I don't think we can do much better than just printing a message in the logs, and adding the jobid is a useful extension.

I'm not against refreshing your simple patch https://review.whamcloud.com/35443 to add the JobID and be done with it.

Another option would be to use a kernel workqueue to do the fid2path lookup almost immediately, and print the message to the console log from a different thread context, rather than trying to do it in the ptlrpcd thread?  Something like INIT_DELAYED_WORK() used by ptlrpcd_watchdog_init() (see Documentation/core-api/workqueue.rst for details).  This would be more complex than the simple (pathname-less) console message, but only a few lines longer and more likely to be able to resolve the pathname if it is available.
