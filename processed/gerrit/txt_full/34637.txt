--- Commit Message ---
LU-10092 pcc: security and permission for non-root user access

For current PCC, if a file is left on the PCC cache, it may be
accessible to other jobs/users who would not normally be able to
access it. (That is,  they access it directly on the PCC mount via
FID as the local PCC mount is basically just a normal local file
system.)

This patch solves this by restricting access on the PCC side and
just depending on the Lustre side permissions for opening a file.
So PCC files on the local mount fs are created with some minimal
(zero) set of permissions. Then, when accessing a PCC cached
file, we do the permission check on the Lustre file, then do not
do it on the PCC file. This should render the PCC files
inaccessible except to root or via Lustre.

--- Message (Patrick Farrell, 2019-04-16 16:48:40.000000000) ---
Patch Set 3:

(5 comments)

So the patch looks pretty good, but I think the choice to use the creds of the user who created the mount minus CAP_SYS_OVERRIDE is wrong - I think we need, instead, to use some minimal set.

In particular, it's possible for a user to have CAP_SYS_ADMIN (for mount) but not CAP_DAC_OVERRIDE or CAP_DAC_READ_SEARCH.  (I *believe* that's the pair required to ignore file and directory permissions.).  That would cause PCC to not work, I believe.

So if possible, I think we should use a constructed set with those two for the PCC access..?  Is there any possibility of using a symlink on PCC to escape the PCC mount with these creds?  Or is creating a symlink on PCC disallowed?  (That would be reasonable, there's no point to caching them)

One other thing: Do we care about retaining CAP_SYS_OVERRIDE when accessing PCC for users that have it?  It seems weird to drop it for them.  So perhaps we should check for it and include it?  Or perhaps it doesn't really matter, since the reserved space and quota checks we care about are on Lustre?

Patch Set 3:

(5 comments)

Our permission implementation is borrowed from linux cachefiles and overlayfs.
We are not using the creds of the user who created the mount minus CAP_SYS_OVERRIDE. We use the creds of the user who created the mount (which is usually root) lowing the CAP_SYS_RESOURCE, where CAP_SYS_RESOUCE means that there is no limits with resource usage.

The reason lowing CAP_SYS_RESOURCE is:
we want to enforce a quota limitation of the capacity usage for each user, group, or project to provide cache isolation. An administrator can,  also specify the quota enforcement on the local PCC file system.

There needs to be an update to the "lctl-pcc.8" man page to make it clear that the user running "lctl pcc add" needs to have read/write permission to the PCC cache filesystem, and that the credentials for that process will also be used for later access to all files in that PCC archive.

Separately, this raises the interesting question of whether it is possible for regular users to run "lctl pcc add" on a local filesystem that they have write permission for, something like:

  adilger@client$ mkdir -p /var/tmp/adilger
  adilger@client$ lctl pcc add /var/tmp/adilger

to temporarily cache my own files into a persistent local filesystem?  I think that would be *very* interesting for some users/sites if this could be configured on a per-user or per-job basis without the need for root intervention, and each user might configure and use their own PCC directory instead of having root set up a long-term PCC filesystem on a client.

That would probably need some extra work (e.g. an "lctl pcc new_id" command to return an unused archive ID for the client-local PCC archive), but is something to consider for 2.14.
