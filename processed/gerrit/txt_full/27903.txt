--- Commit Message ---
LU-9409 llite: Add tiny write support

If a page is already dirty in the page cache, we can write
to it without a full i/o.  This improves performance for
writes of < 1 page dramatically.

Append writes are a bit tricky, requiring us to take the
range lock (which we can normally avoid), but they are
still much faster than the normal i/o path.

Performance numbers with dd, on a VM with an older Xeon.

All numbers in MiB/s.

                8 bytes 1KiB
Without patch:  .75     75
With patch:     6.5     153

I've been struggling with one of the theoretical bugs.  My concern is cl_page_clip, where we might clip a page to not include the 'tiny write' data.

It's used in two places:
1. osc_io_commit_async, for lockless io.  But lockless i/o should always be direct i/o, so there should never be any dirty pages, so this should be safe.

2. vvp_io_commit_sync, when we hit -EDQUOT on the async commit.
The fear is that one write dirties the page (say a 512 byte write at an offset of 3072 bytes in to the page), then we do a tiny write, say 8 bytes at 0 offset in to the page.  Then we do another real write at a higher offset (say 3072 bytes at offset 2048) and this write gets -EDQUOT on the first page, causing us to clip this page.

I've been playing with this for a while, and I'm starting to think it's actually impossible to hit -EDQUOT on a second write to a page that's already dirty in the page cache.

Do either of you know if that's correct?  If so, then I just have to figure out updating file size...
