--- Commit Message ---
LU-2182 llapi: Expand layout api to verify against a target file system

Besides validating the layout users create against the limits of
Lustre add additional functions that can validate the layout
against the target file system. To do this two function are added.
The function llapi_layout_pool_absent can be used to ensure the
layout's pool is actually present on the file system. The second
function llapi_layout_ost_index_valid ensures that the user
specified OST index supplied matches to a real OST on the file
system. This method also test if the OST index belongs to the
layout's pool if one is present.

Patch Set 7:

We shouldn't validate layouts, we should provide interfaces that let users build valid layouts.  A bunch of functions to validate various aspects of an already-built layout will create a messy and awkward interface.  How would an application use such an API in practice?  Would it do a parameter search on each attribute, building layouts with various values until it finds a layout that passes validation? That is backwards.  A sane application would query the API to discover valid striping attribute values then build a layout using them. So lets design an interface that sane applications can use.
        
Instead of answering the question "does this layout have a valid OST index?", let's answer "what OST indices are available?"
  
"What pools are available?", not "Does this layout have a valid pool?"          

etc.
