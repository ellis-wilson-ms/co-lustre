LU-8137 utils: fix llverdev for use on regular files

Allow llverdev to work with regular files to verify data contents.

Add a parameter to allow specifying the maximum file size to use.
Allow the use of unit suffix when specifying sizes as arguments.
If the file size is zero, write until the filesystem is full.
Use pread() and pwrite() to avoid having to deal with file offsets.

Add sanityn test_56a/56b to verify proper operation on a large file.

Update llverdev.8 man page to describe options properly.

Test-Parameters: trivial testlist=sanityn env=ONLY=56a,56b ostsizegb=12
Test-Parameters: fstype=zfs testlist=sanityn env=ONLY=56a,56b ostsizegb=12

--- Message (Andreas Dilger, 2021-05-14 21:33:15.000000000) ---
Patch Set 7:

> Right now the test only accepts one offset value. Would it make
 > sense add an option to accept multiple offsets at which the data
 > needs to tested?

It is possible to use "--offset" and "--size" to limit testing to specific subsets of the file if that is needed, and run llverdev multiple times if different specific ranges need to be tested.

llverdev can already run sparse tests ("--partial" => write 1MB chunk per GB of file by default) to reduce IO time without significantly reducing functionality, but I don't want to make it overly complex.

One of the reasons that it finishes all writes first and then the reads second is that this maximizes the chance of catching data corruption (e.g. offsets > 2^31, 2^32, 2^32 * 4KB, ... are aliased to offset 0).  Otherwise, just writing then reading a small section of data may *appear* to work correctly (write and read the same data at "offset_X") without detecting that the data is being written to the wrong place in the file/disk (e.g. "offset_X & 0xffffffff...").

Also, I'd be happy just to get this patch fixed/landed, and we can work on further improvements incrementally after that.  If you have time to refresh it per review comments, that would be appreciated, otherwise it will have to wait until I have time for it again.
