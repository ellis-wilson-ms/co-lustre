LU-11392 tests: check race for llog_process_thread

The patch adds 10h test at llog_test which runs at 60a sanity.
It reproduces a race between llog_process_thread and llog_add.
The llog should be wrapped so it has old data on disk and zero at
bitmap.
1. llog_process_thread reads part of llog at buffer.
1. process a last record, checks the next record fields
2. llog_add adds a record and marks new record at bitmap
1. check bitmap flag and process the old record from buffer

Patch Set 1:

I understand that this is your process at Xyratex/Seagate/Cray, but it is not the process we have for patches landing to master. We don't (knowingly) allow patches that cause the tests to fail to land.  Of course, there are still cases where patches land that cause intermittent failures, but we try to avoid this by running tests multiple times with different configurations. 

There are two main reasons for this:
- we don't want every second patch to cause test failures during "git bisect" as must be the case in your tree
- we don't want people to get used to ignoring test failures

It is easy enough if you want to keep this practice in your tree - you can submit the patches to Gerrit with:

 Test-Parameters: fortestonly testlist=...

to verify the test/patch works, then resubmit them in reverse order to ensure both patches pass testing. 

Alternately, I've also suggested to modify the "test" patch so that it checks for the failure condition and considers this a "pass" for the test, and lack of failure is an error. Then, in the fix patch it will modify the test to consider the failure case an error, and lack of failure to be a pass (of course it also needs to check for the correct client or server versions).

This works OK for many failures, but it wouldn't be possible to have a pre-fix test patch pass testing if the client or server was crashed/LASSERT/etc. In that case it is still better to have the test be part of the fix, or in a second patch if that is easier for you.
