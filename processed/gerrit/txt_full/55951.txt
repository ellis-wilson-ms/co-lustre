LU-14408 llite: add a Lustre inode cache shrinker

This provides a way to shrink the unused inode cache (inodes not
covered by LDLM locks) for a Lustre superblock.

The following command will drop all the unused inodes for <fsname>:
  # lctl set_param llite.<fsname>-*.inode_cache=clear

The following command will disable the unused inode cache:
  # lctl set_param llite.<super>.inode_cache=0
  # lctl get_param -n llite.<super>.inode_cache
  max_cached_mb: 0
  max_cached_cnt: 0
  unused_inode_mb: 0
  unused_inode_cnt: 0

The following command will enable the inode cache without limit:
enabled: 1
  # lctl set_param llite.<super>.inode_cache=1
or
  # lctl set_param llite.<super>.inode_cache=unlimited
  # lctl get_param llite.<super>.inode_cache
  max_cached_mb: unlimited
  max_cached_cnt: unlimited
  unused_inode_mb: 0
  unused_inode_cnt: 100

The following command will limit the unused inode cache to 5MiB:
  # lctl set_param llite.<super>.inode_cache=5M
  # lctl get_param -n llite.<super>.inode_cache
  max_cached_mb: 5
  max_cached_cnt: 3450
  unused_inode_mb: 0
  unused_inode_cnt: 100

(5 comments)

Is the need for this driven because the current interaction between the VFS inode cache and Lustre is broken?  Or was there a desire to have a specific limit on how many inodes can be cached on a client?  

Is there a problem with the client having *too much* memory and not freeing inodes when there are too many inodes, but there is still a lot of free memory?  I don't think I've heard of this, compared to inodes consuming too much memory, but it seems possible today...

In some respects, I think this patch is a sign that the Lustre inode cache memory usage is broken, and/or that LDLM LRU sizing is broken, because inodes should be OK to use a lot of memory if it is available, and shrink when memory is low.  It seems like the "shrink when memory is low" part has been broken for a long times.

Most users will not know the tunables exist, and it just adds more complexity to configure and more overhead to track this list.  Also, the VFS *already* maintains such a per-superblock LRU, but AFAIK there is some problem with VFS shrinking of llite inodes.

My strong preference would be to fix the VFS shrinking of the llite inodes, rather than implementing our own list to work around this problem.

Have you looked into *why* the inodes are not dropped from memory in the first place?  I'm not sure if there is any benefit to keep an inode in client cache after all of the DLM locks are gone.

Lai, you've worked on this client inode cache problem for some time already for some time, can you please update Etienne on what options might exist for finally fixing this issue?  I see LU-13833 has some potential solution, and we are well beyond 3.0 kernels today.

The last patch try to remove the lock contention from syscalls paths: by removing ll_lru_del from ll_iget() for re-used inode, the cleanup will be done by ll_shrink_inode().

For drop inode path (e.g:  ll_lock_cancel_bits() in case of blocking AST), the patch will not cache the inode if there is contention on the LRU list lock.

> Is the need for this driven because the current interaction between the VFS inode cache and Lustre is broken?  Or was there a desire to have a specific limit on how many inodes can be cached on a client?  

No I don't think the VFS inode cache is broken. But the Linux shrinker are call only in case of  low memory. For system with lot of RAM, this could lead to several millions/billions of inodes cached. This could impact the performance of the cache.

To give you some context, this have been seen in production. Some nodes were declare not healthy because they were consuming to much memory after the end of a job (slurm_epilog). To clean the nodes we were using "lru_size=clear" but it does not clear lustre_inode_cache. So now, we reestablish the"drop_cache 2" to clean the node. The problem began to appear after several months.

So the purpose is to clear the inodes of a specific FS with dropping all. And defines specific limit for use cases consumming a lot of inode (for FS scanner like robinhood): this would avoid dropping cache every minute for those kind of nodes.

> Have you looked into *why* the inodes are not dropped from memory in the first place?  I'm not sure if there is any benefit to keep an inode in client cache after all of the DLM locks are gone.

Yes, for most of the cases we return 0 (not drop) in ll_drop_inode() (with inode_cache=1). We advise the kernel to keep the inode. 
Then the entry is kept by the kernel until there is not enough memory or we umount.

I think, it would make sense for some use cases like the creation file in single directory by several node. The inode LDLM lock will be invalidate for every directory update, so it make sense to cache the entry. Another example would the opencache heat count.

> Lai, you've worked on this client inode cache problem for some time already for some time, can you please update Etienne on what options might exist for finally fixing this issue?  I see LU-13833 has some potential solution, and we are well beyond 3.0 kernels today.

If think that the LU-13833 can be a potential solution to connect the Lustre LDLM cache to VFS, it would make the kernel free some unused lock in case of low memory. The kernel inode shrinker can not evict inode protected by LDLM lock (i_count > 0).
