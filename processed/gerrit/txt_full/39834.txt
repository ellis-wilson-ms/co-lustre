Thanks for the log messages (I cannot help wondering why these weren't included in the original bug report on jira....).

It appears that Linux has two slightly different ways to report the address of the peer of a socket.

The getpeername() syscall and kernel_getpeername() function call 
  sock->ops->getname(...., 1)
and this will fail in the connection is in state SYN_SENT or CLOSE.
I cannot see how the state might be either of these after a successful kernel_accept(), and before the socket is closed. But maybe there is some corner case is some kernels which make it possible.

The SO_PEERNAME getsockopt call uses
  sock->ops->getname(..., 2)
which returns the same information, but doesn't check the state of the TCP connection.  So once the information is available, it stays available.  It would (I think) be safe to use this interface and assume it will work if it had previously worked on the same sock.  But we don't have easy access to that interface within the kernel.... unless we just call sock->ops->getname(..., 2) - which is a bit ugly.

So there doesn't really seem to be anything else to fix.  I'm still a bit bothered - we really shouldn't bet getting an ENOTCONN error.  But I'm not going to lose sleep over it.
