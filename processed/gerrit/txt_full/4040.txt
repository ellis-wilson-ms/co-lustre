Johann, I didn't see previous question until now...

Having two scripts is unpleasant, for sure.  How complex is it to have two different "config subroutine" files for the two quota codes, then a single test script that sources one or the other script:

        if [ $(lustre_version $SINGLEMDS) -gt $(lustre_version_code 2.3.56) ]; then
                . $LUSTRE/tests/quota-config-new.sh
        else
                . $LUSTRE/tests/quota-config-old.sh
        fi

and then a common sanity-quota.sh script calls the appropriate routine to do the actual tests?  These scripts could also add to the ALWAYS_EXCEPT lists if some test is no longer relevant.

That would reduce the overall duplication to just the config routine scripts.

Not a must-have, but definitely desirable if this will work.

--- Message (Niu Yawei, 2012-09-20 07:13:44.000000000) ---
Patch Set 1: (2 inline comments)

Hi, Andreas

It's possible to extracting differences into two config scripts and make all tests into a commen script, but that needs to change every tests, it's quite a lot of work.

For the external function tests, such as basic hardlimit, softlimit test, the old test usually needs to tune the qunit size during testing and sometimes iterate the test many times for different qunit size setting, but that's not valid in new quota anymore.

For the internal mechanism tests, such as quota acquire/release, reintegration/recovery test, most old tests are not valid for new quota anymore. I've adapted them to some new tests with same test numbers.
