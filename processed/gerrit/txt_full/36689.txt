--- Commit Message ---
LU-12944 mdd: pass correct xattr size to lower layers

In mdd_iterate_xattrs(), struct lu_buf allocated to store xattr value
can be reused for multiple xattrs, because it is only reallocated if
it happens to be too small for one xattr.
As a consequence, lb_len field does not represent actual xattr's size.
It has to be adjusted when passed to lower layers.

Patch Set 1: -Code-Review

Hmm, going back a bit on this - do we really need the *lu_buf* to be exactly the right size?  Otherwise this is going to cause problems for e.g. mti_big_buf where that buffer may be reallocated on each use because of different xattr sizes.

The point of lu_buf_check_and_alloc() is to have a buffer that is *at least* large enough to hold the requested size.  Maybe we need to fix the few (?) places that are using lb_len to store the exact size of the buffer to pass the data size separately?  That would not be more overhead than having a separate "lb_len" and "lb_alloc_size" in each lu_buf.

Unfortunately, think that there may be a few places that are allocating lu_buf larger than 64KB (or at least exactly equal to 64KB), so using "u16 lb_len" and "u16 lb_alloc_size" to save space may not be safe.

Do you know specifically which lu_buf is having this problem?

Patch Set 1:

Hmmm, it seems my modification has a larger scope than I initially thought.

However, a use case for this is shown thanks to the new test added in sanity.sh. When migrating files and directories to a different MDT, xattrs set on these files and directories will be corrupted because of a wrong lb_len value.


As far as I understand it, in this use case the problem happens in osp_xattr_set(). This function calls osp_oac_xattr_find_or_add(), and then osp_oac_xattr_assignment(), all with the same struct lu_buf obtained from a call to lu_buf_check_and_alloc() from mdd_iterate_xattrs().

osp_oac_xattr_assignment() calls __osp_oac_xattr_assignment(), which sets oxe->oxe_vallen to buf->lb_len. The direct consequence is that if lb_len is greater than the actual length of the xattr, it will end up storing more bytes than we have to.


I think this problem needs more expertise than I have in this area. Anyway, at least the new test in sanity.sh can be kept for investigation and non regression purpose.
