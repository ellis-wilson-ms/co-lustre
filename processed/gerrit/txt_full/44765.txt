Patch Set 8:

Hmm, I was thinking more about how this works exactly, and I believe the current code will not work properly once the debug log hits the maximum file size.

The debug_daemon file is a circular buffer that is written out to a file in the filesystem in binary format directly from the kernel.  This is done to minimize overhead during debugging (no data copies, less binary->ASCII conversion), but with newer CPUs this overhead may be less critical than in the past.

What it looks like this patch implements a process to read the binary debug_daemon output file and convert it to text.  That will *mostly* work, but only until the binary log hits the maximum file size and the log wraps.  Then the binary->text conversion will stop and all new debug data will be lost (AFAICS).

Also, this has the issue that debug_daemon is still writing the binary output file, and presumably the stdout from "lctl debug_daemon -F" will be written to a *second* output file, so it is actually doubling the amount of debug data written to the local filesystem.

For this to work reliably I think the best way is to *NOT* have debug_daemon write to the circular binary log file, and instead have it read the binary logs continuously from the kernel and then parse and write them to the output file.

The simplest (though sub-optimal) option would be to have "lctl debug_daemon -F" continually dump the debug logs to *different* files, like "lctl dk", but *append* the output to the previous file (or stdout).  That would at least avoid the max size wrapping problem from debug_daemon.

Unfortunately, it looks like "lctl dk" is also writing the binary output to a temporary file (directly from the kernel, so we can't just connect it to stdin) and then converting the temporary file to text in userspace.  So using "lctl dk" internally still doubles writes to the filesystem, but at least the temporary debug file will be deleted when it is converted to text, unlike using debug_daemon.

A better solution (though a bit more work), would be to change "lctl dk" to *read* the binary logs from the "dump_kernel" file and then parsing it to text in userspace without writing it to the temporary file at all.  The parsing would need to be done in chunks because the binary logs may be returned out of order due to per-CPU buffers, different interrupt contexts, and circular buffer reuse.  However, parse_buffer() already handles sorting the messages by timestamp, and doing this more often reduces the amount of sorting needed (e.g. once every 0.1s is better than 1s or 5s).
