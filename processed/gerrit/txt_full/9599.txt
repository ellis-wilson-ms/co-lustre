If we are sending 4kB random writes, I'd rather limit the number of niobufs in the message, rather than sending thousands of random pages in a single RPC.

--- Message (Li Xi, 2014-04-28 14:00:24.000000000) ---
Patch Set 1:

Hi Andreas,

I understand the patch is tricky. Do you have any better idea how to fix this?

There seems no easy way to calculate the exact space needed to save niobufs in in get_write_extents(). Thus, limiting the number of it in advance seems impossible. I guess we need to change more codes in osc_send_write_rpc() to fix this problem, right? For example, when there is not enough room to save the niobufs, it fails and retries by sending these pages in different RPCs. What's your opinion?

Thanks!

--- Message (Andreas Dilger, 2014-05-06 20:02:53.000000000) ---
Patch Set 1: Code-Review+1

Actually, I'm not against increasing the OST max request size, but I also don't think that it makes sense to send thousands of random IOs in a single RPC.  It would probably be better performance to submit the random IOs in parallel so that they can be handled by multiple threads concurrently and sorted in the elevator/controller.
