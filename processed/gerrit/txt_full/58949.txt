LU-12511 build: make fscrypt detection work for native client

The current code when built against the native client fails to
detect fscrypt so no support is built into the tools. This
patch makes the LC_FSCRUPT_SUPPORT test against the fscrypt UAPI
header to see if its at a level we can support natively.

--- Message (James Simmons, 2025-05-08 15:46:08.000000000) ---
Patch Set 2:

If fscrypt in the kernel is not enabled then Lustre needs to turn on its own llcrypt version. 

HAVE_LUSTRE_CRYPTO tells us if the kernel supports the fscrypt API at a mature enough level. RHEL8 has a fscrypt version but its not mature enough which is why we have HAVE_LUSTRE_CRYPTO test.

Looking at the test:

if enable_crypto is in-kernel (use native) then test if fscrypt API is mature enough. If its not then we look at has_is_encrypted which tell us we are 4.14+ kernel which is enough for llcryptfs to work. For the native client, which only builds the utilities, we are interested if the kernel supports a mature fscrypt API so the user land tools build correctly. If the native kernel disabled fscrypt then the tools when run would just fail which is fine. Maybe we can harden it more with testing the connect_flags in the tools for test_dummy_encryption?
