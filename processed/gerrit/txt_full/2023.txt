Patch Set 5: Looks good to me, but someone else must approve

Based on the comment in ptlrpc_check_set() for bd_success, and the code in ost_brw_read() itself, I'm still unclear on how this problem could be hit.

The server does not send the RPC reply until after the bulk get/put are complete, so I don't understand how the bulk message (which should be complete even before the reply is sent) could become unordered compared to the RPC reply.  Do you have any patches that would affect this code in some way?

I'm not totally against landing this, but I also don't like making changes which are not well understood, because it might just be covering up a bug in some other part of the code, or may allow other real errors to pass unnoticed.

Patch Set 5:

We hit that issue with IOZONE test over tcp network.
In that case bulk transfer need more time when need to transfer rpc with reply. But after that we hit a network flap (likely bug in mxl4_ip driver) and bulk transfer is failed. Sometimes it's stick in bulk phase until recovery started and aborted all in flight requests - it's also generate a lots errors.
