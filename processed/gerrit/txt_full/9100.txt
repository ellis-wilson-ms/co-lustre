--- Commit Message ---
LU-4578 ptlrpc: Early replies need to honor at_max

When determining whether an early reply can be sent the server will
calculate the new deadline based on an offset from the request
arrival time. However, when actually setting the new deadline
the server offsets the current time. This can result in deadlines
being extended more than at_max seconds past the request arrival
time. Instead, the server should offset the arrival time when updating
its request timeout.

When a client receives an early reply it doesn't know the server side
arrival time so we use the original sent time as an approximation.

Patch Set 5: Code-Review-1

(2 comments)

This could very well be correct, but we definitely need to have greater improvements to the code comments along with this patch.  Right now the code is pretty thoroughly confusing.  The code has been updated over time without improving the comments along with them.  Going back in time, it looks like the code was a bit easier to understand early on.

So let make sure we fix them up this time.

This comment really needs a rewrite:

/* Fake our processing time into the future to ask the clients
		 * for some extra amount of time */

Instead it should explain that at_measured() will change the service time estimate based on the number we provide.  It should then explain how the client will learn of that change.  This code is seriously lacking in symmetry: The server calls at_measured(), and the client apparently learns of this by calling ptlrpc_at_get_net_latency(req) (more or less).  That is not particularly conducive to understandability.

The comment for at_measured() is pretty terrible too.  It would be great to explain that at_measured() is primarily updating at_current() to the provided value, after enforcing at_min/at_max limits.

Finally, I would also suggest moving the "newdl =" line down to much later in the function as I mentioned inline in the code.
