--- Commit Message ---
LU-5042 ldlm: delay filling resource's LVB upon replay

This patch is an attempt to delay unnecessary filling+resend of
resource's LVB upon replay after Server reboot.
This should avoid recovery to take a very long time when
replaying a huge number of locks and due to all associated LVBs
beeing read from disk. Now resource's LVB is only read upon need
to be sent to a new Client.

(4 comments)

Generally, I think that the idea of postponing the lock resource initialisation is good, but from the implementation view, it may be dangerous if we miss some corner usages.

On the other hand, I am thinking that whether it is worth for us to do that or not. Because for the lock replay, the first thing that we should do is to drop the cached but unused locks on the client side. it saves both the network traffic and the server side disk IO. We should have done that already, but if it is not perfect, we should improved that.

I wonder that whether there are real use cases that millions of active resources are in using before the server crashed. If yes, the server must be quite busy before the crash...
