--- Commit Message ---
LU-14289 ptlrpc: rename cfs_binheap to simply binheap

As the binheap code is no longer part of libcfs, the cfs_ prefix is
misleading.  As this code is local to one module and doesn't conflict
with anything global, there is no need for a prefix at all.  So change
cfs_binheap to binheap.

This patch was prepare using 'sed', then fixing a few text-alignment
issues caused by the loss of those 4 characters.

--- Message (Neil Brown, 2021-02-01 23:41:56.000000000) ---
Patch Set 1:

> My hope is one day this will become apart of the Linux kernel tree for general use.

There is already binary-heap code in Linux: include/linux/min_heap.h.
It is quite different though.  Most significantly the heap is stored in an array in a single memory allocation.  This limits it to a few thousand entries.
The lustre version uses a couple of levels of indirection and allocates new pages as needed, so it can store 4 billion entries.
Is this enormous size really needed?  I don't know but I couldn't easily convince myself that it wasn't.

--- Message (Andreas Dilger, 2021-02-02 03:54:51.000000000) ---
Patch Set 1:

> ... Most significantly the heap is stored in an array in a single

 > The lustre version uses a couple of levels of indirection and
 > allocates new pages as needed, so it can store 4 billion entries.
 > Is this enormous size really needed?  I don't know but I couldn't
 > easily convince myself that it wasn't.

This needs to be able to hold all of the incoming RPCs from potentially tens of thousands of clients, with 8-256 RPCs/client/target.  So in the worst case maybe 20000 * 256 * 32 = 16M items in the heap.  Not quite billions, but definitely more than thousands.
