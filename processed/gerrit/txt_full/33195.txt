--- Message (Andreas Dilger, 2018-09-18 18:07:32.000000000) ---
Patch Set 1: Code-Review-1

This doesn't really implement what was written in LU-8365. The goal of this patch is *not* to have the allocator searching backward, since there is normally no reason to expect that recently-checked groups suddenly have free space.

Instead, the allocator should only be reset to check lower-numbered groups when a significant amount of free space becomes available in an earlier group because of file deletion. That can be handled directly in the block freeing code rather than having to scan for free space continually. This doesn't hurt the current round-robin behavior (which will also loop around to the start of the disk after some time), and doesn't add much overhead in the common case where blocks are being allocated but not freed. Instead, it puts the overhead into the block freeing path, which is typically used when a large number of files are deleted at one time. We don't want to reset the last_group pointer if only a small number of blocks are freed. 

As a starting point, something like the following at the end of the ext4_free_blocks() function ( I put it in the LU ticket) - "if group free blocks > 3/4 blocks per group then save last_group, set last_group = group" and in the alloc path "if num_groups_checked > N and last_group_save > group then set last_group = last_group_save".

The way that "group free blocks" is calculated can start with just "free blocks" for that group, but a better way would be to add up free chunks >= 4MB in size using the mballoc buddy bitmap. That will ensure the group pointer is not reset if the group is full of small fragments. 

Also, please include the previous patch http://review.whamcloud.com/21142 so we can monitor the last_group values under different workloads (eg. only writes w/o deletion, write/delete in a loop, etc).
