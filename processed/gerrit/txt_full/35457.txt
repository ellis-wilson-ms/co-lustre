LU-12518 llite: Accept EBUSY for page unaligned read

When doing unaligned strided reads, it's possible for the
first and last page of a stride to be read by another
thread on the same node, resulting in EBUSY.

Also this could potentially happen for sequential read,
for example, several MPI split one large file with unaligned
page size, sequential read happen with each MPI program.

We shouldn't stop readahead in these cases.

The code in ll_grab_cache_page_nowait misses a case handled by grab_cache_page_nowait().  Maybe it isn't that important here, but it is bad form to use the same name for different behaviour.
Specifically, if add_to_page_cache_lru() fails with -EEXIST, it should retry the lookup.

More generally though, I agree with Patrick's comments.  I don't think we need this function at all - just use grab_cache_page_nowait() and don't worry about the possibility that the allocation failed.
Memory allocations (that are allowed to wait) almost never fail.  They wait for some memory to be freed - possibly by a process being killed.
The main time they fail is when the allocation is being requested by the process that ends up getting killed.

The fact that GFP_NOFS is given does *not* mean that no filesystem page cache can be freed.  I'm fairly sure that clean pages can still be freed, and if a lot of read-ahead has been happening, then there are probably a lot of clean pages.

There is a comment
> I might need add some debug information to see how often this could happen.

I agree.  Until that debug information has been gathered and there is concrete evidence that ENOMEM occurs significantly often, I think we should consider it a corner case, and not worth the maintenance cost of duplicating kernel code.
