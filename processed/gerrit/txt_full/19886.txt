--- Commit Message ---
LU-7988 hsm: defer all new archive hals

All new archive HALs are enqueued on a coordinator list, and the
coordinator will process them once every "wait_event_time"
seconds. This reduces the number of times hsm_find_compatible and thus
cdt_llog_process are called.

Other types of HAI (restore, cancel, ...) are processed as usual. When
one of these is received, the deferred list of archive HAI, if not
empty, is processed first.

Assuming we ultimately want to get rid of the llog altogether, I do not think this patch should land. To me, the ratio complexity/performance is bad: it will be quite hard to undo when the llog is replaced with something else.

I agree there is work to do around hsm_find_compatible() though. Currently, it fulfils 3 distinct goals:

- set the cookie of CANCEL requests to the cookie of the request to cancel;
- set the cookie of NONE requests (used to find out what hsm actions are scheduled on a fid);
- set the cookie of other requests (ARCHIVE, RESTORE, REMOVE) to detect duplicates (I opened LU-9959 about this because I believe it is not doing it correctly).

I think splitting the function into three would be better on the long term. Since it is currently called for each incoming hal, and hal are bound to contain only one type of request, there would not be any performance regression.

After that split. We could work on improving each function separately:
- check the coordinator's active request list first (no access to the llog) for requests to cancel;
- use a mechanism in front of the cdt's llog to detect duplicate requests (bloom filter, cuckoo filter, quotient filter, ...).
