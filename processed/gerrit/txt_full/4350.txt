--- Message (John Hammond, 2012-11-28 20:17:17.000000000) ---
Patch Set 14: I would prefer that you didn't submit this

I believe that this will be broken if the fldb contains 100 or more-ish entries.  (I don't know if that's reasonably possible.  Is it?)  You're feeding the cpu-endian file position to iops->load() which I guess happens to be equivalent to iops->get() here, but keys in fldb are stored big-endian.  Hence incrementing *pos may increment your search key by 1 << 62 which will probably exceed start of the 100-th range.

Your seq_next() function should be updating *pos with the cpu-endian key for the next range.

Because fsp_stop is only cleared in open(), llseek() isn't really supported.

Try running the following command twice:

 multiop /proc/fs/lustre/fld/srv-lustre-MDT0000/fldb oz1r4096

--- Message (wangdi, 2012-11-29 01:15:25.000000000) ---
Patch Set 14:

I think I agree with you we should update pos here. Thanks for reminding me this. 

But the real reason of system hanging( I assume that is the problem you mean? ) during second "
 multiop /proc/fs/lustre/fld/srv-lustre-MDT0000/fldb oz1r4096 "

is that mulitop did not close the file, so some lock is being hold inside the iam (holding by load() actually). Then if it tries to seq_start(load) again, it will cause deadlock inside iam (sigh, it does not document well for this API :( ). So I add put in the seq_stop, which will release the lock hold by load(), the problem should be fixed.

 The reason why you do not see this during cat,  is that iam_it->fini will always release all of the lock, which will be called during closing the file.
