--- Commit Message ---
LU-3299 llite: force lvb_data update after layout change

When a file is restored the layout lock is first
associated with the released layout and after restore
it has to be assocaited with the new layout. This patch
forces lvb_data update in ll_layout_fetch() even if one
is present (case for released->normal state change)

Hi JC, maybe I missed something here. If restore revokes layout lock, why the lvb of layout lock can't be refreshed?

--- Message (jacques-Charles Lafoucriere, 2013-05-29 20:08:33.000000000) ---
Patch Set 1:

In restore case the layout move from a released one to a valid one. So the lvb_data is already allocated. In ll_layout_fetch() the refresh was not done if LDLM_FL_LVB_READY was set or if the  lvb_data was allocated. The patch just do the refresh in these cases.

"Jinshan, could you please also remind me what are the latest rules on when the layout is packed in the completion AST? Is it only when the layout is released?"

Right now, when a layout lock is enqueued, layout can be returned in 2 ways:
1. if the MDT can grant the layout immediately, layout will be packed in DLM_LVB of lock reply RPC;
2. Otherwise, client has to fetch layout in ll_layout_fetch()
