--- Commit Message ---
LU-17175 gss: new 'quick' option to lsvcgssd

Add a new '-q' option (for quick) to lsvcgssd. This is useful for the
Shared Secret Key case, when we want to start the daemon rapidly by
skipping the speed check test for Miller-Rabin rounds used in prime
testing.
When this option is specified, let OpenSSL use its default value for
the number of rounds used for Miller-Rabin prime testing.

> if this is fine to auto start the daemon without doing the speed check. Is it worth doing it at all?

Skipping the speed check test lets OpenSSL use its default value for the number of rounds used for Miller-Rabin prime testing. This can lead to longer prime check testing, and possible request timeouts. This is why we implemented the speed check test, in order to limit the time spent in prime testing. But from a security standpoint, skipping the speed check test is safe.

In case the daemon is auto started, we really need to skip the speed check test, because in this case a gss request is already in progress.

> 
> So, 
> 1 - in the past DH_check becomes too slow and request could timeout
> 2 - so we decided for a mechanism to limit the max run time 
> 3 - now, we are adding a 'quick' option that disable 2) and goes back to 1). I'm missing something...

From OpenSSL 1.1.1c, it increased the number of rounds used for Miller-Rabin testing of the prime provided to DH_check(). It led to observed request timeouts when an SSK flavor is being used, because it makes the prime check roughly x10 longer. So this is an issue for every request being processed.

In order to work around this issue, we implemented a speed check for prime testing. We estimate how long it takes to carry out prime testing with various values for the number of rounds. And we pick a number of rounds for which time spent in prime testing is acceptable, so that there is less risk of request timeouts. In a way it speeds up request processing when an SSK flavor is being used.

But the drawback of this speed check test is that it slows down the startup process of the lsvcgssd daemon. That is because we carry out prime testing of a sample prime number with various rounds, before deciding on the most suitable number of rounds.

Now, in case we are in the process of handling a GSS request and we realize the lsvcgssd daemon is not up and running, we need to start the daemon as quickly as possible. Otherwise the request being processed could timeout. In this particular case we need to skip the speed check test carried out at daemon startup. Of course this is a tradeoff, and it exposes to subsequent requests timing out because their prime testing took too long (as we are using OpenSSL default number of rounds, that makes prime check roughly x10 longer).
