--- Commit Message ---
LU-17632 o2iblnd: graceful handling of CM_EVENT_CONNECT_ERROR

There were examples in the field with RoCE setups which demonstrate
that RDMA_CM_EVENT_CONNECT_ERROR may be received when conn state
is neither IBLND_CONN_ACTIVE_CONNECT nor IBLND_CONN_PASSIVE_WAIT.
Handle this in a more gracious manner: report the event as unexpected
and allow the flow to continue.

(4 comments)

We should never LASSERT as a method of error handling on information that is out of our control, like data received over the wire, from userspace, (or from disk, though that doesn't apply here).  We've moved beyond the stage where "system crashes and we are notified to fix it" is a workable model, unless the system can no longer function at all in that state. 

I'm not against LASSERT on internal state that we control, but definitely not on external state.  Note this is *not* a request to change this patch.
