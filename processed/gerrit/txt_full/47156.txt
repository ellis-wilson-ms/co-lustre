LU-15787 sec: block enc unaware clients on enc files

Prevent encryption unaware clients from manipulating encrypted files
and directories. Those can be old clients, or clients built without
encryption support (intentionally or because they run on an old
kernel).
In the mdt layer, check that clients have the OBD_CONNECT2_ENCRYPT
connection flag, and if not, block access if they try to manipulate
a file or directory that has the LUSTRE_ENCRYPT_FL flag.
The forbidden operations from encryption unaware clients are:
- open
- create
- link
- rename
- migrate
Improve sanity-sec test_54 to test this use case.

(1 comment)

This is definitely an improvement, and a must-have for 2.15.0.

My main concern with this patch is that it needs to add an explicit check in each function that is trying to modify an encrypted file/directory. This has the risk that some existing or new file modification paths are missed. 

Is there a central location (eg. declare transaction) that could check the object flags and client flag to catch all modification attempts?

--- Message (Sebastien Buisson, 2022-04-27 16:33:11.000000000) ---
Patch Set 1:

I agree having to add a check in every place in the code where we need it is cumbersome and error prone in case code is changed or added.

But we do not want to prevent all kinds of access from encryption unaware clients (removing files/directories for instance). Also, I do not see any useful transaction handlers in the mdt layer. There might be some in the mdd layer, but this layer lacks visibility on the export, so I cannot see how to check connection flags from the mdd layer.
