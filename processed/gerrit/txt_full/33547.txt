Patch Set 1:

If you are sure the LMA is corrupted, then you can remove it and let namespace LFSCK to re-fill it with the FID-in-dirent. But there are something to be mentioned:

1) It only works on the MDT. For the inode on OST, removing LMA means no LMA any longer.

2) For the inode on the MDT0000, the OI scrub will generate IGIF for such inode, NOT the FID-in-dirent. Means that if someone has remembered the original FID, then using such FID to access the inode will hit error. Such logic is used for handling upgrade case, you can make patch to adjust the OI scrub behavior for current non-upgrade case.

3) For the inode on other MDT, the namespace LFSCK will recover the LMA from the FID-in-dirent if such EA exists.

Patch Set 6: Code-Review-1

I agree that removing corrupted LMA EA is necessary, but as for how to avoid the OI scrub re-filling the missed LMA with IGIF, we may have more options. Please check the following logic:

static int
osd_scrub_check_update(struct osd_thread_info *info, struct osd_device *dev,
                       struct osd_idmap_cache *oic, int val)
{

...

        if ((val == SCRUB_NEXT_NOLMA) &&
            (!scrub->os_convert_igif || OBD_FAIL_CHECK(OBD_FAIL_FID_NOLMA)))
                GOTO(out, rc = 0);

...

}

So there are two conditions for the OI scrub refilling the missed LMA with IGIF:
1) scrub->os_convert_igif is true, and
2) OBD_FAIL_FID_NOLMA is NOT set.

So if it is just for temporary solution, then the most simple one is to set the fail_loc as OBD_FAIL_FID_NOLMA during OI scrub, that is enough, without any additional patch.

If it is not temporary solution, we can consider adjust the scrub->os_convert_igif. This flag was mainly used for upgrading Lustre from 1.8 to 2.x at that time. But now, most of Lustre users have already switched to Lustre-2.x, so we can add some lproc interface or mount option to notify the OSD logic that this is not upgrading case, or we can detect that via checking some current configuration, then "scrub->os_convert_igif" will be false on the MDT0, that will make the OI scrub to skip this inode.

On the other hand, if we do not support to upgrad Lustre-1.8 directly to Lustre-2.11 or newer, we can remove the logic for re-filling LMA with IGIF, that is more clean solution I think.
