--- Message (Ben Evans, 2015-08-12 14:47:45.000000000) ---
Patch Set 3:

John: This is the first of what will be a series of similar changes in order to better organize the code.  Once that's done, separating userspace, kernelspace and common headers should be a much more tractable problem.  Also note that only the things that require lu_seq_range are now including it.

--- Message (John Hammond, 2015-08-12 15:51:49.000000000) ---
Patch Set 3: Code-Review-1

Sorry, every structure that goes on the wire should be defined in lustre_idl.h. That's the rule and it's a good one.

--- Message (Ben Evans, 2015-08-12 17:10:14.000000000) ---
Patch Set 3:

I'd argue that shoving everything that goes on the wire (along with various functions, defines, etc.) into a giant file violates the Lustre Coding Guidelines, as it is neither beautiful, nor does it do what it says on the tin.

You could argue that because it goes on the wire, I should include seq_range.h in lustre_idl.h so that if something is sending data over the wire, all any piece of code that sends things over the wire has to do is include the one file.  I'd be fine with that.

On the flip side, I'd also argue that things that should never send data on the wire, such as ldiskfs should never include lustre_idl.h as it sends nothing over the wire.  Yet, it does.  It should be including only the things it needs to do it's job.
