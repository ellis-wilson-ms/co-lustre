LU-12460 llite: replace lli_trunc_sem

lli_trunc_sem can lead to a deadlock.

vvp_io_read_start takes lli_trunc_sem, and can take
mmap sem in the direct i/o case, via
generic_file_read_iter->ll_direct_IO->get_user_pages_unlocked

vvp_io_fault_start is called with mmap_sem held (taken in
the kernel page fault code), and takes lli_trunc_sem.

These aren't necessarily the same mmap_sem, but can be if
you mmap a lustre file, then read into that mapped memory
from the file.

These are both 'down_read' calls on lli_trunc_sem so they
don't directly conflict, but if vvp_io_setattr_start() is
called to truncate the file between these, it does
'down_write' on lli_trunc_sem.  As semaphores are queued,
this down_write blocks subsequent reads.

This means if the page fault has taken the mmap_sem,
but not yet the lli_trunc_sem in vvp_io_fault_start,
it will wait behind the lli_trunc_sem down_write from
vvp_io_setattr_start.

At the same time, vvp_io_read_start is holding the
lli_trunc_sem and waiting for the mmap_sem, which will not
be released because vvp_io_fault_start cannot get the
lli_trunc_sem because the setattr 'down_write' operation is
queued in front of it.

Solve this by replacing with a hand-coded semaphore, using
atomic counters and wait_var_event().  This allows a
special down_read_nowait which ignores waiting down_write
operations.  This combined with waking up all waiters at
once guarantees that down_read_nowait can always 'join'
another down_read, guaranteeing our ability to take the
semaphore twice for read and avoiding the deadlock.

I'd like there to be a better way to fix this, but I
haven't found it yet.

Hi Patrick,
 This patch was written to resolve a lockdep warning that I can reproduce easily with mainline. It happens in sanity test 56w, though I'm not sure that running just that test will trigger it.

- an O_DIRECT read causes vvp_io_read_start() to call generic_file_read_iter() which leads to ll_direct_IO which calls iov_iter_get_pages_alloc and then get_user_pages_unlocked.

 vvp_io_read_start holds lli_trunc_sem, and get_user_pages_unlocked takes mmap_sem.

- A page fault will take mmap_sem in __do_page_fault, which then calls ll_fault and eventually vvp_io_fault_start() which takes lli_trunc_sem

These are both read locks on lli_trunc_sem so they only cause a deadlock if a readlock waits for a read lock - which happens with rw_semaphore when something is waiting for a write lock.

So the line you quoted is correct, but could be enhanced by saying this only happens with O_DIRECT reads, and happens in ll_direct_IO.

Your guess about mmap_sem is correct, it is taken in __do_page_fault long before any lustre code is involved.

The "latter" and "former" refer to down_read() calls on lli_trunc_sem. The latter in vvp_io_fault_start, and the former in vvp_io_read_start.

I think all the relevant code is much the same in OpenSFS lustre.

I should re-iterate a comment in the Commit message.
For this issue to actually cause a deadlock (rather than just a lockdep warning) you need three threads to be racing.
One must be serving a page-fault in a particular lustre file
One must be calling ftruncate() on that same file.
One must be doing an O_DIRECT read from the file into a memory buffer which is mmap()ed from that same file.

This is almost impossibly unlikely unless you are deliberately trying to trigger the deadlock.

I'll provide the lockdep report in a separate comment.

OK, actually, I think the fact that we have a queue is a problem...  Readers can jump the queue when they arrive, but only when they arrive.

Consider this order in the queue, with writer1 holding the lock and the others waiting:

reader2
writer2
reader1
writer1 (HOLDING LOCK)

What happens to that second reader?
Writer1 releases the lock, which calls:
wake_up_var
Which, if I'm reading the wait queue code correctly, just wakes up one waiter...
So now the lock & queue looks like:
reader2
writer2
reader1 (HOLDING LOCK)

And reader2 doesn't join reader1 in getting the lock, because it's not woken up.  So we're back in the situation where we started - A reader has the lock and another reader is sleeping behind a writer.

So I think we either need some other truly queue-less primitive or, more likely, we need "wake_up_var_all", which we don't have.  (I suppose it could be open coded.)

wake_up_var_all *does* have the effect of basically making us queue-less by making it a free-for-all.  There's a queue, but only as an implementation detail, we don't respect ordering.

--- Message (Neil Brown, 2019-06-21 04:25:14.000000000) ---
Patch Set 1:

wake_up_var() does wake up all waiters.
A wait queue can have exclusive and non-exclusive waiters.  The exclusive waiter are (almost always) at the front of the queue (woken first), the non-exclusive are at the end (woken last)
A normal wakeup will wake up everything it finds on the queue until it it has found and woken an exclusive waiter, then it will stop.
A wakeup_all() will not stop when it finds an exclusive waiter.  (you can ask to wake up the first N exclusive waiters if you really want to).
So unless something calls ___wait_var_event() passing the 'exclusive' flag, the wake_up_var will wake all waiters.  So in your example, reader2 will be woken.

I agree that performance isn't an issue here.  Conflicts on this semaphore are likely to be rare, and even when there are readers to wake up, there probably aren't many.

I see your point about "queueless" now.  The fact that they are waiting on a queue is an implementation detail.  They are waiting somehow, and they are all woken equally.
