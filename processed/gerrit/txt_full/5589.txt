--- Commit Message ---
LU-2681 fld: shrink seq_type in lsr_flags

In lu_seq_range, lsr_flags is treated to only hold
LU_SEQ_RANAGE_MDT, LU_SEQ_RANGE_OST or LU_SEQ_RANGE_ALL.
Because they can only be 0, 1, 2, using lsr_flags(32 bits)
is clearly unnecessary.

This patch will shrink the current seq_type to 2 bits, so
other flags can be used in lsr_flags in future.

Signed-off-by: wang di <di.wang@intel.com>

(1 inline comment)

Di, I don't understand your reasoning about not consuming another bit for MDT, yet consuming a bit for LU_SEQ_RANGE_ALL (which has no real meaning).

The reason we picked LU_SEQ_RANGE_MDT == 0 was for backward compatibility, but if we need to add a bit to handle the distinction between MDT and OST, then I'd rather use that bit to positively identify MDT ranges.

That said, why does the FLD query need to specify the type at all?  Couldn't the query just return the sequence range, and then the caller decides whether it wants that type of FID?  That will make more sense when we have unified targets and the client doesn't really care what the type is.  Also, there should never be overlapping FID ranges for both OST and MDT types.

I don't really consider this flag to be part of the query, only the SEQ should be the key, and the type is just information that is returned for the specific range.
