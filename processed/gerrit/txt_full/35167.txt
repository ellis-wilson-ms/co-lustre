--- Commit Message ---
LU-11367 som: integrate LSOM with lfs find

The patch integrates LSOM functionality with lfs find so that it
is possible to use LSOM functionality directly on the client. The
MDS fills in the mbo_size and mbo_blocks fields from the LSOM
xattr, if the actual size/blocks are not available, and then set
new OBD_MD_FLLSIZE and OBD_MD_FLLBLOCKS flags in the reply so that
the client knows these fields are valid.

The lfs find command adds "-l|--lazy" option to allow the use of
LSOM data from the MDS.

Add a new version of ioctl(LL_IOC_MDC_GETINFO) call that also returns
valid flags from the MDS RPC to userspace in struct lov_user_mds_data
so that it is possible to determine whether the size and blocks are
returned by the call.  The old LL_IOC_MDC_GETINFO ioctl number is
renamed to LL_IOC_MDC_GETINFO_OLD and is binary compatible, but
newly-compiled applications will use the new struct lov_user_mds_data.

New llapi interfaces llapi_get_lum_file(), llapi_get_lum_dir(),
llapi_get_lum_file_fd(), llapi_get_lum_dir_fd() are added to fetch
valid stat() attributes and LOV info to the user.

Patch Set 6:

When trying to several things in a single place, things gets complex. Thus, instead of doing everything in a single API, I think exporting several simple APIs would be better. Something like:

int llapi_get_lum_file(const char *path, lstat_t *stat, struct lov_user_md *lum, unsigned long long *valid_flags);

This function gets the @stat and @lum for file with path of @path. Both @stat and @lum can be NULL, and they can be both NULL too. If any of them is not NULL, corresponding result will be copied to the corresponding buffer, with @valid_flags set with proper flags. Following flags are being used:

FLAG_STAT_SIZE == 1, size and nblock in stat is valid
FLAG_XXX == 2, xxxxxxx
FLAG_YYY == 3, yyyyyyy
...

There is no need for the caller to give an dir fd or parent fd to llapi_get_lum_file(). The function itself should be able to open the file's directory and call ioctl on the file. And llapi_get_lum_file() should not call lstat() if ioctl(IOC_MDC_GETFILEINFO) fails. Whenever someone calls llapi_get_lum_file() it is trying to use Lustre's private API to get the lum and stat which suppose to be quicker than lstat(). The caller can do that by itself if needed after a failure of llapi_get_lum_file(). 

And for directory, another API can be defined:

int llapi_get_lum_dir(const char *path, lstat_t *stat, struct lov_user_md *lum, int *valid_flags);

It does the same thing with llapi_get_lum_file() except it opens the directory @path and calls ioctl(LL_IOC_MDC_GETINFO) on the directory itself.

We can define llapi_get_lum_file_fd() and llapi_get_lum_dir_fd() in the same way:

int llapi_get_lum_file_fd(int dir_fd, const char *fname, lstat_t *stat, struct lov_user_md *lum, unsigned long long *valid_flags);

int llapi_get_lum_dir_fd(int dir_fd, lstat_t *stat, struct lov_user_md *lum, unsigned long long *valid_flags);

I think in the way, the API will be much more simplified.

What do you think?
