LU-12668 llite: fix EC read handling in VVP layer

Fix multiple bugs in the VVP/llite layer that caused
hangs, LBUGs, and data delivery failures during EC
recovery reads.

EAGAIN from glimpse:
ll_glimpse_size() returns -EAGAIN when OSTs are
deactivated.  Handle this by using the cached inode
size and letting EC recovery proceed via parity
reconstruction.

Data delivery in ec_rd_end:
vvp_io_ec_rd_end() returned early without calling
generic_file_read_iter() when ec_page_list was empty.
Even when no parity reconstruction was needed, the
EC read IO type still requires generic_file_read_iter()
to copy page cache data to userspace.  Remove the
early return.

Infinite loop in ll_readpage:
During CIT_EC_RD, lcc_end_index was 0 (unset), causing
every page to match "vmpage->index >= lcc_end_index"
and return AOP_TRUNCATED_PAGE indefinitely.  Fix by
guarding on lcc_end_index > 0, setting it from
min(ec_inner_end, i_size) in vvp_io_ec_rd_start(),
and returning -EIO for beyond-range pages instead of
AOP_TRUNCATED_PAGE.

Component range constraint:
generic_file_read_iter() in ec_rd_end could read into
pages of subsequent PFL components that have no DLM
locks, causing LBUG in osc_req_attr_set().  Truncate
the iov_iter to the current component's range capped
at i_size, and refresh lcc_end_index before the read.

Lock coverage check:
During EC recovery, readahead may request pages on
deactivated stripes that have no DLM lock.  Add
cl_io_read_ahead_prep() check for CIT_EC_RD pages
in ll_readpage() so lockless pages return -EIO instead
of reaching osc_build_rpc() where they LBUG.
