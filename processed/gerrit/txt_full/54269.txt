--- Commit Message ---
LU-17605 obdclass: do not wait forever acquiring entry

The process of refreshing an entry via refresh_entry() goes through
an upcall/downcall. If the upcall succeeds, we enter a wait queue.
If after that the downcall is never called, we hit the expiry timeout,
and we get removed from the wait queue.
But if the entry is not new, the expiry time will be
MAX_SCHEDULE_TIMEOUT == LONG_MAX, which means an infinite wait.
So avoid waiting forever if an entry could not be refreshed, and call
wake_up_all() if the wait for the ACQUIRING state failed.

Patch Set 1:

> There is no 'grace' period when an entry expires. But there is
 > already a retry mechanism to mitigate an overloaded LDAP server:

The retry mechanism means the client won't get an *error* back if the upcall failed once, but it *does* mean that the client (maybe many of them) may suddenly become stuck waiting for the entry to be refreshed,  even though 1ms earlier that entry was OK to use.

I'm wondering if it makes sense to have some "soft" and "hard" timeout on the entries, where the MDS upcall _tries_ to refresh the entry, but if this times out/fails it will continue to use that entry for some time before it actually blocks the client or returns an error.

We don't want to give users extra time if their account was removed, so if the upcall succeeds but returns an error that the user does not exist, that is totally different than the upcall timing out and 99.9999% of the time the user was not coincidentally removed in those seconds before the entry expired.

A relatively safe approach would be to *try* to update the entry after it has expired, but only block the thread doing the upcall if the entry has been expired more than, say, 5x the expiry interval and the downcall has not completed in a few seconds.

That allows requests to continue processing for a few minutes if the LDAP server is suddenly busy/offline.  This would also allow removing the explicit upcall retry mechanism, since some other RPC arriving for the same UID in a short time will retrigger the upcall again (though it should still be rate limited so that only one upcall can be in progress at one time).
