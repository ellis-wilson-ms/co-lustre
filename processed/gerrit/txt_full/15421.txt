--- Commit Message ---
LU-3534 ptlrpc: mbits is sent within ptlrpc_body

ptlrpc is using rq_xid as matchbits of bulk data, which means it
has to change rq_xid for bulk resend to avoid several bulk data
landing into the same buffer from different resends.

This patch uses one of reserved __u64 of ptlrpc_body to transfer
mbits to peer, matchbits is now separated from xid. With this change,
ptlrpc can keep rq_xid unchanged on resend, it only updates matchbits
for bulk data.

This protocol change is only applied if both sides of connection have
OBD_CONNECT_BULK_MBITS, otherwise, ptlrpc still uses old approach and
update xid while resending bulk.

--- Message (Andreas Dilger, 2015-06-29 07:24:07.000000000) ---
Patch Set 5: Code-Review+1

One idea that came to mind while reviewing this is that it probably makes more sense from a protocol point of view to send the XID as part of the ptlrpc_body, and send the matchbits via LNet. That way the XID stays at the ptlrpc protocol level as it really should be, as it doesn't change if the RPC needs to be resent (since the Lustre recovery protocol really depends of the RPC XID staying constant over resend), and the LNet layer can do what it wants with the matchbits if it needs to be resent (e.g. use previous matchbits - 2^30 for each resend, Lustre initializes the matchbits to be at least 2^50 or more).

(1 comment)

Andreas, I actually think both rq_xid and matchbits should be within ptlrpc_body, rq_xid is not really necessary for LNetPut called by ptl_send_rpc()->ptl_send_buf(), from LNet point of view, request portal doesn't need matchbits (reply and bulk portal need matchbits) because RPC can be fit in any request buffer on server and matchbits is ignored.  So the only reason that we transfer rq_xid to LNetPut while sending RPC is, remote peer needs it as matchbits of reply (and bulk without this patch) and identifier of RPC, it means that rq_xid is just a 64-bit data used by ptlrpc, not by LNet.

The reason that I didn't put rq_xid in ptlrpc_body is because I don't want to consume reserved bytes too quick.
