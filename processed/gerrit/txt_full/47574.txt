--- Commit Message ---
LU-15922 sec: new connect flag for name encryption

Introduce OBD_CONNECT2_ENCRYPT_NAME connection flag for compatibility
with older versions that do not support name encryption.
When server side does not have this flag, client side is forced to
null encryption for file names. And client needs to use old xattr to
store encryption context.

Also update tests in sanity-sec to exercise name encryption only if
server side supports it.

> Sebastien, what is the impact of this patch is not included in
 > 2.15.0?  At worst it would almost immediately return errors if name
 > encryption was enabled explicitly and then used with an old server?
 >  Or is there permanent corruption if this was used with an old
 > server?

Without this patch, I think 2.15 clients accessing 2.14 servers are going to work or fail randomly. When name encryption is enabled on a 2.15 client, it is going to encrypt the file names, and then critical encode them. 2.14 servers will leave that as is, and store those names. This could work when client has the encryption key, although there might be some places in the code on server side that expect a nul terminated string for file names, which is not the case with encrypted names.
And without the key, as soon as name length exceeds 32 chars, it will not work because server side is not able to handle the digested form computed on client side.

And as can be seen from interop tests results like https://testing.whamcloud.com/test_sets/7ad6d961-6a29-45e8-b60f-6e8ffdf3dc63 , "most" of the encryption-related tests in sanity-sec are passing. And they do use name encryption, because of dummy encryption mode. So it shows it is not quite like crashing or returning errors immediately, unfortunately.

Looking at the test results from this interop session, I realize this patch needs a little bit more work. Indeed, in 2.15 we changed the xattr used to store the encryption context, from security.c to encryption.c, to match ext4. But 2.15 clients have to continue using the old xattr name (LL_XATTR_NAME_ENCRYPTION_CONTEXT_OLD) when they talk to 2.14 servers, because those servers do not grok the new name (they do not have a handler for that xattr type).
