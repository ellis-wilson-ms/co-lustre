What is the reason behind that? What issue are you trying to solve?

From a kerberos stand point (i don't know about SK), this is a system service, with a specific config file (/etc/sysconfig/lsvcgss). If you spawn it from the helperusermod context, you will end up with a process running in some context, without the proper config options (think we add the -R realm option to lsvcgss).

From an admin stand point, this is problematic, the 'systemctl status lsvcgss' will return wrong status, and the config file will be ignored. This is orthogonal to system management.

The only use case I can see some benefit is if this is fully transparent to admin, and they do not manage the process at all. Or, if this is a one shot process, that is no more resident.

Patch Set 2:

> Patch Set 2:
> 
> From a kerberos stand point (i don't know about SK), this is a system service, with a specific config file (/etc/sysconfig/lsvcgss).

lsvcgssd daemon works the same for Kerberos and SSK from that perspective. It can be started manually, or as a system service after having customized its options via /etc/sysconfig/lsvcgss.

So indeed, very good point Andreas, the best way to start it from l_getauth would be to call "systemctl start lsvcgss". If admins have configured this service, it will make them happy. And if they have not, well again this is a best effort and lsvcgssd will be started with some default values.

The only problem is to find out if /etc/sysconfig/lsvcgss has been correctly configured. By default (i.e. if not edited) its content is:

   # Optional arguments passed to lsvcgssd.
   LSVCGSSDARGS=''

So it will end up launching lsvcgssd without any args. This will no work for SSK.

The problem is to be able to support all GSS mechs: ssk, krb5, null. This is why I created this additional service "ldaemon@.": to be able to pass the mech as an argument. But I had not realized that calling "/etc/rc.d/init.d/lsvcgss start" is not equivalent to "systemctl start lsvcgss", in a sense that as you noticed, it prevents from using "systemctl stop lsvcgss" afterwards.

So I will modify this patch to get rid of "ldaemon@.", and call 'systemctl start' directly from l_getauth. And in the lsvcgss script, use all mech-related options if $LSVCGSSDARGS is empty, which happens if the daemon has not been configured at all.
