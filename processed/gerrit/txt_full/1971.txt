--- Commit Message ---
LU-993 osd: code cleanup for directory nlink count

- LDISKFS_LINK_MAX is ldiskfs unique, we should not handle it in the mdd
  layer. It's moved down to the osd layer in this patch.

- Remove the declared operation count assertion in OSD_EXEC_OP(), since
  the undo operations are not declared, if there some operation failed,
  the undo operation will trigger the assertion unnecessarily.

- Restore the sub-directory count to 70000 for the sanity test_51b, and
  add test_51ba for testing unlink > LDISKFS_LINK_MAX sub-directories.

--- Message (Andreas Dilger, 2012-01-16 17:13:57.000000000) ---
Patch Set 1:

There are several issues with this patch that prevent acceptance as-is:
- if this is a hack or temporary workaround, there should at least be a clear comment to that effect, AND what needs to be done to fix it properly. Just writing that it is a hack doesn't tell anyone why it is incorrect, nor what is needed to get out of that situation  Otherwise, it depends on the memory of a few busy people to even know that this code is not correct, and it will often be forgotten due to other bugs that need to be fixed.  This also leads to my extreme reluctance to accepting hacks into the code - almost always this results in code that stays as a hack indefinitely, because once the visible bug is fixed the management pressure to fix the next visible bug overrides the need to fix the (now hidden) hack. 
- that said, it isn't clear that this change is incorrect.  There should also be a timestamp update on the parent directory, so it may be that the parent attrs always need to be updated?  I don't know why this wouldn't have failed previously. 
- it appears that this problem is caught by the OSD API debug code, even though the transaction already has credits for the parent director inode due to the declare_ref_add() call. That is great that the debug code tracks the setattr vs. ref_add() calls, but I hope the declare_set_attr() is not causing more credits to be reserved for that inode?  that would add overhead to every transaction unnecessarily. 
- the test case in sanity.sh should be re-enabled as part of this patch, since it is clearly testing a case that is not being tested today. That someone disabled this failing test by reducing the number of directories and not even linking it to an open bug using the existing ALWAYS_EXCEPT mechanism is bad programming practice and leads to hidden bugs like this. That is why we have practices like ALWAYS_EXCEPT - to ensure that hacks like this do not stay longer than absolutely needed, and do not get forgotten.
