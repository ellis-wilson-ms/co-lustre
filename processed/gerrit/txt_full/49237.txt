LU-16463 llite: replace lld_nfs_dentry flag with opencache handling

The lld_nfs_dentry flag was created for the case of caching the
open lock (opencache) when fetching fhandles for NFSv3. This same
path is used by the fhandle APIs. This lighter open changes key
behaviors since the open lock is always cached which we don't
want. Lustre introduced a way to modify caching the open lock
based on the number of opens done on a file within a certain
span of time. We can replace lld_nfs_dentry flag with the
new open lock caching. This way for fhandle handling we match
the open lock caching behavior of a normal file open.

In the case of NFS this code path will always be called with the
internal kernel thread 'nfsd'. If we are called by this kernel
thread set the open threshold to zero which means always cache the
open lock. Once Lustre is only supported on Linux kernels above
5.5 we can remove this special NFSv3 work around.

This is an improvement. This moves handling the open lock cache into the hands of the user. Currently Lustre controls open lock caching on the super block level which can only be tuned via sysfs. Now the application writer can fine control this lock caching themselves using the fhandle API. By default the open lock will behave like before using the sysfs / sb level but if the user wants to start caching the open lock after say 5 opens then the lustre_file_handle can set lfh_thrs_count to 5. This is very powerful.

I imagine it would make sense to use the min() of the two values.  If the filesystem wants every open to be cached, do we really want individual files to over-ride this?  Why?

I should go over what the goals are and what problems exist.

1) We have the ioctl LL_IOC_GETPARENT (ll_getparent) which returns the 
   parent fid. In theory we can also return the parent fid with our 
   fhandle. The parent fid returned is influenced by a linkno. So the 
   question is can we pass in the linkno to influence which parent
   fid is returned.

2) It would be nice to get the original parent FID for when a file 
   gets corrupted and moved to lost_found for recovery purposes.

3) Caching the open lock changes the files behavior due to lighter 
   closes. One of those LSOM (lighweigth stats). This normally happens
   after some set number of opens. fhandle handling for Lustre breaks 
   this.

4) The setting is controlled at the mount level. While it might help 
   applications it could break other applications. Say admins run an 
   application to monitor light weight stats. So this is the motive
   to bring control to the individual application level.

So this gives the background. So what is acceptable with the above?

Patch Set 7: Code-Review-1

(2 comments)

While this is somewhat better than changing the open file handle, the problem is that O_CREAT_NOCACHE's internal use of (O_NOCTTY|O_CREAT) conflicts badly with O_LOV_DELAY_CREATE's use of (O_NOCTTY) together with the caller supplying O_CREAT itself to create a new file with a Lustre layout.

While the side-effect isn't bad (not having an open lock cached on the first open of a new file) I think it will be confusing internally since the code won't know if the intention was O_LOV_DELAY_CREATE or O_CREATE_NOCACHE.

I think the better solution is just to remove lld_nfs_dentry and the "always cache open-by-handle" semantic completely, and let the opencache do its work.  The client shouldn't be set to cache every open by default (maybe only if it is a dedicated NFSv3 server), and it will quickly cache re-open files anyways.
