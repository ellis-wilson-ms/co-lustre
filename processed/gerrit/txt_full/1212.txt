--- Commit Message ---
LU-549 xattr: packaged xattr in single RPC reply

Currently, MDS returns ACL together with other object's attributes
when replies getattr/lookup RPC. As for stating directory object,
client will send additional RPC for default ACL.

As an improvement, MDS will return packaged xattr to contain both
posix ACL and default ACL (for directory). For remote client, the
remote permission is also returned through packaged xattr.

The packaged xattr framework supports to return more other xattrs
by single RPC.

--- Message (Andreas Dilger, 2011-10-19 23:05:37.000000000) ---
Patch Set 7: I would prefer that you didn't submit this

(7 inline comments)

My first main concern is whether adding the DEFACL buffer breaks the RPC network protocol and interoperability with older clients?

My second concern is whether it would be better to create a generic mechanism for returning xattrs to the client, instead of (or in addition to) the DEFACL, so that we could address a number of performance issues related to repeatedly fetching xattrs to the client.  Since xattrs are becoming increasingly widely used for security, samba, NFS, and other reasons (see LU-549), it makes sense to try and address these issues together.

If we have to do extra work to maintain RPC compatibility, then instead of adding a reply buffer, flags, etc. that is only going to fix the DEFACL issue, it would be better to make a generic xattr fetch mechanism in the reply (e.g. as many xattrs as will fit into the reply buffer, given that the server knows the size already), and they can be cached on the client either under the MDS_INODELOCK_UPDATE bit or at some later time under a separate MDS_INODELOCK_XATTR bit.
