--- Commit Message ---
LU-398 nrs: Increase module usage count for external policies

Increase the module usage count for policies that have registered
from other modules; the count is increased once when the first
instance of the policy is started, and decreased when the last
instance of the policy is stopped. This prevents the external
module from being unloaded when an instance of one of the
policies it contains may be handling an RPC.

Patch Set 2: I would prefer that you didn't submit this

(3 inline comments)

For example:
ptlrpc_lprocfs_rd_nrs_crrn_quantum -> ptlrpc_nrs_policy_control -> nrs_policy_ctl -> nrs_policy_ctl_locked ->  policy->pol_desc->pd_ops->op_policy_ctl()

If the code path above is possible before the PTLRPC_NRS_CTL_START command is issued, then we're calling into a module by the op_policy_ctl() pointer without a ref count on that module, since the get_module() is called when handling PTLRPC_NRS_CTL_START. I failed to see why op_policy_start needs to be protected against module removal but op_policy_ctl doesn't need it.

--- Message (Nikitas Angelinas, 2013-03-06 11:44:16.000000000) ---
Patch Set 2: (3 inline comments)

The code path example that Isaac gave is possible, and we would be calling into a module without holding a ref count, but the idea was that since nrs_policy_ctl() and ptlrpc_nrs_policy_unregister->nrs_policy_unregister_locked()->nrs_policy_unregister() are serialized by nrs->nrs_lock, then as long as the user calls ptlrpc_nrs_policy_unregister() in the module's exit() function, this should be a safe operation.

But I agree that is not a clean solution; I will rework it probably by having some of the struct ptlrpc_nrs_pol_desc fields held internally by NRS within the ptlrpc module, such that they can be accessed safely.

--- Message (Nikitas Angelinas, 2013-03-06 13:41:59.000000000) ---
Patch Set 3:

Iow, the main idea with this patch was to keep a reference on the module whenever the policy could possibly be handling an RPC, because taking and releasing a reference for each RPC would be unwanted overhead, and it is not how module refcounting is really supposed to work; i.e. ideally there should be one reference from one module to the other. As mentioned in previous comments, I think this patch needs to be reworked so that it does not rely on the user calling ptlrpc_nrs_policy_register() in module->exit(), possibly by copying some struct ptlrpc_nrs_pol_desc fields inside NRS in the ptlrpc module, unless there is a better idea on how the patch should work.
