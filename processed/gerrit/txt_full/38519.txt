LU-13225 utils: bash completion for lfs and lctl

Add a bash completion for "lfs" and improve completion for "lctl".
Rename the "lctl" completion script to "lustre" since the two
commands share helper routines for fsnames, pools, etc. and install
"lfs" and "lctl" symlinks to the common command file.

The completion prints available sub-commands and their options,
and for some sub-commands it completes available arguments
(e.g. mount points, pool names, and MDT/OST names).

A couple of minor changes to "lfs" and "lctl" usage messages to make
the sub-command options easier to parse.  More needs to be done to
make all sub-commands have proper long options.

There is definitely more that could be added to the completions,
but this is a good start and provides a framework for the future.

The pkg-config tool returns /etc/bash_completion.d

 # pkg-config --variable=completionsdir bash-completion
 /etc/bash_completion.d

And when I build with this patch to use @BASH_COMPLETION_DIR@ in lustre.spec.in allows the RPM to build properly:

 # grep -C1 compl lustre.spec
 %{_datadir}/lustre
 /etc/bash_completion.d/*
 %{_includedir}/lustre

 # rpm -ql lustre-client | grep compl
 /etc/bash_completion.d/lctl
 /etc/bash_completion.d/lfs
 /etc/bash_completion.d/lustre

I'd thought that since autoconf uses:

 BASH_COMPLETION_DIR="`pkg-config --variable=completionsdir bash-completion`"

and this reports "/usr/share/bash-completion/completions" on most systems, this should work properly?  I guess our builders do not have bash-completion installed, so "pkg-config" doesn't return anything and it falls back to the default "$datadir" value?

I think it makes sense to default to "/usr/share" instead of "$datadir" in the fallback case (which is the default for upstream anyway), which will fix this problem for the "bash-completion is not installed" case, and if someone has it installed in a different location then the "pkg-config" output will still work properly.
