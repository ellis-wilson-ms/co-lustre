LU-20034 osc: severe spinlock contention in fadvise path

wake_up(&osc_lru_waitq) is called unconditionally on
every page release from osc_lru_unreserve().  When the
VFS fadvise(POSIX_FADV_DONTNEED) path triggers page
invalidation via ll_releasepage -> osc_page_delete,
this results in one wake_up per page, each taking the
waitqueue spinlock.  With many cores calling fadvise
in parallel from many threads, as rclone does, this
causes catastrophic spinlock contention.

Observed on a 256-core system with 64 parallel rclone
transfers: 69% of all CPU cycles in the spinlock, with
effective throughput well below hardware capability.

Calling fadvise on every block is wasteful - it would
be better to let the filesystem handle cache eviction -
and the kernel fadvise machinery is single-page, which
is very inefficient for Lustre's per-page teardown.

Guard wake_up() with waitqueue_active() to skip the
spinlock when no waiters exist.  This is safe because
atomic_long_add() on cl_lru_left provides the store
barrier pairing with prepare_to_wait().  A BUILD_BUG_ON
enforces the atomic type constraint.

Consolidate osc_lru_del, osc_lru_shrink, and the old
osc_lru_unreserve wake_up sites into a single
osc_lru_unreserve() to prevent callers from bypassing
the guard.
