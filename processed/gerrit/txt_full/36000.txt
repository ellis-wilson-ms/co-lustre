--- Commit Message ---
LU-12718 obdclass: Allow read-ahead for write requests

cl_io_read_ahead asserts that read-ahead can only happen
due to CIT_READ or CIT_FAULT requests.
Since LU-9618, we expect CIT_WRITE requests to also
sometimes trigger read-ahead.
So the LINVRNT() needs to be extended to acknowledge
that.

I think modify LINVRNT() in cl_io_read_ahead() to allow WRITE is fine, something like:
diff --git a/lustre/obdclass/cl_io.c b/lustre/obdclass/cl_io.c
index 83f6c00e30..8d88963263 100644
--- a/lustre/obdclass/cl_io.c
+++ b/lustre/obdclass/cl_io.c
@@ -562,7 +562,8 @@ int cl_io_read_ahead(const struct lu_env *env, struct cl_io *io,
        const struct cl_io_slice *scan;
        int                       result = 0;
 
-       LINVRNT(io->ci_type == CIT_READ || io->ci_type == CIT_FAULT);
+       LINVRNT(io->ci_type == CIT_READ || io->ci_type == CIT_FAULT ||
+               io->ci_type == CIT_WRITE);
        LINVRNT(io->ci_state == CIS_IO_GOING || io->ci_state == CIS_LOCKED);
        LINVRNT(cl_io_invariant(io));
        ENTRY;

Further, I am also wondering why  we hit the bug so late, I think it makes big sense
to add a test, eg small sequential write( < PAGE_SIZE) to cover this.

The LINVRNT checks are special, they're only enabled by a particular config/build option, I think DEBUG_CHECK_EXPENSIVE or something (you can look up the macros if you're curious...), which isn't on by default.

The problem is a lot of them *aren't* expensive.  It would maybe be a good use of time to distinguish the checks that are cheap (simple asserts like this) and pull them out from behind that flag, and leave it with actually expensive stuff like "walk the whole list to check a list invariant".
