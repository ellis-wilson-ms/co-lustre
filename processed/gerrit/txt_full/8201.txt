LU-4198 clio: turn on lockless for some kind of IO

We can safely turn on lockless for Direct IO
and no lock.

Direct IO will still enqueue lock in the server side,
and we could not use lockless for in the following case:

1) If group lock is held before DIO, use lockless will
make us deadlock, so we use group lock instead and trust
this to protect consistecy.

2) Direct IO might fallback to Buffer IO in some cases,
and we will restart Direct IO with normal lock holding

The main motivation for this patch is to support AIO.

The problem with this seems that with following case:

1. buffer write
2. Direct Read(lockless) come in, it will try to write dirty pages firstly, which we don't try to hold
LDLM lock, direct read fail and restart with LDLM lock held. buffer write finish and DIO read submit requests to server side.
3.Server side trigger blocking AST since Client side hold LDLM write lock.
4. Client try to cancel write lock, which will try to release pages? But since CL page is held
by DIO read which means they could not be relased..

So the problem here is if Client side have cached lock, DIO should try to match it firstly, so that
DIO won't restart.

At the same time, AIO could not rely on the cached locks as LDLM lock could be canceled but AIO request might have not sent yet.

So to solve the problem, we might need both, always use lockless IO, but if there is cached lock locally, we need try to match the cached lock firstly...this might be a little tricky..

--- Message (Wang Shilong, 2019-07-31 00:39:32.000000000) ---
Patch Set 35:

> (4 comments)
 > 
 > General question.
 > 
 > What happens if, on a client, I do a buffered read (so I have a
 > lock), then a DIO write to the same area (which is lockless) and
 > then read again?

If all this come into same client:

1. We have buffer read and hold read lock.
2. DIO write come in, osc_lock_enqueue() will call osc_lock_enqueue_wait() to cancel local client
conflict lock even we are lockless before return.
3. READ Lock will be canceled on this client.
4. DIO write continue. to send RPC to server side.
5. if there are other read/write lock conflicts on other client, server will send blocking AST to those clients to cancel lock.
