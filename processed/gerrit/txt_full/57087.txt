--- Commit Message ---
LU-18462 nodemap: sanity checks for positional parameters

A number of nodemap commands just take positional parameters instead
of options:
- nodemap_activate
- nodemap_add
- nodemap_del
- nodemap_test_nid
In this case, we need to check manually that we got the correct number
of arguments, and return a proper error message otherwise.

These commands are also modified to grok named options, in addition to
positional parameters that we will have to keep for some time for
backward compatibility reasons.

> Sebastien, is there any plan (or opportunity in this patch) to reduce the use of positional parameters?  Generally, positional parameters are fragile and should be avoided, but for compatibility they could be kept for some releases and named options should be added for most of the arguments.

I agree positional parameters are not good by nature. There are 4 'lctl nodemap*' commands that still use them:
- nodemap_activate
- nodemap_add
- nodemap_del
- nodemap_test_nid
'add' and 'del' could take a '--name' option, and 'test_nid' a '--nid' option. For 'activate' the command is so straightforward that it is not easy to make it take a parameter. Or maybe we could switch to what Marc is proposing, with a command to activate and one to deactivate?

But indeed, adding named options would need to be done in addition to positional parameters to not break compatibility, and we would have to keep both mechanisms at least for some time.

I'm not against having a *single* positional parameter for a command, like "lctl nodemap_add nodemap_name" since I don't think this is confusing and having to use "--name" can become annoying.  Imagine if we had to use "cp --source fileA --target fileB" every time...

That said, I'm *not* a big fan of commands that take multiple positional parameters, since this can become particularly confusing when some of them become optional.  Back in the day, "lfs setstripe" always needed to specify "stripe_count [stripe_index [stripe_size]]" in order (or something like this), which was a disaster for users.
