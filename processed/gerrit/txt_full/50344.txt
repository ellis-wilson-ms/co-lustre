--- Commit Message ---
LU-16649 llite: EIO is possible on a race with page reclaim

We must clear the 'uptodate' page flag when we delete a
page from Lustre, or stale reads can occur.  However,
generic_file_buffered_read requires any pages returned from
readpage() be uptodate.

So, we must retry reading if page truncation happens in
parallel with the read.

This implements the same fix as:
https://review.whamcloud.com/49647
b4da788a819f82d35b685d6ee7f02809c05ca005

did for the mmap path.

Yingjin pointed out that if we don't revert "check vmpage in releasepage", the test for this doesn't work.  That's true.

As Yingjin noted, the check in releasepage() (from LU-14541) is A) correct, and B) should keep us from clearing pageuptodate on pages which are in use, which is the cause of the EIO here and the SIGBUS which Panda fixed earlier with seqlock + retry trick.

BUT we see both of these problems with the releasepage() check in place.  (We have seen the EIO at a customer site with the releasepage check in place, even though I couldn't reproduce it locally with the releasepage check in place.)

The issue is that we clear pageuptodate whenever we call cl_page_delete().

So I think the problem is with our other calls to cl_page_delete(), which do not check vmpage refcount to see if the page is in use.  If we fix those, we should not need the retry/seqlock in either mmap or regular read cases.  (By the way, it may also be possible to get EIO on *writing* a page if it's already in cache rather than created for the write...)

So I am going to look at those calls to cl_page_delete() and see how we can include checks for vmpage refcount there.

Yingjin, Andreas, Panda,

I think it might make sense to stick with the seqlock()+retry fix for 2.15 and other branches, but we should *not* revert LU-14541.

I agree that "change - including reverting the LU-14541 patch - is correct."

But after the newer kernels (such as Ubuntu 2204) introduces mapping->invalidate_lock, we could get rid of all these seqlock checks (which may have impact on the performance) for the newer kernel.

One blocking is that invalidate_mapping_pages() is called in the kernel without holding this invalidate_lock. The reason is that the newer kernel thought the this function only release pages that are not in use or under I/O.

So I suggest that we can use LCC ((ll_cl_context) to check in ->releasepage() that whether the current page is under I/O (read path) and belong one read context of a process. we can release the page only when the page is not under I/O or in use.
Or a more coarse grain solution, we can account for an inode the number of processes doing read() and fault(): increase this value when call ll_lcc_add and decrease it when remove the LCC. if this count is larger than 1, we could not release the page...
I think this suggestion is doable. I searched the kernel source tree, btrfs_releasepage seems to do the same check.
