--- Commit Message ---
LU-15485 lprocfs: preallocate all lprocfs_stats

Allocating the stats as-needed leads to fragmentation.

lprocfs_alloc_stats() preallocates only for IRQ_SAFE stats.  It does
this so that it doesn't have to try to allocate memory in an interrupt
handler. When the stats are allocated as-needed, this can lead to
fragmentation. Change lprocfs_alloc_stats() to preallocate all stats.

> It looks like LU-17504 will resolve this, no?

If you mean patch: https://review.whamcloud.com/54365 "LU-17504 build: fix array-index-out-of-bounds warning" then I think it will avoid part of this issue, but not all of it. 

 > I see it's the opposite. I wonder if we should use a page pool or
 > slab instead?

No, a pool is only useful for short-lived allocations. These stats allocations are long lived so having a pool would not help.  I don't think a dedicated slab would help either, since most of the allocations are different sizes. 

The conclusion in LU-17504 is that the only user of the IRQ stats was the memory allocation tracking, and converting that to a percpu_counter() reduces the Lustre-specific code and avoids the need for the IRQ usage. 

Ideally, the rest of the lprocfs_stats code would also be moved over to the kernel percpu_counter() code and not implement its own code. That said, some care should be taken that the memory allocation does not get too large, since the alignment can sometimes be painfully large.
