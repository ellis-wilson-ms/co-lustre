LU-957 scrub: OI scrub against ldiskfs

The OI scrub is totally implemented inside osd-ldiskfs,
and driven by the inode iterator. For each object with
normal fid, its fid in the LMA and related OI entry in
the OI file will be compared, if they are inconsistent,
then the OI entry will be updated.

Patch Set 1:

Some comments on this long discussion:

- inode table iteration is (or was) also useful for quota checking.  With the move to e2fsck doing quotacheck, this use case is no longer present.
- readahead vs. OSD thread: I'm fine with prefetch vs a dedicated thread.  That said, we should be starting OI scrub whenever LFSCK is run, so is there much difference?
- I'm fine with using UUID vs. FID for detecting backup/restore.  While it is possible for the UUID to be changed, this isn't a very common case.  It definitely DOES make sense to start OI scrub thread immediately at mount, since it can do a lot of work before clients start sending requests.
- on upgrade from an older Lustre (without OI Scrub) the first mount should do a preliminary OI scrub, which should be fairly fast & efficient since the OI should be correct.

--- Message (Fan Yong, 2012-04-19 01:09:32.000000000) ---
Patch Set 1:

> Andreas Dilger		2:37 AM

> Patch Set 1:

> Some comments on this long discussion:

> readahead vs. OSD thread: I'm fine with prefetch vs a dedicated thread. That said, we should be starting OI scrub whenever LFSCK is run, so is there much difference?

Enable OI files checking/repairing for each LFSCK run, means OI scrub will not be an optional for user space command, but a default requirement for LFSCK triggered by user space command. It can simplify OSD layer logic. But consider ZFS-based case, such dedicated thread is just nop, so do not know whether it is Alex want or not.

> on upgrade from an older Lustre (without OI Scrub) the first mount should do a preliminary OI scrub, which should be fairly fast & efficient since the OI should be correct.

When creates "scrub_file", it may be:

1) a new formated system.

2) upgraded from old version directly.

3) restored from old version backup.

For 1) and 2), it is unnecessary to run OI scrub. For 3), the OI scrub should be run. Unfortunately, the system cannot distinguish above cases by auto detection. So there are three options for that:

a) Trigger OI scrub automatically when mounts MDT in spite of whether it is really needed or not.

b) OI scrub will be triggered by lookup-by-name when invalid OI entry is found later.

c) The sys admin will trigger OI scrub manually after MDT mount up according to the real case.

Which one is better?
