--- Commit Message ---
LU-16026 mdt: remote subdir mount cache ROOT LOOKUP lock

If subdir is a remote directory, the subdir mount is inefficient
because client can't fetch LOOKUP lock, and tries to lookup ROOT
in each file access.

This patch includes changes on both server and client:
* mdt_get_root() returns fileset parent FID along with fileset FID
  if subdir is a remote directory.
* ll_inode_revalidate() replaces parent FID and name if inode is
  ROOT, and is a remote directory too.
* lmv_intent_lookup() sends request to parent MDT if it's looking
  up ROOT of remote subdir mount.
* simplify mdc_intent_lock(): it will try revalidate lock if FID is
  sane, and if no lock found, continue lock enqueue.
* update sanity 247f/247g, add 247h.

(5 comments)

what I don't understand about this patch is why the actual "ROOT/" directory does not have the same problem?  Exposing the parent FID to the client is a bit of a security leak, but more importantly it isn't clear why the client needs a lock on the parent at all? 

Once the fileset name->subdirectory FID lookup is done (once at mount time), the fileset name should not matter anymore, and only the subdirectory FID should be needed.  We shouldn't support anything like the subdir being renamed and replaced with a new directory while it is mounted on the clients.
