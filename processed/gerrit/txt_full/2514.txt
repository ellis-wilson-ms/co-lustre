--- Commit Message ---
LU-744 osc: add lru pages management - new RPC

Add a cache management at OSC layer, this way we can control how much
memory can be used to cache lustre pages and avoid complex solution
as what we did in b1_8.

In this patch, admins can set how much memory will be used for caching
Lustre pages per file system. A self-adapative algorithm is used to
balance those budget among OSCs.

Patch Set 10: I would prefer that you didn't submit this

(4 inline comments)

The problem I see with this patch is that it is moving in the wrong direction.  Administrators want to be able to specify the cache limit for all Lustre filesystems on a node, while adding a cache limit per OSC doesn't really improve anything for them.

At a site like LLNL, they have over 3000 OSCs on the client, so any per-OSC limit will either have to be so small that it hurts performance, or it will be so large that it is much larger than the total RAM, and effectively no limit at all.

I'd rather see more effort put into understanding why Lustre IO pages do not work well with the Linux VM page cache management, and fix that.  This will provide global cache management, and avoid memory pressure for all users.

--- Message (Brian Behlendorf, 2012-07-18 19:17:28.000000000) ---
Patch Set 12: I would prefer that you didn't submit this

(7 inline comments)

Just a few quick comments.

This is better, but I don't think recreating exactly the 1.8 behavior is what we want.  The 1.8 implementation was good enough, but not never exactly what was needed.  I'd like to get it right this time.

For example, this patch preserves setting the cache limit on per-filesystem basis.  Since we typically will mount 3-4 different Lustre filesystem on a client that's a problem.  What we really need is a global limit at the node level as Andreas mentioned.

I'm also skeptical that adding these additional LRUs to Lustre is the right way to go.  We _should_ be able to leverage the existing Linux VM rather than invent our own.

While the kernel still doesn't provide a single sysctl to limit the total page cache size (although there are various patches floating around for this).   This shouldn't be a huge issue as long as the Lustre client can quickly invalidate clean pages.  It's my recollection that it couldn't for large memory systems in 1.8 and that was a significant part of the problem which has to be fixed.  Failing that, these days I'd expect us to be able to use cgroups to effectively manage the page cache size.

The related problem of limiting dirty pages has been addressed for a while now (kernel-2.6.28) with the vm.dirty_ratio and vm.dirty_bytes sysctls.  We should verify Lustre honors this and consider ditching the existing cl_dirty_max limit.

That said, we'll probably still pull the patch as is and give it a little more testing to get you some additional feedback.
