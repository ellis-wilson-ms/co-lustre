LU-4009 osd: support for ZIL

all the updates go to ZIL, osd_sync() and osd_object_sync() flush
the whole log.

it's hard to turn all Lustre transactions into a linear sequence
as within a single TXG (actaul commit unit) Lustre puts many
transactions modifying same resources in arbitrary order (so that
transaction T1 can get into ZIL before transaction T2 while T1
depends on T1).

basically ZIL log contains two type of records:
one stores transaction which is a set of updates (see OUT),
another record is a barrier simulating TXG commit, so
we're sure all running transactions have completed by a
barrier and all potential dependent transactions get into
ZIL by this point. during ZIL replay we collect all the updates
in memory, once a barrier is met we apply all the updates, including
indirect writes.
