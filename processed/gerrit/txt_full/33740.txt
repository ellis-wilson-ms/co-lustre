--- Commit Message ---
LU-11415 socklnd: improve scheduling algorithm

Modified the scheduling algorithm to use all scheduler threads
available. Previously a connection is assigned a single thread
and can only use that one. With this patch any scheduler thread
available on the assigned CPT can pick up and work on requests
queued on the connection.

In conclusion, I think both types of tests need to be carried out to ensure that the code works optimally in both cases.

Patch Set 2:

If we assume that the objective of Lustre is to provide the best overall throughput for all clients and not just one specific client, then 1:1 testing does not make sense.  With this objective, 1:1 is not a subset of 1:N because optimizing 1:1 does not mean you are improving the situation for 1:N.

When a client is talking to OSSs, it views the world as 1:N so the bulk operations should spread out over multiple connections.  If the NIDs hash evenly out over the CPTs (this assumption is a flaw in LNet in my opinion and could be better), then you get a nice balance of work and don't get stuck on CPT locks like you would in the 1:1 scenario.  The lock bottleneck of 1:1 is not a bottleneck for 1:N and, therefore, optimizing for 1:1 won't help 1:N.  Especially if that optimization makes 1:N worse, it is work that is harmful rather than helpful.

Bulk operations are responded to by the server's as they have the resources to to complete them.  So the servers are really in charge of how fast bulk operations happen.  Because servers are managing many clients at once in a typical cluster, they also view the world as 1:N in their responses.  Balancing those out over the CPTs help to optimize network resource use and to avoid locks improving the overall throughput of the Lustre cluster rather than the throughput of an individual node.  Again, my argument is that focusing on the performance of a 1:1 scenario is not realistic for either client or server and optimizing for 1:1 does not imply an improvement of the overall cluster which is the whole point of using Lustre.

I'm not at all disagreeing with this patch and think it is good. I'm disagreeing with the methodology used to measure the performance value of such changes.

That's correct. Previously we had a scheduler per thread. So when you bound a scheduler to a connection, the connection could only use the single thread. Now the scheduler is per CPT and all the socknal_schedulers running on that CPT share the same queues. This way when you bind a scheduler to a connection, any of the threads can wake up pickup a connection from the queues and work on it.
