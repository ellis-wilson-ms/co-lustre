LU-6824 ldiskfs: give warning with dir htree growing

Currently without large dir feature, ldiskfs directory hash tree
will be limited 2 height, this means directory size is limited about
1GB, and in fact users are likely to hit ENOSPC when reaching half of
limit because of bad hash. tested by following scripts.

i=0
filename="aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa"
"aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaabbbbbbbbbbbbbbbbbbbbbbbb"
while [ 1 ]
do
        touch $filename"$i"
        if [ $? -ne 0 ];then
                break
        fi
        ((i++))
done

When directoy size grow about 590M, we hit ENOSPC. Better way is to
add support to e2fsprogs so that we can use large dir feature.
As a walkaround way, this patch try to give warning messages to
console when 10/16 and 11/16 of limit reach.

So this patch will try to give following messages when warning
limit or hash index tree limit reach:

Directory (inode: 8388610 FID: [0x200000401:0x1:0x0]) has approached
maximum limit.

Directory (inode: 8388610 FID: [0x200000401:0x1:0x0]) is approaching
maximum limit.

FID output here is useful for administrators to locate lustre
file path

Patch Set 1:

I think for this patch there needs to be some kind of ratelimit on the ext4_warning() messages that are printed to the console.  One option is to put a flag into the in-memory ext4_inode_info EXT4_STATE_* flags that indicates this message was printed for the 50% full case, and it won't print again until it hits 75% or the inode is dropped from memory and the flag is lost.  It would also be better to use d_path() instead of making a new version of that code.  It should be possible to create a vfsmount from the superblock and the s_root dentry something like:

    struct vfsmount vfsmount = { .mnt_root = sb->s_root, .mnt_sb = sb };
    struct path path = { .mnt = &vfsmount, .dentry = dentry };

that is good enough for calling d_path().


I haven't looked at how much effort is needed for adding the INCOMPAT_LARGEDIR feature to e2fsprogs.  I think handling the 3-level htree in e2fsck is probably relatively simple and can be tested with 1KB blocksize directories with only about 100k files (as local ldiskfs mount, not Lustre).  It might be a bit more work to fix the i_size_high usage for directories instead of only regular files, but I think that is mostly a grep for "i_size" or "i_dir_acl" usage and changing the code.  I think it might be reasonable to get a working version in a few days, and then a few days to make e2fsprogs regression tests.  A few more days would be needed to verify that the 3-level htree + > 2GB directory size was working in the ldiskfs code.

There shouldn't be any existing directory inodes with i_dir_acl != 0, since e2fsck already checks for this, so there isn't any danger to existing directories if this suddenly becomes i_size_high if INCOMPAT_LARGEDIR is enabled.

To use INCOMPAT_LARGEDIR would not need any reformatting of the MDT filesystem, only upgrading e2fsprogs and turning the feature on the filesystem via "tune2fs -O largedir /dev/MDT".  There might need to be fixes to the ext4-large-dir.patch in master (and ext4-pdirop.patch in b2_5) if it doesn't handle this new feature correctly.  Since this doesn't change the network protocol it doesn't need any changes to the client or OST, on the MDT ldiskfs.

--- Message (Andreas Dilger, 2015-10-23 01:53:43.000000000) ---
Patch Set 3:

The inode already has the FID in the "lma" xattr, so it wouldn't need to be passed down.  The problem is that ldiskfs itself doesn't really "know" about the FID, and the "lma" xattr will not exist for non-Lustre filesystems so this patch couldn't go upstream like that, but it probably would be a much smaller patch than the current one.

Another solution would be to fix the osd-ldiskfs caller to fill out the dentry more fully so that d_path() doesn't oops, but prints out the file and/or directory name before returning.

Another option is for ldiskfs to pass up a different error than -ENOSPC to the callers when the directory is full (e.g. -ENOBUFS "No buffer space available", which would be reasonable in this case also) and then have osd-ldiskfs print a more useful error message including the FID for use by "lfs fid2path".

Of course, the client application should also be getting the ENOSPC errors returned if the directory is full, and it has the full path, so it would be better if the application handled this error properly.

The best solution would be to fix e2fsck to allow 3-level htree, so that the directory wouldn't become full, and there is no need to print an error message.
