--- Commit Message ---
LU-14708 ptlrpc: skip unnecessary client eviction

A server does not update last_rcvd file on connection of new clients
synchronously. If the server fails over before the last_rcvd update is
committed, recently connected client may find itself evicted
unexpectedly.

If a client has not cached any data from a server and has not
performed any modifying rpcs to the server - let the client to connect
as a new one instead of considering itself as evicted.

Test to illustrate the issue is included.

The mounting was made async years ago because sync writes are much to slow for HDD OSTs and mounting many clients. For a 10000-client system with 100 IOPS on an HDD OST this would take 100s to mount the filesystem. 

Instead, there is a commit callback and export flag that marks the next client write sync only if the client modified the filesystem right after initial mount, but before the last_rcvd is committed to the journal.

There needs to be some other way to handle the client connection if the server reboots right at mount...

--- Message (Andreas Dilger, 2021-05-27 18:05:15.000000000) ---
Patch Set 1:

Unless something is configured strangely with your OSD filesystem, the commit callback that commits the last_rcvd updates and clears the "need_sync" flag should finish within 1-2 seconds of the client mount.

Note that it was Cray that originally requested this async last_rcvd method, because mounting many clients in parallel was too slow with sync writes for each client (at most 100 clients mounting per second, limited by the HDD seek rate of OSTs).

The main question is whether it is better to handle the very rare case where there is a server crash within 1s after a client mounts (but hasn't actually done anything), or to make millions of normal mounts slower?  Another possibility would be to have the client handle this case more transparently?
