LU-16807 ksocklnd: remove LASSERT that could cause a deadlock

New kernels report the following possible deadlock:

WARNING: possible irq lock inversion dependency detected
--------------------------------------------------------
tuned/1273 just changed the state of lock:
ffffffffa05af070 (&ksocknal_data.ksnd_global_lock){++.?}-{2:2},
		  at: ksocknal_data_ready+0x2b/0x110 [ksocklnd]
but this lock took another, SOFTIRQ-unsafe lock in the past:
 (&tcd->tcd_lock){+.+.}-{2:2}

and interrupts could create inverse lock ordering between them.

other info that might help us debug this:
 Possible interrupt unsafe locking scenario:

	CPU0			CPU1
	---- 			----
lock(&tcd->tcd_lock);
			local_irq_disable();
			lock(&ksocknal_data.ksnd_global_lock);
			lock(&tcd->tcd_lock);
	<Interrupt>
lock(&ksocknal_data.ksnd_global_lock);

 *** DEADLOCK ***

This is due the use of tcd_lock in the LASSERT to ensure
sk_data_ready is set correctly. In all the years of Lustre this
has never been reported. Mostly like if it did report a dead lock
would happen. Since this LASSERT is unlikely we can just remove
it.
