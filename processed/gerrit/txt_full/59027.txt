LU-0000 dbg: Debug Gerrit AI bot in sample patch4

ptlrpc: Add alternate fileset data structure handler

Handling multiple future alternate filesets requires an in-memory data
structure that is synchronized with the nodemap IAM storage for
filesets. Specifically, it is necessary to properly handle gaps in the
cluster idx subid range that is reserved for filesets which occur when
filesets are deleted.

To easily synchronize the IAM filesets with the in-memory nodemap, an
this patch implements an ordered associative array, rb tree in this
case, which is naturally ordered by a node key - in this case the
global fileset id that can later be directly mapped to the IAM cluster
idx subid range for filesets.

This patch sets the maximum number of alternate filesets to 255 (+1
primary fileset).

This patch only includes the management of the fileset rb tree and
includes all necessary functions. The next patch connects the rb tree
to the IAM and add the API for adding and deleting filesets.

    *   **Locking:** The commit adds `nm_fileset_alt_lock` to `struct lu_nodemap` but doesn't show its usage. While usage is likely intended for subsequent patches that modify the tree, it's crucial that *all* accesses (read and write) to the `nm_fileset_alt` rb-tree in future code are properly protected by this lock. The current patch doesn't add any locking calls around the tree operations themselves within `nodemap_fileset_alt.c`, which is acceptable as these are helper functions, but the callers *must* handle locking.
    *   **Error Handling in `fileset_alt_add`:** The function returns `-ENOSPC` if the generated `fset_id` exceeds the maximum. This seems correct, but consider if there are scenarios where a caller might provide a specific ID that is too large – the check should ideally happen regardless of whether the ID was auto-generated or provided. (Currently, it only checks the auto-generated ID before the loop). *Correction:* On closer look, the check `if (fileset->nfa_id > LUSTRE_NODEMAP_FILESET_NUM_MAX - 1)` happens *after* the potential auto-generation, so it covers both cases. This is fine.
    *   **`fileset_alt_resize` Robustness:** The `fileset_alt_resize` function uses `OBD_ALLOC` to get temporary memory for the resized path. If this allocation fails, it prints a `CERROR` and continues to the next node. This means some filesets might remain un-resized if memory is tight during the resize operation. This might be acceptable, but it's worth noting that the memory optimization might not be fully achieved in low-memory situations.
    *   **ID Assignment:** The `get_first_free_id` function iterates the tree to find the lowest available ID starting from 1. For a tree with many entries and few gaps near the beginning, this could be slightly inefficient, but given the maximum of 255 alternate filesets, the performance impact is likely negligible in practice.
*   **Commit Message:**
    *   The commit message is reasonably good. It clearly states the purpose (adding the data structure for alternate filesets), mentions it's preparatory, and outlines the core change (rb-tree implementation).
    *   It correctly identifies the next steps (connecting to IAM and adding APIs).
    *   The `Test-Parameters: fortestonly` tag suggests this isn't intended for production use yet, which aligns with its preparatory nature.
    *   It could perhaps explicitly mention the modification to `lustre_disk.h` and the rationale for the chosen `LUSTRE_NODEMAP_FILESET_NUM_MAX`, but the current level of detail is acceptable.

Overall, this appears to be a solid preparatory commit laying the groundwork for managing alternate filesets in nodemaps. The main review focus for subsequent patches will be the correct integration with IAM storage/loading and the proper use of the `nm_fileset_alt_lock`.
