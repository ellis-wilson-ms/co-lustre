Patch Set 1: Code-Review-1

I would prefer to avoid adding a separate SEL watermark, for the same reasons that Patrick described earlier.  This is just one more thing that users will not know about and/or get wrong, so "doing the right thing" automatically is far preferred.

You are correct that my previous comments were confused, since SEL is not normally *allocating* new objects when it is extending the layout.  

My intent was that SEL should be allowed to continue extending existing components on a "full" OST more than new files/components can allocate new objects there.  The reason is that if *all* files are SEL then they will all "immediately" stop using the OST once it hits the lwm, which I thought might be a problem.

*However*, if one considers that SEL will mostly be used on smaller flash OSTs (e.g. 8-32TB) rather than large disk OSTs (100-500TB+) the 0.1% lwm of a flash OST is only 8-32GB and that is only a second or two of full-bandwidth writes.  That is in the margin of error for a one-second statfs so already close enough to stop using the OST immediately.

In that light I'm OK going back to the original implementation using the existing lwm limit for SEL, rather than introducing additional complexity.  Sorry for the confusion.

--- Message (Patrick Farrell, 2019-06-12 14:50:29.000000000) ---

This brings up a point that I have sort of ignored.

In some ways, checking using:
"available space - required extension space"
is silly.  In most cases, the amount of "extension space" is small compared to the total space on the OSTs, so including it in the calculation has a relatively small effect on the behavior.

If we skipped including extension space in the calculation, it would be easy to copy exactly the stripe allocator behavior with the low and high thresholds.

However, this assumes that extension space (on a per OST basis) is small relative to the threshold sizes.  If extension space is small relataive to the thresholds, then it makes little difference if we include it or not.  But if extension space is large relative to the thresholds, then it's a bad idea to not include it.

Just something to consider.  I think it's best to keep the extension space included to cover the case where it is large relative to the thresholds.

Anyway - I think it's a good idea to use both thresholds like Vitaly suggested, in imitation of the stripe allocator.
