Hey Yu,

Thank you for your comments.

Firstly, you suggestion regarding code duplication with ost-pools.sh is reasonable. However, the goal of this patch is to make test 200 go separately without fails. Moving the ost related stuff to test-framework is not so good idea as it might seem. 
Looks like the history of ost pool testing is as follows: the tests 200 and 201 had been created and added to sanity before the ost-pool.sh, and the code of ost-pool.sh is definitely based on them.
(Interesting that I didn't refer ost-pools.sh while have been working on combining the 200a-h and 201a-c). Probably it is better to remove the test_200 from the sanity suite totaly. In this case we, of course, have to revisit the ost-pool suite and supply it with missing coverage.
Both moving shared functions to test-framework or revisiting missing tests and putting them in ost-pool will take more time than expected for this task. What are the cons and pros? I also didn't plan to refactor both tests. Can we let this patch set go to master and make this refactoring later after we discuss with community what is the best way to do with ost pool testing?

Secondly, while preparing the patch I've been trying to keep  the code of tests 200 and 201 as original as possible, so we do not risk to change the logic. Also, I tried to do not modify the code that is not required to reach the goal, that is why you've seen style issues.

I resolved all your style related comments in the patch set 2. Could you please, review.

Thought?
