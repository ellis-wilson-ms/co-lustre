Patch Set 1: I would prefer that you didn't submit this

I'm concerned with disabling test 101a. The whole point of that test is to ensure that random reads do NOT read too much data. Even of the readahead code is growing reads initially because it thinks the reads are sequential, it should still be tracking the original read size and if the reads are always random 64kB chunks then growing the reads to 1MB is a lot of overhead.

Patch Set 2: I would prefer that you didn't submit this

(2 inline comments)

> I thought the whole point is that read 1M is always more
> efficient, since the rpc round time is almost the same no
> matter how much you will read (4k or 1M)?

I think it is a reasonable assumption that files will be read sequentially until shown otherwise, so small files could start with a full 1MB read on first access, and only do smaller reads if the file begins to have random accesses.

However, I also think that if the file is seeing random file IO, then reading in full 1MB chunks for every read can cause a lot of overhead and memory wasteage if the reads are small. 

Do we have a sanity.sh test for max_read_ahead_whole_mb?  This string doesn't appear in lustre/tests/*.sh, so I suspect not.

However, I also think that if the file is seeing random file IO, then reading in full 1MB chunks for every read can cause a lot of overhead and memory wasteage if the reads are small. "
