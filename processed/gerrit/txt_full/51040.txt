LU-8802 obd: remove MAX_OBD_DEVICES

Remove this arbitrary limit by reimplementing the array as an
Xarray. Xarray can grow and shink dynamically, hence saving
memory and allow for many more OBD devices. There is still
technically a limit OBD_MAX_INDEX, which is xa_limit_31b.max
or around 2 billion. This is far more than is practically
useful.

This patch also adds various iterators for OBD devices, which
are used to simplify code in various places.

Removing class_obd_list() since it is unused. Rename
class_dev_by_str() to class_str2obd() to keep the pattern.
Several class_* functions have been refactored to improve
locking. The larger issue of OBD device locking will be
addressed separately.

Update the OBD device lifecycle test to try loading
more devices (about 24,000 for now).

Currently, adding an additional OBD device is an O(n^2)
operation due to the class_name2dev calls in
class_register_device(). This will be addressed in a future
patch adding a hash table for OBD device name lookups.

Further, OBD life cycle management could likely be simplified
by using Xarray marks. Right now, it is handled by a bit
field in the obd_device struct. Since the scope of the changes
needed to simplify this seem large, this will also be addressed
separately.

Understood that this is WIP, but thought I'd provide some feedback early before you invested too much time to polish this.

I'm happy to see this limit be removed, but I think more could be done to isolate the implementation and improve the interfaces (both semantically and performance wise).

It isn't clear whether using XArray will solve some of the main issues here. Sure it avoids the static pointer array size by having a flexible array size, and should still allow direct by-devno access into the array. I don't recall whether XArray has a mechanism to find empty index values without a linear scan, or if it would be paired with IDR to keep track of free device numbers?

However, AFAIK XArray doesn't address any of the full list/array walking issues that arise from having to do a full scan for device names/UUIDs. IIRC (will have to look again), but Alexey's patch for LU-147 allowed hash table lookups by name/UUID.  Don't be put off by the use of cfs_list_* in that patch, since they were thin wrappers for list_*, and cfs_hash can be replaced with rhashtable.  The reason for the revert was due to a few bugs with the patch right before a release, and it never made it over the hump again to be included. 

I generally think that implementation is a good reference (having now refreshed my memory), though it does still have a hard-coded limit of 65536 devices due to the use of a bitmap for allocating the device numbers. At the time that was justified by the 65532 limit of OST devices, but that no longer holds with hundreds of concurrent mounts on a single node due to containers and per-user mountpoints. If that was replaced by an IDR it could be much more scalable.

IDR only goes up to UINT_MAX at most. It seems like it was deprecated in favor of Xarray according to https://docs.kernel.org/core-api/idr.html#id-allocation, which can go up to LONG_MAX. Xarray seems to have similar features to IDR with regards to finding free index values. Using an Xarray instead of a list makes sense to me, after looking into it more. 

Xarray's don't appear to have a quick way to do name/UUID lookups. The hash tables might be needed regardless of the underlying structure for storing the OBD devices. I think the final approach would end up as a mix of Xarray/rhashtable. Xarray/rhashtable combo would give fast lookups for devno, name, and UUID.

I'm going to refactor this patch to clean up the ugly hacks and try out an Xarray. The hash tables I might do in a third revision.

xa_for_each_marked(&xarray, index, obd, OBD_SET_UP) {
}
for health_check_show().
