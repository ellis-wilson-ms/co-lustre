> Yes 'if (!ptr)' is required over 'if (ptr == NULL)'. The upstream checkpatch reported if (ptr == NULL) or if (ptr != NULL) as style errors. Also required is if (rc) or if (!rc) form as well. This is not reported by checkpatch yet but Greg has demand this change as well.

You're free to make these changes when you upstream patches. But I'll continue to ask for explicit comparisons when I review lustre patches. My feeling is that this is a lazy shorthand which values the time of the author over that of the reader.

(Note also that the upstream tree contains thousands of explicit comparisons.)

Here is something I wrote about this for internal purposes:

I would like to update our coding guidelines to suggest using explicit comparisons in conditionals. That is, I think we should use if (p == NULL), (p != NULL), (rc < 0), (rc == 0) rather than (p), (!p), (rc), and (!rc). While they require marginally more typing, the explicit
comparisons are more readable. By more readable I mean that the the reader can parse and understand them faster than the implicit forms. This is mostly because they make failure look like failure. Compare the explicit comparisons:

 
  p = alloc_gater();
  if (p == NULL)
    ...

  rc = do_stuff();
  if (rc < 0)
    ...

to the implicit versions:

 
  p = alloc_gater();
  if (!p)
    ...
 
  rc = do_stuff()
  if (rc)
    ...

With implicit conversions the way we write the test for failure depends on whether the function returns an interger or a pointer. I think this is bad. BTW it gets much worse when the test is used with && or ||.

 
  if (!rc && itp->d.lustre.it_lock_mode)
  if (!rc && data.mcud_usercount == 0)
  if (!rc || (rc != -ENOENT && i))
  if (loi->loi_ar.ar_rc && !rc)

Note that when the value already has type bool then it should have a word like 'is' in its name and there is no need for comparison:

 
  if (is_registered)
  if (already_registered)
  if (IS_ERR(p))
