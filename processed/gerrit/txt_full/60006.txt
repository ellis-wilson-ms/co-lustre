LU-18357 ptlrpc: Fix fileset llog compatibility on nodemaps

There are currently two issues related to old nodemap llog fileset
handling that can cause inconsistent state across servers during
nodemap synchronization:

(1) - IAM filesets are incorrectly migrated: When a nodemap is deleted
and recreated with the same name within a single IAM synchronization
step, the new nodemap keeps the fileset from the old nodemap. This
occurs because the "nmf_fileset_use_iam" flag defaults to 0,
triggering unnecessary fileset migration (used for llog filesets) even
for IAM filesets. The result is an inconsistent state where MGS has no
fileset while MDS/OSS retain the old fileset.

(2) - Stale llog filesets persist after nodemap deletion in the params
llog: When using llog-based filesets, deleting a nodemap without
explicitly removing its fileset causes the fileset to reappear when a
new nodemap is created with the same name. This is long-standing bug
and occurrs any time a fileset is set on a nodemap that has no IAM
nodemap defined.

This patch addresses both issues:
- For (1), "nmf_fileset_use_iam" is set to 1 by default for all newly
  created nodemaps. Note, nodemaps of older Lustre version with llog
  filesets keep 0 for this flag to allow backward-compatibility. Now,
  llog filesets can be set even if some IAM fileset was set before
  (this was not possible before). However, while IAM filesets can
  "overwrite" llog filesets, the opposite remains not possible.
- For (2), llog filesets are now automatically cleaned up from the
  params llog whenever a nodemap is deleted or when an IAM fileset is
  set (or cleared) via "lctl nodemap_set_fileset". This is done by
  internally triggering the equivalent of
  "lctl set_param -P -d nodemap.NODEMAP_NAME.fileset"
  to remove stale llog entries across all server targets.
  This is only done when we detect that a fileset was indeed set
  via the llog and whenever a fileset is explicitly cleared.
  The latter is important as otherwise stale filesets may not
  be detectable because the MGS is not reading the params llog.

Sanity-sec 27ac was added to test this fix.

Patch Set 5:

(2 comments)

Thanks, Sébastien, for reviewing so quickly! 

> This is certainly not a trivial patch, and if I were to explain it, I would say basically this patch does not keep llog-style filesets, but replaces them with IAM-style filesets, and removes the old-fashion filesets. Is that correct?

Not quite; We can't replace them as we go because of a possible downgrade, where we would then lose the filesets. The easiest "fix" would have been to simply replace the IAM flag default to 1. However, then set_param can no longer be used for filesets, which is not what one would expect from a deprecated option. And this is why it became so complicated.

Instead, this patch now expects that filesets are IAM filesets. It was the opposite before and caused the (1) bug during migration. Now, we allow llog filesets even after IAM filesets were used, but only when no fileset is set. Previously, this was only possible on fresh nodemaps. Although this was not _really_ the goal, it makes our lives easier. Alternatively, we need to resort to another IAM flag to block llog filesets entirely. I think this is not needed, though, and it is simpler as implemented in this patch.

The fix for the (2) bug is only to clean up the params llog after ourselves. Otherwise, the filesets would have stayed in the params llog forever and come back to "life" under various nodemap operations. Fixing (2) is probably the bulk of this patch because we unfortunately have to travel back to the MGS layer to issue a "set_param -P" command.

> Do you plan making a test for that?

Yes, I'll make a test to demonstrate this - this will hopefully also make it clearer.
