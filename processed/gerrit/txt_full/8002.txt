LU-4106 scrub: Trigger OI scrub properly

There is the following race case between osd_fid_lookup() and object
unlink/detroy:

Both RPC service thread_1 and RPC service thread_2 try to find the
same obj_A at the same time. At the beginning, the obj_A is not in
cache. The thread_1 is in osd_fid_lookup() and finds the OI mapping
for obj_A. But before the thread_1 finding out related inode_A, the
thread_2 moves faster and finds the inode_A and unlinks the inode_A.
So the thread_1 will fail to find the inode_A. Under such case, the
thread_1 will try to check OI again to make sure whether related OI
mapping is still there or not. If no OI mapping, then it is normal
becuase someone has unlinked the file by race; otherwise, it may be
caused by file-level backup/restore, then thread_1 will trigger OI
scrub to rebuild OI files.

But we ignored a corner case that the thread_1 recheck the OI files
may just between the thread_2 has dropped the inode_A's referene to
zero and will remove related OI mapping from the OI file. Then the
thread_1 is misguided, and will trigger OI scrbu unexpectedly.

More initial OI scrub for the /ROOT/.lustre directory to make sure
the necessary files/directories for mount are ready before used.

This patch also enhances the ls_locate()/dt_locate_at() interface
to allow the caller to pass some hints to low layer, such as flag
LOC_F_NEW for create, to help the low layer to handle efficiently
and properly.

--- Message (Fan Yong, 2014-01-06 05:43:58.000000000) ---

What the "inode freeing" means? removing the inode from memory or else? Because someone may still in using such inode, even though the inode's nlink becomes zero, the inode maybe still live in memory until the last reference on the inode has been dropped. So inside the osd_object_destroy(), when oi_delete is called, the inode is still in memory (although the nlink is zero).

For the second thread, when it firstly call oi_lookup(), it found the OI mapping, but when it locates the inode with the OI mapping, it found that it is a deleted inode (with zero nlink), then we can make it do another oi_lookup(), there are three cases:

1) the OI mapping has been removed by the first thread, then the second thread knows that the object has been removed by race.

2) the OI mapping is still there, the OI mapping is invalid because of file-level backup/restore.

3) the OI mapping is still there, the first thread just dropped the inode nlink to be zero, but not removed the OI mapping in time yet.

For case 2), we need to trigger OI scrub to checking the OI files; for case 3), we should do as case 1) does.

Our current OI scrub supports to regenerate the whole OI tables from zero, means your can delete the OI files totally under "ldiskfs" mounted. When you mount the device as "lustre", it can auto detect the OI files removing, and trigger OI scrub to regenerate the whole OI tables by scanning the whole device. And we allow (at least partly allow) the client to access the system during the OI scrub, in spite of the normal backup/restore, or the whole OI tables removing/regenerating.

There are three levels "lookup" inside the osd_fid_lookup() in order:
1) Per-thread based OI-cache. When former name based lookup found the name entry in the directory, it would add the FID=>ino# mapping to the thread OI-cache. Then subsequent osd_fid_lookup() (handled by this thread) can use it directly without searching the OI files.

2) If the system in OI scrubbing, and the name based lookup found the right "FID=>ino#" mapping which OI mapping in the OI file has not been rebuilt or regenerated, it would add the "FID=>ino#" mapping to a small global table (or list). Then for the subsequent osd_fid_lookup() (in spite of it is handled by this thread or not), if the handler thread cannot find the needed "FID=>ino#" mapping by checking per-thread based OI-cache, then it will check the small global table next. If found, then no need to search the OI files.

3) If above two steps failed both, then search the (kinds of) OI files.

For normal cases (neither backup/restore nor OI files removing/regenerating), if we cannot find the OI mapping via above three steps, then return -ENOENT. But for the case of OI scrub in rebuilding/regenerating the OI files, it is some different:

(A) For name based request from client, there will be name-based lookup before the osd_fid_lookup(), so if the file is there, the per-thread OI-cache will be filled and without searching the partly rebuilt/regenerated OI files.

(B) For FID based request from client, if some other clients have already sent name based requests for the same object (FID) before, then we can find the right "FID=>ino#" mapping in the pre-thread OI-cache or the in small global table.

(C) For FID based request from client, if it cannot find the needed "FID=>ino#" mapping in the first two steps, it will search the partly rebuilt/regenerated OI files. Three possibilities:

(C.1) lucky enough, the OI mapping for the target FID has been rebuilt/regenerated. Then processed as normal.

(C.2) We did not found the OI mapping in the partly rebuilt/regenerated OI files. If it is for create, the osd_fid_lookup() should return 0 to allow subsequent real create operations.

(C.3) We did not found the OI mapping in the partly rebuilt/regenerated OI files. If it is for non-create, the osd_fid_lookup() does not know whether the target object really non-exist or just because of related OI mapping to be rebuilt/regenerated. Under such case, return "-EREMCHG" to the caller to tell that the OI files are in rebuilding/regenerating and the OSD does not know whether the target object exist or not, please try again after the OI scrub.

Currently, we use "LOC_F_NEW" to help the OSD to distinguish the (C.2) and (C.3). From the OSD view, it does not  (and should not) care about whether the request is from client or from local storage. It just needs the caller to clarify that whether it is for create operation or not.

Be as an online tool, the OI scrub should allow the system to be accessible during the OI scrubbing, which is part of the contract requirement. So "LOC_F_NEW" is necessary for such purpose, in spite of backup/restore or the while OI files removing/regenerating. This patch just follows the existing logic as processing client-side RPC. It does not bring new mechanism and break nothing.
