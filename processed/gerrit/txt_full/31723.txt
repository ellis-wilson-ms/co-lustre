--- Commit Message ---
LU-8324 hsm: prioritize one RESTORE once in a while

Currently, HSM requests are run in the order they are submitted to
the coordinator. This has the undesirable side effect that the more
interactive requests (RESTORE / REMOVE) can take quite a while to be
run when a huge batch of ARCHIVE requests are already queued.

This patch is not a clean fix to LU-8324, it is merely an attempt at
making things bearable while a proper solution is being developped.

Selecting a RESTORE from mdt_coordinator_waiting_cb() does not necessarily mean it will be sent to a copytool (some requests are discarded later in mdt_coordinator()). So the current heuristic is not good enough.

Patch set 2 will fix this by inserting the RESTORE request at the start of the "selection buffer" (`hsd`) instead of its end.

@Sergey
Hi, I am not sure whether adding a test for this patch is a good thing. I only consider this a temporary fix.

But if you need to check the patch actually works, there is a little script attached to the JIRA ticket (analyzer-lu-8324.sh).You may need to modify a few variables at the start of the script to suit your environment; it also expects to be run on a single-node setup with a mounted filesystem ("llmount.sh" with the default config is what I use); and /proc/fs/<fsname>/mdt/*/hsm/max_request should be set to 3 (the default).

The result of running it should be:
 - without the patch: 15 ARCHIVEs are processed before the first RESTORE;
 - with the patch: one of the first 3 processed request is a RESTORE.
