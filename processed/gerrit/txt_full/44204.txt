--- Commit Message ---
LU-14838 osc: Remove lockless truncate

Lockless truncate does not work and cannot be made to work.

Fundamentally, it has no means of ensuring consistency
across clients because it can't force them all to drop
cached data without locking.

It's been off for years - let's just get rid of it.

Patch Set 2:

Maybe I'm missing something here, but the premise of "lockless truncate" was *not* that no locks are taken anywhere, but rather that no locks are fetched by the client, *just* to do the truncate.

There still needs to be a lock on the OST to invalidate the client caches, but this avoids the extra 2x round-trips for the client to lock + truncate + unlock the file under contention.

The workload that triggered the development of lockless truncate was some huge MPI program that was using open(O_TRUNC) on a single file shared across all of the ranks.  Since this was done inside a library, it couldn't be avoided by the application.

Patch Set 2:

Yeah, I understand now that you say that, I did not write this up well.  Thanks for making me think about this more.

Lockless truncate asks the servers to do it, so there's no central coordination - there's nothing that forces all the locks to be held at once.

The issue is two racing truncates to different sizes.  If they do not hold all the locks at some point, they can interleave across stripes, so you can get a truncate to *different sizes on different OSTs*.

This can lead to a size which is not the size requested by either truncate.

Consider a two stripe file with 1 MiB stripes, racing truncates to 3.5 MiB and 7.5 MiB.

The truncates are just sent to the servers, so they can arrive in different orders at different servers.

Viz, a possible ordering:
Truncate to 3.5 MiB, then 7.5 MiB on stripe 1
Truncate to 7.5 MiB, then to 3.5 MiB on stripe 2

And Stripe 1 holds 0-1 MiB, 2-3, 4-5, 6-7, etc…
Stripe 2 holds 1-2 MiB, 3-4, 5-6, 7-8, etc…

So truncate to 7.5 MiB on stripe 1 is the same on that stripe as truncate to 7 MiB (so, after truncate, the size according to stripe 1 is 7 MiB - 1 byte - since it doesn’t hold the 7.5 MiB offset, that’s on stripe 2)

But stripe 2 has been truncated down to 3.5 MiB, that’s the last operation there, so that’s the size there.  It doesn’t really matter exactly what that does on stripe 2 except that it’s lower than the size on stripe 1.

So when Lustre determines the file size, it uses the “furthest”, which is that 7 MiB - 1 byte from stripe 1.

So the size is 7 MiB - 1 byte, because the truncates interleaved.

So we can’t allow that interleaving, and the only way to do it is through serialization across all the stripes.  So, holding all the locks, somewhere, which should probably be the client.
