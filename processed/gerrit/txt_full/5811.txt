unnecessary page eviction and possible bitmap read during
bulk writes.

--- Message (Andreas Dilger, 2013-03-23 17:00:24.000000000) ---
Patch Set 5: I would prefer that you didn't submit this

Some comments:
- instead of adding lru_cache_drain() before every call to ext4_mb_cache_init(), why not put this inside ext4_mb_cache_init()?
- it doesn't seem right that we would need to call lru_cache_drain(). Do we have proper calls to mark_page_accessed() in the mballoc functions?
- this kind of patch should really be pushed upstream to get feedback. If it is a performance problem with ldiskfs it is also true for ext4.
- we are trying to get rid of kernel patches, and this patch will prevent building on an unpatched kernel and on SLES. The use of lru_cache_drain() should have a configure check. 

For a long time I've thought it would be better to avoid putting large files into cache by reducing readcache_max_filesize. This would help avoid cache pressure on the bitmaps, as well as on the inode table and other metadata.

Patch Set 5:

Andreas,

mark_page_accessed() is NOP without a calling a lru_add_drain().
because find_or_create_page put a page to the lru pagevec batch, and wait 
until pagevec full to move to real LRU.
mark_page_accessed() just mark a page referenced (NOT accessed) for these pages (PageLRU() is false).
BUT shrink_inactive_list() have call lru_add_drain() so move pages from lru pagevec to inactive list, when shrink_inactive_list found that page and kill.

We may write a patch without patching a kernel, just use a shake_page(page, 0) as replace to lru_add_drain().
void shake_page(struct page *p, int access)
{
        if (!PageSlab(p)) {
                lru_add_drain_all();
                if (PageLRU(p))
                        return;
                drain_all_pages();
                if (PageLRU(p) || is_free_buddy_page(p))
                        return;
        }

but it's adds some overhead as call drain lru pages at all CPU's.
