LU-19083 ptlrpc: handle duplicate BRW after reconnect

When a client reconnects during a bulk transfer operation (OST_READ
or OST_WRITE), the original BRW request can fail without sending a
reply back to the client. The client then resends the BRW request,
but the server was incorrectly treating this as a duplicate request
and discarding it, leading to hung I/O operations.

This patch modifies ptlrpc_server_request_add() to check the
connection count when handling potentially duplicate BRW requests.
If the client's current connection count is higher than the
connection count in the original request message, this indicates
the client has reconnected and we should process the resent
request rather than treating it as a duplicate.

A new failure injection point CFS_FAIL_PTLRPC_OST_BULK (0xe003)
is introduced to simulate failures during bulk transfers. The
lnet_parse() function is modified to check this failure point
and drop GET/PUT messages matching the OST_BULK_PORTAL index.

Added unlikely() hints to existing failure path checks in
lib-move.c to optimize the common case where no failures are
being simulated.

Test sanity.sh test_500 validates the fix by performing dd
operations with injected failures and OST reconnections to
ensure BRW requests complete successfully without slow reply
warnings.

LNet fail_loc defines moved to a dedicated header so they can be
included in other modules without pulling in all of the other internal
structures and prototypes.

I think you misunderstand the logic in play here.

When we encounter that the existing read/write/... is being executed already, we do not simply drop the reply, we actually update the matchbits of the request that is being processed for the request that is the resend.
The thinking is - by doing such a swap, once we are done with processing ,we don't want the client to then just drop the result of our hard work on the floor because it was impatient and timed out and resent, but instead would get that result straing as if it came from processing the resent request.

The whole plan and pathes you can see in LU-793 and this patch singlehandedly undoes all of that.

I think what you are hitting is the breakage that was introduced by LU-15190? That patch now checks for ALL duplicates without regards if any of the requests being culled out are currently being processed or not breaking the original logic (that had some racing still I guess?), can you please retry your testcase with commit bb83a8af59d30b3f9e6de171eca962316ab7f6f4 reverted please?

There's a bit of a rehash of logic at play between all involved (but with memory long faded) in LU-18072 as well.

Rereading the ticket you filed - I am relatively sure every timeout is a client reconnect so it cannot result in all currently processed requests being dropped as easily, or LU-793 would never work, unless it was a bit of a more modern addition. Looking at the patch that introduced this "Reconnect on bulk" message - in fact it was LU-793, so it should all be accounted for?
