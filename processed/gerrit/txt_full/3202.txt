LU-1575 ptlrpc: fix for the flock policy combinations

The current patch is intended to fix the issue with incorrect flock
policy determining. This issue is observed when 1.8 clients
with updated flock policies try to connect to the 2.x servers.
This patch intended for 2.x servers to let recognition for
updated 1.8 clients.

Patch Set 6: I would prefer that you didn't submit this

Is this patch already in use in production systems using this OBD_CONNECT value?  This conflicts with the value that was recently reserved for nanosecond timestamps. If it IS in production use already, we will need to change the value for the nanosecond timestamp feature, but I would not be terribly happy in this case. It would be a terribly bad idea to use OBD_CONNECT values in production that were not already reserved in master and b1_8 first to avoid conflicts.

As for the patch itself, I need to take a closer look at the interaction between this patch and the others related to flock when I'm back from vacation tomorrow and not looking at it on my phone. I though that there was already a compatibility mechanism in place for this problem with the flock owner, but I don't recall the details without looking at the code and old patches.
