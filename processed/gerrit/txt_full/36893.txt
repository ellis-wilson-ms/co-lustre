--- Commit Message ---
LU-12988 ldiskfs: mballoc to prefetch groups

ahead of scanning. prefething is done in 8 * flex_bg groups, so
it should be 8 read-ahead reads for a single allocating thread.
at the end of allocation the allocating thread waits for read-ahead
completion and initializes buddy information so that read-aheads
are not lost in case of memory pressure.
at cr=0 the number of prefetching IOs is limited per allocation
context to prevent a situation when mballoc loads thousands of
bitmaps looking for a perfect group and ignoring groups with
good chunks.

Patch Set 2:

I think this is looking very promising, but I think it makes sense to merge this with the previous patch, unless we are sure the smaller patch will be useful by itself?

I'm also a bit worried that we are not submitting the IO as efficiently as it could.  For 512k groups with 512k bitmaps (640TiB), we would generate 20000 reads/seeks if reading with flex_bg size of 256 (default for Lustre OSTs).  This is "only" about 200s @ 100 seek/sec to read all of the bitmaps.  If we need to seek for every bitmap this would take 14h, so we need to be sure that the IOs are merged.
