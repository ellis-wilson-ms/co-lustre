LU-16338 readahead: clip readahead with kms

During I/O test, it found that the read-ahead pages reach 255 for
small files with only several KiB. The amount of read data reaches
more than 1MiB.
The reason is that the granted DLM extent lock is [0, EOF], which
is larger than the requested extent. During readahead, the OSC
layer will also return [0, EOF] extent which will clip into stripe
size (1MiB) regardless the actual object size.
In this patch, the readahead range is clipped to the known min
size (kms) on OSC layer during readahead. By this way, the
read-ahead data will not beyong the last page of the file.

Add sanity/101m to verify it.

This patch also fixes multiop to return successfully when reaching
EOF instead of exiting with ENODATA during read.

Hmm.  I see your point about KMS, but I'm not sure it's harmless to read past EOF?

I'm concerned about reading in more pages than there is data in the file.  You say we read 255 pages for files several KiB in size.

When we read pages past EOF, where is that data coming from?  I guess it's likely it's  just zeroes, since ldiskfs knows the OST object file size and we're reading past it.

Next question...

So reading extra pages in to the page cache *sounds* safe, because we don't allow reads beyond EOF anyway, but I'm not sure it is.

What would happen if the client reads these pages holding a write lock, then grows the file size by writing later in the file?  These pages are now inside the real size of the file.

I guess they were zero filled when we read them in, so it's OK.

OK, so: I can't see how reading zeroes in to the page cache past EOF causes any problems, but I am very uncomfortable with reading pages in to the page cache that are beyond EOF, that's why I was concerned about KMS accuracy.  Putting pages in like that is wrong and I'm afraid it will cause problems at some point.

Yingjin, thoughts?  I guess that isn't a regression and doesn't have to be fixed here.  But it makes me nervous :)
