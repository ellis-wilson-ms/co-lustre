--- Commit Message ---
LU-10360 mgs: Dynamic network updates

When a nid or a new Network is added to the servers a
discovery message is sent to all the peers to inform them
of the update. When the MGS receives this discovery update
it can update both it's IR database and llog. This way new
clients can mount the file system using the newly added
interfaces or networks.

Patch Set 5: Code-Review-1

(10 comments)

I think we really need to better understand the pros and cons of updating the persistent config llog when a NID is dynamically added on a server before this patch is landed.

I don't see any problem with updating the IR table, since that is only in memory.  However, rewriting the config llog dynamically could be problematic for several reasons:
- the new NID may be temporary, so we don't want it permanently stored
- updating the config llog repeatedly makes it grow larger over time
- updating the config llog increases the risk that it becomes
  corrupted if the server crashes or there is a bug in the code,
  but there are no tools to "repair" the config llog short of
  rewriting the whole thing

In the meantime, I think it makes sense to separate the MGC IR table update from the llog update in this patch, and push the MGC IR table part of the patch first, so that we can make separate decisions about the two of those changes
