LU-18192 nrs: TBF combination with different granularities

Under current NRS TBF design, a server classifies the incoming I/O
requests into TBF buckets according to the NRS TBF rule with
different rate limits.
The requests in each TBF bucket are scheduled in FIFO order and
the RPC rate is limited per TBF bucket based.

Each TBF bucket is in unit of a TBF type or the combination of
various TBF types. i.e.
TBF type:	TBF bucket
NID     TBF ->	(nid)
JobID   TBF ->	(jobid)
UID     TBF ->	(uid)
GID     TBF ->	(gid)
Opcode  TBF ->	(opcode)
general TBF ->	(nid,jobid,uid,gid,opcode)

The more general the TBF type is, the finer the granularity of the
TBF buckets are.
For example, 100 nodes (NID) means there are 100 TBF buckets for
NID TBF type.
For general TBF type, the number of TBF buckets, in theory, is
the product of the number of various TBF type.
Thus, it will generate much more TBF buckets for the general TBF.

We can define much coarser TBF types according to the requirement
of the users. i.e.
"# lctl set_param ost.OSS.ost_io.nrs_policies="tbf nid+opcode"
"# lctl set_param ost.OSS.ost_io.nrs_policies="tbf nid+uid+gid"

By this was, an administrator can set different combination of
TBF types dynamically, reducing the generated TBF buckets during
scheduling.

Patch Set 10: Code-Review+1

I'm in favor of the added functionality of this patch series, but my main concern is the complexity of the policy specification, and the difficulty any user will have to express a "simple" rule set that actually produces the expected outcome.

IMHO, the "desirable" outcome of NRS for the vast majority of users would be "minimize impact of 'noisy neighbor' on shared storage users".

So while I'm giving +1 on this patch, and want the later patches to continue moving forward (especially the projid and nodemap rules), what I would _really_ like at the end is some ability to have a simple policy that implements "balance IOPS, bandwidth, and latency evenly between users, until those users' IO is completed" and/or "prioritize jobs/projects/nodemaps that have very little IO".

I think one of the major problems with the current TBF implementation is that (AFAIK) it *prioritizes* the RPCs of the noisy neighbor because there are always more RPCs from that client/job/project than can fit into the bucket, so they will always exceed their deadline and be prioritized in the next round instead of being slowed down.
