as SSK or Kerberos can add latency to reconnections.

Imperative Recovery's (IR) core is the ability for the MGS to warn clients about target failovers. This patch is not tampering with this mechanism. But a side effect of IR is to shrink the recovery window. The reasoning is that with clients pro-actively alerted about target failovers, the recovery should be shorter to complete.

Problem is recovery takes more time when a GSS security flavor is enforced. For instance there is a "redo" mechanism directly at lgss_keyring level, in case of connection timeout. Also, due to the fact security contexts for former connections are kept, the reconnection to targets on their primary location fails because of invalid security context.

For these reasons, and to avoid breaking other use cases, the idea of this patch is to keep IR mechanism on - even when a GSS security flavor is enforced, but to not shrink the recovery window, as we know recovery will take "longer".

It is true another approach is to simply tune Lustre timeouts differently, given that a Lustre filesystem has either SSK/Kerberos enforced or not, but rarely switch that back and forth in the filesystem lifetime.

--- Message (Sebastien Buisson, 2023-06-12 14:29:20.000000000) ---
Patch Set 1:

> Patch Set 1:
> 
> > Imperative Recovery's (IR) core is the ability for the MGS to warn clients about target failovers. This patch is not tampering with this mechanism. 
> 
> So why is it not working here? The problem happens because the client tries to connect to each service node and this is taking too long, instead of directly reaching the right server, no?

After losing connection with the targets on the secondary node, Lustre naturally tries the primary node first. And the client already has a GSS context for this connection (kept from the original connection). So the client uses this existing GSS context, but as it does not correspond to any valid (reverse) context on server side (because it was restarted), the connection gets refused.
Note this would not happen without GSS, and this first try would succeed. The reason for keeping GSS contexts for former connections is to support the case where a server has targets reachable via multiple NIDs. If one of the network interfaces fails, the client tries on the secondary NID and immediately reconnects. This is how Lustre internal HA is implemented even without GSS, so not keeping GSS contexts for former connections would break this use case.
Then the Lustre internal HA mechanism goes on a loop through all the failover nodes. All attempts fail because the targets are back to their primary location, and by the time the client tries again the primary node (the GSS-specific "redo" adds more delay), the recovery window is closed and the client gets evicted.

No GSS:
 1. IR tells us where to connect
 2. Connect to that node
 3. if that fails, try the other service node
 4. go back to first one until success

With GSS:
 1. IR tells us where to connect
 2. Connect to that node
 3. Fail due to bad context
 4. .. move to the other node ...

(at some point, discard the context)

 5. Retry on the right node, with a different context, succeed

Q:
 - Could we do instead: "4. Retry with a new context on the same node?"
 - In my test, time between step 2 and 5 sometimes are faster than others, so this is not just a matter of GSS context. (I'm always stopping the same host, and this is always the same 3 targets that fail over).
