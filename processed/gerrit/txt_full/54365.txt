--- Commit Message ---
LU-17504 build: fix array-index-out-of-bounds warning

On Linux kernel 6.5, due to commit 2d47c6956ab3
("ubsan: Tighten UBSAN_BOUNDS on GCC"), flexible
trailing arrays declared like 'lc_array_sum[1];'
will generate warnings when CONFIG_UBSAN & co. is
enabled:

  UBSAN: array-index-out-of-bounds in lprocfs_status.c:1609:17
  index 1 is out of range for type '__s64 [1]'

Since LPROCFS_STATS_FLAG_IRQ_SAFE flag is only used
in one place - obd_memory() counter, we can just
remove it and change obd_memory over to a regular
percpu_counter. This would both simplify the
lprocfs_counter() code, move over to using more
kernel functionality instead of libcfs, as well as
reduce overhead slightly for the memory accounting code.

Patch Set 2: Code-Review-1

While the addition of 8 bytes to the lprocfs_stats counter is not a big deal, I think there is a better fix for this.

It looks like LPROCFS_STATS_FLAG_IRQ_SAFE is only used in one place - obd_memory() counter, and it doesn't need most of the overhead of lprocfs_stats (min/max/average/etc).

It would be better to just remove LPROCFS_STATS_FLAG_IRQ_SAFE and change obd_memory over to a regular percpu_counter.  This would both simplify the lprocfs_counter() code, move over to using more kernel functionality instead of libcfs, as well as reduce overhead slightly for the memory accounting code.

It looks like that would be relatively little effort, just change the accounting macros:

 /* OBD_MEMORY_BATCH is the maximum error allowed per CPU core.  Since
  * obd_memory_sum() is calling percpu_counter_sum_positive(), it adds
  * up the per-core local delta anyway, so the per-core batch size is
  * can be large.  This could be percpu_counter_add_local(), but that
  * only exists in kernel 6.0 and later, and just uses a larger batch.
  */
 #define OBD_MEMORY_BATCH (16 * 1024 * 1024)
 #define obd_memory_add(size) percpu_counter_add_batch(obd_memory, (size), \
                                                       OBD_MEMORY_BATCH)
 #define obd_memory_sub(size) percpu_counter_sub_batch(obd_memory, (size), \
                                                       OBD_MEMORY_BATCH)
 #define obd_memory_sum()     percpu_counter_sum_positive(obd_memory)

I only see obd_update_maxusage() called in the pinger thread on the clients, but I don't see obd_update_maxusage() on the servers.  It might make sense to change this to call obd_update_maxusage() once every 5 seconds as a work item instead of the pinger, so that it always runs on all nodes?  However, that could be a separate patch in case it has some issues and shouldn't delay this cleanup.
 

The "#else /* CONFIG_PROC_FS */" block could also be removed since this no longer depends on lprocfs_counter.
