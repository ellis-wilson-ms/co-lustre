LU-8062 tests: fix recovery-small test_115b fail_loc

The recovery-small.sh test_115b added for LU-7434 in patch
http://review.whamcloud.com/19953 fails intermittently with
"dd success".  A bulk write RPC is infected with a fail_loc value
so that it will block for completion (either req/reply lnet buffers
unlink or bulk unlink) for a few seconds.  Also a delay is injected
before the RPC is sent.  The delay gives some time to inject a
second fail_loc on the server so the server will return an error
for the bulk write RPC.

While the first fail_loc is hit correctly, there are no bulk write
RPCs immediately due to the inserted delay.  The problem with the
test is that the second fail_loc is too generic (OBD_FAIL_OST_STATFS)
and is set to fail only once, so it can return an error on OST_STATFS
RPCs from the MDS occasionally and the bulk write succeeds.  In
other cases it appears the delay for the second fail_loc may be too
short and times out before fail_loc is set properly.

Instead, use the existing OBD_FAIL_PTLRPC_BULK_PUT_NET fail_loc for
writes as well so that this only catches the bulk write RPC.  The
use of OBD_FAIL_PTLRPC_DROP_BULK in the write path would create
testing interop issues, so it is added but currently unused.

Convert cfs_fail_check_set() to a macro, so that the caller's path,
function name, and line number are printed in the logs, since this
is ambiguous if there are multiple locations using the same fail_loc.
