SKIP: ost-pools test_23b Filesystem space 19305612 is larger than 14000000 limit

Per ost-pools.sh:
 MAXFREE=${MAXFREE:-$((2000000 * OSTCOUNT))}

And per autotest:
 OSTSIZE=7235174

It's likely ost-pools test 23b will always be skipped by autotest if the OSTSIZE value set by autotest is large. It seems there is no proper way for the test script to specify the MAXFREE value. Maybe we can figure out a real value which will make the out of space test run within an acceptable time, and set that value as the default one for MAXFREE to limit the OOS test time.

E.g., is 31457280 (30G) reasonable? or maybe other value?

To verify the current patch for the time being, we can add test parameter "ostsizegb=2" into the commit message.

--- Message (Andreas Dilger, 2012-11-16 02:05:13.000000000) ---
Patch Set 4:

Rather than picking a specific (large) size for the limit, which may take a very long time, it would be better to limit the time that the test is running. It wouldn't be a bad idea if the OST size is not too much over the limit (e.g. within 10x the limit) to write some small amount of data (say 100MB) and then calculate how long the test would run. If it is estimated to take less than 10(?) minutes to fill the space then it should be OK to run. 

That will allow it to run if working with a small test system or on real hardware, but will not take hours if running in a small VM.

Yes, rather than spending a long time to run and then exiting the test, it would be better to compute estimated completion time on each loop based on the average rate for all the data written so far. That will avoid overly optimistic estimates for writes to cache at the start if the test.
