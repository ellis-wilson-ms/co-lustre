--- Commit Message ---
LU-3319 procfs: provide framework for seq_file handling

In linux 3.10 kernels and above the proc file system interface
changed. To handle this change we need to move from the current
method of using proc_read and proc_write function pointers to
using the seq_file. This patch provides the base to enable moving
all the lustre proc handling gradually over to this new method
of proc data handling.

Patch Set 4: (2 inline comments)

IMHO, the HAVE_ONLY_PROCFS_SEQ thing really complicates the code, especially every entry of each lprocfs_vars structure has to be surrounded with #ifndef HAVE_ONLY_PROCFS_SEQ.

Can we get rid of HAVE_ONLY_PROCFS_SEQ? For example, in the infrastructure patch, keep all existing APIs, and add struct lprocfs_vars2 that replaces read/write_proc_t with fops, and also lprocfs_register2/lprocfs_add_vars2(maybe others as well) that process lprocfs_vars2 list. Then the infrastructure can support both APIs and callers can be converted from old APIs to new APIs incrementally. In the end when all callers are converted, create a patch to remove the old APIs and rename the new ones to the old name. When that is done, 3.10 kernel procfs change is also supported.

Does this sound possible?
