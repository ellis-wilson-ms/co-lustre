LU-18778 llite: try more supp. gids during atomic open

During a file open, Lustre will typically check the group ID
on the dentry of the file to see if the user has permissions
to open that file. If there is no dentry (or a negative dentry),
Lustre will instead attempt an atomic open. In this case, Lustre
may not know which supplementary group to pass along with the
request.

If the client sees a permission error, the MDS may direct the
client to retry with a server-provided group ID. The client
will perform a local validation with that GID, and attempt
the retry if the user is a member of that group.

In the DNE case, the MDS might not ask the client to retry. In
this case, we get an incorrect permissions error - and we are
not given any other GIDs (from the MDS) to try. To mititgate
this, retry will all of the available supplementary groups.

In the normal (happy) path, we will not experience any additional
requests. But in the unhappy path, the client will attempt more
retries on atomic open permission errors.

Patch Set 3:

There's been a few patches in this area in the past (i.e. supplementary group issues). This is a new edge case with DNE. I think Lustre ought to pass more than 2 GIDs in this atomic open (no dentry) case. If we passed 16 GIDs like NFS, we could avoid a lot of this complexity and retry logic. Most RPCs could stick with 2 GIDs, since they have a cached GID. But these uncached cases could be made faster (i.e. less retries) and more robust if we just passed more GIDs in the first place.
