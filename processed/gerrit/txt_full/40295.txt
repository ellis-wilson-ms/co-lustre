--- Commit Message ---
LU-14045 sec: fix O_DIRECT and encrypted files

Sometimes, we can end up in a situation where
osc_release_bounce_pages() mistakenly consider pages as fscrypt
bounce pages, and tries to free them.
Fix the way we consider bounce pages by always setting the PageChecked
flag on them.
Also remove sanity test_426 from the ALWAYS_EXCEPT list.

Patch Set 2: Code-Review-1

I don't like this at all.  It adds complexity that is hard to follow and make the code more fragile.

I think a better fix would be to make sure O_DIRECT pages have ->mapping set.  Then we could unconditionally call llcrypt_finalize_bounce_page() on all pages.

I see that the mapping *does* get set on these pages, but is then cleared again in cl_aio_end().  As these are private pages which will shortly be freed, why do we need to clear ->mapping ??

You are talking about the case of encrypted files. However, the issue hit by James occurs on normal files. For a reason I could not reproduce, under some circumstances (presumably when doing O_DIRECT IOs), pages of a non-encrypted file end up with no mapping set. So they are mistakenly considered as bounce pages used for encryption.

This behavior seems to be due to newer kernels, as it was not hit with Ubuntu 18 or RHEL8. Not sure how it is possible that Lustre gets pages with no mapping set.

--- Message (Patrick Farrell (personal), 2020-10-20 18:12:28.000000000) ---
Patch Set 2:

Bounce pages are temporary pages allocated so that they can hold encrypted content. They are necessary because pages in the page cache must always contain clear text data. Bounce pages live for the duration of the request, and need to be freed once done. They can be distinguished from 'normal' page cache pages because they do not have any mapping set, this is how fscrypt works internally.

In case of buffered IO, either it is on an encrypted file, and in this case bounce pages are allocated, or it is on a regular file, and in this case there are no bounce pages. So for buffered IO, checking ->mapping on the page is enough to find out if we are dealing with a bounce page or not.

In case of O_DIRECT IO, unfortunately the pages used to carry out the IO are not page cache pages, so we cannot assume they have a mapping set. Which means those pages could incorrectly be considered as bounce pages by fscrypt.

In patch #38967, we brought support for O_DIRECT on encrypted files. Basically, it consists in making use of pages being used for sending RPCs, by setting a proper mapping and index on them, suitable for encryption/decryption. This is done in ll_direct_rw_pages(), and at the same time we set the PageChecked flag on the pages. Reference to the mapping and PageChecked flag are removed in cl_aio_end(), once the IO is done.

So, the idea of this patch is to improve the way we determine if we are dealing with a bounce page or not. This is why in case of O_DIRECT (OBD_BRW_SYNC flag on brw_page), we retrieve the "page cache" associated with the page by calling llcrypt_pagecache_page(). This function simply calls page_private(). Then we check for the PageChecked flag: if it is set, it means we are actually dealing with a bounce page.

To make it more straightforward, we could systematically set the PageChecked flag on bounce pages when they are allocated, that is to say on the return value of llcrypt_encrypt_pagecache_blocks() called from osc_brw_prep_request().
Then, in osc_release_bounce_pages(), we would just have to check the PageChecked flag in all situations (buffered or DIO), instead of relying on whether a mapping is set on the page or not.

How sensible is this approach?
