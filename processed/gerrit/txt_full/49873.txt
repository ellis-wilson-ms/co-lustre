LU-16524 nodemap: add rbac property to nodemap

Add new rbac property to nodemap. Internally this is a mask of allowed
roles. Externally it defaults to all, which means all roles are
allowed, and it can take the following values (multiple can be
specified, comma separated), with the semantic:
- byfid_ops, to allow operations by FID (e.g. 'lfs rmfid').
- chlg_ops, to allow access to Lustre Changelogs.
- dne_ops, to allow operations related to DNE (e.g. 'lfs mkdir').
- file_perms, to allow modifications of file permissions and owners.
- quota_ops, to allow quota modifications.
Apart from all, any role not explicitly specified is forbidden. And to
forbid all roles, use 'none' value.

Update lctl-nodemap-modify man page to mention this new property.

Patch Set 9:

I think I found the reason for the problem hit when running with a ZFS backend. ZFS index files return keys in hash order instead of numerical order. Consequently, records about nodemap properties can be returned *before* the records that actually define the nodemaps. And this breaks nodemap config load - a property cannot be added to an unknown nodemap.

A change was made by patch https://review.whamcloud.com/23849 so that the nodemap config is loaded in 2 passes: first the nodemap definitions, then the nodemap properties. This mechanism needs to be adapted now that we have multiple records with the NODEMAP_CLUSTER_IDX type: NODEMAP_CLUSTER_REC for the nodemap definition, and NODEMAP_CLUSTER_ROLES for the rbac property. This is what this new patchset does in nodemap_load_entries() and nodemap_page_build().
