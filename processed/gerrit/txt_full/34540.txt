LU-10602 llite: Add support for file heat TopN listing

This patch adds file heat TopN listing functionality for each
Lustre filesystem on the client side.
It implements "lfs heat_top" command to show the file TopN listing
with hotest file heat periodically.
And also it implement a llapi interface "llapi_heat_top_get" for
a user to get the file heat TopN listing information via IOCTL.

Patch Set 2: Code-Review-1

> This is not a practical solution to walk all of the inodes in the filesystem to find the top heat on each call.  There might be millions of cached inodes on a large client.  Walking the whole list for each call from userspace may result in stack traces being dumped if it takes too long, and will block filesystem updates because s_inode_list_lock will be very hot.
> Probably what would be better is to maintain a heap of approximately the top 1000 (or some other tunable number) of file heat values for the filesystem, updated when the file heat is updated on a file.  If the heat is larger than the minimum heat value, then it is added to the heap and the minimum value is removed (if there are too many items in the heap), and if the heat is smaller than the minimum heat then it is not added to the heap, or removed from the heap.
> It looks like there should be a new cfs_binheap_insert_min() function similar to cfs_binheap_insert() that does not call cfs_binheap_grow() when the maximum number of entries is reached.
> I think it may be enough to have a single heap based on total operations, as long as it is large enough to hold more entries than the number typically requested from userspace, and the top operations can be retrieved based on the sort key.  Even if the sort key is bytes read/written, there will be a number of operations to read/write the data of the file.

Agreed that walk all of the inodes in the filesystem to find the top heat on each call might cause problem. Agreed that we need to improve this before landing the patch. Because, the worse case could be hundreds of userspace threads reading the topN list causing super hot s_inode_list_lock.

The most critical problem here, is how to maintain the heap of topN list without causing any I/O lantency. This is hard, because everytime that a file is being accessed or updated, the file heat will be updated. That means, if we maintain an heap that needs a file to be inserted into this heap whenever the files heat is being changed, that inserting will needs to happen everytime when any file is being accessed or updated. Even with super quick heap, this operation will introduce significant overhead.

The original design is, instead of triggering walking throught the whole inode list whenever a user-space tries the read the topN lsit, an internal kernel-space thread walks through the inode list from time to time (maybe according to the decay period). Let's assume the time interval of each scanning is M seconds. Then even during the walking process of the inode list, the kernel-space thread will slow down the scanning by sleeping after each inode so as to make sure the whole scanning circle is approximately M seconds.

And when the userspaces tries to get the TopN list, it will get the the list immediately without waiting for any scanning process. This can be easily implemented by copying/dumping the heap maintained by the kernel-space thread.

By using this design, we have a lot of benifit:

1) s_inode_list_lock won't be very hot, because the scanning thread will sleep for a while between scanning of two inodes.

2) No performance impact to I/O procedure at all.

3) User-space tools can get prompt result immediately from the kernel-space without any delay or waiting.

4) Multiple user-space tools can access the TopN list at any interval and won't cause any performance impact.

The only compromise here is the TopN list might has some delay (maximum M seconds). But that is not a critical problem if M is not super large. And any other implementation would have similar delay more or less.

Another option would be to batch the inode heap updates to at most once or twice per update interval, and/or only if the update would move the inode on/off the list. I was just thinking of the "lazy inode update" approach together with a pending list of inodes changed since the last heat update is enough.  

Most updates in ll_heat_add() will only be to the local inode, and they are only added to the pending list once per interval.  It is trivial to check if the inode is on the pending list without any lock, and it only is added to the list isn't already on it.  The code can use an SMP aligned lock and list_head to avoid contention, like blockgroup_lock.

The "lfs heat_top" userspace thread does the pending list processing at most once or twice per heat interval to insert modified inodes into the heap and remove old ones.  That avoids the need for continuous processing in the kernel if/when "lfs heat_top" is not running, and in the worst case there is a single list update the first time an inode is used, and once per heat interval (if "heat_top" is running) or when the inode is freed.

Another option would be to batch the inode heap updates to at most once or twice per update interval, and/or only if the update would move the inode on/off the list. That would allow the heat accounting to be about the same cost as now (increment a per-inode counter or two) without needing to scan all the inodes in the list. Comparing to the scanning speed, I think the users will be more concerned about the the stability of the TopN output. That means, if a file remains as one of the hottest files in the system, it should be remained in the topN list constantly. A user who is checking the topN list would be very surprised if a file shows up on top of the list at one time, and then disappear suddenly in the next period. This kind of vibration in the file list would be really confusing, and would mislead to the wrong conclusion that the file's heat has disappeared during that period. And comparing to the top command based on CPU usage, users will be more sensitive to the vibration, because unlike CPU usage, file heat won't burst and disappear suddenly, it supposes to grow steadily when there are accesses and decay slowly when no access.

However, unfortunately, the heat growing and decaying of each file can't be simply predicted. A file's heat can easily grows to top 1 and remains as top 1 without not being accessed for a very long time, especially when the decay rate is small and period is large. That said, it is not safe to assume that a "hot" file will be kept its presence in any kind of LRU list. Even in a LRU of length 10000 for top100, the top 1 hottest file can still be missing. And this scenario can be easily reproduced: File A is access for millions of times quickly thus has huge value of file heat, but then millions of files are created without touching file A and file A was then expelled from the LRU.

And even more unfortunately, the file heat will always update when there is any I/O or access. That means, the file need to be put into a list (or check whether it is already in a list) every time whenever it is being accessed. This checking and updating is so frequent that it is very unlikely we can avoid introducing significant I/O latency.

It seems to me that the easier way would be avoiding to maintain the TopN list during the I/O procedure. Generating and maintaining the  topN list offline and asynchronously would simplify a lot of things. The only problem left is the scanning speed of the whole inode list. I am not too concerned about this, since as you mentioned drop_pagecache_sb(), evict_inodes(), invalidate_inodes() need to scan the whole list too. If the scanning speed is too slow (e.g. more than hundreds of seconds), umounting would be super slow. And we will need to optimize it any way some how.

And  I guess most likely, the scanning speed of the list without any operation (lock, iget(), unlock and then next loop) would be much quicker than 10 million inodes/s, maybe ~100 million inode/s. And I think the updating interval of 10 seconds would be acceptable. That said, a file system with 10 million active inodes could be updated each 10 seconds with a scanning interval of 1 us. And if each inode costs more than 32KB (I don't know, just guessing) memory in average both for metadata and data page cache, 10 million inodes would cost 320GB memory. And I think in that situation, most likely there will be some memory pressure to cause inode list shrinking (by prune_icache_sb())?

I just find a LRU list of s_inode_lru in "struct super_block", @Andreas and @Yingjin, maybe as another compromise, we can scan the head of this LRU with a limitation of scanning time? That seems better than scanning "s_inodes" in "struct super_block".

I think my modified proposal is solid.

The pending list of inodes would not be an LRU in my proposal, but rather a list of inodes modified within:

     list_interval = max(heat decay period, heat_top interval)

The recently-modified inodes only need to be inserted/removed from the pending list at most once per list_interval, and heat updates will be local to the inode with a lockless check that they are already on the pending list.

The top heat inodes would be kept in a (roughly) fixed-size heap regardless of whether they were modified or not.  Unmodified inodes in the heap would be reprocessed at most once per list_interval.  All inodes would be removed from the pending list when heat_top is run, and maybe added into the heap if hot enough.  This may cause "cold" inodes to drop from the heap.

I think this provides a fairly optimal solution:
- only recently-modified and "top heat" inodes are ever processed
- hot-but-unused inodes are kept in the "top heat" output until cold
- cold/unused inodes do not need heat to be processed at all
- rapid heat updates are totally local to the inode
- minimal global locking (once per active inode per heat decay period)
- can use per-CPU inode list to avoid SMP contention
- no background scanning overhead, only once per heat interval
