--- Commit Message ---
LU-14361 statahead: add statahead advise IOCTL

This patch reuse ioctl(LL_IOC_LADVISE2) for statahead advise.
This allows userspace programs to advise the kernel statahead
of the order that they will be traversing a directory, so that
the client can prefetch inode attributes from the MDT, similar
to what posix_fadvise(POSIX_FADV_SEQUENTIAL) does for file data.

After patched mdtest via adding this statahead IOCTL hint, it
can support mdtest benchmark with regularized file naming format:
	mdtest.$rank.$i
The usage of this statahead advise IOCTL could be as follows:
	open(dir);
	ioctl(dir_fd, IOC_LADVISE2, ...);
	stat mdtest.0.0;
	stat mdtest.0.1;
	stat mdtest.0.2;
	stat mdtest.0.3;
	...
	clsoedir(dir);

--- Message (Andreas Dilger, 2022-09-23 02:53:38.000000000) ---
Patch Set 1:

I don't understand this patch.  If it is improving performance only for mdtest then it doesn't really help users.  Rather than have mdtest use an ioctl to explicitly tune this behavior, it would be better for the client to automatically detect the access pattern.

For example, calculate the differences in sequential filenames from a single thread to calculate skip interval, like strided readahead? 

That would improve mdtest performance, but also improve real application performance, which is far more useful/important.

--- Message (Qian Yingjin, 2022-09-23 08:47:55.000000000) ---
Patch Set 1:

Lustre has readahead, but it also has implemented fadvise (DONTNEED, WILLREAD) for more precise read-ahead control from the user hint.
This patch can be also seemed as a precise stat-ahead from user hint via IOCTL.

Detecting the access pattern for stat() calls is desirable, but it is much more complex to implement, especially for access mode like mutest shared dir stat() workload (stat mdtest.$rank.$I, each rank is a separate process to do stat() call series with regularized naming format.).
Moreover, it is hard to predict how many entries should be stat-aheaded, and when to stop the kernel stat ahead thread...

--- Message (Shuichi Ihara, 2022-09-23 11:45:39.000000000) ---
Patch Set 1:

I guess many applications call readdir(), then stat() to files in general? if readdir() called, Lustre statahead can be triggered. So, it's OK in this case.
But mdtest doesn't call readdir() since they know where file exists and exact all filename. In the end, they call stat() to files without readdir(). 
It would be great if statahead works without any application changes in such cases. This is not only mdtest, but we also need to consider various filename patterns to trigger statahead?

--- Message (Andreas Dilger, 2022-09-24 06:13:54.000000000) ---
Patch Set 1:

Similar to readahead, I suspect that a workload like mdtest could detect a regular file access pattern even without readdir(), by keeping a per-thread cache of the last filename accessed and doing a difference between successive filenames (e.g. compare characters, and find differences if digits, hex digits, ASCII alpha)?

I _think_ we already have some thread-local information in llite (maybe in lu_env that is looked up by PID?) so it would be possible to keep a copy of the filename there (max 256 bytes).  For readahead we use the file descriptor, and for readdir+stat statahead it is also the readdir file descriptor, but I don't think mdtest has an open fd for accessing the directory (unless it is using openat(fd, ...), but I don't think so).
