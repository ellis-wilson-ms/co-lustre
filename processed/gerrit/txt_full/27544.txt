LU-9618 clio: Use readahead for partial page write

When writing to a region of a file less than file size
(either an existing file or a shared file with multiple
writers), writes of less than one page in size must first
read in that page.

This results in extremely poor performance. For random I/O,
there's no easy improvements available, but the sequential
case can benefit enormously by using readahead to bring in
those pages.

This patch connects ll_prepare_partial_page to the readahead
infrastructure.

This does not affect random I/O or large unaligned writes,
where readahead does not detect I/O.

Benchmarks are from a small VM system, files are NOT in
cache when rewriting.

Write numbers are in MB/s.

File per process:
    access             = file-per-process
    ordering in a file = sequential offsets
    ordering inter file= no tasks offsets
    clients            = 1 (1 per node)
    repetitions        = 1
    blocksize          = 1000 MiB
    aggregate filesize = 1000 MiB

New file (best case):
xfsize  ppr	write
1KiB	n/a	59.44
5KiB	n/a	164.5

Rewrite of existing file:
xfsize  ppr	re-write
1KiB	off	4.65
1KiB	on	48.40
5KiB	off	12.95
5KiB	on	143.3

Shared file writing:
	access             = single-shared-file
	ordering in a file = sequential offsets
	ordering inter file= no tasks offsets
	clients            = 4 (4 per node)
	repetitions        = 1
	blocksize          = 1000 MiB
        aggregate filesize = 4000 MiB

xfsize  ppr     write
1KiB	off	11.26
1KiB	on	58.72
5KiB	off	18.7
5KiB	on	127.3

Patch Set 3:

My, readahead is complicated...

After some testing, I can say this:
No, it doesn't.  This appears to be because reads must be greater than 1 page in size to trigger strided readahead...  Possibly something to fix there.

A bit more on strided readahead:
The reads-required-to-activate/recognize aspect seems to make strided readahead tough for small i/o.  4k with jumps doesn't activate strided readahead at all.

8k with jumps (jumps must be large-ish or we stay in window for normal readahead, which is different...) activates strided readahead, but it misses a pretty high % of pages.  The same number of 1 MiB reads misses the same # of pages, so the issue seems to be detecting the pattern.

Anyway, single page reads don't trigger strided readahead.  I also suspect it only queues one stride out, so it's only going to readahead 1 page...  So...  Not much point, then.  If it queued multiple windows out, that would likely be interesting.
