--- Commit Message ---
LU-334 llite: Add LPROC_LL_OSC_{READ,WRITE}.

This patch adds quick and easy access to aggregate OSC BRW (bytes on
the wire) statistics through llite.  To accomplish this, we pass the
number of bytes transferred by a successful request to
cl_req_completion(), and on to ll_stats_ops_tally() by way of the
ccc_req_completion().

--- Message (John Hammond, 2011-09-13 21:59:59.000000000) ---
Patch Set 1:

> Actually I don't understand the purpose of this patch. It seems
> LPROC_LL_BRW_{READ,WRITE} should do stats for how many bytes/pages
> from applications.

I guess I'm not clear on what LPROC_LL_BRW_{READ,WRITE} were
originally supposed to represent.  Outside of lloop, in 1.6 and 1.8
LPROC_LL_BRW_WRITE is not used, and LPROC_LL_BRW_READ was only used to
read a partial page in ll_prepare_write().

> If you'd really like to know the exact bytes transfered over
> network, can you please take a look at ptlrpc_lprocfs_brw() which is
> called in osc_send_oap_rpc() to collect how many bytes prepare for
> transfer.

As a consumer of these stats, I have two issues here.  First, the OSC
BRW stats represent the number of bytes requested.  It's much better
to tally by the number of bytes actually transferred.  Since this way
we don't have to wonder if the user is triggering an error (like over
quota of something) or is really doing a large amount of IO.

Second, I really want four numbers here: cumulative byte and request
counts for reads and writes.  To use the OSC BRW stats for this
purpose on TACC's largest lustre filesystem, I need to read 300
(#OSTs) files.  If I use stdio that means 1800 system calls.  Reading
one stats file takes 0.000080 seconds, reading 300 takes 0.037.

> I noticed that LL_BRW_{READ,WRITE} is actually not used by llite and
> liblustre, I'll appreciate if you can bring it back.

OK, but I'm not sure exactly what to tally.

Would you ACK this if I replace LPROC_LL_BRW_{READ,WRITE} with two new
counters?
