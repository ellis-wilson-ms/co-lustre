LU-19579 hsm: fix HSM recovery for started requests

Started ARCHIVE/REMOVE actions hang indifinity after a failover
(until active_request_timeout).
Copytool llapi_hsm_action_(begin|progress|end) fails with -ENOENT
because the actions is not re-registred in active_requests hastable
at coordinator startup.
RESTORE are processed differently: the current request failed on the
copytool but the action is restarted/resend after the coordinator
startup.

This patch fix that issue by re-register all the started requests in
active_requests hastable. This allows the hsm_progress of older
requests to be processed by the coordinator. A RESTORE action is not
restarted at startup, the coordinator waits the completion of the
current request.

Note:
old requests uuids are lost after the failover, so the requests are
re-registred with an empty uuid. Cancel requests can not be sent
for those cases: the records will be directly set to CANCELED.

Add regression test sanity-hsm 108.
