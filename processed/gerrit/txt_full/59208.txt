LU-18756 sec: add resource id check to oss and mds

This patch includes the resource id check into the relevant code paths
on the oss and mds side. It is therefore included for the following
operations.

On the MDT-side:
- open
- create (file and directory)
- unlink (file and directory)
- setattr
- setxattr
- getxattr
- rename
- link

On the OST-side and on the MDT-side for Data on MDT (DoM) files:
- write
- read
- truncate
- fallocate

Some caveats:
The resource id check is not included for MDS_GETATTR RPCs due to
functional and usability concerns. Specifically for the latter, the
"struct stat" would no longer be filled resulting in "?" when running
"ls -l", which can be misunderstood.

Also, if the check is only enabled on the OST-side, writes are only
denied for "sync"/"fsync"-type operations on a file as the check is at
the server-side. If the check is enabled on the MDT-side, write-access
is denied before the OST_WRITE RPC is sent, i.e., immediately
returning the access denied error code. If a file is still in the page
cache before the check is enabled, a client can still read the local
copy of the file, which is expected.

Sanity-sec test 75a was added to exercise the ID check for the above
cases in several disciplines further testing that access to
neighboring nodemap offset ranges work as expected.

Thanks Sébastien! Hmm, I thought the latest patch covers exactly these cases. I seem to be misunderstanding something, or I've been unclear, so please let me explain my view in more detail.

I don't think we need multiple clients, reconfigurations, or even several nodemaps to test these cases. I actually started with having multiple nodemaps and clients, but then realized it could be done much simpler.

Let's say tenant A is accessing files from tenant B; then this operation does not require or involve the nodemap of tenant B at all. This is because we can only test whether the nodemap rules of the accessing client (here, A) permit us to access an object with some FS_ID for UID/GID. In fact, we do not have the information to know to which nodemap a file "belongs". Whether a trusted client is creating that file and assigning the FS_ID to fit tenant B's offset, or whether the unprivileged tenant B has created that file (that resulted in the same FS_ID for UID/GID) is not relevant for tenant A accessing it.

The test relies on this. As a concrete example, if we have neighboring tenants (offset-wise) such that the range intervals are defined as:
- tenant X offset range: [100,000, 200,000) (tl in the test, i.e., left)
- tenant Y offset range: [200,000, 300,000) (accessing tenant in the test)
- tenant Z offset range: [300,000, 400,000) (tr in the test, i.e., right)

Therefore, (from your example) tenant A accessing files from tenant B is the same for "nodemap_map_id()" as tenant Y accessing Z. Likewise, B accessing A is the same as Y accessing X. And so, we only need one unprivileged tenant (here Y) accessing two neighboring ranges (lower and greater 'alien' offsets) to cover the two cases you mention.

This is why each operation in the test accesses a file from "tl" (< tenant_offset) and from "tr" (>= tenant_offset).

---

PS: This property makes it challenging to have a per-nodemap setting for the ID check, as Andreas suggested, because we don't know which file belongs to which nodemap, and we have no mechanism for disabling access to a specific ID range. We _can_ implement a reverse-mapping for "fs_id -> nodemap foo" for offsets and explicit mappings, but it is not available today.
