LU-14887 llite: Add DIO splitting tunables

The new faster DIO path is great, but benefit is only seen
if I/O is split to multiple RPCs.  Currently, this only
happens if I/O size is > RPC size or if I/O crosses a
stripe boundary.

This requirement for splitting means that there is an
inherent conflict between the desire to do large RPCs and
doing single stream I/O at high speed.

This patch adds a pair of tunables, turning on some I/O
splitting by default while allowing users to control the
degree.

DIO parallelism, at the llite layer, specifying how many
chunks we should try to split a DIO In to
Minimum preferred I/O size, at the OSC layer, specifying
the minimum size to which we should split I/O.

Parallelism is a global control of how much splitting is
desired for best performance, the overall preference
between maximum RPC size and maximum single stream
performance.

The OSC level control is because some OSTs have
dramatically different performance with synchronous I/O,
so for a spinning OST, it may be desirable to enforce a
higher minimum I/O size (and so less parallelism), and the
reverse for a flash OST.

Patch Set 5:

"Patrick, the issue I see here is that this makes clients/threads 'greedy' in terms of sending more RPCs to improve their own bandwidth, without considering whether that hurts aggregate client and/or OST performance due to having to process more/smaller RPCs."

Yes, this is *absolutely* the issue here.

"Ideally, we wouldn't need a tunable here at all, by having some kind of feedback whether increasing the number of RPCs actually improves performance or not?"
I'm going to think about this (now-ish); I have the germ of an idea or two but I'm not sure.  Performance measurement on the fly is challenging.

"The other option (which may be significantly more complex, so it is not a recommendation, just a point of discussion) is whether it is possible to have more parallelism at the syscall level without sending smaller RPCs over the network?"
So, not really - no particular point.  Parallel DIO improves performance a lot vs single large RPCs (even at 64 MiB writes or reads) and that's all about getting those multiple RPCs out.

By creating multiple RPCs at the client, it generates parallelism on the network interface(s) with multiple streams, parallelism on the server CPUs with multiple messages to process, and parallelism on the server storage with multiple writes.

Another huge (and intertwined) benefit is "overlapping" the syncs to disk with sending of data.  Normally we would marshall up the pages for (worst case, I guess) a 64 MiB RPC, then send them out on the network, then process them on the server, then start sync()'ing them.  So we have to wait for that 64 MiB to sync to disk - that is probably our longest wait.

If we split that up in to smaller sizes (say 16 MiB), then obviously we get parallelism, but also, the last thing we do is wait for a 16 MiB sync.  Assuming the disk has enough bandwidth, this should take something like 1/4 as long as waiting for the 64 MiB sync.

We could consider creating multiple streams on the network or to disk for very large RPCs (this could be a very interesting idea), but that's eventually a lot like splitting up to multiple RPCs.  And it would be an entirely different set of work at different layers.
