--- Commit Message ---
LU-4444 tests: Skip conf-sanity/69 on zfs

Because file creates happen slowly on ZFS and the number of files
required to run the test is 100K, this test cannot run in a
reasonable amount of time.

Also bail out of test if createmany fails (possible if MDS or OST is
too small), this prevents the test from just timing out instead.

Test-Parameters: envdefinitions=SLOW=yes testlist=conf-sanity
Test-Parameters: envdefinitions=SLOW=yes testlist=conf-sanity mdsfilesystemtype=zfs mdtfilesystemtype=zfs ostfilesystemtype=zfs

--- Message (Andreas Dilger, 2014-01-14 19:10:00.000000000) ---
Patch Set 1: Code-Review+2

Is this test only so slow in our test environment, or on real disks also?

I don't mind this patch landing, since we are testing this functionality on ldiskfs still (in theory it should be the same). 

A better long-term approach (to be done in a separate patch) would be to run some smoke test (or just leverage some other test creating a bunch of files) at the start on sanity to get an estimate of the create rate in the current system and save it into some global file, like /var/tmp/lustre-create-rate, and use the estimate to determine how long this test will run. If it is too long (say 5 min for SLOW=no, maybe 20 minutes for SLOW!=no) then the test is skipped.

That would allow the test to run in the future if ZFS speeds up, or if it runs on faster hardware.  I think there are a few other tests skipped because ZFS-on-LVM-on-VM-on-single-disk is slow, but that won't always be the case (e.g. developer testing, or if we start to run some tests on real nodes with real disks).
