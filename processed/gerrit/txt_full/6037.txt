LU-2888 lov: using oi_id/oi_seq in lmm

Since pre-2.4 use {oi_id,oi_seq} for FIDs on MDT, which make
it difficult to tell whether it is ostid or FID by only checking
oi_seq, so this patch will separate lmm_oi/FID conversion from
ostid_to_fid.

Check return value of ostid_to_fid, so to avoid compiling
warnning.

Signed-off-by: wang di <di.wang@intel.com>

--- Message (Andreas Dilger, 2013-04-12 08:47:17.000000000) ---
Patch Set 1: Looks good to me, approved

(1 inline comment)

Since we are planning on making new layouts soon anyway, we could just define those with lu_fid instead of ost_id and avoid this complexity in the future.  I guess I'm OK to go back to the old lmm usage given that we want to get 2.4 out the door, and this would otherwise also cause some problems with older clients as well.

In hindsight, it was a mistake not to move from the "IGIF FID" format used in 1.8 (with {inode} == seq instead of "id") directly to use lu_fid here.  I only thought of that a couple of months ago, but it was too late by then since the change had been made for 2.1.0.

Possibly at some point in the future we can have LFSCK convert the old LOV_MAGIC_V1/V3 layouts in the filesystem to the new format so that the use of ost_id can be removed.
