Shadow, I'm going to resubmit this patch with just the autoconf changes, since I have a need to fix the autoconf portability also.

Please submit the portability changes in a separate patch.

Also, there were a few other fixes unrelated to portability that should be in yet another patch.  It makes it more difficult to inspect and land any patch if it has many unrelated changes together, and if there is a problem with one part of the patch it prevents the other parts from landing.

Regarding the platform portability fixes, I'm not against landing them in the platform directories, if there is a real user of this code available.  As yet, the MacOS code is non-functional and unsupported for several years, and the WinNT code is also unreleased.  I understand you also have been working on a FUSE client, but I don't think I've ever seen it released anywhere?  Unless there is usable code on these platforms, these cfs_ portability interfaces are just adding overhead and complexity to the Linux client for no value at all.

The current plan moving forward is to make the libcfs API match the Linux kernel API, and put the portability into the (uncommon) platforms libcfs code.  That is what was done for ZFS-on-Linux using the "Solaris Portability Layer" (SPL) to emulate the Solaris kernel API on Linux, avoiding the need to change the ZFS code significantly.

For example, the ALIGN conflict could be avoided by using:

 #undef ALIGN
 #define __ALIGN_MASK(x, mask)   (((x) + (mask)) & ~(mask))
 #define ALIGN(x, a)             __ALIGN_MASK(x, (typeof(x))(a) - 1)
 
to the posix/libcfs.h header so that it doesn't conflict with the arch-specific code in userspace.  One could even replace the use of cfs_size_round*() functions with ALIGN() at the same time, since they are redundant at this point.

I'm not against portability, but I'm against complexity in the Linux code that so far has shown no benefit to any Lustre users, but has definitely hurt the client performance (namely CLIO).
