--- Commit Message ---
LU-9019 obd: express timeouts in timeout_t

In the latest kernels time_t has been removed since time_t has
been a 64 bit value just like time64_t so no need for it anymore.
To avoid confusion between timestamps and timeout values Lustre
has a timeout_t typedef which is in seconds and is a s32 since
timeouts are generally short. This helps to avoid errors that
has happens in the past with certain math operation between
timeouts and timestamps that lead to overflow and underflow
cases.

In light of the problems with patch https://review.whamcloud.com/36688 "LU-12931 timers: correctly offset mod_timer" it makes sense to refresh this patch to clarify the time values.

Definitely using "time64_t" for a 300s delay is not the right thing.  Lots of places are using "unsigned int" for the delays (e.g. at_early_margin, ldlm_timeout, obd_timeout, etc.) but that doesn't really convey that they are in units of seconds like "time_t".  The main drawback of time_t is that it is "unsigned long" which is larger than necessary for the timeout values (64-bit vs. 32-bit), but probably better to have some indication that it is a time.
