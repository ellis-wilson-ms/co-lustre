--- Commit Message ---
LU-5510 scrub: ldiskfs_create_inode returns locked inode

There was race condition between creating new inode and OI scrub:
the OI scrub may find the new created inode just after the creator
creating it but before setting the LMA EA. Originally, to resolve
such trouble, the creator will set the new created inode's state
as LDISKFS_STATE_LUSTRE_NOSCRUB. But such state is set after the
new inode unlocked. So the OI scrub still has some chance to find
the new created inode with neither LDISKFS_STATE_LUSTRE_NOSCRUB
nor LMA EA.

Be as improvement, this patch makes the ldiskfs_create_inode() to
return the new created inode with lock. The caller can set more
state (not only for LFSCK, but also for other purposes in future)
on the new created inode before unlock it.

Firstly, the patch does not introduce additional patch for ldiskfs, because this patch only updates ldiskfs_create_inode() that is introduced by other former ldiskfs patch.

The second, moving NOSCRUB to osd_object is not good, because from the OI scrub view, it does not care about the object, if OI scrub establishes the object (for every scanned inode) in RAM, it need to touch OI lookup, and cause unnecessary complexity and overhead.

--- Message (Alex Zhuravlev, 2015-01-23 07:49:15.000000000) ---

indeed, if an inode is being created, then it's referenced and presented in LU cache. after that LMA is supposed to be set.

--- Message (Fan Yong, 2015-01-23 07:56:38.000000000) ---
Patch Set 3:

The case is that when the ldiskfs_create_inode() return an unlocked inode to OSD, but before the create handler to set osd_object's flag (NOSCRUB) and the LMA, the OI scrub may have found inode. Then the OI scrub cannot distinguish whether it is really new created inode or not. That is why we want the ldiskfs_create_inode() to return locked inode to the caller.
