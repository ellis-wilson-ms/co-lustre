LU-19956 osc: fix race from direct ops_transfer_pinned clear

osc_completion() cleared ops_transfer_pinned directly
and dropped the cl_page reference in a separate step.
This decoupling allowed a race with osc_page_delete():

  osc_completion:                osc_page_delete:
    ops_transfer_pinned = 0
                                 if (ops_transfer_pinned)
                                   /* false, skip put */
    cl_page_put() /* orphan */

On aarch64 (weak memory ordering), the flag clear could
become visible before the ref drop, causing
osc_page_delete() to skip its cl_page_put() and leaving
an orphaned transfer reference.  The final put in
osc_completion() then hit the cl_batch_put assertion.

The deeper problem is that ops_transfer_pinned was used
as a flag to coordinate which of two code paths would
drop a reference.  A reference should have a single
owner responsible for releasing it -- using a flag to
arbitrate between two paths that might release it is
inherently fragile.

Examining the cl_page state machine shows this two-path
design was never necessary: the transition matrix
forbids CPS_PAGEOUT/CPS_PAGEIN -> CPS_FREEING, so
cl_page_complete() (which transitions to CPS_CACHED)
always runs before cl_page_delete().  Completion is
the sole owner of the transfer reference; delete never
needs to release it.

Fix by using osc_page_transfer_put() in osc_completion()
which clears the flag and drops the ref together.
Replace the osc_page_transfer_put() in osc_page_delete()
with LASSERT(!ops_transfer_pinned) to enforce the state
machine guarantee.

Add sanity test_80a to stress the completion path with
a fail_loc delay and concurrent lock cancellation.

## Fix Summary

### Problem
The patch introduced a compile error because `osc_page_transfer_put()` was being called from `lustre/osc/osc_cache.c` but the function was defined as `static` in `lustre/osc/osc_page.c`, making it invisible to other compilation units.

```
lustre/osc/osc_cache.c:1447:3: error: implicit declaration of function 'osc_page_transfer_put'
```

### Solution
Made `osc_page_transfer_put()` a non-static (exported) function by:

1. **Removed `static` keyword** from the function definition in `lustre/osc/osc_page.c` (line 45)
   - Changed from: `static void osc_page_transfer_put(...)`
   - Changed to: `void osc_page_transfer_put(...)`

2. **Added function declaration** to the public header `lustre/include/lustre_osc.h` (line 538)
   - Added: `void osc_page_transfer_put(const struct lu_env *env, struct osc_page *opg);`
   - Placed it in the `/* osc_page.c */` section with other exported functions from that file

### Rationale
The patch's design requires `osc_page_transfer_put()` to be called from `osc_completion()` in `osc_cache.c` to atomically clear the `ops_transfer_pinned` flag and drop the cl_page reference together. This prevents the race condition described in LU-19956 where the flag could be cleared separately from the reference drop, allowing `osc_page_delete()` to see an inconsistent state on weakly-ordered architectures like aarch64.

By making the function non-static and properly declaring it in the header, `osc_cache.c` can now call it, which is essential for the bug fix to work correctly.
