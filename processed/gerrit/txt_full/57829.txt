LU-18357 ptlrpc: Add read-only flag for nodemap filesets

When adding a fileset to a nodemap, each fileset was previously set to
read-write except if the whole nodemap is set to read-only.

This patch allows setting individual filesets as read-only. Depending
on which fileset applies during a client mount, the fileset's
read-only flag is used to force the mount as read-only. This is done
by setting an -EROFS error code server-side without setting the root
FID if the MDT connection does not match the fileset's ro-flag.

Note, the global nodemap "readonly_mount" property overwrites the
individual fileset read-only flag.

This patch adds the optional "--ro" flag to "lctl
nodemap_fileset_add". "lctl get_param nodemap.NM_NAME.fileset" was
modified to visualize the flag per fileset:

nodemap.c0.fileset=
[
 { primary:     /remote, mode: ro },
 { alternate:   /home },
 { alternate:   /data, mode: ro }
]

Sanity-sec "27e" was added to test this flag. Sanity-sec "27d" was
extended to explicitly test for a read-write mount.

Important implementation details:
The root fid retrieval is now done earlier in
"client_common_fill_super()". Previously, "md_get_root()" was called
after the client already connected to an MDT and OST. Since setting
the mount-point to read-only requires reconnecting to the MDT with the
corresponding read-only flag, "md_get_root()" is now called after
connecting to an MDT (before connecting to an OST) and after the
nodemap may have been set to read-only due to the "readonly_mount"
property.

Further, checking whether subdirectory mounts are server-side
available with the "OBD_CONNECT_SUBTREE" connect flag is now done
after the "MDS_GET_ROOT" RPC is sent (instead of before as
previously).  This guarantees that the connect flags are set and
"OBD_CONNECT_SUBTREE" can be checked.

This patch should resolve Sébastien's concerns. 

FYI, Calling "get_md_root()" on the client twice was more complex than anticipated in the case when reconnecting to MDT0 due to a read-only fileset where the mount is not read-only. 

The reason is that "obd_statfs()" is called before "get_md_root()". After reconnecting as read-only, statfs would use the previously cached value, which caused the mdt0 connect flags to be lost, breaking "get_md_root()" on the client and the server side. To force "obd_statfs()" to not use the cache, its max_age parameter is modified.

Note, this is not an issue with the "readonly_mount" property because "obd_statfs()" failing with "-EROFS" would not populate the cache, so the second call after reconnection cannot use the cache.

I'm not super keen on causing incompatibility with 2.10 since we normally keep this much longer, but this is an uncommonly used feature, and 2.10 isn't that new anymore so possibly it is ok. Knowing what the effects are if a 2.10 client tries a subdir mount would help inform this decision. 
>

>

This patchset no longer causes incompatibility with 2.10 by re-adding the OBD_CONNECT_SUBTREE connect flag check - but this time after the "MDS_GET_ROOT" RPC is sent. IIUC, there are a few things to consider here:
1. #5007 did not remove or add an RPC, it only renamed it but kept the same ID, and so the RPC remains valid regardless of the version. (Even if the RPC was not available on the server, ptlrpc returns EOPNOTSUPP instead of crashing)
2. The difference between before and after #5007 is that if an old server is used that does not  support subdirectory mounts, i.e., before #5007, the subdirectory is ignored and the root fid is returned. This is a potential security concern if the connect flag is not checked when a sub dir is set with an old MDT.
3. This means that we can allow an RPC to be sent. Since the target (MDT) connect flags are set once the RPC is received, we can check for "OBD_CONNECT_SUBTREE" after. With this patch, in case a modern client connects to an old mdt, the root fid would be set server-side, but if a subdir (fileset) is defined client-side and the OBD_CONNECT_SUBTREE is not set, we abort the "get_root" on the client.

Would this be satisfactory?
