--- Message (Andreas Dilger, 2014-06-11 07:05:28.000000000) ---
Patch Set 1: Code-Review-1

(6 comments)

Definitely one leak in the error path that needs to be fixed.

While this will definitely fix the problem on client-only builds, it isn't clear to me how it fixes the problem of mounting a client filesystem if the server RPM was installed (e.g. if the site wants to keep the same kernel everywhere and uses the server RPMs on the clients)?

I was thinking of just biting the bullet and having the lustre.ko module loaded on the server nodes, but this also drags in lov.ko, mdc.ko, etc. even if they are not needed. Unfortunately I don't have much better to offer than possibly splitting up the fstype between client and server, which would help simplify a bunch of this code, but complicates the upgrade process and documentation significantly.

--- Message (Bob Glossman, 2014-06-11 12:27:31.000000000) ---
Patch Set 1:

The issue of client mounts done in rpms installed from server builds is addressed  by the new added code in the client only mount execution path in lustre_fill_super() in obd_mount.c.  symbol_request(force_client_load) forces the load of lustre.ko if it isn't already loaded.
