Patch Set 1: I would prefer that you didn't submit this

Hmm, it seems we have both "struct ldlm_callback_suite" and "struct ldlm_enqueue_info".  I think it makes more sense to merge these two structures, so that ldlm_enqueue_info has proper function pointer declarations for the callbacks, but we use a single "callback argument" structure in the LDLM code.  The ldlm_enqueue_info also includes the lock type and flags, which are common to all of the places where ldlm_callback_suite is also being used.

--- Message (Andreas Dilger, 2013-05-08 19:02:22.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

(72 inline comments)

Quite a large number of coding style issues that would have been caught earlier if the checkpatch.pl output in the commit message was looked at.  Please ensure you have the right commit hooks installed:

 ln -sf ../../build/commit-msg .git/hooks/
 ln -sf ../../build/prepare-commit-msg .git/hooks/

In any case, there is also a potential defect in the patch, because it is passing the address of a structure on the stack back to the calling function, and my reading shows that "const" is not enough to ensure that the structure is allocated in global memory.  It needs to be declared as "static" for that to happen.

This, in turn, opens up a large optimization in ldlm_lock memory reduction, because the static ldlm_callback_suite structures can be shared among all locks of that type, instead of copied separately into every one of the millions of locks in memory.

One other minor change is that it would be nicer if "cbs" were named "lcs" everywhere, so that it matches the structure field prefixes.  I don't think it is an improvement to have two different common names for these structures.
