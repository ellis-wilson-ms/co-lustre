files in the directory $LUSTRE_TESTS_API_DIR with the extension .c are
compiled and linked. The directory is located in the build tree at
./lustre/tests/clientapi.

Li Wei makes a good point - why wait until the test is run (_if_ it is run locally by the developer) to detect build failure, when it could be caught during the build itself?  Catching that at build time takes a lot less resources.

It probably makes sense to add some comments to the test itself that makes it clear the "fix" for a build breakage is NOT to change the test program, but rather to ensure that the test program does not break in the first place.

--- Message (Christopher J. Morrone, 2013-01-15 01:41:06.000000000) ---
Patch Set 7:

I think that it may be somewhat harder to test this at normal build time.  You'll need to be very careful to scrub the build line of all of the lustre internal includes and such.  After all, we've failed to notice for a long time that lustre_idl.h doesn't build in user space, because it builds fine as part of the lustre build system.

So I think thats what Richard was trying to do here, build it more independently so we catch those problems in the future.
