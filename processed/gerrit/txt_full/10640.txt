--- Commit Message ---
LU-3536 lod: Separate thandle to different layers.

Separate thandle into different layers on MDT stack.

The current implementation use single thandle in all
layers, which might cause some issues for cross-MDT
transaction, for example during transaction stop,
it needs to stop local OSD transaction first,
then send remote RPC, because we do not want hold the
transaction, during RPC sending, but once we stop osd
transaction, which might cause this single thandle be
destroyed (see osd_trans_stop()), but all of remote
updates are still attached in this thandle.

This patch will separate the thandle to different layers:
1. MDD thandle will present itself in MDD transaction.
2. LOD thandle will distribute the thandle to all sub-thandles,
   and also attach the update blob buffer to store updates for
   cross-MDT operations.
3. OSP thandle will store the updates for the remote
   correspondent target, and also manage to send them to the
   remote target.
4. OSD thandle will still be the same as the original one,
   relying on the bottom FS to achieve the "atomic".

I don't understand how this can work correctly.

If there is a transaction on the MDD, then all of the updates done on the local OSD need to be in a single transaction, both for the local updates and any llog records for the remote OSPs.  Otherwise, if there is a crash then some of the handles may have committed (e.g. llog updates) but others were not committed (e.g. local OSD namespace updates) and the filesystem will be inconsistent after recovery.

That means there should be a single OSD-level transaction for all of the OSD/OSP devices.

Patch Set 5:

"If there is a transaction on the MDD, then all of the updates done on the local OSD need to be in a single transaction, both for the local updates and any llog records for the remote OSPs. Otherwise, if there is a crash then some of the handles may have committed (e.g. llog updates) but others were not committed (e.g. local OSD namespace updates) and the filesystem will be inconsistent after recovery.
That means there should be a single OSD-level transaction for all of the OSD/OSP devices."

Yes, there is only 1 OSD-level transaction for 1 operation (for the whole transaction), which is the osd_thandle, same as before. The reason for this patch is that current implementation of thandle for remote operation is a bit hacky, because it use single thandle in all layers. For example during transaction stop,

On the one hand, we needs to stop OSD transaction first, then send remote RPC, because we do not want hold the transaction, during RPC sending.

But on the other hand, once we stop local osd transaction, which might cause this single thandle be destroyed. (see osd_trans_stop()), but we actually want to keep this thandle until all of OSP finishes its job(call trans_stop). Though we can use some kind of reference count to "resolve" that, but this makes code a bit complicate, and also make OSP and OSD are "correlated" somehow, which is not very nice. 

So we come up the idea of separating the thandle for different layers. 

1. MDD thandle will present itself in MDD transaction.
2. LOD thandle will distribute the thandle to all sub-thandles, and also attach a update blob buffer to store the updates.
3. OSP thandle will only store the updates for the remote correspondent target, and also manage to send them to the remote target.
4. OSD thandle will still be the same as the original one, relying on the bottom FS to implement the "atomic".

Though there are a few expection here,
1. MDD needs to access the OSD thandle sometimes because llog, i.e. LLOG in MDD does not go-through LOD, instead it will go to OSD layer directly.
2. OSP will need to access OSD thandle sometimes for example update last-rcvd.
I am not sure these two exception will be "fixed" later, but right now, I just add th_storage_th to temporarily resolve this.
