--- Message (Nikitas Angelinas, 2018-11-02 10:27:53.000000000) ---
Patch Set 1:

I had only validated the CRR and ORR/TRR policies by simply adding some debug prints to output the round and sequence values for requests, and manually checking that the policies appear to work. Some of the results of those checks can be seen in slides 10-11 in https://www.eofs.eu/_media/events/lad12/10_nikitas_angelinas_xyratex_lustre_nrs_lad_12.pdf. However, the CRR policy was reworked to batch requests afterwards, and I am not certain that I had checked the updated version for correctness, unfortunately. I believe that I must have, as I tried to check everything in detail at the time, but I am not sure if I have any documented evidence of having done so. I agree that there should be formal tests for the correctness of these policies. I did want to do this, but I stopped working on Lustre shortly after NRS landed and up until recently, so I never really got the chance. I can try to come up with some verification tests. I will look at what the TBF tests do in sanityn to see if similar principles can be applied. If someone has some ideas, please let me know.

One thing that might interfere with the request ordering from these policies is that there is a separate pair of NRS instances per CPT (one for regular and one for hp requests),  and so the resulting ordering may not be ideal with more than one CPT. Iirc, client NIDs might also have an affinity to certain cpts, which could also affect CRR ordering, but I could be mistaken. I am not sure how important this might be, though, and I am also not sure if these policies have been found to be useful in the field? I am guessing that CRR may have tangible benefits, but I am not about ORR/TRR?

James, I will try to review this patch soon; sorry!

I just wanted to say that I think ORR is actually quite important, if we can get it to work well. This linearizes the IO stream for shared-file writes, and for reads of both FPP and SSF files. There is work underway to add FIEMAP to ZFS, so it could significantly benefit from ORR on read because the writes are relatively random in order, and once we can determine disk offsets via FIEMAP we could sort the reads better.
