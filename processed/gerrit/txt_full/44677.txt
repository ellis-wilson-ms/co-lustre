LU-14948 build: Warn about /usr/src/lustre.tar.bz2

When /usr/src/lustre.tar.bz2 exists, make debs (and dkms-debs)
will fail with an error like:

  Extracting the package tarball, /usr/src/lustre.tar.bz2, ...
  ../../generic.sh: line 73: debian/rules: Permission denied
  BUILD FAILED!

Add the current git hash to the lustre tarball, as well as
attempt to remove the conflict from /usr/src.  Failing that,
give a warning to ask the user to remove the conflicting file.

I'm not really familiar with the Debian/Ubuntu build process, but it seems strange to me that the build would depend on storing/using the file from such a "prominent" location, and without an embedded version number, rather than just creating a tarball in /tmp or in the current working directory?

On the one hand, I dislike "workarounds" like this where the code prints an error message for the user to fix something, when the code could just as easily delete or rename the file itself. On the other hand, the build script shouldn't be deleting (nor storing) files from a common location, which might contain the "pristine" sources and will not necessarily be identical to the tarball created by the build.

My preference would be to have the build use it's own temporary tarball and ignore the "/usr/src/lustre.tar.bz2" file entirely, but I don't know how baked into the Debian build process that is.
