--- Commit Message ---
LU-17015 gss: support large kerberos token for rpc sec init

If the current Kerberos setup is using large token, like when PAC
feature is enabled for Kerberos, authentication can fail due to server
side unable to exchange token between kernel and userspace.
This limitation is inherent to the sunrpc cache mechanism, that can
only handle tokens up to PAGE_SIZE.

For RPC sec init phase, use Lustre's upcall cache mechanism
instead of deprecated kernel's sunrpc cache. The upcall calls a new
userspace command 'l_getauth', that fowards the sec init request to
the lsvcgssd daemon via Unix domain sockets.

--- Message (Sebastien Buisson, 2023-09-01 19:14:47.000000000) ---
Patch Set 1:

This patch is functional, but still in draft state. While it does make use of Lustre's upcall cache mechanism instead of kernel's sunrpc cache for rpc sec init, the rpc sec context part still relies on sunrpc cache. And it does not remove now unused code based on sunrpc cache.

Anyway, I was able to use Kerberos authentication on my Lustre test system with this patch applied. While not complete, it should fix the token size problem seen at some sites, as my impression was that it was hit during rpc sec init.

This patch adds a new userspace command named l_getauth. I have not checked if it is properly packaged, but in case it is not (which will be fixed in a further iteration of this patch) the binary can be copied by hand under /usr/sbin/ on all server nodes.

Thanks Aurelien. My intention with using a stream was to avoid copying all data to be sent into one large buffer before the actual write to /proc (functions send_response() and do_svc_downcall()).
But indeed, I overlooked that buffered limitation.

So as you suggest, we could try to change the buffering type. But I am not sure unbuffered is the right one. According to setvbuf man page:

       The three types of buffering available are unbuffered, block
       buffered, and line buffered.  When an output stream is
       unbuffered, information appears on the destination file or
       terminal as soon as written; when it is block buffered, many
       characters are saved up and written as a block; when it is line
       buffered, characters are saved up until a newline is output or
       input is read from any stream attached to a terminal device
       (typically stdin).

Because we are calling write primitives multiple times consecutively for the same 'gss reply', I am afraid unbuffered could be counter-productive. Maybe line-buffered could be suitable, as long as there is no limit on the size of the 'line' (before it gets flushed).
