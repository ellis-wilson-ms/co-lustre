--- Message (Andreas Dilger, 2017-03-07 10:44:10.000000000) ---
Patch Set 1: Code-Review-1

This patch looks like it is just disabling the xattr interface, rather than fixing it.  Please see the upstream patch for what needs to be done here to add in the xattr handler:

 commit 2c563880ea8fdc900693ae372fa07b3894f8ff63
 Author:     James Simmons <jsimmons@infradead.org>
 AuthorDate: Mon Jul 11 09:40:10 2016 -0400

    staging: lustre: llite: basic port to xattr_handler API
    
    Port the xattr functionality to the new xattr_handler API.
    This is smallest changes needed to move to this new API. The
    function ll_removexattr can be replaced by generic_removexattr
    as well since it also uses the xattr_handler set xattr backend.
    To tell the difference between the two cases we test the flag
    passed in for XATTR_REPLACE. The ll_getxattr function is
    replaced by the generic_getxattr function.
    
    Signed-off-by: James Simmons <jsimmons@infradead.org>
    Signed-off-by: Greg Kroah-Hartman <gregkh@linuxfoundation.org>

In general, when updating the client code for changes in the upstream kernel, it makes sense to use the patches landed to the upstream kernel as much as possible, so that the code stays more similar, and it is likely less effort than figuring out a new patch from scratch.

Also, it avoids the effort that will be needed later to resync the master and upstream kernel client if they have different solutions to the same problem.

That said, there are some times when the patches landed into the upstream kernel are not quite correct, or at least no longer correct for master, so it always makes sense to review the upstream patches again before landing to master.
