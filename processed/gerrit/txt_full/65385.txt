LU-12668 lov: fix NULL deref and UAF in EC read recovery

Three independent bugs cause crashes and page-lock leaks under
concurrent fio read stress with OST fault injection:

1. lov_io_ec_rd_start calls ec_encode_data_base via a shared
   decode context sized for err_nr destinations, but only d_j
   recov_ptrs entries are filled when positions on an "error
   stripe" were already Uptodate in cache.  recov_ptrs[d_j..
   err_nr-1] stay NULL and ec_encode_data_base+0x8e NULL-derefs
   writing to the first unfilled destination.

   Build a per-page err_array_j (subset of err_array, same sort
   order) and use the shared recover_ctx only when d_j == err_nr
   (full-stripe failure, common case).  Otherwise build a
   per-page ctx for the d_j actual errors and the first d_j
   entries of parity_used -- any d_j distinct parity stripes
   suffice to recover d_j data positions.  UPTODATE pages stay
   as sources, never destinations.

2. lov_ec_read_parity_stripe submit-failure path calls
   cl_2queue_fini() which puts every cl_page in qin/qout,
   dropping their refcounts to zero and freeing them, then
   falls through to out_err which cl_page_put()s the same
   out_clpages[j] entries.  This is refcount_warn_saturate
   underflow + UAF and corrupts the vmpage lifecycle enough
   that concurrent readers hang on a vmpage lock nobody
   releases.

   Null out_clpages[j] after cl_2queue_fini() so out_err skips
   the freed pages.  Vmpages remain caller-owned and are still
   unlocked + __free_page'd by out_err.

3. lov_ec_read_stripe_pages out_err unlock_pages every
   out_vmpages[j] lacking an out_pages[j] cl_page.  UPTODATE
   pages have that exact shape -- added to uptodate_pages[]
   with no cl_page -- and will be unlocked again by the
   caller's rg_cleanup uptodate_pages[] loop.  The double
   unlock races with concurrent grab_cache_page(), appearing
   as a permanently locked vmpage.

   Skip UPTODATE entries in out_err.  rg_cleanup remains the
   single unlock site for those.

Deterministic reproducer: sanity-ec 44d (dd loop) crashes at
ec_encode_data_base+0x8e within seconds of starting OST fault
injection on an EC 4+2 layout with 2-OST fault masks; 44e/44f
(fio buffered/direct) additionally hang on vmpage locks.  With
these three fixes, 44d passes cleanly.  44e/44f pass at 30s
stress but still show a residual page-lock leak at 60s; they
stay in sanity-ec ALWAYS_EXCEPT pending that follow-up.
