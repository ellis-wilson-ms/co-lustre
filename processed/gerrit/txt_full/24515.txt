LU-8433 nrs: Maximizing throughput via rules with dependency

In order to ensure efficient utilization of I/O resource, we
propose a dependency rule strategy. The command of a dependency
rule is shown as follow:
        "start ruleB <matchCondition> deprule=ruleA
                lowerrate=$r1 upperrate=$r2"

Where 'deprule' represents the rule name of the dependent rule,
which means, 'ruleB' depends on 'ruleA' ; the key 'lowerrate'
indicates the lower bound of RPC rate limited value while the
key 'upperrate' indicates the upper bound of RPC rate limited
value.
The principle is that the real RPC rate limited value of a rule
is dynamically adjusted between the lowerrate and upperrate to
obtain more I/O bandwidth according to the spare I/O capacity that
its dependent rule does not make full use of.

--- Message (Andreas Dilger, 2018-02-20 18:45:30.000000000) ---
Patch Set 3:

I'm definitely in favor of some functionality like this, but it isn't clear that this is the best interface for adding such functionality.

Is there a reason to have "deprule" instead of just adding lowerrate= and upperrate= to "ruleA" directly?

My understanding is that most TBF rules are essentially "uperrate=" rules, in that they specify an upper limit on the number of matching RPCs that can be processed per second.  There is also a "realtime" limit which ensures that rule gets all of the requested RPCs if there is not enough performance to process all of the specified rules, which is very similar to the "lowerrate=" proposed here.  We should avoid having multiple ways to specify the same thing, as that makes it complex for users to understand.

It would be nice to have a mode where the rules are "weights" that specify relative priority, rather than absolute performance, so that servers are not idle when there are outstanding RPCs that could be processed.  This may mean something like "upperrate=unlimited" or "upperrate=idle" or similar, so that RPCs can be processed if there are no other buckets with matching RPCs that have priority.

--- Message (Qian Yingjin, 2018-02-22 02:40:26.000000000) ---
Patch Set 3:

The reason to have a "deprule" is that we can think this feature is to obtain (stolen) the spare I/O capacity that the dependent rule does not make full use of. RuleA only considers the bandwidth utilization of the dependent rule regardless of the load status of the entire server. See the paper "A Configurable Rule based Classful Token Bucket Filter Network Request Scheduler for the Lustre File System" Section 4.3 for detailed.

For something like "upperrate=unlimited" or "upperrate=idle" or similar, it is good feature, but IMOH, it is hard to implement based on current TBF mechanism.
