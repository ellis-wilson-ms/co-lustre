LU-812 llite: O_LOV_DELAY_CREATE conflicts with FMODE_NONOTIFY

Kernel 2.6.36 introduces FMODE_NONOTIFY and defines it as 0x1000000
(commit 12ed2e36c98aec6c41559222e311f4aa15d254b6), this causes the
confliction with O_LOV_DELAY_CREATE which is now defined as octal
0100000000.
This confliction causes kernel remove this flag in build_open_flags,
then "lfs setstripe" a new file will fail with -EEXIST(stripe
already set).

This patch changes O_LOV_DELAY_CREATE to 0120000000(0x1400000)
to make it works for new kernel and compatible with old version
statically linked binary.

Another change is starting to build dynamic linked version of
liblustreapi.so.

--- Message (Andreas Dilger, 2012-08-25 17:11:07.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

This isn't do simple as just changing the bit value. This flag is used by userspace applications and in the Lustre network protocol, and is not tied to a specific Lustre version.

I don't think FMODE_NONOTIFY is exposed to userspace, so some mechanism needs to be in place to use a different value for "FMODE_LOV_DELAY_CREATE" inside the client kernel, and then convert the flag for Lustre to keep the same value for MDS_LOV_DELAY_CREATE over the network.

Within llapi_open_file_stripe() it would be possible to mknod() the file first, then setxattr() on the file before opening it, but that is going to add multiple RPCs to the open process. Alternately, I thought there was another O_ flag added recently which is like "local only" to allow ioctls and such on block device files that do not exist. I can't recall the name right now. 

Since this is a significant issue affecting many different Lustre releases, it should be tracked in its own Jira ticket.

--- Message (Liu Xuezhao, 2012-08-26 03:06:30.000000000) ---
Patch Set 1:

Hi Andreas,

Changing O_LOV_DELAY_CREATE does not affect network protocol between client and server. Because the below logic:

        if (flags & O_LOV_DELAY_CREATE)
                cr_flags |= MDS_OPEN_DELAY_CREATE;

inside mdc_open_pack -> mds_pack_open_flags() already translates client's O_LOV_DELAY_CREATE to server's MDS_OPEN_DELAY_CREATE.
So this change will not bring compatible issues between different version's client and server.

O_LOV_DELAY_CREATE's only usage is pass in a special flag for setstripe a new file. If opened with O_LOV_DELAY_CREATE, then lustre doesn't do the object creation or open until ll_lov_setstripe() ioctl is called. 

user space's llapi_file_open_pool and llite's ll_file_open/ll_lov_setstripe_ea_info already have all needed logic to handle it.

But for kernel after 2.6.36, O_LOV_DELAY_CREATE conflicts with FMODE_NONOTIFY and will be cleared in vfs' do_sys_open -> build_open_flags(). So I change the definition to an un-used bit since it does not affect on-the-wire protocol.

For the "another O_ flag" you mentioned, I think it is O_PATH added by kernel commit 1abf0c718f15a56a0a435588d1b104c7a37dc9bd . It will create the inode but assign an empty_fops (f->f_op = &empty_fops;) which breaks hooking with ll_file_operations, I think O_PATH cannot be used in this case.

Thanks.

--- Message (Andreas Dilger, 2012-08-29 16:15:51.000000000) ---
Patch Set 1:

Shadow,
the problem I see isn't with tools like lfs that are packaged with Lustre, or software that is packaged and versioned at all. Instead, the problem iwill be with user applications that are statically linked with liblustreapi.a and are not under any kind of version control.

It would be possible to add in compatibility code for this, by checking for both the current value and the new value (the value of O_LOV_DELAY_CREATE should be defined with both bits set), so that older binaries will continue to work as long as possible, but newer binaries will be compiled to use both values for some time .  Since newer kernels will mask off the old flag (apparently without any harm) but allow the new value, it should be safe for applications to send both. 

For older releases of Lustre, they could begin to check for either value in the kernel, and send both values from userspace. I don't think there is any way to directly detect from userspace if the kernel is masking this open flag. 

It would also be a good idea to start building a dynamic linked version of liblustreapi.so so that it is possible to fix these sorts of problems without having to recompile the applications. 

I'm of course glad that there isn't a protocol issue, since that avoids a lot of further complexity.

Patch Set 1:

As for the option to use an existing flag, what I was actually thinking of was O_ACCMODE and FMODE_WRITE_IOCTL, which is currently used for floppy.c to open the device only for ioctl() of a floppy drive that has no disk inserted.

It would be desirable to use an existing flag instead of creating a new one, since we may otherwise find ourselves in this situation again in the future, without an easy workaround with no apparent side effects as today. 

One potential concern is that I'm not sure whether it is possible to convert a file opened with O_ACCMODE to having a different open mode afterward or not.  Unlike using some new value for O_LOV_DELAY_CREATE, using O_ACCMODE would affect the core read/write behavior of the file.
