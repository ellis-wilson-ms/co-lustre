--- Commit Message ---
LU-6120 lfsck: notify ever failed server to exit LFSCK

During the first-stage scanning, the local LFSCK instance records
which OSTs have ever failed to respond LFSCK verification requests
(maybe because of network issues or the OST itself trouble). Then
before start the second-stage scanning, the local LFSCK instance
will notify those ever failed OSTs to skip orphan handling since
they missed some OST-objects verification via la_sync_failures().

Originally, after la_sync_failures(), related OSTs will be removed
from the LFSCK targets list, in spite of whether la_sync_failures()
succeed or not, then the subsequent LFSCK notification RPCs will not
be sent to those OSTs. That may cause some OST(s) cannot exit LFSCK
expectedly, and then the subsequent LFSCK start will get failure
since former LFSCK instance has not exit.

> what happens if MDT (where master LFSCK is running) crashed just before the status file is updated? OST got their LFSCK instance running already? will they be "leaked" ?

The LFSCK on OST will detect the LFSCK status on MDT periodically, although such period may be a bit long. So even if the MDT crashed, and failed to recover for a long time, the LFSCK on the OST can still exit finally.

On the other hand, this patch also fixed another "reference leak" on the LFSCK on the OST inside lfsck_layout_slave_double_scan(): if the LFSCK on the OST found its status become LF_INCOMPLETE some earlier, it may leave the double scan without releasing "lfsck->li_double_scan_count", that will cause the LFSCK to hung there for ever.

--- Message (Alex Zhuravlev, 2015-02-04 01:36:01.000000000) ---
Patch Set 2:

right, but what happens if MDT crashed and sysadmin restarted it right when MDT is back while OST is still running old instance?

--- Message (Fan Yong, 2015-02-04 01:49:24.000000000) ---
Patch Set 2:

If the MDT is restarted, then the LFSCK on the MDT will auto resume and skip orphan handling. Under such case, it will not care about whether the OST LFSCK is still run or not. The LFSCK instance on the OST will exit by itself.
