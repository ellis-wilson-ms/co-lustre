LU-80 ldiskfs: large EA support

This patch implements the large EA support in ext4. If the size of
an EA value is larger than the blocksize, then the EA value would
not be saved in the external EA block, instead it would be saved
in an external EA inode. So, the patch also helps support a larger
number of EAs.

This patch is rebased on the latest upstream ext4 codes and adds the following changes into the one under http://review.whamcloud.com/#change,1111:

1) change ext4_set_aops(inode) to ext4_set_aops(ea_inode) in ext4_xattr_inode_create()

2) add unlock_new_inode(ea_inode) into ext4_xattr_inode_create()

3) replace (EXT4_SB(inode->i_sb)->s_es->s_feature_incompat & EXT4_FEATURE_INCOMPAT_EA_INODE)

   with EXT4_HAS_INCOMPAT_FEATURE(inode->i_sb, EXT4_FEATURE_INCOMPAT_EA_INODE)

4) store the EA value in an external inode if value_size > blocksize instead of 1/2 blocksize

The patch does not have the contiguous EA blocks support, which would be added later.

Patch Set 4:

There is still a bug in the code if it causes an oops if there is not enough journal space. 

The calculation of journal credits for the xattr should also be examined closely to ensure it is not too big. It shouldn't overflow on even the minimum size journal transaction (256 credits) when writing 16 blocks.

For the xattr it needs:
- parent inode
- new xattr inode
 - inode bitmap
 - group descriptor
 - superblock or block bitmap (for uninit_bg)
- possibly external xattr block being updated
 - possible freeing external xattr block
 - block bitmap
 - group descriptor 

For each block it potentially needs:
- block itself
- block bitmap
- group descriptor

That means 57 credits for a 64kB xattr in the worst case.
