--- Commit Message ---
LU-11089 obd: rename lu_keys_guard to lu_context_remembered_guard

The only remaining use of lu_keys_guard is to protect the
lu_context_remembers linked list, and always write_lock()
is used.
So rename it to reflect this, and change to a spinlock.
We move keys_fini() out of the locked region in
lu_context_fini() - once we have removed the context from
the lc_remembers list, there can no longer be a race.

This bug was actually introduced in 
Commit 372ef85512dd ("LU-11089 obd: use wait_event_var() in lu_context_key_degister()")

This caused key_fini() to now contain:

		LASSERT(atomic_read(&key->lct_used) > 1);
...
		if (atomic_dec_and_test(&key->lct_used))
			wake_up_var(&key->lct_used);

which doesn't make sense.  If there is a change that dec_and_test will find the the value is used, then sometimes the value must have been 1 - so asserting that it is > 1 is silly.

The part of the commit that introduced the bug is in lu_context_key_degister -  it now calls atomic_dec(&key->lct_used); where it didn't before.  The commit message even explains this:
"Previously ->lct_used would only fall down to 1.  Now we decrement
    it an extra time so that wake_up, which only happens when the
    count reaches zero, ..."

So we need to fix that LASSERT to only require that lct_used >= 1.  Then we can re-apply this commit.
