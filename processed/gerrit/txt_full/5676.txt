--- Commit Message ---
LU-2912 mdd: move fid2path from MDD to MDT

1. Move some linkEA API to obdclass, so both MDD(set linkea)
and MDT(fid2path) can access these LINKEA operation.

2. Move fid2path from MDD to MDT, so it can detect remote
object, and return -EREMOTE to the client. Then the client
will try fid2path on another MDT, and finally the client
will assemble the path fragments from different MDT.

--- Message (Andreas Dilger, 2013-03-12 16:13:04.000000000) ---
Patch Set 1:

If we just generated the path from the FID, I'm not sure why we would immediately do a full fid2path check?  I could imagine that as we get the linkEA that it might make sense to optionally verify the last path component in case of a bad linkEA (since linkEA is not checked by e2fsck). However, there doesn't seem to be a good reason to check the full path each time.

Sure, there could be a race between generating the path and some other operation like rename or unlink, but that is still true immediately after the path is verified, so it provides no real benefit.

Also, we shouldn't be duplicating the LFSCK functionality. I don't mind to verify the linkEA on each access, but let's ensure that it isn't done in several different places.
