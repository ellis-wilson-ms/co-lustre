> Li Xi, would you advise on this if current collector (e.g. collected) can easy do regex for such non-fixed format in single line and parse them?

Indeed, the non-fixed format can not be parsed by Collectd for now, not without some extension. I am thinking of the way to extend Collectd/XML to understand the new format, but it does not look easy.

The fixed format which always dumps 0 for non-exist field would be much easy to parse. We will just need to split the line into fields and the meaning of each field would already be determined.

For non-fixed format, the parsing logic need to determine the meaning by the context, which is out of the capability of Collectd/XML for now.

We might be able to reuse the existing "ExtendedField" mechanism of Collectd to parse the new format, but that needs configuration in collectd.conf, not in XML, which is considered as a configuration per-system. As a format that works for all Lustre installation which is regardless of the configuration, XML should be the place to hold the format information.

So in a word, we need to extend Collectd/XML for the new non-fixed format. 


Following is an example of "ExtendedField" in collectd.conf
<ItemType>
    Type "ost_jobstats"
    <ExtendedParse>
        # Parse the field job_id
        Field "job_id"
        # Match the pattern
        Pattern "(.+)[.]([[:digit:]]+)"
        <ExtendedField>
            Index 1
            Name procname
        </ExtendedField>
        <ExtendedField>
            Index 2
            Name uid
        </ExtendedField>
    </ExtendedParse>
    TsdbTags "procname=${extendfield:procname} uid={extendfield:uid}"
</ItemType>
