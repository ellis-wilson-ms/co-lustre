LU-16210 llite: replace selinux_is_enabled()

selinux_is_enabled() was removed from kernel 5.1.
The commit 39e5bfa add the kernel support by assuming SELinux to be
enabled if the function selinux_is_enabled() does not exist.

This has performances impacts: on older kernel (e.g: Centos7) getxattr
RPCs was not send for "security.selinux" if selinux was disabled.
Utilities like "ls -l" always try to get "security.selinux".

This patch implements the same solution use by XFS in the kernel: it
checks if inode->i_sb->s_security is set to determine if the security
xattr is needed.

It adds also the s_security check for security_inode_notifysecctx() to
avoid calling this function if selinux is disabled (as in
nfs_setsecurity()).

For "Enforcing SELinux Policy Check" functionnality, the selinux check
have been moved in l_getsepol: -ENODEV is returned if selinux is
disabled.

Add a regresion test "sanity test_907" for this use case.

Patch Set 4: Code-Review-1

If I understand correctly, the new solution based on i_sb->s_security does not cover all use cases. It will work only when SELinux is entirely disabled at boot via "selinux=0" specified on the kernel boot command line. If SELinux is set to 'disabled' via /etc/selinux/config, this solution will not be able to detect it, and unwanted/unnecessary getxattr requests will still be triggered.
Because of that, this patch is going to introduce a regression on RHEL/CentOS 7. With the current code, at least it works on all cases on RHEL/CentOS 7 thanks to the selinux_is_disabled() function still present.

So this patch should not entirely get rid of selinux_is_disabled() detection and usage. It should use it when present, and 'replace it' with something based on i_sb->s_security otherwise.

BTW, I am puzzled about Ceph and NFS. The implementation based on i_security was relying on their own implementation, so does it mean their implementation is broken?
Actually, it does not seem that wrong to use i_security to decide on whether SELinux is enabled or not. It is just that i_security can be used for other security modules. So basically, if we stick with the implementation that uses i_security, we can end up getting security.selinux xattrs whereas SELinux is disabled but another security module (LSM) is enabled. And generally speaking, I am wondering if this is desirable to have BPF LSM loaded if not used.

lockdown,capability,yama,selinux,bpf
