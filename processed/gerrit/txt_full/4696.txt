LU-2109 llog: Defer closes of log handles under processing

When an OSP is processing a plain log using llog_process_thread(), the
log may be destroyed during a record cancellation, in which case the
log handle will be freed in osp_sync_process_queues().  This causes
trouble for llog_process_thread(), as the llh_bitmap[] freed together
with the log handle is needed to guide processing.  This patch detects
busy log handles in llog_cat_cancel_records() via non-zero lgh_cur_idx
values and defers closes of busy log handles to llog_cat_process_cb().

Patch Set 1:

the issue happens when some records ahead of the current position are cancelled already.
IOW, when this is not the first pass.

log_cancel_rec() destroys llog only if lgh_last_idx==MAX.


when we enter llog_process_thread() we have the following cases:
1) lgh_last_idx == MAX
    no new records are possible, so using a copy of llh_count is safe
2)  lgh_last_idx < MAX
   a) llog is in use, new records are coming
      this is the first scan, as catalog code do not re-use llogs after reboot
      no cancelled records ahead are possible
   b) llog is from previous boot, no new records
      log_cancel_record() won't destroy loghandle
