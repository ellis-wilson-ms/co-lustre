The overhead is that it has to create cl_lock structure for each I/O. The overhead is significant when doing small like 4KB I/O. Without this patch, I couldn't see any performance improvement in my VM benchmark.

--- Message (Andreas Dilger, 2018-12-18 00:12:31.000000000) ---
Patch Set 1:

IMHO, doing lockless IO is the best way to move forward for handling small random IO from multiple clients.  Essentially, if the OSTs detect lock contention from multiple nodes and the IO size is small, they stop giving out locks and the clients just dump the pages without caching to the OSS.  We don't want to _always_ do lockless IO (e.g. single client random write, or random read of a "small" file), because clients can aggregate multiple writes, and for reads they could do large read IOs into client cache (i.e. LU-11416).
