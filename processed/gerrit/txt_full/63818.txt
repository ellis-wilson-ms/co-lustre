## Summary of the Fix

### Problem
The patch removed autoconf tests for kernel features from versions 4.9-4.18, but left behind compatibility code that depended on those tests. This caused two issues:

1. **`posix_acl_update_mode` conflict**: The code tried to define a static inline version of `posix_acl_update_mode()` because `HAVE_POSIX_ACL_UPDATE_MODE` was never defined (its autoconf test was removed). This conflicted with the kernel's own declaration of the function.

2. **`HAVE_RADIX_TREE_REPLACE_SLOT_3ARGS` undefined**: The autoconf test for this was removed, but the compatibility code still referenced it.

### Solution

**File: `lustre/include/lustre_compat.h`**

1. **Removed the `posix_acl_update_mode` compatibility shim** (lines 238-262):
   - Deleted the entire `#ifdef HAVE_IOP_SET_ACL` block that provided a fallback implementation
   - This is safe because kernel 4.9 introduced `posix_acl_update_mode()`, and since we no longer support kernels before 4.9, all supported kernels have this function built-in
   - The macro wrapper at line 461-462 remains because it handles API differences in later kernel versions (namespace argument) that are still supported

2. **Simplified the `radix_tree_rcu` definition** (lines 470-474):
   - Changed from conditional definition to unconditional: `#define radix_tree_rcu __rcu`
   - This is safe because kernel 4.9 changed `radix_tree_replace_slot` to take 3 arguments and use `__rcu`, and since we no longer support kernels before 4.9, all supported kernels use the 3-argument version

### Rationale

The patch removed support for kernels 4.9-4.18 by removing their autoconf feature detection tests. However, it didn't remove all the compatibility code that depended on those tests. Since the removed tests were checking for features introduced in kernel 4.9, and we're now only supporting kernels 4.19+, we can safely assume these features are always present and remove the compatibility shims.

I removed the compatibility wrapper for `posix_acl_update_mode` from `lustre/include/lustre_compat.h` (lines 213-237).

### Rationale

1. **Root Cause**: The patch removed the autoconf test `LC_POSIX_ACL_UPDATE_MODE` which defined the `HAVE_POSIX_ACL_UPDATE_MODE` macro. This macro was used to conditionally compile a compatibility wrapper for the `posix_acl_update_mode` function.

Since the patch explicitly removes support for kernels 4.9-4.18 and assumes a minimum kernel version where this function exists, the compatibility wrapper is no longer needed. All supported kernels now have this function natively.

4. **Impact**: The code will now use the kernel's native `posix_acl_update_mode` function directly, which is the correct behavior for kernels 4.19 and later.

## Summary

**What was the problem:**

The patch removed support for kernels 4.9 to 4.18, but left a static inline wrapper for `posix_acl_update_mode()` in `lustre/include/lustre_compat.h`. This function was introduced in kernel 4.9 (commit 073931017b49), so it should always be available in kernels 4.19 and later.

The compile error occurred because:
1. The kernel's `linux/posix_acl.h` header (included via `linux/posix_acl_xattr.h`) declares `posix_acl_update_mode` as a non-static extern function

3. This created a conflict: "static declaration follows non-static declaration"

**The fix:**

Removed the entire static inline wrapper for `posix_acl_update_mode()` (lines 213-238 in `lustre/include/lustre_compat.h`) because:
- The function was added in kernel 4.9
- The patch removes support for kernels before 4.19
- Therefore, this compatibility wrapper is no longer needed - the kernel will always provide the function

**Rationale:**

Since Lustre now requires kernel 4.19 or later, and `posix_acl_update_mode()` has been available since kernel 4.9, there's no need for a compatibility shim. The kernel's native implementation will always be available. The patch author mentioned keeping the POSIX_ACL_UPDATE_MODE test for collision avoidance with another patch, but the actual compatibility wrapper code should have been removed along with the other pre-4.19 compatibility code.
