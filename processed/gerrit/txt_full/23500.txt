LU-8569 linkea: linkEA size limitation

Under DNE mode, if we do not restrict the linkEA size, and if there
are too many cross-MDTs hard links to the same object, then it will
casue the llog overflow. On the other hand, too many linkEA entries
in the linkEA will serious affect the linkEA performance because we
only support to locate linkEA entry consecutively.

So we need to restrict the linkEA size. Currently, it is 4096 bytes,
that is independent from the backend. If too many hard links caused
the linkEA overflowed, we will add overflow timestamp in the linkEA
header. Such overflow timestamp has some functionalities:

1. It will prevent the object being migrated to other MDT, because
   some name entries may be not in the linkEA, so we cannot update
   these name entries for the migration.

2. It will tell the namespace LFSCK that the 'nlink' attribute may
   be more trustable than the linkEA, then avoid misguiding the
   namespace LFSCK to repair 'nlink' attribute based on linkEA.

There will be subsequent patch(es) for namespace LFSCK to handle the
linkEA size limitation and overflow cases.

One option to handle the dangling references for migrated FIDs is to keep the "FID forwarding" table longer than just until the next recovery, but until the next LFSCK run.  That would only be needed for files with hard links that are not in the linkEA list.  For MDS removal this table would need to be stored on some other MDS than the one being removed, or alternately we would need to require a full LFSCK run when removing an MDS.  That would be possible when removing an MDS cleanly, but not when the MDS dies unexpectedly.

Another option that I previously discussed with Di in LU-7607 it would also be possible to keep the original FID in the LMA after migration so that LFSCK could repair the links by matching the dirent FIDs to the original FID in the LMA.  Once LFSCK completes a run and no links are found to FIDs in the table, and the entries are old enough the entry could be removed.

If the latter proposal seems useful, it would make more sense to change the implementation here to store the overflow timestamp in a "remap index" (ZAP or IAM) that stores the target FID and/or the timestamp in a small struct, so that we can know when it can be expired from the table.  This has the advantage of consolidating the existing remapping table and moving the overflow timestamp out of the linkEA, so that it doesn't consume space in every single xattr.  That table would need to be used by LFSCK or if OI lookup returns ENOENT.  It _might_ be possible to store the remap table in the OI directly so that there is old FID->new FID mapping, but I don't know how easily that would be implemented.
