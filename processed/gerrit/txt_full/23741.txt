--- Commit Message ---
LU-8569 lfsck: handle linkEA overflow

If the linkEA is marked as overflow (32 bits timestamp in header),
and even if some hard links are removed, we still cannot clear the
overflow timestamp, because the missed hard links entries are still
not in the linkEA. It is the namespace LFSCK's duty to add the missed
entries into the linkEA (if possible) and clear the overflow timestamp
if all the hard links entries are in the linkEA. But it is possible
that there are some new hard links added during the namespace LFSCK
scanning. So the namespace LFSCK needs to compare the linkEA overflow
timestamp with the namespace LFSCK latest reset time (scan the system
from the beginning) in the 2nd-stage scanning. If the latter one is
newer, then means all the hard links entries have been in the linkEA,
so the linkEA overflow timestamp can be cleared.

To avoid the trouble caused by clock drift among MDTs, the linkEA
overflow timestamp will be set as the MDT local time on which the
object resides even if the set linkEA operation is sponsored by
some remote MDT. So we can directly compare the linkEA overflow
time with the namespace LFSCK latest reset time.

The old interfaces between MDD/OUT and LFSCK via lfsck_in_notify()
for the linkEA overflow event are obsoleted and removed.

--- Message (Fan Yong, 2016-11-15 04:23:46.000000000) ---
Patch Set 1:

> I still think it is possible to handle this case with just one or two bits rather than storing the actual overflow time, since we don't really care about what time the linkEA overflowed except in the very rare case of overflowing the linkEA in the middle of running LFSCK.

In theory, it may work as you suggested. As we original did, we can make the MDD to talk with LFSCK directly, or make the MDD to realise that the LFSCK is running, the MDD will notify the MDD some linkEA overflow. But such way is regarded as unclean, and was suggested to replaced by current solution in LU-5802.

The LFSCK needs NOT to know the exact time of linkEA overflow. It just needs to know whether the overflow is happened during the LFSCK or before the LFSCK. So on some degree, we can replace the timestamp with some kind of generation. But usually, Lustre uses 32-bits (or 64 bits) generation, if we just use 8-bits generation, it is quite easy to be reused.

--- Message (Fan Yong, 2016-11-15 04:54:13.000000000) ---
Patch Set 1:

And please note that the new patch uses 32-bits for overflow timestamp, not 64-bits. If we have other requirement for the linkEA in the future, we can use another left 32 bits padding. Even if we only use 8 bits in this patch, the left bits is also less than 64 bits. Not too much different from the current patch.
