LU-2456 lnet: Dynamic LNet Configuration (DLC) IOCTL changes

This is the fourth patch of a set of patches that enables DLC.

This patch changes the IOCTL infrastructure in preparation of
adding extra IOCTL communication between user and kernel space.
The changes include:
- adding a common header to be passed to ioctl infra functions
  instead of passing an exact structure.  This header is meant
  to be included in all structures to be passed through that
  interface.  The IOCTL handler casts this header to a particular
  type that it expects
- All sanity testing on the past in structure is performed in the
  generic ioctl infrastructure code.
- All ioctl handlers changed to take the header instead of a
  particular structure type

I haven't had problems but the livermore guys are not happy with ioctl usage. I added Morrone to comment on this. We have been discussing moving from ioctls for the lnet utilities.

--- Message (Christopher J. Morrone, 2014-07-21 20:56:35.000000000) ---
Patch Set 29:

I don't see a problem with using ioctls for system configuration activities like this that a sysadmin would expect to run locally on every node.

We only have an with ioctls when they are used by the Lustre library to implement functions that normal user applications will be expected to user.  For instance, getting and setting lustre stripe values.

A function shipping system has trouble with ioctls because the buffer sizes are not clearly delineated in the function parameters.  Therefore, in order to function ship most of the Lustre API ioctls someone would need to write custom function shippers for each Lustre ioctl.  That we want to avoid.

But normal user applications aren't involved with configuring lnet.  DLC is a more sysadmin-focussed activity.  I don't have a problem with using ioctls for DLC.
