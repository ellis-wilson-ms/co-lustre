What happens if there really a problem with the ".." entry?

--- Message (Fan Yong, 2014-07-09 01:37:13.000000000) ---
Patch Set 1:

Depends on what the "problem" is:

1) If the MDT device is restored from file-level backup, then there will be no FID-in-dirent for ".." entry. The application can get the FID for ".." from the parent's LMA EA. Only very small performance overhead for that.

2) if the MDT device has a bad "FID-in-dirent", then the namespace LFSCK can repair it via checking with the FID-in-LMA EA.

3) If the FID-in-LMV EA is wrong, then we have no good solution to resolve such trouble yet, because all the OI scrub/LFSCK assume that the FID-in-LMV EA is trustable.

As for other inconsistency, such as ".." entry references an invalid parent, the 1.5 LFSCK does not check such case, it is the local e2fsck tool's duty to guarantee that the <parent, child> directory structure is consistent. But in LFSCK 3, the namespace LFSCK will verify related consistency for remote case.
