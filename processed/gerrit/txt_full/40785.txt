Patch Set 1:

> How does sending an RPC with a bad field size translate to -ENOENT
 > returns on the client? Is there some room for further improvement
 > here?

I can't say I fully understand the mechanism here, but one possibility is for a striped directory it may be that a newly-mounted client doesn't get the any layout for the directory due to the zero-length reply buffer, it tries to do the name lookup in the DNE master directory without success, until another RPC reply (for a regular file or directory) comes in to update the minimum buffer size?  This check is ensuring that there is always *some* space allocated for the layout reply buffer. 

In any case, this is restoring code that existed for a long time, and has been shown to resolve this issue at a customer site, and is extremely low risk.
