LU-13689 hsm: replace cdt_state_lock with cdt_llog_lock

HSM requests should not be added to the CDT llog when cdt_state is in
CDT_STOPPED as the CDT is unavailable, or in CDT_INIT as any HSM
requests in the CDT llog may not have been fully processed and so
cdt_last_cookie may not have been reset appropriately. Setting
cdt_state and checking it before adding an HSM request to the llog
need to be serialized.

mdt_agent_record_add() already holds cdt_llog_lock in write
mode in the path where cdt_last_cookie is read and incremented, so
cdt_state_lock can be replaced with cdt_llog_lock to allow to read
cdt_state safely in this path before adding a request. Doing this
should not add much contention to cdt_llog_lock in the request-adding
hot-path, as cdt_state_lock is only held during cdt_state transitions
and while purging requests from the HSM action list, which should be
rare events. One possible issue with this locking scheme might be the
potential to starve threads servicing the aforementioned
administrative actions in cases where a large number of HSM requests
are being added, but it is unclear whether there would be a noticeable
delay in practice.

This change should address cases where the following two assertions
are triggered in cdt_agent_record_hash_add(), due to HSM requests
being added to the HSM llog while cdt_state is in CDT_INIT:

LASSERT(carl0->carl_cat_idx == carl1->carl_cat_idx);
LASSERT(carl0->carl_rec_idx == carl1->carl_rec_idx);

--- Message (Nikitas Angelinas, 2020-06-25 23:48:17.000000000) ---
Patch Set 1:

I am not sure if RAoLU is used in production, unfortunately. Yes, mdt_agent_record_add() is also called in mdt_handle_last_unlink() for implementing RAoLU and that has the same issue that this ticket is attempting to address, i.e. mdt_agent_record_add() can race with mdt_hsm_cdt_start() which resets cdt_last_cookie and can cause a duplicate cookie to be assigned to the request, leading to the assertions in cdt_agent_record_hash_add() failing. This patch currently allows RAoLU requests during CDT_STOPPED and CDT_INIT; this ensures that files are removed from the archive, as is done currently in master and they don't start accumulating, giving the impression that RAoLU doesn't remove them reliably in some cases. Allowing requests during CDT_INIT has the potential of triggering the assertions, although this is the case with current master as well; looking at this again though, it might be better if RAoLU requests during CDT_INIT just fail with an error message
