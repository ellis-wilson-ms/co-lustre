LU-15803 sec: correctly handle page lock in ll_io_zero_page

In ll_io_zero_page(), we need to make sure we have locked the page,
and it is up-to-date, before zeroing. So modify ll_io_read_page()
behavior to not disown the clpage for our use case. It avoids being
exposed to concurrent modifications.

(1 comment)

So, I believe page is not locked in the DIO case (in general DIO doesn't use the page lock - all of the client functions which call 'lock_page' are only for page cache pages, not for DIO).

For the buffered case...  I'm not really sure if it's locked or not - the caller is setattr.  I'd have to look at the code more and I think the crash implies that sometimes it's *not* locked by the caller.  (Or there's a simpler logic mistake in this function that can lead to the bad unlock)

But basically, we need to start this function knowing whether or not our thread has the page locked.  We can't establish that by asking the page about its lock state - it could be locked by anyone.  (Or if we know 100% that the callers either lock the page or the page doesn't need to be locked from that caller, then we should add asserts to reflect those conditions and not touch the lock in this function)

It turns out the page is locked with the call to grab_cache_page_nowait():

   /**
    * grab_cache_page_nowait - returns locked page at given index in given cache
    * @mapping: target address_space
    * @index: the page index


So if we change the initial value of lockedbymyself from true to false, it will lead to leaving a locked page behind.

Also, in case the page is not uptodate and needs to be read from server, the call to ll_io_read_page() unlocks the page, for a reason I do not know. Maybe Bobijam can comment on that?
So in this case, we need to lock the page again.
