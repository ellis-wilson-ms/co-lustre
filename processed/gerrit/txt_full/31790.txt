I'm wondering if it might be better to make the change look more like upstream.

The gist of the upstream API is that all the timer users stop using the "unsigned long data" and use the from_timer() to get the object containing the timer.

From a backport perspective I think from_timer() and timer_setup() can be emulated in backports.

The only issue is the callback routine would need to be wrapped to add the now unused "unsigned long" as the last arg when indicated.

I have confirmed my idea can work.

Here is the emulation:

#ifndef HAVE_TIMER_SETUP
#define cfs_timer_cb_arg_t unsigned long
#define timer_setup(t, cb, f) \
       setup_timer(t, cb, (unsigned long)t)
#define from_timer(var, t, fld) \
       container_of((struct timer_list *)t, typeof(*var), fld)
#else
#define cfs_timer_cb_arg_t struct timer_list *
#endif

I built this on RHEL 7.3 and 4.14.  (4.14 has the new API as well as the old API).

So the process would be:
- Change to use the new timer API
  * The lustre in 4.15 staging has some examples
  * Also see https://lwn.net/Articles/735892
  * upstream patches to convert to the new API
- Compile on 4.14 to test
- Use cfs_timer_cb_arg_t in the callbacks

In cases where the containing struct cannot be derived use a stub struct.
