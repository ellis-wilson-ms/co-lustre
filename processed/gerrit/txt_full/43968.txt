LU-14752 obdclass: handle EBUSY returned for lu_object hashtable

When the rhashtable grows to a certain size it will be rescaled.
When rescaling you can be returned a ENOMEM or EBUSY error. This
we reported as:

LustreError: 3594004:0:(lu_object.c:2472:lu_object_assign_fid()) ASSERTION( rc == 0 ) failed: failed hashtable insertion: rc = -16
LustreError: 3594004:0:(lu_object.c:2472:lu_object_assign_fid()) LBUG
Pid: 3594004, comm: mdt01_020 4.18.0-240.22.1.1toss.t4.x86_64 #1 SMP Tue Apr 13 17:18:40 PDT 2021
Call Trace TBD:
Kernel panic - not syncing: LBUG
...
Call Trace:
dump_stack+0x5c/0x80
panic+0xe7/0x2a9
lbug_with_loc.cold.10+0x18/0x18 [libcfs]
lu_object_assign_fid+0x3b8/0x3c0 [obdclass]

Add handling the EBUSY case for our lu_object hash.

Neil,
I'm not sure what you are getting at here?  I was just making a friendly offer to help, and trying to keep the planet a few hours further away from heat death for changing a few lines in the commit message.

By no means is this an implication that there aren't enough nodes to run testing (which we provide gratis to anyone who submits a patch), so no need to jump to wild conclusions about "unfit for purpose".

If you are concerned about *over-testing* a patch (which can sometimes be the case, but is surprisingly hard to tell), then Oleg has done some work to map patch changes to test sessions to be run, something equivalent to auto-adding one or more lines like:

 Test-Parameters: trivial testlist=xxx <other options>
 ...

but that mapping is an inexact science.  His test system (Lustre Gerrit Janitor) is a secondary test system that runs a focussed set of tests quickly, but still depends on autotest to run the full set of tests for different kernels, arches, backing filesystems, etc.

If you are concerned about *under testing* a patch, there is also contrib/scripts/gerrit_checkpatch.py that is the framework for running scripts (checkpatch.pl in this case) for some/all patches pushed to Gerrit, and is the basis what Oleg uses to trigger the Janitor builder/test runs.  You would be welcome to provide your own automatic testing nodes for contributed patches, all it needs is a Gerrit account to access the patch list and post comments to the patch (preferably not your personal one in the long run, to avoid confusion).

James,
to answer your question about editing the commit message - because we also trigger custom testing via the commit message, but builds are only kept for a week or so, it will always trigger a new build to avoid the case where the build is not available, and because Gerrit/Jenkins aren't very smart.  However, since I'm an admin in Jenkins, I can edit the commit message and kill other people's builds while they are in their 60s grace period after submission, which preserves the Jenkins/Maloo Verified status on the patch (it will only print "No builds executed" in that case) and avoids the retesting.  In most cases, there are other changes that can also be done to improve a patch, but in some limited cases it is worthwhile to do this dance to avoid a full retest.

> If you are concerned about *over-testing* a patch 

This one.  Not particularly that more tests are done when fewer would suffice, but more that any testing is done when none is necessary.
With James, I was surprised that any tests were run on the last update as only the commit message was changed. Has anything changed there?

But it is more than just that.  I'm not particularly concerned about the power-usage of all the testing - more about the time and hassle.
The false negatives have always been an annoyance, but I understand that it is very hard to avoid them with software as complex as lustre.  I don't see them as a reason to not run those tests, but I do see them as a reason not to run ANY tests if none are needed.  The hassle they cause is a real dis-incentive to make trivial changes.

I would *really* like it if updates that did not change the code - only comments and white-space - did not clear any +1s, whether Code-Review or Verified. (reducing a +2 to +1 is probably OK).  This would make the choice of applying a trivial fix or not much easier.

I am also bothered about the testing burden when I update a patch in a large series.  Remembering to make sure the best "Test-Parameters" is present is an unnecessary burden.  I would like it if a patch were tested only if the preceding patch has already passed tests.  That would avoid a lot of pain with a large series, and reduce the disincentive to fix minor problems.
I might like Test-Parameters to (optionally?) be inherited from one patch to the next - but I haven't really thought this through.

It would also be nice if a rebase that didn't materially change the patch would not remove +1's.  Determining "materially" might be a challenge, and there would still need to be some testing as a rebase can obviously break a working patch.  I don't have a clear picture of how this would work, only a clear annoyance that a rebase is more pain that it should be.  With good development practices (which I think we have) a patch which A/ applies without fuzz, and B/ still builds, it extremely likely to still behave the same way.

More broadly - I have a strong belief that the computers should serve us, not the reverse.  They should make life easier, not more difficult.  The fact that you could offer to do something trivial that was not automatic set my alarm bells ringing.
