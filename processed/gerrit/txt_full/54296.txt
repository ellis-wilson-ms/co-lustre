LU-17612 sec: return keyring errors to userspace

In current code, Linux keyring errors, when using GSS Kerberos,
are all masked under a generic ECONNREFUSED error. That makes
it hard to understand the root cause of the problem
for the I/O caller.

Update the code to propagate errors from request_key() up to
the application.

struct ptlrpc_cli_ctx * gss_sec_lookup_ctx_kr(...) is modified
to now returns a NULL pointer or -errval. This is tested by callers
and propagated. NULL values are still converted to ECONNREFUSED.

Patch Set 1: Code-Review+1

I always have mixed feelings with this kind of changes.
On the one hand, I appreciate it is nice to understand what is exactly going on when authentication fails, especially from an admin perspective.
On the other hand, giving precise and accurate reasons for authentication failures is a valuable hint for hackers. For instance, on many websites today, when you fail to authenticate, the site does not tell you if it's either your login or password that was incorrect, but just that the login/password couple was incorrect. So that a hacker does not know if it is worth trying with the same login and a different password.

It is not exactly the same here, but hopefully you get the idea. And to make sure we are all aware of what we are doing.

Thanks for the warning. After double checking that internally in the team, we think this is safe form a security stand point.

Note that here, we are only exposing the keyring behavior on the client, not the permission checks that the MDT or OST could be doing. Security checks from them are untouched and returned codes from the RPCs are still the samùe. We consider the benefit of reporting these errors is way more interesting that the tiny risk of exposing the keyring content. Also note that the keyring content is already accessible by the current user (keyctl show, cat /proc/keys), and Lustre source code too. So he can already find out the EDQUOT, EKEYREVOKED by himself)
