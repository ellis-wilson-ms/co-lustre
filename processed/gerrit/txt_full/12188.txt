LU-5707 lfsck: recreate LFSCK tracing file for load failure

The extended attribute XATTR_NAME_LFSCK_NAMESPACE for namespace
LFSCK tracing file is enlarged since Lustre-2.7. When downgrade
from Lustre-2.7, the old MDS will get -ERANGE when loads the
XATTR_NAME_LFSCK_NAMESPACE extended attribute from the new
formatted on-disk namespace LFSCK tracing file. Under such case,
the LFSCK should not return failure, instead, it can recreate the
XATTR_NAME_LFSCK_NAMESPACE EA as initialization case.

More further, if the LFSCK failed to load tracing file, it is
unncessary to fail the whole LFSCK, instead, just recreate the
tracing file.

Patch Set 3:

That would probably be better than having to patch all of the released versions of Lustre. Of course, the master code should still be fixed to avoid this problem in the future by deleting the xattr if it doesn't understand it, and just start LFSCK again. Also, it should try to delete the old xattr to avoid leaking space since there may only be 4KB of xattr  space on the inode.  

The code to delete the old xattr should be under a check like:

 #ifdef LUSTRE_VERSION_CODE < OBD_OCD_VERSION(3,2,53)

So that it stops trying to remove the old xattr and that compatibility code can itself be removed.
