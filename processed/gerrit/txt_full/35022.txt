While I agree there may be use cases for "force-release". This patch is not the place to add it. Otherwise, one will use "lfs release -f" all the time, bypassing the "data_version" check.

The check is meant to prevent data corruption between Lustre and its HSM backend. It should not be overlooked so lightly.

--- Message (Quentin Bouget, 2019-06-04 17:03:56.000000000) ---
Patch Set 2:

> In your case, the echo sets the dirty flag.

Not necessarily, and that is exactly the point of the data_version.

Unless you can prove that the dirty bit is correctly set everytime a file's data is modified on an OST, even during failures (eg. client crash, OST failover) you cannot remove the check safely. And if you manage to prove this, then you don't need a "--force" option, just remove the data_version check.

The dirty bit is simply a shortcut to the information provided by comparing two different data_versions. In short, if it is set, you know for sure the file was modified since it was last archived. If it is unset, you know the file _probably_ was not modified, but you cannot be sure.
