--- Commit Message ---
LU-14313 utils: mount error when no server support

When the mount utility was built without server support,
and this causes the mount to fail, an error message should
say so.

The error values returned by main() should be positive as
they are exposed to the user.

Upon mount() failure, errno must be checked to determine
the reason for the failure.  The return value is -1.

Gerrit even generates a URL for "b=NNNN" links:

 http://bugzilla.lustre.org/show_bug.cgi?id=16607

which, until recently, would have taken you to the old bugzilla.  However, it appears that this now redirects the https://jira.whamcloud.com/ but I'm not sure when that happened.  It was probably in the past month or two, since I do occasionally reference it for issues like this, but I guess OpenSFS got tired of hosting it for a once-every-two-month request.

I do have an archive of the public issues from that site.  It reports:

   Brian Behlendorf <behlendorf1@llnl.gov>

   A while ago we noticed the 'retry' option was dropped from 
   mount.lustre when it was rewritten.  We've been carrying a
   patch to re-enable this support since we require it for our
   BGL system.  It would be nice to get this back in 1.6.6.

I myself use this option for my clients, since the kernel mount will give up relatively quickly if the MDT0000 is not available at the time of client mount (it is more forgiving of missing OSTs), and if you need the Lustre filesystem mounted for the client to function then it makes sense to block the boot/mount process and retry until it succeeds.
