--- Commit Message ---
LU-16056 libcfs: restore umask handling in kernel threads

This reverts commit 9013eb2bb5 which incorrectly assumes that Lustre
service threads do not modify umask.  A quick grep shows that umask
is modified in osd-ldiskfs __osd_create().

If some other thread sharing the same fs context is modifying umask
in an incompatible way (which includes all Lustre threads after
this patch) then it will occasionally break created file access
permissions for Lustre.

(1 comment)

> Is this true for ALL kthreads or just a special subset.

This is true for at least all of the MDS threads (and has caused a bery rare bug in 2.14+ that has taken us weeks to isolate), but also any other server threads that may be creating/writing files to storage (llog, last_rcvd for client state, etc). 

It isn't even clear why this change was made in the first place?  The empty fs context for a bunch of server threads is trivial.  The code that changes umask is down in osd-ldiskfs because it is still affected by some kernel VFS umask changing code, while osd-zfs doesn't need it. 

We _could_ just change umask=0 for all the server threads immediately after creation, so they don't mess with permissions, rather than save/restore umask for every operation, but I don't think it matters much where it is some in the end, as long as it is done properly.

I'm not against adding comments.  I would think that a file server in the kernel (e.g. knfsd) *would* be using unshare_fs_struct(), and indeed it is:

 nfsd(void *vrqstp)
{
        :
        :

        /* At this point, the thread shares current->fs
         * with the init process. We need to create files with the
         * umask as defined by the client instead of init's umask. */
        if (unshare_fs_struct() < 0) {
                printk("Unable to start nfsd thread: out of memory\n");
                goto out;
        }

so it is confusing to me why Neil thought that this wouldn't be used by the Lustre code.  However, my conclusion is that this patch was developed against the upstream *client* code, which indeed does not need/change umask, unlike the server code that does.
