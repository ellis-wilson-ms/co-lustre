--- Message (Andreas Dilger, 2018-03-09 01:37:33.000000000) ---
Patch Set 1:

> Benchmark:
 > 8 byte append w/dd:
 > dd if=/dev/zero bs=8 oflag=append of=./testfile

What is the file size here?  I tested with count=1000000 to good effect.

 > 3.6 MB/s w/patch
 > ~720 KB/s w/out patch
 > 
 > So about a 4x improvement for Lustre, in a pretty narrow case.

It would be great to include such information in the commit comment.

My local VM testing of this 8-byte append workload showed ~3.7MB/s to Lustre w/o the patch (tmpfs OSTs), and 6.7MB/s directly to tmpfs.  Running on real hardware (1GigE + HDD OSTs, i5-6600@3.30GHz CPU) it was 1.9 MB/s for Lustre and 21.7MB/s tmpfs, so a bit more of a difference but this is still slow no matter how you slice it.

This makes me wonder if tiny appends is the right thing to be optimizing?  Running with bs=4096 count=100000 appends shows a much larger difference between Lustre and tmpfs (6x in my VM, and 110MB/s vs. 3.0GB/s = 27x on my test node) so there are definitely larger improvements to be had for large appends vs. tiny ones.

--- Message (Patrick Farrell, 2018-03-09 03:43:55.000000000) ---
Patch Set 1:

Sure, I'll put that info in the commit message if/when I clean this up to get it ready for possible landing.

About 4K i/o: Certainly there are larger improvements to be had, but 4K append performs not that differently from normal 4K I/O in my testing, and I haven't come up with any great *simple* ideas for improving write performance for writes of 1 page or greater.

Tiny writes specifically depends on having a dirty page to skip the complex work of prepping a page for i/o on Lustre.  The only way I can think of to do the same at 4K and up is some version of Jinshan's containers idea.
