LU-2190 utils: Map ldd_params to individual props

At mkfs/tunefs/mount time the configuration utilities will access
a .ldd_params configuraton string.  This string consists of space
delimited key values pairs which must be saved somewhere.

For ldiskfs it makes sense to store this string on disk as part
of the binary lustre_disk_data structure.  But the logical place
for ZFS is to use the user dataset properties where all the other
lustre_disk_data fields are stored.

This patch updates the ZFS portions of the utilities to take the
.ldd_params string and break it apart in to individual key value
pairs which can be stored as user dataset properties.  This is
done by simply appending 'lustre:' to the provided param key.

The original .ldd_params field can then be reconstructed by
fetching all the dataset properties prefixed by 'lustre:' and
appending them with their keys.

The advantage in this approach is new parameters can be added
without the need to update utilties.

--- Message (Prakash Surya, 2013-03-07 00:07:12.000000000) ---
Patch Set 1:

I agree, the params _should_ be set/tuned _only_ by the lustre utils.           
The problem with that, is the lustre tools are not user friendly. Using         
zfs get/set is a *much* easier to use interface for our admins. So,             
until the utils are fixed to be "better" this isn't an option we'd like         
to pursue.                                                                      
                                                                                
I'm willing to fix up this patch to meet your needs, but I'm not                
completely sure where to go from here. I see two options:                       
                                                                                
   a) Filter out *known* options which are *not* ldd_params                     
      (i.e. version, fsname, etc.). This would ensure these                     
      are not passed into the ldd_params string. The                            
      "drawback" being anything with LDD_PREFIX that is                         
      *not* known will be passed as ldd_params.                                 
                                                                                
   b) Only pass in *known* options which *are* ldd_params. This                 
      ensures everything in the ldd_params string is valid. The                 
      drawback to this is if a new param is added, the mount util               
      must be modified to understand the new param.                             
                                                                                
I don't think (b) is an option because that is what we had before, IIRC,        
and that obviously did not work. If we do (a), then we run into the             
problem where a new option is added that conflicts with one of the ZFS          
dataset parameters (i.e. data set has "lustre:xyz=on" which is ignored,         
but then the ldd_param adds support for "xyz" so it's no longer                 
ignored).                                                                       
                                                                                
IMO, option (a) is do-able and something I can add to the patch. I just         
want to make sure it will satisfy your needs before I work on it. Will          
that work? Or did you have something else in mind?
