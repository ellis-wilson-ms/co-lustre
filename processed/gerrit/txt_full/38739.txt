LU-13563 wbc: reclaim mechanism for inodes cached in MemFS

This patch adds reclaim mechanism for inodes cached in MemFS.
It works as follows. At the setup time on a client, it creates a
kernel daemon thread dedicated to the reclaim works. When detect
caching too much inodes which has reached a high watermark, it
will wake up the daemon thread to start reclaim work until the
cached number is decreased below the half of the inode limit.

Doesn't the kernel already have threads to do this?  If the WBC uses a kmem slab for the cached inodes, there should already be memory pressure. --- Message (Qian Yingjin, 2021-05-23 09:05:45.000000000) ---
Patch Set 6:

From my own opinion, I think the current Linux kernel only considers the write-back caching for data, not very suitable for metadata write-back caching.
The current Linux kernel write-back caching lacks the following functionalities for metadata write-back caching:
- Export the symbol to trigger asynchronous flush in background explicitly.
- Kernel flusher is usually period based. For benchmark workload, before the flusher is waken up, there may be lots of pending dirty files. It would better to support to trigger the flushing when accumulated a certain count of dirty files (dirty count based flush). 
- Kernel flusher for each filesystem backend is single-threaded.

Our WBC work is mainly for Lustre file system. However, due to the general design concept, it only needs some small modification and adaption for other file system, even local file system can benefit from our WBC design.
For a long term, I think we should implement our WBC mechanism as a separate kernel modules (named wbcFS?) similar with cacahfiles in Linux kernel, provide some common interface for local or networked file system.
