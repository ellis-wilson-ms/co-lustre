--- Commit Message ---
LU-12127 lfsck: skip orphan processing for namespace LFSCK

LFSCK can reconnect a recently-deleted orphan object back
into the normal namespace when it shouldn't. This can cause
access to the deleted data (potential security risk), and
sometimes cause an assertion if orphan is later deleted.

The commit 077570483e75e0610fd45149b926097547c434b8 skips
the orphan object during LFSCK scan. But what needs to be
skipped is only the namespace LFSCK logic for the orphan
object, but the layout LFSCK processing is still necessary
for the orphan object to verify the consistency between the
orphan MDT-object and related OST-object(s). This patch
just does that.

Not all files are removed from ORPHAN/ directory after MDS restart, only if the client(s) or processes holding the file open-unlinked do not participate in MDS recovery. Normally, all files in ORPHAN/ should be in use, unless clients fail or "abort_revovery" is used, since they will be deleted when the file is closed on the MDS (though some bugs exist that don't send the close to the MDS properly, or openlock caches the open handle on the client).

Since this is the expected case that files in ORPHAN/ are in use, they should definitely be in the OI file, so that they can be found with lookup-by-FID for revalidate, truncate, or other MDS RPCs.
