--- Commit Message ---
LU-11449 osc: send RPCs for sparse writes

If we check for full extents rather than dirty page count,
we will not send RPCs for small random writes until the
cache limit is hit.

The solution is to start sending RPCs once there are enough
dirty pages, regardless of whether or not they're
contiguous.

But we need to avoid splitting sequential writes from > 1
writer per stripe, so we have to check for greater than
max_pages_per_rpc.

The compromise chosen here is 4*max_pages_per_rpc, which
will work well for up to 4 simultaneous sequential writers
and still flush random writes fairly quickly in most
configs.

Hm.  The idea of something variable rather than fixed is appealing, but that one probably wouldn't work, and I haven't thought of another.

The key here is that this should be as small as possible (so we start sending RPCs in the random case), while being >= the number of CPUs doing writes to this stripe of this particular object.

This shouldn't be very many anyway, and if it is, they'll get bad performance for other reasons.  We could do 8 instead of 4 to create a bit more headroom, but then in the random case, we would only start RPCs after we had 128 MiB (assuming 16 MiB RPCs) of dirty data.

If there are many active CPUs, it's likely that they're either writing FPP, writing mostly (or entirely) to other stripes, or they're not doing I/O.

If we use num_active_cpus() with some upper limit (eg. 64?) then we would get up to 64MB of dirty pages on the client before an RPC was formed. I don't think that is so bad. This gives some opportunity for merging on the client, if it is a random-but-dense write workload, and allows some piecewise ordering of pages within the file. 

 > This shouldn't be very many anyway, and if it is, they'll get bad
 > performance for other reasons.  We could do 8 instead of 4 to
 > create a bit more headroom, but then in the random case, we would
 > only start RPCs after we had 128 MiB (assuming 16 MiB RPCs) of
 > dirty data.
 
Even with 16 MB RPC size, we cap the number of fragments in a single RPC to 256 (I think) for two reasons:
- limit the size of the RPC request itself, otherwise with 16MB RPCs we would have 4096 niobuf_remote structs in the request, which is 64KB
- add concurrency to the RPC processing. With random IO there is little chance for page merges on the client, but submitting more RPCs to the OSS for processing concurrently allows more threads/CPUs to be active in parallel.  

I guess the proof is in the pudding - did you get any chance to benchmark this before/after the patch?   It may not affect a many-client write bandwidth, but I suspect it would improve single-client write latency.  Something like what Ihara posted in LU-11164  should be run, maybe with fake_io so that you are measuring client performance instead of server performance.

--- Message (Patrick Farrell, 2018-10-01 15:31:30.000000000) ---
Patch Set 2:

I plan to get benchmark data soon and will put some in the commit.

I figured it's worth describing the scenario I'm trying to avoid.  Consider a *very* large sparse file and a system with 2 GiB max_dirty_mb.  That system will allow 2 GiB of dirty data to build up before sending any RPCs, then will send as much as max_rpcs_in_flight and the fragments limit you referenced allows.  (So in the worst case it would be as little as 1 MiB per RPC, 4096*256.)

Those RPCs will go out *very* fast since the data is already in cache.  Very nearly all at once.

Then the OSS/OST will process them, in parallel, and they will all complete around roughly the same time.  The client - which has just been waiting for this - will then refill the cache space this freed up, then do this "burst" of RPCs again when the limit is hit.

So that's the problem - We want RPCs in flight all the time.  Ideally, the client is constantly submitting RPCs and the server constantly has RPCs to write out.  (Obviously in this scenario something is the bottleneck, but the key is both going on at the same time, as much as possible.)

I've verified this behavior by looking at stats, etc, but I need to do benchmarks to verify the benefit.

The main improvement I'm interested in is to reduce the maximum number of fragments for random IO.  I don't think there is a benefit to packing 16k random 4KB pages into a single RPC.  This has several bad effects:
- consumes a huge amount of grant without it being returned quickly
- consumes a large number of journal credits, because 4KB random writes represent the worst-case allocation for files
- prevents merging of IOs at the server
