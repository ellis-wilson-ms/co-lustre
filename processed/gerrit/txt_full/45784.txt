LU-15340 llite: Delay dput in ll_dirty_page_discard_warn

Otherwise we can be final dput and need to wait for pages
to clear which is bad because this is called from ptlrpcd
that is not supposed to block esp. for network traffic as
it can cause livelocks if it happens to be needed to kill
the very same RPC we are waiting on.

Additionally pass in the inode from IO since the page
we are using might come from directio and that is
probably not even a valid inode.

Yes, a page that is referenced can be truncated.  Then truncate_inode_page() will call delete_from_page_cache() and the page->mapping will be set to NULL.
This requires having a lock on the page.  If you don't have a lock on the page, then you cannot trust ->mapping.

Is ll_dirty_page_discard_warn() called with a lock on the page.
If it is, then it shouldn't be possible for the bug to happen.
If it is not, them it shouldn't be doing anything with page->mapping->host.

As it only wants the inode for an error message, I would suggest trylock_page().  If that succeeds, then get the inode, find a dentry and use "%pD4" to print out the dentry.
If the trylock fails, or the inode is NULL, or the dentry cannot be found, just print a generic message.

I wnat to start with a quote of this comment to evict():

 * Free the inode passed in, removing it from the lists it is still connected
 * to. We remove any pages still attached to the inode and wait for any IO that
 * is still in progress before finally destroying the inode.

It strongly implies pages attached to inode do not preclude it from being evicted. Which is I guess natural otherwise how would any non-empty file inode would get evicted?

 > Yes, a page that is referenced can be truncated.  Then
 > truncate_inode_page() will call delete_from_page_cache() and the

Well, we don't have the page locked, we only have the pagewriteback set as we are in the writeout path.

we cannot lock the page as that creates lock inversion with the common "lock page -> wait page writeback" call sequence and we are the thread that would unset the page writeback bit.

So that leaves us with the choice of:

grab page->mapping
test it's not null
if not, work on mapping->host?

Sort of strange we never have seen a crash in this dirty discard function trying to dereference an already freed inode of NULL page->mapping

In fact we don't even need any of that, the only caller of it is vvp_vmpage_error() and it gets inode as one of its arguments so we can just as well pass that down to the ll_dirty_page_discard_warn()

But that leaves us the mystery of if there is inode up the stack - then either there's a reference so there's no final iput possible or we "own" the inode up the stack without a reference?
