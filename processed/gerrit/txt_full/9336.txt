LU-4942 at: per-export lock callback timeout

The lock callback timeout is calculated as an average per namespace.
This does not reflect individual client behavior.
Instead, we should calculate it on a per-export basis.

(9 comments)

It might also be possible with this patch to reduce ldlm_enqueue_min time somewhat (maybe 10s again), so that in the common case servers can get faster notification of client problems and resend more quickly.  We can't reduce it too much with the current implementation (originally it was 2s), since we don't want the server to be spamming the client and not give it a chance to reply to a request before it is timed out.

As discussed in b=3622, we would need more work to allow receiving the reply for any RPC sent with the same XID into the same reply buffer.  However, this patch and the previous one is a good start, and should avoid most of the problems seen in the past.

--- Message (Vitaly Fertman, 2014-08-21 10:12:58.000000000) ---
Patch Set 4:

Andreas, you got it wrong. ldlm_enqueue_min for the server means what is the minimal time we can cancel all the conflicts to satisfy this enqueue, what means not BL AST RPC timeout, but the whole BL AST cycle, BL AST+flushing client's data+cancel. and this is what we also set to lock callback timeout. if you set to 10sec, you get a lot of lock timeouts as this is not enough to cancel conflicts.
regarding the BL AST RPC, its timeout is set according its own AT from the reverse import.
in other words, ldm_enqueue_min should not be set to less than 100sec, at least for the server. regarding the client- we do not care much even if it is large.
