(7 comments)

Needs an llapi_fid2path_cached.3 man page.

That said, I guess this wasn't really what I was thinking for how we would accelerate fid2path lookups.

Firstly, I don't think it is useful to cache fid2path for regular files, since they are unlikely to be accessed repeatedly (or the caller could just cache this itself), and even if they were used repeatedly it isn't clear what to do if there are multiple links.

This will also repeatedly cache the whole string for each pathname, while I was thinking we would have a FID-to-NAME cache for each directory component of the path, where each component could be used to avoid the lookup of all of the common parents when generating the fid2path for thousands/millions of files within the leaf directories if we get the parent FID + one pathname for each file, in addition to the FID of the file itself.  Since that is already available at scan time, it would be more efficient to dump it to the output file rather than having to look it up again.

If this is *already* being used to cache the parent FID lookups, then I guess it is at least mostly doing what I expected, though maybe not quite in the same way.
