LU-19536 osc: enforce dirty limits for DIO writes

The DIO path (osc_queue_dio_pages) only checked server
grant availability before allowing a write, ignoring
both the per-client dirty page limit (max_dirty_mb) and
the global dirty page limit (obd_max_dirty_pages).
This meant DIO could pile up unlimited dirty pages per
OSC, bypassing memory pressure management.

Consolidate all three write-permission checks into
osc_reserve_grant(): server grant, per-OSC dirty limit
(cl_dirty_max_pages), and global dirty limit
(obd_max_dirty_pages).  This makes it the single gate
for "can I dirty these pages?" and ensures every write
path - buffered, sync, and DIO - enforces all limits.
All checks are done before any state is mutated, so
failure is a simple early return with no rollback.

Use osc_reserve_grant() in all three write paths:

 - osc_queue_dio_pages (DIO) - adds both missing dirty
   page checks
 - osc_queue_sync_pages (sync buffered IO) - uses the
   shared helper, fixing a pre-existing grant leak: the
   old code used "osc_reserve_grant() == 0 && dirty <
   max" so when grant succeeded but dirty failed, the &&
   short-circuit skipped into the else branch with grant
   moved to cl_reserved_grant but never consumed or
   returned to cl_avail_grant
 - osc_enter_cache_try (buffered IO) - simplified to a
   single osc_reserve_grant() call followed by
   osc_consume_write_grant(), replacing three exit
   points with manual rollback

When the dirty limit is exceeded, DIO falls back to
synchronous RPC submission (ci_parallel_dio = false),
same as when grant is exhausted.

Add test_398v to verify DIO respects max_dirty_mb by
writing 32MB with the limit set to 1MB and checking the
kernel debug log for the sync fallback message.
