--- Commit Message ---
LU-12017 ldlm: DoM truncate deadlock

setxattr takes inode lock and sends reint to MDS.
truncate takes MDS_INODELOCK_DOM lock and wants
to acquire inode lock.

MDS locks are for different bits
MDS_INODELOCK_UPDATE|MDS_INODELOCK_XATTR vs
MDS_INODELOCK_DOM but they blocks each other if
some blocking lock was present earlier.

If IBITS waiting lock has no conflicts with any lock in the
granted queue or any lock ahead in the waiting queue then
it can be granted.

Use separate waiting lists for each ibit to eliminate full
lr_waiting list scan.

Patch Set 4: Code-Review-1

(1 comment)

Patch needs more work - for the waiting queue the ldlm_inodebits_compat_queue() should check all needed lr_ibits_queueus but not lr_waiting. 

As for patch itself in general, it should do the job after all but it looks a bit overloaded and complicated with all these cycles, extra allocations, etc. - e.g. the lock processing can be called several times for the same lock if it has several bits set, which is also overhead. I prefer to go with Andreas approach and per-resource counters for all bits. That will be faster, no extra lock processing, no new allocations and it will simplify also lock enqueue, not only reprocess. 

Another concern about all of this - it is not deadlock resolver anyway, it hides deadlock situation making it pass in some cases, so I wouldn't close deadlock ticket until deadlock is resolved. This patch helps with deadlock only because second lock has no common bits, e.g. consider that:
1. in waiting queue we have DOM bit, and the same thread try to take XATTR lock after that, if there is other thread in the middle, blocked on DOM bit then XATTR will deadlock and this patch (as any other reprocessing approach) helps, but if lock in the middle has DOM|XATTR then deadlock is not resolvable. The real problem here is the second lock enqueue from the thread already having first lock busy, that is always source on potential deadlock
