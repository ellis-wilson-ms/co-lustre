LU-2850 kernel: 3.8 upstream kills daemonize()

kernel_thread() is a low level kernel function.
Instead of calling kernel_thread() and relying on daemonize()
to create new kernel threads, we should really switch to
kthread_run() and elimilate daemonize().

All callers are converted except for those in
lnet/klnds/gnilnd/gnilnd*, as requested by James.

Patch Set 2: (2 inline comments)

Jeff,

First thanks a lot for reviewing the patch.

As for signal handling semantics difference between kthread_run and kernel_thread(), as I audited all kernel_thread() callers in Lustre, they all call daemonize() the first thing in the child thread. Since daemonize() takes care of blocking and flushing all signals, all child threads will have all signal blocked after calling daemonize().

OTOH, it seems that Lustre never unlocks signals since I don't see any caller of sigdelsetmask() or sigprocmask() in Lustre kernel code. The only signal manipulation places (cfs_block_xxx functions) call sigaddsetmask to increase blocked signal coverage. cfs_restore_sigs() might be an exception, but the sigset passed in is always fetched from current thread, so it doesn't unblock any signals for daemonized()ed threads. So I don't think Lustre ever unblocks signals in daemonized kernel threads.

Therefore, I think we can just achieve the same effect by replacing kernel_thread()+daemonize() with kthread_run().

Does above explanation make sense to you?

Thanks.
