--- Commit Message ---
LU-8299 clio: correct arithmetic in cl_index

cl_index uses a right shift for division, but takes an
signed argument (type loff_t).  If the argument is ever
negative, a right shift doesn't do division.

Make cl_index take pgoff_t as its argument instead.
This is an unsigned type of the same size (and it's what it
returns anyway).

Hm.  That's not what other filesystems do, though that doesn't mean we couldn't.  The address error (the error returned by other file systems and by Lustre with this fix) comes from this check in filemap_fault (called from vvp_io_kernel_fault):
        pgoff_t offset = vmf->pgoff;
        struct page *page;
        pgoff_t size;
        int ret = 0;

        size = (i_size_read(inode) + PAGE_CACHE_SIZE - 1) >> PAGE_CACHE_SHIFT;
        if (offset >= size)
                return VM_FAULT_SIGBUS;

And as you can see, it uses pgoff_t to represent the number of bytes in a file.  So it has the same problem you're describing - On a 32 bit platform, it would limit file size to 2^20 pages.  So maybe that limit is already baked in elsewhere?

--- Message (Andreas Dilger, 2016-06-20 20:34:44.000000000) ---
Patch Set 1:

In the code you quoted:

        pgoff_t offset = vmf->pgoff;
        struct page *page;
        pgoff_t size;
        int ret = 0;

        size = (i_size_read(inode) + PAGE_CACHE_SIZE - 1) >> PAGE_CACHE_SHIFT;
        if (offset >= size)
                return VM_FAULT_SIGBUS;

The "size" calculation is done in i_size_read() units (loff_t) rounded up to the end of the page, and _then_ downshifted to store in "pgoff_t size", so "size" never contains the size in bytes, only the page index.
