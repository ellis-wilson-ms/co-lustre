LU-18924 hsm: fix the crash cause by huge max_requests

Some variables in function mdt_coordinator() should be unsigned
type to avoid memory allocation crash caused by huge parameter
mdt.*.hsm.max_requests.

To avoid such a failure earlier, the sum of MDT max_requests is
limited to 1/8 of total memory.
If it is bigger than this limit, it will be recalculated by this
limit and a useful warning message with memory information and
the limit will be printed.

Also, sanity-hsm.sh test_40 is modified to verify this patch and
stack_trap is added to test_50 and test_100 to restore the default
max_requests value correctly.

Parameter mdt.*.hsm.max_requests does mean the scan request on the MDT, so it should reserve 5KB for each action list.

It seems to be used to determine the number of scan requests to allocate in mdt_coordinator(), but it is used as the number of individual HSM requests in mdt_cdt_waiting_cb(), where cdt_max_requests, cdt_request_count and hsm_action_count are all used for units of HSM requests, i.e. struct hsm_action_item; that function just exits early if cdt_request_count >= cdt_max_requests, so max_requests is used for limiting the maximum number of individual HSM requests on the MDT. I think allocating a scan request (which can point to one hsm_action_list) for each HSM request/hsm_action_item might be done as hsm_action_item includes a variable length hai_data member at the end, for carrying a generic amount of opaque data that can be interpreted by either the copytool or possibly other components in the HSM pipeline, so it might be a way of allocating enough hsm_scan_data structures for the extreme case where the opaque data component is large enough that it only allows each hsm_action_list to hold one hsm_action_item?

I don't know how widely used the opaque data component might be in the field, or how large the data size might be, fwiw; iirc, with an empty data field, each MDS_HSM_REQUEST RPC can hold ~52 hsm_action_item structs, but maybe I'm mistaken. Imho, we could either use the current calculation in the latest patch set, especially since the max_requests allowed seem sufficient for everyone's current use cases, even though the current mechanism does seem to allocate some memory unnecessarily, or make a reasonable assumption about the expected maximum data component size and multiply the number of max_requests, without increasing the risk of running OOM, e.g. a 512 byte data component should allow each hsm_action_list/MDS_HSM_REQUEST RPC to hold at least n number of hsm_action_item structs, so we could multiply max_requests by n in max_requests_store(), but only allocate max_requests/n scan requests in mdt_coordinator()?

Patch Set 15:

> my concern is that the attempt to allocate a single 128G segment in kernel might still cause OOM or other undesirable effects.

This is not a single segment of 128GB, the objects saved in the RAM should not be bigger than 432 bytes in average. However, there is still 1 big array of "struct hsm_scan_request" to build the requests.
So with 1TB of RAM, this max_requests can be set to: (1<<40)/8/(432+40) = 291184223 = 291M
This means that the array size (hsm_scan_data.hsd_request) is: (291184223 * 40)>>30 = 10GB

At the CEA we used a max_requests value between 100-200. But HPE seems to test HSM with max_requests set to 1M (from previous patches). 

So with 2 MDT mounted on the same node, the minimum RAM needed to set max_requests to 1M is:
(472 * 1000000 * 8 * 2)>>20 = 7202MB (8GB)

With RAM/16:
(472 * 1000000 * 16 * 2)>>20 = 14404MB (16GB)

I am not against enforcing max_requests with max_ram/16, if you think it is necessary.
