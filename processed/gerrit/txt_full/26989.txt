LU-9384 ldiskfs: handle expand_extra_isize_ea properly

With project quota introduced on RHEL7, the inode's i_extra_isize
is changed from 28 to 32 for project ID information. Such on-disk
layout changes will trigger ldiskfs_expand_extra_isize_ea() when
modifies the existing object that is upgarded from Lustre-2.9 or
older release.

In theory, the ldiskfs_expand_extra_isize_ea() only needs to move
the inode inline EA with 4-bytes (32 - 28) afterward if there is
enough inode inline space; otherwise, it needs to move some EA(s)
from the inode inline space to another separated EA block that is
referenced via inode::i_file_acl. If the inode::i_file_acl is NULL,
it will allocate a new block for that. Unfortunately, the old logic
of ldiskfs_expand_extra_isize_ea() handles the input paramter
@new_extra_isize (that is 32) as the diff that should be 4. Means
that it tries to move the inode inline EA with 32-bytes afterward.
But there is NOT enough inode inline space for that. So it allocate
inode::i_file_acl block. It increases the blocks used by the inode.
That is why the test_32b found the 'list' outputs are different
after upgrading from Lustre-2.9 or older release.

This patch fixes such wrong logic.
