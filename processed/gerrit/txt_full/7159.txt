--- Commit Message ---
LU-3647 hsm: do not verify data_version for restore

In ll_ioc_copy_end(), it used to verify data_version for restore.
This is not required because layout lock was already held by
restore request on the MDT side so that we can't verify whether
layout is stale. Therefore ll_data_version() uses stale layout to
call getattr to OFD which causes ENOENT, then this error code will
be transferred to MDT to result in restore failure.

(1 inline comment)

Could you explain how you reproduce this error?

ll_data_version(), on restore, will and need to be done on the volatile file. The Layout Lock is held on the regular file, which has a RELEASE layout. There is no way getattr is sent to OFD, has the OST IDs were discarded when the file was released.

--- Message (Jinshan Xiong, 2013-07-29 17:33:38.000000000) ---
Patch Set 1: (1 inline comment)

I used two clients: one as an agent and another one as normal client, and on the normal client, I just release and restore one file. On the 2nd try, it hung at restoring the file.

From what I have seen, the agent client served the restore request and copied file back to lustre, then it called copy_end ioctl to send progress REQ back to MDT. However, it was returned with ENOENT when trying to get data_version(this is because ll_data_version() doesn't refresh layout so it used stale layout to send getattr REQ to OFD), then coordinator received this error and refused to release layout.

Taking a step back, even ll_data_version() were implemented correctly by refreshing layout, it wouldn't work also because the layout was already taken by restoring REQ, so there is no way for it to get one.
