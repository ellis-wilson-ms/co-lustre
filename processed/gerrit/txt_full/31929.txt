LU-10887 mdt: ldlm lock should not pin object

For performance, the MDT needs to cache remote ROOT xattr
with holding xattr ldlm lock. Originally, such xattr ldlm
lock directly back references the object via l_ast_data.
That pins the object in RAM. Unfortunately, such reference
may prevent the ROOT object being released when umount MDT:

kernel: LustreError: 49321:0:(osp_dev.c:1276:osp_device_free())
header@ffff885e724737b0[0x0, 1, [0x200000007:0x1:0x0] hash exist]{
kernel: LustreError: 49321:0:(osp_dev.c:1276:osp_device_free())
....mdt@ffff885e72473800mdt-object@ffff885e724737b0( , writecount=0)

This patch drops such back reference. Instead, we can locate
the object with its FID when need to invalid the cache.

The refcount leakage should have been fixed by https://review.whamcloud.com/#/c/31143/.

Actually in the first implementation of cached remote lock, it doesn't pin object, but Alex thought this is too efficient to lookup by fid each time, and though cached remote lock is only  used by ROOT xattr lock now, it's a general infrastructure, and can be used in other places if remote lock is frequently requested.

In summary, it's not an issue that ldlm lock pins object, but it should guarantee there is no race in object release.
