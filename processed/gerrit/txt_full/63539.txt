--- Commit Message ---
LU-19822 lnet: interfaces can be manually disabled

Implement lnetctl disable/enable to allow admins to selectively
block traffic on LNet interfaces. Currently, the only way to
manually disable an LNet interface is to delete it. An lnetctl
tunable would give admins a network agnostic method to
temporarily block traffic to a subset of clients or block
certain routes.

This patch also implements the needed kernel infrastructure. When
an interface is disabled, sends are routed over the other available
interfaces (if available). recv are dropped outright.

## Summary

**The Fix:**
I fixed the compile error by modifying the `strncpy` call in the `lustre_lnet_set_ni_state` function at line 2840 of `lnet/utils/lnetconfig/liblnetconfig.c`.

**What was changed:**
- Changed `strncpy(data.cfg_config_u.cfg_net.net_intf, intf, LNET_MAX_STR_LEN)` to `strncpy(data.cfg_config_u.cfg_net.net_intf, intf, LNET_MAX_STR_LEN - 1)`
- Added explicit null-termination: `data.cfg_config_u.cfg_net.net_intf[LNET_MAX_STR_LEN - 1] = '\0'`

**Rationale:**
The compiler error `-Werror=stringop-truncation` was triggered because `strncpy` was called with a size parameter (`LNET_MAX_STR_LEN`) that equals the destination buffer size. This is problematic because:

1. If the source string (`intf`) is >= `LNET_MAX_STR_LEN` bytes, `strncpy` will copy exactly `LNET_MAX_STR_LEN` bytes without adding a null terminator
2. This leaves the destination buffer without proper null-termination, which can lead to buffer overruns when the string is later used

The fix ensures:
1. At most `LNET_MAX_STR_LEN - 1` characters are copied from the source
2. The last byte of the buffer is explicitly set to `'\0'`, guaranteeing null-termination
3. This prevents potential buffer overflow issues and satisfies the compiler's stringop-truncation warning

This is a standard defensive programming practice when using `strncpy` to ensure strings are always properly null-terminated.
