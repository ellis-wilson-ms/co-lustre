LU-14095 ssk: default rounds of Miller-Rabin for DH_check

OpenSSL 1.1.1c increased the number of rounds used for Miller-Rabin
testing of the prime provided as input parameter to DH_check(). This
makes the check roughly x10 longer, and can lead to request timeouts
when an SSK flavor is being used.

Instead, use a dynamic number of rounds based on the speed of the
check, evaluated when the lsvcgssd daemon starts. If DH_check()
runtime is fine, just use it instead of our own check.

--- Message (Sebastien Buisson, 2020-11-18 15:50:53.000000000) ---
Patch Set 1:

> Would it make sense to consider increasing the number of rounds as
 > is done with OpenSSL 1.1.1c, but add an internal check for how long
 > this is taking, and exit early if it is too slow?  How long is "too
 > long"?
 > 
 > Basing the early exit on the amount of time taken would still allow
 > doing the full new number of rounds if the CPU is fast, but at
 > least run the old number of rounds on a slower CPU/VM systems.
 > 
 > That shouldn't be _less_ safe than the old code, but I'm assuming
 > OpenSSL had a reason for increasing the number of rounds, so we
 > should try and implement that if possible.

I am not sure how long is "too long", but for instance it is OK if DH_check takes ~30 ms, but not OK if DH_check takes ~600 ms. I will push a test patch to see how long it can take before it starts timing out RPCs.

The reason they decided to increase the number of rounds in OpenSSL is that the composite number being tested is not guaranteed to be randomly generated. In this case, the default number of rounds, based on the number size, might not be enough to claim with a reasonable error rate that it is actually a prime. But in our case, we request users to generate the SSK key with a real source of entropy. So we should not be exposed to this weakness, unless SSK keys have been generated incorrectly.

In order to avoid copying too much code from the OpenSSL sources, instead of adding an internal time check in BN_is_prime_ex(), we could run a speed test when the lsvcgssd daemon is launched, and infer the number of rounds that is going to be used for all later calls to BN_is_prime_ex().

I am a little bit concerned about copying in the Lustre sources some code from OpenSSL, because of their specific license. Is the copyright I added here in sk_utils.c enough?
