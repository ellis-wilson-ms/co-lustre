--- Message (Andreas Dilger, 2018-10-18 19:41:26.000000000) ---
Patch Set 1: Code-Review+1

> rename may leave file to be remote, and considering the
 > complexity, we can just add DoM migration support

I don't think changing the error return code to -EXDEV is much work...

I guess the question is what happens if someone tries to migrate a directory with some DoM files?  Does the whole migration fail because of the DoM inode, or does the migration succeed but DoM files are left remote?  In that case, doing the equivalent of:

    TMPFILE=$(mktemp .$domfile.tmp.XXXXXX)
    cp "$domfile" "$TMPFILE" && mv "$TMPFILE" "$domfile"

would move both the DoM file data and the inode, which is the equivalent of migration.  One minor problem us that the "mktemp" might create the filename on the wrong MDT, so we might want to have an "lfs mktemp -i" helper that creates a temporary file on a specific MDT (maybe just a loop around "mkstemp()" until it generates a name on the correct MDT?).
