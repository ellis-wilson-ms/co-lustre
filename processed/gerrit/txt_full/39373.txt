LU-10391 lnet: extended nids.

This is a tech-preview for high-level review.  It is combination of
about 40 individual patches, presented as a single patch to allow
that review.

A new extended nid is added which supports a 128bit address field.
Much of lnet and socklnd is changed to make use of it, and new
protocol versions are added to contain it.

lnds other than socklnd have had minimal changes, and in particular
there has been no attempt yet to allow them to communicate with a peer
that uses an extended nid.

ioctls for setting up interface and routing have not been changed yet,
so it is not yet possible to configure the new extended nids.

Minimal changes have been make to lustre filesystem code - it still
uses legacy nids.

This patch is presented to get feedback on the approach and the
naming.  I didn't give a lot of thought to names, I just added 'x' for
'extended' in various places.

This patch set passes sanity only using IPv4 addresses, so I don't
think I've broken much.  However I'm not asking for any correctness
review at this stage.

I particularly want to know if this is a senisble way forward, and if
I should start pushing the patches...
I'd really like to be able to push them as a coherent branch, other
than individually....

Thanks.

(28 comments)

So I had a pass over the changes, and they _seem_ reasonable as a starting point (IANALNetExpert).

It would definitely be nice to try and structure the patch series to minimize the number of mechanical changes to the code, if that is possible, to minimize conflicts.

I definitely think the best approach is converting all of the internal structures to use lnet_nid_v6, and only use lnet_nid_v4 explicitly in the "edge" cases where it is needed for protocol compatibility.  This could be done in a few stages:
- adding accessor functions (lnet_nid_same(), lnet_nid_is_lo0(), lnet_nid_is_any(), etc.)
- add struct lnet_nid_v4 and point lnet_nid_t at it, and do not treat it like an int
- use lnet_nid_v4 explicitly in the places where explicitly needed (LNDs, config llog records, APIs)
- add struct lnet_nid_v6 and point lnet_nid_t at it so that it is used everywhere internally
- add new LND types for lnet_nid_v6 (e.g. 256 + LND)
- and start using lnet_nid_v6 where explicitly needed
-- handling IPv6 connections in socklnd, o2iblnd
-- handling lnet_nid_v6 config records in Lustre
