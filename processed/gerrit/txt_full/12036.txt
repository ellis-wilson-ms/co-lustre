LU-5141 hsm: Only regular files should be archived

It is currently possible to ask lfs to hsm_archive a directory,
although this doesn't appear to make any sense, and the posix
copytool rejects it.

Ideally this should be caught early and not being forwarded to
the copytool at all.

So adding regular file check in
lustre/utils/lfs.c
which will report error if the file provided for hsm_archive
is not a regular file

Patch Set 3:

I was discussing this with a few folks at SuperComputing, and actually came up with a very good use case for archiving a directory. If the copytool would take archiving a directory to mean "create a tarball of the directory and its subdirectories, and then archive the tarball" this would allow two significant improvements over the current functionality. 

It would provide an efficient mechanism for archiving many small files. It would also allow removing archived inodes from the MDT filesystem if the copytool deleted the directory and all the inodes therein.

Currently, there is no benefit to archiving a small file except for a backup, and even so this places a large overhead on the archive to handle small files efficiently. With this mechanism to archive a directory into a tarball, it can store many small files (including HSM stub inodes that were previously released after archiving) into a single large file for efficient storage in the archive. It also avoids the need for the MDS to continue growing to cover all the files in the fikesystem and the archive.
