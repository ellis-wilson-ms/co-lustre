LU-12640 obdclass: percpu env race

Following race might happen:

CPU0 thread0
|->lu_env_remove()
  |->if (lu_env_percpu[i].lep_env == env) {
	|->LASSERT(lu_env_percpu[i].lep_task == task);
	|->lu_env_percpu[i].lep_task = NULL;
	  |->switch to thread1
	    |->lu_env_find()
		lei = rhashtable_lookup_fast(&lu_env_rhash, &task,
				lu_env_rhash_params);
		if (lei) {
			env = lei->lei_env;
			lu_env_percpu[0].lep_task = current;
			lu_env_percpu[0].lep_env = env;
		}
		put_cpu();
		 |->switch to thread0
		   |->lu_env_percpu[0].lep_env = NULL;
			after this we got:
			lu_env_percpu[0].lep_task = thread1
			lu_env_percpu[0].le_env = NULL;
			|->switch to thread1 call lu_env_find()
			  |->if (lu_env_percpu[i].lep_task == current) {
				env = lu_env_percpu[i].lep_env;
				put_cpu();
                		LASSERT(env); --------------> here

And another case is:
CPU0 thread0
|->lu_env_find()
  |->lu_env_percpu[0].lep_task = thread0;
     lu_env_percpu[0].lep_env = env0;
	CPU1 thread0 trying to access lu_env_percpu[0]
         |->lu_env_remove()
	   |->if (lu_env_percpu[0].lep_env == env) {
	     |->lu_env_percpu[i].lep_task = NULL;
	CPU0 thread1 call lu_env_find()
	 |->lu_env_percpu[0].lep_task = thread1;
	 |->lu_env_percpu[0].lep_env = env1;
	   |->CPU1 thread 0 continue to cleanup
	     |->lu_env_percpu[0].lep_env = NULL
		Got a state again:
		   |->lu_env_percpu[0].lep_task = thread1
		   |->lu_env_percpu[0].le_env = NULL;

So the problem is that lu_env_find() could race with lu_env_remove()
and we get a state that lep_env = NULL while lep_task not NULL.

Take a spinlock to protect region seems too heavy and hurts performance,
the lightway just check whether we get a NULL pointer before return.

About timeslice based preemption:
This is true for userspace, but not the kernel, unless you've got a special config option (I think it's PREEMPT_RT?).  Without preempt enabled, kernel processes cannot be preempted (interrupts are a special case, but they still run on the same CPU, they just interrupt the current process - not switch to another).  Kernel processes can only switch CPUs if they choose to go to sleep.

Consider cond_resched() - It's a process saying "I can be scheduled if needed".  That's specifically necessary because they would not be scheduled if they did not *volunteer* to be scheduled.

This is also why sleeping holding a spinlock isn't safe (in kernel).   Your process sleeps, and then if every CPU on the system tries to get that spinlock, your process will never be woken up - because those processes keep spinning forever.
