(5 comments)

I'm generally in favor of this, with a few comments/questions online. It seems like some of the headers don't belong in userspace at all (in particular socklnd.h), and in other places we have includes that don't seem to be needed. 

The other concern is what impact this has on other tools (e.g. DVS) that may include these headers from userspace? I'm fine to bring this inline with the upstream kernel, but it also makes sense to include compact headers in our tree so that tools eventually get a #warning if they include the old header but can continue to build, and have a chance to transition to the new layout.

--- Message (James Simmons, 2017-07-20 23:57:35.000000000) ---
Patch Set 3:

(5 comments)

I doubt this will have a impact on userland since lnet headers were never installed on any nodes before. In other words /usr/include/lnet has never existed. Since this has been the case no user land utility or application has been able to program directly the lnet interface. At our center we have always used the lctl/lnetctl tool or abused /proc for lnet. 

As for DVS that is a special kernel module that most likely has to be pointed to the kernel lnet headers to build against. I add the Cray guys so they can respond in this case. If we change the location of the headers it wouldn't break DVS but Cray will need to update the configure testing to see where the headers are located.
