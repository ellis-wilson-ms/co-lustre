Patch Set 1: Code-Review-1

Sorry, related rule/logic for protecting i_size is not clear and inconsistent.

osd_inode_setattr() => i_size_write() may be triggered by ATTR_SET RPC, it does not take the inode->i_lock. osd_ldiskfs_write_record() => i_size_write() may be triggered internally for non-RPC operation, but it takes inode->i_lock. osd_ldiskfs_writelink() => i_size_write() is for creating symbol file, also can be regarded as client sponsored operation. So it seems that taking inode->i_lock is not related whether with from the client side or not?

On the other hand, it is hack way to take different locks to protect the same target (i_size) based on human knowledge from outside of OSD. It may works, but it is not convenient for others to do further development. And if someday the logic for the caller changed, it is possible that forget to adjust the lock protection for i_size_write().

So I would suggest to unify the logic, for example:

int osd_size_write(struct inode *inode, loff_t size, bool append)
{
    int rc = 1;

    spin_lock(&inode->i_lock);
    if (!append || size > i_size_read(inode)) {
            LDISKFS_I(inode)->i_disksize = size;
            i_size_write(inode, size);
            rc = 0;
    }
    spin_unlock(&inode->i_lock);

    return rc;
}
