--- Commit Message ---
LU-14976 nrs: change nrs policies at run time

This patch take extra references on policy to avoid stop a NRS policy
with pending/queued request in it.

It uses a new refcount_t "pol_start_ref" for this purpose to keep
track of policy usage in started state. It enables to safely stop a
policy without "nrs_lock" and avoids to sleep in the spinlock.

It adds a wait queue field "pol_wq" in "struct ptlrpc_nrs_policy" to
wait all queued request in a stopping policy to be drained when
restarting policy with a different argument.

Add test sanityn 77r for this use case.

> ugh, this is a crash galore in test 77r in boilpot. I don't think
 > it ever hit this hard with all the sleeping under spinlock in prior
 > cases, but now I get a crash every 10 minutes

Oleg, I think this patch is only adding a refcount on the NRS policy to avoid it being removed while in use, which would *actually* crash the thread. The stacks you referenced are NMI watchdog timeouts because the cleanup is holding a spinlock while calling a sleeping function. The new test_77r is triggering this use case directly, but if that test was run without the patch the thread would outright crash rather than have threads stuck on a spinlock for a few seconds. So the patch is fixing a real world problem, but causing failures during testing is not good either as it hides other issues. 

That said, I think the issue can be fixed relatively easily.  The refcount could be changed to a kernel refcount_t, and policy start should hold one refcount that is dropped only when it is stopped, so that the refcount will never drop to zero during use (that is not allowed with a refcount_t). That should avoid the need for a spinlock to protect the refcount (it is atomic) so that the policy stop does not call a sleeping function while holding a spinlock.
