LU-17431 nodemap: fileset inheritance for dynamic nodemap

Filesets for a dynamic nodemap are inherited from the parent nodemap.
It is possible to define a primary or alternate fileset specific to
the dynamic nodemap, but only if it is a subdirectory of the parent
nodemap's filesets, or if the parent has no filesets defined.
As we do not want to leave dynamic nodemaps without any primary
fileset defined, the only way currently available is to use
'lctl nodemap_set_fileset'. 'lctl nodemap_fileset_add/del' cannot be
used for the primary fileset of a dynamic nodemap.

Add sanity-sec test_72d to exercise this feature.

Apologies, Sébastien, I think we need to reconsider how to handle adding filesets to child nodemaps. Currently, there are some contradictions when adding filesets to child nodemaps, and it is possible to bypass the fileset inheritance rules applied to some "fileset_add" operations. 

For example, if the parent has only a prim fileset '/primary', the child is restricted to only setting a primary fileset that is identical to or hierarchically under it. However, the child can simply add an alternate fileset that bypasses the primary fileset of the parent, like '/alt1', and access other parts of the namespace that way. The same is true in the opposite case: When the parent only has alt filesets, the child is restricted to setting alt filesets depending on the parent's alt filesets, but the child can set any primary fileset to escape the restrictions of all alt filesets on the parent. 

Given that child nodemaps are generally restricted by the privileges of the parent, I think it would be preferable not to distinguish between fileset types and to only allow the child to create a fileset if it does not escape _any_ fileset on the parent. This would probably also simplify `check_fileset_add_vs_parent()`. 

I realize that this is the opposite of how alt filesets work on normal nodemaps, where they allow access to the namespace outside of the primary fileset. Nevertheless, I don't think we should allow this on child nodemaps because they should only be able to operate within the restrictions of the parent, right?

(The delete case should still be fine)
