--- Message (Andreas Dilger, 2019-06-19 05:48:45.000000000) ---
Patch Set 3: Code-Review-1

(1 comment)

Firstly, there needs to be a second patch in this series that actually removes the cfs_get_random_bytes() function, or it will be dead code. 

Secondly, the distinction between the two functions is not (or at least was not) meaningless.  The problem is that (re)booting 10,000 or 20,000 diskless nodes at exactly the same time may cause the initial RNG state of two or more nodes to be the same, and result in the client UUIDs and/or initial RPC XID to be the same. This used to be seen on Cray Catamount systems, though is less likely on nodes that have high-res clocks and interrupts (which Catamount did not have).  Hence mixing the NID(s) into the PRNG seed.

This can be done better today by adding the client NID(s) into the kernel PRNG entropy pool at startup, and then using the kernel interface. This was done in Neil's original patch series, the rest of which is missing here, and is needed before this patch can land:

  https://lore.kernel.org/patchwork/cover/864255/
