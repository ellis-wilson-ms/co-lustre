--- Message (Timothy Day, 2024-05-18 21:33:25.000000000) ---
Patch Set 2:

> Patch Set 2:
> 
> Why do you say that the LASSERT() checks are not meaningful users of this API, and what is the benefit to remove this method?

These locking LASSERT() checks (and a lot of other checks, I think) are really just development aids that never get turned off i.e. if someone adds a new caller, they use the locking correctly. If the testing exercises each caller at least once and doesn't crash, this should never crash in a real usage.

Having all of these debugging LASSERT() enabled all of the time leads to issues like https://review.whamcloud.com/c/fs/lustre-release/+/55053 that can kill performance. While I don't claim this patch has performance improvements, it's unfortunate that we have no way to differentiate between these development oriented LASSERT() and genuine run time invariant checks.

I suppose it would be better if these development oriented LASSERT() only crashed during testing -  the kernel has something like panic_on_warn. Then in real builds, they could be downgraded to warnings (or just compiled away all together).

I know there exists some option to compile away all custom Lustre debugging. But that's too heavy handed. Maybe we could have a LASSERT_WARN()? With similar semantics as WARN_ON()?

I wouldn't mistake the one case of LASSERT in the page handling hot path as a general performance issue. That code path has been optimized by Patrick and is called millions of times per second to get GB/s rates, so every cycle makes a visible difference. That is not necessarily true of other places. 

I agree that some of the code paths are more for development than runtime, but these LASSERTs can also catch strange race conditions and data structure corruption at runtime. 

There is also the issue of having a "test" build vs. a "production" build, and if we are not testing what is actually being run in production then it could also let bugs escape.
