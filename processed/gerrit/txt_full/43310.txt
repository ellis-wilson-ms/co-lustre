--- Message (Andreas Dilger, 2021-04-15 01:04:25.000000000) ---
Patch Set 1:

The promotion from int to __u64 at compile time is not the problem.  The main issue is that since liblustreapi is a shared library, this would cause an ABI change. That may cause either a runtime failure for applications using this function (eg. copytools or applications that use group lock), or worse - unpredictable results based on whatever garbage is on the stack above the "int" because the kernel is trying to access this as a "__u64".

It is better/safer to keep the current API/ABI for the old code, and add an llapi_group_lock64() function that does it properly, and use the new interface in the tools if needed.  At some point in the future (eg. 3 years) deprecate this interface and new compiles should be redirected to use the 64-bit interface.

--- Message (Andreas Dilger, 2021-04-15 02:58:31.000000000) ---
Patch Set 2: Code-Review-1

I didn't notice that this ioctl is taking "unsigned long" for the argument.  I was looking at ldlm_extent which has "__u64 gid".  That is a bit messy, because "long" == "int" on 32-bit platforms.  

It would be better if "llapi_group_lock64()" actually took a __u64 as the argument to avoid confusion.  We probably don't really need to care about 32-bit CPUs, so or now it would be enough if there is a check like:

        /* check LL_IOC_GROUP_LOCK is OK for llapi_group_lock64() */
        BUILD_BUG_ON(sizeof(long) == sizeof(__u64));

so that this is detected if it ever matters, which is probably never.  We might need to add a definition of BUILD_BUG_ON() to userspace.
