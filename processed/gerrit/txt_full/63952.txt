LU-12669 lov: continue lock enqueue on EC read stripe failure

When a sub-lock enqueue fails during an EC read
(CIT_EC_RD), the lock loop would break immediately,
propagating the error and aborting the entire read.
This is wrong for EC because recovery can tolerate
missing stripes - it only needs to know which ones
failed so it can reconstruct their data from parity.

Change the failure path to set rc = 0 and continue
the loop, so all remaining sub-locks are enqueued.
The failed stripe is already marked LSS_READ_ERR,
which lov_io_ec_rd_start() uses to identify stripes
needing parity reconstruction.

Without this fix, any single stripe lock failure
causes the EC read to fail even when enough parity
exists to recover the data.

Required for tests 42a through 46c: without
this fix, cl_io_lock returns -EIO when any stripe
fails lock enqueue, aborting before EC recovery
can start.
