--- Commit Message ---
LU-4664 clio: consume grant for sync write

Sync write should consume grant if available, otherwise,
it could fail for ENOSPC when there is still available
space reserved by grant.

Additionally, direct i/o writes are sync, and they receive
more grant from the server without consuming grant.  The
result is that direct i/o writes accumulate grant on the
client without ever using it.

Fixing grants for sync writes fixes this as well.

This problem has been discussed several times, and I still don't think this patch should be landed.

1. sync write doesn't need to consume grants at all because there is a process waiting for the writing results;
2. consuming write grants will interfere with async write. Usually direct IO is used to write huge amount of data, now it will consume all the grants on the client side, this is unfair to async write. Also, since sync write RPC can be mixed with granted and non-granted pages, this will make the situation worse;
3. I understand the intention of this patch - but I tend to think this method seems to be incorrect to solve the problem. The correct solution is to make grant shrink working properly. IMHO, It's not terribly wrong when the OST starts to deny sync write RPC if it has been really full, as long as the writer can be notified.

--- Message (Andreas Dilger, 2017-03-14 04:18:18.000000000) ---
Patch Set 12:

> This problem has been discussed several times, and I still don't
> think this patch should be landed.
>
> 1. sync write doesn't need to consume grants at all because there
> is a process waiting for the writing results;

If the OST has granted out all of the available space to clients (including the local client) then sync writes will fail with ENOSPC even though there is unused granted space available for that client.

> 2. consuming write grants will interfere with async write. Usually
> direct IO is used to write huge amount of data, now it will consume
> all the grants on the client side, this is unfair to async write.

This will not cause a problem if sync writes also return grants to the client (enough to cover the full RPC size), if there is space available on the OST.

> Also, since sync write RPC can be mixed with granted and non-granted
> pages, this will make the situation worse;

I'm not sure I understand.  If sync writes return grant in the same manner as async writes, then the client will not run out of grant space for async writes.  It won't be any different than two async writers on the same client.

> 3. I understand the intention of this patch - but I tend to think
> this method seems to be incorrect to solve the problem. The correct
> solution is to make grant shrink working properly.

Sure, that would also be great.  However, I don't think the two solutions are incompatible.  Grant shrink will help release *old* grants on clients not doing any writing, but it will not release recent grants allocated to the (local) client, so the sync write would still get ENOSPC before neecessary due to the clients holding grants.

> IMHO, It's not terribly wrong when the OST starts to deny sync
> write RPC if it has been really full, as long as the writer can
> be notified.

The way the writer is notified is "ENOSPC", even though there is free space in the filesystem.

One option to consider for grant shrink is to use a fake DLM lock to hold grants on each client, and when the OST needs to revoke grant it can send a callback on the grant lock and clients not actively writing can return their grant to the OST.  This is more dynamic than just grant shrink due to timeout, but also more complex to implement.
