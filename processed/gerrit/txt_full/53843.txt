LU-17476 lnet: prefer to use bits only to match ME

In some cases, it has been observed that a reply will arrive
at the portal with the correct match bits, but is dropped by
lnet_parse_put().  This appears to happen with LNet Multi-Rail
peers, each having two separate NIDs.

If a reply arrives with matchbits available and matching, but
the NIDs don't match, confirm the match if the NIDs are found
to belong to the same peer.  This will only happen in cases
where the reply would be dropped entirely, causing hundreds of
seconds of delay until the RPC is resent, so the extra overhead
of checking for a peer match before dropping the request is
only in the error path and minimal compared to the alternative.

Add CFS_FAIL_CHECK() for exercising the match NIDs code.

That is in a hot codepath, but CFS_FAIL_CHECK() is marked unlikely()
and this check is in the error case and _should_ only be hit when
the message would have been dropped anyway, so it seems unlikely to
impact performance in any meaningful way.

At this point I see this as a debug patch, not a fix candidate. According to current design, primary NID should be used for message-to-MD matching. If there is a case when the bits match, but non-primary NID is used, then the bug is in the logic which allows this to happen, not in the logic which is doing the matching.

--- Message (Andreas Dilger, 2024-01-27 21:27:22.000000000) ---

 > According to current design, primary NID should be used for

 > non-primary NID is used, then the bug is in the logic which allows

I agree with you on an intellectual level, and definitely we should identify and fix what is causing the NID mismatch to happen (to avoid the overhead of doing the peer match).

However, on a practical level there will always be bugs in the code, and possibly dynamic changes at runtime which mean the NIDs are mismatched.  As such, I would prefer to land this patch which prints a clear console error in such cases but lets the filesystem continue to be usable, rather than stand on principle and make a filesystem be unusable until we can develop and install a code fix for whatever the problem is.
