--- Commit Message ---
LU-3590 lfsck: repair MDT-object with dangling reference

If the OST-object referenced by the MDT-object is lost, then the
LFSCK needs to recreate the OST-object with the specified FID and
initialize it with the given parent MDT-object FID and owner attr.
Although the new created OST-object is initialized, the SUID+SGID
mode will be kept, which will be dropped by the first modification
RPC, like write/punch/setattr. Then we can distinguish whether the
recreate OST-object has been modified or not.

Patch Set 23:

I tend to agree with Alex on the discussion about LFSCK vs. recovery. I think it makes sense to have LFSCK stop processing during recovery, and only flag new errors after recovery for that target has completed.

This allows the normal recovery mechanisms to complete (all of which are I tended to finish quickly), and only after this point should LFSCK trigger because of inconsistencies. This would be possible by only flagging any errors on the OSD during recovery without starting LFSCK, but then re-verifying these inconsistencies after recovery is done. 

While it is true that there may still be a need for LFSCK to detect and fix inconsistencies after recovery is finished, the extra overhead of doing a full LFSCK run for a problem that is only transient during recovery is very high. At the same time, if LFSCK is making changes to the filesystem during recovery processing it is possible that this could cause recovery to fail (some "orphan" object has an inode created on the MDT that was going to be replayed by the client, and this causes recovery to be aborted).
