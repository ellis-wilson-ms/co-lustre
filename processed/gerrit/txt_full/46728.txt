--- Commit Message ---
LU-15625 sec: set_param -P must not be used for root squash

Only lctl conf_param must be used to set root squash parameters,
because these are the settings that are forwarded to client side.
So prevent use of 'set_param -P' for root_squash and nosquash_nids
parameters.

(1 comment)

-1 because I think just returning an error without a message will not fix the problems of admins getting this wrong (ie. there would still be a support call, just a slightly different one).

At a minimum there should be a useful error message printed. 

However, my preference would be to fix up the "set_param -P mdt.<fsname>*.*squash*" parameters during client llog processing to use "*.<fsname>*.*squash*" so that this "just works". This is done for some other parameters already in class_process_config() but it should be under LCFG_SET_PARAM so it only affects this case. 

Also, catch the "mdt.*.*squash*" case in "lctl set_param -P" in userspace and print a more useful message to use "*.<fsname>*.*" so that the bad records don't get into the config log at all.

--- Message (Andreas Dilger, 2022-03-07 19:51:01.000000000) ---
Patch Set 1:

> If this is the case shouldn't root_squash debugfs files be made
 > read only.

Now that I think about this more, the current patch would break the case where users are setting this parameter correctly using "lctl set_param -P *.*.nosquash_nids=NID" or "lctl set_param -P *.*.root_squash=X:Y" (or separate "mdt.*" and "llite.*" parameters), which is what *should* be documented in the manual. 

The main issue is the use of "mdt.*" in isolation, and the patch http://review.whamcloud.com/5700 "LU-1778 llite: fix inconsistencies of root squash feature" (which I've never really liked) that is adding a "cosmetic" root squash permission check on the client, when this has absolutely no real security benefit.
