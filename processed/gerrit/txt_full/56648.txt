LU-11509 ldlm: Implement LFRU cache eviction

Added frequency count for LDLM lock. If the LDLM lock has high
frequecy, it will moved into ns_priv_list instead of normal
ns_unused_list.
Added LFRU control by setting up min_lock_freq. Added lock
decay to remove the lock from the priv list.

Patch Set 2: Code-Review-1

(10 comments)

Hello Kiet,
thank you for submitting your patch.  It looks like a good start, and don't take my comments as negative criticism, but rather helping to move your patch forward until it can be landed and used in production Lustre filesystems.

I think there is a fundamental gap in the current implementation, that it doesn't appear that l_freq is incremented or decremented anywhere, so the current patch appears to be a no-op (locks are never elevated to the ns_priv_list, and never removed).

Once this is implemented, there needs to be a way to investigate/measure the impact this patch has on LDLM performance.

I've added Yang Sheng as a reviewer, since he has been working on DLM performance recently and can hopefully provide some suggestion on how to test this change, if you don't already have some performance measurements?

At a macro level, something like "compilebench" might be a workload that has a lot of lock/file access, and could benefit by keeping some files in cache more than others (e.g. common .h files instead of .c files that are only acccessed once).  This performance benefit may only be visible if the lru_size is limited and not all file locks can be cached on the client.
