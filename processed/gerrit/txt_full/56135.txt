LU-18163 obdclass: fix sysfs_memparse()/string_to_size()

The patch reworks string_to_size() to avoid string copies and handle
default unit directly in __string_to_size().
Also, this would handle more gracefully fractional value with huge
unit (like 0.5EiB).

This implementation fixes the parsing of the following invalid
strings:
- "10.badbadMib"
- "10MiBbadbad"
- "10MBAD"
- "10.123badMib"
- "1024.KG"

The patch change the way to handle decimal fractional part: a maximum
of 9 digits are supported.

It fixes test_string_to_size_err() to actually return an error if a
test failed and then prevents the module to load.
It fixes obdclass_init() to avoid a crash if obd_init_checks()
failed.

Add regression tests in obd_init_checks.

> This code is a total rewrite which means there are many bugs he is fixing. I like too see each bug issue he is fixing address in a patch at a time instead of fixing many issues at one time. For example the replacement of lprocfs_strnstr with strnstr() could be its own patch.

I agree on lprocfs_strnstr(), I will do a separate patch for this.
But, this is the rewrite of 1 function, so I can't divide the patch more (to keep coherency).

To give some context, when Andreas asked to handle '%' for the LU-14408, I saw the workaround in ll_max_cached_mb_seq_write() and I don't like that. So, why not directly parse the new unit '%' in sys_memparse(): someone else might have the same needs in the future.

I saw several holes in the original implementation string_to_size():
- "10.badbadMib" is accepted
- "10MiBbadbad" is accepted
- "10MBAD" is accepted
- "10.123badMib" is accepted
- '1024.KG' is accepted and the unit selected is "GiB"

I was not sure to be able to modify string_to_size() without breaking something else. So, I saw a good opportunity to rewrite the function by dividing it into smaller reusable parts (string_to_decimal and string_to_blksize). In my opinion, this makes the code more maintainable for future evolutions or fixes.

> I'm not a fan of one line wrappers either around __string_to_size(). This is frowned on upstream since it makes the harder to read.

sys_percent_memparse() is an extension sys_memparse(). These functions are not independent, so it makes sense to call the same functions. Otherwise it would duplicate the same code and make the code less maintainable.

If you prefer, I could to something like:

sys_percent_memparse()
{
...
     rc = string_to_percent();
...
     if (rc > 0)
          return 0;
     rc = string_to_size();
...
}

But this would require parsing the same string several time. So I prefer to add a new unit to __string_to_size() because the change is relatively small.

James, I'm not sure why you've picked this patch to make a stand on, but I don't think is productive compared to spending time on more important code.

How is the userspace tool going to handle a '%' value?  It would have to understand what every parameter means (e.g. that "max_dirty_mb=50%" means that this is a fraction of the total RAM) in order to know what percentage is involved.

In some use cases, it would be possible to convert "1.25M" to "1280K", but how would "max_cached_mb=7.5" (or whatever) be handled in userspace?  Again, "lctl set_param" would need to add a special case that "max_cached_mb" is in MB units and do the conversion to KB in userspace before writing it.

What about (misguided) tools or scripts that are writing to procfs/sysfs files directly, instead of using "lctl set_param"?  Granted, I am not a fan of them, but they definitely exist and I don't want to break them gratuitously.

I don't think doing this in userspace is less complex, and IMHO it is more likely to have a bug.

The biggest complaint should be for the upstream kernel removal of simple_strtoul() and mandatory replacement with kstrtoul() that doesn't allow anything to be after the number, unlike the userspace strtoul().  That is what forced the duplication of all this code.

I could understand having a common function that doesn't allow a suffix and an internal helper that does, but there is no "internal" variant that returns the suffix, which also causes a lot of unnecessary strcpy() for places where a number is embedded in a string (e.g. "START-END" or "NUM:NUM" or similar).
