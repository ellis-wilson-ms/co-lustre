--- Commit Message ---
LU-1415 tests: Cope with ZFS's delayed space updates

Space-related updates, such as changes of dn_used and of
ds_used_bytes, are invisible to opertions like dt_attr_get() and
dt_statfs() until after corresponding transaction groups start to be
synchronized.  This patch makes sure the tests do a dt_sync() for each
OST before checking space usage.

Patch Set 11: Looks good to me, but someone else must approve

Looks good, although there are a few other ZFS issues related to free space which _may_ come up in the future.

1) The free space in your pool will vary based on the activity of other data sets which share the pool.  So for Lustre testing it may be wise for each dataset to have its own pool.

2) When an object is removed the blocks used by that object won't be immediately available for allocation.  However, they will be treated as free in the reporting.  This is to ensure you can always safely rollback the pool by several txgs.  It should be a problem unless you pool is very small and very full.

3) When your removing a object, if that object is part of a snapshot you will not see any space freed.
