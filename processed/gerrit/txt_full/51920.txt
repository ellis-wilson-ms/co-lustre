LU-8802 obd: add strong locking to OBD lifecycle management

The OBD lifecycle code can race in certain conditions.
This is most noticable for class_{attach,detach,setup,cleanup}
and class_{incref,decref}. Implement a new global lock,
obd_lifecycle_management_lock, to stop these different
functions from racing. This lock can reliably prevent
the crashing seen in patch 51895 (caused by attempting
to setup and teardown the same OBD device in different
threads).

OBD devices are setup infrequently, so the potential for
lock contention is low. The time to setup about 10,000
OBD devices is around 10 seconds, with or without this
patch.

Lockless versions of certain functions have been
provided, since certain OBD cleanup and setup
routines must call some of these class_* functions.

Further, class_{attach,detach,setup,cleanup} have
been un-exported to make it harder to abuse these
functions. They were only used in a small number
of places.

--- Message (Neil Brown, 2023-08-10 21:49:05.000000000) ---
Patch Set 1:

Thanks for starting on this.  There is definitely a need here.
I think a mutex might be needed, rather than a spinlock.
The lock is held across class_process_config and I'm fairly sure that can block for memory allocations in some cases.

class_incref() should never need to be locked.  I currently is called with ->obd_dev_lock taken specially for the purpose and I cannot see how that would be useful.  Maybe there is something I'm missing, but if there is - it should be addressed a different way.  We might need a class_try_incref() which fails if the refcount is zero.
Similarly cless_decref() should not need to be locked.  It might need to use kref_put_mutex() or similar to take a lock on the final put, but there is no need for a lock at other times.

I think it would be really helpful to have a clear statement of what the important races are.  That would help us focus on what the best fix will look like.

I'll try to have a deeper look at this locking next week.

Patch Set 2:

Serializing everything using a global lock (like this patch does) definitely seems hacky. Although the alternatives aren't much better, it seems. Serializing just device creation doesn't prevent the crashing from using the ioctl(). I didn't have much luck using a per-OBD lock or mutex. My efforts with obd_dev_lock spinlock still crashed.

Perhaps it's best to land Alex's patch and Neil's cleanup work first, then revisit this? I think the existing locking needs to be cleaned up a lot more before we could properly serialize OBD lifecycle changes without using a global lock.

I think the biggest problem is that if a thread depends on an OBD being in a certain lifecycle state, there's no lock preventing other threads from changing the lifecycle state. Even the lifecycle flags themselves are not consistently checked/set under a lock.

These ioctl() can be used to crash old versions of Lustre also. The corresponding lctl command are already marked as dangerous. So I think not fixing this specific crash now isn't a big deal.

Any thoughts?
