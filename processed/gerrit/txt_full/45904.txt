LU-15388 osd-ldiskfs: wrong dot/dotdot FID for local agent

Wrong FIDs are passed into osd_add_dot_dotdot_internal() in
osd_create_local_agent_inode(). Local agent inode is created to
satisfy e2fsck, and these two FIDs are not used anywhere, which
won't cause any known issue.

> It would be good to have a test case for this.

It seems not easy to produce a corrputed FID on dotdot.

Reason is:

1) osd_add_dot_dotdot() is called with wrong order of fids, and osd_add_dot_dotdot() itself calls osd_add_dot_dotdot_internal() with wrong order too. So, in the case the dotdot has correct FID. So in major cases, dodot FID is still correct?

2) osd_create_local_agent_inode() calls osd_add_dot_dotdot_internal() with wrong order, this will cause wrong FID on dotdot.

3) In osd_add_dot_dotdot(), in case of rename, dotdot is already created, and __osd_ea_add_rec() is called with wrong FID on dotdot. This will cause problem.

Li Xi, the agent inode creation happens when renaming a directory to a new MDT, so something like the following may trigger it:

 # lfs mkdir -i 0 $DIR/$tdir.0 $DIR/$tdir.0/sub0
 # lfs mkdir -i 1 $DIR/$tdir.1
 # ls -lai $DIR/$tdir.0/sub0
 # mv $DIR/$tdir.0/sub0 $DIR/$tdir.1/sub0
 # ls -lai $DIR/$tdir.1/sub0

There would be an agent inode created on MDT1 for the "sub0" directory.
