Patch Set 1: I would prefer that you didn't submit this

I think it would be better to actually ignore unknown parameters, rather than having them prevent the system from mounting. Stopping the muh t has relatively little benefit, but there are definitely cases where an upgrade or mixed environment configuration could cause unknown parameters to be stored in the config.

This would also benefit from a test case that writes an invalid parameter into the config file using "conf_param" and then verified that a new mount does not fail.

Patch Set 3: Looks good to me, approved

Minor style note: for console errors, the preferred format is to print the device name at the beginning, and the error code at the end like:

 CERRROR("%s: ... message: rc = %d\n", device_name, rc);

In this case of global parameters there is not a device name that can be printed, so that part can be skipped, but we're trying to make the message content more useful and consistent we go along.

Not a fatal reason to reject the patch, but would be nice to fix if the patch is resubmitted.

A similar patch is needed for b2_1 as well.
