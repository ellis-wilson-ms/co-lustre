--- Commit Message ---
LU-1302 llog: introduce llog_open

- llog_open is pair to llog_close. Llog handle is allocated inside
  llog_open while llog_free_handle is part of llog_close. Each llog
  even non-existent should be opened and closed.
- llog_exist() is added to API to check if llog file exists
- llog_cat_put is renamed to llog_cat_close with cleaning up empty
  llogs like llog_obd_origin_cleanup did.

Patch Set 1: (5 inline comments)

I am not sure what exactly needs fixing in API? Nevertheless, note, this is port from orion to master, so it is working over OSD already and some not-critical things maybe postponed to later discussion and improvements in master itself.

About API semantic - llog_open and llog_close is like file handle or lu_object semantic. Open just create new handler and associate it with existent llog. Close destroys handle. Maybe that could be llog_handle_get/put but I am not sure that is more clear.

Create becomes very simple and generic - it just does pure create and allowed over OSD only. That is why having thandle is fine here, like that is fine with dt_create in OSD.

Llog library is just extension over dt_object API, all high-level users can use helpers like llog_open_create(), llog_write(), llog_erase() which hides transaction handling.
