LU-7659 hsm: Use netlink for KUC communication

Make communications between MDC and copytools rely on asynchronous
netlink communications instead of pipe based communication. Currently
how the communications is done is considered incorrect as described
here:
https://www.linuxjournal.com/article/8110

This new netlink API uses generic netlink to avoid taking a dedicated
netlink family number. Kernel now initiates communication by creating
socket at module load time and message sending from kernel has
changed from synchronous vfs writing to asynchronous netlink sending.

As for now, copytools subscribe to both netlink & KUC groups.
Copytools registration still rely on ioctls to populate KUC groups,
but no more file descriptors are sent to the kernel to perform VFS
writing.

"BTW how are you testing this to make sure it works? Just running sanity-hsm.sh ?"

I benchmarked and tried a few good candidates mechanisms for Kernel-User communication in https://github.com/YohanPipereau/UserKernelCommunication . This repo was for benchmark, I need to clean up a few things and write a decent documentation. Yet, I have faced a major problem during my testing. If the kernel sends a lot of messages (ex: 10,000), userland sockets will be congested and rise an ENOBUF error (ex : error raised at message 345). This is why netlink is said to be "unreliable" for Kernel to Userland communication as it relies on datagrams and asynchronous sending. To make netlink reliable, we can use ACK and ERROR messages to send again messages when userland sockets are congested. It is also possible to implement some kind of flow control based on counters aware of the available space in the socket buffer before we reach congestion (a bit dirty). 

For the Lustre netlink patch itself, I only tested sanity-hsm and unit testing in lustre/tests/llapi_hsm_test.c.
