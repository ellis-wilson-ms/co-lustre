LU-16981 lod: update llc_stripe_count after ost inactive

If an OST gets deactivated while lod_ost_alloc_qos() is trying to
allocate stripes for a file create, then normally this is caught and
EAGAIN is returned which causes the lod_comp->llc_stripe_count to
get updated to accurately reflect the stripe count. But there is a
race condition and if the OST is deactivated after the call to
ltd_qos_is_usable() but before the stripes are allocated, then
updating the stripe count never occurred.

This causes an LBUG later in lod_striped_create() because fewer
stripes are allocated than the number in llc_stripe_count so it
finds a stripe that is NULL.

The solution is to properly update lod_comp->llc_stripe_count when
the number of stripes created is less than expected.

I'm not sure what the best way is to create a test for this, so I'm open to suggestions on that (if a new test is needed).

Another idea for how to resolve this could be to set lod_comp->llc_stripe_count to 0 prior to the loop `while (nfound < stripe_count)` and then increment llc_stripe_count each time `stripe[nfound] = o` occurs. I haven't tested this method so I don't know if it has issues, but just proposing it in case the current solution isn't ideal. I went with the current solution because it seemed like a less invasive change.

(2 comments)

Thomas, thanks for the analysis and patch. To my reading, this patch is shrinking the race window, which is good, but it is not completely eliminating it. I suspect that your test case in LU-16981 would still fail if the activate/deactivate were running in an independent loop from the creates. 

I think a more complete fix would be to just update the layout with the number of stripes that were actually created in the end, rather than having an LBUG() when reality does not match expectations. That handles any race conditions or failures that might happen during object creation.

--- Message (Andreas Dilger, 2023-07-25 17:13:57.000000000) ---
Patch Set 1:

As for creating a test case, I think what is in LU-16981 is mostly good. Two minor improvements:
- use fallocate to allocate space to the large file since this is *much* faster than actually writing out the data, and add a stack_trap to remove the large file at the end of the subtest
- have two separate loops running in parallel:
* first loop enabling/disabling the OSCs with eg. "sleep 0.1" between them and put into background with a stack_trap to kill it at the end of the subtest
* second loop creating a few hundred fully-striped files (and a stack_trap to delete them all whether the test passes or fails). 

You could run that subtest multiple times from Test-Parameters to ensure it is robust. It doesn't need to create thousands of files if that is what it takes to hit the problem *every* time, just enough that it would fail say 1/10-1/20 runs without the fix, since these tests are already run hundreds of times a day by different patches.
