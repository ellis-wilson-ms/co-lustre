--- Commit Message ---
LU-4459 mdt: reject RPCs sent to wrong MDT

Add a new target handler flag (HABEO_INODE) which requires that the
object sent in the request is an existing local object. In
mdt_object_find_lock() return -EPROTO if the object was found to be
remote.

I'll try to rework this so that it just strengthens the semantics of HABEO_CORPUS.

> I am not fan of adding extra flag here, since it might always cause some compatible problem here.

Several handlers require the fid1 object to exist and be local (several assert on this). So adding a flag that causes then to return -EPROTO rather than crash should not be a compatibility issue.

> Just curious, why do we need client to tell server the object needs to be local?

I don't understand.

> I don't think this makes sense, please clarify why you think we need this flag and what is expected behavior and use cases. Maybe I misunderstood the idea.

The idea was to reject requests sent to the wrong MDT (or with a bad object) early and cleanly rather than crashing because of some failed assertion in lu_object_attr() or somewhere...

It wasn't obvious to me how to fix HABEO_CORPUS in this way. Since the mdt and ost unpacking functions have different expectations about setting tsi_corpus.
