--- Commit Message ---
LU-247 clio: Unaligned dio support

- Port unaligned dio support (b18801);

- Changed the buffered lockless io into direct lockless io, since
  it doesn't make sense to copy the data between page cache and user
  buffer for lockless io, and the overhead of such copy sometimes
  could be heavy (as shown in b18801). Furthermore, dirty page cache
  without holding dlm lock could result in cache inconsistence.

- Fix the problems caused by concurrent dio read & buffered read,
  like LU-3192.

--- Message (Vitaly Fertman, 2013-07-31 13:34:06.000000000) ---
Patch Set 18:

I may think of the following performance issues with this patch:

1. DIO will not gather cached BIO pages to its RPCs, so to send all such pages, new RPC will be needed. it may slightly decrease performance, although not critically.
2. DIO will not find cached pages on the client side anymore, and will re-read them from OST again - such case will be slower.
3. mixed IO happened on the same region, could minimize the fragmentation in this IO and therefore the amount seeks on OST, if sent together. DIO sent separately, will not care about it, so the IO performance on OST will be slower due to amount of seeks. however, this is a very very rare case, so I would not pay much attention to it.
4. DIO and BIO sent through separate engines may lead to well known 1.8 problem, with DIO generates a lot of RPCs, but does not pay attention on max dirty pages and grants; when max dirty pages is reached, BIO starts generating RPC, but its slow due to large amount of DIO RPCs, and new BIO is just blocked, whereas DIO IO is very active - the patch will add this misbalance back again. 

therefore, I think (2) and (4) should be addressed before landing this patch.

Hi, Vitaly

for 2: I don't think direct read should check page cache, DIO means it should bypass page cache, and this patch is doing the right thing. If DIO read uses cached page, what's the difference of direct read with buffered read? In most case, I think DIO user manages cache in userspace by itself, lookup page cache just added unnecessary overhead.

for 4: I don't think there will be heavily mixed DIO and buffered IO in the real world (Did any 1.8 user report such problem?). And actually, if we can tolerate DIO fails for ENOSPC when there is still available grant on the client, the grant consume code for DIO can be totally removed.

--- Message (Oleg Drokin, 2013-08-01 01:40:15.000000000) ---
Patch Set 18:

Indeed, like we discussed with Vitaly already, we should not be looking for cahced pages for direct IO at all.
In fact kernel goes to great pains to flush all such cached pages before starting direct IO, so there should not be any pages like that anyway (unless there's racing IO).

Mixing Direct io and non-direct IO in the same RPC is also somewhat bad idea because generally we want to commit direct io right away on the server, but non-direct io could wait. (and besides, this is a pretty stupid usecase that I don't think needs any optimization).
