--- Commit Message ---
LU-12634 libcfs: force_sig() removed task parameter

Linux 5.3 removed the task parameter for force_sig()
signal: Remove task parameter from force_sig

When force_sig() is not available reset the target thread
default handler to SIG_DFL and proceed to use send_sig(..., 1)
which eventually marshals the same signal to the target task.

kernel-commit: 3cf5d076fb4d48979f382bc9452765bf8b79e740

NOTE: force_sig() is used here instead of a wake_up_process() as tasks
      may be blocked on rpc activity.

It isn't immediately obvious to me that this is correct - I would rather the commit message made it obvious.
A signal might wake an l_wait_event() that wake_up_process() wouldn't.
If the signaller doesn't make the wait condition true, if the l_wait_event call has allowed interrupts, then the behaviour can be different.

There are two cases here: li_task and lad_task.
The wait for li_task will abort on an interrupt, and the condition includes !thread_is_running()
The wakeup of li_task sets the thread flag SVC_STOPPING before triggering the wakeup.
So the condition will become true and the fact that a signal aborted the wait isn't important.

For lad_task there are two waits in lfsck_assistant_engine()
