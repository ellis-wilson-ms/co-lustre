LU-12140 lnet: adds checking msg len

The LNET can't handle a msg with len larger than LNET_MTU.
The next error occured for DOM 1MB
 LNetError: 3137:0:(lib-move.c:4143:lnet_parse()) 192.168.8.1@tcp,
 src 192.168.8.1@tcp: bad PUT payload 1051832 (1048576 max expected)

The patch adds fragment size check.

Patch Set 2: Code-Review-1

that doesn't look right. 1MB DOM stripe size shouldn't cause lnet problems and reducing it to 64K is not a fix but the hiding of problem if it exists. No matter what size is, there should be no such big data transfers. DOM read-on-open feature can transfer data but not more than reply buffer size, if it is bigger than LNET_MTU then probably something is wrong with RPC parameters and they should be fixes instead?

Also that can be already fixed by commit  e0adb618a4b0d0182419a5731fe046e9157b9f51 which prevents reply buffer grow

--- Message (Mikhail Pershin, 2019-05-29 06:03:35.000000000) ---
Patch Set 2:

Without that patch disables reply buffer grow, read-on-open could ask for reply buffer to be up to:
max_reply_len = req->rq_rqbd->rqbd_svcpt->scp_service->srv_max_reply_size;

So it seems srv_max_reply_size is bigger than LNET_MTU, probably it requires more checks while being set. Not just because this particular case but in general to prevent similar things in future. That would be reasonable to limit it by lnet routers limits, I am not sure if that is possible though
