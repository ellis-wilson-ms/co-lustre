LU-9538 utils: Tool for syncing file LSOM xattr

Add a helper tool for syncing file LSOM xattr.
Firstly, register a new changelog user:
lctl --device lustre-MDT0000 changelog_register

After perform some file operations on Lustre file system, run
this tool to sync file LSOM xattr:
llsom_sync -u cl1 -m lustre-MDT0000 /mnt/lustre

Patch Set 1: Code-Review-1

(14 comments)

I think this looks pretty reasonable to start, but needs some fixes.

It needs a man page as well as a test case.

Rather than making this a one-shot program, you may want to add a "--daemon|-d" mode that uses daemon() to detach from the shell, and runs in the background to process changelog entries as they arrive (probably with a configurable delay, maybe 10 minutes by default?), and sleeps when there are no entries.  There is the llapi_changelog_get_fd() call to get a file descriptor that can be used for poll() to avoid doing sleep/check in a loop.

(1 comment)

> All that is to say I don't see much benefit in doing a sync. The info is still likely to be wrong about as often.

The benefit of doing a sync is to minimize the inaccuracy of LSOM.

It is not clear now how much percentage of files will have inaccurate LSOM, because the inaccuracy highly depends on the I/O pattern (parrallel open/close/write/truncate etc) and client evictions. We will be able to figure that out after LSOM is widely.

But no matter how much percentage of files will have inaccurate LSOM, this sync tool will be able to decrease the probability. And I think if there is no MDT failover, the Changelog will be able to catch all close/truncate events, which means, with that assumption, the accuracy should eventually be 100%. That means, by using this tool, eventual consistency can be guranteed with the assumption that MDT doesn't failover. And eventual consistency is very important for policy engines and other scanning tools. Policy engines doen't mind too much if a small part of files are missed during this scanning loop, as long as these files will be found at the next scanning loop. However, if the policy engine can never found some files, that bothers administrator.

And please note this gurantee is achieved with only very small overhead. You can enlarge the "o_min_age" option to one hour, one day or even one week. With that long time period, this tool will only sync LSOM of files which are not being actively accessed.

Of course, this tool requires turning on Changelog feature, and Changelog always introduce the problem of possible space exhaustion of MDT. But that is another question.
