Looking at the test in question, I suspect a lot of different things could make it both more both robust and faster:
- create and execute multiple "dd" copies in the filesystem, since that is the primary goal of that test
- limit the size and/or number of files based on available space, though I'd think that a *total* of 1GB space is not unreasonable?
- slow down the write speed (eg. "bs=4k" and "oflags=sync", not sure if there is a bandwidth throttle for dd like Tim added for "lfs migrate")
- run some other binary in addition or instead of  "dd" copies that are not consuming so much space in the filesystem, but generating locks (eg. "cp -a /etc $DIR/$tdir-etc")
