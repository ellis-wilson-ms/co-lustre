LU-11432 lnet: Assign specific values to enum constants

The enum constants in lnet-types.h are automatically
enumerated from 0. This can break the network
protocol compatibility if mistakenly a new
values is inserted or a value is deleted in the
middle of the list.

This patch also adds a new enum struct lnet_md_options
to pack all the lnet MD options together.

--- Message (Andreas Dilger, 2020-02-03 08:49:47.000000000) ---
Patch Set 3:

(1 comment)

> Doesn't this duplicate the functionality of lnet_assert_wire_constants()?
 > Do we need both?

I think they are totally different.  This is fixing the declaration of these enum constants to be explicit values, which is critical because they are part of the wire protocol and should never change.

However, they are currently declared with implicit values, which is not only dangerous, since the value would change if other constants are added or removed before a given constant, but having the actual values at the point of declaration makes it easier to determine the actual value when looking at logs where they are printed.

The use of named enums is preferred over #defines because it improves code readability and reduces bugs to see "enum lnet_md_options options" as a function argument over "int options" that doesn't give any hint as to what values it should be passed.  Using enums also allows the compiler to have improved type checking, and debuggers can sometimes use the enum to print symbolic names for fields.

As for wirecheck.c/wiretest.c, those are to *verify* that these values do not change due to developer misunderstanding or accidental changes.  Having wire/interface constants checked in wire* makes it clear when the code changes that there is a bug in the patch, or at minimum some unusual change is being made to the code.
