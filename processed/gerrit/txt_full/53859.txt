--- Commit Message ---
LU-17483 gss: refresh req context with already existing one

When we are processing a request with a root GSS context that
has the PTLRPC_CTX_ERROR_BIT bit set, try to replace it with an
already existing context. Such a context can already be up-to-date
thanks to other authentication requests sent to failover NIDs while
the current request was in the delay list. This valid context can be
fetched from the struct ptlrpc_sec.

> Patch Set 2:

> This patch looks promising. I did not spot anything obviously wrong, but I don't understand the code enough to really review it.
> 
> I'd love to have a more context. Could you describe, here, or in the ticket a scenario where this could be useful?

One use case I have identified for this patch is when a client mount command is issued while a target is running on a failover node (i.e. not the primary node). If a GSS flavor for that target is in place, the mount would fail.
I have identified this problem as a consequence of the failover NIDs being tried in parallel. Basically all the requests sent initially to the various failover NIDs are pointing to the same GSS context. And this initial GSS context will not be the right one, because it corresponds to a SEC_CTX_INIT request sent to a NID where the target is not running. A subsequent GSS context is created later, for a SEC_CTX_INIT request sent to the right NID.
