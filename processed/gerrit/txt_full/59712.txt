LU-13653 mdt: add EX lock protection for FLR transitions

Add exclusive lock protection around FLR state transitions to
prevent race conditions between client operations and server-side
layout changes during concurrent write scenarios.

The race condition occurs when multiple threads attempt to modify
FLR state simultaneously - client operations can interfere with
server-side transitions from WRITE_PENDING back to RDONLY state
when the last active writer releases their lock.

Taking the EX lock guarantees there are no clients with AW locks
when we do the layout updates.

Solution:
- Take EX mode ACTIVE_WRITER lock in workqueue before FLR transition
- Exclude EX mode locks from active writer counter to prevent
  recursive transitions
- Release EX lock after FLR transition completes

This ensures atomic FLR state changes by blocking client activity
during server-side layout transitions, preventing corruption of
mirror metadata and ensuring consistent FLR state management.

Changes:
- Modified mdt_blocking_ast() to skip counter decrement for EX locks
- Modified mdt_intent_active_writer() to skip counter increment
  for EX locks
- Enhanced mdt_flr_rdonly_work_fn() with EX lock acquisition around
  FLR state transitions

The EX lock exclusion prevents infinite recursion where the EX lock
taken for FLR protection would itself trigger another FLR transition.
