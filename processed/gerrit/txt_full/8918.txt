LU-4460 mount: fix lmd_parse() to handle comma-separated NIDs

This patch reverts commit 3917e62018878dfffac59ceed70f20b0419945d3,
which cannot handle the upgrade situation that old mountdata already
contains comma-separated NIDs. The correct way to fix the original
issue is to parse comma-separated NIDs in lmd_parse().

The patch also updates disk2_4-ldiskfs.tar.bz2 to make the mountdata
of ost contain comma-separated NIDs so as to verify the patch under
upgrade situation.

--- Message (Jian Yu, 2014-02-12 15:26:29.000000000) ---
Patch Set 5:

I just found the root cause.

In the above failure, the original options were:

user_xattr,acl,osd=osd-ldiskfs,errors=remount-ro,iopen_nopriv,user_xattr,acl,mgs,virgin,update,param=sys.timeout=20,param=lov.stripesize=1048576,param=lov.stripecount=0,param=mdt.identity_upcall=/usr/sbin/l_getidentity,svname=lustre-MDT0000,device=/dev/lvm-Role_MDS/P1

After parsing "param=sys.timeout=20", "s1" pointed to ",param=lov.stripesize=1048576,param=lov.stripecount=0,...", and "s2" pointed to "param=lov.stripesize=1048576,param=lov.stripecount=0,...".

Then memmove(s1, s2, strlen(s2) + 1) made the options become "...param=sys.timeout=20param=lov.stripesize=1048576...".

In fact, "s1" should point to "param=sys.timeout=20,param=lov.stripesize=1048576,param=lov.stripecount=0,...", which will make the memmove operation clear the "param=sys.timeout=20" option correctly.

So, it turns out that the value of "s1" in the long "if ... else if ..." code block should not be changed, and "s3" is still needed. I'll revise the patch back to the changes in patch set 4.
