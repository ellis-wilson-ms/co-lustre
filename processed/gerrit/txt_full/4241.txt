--- Commit Message ---
LU-2093 lod: fall back to RR allocation when QoS fails

lod_alloc_qos() checks is there enough OSPs to satisfy the request
checking OSP state with dt_statfs(), then it tries to reserve
objects on some of them. during the reservation the state of OSP
can change (due to broken connection, for example), then QoS code
might found less ready OSPs than required. this is a valid situation
and LOD should fallback to RR allocation.

sanity/116a added to verify this: dt_statfs() are still reporting
OSPs are good, but no actual object can be created on OSP with
index 1.

--- Message (Andreas Dilger, 2012-10-10 16:14:57.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

I don't understand why there is a need to fall back to RR if the QOS allocation ran out of OSTs?  The OSTs are not going to be more available with RR allocation, or is the RR code trying harder to wait for OSTs to recover?  With the old code if there were at least 3/4 of the requests stripes that could be allocated then it succeeded and continues on. If not, then an error was returned to the caller. This was of course only the case after having waited for all OSTs to complete precreation if they were currently out of objects, but not if they are out of space or offline. 

Please don't name tests test_116() and test_116a(), since it is then impossible to run just test_116(). Instead, rename the old test to 116a and the new one 116b, with a comment that the old test used to be named 116 ("# was named test_116")..
