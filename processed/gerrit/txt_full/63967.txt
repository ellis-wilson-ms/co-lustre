--- Commit Message ---
LU-0000 lov: fix EC recovery for PFL beyond-EOF stripes

Fix two bugs that cause EC recovery data corruption
when a disabled OST is shared between PFL components
and the file does not fill a complete RAID set.

1. lov_io_set_range() expands the EC read extent to
cover full RAID sets for parity calculation. This
causes lov_ecio_add_data_sub() to process data stripes
beyond EOF. If such a stripe is on an unavailable OST,
the lock path marks it LSS_READ_ERR, triggering
unnecessary recovery that corrupts valid data. Fix by
capping the data extent to i_size in
lov_ecio_add_data_sub() and clearing any false
LSS_READ_ERR on non-intersecting stripes.

2. lov_ec_read_data_pages() used component-relative
offsets for lov_stripe_number(), but
lov_ecio_add_data_sub() uses file-level offsets via
lov_stripe_intersects(). For PFL components with
non-zero e_start, this mismatch causes the recovery
function to check the wrong stripe's lso_status for
each page position. Fix by using file offsets with
modulo wrapping to prevent out-of-bounds access.

Also improve test_46d to select only data-only OSTs
for deactivation, avoiding parity OSTs which would
reduce recovery capacity.
