LU-9538 mdt: Lazy size on MDT

The design of Lazy size on MDT (LSOM) does not guarantee the
accuracy. A file that is being opened for a long time might
cause inaccurate LSOM for a very long time. And also eviction or
crash of client might cause incomplete process of closing a file,
thus might cause inaccurate LSOM. A precise LSOM could only be read
from MDT when 1) all possible corruption and inconsistency caused
by client eviction or client/server crash have all been fixed by
LFSCK and 2) the file is not being opened for write.
In the first step of implementing LSOM, LSOM will not be accessible
from client. Instead, LSOM values can only be accessed on MDT. Thus,
no interface or logic codes will be added on client side to enabled
the access of LSOM from client side.
The LSOM will be saved as an EA value on MDT.
LSOM includes both the apparent size and also the disk usage of
the file.
Whenever a file is being truncated, the LSOM of the file on MDT
will be updated.
Whenever a client is closing a file, ll_prepare_close() will send
the size and blocks to the MDS. The MDS will update the LSOM of
the file if the file size or block size is being increased.

Alex, the blocks count may be a few seconds out of date with ZFS, but if the file is held open for even a few seconds then ZFS will start returning some blocks.  The size information will be accurate on at least one client.

In many cases, applications don't actually check the i_blocks value, only the i_size value, so that is enough.  As Qian also wrote, there is a separate tool that watches the ChangeLog and can update the SOM size/blocks after the file is closed (by default it runs a few minutes after the ChangeLog records are written), and the SOM size/blocks will also be updated if any application does an open+stat+close (i.e. if it based some decision on the LSOM data and then verified the file stats before operating on it).

It would also be possible to have LFSCK update the SOM information on every scan, since it has all of this information when it is running (layout, OST attributes, etc).

(major) whereas this patch introduce 2 levels of reliability, I do not agree it is enough. if we actually tried to guess the file attributes, this guess is very valuable for some tools, especially because we have an accurate size in most of the cases - indeed, it is accurate for a single writer and if no recovery is involved - the most popular case, and some other less popular cases; however if we have not tried - the attributes still stored on disk are just wrong. there is no sense to start using attributes if we know the real ones are really different. thus, from my perspective it is very important to separate stale attributes - the open-for-modification should remove the LSOM attrs (set it to SOM_FL_UNKNOWN). this removal can be async though as we still do not pretend to be 100% accurate - still while there is no recovery, it will reach the disk.

> the open-for-modification should remove the LSOM attrs (set it to

The one problem here is that Fortran codes always open files for write, which would invalidate the LSOM data.  If we also reset the LSOM data when they close the file it would two extra writes to the MDT per opened file for what _should_ be a read-only operation.

The question is whether the "stale" data is going to be really inaccurate, or not any worse than before?  If the client doesn't write to the file it is not worse, and even if the client writes to the file, the stale LSOM values are better than no data at all.

> why [cm]time attrs are completely ignored by LSOM?

I thought about this as well.  I think the MDT already has a fairly good idea of the [cm]time, enough for "lazy" operations.  I also figured this patch was getting complex enough that we could add [cm]time updates at some later patch if they were very inaccurate.  

I'm not against updating [amc]time as part of LSOM, though I don't think we need to store them into the "trusted.som" xattr.  They should be safe to store directly into the MDT inode.

Ideally, I wouldn't mind getting the LSOM patch landed for master and then consider to back-port it to 2.10, since it would be very useful for some workloads, doesn't add much overhead, and is fairly isolated in the code.  The more complex it becomes, the harder it will be to accept it into 2.10.

Patch Set 31:

Hi Vitaly,

It's clearly awared that LSOM will be inaccurate in many cases, especially when there are concurrent writes/truncates. However, the LSOM is designed to be inaccurate in this way. The accuracy of LSOM can not be trusted in any time. It is only a way to speedup scanning tools or policy engines. Whenever the MDT scanning tools or policy engines need accurate LSOM, it should check the real file size/blocks.

For example, the policy engine might want to run "lfs migrate" on all of the files that are bigger than 1GB to balance the OST usages. Even though LSOM might be inaccurate, but it is very likely the small files have small LSOM, and large files have large LSOM. In the worst case, all of the LSOM could be wrong. But because most large files have large LSOM, it is very likely the policy engine can find most of the files that are bigger than 1GB. If 90% of the files larger than 1GB have LSOM larger than 1GB, then the policy engine can find out and migrate 90% of the files. And that is totally good enough for OST usage balancing.

After the policy engine gets the "suspected" file list that could be larget than 1GB by scanning LSOM, it can double check by getting the real file size. If the file size is smaller than 1GB, policy engine could skip the migrating.

There might be some files that are larger than 1GB but have <1GB LSOM. The syncing tool of LSOM could help to sync the LSOM eventually, and policy engine might found and migrate those file eventually in the next loop.

Even with LSOM syncing tool, the policy engine could still miss some files that are large than 1GB. But that is totally fine. It is very likely that a very small part of the files are missing. And please note the file system is being changed all the time. Even a tool can print files that are larger than 1GB at the scanning time point, conditions could change in the next second because of writing/truncating. In these kind of use cases, it is impossible to ensure accuracy of 100%, and 100% accuray is unnessary at all for these use cases. To get 100% accuray, the administrator will need to 1) stop all of the I/O, 2) scanning the whole file system using real file size. I can't think up any use cases for that. It is really exclusive and really slow, which I don't think is useful for data management of actively running file system.

I think the current design is alreay enough for a lot of use cases. Of course, improving the LSOM would always be nice. The accuracy that policy engine can get from scanning LSOM could be improved maybe from 99% to 99.9%. But we need to think whether the efforce to decrease the inaccuracy from 1% to 0.1% really worth it. And we currently don't have any data or experience about the accuracy at all. I think we need to land the LSOM feature and use it for current use cases. We will be able to know whether the accuracy is enough or not soon. Enough statistics can be collected soon for making the decision or judge at that time. And if the accuracy is obviously not enough for some use cases or in some corner cases, we can improve it latter at any time.
