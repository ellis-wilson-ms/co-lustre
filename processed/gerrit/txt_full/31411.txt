LU-9341 llite: don't instantiate full layout on append

Currently, append writes instantiate all layout components
to get around the fact that VFS does not provide the offset
for an append write and we can't check file size correctly
before knowing which components we must lock.

Instead of initializing all components, use our current
local size as the offset, then get the layout and
instantiate any required components as normal.  Now, we
lock these components and get the file size as normal in
vvp_io_write_start.

This file size may be more up to date than the local one,
in which case we must restart the I/O so we can instantiate
any components that we would touch at our new offset.

When we restart, we have a file size that is guaranteed to
have been correct at some point during our I/O, because the
only way for it to be wrong is a new component was
instantiated after we checked the layout, in which case
that instantiation happened during our I/O and is a
legitimate race condition.

Racing append writes can conceivably cause many restarts,
but the LDLM locking required should make this rare.

Note this does *NOT* change the LDLM layer locking behavior
of appends, which still take locks to infininity on all
components they are aware of, in order to protect file
size.

Add some tests for this to sanity-pfl.sh, correct other
tests which assume we can't append to incomplete files.
