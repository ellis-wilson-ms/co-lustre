LU-15274 llite: Fix whole file read and fast_io

There is a subtle bug with whole file read.  If the first
few reads of a file are at the back of the file, readahead
state can become stuck such that ll_use_fast_io always says
no.

The file is read in correctly, but this prevents fast read
from being used, which has a large performance cost.

When we're doing a whole file read, we don't need to drop
back from fast read to issue another RPC - The whole file
is already being read in.  We should only drop back to
regular read if a page is not found in cache.

For a particular mmap read case reading 63 MiB of data,
this reduced the time required to read it from 28 seconds
to about 5 seconds, an improvement of about 82%.

NB: I've been unable to reproduce this exact situation in
testing, so I do not have a specific test for this patch.
However, the customer can reproduce it consistently and
this resolves the issue there.
