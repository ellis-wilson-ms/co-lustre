LU-1016 build: no sub-builds in sub-shells

The make program paradigm is to have command failures fail the make.
Invoking a failing "make" in a subshell will cause the subshell
to exit returning non-zero to the invoking shell.  However, the
invoking shell will *NOT* exit.  Therefore, to correctly use a
subshell in a make file scriptlet, use the following template:

  ( do-some-stuff ) || false

this will work as expected.  You want "false" instead of "exit 1"
in case make as been invoked with the "-i" ("--ignore-errors")
option.

WRT the code in question, there really is no need of even using
a subshell.  So this patch fixes two (related) things:

1.  the "for subdir in $(RPM_SUBDIR)" is now done in one process
    and will cause the superior make to stop if any subdirectory
    make fails.

2.  The tests for BUILD_TESTS and BUILD_SERVER for building RPMs
    need to take into account the possibility that these are not
    necessarily defined in all autoMakefile.am's that use the
    autoMakefile.am.toplevel file.

To see the problem, you must first induce an error in a subdirectory build.  One way of doing this is to not provide a value for BUILD_TESTS.  That will cause a scripting failure.  That issue was fixed.  So:

1. That should not have been a failure in the first place.
    When one writes a make file for a subdirectory, they should not
    be required to scrutinize the long, convoluted "toplevel" included
    makefile for obscure variables that must be set "true" or "false".
    After all, only one subdirectory requires that BUILD_TESTS be
    set true, yet its needs force recognition in all other subdirs.

2. When there is a failure due to this or any other issue, the build
    must stop.  Without this patch, it continues as if there were nothing
   wrong and I had to laboriously hunt down the cause. (Which was #1
   above, but has since been fixed without fixing the infrastructure
   problem.)

Patch Set 4:

This will complete successfully:

$ cd $(git rev-parse --git-dir)/..
$ sed -i "/^BUILD_TESTS/s/^/# /" ldiskfs/autoMakefile.am
$ bash ./autogen
$ configure
$ make
$ make rpms

but not really.
The ./autoMakefile.am will source "build/autoMakefile.am.toplevel"
and go through a subdirectory loop that will not notice that when
the rpms-real rule gets invoked in the subdirectory, it chokes and dies.
It chokes and dies because BUILD_TESTS is undefined triggering a
shell syntax error.  But it could be any other failure, too.

1. Failing to define BUILD_TESTS should not lead to an error because
    it is imposing implementation details to various lustre components
    that do not really need to know that stuff.  If you really have to
    make the make fail because someone neglected to add BUILD_TESTS
    values (as was the case a few weeks ago), then at least make the
   error comprehensible instead of a shell syntax error failure.

2. If there is a subdirectory failure, then the make process must stop.
    The current setup marches on as if there is nothing awry.  There
    are two approaches to fixing it, and I chose one of them.

   a) use this syntax when using a subshell in a make file:
        ( cd somewhere ; do-some-stuff ) || false
      if there is a failure, the subshell exits non-zero and runs the "false" command.
      That will tell the shell to exit if "set -e" has been utilized.
      If you do not use the "|| false" the shell *WILL* *NOT* *EXIT*
      even though "set -e" has been specified.

   b) use this syntax, avoiding a subshell make entirely
       (this is what I chose):

        { cd somewhere ; do-some-stuff ; cd - }
