--- Commit Message ---
LU-14123 wbc: alloc fid on the target same with root WBC dir

In this design, all files under a root WBC directory are allocated
with FIDs located on the target (MDT) same with the one where the
root WBC directory located.
And also default LMV on the root WBC directory is not supported.
This avoids cross-MDT updates and eliminate the dependency, thus
simply the future batched metadata update and recovery.

--- Message (Andreas Dilger, 2021-05-22 18:14:33.000000000) ---
Patch Set 1:

Is this a temporary restriction?

One of the potential benefits of WBC is that it has time to make a better decision about where to place inodes and stripe directories, because they are in Memfs first, and the client doesn't need to make a decision at create time of which MDT to use. 

That way, if a WBC subdirectory tree grows large, it could be created on a remote MDT and/or with multiple stripes at flush time, rather than doing extra work to migrate inodes after they are all written to one MDT.

--- Message (Qian Yingjin, 2021-05-23 16:03:13.000000000) ---
Patch Set 1:

Yes, it is a temporary restriction for benchmark purpose.

I agree that delaying to decide the FID at the flush time can make much more better balance for MDT usage. But as you mentioned previously, when the client runs the commands such as "ls -li" or stat(), which needs the inode number information, it needs to flush the file to MDT. If we alloc the FID when create the files in MemFS, then it does not need to flush.
And moreover, we can restripe the directory when the directory becomes extreme large.
See LU-13564 WBC: inode and space grant mechanism under WBC
and 
LU-14141
WBC: Stripe directory not flushed to MDT when it grows large enough
for more information.
