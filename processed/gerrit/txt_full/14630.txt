LU-6397 osc: Remove kms_valid check in osc_enqueue

If kms_valid is not set in osc_enqueue_base, a new lock
request will not attempt to match existing ldlm locks.

This opens up a race on newly created objects where a
client can create two LDLM lock requests where only one
is necessary:

kms_valid is not updated until osc_enqueue_fini is called,
which does not happen until a lock request has received a
reply from the server.

Consider two processes, P1 and P2:
P1 makes an IO request (FX, write first page of a file)
P1 creates an LDLM lock request
P1 does not check for existing LDLM locks because kms_valid
is not set
P1 waits for reply from server
P2 makes an IO request (FX, read second page of the file)
P2 creates an LDLM lock request
P2 does not check for existing LDLM locks (goto(no_match) i
P2 waits for a reply from server
P1 Receives reply, lock is granted
(Lock is expanded beyond the requested extent, so it covers
the area P2 wants to read)
P2 Receives reply, lock is blocked by lock granted to P1
Lock granted to P1 is called back by server, even though it
matches request from P2

kms_valid is checked here because it is currently possible
to have ldlm locks without a corresponding osc object if
the osc object has been destroyed by osc_object_prune.
(Called for drop_caches or under memory pressure.)

This patch changes osc_object_prune to cancel any LDLM
locks associated with that osc object.  This is safe
because if any of the LDLM locks are in use, the
osc object will not be a candidate for pruning.

This allows us to remove the kms_valid check in
osc_enqueue_base.
