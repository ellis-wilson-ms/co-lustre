LU-16605 lfs: Add -0 option to fid2path

Currently fid2path adds '\n' after printing
pathnames. Add '-0' option to fid2path to
add NUL('\0') at the end after printing out
pathnames instead of '\n'. This allows
pathnames that contain newlines('\n') to be
correctly interpreted by binaries like xargs.

Without -0 option:
$ lfs fid2path /mnt/lustre 0x200000401:0x1:0x0
/mnt/lustre/Test
_file
/mnt/lustre/Link_
file

With -0 option:
$ lfs fid2path -0 /mnt/lustre 0x200000401:0x1:0x0 | xargs --null
/mnt/lustre/test
_file /mnt/lustre/link_
file

Test-case sanity/226e added.

Sorry, I didn't understand the original problem that this patch was fixing. 

I think using "-0" for this option is confusing, because "find -0" and "lfs find -0" have a specific meaning - terminate printed strings with an explicit '\0' so that tools like "xargs -0" can handle pathnames with embedded spaces, newline, etc.

It would be better to make the handling here consistent with the other tools so that "lfs fid2path -0" is printing a NUL-terminated filename suitable for "xargs -0" usage instead of just skipping the '\n' completely, since that essentially prints the wrong filename, and could potentially conflict with a *different* filename that doesn't have the embedded newline in it.
