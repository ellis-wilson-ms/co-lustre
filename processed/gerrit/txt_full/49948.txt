LU-16540 llite: move ll_get_user_pages to ptlrpcd

Moving ll_get_user_pages from the top level to running in
the ptlrpcd context has the potential to significantly
improve DIO performance (10% or more).  This requires some
code reorganization and some careful handling of vmpage
assignment, etc.

This patch isn't even at prototype stage yet, I just don't
want to lose it.  Doesn't work yet.

Issues are mostly around the relationships between sdio and
iovec_iters, and setting vmpage in the cl/osc pages.
Particularly, issues around stripe boundaries, RPC
boundaries, and the resulting relationships between the
iovec_iter, the sdio, and the extents/rpcs.

Patch Set 9: Code-Review-1

So I realized - This is an incorrect approach.

Specifically, you can't move this to ptlrpcd because a single sdio - sub dio - can be split across multiple RPCs which can be handled by multiple ptlrpc threads.  That means you need concurrency control to make sure they don't conflict in working on the sdio, and you can't do that because that would mean one of them had to hold a spinlock during potentially sleeping operations (get_user_pages, among other things), so this has to be done differently.

But that's OK - it just means we'll need to farm out the handling and submission of each sdio to a worker thread up at the top level.  That should actually be - I think - pretty easy, and has the potential to *seriously* increase single threaded performance.  Like, possibly several *times* for larger I/O sizes.  So that's neat, if a bit absurd.  But get_user_pages is one of the few significant pieces of work that can't be avoided, so parallelizing it is useful.

Anyway, fun stuff.
