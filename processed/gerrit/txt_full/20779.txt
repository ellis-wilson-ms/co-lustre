LU-8272 ldlm: Use interval tree to update kms

Currently, ldlm_extent_shift_kms does a linear search of
the list of granted locks on the resource when looking for
the lock to use to update the kms.

This can be avoided by using the interval trees which store
the extents of granted locks.  For PW/write locks, the lock
with the highest start must be the lock with the highest
end as well, so we can walk the interval tree in reverse to
almost immediately find the new 'highest end'.

Since the tree is sorted by 'start' and PR locks can
overlap, we cannot easily use the tree to find the PR lock
with the 'highest end'.  So we cannot optimize this case,
but many PR locks with different extents should be rare, so
this is OK (and it is no worse than what we do now).

I thought about this for a while, and concluded that because kms_ignore could be set on any lock, we can't really do the "walk the tree and calculate a new maxhigh ignoring this lock" thing.  Not in a way that's efficient.

So we can't do PR locks as efficiently as I'd like.  In some cases we could still end up walking the whole tree.  It should still be an improvement for PR locks in many cases (since we start at the highest start rather than just walking the list), and it's a huge improvement for the case of many PW locks.

Picking up Vitaly's comment and an email conversation he and I had...

I don't think we care about making the case of lots of PR locks perform well.  Lock ahead/requestlock is *not* going to be used with PR locks.  Since read locks can already overlap, lock ahead just adds overhead.

So the case of many PR locks on one client shouldn't happen.  It's possible, but it will not happen in normal usage.

So I don't think we need to implement the max_high + ignore that Vitaly described.  It's not worth the effort.
