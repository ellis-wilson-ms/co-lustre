--- Message (Neil Brown, 2019-07-08 01:41:29.000000000) ---
Patch Set 1: Code-Review+1

The flag is FS_REQUIRES_DEV, not FS_NEEDS_DEV !!!

The main effect of this flag to the prevent "nodev" appearing in /proc/filesystems.
This might affect installers and similar tools, which probably isn't an issue for lustre.
It also affects nfs exporting, but in a way that is increasing irrelevant.
Finally it affects apparmor and tomoyo security systems.  If the flag is set, then the "device name" passed to mount is assumed to actually be a device name, and access permissions are checked.

We definitely don't want this for the client.
I'm not certain how important it is for the server - probably the security systems are the only affected part that is at all interesting.
I would probably remove it, but when we introduce a different filesystem for server-side mounts, consider re-introducing it.

Maybe yes, but some caveats below.

We don't need/want NFS export directly from the server-side mounts, nor any pathname-based security checks on the servers, since OSTs don't have paths, and DNE MDTs can't generate full pathnames by themselves either, so removing it from the server side seems fine.

On the client side, we *do* want clustered NFS re-export to work even though (paradoxically) the client is a "nodev" filesystem.  I'd generally trust Neil's understanding of NFS more than my own, but it is entirely possible that we are doing something strange due to issues that existed in the kernel 15 years ago that could be done in a better way today.

fs/nfsd/export.c says about this:

    /* There are two requirements on a filesystem to be exportable.
     * 1:  We must be able to identify the filesystem from a number.
     *     either a device number (so FS_REQUIRES_DEV needed)
     *     or an FSID number (so NFSEXP_FSID or ->uuid is needed).
     * 2:  We must be able to find an inode from a filehandle.
     *     This means that s_export_op must be set.
     */

There is (somewhat ugly) code in client_common_fill_super() that sets sb->s_dev based on the MDT name so that it is consistent across clients so that clustered NFS can use the same file handle on all client nodes:

    /* We set sb->s_dev equal on all lustre clients in order
     * to support NFS export clustering.  NFSD requires that
     * the FSID be the same on all clients. */
    uuid = obd_get_uuid(sbi->ll_md_exp);
    if (uuid != NULL)
        sb->s_dev = get_uuid2int(uuid->uuid, strlen(uuid->uuid));

It _looks_ like this is used in the NFS code to generate the FSID via FSID_DEV, even though it appears we _could_ be using FSID_NUM if there was a way to set ex_fsid from the filesystem so that envode_fsid() and mk_fsid() work properly:

enum fsid_source fsid_source(struct svc_fh *fhp)
{
    if (fhp->fh_handle.fh_version != 1)
        return FSIDSOURCE_DEV;
    switch(fhp->fh_handle.fh_fsid_type) {
    case FSID_DEV:
    case FSID_ENCODE_DEV:
    case FSID_MAJOR_MINOR:  
        if (exp_sb(fhp->fh_export)->s_type->fs_flags & FS_REQUIRES_DEV)
            return FSIDSOURCE_DEV;
        break;
    case FSID_NUM:          
        if (fhp->fh_export->ex_flags & NFSEXP_FSID)
            return FSIDSOURCE_FSID;
        break;

static __be32 *encode_fsid(__be32 *p, struct svc_fh *fhp)
{       
    u64 f;  
    switch(fsid_source(fhp)) {
    default:        
    case FSIDSOURCE_DEV:
        p = xdr_encode_hyper(p, (u64)huge_encode_dev
                             (fhp->fh_dentry->d_sb->s_dev));
        break; 
    case FSIDSOURCE_FSID:
        p = xdr_encode_hyper(p, (u64) fhp->fh_export->ex_fsid);
        break;  

static inline void mk_fsid(int vers, u32 *fsidv, dev_t dev,
                           ino_t ino, u32 fsid, unsigned char *uuid)
{                       
    u32 *up;
    switch(vers) {  
    case FSID_DEV:  
        fsidv[0] = (__force __u32)htonl((MAJOR(dev)<<16) |
                         MINOR(dev));
        fsidv[1] = ino_t_to_u32(ino);
        break;
    case FSID_NUM:
        fsidv[0] = fsid;
        break;

I recall some (possibly ancient) reason why it wasn't (possible to be?) done that way.  I have no idea how we would convince NFSd to use FSID_NUM from llite, and I don't see any mention of "fsid", or know how to set "ex_fsid" from the Lustre code.

Note that we do *not* run NFS re-export testing during normal patch review, so we shouldn't take the Maloo +1 as proof that clustered NFS re-export is still working properly.

nfsd certainly has changed behaviour over the years.

Currently, the preferred way for a filesystem like lustre to provide an fsid for the filehandle is through the f_fsid field of the statfs64() syscall.
mountd will try blkd_getno_to_devname() to see if it knows a UUID for the filesystem.  If that succeeds it is usually more reliable (except for btrfs).  If it fail, statfs64() is used.

It appears that lustre() uses an appropriate value in statfs64(), so I think this should all work properly.
Testing is, as always, still a good idea.

--- Message (Neil Brown, 2019-11-20 22:07:03.000000000) ---

> Are you really sure about this?
Yes.
Since about 2008, a couple of years after the bug you linked, nfs-utils has been providing a UUID when one isn't explicit in /etc/exports, so the kernel never has a chance to care about FS_REQUIRES_DEV.
If libblkid cannot find a device matching the st_dev of the filesystem (which it won't for lustre), mountd uses statfs64() to generate a uuid.
Lustre provides the same uniqueness for the fsid returned by statfs64 as it does for the dev_no used were it to depend on FS_REQUIRES_DEV.
