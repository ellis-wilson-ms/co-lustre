LU-11389 lnet: increase lnet transaction timeout

Increase the new LNet Health transaction timeout to the original
50s value, to avoid spurious lnet-selftest failures and expected
false timeouts under load.

Fix the lnet_transaction_timeout module parameter description.

Patch Set 6: Code-Review+1

To expand on my previous comment, if the code is currently experiencing failures with a 5s timeout (which IMHO is definitely too short to have for heavily loaded servers and networks) then increasing the timeout to 50s is definitely an improvement.

It _might_ not be enough for a very large system (in some cases Lustre adaptive timeouts go as high as 900s), but if there low-level LNet messages that can satisfy the 50s timeout without causing failures then this is OK.

So, in summary, I'm not against landing this patch, especially because we are seeing a lot of lnet-selftest failures today, and I suspect that heavily-loaded at-scale testing will also see problems with the current value.

An improvement would be to get the AT peer timeout from Lustre and use that on a per-message basis, rather than having a fixed timeout.

Patch Set 7: Code-Review-1

There are a couple of issues with making the timeout that large:
The intent of health is to be able to fail quickly enough in order to retry. If you set the transaction timeout to 50s, you're not going to have any time to retry. Retry occurs mostly when you try to send but the LND times out on sending. Please remember, the LND timeout is per LND message and it's not per RPC message. 

Also note that RPC timeouts (which rely on successful transmission of an LND message) is already about 5 seconds. So setting the LND timeout to 50s doesn't make sense. The RPC would've failed long before the LND timeout kicks in, and that's what we mostly see in timeout error reports. RPC timeouts multiple times, then finally the LND reports timeouts back up to the ptlrpc layer.

It's a better setup to keep the lnet transaction timeout short, which will allow LNet to retry sending the message multiple times before declaring it a lost cause.

And given that the RPC timeout is 5 seconds, the LND timeout should be below that.

Question: are these errors only with LNet selftest? Are they seen with any file system tests? Selftest doesn't use ptlrpc.

--- Message (Amir Shehata, 2018-12-05 00:58:58.000000000) ---
Patch Set 7: Code-Review+1

I'll +1, but we need to have another patch to properly resolve this. Since health is off by default having a 50s transaction timeout should be ok for most cases. But once health is activated, we need to be more aggressive with retries. I'll work on a patch to introduce two sets of defaults one for active health and the other for inactive case. When health is turned on the health defaults will take effect, and can be further tuned by the user.
