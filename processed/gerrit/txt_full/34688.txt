--- Commit Message ---
LU-12098 mdd: explicitly clear changelogs on deregister

In case of MDS crash in the middle of changelog_deregister, the system
can end up with the changelogs user deregistered, but the changelog
entries not actually cleared. Then the only way to get rid of the
remaining changelogs not used anymore by any user is to register a new
changelogs user and then deregister it.
To protect from this scenario, explicitly clear changelogs used by the
user, before actually deregistering it.

Also add recovery-small test_136 for non-regression purpose.

(3 comments)

After this change, if the user invokes 'lctl changelog_deregister' and the MDT goes down while the changelog records are being cleared, the changelog user will still exist and changelogs will still be growing but the user may not realize it.

I cannot think of a way to prevent that without much bigger changes though.

How about putting a note in the manpage under changelog_deregister warning the user that the process may take a long time, and need to be run again if interrupted by failover or shutdown of the MDT?

nit: Commit message "To protect from this scenario" might be better phrased "To reduce the likelihood of this scenario".

Did you consider them already?

--- Message (Sebastien Buisson, 2019-04-18 09:27:19.000000000) ---
Patch Set 1:

(4 comments)

With this patch, if while doing a changelog_deregister the MDS crashes during the newly added clear phase, then it will be easy to recover from this situation by issuing again the changelog_deregister command, because the consumer is still registered. After all, it makes sense to run again a command that crashed, and it will clean up remaining entries if any, and then actually deregister the consumer.

If the MDS crash occurs after the newly added clear phase, then we could end up with the consumer being deregistered, or not. But we are sure that there are no pending changelog entries with no "associated" consumer. Again, re-running the command that crashed makes sense, and it could just say that no corresponding changelog user exists, or proceed to its de-registration.

So in all cases we are protected from remaining pending changelog entries, and simply re-issuing the changelog_deregister command makes it possible to actually clean things up.
