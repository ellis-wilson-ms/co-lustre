LU-15909 lnet: Refactor lnet_find_route_locked

Refactor lnet_find_route_locked() to take lnet_send_data as an
argument.

lnet_find_route_locked() is responsible for finding the peer NI of an
lnet router to use as a next hop. It takes several pointer arguments
to be used as return values. One is for the selected router peer NI.
We can use the lnet_send_data::sd_gw_lpni field for this instead.

Another is for the lnet_route that has the highest sequence number,
and the function return value is the lnet_route corresponding to the
router peer NI that was chosen. The only thing
lnet_find_route_locked()'s caller does with these lnet_route objects
is use them to update the sequence numbers. lnet_find_route_locked()
can instead update these sequence numbers itself. It may be the case
that the sequence numbers are incremented even though we don't end
up completing this send (e.g. the send could be queued for discovery),
but this is rare and harmless. On a busy system the general round
robin behavior should still hold.

lnet_find_route_locked()'s other function arguments are already stored
in the lnet_send_data object for the send.

lnet_find_route_locked() is renamed to lnet_select_gw_lpni() to better
reflect its purpose.

lnet_compare_routes() is modified to consider whether the routes
being compared are preferred (via UDSP). This helps simplify the
loop in lnet_select_gw_lpni().
