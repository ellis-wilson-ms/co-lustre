LU-16357 obdclass: inform remote node to dump debug log

Currently libcfs_debug_dumplog() can be called to dump debug log of
current node for debug purpose, but sometimes we want to dump debug
log of other nodes as well.

This can be viewed as an extension to libcfs_debug_dumplog(), and it
has three interfaces to dump debug log:
1. lu_client_dumplog(struct lu_context *ses)): inform client to dump
   debug log, this can be called by MGS, MDS and OST code, and it can
   be called in any layers, e.g. mgs, mdt, mdd, lod and etc.
2. lu_mdt_dumplog(int index): inform MDT(s) to dump debug log, index
   -1 means all MDTs, this can be called by client and MDS in any
   layers as well.
3. lu_ost_dumplog(int index): inform OST(s) to dump debug log, index
   -1 means all OSTs, this can be called by client and MDS in any
   layers as well.

The following changes are made in the code:
* add ptlrpc_body.pb_flags MSG_DUMP_LOG to inform peer to dump debug
  log, once MDT/OST receives request with this flag, or any node
  receives reply with this flag, it will dump debug log immediately.
* add per session data lu_ses_data, which is used to pass data between
  layers in MDT/OST, it contains a field sd_client_dump_log which
  indicates whether current session should inform client to dump
  debug log, this is used by lu_client_dumplog().
* lu_mdt/ost_dumplog(int index) is built upon pinger, and they will
  force a ping immediately to target to dump debug log, if index is
  set to -1, ping all MDTs or OSTs.
* add sanity 60k 60l to verify the interfaces, and they also act as
  examples.

I think the idea is good, but I'm not sure the interface makes sense. It looks like the fail_loc needs to be set, then an RPC needs to be sent to a specific server to trigger the dump?  This is not very usable for the admin or script to need an extra RPC to be sent to a specific server, though it is OK for code internally. 

One option would be to add an extension to "lfs debug_kernel --remote <NID>" or similar that sets the DUMP flag and sends a ping RPC to the given NID. It should also be possible to pass something like "MDT0000" to trigger on that MDS, or "mds" or "oss" to trigger on all MDS/OSS.

Also, there should be a way to enable/disable this on the servers, so that containers running as root on the clients cannot trigger log dumps in the servers and fill up the filesystem.  That could be a server side tunable param, or tied to the nodemap "admin" property.

Patch Set 9:

> > Patch Set 8:
 > >
 > > Can we make a command for lctl to dump {mdt,ost} logs from client
 > > by manually?
 > 
 > What benefit is this lctl command compared to ssh to MDS/OSS and
 > run lctl there?

The benefit of having "lctl debug_kernel" do this instead of SSH, is that most clusters do not allow clients to run ssh as root to the server nodes.  This is a very limited interface and unlikely to cause security problems, unlike SSH.

I agree that having an "lctl" interface for this would be very useful, as I'd suggested in my initial review.

 > One option would be to extend "lfs debug_kernel --remote <NID>" or
 > similar that sets the DUMP flag and sends a ping RPC to the given NID.
 > It should also be possible to pass something like "MDT0000" to trigger
 > on that MDS, or "mds" or "oss" to trigger on all MDS/OSS.
 >
 > Also, there should be a way to enable/disable this on the servers, so that
 > containers running as root on the clients cannot trigger log dumps in the
 > servers and fill up the filesystem.  That could be a server side tunable
 > param, or tied to the nodemap "admin" property.  It should be off by default.

It would have been helpful to have this interface for debugging this week, so that if there is an application-level error on the client, it can trigger a debug log dump on the servers immediately to capture the log history.  Otherwise, it is too complex to collect debug logs on all of the nodes at one time.

We might also allow an option (off by default) that would  allow non-root users to trigger a debug log dump, though they shouldn't necessarily be able to read/access the dumped log themselves.  That would also simplify application-level debugging since no root access is needed to capture the logs.
