--- Commit Message ---
LU-12421 lfs: fix ladvise range

lfs ladvise mistakenly treats ranges like 0-1M as [0-1M],
requesting action on 0,1045876, but this is actually 1M+1.

We just need to subtract 1 from 'end' to reflect that
ranges are [x,y), not [x,y].

This is easiest to test with lockahead, as incorrect ranges
show up when requesting adjacent locks, such as 0-1M and
1M-2M.

Add a general set of simple lfs ladvise lockahead tests.

(1 comment)

I think it is free for us to choose one of the two meanings for "-s 0 -e 1048576". The first possible meaning is [0, 1048576] with 1048576 included, the second one is [0, 1048575] with 1048576 excluded. (I prefer the seconds one as what this patch is doing.)

And we are free to choose one of the two possible meanings of "-s 0M -e 1M" too: [0, 1048575] or [0, 1048576]. But no matter which one we choose, it should be the same with "-s 0 -e 1048576". Otherwise, it would be a very big suprise when the user figured out these two expression has different meanings. The reason is: 1M == 1048576 Byte is so obvious and everyone would take that for granted. And thus, users would naturely replace 1M with 1048576 in the command line without thinking about it. And if the behavior becomes different just because of this replacement, users would be confused.

By contrast, I think in the patch, we need to define the precise meaning of "-s $START -e $END". If we can always say "$END byte is NOT included in the ladvise rage", that is very clear to the users and they will be able to write or change the command line without any ambiguity. Potentially, we could support command line like "-s 100G10M9K4 1T9G8K9" with precise definition.

Because of this reason, I don't think we should use differen meaning for "-e 1M" and "-e 1048576".

What do you think?
