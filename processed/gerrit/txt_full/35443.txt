LU-12522 llite: Remove fname from dirty page discard warning

Retrieving the file name in ll_dirty_page_discard_warn() sometimes
causes the calling ptlrpcd daemon to self deadlock.

The daemon issues the warning when it detects an eviction and
there are dirty pages that will be lost. At the point the warning is
issued, the daemon only has access to the page and its inode. It must
therefore do a path lookup, getting references on the dentry and
inode. If, while it holds these references, all other references are
dropped, the daemon is the last referencer and is responsible for
deleting the inode and flushing the dirty pages associated with
the inode. But the dirty pages are already being written by the failed
rpc, so the ptlrpcd daemon deadlocks itself.

The solution is to eliminate the file name from the warning message.
The message includes the fid so the file name can still be identified
if needed. Furthermore, empirical evidence from many different systems
running various Lustre versions shows that ll_dirty_page_discard_warn
seldom, if ever, successfully retrieves the actual file name. The
file name in the warning is typically blank.

well, the logic is somewhat simple.

If there's an unknown (or mostly unknown) crash and backtrace triggers inside any of the files the patch touches, it tries to comment on that line (though it does not really says how deep in the stack trace it is hoping it's obvious from the stacktrace).

Son in this case if we look into the decoded backtrace: http://testing.linuxhacker.ru:3333/lustre-reports/1045/testresults/sanity-flr-zfs-centos7_x86_64-centos7_x86_64/oleg34-client-vmcore-decoded-bt.txt

we can see that:

 #22 [ffff8800c577fda0] ll_getattr_dentry at ffffffffa0d68c42 [lustre]
    /home/green/git/lustre-release/lustre/llite/llite_internal.h: 1331

So it's sort of doing it's thing as designed. I guess I can try to trim it so that it does not happen beyond certain depth and instead posts as a non-line comment, but since it was a newish crash at the time (untriaged, seen less than 21 times) it would have posted anyway since the assumption is new crashes are usually related to the patches they happen in. In ideal world, anyway.
