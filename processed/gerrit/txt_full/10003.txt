--- Commit Message ---
LU-4841 osc: revise unstable pages accounting

A few changes are made in this patch for unstable pages tracking:

1. Remove kernel NFS unstable pages tracking because it killed
   performance
2. Track unstable pages as part of LRU cache. Otherwise Lustre
   can use much more memory than max_cached_mb
3. Remove obd_unstable_pages tracking to avoid using global
   atomic counter
4. Make unstable pages track optional. Tracking unstable pages is
   turned off by default, and can be controled by
   llite.*.unstable_stats.

Comments from the original unstable page accounting patch inspection:

Andreas Dilger		2013-02-08
 
 I guess there isn't an API for "increment that counter by
 page_count"? With larger BRW RPC support landing (4MB now,
 up to 32MB in the future, 8192 pages) this is a lot of
 increments that could be avoided.

 Actually, I see helper macros add_zone_page_state() and
 sub_zone_page_state(), dating back to 2.6.18 vanilla
 kernels, that could be used instead.
	
Prakash Surya		2013-02-13

 Is it safe to assume that each page in the request is in
 the same zone? (i.e. page_zone(bd_iov[i]) will return the
 same for all indexes)?

 I'd hate to introduce a subtle accounting bug by assuming
 all pages will be in the same zone as index 0, when in
 fact that's not the case.

What about traversing the page list and accumulating the pages in the same zone as the first page, and then calling {add,sub}_zone_page_state(page_zone(bd_iov[first])) only once per zone?  I would imagine that in the common case the thread dirtying a file would commonly be scheduled in the same zone so the pages only need to be scanned once.  To make this iteration efficient in the case of multiple zones, it should save the index of the first page NOT in the same zone and only re-scan the remaining pages (if any).  Something like:

        next = 0;
        do {
                struct zone *first_zone;
                first = next;
                
                first_zone = page_zone(desc->bd_iov[first].kiov_page);
                for (i = first + 1, zone_pages = 0; i < page_count; i++) {
                        if (page_zone(desc->bd_iov[i].kiov_page) ==
                            first_zone)
                                zone_pages++;
                        else if (next == first)
                                next = i;
                }
                add_zone_page_state(first_zone, NR_UNSTABLE_NFS, zone_pages);
        } while (next != first);

The other question is why we implement our own limit for unstable pages, if the kernel already has such a limit?  We already impose a limit on the total number of cached pages ourselves, so it makes sense to just leave the unstable page tracking to the kernel?

"What ... Something like:"

I've implemented a similar patch and posted at LU-4841 but I can still see about 5% performance regression; it's okay for me to track it this way. However, this is related to my next comment.... 

"The other question is ... the kernel?"

The problem boils down to the definition of caching pages by Lustre, i.e., if we count unstable pages as part of caching pages. I tend to think we should because they're used by Lustre and nobody else can use them until the transactions are committed. The 2nd reason is that kernel can't do anything if there are too many unstable pages existing, but Lustre can intervene in the problem earlier by sending SOFT_SYNC request.

Patch Set 2:

First off, unrelated to the patch, the Gerrit UI seems a bit insane for general discussion of a change. How do I easily quote a previous comment and address it directly? Sigh...

OK, back on topic..

Andreas, I agree about collapsing the "many" calls into "fewer" calls would probably be good. Jinshan says he's implemented that with some success. Originally I had no data to say that complexity was worthwhile, but it looks like now we do. So, I'm all for making that more efficient.

Regarding the "Lustre unstable page limit". Currently, I don't believe we have a hard limit within Lustre. Since we're using NR_UNSTABLE_NFS, we can rely on the kernel to halt us and prevent an OOM when tht value gets large (and we do rely on the kernel).

We still need the "soft" limit and accounting withing Lustre to send soft syncs to the servers. Otherwise, it's possible for the client to have a "write, wait, write, wait, ..." effect due to the server not proactively committing transactions. As we hit the dirty limit, IO will be paused, and the writeback logic will be kicked. Since writeback doesn't do anything for us in relation to unstable pages, we just have to wait for the server commit and last_committed to be piggy backed on a ping. Not ideal from a performance perspective (but at least we don't OOM).

--- Message (Andreas Dilger, 2014-05-07 18:01:04.000000000) ---
Patch Set 2:

Jinshan, I think it makes sense to keep the NR_UNSTABLE_NFS handling from the kernel in order to better handle the case of application memory usage and throttling the generation of dirty data.  This is different than the Lustre dirty page limit, because that doesn't take into account memory that is used by the application, and is only "3/4 of all RAM".

I think with the more efficient NR_UNSTABLE_NFS acounting we can have the kernel handle the balance between the application memory usage and dirty pages, but our own accounting can cause SOFT_SYNC requests to be sent early enough.

Ideally, if we could access the kernel's NR_UNSTABLE_NFS counter we wouldn't have to track this ourselves, and it would have the benefit to do accounting across all filesystems instead of only on a per-filesystem basis.

Also, upon closer inspection, I really don't like the way the tuning of 'cl_check_unstable' is *only* done on a per-OSC basis. We really need a better way to set this.

Imagine an admin trying to set this to be enabled by default on a given node that is persistent across reboots. They'd have to set up a post-mount script that walks the proc namespace enabling this for each OSC on the client. And this is assuming the client is connected to all OSTs at the time the script is run; what would happen if an OST is down, or is added later, or something else? It would be left disabled for that OSC, right?

I can think of a couple better options off the top of my head:

    a. Add a filesystem level tunable in procfs; e.g. llite/*/unstable_stats

This can be queried at OSC initialization time to set the OSC state. We could keep the per-OSC tunable or remove it, I'm indifferent.

    b. Allow a module option to be specified to override the default globally

This would let the admin to override the default using a modprobe.conf file, and the variable could be queried at initialization time. Of course this wouldn't let the admin change the value for already initialized OSC.
