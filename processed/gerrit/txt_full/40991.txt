--- Commit Message ---
LU-14187 osd-ldiskfs: fix locking in write commit

Restart transaction in osd layer break rules that
locks are taken afer transaction start.

This patch try to fix by moving transaction restart
to OFD layer.

We will record how many extents we declare with
@oh_declared_ext. And during IO we check if it
run out before we restart whole transaction.

We track an average extent bytes per filesystem and
use it as a hint during declare write commit. This
will avoid us restart transaction ofen if filesystem
is heavily fragmented.

Fixes: 0271b17b ("LU-14134 osd-ldiskfs: reduce credits for new writing")

--- Message (Andreas Dilger, 2020-12-21 07:34:34.000000000) ---

> declaration overhead isn't a problem, IMO - this again a very
 > specific case with lots of small fragmented chunks.

Alex, isn't the real problem that with a large RPC, even if the incoming pages are contiguous, we don't know if they will be contiguous on disk or not, so we need to reserve credits for the worst case scenario - that each block is allocated with a different bitmap, block group descriptor, etc.  This is made *much* worse with snapshots, because each bitmap, group descriptor, etc. may also need to be copied (== allocated) for the snapshot file, and each of _those_ blocks may be randomly allocated.

However, in the common case, the pages will be mostly contiguous, so the huge overhead is not needed.  If we can reliably handle restarting the transaction with more credits, but only in the very rare case where the writes are fragmented, then this is a good tradeoff.

It would even be OK for the snapshot metadata copies to the snapshot file to be done in the first transaction, and then restarted, since those do not need to be atomic with the main transaction.  At worst, there will be an identical copy of the metadata blocks in the snapshot file (that were not modified yet in the main filesystem because of the restart), but that is not harmful.

Andreas, probably I misunderstand something, but I don't see how the approach changes that - if we want to be able to allocate/submit large IOs we still have to declare a lot. for example, we've got 4MB contiguous write or 1K pages., like you said, we don't know how many discontiguous blocks that will take, so we reserve propotionally to this 1K (bitmaps, gdps, extents) - otherwise we reserve for, say, 256 blocks at most, then we have to check credits and allocate more, but this is another call to the allocator and all is done in hope that nobody else crosses the same area or blocks can be grabbed by another thread. preallocation helps to some extent, but it's not guaranteed. IOW, large allocations need large transactions. we can improve that with bigalloc, some heruistics like "the more fragmented fs, the more credits", but generally this is a real limitation.
