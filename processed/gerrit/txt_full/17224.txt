--- Message (Emoly Liu, 2015-11-18 07:21:36.000000000) ---
Patch Set 1: Code-Review-1

Lixi, we do cfs_hash_putref() in nrs_xxx_stop(), which will be called when policy fails to startup. Did you hit any memory leak?

--- Message (Li Xi, 2015-11-18 07:55:03.000000000) ---
Patch Set 1:

Hello Emoly,

The problem is if nrs_policy_start_locked >>> policy->pol_desc->pd_ops->op_policy_start fails, then the policy is in the state of the policy will be NRS_POL_STATE_STOPPED. And the policy->pol_desc->pd_ops->op_policy_stop() won't be called later. And also, it seems nrs_policy_stop_locked() or nrs_policy_stop0() will only be called when the policy status is NRS_POL_STATE_STARTED. In this sense, policy->pol_desc->pd_ops->op_policy_stop() won't be called in the future in any way.

--- Message (Li Xi, 2015-11-18 07:59:26.000000000) ---
Patch Set 1:

Also, it doesn't looks right that a function allocates memory and quit without freeing it. Sorry, I didn't produce the problem. But it should be easy to do so. Just add a debug line which returns failure in nrs_tbf_rule_start() should work.

--- Message (Emoly Liu, 2015-11-18 09:37:27.000000000) ---
Patch Set 1:

You reminded me that besides TBF, other policies would have the same issue if 
nrs_policy_stop0 is called as long as  policy->pol_state == NRS_POL_STATE_STOPPING. If so, I suggest we should check the state setting in nrs_policy_start_locked().
