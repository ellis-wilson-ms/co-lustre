LU-1669 llite: Reduce contention on lli_size_sem

Testing shows there is still some low hanging fruit to be had in
terms of increasing the write performance of a single shared file
workload, using many threads on a single machine. With many threads
concurrently writing to a single file, the ll_size_sem becomes a
limiting factor to write throughput.

This change attempts to remove this bottle neck by changing the
ll_size_sem (which is essentially a mutex) to be a rw_semaphore.
This allows multiple threads to read the inode size concurrently,
and the write lock only needs to be taken when the inode size
changes (which is an infrequent event for many single shared file
workloads). Thus allowing much better concurrency and parallelization
out of the vvp_io_commit_write function.

Performance testing shows this change to garner a sizeable
performance boost to write bandwidth. Using FIO on a single machine
with 32 GB of RAM, write performance tests were run with and without
this change applied; the results are below:

+--------+---------+---------+--------+--------+--------+--------+
|    fio v2.0.13   |            Write Bandwidth (KB/s)           |
+--------+---------+---------+--------+--------+--------+--------+
| #Tasks | GB/Task | Test 1  | Test 2 | Test 3 | Test 4 | Test 5 |
+--------+---------+---------+--------+--------+--------+--------+
|   1    |   64    |  452446 | 454623 | 457653 | 463737 | 441509 |
|   2    |   32    |  850318 | 565373 | 602498 | 733027 | 703742 |
|   4    |   16    | 1058900 | 463546 | 529107 | 976284 | 986097 |
|   8    |    8    | 1026300 | 468190 | 576451 | 963404 | 968389 |
|  16    |    4    | 1065500 | 503160 | 462902 | 830065 | 959462 |
|  32    |    2    | 1068600 | 462228 | 466963 | 749733 | 890119 |
|  64    |    1    |  991830 | 556618 | 557863 | 710912 | 847302 |
+--------+---------+---------+--------+--------+--------+--------+

 * Test 1: Lustre client running 04ec54f. File per process write
           workload. This test was used as a baseline for what we
           _could_ achieve in the single shared file tests if the
           bottle necks were removed.

 * Test 2: Lustre client running 04ec54f. Single shared file
           workload, each task writing to a unique region.

 * Test 3: Lustre client running 04ec54f + I0023132b. Single shared
           file workload, each task writing to a unique region.

 * Test 4: Lustre client running 04ec54f + I0023132b + I71e060c1.
           Single shared file workload, each task writing to a
	   unique region.

 * Test 5: Lustre client 04ec54f + I0023132b + I71e060c1 + this
	   patch. Single shared file workload, each task writing to
	   a unique region.

As you can see, this patch and the other two patches in this series
(I0023132b and I71e060c1) increase the write bandwidth for this
workload by factors ranging from ~1.5 to ~2, depending on the number
of threads. This patch alone increases the write bandwidth by about
20% at the highest thread count workload tested (64 threads).

--- Message (Patrick Farrell, 2016-10-07 16:36:25.000000000) ---
Patch Set 5: Code-Review-1

Hm.  At least in my testing on 2.7 and newer, the lli_size_mutex (the current name for this lock) has never been a point of contention.

Not sure why it was for Prakash or what changed.  I see really nasty contention in the page cache at the thread counts where he's claiming an improvement from this patch...  And I also see much, much higher total throughput, though I'm not sure how he had things configured with regards to striping, etc.

The change itself seems sound - rw_semaphores are good, having no fairness issues (and I think having a comparable fast path), I think they're pretty much always a gain where they can be used - but I haven't seen a need for it.  (And perhaps we also don't have the kind of reader/writer ratio where it would matter to make this change.)

I just see no reason for this right now.
