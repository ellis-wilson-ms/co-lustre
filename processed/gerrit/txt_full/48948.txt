LU-16262 tests: Remove sanity-gss.sh

Purpose of sanity-gss is to test just the
GSSAPI code itself. This is done by making
use of the gssnull/null security flavor.
Currently, this is exercised through
non-regression tests with SSK and therefore
sanity-gss.sh is not required. This patch
removes sanity-gss.sh from repo due to above
reasons.

Patch Set 1: Code-Review-1

This is wrong. One test's result should not depend on the flavor global setting. If the global call to set_flavor_all is changed, we do not want to have to change individual tests.

Have you considered using a flavor different from null?

That being said, I do not think it is worth investing time on sanity-gss. We already have sanity-sec, which is functional and exercises the SSK feature. Moreover, sanity, recovery-small, sanity-sec and sanity-selinux are run with SHARED_KEY=true as part of review-dne-selinux-ssk-part-1 and review-dne-selinux-ssk-part-2 test sessions. Which means SSK is pretty well tested today.
On the contrary, Kerberos support is not exercised as part of Maloo testing today. There is sanity-krb5, but we are lacking a KDC in the Automated Test infrastructure. This aspect is currently being addressed, but I am pretty sure we will hit issues with individual tests in sanity-krb5, as this script has not seen any significant update since mid 2019 (see commit 1155c2fee4).
So it would be much more useful to spend time on sanity-krb5 rather than on sanity-gss. It looks like some tests in sanity-gss also exist in sanity-krb5 (for instance tests on 'lfs flushctx'). And maybe it could be useful to add to sanity-krb5 some tests taken from sanity-gss, if proven to be meaningful.
