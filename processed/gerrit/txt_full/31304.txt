LU-10667 llite: Always real_close writeable file

When a file is open by handle, it uses a CW lock for NFS
open cache related reasons.  This causes the client to do
a local close rather than a real close (to the MDS),
because it holds a write lock.

Unfortunately, this means closing a writeable file
descriptor which was opened by handle does not decrement
the write count on the MDS, which results in ETXTBSY when
another client tries to open the file to execute it.

The solution is to always do a real close when the file is
open for write.  This may impair NFS performance slightly
but is necessary for correctness.

Patch Set 2: Code-Review-1

I think it's kind of bad to do this since it does destroy important performance crutch for nfs (did you know that NFS opens and closes the file for every NFS rpc on IO, so potentially for as little as every 1k?)

We could limit the damage some if we only do it for executable files, I guess, so there could be at least a chance somebody would try to execute them.

But I do wonder if we should just check if there's a conflicting lock and try to cancel it if so? We even have old xyratex (or even oracle?) patches for that. https://review.whamcloud.com/#/c/9063/
