--- Message (Andreas Dilger, 2018-05-03 22:50:43.000000000) ---
Patch Set 2: Code-Review-1

While I think that this patch is probably reasonable, removing this parameter completely might be a problem for some sites.  For example, clients running over WAN that have low bandwidth and should not cache too much dirty data, clients where the sysadmin wants to limit memory usage by Lustre, etc.

My recommendation would be to increase the default max_dirty_mb limit to some larger value (e.g. 256M or 1G per OSC), and then making a release with that new default instead of removing the parameter completely.  That has a similar effect as removing the limit, but doesn't prevent anyone from re-imposing the limit on their system if they need it (i.e. we have a safe fallback path instead of breaking things horribly for some users and needing to get patches/builds to them with this change reverted).

The actual limit should be related to the bandwidth of current networks (e.g. EDR IB and OPA @ 10GB/s) so that we can cache enough dirty data to keep the pipes full, but not so much that it takes 10 minutes to flush the client cache.

--- Message (Andreas Dilger, 2018-05-03 23:03:02.000000000) ---
Patch Set 2:

PS: also useful would be to see what the effect is on the bandwidth of a single client writing a single-stripe file on an IB/OPA network (possibly with multiple client threads and "fake write" on the OSS so that client CPU and OSS bandwidth is not the limiting factor).

I suspect that 32MB is a limiting factor for this benchmark, but it may well be that there are other limits and increasing the cache size doesn't help after a certain point.  Note that the client's max grant is normally 4*max_dirty_mb, so this may also play into the performance.

Also, we'd need to see if a client has 10s of outstanding writes on a slow network (bandwidth * 10), and then the user tries to read a file if the read is processed in a timely manner.  Otherwise, we'll also need improvements in the IO scheduler to prioritize reads over writes.

Also, if grant is to become the primary control mechanism for client dirty cache size, we need to finish patch https://review.whamcloud.com/23203 "LU-8708 osc: enable/disable OSC grant shrink" (or similar) so that a single client can't consume all of the grant on a filesystem.
