--- Commit Message ---
LU-18082 utils: fid2path support for path exceeding PATH_MAX

PATH_MAX affects the path length of a single pathname, but it is
possible to exceed this by eg. creating a subdirectory tree, then cd
into it, then create a deep tree below this using a relative pathname.

So fix ll_fid2path and lfs_fid2path to not limit the output buffer to
PATH_MAX.

Add sanity test_154i to exercise this.

This should add a test case that creates a deep directory tree to confirm this patch is fixing the issue.  I tested locally and was able to create a directory tree over 5200 chars long with a loop:

        long=thislongpathnameisforaverydeepsubdir
        max=$((4096 / $(wc -c <<< $long) + 5))
        for (( depth = 0; depth <= max; depth++)); do
                mkdir -v $long$depth || error "..."
                cd $long$depth
        done
        fid=$(lfs path2fid .) || error "..."
        path=$(lfs fid2path $MOUNT $fid)

though create performance slowed down as I got deeper into the tree.  A similar slowdown was seen with a local ldiskfs filesystem as well (though to a lesser extent), so it seems like something in the kernel.

Note that it is faster to create a deep directory tree with a few subdirectories with long names than many subdirectories with a short name, since the slowdown appears related to the number of directory levels and not the length of the pathname. 

I also tried to see if the relative pathname worked with fid2path, but this crashed.  It should have given the last component of the path:

        # lfs path2fid .
        [0x240000404:0x85:0x0]
        # lfs fid2path $(cd ..; pwd) 0x240000404:0x85:0x0
        Segmentation fault
