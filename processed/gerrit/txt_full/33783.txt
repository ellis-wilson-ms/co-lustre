--- Commit Message ---
LU-10070 lod: SEL: Implement basic spillover space

This is a barebones implementation of spillover space.
This allows the creation of extendable layout components,
which are normal layout components followed by "extension
components".  These extension components are never
initialized, instead, when i/o reaches them, the server
checks if there is sufficient space on the preceding normal
layout component, and if so, it modifies the extent of the
component to give space to the preceding component.

If there is not sufficient space on those OSTs, the special
extension space component can be removed, and the next
component of the layout is moved down to meet the existing
component.  This allows i/o to "spill over" to this new
layout component, which is expected to be on different
OSTs.

For multi-tiered systems, this makes it possible to avoid
the situation where an inner tier is low on space, but a
an outer tier has plenty, and PFL files cannot use the
space in the outer tier because the inner is full.

This patch requires the next patch in the series for FLR
support, but does not depend on the other subsequent
patches in this series.

"My suggestion was to use both the high watermark and low watermark similarly in SEL, allowing existing files to continue to expand until their OSTs hit the low watermark (to reduce layout complexity), but push new files away from full OSTs earlier, so that they don't even start to use OSTs that are below the high watermark."

Hmm.  So to be completely clear about what this would mean, all stripe allocation decisions are made by the unmodified stripe allocator.

When the a newly selected set of OSTs (for a newly initialized component) doesn't meet the SEL criteria, SEL has only two choices:
1. If there is a 'next' component beyond, SEL can skip the component it was going to instantiate and use that next component instead (It can "spill over" to that next component).  This is intended for 'oh, an intervening tier is full', an example being a DOM-FLASH-HDD layout where the flash is full(-ish), so the flash component is skipped.
2. It can *reject* the new component and continue to extend on the previous one.  It only does this if there is no following component to spill over to.  (This is the "repeat" case, where it's asking for a copy of the last component, but running it through the stripe allocator to get less full OSTs.)

So...  I guess that is a reasonable choice in the "skip a tier" situation.  In the "repeat" situation...  I guess it's still fine.

Hm.  I started writing this comment from a position of opposition to the idea, but I find that as I finish it, it seems reasonable.  Basically, I think this just means moving the space check for SEL from the low watermark to the high one, making it more aggressive than the stripe allocator itself.

This sounds like a good idea to me.  (This is going to require tweaking a bunch of tests, I suspect...  But other than that, I think it's the trivial swap from reserved_mb_low to reserved_mb_high in one function?)
