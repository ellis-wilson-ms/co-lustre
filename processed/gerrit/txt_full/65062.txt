LU-19921 target: add verification for nodemap

This patch adds gss_iam_get_nodemap(), gss_iam_install_ctx(),
gss_iam_handle_init() and other nodemap changes, preparing the
target side for GSSIAM verification and security context setup.

GSSIAM introduces a novel identity-based authentication model using
a Hybrid RPC Model. Since the data RPCs (after the initial context
establishment) use standard Null framing to avoid signing overhead,
we need a way to enforce fine-grained access control (UID/GID mapping,
fileset restrictions, etc.) on these Null-framed RPCs.

Why use nodemaps instead of directly enforcing the security context?
Lustre's nodemap framework is already the established, optimized
engine for enforcing UID/GID squashing, project IDs, and fileset
(subdirectory) boundaries. If we only attached the security context
directly to the export, we would have to implement a parallel,
redundant access control mechanism within the RPC or GSS layers
just for GSSIAM.

Instead, by creating a dynamic child nodemap for each unique
GSSIAM identity (inheriting from a parent "gssiam" nodemap) and
pinning it to the export, GSSIAM seamlessly integrates with
Lustre's native policy enforcement. The target applies its standard
nodemap rules to the incoming Null-framed RPCs effortlessly.
The dynamic nodemap is automatically cleaned up when the export
is closed or expires.
