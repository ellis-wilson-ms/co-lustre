--- Commit Message ---
LU-4209 utils: fix O_TMPFILE/O_LOV_DELAY_CREATE conflict

In kernel 3.11 O_TMPFILE was introduced, but the open flag value
conflicts with the O_LOV_DELAY_CREATE flag 020000000 added to fix
LU-812 in Lustre 2.4.  O_LOV_DELAY_CREATE allows applications
to defer file layout and object creation from open time (the default)
until it can instead be specified by the application using an ioctl.

Instead of trying to find a non-conflicting O_LOV_DELAY_CREATE flag
or define a Lustre-specific flag that isn't of use to most/any other
filesystems, use (O_NOCTTY|FASYNC) as the new value.  These flags
are not meaningful for newly-created regular files and should be
ok since O_LOV_DELAY_CREATE is only meaningful for new files.

I looked into using O_ACCMODE/FMODE_WRITE_IOCTL, which allows calling
ioctl() on the minimally-opened fd and is close to what is needed,
but that doesn't allow specifying the actual read or write mode for
the file, and fcntl(F_SETFL) doesn't allow O_RDONLY/O_WRONLY/O_RDWR
to be set after the file is opened.

We will keep the 0100000000 flag for backward compatibility until
3.13 is the oldest client kernel that is supported, but drop the
conflicting __O_TMPFILE value of 02000000 since that will cause an
error when running on newer kernels.  The 020000000 has only been
used since Lustre 2.4.0 and always in conjunction with 0100000000,
so any apps that used O_LOV_DELAY_CREATE directly instead of calling
llapi_file_create*() will still work until Linux 3.13 is used.

Patch Set 2:

Any ideas on how to solve this problem cleanly?

We could check for the absence of O_NODELAY, but that isn't very robust if O_NOCTTY is used by some other application. I figured O_NOCTTY would only be used by applications that know they are opening a tty device, but I guess it is also used by apps that might open random files and want to be safe in case they do open a tty device. 

A similar mechanism would be to check a combination of flags, but I don't see another one that is obvious. I looked at FASYNC, but it doesn't look as safe as I'd like. 

A second option is to fall back to creating the objects with a default layout if no ioctl() is called. I sort of thought that was the default behaviour in the past, but I haven't looked at the details for a long time.

There is also the option of actually trying to get a reserved flag in the upstream kernel, but I expect a reply like "don't do that, system calls are cheap" that isn't really helpful.

My preference at this point is the second option of creating objects with the default layout (as if LL_IOC_SETSTRIPE was called with all default options) if the file does anything on the client.

While we can change the test tools, that does not fix an other tools potentially out there.

So I think we should just have O_LOV_DELAY_CREATE to always represent the current mask.

We definitely should just drop the 0100000000 flag completely and never ever remembering about it.

Hm, I imagine since 020000000 is just dropped by the kernel, it's no big deal to actually combine 020000000 and O_LOV_DELAY_CREATE_MASK to get O_LOV_DELAY_CREATE if we really want to, but that's about it.
