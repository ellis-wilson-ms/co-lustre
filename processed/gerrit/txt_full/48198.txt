LU-16091 enc: S_ENCRYPTED flag on OST objects for enc files

Add a dumb encryption context on OST objects being created, when the
LUSTRE_ENCRYPT_FL flag gets set in the LMA, for ldiskfs backend
targets. This leads ldiskfs to internally set the LDISKFS_ENCRYPT_FL
flag on the on-disk inode. Also, it makes e2fsprogs happy to see an
enc ctx for an inode that has the LDISKFS_ENCRYPT_FL flag.

Add a dumb encryption context on OST objects being opened, if there is
not already one, for ldiskfs backend targets. This is done by adding
the LUSTRE_ENCRYPT_FL flag if necessary, at the same time as atime
gets updated. It is some sort of live self-check that fixes OST
objects created with an older Lustre version.

Enhance lfsck to detect and fix OST objects belonging to encrypted
files that are missing the encryption flag. This is implemented in the
MDT-OST consistency routine, as part of the layout checking.

Also add sanity-sec test_62 and sanity-lfsck test_42 to exercise this.

Note this patch does not add any dumb encryption context on OST
objects when the backend is ZFS.

> Can you please add a test case that is calling run\e2fsck on an MDT
 > and OST with encrypted files, so that we know this is working, and

Will do.

This patch also needs to be updated because it currently does not provide a way to set the encryption flag on already existing files.

Moreover, I found a problem with it. In this current implementation, it only works when the client issues a setattr on the file. But I hit some use cases where the client can write content to a file without doing any setattr. For instance with the following command, if $FILE does not exist yet.

   dd if=/dev/zero of=$FILE bs=1 count=1 conv=fsync


 > I'm a bit worried that this may further complicate our lives until
 > the FSCRYPT backup/restore is finished, since it means OST objects
 > will be inaccessible.

That's debatable. Without this patch, doing an OST backup at the ldiskfs file system level would apparently succeed for encrypted files, but only bytes up to the OST object size would be backed up. So you would basically get a corrupted backup of your encrypted files, without any possibility to decrypt or restore them because of missing encrypted blocks after EOF.

At least, with the encryption flag set on encrypted files' OST objects, and in the absence of complete implementation of back/restore for enc files, you would realize that those objects could not be backed up.

To better qualify this, there is the noticeable exception of files whose size is aligned on LUSTRE_ENCRYPTION_UNIT_SIZE == 4096 bytes boundaries. In this case of course, there is no "hidden" data after EOF, and backup/restore of OST objects at the ldiskfs file system level would work as-is.
