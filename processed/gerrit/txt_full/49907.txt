LU-16524 sec: enforce rbac roles

There are 5 different rbac roles defined via nodemap:
- byfid_ops, to allow operations by FID (e.g. 'lfs rmfid').
- chlg_ops, to allow access to Lustre Changelogs.
- dne_ops, to allow operations related to DNE (e.g. 'lfs mkdir').
- file_perms, to allow modifications of file permissions and owners.
- quota_ops, to allow quota modifications.
Enforce these roles by checking the value of the 'rbac' nodemap
property on server side and returning -EPERM if operation is
forbidden.

Add sanity-sec test_64* to exercise these capabilities.

open_by_handle_at in the kernel uses do_handle_open.  (Handles are just fids sandwiched in to a blob of memory...)

And do_handle_open starts with handle_to_path, which turns the fh in to a dentry, (eventually) using ll_fh_to_dentry.

After that, it's just a regular open and I don't think it can be recognized.

And ll_fh_to_dentry is a local operation, or can be.

Hmmm...

Well, I think I see a way, *if this is important*.

ll_fh_to_dentry is *only* used for this, and it calls ll_iget_for_nfs.  ll_iget_for_nfs tags the dentry with lld_nfs_dentry
We can't reuse that dentry tag to recognize this - it would mean breaking NFS export on this client - but we could add our own flag here for requests from ll_fh_to_dentry

Then we could do something in ll_file_open() with a new MDS flag to tag open_by_handle_at requests, so they can be recognized separately.

So that looks doable, if possibly a bit complicated to make sure we force it to go to the MDS and not be handled locally, etc (I don't understand that terribly well)

So, what do we think?  Is it critical to block open_by_handle_at in the first version of this, or could it be left for later?  It is a local root operation that can pierce file permissions...

I wonder, separately, maybe rootsquash already blocks this.  open_by_handle_at calls do_file_open_root, which calls path_openat, would that be blocked by rootsquash, ie, would the path lookup part fail?  Is rootsquash necessary for rbac configs like this to make sense?  ie, will it always be on?  If so, we might already block open_by_handle_at with rootsquash, unless they have full permissions to the entire path.  Something you could maybe test?

> We can't reuse that dentry tag to recognize this - it would mean

That would be at best a partial solution, because it would depend on the client to set this flag, and a suitably-motivated root user could patch that flag out of their kernel module.

 > I wonder, separately, maybe rootsquash already blocks this. 
 > open_by_handle_at calls do_file_open_root, which calls path_openat,
 > would that be blocked by rootsquash, ie, would the path lookup part
 > fail?  Is rootsquash necessary for rbac configs like this to make
 > sense?  ie, will it always be on?  If so, we might already block
 > open_by_handle_at with rootsquash, unless they have full
 > permissions to the entire path.  Something you could maybe test?

Root squash should not be considered a security mechanism, because root on the client can always to "su - $user" to become any UID accessible on the client.

Open-by-FID will already do UID-based permission checks, though not path-based checks (though conceivably it could, like NFS).  That said, with nodemap, the real way to secure the subdirectory is to define non-overlapping UID/GID/PROJID ranges for each subdirectory, and then squash/block any access to IDs outside of the nodemap range.  That allows the MDS to definitively block out-of-tree access without having to do path-based checking.
