--- Message (Alexey Lyashkov, 2014-06-05 11:47:22.000000000) ---
Patch Set 2:

it's very easy - to mark request unlinked we need unlink both buffers and bulk (if need). So we can't do anything until LNet unlink event isn't received. 
so we can't resend or process reply until both buffers and bulk unlinked,
and we can't shutdown client because client_disconnect_export->ptlrpc_invalidate_import will abort in flith (ptlrpc_deactivate_import->abort_in_flight) and wait until requests will finished (aka called interpret) via 
                rc = l_wait_event(imp->imp_recovery_waitq,
                                  (cfs_atomic_read(&imp->imp_inflight) == 0),
                                  &lwi);

it's easy.

--- Message (Alexey Lyashkov, 2014-06-05 11:51:13.000000000) ---
Patch Set 2:

in general it's bug in LNet code. LNet commit MD on initial LNetGet / LNetPut call, but we don't connect such buffer to the network driver so able to unlink buffer quickly, but LNet mark MD busy and stop sending event and unlink buffer.

--- Message (Alexey Lyashkov, 2014-06-05 19:30:15.000000000) ---
Patch Set 2:

look to background https://projectlava.xyratex.com/show_bug.cgi?id=16308.

Caller calls a LNetPut but destination still inaccessible and lnet don't able to release a TX and send an event for ptlrpc. Request hold an import reference and it prohibit an release obd device reference and broke a shutdown.

Patch adds additional flag and forbid to access to the outgoing buffer and delay call interpret function (decrease an inflight rpc count) until both buffers (outgonning and incoming) and bulk buffer will released.
rq_req_unlink == 0 indicate and buffer still unreleased, and ptlrpc_client_recv_or_unlink return true if recv or unlink process in flight.
so
ptlrpc_unregister_reply via ptlrpc_client_recv_or_unlink vill forbid to call after_reply function (and modify outgoning buffer with MSG_RECOVERY flag) or prevent decrease an imp_in_flight checked in invalidate import function.
