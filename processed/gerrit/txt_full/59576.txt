LU-0000 mdd: fix immediate mirror write intent handling

Fix server-side handling of write intent requests for immediate
mirror files. The issue was in mdd_layout_change() where the FLR
state switch statement was not properly handling immediate mirror
files that have combined flags.

Immediate mirror files are created with:
  flr_state = LCM_FL_RDONLY | LCM_FL_IMMEDIATE_MIRROR (0x5)

But the switch statement was checking the full flr_state value,
so when flr_state = 0x5, it would fall through to the default
case and do nothing (rc = 0), instead of calling
mdd_layout_update_rdonly() to transition from RDONLY to
WRITE_PENDING state.

The fix extracts only the base FLR state bits (0x3 mask) for
the switch statement, so immediate mirror files with state 0x5
will correctly match the LCM_FL_RDONLY case (0x1) and call
mdd_layout_update_rdonly().

This allows immediate mirror files to properly transition from
RDONLY to WRITE_PENDING state when receiving write intent
requests, enabling direct writes to work correctly.

Before this fix:
- Write intent requests for immediate mirrors would hang
- Direct writes would result in 0-byte files

After this fix:
- Write intent requests complete successfully
- Direct writes work normally
