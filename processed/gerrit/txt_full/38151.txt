LU-13419 osc: Improve speed of enter_cache_try

When doing writes to many files, one bottleneck on a client
currently seems to be the grant code, specifically
spinning in the lock around:
osc_enter_cache_try

The contention is *just* on osc_enter_cache_try, so there's
no obvious way to refactor the lock, etc.  Instead, we can
look at where time is going in the function.

Two things that stand out:
obd_dirty_pages is an atomic, and expensive:
In my perf tracing, the add_return to this is 50% of the
time in this function.

This can be replaced with a percpu_counter.

These benchmark #s are with the earlier version of the
patch, which mistakenly replaced the atomic with a bare
unsigned long.  I'm not currently able to benchmark the
percpu_counter, but it should be similar.

mpirun -np 36 $IOR -o $LUSTRE -w -t 1M -b 2G -i 1 -F

That's 36 processes on one client, writing to separate
files.

Before patch:
5942 MiB/s
After patch:
14950 MiB/s

Looking in perf, the change is huge:
I go from spending 60% of the time in osc_enter_cache_try
to around 30%, but that's while moving 2.3x the amount of
data per second.

Patch Set 3: Code-Review-1

(1 comment)

As appealing as this patch is performance wise, I realized there is a significant flaw in the logic when I was going to ask you to add a comment by obd_dirty_pages that it is protected by cl_loi_list_lock.

The problem is that cl_loi_list_lock is a *per OSC* lock, but obd_dirty_pages is a *global* counter of all dirty pages on the client.  So holding cl_loi_list_lock may appear to be correct when you are writing to a single-stripe file, but will quickly become broken when multiple threads are writing to different files.

I think an option would be to change obd_dirty_pages to be a percpu_counter, so that it is not having the overhead of an atomic, but it is not strictly accurate at all times.  For obd_dirty_pages IMHO it is OK to be somewhat over/under obd_max_dirty_pages without anyone's hair catching on fire, as long as it doesn't go too far over the limit (possibly this can be predicated on how close we are to the limit?).

Uck, excellent point.  Sloppy on my part.

OK, so, counterpoint/thought:
Do we need obd_max_dirty_mb?

It defaults to 1/2 of RAM (except for small systems (<512MiB), where it's set to 1/4), which is *the same as the default for max_cached_mb*.

So in the default config, this limit is not doing anything.

Additionally, while it's a bit of a poor substitute, we can control max_dirty_mb on a per OSC basis.

So...  Do we need/want it at all?  I'm inclined to "eh, it doesn't seem important".  The only goal that I think can't be met with max_cached_mb is limiting the total amount of dirty data that can be outstanding.

I've heard of people proposing to do this, but I don't think I've heard of anyone doing it in production...?

Patch Set 3:

obd_dirty_page is the sum of all cli->cl_dirty_pages.

Rather than updating the global count and checking against the global limit, we could share the global limit around the the various cli.
e.g. give each cli a share of '0' to start with.  If it wants more, it takes a global spinlock and gets some more if available.
If none is available, it tries to encourage each cli to give some up.

If the limit is rarely reached and most cli stablise on a share that works for them (for a least a few seconds), then you get cheap accounting most of the time, and very expensive accounting very very rarely.

This is vaguely similar to the kernel's writeback throttling (balance_dirty_pages()).  It can be hard to understand but a key idea is that there are a sequence of escalating steps which get more costly but happen less often.
balance_dirty_pages_ratelimited() is the main entrypoint. It (if I recall correctly) keeps a per-process counter and only enters balance_dirty_pages() every so-often - unless memory is know to be tight, then it enters more often.
balance_dirty_pages() usually just checks some global counters which are probably a bit out of data, but if the limit looks like it is getting close, it sums across some per-cpu counters to get more accurate data.  If that shows that it will probably need to wait a little - then it decides to take the full cost of accurate accounting, because it has nothing else useful to do.

I don't think we should follow balance-dirty-pages exactly, but the staged escalation is probably a good pattern to follow.
