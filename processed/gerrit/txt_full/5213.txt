LU-2711 tests: Code execution between tests

Removed code execution between test declarations. The code moved
to corresponding test_<N>_setup and test_<N>_cleanup function,
see description below.

Refactored test-framework to support initialization and
de-initialization blocks: per suite, per test group and per test.
Following functions can be declared in a suite file and executed
automatically:
- tf_setup, tf_cleanup (for whole suite)
- test_<base>_setup, test_<base>_cleanup (for groups)
- test_<testnum>_setup, test_<testnum>_cleanup (for tests)

The run_test function works as "add test" and updates the test
execution list which is used by tf_run command.

The tf_run function implements standard test workflow like:
load configuration, validate parameters and generate filters,
checks configurations, setup and cleanup lustre, runs tests,
reports results.

Initially refactored sanity and sanityn tests.

Added tf-suite a wrapper to be included instead of test-framework.sh
for simplification of test suite declaration. The tf-run script
is used to avoid copy and paste workarounds to run tests from
custom location, so you can use '. tf-suite' instead of computation
of LUSTRE variable before including the test-framework.sh script.
The test-framework.sh is still available to include for backward
compatibility with in suites that are still not refactered.

Added -l list command displaying available tests in a suite
as demonstration, usage ex: lustre/test/tf-run sanity -l

The main reason is that the tf-suite script can't be found by default. But if you provide right path to the tests directory, it will work. This standard path setup in the tf-run.

  /usr/lib64/lustre/tests/auster -v -k sanity --only 0c
  PATH=$PATH:/usr/lib64/lustre/tests ONLY=0c sanity.sh
  ONLY=0c /usr/lib64/lustre/tests/tf-run sanity

as well as the following:

  (cd /usr/bin ; ln -s /usr/lib64/lustre/tests/tf-run)
  ONLY=0c tf-run sanity

Kyrylo,
I agree that this patch has taken longer to land than anyone would like.  Looking at the patch history, I see that there are long gaps when the patch was not being updated or not being reviewed.  Typically the responsibility to get attention on a patch is on the patch owner, though I also see that there are a number of cases where intermittent test failures have caused problems as well.

I suspect one reason that there has been a reluctance to land this patch is because that even though it works properly in the autotest environment, it changes how tests are run manually by developers (see my and James' previous comments about running "sanity.sh" directly fails) but there doesn't appear to be any clear benefits of this change for developers?

I think the second question is what your intention is for fixing the remaining scripts?  There are no other "test owners" to fix the existing tests to your new style of setup/cleanup.  If that isn't done as part of your changes (not this specific patch, but as follow-on patches) then it just means there will be two inconsistent ways of doing test setup/cleanup instead of only one, which would not be an improvement.  The per-subtest setup/cleanup could have been added to run_one() without introducing the whole tf_* code in this patch and breaking the ability to run tests directly, so there needs to be some long-term benefit to these changes (i.e. cleanup and simplification of the test code).

This patch has also moved a long way away from fixing "code execution between tests" as it originally started and is currently described, and should get a new patch summary like "refactor test script initialization" or similar.

In summary, I'm in favour of cleaning up the testing infrastructure, but it needs to be more than just this patch.  If you are willing to continue this work to clean up the other test scripts to use this new style of initialization then I think this patch could be landed.
