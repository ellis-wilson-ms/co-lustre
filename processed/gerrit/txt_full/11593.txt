LU-2614 build: Modified lbuild to build with MLNX OFED
The change will enable the compiliaton of Lustre using
the kernel modules of the Mellanox OFED stack.

Besides running Lustre with another OFED stack, the
Mellanox kernel modules are providing SR-IOV
capability for:

- RHEL/CentOS 6.5 and Linux kernel 2.6.32-431.29.2
- Mellanox Connect-X2 and Connect-X3 HCAs
- HCAs flashed with firmware version 2.9.1000
- Virtual machines set-up with kvm-qemu >= 0.12.1.2
- Fabric using Mellanox subnet-manager version '4.1.5'

The SR-IOV feature allows the usage of virtualized functions
(VF) within an infiniband HCA, and to assign these VF to VMs.
At the moment the 'vanilla' OFED (www.openfabrics.org) doesn't
support this feature.

The change rely on Mellanox's OFED version 2.2.1. The
Mellanox OFED distribution containing the Mellanox kernel
module source files, is downladed to the directory
specified via command line option '--kerneltree'. For
this task a new function 'download_mlnx' has been
introduced.

The content of the kernel-ib-devel RPM store the
compat header files in an other directory than the normal
vanilla OFED. The Lustre compile environment is set-up
correctly by assigning this directory location in
variable CONFIGURE_FLAGS to the configure option
'--with-o2ib' in case Mellanox OFED is used.

The former changes to lustre-lnet.m4 and o2iblnd.h became
superfluous since compat header files have been included
in the LNET source code now.

What I have in mind by modularization is that there would be a number of "implementation" modules (i.e. separate shell scripts), called, say "ib-inkernel", "ib-ofed", "ib-mlnx", "ib-truescale", etc.

Then you would have one single if/elif/elif/.../else block to source the appropriate ib-* script and that script provides the (i.e. common) methods to do the various things needed, like downloading the source, unpacking the source, setting configure command line arguments, building from the source, etc.

This way the main code is clean from ever-growing if/elif/else blocks and just calls methods, the function of which is defined in the respective ib-* file.

This is similar to how lbuild-{rhel,sles,etc.} work.
