--- Commit Message ---
LU-3682 tunefs: prevent tunefs running on a mounted device

This patch is to check if the given device is already mounted in the
following situations:
1) For ldiskfs, get the real path to the given device in case it is a
   symlink or dm device, and return EEXIST if it is already mounted.
   What's more, if the given path doesn't exist, mkfs.lustre won't
   create it automatically. So to avoid breaking the local testing,
   this patch creates the file in test-framework.sh instead.
2) For zfs, just pass the given pool name directly.
3) If the given file is already associated with a mounted loop device,
   check_mtab_entry() should return EEXIST to prevent the next loop
   setup attemp.
Also, this patch adds conf-sanity.sh test_81 to verify the fix.

--- Message (Emoly Liu, 2013-11-18 09:13:13.000000000) ---
Patch Set 3:

Oleg, this is not a bug. The new code does not create loopback files automatically. I ever thought this was a problem and reported it to LU-3991. Please see the comments from Andreas and Alex on http://review.whamcloud.com/#/c/7722/ for patch set #1 and #2.

Andreas said that "I don't think that creating loopback files automatically is a good idea. In the normal use case (instead if during testing) any typo or error in the device name will result in a loopback file being created instead if returning a useful error to the user. For example, if someone accidentally runs "mkfs.lustre /dev/sfa" instead of "sda" this will result in a loopback file being created in /dev that will initially work, but will be very slow and will later fail when the root Filesystem runs pit of space.

I would rather that testers/scripts make sure that the loopback file it created first, and allow mkfs.lustre to return an error if this us not done correctly."
