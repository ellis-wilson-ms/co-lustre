ldiskfs we also clean up more macros.

Patch Set 5:

Chris I looked into what you suggested and found it would takes a lot more surgery to make it work. Yes you can move the lustre macros that are specific for ext4 to ldiskfs-build.m4 but then you run into the issue of the config.h for ldiskfs conflicting with the config.h for lustre itself. You can rename ldiskfs config.h to ldiskfs-config.h and include it but then you run into the issue of PACKAGE_* being redefined. To get around that then you have to do zfs meta type magic to make it work. At that point is it worth the effort especially if the fsfilt layer is going away soon.

--- Message (Christopher J. Morrone, 2013-02-07 18:57:27.000000000) ---
Patch Set 5:

I did not suggest trying to include ldiskfs's config.h from lustre.  As you have learned, one should never attempt to export config.h to another package.

You do, however, need to have ldiskfs export the information in SOME reasonable way.  I created the infrastructure that would make this easy for you to export a new variable in Patch Set 2 of http://review.whamcloud.com/5005 (you really need to look at 2, patch set 3 does not have any of this).

Andreas mark that down for the complexity, largely, I think, due to the lack of consumers of the information.  Perhaps now that you _have_ a consumer that approach could be resurrected.

The key part was the creation of lustre_definitions.h.in.  That is a header that was intended to be used by both ldiskfs, and more importantly installed as part of the future ldiskfs-devel package so that lustre can easily check which features ldiskfs offers.
