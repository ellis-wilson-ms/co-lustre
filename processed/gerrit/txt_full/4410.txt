Patch Set 2: I would prefer that you didn't submit this

Adding more positional parameters makes parsing the options more complex, and also makes using this function more complex.

It would be better to allow the user to specify a FID directly instead of just of the objid for FID-on-OST:

 lctl destroy [0x240000003:0x44:0x0]

or with IDIF FIDs:

 lctl destroy [0x100010000:0x1234:0x0]

It would be easy to detect this case by checking for a FID format first, then fall back to a plain objid if no FID is found.

--- Message (Andreas Dilger, 2012-10-31 08:11:59.000000000) ---
Patch Set 2:

Niu,
while the current OSTs only need an objid today, with the landing of DNE code in 2.4 the OSTs will use regular FIDs to identify objects.  So if the interface is changing today to allow sequences, it may as well be able to handle FIDs in the future, especially if using the normal FID format makes it easier to copy FIDs from debug messages.

Note that there is no reason why allowing FIDs as input should break existing scripts, if it allows both a simple objid and a FID as input:

 lctl destroy <objid|fid> [num]

where "fid" is the normal "0xSEQ:0xOID:0xVER" format.  This can automatically be detected by something like:

        struct lu_fid fid;

        /* Check for "0xSEQ:0xOID:0xVER" first, but if that doesn't work try
         * for just a single objid for test OST objects for FID_SEQ_ECHO. */
        if (strstr(argv[1][0], ':') == NULL ||
            sscanf(argv[1], SFID, RFID(&fid)) != 3)
                fid.f_seq = FID_SEQ_ECHO;
                fid.f_oid = strtoull(argv[1], &end, 0);
                fid.f_ver = 0;
                if (*end || fid.f_oid == 0 || errno != 0) {
                        fprintf(stderr, "error: %s: invalid objid '%s'\n",
                                jt_cmdname(argv[0]), argv[1]);
                        return CMD_HELP;
                }
        }
