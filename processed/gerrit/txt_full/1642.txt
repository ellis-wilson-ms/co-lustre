--- Commit Message ---
LU-820 Increase LU_CDEBUG_LINE so that osc page messages fit.

LU_CDEBUG_LINE at 255 does not quite fit osc-page message out of
osc_page_print that is quite long.
As a result we get annoying "does not end in newline" console messages
when this happens.

Doubling LU_CDEBUG_LINE certainly should be enough.

Oleg, do you know how much memory is used here?  I'd be hesitant to double the memory allocation for a rare case.  Is it worthwhile to track down the debug message that is causing the problem and rather make that shorter?

--- Message (Oleg Drokin, 2011-11-03 05:07:28.000000000) ---
Patch Set 1:

How much memory is used where? In this offending print? seems to be under 260 bytes, which does not mean it might not be a bit higher when I am not looking.
It is likely that upping the limit to say 300 bytes will suffice as well.
I believe these "keys" are allocatd once per lustre-using process, so it should not be terribly bad most of the time.

Unfortunately splitting the call to *printer won't help at all.

The *printer fills an internal buffer (the one referenced by the key increased) until a \n is seen at the end.
So unless we want to split that one line output into several, it won't work.

Printing less data is a double edged sword too. Sure we don't need it most of the time, but what about when we do need it?
