--- Commit Message ---
LU-3382 build: clean unused link dependencies

Compiled executable Lustre files such as:
lst, lstclient, loadgen, lustre_rsync, lctl, lfs,
debugctl and ptlctl are linked against libtinfo.so.5
and libncurses.so.5, however none of the library's
symbols are used. Same issue with multiop which is
linked against librt.so.1, however uses none of the
library's symbols, and liblustre.so which is linked
against libz.so.1 and uses none of the library's
symbols as well.

The fully qualified path in "dh_shlibdeps -l $(CURDIR)/lustre/utils" solved the problem.

There is however, another issue:
The finally built packages have filename, e.g.
lustre-utils_2.4.0-1_amd64.deb, where version 2.4.0-1 is extracted from
debian/changelog [lustre (2.4.0-1)] and this version does not depend on the true Lustre version.

I think it would be a good idea
to extract the version e.g. as follows:

LUSTRE_VERSION=$(echo `git describe` | sed -e "s/_/\./g" | cut -c2-)

and update the entry by means of command "dch" in debian/rules such that build Debian packages have correct filenames
