Patch Set 5:

After investigation, I would tend to think the problem seen with the in-kernel fscrypt library is limited to file migration (sanity-sec test_52 and test_59b). In this use case, Lustre creates a temporary, volatile file to migrate data. For encrypted files, we explicitly set the encryption context of the volatile file (directly on server side) to be the same as the original file, so that data is encrypted with the same key. Problem is Lustre stores the encryption context in the "security.c" xattr, whereas fscrypt expects it to be in "encryption.c". When using the embedded llcrypt lib, it is fine as all parties use "security.c".

So I have rebased this patch on top of the latest encryption related patches on master, to see if using "encryption.c" xattr makes in-kernel fscrypt happy.
