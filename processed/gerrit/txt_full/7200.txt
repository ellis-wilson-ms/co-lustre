--- Commit Message ---
LU-684 tests: replace dev_read_only patch with dm-flakey

The dev_read_only kernel patch is mainly used for testing,
in order to simulate a server crash for ldiskfs by discarding
all of the writes to the device.

Since Linux kernel 3.0, this testing functionality can be
simulated by using "dm-flakey" target for device-mapper,
which supports a "drop_writes" parameter that could be used
in place of our dev_read_only kernel patch.

Patch Set 3:

To clarify, my concern is that we can't change the test framework code to require the new dm-flakey mechanism since this will break interop testing if this mechanism is not working on older versions of Lustre (which are configured by the test-framework code on the server and will not have the right DM device names, nor the check_and_add_to_md() function in its test-framework.sh file).

This also depends on whether the dm-flakey mechanism works on all supported kernels (RHEL6, SLES11 SP2, etc).  I see that someone from Red Hat committed the "drop_writes" mode to dm_flakey in the upstream kernel version v3.0-6027-gb26f5e3, and checking my local kernel I see that the "drop_writes" feature appears in 2.6.32-279.5.1.el6 (RHEL 6.3 and presumably later), but not in linux-2.6.32-131.6.1 (RHEL 6.1) so we should probably be OK on this front.

I find this approach unreliable. could we try to use snapshots instead? say, when rdonly is activated, we take a snapshot and the filesystem is keep going. when rdonly is reset, we rollback to that snapshot. the principal difference is that when rdonly is active we still see correct filesystem while with the current approach we are running on a corrupted filesystem and every time we have to read a block with skipped write we risk to find the filesystem corrupted. we're lucky enough to use this for years, but sometimes it fails..
