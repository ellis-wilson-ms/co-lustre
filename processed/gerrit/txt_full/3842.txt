--- Message (Andreas Dilger, 2012-11-29 19:26:02.000000000) ---
Patch Set 4: I would prefer that you didn't submit this

I would rather remove the mcreate calls from here as well, and fix the real problem in the right place. replay_barrier() is supposed to block all new changes to the system, not introduce extra changes to the system that are hidden from the test. 

If there is a problem with clients missing their sequence, that should be handled early on in the test, if it actually matters for the test. It probably makes sense to go back and see what test(s) failed before the mcreate was added, and just fix those tests instead of adding changes that may break (i.e. make useless, not necessarily cause to fail)  other recovery tests.

--- Message (Mikhail Pershin, 2012-11-29 19:39:10.000000000) ---
Patch Set 4:

That is not so simple because any test can fail, we can't predict when new sequence request will be issued. Anyway I agree that we need to return back and check what conflict caused that, maybe that is critical only for some tests.

--- Message (Andreas Dilger, 2012-11-29 20:19:01.000000000) ---
Patch Set 4:

Realistically, no client should run out of seq/oid space during recovery testing, and AFAIK the client should always prefetch a seq. in that regard it is more likely that the extra mcreate calls will cause the client to run out...

I think the more realistic scenario is when the client was newly mounted (or just evicted?) and hasn't gotten a seq at all. If it is only in the case of a new mount, then that should be relatively easy to fix. I don't think there is a reason for a new seq after eviction, since no other client could use this seq again so it is safe for the client to continue using it.

--- Message (wangdi, 2012-11-29 22:22:43.000000000) ---
Patch Set 4:

Well, in current implementation, client will reset (zeroing) all of the fid client cache (see seq_client_flush) during eviction, so the client does not to know where to start using the old seq after restore the connection to the MDT.  That is why it has to get a new seq before further creation. Probably we can optimization this process later, for example skip enough oid  according to the replay list during eviction, and keep using the old seq after restore etc.

But I agree to remove mcreate and sync from replay_barrier and fix  the specific test if needed.
