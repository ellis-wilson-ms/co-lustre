--- Commit Message ---
LU-1778 llite: enhancement to root squash support

Root squash is currently not properly enforced when permission is
checked from client cache information. This leads to scenario where
root can be allowed file access whereas root squash is activated.

Here is an example of such scenario:
root squash is activated and file F is owned by user U with 0600
permission
- user U stats file F
- user U keeps file F open
- user ROOT reads file F
=> access is allowed whereas it should be denied.

This patch adds root squash settings to llite so that client is able
to apply root squashing when it is relevant.

Configuration of root squash will now require 'lctl conf_param' on
both mdt and llite components:
lctl conf_param $FS.mdt.root_squash="65534:65534"
lctl conf_param $FS.mdt.nosquash_nids="192.168.0.1@tcp0"
lctl conf_param $FS.llite.root_squash="65534:65534"
lctl conf_param $FS.llite.nosquash_nids="192.168.0.1@tcp0"

--- Message (Andreas Dilger, 2013-02-02 21:04:17.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

My point is that having root squash on the client is meaningless.

If some user is root on a client node, and there was root squash "enforced" on that node, then the user could either become the actual user on that client, bypassing root squash:

 [root@client]# su - adilger
 [adilger@client]$ cat /mnt/lustre/adilger/secret_file

and root squash has no value.  Alternately, disable root squash on the client:

 [root@client]# lctl set_param llite.*.root_squash=0
 [root@client]# cat /mnt/lustre/adilger/secret_file

or alternately install a version of Lustre without root squash at all (old or patched code):

 [root@client]# umount -f /mnt/lustre
 [root@client]# rpm -e lustre-client
 [root@client]# rpm -i lustre-client-without-root-squash
 [root@client]# mount /mnt/lustre
 [root@client]# cat /mnt/lustre/adilger/secret_file

In summary, there is absolutely nothing to be gained except code complexity if the user already has root access on the client.  This has to be enforced at the server, and at most root squash can only prevent the user from accessing files owned by root in the filesystem, or other root-only operations.

Only with Kerberos and/or the upcoming UID/GID mapping could root be denied access to _new_ files from that client, and I can't think of any way that root could be denied access to cached files on the client.  Even if the user's keys were only in memory and the kernel itself blocked access from root locally (in an irrevocable manner), the root user could replace the lustre kernel modules with an insecure version and reboot, and then wait until the user accessed secure data again.

The only way to avoid this is to never allow root access on the client in the first place.
