--- Commit Message ---
LU-14565 ofd: Do not rely on tgd_blockbit

tgd_blockbit is recordsize bits set during mkfs.
This once set does not change. However, 'zfs set'
can be used to change the OST blocksize. Instead
of using cached value of 'tgd_blockbit' always
calculate the blocksize bits which may have
changed.

Test-case: sanity/104c added.

Patch Set 1:

>The other question is where is the block size cached on the client, and would it be possible to fix that instead?  One problem with this patch is what happens if the ZFS block size is changed to a value larger than 1MB?

What I found out is that osd_device->od_max_blksz (holds the default recordsize initially) is initialize during the mkfs. If recordsize not passed as option it is put to default 1M. 'od_max_blksz' within the statfs() call is then used to initilize the obd_statfs->os_bsize (block size). This is further used in calculation of blocks/free/avail which is reported. The odb_statfs(client) keeps getting changed as per the setting for IO.

'zfs set recordsize=32K'(etc...) changes the od_max_blkssz under osd_recordsize_changed_cb(). Which is fine. However, it also changes the os_bsize which is used as a 'divisor' for reporting the statfs/df output. Here os_bsize is used for IO as well as reporting the statfs/df output.

IMHO, (presently) od_max_blksz should be read as 'od_current_blksz' here. Which I think is causing the error(miss-reporting). As statfs/df always refer to max. 

Question:

Andreas, Would it be reasonable to have a separate field/value for recordsize and let 'od_max_blksz' be max block size.? Then update the os_bsize as recoredsize and continue with statfs calculation with max_block_size ?  Will this approach work ?

Thanks
