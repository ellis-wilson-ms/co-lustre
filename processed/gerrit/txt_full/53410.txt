Patch Set 4:

Not that it is the fault of this patch, but it is confusing to me that we have both disk-VER-FSTYPE.tar.bz2 as well as a separate set of FSTYPE-TGT-VER.tar.bz2 images.  The disk-* images will be found by test_32* but I'm not sure what the FSTYPE-* images are used for?

As for creating new images, the basic process is:
- build and install eg. vanilla 2.14.0
- run ONLY=32newtarball bash conf-sanity.sh

and this should create a new disk2.14-ldiskfs.tar.bz2 or disk2.14-zfs.tar.bz2 tarball, depending on FSTYPE. 

That said, this normally also needs a manual step first to create "examples" of files using new features, layouts, etc. into test_32newtarball() and then verify them in test_32*() if the appropriate feature marker is in the image. 

That ensures the upgrade process is at least exercising the basic functionality of each feature, if not the cross-product of the same. Things like project IDs/quotas, OST pools, pool quotas, fscrypt files/dirs should all be added to the images and tested after upgrade.
