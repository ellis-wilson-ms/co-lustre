--- Commit Message ---
LU-2430 mdd: add lfs mv to migrate inode.

Add lfs mv to migrate the individual inode from one MDT to
another MDT, and this function will only migrate inode layout
on MDT but not touch data object on OST.

lfs mv -M 1 /mnt/lustre/test1 #move test1 to MDT1.

The directory will be migrated from top to the bottom, i.e.
migrating parent first, then migrating the child.

Add migrate into sanity 17n to check on-disk format.

Add sanity 230(c,d,e,f) sanityn 80 to for migration.

--- Message (Andreas Dilger, 2013-06-27 16:34:09.000000000) ---
Patch Set 5:

John,
"lfs migrate" is already used for migrating a file's data objects to new OSTs. This is transparent to the application and does not change the inode number/FID. On the other hand, this operation moves the inode from one MDT to another and changes the inode number. It can't (currently) be used while the file is in use, because the file namespace lock resource changes with the MDT FID.

This makes sense to me to keep as a separate operation from "lfs migrate". If you have a better name than "mv" I would be open for that.

(12 comments)

I know that the patch has already landed, but some cleanups that should be done in a separate patch.

I think that having both LMV_MAGIC_MIGRATE and LMV_HASH_TYPE_MIGRATE is a mistake, since that is redundant and prevents more interesting uses in the future.  I think it would make sense to have LMV_HASH_TYPE_MIGRATE be a flag so that it is possible to specify both MIGRATE and a striped hash type like LMV_HASH_TYPE_FNV_1A_64.

That would allow using "lfs mv" in the future to restripe from a single directory to a hash directory by walking the source directory, keeping all of the entries that belong to the existing directory in place, and then moving all of the remaining entries to their new parent directory by hash.  

It could even be possible to rehash from one stripe count to a larger one by walking all of the directories and potentially moving them as needed to match the new hash.
