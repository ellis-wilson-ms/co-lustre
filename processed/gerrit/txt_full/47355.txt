--- Commit Message ---
LU-15858 sec: reinstate null encryption for file names

Reinstate null encryption for file names by adding a new llite
parameter named 'enable_filename_encryption', set to 0 by default.
When this parameter is 0, new empty directories configured as
encrypted ignore the filenames_encryption_mode and use
LLCRYPT_MODE_NULL instead, which is a no-op. This LLCRYPT_MODE_NULL
mode is inherited for all subdirectories and files.
When this parameter is 1, new empty directories configured as
encrypted use the normal encryption mode.

To set this parameter globally for all clients, do on the MGS:
mgs# lctl set_param -P llite.*.enable_filename_encryption=0

Also update sanity-sec test_54 to exercise the new parameter
'enable_filename_encryption'.

--- Message (Andreas Dilger, 2022-05-19 20:11:06.000000000) ---
Patch Set 1:

James, the situation is that the filename encryption has exposed a number of sharp corners (e.g. being unable to access the directory at all when MDT is mounted as ldiskfs, no backups at either the MDT or Lustre level, possible issues with LFSCK, etc.) that make it risky to use.  Rather than rip out all of the code, work is underway to pad the corners (e.g. patch 47309) to avoid serious foot-gun activities in 2.15.0, and allow it to be released while more work is ongoing.

This patch doesn't preclude anyone from using filename encryption if they want, but makes it optional (and compatible with what existed in 2.14).

Holding back 2.15 for what is a new feature that nobody has used yet, and is not going to be widely used initially, doesn't make sense.  Since 2.15 is an LTS release, it will continue to get fixes over time that can resolve the issues with filename encryption that are being found.

--- Message (Andreas Dilger, 2022-05-19 23:32:17.000000000) ---
Patch Set 1:

> > LSI_BDI_INITIALIZED already defined with this value in
 > > lustre/llite/llite_lib.c and used for lsi->lsi_flags:
 > >
 > > #define LSI_BDI_INITIALIZED	0x00400000

 > Well, that's terrifying.  Can you move that definition here?

This was moved from the header in patch https://review.whamcloud.com/28511 "LU-9558 llite: port lustre to unified handling of bdi", presumably because LSI_BDI_INITIALIZED is only needed for older clients (< 4.12), but it is definitely at serious risk of collisions.

One (slightly better) option to use a higher bit for LSI_BDI_INITIALIZED (e.g. 0x40000000), to avoid collisions until pre-4.12 clients are no longer supported (mainly el7) and that code is deleted.

Alternately, and safer, would be to also put the LSI_BDI_INITIALIZED definition here under HAVE_SUPER_SETUP_BDI_NAME so that it is clear when it can be deleted.

These LSI_* flags share lsi_flags with LDD_F_* flags (despite the different names) which is just opening ourselves up to bugs in the future, but shouldn't need to be part of this patch.

--- Message (Sebastien Buisson, 2022-05-20 08:44:41.000000000) ---

> Lastly it doesn't consider
