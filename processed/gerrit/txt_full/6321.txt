LU-2914 lfsck: split LFSCK code from mdd to lfsck

The new LFSCK is used not only for namespace consistency verification,
but also for some other purposes, such as file layout consistency, DNE
consistency, and ect.

So it is more reasonable to move the new LFSCK code from mdd module to
another independent module -- lfsck, which can directly talk with lod,
local storage lib and ldlm. MDD (and OFD in next pahse) will register
as lfsck instance for related lfsck component(s), such as 'namespace'.
The lfsck processing will be independent from MDD, except the control:
start/stop/pause.

> Patch Set 8: Looks good to me, but someone else must approve
> can you comment on how you deal with LinkEA and orphans?

Firstly, the LFSCK uses namespace-based directory traversal to perform linkEA consistency check/repair. It only scans the /ROOT and its descendants. So the objects under /PENDING will be skipped.

If the object becomes orphan just during the LFSCK processing such object, then linkEA verification policy will guarantee the consistency. When the LFSCK update the object linkEA (add/del/modify), it will lock (osd_write_lock) the target object, then the LFSCK will perform lookup under its current parent. If the target object has become orphan, then the lookup will return -ENOENT, and then the LFSCK knows that related linkEA is redundant. Otherwise, before the LFSCK unlock the target object, nobody can make the target object as orphan.

Above processing is based on local MDT, if DNE involved, then related object will be skipped. We will handle those cases in LFSCK phase III.
