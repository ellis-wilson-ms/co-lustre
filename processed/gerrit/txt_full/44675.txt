LU-14949 llite: Always do lookup on ENOENT in open

When there is no valid dentry found for a file we want to
open, we perform a full lookup, which goes to the server
and looks up the file by name. When we find an existing
dentry in cache *but the file is not open on the node*, we
do not do a full lookup.  We move directly to opening the
file.

When we open files, we use the FID of the file.  The
problem occurs when a new file is renamed *over* the file
we were trying to open.  This removes the FID we are
trying to open, but the file *name* userspace called open()
on is still present.  In this case, we will return ENOENT,
even though there is a file matching the name used in the
open() call.

The solution is when we get an ENOENT on open (indicating
our open raced with an unlink), we always send ESTALE back
to the VFS, which restarts the open and forces a lookup to
the server (by forcing Lustre to consider the dentry
invalid, see comments in ll_intent_file_open and code in
ll_revalidate_dentry).

This causes a lookup by name, which will correctly handle
the rename, allowing the open to proceed normally.

This should only generate extra retries in the case where a
positive dentry exists on the client but the file has been
removed on the server, ie, open racing with unlink.

This should hopefully be rare.

This looks very reasonable.

The problem itself is not so much the dentry cache as it is the fact we do not lock it on read only use.

So if we were to take a lock reference across the open RPC to ensure the file will stay put - that'll help too but there are very real rasons why we don't want to do that so ESTALE is the way to handle it and cause the restart.

It's possible we can just always pack in the name into the RPC so server does retrying logic internally, but that probably overcomplicates both client and server logic for what seems to be a pretty rare race condition (Except this workload of course)

What is also possible is we could delay unlinks for some time server-side so majority of racing RPCs still can find an object in "open-unlinked" state, but I am not sure we want to go here either unless the performance hit for such specific workloads turns out to be unacceptable with resends

it's kind of strange that the race cannot occur across clients.

It's possible the race is just narrower then, my understanding is this:

 racing client 2: stat b (gets a lock on b, potentially also on the parent)
racing client 1: rename(a,b)
serving thread 1: lock parent dir, a, b
mds: sends ASTs for any locks on parent dir, a and b
racing client 2: open(b) -> find the dentry/inode cache, issues open by fid
racing client 2: cancels local cache sends cancel to server
mds: receives the cancel, kills locks, clear the serving thread 1 to continue the rename
serving thread 1: unlink old b place a into it's place
serving thread 2: open-by-fid request arrives for the old b fid, cannot be found

so you see - if the open by fid arrives BEFORE the cancel then there's no problem and this is a more likely scenario, the race is much harder to hit.

Now when you do local operations we also have ELC so when the rename is PREPARING to send we identify parent lock and a and b locks and cancel them before the rename is event sent, then pack the cancel requests into the rename request.

the end result is we have a racing rename vs open by fid RPCs that is a much wider race.
