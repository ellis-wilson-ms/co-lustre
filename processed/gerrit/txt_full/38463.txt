I'm not sure I agree with the direction of this functional change. I believe per network type timeout + considering ULP timeout in determining the LNet/LND timeout makes more sense. The initial purpose of this is to reduce the complexity for the user from having to independently tune 3 different timeouts, ptlrpc adaptive timeout, LNet timeout and LND timeout.

--- Message (Chris Horn, 2020-05-04 18:02:12.000000000) ---
Abandoned

After discussing this issue with Amir I've realized that the ability to set per-LND timeouts that take precedence over the global timeout obviates the need to have the global timeout be tunable. As such, I'll abandon this patch and rework the series to:
1. Fix the lnet level timeout calculation 
2. Make the lnet level timeout readable from lnetctl/sysfs 
3. Allow LNDs to override the default
