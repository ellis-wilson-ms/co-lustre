--- Commit Message ---
LU-10337 mdt: Allow open of open orphans

Standard open by handle behavior allows opening of open
unlinked files files.  This currently only works in Lustre
if the file is already open on the same node, which is
insufficient.

When an open file is unlinked, we make it an orphan.
These files can be recognized by checking their open count
(mod_count).  It's enough to just make opening these files
possible, because the client cannot look them up to do an
open except when using a file handle.

--- Message (Andreas Dilger, 2017-12-12 11:22:51.000000000) ---
Patch Set 7:

Strictly speaking, when O_TEMPFILE files were added to the kernel (which we haven’t yet added to Lustre...) there was a discussion about this. The conclusion was that unlinked Giles should only be accessible if they were initially created as temp/nameless files, not necessarily all unlinked files, since that may constitute a security risk. 

Such files are marked with a flag in memory to allow them to be linked back into the namespace. 

What is it about open-by-handle that makes it need to access unlinked files?  My thought would be that the open handle has a limited lifespan (tens of seconds, maybe) after which it is not guaranteed to work. We could consider to add a cookie from the MDS into the open handle, a new one is generated every second, and expire each cookie after 10s.

Patch Set 21:

I think the spec you're reading is perhaps incomplete?

Or perhaps the issue is the word "deleted" - Is an unlinked but still open file "deleted"?  You could certainly argue it's not - the data portion is still present on disk and can be read and written.

But perhaps it *is* allowable per the spec to return ESTALE on open unlinked files.

But ESTALE in this situation is not what other file systems do, (XFS and EXT4 both allow opening by handle after unlink, for example).  And note also that this "open by handle of unlinked open file" works if the file is already open on the same client, but not if its open on another client.  Then only that client can open the file.  That seems like a clear violation of Lustre's goal of appearing the same to all clients.  (Not saying we have to follow that perfectly, but it's a goal.)

ESTALE is also incompatible with the requirements of a multi-node userspace NFS server or wrapfs-type system using Lustre for the metadata.  Cray's datawarp and DDN's IME are two examples of that, where the metadata and data activity may happen on different Lustre clients (at least, in datawarp - IME is broadly similar but I don't know the details of the implementation).  That means we have to be able to open unlinked files by handle from multiple nodes or we can't actually read/write them.

So maybe it's not a requirement of the spec, but it is of our use case. :)
