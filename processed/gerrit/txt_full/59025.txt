LU-0000 dbg: Debug Gerrit AI bot in sample patch2

ptlrpc: Use dynamic memory allocation for filesets

Currently, a nodemap's fileset uses a static-sized array which may be
much larger than needed. This patch adds dynamic memory allocation for
a nodemap's fileset to reduce the future memory footprint when
multiple filesets per nodemap are specified.

Note, when reading fileset fragments from the nodemap IAM, the needed
size is not known yet. Therefore, each fileset is initially
preallocated with "PATH_MAX + 1" and then shrunk when the actual size
is know. This is done on the last step, when a nodemap is set active,
e.g., after reading all IAM idx pages.

The nodemap's fileset is further renamed as future "lctl
nodemap_set_fileset" commands can only modify the primary fileset.

### 1. Commit Summary

This commit refactors how fileset names are stored within the `lu_nodemap` structure. Previously, a fixed-size character array (`char nm_fileset[PATH_MAX + 1]`) was used. This commit changes it to use dynamic memory allocation by introducing a character pointer (`nm_prim_fileset`) and a size field (`nm_prim_fileset_size`).

The main motivation is to prepare for future enhancements allowing multiple filesets per nodemap and to potentially reduce the memory footprint compared to always allocating `PATH_MAX + 1` bytes per nodemap, even if the fileset name is short or not set.

The changes involve:
*   Modifying the `lu_nodemap` struct definition in `lustre_nodemap.h`.
*   Updating code in `nodemap_handler.c`, `nodemap_storage.c`, and `nodemap_lproc.c` to use the new pointer and size field, including allocation (`OBD_ALLOC`), deallocation (`OBD_FREE`), and copying (`memcpy`).
*   Introducing a helper `nodemap_fileset_reset` in `nodemap_handler.c` to centralize freeing and nulling the fileset pointer.
*   Adding logic in `nodemap_storage.c` (`nodemap_cluster_fileset_helper`, `nodemap_fileset_resize`) to handle loading fileset fragments from the Identity Access Management (IAM) index. Since the total size isn't known when loading fragments, memory is initially preallocated (using `PATH_MAX + 1`) and later shrunk to the actual size once all fragments are loaded and the configuration is activated.
*   Renaming `nm_fileset` to `nm_prim_fileset` (primary fileset) to reflect that this field will hold the primary fileset when multiple filesets are supported.

### 4. Critical Review

*   **Strengths:**
    *   The change addresses a valid concern about memory usage with fixed-size arrays, especially looking ahead to multiple filesets per nodemap.
    *   It uses standard Lustre mechanisms (`OBD_ALLOC`/`OBD_FREE`) for dynamic memory.
    *   The introduction of `nodemap_fileset_reset` improves code clarity by centralizing the cleanup logic.

*   **Potential Weaknesses / Areas for Improvement:**
    *   **Memory Size Tracking:** The biggest concern is the apparent inconsistency in using `nm_prim_fileset_size`. It *should* track the allocated buffer size for correct freeing. However, several allocation sites (`nodemap_set_fileset_iam`, `nodemap_set_fileset_local`, `nodemap_fileset_resize`) calculate size based on `strlen() + 1`. This might lead to allocating less memory than `nm_prim_fileset_size` indicates (if it was previously set larger) or freeing memory using a potentially incorrect size in `nodemap_destroy` if the size field wasn't updated after a `realloc`-like operation (though `realloc` isn't used here, the principle applies). The code should consistently use `nm_prim_fileset_size` to hold the *allocated* size and update it whenever the allocation changes. Allocations should likely use this field if resizing, or set it after initial allocation.
    *   **IAM Load Complexity:** The strategy of preallocating `PATH_MAX + 1` in `nodemap_cluster_fileset_helper` and then shrinking later in `nodemap_fileset_resize` works but adds complexity and temporary memory overhead during config load. An alternative might involve storing the total fileset size as a separate IAM record, allowing allocation of the exact size needed upfront when processing fragments. This might simplify the loading logic.
    *   **Error Handling during Load:** The error handling in `nodemap_cluster_rec_helper` upon memory allocation failure uses a `GOTO out_nodemap`. While functional, ensuring cleanup (`nodemap_putref`) happens correctly requires careful checking of the surrounding code.
    *   **Naming:** `nm_prim_fileset` is okay, but clearly documenting its role as the *only* fileset currently supported, intended for future expansion, would be beneficial.

*   **Commit Message:**
    *   The message clearly explains the motivation (memory usage, future multi-fileset support) and the core change (static to dynamic allocation).
    *   It correctly mentions the preallocation/shrinking strategy for IAM loading.
    *   The scope is reasonably described.
    *   The `dbg: Debug Gerrit AI bot` part in the subject line is noise and should be removed.
    *   The `Test-Parameters: fortestonly` tag is concerning, suggesting that full regression or integration testing might not have been performed. This significantly increases the risk associated with the change, especially given the memory management complexities. Thorough testing is crucial.

    *   **Lines:**

        ```c
        	fset_len_remain = nodemap->nm_prim_fileset_size - fset_offset;

        	if (fragment_len > fset_len_remain)

        ```
    *   **Issue:** The code calculates `fset_offset` based on `fragment_id`. If `fragment_id` is sufficiently large, `fset_offset` could exceed the allocated size `nodemap->nm_prim_fileset_size`. While the code adjusts `fragment_len` based on `fset_len_remain`, the calculation of `fset_len_remain` itself (`nodemap->nm_prim_fileset_size - fset_offset`) might involve unsigned integer underflow if `fset_offset` is larger than `nodemap->nm_prim_fileset_size`, potentially leading to a large positive value for `fset_len_remain`. If this happens, the check `fragment_len > fset_len_remain` might not trigger the adjustment, and the subsequent `memcpy` could write out of bounds when accessing `nodemap->nm_prim_fileset + fset_offset`.
    *   **Rule:** ARR30-C. Do not form or use out-of-bounds pointers or array subscripts.
    *   **Suggestion:** Add a check to ensure `fset_offset` is within the valid bounds of the allocated buffer before calculating `fset_len_remain` and performing the `memcpy`.

    *   **File:** `lustre/ptlrpc/nodemap_storage.c`
    *   **Lines:**
        ```c
        		fset_size_prealloc = nodemap->nm_prim_fileset_size;
        		fset_size_actual = strlen(nodemap->nm_prim_fileset) + 1;
        		if (fset_size_actual == fset_size_prealloc)
        ```
