--- Commit Message ---
LU-10602 llite: add file heat support

File heat is a special attribute fo files/objects which reflects
the access frequency of the files/objects.
File heat is mainly desinged for cache management. Caches like
PCC can use file heat to determine which files to be removed from
the cache or which files to fetch into cache.
This patch adds file heat support on llite level.

Patch Set 1: Code-Review-1

(21 comments)

I think the API for getting the file heat needs to be improved to make it extensible in the future.

The other issue that I see is that the "heat" that this is measuring is only being stored on the local client, and doesn't necessarily give a good idea of how busy the file is.  It seems this is OK for PCC, since the heat would decide if the file should be imported into the local cache or not, but we should make sure that the interface and data structures are also useful/extensible if the heat is stored remotely on the OSS.  It doesn't seem like this would be useful yet for a client mounted directly on the OSS to manage migrating "hot" files from a cold to a hot storage tier.

How does a client determine which files are "hot" without scanning the entire filesystem?  It seems this API is OK to check current files that we already know about, but there doesn't seem to be a way to find files that are "hot" that should be imported into the local PCC.

--- Message (Li Xi, 2019-03-13 01:36:52.000000000) ---
Patch Set 1:

> It seems this API is OK to check current files that we already know about, but there doesn't seem to be a way to find files that are "hot" that should be imported into the local PCC.

Agreed on this. I have been thinking about adding an interface in /proc or /sys which will dump the top 100 hostest fids. It could be implemented as an IOCTL of the file system too. An interface like "lfs_heat_top" would be good.

--- Message (Andreas Dilger, 2019-03-13 03:28:49.000000000) ---
Patch Set 1:

One option would be to avoid explicit "hot file list" generation, and instead do this automatically when files are (re-)opened or read.  If llite/PCC keeps a running average "hotness threshold" based on free space in PCC and running average of file heat (with some minimum value), and when a specific file heat exceeds the "hotness threshold" it would be automatically mirrored into the local PCC.

That way we don't have to keep a sorted list, just the running average.  That is fine because a newly-accessed file will not have any saved heat if it isn't already in PCC, so it will take some time to build up to the threshold.

--- Message (Li Xi, 2019-03-13 04:58:51.000000000) ---
Patch Set 1:

Instead of implementingt the hotest file list specifically for PCC usage, I think hotest file list would be useful for general usage.

I don't think a complete sorted list based on heat is interesting. The most 100 (or 10000 based on the memory usage) files are interesting enough, since majority of files only have nearly zero file heat. Thus, an heap with limited memory size would be enough to maintain top N hottest files. And all the files that have zero file heats shall not be put into this heap.
