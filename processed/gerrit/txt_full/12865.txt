--- Commit Message ---
LU-5951 clio: update timestamps after buiding rpc

The mtime/atime/ctime in the write RPC has to be updated after
the RPC is built (where xid is generated), otherwise, it could
race with the setattr and updating wrong timestamps on OST side.

Seems this regression was introduced when landing clio code.

Use ofd_write_lock() to protect fmd lookup/udpate in
ofd_punch_object(), otherwise, it could race with ofd_attr_set()
and ofd_commitrw().

Patch Set 1: Code-Review-1

(4 comments)

This patch is definitely causing a large number of tests to fail related to mtime updates, so there must be something missing.

The major difference with the old obdfilter code is that it did the fmd lookup and updated the inode under obd_osfs_lock.  There is no locking here at all to ensure the attributes of "la" are still uptodate at the time that ofd_commitrw_write() -> ofd_write_attr_set() -> ofd_trans_create() ->  dt_attr_set() are called.  That means it may be possible to race with updating the timestamps if there are multiple threads doing updates at this point.

It also seems like updating the attributes has become inefficient in ofd_commitrw_write().  Calling ofd_write_attr_set() always creates a separate transaction when LA_[AMC]TIME are set (i.e. almost always) instead of setting the timestamps as part of the write transaction.  It would be better to change ofd_write_attr_set() to only update the UID/GID if those are set or filter_fid is missing (and only clear LA_UID/LA_GID in that case), and then set the timestamps as part of the write (the "if (la->la_valid)" case that still exists in ofd_commitrw_write().  The LA_[AMC]TIME should not be cleared earlier after ofd_write_attr_set().

I see that ofd_fmd_max_num_seq_write() and ofd_fmd_max_age_seq_write() are not calling ofd_fmd_expire() to expire the old entries as they probably should.  Also, ofd_destroy_export() is calling ofd_fmd_cleanup() twice...
