--- Commit Message ---
LU-12514 ptlrpc: move thread creation out of module initialization

When the ptlrpc module is loaded, it calls ptlrpc_init_portals() which
starts ptlrpcd threads, and starts the pinger work-queue.

We don't need these threads until the module is actually being
used, such as when a lustre filesystem is mounted.

So move the thread creation into new ptlrpc_inc_ref() (modelled on
ptlrpcd_incref()), and call that when needed, such as at mount time.

I had a look in the debug log provided by the janitor, and noticed there is very little network traffic.  Only 7 lines that contain "socklnd" on the client, and none on the server.
Looking for error codes (if only there were easy to grep for !!) I see ptl_connect_interpret() says "recovery of ... failed (-19)".   -19 is ENODEV and a comment in the code says that means "there is no service".

So I think that LNet isn't getting started properly or early enough.

Before this patch, ptlrpc_init_portals() -> ptlrpc_ni_init() -> LNetNIInit() -> lnet_acceptor_start()

is called at module-load time, so lustre is accepting tcp connections very early.
After this patch, the above doesn't happen until a filesystem (whether lustre on the client, or osd on the server) is mounted, or client_obd_setup(), or ldlm_namespace_new() is called.

Presumably we need some other trigger to start things up.

James: has your successful testing been on both the client and the server?
Maybe we could set test parameters to the the patched client against a working server, and the patched server against a working client, and see which has a problem..
