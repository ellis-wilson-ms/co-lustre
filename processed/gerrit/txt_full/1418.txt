In my patch, the new mds layer, as another obd device, will be called "mds". So we can not use "mds" to refer to the OLD mds obd anymore. That is why I rename it from "mds" to "mdd_mds". Since the old MDS code will be removed after orion is landed to master, so mdd_mds will be removed too at that time, with this regard, it probably makes sense to land this patch to 2.4.

But since the old MDS code is only used as the data stack connection on current master, and there are almost nothing valuable for this device under /proc. So even we change the name, we do not need worry too much. IMHO.

root@mds tmp]# find /proc/fs/lustre/mdd_mds/
/proc/fs/lustre/mdd_mds/
/proc/fs/lustre/mdd_mds/num_refs

But one thing I do concern is that those threads_xx proc parameter will be moved from /mds to /mdt in this patch,  if some users set these parameters in the config log (as we discussed in the email), we need resolve this incompatible issue. And this patch does not include this incompatible fix. since there are not much time left for 2.3, it probably makes sense to land this in 2.4 indeed.
