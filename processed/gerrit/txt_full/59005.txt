LU-0000 dbg: Debug Gerrit AI bot in sample patch

Use nodemap IAM for persisting filesets

Currently, two separate commands exist to set a fileset and both
behave differently. "lctl set_param (-P)" works as expected. On the
other hand, "lctl nodemap_set_fileset" updates the fileset only on the
local node and is not persistent. It could also be run from other
server nodes and not only from the MGS. This behavior is inconsistent
with other nodemap commands.

Ideally, we would want to use the nodemap IAM records for persisting
all nodemap properties without relying on secondary mechanisms, such
as the params llog which is currently used for filesets. This patch
adds support for persistence across restarts for "lctl
nodemap_set_fileset" commands. Filesets are split into 28 byte path
fragments which are stored alongside 4 byte fragment metadata into one
nodemap IAM record (which is 32 bytes).

For this, fileset fragments are stored in a "defined" subid range
within the NODEMAP_CLUSTER_IDX type. In anticipation of supporting
multiple filesets, the first fileset subid is reserved for a fileset
header. The subid range is set to support at maximum 256 nodemap
records for fileset fragments.  256 entries support enough fragments
to hold a PATH_MAX fileset in total. Only needed fragments are
inserted into the nodemap IAM.

To support seamless updates from fileset llog to IAM versions, a
nodemap flag was added which allows existing "lctl set_param -P
nodemap.<name>.fileset" to be applied outside the nodemap IAM as long
as no IAM records were set. Filesets from the llog are not migrated to
the IAM but are still compatbiel after an update. "lctl set_param
nodemap.<name>.fileset" is deprecated and uses the IAM once fileset
IAM records are in use.

Sanity-sec "test_27a" was modified to test fileset persistence.

**1. Commit Summary**

This commit refactors how nodemap filesets are persisted in Lustre. Previously, making filesets persistent relied on using `lctl set_param -P nodemap.<name>.fileset=...`, which updated the configuration via the params llog mechanism. The command `lctl nodemap_set_fileset` only applied the change locally and non-persistently. This inconsistency is addressed by making `lctl nodemap_set_fileset` the primary command for managing filesets persistently.

The core change involves storing the fileset path directly within the nodemap IAM (Index and Attributes Manager) records on the MGS, similar to other nodemap properties. Since fileset paths can exceed the size of a single IAM record (32 bytes), the commit introduces a fragmentation mechanism:
- A fileset path is broken into 28-byte fragments.
- Each fragment, along with metadata (fragment ID), is stored in a separate `nodemap_fileset_rec` IAM record.
- These records use the `NODEMAP_CLUSTER_IDX` type with a dedicated range of subtypes (`NODEMAP_FILESET` + fragment index) starting from 512. A maximum of 256 fragments are supported, accommodating `PATH_MAX`.

To handle the transition and maintain backward compatibility:
- A new flag, `nmf_fileset_use_iam`, is added to `lu_nodemap`.
- If this flag is false (default for existing setups), the system continues to respect filesets potentially set via the old `set_param -P` llog mechanism. `nodemap_set_fileset` called via `set_param` will only update the local in-memory value.
- Once `nodemap_set_fileset` (the ioctl version) is used, the `nmf_fileset_use_iam` flag is set, the fileset is stored in the IAM, and subsequent `set_param` calls for filesets will also use the IAM mechanism.
- The `lctl set_param nodemap.<name>.fileset` command is now deprecated, and a warning is issued when it's used.

Helper functions (`nodemap_idx_fileset_add`, `_del`, `_update`, `_clear`, `_rec_init`, `_fragments_add`, `_fragments_del`, `_fragments_clear`, `_info_init`, `_fileset_helper`) are added to manage these fragmented IAM records, including logic to handle partial updates and attempt rollbacks on failure.

Finally, the `sanity-sec` test suite (`test_27a`) is updated to use the new command and verify the persistence of filesets across Lustre service restarts. Wireformat checks (`wirecheck.c`, `wiretest.c`) are updated for the new structures.

**4. Personal Critical Review**

- **Strengths:**
    - **Consistency:** Consolidates nodemap property persistence into the IAM, making `nodemap_set_fileset` behave like other persistent `nodemap_*` commands and removing reliance on the separate params llog mechanism for this specific property.
    - **Backward Compatibility:** Attempts to maintain compatibility with existing setups that used the llog method via the `nmf_fileset_use_iam` flag.
    - **Persistence:** Provides true persistence for filesets set via the `nodemap_set_fileset` command, surviving restarts.

- **Potential Weaknesses & Areas for Improvement:**
    - **Complexity:** The fragmentation mechanism significantly increases the complexity of storing and retrieving the fileset value compared to a simple string storage. This complexity resides in `nodemap_storage.c` and could be a source of future bugs, especially around edge cases and error handling/recovery. Debugging issues related to fragmented records might be challenging.
