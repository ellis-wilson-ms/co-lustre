LU-13440 lmv: add default LMV inherit depth

A new field "__u8 lum_max_inherit" is added into struct lmv_user_md,
which represents the inherit depth of default LMV. It will be
decreased by 1 for subdirectories.

The valid value of lum_max_inherit is [0, 255]:
* 0 means unlimited inherit.
* 1 means inherit end.
* 250 is the max inherit depth.
* [251, 254] are reserved.
* 255 means it's not set.

A new field "__u8 lum_max_inherit_rr" is added, if default stripe
offset is -1, lum_max_inherit_rr is non-zero, and system is balanced,
new directories are created in roundrobin mannner, otherwise they
are created on the MDT where their parents are located to avoid
creating remote directories. And similarly this value will be
decreased by 1 for each level of subdirectories.

The valid value of lum_max_inherit_rr is different:
* 0 means not set.
* 1 means inherit end.
* 250 is the max inherit depth.
* [251, 254] are reserved.
* 255 means unlimited inherit.

However for the user interface of "lfs", the valid value is [-1, 250]:
* -1 means unlimited inherit.
* 0 means not set.
* others are the same.

Add sanity 413c.

Thank you for the patch.  I realized when looking at adding this to "lfs find" that the use of "--depth" is not a good name here, because "find" has a "-depth" option and it is also confusing with "--maxdepth".  It would be better to use a more unique name like "--max-inherit|-X" so that this option does not conflict.

Also, I don't see anywhere in the patch that the "max inherit depth" is decremented?  My thought was that the "max inherit depth" is decremented each time it is copied, until it reaches "LMV_INHERIT_MIN" and then it stops being copied.

I think this looks a lot better, but some minor bugs to be fixed in the argument handling and usage messages.

Have you given any thought about how this would be applied as a default to a new/empty system, and have you tested something like that?

For example, running a test on your local system to fill the filesystem *should* result in all the MDTs being balanced, something like:

    $LFS setdirstripe -D -i -1 -R 3 -X -1 $MOUNT
    $LFS setstripe -E 64k -L mdt -E eof -c 1 $MOUNT
    tgtdir=$MOUNT/tgt
    mkdir $tgtdir
    while true; do
        lfs df
        lfs df -i
        depth=$((RANDOM % 5))
        # find a "random" directory to copy into
        rsync -a /etc/ $tgt/etc-$SECONDS/
        newtgt=$(find $MOUNT/tgt -type d --mindepth $depth
                 --maxdepth $((depth + 3)) | head $RANDOM | tail -1)
        [[ -d "$newtgt" ]] && tgtdir=$newtgt
    done

and let that run for a while to see if all of the MDTs are being used. 

DoM may fill up the MDTs more quickly, but I think most files in /etc are very small, and in theory the DoM component should be dropped if the MDT is too full.

It might be useful to graph the lfs df %used for inodes and blocks for each MDT to see if the usage is growing evenly.
