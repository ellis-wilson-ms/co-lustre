LU-19014 memcg: fix client hang in balance_dirty_pages()

Two nodes (at least) append write a shared file in Lustre with
memcg enabled.
The client randomly hung in balance_dirty_pages() with the
following call trace:
[<0>] balance_dirty_pages+0x2ee/0xd10
[<0>] balance_dirty_pages_ratelimited_flags+0x27a/0x380
[<0>] generic_perform_write+0x150/0x210
[<0>] vvp_io_write_start+0x516/0xc00 [lustre]
[<0>] cl_io_start+0x5a/0x110 [obdclass]
[<0>] cl_io_loop+0x97/0x1f0 [obdclass]
[<0>] ll_file_io_generic+0x4d2/0xe50 [lustre]
[<0>] do_file_write_iter+0x3e9/0x5d0 [lustre]
[<0>] vfs_write+0x2cb/0x410
[<0>] ksys_write+0x5f/0xe0
[<0>] do_syscall_64+0x5c/0xf0

After analyze the core dump of the hung system, we found that the
bdi_writeback data structure (wb) corresponded to the memcg has
pending dirty pages (in state WB_registered | WB_has_diry_io), but
can not write-out the dirty pages and loop in balance_dirty_pages
function.

This is a bug in Lustre memcg code. In OSC/MDC layer, it will stop
to flush dirty pages once found that there are no any unstable
pages.
However, there may be some dirty pages queued in the cache. In
this case, the client should still write back the dirty pages.
Thus the wb stat accounting will be updated and the write process
can continue instead of looping endless.

Moreover, there are some problem in the current Lustre CLIO engine.
When the system or a certain memcg is under memory pressure, the
client just queues the dirty page in page cache or in the current
active extent (OES_ACTIVE osc_extent) when vvp_io_write_commit()/
cl_io_commit_async() is called in ->write_end(). The queued pages
can not be written back even the kernel is trying to flush dirty
pages in writeback via ->ll_writepages().
The client is looping in the following call sequences:

loop:
->write_begin()
->write_end()
->balance_dirty_pages()
  ->Launch file writeback in background but cannot flush any
    dirty pages.
  ->The current process is paused a certain time (i.e. 200ms) as
    the corresponding @wb is dirty exceeded.
-> GOTO loop:

The write progress is very slow: write a page and sleep/pause for
a period of time alternately.

We fix this hang in ->ll_write_end(). When detect the corresponding
@wb is dirty exceeded, the client will submit the dirty pages into
OSC writeback cache. The state of current extent will change from
OES_ACTIVE to OES_CACHE and this kind of extents can be written
back. Moreover, we mark the current extent as urgent, thus it can
be flushed much more quickly.

Fixes: 8aa231a99 ("LU-16713 llite: writeback/commit pages under memory pressure")

I figured out why disabling unstable page accounting avoids the issue.

The issue with the client is that it’s hit the unstable pages limit and isn’t finding out they’ve been committed.  All pages are clean but unstable and it can’t add new dirty pages.  I recall this from working on write back/cgroups support and can’t remember if we ever solved it.

But the issue here isn’t dirty pages on the client or uncommitted pages on the server - it’s that the client doesn’t find out about committed pages in a timely manner.  I’m not sure why it is able to make progress - maybe ping? - at all but that progress is super slow.

I think write back needs to send an empty RPC or sync request, even when there are no dirty pages…?  We both need to request commit AND get a reply back saying pages were committed and do that when there are no dirty pages waiting.

--- Message (Andreas Dilger, 2025-05-16 03:06:24.000000000) ---
Patch Set 2:

Patrick, there _should_ be a pipeline of transaction commits (last_committed_transno) that allow the client to know RPCs have committed, and in turn clean up the associated bulk pages that are persistent on storage.

There is also patch https://review.whamcloud.com/56773 ("EX-10481 sec: set page sync flags to avoid ENOMEM") that I thought implemented a reasonable mechanism to avoid stop-and-go IO traffic.  It sends OBD_BRW_SOFT_SYNC to prod the OST to start syncing early (75% of limit), *before* the client is totally OOM (or at unstable page limit, or whatever limit it might be bumping up against).

Then, if the client gets closer to its hard limit (ideally "num_rpcs_in_flight away") if the OST hasn't yet started higher last_committed_transno in the RPC replies, it clears OBD_BRW_ASYNC (forcing the BRW RPC to be a synchronous write before reply), which should ensure that all previous BRW RPCs should also be committed and the pages can be cleaned up.

I can fully imagine that something has become broken in the page cleanup life cycle over years of kernel updates, since we definitely hit client OOM issues in VM testing.

--- Message (Patrick Farrell, 2025-05-16 19:32:04.000000000) ---
Patch Set 2:

"Then, if the client gets closer to its hard limit (ideally "num_rpcs_in_flight away") if the OST hasn't yet started higher last_committed_transno in the RPC replies, it clears OBD_BRW_ASYNC (forcing the BRW RPC to be a synchronous write before reply), which should ensure that all previous BRW RPCs should also be committed and the pages can be cleaned up."

Yes, I think this is the part that isn't working.  I don't think there's anything that checks these limits and forces a sync - when we hit the unstable pages limit, we use SOFT_SYNC, but soft sync requests soft sync, it doesn't make the RPC sync.

So when we get close, we send a soft sync RPC, but it doesn't force syncing of that RPC, it's a soft sync.  So we don't get a reply and so we stay stuck.

We could either force that RPC to be sync or implement something a bit more complicated.

But ALSO:
The soft sync limit has nothing to do with the kernel VM limits.  It's just using Lustre tunables internally.

So we're not even requesting soft sync, depending on the exact combination of settings. :(

I think the thing to do is Yingjin's change, but also probably something with having writeback send an RPC whenever there are unstable pages, so sending an empty RPC if there are unstable pages but no dirty pages.
