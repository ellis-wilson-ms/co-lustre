--- Commit Message ---
LU-7564 osp: Do not match the lock for OSP

In DNE operation, we do not need match the lock
in the OSP cache, so to lock the remote object
exclusively on master MDT, then other threads on
master MDT will not be able to access the remote
object at the same time.

Patch Set 1:

Di, any changes with COS ? if not, please describe the exact case in the commit message.

--- Message (wangdi, 2016-02-02 18:14:25.000000000) ---
Patch Set 1:

oh, this is not related COS. This is only to make sure no other threads on the master MDT can access this resource at the moment. Without the flag, other threads on the master MDT might access the object, note this is not client side enqueue.

--- Message (Alex Zhuravlev, 2016-02-02 18:16:20.000000000) ---
Patch Set 1:

so what's wrong with accessing such an object concurrently?

--- Message (wangdi, 2016-02-02 18:21:11.000000000) ---
Patch Set 1:

hmm, for single operation, we used to lock the resource exclusively before any operation. I think all MDS stack is based on that, IMHO.

Considering two threads concurrently add two entries with same name under the remote directory. If without this flag, both of threads can proceed, because they can share the lock. then it will end up to meet -EEXIST error until remote MDT insert the name entry, which will cause a lot trouble.

--- Message (Alex Zhuravlev, 2016-02-02 18:25:13.000000000) ---
Patch Set 1:

how is it different from a local case?

--- Message (wangdi, 2016-02-02 18:29:20.000000000) ---
Patch Set 1:

In local operation, MDT will use ldlm_enqueue_local(), which will lock the resource exclusively, not for ldlm_enqueue_cli(), which will only lock the resource on this node, but other threads on the same node can still access the resource, which should not be allowed on the server side.

--- Message (Alex Zhuravlev, 2016-02-02 18:41:08.000000000) ---
Patch Set 1:

please, provide an example

--- Message (wangdi, 2016-02-02 18:56:21.000000000) ---
Patch Set 1:

1. $lfs mkdir -i1 /mnt/lustre/test1
2. client1 create file /mnt/lustre/test1/a and send req to MDT1, handled by thread1
3. client2 create file /mnt/lustre/test1/a and send req to MDT1, handled by thread2
4. thread 1 enqueue the lock of test1 on MDT0, then both thread1 and thread2 can share the lock (without the flag). 
5. both of them will create objects, packing updates, and send it to the MDT0. then one of threads will met -EXIST. Even worse we do not have rollback mechanisms here.

Anyway, local server lock will exclusively lock the resource to the thread, and it should be the same for remote server lock.

--- Message (Alex Zhuravlev, 2016-02-02 19:01:50.000000000) ---
Patch Set 1:

well, those are different locks and they should conflict by definition. ldlm_cli_enqueue() itself just sends a request for this _new_ lock to remote target. so I guess it's ldlm_lock_match() who makes it shared? why do we match here?

--- Message (wangdi, 2016-02-02 19:28:14.000000000) ---
Patch Set 1:

that is exactly what LDLM_FL_EXCL does, not match for this lock.

--- Message (Alex Zhuravlev, 2016-02-02 19:31:05.000000000) ---
Patch Set 1:

sure, but then why do we call ldlm_lock_match() in the first place?

--- Message (wangdi, 2016-02-02 19:52:58.000000000) ---
Patch Set 1:

Hmm, I think the right behavior is

1, if there are no other threads access the resource, then we want to match.

2. In some cases, we still want to cache the lock, after 1 thread release lock.

Hmm, probably LDLM_FL_EXLC is not enough here.
