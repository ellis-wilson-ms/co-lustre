(3 comments)

Amir, Serguei, I'd like your feedback (in addition to Neil's) on my proposal to change the meaning of the "KSOCK_PROTO_Vn" versions to be a bitmask of feature bits.  I think this would allow more flexible protocol changes in the future, similar to how OBD_CONNECT_* flags are used in Lustre, and doesn't add any significant complexity to the code.

The only thing that would need to be decided at this point is whether there are "optional" connection features (e.g. high 16 bits of version) that can appear in the proto version but are ignored if not understood, while the low 16 bits would be "mandatory" to be understood.  

Older peers would treat all of the bits as a "new version" (maybe a very large one) and fail such a connection if it ever was seen, but new peers can mask out the optional bits if there are fields that have some value which is not critical to understand.

In the case of KSOCK_PROTO_TCP6 it would mean that kshm_src_nid and kshm_dst_nid are nid16 values, as are kshm_ips[].

--- Message (Neil Brown, 2021-05-25 03:29:00.000000000) ---
Patch Set 2:

I agree that feature bits can be useful in some circumstances, but they also greatly increases the testing matrix for interoperability testing.
In the case of SOCKLND, I think the protocol is, and should be, quite simple - more so than the OBD protocol.  I feel that feature bits would be over-engineering.

As well as introducing a hello message with 16-byte address, I would like the new hello message to not include nips or ips[] as these are now deprecated.  Would that require two feature bits - 1 each (nid size and extra addresses?  Large extra addresses would never be tested).

I can see that a KSOCK_PROTO_V4 introduced for IPv6 might seem confusing, but I try not to think about it like that.  There are 4-byte addresses and 16-byte addresses, and that is the change that is happening in the hello protocol.
In socklnd, 4-byte addresses are IPv4.  In other LNDs there are (presumably) in other address spaces.  In socklnd, 16 byte addresses are used for IPv6, but other LND's could potentially use them for any address larger than 4 bytes (6-byte Ethernet addresses ???)

So my preference would be to stick with sequential numbering.  But I won't argue strongly against feature bits if others really want them.
