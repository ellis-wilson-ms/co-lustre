LU-12728 ldlm: sticky lock matching

When multiple clients are writing to the same stripe of a
file, Lustre tends to end up with "lock exchange", where
clients writing to non-overlapping blocks still end up
exchanging LDLM locks on that stripe.  This is very bad for
performance, as in the worst case, each client does only
one write for each lock cancellation + sync.

This is mitigated because in practice clients usually do >1
write before the lock is cancelled, improving the writes
per lock request ratio.

This is desirable, and we can encourage it by allowing a
restricted number of i/os to match a lock after the
blocking callback has been received.  Limiting the number
of i/os that can do this means the client will still be
able to give back the lock and avoid eviction.

Making the lock "sticky" in this manner should improve
the ratio of writes to lock requests under contention,
which should significantly improve performance.

Note this change does not directly delay lock cancellation,
so if there are not multiple threads actively attempting to
use a lock when a BL callback is received, there will be no
change in behavior.

(3 comments)

I like the idea.  This is similar to an optimization in the JBD2 layer of ext4, where journal commits for synchronous transactions are delayed for a short time to allow other threads to join the transaction, but the transaction is not delayed if only a single thread is writing to the filesystem.

It will be interesting to see what impact it has on performance.  On the one hand, it _might_ help, but if there are many threads doing interleaved writes (IOR HARD) it is possible that the locks are not larger than a single IO so there really isn't any chance for merging.  This would be best seen in cases where there are a small number of threads contending on the same lock and lock expansion is still being done.

"I like the idea.  This is similar to an optimization in the JBD2 layer of ext4, where journal commits for synchronous transactions are delayed for a short time to allow other threads to join the transaction, but the transaction is not delayed if only a single thread is writing to the filesystem."
Ah, that's actually *very* similar, yeah.  Including the "not delayed" part.

"It will be interesting to see what impact it has on performance.  On the one hand, it _might_ help, but if there are many threads doing interleaved writes (IOR HARD) it is possible that the locks are not larger than a single IO so there really isn't any chance for merging.  This would be best seen in cases where there are a small number of threads contending on the same lock and lock expansion is still being done."
So while there's logic (and it's somewhat complicated) to control expansion, it doesn't seem to work terribly well under contention.  If I understand it correctly, it will always expand to the start of the next lock, except if there are more than 32 locks on the object, in which case it will extend a maximum of 32 MiB.  32 MiB is still a *lot*.  In my experience of looking at this, this code never really gives happy or consistent results with shared file i/o.  (I've never really been confident of how to get better results through modifying it, though, because as bad as lock contention is, requesting a lock for every i/o can be even worse.)

So I think this will still come up for IOR hard.  I'm *hoping* so, anyway, but that's another thing to test.  (It also helps for that case that, at least currently, we have packing on the node so adjacent threads are writing adjacent ranges.)
