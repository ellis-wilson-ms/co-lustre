--- Message (Patrick Farrell, 2019-07-08 16:02:59.000000000) ---
Patch Set 1: Code-Review-1

OK, so I think we're suggesting loo_object_release() and lu_object_is_dying() call orders could be re-arranged by the compiler, while executing on the same CPU.  Is that correct?

Because if so, that's wrong - The compiler is not free to re-arrange function calls, because it has to assume functions can have side effects.  In general, functions are always going to be called in the order specified, because the compiler does not 'see through' the calls and do optimization across functions.  (Inline is different since there's no actual separate function)

If it's a cross CPU issue, that's different, but in that case I think we need smp_wmb here - not just a barrier(), which is compiler only.

Actually, I don't think a cross CPU issue is possible here - This area of code should be single entry due to the lock above:
	if (!cfs_hash_bd_dec_and_lock(site->ls_obj_hash, &bd, &top->loh_ref)) {
