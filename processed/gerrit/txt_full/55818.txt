--- Commit Message ---
LU-18884 osp: use UUIDs instead of NIDs for LWP setup

The osp when setting up LWP uses NIDs instead of UUIDs for
LCFG_SETUP. This is wrong and NIDs can easily over flow the
obd_uuid structure. A side effect of this mistake is that
the ptlrpc conn to UUID mapping just ends up being a NID
looking up a NID which happens to be itself. That is very
wrong. Replace using an NID with a real UUID.

I disagree with patch and think it breaks current nodes mapping. I think we have to make clear what 'UUID' is being used in struct uuid_nid_data in lustre_peer.c and all mapping it provides. That 'UUID' which is bad naming I think, is NODE identificator, not target or server. It defines which NIDs has one or another NODE in cluster. Its purpose was mostly to save info about configured NIDs for each node, so client import can set up list of connections to all nodes where target can be including failovers. Since nodes can't be identified as target name that mapping is using primary NID notion. So my question here - are we on the same page here? Or you are thinking differently about what UUID means in this mapping? I mean struct uuid_nid_data.
What is wrong with patch itself, first, instead of single node mapping it produces now several mappings for each target on that node, say that is node "192.168.0.11@tcp", originally this string was used as UUID for that node, so any mention of that NID in config for any device will end up with the same mapping for node identified by that primary NID. With patch each device will create own mapping with the same list of NIDs identified by device UUID, e.g. "<lwp_name>_UUID" in patch. Which is wasteful in terms of memory usage and need more effors to handle them all.
The second and more critical, such approach doesn't store NIDs by nodes/failovers, all NIDs for all nodes/failovers will be under the same target UUID, e.g. lwp-MDT0000_UUID->NID1,NID2,FAILNID1,FAILNID2,.. instead of current mapping as "NID1@tcp"->NID1,NID2 and "FAILNID1@tcp"->FAILNID1,FAILNID2. That breaks how import creates and uses connection list, which is exactly just list of node 'UUIDs' as NID string. Now it would have single connection with target UUID name it seems, and I expect connection problems due to that.
So before doing any other changes here we need to agree about this functionality how it works now, what is not OK or even wrong with that and what is proposed instead, for me it seems you are going to get rid of nodes mapping and re-organize all more like MGS nidtable but local on client, where each target has own NID list
