LU-13023 pcc: Incorrect size after re-attach

The following test case will result in incorrect size for PCC copy:
- Attach a file with size of s1 (s2 > 0) into PCC;
- Detach this file with --keep option, and the data will retain
  on PCC;
- Truncate this file locally or on an remote client to a new size
  s2 (s2 < s1);
- Re-attach the file again. The size of PCC copy is still s1.

To solve this problem, it need to truncate the size of the PCC copy
to the same size of the Lustre copy which will be HSM released
later after finished the data copy (archive) phase.
This patch also adds the handle for the signal pending when the
attach process is killed by an administrator.

(7 comments)

This patch looks like the minimum to fix the immediately visible problem, and we want to get 2.13 out the door, but also seems like it could be improved.

One question I have is why there wasn't something like a data version check in the PCC/HSM code to ensure that the client considered the PCC archive copy as stale and not use it at all? Yes, my strong preference is that we merge HSM, PCC, and FLR into a single mechanism for having multiple copies of the same data.  I think there are a few steps needed for this:

If the PCC data version does not match the Lustre data versiom, then the PCC version should not be used.  I'd think this should be handled using normal HSM_DIRTY state when the Lustre file is modified, just like with a tape archive.

 > I think we can try to implement data version for PCC to check
 > whether it is stale.

Yes, definitely.  If there is no checking of the data version then re-using the PCC cache after detach is unsafe.

 > I thought you are saying some optimize for PCC.
 > i.e.
 > In current PCC, once a file is attached into PCC on a client,
 > another client access even not modifying data (read operation) will
 > trigger the data restore.
 > We currently use layout generation to determine whether the PCC
 > copy on the client is still valid. But data restore will increase
 > the layout generation. so a read operation from remote clients will
 > invalidate the PCC copy.
 > I think it could be optimized by the following strategies:
 > When archived the file, we store the data version of the file into
 > HSM attrs EA;

Or better the data version can be saved in an xattr on the PCC cache file.  That makes it clear to check whether the PCC copy matches the Lustre copy or not.

 > Upon data restore for a PCC cached file, we could reset the data
 > version of various objects on OSTs after copy data back into Lustre
 > OSTs.

It is difficult to change the data versions of the files in Lustre, since these are based on the last transaction number the object was modified in.  However, it would be easy to change the data version on the PCC cache file to match the data version of the restored file.

 > However, we must ensure the sum of data version of all these
 > objects is equal to the data version storing on the data version of
 > HSM EA attrs on MDT.
 > So when re-attach the file once attached into PCC on the client, we
 > only need to check whether the data version storing on HSM EA on
 > MDT is same as the data version of the file (sum of data version on
 > Lustre OSTs).  If so, we don't need to the data copy phase.

I think the easier case would be if the HSM/PCC copy would be part of FLR, then if the file is modified via PCC it uses the FLR mechanism to mark the Lustre mirror STALE.  When the file is accessed by another client, or ages out of the cache, then the FLR mirror in Lustre needs to be resync'd from the PCC file and will no longer be STALE.  It can then be read from either PCC or Lustre.  If the Lustre copy is modified, the PCC copy is marked STALE/HSM_DIRTY and should either be removed immediately (if no longer needed), or kept for some short time as a backup until the PCC space is needed.
