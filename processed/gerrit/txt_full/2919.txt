--- Commit Message ---
LU-56 lnet: add lnet_*_free_locked for LNet

LNet has a set of internal functions lnet_*_free to release LNet
message or LNet descriptors (EQ, MD, ME), they require caller to
hold LNET_LOCK on calling, this is somehow confusing because
lnet_*_alloc should be called w/o any lock.
This patch add another set of functions named as lnet_*_free_locked
to replace original lnet_*_free which require caller to hold lock,
and still provide lnet_*_free which can be called w/o any lock.

This patch is just the first step work for LNet SMP improvements.

--- Message (Isaac Huang, 2012-06-06 23:59:52.000000000) ---
Patch Set 1:

Liang, sorry I'm late again ;)

The patch looks good, but, it doesn't make much sense to me to have the lnet_*_free_locked() functions. E.g.:
lnet_drop_delayed_put()
......
    lnet_msg_free_locked(msg);
    LNET_UNLOCK();
}
Here it's unnecessary to hold the lock while freeing the msg. I'd rather UNLOCK() earlier:
    LNET_UNLOCK();
    lnet_msg_free(msg);
}
There's quite some places like this. I don't like calling LIBCFS_FREE() with LNET_LOCK() - LIBCFS_FREE() is not cheap. I'd suggest to:
1. Remove all lnet_*_free_locked() functions.
2. Replace their call sites with: UNLOCK(); lnet_*_free(); In some places the changes aren't that straightforward, but I think it's worthwhile.
3.
lnet_xx_free()
{
/* Never call me with lock held */
#if FREE_LIST_STUFF
   lock_some();
   free_list_free(xx);
   unlock_some();
#else
   LIBCFS_FREE(xx);
#fi
}

So that locking is only used when LNET_USE_LIB_FREELIST defined, which is never defined for KERNEL, and for KERNEL all unnecessary LOCK are avoided.

--- Message (Liang Zhen, 2012-06-07 02:15:31.000000000) ---
Patch Set 1:

Isaac, thanks for review.
First, this is just an intermediate patch, the reason that I created a long patch chain is because reviewer can't look into a patch with too many changes together. 

Although calling kfree with hold of locks is not a good style, but in linux it's safe to call it even in interrupt context, that's the reason that we live with this for years, I think most of kfree is no more than add an object to a list.

I actually did try to put LNet objects in a deathrow list if we called lnet_*_free with hold lock, and free them while unlocking, but that will increase complexity of code (LNet will have two types of lock soon, it's possible that we try to lock-B with holding lock-A), and my tests didn't show any observable improvement with this change. So I gave up to do that, the reason I added an extra set of APIs at here is only because I want to make code has less confusion.

In the upcoming patches, LNet will have partitioned data & lock, which means if w/o this change, and  we allocated a LNet object in CPU-partition-A, then try to do something in CPU-partition-B (i.e; forwarding a message), if there's something wrong happend while we holding lock of partition-B, and want to release the object, then we need to release lock of partition-B and re-lock partition-A and call lnet_*_free for the object because it needs to be called with holding of lock, this is really confusing and complex.

 With this change, we can just unlock partition-B and call lnet_*_free, or calling lnet_*_free_locked if the object is allocated in current LNet partition.

Of course, this change still give us a chance to add stuff to deathrow list in the future, just as you said. We can put object on deathrow list while calling lnet_*_free_locked, and define a function of lnet_*_lock which will call:
1) take dead objects from deathrow list; 2) unlock; 3) free objects by calling lnet_*_free.

But as I mentioned above, I didn't see too much improvements with this change, so unless there is a good reason, I prefer not to do this now.
