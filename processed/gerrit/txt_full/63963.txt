LU-12669 lov: fix parity sub-object offset for PFL EC

test_47a creates a multi-component PFL file with two EC
components (comp1 [0,16M] 2+1, comp2 [16M,EOF] 4+2),
writes data spanning both, resyncs parity, disables one
OST per component, and reads back.  Recovery fails
because parity reads use wrong sub-object offsets for
comp2.

Root cause: userspace resync (llapi_ec_write_parities)
writes parity at file-level offsets through
llapi_mirror_write.  The LOV layer maps these via
lov_stripe_offset() to sub-object offsets.  The kernel
recovery path must read from matching sub-object
offsets.

When a component's e_start is not aligned to the parity
stripe width (pcount * stripe_size), different parity
stripes map to different sub-object offsets.  Userspace
writes parity index k at file offset
(e_start + raid_set * swidth + k * stripe_size), and
lov_stripe_number() determines which LOV stripe receives
each index.

Fix lov_ec_read_parity_pages to:
1) Accept a parity file offset (not sub-object offset).
2) Compute start_stripe = lov_stripe_number(parity_foff)
   to find which LOV stripe the first parity index maps
   to.
3) For each LOV stripe i, derive p_idx =
   (i - start_stripe + pcount) % pcount and call
   lov_stripe_offset() to get the per-stripe sub-object
   offset.
4) Set parity_used[j] = p_idx so lov_recover_data uses
   the correct encoding coefficients.

lov_io_ec_rd_start now passes the parity file offset:
  e_start + raid_set * pcount * stripe_size

lov_ecio_add_parity_sub retains the base_poff formula
for sub-IO extent computation, which correctly covers
all per-stripe sub-object offsets.

This commit supersedes 7ba8ad8e3c ("fix parity offset
calculation in EC recovery") which can be dropped.
