--- Commit Message ---
LU-17519 lnet: remove dead peer nis automatically

When a client is decommissioned, it stays forever in the peer list of
the servers and generates a stream of messages like:
lnet_handle_recovery_reply()) peer NI (10.11.12.13@o2ib) recovery
failed with -113
lnet_handle_recovery_reply()) Skipped 1234 similar messages

It can be cumbersome to remove it manually when a lot of clients are
constantly decommissioned. This patch removes a peer NI automatically
if it has been in recovery for a period of time greater than
dead_peer_ni_interval (in seconds).

The default value is 0 and means that this feature is disabled.

--- Message (Chris Horn, 2024-03-21 16:17:13.000000000) ---
Patch Set 1:

(2 comments)

A better way to remove dead peer _NIs_ would be to queue the peer for discovery. If the peer no longer has the NI then it will be deleted by discovery (and really it should have been deleted already, right?). If NI is still configured on the peer then we ought to keep it.

I think the more interesting case is when the peer itself is dead for an extended period. In this case we don't want to just delete some peer NI but rather the entire peer. So perhaps we should check all last alive values of the peer and delete the peer when we haven't heard from any of them?

I agree with Serguei that we shouldn't need a new timeout parameter.
