CLF_JOBID flags allows to check if the field is present or not (old
format) when reading an entry. Jobids are expressed as 32 chars long,
zero-terminated strings. Updated test_205 in sanity.sh.

--- Message (Andreas Dilger, 2013-07-11 23:56:49.000000000) ---
Patch Set 14: I would prefer that you didn't submit this

(13 inline comments)

It appears that this patch does not address interoperability issues with earlier versions of Lustre that I can see.

1) What happens if a new client connects to an old MDS?
2) What happens if an old client connects to a new MDS?
3) What happens if an MDT is upgraded with ChangeLog entries that have mixed record sizes?
4) What happens with old binaries on a new client?

I think problem (1) and (3) are addressed by CLF_HAS_JOBID, but (2) and (4) appear to be missing.  Ideally, if the application doesn't open the changelog with a new CHANGELOG_FLAG_JOBID then either the kernel or library would filter out the cr_jobid field from the changelog record and convert it to the old changelog_rec structure.
