LU-17478 clio: parallelize unaligned DIO write copy

The data copying for unaligned/hybrid IO reads is already
parallel because it is done by the ptlrpc threads at the
end of the IO.  That for the writes is not - it is done
by the submitting thread during IO submission.

This has a huge performance impact, limiting writes to
around 3.0 GiB/s when reads are at 12 GiB/s.

With the iov iter issue fixed, we can do this copy as
part of IO submission.

With this and the patch to use page pools for buffer
allocation (https://review.whamcloud.com/53670), the
maximum performance of hybrid IO is basically the same as
DIO, at least for current speeds.

This means hybrid reads and writes at 20 GiB/s with
current master + this and the pool patch.

Note this requires a funny workaround: If a user thread
calls fsync while a DIO write is in progress, the user
thread can pick that write up at the RPC layer and become
responsible for writing it out, even though that write
isn't in cache.  (Because the write is waiting to be
picked up by a ptlrpcd thread.)

If that DIO write is unaligned, the user thread is unable
to do the memory copy.  It's not an option to have the
thread ignore a ready RPC, so instead we spawn a kthread
to handle this case.

This only occurs when DIO is racing with fsync, so
performance doesn't really matter, and this cost is OK.

Also this is actually a surprisingly simple patch...  I had expected to spend real time on this problem, but using the page pools for allocations means we don't need to parallelize the allocation phase, so this got easier.  (Because the big allocations can sleep, we can't do them in the ptlrpcd threads, so we couldn't parallelize there... But using the pools means allocation is so fast there's no need to parallelize it.  I've confirmed this on one of our DGXes.)

> 
> On the one hand, having buffers available for fast allocation is great, but there can't be an huge number of pages pinned in the pools if we start hitting memory pressure.  At some point we need to start throttling the dirtying thread(s) and/or free pages from the less-used parts of the pool and/or bound the upper size of individual pool entries so that they are only used for rapid small-RPC IO, while large IO where the network/storage transfer time is noticeable can do their own alloc/free?

So, this is a page for every DIO page which is in flight.  Any core or thread can throw multiple RPCs out there, in theory, so no relation to those.

So it could end up being quite a bit, and yes, we probably do need some throttling.  We have shrinkers registered for the pools, so the freeing part is taken care of.

It is in fact large IOs that benefit the most from this, as strange as that sounds.  Essentially, for small IOs, the fixed costs (per-IO setup, round trip time, etc) dominate, then for larger ones, the running costs dominate.  Allocation is a running cost.  With very large IOs, we see IO submission and server activity running in parallel - The network and server are handling RPCs while we're still submitting.

Also worth noting that because we do the network/storage stuff in parallel, it can go *really* fast relative to IO submission.

So...  I don't think we'd ever not want to use the pools - I think the worst case scenario is we add a little overhead to allocation by going through the pools if we're absolutely slammed on memory.  I can sort of imagine how that might bite us, but I think in practice we'll be happy with it.

But we should almost certainly limit the total amount of IO outstanding, which we could limit through the pool.  For example, 1 GiB of IO from one thread can hit 60 GiB/s on a DGX to an AI400 (not hybrid, regular DIO with LU-13814, but the point applies...).  So that's a total of 1 GiB of outstanding IO across all OSTs for a lot of throughput.

So that's all to say we could likely limit the pool size to a few GiB and that would get us plenty of throughput - that would be my preferred method, limiting the pool size.  I'm not sure how we should limit it (like, how to determine the limit), but fairly small amounts (relative to total system memory, anyway) should do.

(For example, if we compare that to max_dirty_mb at 2 GiB per OSC, compare to that 1 GiB of IO getting you 60 GiB/s of throughput...)
