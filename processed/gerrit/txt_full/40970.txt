--- Commit Message ---
LU-14217 osd-zfs: allow SEEK_HOLE/DATA only with sync

ZFS doesn't report valid offset for SEEK_DATA if there are dirty
data, but may report SEEK_HOLE correctly that cause unreliable
results when same offset can be reported as HOLE (correctly) and
also as DATA, incorrectly but because switching to generic approach,
assuming all file is data and hole beyond end of file.

To avoid that we have to sync dirty data when dmu_offset_next()
reports EBUSY and repeat lseek call. Considering that this can
cause slowdown this behavior is controlled via new 'sync_on_lseek'
option. With this option turned off osd-zfs reports that it doesn't
support SEEK_DATA/HOLE because we cannot use unrealiable results
in our tools to copy sparse data

--- Message (Andreas Dilger, 2020-12-17 01:38:20.000000000) ---
Patch Set 1:

Does the client also flush its own cache when SEEK_DATA are called?  I'm wondering if the OSS needs to take a PR extent lock covering the start of the seek to the end of the hole to verify that all data in that region has been flushed?

That might be optimized if the client calling SEEK_DATA is already holding a DLM lock for that region, and the client does its own seek in the page cache for dirty data.

I don't think it is harmful to return the start of a hole, then discover that SEEK_DATA returns the same offset (e.g. in case of a race) if the DLM lock caused data to be flushed to the OST from another client.  That minimizes the region that is locked, and avoids getting any locks on the OSS in the common case if there are no holes in the file.

--- Message (Mikhail Pershin, 2020-12-17 06:03:33.000000000) ---
Patch Set 1:

Andreas, yes, client takes extent lock and flushes own cache but without forced sync on server, that would be overkill. And as I understand, ldiskfs could perform SEEK_HOLE/DATA on pagecache data so doesn't need all data flushed on server but ZFS can't and return EBUSY if there are dirty data one server.

And the same offset for data and hole is quite troublesome, consider we are searching for data chunk to copy, SEEK_DATA returns offset, SEEK_HOLE from that offset should give us data chunk, and if offset the same then chunk size is zero. That would be considered as end of file and zero read, so sparse copy routine stops at that point. Moreover if we will try to continue and issue new SEEK_DAT to find the next chunk, it could return the same offset again and repeat doing so at least until data is flushed in ZFS. I don't think that such "lseek polling" for is what we need.
So the calling txg_wait_synced() looks as optimal solution

--- Message (Andreas Dilger, 2020-12-17 09:35:00.000000000) ---
Patch Set 1:

I think it would be possible for SEEK_HOLE to return what the OST thinks is a hole, then the data is written and the next SEEK_DATA flushes the client data by getting a PR lock covering the hole to verify nothing was written. If data was written into the hole, it returns the same offset, but if no data was written then the hole is skipped.  

I don't think it is as likely for the reverse to happen, and it is not harmful to return data to the client when there is a hole being created. At worst, the client would ready zeroes for that case.
