--- Message (Andreas Dilger, 2014-12-19 09:34:46.000000000) ---
Patch Set 1:

Hopefully this patch allows isolating the root cause of the failure to a specific subtest.

However, if this patch with 10x sanity runs doesn't find the problem, we may have to approach it in s different way. It might be that unmounting after every test avoids whatever race condition is causing the problem. Also, this might cause the testing to be extremely slow. 

We know that the cause of the hang is somewhere before test_132() and the previous test that unmounts the OST (not sure which one that is right now), since this was the only test to report problems until it was disabled and we started seeing failures in test_133g().

It may be better to perturb the testing as little as possible and just insert a single unmount/remount of ost1 half way between test_132 and the previous unmount (or the start of sanity if that isn't clear). Then, we use the test results to bisect the remaining tests depending on whether the added unmount fails or not.
