LU-13642 lnet: Allow static IP specification

Allows selecting an interface by specifying an IP address(NID)
in the lustre.conf (lnet.conf) file or as parameter in the module
start.
example: networks=tcp1(1.2.3.4)

The change does not have any effect on current configurations.
Changes for IPV6 are included, but not activated and tested.
Further IPV6 changes are not integrated.

For further reference please read

IP specification in LNet
https://wiki.whamcloud.com/display/LNet/IP+specification+in+LNet

Test-Parameters: trivial

This looks pretty good - thanks.

However I think you use sockaddr_storage too much.
You should only use it when you need to allocate space to store any address.
So lnet_parse_networks() needs it, and lnet_parse_ipaddress() should accept it as an argument.
But lnet_init_nid() and the functions which call it only need a "struct sockaddr *", because they only read from it, they don't write to it.  So they don't need all the storage, they just need a valid address.
I think this adds one cast to the code where lnet_parse_networks() calls lnet_ni_alloc(), but I think it is worth it.

My (limited) understanding of the issue here is that this appears to be a bug in "ip2nets".  The original issue reported as that ip2nets is selecting an interface based on the IP address, but then selecting the first alias on the network (which is link local) and not the IP address that was given to ip2nets.

That seems like a bug.  I'd prefer fixing ip2nets rather than adding a new and very similar directive.  Firstly, that avoids confusion by users that don't know which one to use, and people would *still* try ip2nets and hit the original bug.

Secondly, ip2nets is more convenient to configure a large number of clients, while "networks=tcp1(IPaddr)" or "networks=NID" both depend on a specific IP address to be specified, which only works for a single node, and would then need customization for each node.
