LU-9658 ptlrpc: Add QoS for uid and gid in NRS-TBF

This patch add a new QoS feature in TBF policy which could
limits the rate based on uid or gid. The policy is able to
limit the rate both on MDT and OSS site.

The command for this feature is like:
Start the tbf uid QoS on OST:
    lctl set_param ost.OSS.*.nrs_policies="tbf uid"
Limit the rate of ptlrpc requests of the uid 500
    lctl set_param ost.OSS.*.nrs_tbf_rule=
 "start tbf_name uid={500} rate=100"

Start the tbf gid QoS on OST:
    lctl set_param ost.OSS.*.nrs_policies="tbf gid"
Limit the rate of ptlrpc requests of the gid 500
    lctl set_param ost.OSS.*.nrs_tbf_rule=
 "start tbf_name gid={500} rate=100"

or use generic tbf rule to mix them on OST:
    lctl set_param ost.OSS.*.nrs_policies="tbf"
Limit the rate of ptlrpc requests of the uid 500 gid 500
    lctl set_param ost.OSS.*.nrs_tbf_rule=
 "start tbf_name uid={500}&gid={500} rate=100"

Also, you can use the following rule to control all reqs
to mds:
Start the tbf uid QoS on MDS:
    lctl set_param mds.MDS.*.nrs_policies="tbf uid"
Limit the rate of ptlrpc requests of the uid 500
    lctl set_param mds.MDS.*.nrs_tbf_rule=
 "start tbf_name uid={500} rate=100"

> I prefer extracting the {ug}id from the request over that
> from the object. Some operations do not carry {ug}id, for
> examples ost_read and ost_punch for the newly created objects.

This is something that should be fixed in the client, to send the UID/GID for these RPCs, and the NRS code can use them.  The server can determine if the fields are valid using OBD_MD_FLUID/OBD_MD_FLGID, and the OST code can ignore them.

This is a relatively harmless change, and I'd be OK to accept a patch to fix this on b2_10 (and you could even backport it to your b_ieel3_0 branch if you wanted).  This implies it should be a separate patch, so it is more easily backported.

> Besides, I think it is also more reasonable for users, because
> QoS policy is aimed for the system users in the most case.

Sure, and fields exist to send this information from the client.

> In my former patch, in order to use the efficient {ug}id, I
> actually pack the {ug}id in ost_body for all the OST requests,
> although it is not so elegant. After seeing the patch you
> refreshed, I think it is also a good method to get the {ug}id
> from the object. So, I combined the two methods, but still
> using the efficient {ug}id as much as possible.

Do you mean "effective {ug}id" here?  That is the actual UID/GID of the running process on the client, and what is passed to the MDS in fsuid/fsgid.

> If we want to limit the requests JUST using efficient {ug}id,
> ONLY the following will be modified.
> 1. for osc->ost side, we should pack the {ug}id to ost_body.
> It can be done in function lustre_set_wire_obdo().

The difficulty with doing it in lustre_set_wire_obdo() is that this would change e.g. a "touch" command that is modifying other attributes of this object.  It needs to be done with the knowledge of what command is being run.

> 2. for mdc->mdt side, we should only handle LDLM_ENQUEUE. maybe
> it is needed to add cr_uid and cr_gid in struct mdt_rec_create.

I don't think any changes are needed for mdt_rec_create.  It isn't possible in POSIX to create a file that has a different ownership than that of the running process, which is already sent as cr_fsuid/cr_fsgid.
