LU-18838 llite: fix statahead thread ctx use-after-free

The statahead thread crashes in atomic_dec_and_lock() at exit because
lli->lli_sax has been nulled (and the ctx freed) while the thread is
still running. Two LIST/FNAME spawn paths in start_statahead_thread()
hand the spawned thread the same single refcount that lives in the
lli->lli_sax slot rather than taking a dedicated thread-side ref the
way the SFNAME-with-existing-slot path and ll_ioctl_ahead() already
do. With multiple statahead threads sharing one ctx (SFNAME / mdtest
shared dir), any peer that legitimately puts the slot's ref drops the
refcount to zero, frees the ctx, and nulls lli->lli_sax out from
under the still-running thread. The thread then re-reads lli->lli_sax
at cleanup and dereferences NULL.

Fix it in two parts:

  - In the LIST/FNAME and SFNAME-empty-slot paths in
    start_statahead_thread(), take an explicit __ll_sax_get(ctx)
    under the same spinlock that publishes lli->lli_sax = ctx so
    the thread truly owns its own ref. Track this with a local
    thread_ctx_held flag and unwind it in the kthread_create()
    failure path. Paths that already had a dedicated thread bump
    are unchanged.

  - In ll_statahead_thread() and ll_deauthorize_statahead_advise(),
    capture ctx = lli->lli_sax once under the spinlock at function
    entry, while the slot is guaranteed populated and stable, and
    release via that local pointer at exit instead of re-reading
    the racing slot.

Add LASSERTs that the captured ctx is non-NULL and that
ctx->sax_inode matches dir as a tripwire for any future spawn site
that forgets to publish the ctx before waking the thread.
