LU-15829 llite: don't use a kms if it invalid.

Lockless DIO don't update a KMS as other IO type does,
it caused a situation when next read don't known a real file size
to be read. Lets avoid using an invalid KMS.

--- Message (Patrick Farrell, 2022-05-22 01:59:18.000000000) ---
Patch Set 1: Code-Review-1

I haven't figured out yet if this patch is correct, but I am confident it's unnecessary.

With DIO, if the read is inside of KMS, great - we perform the read to the server.  If there's data, we return it.  No correctness issue.

If the read is beyond KMS, we do prep_size, which glimpses size.  Now we have valid size information and we either do the read or don't do the read accordingly.

The bug is just if the read is *at* KMS, when we might decide to not send a read.

With DIO, everything is fine if we either:
Send a read (because if the file has been truncated, the read will discover that and give a correct result)
Update the size

The only issue is if a KMS issue causes us not to send a read to the server.

Shaun's patch was both sufficient and correct.

--- Message (Patrick Farrell, 2022-05-24 15:57:18.000000000) ---
Patch Set 1:

Sure, but we already don't send a glimpse if we have the lock.  If we have the lock, the glimpse is satisfied locally without sending a request to the server.

And the glimpse request returns a read lock unless there's a competing write lock.  And  if there is a competing write lock, that lock will be cancelled when the server takes the DLM lock for our read, so we'll get it next time.

Anyway, more importantly:
We *already* use loi_kms_valid here.
If !loi_kms_valid, we return 0 for kms from cl_object_attr_get.  See osc_attr_get().

That means we're *already* using kms_valid, we just return 0 when it's invalid.  We just don't correctly handle kms=0, which Shaun's patch fixes.

So perhaps the right fix would be to special case kms=0.  But there's no need to look at kms_valid in this function.
