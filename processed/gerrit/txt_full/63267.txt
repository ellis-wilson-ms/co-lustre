## Summary

**The Fix:**
I removed the attempt to capture a return value from `ll_dmu_write()` and removed the subsequent error check.

**Rationale:**
The compile error occurred because the patch incorrectly tried to treat `dmu_write()` as a function that returns an error code, when in fact it's a **void function** that returns nothing. 

The `ll_dmu_write()` macro (defined in the autoconf macros) expands to either:
- `dmu_write((os), (object), (offset), (size), (buf), (tx), (flags))` for ZFS 2.4.0+, or  
- `dmu_write((os), (object), (offset), (size), (buf), (tx))` for older ZFS versions

In both cases, `dmu_write()` is a void function. The error message "invalid use of void expression" was triggered because the code tried to assign the result of a void function call to the variable `rc` and then negate it.

Looking at other uses of `dmu_write` wrappers in the codebase (e.g., `ll_dmu_write_by_dnode()` in `osd_scrub.c` lines 2143 and 2351, and `osd_dmu_write()` in `osd_io.c` lines 416 and 1145), they are all called without capturing a return value, confirming that these are void functions.

The fix simply calls `ll_dmu_write()` without trying to capture its return value, matching the pattern used throughout the rest of the codebase.
