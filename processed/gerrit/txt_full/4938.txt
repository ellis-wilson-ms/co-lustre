LU-398 ptlrpc: Add the NRS ORR and TRR policies

The ORR (Object-based Round Robin) policy schedules brw RPCs in
per-backend-filesystem-object groupings; RPCs in each group are
sorted according to their logical file or physical disk offsets.

The TRR (Target-based Round Robin) policy performs the same
function as ORR, but instead schedules brw RPCs in per-OST
groupings.

Both these policies aim to provide for increased read throughput
in certain use cases, either by minimizing costly disk seek
operations (by ordering OST_READ, and perhaps also OST_WRITE
RPCs), but may also allow for improved performance through better
resource utilization and by taking advantage of locality of
reference characteristics of the I/O load.

Patch Set 3:

This patch needs two things:

1. To add the new-style lprocfs functions (i.e. as per Liang's comments in the 'framework and fifo policy' patch, patchset 9, to have one lprocfs file per service, and not per service partition); I should have this ready and uploaded by the end of today.

2. Especially the ORR policy, and less so the TRR policy, need some way of recycling struct nrs_orr_object instances, otherwise given enough runtime, these will grow to unacceptable levels of memory usage. One way to get around this is to temporarily shut down and then restart the policy, such that all previously used resources are freed, but that cannot be a good solution.

I was not sure whether this mechanism should be LRU-based, a simple freeing of each object when its usage counter drops to zero, or something else, so any suggestions are more than welcome.

However, I am hoping that if we _really_ need to, we could perhaps land this patch without having addressed #2 entirely, and it would be ok to submit an extra patch for that after the feature window closes, since minor enhancements and bugfixes are meant to be accepted for a few weeks after the feature window will close, and this shouldn't be too big a change.
