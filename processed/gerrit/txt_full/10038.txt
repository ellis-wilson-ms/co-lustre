LU-4801 ldlm: use mutex instead of spinlock on server side resources

Since server side processing of resources during granting is potentially
very cpu-intensive, it well might be that spinlock is
improper choice and by using mutex we allow other threads to do something
useful instead.

it's not the code that holds the spinlock. The concern is the spinlock vs mutes implementation itself.

In face of heavy contention spinlocks spin wasting cpu cycles that could be better spent doing something else. when a contended mutex is encountered the cpu is yielded to other threads that could do something useful with it.

--- Message (Neil Brown, 2019-10-16 22:46:24.000000000) ---
Patch Set 4: -Code-Review

> In face of heavy contention spinlocks spin wasting cpu cycles that could be better spent doing something else.

Yes of course - in the fresh morning light that is now obvious :-)

Though mutexes are optimized to spin opportunistically - if the thread holding the mutex has running on a CPU, threads waiting for the mutex will still spin for a while - but not as long if the mutex is held for a long time.

It would be better to reduce the time that the lock is held for.  Scheduling might be better than spinning, but it is not exactly cheap.

Are there any identified hot-spots that we could look at optimizing?
