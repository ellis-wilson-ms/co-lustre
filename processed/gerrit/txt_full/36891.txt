--- Commit Message ---
LU-12988 ldiskfs: skip non-loaded groups at cr=0/1

cr=0 is supposed to be an optimization to save CPU cycles,
but if buddy data (in memory) is not initialized then all
this makes no sense as we have to do sync IO taking a lot
of cycles.  also, at cr=0 mballoc doesn't store any avaibale
chunk. cr=1 also skips groups using heruistic based on avg.
fragment size.
it's more useful to skip such groups and switch to cr=2 where
groups will be scanned for available chunks.

using sparse image and dm-slow virtual device of 120TB was
simulated. then the image was formatted as OST and filled
using debugfs to mark ~85% of available space as busy.
mount as OST w/o the patch couldn't complete in half an hour
(according to vmstat it would take ~10-11 hours). with the
patch applied mount took ~20 seconds.

--- Message (Andreas Dilger, 2019-11-29 09:47:30.000000000) ---
Patch Set 2:

> not sure about this 75% rule.  I think it still makes sense to use
 > cr=0 as it's cheap and fast if data is in memory.

Originally Artem's patch was only an interface from userspace that said "skip cr=0" now, and there was a userspace daemon to control this in real time, but that is potentially complex to set up.  The static 75%/85%/95% thresholds were convenient and provided similar speedups as having a userspace daemon, but without any need for userspace tools.

I think with this patch to skip groups with cr=0 when the bitmaps are not loaded, that might make the first stage of his patch obsolete.  I suspect he was seeing similar symptoms as us with the millions of bitmap loads, and chose to use a different heuristic (free space) for skipping cr=0 and then moving to cr=1.

As the filesystem fills beyond 85% and 95% full, it will progressively skip cr=1 and cr=2 and then just try to find any file allocation.
