LU-6138 lfsck: NOT hold reference on pre-loaded object

To improve the LFSCK performance, the LFSCK main engine will pre-load
the object locally or remotely, then generate related LFSCK request
that reference the pre-loaded object, and then push the request into
related LFSCK pipeline. The LFSCK assistant thread will handle the
LFSCK request some later asynchronously.

Originally, the LFSCK request holds the pre-loaded object reference,
so the assistant thread can handle it directly without locating the
object by FID again. But holding the object reference will cause the
object cannot be purged out RAM. If some LFSCK request has held the
object, and some other unlinked the object before the LFSCK assistant
thread handling the LFSCK request, then the unlinked object will be
cached in RAM until the last reference released. Because the LFSCK
main engine and assistant thread run asynchronously, we do not know
when the LFSCK request that holding the object reference will be
handled. If the assistant thread needs to locate the object with
the same FID before that, it will fall into self-deadlock for ever.

Patch Set 5:

> nasf, I think if the OI Scrub wasn't allowed to get so far ahead of layout verification then cache flushing wouldn't be a problem. I think part of the current problem is that OI Scrub is not limited by the async window and it can produce too many lfsck_layout_req's to fill the MDS memory and push OSD objects out of cache.
If OI scrub was throttled when it got too far ahead of the layout engine then the OSD objects would still be in cache even without any reference on them.
That is something for a separate patch I think, since this one should land as-is to at least get the benefit of not keeping a reference on the parent object.

Andreas, if the OI scrub is under urgent mode, such as rebuild/recreate OI files, then it will run as fast as possible, under such case, the OI scrub may be much ahead of other up layer LFSCK component (namespace LFSCK/layout LFSCK). But if it is NOT under urgent mode, then the OI scrub speed will be controlled via the OSD layer pre-fetching cache size that is SCRUB_WINDOW_SIZE (1024). So the OI scrub will be at most SCRUB_WINDOW_SIZE ahead of the other up layer LFSCK component.
