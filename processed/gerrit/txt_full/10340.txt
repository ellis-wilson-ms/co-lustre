before mdo_is_subdir, otherwise the un-exist object will cause
LBUG in mdd_is_subdir.

Patch Set 2: -Code-Review

> No, it should check before mdd_is_subdir, see mdd_is_subdir->mdd_object_type->lu_object_attr-> LASSERT(lu_object_exists(o) != 0);

right, I missed that one.
I also missed the case you run into, as it is not properly described.

1st, it is not related to ldlm locking, because e.g. destroying of orphan on close does not take ldlm lock at all; also because we never remove EXISTS flag - so it is safe to check if flag is set without such a locking.

2nd, !EXISTent object for known fid may only appear if the tgt dir has been  removed, in this case  our object_find gets an absolutely new allocated object without EXISTS flag set - this is what gets missed once my lock ordering patch applied.

3rd, the same may happen with src dir too, but the previous code had only the check for tgt dir, because src dir is protected by lookup - it returns an error for src child lookup.
