LU-19193 osd-zfs: revert LU-15963 use contiguous chunk

When using pools containing 106 HDDs in two draid2:11d:1s:53c vdevs
with two NVMe special devices, we found obdfilter-survey produced
large numbers of stripes (plain files) with a data block size of 4K.

Since that pool configuration stores data in minimum allocation units
of 44K, this resulted in 40K of wasted space per block, and a much
larger block count than is appropriate.

The issue was found to be in LU-15963 osd-zfs use contiguous chunk to
grow blocksize

If only RPCs with a non-zero offset (off != 0) are seen by
osd_grow_blocksize() when first writing to the file, then
obj->oo_next_blocksize is never set, and osd_grow_blocksize() then
fails to picks a new block size, which means everything gets written
out as 4K blocks. 4K is the default value regular file objects are set
to when they're first initialized.

This reverts commit dacc4b6d384cbe6376a4cf106cc63ad1ac0cd23d.
