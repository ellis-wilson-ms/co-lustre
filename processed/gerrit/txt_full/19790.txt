--- Commit Message ---
LU-7473 acl: increase ACL entries limitation

Originally, the limitation of ACL entries is 32, that is not
enough for some use cases. In fact, restricting ACL entries
count is mainly for preparing the RPC reply buffer to receive
the ACL data. So we cannot make the ACL entries count to be
unlimited. But we can enlarge the RPC reply buffer to hold
more ACL entries. On the other hand, MDT backend filesystem
has its own EA size limitation. For example, for ldiskfs case,
if large EA enable, then the max ACL size is 1048492 bytes;
otherwise, it is 4012 bytes. For ZFS backend, such value is
32768 bytes. With such hard limitation, we can calculate how
many ACL entries we can have at most. This patch increases
the RPC reply buffer to match such hard limitation. For old
client, to avoid buffer overflow because of large ACL data
(more than 32 ACL entries), the MDT will forbid the old client
to access the file with large ACL data. As for how to know
whether it is old client or new, a new connection flag
OBD_CONNECT_LARGE_ACL is used for that.

Patch Set 2: Code-Review-1

Hmm, maybe I missed something important. But I didn't see any size limit on client side. And there is no client side in your patch either. So this looks like a server side limitation. That makes me think that OBD_CONNECT_OLD_ACL and LUSTRE_POSIX_ACL_MAX_SIZE_OLD might not be necessary.

--- Message (Fan Yong, 2016-04-26 12:07:04.000000000) ---
Patch Set 2:

> Hmm, maybe I missed something important. But I didn't see any size limit on client side. And there is no client side in your patch either. So this looks like a server side limitation. That makes me think that OBD_CONNECT_OLD_ACL and LUSTRE_POSIX_ACL_MAX_SIZE_OLD might not be necessary.

The ACL entries count is restricted by both the RPC reply buffer size and the backend ACL EA space. Originally, the RPC reply buffer size was defined as LUSTRE_POSIX_ACL_MAX_SIZE which is related with the hard coded ACL entries count limitation (32). That is why you can create 32 ACL entries at most at that time.

But in fact, we can redefine the RPC reply buffer size independent from the old LUSTRE_POSIX_ACL_MAX_SIZE, and make it the same size as backend ACL EA size, that is one block at most for ldiskfs backend. Then you create more ACL entries.

It is NOT true that the patch is server side only, because both client and server share the same RPC layout which is defined in the layout.c, such file depends on lustre_acl.h. As you can see, we redefined LUSTRE_POSIX_ACL_MAX_SIZE in lustre_acl.h, then client behaviour is affected also. To be clear, you can apply the patch on your system for verification.

> Was any testing performed to ensure the RPC sizes do not grow uncontrollably?
The ACL size is restricted NOT exceeds "ocd_max_easize", so the RPC size will not grow uncontrollably unless the "ocd_max_easize" is out of control.


> What's the mechanism to curtail back rpc sizes once we see a few big xattrs that are otherwise rare on a given fs?

There is not shrink mechanism for the RPC buffer size for ACL, it will be increased as "ocd_max_easize" increased. We do not know what the next ACL size will be, we can make some assumption, but it may be wrong. If such case happen, the ACL cannot be returned to client. That needs some protocol to negotiate between client and MDT for the large ACL. I am not sure whether it is worth to complex the protocol for that.

In fact, such mechanism does not exist for current implementation. The current RPC buffer size for ACL is fixed size as LUSTRE_POSIX_ACL_MAX_SIZE, the new size with the patch is "ocd_max_easize".
