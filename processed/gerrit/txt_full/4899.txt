--- Commit Message ---
LU-1866 ldiskfs: allow dotdot to be updated with data appended

The original implementation assumed that there was enough space
to hold dotdot entry for rename. It is not always true when new
name entry has extra data appended, such as FID-in-dirent, but
old name entry has not. The old patch may cause directory crash.

Would it be possible to avoid this patch entirely if rename just dropped the extra FID data if there was no space in the  ".." dirent?  This is only needed for upgraded 1.8 filesystems, right?

While it definitely makes sense to store FID-in-dirent for normal directory entries, since ".." is not as frequently accessed as directory entries it may not be critical for performance to avoid the ".." lookups for upgraded 1.8 filesystems.

--- Message (Fan Yong, 2013-01-04 12:26:03.000000000) ---
Patch Set 1:

We can ignore the FID-in-dirent for dotdot entry, since ".." is not frequently accessed.

But this patch "ext4-dir-dotdot-update.patch" is used to replace the old patch  "ext4-hash-indexed-dir-dotdot-update-rhel5.patch". It is not only for performance, but also for correctness. Because the old patch did not consider the case of rename after MDT file-level backup/restart: there is not enough space to hold the FID-in-dirent. For old Lustre-2.x, rename after MDT file-level backup/restore will either trigger ASSERT() or crash the directory page. The new patch resolve the potential bugs.
