LU-3270 statahead: statahead thread wait for RPCs to finish

Statahead thread should wait for inflight statahead RPCs to
finish before quit, because statahead RPC callback will access
data allocated in context of statahead thread, and this can
also guarantee all referenced inodes are put in client umount.

Several other fixes and cleanups to statahead:
* when only_unplug is set for do_statahead_enter(), it should
  not unplug the entry found, because the inflight statahead
  RPC may not finish yet.
* fix the bug in LU-2272: ll_intent_drop_lock() may sleep, which
  should not be called inside lli_sa_lock.
* removed sa_entry.se_refcount because now statahead threads wait
  for RPCs to finish, it's always safe to access sa_entries, and
  scanner is the only one that removes sa_entry.
* simplify sai refcounting:
  - statahead thread holds one refcount.
  - do_statahead_enter()/ll_deauthorize_statahead() takes refcount
    before use.
* add lli_sa_enabled flag to enable/disable statahead for current
  opened dir.

Patch Set 1:

It looks really DANGEROUS to return ll_file_release before statahead thread quits, because then we will have nothing to protect ll_inode_info from being freed, and the statahead thread relies on its existence for sure. Should there be an extreme case that an RPC returns so late that the inode is recycled, we may crash in statahead thread accessing ll_inode_info.

Back to LU-3270, I think it is more proper to group whatever is necessary to release an RPC into a common structure and have it ref-counted. Then in the case of stopping statahead thread with in-flight RPCs, such structure can be accessed and freed safely when RPC returns/timeouts.

The original implementation is almost lockless for the "ls -l" thread to interact with the statahead staff except for ll_sa_entry_fini().
