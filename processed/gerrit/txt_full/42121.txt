LU-13594 obdclass: Add OOM handler for obdclass

This patch adds OOM handler for obdclass. The handler
currently only prints max memory that was used by obdclass
along with current memory being used before attempting
to kill the user process.

Currently, when the handler is kicked in the output under
dmesg would look like:

Output:
~~~~~~~~
...
Mar 21 07:02:02 devbox kernel: obd_memory max: 244859953, obd_memory current: 0
...

Hi Aurelien,

>I'm wondering if these 2 `pr_info()` add value as this is already called in the oom global handling code.

The idea was to provide little more info that we are indeed calling the OOM handler. other than "Out of Memory: Killed process". 

>But when CONFIG_PROC_FS is NOT defined, this function becomes a no-op. I'm wondering if we should not just wrap the all content under the #ifdef, or even don't defined the callback at all?

Since, obd_memory_sum()/obd_memory_max() rely on /proc to be present it will indeed become a no-op when /proc is not available (maybe just crashed before OOM is triggered and libcfs/obdclass is victim). Does it have any helpful info other that the pr_info() string then? No, it does not not. Only thing IMO happens when pr_info() is outside of #ifdef's is for the same reason to provide little more info that we are calling the registered OOM handler.
