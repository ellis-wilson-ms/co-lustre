Patch Set 3: Code-Review+1

Of course, I'd be happy if you accidentally changed all of the UAPI headers to LGPL during this change... :-)

If you have the cycles, it would be useful to go through the commit history for each of the UAPI headers to see the committers, and send out a separate patch for each header to change the license to LGPL with reviewers from each of the original submitters (or someone from that organization). 

The original reason my license change PR patch (see LU-5969 for history) didn't ever finish is that it tried to get signoff from *everyone* on every header and liblustreapi*.c file at once, rather than doing it incrementally for the individual files on an opportunistic basis.

I think doing it separately for each file has much more chance to make forward progress, since there may be only 1-2 "real" contributors to each file.  I exclude "trivial" updates to any file as not affecting the copyright status (e.g. whitespace changes, simple function renames, etc.)

For the UAPI headers in particular, I don't think GPL vs LGPL is meaningful. As long as the UAPI headers are just a representation of the kernel's interfaces with user space, proprietary applications should be able to use them without worrying about GPL copyleft. I don't think any of Lustre's UAPI headers have tons and tons of original library-like functionality.

I think we'd be better off focusing the LGPL for the Lustre shared libraries. Or just writing a new library under some BSD license or something.

For most cases I would agree.  All the legal info I've read is that constants and API definitions are "not copyrightable" (or Linux would have been screwed for POSIX compatibility).

However, the mere presence of the GPL license in the header can cause confusion so I would rather make it unambiguous.

 > I think we'd be better off focusing the LGPL for the Lustre shared
 > libraries. Or just writing a new library under some BSD license or

I've tried to make sure that everything new going into liblustreapi_*.c is LGPL, but nobody has had time/motivation to go through the existing liblustreapi.c code to rewrite it or at least split it up into separately-identifiable files and then request signoff from the original authors for the license change (which I don't think would be impossible given that there are still only a limited number of significant contributing organizations for this code).
