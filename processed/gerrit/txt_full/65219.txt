LU-12668 llite: force AIO sync on EC reads for recovery

For sync DIO, EC recovery on read failure runs inline in
ll_file_io_generic after cl_sync_io_wait_recycle.  AIO has
the same recovery requirement but cannot run from the sub-DIO
completion path -- ll_file_io_generic returns -EIOCBQUEUED to
the VFS before the BRW completion fires.

Rather than building a workqueue-based async recovery path,
make AIO reads on EC files behave as sync DIO: after cl_io_loop
runs cl_io_iter_init (which is where lov_io_mirror_select sets
io->ci_cross_ec), if it is an AIO read on an EC layout that
got -EIOCBQUEUED, set cda_no_aio_complete and cda_creator_free
to disarm the async ki_complete path, then drop into the same
wait + restart code sync DIO uses.  The VFS calls ki_complete
itself when we return.

The flag flip is race-free because the submission ref on
cda_sync is still held -- end_io cannot fire until we drop it
in cl_sync_io_wait_recycle.

This disables async semantics for AIO reads that hit the EC
recovery path, but those reads are already paying for parity
reconstruction so the lost AIO concurrency is small.  As a
side effect, the range_lock held by ll_file_io_generic now
remains held across the EC restart, which is what we want.
