LU-12669 lov: fix OOB access in EC recovery for PFL layouts

lov_ec_read_data_pages() passed an absolute file offset to
lov_stripe_number() to compute the starting stripe of a
recovery group. For PFL components that don't start at
file offset 0, the stripe number is wrong because
lov_stripe_number() computes (offset % stripe_width) /
stripe_size, and the component's extent start may not
be aligned to a stripe width boundary.

For example, with component 2 starting at 4M and having
stripe_size=4M with 4 data stripes (stripe_width=16M):
lov_stripe_number(4M) returns 1 instead of 0. The loop
then accesses stripes 1,2,3,4 where index 4 exceeds
lo_nr (4), causing an out-of-bounds read of lo_sub[4].
This reads a garbage pointer (e.g. 0x3836f529242869f5)
and dereferencing it for lso_status triggers a general
protection fault.

Fix by subtracting lsme->lsme_extent.e_start from soff
to make it component-relative before calling
lov_stripe_number(). This ensures recovery groups
always start at stripe 0 within their component.

Resolves kernel crash in test_46d with mixed
stripe sizes in PFL layout (bead 2-m5z).
