(2 comments)

James, it may depend on how the application is linked?  I think I would agree that using .so directly is probably wrong, but may be an artifact of some older packaging of liblustreapi.so without versioning (e.g. before v2_10_56_0-113-g7b659a1e68 "LU-5541 build: move libcfs and liblustreapi over to libtool")?  It would be useful to check old lustre RPMs to see when .so.1 was added.

I see recent Lustre utilities are linking to the actual library version:
$ ldd $(which lfs)
	liblustreapi.so.1 => /lib64/liblustreapi.so.1 (0x00007fee991aa000)
	liblnetconfig.so.4 => /lib64/liblnetconfig.so.4 (0x00007fee98f86000)

which is what I would expect.  Otherwise, if the application uses .so (instead of .so.1) then the application may break if the .so symlink points to .so.2.

That said, we may need to include the .so symlink into the main package for some versions until older binaries are retired?

We have tried our Cray compute nodes with 2.14 for our internal testing and they didn't have the lustre-devel rpms installed. Same for the 2.12 version. We have special libraries that are linked to liblustreapi and we have run latest IOR that links to liblusterapi as well. In all cases the applications always ran with no issues.

Now I do agree liblustreapi linking is a bit fragile. I have played with pkgconf setup for Lustre so projects could use that. It would always provide the correct linking and include paths. 
Even if you someone built lustre with

Let me see if I can get cycles to push that work. Note projects would have to use pkg-conf in their builds to take advantage of this. I recommend HPE here to update their build system after such a patch landed.
