LU-17022 obdclass: start converting obd flags to a bitmap

struct obd_device has multiple single-bit flags.  This usage is not
normal in Linux and requires a spinlock when there is any chance of
concurrent access.  This makes access from bh or irq context awkward
as all spinlock need to be protected.

It is more normal to declare an unsigned long, or a bitmap, and use
set_bit, clear_bit, test_bit, etc.

This patch starts the process with obd_attached and obd_set_up
becoming flags OBDF_ATTACHED and OBDF_SET_UP.

In one case test is followed by setting the flag to zero.  This is now
done with test_and_clear_bit(), so the lock isn't needed.

I was looking at some of these flags when I was looking into locking issues with the MAX_OBD_DEVICES patch. It seemed strange to me that obd_attached, obd_set_up, obd_stopping, and obd_starting are separate flags rather than one combined `OBD state` variable.  (not attached, still alloc'ing) -> obd_attached -> obd_starting -> obd_set_up -> obd_stopping -> (free OBD)

Is that the right mental model? Combining these might make reasoning about OBD state easier.

One variable does make more sense.  It is harder to implement though.
This patch, and the series it is part of, converts all these flags to use atomic test_and_set type operations so that no spinlock is needed.  There is no way to atomically update a set of 2 or 3 or 6 bits in a long so we would have to move it to a different word of memory and use a spinlock. 

Just think of it as a base-one number.  i.e. a 6bit field with values 000000 000001 000011 000111 001111 011111 111111.
