--- Message (Johann Lombardi, 2011-05-13 04:51:34.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

Fanyong, could you please clarify why we need this patch and how it is related to 1.8/2.1 interop?
The problem with this patch is that errors on async i/o won't be returned on fsync any more if a sync write failed at some point, e.g.:

sync_write(fd)   -> EIO
sync_rewrite(fd) -> success
async_write(fd)  -> success
(async writes failed in the background)
fsync            -> success

lli_write_rc (unlike lli_async_rc) is never cleared.

--- Message (Fan Yong, 2011-05-18 02:49:15.000000000) ---
Patch Set 1:

Johann, sorry I do not understand your question. The purpose of the this patch is to prevent error reporting repeatedly, which is not related with 1.8<->2.x interoperability. For example, if some async write failed, then it is unnecessary to report error when close, which will trigger ll_flush(). On the other hand, if sync/async write succeed, then "lli_write_rc" will be cleared.

--- Message (Johann Lombardi, 2011-05-18 12:26:31.000000000) ---
Patch Set 2:

> if some write fails with ENOSPC/EAGAIN/EDQUOT, or something
> else, which maybe caused by former async write operations, > then application usually closes related file and exit.
> At such time, since the application has known write
> failure, why the close operation return failure again?

The application might not consider some of those errors as fatal or could think that only the last write was unsuccessful. Therefore i think we should still notify the application of other errors on async writes (like EIO) that might have happened.
Also, EINTR can be returned on write(2) and should not mask previous errors.
