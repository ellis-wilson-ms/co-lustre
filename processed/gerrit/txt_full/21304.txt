--- Commit Message ---
LU-7650 o2iblnd: handle mixed page size configurations.

Currently it is not possible to send LNet traffic between
two nodes using infiniband hardware that have different
page sizes for the case when RDMA fragments are used.
When two nodes establish a connection they tell the other
node the maximum number of RDMA fragments they support.
The issue is that the units are pages, and 256 64K pages
corresponds to 16MB of data, whereas a 4K page system is
limited to messages with 1MB of data. The solution is to
report over the wire the maximum number of fragments in
4K unites regardless of the native page size. The recipient
then uses its native page size to translate into the
maximum number of pages sized fragments it can send to
the other node.

(3 comments)

I believe this will be much easier to follow if a big comment is added to the two structure variables, ibp_max_frags and ibc_max_frags, explaining what these variables are now storing.  From my interpretation:

ibp_max_frags is the max number of 4k fragments for the peer,
ibc_max_frags is the max number of page-sized fragments for the connection.

Very important to no mix these up as I suspect has happened in the patch.

--- Message (Doug Oucharek, 2016-08-11 23:13:47.000000000) ---
Patch Set 4:

Also:

Be sure to check kiblnd_create_conn(); it is setting ibc_max_frags to the peer value of ibp_max_frags without doing the shift.

Look at the IBLND_REJECT_RDMA_FRAGS case in kiblnd_check_reconnect(): it is incorrectly updating the peer's ibp_max_frags from frag_num which was derived from the sender's ibc_max_frags.

--- Message (James Simmons, 2016-08-12 00:16:29.000000000) ---
Patch Set 4:

I see what I did wrong. So IBLND_MAX_RDMA_FRAGS is meant to be in 4K chunks. This is due to kiblnd_init_rdma() needing to handle the 256 4K pages that need to be sent from the local Power8 to the x86 remotely. So we have to allocate the worst case which is 4K pages in kiblnd_create_tx_pool().  So its kiblnd_cfg_rdma_frags() that is wrong. It should be return the number of fragments for the native page size. Also the default value returned for kiblnd_cfg_rdma_frags() should be right on the Power8 platform. The reason I didn't see these issues was that I tune the number of frags.
