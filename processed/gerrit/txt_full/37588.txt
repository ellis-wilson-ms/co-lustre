--- Message (Neil Brown, 2020-02-17 23:06:30.000000000) ---
Patch Set 1: Code-Review-1

Arg.. I clearly didn't understand this code properly when I patched it..

Clearly there can be multiple threads running at once, though if they have the same start time (in seconds), only one will do actual work.
And that work is serialized by cfs_tracefile_sem.

There is room for a lot more cleaning up here.

Using this mutex is not a bad start, but it does have a problem.  The function is clearly designed to be interruptible, but mutex_lock() isn't.

You could use mutex_lock_interruptible(), but I think I'd prefer mutex_trylock(), and if you cannot take the mutex, assume a thread is already running, and don't bother waiting.

--- Message (Alex Zhuravlev, 2020-02-18 05:01:14.000000000) ---
Patch Set 1:

Neil, I though about try-lock approach, but that would mean that sometimes LBUG() won't get dumped: one thread called to libcfs_debug_dumplog() and right before mutex_unlock() anther thread hits LBUG(), wants to dump, but doesn't because the mutex is still locked - no LBUG() and potentially interesting preceding lines in the dumped logs..

--- Message (Neil Brown, 2020-02-18 05:21:22.000000000) ---
Patch Set 1:

> but that would mean that sometimes LBUG() won't get dumped:

I think that is already possible.

libcfs_debug_dumplog_internal() gets the time in seconds, and if that is the same as last_dump_time, it doesn't call cfs_tracefile_dump_all_pages().  So if one libcfs_debug_dumplog() happes at the start of a second, and another happens near the end, then the second will be ineffective.

Maybe it is worth putting the effort in to tidying this up properly, but for that we would need a fairly clear statement of what is needed, specifically in the face of signals.
Also, should libcfs_debug_dumplog() wait for just one thread to complete, or for all threads to complete, or for all threads that started before the one it started or ....
