--- Commit Message ---
LU-11290 osc: Batch gang_lookup cbs

The osc_page_gang_lookup call backs can be trivially
converted to operate in batches rather than one page at a
time.  This improves cancellation time for locks protecting
large numbers of pages by about 10% (after landing
another optimization (LU-11290 ldlm: page discard speedup)
it shows 6% for canceling a lock for 30GB cached file ).

Truncate to zero time (with one lock protecting many pages)
was improved by about 5-10% as well.  Lock weighing
performance should be improved slightly as well, but is
tricky to benchmark.

--- Message (Patrick Farrell, 2018-09-05 19:26:36.000000000) ---
Patch Set 1:

Sure, here's how I tested it.

Write the file on one mount (node has plenty of memory):
cd /mnt/cent7s01/; rm -f twostripes; lfs setstripe -c 2 twostripes; dd if=/dev/zero bs=1M count=16000 of=./twostripes; sync

Read the file from another mount, which cancels the lock and then reads the file:
cd /mnt/cent7s01_2/; time cat twostripes

Drop caches and read the file again:
echo 3 > /proc/sys/vm/drop_caches; time cat twostripes

The lock cancellation time is the difference between the first read and the second read.  On my VMs, it’s about 2.8 seconds without this patch, and about 2.5 with it.

Truncate is much simpler, just write to the file and then truncate it.  But the benefit was smaller as well.

"Maybe because I don't have enough background knowledge, but it isn't obvious to me why this patch improves the performance. lu_ref_del()/cl_pagevec_put() costs too much time and delay the callback waited by the other client?"

Well, I mean, the entire process of getting rid of pages takes too much time.  The reason this patch improves performance is that without it, these callbacks are called for every page, and for large files, that's a lot of pages.  So batching them reduces the number of function calls by a factor of 14 (pages in a pagevec).

"Test is very smart, but is it possible to scale up the test to like hundreds of seconds? That would be more persuasive than 0.3 second improvement."

Unfortunately, no, because what we're doing is clearing pages from the page cache, so we're limited by client memory.  We could scale it up larger with bigger nodes, but not by that much.  Those nodes had like 20 or 32 GiB of RAM?  Allowing me to have all 16 GiB of file in cache.  The biggest nodes available are still only like 128 or 256, so ~10 times bigger is the most we could go.  In my testing, the timing (as a %) was *very* consistent.  I did run the test a bunch of times. :)

This isn't a huge improvement, but it's a simple patch, and it should help some workloads.  (It was of interest at Cray primarily because Cray drops the client locks at the end of each job, so this benefit can be significant over time.  I'm still pushing it now because it's such a simple patch, and does help some.)
