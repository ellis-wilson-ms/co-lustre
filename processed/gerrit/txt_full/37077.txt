--- Commit Message ---
LU-13094 mdt: handle ENOENT lookup failre correctly

When MDT handles the IT_LOOKUP or IT_GETATTR intent lock, it will
ignor -ENOENT error when DISP_LOOKUP_NEG is set. This will cause
the client caches and revaidate a non-existed dentry wrongly.

This patch corrects this bug, let MDT return -ENOENT directly.

Signed-off-by: Qian Yingjin <qian@ddn.com>

Qian, consider the (very common) case where PATH includes several directories in Lustre (for bash for Python). Every time that some process tries to find some binary or library it will look up that name in every directory in PATH. If the negative entries are not cached then each lookup will cause an RPC to the MDS, which causes a very high load of lots of clients are doing this.  Oleg originally developed the negative dentry cache for Lustre because of severe performance problems when many clients were running Python binaries with tens of library paths in Lustre. 

There was also a discussion in upstream linux-fsdevel about this issue for non-Lustre filesystems (https://lwn.net/Articles/758847/ ), and it seems that even the kernel does not yet shrink these entries.  this hasn't been a problem in the past 10+ years that Lustre has this feature, not in the kernel for the past 20 years, so it seems unlikely that workloads will suddenly trigger this issue.

Is this causing a particular problem with WBC, or what has raised the need for this patch?
