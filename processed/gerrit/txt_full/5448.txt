even if the fo_destroy_in_progress was cleared already, otherwise
the last_id on MDS will not be synced with the last_id on obdfilter.

--- Message (Andreas Dilger, 2013-02-17 16:28:45.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

I don't understand this change. The only way that destroy_in_progress can be cleared is if done other thread has raced and gone the orphan cleanup already. This can happen in rare cases if one precreate thread is taking a long time to complete, and the MDS sends another precreate request and they are both trying to do the same thing. 

The only purpose for this flag is to detect if the orphan cleanup thread was blocked on fo_create_lock, so if it is ignore in filter_destroy_precreated() then there would be no point in having it at all. 

Maybe Mike recalls if there is some other bug in this code?

--- Message (Niu Yawei, 2013-02-18 02:13:53.000000000) ---
Patch Set 2:

The point of fo_destroy_in_progress is to abort inflight precreate, thus orphan cleanup thread can get fo_create_lock to start orphan cleanup asap.

As I mentioned in LU-2823, in mds failover case, if a orphan cleanup for original mds can't finish before the failover done, the new orphan cleanup for the failover mds will be skipped and the failover mds will have wrong last_id at the end.
