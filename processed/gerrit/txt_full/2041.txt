--- Commit Message ---
LU-957 scrub: Ancillary work for new LFSCK

1) New debug sub-system - 'D_SCRUB'
For recording Lustre scrub running trace.

2) New MDT mount option - 'noscrub'
To disable triggering LFSCK automatically when MDT mounts
up or by RPC which accessed inconsistent OI mapping entry.

3) Parameters/flags for scrub dt iteration APIs.

4) Any object which will be used when Lustre server mounts
   should not be added into OI files to guarantee that the
   Lustre server can startup even if OI files corrupt.

Alex, I would argue that aborting all scrub activity because of a single error is not very useful.  The whole point of OI scrub and LFSCK is to repair a filesystem that is inconsistent for some reason.  It is entirely possible that some kinds of unrecoverable error will be hit on fixing a single OI entry, but that only affects a single file, and the rest of the OI should be repaired as best it can be after recording and/or reporting an error.

Similarly, the LFSCK object iteration needs to continue so that it can repair the distributed filesystem state as much as possible.

I think that part of the disagreement between Alex & Fan Yong is that Alex wants a "bottom up" approach to the OI Scrub patches, something like:
- implement the inode iterator with a simple no-op user
- implement the OI scrub/repair inside the OSD only
- rate control inside OSD also rate controls callers
- implement MDD-level inode iterator on top of OSD index ops

Alex, we need to get past this impasse where you disagree with every patch of Fan Yong's.  Can you two please get on the phone and discuss specifics of how this functionality can be implemented in a manner that is agreeable to both of you.  If we need to set up a larger meeting, then please arrange with Richard/Bryon so that this can move forward.

Alex, I typically agree with your statements. One thing to clarify - there are two mechanisms by which an inode traversal are initiated.

First when the OSD starts up and detects that an OI is missing, or if during runtime it is detected that an OI entry is incorrect (e.g. inode is missing or has wrong FID in LMA, I don't think the cost of checking this on the first read from disk would be too much).

The second reason an inode iteration (not an OI iteration as you wrote) is if there is a consistency scan and/or distributed repair. This is not the OI that is opened, but should instead be a local FID that represents a virtual index with all of the inodes in it.

--- Message (Andreas Dilger, 2012-04-06 07:53:30.000000000) ---
Patch Set 5:

Alex, my comment was mainly that you were incorrectly referring to "OI iteration", but OI scrub/LFSCK are doing "inode table iteration".  In order for LFSCK to do inode table iteration, it needs to open some local FID as index that contains all inodes in the filesystem.  The inodes themselves will still use their normal FIDs, only the "all inodes" FID is treated specially.

My thought is that whenever LFSCK opens this special FID the OSD should start the OI verification.  Then, no other API is needed between LFSCK and OSD to signal scrub handling.  Of course, the OSD can do an urgent OI scrub itself if it finds an error internally (FID->OI->inode->LMA_FID mismatch or missing/broken OI file).
