--- Commit Message ---
LU-9306 tests: sanity-hsm, register traps in a better order

Traps are used in tests to define actions to perform on cleanup.
The copytool_setup() functions registers its own trap in order to
ensure any copytool that is launched gets killed at some point.

Unfortunately there is no built-in way to register several traps.
Therefore when test_24d calls copytool_setup() it overwrites
cleanup_test_24d.

This patch is a temporary fix to this issue in sanity-hsm.

Patch Set 4: Code-Review-1

(1 comment)

There is still a need to better register traps, in test_24d more than the other tests (as we saw that when 24d does not properly clean itself up, it may induce many other unrelated failures).

There are other issues as well:
 * most tests register traps after the setup action was performed, that means that if a test fails between the setup and the trap registration (thourgh <Ctrl-c> for example), the test will not clean up properly, and there may be side effects on the next test run.
 * test_12q has pretty much the same problems as test_24d ;
 * the current fix of "calling the previous trap handler from the current trap handler" is tedious and error prone ;
 * trap handler are explicitely (and unnecessarily) called at the end of tests

The current patch is far from correcting all these issues, and since there is no rush to fixing it now, I will work on a better fix.

In a nutshell, changes are needed, it will take some time but this is not a blocking issue anymore.
