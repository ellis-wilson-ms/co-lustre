--- Commit Message ---
LU-5648 ofd: In destroy orphan case always let MDS know last id

It appears that if we think MDS asked us to delete too many
precreate objects (more than 2 MAX_PRECREATE counts)
then we don't do the precreation, but do not let MDS know it needs
to advance the last_id forward by quite a bit.
This lead to reassigning these objects to other files after recovery.

--- Message (Andreas Dilger, 2014-11-10 20:29:31.000000000) ---
Patch Set 2: Code-Review+1

Alex, this check has been around for ages, and very rarely, if ever has it been a problem.  There was one bug LU-2251 which caught a corner case that was fixed in commit b33cba8714.  The chance of the MDS precreating so many objects on a single OST is very small, given the MDS will be allocating objects from many OSTs at once.

The current behaviour is "more safe" than increasing the orphan cleanup limit (at worst leaving orphan objects on the OST that LFSCK needs to clean up), compared to incorrectly deleting hundreds of thousands of objects if the orphan cleanup limit is too large.  I don't see a reason to block this patch for a potential problem that might never actually be hit.

If you think there is a problem with too many uncommitted precreates (which is only a danger with osd-zfs I think), I'd prefer to have a commit callback that tracks the number of uncommitted precreated objects, and if the uncommitted create count exceeds OST_MAX_PRECREATE / 2 it forces a commit after replying to the MDS for the precreate, rather than increasing the limit further.  That should be done in a separate patch, since I think the two problems are somewhat independent.

Patch Set 2: Code-Review+1
