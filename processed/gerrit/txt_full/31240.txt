--- Commit Message ---
LU-10649 llite: yield cpu after call to ll_agl_trigger

The statahead and agl threads loop over all entries in the
directory without yielding the CPU. If the number of entries in
the directory is large enough then these threads may trigger
soft lockups. The fix is to add calls to cond_resched() after
calling ll_agl_trigger(), which gets the glimpse lock for a
file.

If I understand correctly, your concern is that the user space thread will out pace the AGL thread when it yields the CPU and the work of the AGL thread when it resumes will be wasted and possibly slow down the user space thread. Seems to me, any work ahead operation runs the risk of being unnecessary/wasteful. It's only a guess that user app will continue doing the same work in a predictable order. Also seems unlikely that the user thread (ls -l) will be able to out pace the AGL thread since the user thread will start after the AGL thread, does more work including that done by the AGL thread, will wait for network traffic on each stat, and is subject to the same kernel scheduling constraints.

In the customer case for this bug, stopping the AGL thread would hurt performance. Most CPUs are idle; the active threads are SA, AGL and the user space scan process. The cond_resched just gives the kernel a chance to reset its timers and prevent the soft lockup. The AGL thread does not yield the CPU. I can't predict what might happen on a fully loaded system - too many variables to consider.
