--- Commit Message ---
LU-2317 mgs: both callbacks are to be fixed

The commit ORNL-27 introduced a new lock enqueue which drops a
reference on it in the blocking callbacks. however, it turned out
to be racy as if a lock is taken quickly, callback is called twice,
from enqueue completion and from conflicting lock cancel, and lock
reference is decremented twice.  LU-1259 tried to fix it, but only
for 1 callback.

Fix it for mgs_completion_ast_config as well.

--- Message (Jinshan Xiong, 2012-12-07 18:50:20.000000000) ---
Patch Set 2:

Hi Vitaly, you may miss my first comment. Actually the way you do in the patch is exactly what I did in the first place that the notify thread requested and then released IR dlm lock.

However, in the IR scalability testing where we had ~50K clients, I found that clients could hardly be granted with IR locks because there was a burst of enqueuing request so that the CPU was extremely busy. Therefore the notify thread couldn't be scheduled to release the lock for quite a long time. This issue was fixed by:
1. cancel IR lock on completion ast
2. raise the thread priority of notify thread

Based on the factor that there is no such scalable environment to verify this patch, I'm reluctant to do it this way.
