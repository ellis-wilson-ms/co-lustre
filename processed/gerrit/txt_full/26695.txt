Patch Set 4: Code-Review-1

I don't think an ioctl is a correct approach here.
Instead make the module parameter adjustable and write to /sys/modules/lnet/parameters/... when you need it changed.

--- Message (Amir Shehata, 2017-05-05 04:00:52.000000000) ---
Patch Set 4:

This would be the only parameter which uses sysfs. We don't have any infrastructure to handle changes using sysfs. And I'm not sure why sysfs would be the correct approach as opposed to IOCTL. I understand that this particular case might not be complex, but where do you draw the line? I would rather stick with a uniform approach, rather than have some parameters configurable via sysfs, others procfs and others ioctl.

--- Message (Oleg Drokin, 2017-05-05 04:13:34.000000000) ---
Patch Set 4:

From the kernel perspective: you already have the code to handle the module parameter - ioctl is code duplication.
For userspace it does not make much difference for you to open some device and do ioctl on it or to open some file in /sys and write into it, same amount of work. lustre_lnet_config_max_intf() is a whoel new function that is really not reusing any primitives here (other than l_ioctl? to save on the file open).
It looks like there are some other parameters handled that way that we better realign to be set via modules parameters only. Added benefit: you can change those outside of any pesky tools and people actually know how to do it.

We do not have anything in procfs (or should not have and are rapily migrating out, anyway).

--- Message (Olaf Weber, 2017-05-05 17:32:08.000000000) ---
Patch Set 4:

Putting anything related to LNet into sysfs at this point of the release cycle seems to me to be a bit risky. If nothing else, we would be deciding the supposedly-permanent sysfs structure for LNet in a hurry. This in addition to rushing all required supporting code.

Therefore, while I recognize the desire and need to move away from procfs, I do strongly recommend postponing sysfs support in LNet to a following release: 2.11 and maybe also 2.10.mumble. Moving to sysfs is important, but is it really that urgent?

--- Message (Amir Shehata, 2017-05-05 20:32:04.000000000) ---
Patch Set 4:

I would also question the use of sysfs for complex configuration, like network configuration, peer configuration, etc. The approach that we've been taking is doing all the heavy lifting of parsing textual configuration in userspace and pushing that down to the kernel in an easy to use binary format. Can this be done in sysfs? I think we need more time to investigate and come to a decision about what is the best holistic approach to take regarding LNet configuration, since I still think that having some configuration via IOCTL and others via sysfs is not ideal.

Therefore, I think deciding that there is a need to move to sysfs, should be postponed until we have had time to do a proper investigation. I agree that this should not be rushed in 2.10.

--- Message (Amir Shehata, 2017-05-05 20:45:48.000000000) ---
Patch Set 4:

Also please note that all of lnet configuration is under /proc/sys/lnet/. not under /sys/

--- Message (Andreas Dilger, 2017-05-08 20:13:13.000000000) ---
Patch Set 4:

Anything that is a module parameter (as lnet_max_interfaces and lnet_numa_range already are) already has an entry in /sys/module/lnet/parameters so there isn't any extra infrastructure to change, and the /sys/module/{module}/parameters path is already fixed for a long time so we don't need to worry about that changing.

The only change needed is in the previous patch to make the module parameter writable instead of read-only.
