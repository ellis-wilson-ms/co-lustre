--- Commit Message ---
LU-17999 lnet: prevent race in access to peer rtrcredits count

Refactor lnet_parse_forward_locked and lnet_post_routed_recv_locked
to have the code which checks and acts on peer rtrcredits in a single
spot, in order to avoid the race when the count is decremented
(by another thread) after being checked initially for the purpose of
"eager receiving" the message, which might cause an assert on
msg_rx_ready_delay to get triggered.

This race is possible if messages from the same peer NID are being
processed on different local NIs mapped to different CPTs.

Patch Set 1: Code-Review-1

I think we need to better understand the problem before removing these asserts. They were put there to enforce some logic in the code that is apparently broken. I would like to better understand this.

The credits are only decremented in lnet_post_routed_recv_locked(), and they are only incremented in lnet_return_rx_credits_locked().

When the credits are checked in lnet_parse_forward_locked(), we are holding the net lock from:

cpt = lnet_nid2cpt(from_nid, ni);

we continue holding this lock through the call to lnet_post_routed_recv_locked(), so we cannot race with other threads that are parsing messages from this same peer NI.

The other path to lnet_post_routed_recv_locked() is when credits are returned upon message finalization. The cpt from above is saved in lnet_msg::msg_rx_cpt via lnet_msg_commit(). We lock this same cpt in lnet_msg_decommit() before going down the credit return path. However, when returning router buffer credit we do not check if messages in on the lnet_rtrbufpool::rbp_msgs list have a different cpt stored in msg_rx_cpt. This is probably where the bug is.

We don't have this problem with the peer credits because we check for this case in lnet_return_rx_credits_locked() and will swap net locks if needed:

                       msg2_cpt = msg2->msg_rx_cpt;
                        spin_unlock(&lp->lp_lock);
                        /*
                         * messages on the lp_rtrq can be from any NID in
                         * the peer, which means they might have different
                         * cpts. We need to make sure we lock the right
                         * one.
                         */
                        if (msg2_cpt != msg->msg_rx_cpt) {
                                lnet_net_unlock(msg->msg_rx_cpt);
                                lnet_net_lock(msg2_cpt);
                        }
                        (void) lnet_post_routed_recv_locked(msg2, 1);

I think it is possible that credit counts are positive when lnet_parse_forward_locked() checks them, but become non-positive while lnet_post_routed_recv_locked() spins in spin_lock(&lpni->lpni_lock). This can happen with MR, i.e. multiple NIs per peer: lpni_rtrcredits access is protected only with lpni_lock (rbp_credits is protected with cpt lock which should be ok).

The purpose of lnet_parse_forward_locked() checking the credit counts seems to be to trigger “eager_recv” LND handler if available. The message is then marked as “ready to be delayed” regardless if LND handler got executed or not. Only GNILND appears to have this handler. (This part is confusing to me, because I don’t understand why the same message is being put on the delay queue if GNILND is able to process it “eagerly”. Maybe this is another bug?).

If the previous two statements are correct, then perhaps lnet_post_routed_recv_locked() should be allowed to trigger “eager_recv” of the message on negative credit count?
