LU-20050 ptlrpc: fix sepol_helper overflow and import ref leak

commit 8a6788b508 ("LU-20050 sptlrpc: fix positive errno in
sptlrpc_sepol_get()") fixed the case where sepol_helper() returned a
positive rc that reached ERR_PTR() undetected by IS_ERR(). Two related
bugs remain.

First, ret is declared as signed short but call_usermodehelper()
returns int. When l_getsepol exits with a code >= 128, the int return
value (exit_code << 8) overflows signed short to a negative value
outside the IS_ERR() range [-4095, -1]. For example, exit code 231
(observed on RHEL 10.1 when SELinux denies l_getsepol) produces
ret = -6400 after truncation. sptlrpc_sepol_get() passes this to
ERR_PTR(-6400), which IS_ERR() misses, and the caller dereferences
it as a struct pointer, causing a kernel panic.

Fix both the type and the formula in sepol_helper():

  int ret;
  rc = ret < 0 ? ret : -(ret >> 8);

With int ret, no overflow occurs for any exit code. If
call_usermodehelper() fails at the kernel level, ret is already a
negative errno and is returned as-is. Otherwise the exit code is
negated to a proper negative errno within [-4095, -1].

Second, the IS_ERR() branch in ptlrpc_connect_import_locked() calls
ptlrpc_request_free(), which only returns the request slab and does
not release the import reference acquired by ptlrpc_request_alloc()
via class_import_get(). The leaked refcount prevents
client_obd_cleanup() from being called, leaving the LDLM namespace
kobject stranded in sysfs. A second mount attempt fails with -EEXIST.

Fix by replacing ptlrpc_request_free() with ptlrpc_req_put() in the
sepol error path. ptlrpc_req_put() drops rq_refcount to zero and
calls __ptlrpc_req_finished(), which releases the import ref via
ptlrpc_req_drop_import_ref().
