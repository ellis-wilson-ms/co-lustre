--- Commit Message ---
LU-1267 lfsck: framework (2) for MDT-OST consistency

The LFSCK can talk with OSP directly, then the LFSCK on the MDT can
control/monitor the specified LFSCK instance on other targets (MDTs
or/and OSTs) without breaking dt_device APIs nor making OSP to know
the LFSCK things, and simplify the handling of remote OST-object or
MDT-object. For that, each OSP will register to the LFSCK will they
are added into the system. The LFSCK maintains such target table in
RAM with the similar logic as the LOD does.

--- Message (Fan Yong, 2013-12-18 08:25:12.000000000) ---
Patch Set 2:

There are several uses case for LFSCK to talk with OSP, for example:

1) The LFSCK on the MDT needs to control the LFSCK on another MDT and OST, such as start, stop, status update, and so on. If LFSCK cannot talk with OSP directly, it needs to talk with LOD, then to OSP, which require the dt_device APIs support. Unfortunately, we have no such APIs.

2) The LFSCK needs to access some specified OSP, not all OSPs, current LOD interfaces do not support that.

3) The LFSCK on the MDT needs to know the status of LFSCK on other target, it needs to know which targets are participating in the LFSCK. So it needs to record something for related OSPs. According to our former discussion, the OSP should not know about LFSCK, so we cannot add something to OSP. So with this patch, we can make all things processed inside LFSCK.

And so on.

Originally, I used LOD-OSP stack for that and introduced some dt_device APIs for that. But you said that it was not acceptable and suggested that we can resolve it via talking with OSP directly and not care about whether there will be some repeated code or mechanism as LOD.

--- Message (Fan Yong, 2013-12-18 11:03:42.000000000) ---
Patch Set 2:

When the LFSCK on the MDT triggered the LFSCK for MDT-OST consistency verification, it will notify related targets (OSTs and MDTs) to start the scanning also. But it is not all the other target will join the LFSCK, some targets may be not start up yet, some targets may be failed to start the LFSCK, or the administrator may just want to start LFSCK on single MDT instead of all the MDTs, and so on. So after the start notification, the LFSCK on the MDT needs to know which targets involved in the LFSCK, such information was recorded inside OSP originally, now inside the LFSCK with the patch.

And after the first phase scanning, the LFSCK on the MDT will scanning orphan OST-objects, but before that, it needs to know whether other targets have finished the first phase scanning or not, then it query them according to its know targets which involved in the LFSCK scanning.

On the other hand, the LFSCK instances on different targets scanning each own device in parallel, when someone finished its scanning, it will notify the others the status, which also depends on to know which targets have involved in the LFSCK.

So either we record above information inside LFSCK (with this patch) or via OSP as we did before.
