LU-1365 tests: LARGEDIR and 3-level htree for local ldiskfs

Add conf-sanity.sh test for the LARGEDIR feature with Lustre + ldiskfs
to exceed both 2-level htree (with 1024-byte blocksize to reduce the
number of entries per htree level) and 2GB directory size (marked SLOW
since this takes a while to run). This is done using a smaller number
of hard-linked inodes (nlink_max = 65000), to avoid the overhead of
accessing and caching a large number of different inodes.

Running multiple threads on a single directory is serialized by the
VFS directory mutex, so the tests are run in parallel on all available
clients and mountpoints to speed up the test as much as possible.

In total that test session ran over 6h, and the new test_125 ran over 3600s, which I think is the longest time limit for a single subtest.  By my calculations, the test had finished creating (20 * 60000) = 1.2M files, but this would only be less than half of the required 2GB directory size needed for this test.  The file create rate is also steadily dropping (four threads running in parallel together create files at 4x the shown rate, skipping some for brevity):

 total: 60000 link in 392.10 seconds: 153.02 ops/second
 total: 60000 link in 544.24 seconds: 110.24 ops/second
 total: 60000 link in 629.46 seconds: 95.32 ops/second
 total: 60000 link in 698.56 seconds: 85.89 ops/second
 total: 60000 link in 745.45 seconds: 80.49 ops/second

so that means the test would take more than 2h to complete in the VM environment, likely more than 3h, which is just too long to run for these test sessions.

If you have run this patch to completion on your (likely faster) real hardware test system, then I'm OK that the current test will at least run properly.  I don't think there is anything we can do to speed this specific test up, since splitting it over multiple directories or MDTs will defeat the whole purpose.

My suggestion would be to add a "speed test" after the first loop of test_125 to measure how fast we can create files, and if it won't complete relatively quickly the test should reformat the filesystem and be skipped:

 files * blocksize / (filenames_per_block * block fill ratio) / time =
 4 * 60000 * 4096 / (4096 / 264 * 3 / 4) = 21845333 bytes = 87273KiB / 150s = 580 KiB/s

This means it would take 2048 * 1024 / 87273 = 24 loops, or at least 24 * 390s = 9360s to create all the files.

The "speed test" could just time the first loop and "stat -c %s $DIR/$tdir" to get the the directory size and estimate the time to create 2GB of files.  Since the create rate is declining over time (already half speed after 5 loops), so we have to be very pessimistic with estimates.  Something like if initial estimated completion time is more than 20 minutes after the first loop it should abort and the test can be marked skipped.  That allows the test to be run automatically if the hardware is fast, but avoids wasting hours when the systems are slow.  This kind of functionality is unlikely to break, so even an occasional test being run would be enough.


I also note that test_106 that was added in patch https://review.whamcloud.com/26132 "LU-7001 osp: fix llog processing" is taking over 40 minutes to complete, since it is creating and deleting 5x64768 = 323840 files.  This test should definitely also be added to the EXCEPT_SLOW list (with "40" minute elapsed time in comment).  I think an llog will wrap after about 64000 records, so the test could probably be shortened as well to only create 100k files, and create files in parallel as test_125 does (divide up total files between threads)?  That would improve the create rate from 300/s to 600/s seen in test_125.
