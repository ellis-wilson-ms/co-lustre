LU-9230 ldlm: speed up preparation for list of lock cancel

Keep the skipped locks in lru list will cause serious
contention for ns_lock. Since we have to travel them
every time in the ldlm_prepare_lru_list(). So we will
use a cursor to record position that last accessed
lock in lru list.

(4 comments)

I am not sure I like the approach in this patch.
First hte "lru_nowait" list does not mean we can cancel the locks there without any waiting, it's more like "might be nowait but we need to check."

But in reality we have three classes of locks:
 - always nowait: read only locks (but not readahead), and most of metadata locks but open locks fit here
 - always wait: - metadata open locks fall here
 - transitive: - PW locks with data and active readahead fall here

Now if we onyl had the first two we might have two lists and triage the locks at LRU adding time to add to one list or the other, but the transitive locks ruin this whole thing since they might be "need to wait" at lru time and then "nowait" at the time we actually want to cancel them. This is somewhat sidestepped by current code by only triaging them once while in LRU and never changing the state after that which is kind of suboptimal in its own right too.

So I wonder if an alternative approach of kicking the blocking locks at the end of LRU list would be better and allow us to retain a single list for simplicity too?

On the surface this is good for data locks - discovering active readahead and actively dirty pages in a lock when we consider it for removal should be equal to the lock being used so why not move it to the end of LRU? It's a bit more complicated with open locks since those would be considered always used then and always sit at the end of the list - would that be ok?

We can then even stop at the first blocked lock, move it to the end of the list and stop adding stuff to our ELC (but what do we do at recovery then where we do need to traverse the entire list? It's a different traversal so should be ok I guess?).

either way we should also drop the FL_SKIPPED flag and related macroses too then as no longer used.

Vitaly, what do you think about it?
