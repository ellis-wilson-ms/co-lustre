(1 inline comment)

I have no fundamental objection to this patch landing, though my inspection is by no means comprehensive as I am not familiar with the MD RAID code myself.

Beyond the one bug pointed out inline that blocks this patch from landing, some things to point out:
- we are trying to get to a patchless server kernel, and adding more patches will not improve things
- having clear descriptions at the beginning of each patch makes it much easier to understand (e.g. bh_lru_size_config.patch)
- it makes sense to try and submit these patches to the upstream kernel so that they don't need to keep being updated

I don't think these issues should prevent the patches from landing, but this is something to address for the next release.

Denis,
my comments and questions in the previous two inspections were not addressed (e.g. descriptive comments for each of the patches, performance tests, plans for upstream submission).

Is there any plan for Xyratex to submit these patches to the upstream MD RAID maintainer?  We are trying to remove all Lustre patches to the core kernel, so that users can install Lustre against a vendor kernel, and these ones are a big step in the opposite direction.  Another option would be to create a separate MD RAID module with these patches, so that it can be installed with an unpatched vendor kernel instead of having to patch the kernel itself.

Do you have any benchmarks which show the improvement that is gained from these patches against the mainline MD RAID code?  That would be necessary for upstream submission as well, so it is time well spent.
