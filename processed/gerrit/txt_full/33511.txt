LU-11566 utils: increase llog_print buffer size

The current implementation of "lctl llog_print" uses a fixed buffer
size of MAX_IOC_BUFFER=8192 bytes, and returns the entire config
in a single buffer.  However, if the configuration is complex, it
may overflow the preallocated buffer, which is not handled properly:

    WARNING: PID: 5457 at lib/vsprintf.c:1741 vsnprintf+0x691/0x6a0
    CPU: 1 PID: 5457 Comm: llog_process_th Kernel: 3.10.0-862.14.4
    Call Trace:
     dump_stack+0x19/0x1b
     warn_slowpath_null+0x1d/0x20
     vsnprintf+0x691/0x6a0
     snprintf+0x49/0x70
     class_config_yaml_output+0x22a/0x430 [obdclass]
     llog_print_cb+0x415/0x4f0 [obdclass]
     llog_process_thread+0x892/0x15a0 [obdclass]

This is not ideal, but it is too late for a proper fix before the
release.  For now, increase the buffer size incrementally to hold
the config if possible.

Fix the buffer printing to properly detect the overflow case, and
return -EOVERFLOW if the buffer is not large enough, so that the
caller can try again.

--- Message (Andreas Dilger, 2018-10-30 19:32:58.000000000) ---
Patch Set 2: Code-Review-1

> Wouldn't it make more sense to repeatedly call the ioctl with a
 > maximum number of lines?  llog_print is in the warning/debug
 > section of lctl, so even if it is slow, it's not critical-path.

Sure, but it restarts the llog printing from the beginning each time.  We'd have to change the interface to allow it to continue from where it left off.  Probably the easiest would be to do it one record at a time, so that each line was always guaranteed to fit into the buffer, but it would need more changes both to lctl and the kernel.

That said, the current approach isn't as simple as I thought it would be, and is failing when the buffer size is large, so I need to debug it more.

If you want to look at the "return one line per ioctl()" or YAML-formatted debugfs file approach in parallel that would be great.  We want the simplest fix possible for 2.12 so that it might still have a chance to be included into the release, and  larger improvements should be targeted for 2.13.

--- Message (Andreas Dilger, 2018-11-20 22:24:43.000000000) ---
Abandoned

Abandon this approach.  It turns out that llog_print already has a mechanism to specify start/end records, so that should be used instead.
