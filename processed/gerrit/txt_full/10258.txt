--- Commit Message ---
LU-5018 libcfs: don't depend on configure for lustre_user.h

Don't depend on configure to detect the presence of <linux/types.h>
in order to define the __u8, __u16, __u32, and __u64 types.  Instead,
just define these types directly if <linux/types.h> is not included
already.

The <linux/types.h> header is available on Linux systems with GCC
installed, which is required to build Lustre anyway.  On non-Linux
systems they should create an appropriate <linux/types.h> compat
header in the include path.

--- Message (Christopher J. Morrone, 2014-05-12 23:20:18.000000000) ---
Patch Set 3: Code-Review-1

This patch removes defines like HAVE___S8, but leaves behind dead code in the form of all of the consumers of those defines.

I can perhaps get on board with using the __[su]XX style types in user space for ioctl structs, even though I think C99 style would really be saner.  But I think that we need to then have the Lustre headers in user space perform the correct inlude:

#include <linux/types.h>

I do not think our first move should be to define those values ourselves.  If you look at our definitions, I would say there is a pretty reasonable probably that we aren't doing them correctly for all architectures.  Sure, we hacked it once for PPC, but what about all the rest?

--- Message (Andreas Dilger, 2014-05-13 00:36:20.000000000) ---
Patch Set 3:

The reason I'm hesitant to include <linux/types.h> in userspace is that this is not guaranteed to be installed. This behaviour is similar to what e2fsprogs is doing. The reason to keep the #ifdef HAVE___U32 is to avoid clashes with the ext2_types.h header - I've just sent a patch upstream to get those headers to also #define these constants to avoid name clashes. 

The other option would be to always include <linux/types.h> instead if making it conditional, and then install our own header that #defines _LINUX_TYPES_H to avoid later name clashes.
