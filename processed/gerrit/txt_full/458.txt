LU-118 clear_inode: BUG_ON(inode->i_data.nrpages)

This patch does a lot of fixes and cleanup:
 - grab an inode refcount by top cl_object, so that it can't go away
 - cl_object now can be purged by shrinker
 - change the way to create cl_object, now it will be created in
   ll_file_open, ll_setattr_ost, etc.
 - revise lu_object LRU, busy objects are not in LRU list any more
 - cleanup the code by defining cl_object_kill

Future work:
 - now pages and locks are holding a refcount against cl_object, this
   causes cl_objects can't be purged by shrinker if there exists pages
   and locks for an object. This is very bad, it needs fixing.
 - fairness of lu_object lru algorithm - lru is now tightly coupled with
   hash, must fix.

To wangdi: I've thought about your proposal, there are two difficulties to hold an inode refcount for each page:
1. the overhead of grabbing an inode refcount is not trivial. From the implementation of igrab(), it has to hold a global spinlock inode_lock, this would make it super busy;
2. The race between clear_inode() and cl_lock_page_out(). This is a livelock case where the former is called when the final inode refcount is dropped(we may do this inside cl_page_put()), then clear_inode will clear cl_object by waiting for the object refcount to reach zero. However, the same process is actually hold the cl_lock to call cl_lock_page_out(). We may make extra effort to fix this problem.


Anyway, I realize this patch is too long to be inspected. I should have split the patch into multiple parts.

Let's see Wally's test on a system with less memory, with the patch of ll_delete_inode(). If we can survive, maybe we can drop this and use ll_delete_inode() solution, though it's a workaround one.
