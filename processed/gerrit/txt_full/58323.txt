LU-18456 lustre: Implement recycle bin for deleted files

This commit introduces a recycle bin feature to prevent permanent
loss of data due to accidental deletion. Instead of immediately
removing files when users execute 'rm', the files are moved to a
special '.recycle_bin' directory where they can be recovered.

The feature is implemented as follows:
1. Client-side filter that intercepts unlink operations and renames
files to the recycle bin when enabled
2. New proc entry 'llite.*.enable_recycle_bin' to toggle the feature
(0=disabled, 1=enabled)
3. New command set 'lfs recycle' with the following subcommands:
  - enable: Turn on recycle bin for a filesystem
  - disable: Turn off recycle bin for a filesystem
  - status: Show current recycle bin status
  - list: Show files in the recycle bin with original names and deletion times
  - restore: Recover a file from the recycle bin
  - purge: Remove files older than a specified time from the recycle bin
  - empty: Clear all files from the recycle bin
4. Tests in sanity.sh to validate all recycle bin operations

Saved filenames include original name  and timestamp to
ensure uniqueness and provide metadata for recovery.

The recycle bin feature is disabled by default and can be enabled
with 'lfs set_param llite.*.enable_recycle_bin=1' or 'lfs recycle enable'
command.

Patch Set 2: Code-Review-1

One issue with implementing this functionality on the client side is that it isn't possible to determine (race free) whether this is the last link to a file or not.  It would be useful to have a parameter (on the MDS) whether all deleted hard links to a file should be moved to trash, or whether only the last unlink should move the file to trash?

It should be possible to use the fileset from the client export's nodemap to determine any subdirectory mount to determine which trash directory to move the file to.  However, if the subdirectory mount is implemented with a bind mount of a subdirectory then it might not be possible to access the .Trash directory in the filesystem root, but it might be possible to access the deleted file by FID?  Definitely once we have a per-directory virtual .Trash directory it will be accessible everywhere.
