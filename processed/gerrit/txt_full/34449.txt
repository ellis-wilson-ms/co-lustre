LU-12090 utils: lfs rmfid

a new RPC_REINT_RMFID has been introduced by the patch.
it's supposed to be used with corresponding llapi_rmfid()
to unlink a batch of MDS files by their FIDs. the caller
has to have permission to modify parent dir(s) and the objects
themselves.

A bunch of small stuff in the tests & it kind of looks like we still let people ignore path permissions with this.  I still don't think letting regular users ignore path permissions is acceptable.

Consider this common case:
User directories are set up by the admin with permission specific to the user
Users create directory trees under their user directory (without permission inheritance, which is default)
Their files are protected only by permissions on the path.

Otherwise, anyone who can guess the fid (which is allocated predictably) can delete them.  Or if a user mistypes their own fid, they can delete other users files.

I ran permission check oriented tests with this patch, and found out the following:


root cannot 'lfs rmfid' (-EACCES) a file for which rm would work:


   # ls -lR /lustre
   /lustre:
   drwxr-xr-x 2 user0  user0  4096 Jun 26 22:48 user0
   
   /lustre/user0:
   -rw-rw-r-- 1 user0 user0 0 Jun 26 22:42 file3


Other use case where I get -EACCES with 'lfs rmfid', whereas rm works:


   # ls -lR /lustre
   /lustre:
   drwxr-xrwx 2 user00 user00 4096 Jun 26 23:10 user00
   
   /lustre/user00:
   -rw-rw-r-- 1 user00 user00 0 Jun 26 23:10 file4
   # lfs path2fid /lustre/user00/file4
   [0x200000402:0x8:0x0]
   
   $ id
   uid=500(user0) gid=500(user0) groups=500(user0)
   $ lfs rmfid /lustre "[0x200000402:0x8:0x0]"
   rmfid([0x200000402:0x8:0x0]): rc = -13
   rmfid(): rc = -13
   $ rm -f /lustre/user00/file4
   $ echo $?
   0


Is that because the user issuing the 'lfs rmfid' command is not the owner of the file?
This is weird as regular 'rm' does not care about permissions on the file itself.


On a side note, I am in favor of the already expressed idea of a new nodemap property that would decide whether the client is capable of doing this kind of operations. We could imagine a more generic implementation, like a bit mask that would let clients use different subcategories of the llapi. For instance, the use case I have in mind is to control which clients can consume changelog entries (= audit records).
