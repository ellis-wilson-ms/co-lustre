--- Commit Message ---
LU-9437 lfsck: handle LMV EA for migrating directory

For the in-migration directory, its LMV EA contains not only the
LMV header, but also the FIDs for both source and target. So the
LMV EA size is larger. The lfsck_read_stripe_lmv() logic need to
handle such case properly.

--- Message (Andreas Dilger, 2018-02-16 21:21:11.000000000) ---
Patch Set 2: Code-Review-1

This change doesn't seem very flexible. If the LVM size changes again at some point in the future, then the LASSERT() will fail and someone will have to make a patch to change the buffer size manually.

Why not just use the existing large LOV buffer, which is dynamically allocated to match the largest size of xattr seen? This should be OK, since this code path is dealing with a striped directory, while the other buffer is only for regular files, but if that isn't possible to share for some reason, then at least the large buffer size should be allocated dynamically as needed, and then used for all xattrs instead of using it only when the regular xattr buffer is too small. That avoids the extra calls to fetch the xattr in most cases.
