--- Commit Message ---
LU-18072 ptlrpc: do not search for duplicate cancel requests

Cancel requests don't have any max inflight limitations, so
really could arrive in huge numbers and if they are also have a resent
flag, that leads to a lot of very expensive duplicates search that is
totally unneeded at the same time, so let's skip the check for cancels.

> Patch Set 1:
> 
> isn't this just delaying potentially more expensive handling of LDLM_CANCEL by regular means?

no, I don't think so.
1. the cancel itself is not single threaded unlike the duplicates search (per-export spinlock while searching)
2. cancels (and requests in general) are eagerly dropped if they are from a stale import anyway (We see this aplenty in the logs when thi bug hits in particular)
3. the "expensive" part in the disk access (cache flush) only could happen once anyway since nobody else could find the then invalid handle once done (and even before I think)
