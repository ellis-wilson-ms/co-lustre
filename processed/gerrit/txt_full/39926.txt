--- Commit Message ---
LU-13669 readahead: increase ra window progressively

For KX applications(https://kx.com), use default read ahead
per window will hurt performance, we only get good numbers with
1M window size.

The reason behind that is it is not really sequenetial mmap read,
however sequential read RPC size increased as one RPC size even
first time, this potentially cause a lot of page discard.

To overcome problem like above, RPC size will be started as 32K
and increased twice every time before it reach RPC size(16M etc).

Once we reach RPC size, it will grow one RPC size each time as before.
With this patch, we got same performance numbers with 1M and 64M
read_ahead_per_file_mb.

--- Message (Andreas Dilger, 2020-09-26 00:11:19.000000000) ---
Patch Set 1: Code-Review-1

This needs to include the performance improvements seen during testing into the commit message, to make it clear what the benefit of the patch is.

Did you run *other* benchmarks with this patch applied?  I'm concerned that with an initial RPC size of 32KB it will take 4 RPCs _just_ to read the first MB of data, so I think this _could_ hurt very common workloads like sequential reads of few-MB files.

Instead, I wonder if the readahead algorithm needs to be a bit smarter, so that it *doesn't* grow the readahead window as aggressively, or maybe even shrinks it, when there are a few sequential 32KB reads but they never get larger than this?

Tracking something like the strided readahead window size, but using a *weighted average* that decides the most common IO size would likely help.

Then, if e.g. the reads are always 24x4KB=96KB of partly-sequential mmap IO, but offset is random in between, the client would prefetch 96KB on each non-sequential read, instead of only 4KB for the page and then get a "false" readahead with the next 23 sequential reads and then reset to 0 after the next seek.

In this case, it would _also_ be OK if the default readahead was growing quickly like it does today (e.g. up to RPC size after 2 or 3 reads), since the first few accesses are "learning" and we need to optimize for the common case.  The important thing is that after each non-sequential access, the (end - start) size is added to a weighted average RA window size for that fd that is used to help the next readahead.  The RA window size could be IO size or RPC size initially, but "learn" from the IOs after that.

I'm going to mark this -1 for now, but I'd be willing to change my mind if there was more information about the behavior for other workloads.  It may be that "max_read_ahead_whole_mb" will handle the case of files < 2MB, and for very large files the extra RPCs will not matter, so files in the 4-8MB range would be most important to test.

Patch Set 2:

So, I don't think this patch will improve *performance* in almost any specific benchmark.  It's still a good idea, I think.

It's going to do two things.

1. Improve *fairness*/reduce wastefulness.  If a process reads just a small part of a file today, the existing logic will pull in a full RPC.  That's generally good for performance *of that process*, at least if it reads more than a few pages.  But it can be wasteful in many situations if all those pages aren't used, so at a system level it could improve things.
2. Improve handling of patterns we *don't* detect correctly, situations where something 'fools' readahead in to thinking it sees a pattern but it's mostly missing.  Right now, the first readahead we do is a full RPC, so as much as 4096 pages.  If readahead is mispredicting, that's a lot of data to pull in.

I suspect this is the case for KX - It's not showing up as totally random (perhaps reads are within the "loose_sequential_read" window of 8 pages in either direction), but it's not being predicted correctly either.  So shrinking the window makes it less wasteful.

--- Message (Andreas Dilger, 2021-10-03 20:37:06.000000000) ---
Patch Set 2:

Patrick, the "read full RPC" used to be shorthand for "read one MB", which was the same as a RAID stripe, which was the most efficient unit of disk IO due to seeking.  With larger RPCs (4MB/16MB/+) this is probably overkill for some workloads.

What would be ideal is having readahead not be all-or-nothing for pattern detection for a process.  Clearly a *single* misprediction at the start of reading a file shouldn't be the end of the world either way, though I would expect that _most_ applications do sequential reads over random reads, so that should be the default.

However, we shouldn't _keep_ mispredicting this behavior repeatedly for the lifetime of an application, and definitely not for a single file.  If an application keeps on faking readahead out and then going random (i.e. readahead window keeps growing too large, resulting in discarded/unused pages), this should become increasingly "sticky" and make it start with a smaller window and harder to grow the readahead window too large.

This could be done with a decaying average for the initial readahead window that starts at 1MB, but shrinks quickly when the readahead window is reset due to random IO, and grows more slowly.  For sequential reads this initial value would only ever be used once.  

Ideally this could be saved on a per-process basis, but it might be enough to do it on a per-client basis, assuming that an application is running on the same node for some time?
