--- Message (Andreas Dilger, 2023-05-11 21:55:44.000000000) ---
Patch Set 1: Code-Review-1

(10 comments)

Consider the case of

    lfs find ... -printf "... %LP ... %p"

so it is processing a bunch of files and expects a specific output format for later processing, and then there happens to be a special file in there (socket, FIFO, char, block) that don't have a projid set.

As a user, I'm not sure I'd want that to be considered a "failure" and not print anything for "%LP"?  I'd still want "lfs find" to print the same number of fields as requested so that later processing of the output is not broken.

My preference would either be that it prints a projid -1 (INVALID_PROJID) and possibly prints a message to stderr, but it should definitely continue to print the rest of the fields, and process the rest of the files.

Alternately, it could print "0" in this case, because all files that do not have an explicit projid set will use the default projid = 0 internally.  It is just the inability to call the ioctl() on the special file that is the problem, and not the projid itself.  In some sense, this is "adding" an error condition that didn't previously exist without projids.
