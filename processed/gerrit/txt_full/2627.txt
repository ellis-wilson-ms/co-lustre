--- Commit Message ---
LU-903 ldlm: inode references moved to resource

There is a race condition while get_attr after cancel_lru_locks
and sysctl drop_caches. ll_clear_inode clears l_ast_data for
ldlm lock and this lock can't be canceled because "inode == NULL".
ll_mdc_blocking_ast finds such lock. As result DCACHE_LUSTRE_INVALID
is not set and lookup returns wrong inode.

This patch moves inode structure reference to
"ldlm_lock::l_resource::lr_lvb_inode". This prevents from different
inode references for same resource's lock.

Unfortunately we cannot change osc part so easily as CLIO stores osc-lock pointer there instead of inode, see osc_lock_build_einfo
    einfo->ei_cbdata = lock; /* value to be put into ->l_ast_data */

llu part which handles extent locks still uses inodes, calling lov/osc directly, not through CLIO, however osc part is common for CLIO and such a direct call, i.e. osc_enqueue_base() still stores data in lock, thus we cannot take it from resource in llu

Thus we need old ldlm_inode_from_lock() and new function that get inode from resource. Unfortunately,  ldlm_inode_from_lock already used, so  ldlm_inode_from_resource looks like good choice for me.
