LU-11811 mdt: Add a proc entry to set MDT to readonly

It is sometimes desirable to be able to mark the filesystem read-only
directly on the server, rather than remounting the clients and setting
the option there.  This can be useful if there is a rogue client
that is deleting files, or when decommissioning a system to
prevent already-mounted clients from modifying it anymore.

Add the mdt.*.readonly parameter to allow setting the MDT read-only
immediately if set to 1.  All future MDT access will immediately
return -EROFS until the parameter is set to 0 again.

(8 comments)

I think you would need to also add a readonly parameter for the OSTs, since clients may be writing to files that they already have open.

I was just thinking that it may be desirable (in a separate patch) to be able to cancel all LDLM locks from the server as well (e.g. lctl set_param ldlm.namespaces.*.lru_size=clear would work on the server to send blocking ASTs for all locks on that namespace), to ensure that clients need to contact the MDS to e.g. reopen files for write instead of using locally-cached locks.

Also, separately, it might be useful to have a per-export parameter mdt.*.exports.<nid>.readonly to allow setting a single client read-only if needed.  This could be set directly by the admin on the export, or via a nodemap on a range of NIDs, similar to how nmf_allow_root_access is handled.

IN any case, -1 is because of test version check is broken for master and will cause interop test failures.  I think readonly for OSTs would be easily done and useful to add to this patch.
