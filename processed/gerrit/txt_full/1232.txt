1.8 code supports the feature not since 1.8.0, so it was disabled
by default in upgrade cases. however, as we maintain the upgrade/downgrade
on 1 version only, it is already safe to enable it by default.

--- Message (Andreas Dilger, 2011-08-25 18:08:43.000000000) ---
Patch Set 1: Do not submit

Setting this feature on first mount needlessly breaks compatibility, and may cause problems for other filesystems.  

It is much better that a the user run "tune2fs -O dir_data" on the filesystem to enable this explicitly, or that it is done as part of the operation of the "upgrade 1.8 to 2.0" tool.

--- Message (Vitaly Fertman, 2011-10-03 18:19:41.000000000) ---
Patch Set 1:

it seems there is no compatibility issues, because:

1. this lands to 2.x code;
2. dirdata support exists since 1.8.2 and we support upgrade/downgrade
on 1 version only;

===
On Sep 2, 2011, at 10:05 PM, Andreas Dilger wrote:

One concern is that the dirdata feature, while having a reserved
feature flag in upstream e2fsprogs to avoid future conflicts, is
not currently likely to be accepted upstream.  Right now there is
no code other than Lustre that needs this extra space, and it adds
some complexity that is otherwise useless.

There is some chance that the upstream e2fsprogs may still accept
the code to validate extra dirdata fields, but I'm having delays
in getting the MMP checking patch included upstream even though
the MMP feature itself was included into linux 3.0, so getting the
dir_data support in upstream e2fsck will take a long time.
===

3. this is enabled by default in new created 2.1 fs, so enabling it 
on the first mount of 2.1 fs after upgrade is not worse.

4. it is better to have it enabled by default if there are no more issues,
as files are handled faster when having dir-in-dintry.  to be clear,
existing data are not changed here, just new files will get it.
