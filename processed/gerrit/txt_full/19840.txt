When support for newer kernels was backported from the
upstream kernel it lacked any of the enhancements done
for newer version of lustre. This work makes the newer
kernel support lockless writes like the rest of the
lustre llite code.

Patch Set 3: Code-Review-1

Mmm, I think I disagree...
I think there's a race where someone can set the suid bit while we're calling __generic_file_[aio_write/write_iter]...

That would let people trigger the suid modification code from __generic_file_[aio_write/write_iter], when i_mutex is not held.  I don't think that's OK, whether or not we checked IS_SEC first.

Other file systems check IS_SEC and directly decide whether or not to do the suid modification, taking i_mutex if they are doing suid modification (so they will never do the suid modification without i_mutex held), but we'd check IS_SEC outside of __generic_file_[aio_write/write_iter], and it could change - Then we'd call the suid modification code without i_mutex held.

Other file systems avoid this by implementing their own version of __generic_file*.  It's looking more and more like we can't avoid that.

Various in kernel file systems do this, and some future day when we add LU-1669 to the kernel, we'd need to do it in the kernel as well.

--- Message (James Simmons, 2016-04-29 18:38:20.000000000) ---
Patch Set 3:

So we need to ALWAYS take the lock. Then lock-less is not really possible is it.

--- Message (Patrick Farrell, 2016-04-29 18:42:43.000000000) ---
Patch Set 2:

Well, not *completely* lockless, but there's an enormous difference between taking the lock around a few function calls resulting in changing one bitmask and taking the lock around the entire write.

I suspect that if we do what XFS does and take it just for a few operations, we couldn't even measure the performance drop that resulted from taking the lock for such a short time.  And actually, if we check the IS_SEC thing before taking the mutex, it would only matter for SUID files, which are rare anyway.

--- Message (Andreas Dilger, 2016-04-29 21:54:01.000000000) ---
Patch Set 3:

Since the chance of having SUID/SGID files in Lustre is very low, I'm not against simplifying the code by deciding whether to call generic_file_aio_write() or __generic_file_aio_write() based on a likely(IS_NOSEC()) check at the start of the write.  It looks like we would need to add MS_NOSEC to the mount flags and explicitly set S_NOSEC in ll_update_inode() if !is_sxid(inode->i_mode) is true. MS_NOSEC goes back to 2.6.39, but doesn't appear to be in RHEL6.x that I checked, so we would probably want to add compatibility handling for the older kernel.

As for the concern of setting SUID/SGID flags on a file in the middle of a write when we don't hold i_mutex, I think that is always racy regardless of whether i_mutex is held or not.  XFS only checks IS_NOSEC() at the start of the write, but it doesn't hold i_mutex over the write if IS_NOSEC() is true, so it doesn't protect against SUID/SGID being set again in the middle of a lockless write.
