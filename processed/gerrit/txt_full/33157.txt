--- Commit Message ---
LU-11193 llog: Do not write to read-only devices

Check if device is read-only before trying to start a transaction
on a devices.  Lustre snapshots are generally read-only.

When underlying device is read-only we check in the following places:
* llog_destroy, llog_write - return EROFS
* llog_open_create - on the create sized we return EROFS
* llog_cancel_rec - we return 0 which means success where 1 means
"success & log destroyed"

--- Message (Fan Yong, 2018-09-14 14:01:11.000000000) ---
Patch Set 1: Code-Review+1

> Why return 0 for some cases, -EROFS in others?

There are two main cases for the modifications against the read only device:

1. Client sponsored modifications. For these kinds of modifications against read only device, the server will return -EROFS. That will make related client/application to know the failure. But the server can run normally.

2. Servers internally sponsored modifications. Most of these modification may happen during the system mount. If we return -EROFS to related caller, it may cause mount failure. Because it is not easy to strictly distinguish who sponsors the modification, then as long as it may affect the server mount or system running, then we will drop related modifications silently.

For this case, the llog related modifications may be related with Lustre configuration or DNE update logs. So "RETURN(0)" can avoid failing the mount or recovery. If we can have more hints for the modification sponsors, we can distinguish them more detailed. But now, skip the modification silently may be the safe way.
