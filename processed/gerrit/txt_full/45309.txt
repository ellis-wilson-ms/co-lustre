--- Commit Message ---
LU-15133 osp: only deactivate OSP on LAST_FID error

ofd_get_info_hdl() should return -EFAULT upon LAST_FID error, which
is the same as LAST_ID error.

osp_get_lastfid_from_ost() should deactivate OSP only upon -EFAULT,
which means reading LAST_FID on OST failed. This can avoid unnecessary
admin intervention.

Add sanity 27S.

--- Message (Andreas Dilger, 2021-10-20 20:41:37.000000000) ---
Patch Set 1: Code-Review-1

(4 comments)

It would be useful to have a test case for this, which drops the OST_GET_INFO RPC for some reason, or otherwise returns an error.

I _think_ there is a bug in the patch, because osp_get_lastfid_from_ost() is requesting KEY_LAST_*FID*, but that does not return -EFAULT on error.  I see that KEY_LAST_*ID* returns -EFAULT on error, but that is the compatibility interface for very old MDTs and should be removed at this point.  It probably makes sense to return -EFAULT for the KEY_LAST_FID case also, so that the MDS can distinguish this from other local/network errors.  There would be a minor compatibility error for some time, but not worse than the current situation where every error results in an inactive OST.
