--- Commit Message ---
LU-1267 lfsck: enhance RPCs (3) for MDT-OST consistency

The LFSCK on the OST uses LFSCK_NOTIFY RPC to notify the LFSCK
on the MDT about the LFSCK progress for the layout consistency
verification. And uses the LFSCK_QUERY RPC to query the LFSCK
status on the MDT.

The LFSCK RPC from OST to MDT is sent via the reserse connection
from OST-x to MDT-y.

--- Message (Andreas Dilger, 2013-12-07 00:45:27.000000000) ---
Patch Set 13:

Since the MDT is requesting status from the OST every 30s, it doesn't really make sense to me that the OST needs to call SET_INFO itself to send the status.  Why not just return the status the next time that the MDT asks?  Since the OST-side LFSCK is driven entirely by the MDT, there is no harm if the OST LFSCK is sitting idle for some minutes until it hears from the MDT again.

AFAICS, that avoids the need for some hundreds of lines of code in this patch.

--- Message (Fan Yong, 2013-12-14 01:00:30.000000000) ---
Patch Set 13:

1) On OST side, the rbtree for handling orphan OST-objects may occupy a lot of memory. Under such case, it is better to release those memory as fast as possible.

2) My original idea is that, the RPC from OST to MDT for LFSCK event notification is the main mechanism, the RPC from MDT to OST for LFSCK query per 30 seconds is the backup in cases of the OST failure of sending RPC to MDT or MDT failure to receive the RPC from OST.

3) 30 seconds internal for LFSCK query internal may be not fixed, when the system become larger, we may consider to increase the internal in the future.

4) This RPC can be removed relative easy if it is not much useful in the further. But reverse case of removing it now and adding it back when necessary may be some trouble because it will cause interoperability issues.
