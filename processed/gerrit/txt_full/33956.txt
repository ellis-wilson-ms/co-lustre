--- Commit Message ---
LU-11739 lod: don't inherit default layout from root directory

There is no need to inherit the default directory layout from
the root directory when subdirectories are created therein.
This consumes xattr space on the subdirectories, and makes it
more complex to change the filesystem default layout in the future.

This patch fixes the above issue in lod_ah_init() to check if
the parent directory is the root directory and not copy
the default layout xattr to the new subdirectory.

Since I could not reproduce the failures manually, I submitted a for-test-only patch in https://review.whamcloud.com/34037 to reproduce the failures and gather full debug logs. Test results showed that the failures didn't occur consistently.

In the failed test session, the main issue is that sometimes creating a sub-directory under root directory failed as follows:

 mkdir: cannot create directory '/mnt/lustre/d311.sanity': Invalid argument

Lustre debug log on MDS showed that:

 (lod_object.c:5164:lod_declare_create()) Process leaving via out (rc=18446744073709551594 : -22 : 0xffffffffffffffea)

The "-22" error code came from the following codes in lod_declare_create():

                                cfs_foreach_bit(ltd->ltd_tgt_bitmap, i) {
                                        tgt = LTD_TGT(ltd, i);
                                        if (tgt->ltd_index ==
                                                lo->ldo_dir_stripe_offset) {
                                                found_mdt = true;
                                                break;
                                        }
                                }


                                /* If the MDT indicated by stripe_offset can be
                                 * found, then tell client to resend the create
                                 * request to the correct MDT, otherwise return
                                 * error to client */
                                if (found_mdt)
                                        GOTO(out, rc = -EREMOTE);
                                else
                                        GOTO(out, rc = -EINVAL);

It means the MDT indicated by lo->ldo_dir_stripe_offset can not be found. This review-ldiskfs test session has only one MDT, but in its debug log:

 (lod_object.c:4739:lod_ah_init()) Process entered
 (osd_handler.c:314:osd_idc_find_and_init()) lustre-MDT0000: FID [0x200000007:0x1:0x0] not in the id map cache
 (lod_object.c:4685:lod_striping_from_default()) striping from default dir: count:1, offset:2, hash_type:2
 (lod_object.c:4815:lod_ah_init()) final dir stripe [0 2 2]
 (lod_object.c:4817:lod_ah_init()) Process leaving

The ldo_dir_stripe_offset is 2, which is incorrect! What's more, lod_striping_from_default() with the above output should not be in the code path because lds->lds_dir_def_striping_set should not be true (lod_get_default_striping -> lod_get_default_lmv_striping was skipped by fid_is_root(lod_object_fid(lp) condition).

It looks like a previous sub-test changed the root directory layout. I added more debug codes to do further investigation.
