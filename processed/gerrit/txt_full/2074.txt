lol~ you can absolutely choose to use radix tree for free. I meant to say  radix tree is designed to manage sparse indices. Page index is a perfect match. However, for UIDs, it would be in a format of 0x1,2300,0001~0x1,2310,0000 assuming it has to support 100K users. That means most of uid is centered in a specific region and radix tree will be degenerated to an array because it has narrow top nodes and fat and contiguous leaf nodes.

--- Message (Andreas Dilger, 2012-05-15 18:00:34.000000000) ---
Patch Set 7:

If this patch needs to be resubmitted (due to failure in sanity-quota.sh, which AFAIK is not a common failure), please use tabs for indentation.

Also, I wonder that a radix tree is the right data structure for this use, compared to a hash table?  How many UID/GID values are expected to be close to running out of quota at any given time, and is there any chance to have large contiguous ranges of them (which is AFAIK the best use case for a radix tree)?

It _should_ be that Liang's SMP-optimized cfs_hash implementation resolves the performance issues, and it makes sense to avoid extra complexity and different data structures unless there is a need for them.  My understanding is that RCU adds a fair amount of complexity to the code and is not necessarily easy to keep working correctly because of the non-obvious rules for lifetime of the data structures.

--- Message (Johann Lombardi, 2012-05-15 20:51:37.000000000) ---
Patch Set 7:

Sure, i'm fine to try again with a hash table and Liang's improvement, although i really doubt it will perform better than RCU with no locking in most of the cases. It is also worth noting that radix trees are now commonly used in lustre since CLIO uses it to manage cl_pages.
