LU-5275 lprocfs: remove all non-seq file functions

With the completion of the move to seq_file based
proc handling for lustre we can remove all the
no longer used non-seq_file handling routines.
Rename lprocfs_try_remove_proc_entry to match the
new function in newer kernels and RHEL6.6 that
does a similar thing (remove_proc_subtree).

(1 comment)

I have been doing some thinking about this since this is not a easy problem to solve. The main problem is that in newer kernels the proc layer is abstracted and hidden away. You can't see the proc tree structures or handle that tree locking your self. To do the same functionality of lprocfs_try_remove_proc_entry with newer kernels we have to create a new abstraction on top of the proc one with its own locking and some type of data base to track everything. A simple reference count will not work. Basically we have to create our own proc like system. 

So the conclusion is that procfs is not the system to use for newer kernels since what Lustre does is beyond procfs scope of what it can do. At this point I think looking at using sysfs on new kernels is the way to go. Looking at sysfs I see things can be managed as groups so we can have a root structure be shared with different groups.

Actually, I think there is a relatively simple solution that doesn't involve hacking procfs or requiring an immediate conversion to sysfs (while desirable I don't think there is time before 2.7.0).  The basic solution would be for obdclass to track these shared OBD device proc root devices. The simples way would be a bit of a hack and just have global counters for these two devices and only delete the proc root when the refcount is zero. A slightly better fix class_register_type() to create a simple hash table that tracks the refcount of all of the OBD root device types.

That keeps the complexity inside this specific piece of code, and doesn't need any changes to the underlying proc code.
