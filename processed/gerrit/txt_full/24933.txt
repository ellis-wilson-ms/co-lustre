--- Commit Message ---
LU-8964 llite: Make readahead async

Removed current Lustre implementation of readahead and replaced it
with in-kernel implementation. Adopted code for in-kernel readahead
usage. Perform readahead requests asynchronously from current I/O.

In combination with parallelization of generic I/O this patch allows
significantly improve read performance for KNL.

--- Message (Andreas Dilger, 2017-02-08 23:37:51.000000000) ---
Patch Set 5:

I've long been an advocate of using the in-kernel readahead code rather than duplicating our own version.  However, if the lack of strided readahead is going to cause significant performance loss then we need to re-examine this.  It wouldn't be bad to also have backward strided readahead.  The other areas that I think may be significantly impacted are the loss of whole-file readahead if many small files are read, and RPC and stripe-size alignment of reads (not sure if that is included here or not).  

The kernel's readahead implementation will unfortunately split readahead ->readpages() calls into 2MB chunks regardless of what the SBI ra_pages value is, which will be sub-optimal if the RPC size is 4MB or larger.

It would be useful to see what kind of RPC pattern is generated for the kernel readahead vs. the current Lustre readahead code for a few read patterns besides linear streaming reads, for example strided, random, and many small file reads (e.g. 1MB or 2MB).

We may consider to push changes to the upstream kernel to address some of the shortcomings in the kernel readahead, but it would take some time before this became available to patchless Lustre clients unless we also convince vendors to backport the changes (which is considerably easier if there are significant performance gains).
