LU-10131 llite: Update i_nlink on unlink

Currently, the client inode link count is not updated on
last unlink.  This is fine because the dentries are all
gone and the inode is eligible for reclaim, but it's still
incorrect.  This causes two problems:

1. Inode is not immediately reclaimed
2. i_nlink count is > 0 for a fully unlinked file, which
confuses wrapfs

On last unlink, the MDT sends back attributes.  Use the
nlink count from these to update the client inode.

Remove null check inherited from ll_get_child_fid, because
the inode should never be null on an unlink.

Re-enabled test 76, which passes with this patch.
Removed slab allocator tuning from test_76, because slab is
no longer the default Linux allocator.

So you are suggesting a situation where there is dirty data waiting to be written out and it's lost because of an unlink?  Dirty data that would normally be readable to another client because it's still got the file open?

That's a real race condition, but it exists already, I believe.  Currently, if we do a getattr, we find the value of nlink is zero - So if someone with this file open does a stat then closes it, this destruction would occur.

Imagine a library that writes data then does a stat after the write.  That's not that rare, and once that library closed the file, it would lose its data.  That's not affected one way or another by this mod.

Perhaps I'm misunderstanding how we stay safe from this today.  (Also, if there are dirty pages, there is an LDLM write lock.  I think that probably doesn't affect the race you're mentioning, though.)
