LU-2028 lnet: Potential data corruption in 'o2iblnd'

Potential data corruption in 'o2iblnd' (the IB LND driver)
when using pre-mapped DMA buffers

Implemented 'kiblnd_dma_sync_single_for_cpu/device' wrappers
depending on whether or not 'HAVE_OFED_IB_DMA_MAP' is defined.
Added the call 'kiblnd_dma_sync_single_for_cpu' in 'kiblnd_unmap_tx'.
This is the most appropriate place to make the call, as at the time of
this call, the device has completed accessing this DMA area, and
before it will access it again to stream the next message.
Added the call 'kiblnd_dma_sync_single_for_device' in
'kiblnd_post_tx_locked', just before calling the IB API function
'ib_post_send'. This ensures that ownership of the DMA-mapped
memory region is handed over to the IB device, at the latest possible
moment.
Similarly, added dma_sync_single_for_[cpu|device] for 'rx_msg',
to ensure that transfer of ownership of that DMA-mapped memory
is transferred at the right time between the CPU and device.
