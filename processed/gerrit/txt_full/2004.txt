This patch defines two new macros for printing object id in the same
format as DFID. The typical usage is "blah "DOID" blah", POID(ost_id).

Patch Set 1: I would prefer that you didn't submit this

(7 inline comments)

I'd be really happy to see this land, since I've had a very similar patch that I was working on.  Having a struct ost_id instead of the separate id/gr fields makes some of the code cleaner, and this is a good start toward fixing it.

There is also similar code that uses "POSTID" which could be replaced with DOID/POID, but a few places don't have a struct ost_id to work with so can be left until they can be cleaned up in the FID-on-OST world.

For a follow on patch I'd like to remove the places that use the compat macros like lsm_object_id, lsm_object_seq, lmm_object_id, lmm_object_seq, o_id, o_seq, etc. so that they can just use ost_id and/or FIDs internally.  In many cases, this will replace separate assignments with struct copying.

With Di's patch to use FID-on-OST, it may even make sense to change lustre_stripe_md to use a struct lu_fid internally instead of ost_id, to avoid the need to convert the FID repeatedly.  I'm still not sure that we have the FID<->ost_id conversion correct throughout the code, since we are still using seq == 0 everywhere, but this will probably fall out as soon as Di's patches are landed.

Needs to be rebased on master anyway, so including minor fixes...
