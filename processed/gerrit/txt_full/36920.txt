--- Commit Message ---
LU-12303 lnet: recover health at same rate as dec

When there is a failure to send to or over an interface the
interface's health value is decremented by lnet_health_sensitivity.
Originally, when it recovers the health value goes up by 1.
This patch makes incrementing and decrementing health value
symmetrical, by lnet_health_sensitivity.

In this way if a site wants to stop using and start using an interface
quickly, then it can set lnet_health_sensitivity to a large value
compared to the health value maximum, which is hard coded to 1000.
For example setting lnet_health_sensitivity to 500 will reduce the
health value of an interface to 0 after two failed sends and
reset it to maximum health value after two successful sends.

Test-parameters: trivial

--- Message (Andreas Dilger, 2019-12-04 03:30:24.000000000) ---
Patch Set 1:

If the health decrements at the same rate it increments, then it isn't clear why we would have health sensitivity at all?

My concern here would be that if decrement and increment are the same, then a flakey network dropping 50% of packets will be "healthy" 50% of the time.  I can imagine that we don't want the interface to take hours to recover (which is the case today), but it should definitely not be symmetrical IMHO.

As a starting point, maybe increment at 1/4 or 1/8 of the decrement (min 1), so that if an interface is losing more than 25% of the packets it will quickly be stuck at 0 until it improves.  Alternately, increment at a slower rate than every successful packet (e.g. every second there is traffic without no packet loss increment the health).
