LU-10968 hsm: create external HSM queue interface

Add more, consistent, information to messages
including HSM action.

Create an interface for an external HSM queue
Create a demonstration external HSM coordinator

with Lustre

by specifying COORDINATOR=external on the command line

You must run the external coordinator on the MDS

Handles all current HSM activities.  No changes to

Hi Ben, this looks like impressive work! Could you walk us through it a little?

Do you think you could piece up some sort of documentation for the userspace interface (though a reference implementation is already a very good first step).

I have two "high-level" questions/remarks:

- Why do you still use a separate thread to manage requests? As I understood it, the idea of having an external coordinator meant the kernel was left with very little to do (basically, relay RESTORE requests and take the layout lock appropriately). I understand there still needs to be some code left in the mdt layer, but why a separate thread? Why not relay requests (and take locks) immediately as they arrive on the MDS (ie. in the RPC handler)? It seems wasteful to me to leave a scheduling thread when scheduling is not needed anymore.
- Netlink is not a reliable mechanism when used to send messages from the kernel to userspace (ie. messages can be dropped). IIRC, userspace has to call setsockopt() to activate some feature so the kernel side can be made aware that contention is happening and messages are being dropped. Since I cannot find any call to setsockopt() I am a bit worried that this problem is not currently handled by your implementation.

I have not yet read all the code, so I apologize in advance if the answers are obvious.
