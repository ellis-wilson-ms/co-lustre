--- Commit Message ---
LU-482 replay-dual test_0a failed

Running LVM on top of VM hypervisor has the write caching and write
reordering problem for kernels prior to 2.6.33.
This might corrupt the journal or fs metadata and lead to a mount failure
in first replay-dual test being run (typically test_0a).

Adding 10 seconds delay after the mount should be enough for the
changes to be flushed to disk.

--- Message (Andreas Dilger, 2011-12-01 05:16:53.000000000) ---
Patch Set 3:

I object to this workaround being landed.  If the problem is with a specific kernel version (< 2.6.33) and a specific test environment (LVM and/or inside a VM) the delay should be conditional on those factors.

Otherwise we may be hiding all kind of other problems, either now or in the future, because we are testing in an environment that does not necessarily reflect normal usage (i.e. waiting 10s after mount before doing any operations).

In the past we had race conditions caused by clients creating MDS inodes with stripes on OSTs that they hadn't connected to yet, and they would get EIO when trying to write to the file.  I don't want to hide that kind of bug with such a sleep.

The other benefit of making the sleep conditional on the kernel version is that it will turn itself off once we update to newer kernels on the server.  We can't depend upon anyone to remember that this workaround is there and take it out at some point in the future, so at worst it will become a no-op check for some time before someone notices it and removes it.  The conditional check will make an "active comment" for the reader.

Please submit a follow-up patch which makes this timeout conditional.  It should also reference LU-482 as to why the timeout is there.
