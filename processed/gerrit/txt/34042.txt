--- Commit Message ---
LU-11526 rpc: support maximum 64MB I/O RPC

On newer systems, some block drivers allow max_hw_sector_kb to
be up to 65536KB (64MB) to the underlying storage. To maximize
driver efficiency, Lustre should also have bump up maximum I/O
RPC size to 64MB.
Clamp max_read_ahead_whold_mb not to exceed
max_read_ahead_per_file_mb

Except max_cached_mb must be larger than max_dirty_mb.  If we cannot
accumulate enough dirty pages on the client to make a full RPC, then
having a larger RPC size doesn't help.  For widely-striped files, we
may want to tell the OSC layer to use a smaller RPC size, or have it
automatically send smaller (but well-formed) 2^n MB RPCs when there
are a lot of dirty pages, but not enough data for a full-sized RPC?

In essence the bigalloc chunk size is akin to the ZFS blocksize - this is the minimum allocation unit of the filesystem. In that regard, it makes sense to try and submit IO at least that size, though bigalloc is actually better than large blocks because the read-modify-write unit remains at the blocksize (4KB), so sub-chunk writes do not incur a large penalty in performance. 

See also comments on bigalloc in LU-12103.

I agree with what you write, but if there are issues with the larger RPC size linked with bigalloc usage, that should be seen when people start testing bigalloc (i.e. before it is deployed at a customer site).  As yet, I doubt that bigalloc is even functional with osd-ldiskfs, so I don't think there is a pressing concern.

Also, I don't think that bigalloc chunk size will be deployed at 32MB or 64MB to start, but rather in the range of 128KB or 256KB (i.e. 32x or 64x current blocksize) so that it improves allocation performance but doesn't waste too much space for fractional blocks.  That means it will not affect the RPC size until bigalloc chunk size over 4MB, which I think is a long ways off.

If issues arise with too-large RPC sizes, it is always possible to de-tune the RPC size after the fact, the chunk size just sets the minimum default RPC size, but having RPCs smaller than the chunk size will not crater performance like it does with sub-block ZFS writes.
