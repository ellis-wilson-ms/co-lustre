--- Commit Message ---
LU-9771 ldlm: fix the issues introduced by try bits

ldlm_process_inode_bits_lock() should take the try bits in the
waiting list into consideration. If there is a lock in the waiting
list having try_bits conflicted with the requested lock, the try_bits
of that lock should be cleared.

Other improvements in this patch are to grant try_bits in atomic, and
mdt_object_lock_try() should never return negative for try lock only,
and code cleanups.

Patch Set 3: Code-Review-1

We discussed this one with Mike and Jinshan and this is what I think needs to be done (Mike volunteered to update the patch and create the follow one patches to address ordering/bundling concerns if any):

Obviously the bug of not looking for bits in the waiting list needs to be fixed.

The "all or nothing" behavior for bits might be added if there is a need (not yet clear) but it certainly should not be the default because that's not how that feature was originally envisioned.
That said, all users are to be properly audited to ensure there are no undesired dependencies. All users should handle all such dependencies themselves.

I particular the dom+layout lock ordered dependence needs to be checked even though Mike thinks the code would be correct in the face of dom bit without layout bit due to clio mechanics. Nevertheless once we have bundling of file data into ldlm enqueue reply we will need to have layout lock required on the server so might as well prepare for this now.

the layout+lookup example from the patch comments seems to be not super clear. If we give lookup to the lock just to ensure it's more easily cancelled (with explained drawbacks) - that's going to break with lock conversion code anyway. Additionally it sound like quite a kludge that was working around our LRU deficiencies that were mostly fixed since then (or so I think).
