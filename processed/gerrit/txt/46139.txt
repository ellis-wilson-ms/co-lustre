(9 comments)

> OK!  I wasn't sure if we shouldn't run this in the various other
 > configs, eg, ZFS?

True, but then you could add a second line for that:

 Test-Parameters: testlist=sanity fstype=zfs

and still avoid 80-90h of testing for the patch, and associated test failures.

On the one hand, you are totally right that the ZFS testing would be important (I'd totally overlooked this), since fallocate space reservation doesn't work on ZFS (https://review.whamcloud.com/36506 is not landed yet).  On the other hand the test would be a no-op because check_set_fallocate_or_skip() will skip ZFS backends...

I'd be OK with the test, despite some possible improvements, but it looks like it has failed the Janitor testing (maybe kbytesfree-kbytesavail > 50M there?), so it will need to be refreshed in any case.

There are also a bunch of other tests that are writing large files with "dd" that could be similarly optimized in later patches.  Maybe adding a test-framework.sh::create_large_file() helper function in this patch would be in order that could be used elsewhere (it should fall back to "dd" if fallocate is not available).  The oos.sh and oos2.sh could always be run with fallocate, as they are currently often skipped because the OSTs are too large.

I see sanity-hsm.sh has "create_file()" that should be changed to use create_large_file(), and an unused delete_large_files() helper that looks a bit dangerous (it should be changed to take an explicit pathname) and could be registered via stack_trap to do cleanup for subtests that use create_large_file().
