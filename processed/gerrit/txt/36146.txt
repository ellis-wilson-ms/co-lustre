LU-12275 sec: deal with encrypted object size

Problem with size of encrypted file comes from the fact that
an encrypted page will always contain PAGE_SIZE bytes of data,
even if clear text page is only a few bytes. And server infers
object size from content of encrypted page.

The way to address this is the following. Upon writing, when the
client encrypts the page representing the end of the file, it puts
into o_size info of the request's body, the size of the clear text
version of the file. On server side, this information is used to
adjust isize of the object, but still storing the complete pages
on disk.

Well, consider that the size is a single value, while the writes themselves cover blocks and are racing.

If we assume a file that is 1K in size, then write 1 is a single 1K write at offset 1K, and write 2 is a single 1K write at offset 2K, then there is no conflict between writes 1 and 2.  But of course, the size is 3K at the end, so it is important that at the end, we have the size from write 2.  (Or, if we put write 1 at offset 2K and write 2 at offset 1K, it would be important that at the end we have the size from write 1.)

If the writes overlap and really are racing, then it is a legitimate race condition.

So if the writes overlap, say both at offset 1K, then which data is kept is a legitimate race.

Consider a case like this:
Both writes at offset 1K.

Write 1 is 1K in size, and is just "AAA[...]".  Write 2 is 2K in size, and is just "BBB[...]".  The resulting file size is of course 3K, but what is in the file from offset 1K to 2K?  The only guarantee is that it is either *ALL* from write 1 or all from write 2 (AAA, BBB).

This is assuming the two writes are truly occurring at the same time, ie, the syscalls overlap.

Patch Set 3:

> So what if we sent over the actual *size*...? And then we would
 > update this diff using the same rule as the size (ie, we only
 > update enc_diff when the file size is changed, meaning when the
 > file size *grows* [on writes, truncate of course has its own
 > separate handling]). If page size has to be the same on client &
 > server, we could send *just* the size, otherwise, we could send
 > the size & the diff, but only write the diff when the size would

In essence we *already* send over the (potentially) new size with every write in the niobuf offset+length, which is what osd_write_commit() uses to determine the new file size.  I think that "all" we need to do is adjust this by sizediff as part of the BRW RPC, and store the new size directly on the inode.

We already have proper exclusion of i_size updates on the OST (via inode->i_lock spinlock), and luckily this is *not* a hot path because only one (or a few) threads are writing at end-of-file, and the majority of other threads are writing in the middle of the file.

The one caveat is that we would now be storing some data beyond i_size, but even for ZFS I think that would be OK, since the OST recordsize is always going to be at least a multiple of 4KB, and the "size" of a ZFS object (as opposed to the blocks) is a "higher user-level concept" that it defers management of to Lustre (or ZPL).

For 64KB PAGE_SIZE clients and 4KB block OSTs, together with my proposal that this be handled as 16 separate 4KB encryption chunks, we could still safely assume that any data past the end of the 4KB block overlapping i_size is zero and not send it to the OSTs.  That means we only ever need to write the full 4KB block at i_size to disk plus the actual file size, which is exactly what ext4/ldiskfs already does.

"Anyway, I think I really need to write tests for the encryption/decryption feature. It will be easier for development and testing of alternatives to this first implementation. We're discussing a few different races/conflicts.  The simplest one, which isn't even really a race, is just where the first write is "deeper" in the file than a later one.  So write 1K at offset 5K, then write 1K at offset 3K from another client (this is a harder test than doing so from the same client).  You can simply wait for the first write syscall to return - No need for anything else.  Obviously file size should not change after the first write.

Really, as long as there is correct locking around file size changes, this is the only case that needs to be handled.

Basically, size can change in three ways:
It can go *down* from a truncate/setattr
It can go *up* from a truncate/setattr
It can go *up* when a write ends beyond current size

So truncate is special in that it can change size arbitrarily, but writes can only ever increase size.  So we only change size on writes when a write is beyond current size.

My concern about changing i_size on the object is for subsequent reads. Let’s take a simple example by considering a clear text page that has only 32 valid bytes at the beginning (the rest of it being zeros). We have to encrypt the whole page (encryptionchunk size is PAGE_SIZE), which produces an encrypted page full of PAGE_SIZE data. Later on, in order to recover clear text data, we have to read the whole encrypted page (PAGE_SIZE bytes) and apply the decryption algorithm. This will produce PAGE_SIZE bytes of decrypted data: the first 32 valid bytes and then zeros. If you only read and decrypt the first 32 bytes from the encrypted page, the decryption will not produce the correct result (decryption chunk size is PAGE_SIZE).
So, back to adapting i_size on the object, upon later read it will only return i_size bytes to the upper layers, forgetting the end of the last page. This will result in incorrect decryption by the client. We could imagine a mechanism where the OSD layer always returns data as a multiple of client PAGE_SIZE (implying the server knows this info), but only for objects that belong to encrypted files, which means setting a special flag on the object when created (not sure how that’s possible). And as in current implementation, i_size would be used to adapt lvb_size returned to the client, letting the client know how many bytes are actual data.

Regarding the current patch, please be aware that size diff is only sent by the client when the last page in the object is not full. So we cannot simply end up storing the wrong size diff on the object in case of consecutive writes at offsets in "decreasing orders".
