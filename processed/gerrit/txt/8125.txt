--- Commit Message ---
LU-3527 nodemap: idmap management functions

This adds support for uid/gid mapping management to the nodemap
module. This mapping is added as four rbtrees in the nodemap
structure for forward and reverse mapping of UIDs and GIDs.
Postorder traversal for rbtrees have been added for cleanup
upon nodemap removal.

lctl commands have been added for adding, removing, and testing
maps.

Unit tests have been added to test the maps in all the relevant
modes (allow root access, trust client ids).

Patch Set 2: (3 inline comments)

Also not a complete review. Josh can you rebase this series? I can't cherry pick #2 onto #1 cleanly. Here are some general comments for the 3 patches in gerrit and those on their way.

Don't repeat yourself. Rather than putting nodemap name policy and cfs_hash_lookup() calls everywhere, please define a single nodemap_lookup() function that has this policy and use that everywhere.

Avoid C++ keywords as variable names (new and this). Note I don't expect any of this to be compiled as C++. It's just *really* distracting if you're at all used to C++.

Every char * parameter should be const char * unless it really can't be. It it really can't be then it should be followed by a size_t size parameter.

Generally use const as a hint to the reader that these are the input parameters and those are the output parameters.

Use bool variables and return types as much as possible. If it returns bool then there is no error to check and handle which is awesome.

Make every function static if possible.

Keep most of the structure definitions in nodemap_internal.h.

Object constructors should call INIT_LIST_HEAD() on all list members. 

Everything of the form OBD_ALLOC(idmap, sizeof(struct idmap_node)) should be OBD_ALLOC_PTR(idmap) instead.

Whenever you call OBD_ALLOC*() check for errors on the very next line.

In general prefer sizeof(*ptr) to sizeof(struct something).

Don't put parens around tests where they're not needed: if ((foo == 1) && (bar == 2)) should be if (foo == 1 && bar == 2). Using the extra parens suggests that suppressing the "suggest extra parens around assignment" compiler warning. Which you're not and we don't like assignment in conditions anyway.

Also &(ptr->field) is two parens too many.

while (p) should be while (p != NULL).

if (rc) should be if (rc < 0) usually.


When we do

 rc = something();
 if (rc < 0) {
         /* ... */
 }

There is no space between the function call and the if stateement.

99% or error messages should have the form

 "cannot SOMETHING: rc = %d"

Not "Cannot ...", or "Could not ...", or "SOMETHING failed ...".

Patch Set 4:

(13 comments)

In practice, we've seen a couple hundred idmaps total. The use case that we've seen is mostly smaller, private clusters, data collection instruments or VMs owned by labs on campus that want access to shared space, but administer their own machines. There are a couple cases of using it across the WAN, and its useful, just not as common. Either way, it is a relatively small number of individual users we have to map.

A rough estimate is 2000 idmaps per 1 MB. That would be the upper bound of what I could see IU's mapped file system ever using. That said, once persistent mappings are kept in index files, I could see keeping an upper bound on the number of mappings that are kept in memory and aging them out and fetching them from the indexed file on a per need basis.

For the ioctls, if the maps are persistent in index files, only changes (adds, deletes, etc) would get ioctls. The use case we get currently is mapping a client, and then over the next few days, we get a modest number of idmap requests (1-10). I am okay with adding functionality that would push an array of idmaps in a single ioctl, but at this point, it wouldn't get used.
