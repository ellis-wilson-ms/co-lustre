LU-17173 utils: cleanup lfs flushctx

When lfs flushctx is called without mount points, build the list of
all mounts first, and then call the ioctl to flush associated
contexts. Otherwise fetching the mount points unfortunately refreshes
the contexts being flushed, because the mount points are being
accessed.

The problem with non-regression tests is sanityusr = $USER0 is defined with its HOME as /mnt/lustre on the Maloo test clusters. So any command run via 'su - $USER0' or 'sudo -i -u $USER0' automatically triggers an access to Lustre.

This is bad, especially because it interferes when we try to flush Lustre gss contexts: the gss contexts end up being refreshed instead of being destroyed. And most of the time, when the user is not authenticated yet, it gives errors such as:

   Running as sanityusr: keyctl show @s
   su: warning: cannot change directory to /mnt/lustre: Operation not permitted

So we might want to change the HOME defined for sanityusr (and sanityusr1), to be outside of Lustre.

Unfortunately it does not work when we use 'runas lfs flushctx'. The call to execvp done in runas 'inherits' (i.e. does not reset) some bits from the user credentials. In particular, and up to Linux 5.2, the user_struct struct contains a uid_keyring field, and when Lustre retrieves the keyring from kernel space, we end up using that of the user that called runas (usually root in our tests).
On the contrary, using 'su -' or 'sudo -i' (or ssh or possibly other methods) makes a proper, clean user environment.

> On the contrary, using 'su -' or 'sudo -i' (or ssh or possibly other methods) makes a proper, clean user environment.

Not that surprising as keyring was design to be inherited by sub processes. 
su - and sudo both likely reset that relying on the pam_keyinit (distro dependent, but very likely). But I thought that pam_keyinit was only resetting the session keyring.

'keyctl session' is supposed to be able to create named session keyring and attach to existing ones, which would be very useful for the testing, but I'm not able to make it work the way the man page describes it. I did not look at the underlying kernel function to try to understand though.

That lets the option to write the tests in a different way, where we stop switching between root and user context. All actions related to the same keyring should be done under the same runas.
