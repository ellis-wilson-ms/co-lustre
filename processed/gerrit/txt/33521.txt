--- Message (Andreas Dilger, 2018-12-04 04:18:35.000000000) ---
Patch Set 4:

> It makes me a bit sad, in context, that we expect (both Cray and
 > DDN, I think) to have the page cache disabled most of the time on
 > future hardware.  This is a nice improvement for page cache usage,
 > and if I thought it would get used, I'd stand it up on real system
 > and benchmark it to see how much it helps.

I think that this is quite a reasonable tradeoff, namely that RAM is used for small reads/files and repeated reads, and bypassed for the common case of very large IOs where the hardware can already run fast without the cache, and inserting/removing GB/s of single-use data from the cache is pure overhead.

That said, there still needs to be some integration between "cacheless reads" and "lfs ladvise willread", and some metric for repeated reads so that this still populates the cache on the OSS, and doesn't just read the data from disk and promptly discard it.  However, this patch is not the place for that.  Rather patch https://review.whamcloud.com/32875 "osd: capability to enable disable cache on fly" is where that is needed.
