--- Commit Message ---
LU-11104 mdt: rename may cause deadlock

In rename locking, there are two situations we need to lock target
parent before source parent:
1. source parent is subdir of target parent.
2. source and target parents are both stripes of the same directory,
   and stripe index of source parent is after that of target parent.

But the check for the second situation is missing, which may cause
deadlock if another thread is taking stripe locks of their parent.

Cleanup mdd_is_subdir().

Add sanityn.sh test_81b.

--- Message (Andreas Dilger, 2018-06-13 06:01:03.000000000) ---
Patch Set 1:

Up until Lustre 1.8 we didn't have this complex parent/child locking order, which is often difficult to get right.  Instead, we always locked (IGIF) FIDs in numerical order.  In some rare cases it meant that we'd need to get the parent lock, do the child name->FID lookup and find that the parent FID < child FID, then either do a trylock on the larger child FID, or drop the parent lock and lock in child FID, parent FID order.  In some cases, the child FID is known in advance and we can lock both in order.

I often wonder whether it would make sense to go back to that design, whenever I see issues with the parent/child locking order...

Now we have too many locks, e.g., master object lock, stripe lock, remote lookup lock, and child lookup needs to be done under parent lock, so logically there are lock orders. This deadlock is introduced with striped directory, and since deadlock is not hard to debug, even there are other corner cases, I think it makes sense to order lock by logic than FID.
