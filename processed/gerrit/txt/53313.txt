LU-1158 general: support nanosecond timestamps

Change most s64 timestamps in memory from epoch seconds to
epoch nanoseconds. Change the variable names from
xxx_(a/m/c)time to xxx_(a/m/c)time_ns to indicate this change.

struct ost_lvb, mdt_body and obdo are special. They support
nanosecond timestamps by adding additional xxx_a/m/ctime_nsec
fields. These time_nsec fields are only the nanosecond part of
timestamps. Original xxx_a/m/ctime is kept as the epoch seconds.
They are treated differently because they don't have regular
pack/unpack functions, or sometime they are dereferenced from
message body directly without packing/unpacking.

Also rename obdo.o_ioepoch to o_padding_3 because this field was
unused since commit v2_7_50_0-38-gd5d5b349f23e and was never used
in production.

To cooperate with client/server without this change,
OBD_CONNECT_NANOSEC_TIME connection flag is enabled.
Only if both client and server support the flag, the connection
has this flag.

When a time_ns field is packed to connection without
OBD_CONNECT_NANOSEC_TIME flag, time_ns should be converted
from nanosecond to second so that peer client/server can
recieve epoch seconds as before.

Opposite coverting happens during unpacking timestamps from
connection without OBD_CONNECT_NANOSEC_TIME flag.

Now the a/m/c timestamps of lustre file/dir should have
non-zero nsec fields.

Change sanity.sh/test_39 series test cases to check nsec
field in timestamps.

Remove utimensat_08 from pjdfstest ALWAYS_EXCEPT list for new
clients and servers.

Old client without nanosecond timestamps supporting should be
able to work with this version of server reporting only seconds.

Old server without nanosecond timestamps supporting should be
able to work with this version of client reporting only seconds.

I'd like to move this patch forward again, as time permits.

The main drawback of this version of the patch is that it is using all of the remaining reserved fields in mdt_body, when we don't really need 96 bits of timestamps for billions of years into the future.

My recollection is that this is because it is hard to detect if the existing __u64 timestamps mdt_body and ost_lvb have already been converted to nanosecond format or not?

It may be possible to handle this by checking the content of the timestamps themselves?  For example, it is impossible for mbo_btime and mbo_ctime to be set any time before 2001 (when Lustre was created), so they would always store a value >= 0x40000000 seconds (or 0x40000000 << 30 nanoseconds), and this could be used to check if the timestamps have been converted to nanoseconds yet or not.

There is a slight issue that ext4 itself didn't have btime (i_crtime) until 2007, but it existed on disk for many years before it was exposed to the client in 2020, and these will all predate any client that supports nanosecond timestamps.   In very rare cases where mbo_btime == 0, it would be possible to set mbo_btime on the client with (0x40000000 << 30) nanoseconds and not set OBD_MD_FLBTIME, to avoid it being considered valid.

Also, btime is never sent to the servers (only on getattr), so even if mbo_btime is modified it will never corrupt the actual btime field on the server.

For ost_lvb it does not have btime, but ctime should also never be *less* than the actual file creation time, so it will also have seconds > 0x40000000 (or nsec > 0x1000000000000000 if it has been converted).

This would avoid using the dedicated nseconds fields and still give us nanoseconds timestamps up to Tue May 29 19:53:03 MDT 2514 (0x3ffffffff seconds).
