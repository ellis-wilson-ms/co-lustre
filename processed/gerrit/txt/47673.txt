Patch Set 2:

I don't understand the point of this patch.  Which wake_up event might be lost?

The only place out side of this thread which can set sai_task to NULL is in ll_deauthorisze_statahead(), and it then wakes up the task.

The code that you patched is in a loop that already tests for sai_task.  If sai_task is set to NULL between the test already present and the test that you added, then a wakeup will have happened, and so the task will be TASK_RUNNING and schedule will not block.

Patch Set 2: Code-Review-1

I still don't like this fix.  Without understanding what the real problem is, we cannot be sure that the change really fixes it.

May we need a memory barrier.
i.e. change the "sai->sai_task" test in the while loops in ll_statahead_thread() to smp_load_acquire(sai->sai_task), and change the NULL assignment in ll_deauthorize_statahead() to smp_store_release(sai->sai_task, NULL).

Note that whatever is causing the problem you seem to be noticing in the main loop could cause a similar problem is the loop just before the EXIT;
