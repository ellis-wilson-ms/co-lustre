LU-793 ptlrpc: Allow clients to reconnect with RPCs in process

Since not letting clients with pending RPCs to reconnect caused a lot
of problems, this patch allows such a reconnection and would only
refuse to process requests with the same xids that are already being
processed.

--- Message (Andreas Dilger, 2012-03-10 04:33:20.000000000) ---
Patch Set 3: Looks good to me, but someone else must approve

Do you think the max xid tracking is very useful?  The client will already be using the RESENT flag, so at worst when there are resent reqs the server will need to scan the list.

Either it would need to do this anyway, Or in the rare case the client req was lost the the list would be shorter, since the client couldn't have sent too many reqs in the first place. Not having the xid tracking removes an extra atomic op for every RPC (except in the rare case of reordered reqs), so IMHO it is better to make the common case more efficient than the failure case.

--- Message (Oleg Drokin, 2012-03-13 01:32:26.000000000) ---
Patch Set 3:

The xid tracking is probably indeed not so useful, though I don't think it brings all that much overhead either.

I imagine a major case for it would be during recovery, where there is a big request stream coming in with all requests being marked as RESENT.
We cannot easily skip the replay ones because clients do restart recovery at times and that might happen with a request being stuck as well.

As such I am torn on this topic.
