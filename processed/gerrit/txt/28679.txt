--- Commit Message ---
LU-10186 osd-zfs: move LAST_ID OI mapping out of oi.xx

Originally, the OI mapping for LAST_ID was put under the
oi.xx file, it may cause some issues:

1) If we remove the oi.xx file, then before the OI scrub
   scanned the whole device, we can not know the last ID
   for related sequence. That will fail the system start
   as to the OI scrub will be stopped by force.

2) It is incompatible with ldiskfs backend, that will cause
   trouble after migrating from ldiskfs-based device.

This patch moves the LAST_ID OI mapping from oi.xx file to
/O/<seq>/LAST_ID as ldiskfs backend does.

--- Message (Fan Yong, 2017-12-06 07:43:52.000000000) ---
Patch Set 13:

> actually I miss one thing.. we can't really use a filesystem migrated from ldiskfs before OI is rebuilt?

Depends on which OI. For the OI mappings used by the mount process, such as the LAST ID, if not ready or crashed, the mount will be affected. As for other OI mappings those are for normal client visible objects, they are not so important, means the client can access related objects even though related OI mappings have not been rebuilt yet. Because usually, there will be lookup FID by name before locating the object. But if the client triggers pure FID based operation before related OI mapping rebuilt, such as "fid2path", then such operation will be blocked until related IO mapping rebuilt. The same behavior as ldiskfs based case.

It is not the case that we have to mount the target right away just after restored or migrated from ldiskfs-based backup, instead, what we want to support is that we will allow the client to access the system during the OI rebuilding, in spite of when the mount is, and in spite of whether it is restored or just routine OI consistency scan.
