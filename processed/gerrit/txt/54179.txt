LU-17379 lnet: drop discarded msg if pending

LNET peer puts messages into pending queue while
it is discovering. That pins ptlrpc request as well
until peer is discovered. Thus a connect RPC to not
alive peer is stuck until peer discovery timed out
despite own RPC timeout. Moreover that means no connect
attempt to other peers are made for that time.

E.g. mount command with fake mgsnids after 30s:
nids_stats:
   "192.168.0.112@tcp": { connects: 1, ... sec_ago: 31 }
   "192.168.0.113@tcp": { connects: 0, ... sec_ago: never }
   "192.168.56.22@tcp": { connects: 0, ... sec_ago: never }

After 30s it is still stuck with first NID and never tried
any other, despite connect RPC itself was expired after 10s
by ptlrpc.

Meanwhile, during ptlrpc_unregister_reply() we discard lnet
memory descriptor for request if reply is unlinked already.
That can be used on LNET side to drop messages with discared
MD if they are in peer pending queue. Thus such messages
will finish unpinning the related request.

Patch adds such check in already existing function just to
illustrate how that works. Proper solution is to be discussed

Results with patch:
nids_stats:
   "192.168.0.112@tcp": { connects: 2, ... sec_ago: 11 }
   "192.168.0.113@tcp": { connects: 1, ... sec_ago: 21 }
   "192.168.56.22@tcp": { connects: 1, ... sec_ago: 11 }

After the same 30s we had 4 connect attempts with all
failover NIDs tried
