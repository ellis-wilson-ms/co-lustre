--- Commit Message ---
LU-4664 clio: consume grant for sync write

Sync write should consume grant if available, otherwise,
it could fail for ENOSPC when there is still available
space reserved by grant.

Additionally, direct i/o writes are sync, and they receive
more grant from the server without consuming grant.  The
result is that direct i/o writes accumulate grant on the
client without ever using it.

Fixing grants for sync writes fixes this as well.

This problem has been discussed several times, and I still don't think this patch should be landed.

1. sync write doesn't need to consume grants at all because there is a process waiting for the writing results;
2. consuming write grants will interfere with async write. Usually direct IO is used to write huge amount of data, now it will consume all the grants on the client side, this is unfair to async write. Also, since sync write RPC can be mixed with granted and non-granted pages, this will make the situation worse;
3. I understand the intention of this patch - but I tend to think this method seems to be incorrect to solve the problem. The correct solution is to make grant shrink working properly. IMHO, It's not terribly wrong when the OST starts to deny sync write RPC if it has been really full, as long as the writer can be notified.

Niu: in order to calculate grant to write pages, we have to take the block size on OFD into consideration. When writing to a ZFS block, it has to reserve the grant of the size of (max) ZFS block, plus some tax needed for COW. For instance, if the client wants to write pages from 0 to 256 to ZFS with block size 128KB, it will reserve 128KB grant for block and 112 KB for COW overhead when the first page is being written, and then the next 31 pages(a block has 32 pages) will not consume any grant but only {cl,obd}_dirty_pages. And so on for the 33th page that will go to the next block.
