LU-4694 hsm: Cleanup codes about return value

errno migh be zero when llapi_hsm_action_get_fd() returns a failure.
ct_archive() does not handle it correctly, which might cause HSM take
the operation as successful even if it actually fails.

The patch also cleanup other codes about return value.

Hi Robert,

I understand what you mean.

Most of the time, we pass negative return values to CT_ERROR, but sometimes, we pass positive values. I know negative or positive is esssentailly the same, because of the implementation of llapi_error. However, it feels better to keep these values all negative neatly (or all positive in reverse). For example, what if we need to change the implementation of CT_ERROR or llapi_error in the future?

Thanks,
Li Xi

--- Message (Robert Read, 2014-03-21 15:54:58.000000000) ---
Patch Set 1:

In userspace errno is typically (always?) positive though. It is common (though not standard) for functions to return -1 on error, but the actual error code would be a positive value passed in errno.  I suppose we might want to define our custom error codes as negative numbers, but negating errno itself seems wrong and potentially confusing.

--- Message (Oleg Drokin, 2014-03-28 23:10:40.000000000) ---
Patch Set 1:

Biggest concern I have is that we need to closely watch return values on error not to be in the space that's valid non-error values.

So e.g. the function returning an FD cannot really return -ENODATA because that's a valid (though unlikely) file descriptor. The only indication of error possible is -1 (and errno set appropriately).
This also means a lot of current logic that checks fd to be >=0 for to check for validity is technically incorrect.

So we likely should drop all -errno from the error prints (esp. since there's abs() call inside anyway).
As for actual return values, I guess we really need to do something consistent and returning a negative error almost makes sense (but see the caveat that I started with).

--- Message (John Hammond, 2014-03-31 01:25:24.000000000) ---
Patch Set 1:

Oleg,

Huh? -ENODATA will never be a valid file descriptor. The last thing we want it to encourage is rc == -1 checking since that will be broken when an -errno is returned. Negative ints are not and will not be valid file descriptors on Linux, ever.

Returning -errno of failure is the sanest possible practice here.

With regard to io offset sizes it's different. There we should be using size_t in, and ssize_t out like the rest of the civilized world. No more int for everything.
