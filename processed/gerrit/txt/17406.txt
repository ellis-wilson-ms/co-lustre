LU-7256 tests: wait current LFSCK to exit before next test

During the sanity-lfsck tests, some test cases only check the
LFSCK status on MDT0, and go ahead if the status matches the
expected one. For DNE cases, such check maybe not enough, and
may leave un-finished LFSCK instances on other MDT(s). It may
cause the following trouble:

1) When move to next test case, the un-finished LFSCK instance
   may cause new LFSCK instance failure or other strange LFSCK
   check/repair behaviour.

2) If it is the last test case, and some MDT(s) umounted, then
   when some un-finished LFSCK instance on another online MDT
   needs to talk with some umounted MDT, then related RPC will
   trigger reconnect to the umounted MDT, and the LFSCK engine
   will hung there till such MDT mount again. That is the case
   for this ticket hit. In fact, we should allow the server to
   umount even though some LFSCK instances run on other nodes.
   The LU-6684 patch (http://review.whamcloud.com/#/c/17032/)
   will handle that more properly.

This patch is mainly for adjusting test scripts to wait all
the LFSCK instances to exit before the next test case.

There are two options to wait all the LFSCK instances to exit:

1) Check the LFSCK status via related lproc interface on each
   target (MDT/OST) one by one.

2) Export some new interface to check all the LFSCK instances'
   status via single command.

We choose the later solution. Because it is convenient for the
sys-admin. The new interface is 'lctl lfsck_query'. Its usage:

lctl lfsck_query <-M | --device MDT_device> [-h | --help]
                 [-t | --type check_type[,check_type...]]
                 [-w | --wait]

options:
-M: device to query LFSCK on.
-t: LFSCK type(s) to be queried (default is all).
-w: do not return until LFSCK not running.
-h: help message.

> I suspect that this is a common problem that will be hit by users as well, since there is no easy way to monitor for lfsck progress and status. I think this also came up during the LFSCK design review, that there should be a command like "lctl lfsck_status [--wait]" or similar that will return the current state of lfsck on all the nodes, and optionally wait for it to be completed.

It is common issue. It is NOT reasonable to require the user to check whether the LFSCK on other servers has run completed or not when he/she wants to umount current MDT. More commonly, it is normal that some server(s) crashed during the LFSCK, then if without patch, umount other MDT(s) may be affected.

I agree with you that "lctl lfsck_wait" is some helpful. But it will not fix the root issues mentioned above. Instead, your suggestion in LU-6684 is more suitable, means that we should allow the LFSCK to be stopped in spite of whether remote partners alive or not. That is why I made the patch http://review.whamcloud.com/#/c/17032/.

There are two possible way to implement the "lctl lfsck_wait"

1) If we implement that at user space, we need to know which targets (MDT/OST) joined the LFSCK. And even if the LFSCK runs on two MDTs, we still need to know whether they belong to the same LFSCK run or not. For that, we can introduce some generation for the LFSCK, but it may introduce some compatibility issues.

2) If we implement that at kernel level, then we need to keep related LFSCK data structure even if the LFSCK completed (PHASE2 done) on this server until all the LFSCK completed on all joined LFSCK servers. That means we need another PHASE3 after the PHASE2 done, the LFSCK state machine will be changed for that. That may cause some compatibility issues.

If we have to implement "lctl lfsck_wait", then I may choose the latter one. What is your suggestion? or whether it is worth to do that?
