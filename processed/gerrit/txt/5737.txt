--- Message (John Hammond, 2013-05-06 19:28:43.000000000) ---
Patch Set 2: No score

This patch makes me a little uneasy now, since from the tarfile's POV the endianness of *.lov is arbitrary.

It would be good to see that ll_setxattr() properly accepts lov_user_md in both versions, in both byte orders, on files and directories. BTW, I can see that it does not currently.

An alternative is to have getxattr always return whatever it says on disk and let user space do the correction. The "honesty" of this approach appeals to me.

Re Andreas' comment about making the endainness conversion more clear, I would like to see (and have been working on) functions that take a lum/lmm of any version and endian, and just convert it to host/disk endian. This takes the all of the LOV_MAGIC != cpu_to_le32(LOV_MAGIC) type tests out of line and seems a lot cleaner to me.

--- Message (Ned Bass, 2013-05-06 20:59:48.000000000) ---
Patch Set 2:

I really wish Lustre would have taken the approach Rob Pike advocates to deal with endianness.

http://commandcenter.blogspot.com/2012/04/byte-order-fallacy.html

That is, use a known byte order for over the wire data streams, then run *the same code* regardless of local endianness to extract data from the stream.  Yes, servers would do some unnecessary bit shifting, but we would have cleaner code and fewer bugs.

Regarding swabbing in user space.  I see the appeal, but I would object based on the lack of symmetry with the write path. The library doesn't have to know anything about endianness when it writes the data, so it shouldn't have to convert it back when it reads it.
