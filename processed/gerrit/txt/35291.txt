Patch Set 9:

I found out the root cause of the lfm_value truncation issue. It's in ll_getxattr_lov():

 static ssize_t ll_getxattr_lov(struct inode *inode, void *buf, size_t buf_size)
 {
                 //......
                 magic = ((struct lov_mds_md *)buf)->lmm_magic;
                 if (magic == LOV_MAGIC_COMP_V1 || magic == LOV_MAGIC_FOREIGN)
                         goto out_env;
 
                 ((struct lov_mds_md *)buf)->lmm_layout_gen = 0;
 out_env:
                 cl_env_put(env, &refcheck);
 
                 RETURN(rc);
                 //......
 }

Because the magic was not swabbed, if condition became false and made 16 bits zero in the buf.

Patch Set 9:

> if (magic == LOV_MAGIC_COMP_V1 || magic == LOV_MAGIC_FOREIGN)
 
I really don't like the "if it isn't this specific magic, then it must be (something else known)" kind of check. As we add more and more different layouts, the chance of getting that wrong is very high. Instead, we need to change such checks to be "if this is a specific magic that I know, do something; else if another specific magic, do something else" and skip any layout that is not understood.
