LU-13903 utils: separate out server code for lctl

The utility lctl is used by both client and server management but
the code is not cleanly separated for each case. This breaks
building lctl for the native Linux Lustre client since it only
contains client related work. Fix up lctl to be slimed down for
client only targets.

--- Message (Andreas Dilger, 2021-04-06 14:55:01.000000000) ---
Patch Set 3:

> How so? Why can't you supply tools that were built against a full
 > lustre release?

Using the "full" tools would provide even better testing, since it would ensure that the code for the upstream client is compatible with the master branch (or any other branch).  If that doesn't work then we have serious usability issues that need to be fixed.

--- Message (James Simmons, 2021-04-06 16:26:07.000000000) ---
Patch Set 3:

The rule upstream is not to provide things not used in the current kernel code. This means client only related data in the UAPI headers. Their is no way around this. Now we could force the utilities to use only the UAPI headers in the OpenSFS tree but that means the kernel UAPI will be ignored and never tested. Future distros with the native Lustre client could attempt to build against potential broken UAPI headers. Also we will have build issues with the native client since the UAPI headers in the kernel tree and OpenSFS will collide.

--- Message (James Simmons, 2021-04-06 16:27:41.000000000) ---
Patch Set 3:

Also I have thought about splitting the UAPI headers into server only headers but we still have userland tools needing server headers for client only builds.
