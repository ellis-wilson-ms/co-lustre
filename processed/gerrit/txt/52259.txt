(7 comments)

A lot of this code is mechanical/compat changes, which is fine, but it isn't clear to me whether this is just "going through the motions" to make the CLIO code "compatible" with folios, or whether this will actually allow us to *benefit" from folios?

Since the Lustre client is doing a LOT of high-bandwidth IO with (often) large read/write requests on machines with hundreds of GB or TB of RAM, we could significantly improve performance and reduce overhead by leveraging folios to manage large numbers of pages together.  If we only handle each folio as a single page, then the code will give the _appearance_ of using folios, but without any of the benefit.

I'm not saying that this patch *HAS* to implement the full usage of folios, just adding compatibility with newer kernels is enough for a +1 if it works and does not break any existing kernel (which appears to be the case here).

However, because we are moving into kernels that have folios we also shouldn't lose sight of the reason that folios were created - to reduce the overhead of managing millions of pages to drive GB/s IO speeds.

If the answer is "we can't do anything significant with folios until we don't need to support older kernels that don't have folios", then that is what it is, but let's not forget about this in the future.
