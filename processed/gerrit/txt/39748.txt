LU-13931 hsm: get copytool process name correctly

In the patch "LU-13688 hsm: handle in-tree executed copytools
correctly", it adds $HSMTOOL_KILL used to kill the copytool
process with "lt-" prefix when found it was from in-tree build
testing.

However, when compile with "--disable-shared", the copytool is
built in the static way and the in-tree copytool is running with
process name without "lt-" prefix. Thus it breaks in-tree static
build HSM related testing.

In this patch, we obtain the copytool process name via filter the
output of running copytool with "--help" option.

> The following is the results I got via libtool execute:
 
 (...)
 
 Also the output of 'libtool execute lhsmtool_posix' is "lhsmtool_posix", but the process name is still "lt-lhsmtool_posix"...

Yes, that's what I was trying to explain. The output of 'libtool execute echo lhsmtool_posix' is "lhsmtool_posix" because, unless you run the command from the lustre/utils directory, libtool does not see any file named "lhsmtool_posix".

In other words, "libtool execute" does not use PATH to resolve filepaths.

To solve this, one could simply use an absolute path instead of a relative one.

Of course, this raises other issues, such as: what if the path to lhsmtool_posix is not the same from one client to another? What if the machine running the test script uses the in-tree version of the script, but remote clients only have the installed version? etc...

Anyway, my original point is that a few years ago, we used to poke and guess the copytool's file name, and then we stopped doing it because it caused problems.
This patch reminds me of the old days, so I am a bit worried we are going backwards rather than forward.
