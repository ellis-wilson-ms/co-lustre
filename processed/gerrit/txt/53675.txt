--- Commit Message ---
LU-17276 ldlm: don't release ldlm_interval buffer

We will release the ldlm_interval buffer while insert
same extent in tree. But it may be reused late. We have
to alloc it again in such case. So just link it to interval
node and assign it back when the lock was remove from tree.

This patch is hard to understand.  As well as doing the thing that is described in the commit message (which may well be a sensible thing to do), it refactors a lot of code.  That makes it hard to follow.  Refactoring the code is probably a good idea.  Avoiding the allocation is probably a good idea.  Doing them both in the same patch is a bad idea.
Storing the spare node in struct ldlm_interval is also a bad idea.  It wastes 2 pointer which are hardly every used.  It could be easily changed to only waste one pointer - a single pointer to the allocated node - but even that isn't necessary.

If we ignore all the refactoring, I would suggest changing ldlm_resource_unlink_lock() to take a second argument - a 'struct ldlm_interval **' (or maybe a void** if that avoid extra #includes).  If this is NULL, it does exactly what it does now.  This pointer is passed to ldlm_extent_unlink_lock().  If it is not NULL, then ldlm_extent_unlink_lock doesn't call ldlm_interval_free() but instead stores the node in the location pointed to.

ldlm_lock_enqueue never tries to allocate 'node' but instead passes &node as the new arg to ldlm_resource_unlink_lock() (All other callers leave it as NULL).  This way it will have a useful value when ldlm_interval_attach() is called.

That should be all that is needed.

If you want to refactor the code in a separate patch, please do.  But keep that separate from the change in behaviour.

--- Message (Neil Brown, 2024-02-05 04:31:23.000000000) ---
Patch Set 1:

I think I've work out what this patch is doing.

The current code allocates ->l_tree_node for any extent lock when the lock is created.  This is used to link the lock into the extent tree.  If two locks with identical extents are in the tree then one of the ->l_tree_nodes is discarded and the remaining one is shared between the locks.
However is a lock is removed from the extent tree but not immediately destroyed, then it won't have a ->l_tree_node which is bad, so one has to be allocated.  This is clumsy.

With the new code we never free the ->l_tree_node until the lock itself is destroyed.  Instead when two locks in the extent tree have the same extent there is a "main" ldlm_interval which all the locks refer to with their ->l_tree_node and all the other ldlm_intervals are linked together with the ->l_free list.  Whenever a lock that shares an ldlm_interval is detached from the interval tree, the first l_free ldlm_interval is removed from the free list. and attach to the lock.

So it makes sense and is probably correct.  It would a lot to have the above description in the commit message.

However it still wastes space.  You don't need a list_head in each ldlm_interval - only a pointer to an ldlm_interval.  Maybe call it "struct ldlm_interval *l_next_free;".
The ldlm_interval that is in the interval tree points to a free ldlm_interval, and that one points to another free interval and so on.  We always insert at the top of list, and remove from the top of the list.

But it would be much much better to discard struct ldlm_interval completely and just allow locks with the same extent to each be independently in the interval tree.
https://review.whamcloud.com/c/fs/lustre-release/+/33221 does this. Maybe it is time to try to get that patch accepted again.
