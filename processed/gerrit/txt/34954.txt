--- Commit Message ---
LU-8585 utils: enable fileset use with liblustreapi HSM handling

Currently using the special .lustre/fid/X path doesn't work with
filesets but the llapi_fid_* API in liblustreapi does. Move the
HSM handling in liblustreapi to the llapi_fid_* handling. While
sites that need to use fileset can create their own copy agent
that uses the llapi_fid_* most sites are reluctant to replace
liblustreapi with their own version that can work with fileset.
Note sanity-hsm test will not work FILESET since the copy agent
for testing will not be updated to use the llapi_fid_* API.

(4 comments)

I think the concept of open-by-FID is still important to keep.  It shouldn't be replaced with fid2path+open-by-path, since that has more overhead and may be less reliable due to the pathname to a file changing after it is generated (e.g. TOCTOU).

One of the reasons why .lustre/fid (and therefore open-by-FID) is not visible in subdirectory mounts is that this allows a POSIX access check violation if a file can be opened by FID if not all of the parent directories are accessible to that user.

It makes sense for subdirectory mounts/nodemap that the ability to allow open-by-fid should be configured on the nodemap itself.  That allows sysadmins to enable/disable this functionality as needed, but outside the control of the client (which may be a user-supplied image with root access, so a mount option isn't enough).  The "user_fid2path" mount option is only to decide whether regular users can access this functionality or not, but does not affect root access on that mountpoint.
