--- Commit Message ---
LU-17711 osd-ldiskfs: do not delete dotdot during rename

Since upstream kernel commit v5.12-rc4-32-g6c0912739699
ext4_dir_entry_2 after rec_len will be wiped when deleting
the entry.

This creates a problem with rename, when we delete dotdot
first and if it's a dx dir, kernel will wipe entire dx_root
in the block after dotdot entry.

We can just update the dotdot entry in-place without deleting.

For dx dirs, ext4_update_dotdot() takes care of dotdot and
inserting dotdot is an update, use it for linear dirs also.

Rewrite ext4_update_dotdot() to get a few fixes:
*use ext4_read_dirblock to get the first block.
*do not assert on data read from disk, we check the dot and
dotdot entry and if anything looks wrong, we return -EFSCORRUPTED.
*make sure the change is journalled.
*set metadata_csum correctly for dx dirs.

Update ext4-data-in-dirent.patch, if dotdot entry has no space
for dirdata, try to expand the dotdot entry by moving the
entries behind it, or move the dx_root for dx dirs.

Add conf-sanity/154 to verify that the ".." entry was updated
properly after restore, including with an htree split directory
with dx_root entry.

Patch Set 1:

The insert of dotdot without removing first is essentially a replace
in ldiskfs.
But yes from the Maloo test results this is probably not enough.

LU-17711 actually has little to do with dir_data.
If the dir is a indexed dir, the dx_root which is the root of the htree
sits after the dot and dotdot entry, with the dotdot entry's record length
set to cover till the end of the block.
So removing the dotdot entry by calling ext4_delete_entry directly from lustre
would wipe the htree root after that, corrupting the htree. Which is true even without dirdata.

--- Message (Andreas Dilger, 2024-04-11 05:56:21.000000000) ---
Patch Set 1:

I agree with Dongyang that this is actually the "correct" fix and not just hiding the bug.  If any other entry is removed and re-added, then if it doesn't fit in the same space a different spot will be used.  It is only the "." and ".." entries at the start of the directory file that *have* to be replaced in-place (exactly because of their well-known location), and the fact that htree indexing "hacks" the ".." entry to store the index root after the directory entry.  Ext4 itself will never delete "..", and this was done in the MDT code for "correctness" but is probably also the wrong thing.

Another option would be to change osd-ldiskfs to never actually delete "..", in case some other OSD back-end filesystem has a different opinion on whether deleting ".." should be done.

 - I had missed the difference between ldiskfs_next_entry which uses the
rec_len field of the dir entry, and LDISKFS_DIR_ENTRY_LEN() (used to get
the htree) that computes the real length of the entry and which we use
to get the htree position.
And to make things even more confusing I was also thinking the dx_get_dx_info

placed after the FID at the position correctly computed from dir entry
len.

So, not replying to each points individually but yes this all makes much
more sense: we mustn't clear '..', at least not the full rec_len of '..'.

In theory we _could_ clear the actual '..' data, that is LDISKFS_DIR_ENTRY_LEN()
bytes from the start of the entry, but not the full rec_len...
(assuming entries are reliably cleared, it's wasteful to always clear
the full rec_len, because if you e.g. have entries A B C and you delete
C then B you'll clear C's memory twice)
BUT in that case what I was saying about keeping track of the dir data length
goes a full loop and makes sense: we still need to find the htree at namelen
+ fid size, so clearing '..' that way would yield an offset in the position
and we actually need to be careful that if we are to replace .. it needs to
have the same amount of dirdata, which is true for lustre (we'll add
another dir with LUFID), but not in the general case.

For other entries, we're losing the deleted entry's "actual size" but
that doesn't matter because (I _finally_ understood what this pde was):
1/ if there was another entry before in the block, we add the entry's
rec_len to the previous entry, and this includes dir data size as well,
so the previous entry correctly point to the next entry
2/ if there wasn't we're clearing the inode number and that is
properly skipped.
