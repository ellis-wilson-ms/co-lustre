Patch Set 2:

I'm still looking into this, but I'm not yet convinced this actually tests the failure fixed by https://review.whamcloud.com/#/c/fs/lustre-release/+/51559/

It looks like calling "lfs setstripe" on a file directly does not go through lod_get_default_lov_striping() and thus doesn't set the relevant memory in the MDT thread's environment (struct lod_default_striping *lds = lod_lds_buf_get(env)). I think to set the memory and trigger failure requires setting specific striping on a directory and then creating a file in that directory.

So I think to test the failure would require something like this instead: 

save_layout_restore_at_exit $MOUNT
$LFS setstripe -o 0,1 $MOUNT
touch $DIR/$tfile.{1..128}

I'm not 100% confident on this yet, so let me know if you disagree and think this will hit the failure. (I did test this several times locally without the fix in place and it never failed for me.)

This is a guess, but I think a possible sequence of events in the failing case may start with these events in test_27H:

$LFS setstripe -o 0 -o 2 $DIR/$tdir || error "setstripe failed"
touch $DIR/$tdir/$tfile
