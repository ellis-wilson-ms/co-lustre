LU-12518 llite: support page unaligned stride readahead

Currently, Lustre works well for aligned IO, but performance
is pretty bad for unaligned IO stride read, we might need
take some efforts to improve this situation.

One of the main problem with current stride read is it is
based on Page Index, so if we hit unaligned page case,
stride Read detection will not work well. To support unaligned
page stride read, we might change page index to bytes offset
thus stride read pattern detection work well and we won't hit
many small pages RPC and readahead window reset. At the same
time, we shall keep as much as performances for existed cases
and make sure there won't be obvious regressions for
aligned-stride and sequential read.

Benchmark numbers:
iozone -w -c -i 5 -t1 -j 2 -s 1G -r 43k -F /mnt/lustre/data

Patched                 Unpatched
1386630.75 kB/sec       152002.50 kB/sec

At least performance bumped up more than ~800%.

Benchmarked with IOR from ihara:
        FPP Read(MB/sec)        SSF Read(MB/sec)
Unpatched 44,636                7,731

Patched   44,318                20,745

Got 250% performances up for ior_hard_read workload.

I don't know if it can be addressed by this patch, but it would be useful if there was some "margin of error" for the unaligned readahead.

Consider the case of e.g. scanning a video stream at high speed.  The compressed frame size is not necessarily constant, so it will seek and read to an approximate offset, read data to resync with the stream (find start-of-frame marker), then seek to the next offset.

If there some acceptable margin of error (based on the readahead size, say +/- 5-10%) then the strided readahead will continue to run.  It would be desirable to use a decaying average for the stride size and interval (e.g. new_stride = (7 * old_stride + last_stride)/8 or similar).

That is what the margin of error is for.  A miss within more than +/- 5% (or similar) of the readahead window would still be considered a readahead hit (with a missing page inside the window), but a miss outside the margin would be a miss.

Since the readahead stride and window size would not be fixed values, the decaying averages would handle the differences should keep the readahead on track.  It might make sense to also track the variance between read requests (starting with a 5% value), so that there is no error margin for exact readahead, but an appropriate margin if the read requests are not consistent.

I can understand if this might be better as a separate patch, but I was thinking that the conversion to byte-level readahead values may cause accidental readahead misses for even small errors (e.g. off-by-one anywhere in the code)/
