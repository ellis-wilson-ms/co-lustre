LU-13127 ptlrpc: don't require CONFIG_CRYPTO_CRC32

Don't require CONFIG_CRYPTO_CRC32 to build if not configured,
as it may not be available for all kernels and is easily fixed.

Consolidate the early reply code in sec_plain.c to also call
lustre_msg_calc_cksum() to reduce code duplication.

--- Message (Andreas Dilger, 2020-08-09 18:46:57.000000000) ---
Patch Set 4:

> I am just wondering if there is any reason to keep on using

 > Should it be a problem to use crc32_le() in all cases?

I don't mind either way. This was just the easiest for getting the compile working again.

It might be worthwhile to run a quick test to see which one is lower overhead for the small ptlrpc_body. I suspect crc32_le() is going to be lower overhead, since it doesn't need all the setup that CryptoAPI has.

--- Message (Andreas Dilger, 2020-08-11 00:05:38.000000000) ---
Patch Set 4:

> It might be worthwhile to run a quick test to see which one is
 > lower overhead for the small ptlrpc_body. I suspect crc32_le() is
 > going to be lower overhead, since it doesn't need all the setup
 > that CryptoAPI has.

I ran a quick benchmark added into cfs_crypto_performance_test() for CRC32 using cfs_crypto_hash() and crc32_le(), and not surprisingly the crc32_le() code is faster.  What is a bit surprising is *how much* faster it is for the ptlrpc_body buffer size used for early replies:

  Lustre: Crypto hash CRC32 speed = 182 MB/s
  Lustre: crc32_le CRC32 speed = 2017 MB/s

so it seems pretty clear that we can get 10x the speedup <=> 1/10 the overhead from preferring to use crc32_le() in this code if it is available.  In theory, crc32_le() may not be available if CONFIG_CRC32 is not set, but we are no worse off in that case.
