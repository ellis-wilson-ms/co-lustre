LU-9341 llite: Do not lock the world for append

This patch does a few things to guarantee append write
atomicity from multiple processes and multiple clients
without having to instantiate and lock the entire layout.

1. Before starting an append i/o, the client requests a
lock to EOF on the first object of the file.  It finds the
first object by requesting a lock on the first byte of the
file in clio, then switching to an EOF lock request at the
LDLM layer.  This lock is held throughout the i/o.
This guarantees exclusion with appends from other clients.
Thanks to Bobijam for this idea.

Locking to EOF at the LDLM layer is necessary so we are
guaranteed we can use this lock for any i/o to this object.

2. After taking this lock, the client gets the file size
and sets the position of the append i/o and does it like
any other write.  This means it does not check size again if
the i/o is restarted or for each stripe of the i/o
separately.  This guarantees the write is all in one place
in the file.  (This does not prevent non-append writes
from writing in the middle of this space, but this is always
the case and is a legitimate race.)  Because getting the
size occurs under the "append" lock (see 1.), no other
append will check size until this write is complete.

3. To guarantee exclusion with other append writes on the
same client, which could share the LDLM lock, we convert
append writes to hold the range lock across i/o restarts.

Note that because clients without this special
"append lock" behavior attempt to lock the full layout,
including the first object, we achieve exclusion with
these clients as well.

N.B. This isn't quite done, but except for released files,
I believe it's correct.
