LU-12187 lod: implement write intent for EC files

Implement Phase 2 of FLR state management for EC
files: handle write intent to transition from RDONLY
to WRITE_PENDING state and mark parity components as
stale.

When an EC file in RDONLY state is written to, the
system now:
1. Sends a write intent RPC to the MDT
2. Transitions the file from RDONLY to WRITE_PENDING
3. Marks parity components as STALE (need resync)
4. Allows the write to proceed

Changes:
- Update lod_declare_update_rdonly() to handle EC
  files directly instead of delegating to
  lod_declare_update_plain()
  * For MD_LAYOUT_WRITE: mark all parity components
    stale and transition to WRITE_PENDING
  * For MD_LAYOUT_RESYNC: transition to SYNC_PENDING
- Update lod_declare_update_plain() to only handle
  plain (non-FLR) files
- Update lod_declare_update_write_pending() to handle
  EC file resync (transition to SYNC_PENDING)
- Update lod_declare_update_sync_pending() to handle
  concurrent writes and resync completion for EC files
  * Clear stale flag from parity components after
    userspace resync completes
- Update lod_stale_components() to mark parity stale
  in all mirrors
- Update lod_prepare_resync() to include parity
  components in resync extent checks
- Update client-side lov_io_mirror_write_intent() to
  skip mirror logic for EC files (no actual mirrors)

Tests:
- test_5a: EC file creation and initial RDONLY state
- test_5b: Write intent handling and parity staleness
- test_5c: State persistence across operations
- test_5d: Multiple writes in WRITE_PENDING state
- test_5e: Resync flow (placeholder for Phase 3)

Test-Parameters: trivial
