LU-17015 sec: fix PTLRPC_CTX_STATUS_MASK

PTLRPC_CTX_STATUS_MASK should not include PTLRPC_CTX_NEW_BIT, which is
a bit index and not a value. Also, according to code in
sptlrpc_req_refresh_ctx():
	if (unlikely(test_bit(PTLRPC_CTX_NEW_BIT, &ctx->cc_flags))) {
	   if (ctx->cc_ops->refresh)
	      ctx->cc_ops->refresh(ctx);
	}
a context needs to be refreshed if it has the PTLRPC_CTX_NEW_BIT bit.
So the function to check if context is refreshed, cli_ctx_is_refreshed
should not return true if the PTLRPC_CTX_NEW_BIT bit is set.

In the end, do not replace PTLRPC_CTX_NEW_BIT with anything else in
PTLRPC_CTX_STATUS_MASK. Having PTLRPC_CTX_NEW_BIT was a no-op (bitwise
OR with 0), but this was working as expected. Just cleanup the code to
avoid headaches.

That's probably fine, but this state machine is hard to audit. 
I would not say everything is working fine :) so it could also be you found a corner case which is not addressed properly. 

Another way to fix it is to re-add the proper PTLRPC_CTX_NEW flag, and update cli_ctx_is_refreshed() to check for the status not being NEW. That way, the STATUS_MASK is really covering all the different states a context could have.

--- Message (Sebastien Buisson, 2023-10-11 15:34:10.000000000) ---

PTLRPC_CTX_STATUS_MASK is only used by cli_ctx_status():

   cli_ctx_status[1023]           return (ctx->cc_flags & PTLRPC_CTX_STATUS_MASK);

And cli_ctx_status() is only called by:

   cli_ctx_is_ready[1029]         return (cli_ctx_status(ctx) == PTLRPC_CTX_UPTODATE);
   cli_ctx_is_refreshed[1035]     return (cli_ctx_status(ctx) != 0);

So I believe PTLRPC_CTX_STATUS_MASK is really made to filter out PTLRPC_CTX_NEW.
