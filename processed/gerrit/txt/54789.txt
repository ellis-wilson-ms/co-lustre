--- Message (Andreas Dilger, 2024-04-15 08:59:56.000000000) ---
Patch Set 1: Code-Review-1

(5 comments)

Could you please explain the reasoning behind this more.  Forcing a new SEQ on all of the MDT-OST connections is a very heavyweight operation, sometimes taking 30 minutes or more.  I think the urgency of getting the MDT precreate changes landed is behind us, so it would be better to fix this properly rather than continuing to make test script changes that are working around behavior issues in the code.

For overstriped files it seems the problem is that it not possible to start a new OST sequence in the middle of file creation, which is understandable I think.  What is the outcome if the sequence actually runs out during overstripe file creation?  Hopefully it is only that the file will create e.g. only 8 objects per OST that instead of 16 objects per OST, and not crashing or similar?  I think in _most_ cases this would be fairly rare (overstriping itself is mostly used only for benchmarking), and the default SEQ size is 32M objects, so it won't run out.

It would be best if the MDS itself checked that each OST had at least, say, 5x (to handle parallel creates) the overstripe count of OIDs left on each OST before starting to creating an overstriped file, and otherwise wrap the OST to the next SEQ (which it should do directly instead of creating/deleting junk objects).  This would actually solve the problem for *real* users of overstriped files, rather than papering over it in the test scripts.

Failing that, force_new_seq_all() should be optimized to avoid "burning through" all of the OIDs when this isn't needed.  Doing it only for OSTs that are low on OIDs, and *not* doing it for OSTs that have thousands of OIDs they could still use.

Per my other comments, I don't think this is actually the right fix for this issue, but there *are* improvements that could be done to speed up/eliminate the need force_new_seq{_all}().
