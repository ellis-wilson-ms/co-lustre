LU-6254 ldiskfs: only remove page from cache after I/O

When read cache or write through cache is not enabled for OSD,
pages might be removed from cache when preparing the I/O. This
isn't a good behavior especially when using features like
ladvise. This patch change the behavior so that pages will be
removed from cache when being released.

(12 comments)

Alex, I don't think it makes sense to drop pages from the page cache or mapping in preprw (_before_ the RPC is finished).  That can't actually reduce memory usage because there is a reference on the page already.  All that can happen is a new page may be added to the page cache for the same mapping, since this existing one was removed already.

It seems to me that once pages are read from disk they should be unlocked, so that other threads could also send them to different clients if multiple threads are reading the same pages, and the removal from cache should happen after the RPC is finished (in commitrw) so there is the maximum chance that the pages will be reused.

The -1 on this patch is not because of the move of the generic_error_remove_page() call (which I agree with), but rather because of the removal of the separate read/writethrough cache tunables, and renaming the readcache_max_filesize tunable, which may break userspace and config logs.

> no kswapd thread anymore

Assuming every page gets evicted from the pagecache (because read_cache_enable=0), not seeing kswapd run is _not_ good news.
Assuming there are as many pages to evict from the pagecache before and after this patch, and kswapd is not running, eviction is probably done in the 'direct reclaim' code path of the memory allocator. That code path is famously bad performance-wise (each thread trying to allocate memory will try to free some pages but they end up all competing for the same spinlock, which raises CPU's load).

> I think when if OSS CPU utilization is very high, kswapd thread can impact performance

I think it is the other way around, when CPU utilization is very high, kswapd is not scheduled as much as it should, which ends up triggering 'direct reclaim', which in turn raise CPU's load even more.

> I understand that disabling cache can save CPU cycles

Pages are always allocated in the pagecache, it does not matter that caching is disabled. So to me, disabling caching only gives you the worst of both worlds: high CPU utilization and... no caching.

@Alex Zhuravlev
> first of all, given the pages are removed from the pagecache immediately

Pages are always allocated with find_or_create_page(), which allocates a page and puts it in the LRU list. Calling generic_error_remove_page() invalidates the mapping of the inode so that no one uses the data but does not remove the page from the LRU list.

> there should be no need to scan them to identify unused once

So, I am no so sure about this.
Even then, from the perf profiles I have, the most costful operation is to remove a page from the LRU list. The "zone->lock" spinlock is more than highly contended.

> the patch (...) only changes the timing _slightly_

I think so too.

--------------------

@Li Xi
> 1. generic_error_remove_page() is called by osd_read_prep()
> 2. kswapd notices that, and tries to cleanup the page.

I don't think kswapd is woken up when generic_error_remove_page() runs. From what I can see, the function wakeup_kswapd() is only called from __alloc_pages_slowpath().

> However, since osd_read_prep() still needs to use it for a long time to do I/O, kswapd fails.

I don't think kswapd tries to reclaim pages which are curently being used. Otherwise, it would try to reclaim everything, and fail most of the time.

To revive the discussion on this patch, I think there is one important case that the move of generic_error_remove_page() helps in the "nocache" case.

If there are multiple clients reading the same data, then keeping the pages accessible in the mapping allows the OSS threads to re-use the same pages for multiple RPCs, while the current "drop from mapping immediately" behaviour does not allow.  With the current code, the same pages would be re-read from disk for each RPC.

In any case, if we are seeing 15-20% read improvement with this patch, regardless of whether we understand why that is, I think we should really consider landing it (after minor fixes).
