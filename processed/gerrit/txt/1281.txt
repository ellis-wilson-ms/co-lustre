LU-613 clio: Client dead-lock during binary exec

The root cause is clio takes attr lock and i_size_sem in reverse
order.

In my patch, I find out there is no deadlock issue any more in fault
and truncate path, so it holds i_size_sem in fault path to fix this
problem.

--- Message (Jinshan Xiong, 2011-09-26 19:32:51.000000000) ---
Patch Set 2:

Thanks for the comments. I didn't find it in git log.

From this statement, it seems the original intention to not grab lli_size_sem is to avoid blocking ll_nopage by grabbing a semaphore, and clio accidently violated this. Also, this commit(5654) introduced a deadlock problem where truncate path will hold lli_size_sem and then page lock, and refresh_count path will hold page lock and then lli_size_sem, I guess the following commits fixed this problem but I didn't see it from git log. I tend to think this is the "deadlock" problem in the comment.

However, from the comments in ll_get_extent_lock() at b18 branch, definitely people were worrying about the sequence of taking lock lli_size_sem and lov_stripe_lock. This is reasonable since we usually grab lli_size_sem and then lov stripe lock. The confusing part is that from commit message, it stated to avoid blocking in ll_nopage but from comment in code it stated deadlock avoidance.
