--- Commit Message ---
LU-12945 lnet: Limit use of tcp_sendpage

Upstream kernel CONFIG_DEBUG_VM added a test for PageSlab with
the reasoning:
    sendpage was not designed for processing of the Slab pages,
    in some situations it can trigger BUG_ON on receiving side.

sendpage is only used for single page sending of 'zero-copy'.
If the page is not PageSlab continue to use sendpage.

For all other cases use kernel_sendmsg

kernel-commit: a10674bf2406afc2554f9c7d31b2dc65d6a27fd9

Patch Set 4: Code-Review-1

I don't think this preserves zero-copy behaviour.

sendmsg will allocate a skbuf and copy data from the provided iov into that skbuf - so at least one copy.
sendpage allocates a skbuf header and attaches the pages to it, incrementing the refcount on the page.  The page can then be DMAed directly to the network case.  So it does not cause a memory-to-memory copy.

sendpage can only be used when you "own" the page and know that it will not be reused until it has been completely freed.
When you allocate memory with kmalloc, you don't own the whole page.  When you free the memory, slab/slub might reallocate it without first freeing the whole page.

I think that this patch can be much simpler.

Change
 if (tx->tx_msg.ksm_z;_cookies[0]) 
to
 if (tx->tx_msg.ksm_z;_cookies[0] && !PageSlab(kiov->bv_page))

However it might be worth auditing the code and checking the source of all buffers that zc is enabled for, to be sure that we really own the page even when it is not a PageSlab().
