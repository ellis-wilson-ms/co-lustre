--- Commit Message ---
LU-521 lnet: make LST support variable page size

LNet selftest can't support variable page size because it sends
number of pages over the wire.
We have to change wire format to fix this but it will have
compatibility issue, so this patch also implemented "session features"
for LST to resolve compatibility issue, only new version LST
can understand new bulk RPC format, although new LST nodes still can
communicate with old LST nodes.

The Variable page size feature can be turned on by setting
LST_FEATURES to LST_FEAT_BULK_LEN (1). This feature is off by default.

Please see the Jira ticket (LU-521) for more details.

(2 inline comments)

It is great that this is being added, but it should not be defaulting to an incompatible mode that breaks LST in environments that do not have the same Lustre version everywhere.

Instead, this should default to the old/compatible mode initially, and only if LST_VERSION=1 is set everywhere should it use the new version.  Then, add in a check for a future version of Lustre (e.g. 2.6.50 or so) that will print a #warning and change the default version to the new one:

#if LUSTRE_VERSION_CODE >= OBD_OCD_VERSION(2,6,50,0)
#warning "changing LST protocol to use bytes instead of pages, see LU-521"
#define LST_PROTO_VERSION LST_PROTO_VERSION_1
#else
/* use the old compatible version for now */
#define LST_PROTO_VERSION LST_PROTO_VERSION_0
#endif

This is dependent on whether LST itself detect that the new protocol version is in use or not, otherwise it doesn't really help.

--- Message (Isaac Huang, 2011-12-22 19:00:28.000000000) ---
Patch Set 4:

Liang, about the LST_VERSION and "session version number" design. Personally I think it's too complex.

The original design decision on LST version compatibility was that LST should not handle different versions. Instead, all LST test nodes are supposed to run a same version. If during a rolling upgrade, some nodes have older LST version, then the user should simply upgrade them all to a same LST version. This is easy because:
1. Unlike Lustre, LST is not a long running service, so nothing gets interrupted.
2. Test nodes can be upgraded by simply copying a new LST kernel module, without changing anything else.

So I'd suggest to simply increase SRPC_MSG_VERSION, then all old test nodes will get an EPROTO when the console tries to add them to the session. Of course, console should give a meaningful error message like "please upgrade LST to version X on these nodes......".

In short, it's very easy to upgrade all test nodes to a same LST version (without upgrading Lustre), it doesn't interrupt any service, and it doesn't affect Lustre in any way as LST is completely independent. So all we need to do is increase SRPC_MSG_VERSION and complain about it loudly upon old nodes.

I believed the simplicity of upgrading LST everywhere didn't justify the complexity of version compatibility in LST. Unless my assumption is wrong, I'd like to keep the version code simple - just grouch about it!

I don't think a backport is necessary.

LST is self contained. In the case you described above, a recompilation of LST should suffice:
1. cp -a 2.3/lnet/selftest/ 1.8.7/lnet/selftest/
2. rebuild lnet_selftest.ko for 1.8.7
3. copy lnet_selftest.ko to all 1.8.7 nodes.

This should be documented somewhere, maybe in lnet/selftest/README. There might be some minor problems like cfs_list_entry is probably list_entry in 1.8.7 - all  can be easily worked around with a simple wrapper header.
