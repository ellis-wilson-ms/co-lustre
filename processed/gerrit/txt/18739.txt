--- Message (Andreas Dilger, 2016-04-13 07:23:59.000000000) ---
Patch Set 2: Code-Review+1

(4 comments)

Some possible improvements to the patch.  The commit comment could be edited in Gerrit before landing, but if the patch is refreshed the error message should be improved.  

Not against this patch, but it is a bit surprising that punch can take so long?  We used to have this problem when using block-mapped files, before extent-mapped files existed, but never since then.

Have you looked into why the truncate takes so long?  Is it because the extent tree is very fragmented, because the file is huge, because the OST is overloaded, other?  Having information like this in the commit comment makes it easier in the future to understand why the change was needed.

Back in the 2.4 kernel days before extents we also had an ldiskfs patch to push unlinks to a background thread so that the client wouldn't block waiting for the unlink to finish.  The only patch I can find is in a mailing list:

http://lists.openwall.net/linux-ext4/2008/04/24/8

since the Lustre Git history does not go back so far.  I don't know if this is something that interests you or not.
