--- Message (Andreas Dilger, 2021-04-02 04:56:07.000000000) ---
Patch Set 4:

> looks like ZFS file's block number is not accurate to fit the test.

ZFS does not allocate blocks before returning from a write RPC, so there may be a significant delay between writing the file and the block count being updated if the DLM lock is not canceled on the writing client.  We currently do not consider updating the block count to be a reason to invalidate the DLM lock, unlike the file size.

The next time that the file is opened/closed should cause the LSOM blocks to be updated, assuming it is larger.

 > If 'lfs mirror split' does an additional open, data version
 > flush, stat, close, etc then will it be set correctly?

John, if the SOM state is LAZY/STALE then the next close will update it to LAZY with the correct block count if it is larger.  If SOM block count > client close block count then it will *NOT* be updated (LSOM will increase blocks/size only).

So, if it is marked LAZY/STALE then the blocks need to be reduced enough (e.g. blocks / num_stripes * (num_stripes - 1) - fudge), and that will be used for lazy stat and lipe.
