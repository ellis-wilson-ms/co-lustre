--- Commit Message ---
LU-2756 ost: buffer utilization rate of OST rqbd

This patch covered a couple of things:
- different request buffer size for different OST services
  only requests on OST_IO_PORTAL need large buffers

- add extra 46K to OST_IO_BUFSIZE (buffer size is increased to 64K)
  size of OST_IO_MAXREQSIZE is about 18K, if OST_IO_BUFSIZE is only
  a little bigger than OST_IO_MAXREQSIZE, then each rqbd can only fit
  in 2-3 request even there are 18K bytes left in this rqbd, memory
  utilization rate is low. Extra bytes can make buffer utilization
  rate increase to 72%. Also, it can avoid version compatibility
  issue in the future if we want to increase request size again.

--- Message (Andreas Dilger, 2013-03-21 22:37:26.000000000) ---
Patch Set 2:

In the past we were limited to 256 niobufs (1M / 4kB), but with the large RPC size this may be quite large. The maximim number of niobufs in a request is actually PTLRPC_MAX_BRW_PAGES, assuming that we want to allow a fully fragmented IO request to be sent as a single large RPC. 

From the hard drive point of view, this is not much better than multiple separate RPCs, so it might make sense to limit the number of niobufs in a single RPC to some reasonable number like 1024. That is equal to the maximum pages in a 4MB RPC, but avoids the huge request size needed for a fragmented 32MB RPC (upwards of 128kB for the request)..
