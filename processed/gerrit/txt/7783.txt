LU-3970 ldlm: Add parallel drop_caches proc interface

Added a proc interface at /proc/fs/lustre/ldlm/drop_caches which
clears lustre caches for all namespaces in parallel by spawning
threads to work on a queue of namespaces.  It only adds to the work
queue namespaces which are in the active list because those are the
only ones which should have any locks.  The proc interface is
write-only and is used as follows:

echo 1 > /proc/fs/lustre/ldlm/drop_caches

The algorithm used by the new drop_caches interface is as follows.
When drop_caches is written to, ldlm_drop_caches is called.  That
function calls ldlm_dc_get_workq to get a work queue of work items
containing namespaces to be cleared.  ldlm_dc_get_workq copies the
current list of active namespaces in the workq so that the
ldlm_namespace_lock can be held as briefly as possible.  Then
ldlm_drop_caches spawns a number of threads based on the number of
CPUs.  Each thread accesses the workq, so access to the workq is
synchronized with an atomic counter in ldlm_dc_get_work_item.  When a
thread gets a work item, it clears the associated namespace using
ldlm_ns_drop_cache.  Once all the work items have been processed, the
threads return.  ldlm_drop_caches then cleans up and returns. Any
errors encountered are returned by the proc interface.

In addition, the code from lprocfs_lru_size_seq_write which clears the
caches when "clear" is written to lru_size was moved into the function
ldlm_ns_drop_cache.  This was done because the same logic is needed by
the cache clearing threads.

Two tests, test_124c and test_124d, were written to test the two
different ways of clearing lustre caches.  Each test creates many
files and then each clears lustre caches.  test_124c clears caches
serially by writing "clear" to each namespace's lru_size interface.
test_124d clears caches in parallel across namespaces by writing "1"
to /proc/fs/lustre/ldlm/drop_caches.  After clearing caches, each test
then checks that the number of unused locks is zero.
