a mirror file, this patch makes client mark this mirror as invalid
and skip it.

--- Message (Patrick Farrell, 2023-06-19 15:02:31.000000000) ---
Patch Set 4: Code-Review-1

I think we should figure out how to have a test for this - the unknown component instead of error thing is a good idea but it changes mirror behavior (and not just mirror behavior, right?)

Examples of questions for a test to ask (I am not asking these questions, these are suggestions for tests): Can the client safely do IO to a file with an unknown component in a mirror?  Does the "switch to other mirror" actually work?  What happens for IO to an unknown component when there is no mirror?  Does this fail gracefully?  Can the client do ls, getstripe?  Can the client delete a file with an unknown component?  Should it be allowed to?  What does getstripe output look like?  What happens if the client tries to create a file in a directory with an unknown component set in the layout?

I ask because unknown component handling is something that will get a little bit of testing during manual interop testing for CSDC, then will never be tested again if there's no sanity testing for it.

Patch Set 4: Code-Review-1

We discussed how to add a test case for this. Adding an OBD_FAIL that will change the trusted.lov lmm_layout = fail_val, and a different OBD_FAIL that sets lmm_magic = fail_val (if we don't have any such check).  This will allow writing tests with unknown file layouts for regular files, mirrored files, etc.

One tricky part is that we want to be able to distinguish between clients that don't understand the layout and MDS, so there may need to be two (or more?) different sets of OBD_FAIL checks that run in the MDS LOD and client LOV code to get complete coverage.
