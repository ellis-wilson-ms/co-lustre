--- Commit Message ---
LU-20 osd-ldiskfs: Make readonly patches optional

This patch creates additional module called osd_ldiskfs_debug
that would call symbols from read_only patches.
The idea is you would only insert it for debugging sessions
that require the functionality and using the patched kernel.

For production use and performance runs you would then skip this
module and weak_updates magic would allow normal osd_ldiskfs
module to be inserted and run normally (same for performance runs).

The new module would be rolled into lustre-tests rpm

Test-Parameters: mdtfilesystemtype=ldiskfs ostfilesystemtype=ldiskfs

OK, now you are just going to hate me. :)

Honestly, I'm not a fan of this approach.  I think that we really need to keep most autoconf configure-time changes out of the decision making process in generating spec file details.

The only changes that can be made by autoconf to the spec files are those that fit into the realm of the "configure --enable-dist", which bypass the more extensive system tests of "configure".  This is almost certainly going beyond that rule.

Basically, you are introducing a bit of a chicken-and-egg problem.  We should be able to do a "make dist", and then the resulting tarball can be used anywhere.  But that tar ball, with this patch, will already have some decisions made in the spec file.  We can't necessarily just do a straight "rpmbuild" using the tar ball and provided spec file.  We may need to go through a whole round of "make configure" only to make the spec files, and then go through _another_ whole round of "make configure".

You're making it so we _have_ to run the full configure process twice.  We want to avoid that.
