LU-13643 ldlm: implement LDLM_CANCEL_LVB protocol

TODO: Rewrite Augment generated patch description, which is
WAY too long

This patch implements the LDLM_CANCEL_LVB protocol extension
to enable clients to send additional data (LVB - Lock Value Block)
to the server during lock cancellation. This is particularly
useful for Active Writer locks in immediate mirror scenarios
where clients need to communicate mirror state information
during lock release.

Key Features:
- New LDLM_CANCEL_LVB opcode (108) for enhanced lock cancellation
- Client-side ldlm_cli_cancel_req_lvb() function for sending
  LVB data with lock cancellation requests
- Server-side ldlm_handle_cancel_lvb() handler for processing
  LVB data from clients
- Automatic detection and use of LDLM_CANCEL_LVB for
  Active Writer locks
- Wire protocol extensions with proper versioning

Client-side Implementation:
- ldlm_cli_cancel_req_lvb(): Enhanced cancel function that
  includes LVB data in the cancellation request
- ldlm_should_use_cancel_lvb(): Logic to determine when to
  use the enhanced protocol (currently for Active Writer locks)
- ldlm_prepare_cancel_lvb_data(): Prepares mirror state data
  for transmission to server
- Integration with existing ldlm_cli_cancel_list() to
  automatically use LVB cancellation when appropriate

Server-side Implementation:
- ldlm_handle_cancel_lvb(): Processes cancellation requests
  with LVB data, extracts and logs the received information
- Integration with existing LDLM request handling infrastructure
- Proper request format validation and error handling
- Statistics tracking for the new operation

Wire Protocol:
- RQF_LDLM_CANCEL_LVB request format with DLM_REQ and DLM_LVB
- Maintains backward compatibility with existing LDLM_CANCEL
- Proper message field definitions and size handling
- Updated wiretest.c assertions for protocol validation

Use Cases:
- Active Writer locks can communicate mirror synchronization
  status during lock release
- Immediate mirror files can report completion status
- Future extensions for other lock types requiring state
  communication during cancellation

The implementation follows Lustre's existing LDLM patterns:
- Uses standard ptlrpc request/reply infrastructure
- Integrates with existing lock cancellation flows
- Maintains proper error handling and logging
- Includes comprehensive debug output for troubleshooting

Testing demonstrates successful operation:
- LDLM_CANCEL_LVB requests are properly formatted and sent
- Server correctly receives and processes LVB data
- Automatic fallback to standard LDLM_CANCEL when LVB
  is not needed
- Wire protocol validation passes all checks

This provides the foundation for enhanced client-server
communication during lock lifecycle management, enabling
more sophisticated coordination for features like immediate
mirror writes and other advanced Lustre functionality.
