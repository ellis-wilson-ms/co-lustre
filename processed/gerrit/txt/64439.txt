LU-19847 ldiskfs: dirent and fscrypt+case_insensitive

EXT4 occupies space in dirent just after the name with
a hash for simultaneous fscrypt and casefold support.

It was discussed with the EXT4 community that it is
possible to move the hash to dirdata. It could be the
second (or third, if 64-bit inode count) user of
dirdata.

At the same time, the hash placed after the file name
should also be supported.

This patch makes LDISKFS ready for such a hash present
in both variants. While it is not currently possible
to enable fscrypt + case_insensitive on LDISKFS, it is
useful to verify that LUFID works well with such code.

fscrypt + case_insensitive support with dirdata
enabled is checked in a special xfstest sent to EXT4
with the dirdata patch.

## Summary of the Fix

### The Problem

The compile error occurred because of a mismatch between macro definitions:

1. **The patch introduced**: A new configure-time check `LDISKFS_DIR_REC_LEN_WITH_DIR` that indicates `ext4_dir_rec_len()` now takes 2 arguments (directory entry + directory inode) in newer kernels (6.12+)

2. **The old code had**: A macro `LDISKFS_DIR_ENTRY_LEN_(de)` that only takes 1 argument (just the directory entry)

3. **The incorrect code tried**: To call `LDISKFS_DIR_ENTRY_LEN_((de), NULL)` with 2 arguments, causing the compile error:
   ```
   error: macro "LDISKFS_DIR_ENTRY_LEN_" passed 2 arguments, but takes just 1
   ```

### The Fix

Changed line 7502 in `lustre/osd-ldiskfs/osd_handler.c`:

**Before:**
```c
# define LDISKFS_DIR_REC_LEN_DIR(de)	LDISKFS_DIR_ENTRY_LEN_((de), NULL)
```

**After:**
```c
# define LDISKFS_DIR_REC_LEN_DIR(de)	LDISKFS_DIR_ENTRY_LEN_((de))
```

### Rationale

The fix correctly handles the three cases:

1. **When `LDISKFS_DIR_REC_LEN_WITH_DIR` is defined** (new kernels 6.12+):
   - Uses `ldiskfs_dir_entry_len((de), NULL)` - an inline function that takes 2 arguments

2. **When `LDISKFS_DIR_ENTRY_LEN_` macro exists** (older kernels):
   - Uses `LDISKFS_DIR_ENTRY_LEN_((de))` - the macro that takes only 1 argument
   - This macro internally handles calculating the entry length including dirdata

3. **Fallback case** (very old kernels):
   - Uses `ldiskfs_dir_rec_len((de)->name_len, NULL)` - calculates from name length

This maintains compatibility across all kernel versions while respecting the actual signature of each macro/function variant.
