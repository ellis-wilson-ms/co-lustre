LU-4647 nodemap: fix problem with node reclassification

nodemap_add_member can't be used to move an already hashed member
to a new nodemap, so this patch copies the needed functionality to
nm_member_reclassif_cb. This also adds a mutex lock for reclassifying
so that there is only one nodemap reclassifying at a time.
Reclassifying takes a lock on a nodemap's nm_member_hash, so a
deadlock could arise if one nodemap is trying to add a member to
another nodemap, and that second nodemap is also reclassifying and
eventually tries to add a member to the first nodemap.

--- Message (Kit Westneat, 2014-11-05 18:40:24.000000000) ---
Patch Set 1:

Thanks for the review. The situation that the lock tries to protect against is if two different nodemaps are reclassifying simultaneously. Say an export in nodemap1 needs to move to nodemap2. However, the export can't be added to nodemap2's hash while nodemap2 is reclassifying, because iterating the hash locks the hash, so nodemap1's reclassify blocks. If nodemap2 then finds a member export that needs to move to nodemap1, it will then try to add it to nodemap1's hash, block, and deadlock.

So the lock is there to try to prevent that situation, but I wasn't sure the best way to prevent it.

--- Message (Andreas Dilger, 2014-11-25 21:25:39.000000000) ---
Patch Set 1: Code-Review+1

Nathaniel, I see the reclassification lock as analogous to a cross-directory rename operation in the filesystem.  The kernel still only has a single global lock to avoid problems with parent/child renames, and nodemap reclassification is probably going to happen a LOT less than rename.

AFAIK, nodemap reclassification (i.e. moving client(s) from one nodemap to another) is the only operation that will affect more than one nodemap at a time.  All of the others (client connect, disconnect/evict, lookup) will only affect a single nodemap, so adding serialization to those high-traffic operations may impact "normal usage" performance.  The current solution only impacts "extremely rare usage" performance (serializing two concurrent reclassifications) so I think it is a reasonable fix.
