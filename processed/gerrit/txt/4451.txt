LU-2266 tests: Wait for client reconnect in recovery-small 27

In commit de311fb7 a bug was introduced where we no longer wait
for client reconnect and instead the check is such that the client
have not attempted reconnect yet. As such the test is a noop most
of the time.
Correct the waiting condition to include CFS_FAILED mask.

Additinally fixed a race where MDS might reconnect to OST faster than
when we are able to set fail_loc (which is very possible on multinode
setup, original test was designed for a single node test)
(I belive this should also work on 32bit clients, since we converted to
printing unsigned longs now)

Also fix signed-unsigned issue in proc_fail_loc

Patch Set 2: I would prefer that you didn't submit this

> Ok, so it appears that unlike in a local setup, when we failover a remote MDS, it also somehow manages to reset fail_loc (modules unloaded/reloaded?)

Per facet_failover(), for SOFT failure mode, it mainly just does unmount and mount, without unloading and reloading modules. However, it will take more than 30 seconds to reach to the mount operation.

> YuJian, what do you think would be the best way for me to retain fail_loc before mds restarts so that we actually hit that failure point?

Per OBD_FAIL_TIMEOUT(OBD_FAIL_OSC_SHUTDOWN, 10), if OBD_FAIL_OSC_SHUTDOWN was hit, it will sleep 10 seconds. So, we have to make the "lctl set_param fail_loc" operation close to or right before the mount operation which is performed in mount_facet().

Then we have to pass the fail_loc parameter along the path of facet_failover()->mount_facets()->mount_facet() and set the parameter inside the mount_facet() function right before the mount operation. Do I understand correctly?
