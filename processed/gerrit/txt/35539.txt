--- Message (Andreas Dilger, 2019-07-17 21:26:02.000000000) ---
Patch Set 1: Code-Review-1

(7 comments)

This again doesn't make sense to me.  It is removing configure checks for 4.0 and 4.2 kernels, but leaving behind checks for 2.6.32 kernels.

To my thinking, rather than focus on removing the lustre_patchless_compat.h header, which isn't particularly RHEL6 related, it makes more sense to start removing the configure checks for the oldest kernel (2.6.28) and then working forward with each patch to remove the configure checks and compat code for each newer kernel.

Once this has removed support for kernel 2.6.32, then the corresponding RHEL6 ldiskfs and kernel patch series can be removed as described in LU-12561 (separate patches, since they will be huge), and then move forward to kernel 2.6.39, and forward if we agree that kernel 3.0 + SLES11 is also too much effort to support.  We might consider removing only the server support (ldiskfs + kernel patches) and leave the SLES11 client support if it isn't a huge burden.
