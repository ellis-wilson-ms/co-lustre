--- Commit Message ---
LU-13363 lod: do object allocation in OST pool

Currently, the ltd->ltd_qos.lq_same_space boolean that decides
whether the LOD QOS allocator is active for an allocation or not
is tracked for the entire LOV. But when a pool is specified, this
judgement should be tracked on a per-pool basis.

sanity.sh test_116c is added to verify this patch.

Test-Parameters: ostcount=6 testlist=sanity env=ONLY=116c

Patch Set 1: Code-Review-1

I agree with Alex that we shouldn't be recalculating the penalty for every allocation.  That would be too much overhead for normal files.  

I think the main case we need to decide is whether the free space is imbalanced between OSTs in the pool *before* we get into the QOS code.    It doesn't really need to get into calculating the penalties, and the free space won't change if !qos->lq_dirty, so this decision shouldn't have to be re-calculated more than once per 5s or so.
