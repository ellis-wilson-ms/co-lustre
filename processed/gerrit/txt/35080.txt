LU-11673 tests: quote argument of -n and test fix

Inside the single bracket test function '[', the ‘-n’ flag
problems arise with unquoted arguments.  The -n argument
should be quoted or use double brackets for the test.

Quote the ‘-n’ argument in test-framework.sh functions.
This simple correction caused a few tests to fail.
Fix sanity test 65k to use the correct facets and check
for the mgs facet in convert_facet2label() to fix
replay-single test 58b.

(1 comment)

I’ve tracked down the change that causes multiple tests to fail. The issue is in convert_facet_2label() in the (original) code:

	local varsvc=${facet}_svc

	if [ -n ${!varsvc} ]; then
		echo ${!varsvc}
	else
		error "No lablel for $facet!"
	fi

The patch added quotes around ${!varsvc}.

sanity test 65k fails when we quote ${!varsvc} in above code. For this test, facet = ost0. So, varsvc = ost0_svc . ${!varsvc} says use the value of varsvc as a variable and get the value of that variable. In this case, ${!varsvc} = $ost0_svc which is empty. When we quote the variable ${!varsvc} in the [ -n ] test, the test is false because the variable is empty and we get the error found in the else branch. 

In the original case where we didn’t quote the –n argument this test passed because, ${!varsvc} still evaluates to empty, [ -n ${!varsvc} ] turns into [ -n ] and bash evaluates [ -n ] as always true and an empty string is returned.

This is the exact reason that I created this patch; to guard against this always true behavior. Now I need to understand what should be done in the cases where tests fail.

--- Message (James Nunez, 2019-08-02 19:44:03.000000000) ---
Patch Set 3:

Here’s the issue with each test that fails due to quoting the variable in [ -n ${!varsvc} ]:
sanity test 65k
The test is failing due to the call “wait_osc_import_state mds ost0 FULL”. wait_osc_import_state() will eventually call “convert_facet2label ost0” and returns an empty string.
Proposed Solution: increate ost index before calling wait_osc_import_state()
replay-single test 58b
The test is failing due to the call “  wait_clients_import_state ${CLIENTS:-$HOSTNAME} "mgs" FULL” at the end of the test. wait_clients_import_state eventually calls “convert_facet2label mgs”. When we have a separate MGS, “convert_facet2label mgs” returns MGS as expected. What the MGS and MDS is combined, “convert_facet2label mgs” returns an empty string.
Proposed Solution: Add a check in convert_facet2label(): if facet = mgs and combined_mgs_mds, then change facet = mds1
