LU-18606 ldlm: Fix atomic to refcount port for NRS/TBF

Port from atomic_t to refcount_t was showing a refcount_t
error/warn under dmesg. This was seen under two instances.

First, under nrs_tbf_res_get() when a policy tries to
find a client or add new client. Under this if the client
with matching key is alreay present it continues to use
the existing client and discards the newly created one.
In this case during discard, it decrements the refcount
which is not required.

This patch removes the refcount_dec() call when client
is already present and add a refcount_inc() call when
new client is added

Second, CFS_HASH_NO_ITEMREF flag is used to indicate that
the hash table should not automatically manage reference.
cfs_hash_create() was called with missing CFS_HASH_NO_ITEMREF
flag which was causing extra *_put to be called on ptlrpc_exit()

ptlrpc_exit
  cfs_hash_destroy
    cfs_hash_bd_del_locked
      cfs_hash_put_locked

For this, the patch add CFS_HASH_NO_ITEMREF flag to all calls
to cfs_hash_create() under NRS/TBF

While the straight port (7375cd6da6329e) from atomic_t to
refcount_t makes this bug stand out in 4.18 and 5.0 kernel
(not seen on 3.10) the bug itself is not due to 7375cd6da6329e

I don't think this is right - the real bug is that someone is trying to release a reference incorrectly. I think the source of the warning is here:

2978         nrs_tbf_cli_init(head, cli, req);
2979         tmp = head->th_ops->o_cli_findadd(head, cli);
2980         if (tmp != cli) {
2981                 refcount_dec(&cli->tc_ref);
2982                 nrs_tbf_cli_fini(cli);
2983                 cli = tmp;
2984                 if (IS_ERR(cli))
2985                         return PTR_ERR(cli);
2986         }

In this case, refcount_dec() will always emit an error. We should either refcount_dec_if_one() and check the return code (i.e. https://review.whamcloud.com/c/fs/lustre-release/+/56808) or not initialize the refcount until we're ready to give out references.
