## Summary of the Fix

### Problem
The patch introduced two new helper functions `folio_from_iter()` and `folio_to_iter()` in `lustre/obdclass/cl_io.c` that unconditionally used folio-related types and functions. However, when `HAVE_FOLIO_BATCH` is not defined (on older kernels), the compatibility macros in `lustre/include/lustre_compat.h` define:
- `page_folio(page)` as just `(page)` - returning `struct page *`, not `struct folio *`
- `folio_page(page, n)` as just `(page)`

This caused type mismatches and undefined function errors when compiling for older kernels.

### Rationale for the Fix

The fix uses conditional compilation to handle both modern kernels (with folio support) and older kernels (without folio support):

1. **`folio_from_iter()` function:**
   - When `HAVE_COPY_FOLIO_FROM_ITER_ATOMIC` is defined (modern kernel): Uses actual folio types and functions (`page_folio()`, `folio_page_idx()`, `copy_folio_from_iter_atomic()`, `flush_dcache_folio()`)
   - When `HAVE_COPY_PAGE_FROM_ITER_ATOMIC` is defined (intermediate kernel): Uses page-based API (`copy_page_from_iter_atomic()`, `flush_dcache_page()`)
   - Otherwise (older kernel): Uses the legacy `iov_iter_copy_from_user_atomic()` with page-based API

2. **`folio_to_iter()` function:**
   - When `HAVE___FILEMAP_GET_FOLIO` is defined (modern kernel): Uses folio types and `copy_folio_to_iter()`
   - Otherwise (older kernel): Uses page-based `copy_page_to_iter()`

3. **Key changes:**
   - Moved `struct folio *folio` and `int pgno` variable declarations inside the conditional blocks where they're actually used
   - Changed `folio_page(folio, pgno)` to just `pg` in the non-folio code paths
   - Changed `flush_dcache_folio(folio)` to `flush_dcache_page(pg)` in the non-folio code paths

This approach ensures the code compiles correctly on both old and new kernels by only using folio-specific types and functions when the kernel actually supports them.
