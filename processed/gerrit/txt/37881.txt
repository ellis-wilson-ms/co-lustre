--- Commit Message ---
LU-9859 libcfs: remove conditional compilation from libcfs_cpu.c

libcfs_cpu.c manages CPU partitions. In the !CONFIG_SMP case, most
of this disappears and 'static inline's from libcfs_cpu.h are used.
However we still allocate a 'struct cfs_cpt_table' which is used
to keep the NUMA node mask. Event with SMP disabled Lustre wants
to handle any NUMA nodes available.

This patch removes all the !CONFIG_SMP code from libcfs_cpu.c and
conditionally compiles the whole file only when CONFIG_SMP.
We no longer allocate a 'struct cfs_cpt_table' on !CONFIG_SMP,
and don't even declare a structure.  The name "cfs_cpt_tab"
becomes always "NULL", which allows some code to be optimized away.

This means that cfs_cpt_tab can sometimes be NULL, so we need to
discard the assertion that it isn't.

I don't think autoconf is that smart. Do you know a way to do it? As for why cfs_cpt_table is not NULL is that while we might have one core, SMP disabled, that doesn't mean we have only one NUMA node. In fact some platforms like the Intel Phi have node nodes with no associated cores. So we have to handle such platforms.

--- Message (Neil Brown, 2020-04-17 00:31:44.000000000) ---
Patch Set 5:

> Do you know a way to do it?

I gave you a way to do it.  Defining thing-${CONFIG_FOO) to the .o files to be included if CONFIG_FOO is y, then including $(thing-y) in the final target is the standard way of doing conditional compilation in the kernel.

As for cfs_cpt_tab being needed on non-SMP machines, I don't see it.
cfs_cpt_tab is initialized either by cfs_cpt_table_create() or cfs_cpt_table_create_pattern().
The former will abort if the number of partitions given is larger than that number of cpus.  So it is limited to 1 on non-SMP.
The latter is more complex so it is hard to be sure, but it often ignores nodes with no CPU, and has an error
 "No online CPU is found on partition %d"

So it looks to me like you can never have a node listed in cfs_cpt_tab which doesn't have any online CPUs. 

Can you explain to me how cfs_cpt_tab will be used when there is only one CPU?

--- Message (James Simmons, 2020-04-17 14:23:47.000000000) ---
Patch Set 5:

The OpenSFS branch doesn't use Kbuild to build its kernel modules. We use autotools - https://www.gnu.org/software/automake/manual/html_node/Usage-of-Conditionals.html. So the kernel approach will not work for us.

The Lustre CPT implementation handles both cpumask_var_t AND nodemask_t. Its completely valid to have systems with one core and two NUMA nodes for example.
Your original patch hard codes the UP case that the nodemask_t is NULL which is wrong in the example I gave you. This patch fixes that bug by treating the nodemask_t as the cfs_cpt_tab so things can remain inline for the UP case. In any case a proper nodemask_t as to be returned by cfs_cpt_nodemask().
