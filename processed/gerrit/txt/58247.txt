--- Commit Message ---
LU-17916 osc: switch to struct folio

Switch from struct page and page based API to struct folio
and the folio API.

When getting user pages with get_user_pages et. al. as
is used for direct i/o mechanical conversion of page to
folio has to consider the case of huge pages and the
same folio if retrieved for many pages.

So the original offset to the page must be passed down
and used when working with the pages in bulk r/w and
page encryption.

This prepares the osc brw for handing such folios.

> How does the folio conversion intersect with https://jira.whamcloud.com/browse/LU-13814?

Conflicts I and/or Patrick need to understand how to resolve.
As a 'struct page' the GUP <-> brw path is self contained, as a 'struct folio' the extra 'offset' or 'pgno' needs to be retained.
Currently the cl_page is taking that role of holding the folio offset(pgno) and passing it forward to the brw struct. However LU-13814 will drop cl_page so some other intermediary will handle the mapping.

Ideally there would be a way to drop a large folio into the the list of 'pages' and create  create multiple brw_page entries, or make the brw_page entries offsets and extents.

The best is if cl_page described a folio directly and could pass that down, however there is a lot of PAGE_SIZE logic around combining and splitting extents that also needs adjusting.

RE LU-13814; we'll have to work that out.

That series speeds up DIO by about 5x, but it does by removing the usage of *lists* of cl_page and replacing it with a single block of - eventually - basically BRW pages, allocated and freed all at once.  No pointer chasing, etc.

I haven't thought too hard about this, but it doesn't seem like that's fundamentally incompatible with folios, we'd just need to move the abstraction Shaun's created out of cl_page and (essentially) down to brw page.  After LU-13814, buffered IO uses cl_page, but DIO does not.  Moving the abstraction there would increase the size of brw page, but, I mean, sometimes new functionality does that.

Shaun, assuming you were fairly free to modify brw page, does that seem doable to you?

(LU-13814 initially uses an array of osc async pages rather than brw pages, renaming them osc transfer page since they're not actually async.  But *eventually*, at the very end, there's nothing in the osc transfer page but a BRW page.)
