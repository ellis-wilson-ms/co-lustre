LU-1199 build: Clean out the build directory

Clean up the build directory.  Move in the direction of reserving
"build/" for the special-purpose Makefile that lives there.
Eventually we could rewrite the autoconf tests to eliminate that
Makfile, and the build directory could disappear altogether (after
finding homes for anything else that is left).

The autoconf m4 file move into a top level "config" directory.

Most other things that have moved are put in the new "contrib"
top-level directory.  For instance, "contrib/lbuild" contains
all of the lbuild-related files, and "contrib/git-hooks" contains
the git hooks for lustre developers.

Most of the moved files were unchanged, however the lbuild scripts
needed some tweaking to deal with the new location.

Because of the way that Intel's build farm and git hooks expect to
find certain files in fixed locations I have had to leave a few
symlinks in place that point to the new locations.

Patch Set 5:

To be honest, I'm not really clear on the value of moving everything out of the "build/" subdirectory into the "contrib/" subdirectory.  To me, "lbuild" would seem to be clearly related to "build", and the commit hooks are more related to "building Lustre" than "external contributions" since these are needed for every patch submitted for Lustre.

IMHO, "contrib/" is for code/patches that are indirectly related to the Lustre code, or are not going to be applied to the tree, but may be useful for others to use (e.g. HSM copytool, ADIO drivers, wireshark, some of which are under "lustre/contrib" today).

I can definitely see value in putting a bunch of related files into a subdirectory.  I'm willing to be flexible if there is some concrete reason behind this, but I can't see any benefit from essentially renaming the directory from "build" to "contrib"?

--- Message (Christopher J. Morrone, 2013-02-25 18:14:31.000000000) ---
Patch Set 5:

My take on it is that lbuild is _not_ part of the Lustre tree's build system.  Nothing in the lustre tree requires or employs lbuild as part of the build process.

Instead, lbuild is a higher level of scripting primarily employed by Intel's build farm to build the Linux kernel, lustre, ofed, etc., whatever.  As such, is should not be intertwined with the specifics of the Lustre tree's build system.

Also, others might find it useful (although it is very clearly not written in a way to be easily usable by anyone else).

If I could claim that no one else will want to look at lbuild, then I would argue for removing it from the tree all together.  But seeing as it may have value for others, contrib seems to me to be the most appropriate place for it to live.

--- Message (Christopher J. Morrone, 2013-02-25 18:26:38.000000000) ---
Patch Set 5:

And about commit hooks:

I would argue again, that nothing in Lustre's build system uses them.  Lustre's build system is very serpentine, and difficult to understand.  I'd like to reduce confusion as much as reasonable by keeping it separate from things that aren't part of the Lustre tree's build process.  To me, "build" means compilation and/or packaging.

In my mind, I would call the commit hooks a "developer tool", not something that is part of the tree's build system.

Perhaps that means they should live in a top level "scripts" directory instead of "contrib".  I'm willing to consider other options.  But I'm not fond of them being in "build".

Furthermore, the "Makefile" in "build" has a very special, and difficult to understand usage.  It is called in at least 4 very different contexts, if memory serves.  Every time we allow something in "build" someone is going to be tempted to modify that Makefile, and that should be discouraged.

I would honestly like to make that Makefile go away entirely, but I won't have the time to do that for 2.4.  In the mean time, keeping it as isolated as possible is the next best thing.
