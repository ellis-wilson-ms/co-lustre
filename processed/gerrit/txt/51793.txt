LU-17000 misc: remove Coverity annotations

These Coverity function annotations were added
around 10 years ago. Since then, Coverity seems
to produce less false positives. Out of about 20
annotations, only 3 warnings get surpressed.
Thus, the applicability of these annotations
should be re-evaluated.

Coverity has more advanced tools now for reducing
false positives. Various Lustre functions and
macros could be modeled rather than using
function annotations. But first, we need to get
a good idea of what kinds of false postives are
being generated.

https://scan.coverity.com/tune

Thanks Patrick. I'll have more static analyzer fixes (at some point). I always thought static analyzers and compilers were neat. I like to try out new ones when I have a chance. So far I've tried or looked at:

 1) Clang/LLVM (really useful, great tool ecosystem, I use it for all my patches)
 2) Clang plugins (custom ones are easy to write, although I never did get the old Lustre plugins working right)
 3) Coverity (some warnings are really helpful, some not. You can see them here https://scan.coverity.com/projects/lustre if you're curious)
 4) Cppcheck (I didn't think it was very helpful at all)
 5) Sparse (Could not get it working, but I didn't try very hard)
 6) frama-C and some formal verification tools. Seemed interesting, although I don't think those would be practical.
 7) ARM MTE (https://learn.arm.com/learning-paths/smartphones-and-mobile/mte/mte/) which can be tested on QEMU I believe. It's supposed to be hardware support for memory safety. Haven't tried it yet.

If you know about any other interesting tools I could try, let me know.
