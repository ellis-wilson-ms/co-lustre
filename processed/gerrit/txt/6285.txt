LU-3268 lod: return root LOV EA in little-endian

In lod_xattr_get() when constructing LOV EA to return for the
filesystem root, store members in little-endian byte order for
consistency with EAs from disk.

--- Message (John Hammond, 2013-05-08 15:01:20.000000000) ---
Patch Set 1:

> hmm, actually I thought we should do swabbing on demand on the client to save cpu cycles on the server. correct me if i'm wrong...

You are not wrong, but I still hope to convince you on this approach.

1. This has better interoperability. The client /should/ do swabbing on demand. However when a 2.x client swabs LOV xattrs, it does so in a way which assumes that they come over the wire in little-endian. From ll_dir_getstripe():

        /*
         * This is coming from the MDS, so is probably in
         * little endian.  We convert it to host endian before
         * passing it to userspace.
         */
        /* We don't swab objects for directories */
        switch (le32_to_cpu(lmm->lmm_magic)) {
        case LOV_MAGIC_V1:
                if (LOV_MAGIC != cpu_to_le32(LOV_MAGIC))
                        lustre_swab_lov_user_md_v1((struct lov_user_md_v1 *)lmm);
                break;
        case LOV_MAGIC_V3:
                if (LOV_MAGIC != cpu_to_le32(LOV_MAGIC))
                        lustre_swab_lov_user_md_v3((struct lov_user_md_v3 *)lmm);
                break;
        default:
                CERROR("unknown magic: %lX\n", (unsigned long)lmm->lmm_magic);
                rc = -EPROTO;
        }

2. This is less surprising. For every file or directory other than the root the server will store the LOV EA on disk in little-endian and hence will put it on the wire in that order.
