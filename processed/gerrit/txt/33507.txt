Patch Set 6:

> one observation is that the policy must be very selective and smart
 > mixing cacheful and cacheless IO within a single object.  otherwise
 > direct writes would need to invalidate pagecache which makes the
 > whole thing useless (as we want to avoid pagecache costs).
 > another option could be slow, expensive and rare switches between
 > cacheful and cacheless modes involving pagecache invalidation.

I'm fine with relatively rare cases where switching between cached/uncached IO is needed, similar to how we decide readahead is sequential or random, with some hysteresis to avoid changing too quickly between modes.  My expectation is that changing from uncached to cached is relatively easy, but the reverse is more work (purge page cache).  Unfortunately, it would be better if going from cached to uncached was easier, since we typically start with smaller files and don't know if they will grow large or not.

I think a reasonable compromise is if the object is empty and gets some large initial IOs (e.g. few MB) then it can make a decision to be uncached, and would not change unless e.g. the sum of random writes > 4x the sum of sequential writes (or the reverse to change from cached writes to uncached if the initial writes are random).

This allows some possibility to change the behaviour of a file that still has cached pages in memory, while avoiding the cost of switching if it isn't needed.  I think most files would keep their behavior for their lifetime in cache, but it wouldn't be hard to flag files that should be treated one way or the other via ladvise or some heuristic based on previous access patterns.

The advantages of having policy for it are:

1) It is automatic as soon as the administrators finbished the configuration of the rules.

2) The rules can be very flexibily.

I think ladvise won't be able to get the these advantages. Because ladivse needs to be issued by an external tool, it has some disadvantages

1) The external tool + ladise won't be able to get information about the I/O request (i.e. UID/GID/JOBID etc.)

2) The external tool will need to have similar policy itself to be smart.

Of cource, we could use ladvise as an interface of the configurating the rules of the policy. That is another story.
