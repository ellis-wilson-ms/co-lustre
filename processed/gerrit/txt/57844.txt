--- Commit Message ---
LU-10026 ptlrpc: verify large allocations are aligned

If we were ever to do an allocation with kmalloc(), we could
get non-page-aligned memory.

We haven't seen any problem yet, but we are trying to be ready for
this in advance. The arguments from https://lwn.net/Articles/787740/
also look strong.

Let's add an assertion to illuminate this dangerous behaviour.

Patrick, are you sure that kmalloc(>=8192) will *not* allocate PAGE_SIZE aligned memory? 

I don't think we will ever use allocations in the page pool smaller than 1 page, and alternately we could just allocate the 1- and 2- page sizes directly with alloc_pages?

James,
this is a pre-allocated page pool, so the allocation performance isn't  critical.  However, I'm not sure the use of vmalloc() for the small allocations is required, and kmalloc() is more efficient if it is available (which should typically be the case if the filesystem is mounted at boot time).

-1 until this is better understood.

--- Message (Patrick Farrell, 2025-01-23 20:25:56.000000000) ---
Patch Set 1:

I don't *think* so, though I guess I'm not 100%.  Here's an LWN quote from 2019:

"In particular, Babka wanted to discuss when kmalloc() should return objects with their "natural alignment". What that term means was not actually defined until near the end of the session; presumably nobody will object to a slight bit of reordering at this point. Natural alignment for an object means that its beginning address is a multiple of each of up to three different quantities: ARCH_KMALLOC_MINALIGN (eight bytes, by default), the cache-line size (for larger objects), and the size of the object itself when that size is a power of two. The actual required alignment will generally be the least common multiple of the three."

https://lwn.net/Articles/787740/

So does it make sense to use "alloc_pages()" in the 1/2 page cases, since we know it is always at least one whole page being allocated, or is that over-complicating the matter when contiguous pages cannot be allocated?

I see in my el8.10 slabinfo that all of the kmalloc-SIZE slabs are properly aligned, even down to 256 bytes, but I don't have any debugging enabled:

 # name     <objsize> <objperslab> <pagesperslab>
 kmalloc-8k      8192            4              8
 kmalloc-4k      4096            8              8
 kmalloc-2k      2048           16              8
 kmalloc-1k      1024           16              4
 kmalloc-512      512           16              2
 kmalloc-256      256           16              1

so this shows the 4k slab is using 8-page slabs, and allocates 8x 1-page objects per slab, and likewise the 8k slab is using 8-page slabs and allocates 4x 2-page objects per slab.
