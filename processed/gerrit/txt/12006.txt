LU-5231 hsm: incorrect progress reported by lfs hsm_action

When hsm_action command is executed on a file on which
hsm archieve or hsm restore was going on, the progress report show
was wrong.

The issue is hsm_action gets the progress report (how much data
moved) from kernel(through ioctl) and the size will be updated
through ioctl from copy tool.
Basically copy tool is sending amount of data moved
(progress report) properly but the data which is sent from copy tool
to kernel was getting added every time with previous value rather
than simply assigning the value.
changes made: Add operation is taken out from the current code
fix is in: lustre/mdt/mdt_hsm_cdt_requests.c

The issue is not only with 5GB file but with all the files. For small
files it was not observed because interval for ioctl on copy tool was
long(30 sec),
changes made: reduced the time interval to 10 sec to get some good,
frequent progress report on client side
fix is in: lustre/utils/lhsmtool_posix.c

also final update about the progress was not used to sent to kernel.
changes made: Once the copy operation is finished, final ioctl is
sent to kernel
fix is in: lustre/utils/lhsmtool_posix.c

Patch Set 2:

Bug can be reproduced by following steps :

1. Create a file of size 5GB (ioctl interval time on copy tool agent is long currently, that is the reason why we don't get the progress report for small files. so considered 5GB file  ).
2. Once the file is created then archive it "lfs hsm_archive file_name". During the archive process check for the status of the file using
"lfs hsm_action file_name" (Keep on checking for the status again and again, after some several checks bytes moved will be larger than the actually file size)

Observation : Progress achieved (how much data is copied) will be wrapped into hsm_extent struct and ioctl will be performed from copy tool agent in certain intervals. So going through all these extents and adding each extent size will give wrong value (In fact it is not necessary to go through all these extents, as the ioctl which is very recent (very recent extent) gives the total data moved (end - start)).
