LU-12029 utils: l_tunedisk only tune mounted target

The udev rule 99-lustre-server.rules applies Lustre disk tuning on
every change/add Lustre device.
This patch restricts the tuning on mounted ldiskfs device and does not
use debugfs/e2fsprogs API to determine if the device is Lustre.

If the same device is seen on a lot of server, this can hang the
target because of the concurent small read (4K) created by
debugfs/e2fsprogs API.

Add regression test conf-sanity 125b

Test-Parameters: testlist=conf-sanity env=ONLY=125a,ONLY_REPEAT=10

(3 comments)

I'm not sure I understand this. How could l_tunedisk ever run on a mounted device if the udev rule is run at device insertion time, before Lustre could possibly have mounted it?

We used to only do this at mount time in mount.lustre, but then it was moved to a separate udev rule to avoid problems with changing the device from what the admin wanted (in some cases a large max_hw_sectors_kb is advertised by the device but it doesn't work properly), and "-o max_sectors_kb=0" was added to avoid mount.Lustre from ever changing the value. I _think_ that is now the default. 

At this point I think that it would be better to just remove the max_sectors_kb tuning from mount.Lustre and l_tunedisk completely and it is up to the admin/var to configure the system properly.  This has been an endless source of problems and "solves" a problem that is less critical in newer kernels. 

Separately, a kernel patch should be made to fix device mapper/MD to set max_sectors_kb from the value on the parent to sub-devices if they are (re-)added to a volume/multipath after the tuning has been applied.  That is the reason for l_tunedisk to exist - so it can work around a bug that should really be fixed in the kernel.

Once the kernel patch exists, it can be backported to the production kernels as needed and this whole mess removed.
