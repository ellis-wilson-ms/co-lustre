--- Commit Message ---
LU-16538 dom: return PR or PW mode lock with OPEN|DOM bits

In this patch, it returns PR lock mode for read open or PW lock
mode for write open combining OPEN bit with DOM in an ibits lock
to a client for DoM files.
By this way, it can reduce an extra lock traffic for the
subsequent data I/O.

Add sanity/test_435 to verify it works correctly.

--- Message (Qian Yingjin, 2023-02-16 15:23:34.000000000) ---
Patch Set 1:

What's about to change "dom_lock" to "dom_only" files?

I agree that returning PW mode OPEN bits lock to a client may have bad impact on the performance for shared access mode. 
But DoM-only files are usually small. DOM lock is a [0, EOF] lock for DoM-only files. The impact on DoM-only files may be not so big.
The reason I want to grant a PW OPEN|DOM ibits lock to a client is for the next step work in LU-16429.
In the future, I will add some open flags:
- MDS_OPEN_WILLREAD
- MDS_OPEN_WILLWRITE
- MDS_OPEN_WILLSTAT
These flags are set by a client and used  to advise the MDT to grant PW or PR OPEN lock if known or detect that an application on the client will open|create or read/write, stat the files  (single thread process like "cp" files one by one serially) in the future. With OPEN bit combined with DOM or UPDATE|PERM bits in an ibits lock granted to a client via a kind of ahead operations, the subsequent operations can be performed locally without any RPC contract to  MDT.
By prefetching or ahead operations, we can tun some serial operations (cp, grep, or mpiFileUtils/dcp) into parallel or batching operations, and improve the overall performance especially for massive small files workload.
