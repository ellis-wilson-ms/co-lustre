## Summary

**What was the compile error:**

The patch introduced a compile error in `lustre/llite/pcc.c` because the `ll_filemap_get_folios` macro was incorrectly defined for older kernels that don't have `HAVE_FOLIO_BATCH` and `HAVE_FILEMAP_GET_FOLIOS`.

The error occurred because:
1. The macro referenced a non-existent config option `HAVE_PAGEVEC_LOOKUP_THREE_PARAM`
2. One branch tried to call `pagevec_lookup(pvec, m, s, PAGEVEC_SIZE)` with 4 parameters
3. But the kernel's `pagevec_lookup()` function expects only 3 parameters: `(pvec, mapping, start_ptr)` where `start_ptr` is a pointer to the start index

**The fix:**

I removed the `#ifdef HAVE_PAGEVEC_LOOKUP_THREE_PARAM` conditional and its associated 4-parameter version, keeping only the correct 3-parameter version that passes the index as a pointer (`&s`).

**Rationale:**

The `pagevec_lookup()` API in modern kernels (since around 4.14+) uses a 3-parameter signature where the third parameter is a pointer to the start index. The function updates this pointer to indicate where to continue the next lookup. This is the standard API that should be used when `HAVE_FOLIO_BATCH` and `HAVE_FILEMAP_GET_FOLIOS` are not available.

The fix ensures that:
- `ll_filemap_get_folios(m, s, e, pvec)` expands to `pagevec_lookup(pvec, m, &s)`
- The `s` parameter (index) is passed by reference as expected by the kernel API
- The macro works correctly with the usage in `pcc.c` where `index` is a `pgoff_t` variable
