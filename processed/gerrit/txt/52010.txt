Patch Set 20:

I would like to see the kernel libcfs disappear completely.
Having a "compat.ko" to support older kernels certainly makes sense.

I'd like the debugging code to all go in favour of tracepoints, but that has to be a long-term goal I suspect.  I would be in favour of moving everything that cannot be discarded into either 'lnet' or 'lustre' or a new 'compat' module.

Speaking of modules - there are far too many. mgc lov mdc osc lmv fid fld  ptlrpc obdclass don't all deserve to be separate modules.  They should be part of "lustre".
The client should only have "lustre" "lnet" and any k*lnd modules that it needs.
The server would also have "lustre-server" and "osd-*" and possibly "ldiskfs" (which will eventually be just "ext4" - but that's also a long-term goal).

I think some aspects of Lustre debugging are more ergonomic than native Linux debugging (a lot of it could be more refined). A lot of that experience could be recreated in user space I suspect, regardless of what the kernel is doing.

I've thought developing some kind of distributed tracing for Lustre would interesting (along the lines of opentelemetry https://opentelemetry.io/docs/concepts/signals/traces/). So I think retaining some custom debugging could let us explore some interesting ideas.

As for the modules, it's hard to say which should be bundled because of the common dependencies between client and server (ptlrpc, obdclass, etc.)
