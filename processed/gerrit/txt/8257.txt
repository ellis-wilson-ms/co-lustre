--- Commit Message ---
LU-4185 llite: Revise create with no open optimization

Currently ll_lookup_nd just returns a negative when we are trying
to create something with no open (read mkdir). This is all fine most
of the cases, except if the directory where we are trying to do is
not writeable by us. In that case vfs_create would return EPERM
seeing as how a negative dentry means the create cannot proceed.
But in reality if there is an existing name there that we just did
not have cached, the proper return is EEXIST that could only happen
if we did do the lookup.

So amend the optimization to only take place if the directory is
writeable by us, otherwise do the full lookup.

--- Message (Zhenyu Xu, 2016-06-06 15:15:22.000000000) ---
Patch Set 6:

> Does Oleg's patch http://review.whamcloud.com/20354 fix this problem?

No, it doesn't.

The issue is that with non-open create op from the client, vfs first lookup the dentry, our default ll_lookup_nd() will return a NULL dentry in this case, to save a intent lock RPC to MDS, and leave the server work to later create RPC; then vfs goes to vfs_create(),  and it check may_create() to verify the create  permission, and fails in that spot.

Without the optimization, ll_lookup_nd() will go through to ll_lookup_it() which issues the intent lock RPC to MDS, and MDS will returns EEXIST to client, so that the application report EEXIST before checking  the create permission.

--- Message (Andreas Dilger, 2016-06-06 18:13:10.000000000) ---
Patch Set 6:

Since the permission check goes into ll_inode_permission(), could we fix that to return the correct error code (or no error code at all and leaving the check to the MDS) without breaking the optimization?

--- Message (Zhenyu Xu, 2016-06-07 03:22:37.000000000) ---
Patch Set 6:

may_create(parent_dir_inode, child_dentry) -> inode_permission(parent_dir_inode, MAY_WRITE | MAY_EXEC)

It contains no lookup data info in it, ll_inode_permission() lacks the context of this permission check, it only want to know whether the application has the permission to write in the parent directory. I cannot think of a way other than bypass the exact  (MAY_WRITE | MAY_EXEC) check in ll_inode_permission(), but I am not sure whether it would cause another issue.
