--- Commit Message ---
LU-18400 tests: enable panic_on_warn

Enable panic_on_warn to catch kref bugs (and other
strange warnings) that might otherwise get ignored.

This is enabled by default for all test suites.
However, explicitly disable it for test suites that
still have warnings. This will prevent future
regressions while enabling developers to address
existing warnings incrementally.

--- Message (Andreas Dilger, 2025-06-20 21:46:03.000000000) ---
Patch Set 6:

We strongly need to consider some resources outside of WC/DDN for running potentially expensive testing options like this.  With increasing numbers of external Lustre developers repeatedly pushing patch updates for small changes to their patches, we are regularly seeing test backlogs during the week of nearly a full day for the ~300 test clusters that we run internally, and they only catch up on the weekend.

I'm not against added testing by any means, but we do not have an infinite supply of test hardware either.

> I'm not against added testing by any means, but we do not have an infinite supply of test hardware either.

I think it makes sense to rate limit individual developers and allow developers to kill their own test sessions. That'd reduce the blast radius when someone pushes a huge series or a lot of small patches. If someone does push tons of patches - they can either wait a long time, stop some of their sessions, or run their testing on their own hardware. You could implement prioritization as well i.e. prioritize WC/DDN developers over external developers or prioritize testing important bug fixes.

I'm not sure how difficult it'd be to implement that in Maloo. But I think it'd be the most effective solution to long test backlogs.
