--- Commit Message ---
LU-1760 utils: Avoid Mis-leading LNet self test error message

Executing "lst new_session" when lnet-selftest module is not yet
loaded can create confusing error messages like "Invalid parameters
list in command line".

This change attempts to load the lnet-selftest module from the
"lst new_session" command on error, and tries again to create the
session.

I think it'd be possible to find out by IOCTL(2) return that the LSTIO_SESSION_NEW is not supported, which indicates that the module has not been loaded.

At the end of libcfs_ioctl_int, the default branch for the switch:
        default: {
                struct libcfs_ioctl_handler *hand;
                err = -EINVAL;
                cfs_down_read(&ioctl_list_sem);
                cfs_list_for_each_entry_typed(hand, &ioctl_list,
                        struct libcfs_ioctl_handler, item) {
                        err = hand->handle_ioctl(cmd, data);
                        if (err != -EINVAL) {
                                if (err == 0)
                                        err = libcfs_ioctl_popdata(arg, 
                                                        data, sizeof (*data));
                                break;
                        }
                }
                cfs_up_read(&ioctl_list_sem);
                break;
        }

The default value for 'err' should be -ENOTTY, the error code for inappropriate ioctl command according to POSIX. I believe libc6 strerror(ENOTTY) would say "Inappropriate ioctl for device", which is more appropriate. Also, in this case, lst should remind user to load the module not only on the current node but also on all test nodes - the user may very well forgets to load it on test nodes too, which would cause another likely hard to decipher error later on.

Patch Set 2:

Isaac:

I did some printing to see if that err number gets propagated back.  It doesn't.  All errors get converted to a -1 and errno does not seem to reflect that error (ENOTTY).  Also, the errno changes based on the conditions.  For example, if neither lnet nor lnet-selftest are loaded, we get ENODEV since the lnet device is missing.  If lnet is loaded but lnet-selftest is not loaded, we may get EPROTO (Network no available).

Coming up with a good way to predict when lnet-selftest is loaded or not does not seem simple.  That is why I just stuck to attempting to load it and see if that resolves the issue.

As to the compilations errors this change is getting, seems one of our build systems is complaining that I am not paying attention to the return code from the system() call.  I will fix.
