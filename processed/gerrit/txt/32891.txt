LU-11186 ofd: fix for a final oid at sequence

There was an error at the end of sequence range and last oid
0xffffffff can't be created. The 0xffffffff is a valid oid, and
sequence update happens only if it is created.

LustreError: 11756:0:(ofd_objects.c:217:ofd_precreate_objects())
lustre-OST0000:0xfffffffe:10737419264 hit the OBIF_MAX_OID (1<<32)!
LustreError: 11756:0:(ofd_dev.c:1764:ofd_create_hdl())
lustre-OST0000: unable to precreate: rc = -28

The patch fixes this error.

The conf-sanity 122 is added for checking sequence update.

It isn't clear what happens in the overflow case?  The OST returns -ENOSPC one object later, but I don't see in ofd_create_hdl() or ofd_precreate_objects() where this is handled, except to set sync_trans and return -ENOSPC if no objects could be created at all.

Given the breakage seen in this bug, it is clear we have no test for this OID wrap-around.  Could you please add a test case for this.  Probably the easiest is to add a OBD_FAIL check that bumps up the OID value on both the MDS and OSS to (2^32-16 or similar), so we aren't stuck creating 2^32 objects.
