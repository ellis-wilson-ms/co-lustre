--- Message (Andreas Dilger, 2011-05-24 18:30:23.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

I think the current structure of this patch risks ignoring errors that were not seen by the application:

write1 - return error, set lli_write_rc, application rewrites
         data for file
write2 - return 0 (async RPC error not seen by application)
fsync  - does not return error because lli_write_rc is set

It would be better to have ll_file_write_aio() check and clear the lli_async_rc() when it returns an error to the application.

Even better than this (probably only for 2.x) is to get rid of lli_async_rc and loi_ar.ar_rc and use the normal kernel "page error" and "address space error" mechanisms to track errors.  That allows the errors to be cleared on a per-page basis if the pages are rewritten, and sync_page_range() will also return errors only for pages within that range instead of an error on another random page in the same file.

--- Message (Fan Yong, 2011-05-25 14:03:34.000000000) ---
Patch Set 3:

I rename per-inode structure based 'lli_write_rc' as per file-structure based 'fd_last_write' to track the last write/fsync result (success or failure) through the 'file' structure. So when 'sys_close()' is called against the 'file' structure, we know whether the caller has known write/fsync failure or not, and then avoid report confused failure.

As for 'lli_async_rc', they do not conflict:
1) if 'fd_last_write' is set, then means last write/fsync failed, and the caller has known that, so 'sys_close()->ll_flush()' returns success.
2) if 'fd_last_write' is reset, then process as original logic

Is anything wrong?
