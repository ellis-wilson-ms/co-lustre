Patch Set 2:

Consider the following case:

1) At time point_A, the administrator dump lfsck status: the scanned_objs is 'cA', the position is 'pA'.

2) At another near time point_B, the administrator dump lfsck status again: this time the canned_objs is 'cB', but the position is still 'cA'.

If I am the administrator, I will suspect that it is Lustre BUG. Because I cannot imagine why some object has been scanned but position NOT moved? The time point_B depends on the frequency of the cached position to be updated and the schedule delay.

On the other hand, from the test view, updating the cached position by the lfsck engines itself may be NOT good, because there will be some "fail_loc" to make the lfsck engine to sleep sometime and slowdown the lfsck progressing. But such sleeping will affect the cached position updating, then many of test cases may need to be reconsidered and re-implemented.

In fact, although we will NOT update the cached position very frequently, we have to check (after each object scanned) that whether it is time to update the cached position. Although such check will not be much expensive, the total cost for all the scanned objects may cannot be ignored.

So my idea is that: if we have to dump the 'position' information, we should give a relative accurate position which will be updated together with the objects scanning to NOT confuse the administrator and simplify the test/implementation.

Current mechanism and implementation is a "lfsck" status snapshot with very low cost, which will query the position only when requested.
