(1 comment)

Personally I think I don't really agree with this.

As I commented elsewhere, in my view we should not attempt ELC for glimpses as there's no possible conflicts, and since glimpses are the other kind of locks we send in parallel, we should be very careful with what we do while doing it - i.e. doing funny stuff (like walking/cancelling locks) should be off-limits.

the extra "no dirty pages" check you added should not really help, right?
The previous check was for READ locks -- i.e. there was no chance there would have been any dirty pages anyway, right?
Now we select write locks too and add a racy "no dirty pages" check. Potentially even making things worse?

"I don't get your comment about "unrelated" io during lock enqueue. Where would that IO come from? Why would that lock be chosen to be cancelled? Certainly LRU policy looks like it would only allow read locks to be selected for lru cancel (before the patch)."

I think you're confused this with cancel_on_recovery ;-)

OK, say dlm lock foo_A is a lock of resource foo which is currently being queued; and dlm lock bar_B is a write lock on unused list of resource bar. With ELC, enqueuing foo_A will cause bar_B to be canceled, and canceling bar_B will generate write back RPCs.
