--- Commit Message ---
LU-397 Set 'lsr_flags' as 'LU_SEQ_RANGE_MDT' for old 2.0 client

Lustre 2.x introduces server identifier into lu_seq_range to distinguish
MDT and OST fids. But for old 2.0 client, it does not specify such
identifier/flags. So use 'OBD_CONNECT_SEQ_RANGE' to distinguish whether
it is old 2.0 client or not. For old 2.0 client, set the 'lsr_flags' as
'LU_SEQ_RANGE_MDT' by default.

> Compatibility with liblustre is not a concern, because it already will refuse to connect if the version is different by +/- 0.3 or so.

Seems you misunderstand me. Currently, liblustre client does not support 64-bit dir hash, for both the old 2.0 liblustre client and new 2.1 liblustre client. And 'OBD_CONNECT_64BITHASH' is used for telling the MDS whether the connected client (normal/liblustre) support 64-bit dir hash or not. So means new 2.1 liblustre client does not specify 'OBD_CONNECT_64BITHASH' when connect.

If we want to redefine 'OBD_CONNECT_SEQ_RANGE' as 'OBD_CONNECT_64BITHASH' to distinguish old 2.0 client or new 2.1 client for fld lookup interoperability issues, that means new 2.1 liblustre client also need to specify 'OBD_CONNECT_SEQ_RANGE', which conflict with above assumption.

So I have to make some hack: let new 2.1 liblustre client specify 'OBD_CONNECT_SEQ_RANGE' when connect, and it is new 2.1 server's duty to distinguish liblustre and return 32-bit dir hash for it. It works, but breaks the original meanings of 'OBD_CONNECT_64BITHASH'.
