LU-17643 gss: make a local copy of the sptlrpc llog

Make a local copy on server side of the sptlrpc llog, so that
the targets that do not manage to connect to the MGS know at least
which security flavor to accept from clients.
This needs to pass the super_block to config_log_find_or_add().

Add sanity-sec test_70 to check that sptlrpc llog on MDS and OSS side
is equivalent to the one from the MGS.

This must rank as one of the higher value 2-character patches, at least as far as Kerberos goes...

You could add a test case for this, even just something like:

 test_NNN() { # LU-17643
        # this doesn't really *require* ldiskfs, but that is easier
        # to just check this directly without unmounting the target
        [[ "$ost1_FSTYPE" == "ldiskfs" ]] ||
                skip "need ldiskfs to use debugfs"

        # chec ost1 to avoid false positive if MGT is on MDT0000
        do_facet ost1 "debugfs -c -R 'ls -l CONFIGS' $(ostdevname 1)" |
               grep sptlrpc || error "sptlrpc not copied to ost1"
 }
 run_test NNN "confirm that sptlrpc config is copied to target"

More complete would allow "lctl --device %$FSNAME-OST0000 llog_print $FSNAME-OST0000" to be run on something other than the MGS so we could dump their local config logs (which might also be handy in the field).

This could be run relatively late in sanity-sec.sh so that the file copy is sure to be saved to the disk device, instead of right after the first mount when it might still be in the journal or loopback device cache.

A slightly better test would compare that the size of the sptlrpc log was the same on the MGS and OST0000, or even use "debugfs -c -R 'dump CONFIGS/sptlrpc $TMP/sptlrpc' $(ostdevname 1)" to dump the logs and compare them (though I'm not sure they will be binary identical, or possibly have creation timestamps, so "llog_reader" would need to be used to compare only the actual records.

A more functional test would inject a CFS_FAIL_CHECK() into the mount path to drop the sptlrpc config reading from the MGS.
