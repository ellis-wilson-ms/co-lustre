LU-17995 osd-mem: in-memory OSD

Implement a simple in-memory OSD for Lustre. All objects for the
storage servers are maintained in a memory cache rather than
persistent storage. This enables Lustre servers to be run without
any backing filesystem (i.e. ldiskfs or openZFS) or physical
storage.

The main goal of this OSD is to make it easier to test Lustre on
kernels which don't readily support ldiskfs or openZFS. Many
Lustre functional tests do not rely on a specific storage backend;
they are more concerned with the interaction with different
components over the network.

It will also make it simpler to set up a new Lustre development
environment by reducing the dependency on third-party projects
to create a Lustre server.

This OSD should be considered experimental and has been
explicitly marked as such. The outstanding work is listed
in lustre/osd-mem/TODO.

You don't keep data after unmount.  It is explicitly a temporary OSD, partly for testing, but also partly for large "shared memory" usage across a cluster, or workloads where performance is more important than durability.

In the future, I'm going to decouple the dataset lifecycle from the mount lifecycle. So we'll be able to do something like:

 # Create dataset
 mkfs --osd mem mydataset

 # Mount and write a bunch of data
 mount -t lustre mydataset /mnt/mymountpnt
 umount /mnt/mymountpnt

 # Mount and see the data is still there
 mount -t lustre mydataset /mnt/mymountpnt
 umount /mnt/mymountpnt

 # Delete dataset
 tunefs --osd mem --delete mydataset

This will make it easier to restart Lustre services during sanity.sh tests. We'll also be able to modify dataset properties before the mount (to handle configuration that would normally happen via the superblock).

But in the near term, I'm hoping to land the OSD more-or-less as-is - so I can iterate with smaller, more manageable patches.

> It seems that you are using @osd_object to store file data.
> However, these objects are just a caching and managed by @lu_sites.
> And these cache may be shrank by Linux kernel shriner under memory pressure or manual shrink command.
> 
> Thus, I think we need a mechanism to pin these @osd_object to not be evicted from memory wrongly until the last unlink or destroy on the objects.

An @osd_data object has the real file data. We manage the lifecycle of @osd_data separately from @osd_object. So upper layers can freely discard @osd_object with osd_object_free(). Then create a new @osd_object later to access the object again. We'll re-associate the @osd_object with @osd_data with our object index i.e. a big hash table. The @osd_data only gets freed on an explicit osd_destroy() or on umount (currently).
