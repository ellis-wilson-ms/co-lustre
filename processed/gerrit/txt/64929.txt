LU-20048 tcu: Trash Can for encrypt files and directories

When an fscrypt directory is deleted, the encryption context on
the directory must be copied from the parent directory to the new
pFID stub directory when it is first created, so that the
encrypted filenames and contents can be accessed properly via the
".Trash" directory, as well as if an encrypted file or directory
is undeleted.

Add a test case sanityn/118v to verify that:
- Users can access files that moves into Trash Can via ".Trash"
  regardless of whether the files are encrypted or not.
- For a unlocked encrypted directory, a user can access the
  encrypted files in the Trash Can via ".Trash" directory in the
  form of ciphertext. The files in Trash Can can be restored via
  the "lfs trash unrm" command.
- For a locked encrypted directory, a user can access the
  encrypted files in the Trash Can via ".Trash" directory in the
  form of plaintext. The files in Trash Can cannot be restored
  via the "lfs trash unrm" command as the rename() operations is
  not allowed for the locked encrypted directory.
