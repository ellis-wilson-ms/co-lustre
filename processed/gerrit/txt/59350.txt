> 
> I don't understand the benefit of the npages change, or, importantly, replacing the page in the radix tree, I don't see how it could fix a problem

From Lustre's POV, there should no problem in the osc_page management in the radix tree. When a vmpage is get from VFS, cl_page_find() is called to alloc cl_page for it, if the vmapge has been initialized in Lustre, the vmpage->private would be the cl_page and it just return it. If not, it would call cl_page_alloc() to allocate a cl_page and call coo_page_init() to associate
the cl_page with the vmpage.
  cl_page_alloc
        vvp_page_init
           vmpage->private = cl_page
        osc_page_init
           radix_tree_insert(oo_tree, index, osc_page)
When the page is to be deleted
  cl_page_delete
        osc_page_delete
            radix_tree_delete(tree, index)
        vvp_page_delete
            vmpage->private = 0
and oo_tree insertion/deletion is protected by a spinlock @oo_tree_lock

While in a customer site, with certain kernel, they have some threads stuck w/o finish. There is a evidence in the log that the osc_page_delete failed to delete the item from the radix tree, and further when the page with the same index is needed, osc_page_init() keeps in a infinite loop failing to allocate cl_page for it in osc_page_init() reporting -17 (-EEXIST) from the radix_tree_insert(). While with another kernel, they reported that this issue  disappears.

Failed to find what changes in the kernel radix tree implementation, I come up with this workaround patch per se, and customer reported that w/ this patch, their system has been running for 3 days w/o re-hit this issue.
