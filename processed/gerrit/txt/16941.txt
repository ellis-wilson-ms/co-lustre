--- Commit Message ---
LU-5092 nodemap: handle config changes while mid-flight

This adds the ability to detect when the nodemap configuration has
changed between messages for very large configurations. If the
configuration is over 1MB (about 25k entries), it needs to be sent
in multiple messages. If the configuration changes in-between these
messages, then the client needs to abort the previous get config
operation and restart.

Patch Set 4: Code-Review-1

Kit, there isn't an easy way to make individual test timeouts longer.  The test timeout is already 1h, and we don't want any one test taking a huge amount of time since we will be running them on every patch.

You should change the test to run more quickly.  One improvement is to run the nodemap_add and nodemap_del loops on the MGS side, rather than on the client and ssh over to the MGS 32000 times.  Something like:

 do_facet MGS "seq -f "c%g" $large_i | xargs -n1 $LCTL nodemap_add"
 wait_nm_sync c$large_i admin_nodemap

 do_facet MGS "seq -f "c%g" $large_i | xargs -n1 $LCTL nodemap_add"

Also, for "normal" tests you can reduce large_i and only run the large number of nodemaps for "full" tests, like:

 [[ "$SLOW" == "yes" ]] && large_i=32000 || large_i=5000

or some reasonably small number that will allow the test to complete in about 5 minutes.
 wait_nm_sync c$large_i admin_nodemap
