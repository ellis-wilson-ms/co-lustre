LU-16374 enc: align Base64 encoding with RFC 4648 base64url

Lustre encryption uses a Base64 encoding to encode no-key filenames
(the filenames that are presented to userspace when a directory is
listed without its encryption key).
Make this Base64 encoding compliant with RFC 4648 base64url. And use
'+' leading character to distringuish digested names.

This is adapted from kernel commit
ba47b515f594 fscrypt: align Base64 encoding with RFC 4648 base64url

To maintain compatibility with older clients, a new llite parameter
named 'filename_enc_use_old_base64' is introduced, set to 0 by
default. When 0, Lustre uses new-fashion base64 encoding. When set to
1, Lustre uses old-style base64 encoding.

To set this parameter globally for all clients, do on the MGS:
mgs# lctl set_param -P llite.*.filename_enc_use_old_base64={0,1}

The reason to allow *decoding* names encoded in the old format would be to allow a single encoded filename to work across both old and new clients.  This might be important if some process is running on old client node and the filename is transferred to a new client node.  

It wouldn't work in the opposite direction unless we add a tunable parameter to force old name encoding until all clients are updated. My original suggestion was to just allow compatible decoding, which I think is very low overhead, but is potentially hard to detect. Having a parameter like "llite.*.fscrypt_base64=1" or "=2" would flip between v1 and v2 encoding style.

--- Message (Sebastien Buisson, 2023-01-13 15:54:48.000000000) ---
Patch Set 1:

I made some tests with old-fashion and new-fashion base64 encoding/decoding. And it seems pretty tricky to support both.

Let's consider the following input buffer: "\x01\x02".
If an old client encodes this, it gives: "BIA". Note the encoded string does not contain any of the "+,-_" chars that differ between old and new-fashion implementations.
Now, if a new client decodes this, it produces: "\x04\x80". This is not the initial buffer's content, but still a valid base64-decoded string.

The reason behind this is that not only the old and new-fashion implementations use a different encoding table, but also they divide binary data into 6-bit chunks differently.

So I do not know how the new client could "detect" it is handling an encoded name built by an old or new client.
