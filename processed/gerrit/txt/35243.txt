--- Commit Message ---
LU-12443 ptlrpc: fix reply buffers shrinking and growing

The req_capsule_shrink() doesn't update capsule itself with
new buffer lenghts after the shrinking. Usually it is not
needed because reply is packed already. But if reply buffers
are re-allocated by req_capsule_server_grow() then non-updated
lenghts from capsule are used causing bigger reply message.
That may cause client buffer re-allocation with resend.

Patch does the following:
- update capsule lenght after the shrinking
- introduce lustre_grow_msg() to grow msg field in-place
- update req_capsule_server_grow() to use generic
  lustre_grow_msg() and make it able to grow reply without
  re-allocation if reply buffer is big enough already
- update sanity test 271f to use bigger file size to exceed
  current maximum reply buffer size allocated on client.

--- Message (Sebastien Buisson, 2019-06-25 08:36:44.000000000) ---
Patch Set 1:

Indeed, the security context is stored as an extended attribute, which max length is defined as 64k in:
usr/include/linux/limits.h:15:#define XATTR_SIZE_MAX 65536 /* size of an extended attribute value (64k) */


However, Linux Netfilter defines max length of SECMARK security context to 256:
usr/include/linux/netfilter/xt_SECMARK.h:14:#define SECMARK_SECCTX_MAX  256
include/uapi/linux/netfilter/xt_SECMARK.h:14:#define SECMARK_SECCTX_MAX 256


So my perception is that we could use a 256 byte large buffer for our SECCTX in open (and other) requests. If it happens that this buffer is too small (security context as stored on the server is larger than 256 bytes), then the security context could simply not be returned along with the open (or other) reply, and the client would issue a regular getxattr request after the initial open. Our optimization would not work for very large security contexts, but it would save a lot of memory and would still be beneficial in a vast majority of cases.


Of course, when SELinux is not enforced on Lustre client side, the buffer for SECCTX is not necessary, and might be shrunk to 0 thanks to a previous patch from Mike IIRC.


Would you need me to try to make this change in request's SECCTX buffer size?
