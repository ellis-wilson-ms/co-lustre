LU-10329 osd-ldiskfs: REMOTE_PARENT_DIR scalability

Put remote objects under /REMOTE_PARENT_DIR2/<seq>/ from 2.18,
other than /REMOTE_PARENT_DIR now. This can avoid contention and
increase scalability.

Add a tunable osd-ldiskfs.*MDT*.enable_remote_obj_map for this, and it will
be enabled by default from 2.18.

Add environment variable ENABLE_REMOTE_OBJ_MAP for test scripts, when
it's set to 1, MDT will be mounted with this enabled.

Add sanity 154j.

Update sanity-lfsck 23d.

Test-Parameters: mdscount=2 mdtcount=4 env=ENABLE_REMOTE_OBJ_MAP=y testlist=sanity,sanityn,replay-single,replay-dual,sanity-lfsck,sanity-scrub,sanity-hsm
Test-Parameters: mdtcount=1 env=ENABLE_REMOTE_OBJ_MAP=y testlist=sanity,sanityn,replay-dual,sanity-lfsck,sanity-scrub,sanity-hsm

Patch Set 3:

> I'd choose the first option: do not add another level of 'dN/',
 > and put all remote objects of the same seq just under 'seq/'.

This will mean that for each remote file or directory inode, it will need both an agent inode plus a parent directory inode + 4KB block.
There is no guarantee that there will be many remote FIDs sharing the same SEQ directory, since each client creating files will use a different SEQ.  Having 2 new allocated inodes per remote inode will be bad, for example migrating inodes to a remote MDT might *increase* inode usage on the local MDT.  Also, allocating a new 4KB block for each remote inode could be the worst part on an MDT without inline directories (which we haven't yet implemented for osd-ldiskfs).  The other issue is that there may be a *lot* of SEQ directories if they only have a few FIDs in each one.

That is why I think the "aggregate multiple SEQ into one subdir" would be better.  Using a 16-bit mask could even name the aggregate directory like "0x2000004XX" by actually overwriting the last byte of the SEQ with "XX".  That would be a maximum of 32M remote objects per "XX" dir if no "dN/" is used, which _might_ be high, but is probably OK.  I suspect in almost all cases there would never be 32M remote entries in a single SEQXX directory real life.  Even 10M entries per SEQXX directory is still fine and *much* better than today with *every* remote object going into REMOTE_PARENT_DIR.  Using only a 4-bit mask like "0x20000040X" would share only 16 SEQ and at most 2M entries per SEQX directory, but this might still only be used by a small number of FIDs and is still not very efficient.

The MDS would also need to remove such directories when they are empty, so that they do not accumulate over time.  For larger SEQXX directories that would be less of a problem, for smaller SEQX or SEQ directories that could be a big problem.

> So how about skip seq level, just put 32 'dN/' directories under
 > 'MDTxxxx'? It should fit current supported filesystem size, and
 > doesn't need to consider empty seq dir reclamation.

Yes, that would probably work, unless there are crazy numbers of remote objects.  If every object was remote, then max 4B / 32 = 134M per directory, but that is *very* unlikely.  Even so, it will scale with MDT count (due to one directory per MDT) and also be 32x better than the current REMOTE_PARENT_DIR.
