--- Message (Amir Shehata, 2017-06-20 19:46:43.000000000) ---
Patch Set 2: Code-Review-1

- As I understand, debugfs isn't expected to supply a stable interface, and since we have a userspace utility (lnetctl) that's gathering this information, then it should be interfacing with a stable ABI, which makes an argument for using sysfs for statistics. (https://www.kernel.org/doc/Documentation/filesystems/debugfs.txt)

Also LNet statistics as a whole should not be optional, that includes the router statistics. These statistics are required for understanding the cluster. Putting it in debugfs makes a statement that they are not required, since debugfs is not mounted by default. I think that's the wrong strategy.

We've already started the work to move our statistics to sysfs, and I do not think it's a good idea to spread LNet statistics over the different file systems. We should be picking a specific strategy and moving forward with that.

Having statistics in sysfs I believe makes the most sense.

It also makes sense to put the effort into properly designing the interface if we're going to be living with it for the foreseeable future. I attached a document outlining the design (which we're still actively working on) to this ticket: https://jira.hpdd.intel.com/browse/LU-9680

--- Message (Olaf Weber, 2017-06-20 20:01:30.000000000) ---
Patch Set 2:

As this change demonstrates, moving stuff from procfs to debugfs is easy, so works as a quick and dirty fix to demands that procfs not be used for miscellaneous items. But it also means this is mostly a cosmetic change: if the interface design was a bad idea in procfs, then moving it to debugfs merely perpetuates the problem in a different location. And as Amir points out, if the data provided through this interface is required to be able to manage a cluster, then it shouldn't be in debugfs.

--- Message (James Simmons, 2017-06-20 20:07:44.000000000) ---
Patch Set 2:

You do understand only ONE single piece of data can be placed in an sysfs file. That data is only the raw data. What the data represents is in the sysfs file name. That is why upstream all the lnet stats were moved to debugfs. So you will not have a single stats file for sysfs.

Actually the libcfs user land library which is always links to liblnetconfig will mount debugfs if its not available. See the function cfs_get_param_paths() in libcfs/libcfs/utils/param.c. Debugfs is critical for lustre to function so it will always be available. In the future the trace point work will only make  more debugfs essential.

--- Message (Amir Shehata, 2017-06-20 21:43:31.000000000) ---
Patch Set 2:

IMHO, we're not basing our discussions on feelings. We're merely trying to determine based on the documented purpose of debugfs and sysfs on where is the best home for LNet statistics. As far as I can tell the evidence points to sysfs.

Your point regarding not breaking existing scripts is valid. So we will need to maintain backwards compatibility. However, moving forward with future development it makes logical sense to make the best design decisions possible. That's why we're proposing a design for how to maintain LNet statistics. lnetctl can then be used as the front-end for collecting, grouping and displaying these statistics, thereby hiding the details from the user. Eventually, I would say that this approach should deprecate the current model. This is design effort is particularly important in light of LU-9680.

As Olaf mentioned, this patch could be a stopgap solution until the proper design is implemented. Thoughts?

--- Message (Olaf Weber, 2017-06-20 22:00:12.000000000) ---
Patch Set 2:

Once upon a time, procfs was optional, or at least supposed to be optional. Now debugfs is supposed to be optional, and I at least expect it to follow the trajectory of procfs. In my opinion, using debugfs for non-debug items is a mistake, and merely replicates the kind of unstructured interface that people have argued against in its IOCTL and procfs forms before this. So right now debugfs is the designated garbage can interface, and as it turns out garbage can interfaces tend to be indispensable, so there the only real question is when the kernel side will start cleaning up debugfs and moving the garbage to yet another file system.

As a quick and dirty fix to "you must evacuate profs now" just moving to debugfs works, and implies only relatively small changes to code that used procfs before. That's not a bad thing, and one reason I did not -1 this review.

One question worth documenting the answer to: where do people using the upstream kernel bits expect to get their userspace from?
