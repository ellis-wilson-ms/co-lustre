LU-16314 debug: Enable optional unhashed pointers

This patch takes a page out of the kernel trace debug
playbook to rewrite format strings and change %p -> %px
on-the-fly when:

   libcfs_debug_raw_pointers

is enabled.

The module parameter can be viewed and modified by root
via lctl:
    lctl get_param debug_raw_pointers
    lctl set_param debug_raw_pointers=1

Since nothing uses the return value from libcfs_debug_msg
change it to void.

Use percpu pre-allocated buffers for holding modified
format strings to avoid kmalloc/kfree as well as avoid
bloating stack usage.

I've got my pedant's hat on now.

The "85" is the (supposed) average message size of the output message - after sprintf processing.
The length this patch considers is the input message - before sprintf processing.
So "85" isn't really relevant.
A quick analysis of all strings in .c files that aren't in "utils" directories and that follow a capital letter and open parenthesis and contain %p suggests the mean interesting input message size is about 34 and the median 33.  But those are average across the code, not the average across a runtime period so they might not be meaningful.
The max is a little under 179.  If the 812 strings I found, 790 are 70 bytes or less.

So allowing twice 85 seems excessive.   There is one message that would reject.
Allowing 85 would reject about 13 messages.  Allowing 70 would reject a little over 20, still well under 1%

Given that debug messages can be deep in the call stack, I think that we shouldn't be excessive in over-provisioning this on-stack array.
