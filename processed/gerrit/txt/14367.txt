--- Commit Message ---
LU-6431 obd: Do not block in !RUNNABLE state

__l_wait_event() should only set !RUNNABLE state for a task
around schedule functions, not around various callbacks
that could actually call into blocking codepaths and cause the
state to be lost.
This is seen from new kernel's CONFIG_DEBUG_ATOMIC_SLEEP logic.

Additionally get rid of unused __wsate variable and replace
waitq_wait/waitq_timedwait with schedule/schedule_timeout calls
there.

further about schedule returning us to RUNNING state - that seems to be true, but there are certain bits of code in the kernel that assume it might be untrue.
e.g. in the kernel/locking/mutex.c:
         * If we fell out of the spin path because of need_resched(),
         * reschedule now, before we try-lock the mutex. This avoids getting
         * scheduled out right after we obtained the mutex.
         */
        if (need_resched()) {
                /*
                 * We _should_ have TASK_RUNNING here, but just in case
                 * we do not, make it so, otherwise we might get stuck.
                 */
                __set_current_state(TASK_RUNNING);
                schedule_preempt_disabled();
        }

So do you think we should just drop the restoration of the task state altogether?
