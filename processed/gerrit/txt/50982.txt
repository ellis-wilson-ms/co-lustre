LU-13031 jobstats: store jobid in xattr when files are created

This change stores the jobid of the process that creates a file in an
extended attribute in the file's MDT inode, at file creation time.

The name of the xattr is determined by a new sysfs parameter
"mdt.*.job_xattr" so that the admin can choose a name that does
not conflict with other uses they may have for a given xattr.
The default value is "user.job". A value of "NONE" means that
the jobid will not be stored in the inode.

If the name is in the user namespace "user.", then the name portion
can be up to 7 alphanumeric characters long. The admin can choose
the trusted namespace to prevent users from modifying the value,
but only "trusted.job" is allowed in this namespace.

Allowing users to modify the contents of the xattr is helpful so
that the jobid can be preserved even when files are moved with tools
like `cp` or `rsync`, and when copied from one filesystem to another.

As we discussed at LUG, I'd like it if we could come up with a non user xattr option, since it's quite weird to change a user xattr from the kernel.  (It just seems like something that might burn us or, eg, the upstream kernel people would get mad about.)

Actually, that's another complication - What happens if we're writing a user xattr and the client isn't mounted with user_xattr?  I'm guessing the user just doesn't see the xattr but we still create it on the MDS.  I wonder how it works displaying it on the client.  Something to consider.

(3 comments)

Andreas -- I just saw your comments about using "none" over disable. I chose "disable" over "none" to turn the feature off because that's consistent with the jobid_var parameter, but I can either replace that with "none" or add "none" as another option to turn it off. I'll also address your other comments too. But here are the rest of my notes.

- I removed the llapi functions because I agree that they do not make sense when the xattr name is a parameter (unless there's an easy pre-existing way to share MDT parameters with clients, but that is probably overkill). Any function to query the jobid would need to specify the xattr name and at that point there is no difference from getxattr(2).

- I found that it appears not actually possible to store a system.job xattr (At least with the ldiskfs backend). It looks like there is no system namespace xattr handler for ldiskfs. So I only enabled "trusted.job" and "user.*" for this xattr. I briefly tested the ZFS backend and it didn't look like system.job worked there either but I didn't investigate why yet.

For mounting lustre with nouser_xattr:
- Currently, if the *client* mounts the filesystem with nouser_xattr, this xattr will still be created (assuming the job_xattr and jobid_var parameters are set). This might be undesirable, but I'm not sure if there's a good way to get the client mount parameters in this server code. I can explore that more if it's important however. (The xattrs are not visible unless the client remounts with user_xattr, in which case they can see the previously created xattrs.)

- Mounting the MDT's underlying ldiskfs filesystem with nouser_xattr appears to have no effect. It appears to be unsupported (mkfs.lustre with --mountfsoptions=nouser_xattr returns "mkfs.lustre: Error: mandatory mount option 'user_xattr' is missing, but I found that I was able to specify --mountfsoptions=user_xattr,nouser_xattr). 
- Mounting an underlying zfs filesystem with nouser_xattr also did not appear to have an effect to me. 

However, if I am missing something and this xattr should be disabled when the backend FS has nouser_xattr, please let me know. It look like there is a "mo_user_xattr" field in struct mdt_device but I couldn't figure out how to get it to be 0 (mounting the underlying MDT fs with nouser_xattr didn't appear to do so in the tests I ran, but I might be missing something).
