LU-13799 llite: Move free user pages

It is incorrect to release our reference on the user pages
before we're done with them - We need to keep it until the
i/o is complete, otherwise we access them after releasing
our reference.  This has not caused any known bugs so far,
but it's still wrong.

So only drop these references when we free the aio struct,
which is only freed once i/o is complete.

Also rename free_user_pages to release_user_pages, because
it does not free them - it just releases our reference.

This also helps performance by moving free_user_pages to
the daemon threads.  This is a 5-10% boost.

This patch reduces i/o time in ms/GiB by:
Write: 18 ms/GiB
Read: 19 ms/GiB

Totals:
Write: 180 ms/GiB
Read: 178 ms/GiB

mpirun -np 1  $IOR -w -r -t 64M -b 64G -o ./iorfile --posix.odirect

With previous patches in series:
write     5183 MiB/s
read      5201 MiB/s

Plus this patch:
write        5702 MiB/s
read         5756 MiB/s

Oops, thanks for pointing that out.  Needed an older version of the patch and forgot to update the message.  I will fix this one now and consider updating them all to µs and MiB/s when next I’m looking at the whole series.

The reason to switch to bytes/second (ignoring unit modifiers ;) ) is because it gives an absolute rather than relative measurement.  I can say "It takes x ms to move 1 GiB of data and this patch improves that by y ms.".

What I mean by absolute vs relative is:
After the first patch in the series, the time to move 1 GiB is 254 ms.  This corresponds to a performance of ~4.0 GiB/s in IOR.

At this point, if a new patch reduces the time for 1 GiB by 10 ms, 10 ms/GiB is a 4% improvement (10/254).  The resulting IOR performance is 4.23 GiB/s, or +230 MiB/s.

Now consider if, instead, a patch giving 10 ms/GiB reduction is added after the last patch in this series.  The time - after the last patch in the current series - to move 1 GiB is 118 ms (corresponding to 8.6 GiB/s in IOR).  At this point, 10 ms/GiB is an 8.5% improvement, and the resulting performance is 9.4 GiBs, or +800 MiB/s.

So if I report that a patch increases IOR performance by (for example) 5%, that is relative to the current performance, ie, the same patch won’t always give 5%.  But a give patch always gives a certain ms/GiB improvement.  (Assuming there are no more complex situations where one patch changes the behavior of another.  That’s the case here - All of these are simple additive improvements.)

So a patch which reduces time/GiB by 5 ms might not be much if we’re currently at 250 ms/GiB, but the same patch looks much more appealing if we’re at < 100 ms/GiB.

There are a few different things to like about ms/GiB, I think.  It’s not as intuitive, but it’s absolute, and I really like that it counts down to zero, rather than up towards infinity like GiB/s does.

The specific motivator is that I had to reorder these patches to resolve a bug Shilong pointed out, which screwed up the performance results in individual commits (it made them entirely nonsensical, in fact).

So, I cheated a bit and used the measured GiB/s #s to calculate ms/GiB for each patch.  I used these to calculate the IOR results from the reordered patches, since I can’t re-benchmark them (I don’t have that hardware any more) - The final IOR result from all the patches is the same as I originally measured directly.  It’s just the intermediate IOR numbers I calculated.
