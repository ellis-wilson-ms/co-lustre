--- Commit Message ---
LU-3335 scrub: OI scrub on OST

OI scrub should has the ability to handle kinds of OI, including
both the OI files on MDT and the /O directory on OST.

We trust the FID in LMA for both MDT objects and OST objects. So
if some /O sub-item does not match related LMA, then the /O will
be updated, instead of the LMA.

To guarantee that the OI scrub can run without MDT0 involved for
FLDB, the OST object needs to store some flag in its LMA to tell
OI scrub that it is for an OST object, no need to query the MDT0.

Since the LMA has never been used for object lookups, but rather only the O/0/d*/ "names" of the objects, why do you want to trust the LMA over the name of the object?  Is this only for consistency of the code compared to MDT OI scrub?

As for storing a flag on OST objects, I'm not against this, since they are treated differently internally. It would be good to ensure that this is handled in a consistent manner with the OSD API. For compatibility with existing filesystems, objects with the "fid" xattr should also be treated as OST objects, or for that matter it could be that objects on OSDs that are marked with OBD_INCOMPAT_OST should all be treated as OST objects unless there is reason to think otherwise (e.g. "link" xattr). 

I have previously discussed adding an "object type" parameter to the API during object create, but Alex has resisted this, even though it is more consistent with how objects are created in ZFS. Passing the object type at CREAT time would also avoid the requirement of a "back door" method for the OSD to access the FLDB to decide how to treat the object internally, which I think is kind of a layering violation (why should OSD internals depend on Lustre network state?).

Some main comments:
- it is going to be a common case that an OST object does not have a "fid" or "lma" xattr, since this only happens if the object is modified. This would corrupt the OST to give it an LMA FID. At a minimum, the LFSCK needs to check if the device is an OST or MDT device to decide what action to take for objects without any self-FID.
- storing both filter_fid_old and lma in the 256-byte OST inodes will cause significant performance problems. Either filter_fid_old needs to be rewritten, or no new LMA should be added to old objects. I don't think there is any backward compatibility problem with rewriting filter_fid, because this is only used by e2fsprogs, which would have had to be upgraded anyway. 
- there is _already_ a form of FLDB stored on the OST itself - this is the "O" directory - it keeps the list of SEQ numbers that are stored on the local OST, and we shouldn't need to go outside the OSD to figure this out, except in the case of a new SEQ allocation.
