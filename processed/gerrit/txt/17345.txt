LU-7470 nrs: extend TBF with NID/JobID/OPCode expression

Extend the NRS TBF policy to support complex rules with
expressions of NID/JobID/OPCode conditions.

Start the generic extended TBF policy:
    lctl set_param ost.OSS.ost_io.nrs_policies="tbf"

Following rule is valid:
    lctl set_param ost.OSS.ost_io.nrs_tbf_rule=
	"start $NAME opcode={ost_write}&jobid={dd.0},
	 nid={192.168.1.[1-128]@tcp 0@lo} rate=100"
In the rule:
"&" represents the conditional conjunction;
"," represents the conditional disjunction;

Patch Set 1: Code-Review-1

(3 comments)

I definitely like the idea of specifying more complex rules here, but I think it makes sense to take some time to decide the best way to implement this for the future. I had suggested something similar in my previous reviews of TBF, and I'd like to avoid the need for future incompatible changes that might be added if things like exclusions and wildcards are not added, or at least planned for, with this change. 

Are there any plans to add UID/GID/PROJID rules in the future?

Also, there definitely needs to be a test case for this new functionality added to sanity.sh along with the other TBF tests, so that we know it works properly now and in the future. The rest of the comments are mostly a discussion to ensure this is implemented correctly before landing, but the tests are required.

--- Message (Andreas Dilger, 2016-02-24 19:26:44.000000000) ---
Patch Set 1:

I think the other important functionality needed to make TBF usable in practice is the ability to insert rules in different orders, and not just at the end.

--- Message (Li Xi, 2016-02-25 01:34:06.000000000) ---
Patch Set 1:

That is right. And a dull but workable walk-around of this is: 1) remove all rules, 2) sort the rules in the order we want 3) insert rules back in the right order.

That is why I think another problem is more urgent: JobID based TBF policy has a problem caused by the in-flight-RPC per OSC. The jobID that has high rate limit will get similar performance like the slow jobID, because eventually all the in-flight-RPC will be slow RPCs. We are currently working on a client-side mechanism to fix or at least relieve the problem. I hope it will turn out useful...
