--- Commit Message ---
LU-1626 lov: fix lov request set finish check race

When several lov_request callbacks are called, if one of them is
the last lov_request in the set, lov_finished_set() checks for
all of them will return true, while the following action is supposed
be called only once for the set, in this case the assumption is broke
and the lov request set's refcount is wrong.

This patch fixed another glitch, in qos_remedy_create(), when we use
OST pool, the ost_idx value does not initialied correctly.

--- Message (Zhenyu Xu, 2012-08-01 02:54:33.000000000) ---
Patch Set 2:

> So why go to this complicated approach instead of adding a refcount to the set for every request in flight (and conversely substracting one for every completion)?

Even we use refcount approach, we still need make sure that the lov_finished_set() check in the cb_xxx_update() returns 1 for the last updated request, since cb_xxx_update() for each request is asynchronous callback.

That's what idempotent parameter of lov_finished_set() and lov_request_set::set_finish_checked for. When lov_finished_set() is used in cb_xxx_update(), it meant to return true for the last updated request, we use non-idempotent check. 

> You approach is also potentially racy in a way that if there are no more refocunts held outside of the one you are releasing then the race would cause the racing request to access freed set.

if cb_xxx_update() are not run for every request, the set's refcount will not be released, that's the design purpose of lov_request_set.
