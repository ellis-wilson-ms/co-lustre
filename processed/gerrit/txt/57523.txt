LU-18357 ptlrpc: Use dynamic memory allocation for filesets

Currently, a nodemap's fileset uses a static-sized array which may be
much larger than needed. This patch adds dynamic memory allocation for
a nodemap's fileset to reduce the future memory footprint when
multiple filesets per nodemap are specified.

Note, when reading fileset fragments from the nodemap IAM, the needed
size is not known yet. Therefore, each fileset is initially
preallocated with "PATH_MAX + 1" and then shrunk when the actual size
is know. This is done on the last step, when a nodemap is set active,
e.g., after reading all IAM idx pages.

The nodemap's fileset is further renamed as future "lctl
nodemap_set_fileset" commands can only modify the primary fileset.

Hi Andreas, Sébastien, and Yingjin. This patch starts off the multi-fileset patch series, based on storing the filesets in the nodemap IAM.  This patch focuses on allocating the existing fileset dynamically (which will be similar for multi-alt-filesets later).

Whenever you have some time, could you please review it?

The patch series is planned as follows:
1. Change nm_filesets field to primary fileset with properly sized allocation. (#57523)
2. Add global fileset ID to offset subid space. (#57527)
3. Add "add" and "delete" fileset operations for primary and alternate filesets. The latter is modified with the --alt flag. Alternate filesets will not have any effect yet. (in progress...)
	- get_param is modified accordingly.
	- "lctl nodemap_set_fileset" is deprecated.
4. Add mounting rules. Primary and alternate filesets will now be applied appropriately.
5. Add "lctl nodemap_fileset_modify" for an in-place fileset update.
6. Add "lctl nodemap_fileset_clear" fileset operation to wipe all filesets.
7. Add header processing for filesets in the nodemap IAM for future fileset flags.
8. Add the read-only flag for filesets.
9. Extend the "lctl nodemap_info" to show all nodemap values and not only property names.
