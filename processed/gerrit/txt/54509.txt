LU-17431 nodemap: allow handling an id map for a dynamic nm

Adding or deleting an id map for a dynamic nodemap, by sending the
appropriate ioctl to the MDS or OSS device.
On kernel side, we need to allow handling an idmap for a nodemap even
if the nodemap config file is not accessible (i.e. we are not on the
MGS). This dynamic nodemap is not stored on disk.

> My idea was to require the '-d' option as a 'be aware of what
 > you are doing' thing, when commands are launched from non-MGS
 > servers. So I guess this is not convenient by design :)
 >
 > If you think this is not worth it, and more a burden than
 > a help, I can change the current implementation.

I always find commands which report an error like "you are doing X but you need to do Y" annoying, if they could have just done "Y" directly itself (and there is no significant harm in doing so).  In this case, if the nodemap was created as dynamic in the first place, and the kernel/tools know that already, then having to specify "-d" each time is redundant.

 > Please let me know and I will modify the whole series
 > (and resubmit once the RHEL licensing issues are fixed).

I think enough of the series is ready to land that it could also be possible to just make "-d" optional for most of the commands with a separate patch later on in the series.  However, since the patches are all "trivial" then resubmitting and retesting won't be too bad either.  Up to you.
