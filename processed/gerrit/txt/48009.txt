PS: I'm not against splitting up large chunks of work into separate patches, if that makes sense, to help reviewers focus on one part of a complex change.  My main objection is the original patch having known bugs in it (which will be reviewed and maybe waste reviewer and test time), and getting significant changes later on that also invalidates the reviewer's "mental state" that was built upon the original (but actually obsolete) code.

Chris, I would imagine you have the full development history in one of your local branches that won't be going away any time soon?  Like I wrote earlier, it isn't clear to me what benefit having a history of the bugs really brings to the table here, especially since the code isn't useful until the very end of the series.

In upstream Linux development they would ask for this to be split into a series of patches that logically separate the code into stages for review.  That takes a lot of time, and isn't a clear win here since I don't know if anyone outside Cray/HPE can use this code?

In any case, all of the patches have passed build and "test" (though none of the code is actually being tested, just a validation that the existing sanity testing is not broken), so there would be a negative benefit to resubmitting the patches in a different form at this stage.  

For better or worse, there is no "fast track" to landing patches without a full build and at least basic testing.  Ideally, HPE would have a build/test system with suitable hardware+software that would grab patches from Gerrit and run local testing on them (ideally all patches, but at a minimum patches that change lnet code) so that there wouldn't be build or functionality regressions due to other changes in the tree.

Please have it reviewed ASAP so that it does not become obsoleted for some reason and needs to be resubmitted.
