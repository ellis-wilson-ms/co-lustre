LU-17629 lnet: netlink lnetctl ping doesn't resolve hostname

In my test environment, each node is assigned two IPs via
DHCP - one IP per private network (192.168.121.0/24 and
172.28.128.0/24). The /etc/hosts file is set such that
the 172.28.128.0/24 is prefered, since this is to be used
for Lustre traffic:

 -- truncated --
 172.28.128.38 mds-1
 -- truncated --

Without the patch, `lnetctl ping mds-1` returns:

 ---
 manage:
 -     ping:
       errno: -22
       descr: ! "Invalid input data or parameter"
 ...

`lnetctl` is failing to correctly resolve the hostname.
With the patch:

 ping:
 - primary nid: 172.28.128.38@tcp
   Multi-Rail: false
   peer_ni:
   - nid: 172.28.128.38@tcp

Also, add a test to sanity-lnet to ensure this doesn't break
again in the future.

> This is the wrong way to solve this. libcfs_ip_str2addr() needs to fixed to find the correct address.

The issue isn't that libcfs_ip_str2addr() is wrong. It's that this function seemingly never gets called in the current code. In fact, libcfs_str2nid() used by this patch is just libcfs_ip_str2addr() under the hood.

If you look at how yaml_lnet_ping() is called, it takes the command line arguments directly and interprets them as an array of NIDs without any parsing. So if any of the arguments are host names, they don't get resolved. If the arguments are comma-separated, they don't get split. The non-netlink version correctly handles these cases.
