LU-15748 osc: fallocate interop for 2.14 clients

fallocate() start and end are passed in o_size and o_blocks
on the wire.  Clients 2.15.0 and newer should always set
the OBD_MD_FLSIZE and OBD_MD_FLBLOCKS valid flags, but some
older client versions did not.  We permit older clients to
not set these flags, checking their version by proxy using
the missing OBD_CONNECT_TRUNCLOCK to imply 2.14.0 or older.

Arshad, your test cases talk about old client with new/old server, but in this case the server is being modified so it should be new/old client with new server?  

My intent was that old clients that did not set the OBD_MD_FLSIZE+BLOCKS flags (but otherwise sent a correct fallocate request) would be allowed to continue, while new clients that did not set those flags (for whatever reason) would get an error. That allows the addition of these sanity checks without breaking interop. 

 > Old client (pre-2.14) with Newer server !(OBD_CONNECT_OLD_FALLOC is set) =
 > Continue to process - This will fail further down

Could you explain further?

On the plus side, "broken interop" is mostly that fallocate() returns EOPNOTSUPP, which real applications should be able to handle (though of course with reduced functionality).  The problem is that fsx only checks for fallocate() functionality at startup and considers later errors fatal, which is causing 2.14.0->master interop test failures.  

It would also be good to understand why the initial fallocate(mode=0) is passing, but later calls fail.  But I don't want to fix fsx to ignore this, rather to fix the OFD code to work properly.
