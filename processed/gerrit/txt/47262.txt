LU-14541 llite: Check vmpage in releasepage

We cannot release a page if the vmpage reference count is
>1, otherwise we will detach a vmpage from Lustre when the
page is still referenced in the VM.

This creates a situation where page discard for lock
cancellation will not find the page, so we can get stale
data reads.

This re-introduces the LU-12587 issue where direct I/O on
a client falls back to buffered I/O if there are pages in
cache, since it cannot flush them.  This is annoying but
not a huge problem.

Fixes: e59f0c9a245f ("LU-12587 llite: don't check vmpage refcount in ll_releasepage()")

Reverting that patch does not affect the issue.  The test fails immediately with that patch reverted and no other fix.

The ordering requirement is:
own
delete
discard
disown
put

That patch follows that ordering.  It causes page to not be freed as quickly by delaying the final cl_page_put, but the page has been discarded, so it's OK to do not the final put.

We cannot *force* a page to be freed.  We can only discard it - that's what matters.

Patch Set 8:

So I’m taking a slightly different route here.

Changing the required reference count to > 2 resolves the data inconsistency 100%, but hurts drop_caches and sometimes causes issues with hangs with offline OSTs (That’s the FLR test issue).

So, instead, I’m going to do > 3 references (2 + 1 for caller) required for not releasing in ll_releasepage.  This makes this patch a revert of LU-12587.

So, what does that mean?  With 3 references as the limit, it is hard to hit the data inconsistency, but not impossible.  Without this change, John’s test (rw_seq_vs_drop_caches) hits the issue immediately (after 2-3 writes), with this change, we hit it after a few hundred writes.  This takes us back to the state before LU-12587.

I think this is OK because it looks like the bug was introduced in Lustre 2.0(!), and it was never hit until the LU-12587 change exposed it more widely.

I’m still working on figuring out the remaining issue, but I don’t think we should hold the release for it.  We released Lustre 2.0-2.13 with it like this.  (2.14 had LU-12587 in it, and so was worse)
