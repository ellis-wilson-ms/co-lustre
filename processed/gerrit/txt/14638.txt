--- Message (Frank Zago, 2015-05-05 15:25:53.000000000) ---
Patch Set 2: Code-Review-1

Although this patch works, it breaks the interface API as James mentioned. However I think there is a better way to skin that cat. If you use a combination of (pid, which lustre fs, lk_wfd), you can uniquely identify the connection that has to be closed. The current code in liblustreapi_hsm.c set lk_wfd to -1 before closing, so that can be interpreted as close any that matches the pid for that lustre fs. Otherwise only close the one that match. Only a minor change should be done in userspace to not set lk_wfd to -1. Unless an existing application opens twice the same filesystem, that should be compatible with existing compiled applications.

int libcfs_kkuc_group_rem(int uid, int group, int rfd, void **pdata)
{               
        ...

        if (uid == 0) {
                /* Broadcast a shutdown message */
                struct kuc_hdr lh;

                lh.kuc_magic = KUC_MAGIC;
                lh.kuc_transport = KUC_TRANSPORT_GENERIC;
                lh.kuc_msgtype = KUC_MSG_SHUTDOWN;
                lh.kuc_msglen = sizeof(lh);
                libcfs_kkuc_group_put(group, &lh);
        }

        ...
}

btw, in "libcfs_ukuc_start", the "lk_uid" is not set as the UID of the calling process, it's the PID of it.
