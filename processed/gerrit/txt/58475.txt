LU-18831 llapi: add llapi_layout_set_by_fd

In order to set specific layout attributes during restore (for example
the poolname to restore on flash pools) we need a function to set the
stripping of an open volatile because the function
llapi_hsm_action_begin creates the volatile for us. We can already
specify the flag O_LOV_DELAY_CREATE through llapi_hsm_action_begin.
This patch adds llapi_layout_set_by_fd(fd, layout) so that the
copytool can call it after having called llapi_hsm_action_begin.

> It would be useful to know where the open fd is coming from?  Is that from some volatile file creation?

The use case for this function is during restore. llapi_hsm_action_begin creates the volatile for us. We can give the open params to the function which means that we can use O_LOV_DELAY_CREATE (as the lhsmtool_posix copytool does). Our use case is to be able to restore directly on flash instead of the default pool which can be HDD. Our users often do restore + migrate instead of simply restore.

So what we would like is to be able to do this in copytool code:

layout = llapi_layout_alloc();
llapi_layout_pool_name_set(layout, "myflashpool");

llapi_hsm_action_begin(..., O_LOV_DELAY_CREATE);
llapi_layout_set_by_fd(fd_of_restore_volatile, layout);

AFAIK, the only alternative is to add a new llapi_hsm_action_begin_with_layout() function but this would modify the HSM API. This version keeps the HSM API unchanged and can benefit other use cases as well. My understanding of the lustre API is that right now, we can't set a custom layout on an already opened file (with delay create of course). We can only create a file with a custom layout.

> The new patch https://review.whamcloud.com/58514 ("LU-18842 lustreapi: create volatile file with a layout") is adding a function that creates+opens a new volatile file with the specified layout and returns the fd to the caller. I don't think this patch would solve our use case.
