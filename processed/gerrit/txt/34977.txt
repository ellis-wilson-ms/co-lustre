LU-12347 ldlm: add extra request slot for rpc

lustre’s write may want to enqueue intent lock while having osc or mdc
ldlm lock held via:

cl_io_loop
  cl_io_lock                    <- ldlm lock is taken here
  cl_io_start
    vvp_io_write_start
    ...
      __generic_file_aio_write
        file_remove_privs
          security_inode_need_killpriv
          ...
            ll_xattr_get_common
            ...
              mdc_intent_lock   <- enqueue rpc is sent here
  cl_io_unlock                  <- ldlm lock is released

That may lead to client eviction. The following scenario has been
observed during write load with DoM involved:

- clientA:write1 writes to file F and holds mdc ldlm lock (L1) and
  runs somewhere on the way to
  file_remove_privs()->ll_xattr_get_common()

- clientB:write is going to write file F and enqueues DoM lock. mds
  handles conflict and sends blocking ast to clientA

- max_rpcs_in_flight new writes to file F come up on clientA. They
  acquire request slots (ldlm_cli_enqueue()->obd_get_request_slot(),
  submit rpcs, but those are held on mds until the above conflict is
  resolved

- clientA:write1 reaches
  ll_xattr_get_common()..->ldlm_cli_enqueue()->obd_get_request_slot()
  and stucks as all max_rpcs_in_flight slots are busy and will be
  freed only when L1 is canceled. While L1 will be cancelled only when
  clientA:write1 completes with ll_xattr_get_common() and
  cl_io_unlock

- mds evicts clientA as not canceling L1

The fix is to allow ll_xattr_get_common to send intent lock rpc using
extra rpc slot in case when there are pending writes to that file.

The similar deadlock is possible with ptlrpc_get_mod_rpc_slot().

It is fixed with adding IT_GETXATTR to list of operations which do not
need mod rpc slot.

Cleanup in struct osc_lock definition: remove ols_has_ref field as it
is not used.

Tests to illustrate the issues are added.

(2 comments)

The code in ll_setattr_raw using trylock seems to be trivially wrong if there is just another thread calling setattr.  We would unlock the lock belonging to the other thread.

I'm also kind of concerned by the use of write count and NOSEC, though I haven't looked at it closely.  It seems hard to see how it's correct to do that NOSEC manipulation depending on the number of write operations, though I might be wrong.

I think we need a different approach here.

In similar cases on the OST side, we've created another portal for things like this, to give the RPC another option for sending - Is something like that possible here, to allow this operation to complete?

What if we just called file_remove_privs early on write, and just restart the op in the case where we're called in the write path?  I don't really like that because it "open code" part of the write function - file_remove_privs is supposed to be called *after* other checks...
