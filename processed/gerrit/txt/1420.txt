LU-719 ldiskfs: fix error message rate-limiting logic in kmmpd

Current logic would print an error message only once, and then
'failed_writes' would stay at 1. Rework the loop to increment
'failed_writes' and print the error message every
s_mmp_update_interval * 60 seconds, as intended according to the
comment.

--- Message (Andreas Dilger, 2011-09-24 16:53:04.000000000) ---
Patch Set 1:

As for resetting failed_writes when there is no error, I thought about this but I don't think it is worthwhile to do.  In most cases, no message is printed after the first one, so it won't show any difference for the successful writes (which is fine).  However, in the case of alternating failure + success all that would happen is that the error message would be printed a lot more often, which I don't think is useful.

--- Message (Nikitas Angelinas, 2011-09-24 17:41:30.000000000) ---
Patch Set 1:

Yes, I agree. I was thinking about the case where an error of type A occurs, it is somehow addressed by the administrator without incurring any downtime, and later at some point an error of type B occurs; by resetting failed_writes to 0 on every successful write, the administrator could be notified a little bit sooner of error type B. But this is probably not that important, and if as you suggest there are error scenarios where alternating failed and successful writes may take place, then this would defeat the purpose of having the rate-limiting loop in the first place. Any intermittent errors may not get logged, but I suspect that may be ok, and the real errors we want the administrator to take action against are serious, persistent/recurring ones, so assuming that assumption is valid, it makes sense that the logic is optimized for the most important use case.
