--- Commit Message ---
LU-5259 ptlrpc: request gets stuck in UNREGISTERING phase

Exit condition from UNREGISTERING phase is releasing of
both reply and bulk buffers.

Call ptlrpc_unregister_bulk() if ptlrpc_unregister_reply()
wasn't completed in async mode before switching to
UNREGISTERING phase.

--- Message (Chris Horn, 2014-06-26 18:01:55.000000000) ---
Patch Set 1:

The bug fix looks correct to me, but I think we should do more to reflect the semantic change introduced by LU-5259

ptlrpc_client_recv_or_unlink() used to indicate whether the reply buffer has been unlinked, but with LU-5259 this is no longer completely true. If we get 0 back we know the reply buffer has been unlinked, but if we get 1 back the reply buffer might still be unlinked (or it might not). It seems to me that ptlrpc_unregister_reply() only needs to know whether the reply has been unlinked, but ptlrpc_check_set() needs to know that LNET has dropped its reference on both the reply and outgoing buffer.

I think we should make this distinction clear in the code. At the very least we should edit/add comments to make this clear, or, ideally, we should change the code.

We could remove rq_req_unlink from ptlrpc_client_recv_or_unlink and explicitly check it in ptlrpc_check_set where appropriate, so ptlrpc_client_recv_or_unlink goes back to its original meaning.

We could add an additional inline function that checks all three variables and use that in ptlrpc_check_set() instead of ptlrpc_client_recv_or_unlink (ptlrpc_client_all_unlink or something?).
