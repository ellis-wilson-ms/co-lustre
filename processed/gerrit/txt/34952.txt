--- Commit Message ---
LU-12328 flr: avoid reading unhealthy mirror

* Fix an error in lov_io_mirror_init() which would wait unnecessarily
  if we're retrying the last mirror of the file.

* In osc_io_iter_init() we'd check its OSC import status so that the
  read path can quickly switch another mirror.
  sanity-flr test_33b is added to test this case.

* And with all mirrors have been tried, we'd turn off the quick switch
  so that when all mirrors contain bad OSTs, the read will still try
  its best to get partial data from a component before trying another
  mirror.
  sanity-flr test_33c is added to test this case.

Patch Set 1: Code-Review-1

(2 comments)

This needs a test case to check whether reading the (uncached) file is generating a lot of RPC timeouts, or only a single one for the file.

I also don't like the approach of marking the whole mirror offline if only one OST is inactive.  This makes it possible/likely that the whole file is inaccessible if there is one OST (or multiple OSTs on the same OSS) used for different components of the same mirror.  Keeping a separate copy of the OST accessibility state also means that the copy will be stale for possibly a long time, or will need extra effort to activate again.

One benefit of having a single "avoid" state is that if a second OST becomes unavailable we should round-robin across the mirrors (sticking with one mirror until it has a failure, mark it "avoid", then trying the next mirror until we've tried all but "avoided" mirrors, and finally retrying even the "avoided" mirrors), so we may be able to make forward read progress if there is still one good copy of the data somewhere.

Based on Jinshan's comments to the list, it might be useful to investigate whether there is a bug at the OSC layer that we aren't marking the OSC disconnected, or not immediately failing RPCs to an inactive OSC for FLR reads (i.e. "rq_nodelay/rq_noresend") and only sending a waiting RPC after having tried all of the mirrors (so we don't spin on the client trying all the copies in a loop).

Patch Set 2: Code-Review-1

(2 comments)

This needs a test case to make sure that it is working properly.  Something like creating a large file with two or more mirrors (e.g. write for 60s, then resync mirrors), flush the client cache and measure the time to read the file, then mark one of the mirrors offline and wait until the client marks the OST inactive, flush the cache and re-measure the time taken to read the file (should not be longer than the original read by more than 10%), and repeat the test the same for the other mirrors (just to ensure that the read wasn't lucky and always trying the working mirror).  This test should fail on the current code, based on the comments in LU-12328.

My preference would be to fix the underlying problem (that IDLE imports ignore rq_nodelay) somehow, and then have the CLIO layer retry the IO quickly if the lower layer returns an error.  That avoids the problem of CLIO skipping a whole mirror just because one OST is offline, and only skips an OST if it is actually offline.

In theory it should be possible to read all data in a case where the data is alternating between two OSTs for each of the two mirrors, but each OST has a full copy of the data in two different objects:

  lfs setstripe -N -E -1 -c 2 -o 0,1 -N -E -1 -c 2 -o 1,0 $DIR/$tfile

This should allow reading the data if either OST0000 or OST0001 is inactive.  It wouldn't be possible to write to such a file if either of those OSTs is offline, but that isn't what is being tested here.

This seems to be replicating the whole LOV offset calculations, which are not simple. 

Why not take Jinshan's suggestion and just let the IO be submitted, and return immediately if the OSC device is inactive?  That avoids doubling the LOV offset code complexity, and avoids the case of skipping all mirrors if most of the IO submissions could complete. 

The fix would be down at the OSC/ptlrpc layer to handle IDLE devices better, rather than at the LOV layer.
