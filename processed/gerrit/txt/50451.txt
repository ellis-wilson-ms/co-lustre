--- Commit Message ---
LU-16671 osc: fix unstable pages for short IO

Unstable pages was written with theoretical support for
short IO (ie, no bulk, data-in-rpc, LU-1757), but since the
short IO code wasn't merged until years later, they were
probably never tested together.  And when you do, it
crashes.

In truth, short IO has no separate pages to be tracked,
which is why this is crashing.  This means that small write
RPCs won't be tracked in unstable pages, but that's a very
minor limitation and unlikely to cause trouble.  (and since
RPC allocations are not 'pages', they're just malloc'ed,
there's no good way to track them anyway)

Fixes: 70f092a ("LU-1757 brw: add short io osc/ost transfer.")

--- Message (Andreas Dilger, 2023-03-29 00:14:18.000000000) ---
Patch Set 1:

> But that means we can have stale uncommited pages, and so on a
 > replay we could push them to the server, potentially overwriting
 > a later write...  Correct?

The client's pages don't have to be *clean* at the time of the blocking callback.  However, they *do* have to be at least sent to the OST by the time the client sends the lock cancel.  The "obdfilter.*.sync_lock_cancel" parameter decides whether the OST should do a sync before the DLM lock is granted to another client.  If sync_on_cancel=enabled/blocking then the reply to the client's cancel *should* contain an updated last_committed, and the client could immediately drop the RPCs pinning the pages at that point.

Unfortunately, obdfilter.*.sync_lock_cancel=never is the default.  It isn't clear if that can lead to an actual recovery error *if all clients participate in recovery*.  The former client's write RPCs sent before the cancel will have a lower transno and be replayed in order before any write RPCs from the new client holding that extent lock.

Yes, the old RPCs might overwrite newer data on disk *temporarily* from stale pages, but the second client's RPC replay would overwrite with newer data again.  Note that the "new data on disk" would be (mostly) inaccessible since the block pointers would not even be in the inode, because (for extending writes at least) they would have been lost when the OSS crashed.  If, at any time, a journal commit happened on the OST to save the block pointers, then last_committed would also be atomically increased, and both client's write RPCs would be dropped before overwrite due to having a lower transno, and all is well.

The case for overwrites in the middle of a file for previously-allocated blocks is more precarious, in that if the new client does not participate in recovery (crash/eviction), then the old client may overwrite new data with stale data from cache.  However, this is a bit of Schrödinger's writes, since no fsync must have happened from the new client before crash/eviction (or last_committed would have increased), so there is no way to really know externally if those writes made it to the OST and were overwritten, or they never made it at all.

 > What about dlmlock replay, etc?  Does that save us?  The node
 > which could try to replay the write doesn't have the dlmlock
 > any more, would that cause the write to be discarded?

No, the RPCs will be replayed in transno order for that target, regardless of DLM lock state, but per my comments above I think this is handled properly.

 > Rolling this back:
 > I still feel like either we don't have this gap, or it's somehow
 > very small.  This feels like a gap we could drive a truck through
 > if it's really there.

My main concern was when you mentioned that short write pages are not pinned, but I now *think* that is OK and was just a misunderstanding on my part:
- for short DIO writes, the short write data is copied into the RPC buffer and is available for replay until OST commit, *regardless of whether the write is sync on the OST or not*.  That is good because it means the ior-hard-write optimization to make short writes buffered on the server will not break anything.
- for short buffered writes, the data is copied into the RPC buffer *at the time of write* and is usable for replay until commit, but it should *also* be copied into the page cache for reads.  I haven't confirmed this (I think that is more in your area), but it makes sense otherwise it isn't really a "buffered" write at all.

I thought "unpinned short BIO pages" == "no other copy of the data on the client", but I don't think that is correct.

--- Message (Patrick Farrell, 2023-03-30 16:23:17.000000000) ---
Patch Set 1:

Thanks for the clear walkthrough on the consistency stuff - I believe I follow.

-------
"for short buffered writes, the data is copied into the RPC buffer *at the time of write* and is usable for replay until commit, but it should *also* be copied into the page cache for reads.  I haven't confirmed this (I think that is more in your area), but it makes sense otherwise it isn't really a "buffered" write at all."

Yes, it's actually more like this - for buffered IO, everything goes in the page cache *first*.

For writes, the data is copied to the page cache, then we make RPCs from it.

For short IOs, the data is copied again - in to the RPC body - but for regular bulk IOs, we do RDMA from the page cache pages, which are already in the cache at that time.  The page cache pages are available for *read* on the client after the data from userspace has been copied in to them (making them dirty).  This is separate from any RPC formation and sending to the server we might do.

Reads proceed identically, in that page cache pages are inserted and allocated, then we build the RPCs.  It's just that those pages are locked and are not marked uptodate until the RPCs occur and the data is copied to those pages.

That explanation feels a little muddy to me; hopefully it's clear.  Here's another pass at it, without only limited simplifying omissions...

The first step for buffered IO is always inserting pages in to the page cache.  For writes, the pages are locked when we start, but they're unlocked once we've dirtied them with the user data, so they can be read.  RPC formation happens at some other time - maybe immediately, maybe a bit later, but it's not really connected.  For reads, the pages are also locked on creation, but they're not unlocked until we've actually done the reads from the server and the data is in the pages.  (I'm sure this explanation raises questions, but hopefully it answers some too. ;))
