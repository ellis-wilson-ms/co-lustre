LU-13745 llite: avoid lock ldlm twice in splice read

With no .copy_file_range vfs API implemented in Lustre, glibc func
copy_file_range calls do_splice_direct()->splice_direct_to_actor()
->do_splice_to()->ll_file_splice_read().

While the call chain of ll_file_splice_read()->ll_file_io_generic()
->generic_file_splice_read()->ll_file_read_iter()
->ll_file_io_generic()

And that would try to get ldlm lock twice in ll_file_io_generic(), so
that hang ensued.

Fixes: ccac422cf9dd ("LU-11621 utils: optimize lhsmtool_posix with copy_file_range()")

Patch Set 2: Code-Review-1

There are some problems with this patch:

1) with lower version of kernel, eg rhel7 3.10 the call trace is:

|->ll_file_splice_read()
  |->ll_file_io_generic()
   |->generic_file_splice_read()
      |->ll_readpage
           ....

With this trace, I think we should get LDLM lock, because there is no twice lock holding here.

2) And higher version of kernel, I think it probably not LDLM lock twice, but this looks like of deadlock, we should rework to avoid the dance like  ll_file_io_generic is called several times.

3)It might be better to at least to include original customers reproduced tests to Lustre, as it seems there are few tests in Lustre to verify splice.
