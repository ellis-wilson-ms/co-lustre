LU-18357 ptlrpc: Add in-place modification of filesets

To modify a nodemap's fileset, i.e., its subdirectory, type, or
access permissions, the fileset needs to be removed and re-added.

This patch adds the "lctl nodemap_fileset_modify" command to modify a
fileset in-place. For example, the command

lctl nodemap_fileset_modify --name nm1 --fileset "/remote" --rename
"/othersite" --ro --alt

renames the existing read-write primary fileset "/remote" to
"/othersite", making it alternate and read-only.

A new sanity-sec test 27f is added test this command.

Thanks Marc, looks good overall. Just a few questions/comments I have.

First, just to mention that for the upcoming patch that will provide support for dynamic nodemap, the use case presented in the commit message:

lctl nodemap_fileset_modify --name nm1 --fileset "/remote" --rename
"/othersite" --ro --alt

would return an error for dynamic nodemap if the parent has a primary fileset. Indeed, that would leave the dynamic nodemap with no primary fileset, escaping the parent's namespace.

Actually, I am even wondering. In the general case, do we really want to allow converting a primary fileset into an alternate fileset? That leads to "transparently" removing the primary fileset, which could be dangerous?
