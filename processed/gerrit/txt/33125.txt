--- Message (Oleg Drokin, 2018-10-10 01:43:00.000000000) ---
Patch Set 4:

I think this needs some better explanations.

1. If we have ELC avoiding convertable locks, why not jsut issue a downgrade request right from the ELC? That would save us an RPC roundtrip.

2. skipping ELC for a DOM lock sounds like a not so smart idea since the server absolutely WILL turn around and will ask for the same lock to be cancelled (and a corresponding data flush), only an RPC-roundtrip later. So what's the gain here, again? Why not initiate a data flush preemptively if we know we'll need to do it anyway?

--- Message (Mikhail Pershin, 2018-10-10 07:27:22.000000000) ---
Patch Set 4:

About 1) - that is exactly what I am planning to do - start convert(downgrade) from ELC but that is not trivial task. For now this check just helps to avoid ELC vs converting issues. You are right about it can be non-optimal but consider that converting works for DOM files only so far.

2) skipping locks with DOM bits has own reasons - ELC cancels own bits, UPDATE most likely, and that doesn't mean data flush but it will do flush upon canceling whole lock for no reasons. So with DOM bit it is better to allow server do convert and extra RPC round trip just to prevent data flush on every metadata update.

The 2) is closely related to 1) in terms of making ELC able to convert locks instead of pure cancel and I am working on that task as final solution, for now consider this patch as quick fix for known problems.
