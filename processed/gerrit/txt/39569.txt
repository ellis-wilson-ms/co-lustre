--- Commit Message ---
LU-13742 llite: do not bypass selinux xattr handling

Without the hint from selinux_is_enabled() to determine if selinux
is running at boot the performance fix from LU-549 to skip handling
of selinux xattrs cannot be correctly handled.

The correct path is to act is if selinux is enabled.

This fixes a bug introduced by LU-12355 that now exists in
RHEL 8.2 kernels where clients have enabled selinux.

--- Message (Andreas Dilger, 2020-08-06 22:38:57.000000000) ---
Patch Set 2:

The main potential issue with this patch is the huge number of getxattrs that will be done.  AFAIK, the "security" xattrs are not currently put into the xattr cache on the client, so each file access might result in an RPC, which would be brutal.

It would be worthwhile to check the mdc.*.stats after this patch is applied and e.g. "rsync -av /etc /mnt/lustre" and do some things in the client filesystem, then check the number of mdc_getxattr vs. ldlm_ibits_enqueue (open) RPCs.  I currently have mdc_getxattr about 1/7 of ldlm_ibits_enqueue on my RHEL7 test system.

--- Message (Sebastien Buisson, 2020-08-07 07:10:52.000000000) ---
Patch Set 2:

Note that this patch does not change client's behavior when SELinux is enabled. It only impacts behavior when SELinux is disabled, for clients that would still try to get or set security.selinux xattr.

So basically, this would happen for a client on which SELinux has been disabled, and that is manipulating files that were created on nodes where SELinux is enforced.
