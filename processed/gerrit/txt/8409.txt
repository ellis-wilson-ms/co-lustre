--- Commit Message ---
LU-4223 utils: fixing loop leaking in utils

1. If the file is being opened by popen, it should use
pclose instead of fclose to close the file, to make sure
the process created by popen is closed after pclose, then
to avoid loop device is being hold on release.

2. Give another try in loop_cleanup in case there are still
some process going on with the loop.

3. wait loop device to release before continue conf-sanity
32c.

4. Add losetup -a to list loop dev information when the
test(conf-sanity 32) fails.

--- Message (John Hammond, 2013-11-27 22:14:47.000000000) ---
Patch Set 3:

The right thing here may be to use LO_FLAGS_AUTOCLEAR (see http://lxr.free-electrons.com/source/drivers/block/loop.c#L1011) but that will require using ioctl() rather than system("losetup -d ..."). 

Moreover, in newer kernels calling the detach ioctl on a loop device will with reference count greater than 1 will set the autoclear flag and return 0. This may confuse a second 'losetup -d' invocation which finds the device already detached. This is all just terrible.
