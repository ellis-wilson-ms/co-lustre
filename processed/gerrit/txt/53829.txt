LU-17471 osd: add symlink for brw_stats

Add symlink at /proc/fs/lustre/osd-*/*/brw_stats to
/sys/kernel/debug/lustre/osd-*/*/brw_stats to fix
the compatible issue of the previous utils that are
still using the old proc entry.

> Seems reasonable. One question: why the code duplication? Is there
 > no where to put a common osd code? I've noticed other code
 > duplication between ldiskfs and zfs.

It has been discussed a couple of times, but nobody has ever done the work to separate it.  Definitely there is quite a bit of duplication and it could be consolidated into e.g. osd-lib or something and then it is statically linked into both OSDs.  I don't think (m)any sites ever run *both* osd-ldiskfs and osd-zfs at the same time, so having yet another module/library isn't going to be useful in any practical way, but sharing the code could definitely be helpful to reduce bug fixing.

Maybe one day we'll get another OSD where it will be useful.  At one time osd-btrfs was a possibility, but it never got beyond the toy stage, and btrfs itself isn't really a scalable/performant filesystem.  

I think very reasonable targets would be osd-ramfs (totally stripped down tmpfs-backed OSD with no recovery at all, instead of ldiskfs-onloop-on-tmpfs, for aggregating GB of RAM from servers), and/or osd-nova/osd-pmem (OSD on NVRAM for MDTs, if that ever becomes a thing again).

> Sadly this can never go upstream. It's considered a security risk.

You are misunderstanding the patch. This is only creating a symlink, so it isn't changing the access permissions on /sys/kernel/debug, so CANNOT possibly create any kind of security risk.  The symlink still needs to be followed by root, as the actual brw_stats file is still being accessed from /sys/kernel/debug. The only reason for the symlink is to allow applications that hard code pathnames to use /proc/fs/lustre. 

 > To make it less of an issue can you create a symlink into
 > /sys/fs/lustre instead since one day /proc/fs/lustre will go away.
 > I think that is reasonable.

The patch is creating a symlink to where the brw_stats file exists. I don't see how creating a symlink to a directory tree that doesn't contain brw_stats could possibly help? And surely you are not suggesting that brw_stats moves into /sys/fs/lustre, as that would make the kernel folks heads explode (and also not work because it can be larger than the 4KB maximum).

Whether /proc/fs/lustre exists when the patches are submitted upstream is a totally separate issue. It exists today, monitoring tools that depend on it exist today, and until they are updated to use /sys/kernel/debug or some other API then the stats move has broken those tools and fixing them is the short-term priority.
