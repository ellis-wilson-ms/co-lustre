--- Commit Message ---
LU-15808 ptlrpc: wait condition should not sleep

ptlrpc_set_wait using a potentially long running condition
ptlrpc_check_set() that can also block.

If it does block during ptl_send_rpc() it could potentially
trigger a warning:
   do not call blocking ops when !TASK_RUNNING

Instead provide a no sleep variant of ptlrpc_check_set that
adds the problematic requests to a list for the caller to
dispatch.

Defer calling ptlrpc_at_recv_early_reply()
In case of req needing early reply add it recv list for the
caller to process.

Defer calling ptlrpc_send_new_req() to avoid kvalloc() while
sleeping in wait() loop.

I wonder if there might be a much simpler way to resolve this.
The problem is the wait_event macros in ptlrpc_set_wait() which call ptlrpc_check_set() in  context where it is not allowed to wait.  But sometimes it does wait and Oleg suggests that even after this patch it might still wait.

It might be best to change these wait_events to use wait_worken() much like ptlrpcd() does.
Then any waiting in ptlrpc_check_set() would be perfectly acceptable, and we would still try it again whenever a wakeup arrived.
