Hi Timothy,
The right way to handle this is to "Associate" those LU numbers with the failed *subtest* and/or "Accept" or "Reject" the correct/incorrect auto-associated failures in Maloo directly, and then comment on the test session that actually failed instead of the one that passed. 

That allows me to use the failed test URL directly from my email to the failed session to restart it, rather than having to go via Gerrit to find which sessions failed.  

We have relatively fewer customers using ZFS at the moment, so understandably there is less time for us to work on test failures related to ZFS only.  If you want to dig into some of the issues related to ZFS you should consider to run some patches that add debugging to the test scripts and/or code, and add one or more lines of:

 Test-Parameters: fortestonly fstype=zfs testlist=sanity-quota

(each online runs as a separate session), possibly adding "env=ONLY=N,ONLY_REPEAT=M" if it can be reproduced by running only that subtest. 

I think some of the ZFS tests are just very slow due to IO contention in the VMs, and Alex was trying to address this with a few patches, but could use some help to review and test them:
- https://review.whamcloud.com/49061
- https://review.whamcloud.com/47768
