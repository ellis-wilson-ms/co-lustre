LU-17902 nrs: add NRS TBF policy for nodemap

For isolation of workloads across multiple sub-tenants of a FS,
it would be useful to allow registering a NRS TBF rule for a
nodemap, so it is easy to set resource constraints directly per
tenant instead of having to specify NIDs or UIDs (which are only
indirectly related to the nodemap/tenant and may become stale).

And add a test case sanityn/test_77ke to verify it works as expected.

Patch Set 5:

"where in the code the nodemap id is reset in the TBF rule when a nodemap is recreated with the same name?"
We do not do nodemap id reset in the TBF rule. It is the user's duty to fix it.
i.e. start nodemap_rule nodemap={mynodemap} rate=100
when A user call "start nodemap_rule nodemap={mymodemap} rate=100", it will report rule nodemap_rule EEXIST.
then the user will remove or change the rule with name "nodemap_rule".

> Yes, I was thinking about that as well.  The nodemap ID would not change very often, so it
> would be possible to do something to reset the ID number in the TBF rules in the very rare
> case that the nodemap was removed and re-added.
I thought the idea would be to make that id reset automatic, no?

In your example, unless admins use a nodemap name that does not exist, the command would always work. The actual problem we want to address is when a nodemap with a given name is removed and then recreated with the same name, *while* a TBF rule associated with this nodemap is in use. As the nodemap id is going to change in this process, the TBF rule would silently stop working.
