LU-16789 utils: use AIO for 'lfs migrate'

Create a mini async IO framework with ll_aio_ prefix. The framework
is a wrapper layer of libaio to run asynchronous IO. However, it
can also run traditional synchronous IO with POSIX APIs without
any code change if io queue is initialized with sync_mode=true.
Or if libaio is not availble on the running system, the framework
will fallback to synchronous mode automatically.

Create function ll_aio_copy() to copy data from one fd to another.
The function can work in async/sync mode, with different buffer size
and different number of async IO tasks. The function also integrates
checking file, stats report and IO throttling.

Due to LU-18032, the last chunk of file may not be aligned so cannot
be read/written by libaio. So the last partial chunk will be
read/written in sync mode with POSIX API in sync mode even if
ll_aio_copy() is called in async mode.

In async mode, 2 more fields are appended to stats report:
read_tasks and write_tasks, which indicate how many tasks are pending
on reading/writing data when the stats is printed.

Replace migrate_copy_data() with an AIO version of
migrate_copy_data_aio() based on ll_aio_copy().
Command 'lfs migrate' and 'lfs mirror extend' will benefit from it.

Add --aio-tasks=TASKS parameter for 'lfs migrate' and
'lfs mirror extend' command. It tells how many AIO tasks will be
started to transfer data.

But if --aio-tasks=0, the IO will fallback to synchronous mode. All
the read/write will be performed in synchronous mode sequencially.

Add --chunk-size=SIZE_MB parameter to specify the buffer chunk size
in MB used to transfer data.

I don't think it is bad if "lfs migrate" and "lfs mirror extend" can use the new AIO engine, even if "lfs mirror resync" cannot.

The issue with resync is that it is a "read once, write many" function, so it cannot just work with a single write FD.

It might be possible to restructure this code to allow the migrate/extend code to use it with a single write FD, and pass a flag when it is resync to switch between the mirror IDs?  Note that writes to the mirror ID on a single file is using the *same* FD, but calls an ioctl to switch between the mirrors.

In the resync case, it might make sense to have the AIO submission contain which mirror ID it is being written to (*IF* that is possible), and then the IO completion for the first mirror resync write will be re-used to do the second mirror ID resync write, ..., until all stale mirror copies are written?  In the migrate/extend case there would only be one write and the AIO submission would immediately finish.

The question is whether this can be handled with the iocb(x) itself, or does it need to change the file descriptor each time?  If it needs to modify the file descriptor, then this is racy with multiple AIO writes in flight...

There are 2 major differences:
- 'lfs mirror resycn' need to copy data for every component. 'lfs migrate' and 'lfs mirror extend' simply copy the whole file.
- 'lfs mirror resync' using single fd and switch between the "default mirror" and "target mirror". 'lfs migrate' and 'lfs mirror extend' provides 2 fds and copy from one fd to the other.

So my plan is to create a function to copy a segment of file in parallel. 'lfs migrate' calls it from 0 to the end of file. 'lfs mirror resync' calls it for every component. 

Here the assumption is that the active mirror is chosen per fd, not per file. So I can dup the src fd for every component and set active mirror for the new fd.

Please check the kernel implementation of the "lfs mirror write" and "... resync" code to see how they are handling mirror selection. I don't think it is possible to submit read or write to different mirror copies at the same time using different FDs, but I could be wrong.  Possibly this is something that could be fixed/enhanced in the llite implementation?

Otherwise, the AIO code would need to do a chunk of reads from a valid mirror first, then switch to a different mirror and write and sync all of the pages, then switch to the next mirror, ... 

It would of course be better if all of these read and write operations could overlap.  If the mirror ID is not tracked per FD, and is instead tracked on the whole inode, then this also opens race conditions if another process tries to read or write a file during resync. It may be that this is causing it to access an unexpected mirror, but I don't recall the details of the implementation and it is too late for me to dig into it now.
