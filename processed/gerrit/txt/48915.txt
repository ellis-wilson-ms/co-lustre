LU-16251 obdclass: fill jobid in a safe way

jobid_interpret_string() does not fill jobid in an atomic way.
So in lustre_get_jobid() give it a buffer first, then copy the
buffer to jobid as a whole.

It looks like there may also exists a chance of racing jobid updates for lli_jobid in ll_readahead_handle_work().  This might be a good place to add a jobid_name_is_valid() check to avoid clobbering lli_jobid with the name of a worker thread.

Were you able to reproduce this problem in your testing, or is this a best guess at what the problem might be?  If you have a reproducer, that should be added as a test case for this patch.

If you don't have a reproducer already, you could try a test with two threads, each with a different jobid (also different lengths), both do sync and/or buffered reads and writes to the same file in parallel, and then check the resulting jobstats to ensure that only the two jobids are reported, and not some mix of the two jobids.

I _suspect_ that even racing memcpy() would be OK, since both threads would do start-to-end byte/word copies in the same order and one thread would finish before the other, so I think  this is probably OK.

I *do* think that excluding the "kworker" threads from the list of valid jobid names would probably also reduce/avoid this problem.  Having a test case would of course give us a much better idea of where the bad name is being introduced.
