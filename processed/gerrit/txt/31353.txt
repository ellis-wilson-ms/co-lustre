--- Commit Message ---
LU-10681: Disable tiny writes for append

Unfortunately, tiny writes do not work correctly with
appending to files.  When appending to a file, we must take
DLM locks to EOF on all stripes, in order to protect file
size so we can append correctly.

If we dirty a page with a normal write then append to it
with a tiny write, these DLM locks are not present, and we
can use an incorrect size if another client writes to a
different stripe, increasing the size without cancelling
the lock which is protecting our dirty page.

We could theoretically check to make sure the required DLM
locks are held, but this would be time consuming.

The simplest solution is to just not allow tiny writes when
appending.

Also add option to disable tiny writes at runtime.

--- Message (Patrick Farrell, 2018-02-28 14:44:02.000000000) ---
Patch Set 5:

Oleg,

Still planning to make/explore other changes, but was thinking about this for appends last night.

The basic idea of tiny writes is that it finds an already dirty page and knows that the work of creating + tracking it is done, so it can use that page without doing that work.

If tiny writes does not find a page where the write is going, it cannot handle the write and must defer to the normal write path.

For appends, we must know the correct file size to have the offset to write at.  If there are other non-append operations racing with us, it's enough to have *a* correct file size from some time during our write.  But as you pointed out, glimpse is not enough to guarantee that (we could complete a glimpse to one stripe and before we do glimpse to a different stripe an i/o could happen to the stripe we already glimpsed and then to the stripe we're going to glimpse, all before we glimpse that second stripe - So we are not guaranteed to have a file size that was *ever* correct.).

So, we must do LDLM locking to get the correct file size, both so we can do the append write correctly and so we can check for a dirty page.

We also have to take care w/the layout (check out my patch for LU-9341), meaning we need to check the layout, lock it, get file size, then check the layout and verify it hasn't changed (and restart if not).  If it hasn't changed, then we know the file size we got is correct.

So now, after all this work, we can check that there is a dirty page to append to.  And if there is not, then we have to start a normal I/O.

I think this would still offer performance improvements for a logging a few bytes at a time case, but it might hurt performance in other cases, and it's comparatively complicated.
