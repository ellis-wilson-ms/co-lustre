--- Commit Message ---
LU-18885 mgs: use Xarray to store NID lists

Re-organize in-memory NID lists storing at per-network
basis with network id as Xarray index. NIDs are stored
in string form, suitable for IPv6 and hostname formats.

NID filtering by network is done at list update time which
is infrequent event but a possible fetching by network
becomes simple and concurrent for clients reads

Patch implements:
- xarray nidlists management on MGS
- incoming nidlist filtering and storing by networks
- NID lists fetching for IR with framework for network
  filtering, but without real filtering, all target NIDs
  are provided to client (out of scope of this ticket)

> Hi Mike, you mentioned elsewhere that you are investigating an
 > issue with Xarray and an older kernel...
 > What is the current status on this?

Marc, this patch has no such issues. But in general, issue was seen in patch https://review.whamcloud.com/fs/lustre-release/60019/ and the difference was the using full netid as index for Xarray. Full netid consists of (nettype << 16) and netnum, so for tcpN it starts with 131072 instead of 0 like in current code (I am using just netnum).
I think that is the main reason of Xarray issue.
Next, there is no issue with in-kernel Xarray, but only with Lustre compat Xarray code which is used for 3.10 kernel, so can be result of changes made to that code.
Third, issue was resolved by explicit call to xa_erase() before OBD_FREE() in mgs_nidtbl_fini_fs(). Initially I didn't do that thinking that following xa_destroy() will cleanup everything, but somehow skipping xa_erase() cause mess in Xarray.
So this particular patch had no such problems, but I will update it too just to be sure
