LU-10948 llite: Introduce inode open heat counter

Initial framework to support detection of naive apps that
assume open-closes are "free" and proceed to open/close
same files between minute operations.

We will track number of file opens per inode and last time inode
was closed.

Initially we'll expose these controls:
llite/opencache_threshold_count - enables functionality and controls after how
                                  many opens open lock is requested
llite/opencache_threshold_ms    - if any reopen happens within this time (in
                                  ms), the open would trigger open lock request
llite/opencache_max_ms          - If last close was longer than this many ms
                                  ago - start counting opens from zero again

Once enough useful data is collected we can look into adding a heatmap
or another similar mechanism to better manage it and enable it
by default with sensible settings.

I'm not so worried about test breakage just because of different behavior than when the test was first written, if it means that we are getting improved functionality.  Being able to cache the open on a client reduces the RPC count to the MDS for workloads that need it.  It may (partly) be that tests are failing because of the strange exponential growth of the heat?  3 opens would result in a heat of (1+2+3=6).

For LSOM, open cache doesn't really "break" LSOM, just makes the size more lazy to be stored on the MDT.  That isn't really an "error" because the file is still being modified by the client during this time, and the LSOM should be updated eventually on the real close.  

Similarly, for Changelogs, there will be at least one open record written, and even without opencache there is not strict accounting of opens in the Changelog.

I'm not sure how opencache interferes with HSM?

I think the affected tests could be fixed by adjusting the heat threshold for the tests where it matters.
