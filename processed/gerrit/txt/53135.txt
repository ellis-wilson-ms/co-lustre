LU-13307 nodemap: have nodemap_add_member support large NIDs

Currently when mounting lustre using IPv6 address fails with

Lustre: 27361:0:(nodemap_handler.c:395:nodemap_add_member())
  lustre-MDT0000: error adding to nodemap, no valid NIDs found
LustreError: 11-0: lustre-MDT0000-osp-MDT0003:
  operation mds_connect to node 0@lo failed: rc = -22

This was due to no nodemap being set so the ptlrpc layer was not
seeing any new peers. Adding minimal support to nodemap allows
mounting.

Indeed, in this case I was about to propose the following: try to make a patch that bypasses nodemap if it detects an IPv6 address. With a clear warning in the logs/console/etc. I think it is safer than implicitly considering all IPv6 NIDs as part of the 'default' nodemap. The main reason is that usually, and even if it is not a good practice, people grant all admin rights to the 'default' nodemap, and only make dedicated nodemaps for the clients, with specific access rights. The good practice is to create a dedicated nodemap for servers as well, with all admin rights, and grant as few privileges as possible to the 'default' nodemap.

Yes, a one-shot message like you suggest is fine with me.

It is true that if IPv6 NIDs cannot be added to any nodemap (because IPv6 is not supported in nodemap), it would make sense to consider them as part of the 'default' nodemap. I just wanted to draw attention on the fact that the 'default' nodemap can be too restrictive or too permissive for some client nodes. So with IPv6 clients it is important to correctly set properties on the 'default' nodemap.

So in the end, apart from the few comments I made about use of id.nid that would need to be clarified, this patch looks fine regarding the idea to assign all IPv6 NIDs to the 'default' nodemap. I think this would be worth mentioning in the commit comment.
