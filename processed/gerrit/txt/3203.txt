LU-1575 ptlrpc: client fix for the flock policy combinations

The current patch is intended to fix the issue with incorrect flock
policy determining. This issue is observed when 1.8 clients
with updated flock policies try to connect to the 2.x servers.
This patch is intended for the 1.8 clients to be recognized on the
updated 2.x servers.

--- Message (Andreas Dilger, 2012-08-20 19:49:14.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

(4 inline comments)

The problem with this b1_8 version of the patch is that our b1_8 code does not include the original change from b=22040 attachment 30902 which introduces the separate flock_wire_policy_data.  It does not appear correct to be  sending the OBD_CONNECT_FLOCK_OWNER from our unpatched b1_8 without actually sending the owner data.

I'm fine with landing the reservation of OBD_CONNECT_FLOCK_OWNER to b1_8 (pending Johann's approval) in http://review.whamcloud.com/3723, but the client can't send this at connect time until it is actually doing the right thing.

That is one of the main benefits of using an OBD_CONNECT_* feature flag compared to checking version numbers - e.g. 1.8.5 or 1.8.7 clients may or may not have the 30902 patch applied, but if there is a proper OBD_CONNECT flag negotiation at client connection time, the server will know how to act with each client, even if there is a mix of patched/unpatched clients at the same site.
