--- Message (Richard Yao, 2013-03-17 02:40:55.000000000) ---
Patch Set 1: Looks good to me, but someone else must approve

Changes in a library's ABI often require changes in the code of various dependencies to support them. On the other hand, failing to require a specific soname often results in incomprehensible runtime failures whenever an ABI change occurs. Requiring that the code be changed should the soname change is preferable to the runtime failures that will result should the ABI change under the binaries.

On a semi-related note, the use of dlopen() outside of the program loader breaks things that rely on ldd. This generally causes headaches for us downstream. Some examples of headaches that this causes us are as follows:

1. We have a tool called revdep-rebuild that relies on ldd to ensure that all system dynamic link dependencies are satisfied. It will invoke our package manager to rebuild any packages that contain binaries that depend on missing libraries.

2. We have a package manager feature in development called preserved-libs that relies on ldd to detect when updating/removing a package will break a dependency and preserve the library to prevent runtime failures.

3. Our initramfs software has a function called copy_binaries that relies on ldd to copy all dependent libraries into our initramfs. This avoids synchronization issues between our initramfs software and changes in dependent packages that prevent reliable initramfs generation.

The best thing for us downstream would be to rely on the program loader to dlopen() these libraries. I can see us patching the configure script to provide an option to do that in the future. If/when we do, that patch would be submitted for upstream review. In the interim, it would be desirable to specify the soname version. That will ease the pain associated with an ABI change in the ZFS libraries should one ever occur.

--- Message (Andreas Dilger, 2013-03-17 16:42:18.000000000) ---
Patch Set 1:

The reason we do not use normal dynamic linking for the ZFS libraries is so that we can build with both ldiskfs and ZFS support into the tools without having warnings or errors if users do not install the ZFS libraries. 

In a case such as yours, where you are only building with ZFS support, it seems reasonable to conditionally remove the dlopen() calls and allow the linker to do normal linking so that ldd will show the proper dependencies.
