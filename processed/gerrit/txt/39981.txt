--- Commit Message ---
LU-13972 o2iblnd: Don't retry indefinitely

If peer is down don't retry indefinitely. Use the retry_count
parameter to restrict the number of retries. After which the
connection fails and error is propagated up.

This prevents long timeouts when mounting a file system with
nodes which might have their NIDs configured in the FS, but the
nodes have been taken offline.

RE:I think we should at least adjust the description of the retry_count parameter since it is going to have an additional use. Or, rather than overload this parameter to serve multiple purposes, maybe we should create a new one? Do we even need a parameter for this functionality?

I thought about that and I don't think its necessary. 
retry_count
The maximum number of times that a data transfer operation 
should be retried on the connection when an error occurs. This setting 
controls the number of
times to retry send, RDMA, and atomic operations when timeouts occur. 
Applies only to RDMA_PS_TCP. 

In my thinking there is no need to separate the RDMA retry from the number of times we retry connection establishment. The added parameter will increase the configuration complexity and not give any extra flexibility. This way the number of retries for both the connection establishment and operations on the connection are the same. I do overload the parameter, but I think it makes sense in this case.

The other reason is that from what I have seen users hardly change this parameter.
