LU-2684 fid: unify ostid and FID

Since 2.4 will support FID on OST, so this patch will try to unify
ostid and FID, so one day, we will use FID to identify the object
everywhere.

Because both ostid and FID has 128 bits long, so we will re-define
ostid as union,

struct ost_id {
    union {
	    struct ostid {
		    __u64   oi_id;
		    __u64   oi_seq;
	    } oi;
	    struct lu_fid oi_fid;
    };
};

If oi_seq == 0, it will still use <oi_seq, oi_id> to locate the
object as before. And when building reside, it still keeps the
old way res[0] = obj_id, res[1] = obj_seq;

If oi_seq != 0, it will use FID(oi_fid) directly to locate the
object, and use FID to build resid directly, so it will be unified
with META lock resid.

Remove other direct _id and _seq accessment from the code.

--- Message (Andreas Dilger, 2013-03-29 17:04:27.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

Di, I think you misunderstand my intention. In ostid_{id,seq}() and osc_build_resid() the goal is to change the order of the oid and seq fields within struct ostid itself so that it looks like an lu_fid.  That would mean that oi_id would hold the f_seq value, and oi_seq would hold the f_oid and f_ver value iff oi_seq != 0. 

That would mean that osc_build_resid() would continue to use the same order of fields from ostid to generate locks (i.e. name[0]=oi_id or f_seq, and name[1]=oi_seq or f_oid|f_ver) for oi_seq != 0. 

The benefit of this is that old clients will continue to get the right locking, but new clients (when DNE is enabled and OST objects have seq !=0) will use lu_fid for any ostid where oi_seq !=0. 

It would probably be easier if you made struct ostid a union between the current ostid and lu_fid, so that this is immediately available everywhere ostid is used, and then ostid_id() and ostid_seq() can access this internally as an lu_fid (with fid_seq(), fid_oid()) whenever oi_seq != 0.
