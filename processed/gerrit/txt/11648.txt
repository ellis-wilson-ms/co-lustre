--- Commit Message ---
LU-5560 llite: basic support of SELinux in CLIO

Bring the ability to properly initiate security context
on SELinux-enabled client and store it on server side via
extended attribute.

Security context initialization is not atomic, but that would
require a wire protocol change to send security label in the
creation request.

Filter out security.selinux from xattr cache as it is
already cached in system slab.

Patch Set 16:

Looking into the code, I can see that files are created on MDT at lookup time with "open|creat" intent. This happens before the inode is created on client side. And the problem is that security_inode_init_security() requires an inode to retrieve the security context to associate to the file.

However, in RHEL7 a new security primitive was introduced: security_dentry_init_security(). It could be called from ll_lookup_it() as it only needs a dentry.

So the only viable solution to address your concern would be in RHEL7, and would consist in calling security_dentry_init_security() from ll_lookup_it(), and then send the security information with the lookup request (presumably creating a new request type) so that the MDT can set the xattr in the same transaction as the file creation.


Speaking of the implementation done in the current patchset, I agree it can end up with security.selinux xattr not set, for instance in case of crash of the client node creating the file (before the call to ll_setxattr()), or in case of concurrent access from another client node, which could see the file with no security.selinux xattr set yet.

But not having security.selinux xattr set does not create any flaw from a security point of view. Indeed, such a file will be assigned a default security context, and this is the responsibility of the SELinux security policy to be very restrictive with files that have the default security context.

Moreover, in the case of a client crash, it is the security administrator's responsibility to relabel the files.


To conclude, for all these reasons I would advocate to land the current implementation as a first step towards SELinux support on the client side. And as a second step I will implement the atomicity of security.selinux xattr setting on RHEL7 as explained above, given that it will require a lot of work for me.

What do you think?
