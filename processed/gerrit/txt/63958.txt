LU-12669 llite: check lock coverage in ll_readpage for EC reads

During EC recovery, generic_file_read_iter() may call
ll_readpage() for pages that map to deactivated stripes.
These pages have no DLM lock because the lock was skipped
via cld_heed_error during lov_io_lock().  When such a page
reaches osc_build_rpc(), osc_req_attr_set() hits LBUG
because osc_dlmlock_at_pgoff() returns NULL.

This can happen when:
- EC recovery partially fails (e.g. fault injection on
  a stripe causes its sub-IO read to fail, so the data
  cannot be reconstructed and is not placed in the page
  cache)
- The kernel's readahead reads a page that maps to a
  deactivated stripe within the i_size boundary

Fix by adding a cl_io_read_ahead_prep() check for every
CIT_EC_RD page in ll_readpage().  This verifies DLM lock
coverage before submitting the page for I/O.  Pages
without lock coverage return -EIO, which the kernel's
readahead absorbs gracefully.

Also add CIT_EC_RD to the LINVRNT assertion in
cl_io_read_ahead_prep() so it can be called for EC
recovery I/Os.

Resolves LBUG in tests 42d, 46d, and 47a
through 47d where readahead pages on deactivated
stripes had no DLM lock.
