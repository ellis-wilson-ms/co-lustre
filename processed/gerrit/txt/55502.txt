--- Commit Message ---
LU-17922 utils: added idmap range functionality

Added the ability to a declare a range when adding idmaps to a
nodemap. The syntax is:
 <clientid_start>-<clientid_end>:<fsid_start>[-<fsid_end>]

The uid_end value is optional. In practice this looks like:

nodemap_add_idmap --name test --idtype uid --idmap 500-510:10000

It is also now possible to delete idmap ranges with the
nodemap_del_idmap command as well with the same syntax.

Patch Set 5: Code-Review-1

(7 comments)

This looks much better now with the range count.

There is still the question of how to avoid storing individually each idmap generated from a range. This version of the patch calls nodemap_add_idmap() for every single couple in the range, which means we are storing 1:1 mappings individually.

One idea the store the range count would be to convert nir_padding1 to nir_id_count in struct nodemap_id_rec. This would make id mapping storage more scalable, but would require extra work in case a fraction of an existing id map range is deleted. As a first step we could only support deleting whole ranges (I would not find it offending to require the admin to recreate the mappings properly in case the ranges need to be adjusted).

Maybe the ability to store the range count directly could be handled in a subsequent patch?
