LU-5851 build: ship osd-related mount DSO out of lustre-osd

Since LU-4606 changes, an osd-specific DSO for mount/mkfs/tunefs
commands is shipped as part of each lustre-osd specific RPMs. This
makes Lustre Server DKMS-based installs to be unusable since only
the osd-specific module is build but not this library.

To fix this, now ship the osd-specific DSOs as part of generic utils
RPM, alos required to be install in addition of lustre-dkms one.

Then to allow this to be functional, I had to authorize init() method
of each DSO to fail during load because of missing externals (like
zfs/spl modules for mount_osd_zfs.so, ...), then let
backfs_mount_type_okay() allow/error upon explicit back-end hooks
usage, and ignore if Client mount.

Also, remove vprint/verrprint macros duplicate definitions.

Patch Set 14: Code-Review-1

(5 comments)

My goal for including the .so in the DKMS package was to keep the support for OSDs self-contained.   I was thinking "DKMS" => "build binaries when RPMs are installed" and not necessarily "build only kernel modules".  Having just a single RPM for each of the ldiskfs or ZFS functionality is just a cleaner packaging and less complexity for users, but it seems I'm in the minority here.

The current approach has the limitation that if ZFS or ldiskfs are not available when the main userspace package is built then the OSD-specific mount_utils_{type}.so will not be available later on even when the OSD-specific RPM (DKMS or binary) is installed.  That is not a loss for us today (we always build for both ldiskfs and ZFS even if we don't distribute the ZFS binaries), but I thought it could avoid problems with user builds, or adding new OSDs in the future (e.g. btrfs).

In any case, this patch needs to be fixed for other reasons (inline), but I'm fine with this approach if that is what others prefer.
