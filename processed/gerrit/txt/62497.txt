LU-7266 ldlm: Fix LDLM pool Server Lock Volume

In the previous implementation of SLV/CLV, there exists misalignment
between SLV and CLV, causing their comparison failed to produce the
desired lock revocation behavior.

The goal of this patch is to fix the impl, so we could
1. Enable clients to retain sufficient locks to accelerate their
   computation when resources are abundant;
2. Properly revoke locks from clients during memory pressure.

Implementation changes:
- Align lock granularity: both SLV and CLV now use sizeof(ldlm_lock)
  as the base unit
- Align SVR_POOL_MAX_AGE with CLI_LRU_MAX_AGE: reduce from 36000s
  to 600s (LDLM_DEFAULT_LRU_MAX_AGE) for consistency
- Replace gradual adjustment based on grant_plan (which exhibits
  significant delay) with a pressure-based method that considers:
  * Memory pressure: ratio of total-available/total-granted locks
  * Client count: number of connected clients
  This approach ensures proper alignment between SLV and CLV

New SLV calculation formula:
  slv = (lock_limit * pool_max_age * adjust_factor) / client_count

Where adjust_factor = exp(4 * (available/granted - 1)), which:
- Increases SLV when resources are abundant (available >> granted)
- Decreases SLV under memory pressure (available << granted)
- Maintains stability when available ~= granted, and by `-1` we could reach the balance target (adjust_factor = 1)
- Apply exp() transformation to smooth and amplify values which helps to
  * Enable clients to retain sufficient locks to accelerate their computation when resources are abundant, and
  * Actively revoke locks from clients during memory pressure.

Backward compatibility:
- Parameter 'slv_pressure_adjust' is provided to fallback to the
  previous implementation if needed

PENDING TO DO ==============================================================
Currently SLV/CLV treat all clients equally.
Under pressure, large and small clients may keep a similar number
of locks, which is not ideal for HPC workloads.

Introduce a proportional target:
target = CPU_cores × locks_per_core

CLV takes actual_locks / target into account, so clients retain
locks relative to their capacity rather than in absolute terms.

To avoid oscillation under sustained pressure, locks_per_core
must not be fixed. Since only the server has global visibility,
it dynamically determines this value and embeds it into SLV.

SLV layout extension (backward compatible):

bit 63..60 : 0xF, for compatibility, old SLV not fill this part,
bit 59..48 : reserved
bit 47..40 : locks_per_core
bit 39..0  : original SLV

This improves proportional fairness while keeping server-side
pressure control centralized.
