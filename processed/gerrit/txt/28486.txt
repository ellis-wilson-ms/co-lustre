LU-9868 llite: handle DCACHE_PAR_LOOKUP in ll_dcompare

ll_dcompare is used in two slightly different contexts.
It is called (from __d_lookup, __d_lookup_rcu, and d_exact_alias)
to compare a name against a dentry that is already in the dcache.
It is also called (from d_alloc_parallel) to compare a name against
a dentry that is not in the dcache yet, but is part of an active
"lookup" or "atomic_open" call.

In the first case we need to avoid matching against "invalid" dentries
as a match implies something about ldlm locks which is not accurate.
In the second case we need to allow matching against "invalid" dentries
as the dentry will always be invalid (set by ll_d_init()) but we still
want to guard against multiple concurrent lookups of the same name.
d_alloc_parallel() will repeat the call to ll_dcompare() after
the lookup has finished, and if the dentry is still invalid, the whole
d_alloc_parallel() process is repeated.  This assures us that it is safe
to report success whenever d_in_lookup().

With this patch, there will never be two threads concurrently in
ll_lookup_nd(), looking up the same name in the same directory.

This base patch works fine. Its the next patch in the series that is broken : "lllite: use d_splice_alias for directories." Well the patch is not broken it is lustre that is broken. 

The reason for that is the "fids" located in .lustre/fid refers to a directory, then that directory will have 2 names in the dcache - the name in .lustre/fid, and the "real" name.  The dcache does not permit this.  You cannot have two dentries pointing the the same directory inode which is currently happening. This is a serious design flaw in lustre. Currently only the statx -> ll_look_nd() -> ll_splice_alias() encounters this issue.

I'm working on a work around but the real fix is to move away from .lustre/fid and use the fhandle api which is now supported with all kernels and does the same thing as fid2path and path2fid

I wouldn't go so far as to say that the fhandle API does the same thing as fid2path.  The name_to_handle_at() function does the same thing as path2fid and open_by_handle_at() allows open-by-FID (for root users only, while .lustre/fid can be configured to allow non-root users), but (AFAIK) this doesn't allow resolving a pathname from a FID, which is very useful, and it almost certainly doesn't allow resolving multiple hard links from a FID like Lustre does (that needs special support in the underlying filesystem).

Newer kernels are currently broken with .lustre/fid due to the fact we have more than one directory inode pointing to the same parent dentry much like a hard link.
