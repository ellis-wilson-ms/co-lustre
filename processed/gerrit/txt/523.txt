--- Message (Andreas Dilger, 2011-05-19 07:16:54.000000000) ---
Patch Set 2: Looks good to me, but someone else must approve

(1 inline comment)

I'm OK with the idea of this patch, though I agree it is impossible to get the right size for all customers, but I think this is likely to help in many cases.  I wouldn't mind removing the old comments, but it is not a blocker.


It would probably be better to keep some kind of "hit count" on the inode to help decide whether the pages are going to be re-used, even for large files.  One possibility is every time a client needs to read e.g. page index 0 from a file the in-memory inode use count is increased, and at some use count threshold even a large file is kept in cache.  If many clients are reading the same large file the use count would go up quickly and it would be kept in RAM.  

By keeping the use count in the in-memory inode, there would be some automatic "use count aging" as the inodes were evicted from cache.  The next accesses to this file would be uncached unless there are again many clients accessing the file.

That is work for a separate patch.  I think this should go in for 2.1 so that we can determine if it is helping performance, and whether we need to change the default size value.
