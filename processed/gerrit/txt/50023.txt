LU-12275 sec: remove bio functions in fscrypt compat

Remove libcfs/llibcfs/crypto/bio.c since direct block device access
is not needed for client builds, and the use of stuct bio on the
client adds unnecessary complexity to portability.

> Can you say why this matters?  Is it causing a problem or is it
 > just a cleanup?

The reference to bio_alloc() on the client is causing compile issues with different kernels and adding unnecessary portability complexity to the client code.

This function is actually unused (it is only needed for fscrypt working directly on block devices) and could probably be under "#if 0" (or deleted completely) but this is a "less intrusive" version until I hear from Sebastien that it could be removed.

In lite we implemented our own "zeroing" function, ll_io_zero_page(). The function defined in llcrypt is not used by us, so it can be disabled. It would likely be possible to remove the bio.c file from autoMakefile.am, but this may also be confusing if the file exists in the repo but isn't compiled. That would still show up with "grep" and cause developers to think that bio_alloc() needs compatibility handling on the client, when it is unnecessary. 

My preference would be to delete it completely, and we can always recover it from git or from the upstream kernel if it becomes useful in the future.
