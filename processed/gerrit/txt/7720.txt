--- Commit Message ---
LU-3935 scrub: support dryrun mode OI scrub

The administrator can detect whether there are any inconsistent
OI mappings but without repairing them via the command:
lctl lfsck_start --dryrun on -M <$device>

Under default mode, dryrun is "off". When the device is mounted,
the initial OI scrub will detect whether there are upgrade case,
or file-level backup/resotre case, or OI file(s) recreated case.
If found some inconsistency, it will trigger non-dryrun mode OI
scrub automatically. If the administrator does NOT want to such
auto non-dryrun OI scrub to avoid some unexpected repairing, he
(or she) can specify "-o noscrub" when mount the device.

Patch is pretty good.

One improvement I was thinking about is to make it impossible to modify the OSD in DRY_RUN mode. With e2fsck in dry-run mode the device is opened read-only and if any code is not checking for dry-run mode before trying to fix some problem it will generate an error. I was wondering if it was possible to set some flag in the LFSCK env that will return an error if there are any calls to declare transactions or similar.

--- Message (Fan Yong, 2013-09-22 03:27:12.000000000) ---
Patch Set 1: (3 inline comments)

Because the OSD device is not mounted as readonly, and our OI scrub is an online tool. It is not easy to find an unique place to control the system modification without depending on the callers' checking. We needs to know whether the modification is from OI scrub or for normal RPC handling, that needs some additional parameter(s) from the callers, if some OI scrub code forgot or give wrong parameter(s), then related control will be failed. Such additional work is not cheaper than checking "dryrun" mode by OI scrub itself.
