LU-4684 dne: support migrating striped directory

To support migrating striped directory, both server and client
are changed.

Client:
* directory migration is similar to create, so add stripe-count
  and mdt-hash arguments to 'lfs mv/migrate'.
* MDS_REINT_MIGRATE rpc layout changed: client will pack struct
  lmv_user_md.

Server:
* because we can't create striped dirctory remotely, directory
  migration is done on target MDT, but non-dir files are still
  migrated on source MDT because regular file may open lease
  if busy.
* MDT migrate needs to lock source and all its stripes, if source
  is remote, it will check its layout to know whether its striped,
  if so, it will iterate its stripes and lock them.
* previously the source LMV will be enlarged to contain stripes
  of both source and target, now it's changed to store stripes
  of both in target LMV, and target then restripes dirents from
  source stripes to target stripes, when it's finished it
  will delete source stripes from its LMV. To implement this,
  a new field lmv_restripe_offset in struct lmv_mds_md_v1 is
  added, which indicates what stripes are from source, and what
  are from target. And when restripe is finished, the target
  LMV will be updated.
* restripe from N to 1 MDT needs special handling: the target
  will be left as a striped directory with single stripe in the
  end, though this won't cause any issue, it's confusing to user,
  so turn it into a normal directory after restripe.
* since migrating directory contains stripes from both source
  and target, directory can contain 2 stripes on the same MDT,
  and there can also be gaps between stripes, so the premise
  for striped directory is not true any more, to adapt to this,
  stripes lock needs to be updated, this is reflected in function
  mdt_reint_striped_lock().
* non-directory file is migrated in one transaction now, as
  simplifies migration interrupt resume.
* many rewrites and cleanups.

--- Message (Andreas Dilger, 2017-04-11 20:17:14.000000000) ---
Patch Set 1:

My thought is that directory migration should be very similar to directory restriping.  Essentially, create a new layout for the directory that includes old and new shards (directory stripes), then iterate over all of the entries and move them to the shard that they belong to in the new layout.  This ensures that if the (long and not atomic) operation is interrupted in the middle all of the entries can still be found in either the old or new location (which is the reason for LMV_HASH_FLAG_MIGRATION - so that the client will try all shards to find the entry).  It would even allow migration to happen while the directory is being accessed.

With directory restriping, only the existing name entries (not inodes) would be moved to the new shards, and newly-created entries would be created on the correct MDTs.  When moving from a non-striped directory to a striped directory, the non-striped directory would be the master entry, and it would gain shards to hold all of the name entries.  When moving from one stripe count to another (or changing hash functions), then new shards would be added at the start, all entries are migrated according to the new hash, then if the stripe count is reduced then the (verified-to-be-empty) last shard(s) would be removed.

With directory migration the hash function would need to be enhanced to add new shard(s) at the end and store an "offset" or similar parameter in the LMV EA, and restripe from the source shards to the target shards.

The inodes themselves would also need to be moved, but that is largely independent of how the directory layout causes name entries to be hashed.  Essentially, the DNE code would be told whether only the name is being migrated, or if the whole inode needs to be migrated also.

We might consider optimizing the case of removing only a single shard (e.g. MDT removal) during migration, where the new shard is added at the end, entries are moved from the old shard to the new one, and then the new shard replaces the old one in the layout.  During this time, the hash function should keep the names mapped to the existing shards so that they do not all need to be migrated.

--- Message (Andreas Dilger, 2017-04-13 08:02:36.000000000) ---
Patch Set 1:

I agree with Fan Yong that name migration should really just be treated very similar to a remote rename operation handled on the MDS.  I don't think we need a special client, just an RPC that triggers a "rehash" of the name from the current directory shard to the new destination.  In some cases like restriping from few to many stripes, they source and target shard be the same and it is a no-op.

Since we can already do renames properly while the directory is in use, I don't think this should be very complex to implement.  

The main area of development would be to create a new layout hash to handle the transition from one layout to another.  I'd thought about the idea of having two completely separate layouts (source and target layout), but that ends up moving all of the entries each time, while restriping within the a single layout can avoid moving all of the entries.  This is especially true in the (IMHO common) migration case where only a single MDT would be removed from the layout.

For restriping the name migration is all that is needed.  I think that the more complicated part is the inode migration, but AFAIK that is already implemented in the existing DNE2 code and it would just need to be isolated from the rest of the migration code so that the inode is migrated separately.
