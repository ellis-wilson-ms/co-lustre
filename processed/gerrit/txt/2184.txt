LU-1128 ldlm: return -1 for server pool shrinker

For ldlm server pool shrinker, we just use it to decrease SLV,
but never reclaim any memory directly, so it should always return
-1 to inform the kernel to break the shrink loop.

--- Message (Andreas Dilger, 2012-02-28 17:17:51.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

(3 inline comments)

I don't think that this patch will fix the actual problems, but it may hide the current symptoms.  My concern is that it might cause the clients to drop too many locks when memory pressure arrives on the OSS/MDS.

I suspect (based on previous observations, and the results shown by Chris in LU-1128) that the SLV calculation is totally incorrect and needs to be re-examined for correctness of design and implementation.

I would be OK with having a simple patch that shrinks SLV more aggressively, and returning "-1" to avoid busy-looping, but I don't think that setting it directly to an arbitrary value will produce a stable system.
