--- Message (Andreas Dilger, 2011-09-20 22:13:41.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

(6 inline comments)

The commit summary for this patch implies that opening the llog read-only will avoid corruption of the llog file, but I don't think that is an accurate description of the problem being hit, nor is it describing what this patch is actually fixing.

If I would guess from reading the patch, it seems that it is using the LLOG_CREATE_RO flag avoiding the creation of a new llog file, when the original one was not successfully written?

I think it would be a lot more understandable if this method was renamed to llog_*open() instead of llog_*create().  The LLOG_CREATE_RO flag name doesn't really mean anything sensible.  It might be better to have open flags like LLOG_OPEN_RO = 0, LLOG_OPEN_CREATE = 1, and LLOG_OPEN_RW = 2 (it doesn't make sense to use LLOG_OPEN_RO with the other two flags).

Also, having a test case that demonstrates this problem would be excellent, so we know it remains fixed.
