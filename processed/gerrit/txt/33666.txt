Patch Set 10:

> The patch is mostly working but I'm seeing failures for sanity test
 > 200. Only change is the removal of the FILESET test. I tried
 > running ONLY="200" sanity.sh but it passes. What am I missing.

The "subtest-change" sessions are running any new or modified subtest in a loop for 30 minutes, to avoid landing patches with intermittent test failures.  It looks like there is some cruft left behind after the first iteration, and it fails on the second pass:

 https://testing.whamcloud.com/test_sets/6a7a9905-6455-442d-b961-48563f4e25b2

 lfs setstripe: setstripe error for '/mnt/lustre/d200.pools/file_tst/spoo-22': stripe already set
 lfs setstripe: setstripe error for '/mnt/lustre/d200.pools/file_tst/spoo-23': stripe already set
 lfs setstripe: setstripe error for '/mnt/lustre/d200.pools/file_tst/spoo-24': stripe already set
 Checking 'lfs df' output
 Creating files in a pool with relative pathname
 lfs setstripe: setstripe error for '/../mnt/lustre/d200.pools/dir_tst/f200.sanity-1': stripe already set
 sanity test_200: @@@@@@ FAIL: unable to create /../mnt/lustre/d200.pools/dir_tst/f200.sanity-1 

Possibly this is due to the "run_one_logged()" function running the "rm -rf" on the dir inside/outside the subdirectory mount, and this is missing some files/dirs that were created on the first iteration of the test.  These would need to be removed explicitly by the subtest so that they don't cause problems on the later iterations.
