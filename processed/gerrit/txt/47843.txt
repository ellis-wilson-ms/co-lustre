LU-15971 llite: match lock in corresponding namespace

For remote object, LOOKUP lock is on parent MDT, so lmv_lock_match()
iterates all MDT namespaces to match locks. This is needed in places
where only LOOKUP ibit is matched, and the lock namespace is unknown.

(1 comment)

Uhh, I misunderstood the test failure seen in the next patch. The real issue is: open fetches OPEN|LOOKUP lock for remote object, and this issue is addressed by https://review.whamcloud.com/#/c/40764/.

The failure in subsequent patch is:
1. client has two locks of the same remote object: one is LOOKUP from normal lookup, the other is OPEN|LOOKUP from open.
2. The first lock was revoked, but client matches the second, so the dentry is not invalidated.
3. Then obsolete default LMV is seen.

But IMO this patch is still good, because currently lmv_lock_match() iterates all namespaces to match locks, which is a waste and is inefficient if there are quite a number of MDTs. Actually in all the places the corresponding LDLM namespace is known.
