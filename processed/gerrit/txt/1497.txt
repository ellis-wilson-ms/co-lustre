--- Commit Message ---
LU-359 llite: no close error if application has known failure

Don't return error again when close if the application has known
former write failure.

Rename the per-inode based 'lli_write_rc' as per file-structure
based 'fd_last_write' to track the last write/fsync failure.
When 'sys_close()' is called against the 'file' structure,
we know whether the caller has already known former write/fsync
failure or not, then avoid potenical rdundant error handling,
like confused error message.

Miss to set 'lli_async_rc' if ll_writepage() failed.

Patch Set 8: (3 inline comments)

In addition to my comments, I think there is another potentially unexpected failure case:

process1:
open file
try write, generate error (recorded in async_rc);
(not closing yet)
process2:
open file
try write, succeed.
close (will get error from async rc because of process1)

to avoid most unexpected pasrt of it, I guess we can also check async_rc in open and set "saw error" in struct file if async_rc is set.
Though I guess we don't have a clear way to distribute errors properly between two processes if both of them are writing into the same file on the same node and some writes fail only for one of those processes.

--- Message (Fan Yong, 2012-08-08 05:14:17.000000000) ---
Patch Set 8: (3 inline comments)

> In addition to my comments, I think there is another potentially unexpected failure case:
> process1: open file try write, generate error (recorded in async_rc); (not closing yet) process2: open file try write, succeed. close (will get error from async rc because of process1)

For most cases, write() called succeed does not means the data are really written back to storage. When close() error, we cannot distinguish whether it is process1's async failure or the process2's.

There is no way to totally distinguish which process is the owner of the async error. This patch is just try to avoid redundant error message when close().
