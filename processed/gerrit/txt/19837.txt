--- Commit Message ---
LU-8162 llite: lock i_mutex when suid bit changes

In SLES12 __generic_file_aio_write assumes i_mutex is locked.
In Lustre it is not, as protections are handled elsewhere.
This causes a stack dump the first time file_remove_suid() is called,
and we'd rather not see that.

(1 comment)

I do want to point out the reason for this new lustre_**() function is to take the lock. What Ben produces still takes the lock mind over over a small range of code but it will still hurt performance.

I took a look at the xfs code and I noticed that file_remove_suid() only needs to be called when IS_SEC(inode) is false. We could instead have something like

if (!IS_SEC(inode))
     generic_file_aio_write(...)
else
     __generic_file_aio_write(...)

This way the pain is only for those files with suid is set which is rare. The bonus unlike the posted patch here is that we avoid taking the mutex expect for that rare case.

Yes I need to devote cycles to fixing the newer kernel issues. I will do that under the JIRA ticket opened for that issue.

If you want to use the CRAY ticket you need to make it public. I will need to push this patch upstream and having a ticket that can't be viewed by the public is a no-no.
