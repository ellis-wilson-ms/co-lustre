> Hi Neil, can you please elaborate on "the better way"?

Not easily.  I think this approach, of tracking which thread hold the lock and taking it if this thread doesn't already hold it - feels clumsy.  It isn't a common pattern so it is harder to reason about.
The original patch looked OK, but already a problem has been found that needed fixing.  How can we be sure there aren't more?  It is a solution that doesn't feel like it will be maintainable.

I cannot suggest a better way without a deeper understanding of the code than I have.  But I feel sure there must be a better way.  We need to step back from the particular symptom and look at the what the code is really trying to achieve.

The core of the problem seems to be that we need a lock to call truncate_inode_pages(), but sometimes we don't have the lock.
I think that if we have a "struct cl_io" pointer, then we know we have the lock and it is safe to call truncate.  If we don't - what then?
The current approach is to try to take the lock - which gets clumsy.
Do we need to?  Can we just do something else?
One of the problem paths involves lov_layout_change().  It seems that in some circumstances it is OK to simple set LO_LAYOUT_INVALID and not call lo_layout_change().  Could we do that also if we won't have a 'struct cl_io' ??

i.e. rather than taking the lock if we don't already have it, can we somehow delay the work to some other time when we do have the lock?
I really don't know as I only have a vague idea of why we might want to truncate the inode pages when the vfs hasn't already taken the inode lock for us.

Maybe a good starting point is a clear answer to that question.  In what circumstances would there ever be a need to truncate pages when the VFS isn't expecting us to?
Is a layout change the only situation, or are there others?
Is there some other way to respond to a layout change, or to any other event which might need pages to be truncated?
