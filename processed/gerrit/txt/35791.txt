A "struct file" - sometimes called a "file description" in man pages - can certainly be shared by multiple "file descriptors" and by multiple processes or threads.
If you "dup()" a file descriptor, both descriptors refer to the same struct file.
If you "fork" a process, the corresponding descriptors in the different processes refer to the same struct file.
If you have a process with threads and one thread opens a file, other threads can access through that file descriptor and all access the same struct file.

So yes: one process could have outstanding IO while another process releases the lock.  I'd probably call that an application bug.

Neil,

Used to be I could find http://lists.lustre.org/pipermail/lustre-devel/2006-May/000071.html
That in lustre-devel archive.  It's a design description from Oleg.  But that link is now dead and the available archive doesn't go back that far and other googling failed...

So, I guess I'll write something here.

Group locks are definitely mandatory.  They block all other i/o while they're being held.

Here's a bunch more about them...  *small sigh at the loss of Oleg's doc*

They're a special kind of LDLM exent (data) locks (LDLM is the client-server locking that is the main consistency mechanism in Lustre) that can be requested from userspace.  They're a kind of exclusive lock, like write locks, where they cancel all other locks (they're also full file locks, so while they're exent locks by *type*, the extent is always 0-EOF).  But whereas other types of LDLM locks are cancelled on conflict, group locks are not.  They can also be *shared* between multiple clients if they give the same group number.

So, for example if two clients want to write the same block, the usual consistency mechanism for that one gets a write lock, then the other has to cancel it first, forcing write out.  If those clients are sharing a group lock on a particular file, then they can both just write to that block and no consistency protocol is applied.  Just a race.  Also, read-after-write visibility (including clearing from cache previously read data on write) is not enforced between group lock holders, because again that's done via lock cancellation and with group locks everyone with one has the necessary lock for any operations.

What group locks are *for* is situations where the application knows it will never access the same block, but will, for example, have a lot of very nearby writes.  It's hard to lock efficiently for that in the normal manner - Lustre tries to grant locks "optimistically", giving bigger locks than required, so subsequent writes can use existing locks rather than requesting new ones.  But certain patterns - like repeated adjacent writes - 'defeat' this, resulting in false sharing of locks by i/o that never actually overlaps.

So group locks are basically a super dangerous way to let applications sign up to mostly disable the inter-node POSIX write visibility/ordering guarantees of Lustre.  (They also make file size unreliable.)

A little more about this area specifically, you've likely picked this up already, sorry if so:

When a group lock is requested on a file descriptor, the associated struct file is tagged with a flag saying it's group locked and some other info (see struct ll_grouplock), mostly the group id.  (Which, in case it's not clear, is not a group in the user/group sense, just a magic number intended to identify everyone who wants to share the lock.)

Then when i/o is done to that struct file, it is tagged with the group lock info, and allowed to match the group lock.  If i/o is done on another struct file (other than the one which asked for the group lock), it doesn't have the associated stuff, and doesn't get tagged.  That i/o blocks until the group lock is released.

So group locks are absolutely mandatory.  They're a special hook in to the file system given to applications (by request of some application folks) to let them do dangerous things.  (About lock ahead, there's some pretty decent stuff online on that if you're interested.  It's related but not relevant to this conversation.)

And yeah, it's possible to hang your system if you ask for a group lock and then don't release it.  Note that they are released on final close, see ll_md_close (called from Lustre's .release).  The assumption is an application environment where a non-exiting process is eventually killed automatically.  (And if a node holding a group lock goes away, then it's eventually cleaned up via eviction.)

The bug comes in when the same client gets two group locks with the same group.  Normally, a *second* *matching* group lock request on the same client just increments the reference count on the existing (LDLM layer) group lock, and that's fine.  But if the existing lock is being cancelled, at a certain point in the revocation, it can no longer be matched by new i/o.  (The basic way lock cancellation & reference counting works is the lock is marked for callback, then it can't match new i/os, but can still have pages under it.  Then any pages under it are written out/destroyed as appropriate, *then* it's ready and can be gotten rid of.  At this point, the client can tell the server the lock is gone.)

Anyway, the second request doesn't match the existing lock, so it goes out to the server and asks for another one.  Because the group ids match, the server grants the lock.  This makes it possible to get two locks on the same pages on the client, which doesn't work for a lot of reasons.  The LDLM design is such that the client isn't supposed to wait on the *client* side before sending a request to the server.  Deciding whether or not to wait is the servers' job.

That's why Alex's patch A) forces the process doing the last "put" to wait for the group lock to be 100% destroyed (this is normally done async), and B) prevents other group locks from being requested until that's complete.

The question is can an i/o leak around what he's got.  I still think yes because of the fd stuff we discussed, but I'm not sure...?  (I saw your comment Alex - I'm not sure if the process spawning here fits the kind of process cloning & fd cloning we're talking about.  I can try to take a look later.)

The other question is this:
Why doesn't the server prevent a client from getting two group locks on the same file (and make the client wait)?  It could just check the client node, which is recorded as part of the lock.  The answer is that group locks may be used by thousands of nodes at once, and every group lock request would have to walk the whole list to make sure no other locks from that client are present.

That said...  Alex, this does sort of seem like it should be a server side thing, since the server normally handles making clients wait for locks.  I've avoided trying to solve it there for the reason I just gave - walking long lists for *every* request.  (So if 10,000 clients request group locks on a file, you'd walk the list 10,000 times, longer each time.)

Could we just hang a list of group locks off of each export (? is that the per client thing?  I think so...) on the server side, so it's per-client?  Then that list could be walked as part of checking group lock compatibility.  I suppose it's possible for one client to hold a bunch of group locks (on different files), but that seems very unlikely, given their special nature?

Lots to respond to here ...

1/ To answer "Can you highlight the code that suggests that race?  AFAIK, such a race doesn't exist."

I was particularly referring to 
  		CERROR("another thread just won the race\n");
in ll_get_grouplock(), which suggests a race is possible.
The "race" is that the threads try to get a group lock on the one file description.
Both succeed (which I wrongly assumed the server would prevent) and the "loser" immediately drops the lock that it received.

2/ I don't think the server should prevent a client from getting multiple group locks with the same groupid. (though I think the performance issue raised is irrelevant - a different data structure would fix that).
The whole point of the group lock, it seems, is to allow multiple co-operating processes to advise the server of their decision to co-operate, so that the server gets out of their way.  There is no reason these co-operating processes cannot be on the same client.

3/ The problem, I think, is that an *inode* must never have two grouplocks, but the current code at best ensures that a given *struct-file* never has two grouplocks.  The patch adds per-inode locking to prevent any attempt to request a grouplock while a grouplock is held. This is quite independent of which struct-file requested the group lock.

4/ Your point, Patrick, that the client should generally not wait on the client before sending a request to the server is worth exploring, but I think a key issue is that this is part of the LDLM design.  This problem, on the other hand, is part of the local inode cache management design.
The reason that the client doesn't want to hold two group-locks on the same inode is not (I assume) because of any property of LDLM, but simply because it has no mechanism to keep cache associated with one lock separate from cache associated with the other other lock.  So this is a completely local problem and needs a completely local solution.

5/ You say "Alex's patch A) forces the process doing the last "put" to wait for the group lock to be 100% destroyed".  I cannot see that in the code (which certainly doesn't mean it isn't there).  Can you point it out to me please?

6/ You ask "can an i/o leak around what he's got".  My guess is "not" because the "fd stuff" seems largely irrelevant now that I understand that the interlock is per-inode.
Surely any new IO request needs to be associated with some lock.  If the group-lock is no longer available, then either another lock will have to be created (as presumably happens with normal writes), or the write will fail.

7/ A lot of what we have discussed here should really be in the commit description.
Something like:

  If two threads each request a group lock on the same inode for the same
  group-id, the server may grant them both (as group locks for the same group
  id never conflict).  This can result in one inode having multiple group locks
  (possibly only transiently).
  This is a problem because of "a lot of reasons" that I don't fully understand yet.

  To avoid this situation we hold a new per-inode lock - lli_group_mutex - whenever
  adding or removing a group lock, and ensure that we only attempt to have one at a
  time.

8/ It looks odd to me that the patch adds an effective reference counter on the lock that it holds (lli_group_users), but still goes ahead and calls cl_get_grouplock multiple times.  I gather from your comments Patrick that this will increment some refcount at some other level, but it still seems confusing.
I would:
  take the mutex.
  if lli_group_users is > 0 and lli_group_gid matches, then increment lli_group_users,
     drop the mutex, and return.

What is the value in calling cl_get_grouplock every time?
