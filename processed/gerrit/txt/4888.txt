--- Commit Message ---
LU-2360 ptlrpc: a few fixes for req_history read

This patch fixed a few issues in seq_file operations of ptlrpc
request history:
- start() function always start from CPT 0 despite of start position,
  this might cause endless read.
- ptlrpc history sequence can't be directly used as read position of
  seq_file, because seq_read() can increase position to indicate
  reading of the next element, this will actually change CPT ID of
  sequence, which means it's possible that next() will never be able
  to finish iteration of requests on a CPT in some case.

--- Message (Andreas Dilger, 2012-12-23 17:30:54.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

Liang, the difficulty with putting the CPT number at the top of the sequence is that it now makes it difficult in userspace to order the RPCs correctly. The reason the CPT number was in the low bits was that this allows the RPCs to be printed in roughly time order.

Probably printing the RPCs in time order is more complex than needed, especially if the order is not critical for all users, but it should be possible to sort them in some manner.

One way to do this is to switch the CPT number into the low bits again only _when printing_ so that the seq pos works properly internally, but it is still possible to sort the entries by sequence in userspace if needed.
