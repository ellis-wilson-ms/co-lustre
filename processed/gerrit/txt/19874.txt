Patch Set 2: Code-Review+1

Have you tried running with "lctl set_debug=+lfsck" on a clean filesystem and during sanity-lfsck to see how much is printed on the console?  I would prefer if we could print all of the fixes that LFSCK is making to the console, so that it is easier to track, and to make administrators aware of when their filesystem is inconsistent. 

However, we can't do this if LFSCK is printing too many non-critical status messages to the console (e.g. one per inode handled, or too many repeated messages when the same problem is fixed many times). It may be that using CDEBUG_LIMIT(D_LFSCK, ...) is needed for some cases?

Currently, if we do not drop some LFSCK logs, it will print too much things if we directly put it into console. The large quantity of logs includes:
1) OST side object access bitmap updating. That is not necessary, can be dropped.
2) rebuild linkEA for upgrading case.
3) rebuild OI/fid-in-dirent for backup/restore case.

If we limit above case LFSCK logs, then can print LFSCK repairing logs to console directly.
