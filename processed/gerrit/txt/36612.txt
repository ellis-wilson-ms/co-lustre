--- Commit Message ---
LU-8585 tests: re-enable fid2path sanity test for subdir mount

When a subdirectory is mounted, the .lustre directory cannot be
seen. With the move of llapi_fid2path() from using .lustre/fid to
name_to_handle_at() previous sanity test that failed due to
file sets being enabled should now pass since name_to_handle_at()
should be file set aware. We update a few test applications to
work with llapi_open_by_fid() if we can't access .lustre. Same
is done with liblustreapi.so as well.

Patch Set 6: Code-Review-1

I am not sure this patch should be integrated as is. From what I understood, the original intent of LU-8585 is to increase the test coverage. But by switching the open(".lustre/fid/...") to llapi_open_by_fid(), you are effectively removing the .lustre/fid tests from the tests that you reenabled in this very patch. A good example of this is test 233a which was testing .lustre/fid. So now, it is a test for llapi_open_by_fid. Maybe llapi_open_by_fid needs more test coverage, I don't know. But it doesn't seem to be the right place/way to do it.

Furthermore, .lustre/fid is used in production (I know of at least 4 projects at CEA alone that use it directly). We should keep the tests we have for it. And at the moment, it means that those tests cannot be run in a subdirectory mount.

I think that before landing this patch, we should decide whether we want .lustre/fid in subdirectory mounts or not (which would make things simpler). And maybe find a way to increase the test coverage for filesets without reducing it for .lustre/fid. Rather than the other way around: remove .lustre/fid then find a way to introduce .lustre/fid (or similar) to subdirectory mounts.

For me, this patch is confusing two issues: increase the test coverage and allow some functionalities relying on .lustre/fid to work on subdirectory mount. It looks like deciding what we want/can/should do in a subdirectory mount about a .lustre/fid like functionality should be decided before this patch can be landed.

As for the HSM use case, I'm curious if this is a real use case or simply an example of something that is theoretically impossible but not useful in practice? It is possible to have several archive IDs for each group of users and one group of independent copytools per archive ID. Yes, they won't use the subdirectory mount but no user will have access to the copytools anyway.
