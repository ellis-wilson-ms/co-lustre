Patch Set 2: Code-Review-1

No, I think that's a misunderstanding of how the code works here and/or LDLM lock counts.

This code adds a maximum of 2 messages per queued lock.  Most of the time, most files have 1 lock on them (seriously).  Sometimes they have a few, but there are various things that help us not need to look at the whole queue in those cases.

Most of the time this will have very little impact on debug message traffic.

The issue is more "is there a case where this could explode", and after reflection, I think the answer is yes - there are some cases on the server side where we might have thousands of 'open' locks and a request which matches none of them but we print them all out.  An N^2 thing as we add new locks, printing every lock for every new lock.

So I'm going to add a tunable to determine how many times this is allowed to print per new lock, and set it by default to, say, 5.  That will get us the debug we need 99% of the time while preventing blowups.
