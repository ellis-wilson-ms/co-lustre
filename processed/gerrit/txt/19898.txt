Patch Set 3: Code-Review+2

Except in unusual cases, if patches fail testing we require that they be retested to ensure that we don't miss test coverage (e.g. all the tests after recover-small test_23 that didn't get run because of LU-7903).  We are working to allow retesting only the one test that failed, but until that is available the patch has to redo all of the testing.

Doing the full retest also helps to catch patches that have their own intermittent failures.  This also provides an change to make minor updates to the patch when it is resubmitted for testing.

Since this patch is a month old, the build is no longer available, and it just rebuilding it would likely just end with a failure due to LU-6937 because testing has moved from RHEL6 to RHEL7, so I've rebased it onto the latest master and it will be built again and retested.

Doing the rebase/resubmit of a patch after a test failure is something you can do for your own patches.  It is also possible to request a patch to be retested shortly after a failure, maybe within a week of the original test failure while the RPMs are still available.
