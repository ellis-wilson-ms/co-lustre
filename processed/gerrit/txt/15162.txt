--- Commit Message ---
LU-6602 update: split update llog record

If the update llog record size exceeds the limit (llog
chunk size), which usually happens when creating striped
directory with large stripe count , then it will be split
into mulitple records, and these update records will have
the same batchid.

During recovery, these records will be combined and
constructed into the updates of one operation, then
do update replay.

Allow multiples stripes in a single MDT, so it can verify
creating large stripes in small scale test environment.

Add sanity 300j/300k and replay-single 116 to verify it.

I haven't closely reviewed all of the details of the implementation (hopefully Alex will understand all of it).  My main concern is about the protocol/compatibility change.  At a minimum, this should change a magic value somewhere so that existing systems (e.g. James') would not try to recover from older logs.

If this is only changing the recovery llog files on disk (which are consumed internally to the MDS that wrote them) and not the network protocol, then it should be enough to just change the magic and ensure that recovery code checks the magic and either skips old ones or requires the recovery complete on the old version before starting the new one.

If this is changing the OUT network protocol that is also in use by LFSCK to OSTs then we need to work harder to ensure interoperability.
