LU-4423 lnet: Better cookie gen

api-ni.c uses do_gettimeofday to get a 'cookie' or timestamp.
This patch replaces it with ktime_get_ns for the following reasons:

1. ktime_get_ns returns a __u64 which is safer than 'struct timeval'
   which will overflow on 32-bit systems in year 2038 and beyond.
2. Improved resolution: nsecs instead of usecs.
3. Reduced compute: ktime_get_ns is faster than the multiply/add
   combination used in this function

Linux-commit: 9056be30542bfff51190bdda67088f319cf4c9f5

Drop unneeded wrapper function Remove the function
lnet_create_interface_cookie() and replace its call
with the function ktime_get_ns().

Linux-commit: 7bcd831b8579212303ec7c30e975432b914493dc

The ln_interface_cookie is used to ensure that a node can tell whether
the following sequence of events has happened:

node sends GET or PUT to peer
node is rebooted
peer sends REPLY or ACK to node

The ln_interface_cookie is set once, when LNet starts, and remains
unchanged afterwards. To avoid accidentally obtaining the same cookie
after a reboot, the code generated ths cookie using ktime_get_ns().
Once generated, the value of the cookie is not interpreted, only
compared for equality. Olaf Weber reported that due to the use of
ktime_get_ns() a small chance exist of generating a cookie of identical
value across reboots. Using ktime_get_real_ns() removes any chance of
this from happening.

--- Message (Olaf Weber, 2017-01-05 09:35:04.000000000) ---

As far as I can tell, the ln_interface_cookie is used to ensure that a node can tell whether the following sequence of events has happened:

- node sends GET or PUT to peer
- node is rebooted
- peer sends REPLY or ACK to node

The ln_interface_cookie is set once, when LNet starts, and remains unchanged afterwards. To avoid accidentally obtaining the same cookie after a reboot, the current code uses do_gettimeofday() to obtain the wall clock time. Once generated, the value of the cookie is not interpreted, only compared for equality.

Effectively what we want for the cookie is a 64 bit value that's unlikely to be the same across boots.

With that in mind:

  1. ktime_get_ns returns a __u64 which is safer than 'struct timeval' which will
     overflow on 32-bit systems in year 2038 and beyond.

For this particular application the overflow would not matter. Avoiding it doesn't hurt either, and we can assume that the timeval-using interfaces will go away so we'd have to do something here in any case.

  2. Improved resolution: nsecs instead of usecs.

ktime_get_ns() gives the ns the system has been running (awake) since boot. When LNet is started at boot time this increases the likelihood of a cookie collision, though the added resolution helps to ameliorate this risk.

  3. Reduced compute: ktime_get_ns is faster than the multiply/add combination
     used in this function

Largely irrelevant since this is a single call during initialization, as opposed to a performance-critical path.

Would ktime_get_real_ns() not be a better choice?
