--- Commit Message ---
LU-6200 ofd: recreate objects for setattr

the objects synchronization at MDT was triggered after the recovery
completed, then it's possible that the corresponding objects don't
exist when the write or setattr operation from clients was sent to
the OST just after the recovery was over. this patch will recreate
missing objects in these cases.

envdefinitions=SLOW=yes,ENABLE_QUOTA=yes \
mdtfilesystemtype=zfs ostfilesystemtype=zfs \
mdsarch=x86_64 clientcount=4 osscount=2 mdscount=2 austeroptions=-R \
failover=true iscsi=1 testlist=recovery-mds-scale

--- Message (Fan Yong, 2015-04-27 09:47:35.000000000) ---
Patch Set 3:

> frankly, I don't understand why setattr should recreate all the objects? why not just the needed one like we do in write path?

If the client write RPC only create the missed OST-object, but not update the LAST_ID file, then it will cause the real last_id to be greater than the last_id value that is stored in the LAST_ID file. Such case may misguide the LFSCK to regarded the system as inconsistency.

--- Message (Alex Zhuravlev, 2015-04-27 09:50:49.000000000) ---
Patch Set 3:

I'd think LFSCK shouldn't step in until the regular recovery is completed ?

--- Message (Fan Yong, 2015-04-27 09:53:57.000000000) ---
Patch Set 3:

Normally, LFSCK should not. But we cannot assume that the recovery will succeed always. So if the recovery failed, and the LFSCK start after that. It will be trouble.

--- Message (Alex Zhuravlev, 2015-04-27 11:20:37.000000000) ---
Patch Set 3:

it's still not clear to me why should we recreate _all_ preceding objects.

--- Message (Fan Yong, 2015-04-27 11:56:05.000000000) ---
Patch Set 3:

It should not recreate _all_ preceding objects. Instead, It recreate the objects from last_id to the given OID, and the last_id will be updated as the object recreated.

--- Message (Alex Zhuravlev, 2015-04-27 11:57:05.000000000) ---
Patch Set 3:

ok, let me reword - why it needs to recreate few objects at all?

--- Message (Fan Yong, 2015-04-27 16:03:48.000000000) ---
Patch Set 3:

If it only recreates the object to be written or setattr, then whether it should update the LAST_ID or not? If not, then it may cause subsequent LFSCK trouble as described above; if updates the LAST_ID but only re-creates one object, then may cause the normal recovery thread to miss to re-create some lost OST-objects since LAST_ID has been updated (increased).
