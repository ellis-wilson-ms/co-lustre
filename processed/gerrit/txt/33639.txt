--- Commit Message ---
LU-11582 llite: protect reading inode->i_data.nrpages

truncate_inode_pages() looks up pages in the radix tree without
lock, and could miss finding pages removed from the radix tree
by __remove_mapping(), so that after calling truncate_inode_pages()
we need to read the nrpages of the inode->i_data with the protection
of tree_lock.

Since it could still be in the race window of __remove_mapping()->
__delete_from_page_cache()->page_cache_tree_delte(), before the
nrpages being decreased.

--- Message (Patrick Farrell, 2018-11-19 20:26:01.000000000) ---
Patch Set 2: Code-Review+1

Ugh.  I dislike that it looks like (in llite_lib) we implement truncate_inode_pages_final.  We do not.

Older kernels that don't have truncate_inode_pages_final [CentOS 6, so not involved here] just use truncate_inode_pages.  truncate_inode_pages_final is a much more complicated function added to support a new feature.

I guess it bothers me because providing a function suggests we implement that function, as opposed to something like:
#define truncate_inode_pages_final truncate_inode_pages

Which makes clearer that there's just a mapping.

But the generated code is the same, and it's all old news anyway.  (Though related - Moving to newer kernels meant the workaround had to be moved out of that #ifdef'ed code, hence this patch.)

And this patch itself looks correct to me.
