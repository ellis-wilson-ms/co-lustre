Patch Set 1: Code-Review-1

(1 comment)

I really don't like this.  It makes the code much more complex, and possibly causes bad memory access.

It would be better to get the jobid from the file that is being read, since it *has* to have been accessed by this process before the readahead started:

                /*
                 * Whenever a process try to read/write the file, the
                 * jobid of the process will be saved here, and it'll
                 * be packed into the write PRC when flush later.
                 *
                 * So the read/write statistics for jobid will not be
                 * accurate if the file is shared by different jobs.
                 */
                char           lli_jobid[LUSTRE_JOBID_SIZE];

It may be that the readahead threads need to be added to jobid_name_is_valid() so that the readahead threads do not try to "save" their names into lli_jobid.
