Hm, I thought replication was supposed to have a blocking mode (uninterruptible) and an interruptible mode...?  I am just working from memory here.

But I don't think we should automatically switch all replication to being blocking, it should be a choice.

Consider a background replication tool that is doing automatic tiering.  We do not want this tool to prevent a user from accessing a file - We would want this tool to 'lose' the competition for the file, and deal with the file later, rather than force the user to wait.

So I think this should be switchable.

One other possible issue - How long is this write lock held?  Will the PW lock be prolonged correctly?  (I think this is OK, just trying to be sure.)

--- Message (Alex Zhuravlev, 2020-08-06 19:19:38.000000000) ---
Patch Set 1:

this is compatibility table (lustre_dlm.h>):

 * <PRE>
 *       NL  CR  CW  PR  PW  EX
 *  NL    1   1   1   1   1   1
 *  CR    1   1   1   1   1   0
 *  CW    1   1   1   0   0   0
 *  PR    1   1   0   1   0   0
 *  PW    1   1   0   0   0   0
 *  EX    1   0   0   0   0   0
 * </PRE>

PW is compatible with CR, not compatible with PR and PW

then, regular open (mdt_object_open_lock()) uses the following modes:

		if (open_flags & MDS_OPEN_LOCK) {
			if (open_flags & MDS_FMODE_WRITE)
				lm = LCK_CW;
			else if (open_flags & MDS_FMODE_EXEC)
				lm = LCK_PR;
			else
				lm = LCK_CR;

i.e. lease lock (taken by mirroring, should be incompatible with FMODE_WRITE (CW) and FMODE_EXEC (PR)
and compatible with FMODE_READ (CR) ?
