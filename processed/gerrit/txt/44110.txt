--- Commit Message ---
LU-14798 lnet: add LNet GPU Direct Support

This patch exports registration/unregistration functions
which are called by the NVFS module to let the LND know
that it can call into the NVFS module to do RDMA mapping
of GPU shadow pages.

GPU priority is considered during NI selection.

Less than 4K writes are always RDMAed if the rdma source is
the gpu device

--- Message (Andreas Dilger, 2021-06-30 00:23:40.000000000) ---

(3 comments)

> Why to NVIDIA specific. Where is the Arm and AMD love?

I'm not against supporting different kinds of GPUs, but that work hasn't yet been done.  If that is of interest to you or someone else, then I'd be happy to look at patches, but I don't think that should block this patch from landing, the same way that the existence of in-kernel OFED doesn't prevent a large number of systems using MOFED.

 > Also another issue is it can compete with CUDA application on a
 > compute node. The best way to handle such a future is really to
 > have two GPUs in a node. One for I/O and another for CUDA user land
 > applications. In this case we need a way to select which GPU to
 > use. This approach doesn't leave much of a choice.

I don't think anyone wants to dedicate GPUs for IO (IMHO), that is what the CPU is for, but as before I don't think future potential development should prevent these patches from landing.  This code has been tested extensively with GDS/CUDA, but of course if there are future optimizations then those could be discussed when they are available.

The code is structured to be minimally intrusive into the rest of Lustre and LNet so that there are not GPU-specific checks sprinkled throughout the code.  Other implementations of lnet_get_dev_prio() and lnet_is_rdma_only_page() unrelated to GPUs or NVIDIA are possible, e.g. NUMA->HCA affinity on large many-socket systems.
