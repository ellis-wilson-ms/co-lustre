--- Commit Message ---
LU-5895 lfs: prevent migration of a released file

If a file has been released, and a migrate operation is performed on it,
the file will be implicitely restored, then swapped, but this leads to
the following error:

 lfs: <file name>: data version changed during migration
 error: migrate: migrate stripe file '<file name>' failed

Before trying to migrate a file, check whether it is released and
prevent the migration in this case.

Patch Set 4:

We don't need to do OST migration for a released file if an admin is removing an OST, since the released file does not actually use any OST.  It would be desirable (in a separate patch) to be able to update the layout on a released file according to the migrate request.  Is there a ticket open for that yet?

Separately, the other question is what to do with MDT inode migration?  If an admin wants to evacuate an MDT in DNE (or e.g. migrate user subdirectories off MDT0000) I believe that the current code will block the migration because the FID will change, but we should figure out some way to handle this.  If we could update/deprecate the lhsm-posix tool and move to archive identifiers store in Lustre xattrs instead of Lustre FIDs stored in the archive, we could allow the migration to proceed.
