LU-16138 kernel: change rq_integrity_vec to respect the iterator

Add the upstream kernel fix cf546dd289e0f6d2594c25e2fb4e19ee67c6d988
to fix the integrity metadata buffer corruption seen on some nvme
devices.

This kernel patch is optional, the bug exists from rhel9.1 to rhel9.5.
The fix was backported since rhel9.6.

This bug only affects us when:
1. the nvme device has T10PI integrity support, and
2. the nvme device reports a max_segments smaller than BIO_MAX_VECS
(currently 256), which will make the bio split.

I took a closer look and limiting the segments/number of bio_vec is only part of it,
bio_split could still happen if bio size is bigger than max_sectors from the queue limit as well.
It still feels wrong to construct the bio in osd and make sure it won't get split: upper layers should be able to submit a large bio and it's up to block layer to do the splitting if necessary, osd should not worry about bio splitting.

Also this kernel patch is an upstream fix, unlike the T10-PI patches removed previous which are custom and never going to make upstream, and it only affects rocky9.1 to 9.5, due to the changes of multipage bio_vec.

I marked this is just optional because the bug triggers when the nvme device supports T10PI *and* it's max_segments smaller than 256, in real world deployments we are more likely to see
devices with max_segments way over 256, like 2048.
