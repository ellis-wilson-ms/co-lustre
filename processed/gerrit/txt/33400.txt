LU-11548 llite: increase readahead default values

It is commonly recommended to increase the readahead tunables
for clients to increase performance, since the current defaults
are too small, having been set several years ago for slower
networks and servers.

Increase the readahead defaults to better match values that are
recommended today:
- read_ahead_max_mb increased from 64MB to 1GB by default,
  or 1/32 RAM, whichever is less
- read_ahead_per_file_max_mb is increased from 64MB to 256MB,
  or 1/4 of read_ahead_max_mb, whichever is less

Modify the constant names to better match the variable and /proc
filenames.

Fix sanity test_101g to allow readahead to generate extra read
RPCs, as long as they are the expected size or larger.

--- Message (Patrick Farrell, 2019-01-11 16:45:18.000000000) ---
Patch Set 1: Code-Review+1

I ran these #s by a Cray performance/benchmarking guy, and his comment was basically:
These are the values he would set for maximum single client performance during an acceptance, but that they are probably too high for a large system & would get tuned down a bit lower later.

He likes the change (as do I).  Just a nugget to consider re: conflicting priorities.

--- Message (Patrick Farrell, 2019-01-11 21:07:39.000000000) ---
Patch Set 1:

Hmm, actually got a further note with a different opinion from the performance guy I mentioned:
"I would like something less aggressive for the global readahead setting. I think 256MB seems reasonable for a default as a middle ground. I get that 64 can’t always drive an OST even with a handful of clients. I’d prefer the single file size be equal to the global size though, i.e. 256MB, there are plenty of single process I/O server type workloads. Perhaps there’s a reason not to set them equal that’s not obvious to me. Hopefully the RA requests are balanced and one file doesn’t starve out others in a many-process workload."

I have no *strong* feeling here, as people can always tune, but if I were picking it, I'd go with the suggestion of 256 MB.  Anyway, either is better than the current settings.

--- Message (Andreas Dilger, 2019-01-12 07:04:50.000000000) ---
Patch Set 1:

My reasoning behind keeping the per-file RA lower than the total RA was to avoid the case where one thread of a multi-threaded process starts readahead slightly sooner, and then consumes all of the readahead credits for the node. That means one thread will have a large imbalance in terms of data read, while the remaining threads are totally blocked before they get any data at all.
