--- Commit Message ---
LU-9680 obdclass: user netlink to collect devices information

Our utilities can report to users a device list with various bits
of data using the debugfs file 'devices'. This debugfs file is
only by default available to root which prevents regular users
from collecting information. Enable non-root users to collect
the same information for lctl dl using netlink. The advantage of
using netlink is that it also removes the 8K ioctl limit. Add the
ability to present this data in YAML format as well.

Patch Set 11:

Well it depends on the situation. sysfs is perfect for simple configurations settings and can be used to send events to udev to be notified of changes. This is the kind of thing admins love to script and monitor

Now for more complex things like this using sysfs is not the ideal solution. Upstream the more complex solutions were moved to debugfs instead which is not what for.  The reason for this is the same why we use procfs. ioctls can only return limited size of data (32K in newer kernels) vs unlimited return size of procfs/debugfs. Using ioctls in this case works because we return a single data item which requires us to repeatedly call the ioctl. If we really want to move away from debugs/procfs for complex data the best solution is netlink which doesn't have the size limitation. Also netlink works way better with I/O forward systems than ioctls. That is an approach to consider for some later work.

Patch Set 51:

This is still a work in progress. As you can see with the follow on patch most of the netlink handling is now internal to libnetlink library itself. I still need to make the netlink callbacks generic with YAML handling. 

To make this possible we have to handle the key value format of YAML with the netlink packet. So a few approaches can be done. 

1) expand kernel netlink could to create the key names in the packet. This can 
    increase the network traffic size and we have to paste together to data entries 
    into the one YAML string.

2) Make all netlink values created in the kernel proper YAML strings. The trade off 
     here is that we box in netlink handling to do only YAML. Also if we push this 
     YAML <-> netlink handling to libyaml itself they might not like this approach.

3) Create a yaml emitter write handler that caches the layout and when data is 
     received back use the input emitter format to create the key names part of
     the YAML string.

I personally lean toward 3 but feedback is welcomed.
