LU-9789 obdfilter: add cluster ID to JobID

Add option to jobid_node string for a cluster ID

	%n = nodemap

This uses any nodemap settings that have been setup
to determine the local cluster name based on the local nid.

Change nodemap to store the configuration on each client,
when the configuration is sent, check to see if the local
nid matches, and set it for the jobid to use.

Patch Set 2: Code-Review-1

I'm not really a fan of exporting the nodemap to the client - they shouldn't really know about what nodemap they are part of, and this may also change as the system is running.  This also has the problem that no existing systems would have the nodemap name stored in the client config log.

As an alternative, you could add server-side replacement of "%n" in the JobID, similar to what was done in the original https://review.whamcloud.com/29150 patch?

That implies that we might consider to pass unknown "%x" formats through to the server, which I had in the very original version of my patch, but I removed it at the last minute.  Alternately, we might specify a "jobid_extra" (or something better) on the servers that allows "%m" and "%n" and other future parameters to be added to the JobID, similar to how "jobid_name" is handled today.  We could allow "%j" to be used for the client-supplied JobID, so that users have complete flexibility in how the resulting server-side JobID is presented (prefix, suffix, '-' or '.' separator, static string, whatever).  The "jobid_extra" on the server should default to "%j" so that it just copies the client-supplied JobID as-is, to match existing behavior.

Note also, per my comments in https://review.whamcloud.com/29150 that you should use "%m" for "nodeMap", since I'd like to keep "%n" for the client NID.

So, I don't see how this is any different than the previous version I reviewed?  If the nodemap is just a string as set on the client via set_param at startup, then this could just as easily be achieved by setting the jobid_node to statically hold "nodemap_name%..." instead of going through the nodemap infrastructure.

IMHO, the value of adding the nodemap is on the server side, since the server knows the nodemap that the client belongs in (virtualized/containerized clients may not get this info).

Also, there is no test for this functionality to know it is working properly.
