LU-11760 ofd: limit num of objects to create in 1 transaction

Set flag th_sync when the number of objects to create per
sequence reaches OST_MAX_PRECREATE in one transaction.
It is needed to avoid gaps after OST failover.
See details in LU-11760.

> this looks like a workarond, IMO. we have a mechanism to track

 > differentiate lost transaction vs corrupted LAST_ID ?

I'm not sure what you mean?  The 5x OST_MAX_PRECREATE check in the existing code is meant to handle the case that the MDT lov_objids and OST LAST_ID are corrupted or lost offline, or the MDT or OST is restored from backup with a very different value than the OST expects.

The goal of the check is to ensure that the OSS doesn't precreate millions of objects at startup if it gets a request from the MDS where the lov_objids value for MDT last_objid is much larger than the OST LAST_ID.

However, because OSTs are now very fast and in some cases can precreate more objects than 5x OST_PRECREATE before there is a journal commit, which triggers the "don't create more than 100k objects during recovery" and potentially loses data (though I also believe that the OST will handle the case when there is a write to a missing object.

I don't see how the transno can help in this case, since not every transaction is a create>

It's true that the MDS could also throttle precreate RPCs if it notices that the number of OST objects is too high.  However, this doesn't help the OST, which is also trying to avoid the case where there are RPCs sent from the MDS that don't make any sense to the OST (e.g. recreate 2M objects during recovery if the MDT or OST are corrupt).  Either they need to negotiate an object limit at connect time, or the OST need to prevent this from happening in the first place.

--- Message (Andreas Dilger, 2019-07-09 17:49:14.000000000) ---
Patch Set 3:

Alex, I'm not sure what you mean by your comment "then what's the point to assert?"

There is no assertion in the patch, it is just causing the object precreates to commit in the rare case that 500k objects have been created without a transaction commit.  That should be very unlikely in real workloads, since the clients will be doing some IO to the files those objects were allocated to in between the object precreates.

This is only really a problem for synthetic workloads like "mdtest" that do not do any IO to files, and I think it solves a real issue and should be landed.

For the longer term solution, I would be happy to see a patch that allows the MDS and OSS to negotiate a larger ost_max_precreate value at connect time via a new OBD_CONNECT2_MAX_PRECREATE flag and ocd_max_precreate field.  For example, the OST could track the maximum number of objects precreated in a single transaction, then save this in a new field in the last_rcvd file, and send 5x that number to the MDS at the next connect.  We could also add a tunable parameter that is read/written to last_rcvd if it is larger than the current value.
