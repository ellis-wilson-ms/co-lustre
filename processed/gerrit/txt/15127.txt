--- Commit Message ---
LU-4865 zfs: grow block size by write pattern

This patch grows the block size by write RPC. The osd-zfs blocksize
used to be fixed at 128KB, which is too big for random write and
too small for seqential write.

This patch decides the block size by the first few RPCs. If the first
few RPCs are sequential, mostly it will use bigger block size;
otherwise, smaller block size will be used.

I think this change is more on space consumption wise instead of performance wise. Back to your question, if the file was initially written by non-sequential write, the underlying blocks will be non-contiguous so the we'll get bad reading performance anyway. In that case, my patch chooses to use smaller block size, which is the minimum contiguous size from the first few write RPC, to reduce the overhead of COW.

--- Message (Brian Behlendorf, 2015-06-03 21:37:26.000000000) ---
Patch Set 1:

To my way of thinking this change is all about performance.  The block size may only be increased as long as the total file size is still less than a single block.  Once multiple blocks have been written your locked in to the block size.  So if you end up creating a multi-gigabyte file with a tiny block size your performance will suffer.

Also just because the blocks came in non-contiguous that doesn't means they'll be non-contiguous on disk.  If they happen to have ended up in the same TXG, which isn't unlikely, the will be ordered and written next to each other on disk.

It's in general difficult to guess the right block size to use due to lack of sufficient data - as you said, the block size will be locked up once the 2nd block is written. So my policy is if it looks like sequential, then choose big block size; otherwise, then choose smaller one to avoid CoW cost.

Please also notice that the maximum block size should be determined by RPC size. Otherwise, assuming the RPC size is 1MB, the initial sequential writer might extend the block size up to 16MB, which will cause the following 1MB-writer to read 15MB from disk to overwrite 1MB data.

This is a good discussion, I will revise the patch later.

--- Message (Brian Behlendorf, 2015-06-04 17:02:55.000000000) ---

(4 comments)

Just FYI at the moment the latest ZFS code will not let you set the block size higher than 1M.  This is controlled through the 'recordsize' property.  If you want to use 16M blocks you'll need to increase the zfs_max_recordsize module option.  There's a good chance we'll relax this in the future but we want to get more comfortable and understand the performance characteristics of much larger blocks first.

As for looking up a specific object number you can use 'zdb' to dump the contents of an OI.  So if you have the FID for the object and you know what OI it's in, you could run 'zfs -vvv lustre-ost1/ost <OI object>' and grep/awk out the <FID>=<Object ID> you need.

Alternately you could mount a snapshot of the ost1 via the ZPL.  The and OIs will be visible as directories, and the file names in those directories with be the FIDs each of which refers to the object.  So I'd expect something roughly like this to work.

zfs snap lustre-ost1/ost@debug
mount -t zfs lustre-ost1/ost@debug /mnt/debug
find /mnt/debug -name FID -exec stat {} \;
umount /mnt/debug
zfs destroy lustre-ost1/ost@debug

The inode number in stat output will map directly to the object number.

--- Message (Brian Behlendorf, 2015-06-16 16:38:11.000000000) ---
Patch Set 3:

The maximum block size supported by the on-disk format is 16MB.  In practice the zfs 'recordsize' property which is set on a per-filesystem/volume basis is used to limit this to some smaller power-of-two value.  128K is the default, 1M has been shown to strike a good balance, 16M is the hard maximum.

The posix layer will grow the block size up to the value specified by the 'recordsize' property.  In practice this has worked very and it's the interface users will expect.  For example, if they know an OST will only store small file they could set the record to be small and do the opposite for large files.

I think the point Alex was making is that Lustre should behave the same way the posix layer.  The block size should be grown solely based on the object size.  I tend to agree, let's not over complicate this when we have a simple solution which is already known to work well.
