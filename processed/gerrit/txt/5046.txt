--- Message (Andreas Dilger, 2013-01-17 17:26:04.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

The change in OSD_EXEC_OP() make the debugging there totally useless and just pure overhead. I believe it may also cause assertion failures on ZFS. If there are cases where there can be rollback operations on an inode, then those operations should also be declared. This might introduce some overhead due to double reserving blocks for the same inode or directory, so two possibilities exist:
- add a method to declare an operation on an inode that has already been modified in this transaction and not consume extra credits (e.g. With a "rollback" flag that indicates to also declare the reverse operation)
- internally cache the inode number that has already been declared, and avoid accounting for it twice

The second option seems desirable, since it would also reduce the transaction credits for normal operations (e.g. rename within same directory, or multiple setattr on the same inode). 

In sanity.sh, the change to access osd-ldiskfs will break interop testing. Please use {get,set}_obdfilter_param() to handle the different parameter names between branches.
