LU-14535 quota: get all quota info in LFS

This patch adds option "-a" for LFS to get the quota info of
all quota IDs. it iterates quota setting saved in global quota
setting files "quota_master/md-0x0" and "quota_master/dt-0x0"
from QMT and iterates the quota usage info saved in acct quota
files in the backend FS (LDiskFS or ZFS) from QSDs, then merge
the two kinds of quota info at client and print it in the similar
way as "lfs quota -u|-g|-p".

the rw_semaphore "qmt_pool_info.qpi_sarr.osts.op_rw_sem" has been
acquired in qmt_pool_recalc (read mode), but it was acquired once
more in qmt_seed_glbe_all (read mode) and will be stuck if there
is a pending write mode lock acquisition from another thread.

(2 comments)

You can't pass that amount of data back through an ioctl. ioctls in Linux only have a 32K buffer limit per call. The way Lustre normally gets around this is by calling the ioctl over and over again with an incrementing index argument until an error happens. See OBD_IOC_GETDEVICE for example. Another option for dealing with this amount of data is by using Netlink.

--- Message (Andreas Dilger, 2021-03-20 07:52:53.000000000) ---

(18 comments)

It isn't quite clear to me how this all works, but a reasonable model to follow is like FIEMAP and readdir.  

Userspace supplies a buffer with a header that describes how many entries fit into it, and the starting/ending range of quota IDs, types, etc. Typically this might be "[0,~0]" to get all records, but maybe allowing a sub-range would be useful. 

 The kernel replies with a filled in buffer that has not more than the requested number of entries, and the count of entries returned.  If multiple types are requested (u,g,p) then they should be in a specific order so that there is a clear way to restart the iteration.

If we think it makes sense to print eg. u=1000, g=1000, p=1000 together because they are likely to be related, then they should be returned together, and the kernel should not put a partial set of records into the buffer if there isn't enough room for all of them. That would make restarting easier, since it would only need the id number to restart. 

If more entries are needed, userspace calls again with the header having the starting range at one more than the last record returned (and the type of the last record plus any later types, if needed).
