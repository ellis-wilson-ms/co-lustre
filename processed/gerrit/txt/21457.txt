LU-8384 scripts: Add scripts to systemd for EL7

When rebooting a lustre client where Lustre filesystem is still
mounted, the shutdown hangs. This patch create a systemd service
that unmount the Lustre filesystems and unload the Lustre modules
when system is shutdown.

Patch Set 1: Code-Review-1

(1 comment)

I think that we need to stop just dumping new files in the scripts directory.  We need to have a little more directory structure, and separate the sysv files from the systemd files.

We should also only package and install the systemd files on systems that use systemd.  Likewise for sysv files.

Next, I don't think that this is the correct approach for client side mounts.  systemd uses ".mount" units for mount points controlled by systemd.  I think the approach in this patch of making a fake .service is a little to hackish.

--- Message (Christopher J. Morrone, 2017-01-21 17:30:37.000000000) ---
Patch Set 1:

I'm not suggesting that the Lustre packages introduce .mount services.  Instead, I am suggesting that managing mounts of any kind is the job of the sysadmin through fstab or systemd .mount files.  Thats just standard practice.  Introducing a fake .service as in this patch just isn't proper use of systemd services.

It would certainly be nice to treat _lnet_ like a network service, and have a systemd init script that genuinely starts and stops lnet, and can be used to express ordering dependencies between lnet and lower level networking interfaces.  We have just an lnet systemd service file under way here at LLNL.

Longer term it might be nice to treat Lustre _servers_ like other sorts of servers.  Some folks might want a start the Lustre servers from a systemd service.  But there there is long standing foundational work probably needed to make that work well.  Lustre client mounts and server mounts should probably have different file system type names, for instance.  Ideally starting a lustre server wouldn't be through the "mount" command at all to make it more like starting other servers.

But then again, most people that run servers are probably going to want fail-over (I assume), so an OCF Resource Agent script for use by pacemaker is probably the more important demand to start/stop of servers under EL7.  That is certainly how LLNL is doing it, and I know that Intel has some work in that area too.
