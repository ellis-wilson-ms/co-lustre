LU-19179 o2iblnd: sort out map_on_demand usage

In older kernels map_on_demand was used to limit the number of
RDMA fragments for global memory regions. That approach was
removed in modern kernels so the meaning of map_on_demand changed
to act as a switch to select between FastReg / FMR or the global
memory regions. Even more recent kernels removed the global memory
regions so map_on_demand should always be 1 for all cases. In
reality map_on_demand is an obsolete parameter.

Since we need to support an older configurations that a user might
push update the code to behave properly. For the latest kernels
just ignore the user setting any value and always internally user
the default map_on_demand value of 1.

Starting with 4.9+ kernels global memory region support was dropped.
We can now safely drop global memory regions support.
