LU-1951 mdd: fix for error handler of mdd_rename

If mdd_rename() failed to unlink target file/dir, it will try to
revert everything including insert target file/dir back into target
directory, but it didn't restore nlink count of target, which will
leave a file/dir under target directory with wrong nlink number.

Another thing is fixed by this patch is, mdd_attr_check_set_internal()
didn't release mdd_write_lock() while jumping to error handler.

--- Message (Andreas Dilger, 2012-11-01 16:50:05.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

Three changes that should be made to this patch:
- tp_ref should be changed to a bool type, along with is_dir
- a new tp_locked bool should be added there as well, to track whether the dir is still locked or not during cleanup, instead of unlocking in the error handler
- all of the console CERROR() messages should start with the device name and end with the error code so that it is possible to identify the failing device when there are multiple MDTs on the same node, like:

        CERROR("%s: some error message: rc = %d\n", ...);

The reason to not unlock in the error handler is twofold:
- it simplifies the error handling by not repeatedly doing the unlock
- it avoids potential race conditions after dropping the lock in the error case that some other process will begin using this object before it is relocked during the cleanup
