--- Commit Message ---
LU-17940 gss: get rid of root key in all cases

The root key associated with a GSS context (gck_key) is used to pass
information between kernel and userspace during GSS context
negotiation.
Whether the GSS context negotiation went well or not, the context and
the key used in this process should be unbound once done. And this
should mean unlinking the key but also directly invalidating it
instead of just revoking it, to make sure the key is ignored by all
searches and other operations.
For the same reasons, invalidate the key when the GSS upcall times
out or the context pre-initilization fails.

--- Message (Sebastien Buisson, 2024-07-02 13:37:51.000000000) ---
Patch Set 3:

> Patch Set 3:
> 
> Unconditionally unlinking the root key seems rather good.
> 
> But why is invalidating better than revoking? Or said otherwise, when is it appropriate to revoke versus invalidationg? 
> 

Here are the comments in the kernel sources associated with invalidate:

   /**
    * key_invalidate - Invalidate a key.
    * @key: The key to be invalidated.
    *
    * Mark a key as being invalidated and have it cleaned up immediately.  The key
    * is ignored by all searches and other operations from this point.
    */

and revoke:

   /**
    * key_revoke - Revoke a key.
    * @key: The key to be revoked.
    *
    * Mark a key as being revoked and ask the type to free up its resources.  The
    * revocation timeout is set and the key and all its links will be
    * automatically garbage collected after key_gc_delay amount of time if they
    * are not manually dealt with first.
    */

Once the GSS context negotiation is done, we really need to get rid of the key ASAP, so that the next GSS context negotiation will use a new key to pass information between kernel and userspace. This is why invalidate is more appropriate than revoke.

> After this patch, only 2 places remain where key_revoke_locked() is called:
> - gss_kt_revoke() is called with the flag already set actually

Indeed, gss_kt_revoke() gets called by kernel's key_revoke(), which sets the flag beforehand. So the call to key_revoke_locked() is useless here. 

> - flush_user_ctx which is calling kill_key_locked() just before.

In case of a context flush, I prefer to leave the call to key_revoke_locked() because of this comment:

   /* kill_key_locked() should usually revoke the key, but we
    * revoke it again to make sure, e.g. some case the key may
    * not well coupled with a context. */
