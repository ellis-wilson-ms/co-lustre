LU-14516 mgc: configurable wait-to-reprocess time

so we can set it shorter, for testing purposes at least. to change
minimal wait time MGC module option 'mgc_requeue_timeout_min'
should be used (in seconds). additionally a random value upto
mgc_requeue_timeout_min is added to avoid a flood of config re-read
requests from clients. if mgc_requeue_timeout_min is set to 0,
then random part will be upto 1 second.

ost-pools: before: 5840s, after:a 3474s
sanity-flr: before: 1575s, after: 1381s
sanity-quota: before: 10679s, after: 9703s

(1 comment)

looks good.  it makes me wonder if this should be auto-tuned in some way based on the number of clients, but since it is a client-side tunable that is more complex.

There are a number of other places where the client knowing (roughly) how many clients there are would be useful (e.g. max_dirty_mb, default timeout, etc.), so I wonder if sending the current client count in some reply would be useful.

Using obd_connect_data isn't great, because clients that mount first will have the wrong number of clients, and this is rarely updated.  Using OBD_PING could work, or OBD_STATFS (though it isn't clear if that is called regularly on compute nodes).

In this case, it _might_ be possible to piggy-back the client count in the config lock cancellation callback, since that would be sent _just_ at the time it is needed, but it would be useful to have this information at other times as well.
