LU-181 obdclass: fix portal_handle memory wastage

This patch is the first step of fixing memory wastage in Lustre.
We can save 20 bytes for each portals_handle(on 64bits system)
by reusing members, which could be over a hundred megabytes
on system with millions of ldlm_locks because portals_handle is
embedded in ldlm_lock.

Patch Set 2: I would prefer that you didn't submit this

(3 inline comments)

This is a good fix, and I think we can probably do a lot more similar changes to reduce the size of ldlm locks (in separate patches of course).  I think with a minor change of struct packing it would be possible for this change to save another 8 bytes in all of the structs that are using portals_handle.


I'd be thrilled to see similar improvements in later patches for other commonly used structs (e.g. ldlm_resource, ptlrpc_request, ptlrpc_reply_state, lu_object_header, cl_page, cl_lock, etc).  An excellent tool for this is called "pahole" (from the "dwarves" RPM package) that examines structures for packing issues.  I found the following quite useful:

  pahole -RS -H2 lustre/obdclass/obdclass.ko
  pahole -C ldlm_lock lustre/obdclass/obdclass.ko

It would be great to get struct ldlm_lock below 512 bytes so that it can pack into page-sized allocations more efficiently.  Other important structures may also see some benefits from similar shrinkage.
