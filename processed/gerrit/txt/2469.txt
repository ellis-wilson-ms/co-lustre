Patch Set 3: I would prefer that you didn't submit this

My concern is that: according to your patch, if the client cannot connect to one OST for the initial connection, then it will wait for one obd_timeout; if it cannot connect to another OST, then will wait for another obd_timeout; so the max wait time may be N * obd_timeout. For very large cluster, such case may be not rare, should we consider to avoid waiting for long time?

On the other hand, please replace "space" with "tab" in your patch as required in Andreas email.

--- Message (Andreas Dilger, 2012-08-04 07:34:14.000000000) ---
Patch Set 3: I would prefer that you didn't submit this

(1 inline comment)

Except for the minor code style issue, I like this patch a lot.  It will probably solve many problems that we have hit in testing in the past, where the client would hit an inactive OSC during initial mounting, and then return an error.  We worked around this problem in the past by adding delays in the test scripts, but that isn't going to fix the problem in real usage like this patch should.

I agree with Fan Yong's concerns about waiting serially for each initial connection, but the main question is whether this will happen in real life?  If the client tries to connect to all of the OSCs in parallel, they should all either connect or fail at the same time, and with this patch the client would wait for the first OSC to connect (up to obd_timeout, but re-checking every 1s interval), and after that the other OSCs should have already connected or failed.

Presumably, the client will not wait the full obd_timeout each time it tries to do an operation on an OSC that is not currently connected, but rather only on the initial connection to the OST?

--- Message (Andreas Dilger, 2013-11-13 17:21:52.000000000) ---
Patch Set 9: (2 inline comments)

I don't think the performance impact is as bad as one might initially think. The connections will be happening in parallel, so the imp_connection_tried flag should be set on all of the exports after a single timeout. If the client doesn't send any RPCs immediately after mount, then it is likely that it will already be set by the time if the first check, and no waiting should happen at all.  Have you actually seen problems with this patch under real usage, or is your objection only based on code inspection?  It would be more useful to have comments to improve the patch while the it is under development, instead if 6 months after it has landed. 

On the other hand, this avoids a problem that is often seen in testing, and I imagine in real life also - that the application gets errors when trying to send an RPC to a target that is not fully connected yet.

Shadow, I agree the layering isn't great.  Upcalls are not much better, then ptlrpc layer will know about LOV target structure. I'm happy if you submit a patch to improve it. We will not be reverting the patch, but an improvement patch could be landed on top of it.

--- Message (Jeremy Filizetti, 2016-11-16 22:19:44.000000000) ---
Patch Set 3: I would prefer that you didn't submit this

After testing this patch for a while I agree that the serial waiting isn't ideal.  I think it should wait a maximum of 1 * obd_timeout for all targets since operations are happening all concurrently to connect/reconnect.  

With the default obd_timeout as 100 seconds waiting even that long can be a burden especially since our desire with this bug was to enable the automounter for use.

The patch doesn't prevent repeated waiting on the OSC either, because imp_remote_handle never seems to get set to non-NULL.
