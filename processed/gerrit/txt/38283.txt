LU-13131 osc: Do not wait for grants for too long

obd_timeout is way too long considering we are holding a lock
that might be contended. If OST is slow to respond, we might
get evicted, so limit us to a half of the shortest possible
max wait a server might have before switching to synchronous IO.

(2 comments)

Now that we understand the failure mechanism, it may be easy enough to add a test case to sanityn.sh for this, possibly even with an existing fault injection hook. 

Set the bulk RPC processing timeout to wait, say, 3/4 or 7/8 (or maybe 5/4) of obd_timeout. Start sync writing to a striped file from one client, until all of the RPCs from that client are blocked/slow on the OSTs. Then, write from the second client to the same file, which will cause lock callbacks.  This should hopefully get us into the situation you observed, and without this patch the first client will be evicted, but will not if the patch is applied.
