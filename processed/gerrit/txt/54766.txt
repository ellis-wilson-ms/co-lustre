LU-17736 tests: Fix sanityn 104 for test machines with auditd

getfattr performs one stat followed by two getxattr syscalls against
the provided file.  Normally, the stat results in no getxattr calls
internally (as it's not something stat is required to return).
However, if auditd is enabled AND one of the rules includes a
filesystem-specific rule such as watch directory X and record if it's
modified, then for every lookup (each of the three syscalls includes
one) an additional getxattr will be performed, resulting in 5 total
getxattrs.  Because there is significant fuzz here, revise the check
to be at minimum the two "expected" getxattrs but allow for more.
Comments have been added explaining this.

I didn't see the comments here before I commented on the other patch (too many emails in my inbox).  Some things are worthwhile to clarify for the future:

 >  I made the updates, git rebased and squashed down to a single commit
 > git rebase --interactive HEAD~2 # mark bottom-most with squash

Since the second commit will have a new Change-Id then you need to be careful which one to keep, and it will be tempting to keep the one that is at the bottom of the combined text, but that would be wrong.

I find it *much* easier to make edits directly on the affected patch, then use "git commit -a --amend" so that those changes are directly merged into the previous commit (which will preserve the commit message and its Change-Id, but give it a new commit hash, as any change does) rather than creating a separate new commit.  That avoids the overhead of the "git rebase" step and merging the two commit messages (though you still get an chance to edit the text if you've made some substantive change).  

If you are confident of your changes you also can add "--no-edit" which will just keep the previous commit message without any interaction at all.
