--- Commit Message ---
LU-10177 lfs: support DOM-to-OST migration

Migraton of DoM file to an OST file is possible with
'lfs migrate' command in general but would cause space loss
on MDS, because DOM local inode blocks are not deleted.
Patch doesn the following:
- adds truncate for local MDS inode after migration.
- check and prevent migration from non-DOM to DOM file,
  that will be done later with mirroring.
- prohibit 'lfs swap_layouts' command if either file has
  DOM component, 'lfs migrate' should be used instead

--- Message (Andreas Dilger, 2018-08-11 17:48:54.000000000) ---
Patch Set 2: Code-Review-1

(5 comments)

The FID ordering check is not reliable. If we can only check this reliably on the client, then I guess that is what we have to do. It isn't clear to me why we swap the FIDs on the client instead of only the server?  In any case, better to have extra denial on the client in the future when MDS allows this. 

The one other concern is that swap from DoM component to non-DoM component requires a full-file copy to migrate the ~1MB of data that will be stored on the MDT. It would be better if such a migration were handled by using FLR resync method to open just the first OST stripe and write MDT data there, then do the DoM truncate.  This would also allow DoM+FLR to work together. 

I don't insist on this for 2.12, since I'm happy if we can get any DoM migration into 2.12, but something to think about for the future.

(2 comments)

You are right that the need for allowing DoM size changes is not as strong as removing the DoM component from a large file. That is why I'm interested in this first patch to land with the limited functionality now, and we can add those other abilities later. 

There are a few cases where we would want to be able to change the DoM component.  With FLR it may be useful to have a small flash write mirror component at the start of the file for an index, icon, or other start-of-file header information. I think the biggest benefit may be that users don't need to pay as much attention to what they are doing, and it will still work properly, rather than getting errors back and they can't do what they want.
