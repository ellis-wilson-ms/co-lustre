One question that I have about this patch, which might be answered by a better commit comment and/or comments in the code, is what happens in the original case of restarting the IO because the lock was cancelled due to shared IBITS locks?  The point of the check in can_populate_pages() was to restart the IO from the beginning if the objects migrated to new OSTs so that the in-flight IO was not lost (it is the responsibility of the migrator to copy the rest of the file data).  It looks like this case is still OK (vvp_io_init() and vvp_io_fini() are still checking if the layout generation actually changed.

What seems to be missing (or maybe I just don't see it) is restarting the IO when the LAYOUT lock bit is cancelled _without_ the layout actually changing (which I think was the original cause of LU-6389).  In that case, the client will re-fetch the layout lock, maybe again combined with UPDATE, or maybe separate.  I don't think io->ci_need_restart will be set (nor should it, because the whole IO doesn't need to be restarted, only a continuation of the current IO).  How does this patch address this case?
