LU-19079 sec: update nodemap membership on flavor change

When a security flavor is updated on an export (e.g. from null to skn)
the client needs to refresh or create a gss context. Now that we have
GSS based nodemaps, it is useful to lookup the nodemap entries when
this happens, to see if the client now belongs to a different nodemap,
based on its gss context.

Add sanity-sec test_105 to exercise this feature.

## Summary

### The Problem
The patch introduced compile errors because it tried to access `req->rq_svc_ctx->sc_nodemap`, but the `struct ptlrpc_svc_ctx` doesn't have a member named `sc_nodemap`.

### The Root Cause
The nodemap name is stored in the `gss_rsi` structure (in the `si_nm_name` field) during GSS initialization, but this information is **not** transferred to the `gss_rsc` or `gss_svc_ctx` structures. The `ptlrpc_svc_ctx` structure also doesn't have any nodemap-related fields.

### The Fix
Instead of trying to access a non-existent `sc_nodemap` field, the fix:

1. **Added the necessary include**: Added `#include <lustre_nodemap.h>` to get the definitions for `LUSTRE_NODEMAP_NAME_LENGTH` and the `nodemap_test_nid()` function.

2. **Used `nodemap_test_nid()` to get the nodemap name**: This function determines the nodemap name from the client's NID (Network Identifier), which is the same approach used in `gss_svc_upcall_handle_init()` during GSS initialization.

3. **Passed the nodemap name to `nodemap_member_switch()`**: The nodemap name obtained from `nodemap_test_nid()` is then passed to `nodemap_member_switch()` to update the export's nodemap membership.

### Rationale
This approach is consistent with how nodemap names are determined elsewhere in the codebase. The `nodemap_test_nid()` function looks up which nodemap a client belongs to based on their NID, which is exactly what's needed when a security flavor changes and we need to re-evaluate nodemap membership based on the new GSS context.

**Rationale:**

This field corresponds to the `NM_FL2_GSS_IDENTIFY` flag that already existed in the `enum nm_flag2_bits`. However, the actual boolean field in the `struct lu_nodemap` was missing. 

The fix adds this missing field following the same pattern as other similar flags like `nmf_fileset_use_iam`, `nmf_deny_mount`, and `nmf_readonly_mount`. The field is properly integrated into both the serialization (writing nodemap config to disk) and deserialization (reading nodemap config from disk) code paths to ensure the flag is persisted correctly.
