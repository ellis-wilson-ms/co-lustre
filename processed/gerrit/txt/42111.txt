--- Commit Message ---
LU-14536 obi2lnd: don't try to reconnect if there's no listener

For each discovery we try to reconnect up to retry_count times,
default to 5. during MDT mount process conf log, there will be
multiple discovery made for each OST.
If the OSTs are not up, the mount will have a long time out.

It looks from the comment that with 42111 the mount improves by 6 seconds? Is my understanding correct? I didn't get the 30 minute comment. Was the 30 minute mount time without 42109?
My concern with this change is that in some conditions, when the nodes are coming up simultaneously there will be 0 retries on connections which otherwise would've succeeded. Say in the case when the a node is just coming up and you try to connect, but the listener isn't there yet.
This patch prevents retries completely. While there currently exists a tunable to set the retry count to 0 (or less than 5)
Now I understand this is a narrow race condition, but I'm unsure of the impact of this patch generally. Would it make sense for the affected site to simply set the number of retries to 0?
I'll +1 this for now, but I'm unsure if it'll cause other problems.
Chris, what's your take on this?
