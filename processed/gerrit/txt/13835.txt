--- Commit Message ---
LU-6245 libcfs: create userland and kernel string operations

Additonal string handling and NID string parsing code are both
used by kernel space and user land. This prevents us from
moving forward for cleaning up the userland and kernel space
headers. With the code duplicated for both environments we
can then clean up the headers independently. Since NID string
handling is only done for LNET we move it there.

> I did some digging to sort this out and discovered for all the kernels we support, SLES11 SP3 and up, they all define their __uXX with the uintXX_t types in linux/types.h. So uintXX_t is most definitely safe to use instead of the __uXX format. I figured this was the case since over the last few years I have seen the move to uintXX_t types in the kernel code itself.

In the 2.6.32 kernel the uintXX_t types are defined in terms of the __uXX types:

http://lxr.free-electrons.com/source/include/asm-generic/int-ll64.h?v=2.6.32
http://lxr.free-electrons.com/source/include/asm-generic/int-l64.h?v=2.6.32
http://lxr.free-electrons.com/source/include/linux/types.h?v=2.6.32

In userspace they are defined by libc any may not use the same underlying type. On my x86_64 CentOS 6.4 machine kernel/uint64_t != user/uint64_t.

> The reason I prefer to use uintXX_t is that some the data structures that are currently duplicated can be later moved to a user land header that can be used by both user land and kernel space.

If that header includes linux/types.h then it can still use the __uXX types and it can be user by kernel and userspace. linux/types.h and the __uXX types are designed for this purpose.

> Has a bug been filed to glibc since they have wrongly implemented uint32_t?

uint32_t is always 32-bits. The issue is that __u64 and uint64_t are sometimes unsigned long and sometime unsigned long long. Plus the fact that we have these stupid LPU64 type macros and other crap like pointer compatibility. linux/types.h defines the __uXX types for the purpose of shared user and kernel space data structures. So I think we should use that.

Also note that when userspace includes linux/types.h then the uintXX_t typedefs will be skipped. Instead userspace gets the uintXX_t typedefs from stdint.h.
