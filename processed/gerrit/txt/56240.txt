LU-18201 hsm: yield cdt_llog_lock in coordinator thread

The ccordinator thread holds cdt_llog_lock in write mode while
scheduling requests or updating records in the HSM actions llog; if
there is a large number of requests, other MDT threads that also need
to take the lock in order to e.g. mark an action as complete, add a
new action, or print the contents of the llog can be starved and get
blocked. Yield cdt_llog_lock whenever a tunable number of requests
have been handled to avoid this.

Thank you for the reviews and apologies for this late reply. As Andreas and Etienne have suggested, this patch is probably not thread-safe in all cases; furthermore, it doesn't really address the underlying issue of cdt_llog_lock contention. Taking a closer look at debug logs from field reports, I think that most/all of the occurrences of this issue are due to the coordinator updating HSM request state in batches, instead of one-by-one, which can cause a large amount of work under cdt_llog_lock and starve other MDT threads. There is a hash table, cdt_agent_record_hash, that aims to avoid scanning the llog unnecessarily, by mapping HSM request cookies to llog record locations, but its usage seems suboptimal, as a single set of an llog catalog and record index is used for the whole batch of requests in mdt_agent_record_update(); if the HSM request records in the llog are in discontiguous locations, the some of the llog will be scanned unnecessarily and since the number of records that can be updated in a single call to mdt_agent_record_update() can grow fairly large and the offsets between the records in the llog can also be fairly large, the overall amount of work required to perform the updates can become problematic.

We are working on a different set of patches to address this issue, that we are hoping to be able to submit soon, so I will abandon this patch.
