LU-8855 llite: return small device numbers for compat stat()

The compat_sys_*stat*() syscalls will fail unless the devices majors
and minors are both less than 256. So in ll_getattr_it(), if we are in
32 bit compat mode then coerce the device numbers in to the expected
format.

Patch Set 2:

Oleg, any comment on this?  I recall you worked on NFS export a fair amount in the past.

I agree the current patch is the easiest solution, and if you agree then I'll change my review.


We could use major=146 relatively safely for Lustre s_dev. This doesn't apply for rdev, which should just be truncated as the current patch does, and is unlikely to be the source of the problem.

I see that the NFS client mount is calling nfs_set_super->set_anon_super->get_anon_bdev() to allocate a device number from the major=0 space (which can handle up to 2^20-1 minor devices, though only 2^8-1 minors could pass through old_valid_dev()), but it appears after a fair amount of searching that the major=146 (=0x92) space is currently unused.

While using major=146 has some small potential to cause problems, it is at least intended to be used for this purpose, and it is less likely to have conflicts than just using a random major/minor device masked from the existing 64-bit device numbers.

Since we already have different inode numbers returned for 32-bit and 64-bit userspace, I don't see that having different device numbers is an issue - the inode identifier is already inconsistent between the two.  It probably would have been better to use the major=146 device space even for 64-bit userspace, but that is water under the bridge and we shouldn't change it now or it will introduce potential incompatibility between versions.  We might do this for a future 3.x release if there is some major protocol break at that time (worth a comment, and possibly a version check with a #warning so that is noticed?).

I agree it we are unlikely to have 32-bit du/find on 64-bit clients, but there is a possibility of 32-bit clients.  I guess the one reason to stick with the existing "truncate" approach is for consistency with real 32-bit clients.

Andreas, we also call set_anon_super, but unlike nfs we doit vi mount_nodev since we don't try to match underlying mounts to coalesce them.

The thing is we are setting our s_dev to some big value outselves to enable nfs export clustering:

        sbi->ll_sdev_orig = sb->s_dev;

        /* We set sb->s_dev equal on all lustre clients in order to support
         * NFS export clustering.  NFSD requires that the FSID be the same
         * on all clients. */
        /* s_dev is also used in lt_compare() to compare two fs, but that is
         * only a node-local comparison. */
        uuid = obd_get_uuid(sbi->ll_md_exp);
        if (uuid != NULL)
                sb->s_dev = get_uuid2int(uuid->uuid, strlen(uuid->uuid));


So I guess we can just use sbi->ll_sdev_orig when writign the sdev for userspace
or we can just cast and nothing bad would happen either.
Even if there is bdev clash for some luck, ther's also different fstype and and fsid in play so smart applications would notice
