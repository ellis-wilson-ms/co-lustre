--- Message (Andreas Dilger, 2018-11-27 21:29:48.000000000) ---
Patch Set 1: Code-Review-1

As I mentioned in LU-11706, I think "experimental" features is far too broad a bucket to put multiple features into.  The only thing I could imagine being useful is if there was a mask that filtered the OBD_CONNECT_\* features from being enabled by a client or server.

Having a separate per-feature enable/disable tunable (e.g. mdt.*.enable_remote_dir, mdt.*.enable_striped_dir, lod.*.dom_stripesize, mdt.*.dom_read_open, osd-ldiskfs.*.auto_scrub, etc.) allows enabling/disabling a specific feature for testing, and even long after the feature is no longer considered "experimental" it may be useful to disable it in the field if there is some problem found with it.

--- Message (Andreas Dilger, 2021-04-09 19:53:41.000000000) ---
Abandoned

I don't think this patch has any meaning/usefulness by itself.  Rather than have a blanket "experimental features" tunable (which is unclear what is "experimental", and that changes on a per-release basis, it is better that major features have a tunable that can adjust internal parameters and/or turn off the feature completely if needed. That has been the case for most new features (eg. DNE remote directories, DNE striped directories, DNE directory auto-split, idle_timeout, fallocate, etc).
