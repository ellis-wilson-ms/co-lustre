So based on the racer-long results we can see this is extremely effective way of making racer test pass. BUT! the obvious problem I see is this:

The racer test is demonstrating a real problem - when import goes down, processes that depend on them get stuck and no amount of kill signals is going to help (e.g. see here when we try to just normally kill processes at end of racer with fuser -k https://testing.whamcloud.com/gerrit-janitor/21422/results-retry2.html  - a nunch are stuck in various lock waits and revalidations).

Creating "we just do a special signal that only racer will do to make racer pass" is putting this problem on it's head. Instead of fixing the real problem, we just "fix the test". What I think we should do is we ensure when the import goes down whoever uses it errors out and cleans up no matter if it's part of racer or any production workload.

Am I getting it all wrong? Can you present a real compelling scenario why special-casing racer like this is a good idea?

Patch Set 6:

Here we aren't just fixing the test. We use kill properly with Lustre.
The issue with kill is following.
A client sends lock enqueue and waits for the reply. But it receives a kill signal before the reply and just drops the request. Everything works fine until the reply has BL AST bit set. There is no one to parse the reply and to cancel the lock. And the client gets evicted. 
The obvious fix it not to set BL AST bit in the reply, instead the server can send separate BL AST request and it will return an error as there no such lock handle on the client and the server will cancel it also. But it involes new request and can be slow.
So here special signal handling is used to deliver fatal signal to an application from one but not to abort lock handling.
Other fatal signals will terminate the syscall immeadiately but can lead to eviction

I agree with Oleg that this seems to be fixing *only* racer, but leaving the same problem in place for real applications that are doing the same thing, which result in the client being evicted when it shouldn't be.

 > yes, it is a known bug and it is at least 10 y.o. and I don't see
 > any reasonable way of fixing it.
 > On the other hand fatal signals shouldn't be used only in case of

Maybe I'm missing something, but can this patch be *exactly* the way to fix the BL AST problem?  Rather than restricting the patch to only racer, it could be changed so that *any* fatal signal (not just SIGUSR2) will "not interrupt RPCs in progress and shutdown <any> threads"?  That way the client is not evicted, but the user gets what they want (process exits from blocked syscall), and RPC will complete or time out in the background.

I realize that I previously asked for the SIGUSR2 check to be limited only to racer (OBD_FAIL_RACER_SHUTDOWN was added for this reason), but I thought the SIGUSR2 was being added for random fault injection (to improve Lustre testing), and not as a way to work around a bug in Lustre signal handling.
