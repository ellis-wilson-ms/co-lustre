--- Commit Message ---
LU-17048 mdd: protect layout change in MDD layer

We need to detect changes to the LOD layout in between transaction
declaration and when the objects are locked during transaction
execution. Otherwise, if another thread has modified the layout
of an object used by the transaction then the declaration may
be incorrect.

This patch save objects' layout generation in transaction delaration
phase, and check whether they have been changed by others in the
transaction execution phase, if that's the case, the transaction will
be retried for several times.

Fixes: b7bd4e3422 ("LU-14621 mdd: fix lock-tx order in mdd_xattr_merge()")

Patch Set 2: Code-Review-1

(10 comments)

Alex, I think this patch is undoing the lock ordering fix you landed in LU-14621, but it also seems important that the file layout is locked between preparing the transaction and starting it.

The only solution short of changing all of the lock ordering is to have a retry loop, which saves the layout versions (or important parts of the layout itself), then re-verifies them after the locks are held, and restarts the operation if they have changed.

I haven't dug into the details if this will cause more than an empty transaction handle (in the rare case that this is hit outside of racer), or if there are other side-effects (e.g. transno update, etc)?
