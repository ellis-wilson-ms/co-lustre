--- Commit Message ---
LU-2442 kernel: protect i_dquot with i_lock

Remove dqptr_sem (but kept in struct quota_info to keep kernel ABI
unchanged), and the functionality of this lock is implemented by
other locks:
* i_dquot is protected by i_lock, however only this pointer, the
  content of this struct is by dq_data_lock.
* Q_GETFMT is now protected with dqonoff_mutex instead of dqptr_sem.

--- Message (Jeff Mahoney, 2013-02-26 03:40:14.000000000) ---
Patch Set 5: I would prefer that you didn't submit this

(1 inline comment)

While I do believe that there is lock contention in the initialize case, I'd rather see performance numbers before actually implementing this change using i_lock.

As a first step, we can just replace the super-global dqptr_sem with a per-inode rw_semaphore and see how that works. Since it's trading a rw_semaphore for another, we don't need to deal with all of the other API changes until we see real benefit. Then we can look at the API changes. It may well be that the best way to do it, if we're already claiming a reference for each dquot struct before submission, to clone inode->i_dquot before dirtying and then we can avoid the silly locking in inode_mark_dquot_dirty.

--- Message (Lai Siyao, 2013-02-27 06:31:50.000000000) ---
Patch Set 5:

It's not easy to replace the global dqptr_sem with a per-inode semaphore because in functions like remove_dquot_ref() inode_lock needs to be taken before access inode, however inode_lock is a spinlock.

I'll update the patch, but still with i_lock.

--- Message (Jeff Mahoney, 2013-02-27 14:05:45.000000000) ---
Patch Set 5:

I'm not sure how you came to that analysis.

Replacing the per-super rw_semaphore with a per-inode rw_semaphore is straightforward. Outside of the initialization, it's literally a search/replace operation.

We *don't* need to take i_lock before touching the inode generally. inode->i_lock only protects a few of the inode fields; This patch is extending inode->i_lock to protect another one of them.

I've been talking this over with Jan Kara, who's in my team and the author of much of the quota code. The path forward is probably to drop the rw lock entirely and use SRCU instead.

--- Message (Lai Siyao, 2013-02-27 16:33:17.000000000) ---

Could you take a look at drop_dquot_ref()? The lock order is dqptr_sem -> inode_lock, if dqptr_sem is replaced with a per-inode semaphore, it will be taken inside inode_lock, that's not viable. What's your suggestion?
