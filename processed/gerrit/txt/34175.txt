ldiskfs:

ldiskfs+DNE:
- sanity, ...

ZFS:
- sanity, ...

ZFS+DNE:

I definitely DO think there is added value from running the regular tests against the master-next branch, so that we find any issues with "trivial" patches before they are landed, or bad interactions between patches.

More diverse testing is good in general - just after starting this effort we already found: memleaks, the broken sbi_flags, some weird sanity-gss crash in test 150 and this list is likely to grow as I am looking at the rest of the test results (there's also LU-11940 I guess).

This is not intended to replace the autotest either since the configurations are more diverse in autotest (at least for now), but more like augment it and also hopefully provide more timely responses to obviously broken patches (that is caught with having kernel run with debug options enabled - something that traditionally autotest was not able to do, even though it looks like Minh renewed his efforts in the area) and in general highly parallel "fanout" nature of the testing is ore conductive to getting results to people faster (given enough nodes, obviously).

I likely will trim the patch list some after I am happy with the results of everything and then of course I already have all this logic that forces a particular test to be run if it was modified even if various test params were provided that requested not to do it.

Additionally experimenting with this - I am having tighter control over log collection (read I have 100% success rate with getting syslogs and kernel console logs + all the crashdumps too of course) and some other benefits. I hope eventually some of that knowhow could be transferred into our maloo instances as well so there are fewer missing logs. I also hope to again make this information more easily seen in the gerrit as appropriate (e.g. nowadays to get to any failure results you need to navigate too many links, some of which are slow too)

Another side goal here is to see if I can actually make all the logs searchable (I know I can pull existing maloo run logs, but they are somewhat limited) + run some other statistics to quickly highlight when new problems appear (hopefully even if they did not result in failures, like perhaps showing any previously unseen messages as an additional safety check, running generic "if you see this in the logs - it's bad" queries too).

So ...I guess pie in the sky here is to make Lustre developer life easier by showing them more info faster (and then to make the gatekeeper life easier to show them more info easier as well). I am really trying to scratch a lot of my own itches with this if you'd look at my imaginary roadmap ;)
