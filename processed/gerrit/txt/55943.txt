--- Commit Message ---
LU-18109 utils: adding nodemap offset capability

nodemap offset: the ability to add an offset range to nodemaps.
When an offset is defined, idmaps can only be set inside of their
given range. This is defined as:

lctl nodemap_add_offset --name nodemap --offset 100000 --limit 99999

And removed as:

lctl nodemap_del_offset --name nodemap

Test sanity-sec 27ab is added to exercise this new feature.

Oh indeed, it looks like Andreas and I think the opposite :)

I am adding Stephane Thiell in cc, maybe he can shed our light on this. The question is to understand what is the most "natural" way for admins when specifying explicit id mappings with mapping offsets. Andreas wrote:

   I was thinking that specifying the idmap values *without* taking the offset into
   account actually would make more sense for the admin, for a few reasons:
   the idmap entries would be specified in the "natural" format for the client admin
   requests (e.g. "storage admin, please remap my ID X to Y") since they will not
   necessarily know their offset range.  I think this would avoid potential errors,
   especially if the offset is not a nice number like "100000" that an be easily
   added/subtracted from an ID.

However for me it is the opposite. When admins specify explicit id mappings, these mappings should not be offset. Otherwise id on server side (file-system id) would not match the one configured by the admin, and that could be disturbing IMO. Indeed, admins are used to defining id mappings in the form CLIENTID:FSID. But if offsets are applied on top of this, this could not be considered an "FSID" anymore.
