--- Commit Message ---
LU-12659 ldlm: change log level for "lvbo_init failed" back to CERROR

The log level of LVB initialization (lvbo_init failed) was changed to
CDEBUG/DLMTRACE in commit 8739f132 (LU-5042 ldlm: delay filling
resource's LVB upon replay), and that change makes users hard to
notice the fail until analyzing the debug log or application report
it.

Here we restore it back to CERROR, so that the error will be early
detected by user rather than the real access happen.

Test-parameters: trivial

Fixes: 8739f132 ("LU-5042 ldlm: delay filling resource's LVB upon replay")

--- Message (Li Xi, 2019-08-26 02:38:15.000000000) ---
Patch Set 1:

> the problem is we DO get questions about "What is this error, isthere anythign I can do?

Understood it would be a little bit embarrassing when getting this question and only having a answer no better than "Sorry, nothing much can be done for this error message" or "I need a lot more debug message to understand the situation". I have been in that situations for several times myself. A better error message should include both possible root cause and possible solution(s) for the problem. So agreed that "lvbo_init failed for resource : rc = xxx" is not very helpful. Something like "lvbo_init failed for resource : rc = xxx, object XXX of file XXXX might be broken, please use command XXX to recover the file XXXX" would be much better.

We won't be able to accomplish the whole task at one stroke, but we should keep on improving the error messages. A lot more error messages need to be improved. I've got a lot of compains from multiple resources (supports/sales/customers) about this. I think one policy that could help to improve the situation is to enforce the following rule:

1. Whenever there is an error message, informations should be dumped as much as possible, not necessary all in the kernel message, but also could be as a debug file. Something like "...., please refer to dump file XXX for more information".

2. Whenever there is an error message, at least one advice should be printed. Something like "please use command XXX to recover/check the file XXX" or "please contact your support to solve this critical situataion" or "please shutdown you system immediately with command XXX to avoid data lose" etc.

3. Provide a centralized database which can be used to search for the possible solutions of error messages. Currently, we (including the support people) are kind of using Jira for this purpose, but the database should be in a more centrailized and normalized format.

However, as we can see these won't happen easily. And the improvement isn't easy even for this specific patch. So, comparing to printing zero error message and letting users have broken objects without notification, I think having the message printed is fine. It would be nice if Lustre has internal mechanism to fix the problem silently. But omitting error messages (even obscure messages) without doing anything won't make the error disapear. That is true for this patch. And even more important, omitting this error message could potentially improve the risk of the system, because a lost object usually means broken hardware/software. I think that is the reason why we are getting the request to print the error.

--- Message (Andreas Dilger, 2019-10-04 14:43:55.000000000) ---
Patch Set 1:

One possibility that comes to mind to distinguish this race condition is if the client sends the MDT parent FID in its RPC then the OSS can verify if the MDT inode still exists or not, like LFSCK does. The filter_fid would be home at this point, so it would not be usable to determine the parent FID.  If the MDT parent inode still exists then this is an error, otherwise it is a harmless race.  This would need an extra OSS->MDS RPC, which could cause issues if the MDS is offline or busy, but should only happen in the rare case of a missing object. 

Another alternative would be to keep the DLM resource around for a short time after unlink (eg. a few seconds) to allow detecting the "just deleted" case from the "has not existed for a long time" case. This would need some asynchronous process to clean up the resources after a timeout, and would consume memory, so probably not ideal.
