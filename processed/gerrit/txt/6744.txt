LU-3495 ptlrpc: Add history counter of handled requests

This patch helps to calculate the request handling rate
of services between different time intervals. It collects
enqueue RPC number every 10 seconds and keeps the PRC number
during last 100 seconds,1000 seconds and 10000 seconds by
default. The time intervals can be set when inserting module.
And the statistics can be accessed by /proc.

Hi Oleg,

That's a good point. But I guess atomic increasement will not cause performance problem here.

In order to check the performance of atomic increasement, I write following quick patch. It turns out that we can do atomic increasement more than 100,000,000 times per second. RPC rate is not able to reach even 0.1% of this rate, i.e. 100,000 RPC/s. Typically, the RPC rate of a OSS who can provide throughput of 1GB/s is about 1000 PRC/s. I know my estimation is far from accurate. But maybe it makes sense? What's your opinion?

Here is the patch. Please do not run it on production environment because it will crash the kernel when umount. :-)

===================================================================

+}

--- Message (Liang Zhen, 2013-07-29 05:48:02.000000000) ---
Patch Set 4: No score

I think Oleg's concern is that performance issue if atomic operations performance under race, e.g. many service threads (all CPUs) try to change it at the same time. 
We have made many efforts to reduce number of global locks/atomics, and we don't want to lose performance improvements by adding new global lock/atomic.
This can be avoid by two different options:
- use per-cpu-partition counter instead of global counter, for example, put this counter in ptlrpc_service_part instead of ptlrpc_service, this will require more changes then this patch because we have to calculate for each partition then summarise results.
- make this counter to be tunable and it's disabled by default, user can turn it on at runtime via proc or sysctl, this is more like a workaround but it requires less change.
Personally, I will be OK with either way.
