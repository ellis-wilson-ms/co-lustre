LU-19827 lnet: Fix MR forwarding

The change https://review.whamcloud.com/57492 introduced a regression
in the multi-rail forwarding code. MR forwarding allows a router to
select a different target interface for a message when the one chosen
by the sender is down or unhealthy.

Prior to the above change, lnet_is_peer_ni_alive() would check both
the down and unhealthy criteria, but now it only checks the down
criteria. We need to add a check for the peer NI's health value.

There was also a second bug here where we were inadvertently checking
the MR capability of the destination peer rather than the source peer.
This is also fixed.

Add sanity-lnet tests to validate MR forwarding feature

## Summary

**Fix Applied:**
Added the missing header include `#include <linux/libcfs/libcfs_fail.h>` to `lnet/lnet/lib-msg.c` at line 16.

**Rationale:**
The patch introduced a call to the `CFS_FAIL_CHECK()` macro on line 903 of `lnet/lnet/lib-msg.c`:
```c
likely(!CFS_FAIL_CHECK(CFS_FAIL_RTR_HEALTH_INC))
```

However, the `CFS_FAIL_CHECK` macro is defined in `include/linux/libcfs/libcfs_fail.h`, which was not included in this file. This caused the compiler to report an "implicit declaration of function" error.

The fix follows the same pattern used in other lnet source files like `lnet/lnet/lib-move.c`, which also includes `<linux/libcfs/libcfs_fail.h>` to use failure injection macros. This is a standard practice in the Lustre codebase for any file that needs to use the CFS failure injection framework.
