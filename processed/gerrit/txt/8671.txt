LU-5526 ldlm: fix bug in LDLM_FL_AST_MASK

when the blocking ast is sent to the clients, the server set
LDLM_FL_DISCARD_DATA in ldlm_request->lock_flags instead of
LDLM_FL_AST_DISCARD_DATA, which is contained in LDLM_FL_AST_MASK
incorrectly and the dirty pages at clients can't be discarded.

--- Message (Jian Yu, 2014-01-02 01:43:25.000000000) ---
Patch Set 1:

Comments from Andreas in http://review.whamcloud.com/#/c/8495/ :

This patch worries me a bit. The LDLM_FL_DISCARD_DATA is ok to send in an AST RPC to the client holding the lock, but it should never be saved in the lock itself. Otherwise, some operation like truncate(fd, 0) might incorrectly save this flag into the lock and all future AST RPCs could contain this flag. That is why the internal LDLM_FL_AST_DISCARD_DATA is a separate flag and is never sent to the client directly, but rather used to see if LDLM_FL_DISCARD_DATA is set and use that to set LDLM_DISCARD_DATA.

I don't know if the DLM code has changed enough to avoid this probl in another way.
