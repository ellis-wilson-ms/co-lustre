LU-569: Make lu_object cache size adjustable

lu_object cache is specified to consume 20% of total memory. This
limits 200 clients at most can be mounted on one node. We should make
it adjustable so that customers have a chance to configure it by
their needs.

--- Message (Fan Yong, 2011-08-10 02:13:25.000000000) ---
Patch Set 5:

I cannot imagine what the benefit is from the patch.

Honestly, current hash implementation maybe cannot support 200+ logic clients on single node. But it is normal, because no customer wants to mount 200+ logic clients on single node today (except for testing). If it is really necessary to mount so many logic clients on single node, it is not on small node, it should be on very fat node, like SGI-XXX, maybe with hundreds of CPU cores and dozens of TB RAM, at that time, memory pressure is not big problem.

On the other hand, your patch will hurt hash performance. Because you shrink bucket bits of hash table, less hash buckets mean longer hash list in each hash bucket, which increases the average hash searching time.

For example, when you traversing large directory with 10M items. If all items are cached on client, if bucket bits is 15, then average hash bucket depth is 2^21/2^15=64, but if bucket bits is shrink to 13, then such depth is 2^21/2^13=256. Because we cannot control items to be scattered evenly to all hash buckets in the hash table (depends on hash function), so the really hash bucket depth is larger, maybe thousand, that is terrible.

As I understand, the patch is useful for your IR test, but maybe not for the real world configuration. So it is better to consider the performance issues more.
