LU-18183 nrs: rule-based share rate limit for NRS TBF

In the current NRS TBF desingn, a server classifies the incoming
I/O requests into TBF buckets accroding to the TBF rule.
And the requests is scheduled and the RPC rate is limited per TBF
bucket based.

Each TBF bucket is in unit of TBF types (nid, gid, uid, jobid or
opcode) or their combinations. Each TBF type unit has its own
separate TBF bucket and separate rate limit setting by TBF rule.

However, there is  a requirement that all I/O requests matching
the conditions of a special TBF rule with a shared rate limiting.
i.e.
lctl set_param ost.OSS.ost_io.nrs_tbf_rule="start sharerate nid=
	{192.168.25.[32-64]@tcp} rate=3000 share=1"

It defines a rule with a dedicated shared rate limiting TBF bucket
that all I/O requests coming from the NID range 192.168.25.[32-64]
has a shared rate limit of 3000 IOPS with the new option "share".

--- Message (Etienne AUJAMES, 2024-09-13 10:09:05.000000000) ---
Patch Set 1: Code-Review-1

I really don't like how it is implemented. Matching the share rule for every incoming request is too expensive and it will not scale (this is a scheduler).

The actual logic to handle to enqueue incomming request is:
* construct the key for the request
* try to find the matching cli (TBF class) into the hashtable
* if the entry exist, directly enqueue the request in the cli
* if not, match the rule. Create the cli with the rule rate and enqueue

I think we can conserve this logic for share rate rule.
We can create a share cli and store a pointer to the share cli in the hashtable. Each hashtable entry can take a ref on the share cli.

Normal rule (tbf jobid):
  |req_job1| --hash--> |cli_job1|

  |req_job2| --hash--> |cli_job2|

Share rule (tbf jobid):
  |req_job1| --hash--> |entry_job1| --|
                                      |--point--> |share_cli| 
  |req_job2| --hash--> |entry_job2| --|

This can be managed with an "union" for hashtable entries.

Patch Set 1:

I have a new thought to reduce the contention, using two level class.
One level is a TBF bucket (or class, a nrs_tbf_client) for point hash lookup for each jobid, another level is rule-based TBF bucket (or class, a nrs_tbf_client) with sharing rate. Level 1 is only doing lookup, Level 2 is doing rate limit.
Once found a Level1 bucket, we can identify the rule, from the rule, we can identify the share Level2 bucket, and then adding the request into Level 2 bucket in FIFO order.
