LU-16680 llite: add lowmem_sync feature

The lowmem_sync feature makes the client drop to sync io
when it has reason to believe it is in urgent memory
reclaim.  This allows Lustre to handle low memory states,
such as those enforced by cgroups, much better.

We determine this by noting when a thread which is trying
to allocate memory (has PF_MEMALLOC set) is calling in to
the Lustre releasepage(), indicating we're experiencing low
memory.

Then we drop back to sync io for the next RPC.  This has
two benefits:
It rate limits the client, since this thread won't submit
more IO while waiting for the sync RPC
It forces a commit on the server side and returns notice of
that commit with the sync RPC completion,  which allows us
to free our uncommitted pinned pages on the client.

Without this patch, Lustre on my VMs requires the cgroups
memory limit to be at least 512 MiB or IO at any size is
eventually killed.

It also thrashes badly at that limit, performing extremely
poorly as it spends most of its time in memory reclaim.
(On my VM, 60 MiB/s without the patch, vs 200 MiB/s with
the patch)

With the patch, 64 MiB of memory is enough to safely do IO.
There is a performance hit when running at the memory limit
but that is expected and is less severe with this patch
than without it.

Remaining todo:
Test this on real hardware:
1. Test with cgroups at speed to make sure this works well
at higher speeds
2. Loaded single client (many threads doing IO to the same
file with a full page cache)
3. And with multiple clients to one OST, to see the cost of
the extra sync operations

Still marked fortestonly because imo we shouldn't land this
until we've had a chance to verify perf on real hardware.
Working on that.

Patrick, for your patch the main reason to send early soft-sync (IMHO) is to avoid "stop-and-go" page submission (async, block on sync write until page commit, async, ...). 

One way to handle that "intelligently" might be to track the number of dirty pages (or uncommitted RPCs if that is easier) and how long it takes for a sync write to complete for a last_committed update, so the client can develop an adaptive threshold where the soft-sync could be sent before the client "hits the wall".

Essentially, if a sync RPC takes N seconds to return a last_committed update, and the client is approaching the "sync write threshold" (pages or uncommitted RPCs) at the current rate, how many RPCs in advance does it need to send a sync/soft sync so it gets the last_committed back and can unpin pages "just in time" before the kernel starts becoming unhappy and think it needs to start OOM killing?

I don't doubt that this patch is already avoiding the OOM kill problem, but it could potentially be improved further. 

It would be interesting to extract eg. the write syscalls, the dirty page count, the RPC sending times, and when a sync RPC is needed, and plot these over time to see if page dirtying is blocked for seconds at a time while the client waits for a sync.  This could be run for the default config, with "fake writes" and a RAM disk OST to simulate fastest performance, and also with a delay on the RPC processing (or multiple concurrent writers) to simulate a busy server.

--- Message (Patrick Farrell, 2023-04-02 15:31:10.000000000) ---
Patch Set 7:

So, I think "stop and go" page submission isn't as bad as it sounds?  We're going to hit the limit, then we're going to cause a sync, rate limit IO, and free all of our uncommitted pages.  

This means we've basically freed everything we can reasonably free - I mean, maybe we didn't send the shrinkers on a hunt for the last few bytes of slab cache, but we hit all the big stuff - and so we go to back async submission.

So this ensures the sync is 'maximally effective' in terms of how much memory it frees, at least from the view of this particular client.  I'm not saying this is perfect - it's clearly not - but I think it's better than it looks at first glance.

It also doesn't seem like we *need* to get out 'ahead' of this to avoid being OOM killed - in my testing, when we get memory pressure, we seem to always find enough memory to do the necessary sync RPC.  (Not that you're saying we do, just noting it.)

Also, in my testing, this rate limiting is less painful than grubbing for the last few dregs of memory,  because even when you don't hard OOM, the user process spends most of the time in the mm code looping around trying - and failing - to free uncommitted pages.  (Literally, in reclaim, uncommitted pages are sitting on the inactive list, then are moved to a list of 'try to free' pages, then releasepage is called on them, and since we can't unpin them, the kernel can't free them and they are moved *back* to the inactive list.  Over and over - it's very time consuming.  This is avoided if we mark them as "in writeback", but that has its own complexities.)

So at least in my local testing, this patch makes performance when near-OOM much better.  I still need to try it on a real system and with many clients.  (I did try it with many stripes and multiple files & processes, but still locally.)

So I guess I'm advocating for this being a big improvement in both OOM-behavior and (I think) performance in near-OOM states, and I want to see how that plays out before getting in to improving it further.  (I'm also partial to this dead simple "bounce off the limit" behavior if it turns out to be 'good enough'...)

Patch Set 7: Code-Review-1

Hmm, good I made sure to do performance testing on a real system.

The lowmem_sync approach has a problem at scale with cgroups.  Specifically, cgroups seems to only be freeing *just enough* memory so it can do the next allocation, rather than freeing all of the no-longer-unstable-pages.  So we are repeatedly going in to memory allocation + free memory even though there seems to be plenty of freeable memory.

I'm still reflecting and seeing if there's something else we can do here, but at the moment, cgroups seems to be asking to free just 1 page at a time, so we don't pull back from the memory limit, even though many pages are no longer unstable and could be freed - we just stay stuck at the memory limit, going in to memory reclaim for every new allocation.

By the way, replying to a comment *somewhere*:
sending a buffered (ie, not O_DIRECT) RPC via the sync path *does* force a sync on the server.  The requirement is just the absence of the OBD_BRW_ASYNC flag, which is only set in the async path *after* the point where we divert to synchronous (if there's no quota or if we're doing a lowmem sync).

We’d have to track unstable pages per OSC, but that’s not too bad (and I can test out the idea without implementing that).

So…. We won’t rate limit ourselves under memory pressure any more, but the difficulty in getting memory kind of does that for us.  We'll just make sure what's in the page cache is freeable, which should keep us from getting OOM killed.  I'll see what performance is like and make sure we don't get OOM killed.

What happens if cgroups is not active?  It seems like a similar mechanism is still needed to trigger sync writes as memory pressure increases on the client.

IMHO, a combination of OBD_BRW_SOFT_SYNC (to start "early commit" if OST can be responsive enough) and then fully sync writes (to avoid client OOM if it is really low) would be best.
