> My code have fail also with _SOME_ combinations. But you think this

It's true that this improved test still fails in *some* combinations even with the current code, but it fails "less badly" and "less often" than the new patch.  With ostcount=6+stripecount=4 for the current code it creates 111/222/111/223/111/222 first-stripe objects for 1000 files (so some OSTs have 11% of objects, others have 22% of objects), but at least all OSTs are being used.

The new patch *also* fails for ostcount=6+stripecount=3 with 0/500/0/0/500 first-stripe objects (so most OSTs have 0% of objects), and for ostcount=8+stripecount=4 with 0/0/500/0/0/500 (also 0% of objects), and for ostcount=9+stripecount=3 with 0/0/333/0/0/334/0/0/333, and possibly other combinations because they were not all tested.

 > Andreas, this code will be never good except it will be very
 > complex (which soo bad from perf).

The current code is not *perfect* for all combinations, but at least gives a reasonable distribution for all cases.  The new patch is *bad* for many combinations, and would lead to imbalanced OSTs.

 > it's because - modules by even number (number of OST is always
 > even), too bad for any hash function 
 > From other side - this code tries to solve a Knapsack problem.
 > which don't have a good result for fast time.

I don't think we need to have a perfect solution to the problem.  My request is that the patch is not worse than the current code.   I have a simple update to your patch to keep the same behavior as the current code, that will not hurt performance significantly.

It is also important to note that saving a few cycles in OST selection is pointless if the OSTs are imbalanced, which will be *much* slower than the few cycles saved.  Also, OST object creation itself is probably the major bottleneck here, and not the few cycles for selecting which OST to use.
