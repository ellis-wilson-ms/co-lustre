LU-18109 nodemap: fix idmap offset for root

root must be mapped and offset just like other ids, so that all ids on
the client in the range [0, offset_limit-1] are mapped to the
filesystem id range [offset, offset+offset_limit-1].
This means that as soon as an offset is defined on a nodemap, root is
necessarily mapped:
- if admin=1, root id is offset
- if admin=0, root is squashed then offset.
Similarly:
- if trusted=1, ids are mapped (with any explicit mapping) and offset
- if trusted=0, ids are squashed then offset.

Moreover, add_offset and del_offset operations are only allowed on the
MGS or for a dynamic nodemap.

Finally, enhance sanity-sec test_27ab to test id mappings with offset.

> Is that going to be added afterward via some mechanism to verify the source and target IDs are valid *before* the root ID is squashed, and then add a capability (e.g. CAP_CHOWN, or as needed) to that thread to allow the operation to complete even though it is squashed?

The idmap offset feature clearly does not allow root on the client being root on the file system server side. So we will have to provide a way to allow some "local admin" to perform root-like operations such as changing file owner/group, access rights, etc. That would rely on Unix capabilities if we detect that the mapped user is actually root on the client. And the subset of granted capabilities would be configurable via a new nodemap property.
It is important to make the difference between map and squash however. If admin=1 on the nodemap, root id is mapped and offset, then this is fine to grant capabilities. But if admin=0, root is squashed (and offset), and in this case no capabilities would be granted.
