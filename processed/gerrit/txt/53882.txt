--- Commit Message ---
LU-10391 obd: Update lmd_parse to handle IPv6 NIDs

lmd_find_delimiter()/lmd_parse_nidlist() are updated to handle IPv6
NIDs.

class_parse_value() is updated to handle IPv6 NIDs that begin with
'::'. Also, when parsing a Lustre network name, the buffer needn't
contain an '@', nor do we want to search for '@' when locating the
next delimiter.

Fixes: 101f6e8 ("LU-10391 obdclass: handle large NIDs for mount strings")

(1 comment)

This should really add some test cases to confirm the new parsing is working properly.  I think it is a bit too complex to just look at the code and be confident it is handling the various cases correctly.

I was thinking about this issue recently, but maybe I didn't write it anywhere - do we want to introduce a *different* NID delimiter now that ":" is used by IPv6 NIDs?

I think we used ':' because it was already in the "mgsname:/fsname" string, but it wouldn't be terrible to say that if mounting with IPv6 numeric NIDs (as opposed to hostnames) that you MUST use '+' or [] to delimit IPv6 NIDs instead of ':'?  That would be unambiguous and avoid messy parsing heuristics.

That said, it would be even nicer to get the name resolution working properly (LU-16738) so that it is possible to specify a hostname like "testfs-mgs@ib0:/testfs" that can be resolved to multiple IPv4 and/or IPv6 addresses instead of having to type (and parse) a huge mess of IPv6 addresses.

Third idea - just deprecate the "param=XXXX" mount option completely (which looks to be the only place this is used)?  Do we really need to be able to set arbitrary parameters at mkfs/mount time these days?  I think "lctl set_param -P" is pretty robust and I don't call "param=" being used recently, but I could also be missing the forest for the trees...
