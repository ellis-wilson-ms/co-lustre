--- Commit Message ---
LU-5134 utils: Add parallel option to lctl set_param

Add a "-t" option to lctl set_param to enable setting multiple matched
parameters in parallel. When called with "-t", lctl will set up a work
queue of matched file names and spawn a fixed number of threads per
CPU. Each thread will pull items off the work queue, write to the file
associated with each work item, and return when there are no more
items on the work queue.

A field called po_parallel_threads is added to struct param_opts to
indicate the number of threads set_param should run in parallel. If in
parallel, jt_lcfg_setparam initializes a work queue and passes it to
do_param_op, which adds each matched item to the work queue. Once
jt_lcfg_setparam has called do_param_op for each param-value pair, it
passes the work queue to sp_run_threads, which creates threads, each
of which call write_param to set the parameter. If not in parallel,
jt_lcfg_setparam does not pass a work queue to do_param_op, and
do_param_op directly calls write_param on each matched param.

param_display was renamed to do_param_op to more accurately reflect
what it does.

If lctl is compiled without pthread support, "lctl set_param" will
still accept the "-t" option, but it will print a warning message, and
it will set the parameters in series.

The new "-t" option to set_param was documented in the lctl usage and
in the man page.

I see your point.  However, we want a single command/interface we can call from our job scheduler cleanup code to clear Lustre caches on a compute node before the next job is scheduled, and we want this to happen quickly to avoid excessive cleanup times.

Initially, I implemented a parallel drop_caches interface in kernel space which cleared all caches in parallel.  Oleg did not like that change and suggested that we implement a generic parallel set_param that would meet our needs.  That's what this patch does.  Please see Oleg's comments in LU-3970 and the associated patch for more information and then let me know what you think.

John, I will also add that for portability between versions of Lustre where the parameters have changed location (procfs to sysfs/debugfs), an implementation of such a script in bash would probably be smart to use the "lctl set_param" command to set each parameter in parallel.  I am curious about how this would perform against this patch which uses pthreads and requires only one execution of "lctl set_param".  I will do such a performance comparison on my VM and post the results in LU-5134.

Even without a significant difference in performance, it would be highly preferable if we could have a single interface to do this rather than requiring our job cleanup code to do the parallelization.

The current design of the --thread feature revolves around the concept of forking the command and executing it using the provided options. However, the set_perm functionality relies on parsing paths using the llapi_param_get_paths mechanism, these paths  then gets added to work queue. This queue is subsequently processed by a thread.

Integrating this parsing and work queue logic into the existing --thread approach would unavoidably complicate the --thread code. We'd need to tailor the parsing process specifically for the set_perm case, and then add the logic to to create the thread to process these work queue.

I've outlined my concerns here, but please do let me know if there are any aspects I might be overlooking or if further clarification is required. Thanks for your review and suggestions.
