LU-2950 lnet: added test case to test configuring routes

Added a test case in conf-sanity.sh to test the new scripts
lustre_routes_conversion and lustre_routes_config. The test case
takes in a sample routes file which has the old synatx, run the
lustre_routes_conversion script to convert to the new syntax, then
runs the lustre_routes_config script in --dry-run to ensure that the
script configures the routes

--- Message (Andreas Dilger, 2013-04-06 05:51:22.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

Thanks for working up a test case.  Some minor areas for improvement:
- the test input file could be generated by a "here document" from the test itself, so that there is no complexity with packaging and finding the input file:

        cat <<- OLD_LNET_CONFIG > $TMP/old_lnet_config
                tcp1 23 192.168.213.1@tcp:1; tcp5 34 193.30.4.3@tcp:4;
                tcp2 54 10.1.3.2@tcp;
                tcp3 10.3.4.3@tcp:3;
                tcp4 10.3.3.4@tcp;
        OLD_LNET_CONFIG

- the test file needs to validate success or failure more stringently, since it currently appears that it could fail either of the test phases with only some messages to stdout, but without actually reporting any error to the test framework via error(). If either the first or second script completes with rc != 0 it should be considered an error.
- the output from the second script should be validated for correctness, either by comparing it with diff against a known-valid output, or at least doing some sanity checks on it (e.g. number of routes parsed to catch failures with multiplie routes on one line, interfaces and gateways used, hop count, etc)
