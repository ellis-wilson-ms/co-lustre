--- Message (Lai Siyao, 2016-09-21 12:16:45.000000000) ---
Patch Set 5:

IMO xattr cache is in the former category.

--- Message (Alex Zhuravlev, 2016-09-21 12:24:01.000000000) ---
Patch Set 5:

we must ensure that the same thread can't get a stale entry. like we discussed before: LOV EA is set by LOD, few CPU cycles later MDT fetches the same LOV EA to return to a client. I guess one option is to add new entries to the list's head always so lookup meets them earlier. but frankly I'm not sure why do we need to replace.

--- Message (Alex Zhuravlev, 2016-09-21 12:25:21.000000000) ---
Patch Set 5:

ah, in theory the following case is possible: some layer deletes EA, but another layer in the same thread tries to access it right away. in this case we'd return a stale EA? IMHO, it makes sense to have an invalid flag to prevent such a case.

--- Message (Lai Siyao, 2016-09-21 12:29:44.000000000) ---
Patch Set 5:

I don't think the same thread can find a deleted entry.

e.g.,
1. reader iterated list to lookup entry, and it just scanned the first one.
2. writer added new entry into list head.
3. reader won't have chance to find the new one, but the old one, if the stale one is not allowed to use, then it has to lookup from head again.

This is why replace is needed.

--- Message (Alex Zhuravlev, 2016-09-21 12:36:24.000000000) ---
Patch Set 5:

I believe if we add to the head, then replace is not needed - the subsequent lookup starting from the head will meet the new entry earlier. what does "replace" buy us?

--- Message (Lai Siyao, 2016-09-21 12:47:23.000000000) ---
Patch Set 5:

yes, subsequent lookup will find the new one, but concurrent lookup may not be able to find either new or old one.
