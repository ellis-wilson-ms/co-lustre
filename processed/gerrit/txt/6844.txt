--- Commit Message ---
LU-3504 libcfs: alloc kmem atomic when owning resource lr_lock

This change fixes a regression introduced by the following commit:
    commit 2c7a41a5c5958d36fa839ea91f3e63375cbd90fc
    Author: Vitaly Fertman <vitaly_fertman@xyratex.com>
    Date:   Tue Jun 26 01:47:50 2012 +0400
        LU-1157 ldlm: replace waiting flock lists by hashes

This commit introduced the possibility of a dead-lock which
can occur on MDS handling 1st FLock request for an export,
owning resource lr_lock spin-lock from lock_res_and_lock()
the kmem allocation of the new hashing stuff for FLocks is
not done atomic and thus can resched() allowing competitor
threads to spin forever on all available CPUs.

The fix is to force all related kmem allocs to be done
atomically.

Patch Set 2: I would prefer that you didn't submit this

Please don't force cfs_hash to use atomic allocator, lustre can have very large hash table and can have hundreds of thousands cfs_hash instances.
I think this bug should be fixed in a different way, first it is wrong to create hash table inside ldlm_flock_completion_ast()->ldlm_flock_blocking_link()->ldlm_init_flock_export(), I can't see which lock is protecting this assignment:
 "exp->exp_flock_hash = cfs_hash_create(...)"
I think this assignment should be protected by a lock of obd_export, not by lock_res_and_lock(). 
my hunch is, we should create cfs_hash() in a different place, e.g. ldlm_flock_completion_ast() before holding lock_res_and_lock() and calling ldlm_process_flock_lock()

--- Message (Prakash Surya, 2013-07-15 15:18:15.000000000) ---
Patch Set 2:

Liang, I agree we should move cfs_hash_create() out from under the spin lock. That's unanimous I think, but that solves only half of the current issue with ldlm_flock_blocking_link()

Without changing the hashing allocation to be atomic, we still run the risk of sleeping when calling cfs_hash_add(). We *cannot* sleep unless we can refactor the code to drop the spin lock.

Perhaps we need to introduce a way to force the allocations to be atomic using a more granular approach. That way it would only affect the necessary allocation, making them atomic, but not any other calls (e.g. using a parameter or hash initialization flag).
