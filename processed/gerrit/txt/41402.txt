--- Commit Message ---
LU-14119 osd: add mount option "resetoi"

OI files on zfs are special, and they can't be deleted by user space
tools like rm. Sometimes the OI files may contain stale OI mappings,
and they needed to be removed for namespace consistency. Add a mount
option 'resetoi' to recreate OI files on mount time, and it will
support both ldiskfs and zfs. This should be the standard way to
recreate OI files, other than mount as backend filesystem and unlink
them manually.

Add sanity-scrub 17.

--- Message (Andreas Dilger, 2021-02-03 23:26:14.000000000) ---
Patch Set 1: Code-Review-1

> It's simpler to recreate OI files than fixing specific entries, and
 > user may not know which entry is wrong only by logs.

I don't expect the *user* to fix the bad entry.  In a case like this where a bad FID->inode mapping is found (i.e. FID X points to inode Y, but inode Y has a different FID Z) then it would be better if the bad FID entry was deleted from the OI file by the OSD itself.

I'm not saying that we _shouldn't_ allow OI rebuild for ZFS (which is why I gave this patch +1 instead of -1), but it would be better to avoid it if possible.

However, I now realize that there is no test for this change.  Since it is modifying the filesystem in a way that might not be recoverable, please add a sanity-scrub test that tests with "-o resetoi" on the MDT for both ldiskfs and ZFS and then verify that the OI files are rebuilt.

One way to verify the OI files are rebuilt properly is to get a file's FID before remounting with "-o resetoi", then use "lfs fid2path" afterward to verify it returns the correct path.  The fid2path _should_ block with -EINPROGRESS until that FID is inserted into the OI, but the test still needs to wait until scrub is finished so that it doesn't cause problems in the next subtest.
