--- Message (Andreas Dilger, 2012-12-12 23:51:06.000000000) ---
Patch Set 7: I would prefer that you didn't submit this

The way this patch is structured will cause a protocol change for the DLM locking resources.  This would also need patches for all previous clients that interact with a FID-on-OST OSS, which isn't practical.

A better approach would be to replace the struct ost_id {o_id, o_seq} fields with struct lu_fid (in a union for the short term).  This keeps the "object identifier" as a 128-bit value, but for FID-on-OST values (i.e. old o_seq != 0) ON THE SERVER the struct lu_fid is used, while legacy o_seq == 0 objects use the ost_id ordering.

This avoids the need to change the client, and it looks like the only fallout would be that messages printed on old clients would have oid and seq reversed for FID-on-OST servers (mostly harmless?), but the DLM locking would be consistent for all clients.

This would also allow consolidating the OST object handling to use struct lu_fid instead of having to deal with struct ost_id.

--- Message (Andreas Dilger, 2013-01-09 09:37:52.000000000) ---
Patch Set 15: (13 inline comments)

It would be better if there was a preliminary patch to convert the relevant structures to use an ost_id, and use struct assignment to copy the ost_id between different structures.  For this patch, that would mean struct lov_mds_md and lov_stripe_md, along with struct obd_ioobj would need to be converted to contain an ost_id instead of separate id/seq fields.

Then, this patch would add ostid_build_res_name() and ostid_res_name_to_id() with an ost_id as a parameter instead of separate id/seq fields.

A third patch would change the client/server code to clean up printing of id/seq fields to use a macro like DFID/PFID, as was done in http://review.whamcloud.com/2004 would allow converting these to FIDs more easily in the the later patches.

These first three patches do not change any code functionality compared to the existing releases.  It would be good to at least get those patches included.

Finally, a patch to make ost_id a union with an lu_fid (with swapped id/seq) for new FID-on-OST objects (i.e. o_seq != 0) would convert the wire protocol to use seq/oid instead of id/seq for newer objects.  That would make lock resources matched between MDT and OST objects, and the OST protocol would _really_ be using FIDs instead of a FID packed into an ost_id in reverse order.

This last patch would not change any behaviour for existing clients/objects if it is landed for 2.4 with the FID-on-OST code, since it would only change new objects with new server code.  If it doesn't land with FID-on-OST then there won't really be any easy time to change it in the future after we have seq != 0 objects.
