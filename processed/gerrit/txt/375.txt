LU-50 ldiskfs: pdirops patch for ldiskfs

Single directory performance is a critical for HPC workloads. In a
typical use case an application creates a separate output file for
each node and task in a job. As nodes and tasks increase, hundreds
of thousands of files may be created in a single directory within
a short window of time.
Today, both filename lookup and file system modifying operations
(such as create and unlink) are protected with a single lock for
an entire ldiskfs directory. PDO project will remove this
bottleneck by introducing a parallel locking mechanism for entire
ldiskfs directories. This work will enable multiple application
threads to simultaneously lookup, create and unlink in parallel.

This patch contains:
  - pdirops support for ldiskfs
  - N-level htree directory
  - integrate with osd-ldiskfs

For testing purposes, it might be useful to run local ldiskfs patches with 1kB blocksize and 255-character filenames.  This would exceed 2-level htree at around 48k entries, which is practical for a test in conf-sanity.sh.

There need to be some follow-on bugs filed and fixed for this code in order of priority:
- test LARGEDIR and 3-level htree for local ldiskfs in conf-sanity.sh with 1kB blocksize
- e2fsprogs support needs to be added for 3-level htree
- e2fsprogs support for directories larger than 2GB
- e2fsprogs test for 3-level htree e2fsck, corruptions
- add LARGEDIR feature to "f_random_corruption" test
- once e2fsprogs is released and available for download, a patch is needed to enable large_dir on new filesystems with mkfs.lustre (probably for 2.3 since I don't think the maximum directory size is a limit for most users today)
- updates to the user manual and release notes to describe how to enable the LARGEDIR feature with tune2fs
- separate the INCOMPAT_LARGEDIR features (>2GB, 3-level htree) patches from the htree-lock patch so that they can be submitted upstream

Patch itself is fine, though very big.

If we were doing this again, I would insist on the patch being split up into several smaller functional pieces, for example:

- add support for 3-level htree, >2GB dirs (for upstream)
- conf-sanity ldiskfs test for this with 1kB blocks, long names
- move oo_ext_idx_sem to osd_index_{lock,unlock} wrappers
- add htree lock (unused initially)
- add some built-in unit tests for this code
- change osd-ldiskfs to use htree lock
- add Lustre-level tests for this code

Each commit would have a proper description of what is added and the expected performance/functionality gains.

This makes the task of inspecting and testing this code a lot easier.

That said, I've done the inspection of this large patch several times, and the changes below could be landed in a follow-on patch since they are not defects.  I'll leave it to Oleg to decide whether these are needed for landing or not.  If the patch has to be rebased for some reason they should be included.
