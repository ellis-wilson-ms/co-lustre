What is the *BENEFIT* of such a change?  So instead of having shared code in the existing libcfs.ko module, now we depend on a *different* module lustre_compat.ko, it breaks every test interop run, and we don't have fewer modules, probably all of the functions will be renamed, which breaks existing patches, etc...

The libcfs module was intended for low-level infrastructure, and code shared between lnet and lustre modules, and (at times) also used in userspace.  That was fine until sharing code between userspace and the kernel was verboten and it had to duplicated (e.g. libcfs_netstrfns in libcfs/libcfs/utils/nidstrings.c and lnet/lnet/nidstrings.c).  Consolidating that duplication again would be a good cleanup.

I'm all in favor of cleaning up the code, but it has to have a clear benefit instead of just being churn.  IMHO, adding a separate lustre_compat.ko module is just churn.  Reducing the number of modules  and reducing code duplication would be a win.
