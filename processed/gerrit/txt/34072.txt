LU-11836 ldlm: fix enqueue reply vs bl_ast race

In general this race may occur when enqueue replies with
replaced lock resource and BL AST happens at the same time
on that lock. If BL AST arrives on client earlier than
enqueue reply then callback will be called on parent lock
with no cancel returned to the server causing lock timeout.
Moreover, if lost enqueue reply will be resend that may
cause deadlock:
- enqueue #1 issues lock on server but with resend
- process #2 cause BL AST to it before resend
- enqueue #1 resend will wait for parent lock on server
- process #2 holds parent lock busy waiting for cancel
  which will never come. Deadlock

This situations are very rare but become visible with DOM
because it uses more conflicting lock modes and read-on-open
may cause resends due to buffer grow on server.
