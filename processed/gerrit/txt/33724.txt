LU-11703 target: force transaction commit on grant lack

The following test
    addr = mmap(.., MAP_SHARED, ..);
    for(;;)
        memcpy(addr, ...);

may end up with receiving SIGBUS. This is because ll_page_mkwrite()
(lustre implementation of page_mkwrite method of vm_operations_struct)
is unable to handle case of insufficient grants.

An OST allocates grants based on amount of space available on a
backend filesystem reduced by amount of space already granted to
clients (tgt_grant_space_left()). The amount of space granted to
clients gets released in transaction commit hook (tgt_grant_commit()).
If the OST manages to grant almost all available space to clients in
time interval between subsequent transaction commits - it degrades
grants allocation so that clients end up with ll_page_mkwrite failure.

The fix is to sync a target which is close to free space exhaustion
regardless to whether pages to be written back consume grant space or
not. This is because in case of mmaped i/o there is no way to switch
to sync write.

The case illustrating the issue is added to mmap_sanity test.

(3 comments)

Two things that I think are wrong with this patch, since I don't think it is actually solving the problem that is reported in LU-11703.  It may be solving _some_ issue, but I'm not sure it is the right one:
- the mmap overwrite of the file should not be consuming more space or grant on the client, since the same blocks are repeatedly being overwritten.  This indicates that there is actually some bug in the grant consumption on the client.
- the client _may_ still run out of grant if the filesystem is actually full, so SIGBUS should not happen if the file is already preallocated.  At worst the client should do a sync write in this case, but I suspect if the grant consumption issue is fixed then this won't be a problem.  Under normal circumstances, the client might consume grant, but it should be returned to the client immediately if the blocks are already allocated.
