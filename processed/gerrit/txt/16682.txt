--- Commit Message ---
LU-7236 ptlrpc: idle connections can disconnect

 - when new request is being allocated ptlrpc initiates
   connection if it's not connected yet
 - if the import is idle (no locks, no active RPCs, no
   non-PING reply for last osc_idle_timeout seconds),
   then pinger tries to disconnect asynchronously
 - currently only client-to-OST connections can be idle
 - lctl set_param osc.*.idle_timeout=N controls new feature:
   N=0 - disable
   N>0 - seconds to idle before disconnect
 - lctl set_param osc.*.idle_connect=N to reconnect if idle
   (N is positive number)
 - OSC module parameter osc_idle_timeout controls default
   idle timeout and set to 20 seconds by default

Looks fairly straight forward.  Do you have any tests that actually verify this?  One option would be to make "idle_disconnect_sec" hold the time before disconnect, and then set it very low for a test?  I'd think that this may otherwise cause the clients to disconnect right at mount time if they don't have any locks.

At one time when we did purely on-demand connections after mount, several sites didn't like this because it gave the false impression that the filesystem was working properly, when in fact some OST is totally offline and the client didn't know since it never tried to connect.

In that light, clients should at least try to connect to all targets initially, even if they will disconnect after some shorter interval if they haven't been used.
