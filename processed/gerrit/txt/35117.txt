LU-12410 lnet: Fix lnetctl peer add/del nid parsing

The arguments to lnetctl peer add/del --prim_nid and --nid options
are currently treated as ip2net expressions. This prevents us from
defining peers with NIs that do not use ip addresses (currently just
gnilnd). Add a proper code path for handling nids rather than ip2net
expressions. Add a more informative error message when un-supported
net types are specified in an ip2net expression.

Test-parameters: trivial

Patch Set 3: Code-Review-1

I think there is value in allowing the generic ip2nets syntax. This patch removes the ability to add a list of expressions on the command line and in YAML. You'll have to specify ip2nets parameter, which is there for backwards compatibility. The original intent was to standardize on that syntax because an explicit NID is a specific instance of the syntax.

I think a better solution is to modify the code to handle different types of addresses. A function addr2nets can be created. The net in the NID can be extracted, and a lookup table implemented to lookup the type of net and call the appropriate address translation function. This will be useful if we add support for other types of addresses in the future.

Look at libcfs_netstrfns in libcfs/libcfs/util/nidstrings.c

This way the appropriate address parsing function is called depending on the net type.

Then we can allow:
lnetctl peer add --nid 192.169.22.24@tcp,192.169.22.[75-78]@o2ib,[1-3]@gni
