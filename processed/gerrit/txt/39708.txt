LU-10810 clio: SEEK_HOLE/SEEK_DATA on client side

Patch introduces basic support for lseek SEEK_HOLE/SEEK_DATA
parameters in lustre client.

- introduce new IO type CIT_LSEEK in CLIO stack
- LOV splits request to all stripes involved and merges
  results back.
- OSC sends OST LSEEK RPC asynchronously
- if target doesn't support LSEEK RPC then OSC assumes
  whole related object is data with virtual hole at the end
- lseek restores released files assuming it is done prior
  the file copying.
- tool is added to request needed lseek on file
- basic tests are added in sanity, sanityn and sanity-hsm

--- Message (Andreas Dilger, 2020-08-22 16:35:43.000000000) ---
Patch Set 2:

I haven't had time to look at this patch yet, but wanted to bring up an important issue. Since SEEK_HOLE and SEEK_DATA are used for copying files, it is important that the result is consistent with the file data. That means the client doing the seek must have a read lock on the extent that is being seeked over, or the OST needs to hold a lock for server-side locking. 

That will flush pages from client cache and prevent writes to fill holes or extend the file, at least as long as the end of the syscall.

--- Message (Andreas Dilger, 2020-08-22 16:36:49.000000000) ---

Patch Set 2:

I don't think we can ever be correct in the case of concurrent writes, since there will *always* be a window between the time the SEEK was done and when the application can process the information. I think the main concern is to ensure that there is no data written *before* the SEEK that is not accounted for.

If the client or OST is causing previously-written data to be flushed to disk and taken into account, then that is the best that can be expected.
