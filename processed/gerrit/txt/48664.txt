LU-16191 socklnd: limit retries on conns_per_peer mismatch

If connection initiator has a higher conns-per-peer setting than
its peer, don't try to create extra connections forever as the
peer will keep rejecting them. A few retries should suffice to
resolve a valid race.

--- Message (Serguei Smirnov, 2022-09-27 16:50:00.000000000) ---
Patch Set 1:

> Does this logic only kick in when we open the minimum 3 connections
 > in typed mode? If not, should it?

The logic addresses the situation when EALREADY is received (hardcoded N=3) N times in a row regardless of connection type. The assumption is that if EALREADY is received N times in a row, it is not because of the valid "race", but because of maxing out connections on the passive side, e.g. valid "race" should get resolved sooner. So there's no point in retrying the same connection forever due to EALREADY. Different non-success rc starts exponetial retry timeout increase mechanism, so it is not affected by this logic.
