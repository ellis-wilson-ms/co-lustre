LU-18643 tests: Do not create subdirectory on client mount

When calling zconf_mount_clients() with a FILESET, the Lustre client
is mounted at the corresponding subdirectory specified by the FILESET.

However, when the subdirectory does not exist, it automatically
creates it transparently. This may hide bugs, e.g., when a test needs
to verify that mounting against a non-existing directory is not
possible.

This patch removes the silent creation of the directory on client
mount, so that it is the caller's responsibility that the subdirectory
exists before mounting. Currently, no tests are relying on this
functionality as they already create the subdirectory themselves.
Therefore no test needs to be modified.

Currently, none of the tests that call `zconf_mount()` or `zconf_mount_clients()` rely on this transparent subdir creation and already create the subdirectory themselves. So I didn't need to update any tests. Based on the current Maloo results, I didn't seem to miss any.

When Maloo successfully finishes, I will update the commit message to make this more clear (Sébastien explained to me how this can be done without rerunning Jenkins and Maloo).

For a bit more context: In the multi-fileset test from the previous patch, there is a test case to mount a subdir that doesn't exist. While the test case works, it is only because a non-privileged nodemap is used that isn't idmapped. Therefore, the transparent "mkdir" during mount fails (EPERM), and the subdir won't exist on mount. As soon as the nodemap is idmapped or is an admin nodemap, these tests will fail (this will be relevant in the following ro-fileset patch).
