One thing we've talked about recently is to split conf-sanity into several parts (LU-14772):
- subtests that do "reformat; something; cleanup; reformat"
- subtests that only do "setup; something on existing config; cleanup"
These are intermixed in conf-sanity today, and are relatively sensitive to changes in ordering, etc.
- upgrade tests from previous versions

There are three main reasons for splitting up the tests:
- shorten the duration of conf-sanity so that it can run in parallel in 1/3 or 1/4 of the wall-clock time
- remove the second "reformat to default" in the subtests, which is a waste of time if the next test does another reformat itself
- avoid unnecessary "cleanup; setup" if the subtests are using the same configuration and do not need this

That might be a useful precursor to adding in a set of subtests that require a separate MGS to run, so they can do this at the beginning
