patch, it was set to 50s. This modification will also modify

--- Message (Andreas Dilger, 2021-12-08 08:48:41.000000000) ---
Patch Set 1:

This patch should also increase the retry count to 3, and fix the lnd calculation to remove the "+ 1" in the divisor, so it doesn't get too small.

I was looking at https://review.whamcloud.com/37253 that originally was increasing the timeout to 150s and retry count to 3.  

The unanswered question there was whether it is possible to have the LNet-level retries use the same match bits/reply buffer as the original message, so that *either* of the sent messages could reply into the same buffer?  That would handle the case where the RPC handling is taking a long time, but a reply is sent eventually (eg. in 60s when timeout is 150s/3=50s, so 3 resends never finishes), as well as the case where the first message is lost and the second send is needed.

In that case, the LNet level timeout and resend would *improve* reliability, rather than the current case where it can *hurt* reliability because it is prematurely timing out messages that would have eventually succeeded.

--- Message (Chris Horn, 2021-12-08 17:42:15.000000000) ---
Patch Set 1:

The timeout hierarchy is: LND Timeout < LTT < PtlRPC timeout. PtlRPC timeout needs to account for both the time on network as well as some processing time on the server. Say the processing time is 5 seconds then we have PtlRPC timeout ~ LTT + 5 = 155, so at_min should be 77-78 seconds. Are you planning to modify the default at_min as well? Have you considered how these timeout increases impact client reconnects when there is some packet loss or in routed configurations when a router goes down?

I'm suspicious of the assertion that an LND timeout of 50 seconds is desirable. 50 seconds is an eternity in the network. Do we have data showing that network transactions really require this length of time to complete? If 50 seconds is actually desirable then my suggestion would be to decrease retry count in order to achieve the desired LND timeout.

LND timeout == 50 can be obtained with either LTT = 101 and lnet_retry_count = 1, or LTT = 51 and lnet_retry_count = 0. Why is 150 LTT and lnet_retry_count = 3 the best configuration?

In short, it isn't obvious to me that these changes make for good defaults. Perhaps some support data could be added to the linked ticket?

--- Message (Andreas Dilger, 2021-12-09 09:58:20.000000000) ---
Patch Set 1:

Chris, I can't say for sure that the timeout=150s, retry=3 is the best option, but definitely timeout=50s, retry=2 ends up with 13s timeouts.

Note that the "transaction timeout" is not strictly about network message latency, but rather the send and reply of Lustre RPCs, AFAIK.  The problem, IMHO, is that having a retry that cuts the timeout interval in 1/2 or 1/3 or 1/4 and starts from the beginning makes reliability *worse* in some cases.  If it were possible to resend the same message at the LNet level, but keep the same reply buffer, then a long-running RPC would still be able to reply into the same buffer, and if the retry was sent, either the server would reject it, or the reply buffer would be "used up" and the client would reject it.

--- Message (Chris Horn, 2021-12-09 21:52:51.000000000) ---
Patch Set 1:

> timeout=50s, retry=2 ends up with 13s timeouts.

Small correction. In master, LND timeout = (lnet_transaction_timeout - 1) / (lnet_retry_count + 1) = 49/3 = 16.

> Note that the "transaction timeout" is not strictly about network message latency, but rather the send and reply of Lustre RPCs, AFAIK.

I don't agree with that, but may we just have different ideas about what the "send and reply of Lustre RPCs" encompasses. Consider the following:

Client:
  PUT(rpc) -> (LTT) -> Server

Server:
  PUT(reply) -> (LTT) -> Client

The deadline on the client's RPC is when the client expects to receive the reply from the server, right? If so, we can see that the deadline on the client's RPC needs to account for 2*LTT plus server processing time. Before LNet health this would have been 2*LND timeout plus server processing time. With 2x resends, it is 6*LND timeout (3 for client RPC, 3 for server reply) + server processing time. With 3x resends, 8*LND timeout, etc.

> The problem, IMHO, is that having a retry that cuts the timeout interval in 1/2 or 1/3 or 1/4 and starts from the beginning makes reliability *worse* in some cases.  If it were possible to resend the same message at the LNet level, but keep the same reply buffer, then a long-running RPC would still be able to reply into the same buffer, and if the retry was sent, either the server would reject it, or the reply buffer would be "used up" and the client would reject it.

I'm a little confused about this. If LND timeout is hit, and LND sets a health status that tells LNet to resend it, then LNet does not do any unlinking. It simply resends the message so there shouldn't be any problem with replies being dropped. And really, that shouldn't be any issue anyways since LND timeout means it couldn't deliver the message to begin with...

So maybe where you're running into trouble is with the LNet response tracking feature? Can you provide more detail about where/how you're running into these issues?
