LU-18746 flr: allow merge stale mirror

This patch makes it possible to merge a stale mirror to an existing
file, so that we can append several mirrors w/o writing data to
those mirrors first, and after we've finished the merge we'd do the
mirror resync later.

There should probably be a "lfs mirror extend" option that adds a mirror to a file but doesn't try to resync it (e.g. for HSM agent/lamigo to do the resync on a different node).

(5 comments)

I don't like that this flag is added as a separate ID in lil_ids[] instead of a flag of lil_flags.  That could cause problems in the future.

It probably makes sense to add a flag that means "allow stale" and could also be used with "LL_LEASE_LAYOUT_SPLIT" or other operations if needed:

        LL_LEASE_ALLOW_STALE = 0x20,

and then add this to LL_LEASE_LAYOUT_MERGE if needed.

Since lil_flags is only used on the client, there is little concern about interop.

At worst, if a new "lfs" command is run and sets this flag in lil_flags on an old client that doesn't understand it, then the file will be closed without creating the mirror, which is relatively safe in this case since the mirror contains no data.

What if this is run on a new client with an old server that doesn't know understand the "stale" flag?  Would the empty mirror be attached to the file and potentially cause data loss (e.g. the usable mirror could be removed since the empty mirror is not marked stale but has no data)?

@robert, this functionality would also be useful for HSM once the "HSM-is-FLR-mirror" functionality is merged.  It would allow attaching a stale mirror to the file layout and then processing that mirror before it is marked "live".

One of the benefits of doing this vs. having a disconnected volatile object is that we don't need to pass the (potentially very large and complex) file layout around to the agents, just "resync mirror" (or maybe "resync mirror N" if the agent can't figure that out itself).
