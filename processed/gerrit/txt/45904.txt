--- Commit Message ---
LU-15388 osd-ldiskfs: wrong dot/dotdot FID for local agent

Wrong FIDs are passed into osd_add_dot_dotdot_internal() in
osd_create_local_agent_inode(). Local agent inode is created to
satisfy e2fsck, and these two FIDs are not used anywhere, which
won't cause any known issue.

Li Xi, the agent inode creation happens when renaming a directory to a new MDT, so something like the following may trigger it:

 # lfs mkdir -i 0 $DIR/$tdir.0 $DIR/$tdir.0/sub0
 # lfs mkdir -i 1 $DIR/$tdir.1
 # ls -lai $DIR/$tdir.0/sub0
 # mv $DIR/$tdir.0/sub0 $DIR/$tdir.1/sub0
 # ls -lai $DIR/$tdir.1/sub0

There would be an agent inode created on MDT1 for the "sub0" directory.

Patch Set 4: Code-Review+1

It looks like this fix is correct, but the calling convention of the various "*_dot_dotdot_*" functions is inconsistent:
- ldiskfs_add_dot_dotdot(..., parent, child, child_fid, parent_fid)
- osd_add_dot_dotdot_internal(..., child, parent, child_fid, parent_fid, ...)
- osd_add_dot_dotdot(..., child, parent, child_fid, parent_fid, ...)

It is also error-prone that ldiskfs_add_dot_dotdot() is using "data1" and "data2", and not "data_dot" and "data_dotdot" or similar, to make this more clear what each argument is used for.

It would still be good to get a test case for this, but at least LU-15579 confirms that this patch is fixing a visible bug.
