LU-13717 sec: filename encryption - digest support

A number of operations are allowed on encrypted files without the key:
- read file metadata (stat);
- list directories;
- remove files and directories.
In order to present valid names to users, cipher text names are base64
encoded if they are short. Otherwise we compute a digested form of the
cipher text, made of the FID (16 bytes) followed by the second-to-last
cipher block (16 bytes), and we base64 encode this digested form for
presentation to user.
These transformations are carried out in the specific overlay
functions, that now need to know the fid of the file.

As the digested form does not contain the whole cipher text name,
server side needs to proceed to an operation by FID for requests such
as lookup and getattr. It also relies on the content of the LinkEA to
verify the digested form as received from client side.

Are there any negative security implications of the "by FID" operations?  For example, could a malicious client be able to unlink files by FID that it wouldn't already be able to unlink by other means?

I don't _think_ so, because these operations are still "path based" in that the "by FID" operations are done within a single directory, and the FID is only used to find which child entry in the directory is affected.

It would be ideal if the server didn't need to compute the hash digest of the filename itself (i.e. if the client sent part of the cyphertext to allow the server match the cyphertext in the linkEA), but I guess any partial *name* match is always prone to error (e.g. "foo01234.jpg" and "foo03456.jpg" might be fooled by either the leading 4 or trailing 4 characters, and this is not an unusual filename pattern).

(2 comments)

Patrick, to quote Andreas' latest comment: these operations are still "path based" in that the "by FID" operations are done within a single directory, and the FID is only used to find which child entry in the directory is affected. This is what the find_name_matching_hash() function does, by looking at the LinkEA.

I do not think that this patch poses a security issue from that perspective. Lustre already supports 'rm by FID' and it _should_ check for permissions etc.

Regarding the need for the server to compute the hash digest, the problem is the client only knows the digested form of the cipher text, made of the FID (16 bytes) followed by the second-to-last cipher block (16 bytes). So as you can imagine, because it is partial this second-to-last cipher text block cannot be used to uniquely identify a file name.
