LU-1866 obdclass: framework for pin/unpin object in/from RAM

For the given object, to efficiently find out whether some linkEA
entry has been verified or not by the LFSCK, the LFSCK needs to
record the linkEA entries verification histrory, especially for
multiple-linked objects.

For such purpose, the LFSCK will pin the object in RAM and attach
its linkEA entries verification histrory again the object. These
pinned object will not be purged out from RAM automatically unless
there is memoey pressure, and there are no other free objects can
be discarded to release memory.

Define some wrap functions for accessing lu_object_header::loh_flags

--- Message (Fan Yong, 2013-01-05 14:11:39.000000000) ---
Patch Set 3:

For multiple-linked object, the LFSCK will do multiple times lu_object_find() and lu_object_put() against the same object. If pinned/unpinned is driven by such two functions, then the object may be released between the prev-find() and next-put(). That is not what we want.

What we need is to keep the multiple-linked object in-RAM until all the name entries pointing to the object have been verified. There are several ways can be used for that:

1) The way like my patch does. The advantage is that the purge thread can release them if RAM pressure.

2) Hold additional reference on the object, and do not call lu_object_put() until all the name entries pointing to the object have been verified. But we need introduce some mechanism to guarantee that the purge thread can release them if RAM pressure.

I do not think the later way can be simpler than my current patch.
