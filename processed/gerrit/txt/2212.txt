LU-1144 ptlrpc: implement a NUMA aware ptlrpcd binding policy

This is a complement to the ORNL-22 general ptlrpcd threads pool
support.

This patch improves the PDB_POLICY_NEIGHBOR ptlrpcd binding policy,
when the kernel supports NUMA. In that case, the ptlrpcd threads are
all binded and have threads in the same NUMA node setup as partners.

Patch Set 7:

Sorry, my comment about the PDB_POLICY_NUMA not being used in this patch was due to confusion with PDL_POLICY_LOCAL (which is used by the RPC submitters).

The current PDB_POLICY_NEIGHBOR code was a simple mechanism intended as a placeholder that approximated a NUMA-aware policy until a proper NUMA-aware policy was added - see the comment at the beginning of ptlrpcd_bind().  The NUMA thread binding is being done in a separate project that is adding NUMA awareness to multiple levels of the Lustre stack (libcfs, LNet, ptlrpc, etc).  Since you have submitted a patch for this already, we should use it until the time that the CPU affinity APIs are added to libcfs.

The existing code for PDB_POLICY_NEIGHBOR can continue to be used if CONFIG_NUMA or HAVE_NODE_TO_CPUMASK is not available, since it still provides some cache affinity.

--- Message (Gregoire Pichon, 2012-03-29 07:27:53.000000000) ---
Patch Set 8:

You can determine indirectly that NUMA is supported by Lustre by looking if ll_ost_io threads are uniformly spread over the system NUMA nodes.

The cpu affinity of the ll_ost_io threads is conditionned by the same #define (CONFIG_NUMA and HAVE_NODE_TO_CPUMASK) than the NUMA implementation of the PDB_POLICY_NEIGHBOR binding policy.

To know if your system is NUMA, simply run 'numactl --hardware'.
