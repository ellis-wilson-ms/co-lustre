for ldiskfs there is a chicken-and-egg problem for configuring persistent mount options. The current approach is for mount.lustre (via libmount_utils_ldiskfs) to read lustre_disk_data from ldiskfs to parse the mount options, then pass them as options to the mount. --- Message (Andreas Dilger, 2021-09-03 22:52:56.000000000) ---
Patch Set 4:

> The server code has no idea this even exist :-( What could go
 > wrong?Seriously the server code should take advantage of this. Also
 > we would have this functionality with ZFS as well which is a win.

ZFS internally stores this fields of this structure as properties on the dataset, which can be read directly without mounting the filesystem, so lustre_disk_data is only read/written directly on ldiskfs.  It is also used as the common data structure in the user mount tools for consistency between ldiskfs and ZFS.

Whether the in-kernel code on the server accesses this directly or not, this data structure forms part of the uapi since it is read directly from the storage as part of mounting each target (OST, MDT), and the exact format of the structure is critical to maintain correct operation, no different than if it were read via ioctl or from a network packet.  The only reason that isn't done is because this information is needed *before* the mount, so it is retrieved via ext2fs_file_read() directly from the block device, but that doesn't make the data structure any less critical as an ABI for configuring the target, and needing to be kept stable like other API data structures.

Conversely, the rest of mount_utils.h is in-memory data structures that are used only internally by the mounting code.
