--- Message (Sebastien Buisson, 2019-06-26 16:14:59.000000000) ---
Patch Set 3: Code-Review+1

I am fine with this, although Lustre behavior will be slightly changed if someone sets send_sepol to non 0 on a client on which SELinux is disabled. Without this patch send_sepol is silently ignored on a client with SELinux disabled, whereas with this patch having send_sepol != 0 on such client will trigger an error message in the console (originating from l_getsepol) and the client request will fail.
Maybe l_getsepol could be improved in that perspective, to print a clearer message when SELinux is disabled and return a suitable err code.

I think it is ok to consider that setting send_sepol to non 0 on a client on which SELinux is disabled is a user error.
