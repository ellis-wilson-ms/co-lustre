LU-16714 utils: Improve DIO/BIO choice in migrate

We default to direct IO in migrate_copy_data, but this is
harmful for performance on small sizes (Roughly < 32 MiB).
Switch to BIO for these sizes, and make this tunable,
because the correct cutover size between DIO and BIO can
differ substantially from system to system.

> does it really matter if it takes .2 seconds vs .1?

It does if you migrate a million files...

 > If lfs handled sets of files it might make more sense, but then
 > don’t repeatedly allocate buffers…

I had discussed with Patrick briefly to add AIO/DIO copying to migrate, which would allow keeping N reads and writes in flight at any time. Potentially this could also be combined with AIO handling for small files so that it would migrate multiple small files in parallel, and call layout swap after the last write of each file completes. 

That would definitely improve the throughput for many small files, and there will typically be many on the command-line at any time. That said, it may not be much better than using "xargs -P N", except it would balance better between large and small files.

Patch Set 7: Code-Review-1

(1 comment)

Sorry for the late flip-flop, I just realized that internally removing the O_DIRECT flags from the file descriptors may not be safe.

For example, for reading mirrored or encrypted file descriptors the data cannot be read/written via the page cache, and require the use of O_DIRECT.  I would like to keep the "IO copy engine" in a single place rather than duplicating it 5x in lfs for different commands.

I think this O_DIRECT decision needs to be done up in migrate_open_files() so that the fds are not messed with for callers that have special requirements.
