LU-13216 ptlrpc: sptlrpc_req_refresh_ctx's timeout semantic

sptlrpc_req_refresh_ctx() is only called with two possible timeouts
in the current code: either we do not wait at all, or we wait
indefinitely.
So change semantic of sptlrpc_req_refresh_ctx()'s timeout parameter,
to be more consistent with wait* primitives. timeout == 0 means
'do not wait at all', and timeout == MAX_SCHEDULE_TIMEOUT means
'wait indefinitely'. Other timeout values are unsupported.

Consequently, remove code that handles the case of a non zero and
non infinite timeout. And remove the unsed ctx_refresh_timeout()
callback.
It solves a problem in the current code, because in case of indefinite
wait, sptlrpc_req_refresh_ctx() is not supposed to try to refresh the
request, so it must not call the ctx_refresh_timeout() callback.

Patch Set 1: Code-Review-1

That's not the only thing wrong with the earlier patch.
If timeout can be zero, we need to wait differently.
l_wait_event() waits indefinitely if the timeout is zero.
wait_event_idle_timeout doesn't wait at all.

The only timeout values passed to sptlrpc_req_refresh_ctx are 0 and -1.

0 means "wait indefinitely"
-1 seems to mean "don't wait at all"

Maybe we should change the callers to pass "0" for "don't wait at all" and "MAX_SCHEDULE_TIMEOUT" for "wait indefinitely.

--- Message (Sebastien Buisson, 2020-02-10 13:49:04.000000000) ---
Patch Set 1:

Hmm, I thought initially that wait_event_idle_timeout() would be like a no-op in case of zero timeout, but it turns out it is more like wait_event_idle() in this case...

Maybe changing the semantic of sptlrpc_req_refresh_ctx()'s timeout parameter would make sense, to be more consistent with wait* primitives. But still, I think we cannot call wait_event_idle_timeout() with timeout = MAX_SCHEDULE_TIMEOUT, because this function masks all interrupts. So behavior would differ from initial implementation.

Also, with the initial implementation, if a non-zero timeout is specified, on_timeout handler is called once timeout expires. If false is returned, we wait indefinitely with non-fatal signals blocked. And only if on_timeout handler returns true we return -ETIMEDOUT.

I will push a new version that checks on timeout value, and calls directly l_wait_event_abortable() in case it is MAX_SCHEDULE_TIMEOUT. Maybe there is a better approach, feel free to update the patch if you want, or if this new version has issues.
