--- Commit Message ---
LU-12521 llapi: add separate fsname and instance API

The llapi_getname() function returns the combined fsname and client
instance as one string, which is fine when using the entire string,
but the output cannot be safely parsed into separate fsname and
instance strings in all cases.

Introduce new llapi_get_fsname() and llapi_get_instance() functions
that return only the fsname and instance strings, since the source
string returned from the kernel can be unambiguously separated before
it is returned in a combined string via llapi_getname().

Fix the lfs_getname() '-n' and '-i' options to use the new routines
rather than parsing the output from llapi_getname().

Add man pages for these functions.

Fixes: 2a4821b836c8 ("LU-12159 utils: improve lfs getname functionality")

I've made an update to the patch that should address this problem. 

 > Besides this being a ASLR violation you need to make it clear to
 > the end user that the instance is not consistent across nodes for

 > remounts. Sadly these properties make the instance pretty useless.

James, you are conflating the instance with the FSID, which *is* consistent across nodes and mounts. The instance is *intended* to be different between each mount, so that if the same filesystem is mounted multiple times on the same client then it will have a different instance identifier. The FSID is the same across all mounts of the same filesystem and is useful for eg. NFS file handles. 

Separately (not addressed by this patch) the client has a per-mount UUID that identifies the client to the servers. This is currently different than the instance, but I think we could make the instance the same as the UUID, though it might potentially make parsing the instance more complex.

I've clarified this in the man page a bit. The current patch is no more of an ASLR violation than it was before the patch. This will be fixed when the instance is not related to a kernel address, which is a totally separate patch.

The lustre_user.h::LUSTRE_MAXFSNAME is only 8 chars, and while it would be nice to increase fsname to 16 chars or more in the future, it is currently restricted by the 16-byte ext4 s_label field that also needs to hold the 8-byte device type/number (e.g. -MDT0000).

 > So the external representation needed to be the binary UUID
 > representation which is UUID_MAX (16) in size.

The lustre_user.h::UUID_MAX is 40 bytes, which is large enough to fit the string representation of a real UUID.  To avoid potential conflicts for that #define with something else, we should consider renaming it LUSTRE_UUID_MAX or something in our headers/code, since the current name is rather generic?  I don't think we want to use the binary UUID as part of the printed sysfs filename...

So, I don't _think_ there is a limitation in the code, but maybe I'm missing something.

My discussion here was about the temporary buffers used by llapi_getname(), which might be linked into a userspace application independent from what the kernel is returning.  Previously, the full buffer was supplied by userspace and copied into directly.  With this patch there are two temporary buffers for the fsname and instance, so that the code is more modular, and I just didn't want them to be too small.

I'd prefer to go ahead with this patch, and we can work out the details of the instance-is-UUID change later as that is implemented.  I don't think it will change the APIs being added here, which take the buffer sizes as arguments.
