LU-13151 mdt: add parent FID to Changelog recordss

Use the link EA to add the parent FID to ChangeLog records, including
MTIME, TRUNC, and SATTR.

Some tools that maintain copies of filesystem metadata in an external
database monitor changelogs for changes to the filesystem, in order to
determine files that need to be rescanned. This can result in a large
number of small updates to the external database that can reduce the
tool's ingest performance. It might be beneficial to instead track and
scan complete directories that contain modified files and update the
external database using bulk operations. Adding the parent FID to
MTIME changelogs allows to more efficiently determine the parent
directories for some types of file data modifications, by issuing
OBD_IOC_FID2PATH once for each parent FID, instead of once for each
file FID.

I think that the expectation here is that having the parent FID in the Changelog record avoids the need to call FID2PATH on every file in the Changelog, and instead call it once on the parent directory FID to generate the majority of the path (possibly caching the pFID->path component), and then append only the filenames from LinkEA for each file.

Alex had also started to work on an optimized version of fid2path in patch https://review.whamcloud.com/34740 that would cache the directory component rather than generating it new each time.  Having a common implementation of optimized fid2path lookup would be useful for everyone.  Getting the pfid along with each Changelog record would help this as well.

> I think that the expectation here is that having the parent FID in the Changelog record avoids the need to call FID2PATH on every file in the Changelog

I understand that, but my question is rather: why would you need to call FID2PATH in the first place? MTIME indicates a data change, how are paths involved at all?

If I understand Nikitas' message correctly, getting the path is just a means to an end: it allows knowing the path to the parent, which is then used to group re-scans of the FS.

What I still do not understand is why grouping by directory is better than grouping by chunks of N updates. Unless maybe, you use open(parent) + readdir() + openat(parent_fd, child) periodically for your updates. But that seems much more wasteful [*] than just using llapi_open_by_fid() for each MTIME record. You can still batch the calls to llapi_open_by_fid() if it helps the database.

[*] and error-prone: entries move around in the namespace, an entry that was under directory X at some point might be under Y when you decide to look for it.

My guess is that if you want to selectively replicate subdirectories to a remote system, or if there are different policies on a per-directory basis, then knowing the parent FID immediately in the ChangeLog, rather than having to look up the parent FID(s) from the inode, then the records can be quickly processed/skipped depending on the pfid.

There are of course some potential issues with that proposal (e.g. if there are hard links in different directories with different policies, but that is a corner case.
