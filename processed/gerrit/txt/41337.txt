--- Message (Andreas Dilger, 2021-01-28 03:27:27.000000000) ---

(4 comments)

In theory, it would be possible to use fid2path to build the pathname incrementally with multiple calls.  It is already able to stop the upward pathname traversal for a specific parent directory so that it works properly when called against a subdirectory FID (e.g. "lfs fid2path /mnt/lustre/subdir [FID]", when the root is /mnt/lustre).

In theory, fid2path could be hoisted entirely into userspace by just reading the "trusted.link" xattrs directly (or an ioctl equivalent) for each path component, then recursively doing the same for each parent FID to assemble the full pathname.

We've discussed doing this in llapi_fid2path() or a new llapi_fid2path_cached() function to speed up the pathname generation for many FIDs in a list.  That way, the client would build up a cache of FID->parent pathname mappings, and could reuse them for each child fiD.
