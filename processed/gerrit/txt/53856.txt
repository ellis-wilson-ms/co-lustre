LU-17478 osc: add dio_exts tracking

The addition of unaligned DIO means we'll need to copy user
data in order to do some DIO RPCs.  This means we can't do
this from userspace threads (except the one which submitted
the io) and must do it from ptlrpcd threads.

This is fine for normal IO, but a problem comes in when a
user thread calls fsync.  All DIO extents are high priority
and are synced immediately, but it's possible for an fsync
to find them before they are pulled in to an RPC.  In that
case, the user thread will try to do the DIO RPC rather
than leaving it to be picked up by the next ptlrpcd thread.

This causes a memory violation as one user thread tries to
take on the memory ("use_mm") of another one, which isn't
allowed.

The solution to this is to move DIO extents to their own
extent queue and exclude this from usage by user threads.
