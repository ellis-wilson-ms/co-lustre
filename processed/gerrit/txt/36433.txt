LU-12275 sec: control client side encryption

Client enables encryption by default. However, this should be
possible only if server side is encryption aware.
Moreover, we want to give the ability to decide which clients can
make use of encryption, by extending the nodemap mechanism with a
new 'forbid_encryption' property, set to 0 by default.

(1 comment)

The only reason to refresh this is for the kstrtobool() usage.

One other question is whether the parameter should be "allow_encrypt" or "deny_encrypt", or something more neutral?  The main reason I ask is whether we'd want to allow this feature to be enabled for clients by default in the future if the backup/restore issues are fixed?  I guess at that point we could just change the default and the admin would have to set "allow_encrypt=false" for clients that can't use this.

--- Message (Sebastien Buisson, 2019-10-14 12:28:14.000000000) ---
Patch Set 1:

(1 comment)

The proposed name for the new nodemap property is 'allow_encryption', and it is set to 1 by default, including on the default nodemap. I would not like to have something more neutral like 'encryption', because it is important to understand that it is not about enforcing encryption, but about allowing (or forbidding) it. Indeed, from a server standpoint, we are not able to impose that every file or directory is being encrypted. All we can do is to control whether clients enable encryption at mount time, and prevent them to do so if we do not want encryption at all from that given group of clients.
