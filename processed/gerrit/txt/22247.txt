--- Commit Message ---
LU-2525 ldlm: avoid reordered async flock lock and unlock

For async flock case, the unlock request may be triggered
before related lock RPC handled by the MDT. If such case
hanppend, it will leave a ldlm lock on the MDT side that
cannot be released until the client is umount, or the
client is evicted by the MDT.

To avoid the trouble, before sending the flock unlock RPC
to the MDT, we search related flock ldlm lock locally. If
it is still in 'lr_enqueueing' list, then means related
flock lock RPC has not been handled by the MDT yet. Wait
there until related lock is handled by the MDT or failed
or cancelled.

why to re-write the current code? this patch is not integrated into already existing code well, the right way would be to adjust the current code to what is missed - to add ll_file_flock_async_unlock() if you think it is missed somewhere. otherwise, this patch will have to become much larger due to a deeper integration.

all in all, I understand that there could be different implementations and this is one of them, however I do not see much benefits of it over the original one, I also commented in the original gerrit ticket that there is a current exported VFS (not NFS) API which is supposed to be used by any callers, thus I do not think it is worth to re-write the existing patch. and what is the most important, it must not be a blocker for async flock feature sign-off.

although if you have some time for this and you would invest your time into adding a new test checking the VFS API has not changed, this is what would be useful.
