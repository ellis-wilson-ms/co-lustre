--- Commit Message ---
LU-8066 obd_type: discard obd_types linked list.

As all obd_types are kobjects in the lustre_kset kset,
they are linked together in that kset and don't
need any extra linkage.

There are non-obd_type objects in lustre_kset, added by
class_setup_tunables().  These have a different ->ktype, so we are
careful to only return objects with the correct ->ktype.

As kset_find_obj() returns a counted reference, we need
to put that reference when done.

On the server side it is possible to have an obd_type partially
initialized by one subsystem and latter fully initialized by
another subsystem. We use typ_sym_filter to notify us if the
obd_type is only partially setup. If it only paritially setup
then we let the original subsystem that created the obd_type
to clean it up. If the obd_type was latter completely setup
then we let the latter subsystem do the cleanup for us.

(1 comment)

That crash is interesting....
The cause of the crash is:
 BUG osc_thread_kmem ... Objects remaining in osc_thread_kmem on kmem_cache_close()

So something was allocate from osc_thread_kmem, not freed.
However - this is happening at module-unload time, and every time
something is allocated from the kmem_cache, a ref is taken on the module.  So the modules *cannot* be unloaded while there are object still allocated....
Unless.... something is mucking up the module refcounting.
I cannot see any problem with the refcounting through ->lct_owner.
The refcounting in osc_iocontrol() is unnecessary but not broken.
That leaves the refcounting through o_owner.
Every time we take a ref there, we increment typ_refcnt.
Every time we drop a ref, we decrement typ_refcnt.
These happen from class_newdev and class_freedev.

But there is another call to class_put_type() which drops a ref on the module from lu_stack_fini().  It is less obvious that this is correct.
In fact, it decrements typ_refcnt before calling class_put_type(), and class_put_type() decrements it again.  That certainly looks wrong.

However .... ldt_obd_type is never set, so that code in lu_stack_fini() is never run.  So it cannot cause a problem with the module refcount.

So we are back to square one ... it appears that the module refcount is decremented too much, but it isn't clear where.

This patch does make changes to class_get_type(), which takes a ref on ->o_owner, but I cannot find a problem with that change.

I'm stumped - for now.

Patch Set 7: Code-Review-1

(2 comments)

I think I understand the crash now.
My previously analysis was incomplete because I didn't allow for the use of LCT_NOREF, which is quite wide-spread.
For LCT_NOREF usage, we depend on lu_context_key_degister to free keys, and this is called from class_ktype.release.
So it seems likely that a one of these kobjects isn't being put enough times.

There are two different kmem-caches which have sometimes cause problems: osc_thread_kmem and lov_session_kmem
These are in the only two modules that are affected by kobject sharing: osc shares with osp and lov shares with lod.

As I describe in a separate comment, we are a couple of kobject_put() calls missing in class_register_type.  I think that one that should be near line 287 is causing the problem.  When lod is loaded before lov, or osp before osc, the refcount gets messed up and the .release function is never called. So when there modules get unloaded, the are allocations that haven't been freed.
