LU-9098 lnet: Move lnet_routes.conf to /etc

lustre_routes_config now skips lines which are blank or
have a first non-whitespace character of #. This enables
comments to be placed in /etc/lnet_routes.conf. Also,
provide a sample lnet_routes.conf.

The SysV lnet script now expects lnet_routes to be
in /etc. Updated docs to reflect the new location.

From my understanding, directly sourcable files belong in /etc/sysconfig. /etc is used for configuration files which require parsing to be used. lnet_routes.conf is not sourcable and requires parsing which is why I am suggesting to move it to /etc.

A good (and more in depth) description regarding the difference can be found at http://0pointer.de/blog/projects/on-etc-sysinit.html.

--- Message (Christopher J. Morrone, 2017-02-21 22:05:03.000000000) ---
Patch Set 3:

Yes, there are quite afew things about lnet_routes.conf that could have been better.

I would argue that we don't want an /etc/lnet subdirectory, because ultimately we don't want a proliferation of lnet configuration files.  One /etc/lnet.conf file should be the long term plan.  /etc/lnet_routes.conf is a hackish (but necessary at the time) solution that can be retired in the near future when we go to lnetctl for configuration.  When we have lnetctl use as the primary configuration interface, we should be able to configure everything about lnet (including what has to happen with lnet_routes.conf right now) in a single lnet.conf file.

Possibly there could also be an /etc/lnet.d as well, if needed at some point.

Should lustre_routes_config be renamed and moved inside the source tree?  If we planned to make that a long-term solution, almost certainly.  I personally don't care much since I think the goal should be to eliminate that script entirely.

The main purpose for the changes here are to lay ground work for systemd patches that are coming later.  Because lustre_routes_config could not accept a file containing only comments and whitespace, it imposed a requirement that the script be called conditionally.  That was easy with SysV scripts, which are literally shell scripts.  With systemd, it imposed an extra burden of intermediate scripts that the systemd unit file needed to call in order to avoid an error from lustre_routes_config.  This change allows a call to lustre_routes_config to be made in a systemd unit file, and lustre_routes_config will no longer return an error when the config file is just comments and whitespace.

In short, yes, there are lots of additional things that could be done, but since lustre_routes_config is going to be obsolete soon, I would suggest that work can either be done in another patch, or not at all.
