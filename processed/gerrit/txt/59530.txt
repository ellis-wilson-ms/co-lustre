--- Commit Message ---
LU-19062 llapi: add layout pattern string functions

Add llapi_lov_pattern_string() to print arbitrary pattern flags to
a string rather than layout2name() which only can print specific
hard-coded combinations of patterns.

Add llapi_lov_string_pattern() to convert layout pattern names to
flags.

Add enum lov_pattern that holds LOV_PATTERN constants, and use it.

Add description of patterns to lfs-getstripe.1.
Restore "-L, --layout" argument listing to lfs-setstripe.1.

--- Message (Andreas Dilger, 2025-06-13 16:06:34.000000000) ---
Patch Set 7:

James, the size of the enum is constrained by the values used, and

 LOV_PATTERN_DEFAULT =		0xffffffff

will ensure it is a u32.  Also, there are checks in lustre/{utils,ptlrpc}/wiretest.c to verify the field sizes:

         LASSERTF((int)sizeof(((struct lov_mds_md_v1 *)0)->lmm_pattern) == 4, "found %lld\n",
                 (long long)(int)sizeof(((struct lov_mds_md_v1 *)0)->lmm_pattern));
        LASSERTF((int)sizeof(((struct lmv_desc *)0)->ld_pattern) == 4, "found %lld\n",
                 (long long)(int)sizeof(((struct lmv_desc *)0)->ld_pattern));    


so there is no danger that the compiler could change the size randomly without it being noticed.

It would be possible in GCC to specify a field size for the enum, like:

       enum lov_pattern       lov_pattern:32;

as is already used internally in some places to reduce struct size: 

       enum ldlm_mode         l_granted_mode:9;

but not all compilers/tools like this yet, so I didn't add it to the lustre_idl.h file for maximum userspace portability.
