LU-13118 lustre: use UUID for naming instead of internal pointers

According to ASLR it is considered a security violation to publish
internal kernel addresses to user land. Currently lustre uses
internal pointer addresses to create an unique identifier that
ends up being used to build the sysfs tree. This patch switches
this to using the UUID generated. The reason for using UUID is
that in the ptlrpc layer there is a mapping between NIDs and
UUIDs and many client interfaces have the same UUIDs so we
could cheat and use the device name instead.

Instead of using a pointer as the instance uniquifier, we could just use a pseudo-random 64-bit number?  The main reason to use a struct pointer was that we knew it would always be a unique value on each client no matter how many filesystems it had mounted, while a purely random number could hit the birthday paradox every few billion mounts.

We will need to use something more than some random number. Lustre internally has this mapping of:

mgs (logs -> lcfg using UUID) + ptlrpc (uuid -> conn) + (uuid -> obd device)

Mind you that mapping is broken in that instead of UUIDs nid strings are places into struct obd_uuid which is wrong. Also the mapping in ptlrpc of UUID to a conn just becomes a NID to NID mapping. The problem with virtual client devices is the UUID are all the same. So we need to think of clever way to things like the uuid to conn mappings.

Need to think about all this.

Patch Set 7:

I don't know if there are user scripts that depend on this being in the form of a long hex number or not. Anything we have to change in test-framework.sh or the test scripts is usually an indicator that some user script may also depend on the same. 

In particular, a UUID with more '-' in it could confuse parsing this parameter.

That said, using the client UUID has the benefit that this is used/referenced in various places on the servers (eg. log messages and such) so it would make it easier to trace log messages on the server back to a specific client mount...

If the next release is going to be "Lustre 3.0" instead of "Lustre 2.17" then that might be justified.  Alternately, we could have a tunable like "enable_instance_uuid" that shows the instance as a UUID instead of a pointer (or pointer-like value) so that this can be enabled/disabled in the field as needed.  For 2.x it would be disabled by default, and for 3.x it would be enabled by default.
