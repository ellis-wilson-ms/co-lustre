LU-12349 llite: console message for disabled flock call

When flock option is disabled on a Lustre client, any call to
flock() or lockf() would cause a return value with failure.
For applications that don't print proper error message, it is
hard to know the root cause is the missing flock option on Lustre
file system. Thus this patch prints following error message to
the tty that calls flock()/lockf():

"Lustre: flock disabled, mount with '-o [local]flock' to enable"

Such message will print to each file descriptor no more than
once to avoid message flood.

In order to do so, this patch adds support for CDEBUG_LIMIT(D_TTY).
It prints the message to tty. When using this macro, please
note that "\r\n" needs to be the end of the line. Otherwise,
message like "format at $FILE:$LINO:$FUNC doesn't end in '\r\n'"
will be printed to the system message for warning.

> IF it's a sysadmin making that change (to disable flock on by default) - well, it's their job to inform the users as such as well.

As far as I can see, even sysadmin doesn't know preciesly whether an application will be affected by disabling flock or use localflock. We've asked for multiple times when sysadmin wants to "optimize" the flock option but isn't sure whether it will affect the application or not. Without this patch, there is no easy way to tell whether a failure of appilication is caused by disable flock or not. With this patch, it is much easier for an user to notice that the failure of his application is caused by the disable flock feature.

> Should we just mount with -o flock by default 
Sure, we should. But unless we remove -o noflock completely, it is still possible that flock feature is disable and causes problem to the application.

> Informing users is all well, but if they are not listening (batch mode and such) - there's no way to inform them anyway I suspect so should we really try to create complicated mechanisms to combat that in hopes some additional (small?) percentage of users would get the message as well?

I don't think the percentage of users that will be better notification is small. The users that never use flock won't be bottered by this patch, which is desirable. But the users that have failed applications caused by missing flock feature can easily find the message as soon as they run their application in none-batch mode (or even in batch mode without closing their tty.). Without this patch, the failure of flock syscall is completely silent. There is no easy way to find the cause, unless using "strace" or other debug tools.

And judging from the size of this patch (+60 lines), the mechanism is not so complex. And this mechanism can be easily reused in other circumstances that needs to notifiy the users about emmergencies, e.g. quota/space/QoS outage. So I think it worths to maintain extra 60 lines of codes to get the benifit of this new feature.
