LU-15595 lnet: Tweak route updates in lnet_notify

When notified that some lpni is DOWN we can set routes down as
appropriate.

When notified that some lpni is UP we currently set those routes as
UP, but I think this is probably too aggressive. We should instead
queue the router for discovery. Since we know the lpni is UP, we
should be able to discover it successfully and get an accurate
accounting of route status through the gateway.

Patch Set 8: Code-Review-1

I think there is a potential race condition with MR routers.

  A@o2ib -
           \
             - Router - C@gni
           /
  B@o2ib -

LNet peer may have single lnet_route:

  net           gni hops 4294967295 gw                          A@o2ib up pri 0

lnet_notify for A@o2ib or B@o2ib can impact route aliveness

Suppose A is down, and B is up.
Thread 1:
lnet_notify (A@o2ib, UP)
Thread 2:
lnet_notify (B@o2ib, DOWN)
Thread 2 executes first through lnet_is_gateway_net_alive so llpn_alive = false
Thread 1 then executes through lnet_is_gateway_net_alive so it sees llpn_alive = true
Thread 1 exits
Thread 2 marks all routes down, incorrectly. (edited) 

It looks like there is a bug here, and we could solve it by using an atomic counter.

--- Message (Chris Horn, 2022-06-02 19:18:22.000000000) ---

Actually, the race is when both NIs are UP and we get notified that they are down.

Suppose A is up, and B is up.

Thread 1:
lnet_notify (A@o2ib, DOWN)

Thread 2:
lnet_notify (B@o2ib, DOWN)

Thread 2 executes through lnet_is_gateway_net_alive so llpn_alive = true

Thread 1 then executes through lnet_is_gateway_net_alive so it sees llpn_alive = false

Thread 1 marks all routes down, correctly

Thread 2 marks all routes up, incorrectly.

We could use counters to solve this, but that would require keeping track of # of local NIs and # of remote NIs, separately. Then route is alive iff (#local NIs UP > 0) && (#remote NIs UP > 0).

Alternatively we don't allow concurrent access to this route update logic (go back to locking cpt0, or use some spinlock on the peer).
