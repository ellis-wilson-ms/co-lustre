--- Commit Message ---
LU-11102 ldlm: don't skip bl_ast for local lock

The previous commit 954cc675 skips bl_ast for local lock but
there are cases on MDT when local lock can become a client lock,
see mdt_intent_lock_replace(). In that case the client should be
notified if lock is a blocking lock.

Patch reverts commit 954cc675 and provides alternative solution.
During downgrade to COS the lock renews own blocking AST states
and start reprocessing. Any new lock conflict will cause new
blocking AST and related async commit as needed.

I am not sure I like the solution chosen in this patch (in fact in the original already landed LU-11102 patch). as the result it gets many special handlings which I do not think are necessary. we had a talk with Mike about it and he agreed the following idea sounds reasonable: 

- the logic of the ast_sent flag could be restored, thus the current patch is not needed;
- to not miss the bl_ast calls for a COS lock, we could make some checks at the downgrade time;
- thus, e.g. we could just drop the ast_sent flag (altogether with zroing blocking_lock and removing from the l_bl_ast list) and reprocess at the time the lock became the COS one;
- it opens a way for future conflicts to run bl_ast as well as to check if there is any conflicts already sitting in the waiting queue;

--- Message (Lai Siyao, 2018-10-09 03:21:22.000000000) ---
Patch Set 5:

Vitaly, the race still happens before lock downgrade to COS mode, for example:
1. a PR lock1 conflicted with an PW lock2, then ran bl_ast, but did nothing.
2. a PW lock3 from a distributed operation came, but found ast_sent set, then waited.
3. lock2 was downgraded to COS mode, but didn't trigger CoS.

Now lock3 still needs to wait lock2 to release upon transaction commit. The root problem is if a conflict lock won't trigger CoS, it shouldn't run bl_ast, otherwise subsequent conflict lock can't trigger CoS but wait.

--- Message (Vitaly Fertman, 2018-10-09 12:03:02.000000000) ---
Patch Set 5:

> The root problem is if a conflict lock won't trigger CoS, it shouldn't run bl_ast
this should not be a problem, it should be a delayed cancel as on the client side - a lock is just to be marked as pending in meanwhile (well, it is already marked as ast_sent, probably enough)
> otherwise subsequent conflict lock can't trigger CoS but wait.
this is because you does nothing at the time of downgrade - like an the client side when you finish the work and the lock is marked pending it is not placed to LRU but cancelled instead; the only difference is that after the downgrade the original PR conflict does not conflict anymore with COS, and you just need to understand if there are other conflicts in the waiting queue and be ready for future conflicts. thus, I would  consider the COS lock as a new one (instead of replacing it by a new lock, it is enough to clean the blocking info on it) and do reprocess_all. as there will be no ast_sent on the time of reprocess, the next bl_ast should be triggered well
