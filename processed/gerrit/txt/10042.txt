LU-4675 lfsck: new pattern flag for partially repaired file

When the layout LFSCK repairs orphan OST-object, if the parent
MDT-object was lost, then it will re-create the MDT-object and
regenerate the LOV EA and fill the target LOV EA slot with the
orphan information, and fill other slots with zero (LOV hole);
if related LOV EA slot is invalid or hole, then it will refill
the target LOV EA slot; if the target slot exceeds current LOV
EA tail, then extend the LOV EA, and fill the gaps as zero.

Some of the LOV EA holes may cannot be re-filled finally because
of lost some OST-objects. And even if they can be re-filled, but
there are still some possible race accessings from client before
the re-filling. If the client access the LOV EA with hole(s), it
may cause some strange behaviour, such as trigger LBUG()/LASSERT()
on the client.

So we will make the client to be aware of the LOV EA is incomplete.
We introduce a new LOV EA pattern flag LOV_PATTERN_F_HOLE for that:
any time when the LFSCK repairs the LOV EA with hole(s), the LOV EA
will be marked as LOV_PATTERN_F_HOLE; when all the holes in the LOV
EA are refilled, the LOV_PATTERN_F_HOLE will be dropped.

For a new client, it recongizes the pattern flag LOV_PATTERN_F_HOLE,
then it can permit/forbid some opertions on the file with LOV holes:

1) Normal read/write the file with LOV EA hole is permitted, but the
   application will get EIO error when read data from the dummy slot
   or write data to the dummy slot.

2) The users can dump the recovered data via some common read tools,
   such as "dd conv=sync,noerror".

3) Append data to the file which has LOV EA hole will get EIO failure.

4) Other operations will skip the LOV EA hole(s), and will not get
   failures, such as {s,g}etattr, {s,g}getxattr, stat, chown/chgrp,
   chmod, touch, unlink, and so on.

For an old client, since it will not recognize the new pattern flag
LOV_PATTERN_F_HOLE. So the LOV EA with hole will be dicarded with
failure, but it will not cause the client to be crashed.

Sorry, the commit message may be some confused. "The modification which will change something on OST-side will be denied" means "-EIO", not "-EPERM". There are two main reasons for that:

1) We do not want to hide the data lost case by creating OST-object implicitly unless it is specified when the LFSCK is triggered.

2) Even though the administrator requires the LFSCK to create the lost OST-object, the LFSCK may still cannot know on which OST to create the OST-object, because the LFSCK cannot repair the OST-pool information in the LOV EA. If the LFSCK creates the new OST-object on the OST which does not match the original LOV EA, it will hide the failure and may misguide the applications.

So the LFSCK will not create OST-object for the LOV EA hole. Since the file with LOV EA hole is incompletely, it is better to keep the file (OST-object) unchanged. So "The modification which will change something on OST-side will be denied".

In fact, I do not know that is the best solution. Then what's your idea for that?

--- Message (Fan Yong, 2014-05-20 09:14:13.000000000) ---
Patch Set 13:

Oleg:

1) When the LFSCK set/clear the LOV_PATTERN_F_HOLE against the LOV EA, it has already taken the layout ldlm lock which is done by the caller via lfsck_layout_lock(). So your worry about race cases have already been considered and handled.

2) About the LFSCK connect flag. Originally, I used new LOV EA magic (called MAGIC_HOLE) for the LOV EA with hole, then old client cannot recognise it, and will report failure automatically. But Andreas pointed out that such patch will prevent the LFSCK to repair other LOV EA corrupt cases. For example, if the original LOV EA is V3 magic, but it has hole for some reason, and the LFSCK needs to mark it as MAGIC_HOLE  temporarily to indicate that there is a hole in the LOV EA, and such new magic should be replaced with its original magic when the hole is re-filled as the LFSCK processing. But at that time, the LFSCK cannot know whether set the magic as V1 or V3. So Andreas suggested to use new LOV EA pattern flag to indicate that. Unfortunately, the old client does not report failure for its unknown LOV EA pattern flag, and will continue to use the LOV EA with hole, then it will trigger LBUG() on the old client. That is why the new patch checks whether the client is old or not, and deny the accessing LOV EA with hole for old client.
