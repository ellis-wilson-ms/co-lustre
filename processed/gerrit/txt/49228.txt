LU-16339 quota: notify OSTs until lge_qunit_nu is set

There is a window when locks are not granted yet, but
lqe is set to qmt_reba_list to send updates to OSTs.

t1: lqe_init()->qmt_setup_lqe_gd->qmt_seed_glbe()
t1: lqe_init()->qmt_setup_lqe_gd->qmt_id_lock_notify()
t2: qmt_glimpse_lock() lustre-QMT0000: no granted locks to send glimpse
t1: ldlm_lock_enqueue()->ldlm_granted_list_add_lock() ...

If lge_qunit_nu was set to 1 in qmt_seed_glbe and appropriate qunit
is equal to the least_qunit, new qunit won't be sent to OSTs
and finally lqe_revoke will not be set causing endless -115 errors.
The fix calls qmt_id_lock_notify into qmt_dqacq0 for an lqe that has
set lge_qunit_nu or lge_edquot_nu.

Add test 85 into sanity-quota to check that write
doesn't hung if qunit initial value is equal to
the least_qunit due to small block hard limit.

Sergey. it looks like all of your recent patches have conflicts in sanity-quota due to adding tests at the end of the file.  If you are trying to avoid dependencies between the patches, one option would be to add the tests in different parts of the file so that there is no context conflict between them.

For example, there are gaps in the sanity-quota.sh test numbering at 14, 25. 28, 31, 36, 42, 58 that go back several years and could be filled.  You could add sub-subtest numbering if any of these tests are related to existing tests (e.g rename test_23 to test_23a and add test_23b, or similar).

Also, to avoid similar issues in the future, it makes sense to leave some gaps in the numbering at the end of the file, so that there are discontiguous ranges to add tests in the future.  For example, instead of adding test_84, 85, 86, 87, you could add test_100, 125, 150, 175.  While these would still conflict today, as soon as any of them landed then future patches could be added in the gaps without conflict.
