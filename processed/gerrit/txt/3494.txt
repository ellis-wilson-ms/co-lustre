--- Message (Andreas Dilger, 2012-08-24 08:05:53.000000000) ---
Patch Set 2: I would prefer that you didn't submit this

(10 inline comments)

As a general comment, while I think this is a very useful patch and I think we need to get this done, it will be some time before we can safely use any of the flags reclaimed by this patch, since older clients may set them sporadically and send them over the network.

Some of them may in fact be completely unused in 1.8/2.x so could start to be used sooner.  It makes sense to document which flags are completely unused, and which ones will only become unused in 2.3 or 2.4 (depending where this patch lands) and cannot be used until we no longer require interoperability with those releases.

Another possibility is to send an "OBD_CONNECT_LDLM_FLAGS" flag for some time to indicate that the client/server have moved these internal flag values out of the wire flags, and they are safe to use.  At some point (e.g. 2.9.50) we could assume that the flags are "clean" regardless of whether OBD_CONNECT_LDLM_FLAGS is sent, but continue to send it for interop with older clients/servers, and by 3.4.50 just stop sending or masking OBD_CONNECT_LDLM_FLAGS entirely.  That schedule could be moved up if this patch gets into 2.1 and 2.3, and having the OBD_CONNECT_FLAG would avoid the need to check version numbers and allow porting this patch to any release.

--- Message (Andreas Dilger, 2012-09-12 04:01:34.000000000) ---
Patch Set 5: Looks good to me, approved

(1 inline comment)

One question that remains unanswered is when can we begin to start using these newly-liberated flags on the network?  Otherwise, the old clients/servers will be sending their internal flag values over the network, and confusing the peers that are expecting particular wire-value flags.

Oleg, what is your thought about also landing a patch like this for b2_3 and b2_1, maybe b2_2 also?  That would mean that we could start using these flag values around 2.5, if we assume that there is no longer interoperability requirements with 1.8.x after 2.4.x.

Possibly a better solution is to define a new OBD_CONNECT_LDLM_FLAGS (0x4000000000000ULL) which signals that the client/server has applied this patch and is safe to use newer flag values.  Initially, nothing is requiring this feature, but it will help a great deal in the future when we DO need to use one of these flags for some reason.

Patch Set 5: (1 inline comment)

"Otherwise, the old clients/servers will be sending their internal flag values over the network, "

actually, only non-wire flags get changed value, so no wrong flag will be sent over network. the issue is opposite, new client will send a new flag which will be considered as an old non-wire flag by old client, but will never be sent back over network by old client.

I think each new flag could be considered separately when introduced. at that time a particular "Used to be" line is to be removed and this will play a role of reminder to check when this value was used last time and still is use somewhere.

besides that, such a new features which will send new flags over network usually get a new obd_connect flag, i.e. seems to be well protected.
