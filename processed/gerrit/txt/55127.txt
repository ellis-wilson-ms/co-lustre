--- Commit Message ---
LU-17852 gss: do not use expired reverse gss contexts

On server side, a reverse context matches a gss context established
on client side. These reverse contexts have a expiration time, and are
replaced with fresh ones when they expire.
So get rid of expired reverse contexts when we find them in the
gsk_clist. And when we look up for a context, do not continue using
the current one if it is expired.

Add sanity-krb5 test_200 to check the expired reverse contexts.

--- Message (Sebastien Buisson, 2024-05-17 05:51:27.000000000) ---
Patch Set 1:

> Patch Set 1:
> 
> This makes sense, but makes me wonder what happens if this timeout expires before the client has completed sending the RPCs (e.g. in a recovery situation)?  Do the clients regenerate the RPCs with new context on resend, or do they keep retrying with the old context, or (worse) get an error and fail once the context has expired?

If the timeout expires before the client has completed sending the RPCs, what happens is what is described in LU-17852.There will be a "Invalid gss ctx" message on the client, associated with a "client might have lost the context (NO_CONTEXT), retrying" message on the server. Then the server will retry with the newest context.

I will improve this patch to also fix sec_lookup_root_ctx_kr(), so that we check for expiry of current context (gsec_kr->gsk_root_ctx). Otherwise, sec_lookup_root_ctx_kr() always uses gsec_kr->gsk_root_ctx if it is defined, even if it has expired.
