> Perhaps it's the removal of the second wait_for_completion()? Not sure... Do you have more details on how I could reproduce the crash?

My guess is that the removal of the second wait_for_completion() is part of the problem, but even restoring that isn't a complete fix.

Suppose some other CPU is calling lnet_ping_event_handler() just as the wait_for_completion_interruptible_timeout() is either interrupted or times out.
Suppose further that the other CPU has assigned 1 to pd_unlinked, but not yet called complete().
Now lnet_ping (either the old or new code) will see that pd_unlinked is set, so it won't call LNetMDUnlink() or wait_for_completion(). As pd.replied is zero the function will soon exit and  the stack where 'pd' lives could be used.

Then the other CPU finally gets around to calling complete() and it sees a corrupted completion and dies.

I think that we need to capture the return status of wait_for_completion_interruptible_timeout() in a "long" and after the pd_unlinked test, check if the result is <= 0 and if so, wait_for_completion().
i.e.

long res;

res = wait_for_completion_interruptible_timeout(....);
if (!pd.pd_unlinked)
    LNetMDUnlink(pd.mdh);
if (res <= 0)
     wait_for_completion(...)
if (!pd.replied) {
    .....

i.e. we must *always* successfully wait for completion.  Calling LNetMDUnlink() expedites the completion, but doesn't guarantee it has happened.
In fact it is safe tocall LNetMDUnlink() if the md has already been unlinked, so I would make it.

res = wait_for_.....
LNetMDUnlink(pd.mdh);
if (res <= 0) wait_for_completion();
....

and discard pd_unlinked.

Do we have a stack-trace from the crash so that I can confirm my hypothesis?
