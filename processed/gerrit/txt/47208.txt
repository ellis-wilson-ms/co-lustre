I think it is reasonable to run racer together fscrypt on a regular basis, but it can't be enforced for all patch review (i.e. not in sanity-sec), because racer itself occasionally fails even before fscrypt is involved.

It would make sense to modify racer.sh to configure fscrypt for a separate test_2() run (providing proper keys to at least _some_ of the processes), and then add a dir_encrypt.sh command that tries to create+enable on a new directory.  Whether it can do this before a second process tries to create files therein is a separate issue, but at least sometimes it would succeed, and it shouldn't crash/deadlock or have other issues beyond returning an error to userspace.
