LU-2456 lnet: DLC libyaml code

This patch adds support for the 3rd party library libyaml.
This library is licensed under the MIT License.
libyaml is used for YAML text parsing.  Another layer, cYAML,
will be built on top of it to allow the building of an
internal representation of a YAML block which can be stored
queried, etc.

--- Message (Robert Read, 2013-10-28 17:03:40.000000000) ---
Patch Set 5: Do not submit

There are several reasons this is a bad idea:

 - duplicates (and potentially conflicts with) the readily available libyaml package
 - unnecessary code bloat in lustre tree
 - increase testing load
 - different license 
 - sets a very bad precedent 

As we enhance the user space tools in Lustre it is likely (indeed, desirable) to increase our use of external libraries and tools.  We need to learn how to manage dependencies correctly, and pulling them into our source tree is not the right way. 

If, for some reason, we need to customize this library, then the correct solution would be to create a new git repository for it and track our changes relative to upstream there. Then we'd build and test packages for libyaml separately from Lustre.

I agree with both Robert and Chris.

I see several ways of doing this:

1. Provide a link to the libyaml library source code, and it would be the responsibility of the user to compile and install it on the Lustre machines.
2. Provide an RPM precompiled on the supported platform, and the user will need to install the RPM on the Lustre Machine.  the link can be on the lustre wiki.
3. Package the libyaml library within one of the lustre rpms, so that it gets installed as part of the lustre install.

I prefer 3, because the user needs not have yet another step for installation.  This dependency is then transparent to the user.

It becomes the responsibility of Lustre packaging to provide the proper compatible libyaml.  This way if in the future we upgrade to a different libyaml version (only one version exists now) then this upgrade becomes transparent to the user.  

thoughts?

I don't know why you are assuming that correct packaging implies "manual" installation of anything.  But I suppose it all depends on what "manual" means to you.  I don't consider yum particularly manual.  It is about as automated as these things get.

People _can_ choose to avoid the automated tools and do it some more manual method, but we're by no means forcing it on them.

On the build side, a good build system passes the lustre-*.src.rpm to mock to build the binary rpms.  The .src.rpm should has "BuildRequires: libyaml" (or whatever is correct).  Seeing that, mock automatically installs libyaml using yum.

Similarly, a person installing the binary rpms of lustre would do "yum install lustre".  Since at least one of those binary rpms has a "Requires: libyaml", yum automatically installs the required libyaml rpms as well.

That isn't "manual" in my book.

I'm all for retrieving a src.rpm file during the build, but am not sure the package manager is the right way to go for 3 reasons:

1- My understanding is many sites don't have direct access to the internet so how can the interaction with a package manager be integrated seamlessly into a build system?  If you have to carry files to a system on a USB key, I fail to see where the package manager buys you anything.

2- This goes against the open source freedom ideology, but I don't feel a product as complex, and critical, as Lustre should use a package manager to grab code which is not "exactly" the same as we tested with in-house.  I know you can specify version numbers to a package manager, but it seems better overall to just get a src.rpm with all the other build stuff that has been used in all pre-release testing.

3- Is there a universal package manager available for all the distro's using Lustre today (does everything support "yum")?  What about customers who might be rolling their own vanilla kernel like Fujitsu does?
