LU-11912 ofd: reduce LUSTRE_DATA_SEQ_MAX_WIDTH

Reduce LUSTRE_DATA_SEQ_MAX_WIDTH from ~4B to ~32M
to limit the number of objects under /O/[seq]/d[0..31]
dir on OSTs.
This makes the directories stay optimial for ldiskfs,
to avoid going into the largedir/3-level htree territory.

Remove the hard-coded LUSTRE_DATA_SEQ_MAX_WIDTH checks
in ofd, make them check the seq->lcs_width which is
a tunable set to LUSTRE_DATA_SEQ_MAX_WIDTH by default,
allow the value up to IDIF_MAX_OID if a larger seq width
is needed.

Use the odbo->o_size in the OST_CREATE rpc reply on ofd,
to update osp with the current seq width setting.
osp then uses this seq width to determine when to rollover
to a new seq.

The seq will rollover when the seq width is exhausted,
the default is LUSTRE_DATA_SEQ_MAX_WIDTH.
For seq >= FID_SEQ_NORMAL objects, the upper limit of
seq width is OBIF_MAX_OID,
For IDIF/MDT0 objects, the upper limit is IDIF_MAX_OID.
The seq FID_SEQ_OST_MDT0 will change to a normal seq after the
rollover.

Fix osp_precreate_reserve when the last precreated is the end
of the seq and the osp_objs_precreated can not host all
the requested objects, the mdt thread would stuck:
it wakes up osp precreate thread in a loop for progress,
but osp thread will not try to do anything until the seq
is used up. This can be seen easier when seq->lcs_width is
set to a low number and try to create an overstripe with stripe
number bigger than seq->lcs_width.

Fix the precreate thread spinning when the precreate pool
is at the end of the seq, and is nearly empty.

Change the seq->lcs_width to 16384 for all tests in
test-framework.sh, except a few slow tests to avoid timeouts,
and some overstriping tests creating LOV_MAX_STRIPE_COUNT to
avoid overstriping creating less objects than expected,
when precreate pool is at the end of the seq, and there are
not enough objects.

Fix the problem where seq could still change after
replay_barrier. To achieve this, introduce new fail_loc
OBD_FAIL_OSP_FORCE_NEW_SEQ and force_new_seq/force_new_seq_all
to drain the objects in the precreate pool then rollover to a
new seq. This applies to a bunch of test suites heavily using
replay_barrier.

> how do we tell the test system to set seq width to 4096 on all the OSTs?
 > I looked up Test-Parameters wiki and seems it doesn't do this sort of things?

I was thinking of doing this permanently for all tests by changing test-framework.sh, not just for this patch with Test-Parameters.  

Something like the following right at the end of mount_facet() whenever an MDT is mounted:

        [[ $facet =~ mds ]] &&
                do_nodes $(comma_list $(osts_nodes)) "$LCTL set_param -P seq.cli-cli-$FSNAME-OST*-osc-*.width=4096"

though there is a potential ordering issue since (AFAIK) these seq.cli-cli-*.width entries are created when the MDT connects to the OST.  That means if the OSTs are mounted after the MDTs they would not exist at MDT mount time, and similarly if OSTs are mounted before MDTs t then the MDS connections would not be set up correctly yet.  This will also be an issue if some customer wants to change this parameter, so it would be good to have a proper solution.

One opition is having an obdfilter.*.seq_width parameter that is copied when each MDT connects?  However, if it is changed after the MDS connects, it wouldn't be updated until the next connection.  Not great, but unlikely to be a significant problem for production use.   

That would change the test-framework.sh::mount_facet() code to:

        [[ $facet =~ ost ]] &&
                do_facet $facet "$LCTL set_param obdfilter.*.seq_width=4096

One other issue I realized when thinking about how the different IDIF_MAX_SEQ and OBIF_MAX_SEQ values are handled, is that using IDIF SEQ values with a smaller seq_width will NOT actually change MDT0000 object creation to a new O/* directory tree when f_seq is incremented.  That is because 2^16 IDIF f_seq values from MDT0000 *all* map to the same O/0/d* directory tree, so limiting the values to seq_width will only cause a gap (4B - seq_width) in the OID number, but continue to be created in the same O/0 directory tree until MDT0000 creates 2^16 * 32M = 2T objects (or 2^16 * 4096 = 256M objects during testing).

In order to get MDT0000 to properly create OST objects in different O/<seq>/d* directories, it should stop using the IDIF sequence right from the beginning, so I think it makes sense to implement LU-14692 first and base this patch on top of that one.

Yes the SEQ rollover is synchronous, no new FID allocation until it's committed. If we are looking at losing journal commit or write cache loss in controller we are in a deeper trouble already.

When the SEQ rollover to a new one, it's not always the immediate next value, the SEQ client got a range and when the range is used up, a new range is allocated for it. So I don't think we can relax the LASSERT to handle/ignore the small differences.

Now the risk of seeing old SEQ during replay, I think there should be some ordering in place,
when the SEQ rollover is committed, the transactions before it should be committed first so clients will free the RPCs using the old SEQ, as replay is not needed for them.
Plus the SEQ roller commit being synchronous is the other half, so clients won't have the replay RPC with new SEQ but after crash server only sees the old.

So sounds like the SEQ rollover commit should just be like the journal commit log with block barrier in place? I'm not sure if this ordering is already in place. Andreas do you know if we already have this or not? I will also investigate.

Yes during the test runs the SEQ rollover can happen a lot, mostly in sanity. and it's a bit random(which is good) when we have different number of OSTs.
