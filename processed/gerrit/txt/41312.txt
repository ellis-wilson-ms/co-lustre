--- Commit Message ---
LU-14359 hsm: support a flatter HSM archive format

Add versioning (v1 and v2) to the HSM archive format (directory
layout):
  v1: (oid & 0xffff)/-/-/-/-/-/FID
  v2: ((oid ^ seq) & 0xffff)/FID

v1 is the original layout and the default. v2 is the new layout which
should be selected for new installs.

Add an option --archive-format to select the archive format.

Add YAML configuration file support to lhsmtool_posix with properties
achive_format and archive_path. Add an option --config to set the
config file.

Adapt sanity-hsm and test-framework to allow testing of both archive
formats.

Patch Set 4: Code-Review-1

Sorry for the late change on this patch, I realized is that the "OID" values will not be very evenly distributed across the available space.  *Every* client creating a file will use dir .../0001, then .../0002, but only clients that create 65535 of 131071 files in a single mount will use .../ffff.  That would put a lot of FIDs (maybe millions) into the .../0001 bucket, but maybe none in .../ffff.  

It would be better to use something like (OID & 0xffff)^(SEQ & 0xffff) to get the most active bits in the FID being used at once.  The low bits of the SEQ should be relatively random since it increments with every client mount, but SEQ will not change quickly for files created by a single client.
