LU-19469 osc: add tunable for size threshold to cache hole

Add a tunable for the minimum page count threshold to determine
whether cache a hole extent on the rbtree of an OSC object on a
client.
	lctl set_param osc.*.hole_extent_kb=128

The above command will set it with 32 pages (128KiB in size).
The default value is 64KiB.

## Explanation of the Fix

**The Problem:**
The compile error occurred because the `LUSTRE_RW_ATTR(hole_extent_pages)` macro on line 519 creates a static variable named `lustre_attr_hole_extent_pages`, but this variable was never referenced anywhere in the code. The compiler, with `-Werror=unused-variable` enabled, treats unused variables as errors.

**The Root Cause:**
The patch added the `LUSTRE_RW_ATTR(hole_extent_pages)` macro to create the attribute structure, but forgot to register it in the `osc_attrs[]` array where all the OSC (Object Storage Client) sysfs attributes are listed.

**The Fix:**
I added the missing entry `&lustre_attr_hole_extent_pages.attr,` to the `osc_attrs[]` array on line 916. This ensures that:

1. The static variable `lustre_attr_hole_extent_pages` created by the macro is now properly used
2. The new `hole_extent_pages` tunable will be exposed in the sysfs interface for OSC devices
3. The compile error is resolved since the variable is no longer unused

**Rationale:**
This follows the established pattern in the Lustre codebase where:
- `LUSTRE_RW_ATTR(name)` creates the attribute structure with show/store functions
- The attribute must be registered in the `osc_attrs[]` array to be exposed to userspace
- The placement in the array (after `grant_shrink_interval` and before `max_dirty_mb`) maintains alphabetical ordering, which is consistent with the existing code organization

The fix is minimal and follows existing conventions without introducing any new functionality beyond what was intended by the original patch.
