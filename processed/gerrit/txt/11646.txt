LU-0000 ldlm: prolong timeout of waiting lock

this patch covered a couple of workarounds for testing only:

- when server sends back lock reply for ldlm_lock_enqueue, and if
  the lock already has queued blocking AST, we should make sure
  timeout of lock expire timer is long enough at least for client
  to resend lock enqueue, otherwise if reply is lost or congested
  on router for long enough, then client will be evicted.

  This workaround can't help for the multiple messages lost case.

- there is no guarantee that req::rq_timeout is lower than
  ldlm_bl_timeout, which means it is possible that lock wait timer
  is expired but RPC is not timedout yet.
  This patch will make sure ldlm_add_waiting_lock can wait longer
  than both ldlm_bl_timeout and req::rq_timeout
