LU-12514 utils: add "lustre_tgt" filesystem type

Add a "lustre_tgt" filesystem type separate from the "lustre"
filesystem type for forward compatibility to allow the client and
server mount code to be disentangled.  This does not actually make
any changes to the mount code itself, but allows testing interop
and fallback after an upgrade if the new filesystem type is used.

Remove the check for the "lustre_lite" filesystem type from
llapi_is_lustre_mnttype() since that was unused since Lustre 1.4,
and replace it with a check for filesystem type "lustre_tgt".

Mount with the "lustre_tgt" filesystem type in test-framework.sh
if the /sbin/mount.lustre_tgt binary is available on the servers.
For now it falls back to using /sbin/mount.lustre if unavailable.

James, I don't think we can force people to use "lustre_tgt" retroactively, but I'm on board to backport this patch to 2.12 if possible.  That gives us a wider range of versions that support the "lustre_tgt" filesystem type, and we can move over to using it to signal a target filesystem mount sooner (e.g. maybe in 2.15).

Doing the conversion automatically in mount.lustre based on the target type and the availability of "lustre_tgt" in /etc/filesystems (and print a deprecation warning if "lustre" is used) is doable.  In that case, which I hadn't thought of, we could probably do this in 2.14 since it won't immediately break systems when they upgrade/downgrade.

I think it is too late to actually clean up the mount code in the kernel for 2.13.  This patch is essentially just an alias for the existing filesystem type and doesn't really change functionality for existing systems.
