Alex, the incompat flags are per object. If there is some future incompatibility with that object, then the access to the object will be blocked. Examples of incompat flags include remote directories (which _would_ block access to the whole directory, because 2.3 and earlier clients do not understand DNE), HSM released objects (because old clients do not understand layout lock or operations to fetch such objects), and other similar things in the future. 

What is under question here is not the LMA incompat flag itself, but the cost of fetching the LMA. If the cost is low, then simple lookup is enough. If the cost is high, then we need to cache the xattrs. This would also be true of LOV EA, LMA for FID access, ACLs, etc. this would be a generic ZFS problem that needs a generic solution.

--- Message (Andreas Dilger, 2013-04-05 00:46:42.000000000) ---
Patch Set 5:

The goal of the lma_incompat flag is to flag objects (files or directories) with features that some older version of the code does not understand. This is in preference to marking the whole filesystem as incompatible or read-only if some limited number of inodes have an incompatible feature after an upgrade and downgrade.

Access to the object with the incompat flag should be blocked (returning an error to the client).  That means if some directory (e.g. DNE remote dir) is accessed by a 2.3 or earlier MDS it will return an error to the client, rather than trying to access it and probably doing the wrong thing.
