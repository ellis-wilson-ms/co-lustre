Patch Set 1:

Can we really depend on the ordering of the components in the layout to be constant?  What if other mirrors, EC, etc are added/removed on an existing file?  It would be more robust if the EC components positively identified the data components they referenced, either by storing the data component ID in the EC component, or this being implicit by the geometry of the EC component (e.g. number of D+P stripes match, and the stripe size is the same).

In one regard, the EC components are not "specific" to a particular data component, as long as the geometry matches, since the data should _generally_ be the same in all data components even if their layout is different.  However, if we want e.g. EC for old versions of the file (NOSYNC mirrors) then the EC component(s) must be associated with specific data component(s) (possibly by the component timestamps being the same).

Sorting the data first followed by all of the parity is just a matter
of consistency. For instance, if I need to look at all of the data components
and they're sorted, I can start at the beginning and loop until I hit a
parity component. If they're not sorted, then I have to loop through all
components and make sure I ignore parity components.

Either way works but it's not consistent in the code currently. Some places
assume it's sorted and others don't so we need to establish which is correct.
