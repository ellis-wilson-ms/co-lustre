LU-921 llite: warning in case of discarding dirty pages

when a client is evicted, dirty pages may get silently discarded,
the caller of successful write(2) will not know that the data he
wrote have been discarded due to eviction before it can be flushed
to the OSS.

--- Message (Andreas Dilger, 2011-12-30 08:28:12.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

A number of issues, but I can't post them inline from my phone:

- the ccc_dirty_page_warn field to track whether the error message is printed could be a flag or a bitfield
- the error message printed to the console should put the device name at the start of the message and print the FID instead of the inode number (like "%s: file FID "DFID" (%s) may lose data due to eviction" or similar).  The inode number may not be unique. 
- the test isn't checking whether an error message is being printed or not, which is the primary reason for this patch.
- did you verify that no error message is printed on the console in case of dirty page eviction due to lock revocation for truncate?  It used to be that the client would get a flag on the lock from the OST if the lock was being cancelled due to truncate so that it wasn't wasting time to write dirty pages that would immediately be discarded. I believe these are in sanity test 42, which was disabled because it was racy. 
- the test script needs to wrap at 80 columns.
