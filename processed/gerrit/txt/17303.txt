--- Message (Gregoire Pichon, 2015-12-02 14:23:24.000000000) ---
Patch Set 1: Code-Review-1

I don't understand the logic of the patch.

If the reply is saved in the ted_reply_last, it means the reply has been released because it will not be used anymore to reconstruct reply in case of resend. It is saved only to record on disk the highest transno of this export.

The reply is released if and only if the same tag has been reused (tgt_handle_tag()) by a request or the last_xid of an incoming request indicates the client did receive replies for lower xid (tgt_handle_received_xid()). In both cases, it is not expected that the client resends the request with that xid.

--- Message (Hongchao Zhang, 2015-12-03 10:07:36.000000000) ---
Patch Set 1:

the scenario is as follows,
1, during recovery, the client sent the request with its transno (say 141733920776) to MDT to replay
2, at MDT, the sent request was queued to "obd_req_replay_queue" to wait for replaying
3, after some while, the request was replayed and its reply was added to "ted_reply_list" and 
    "ted->ted_lcd->lcd_last_transno" was also set to the transno of the request (141733920776)
4, the client resent the request for the waiting in the "obd_req_replay_queue",  "tgt_handle_received_xid"
    was called in "tgt_request_handle"
5, in "tgt_handle_received_xid", the replay will be released for "xid" in the cached reply is equal to the "xid"
    of the resent request.
    
    int tgt_handle_received_xid(struct obd_export *exp, __u64 rcvd_xid)
    {
        ...
        mutex_lock(&ted->ted_lcd_lock);
        list_for_each_entry_safe(trd, tmp, &ted->ted_reply_list, trd_list) {
                if (trd->trd_reply.lrd_xid > rcvd_xid)     <--- here
                        continue;
                ted->ted_release_xid++;
                tgt_release_reply_data(lut, ted, trd);
        }               
        mutex_unlock(&ted->ted_lcd_lock);
        ...
    }

6, the reply is released in "tgt_release_reply_data" from "ted_reply_last" and put to "ted_reply_last" for
   its transno equals to "ted->ted_lcd->lcd_last_transno"

    static void tgt_release_reply_data(struct lu_target *lut,
                                   struct tg_export_data *ted,
                                   struct tg_reply_data *trd)
    {
        ...
        if (trd->trd_reply.lrd_transno == ted->ted_lcd->lcd_last_transno) {   <--- here
                /* free previous retained reply */
                if (ted->ted_reply_last != NULL)
                        tgt_free_reply_data(lut, ted, ted->ted_reply_last);
                /* retain the reply */
                list_del_init(&trd->trd_list);
                ted->ted_reply_last = trd;
        } else {
                tgt_free_reply_data(lut, ted, trd);
        }
        ...
    }

7, then the resent request can't find the reply and was processed by MDT as normal request, which caused it
    to fail with incompatible "Version" of the corresponding inode (-EOVERFLOW -75) just as shown in LU-6841
