LU-11850 obd: use netlink to get lustre stats

This adds the ability to collect performance metrics from lustre
in another way then from proc / debugfs files. The move to debugfs
has limited the scope of access to only root. Additionally there
is an expensive cost accessing many virtual file system files to
collect that data. Netlink will scale much better in this case as
well as offer a much more flexiable API.

The new ldebugfs_stats_alloc() replaces lprocfs_stats_alloc() and
registers the stats to be accessible BOTH throught debugfs AND
through Netlink. The new global "lstats_list" contains a list of
all registered sets of statistics, so it mirrors a subset of
debugfs. Netlink access can report on any statistics registered in
lstats_list.

Patch Set 10:

So the core work is done but one issue remains which I'm hoping to advice on how to handle the different types of stats. I wrote the code to be used by different types of stats. Well its not exactly the case for lustre_stats_start() which gathers which stats to collect. So I can see three ways to collect different stat types.

1) Register a callback to collect the stats depending on the source sent 
   from userland. The current code has something like that now. It gets   
   messy when '*' is requested which means gather everything.

2) Have a separate lustre_stats_start for each stats type. This means 
   having multiple netlink family (ldlm, llite, etc) and '*' would have
   to create multiple request from user land.

3) Create one big list to transverse which contains every stat type. 

Which would you perfer?

Patch Set 10:

The purpose of this patch is to restore lctl get_param for stats debugfs files for non root user. I'm just using the same approach as the IPv6 and external coordinator work in using a Netlink -> YAML approach.

So for external utilities that depends on how they are gathering the stats. If they are reading the debugfs files then they are out of luck.
If they use llapi_param_get_value() then its workable. I didn't modify llapi_param_get_value() in this version of the patch. Also the API is written as such that you can just use the Netlink API directly. So the approach is very flexible and the end user has choices.

--- Message (Neil Brown, 2021-10-08 04:58:10.000000000) ---
Patch Set 10:

I think I would prefer the "one big list" option - assuming I understand it correctly.  It would be multi-layer list just as the current debugfs approach has multiple directories.

Each module would register The lprocfs_stats structure much as (some of them) currently call debugfs_create_file(...ldebugfs_stats_seq_fops).
So the core code would have a list of these and can iterate through them to find the requested stats.

Registering, unregistering, and iterating would all take the same lock, so that there would be no risk of a structure being freed while the stats in it were being reported (as the current code allows - but I understand it is only demo code at this point).

First the easy question. The reason for lstats_list is due to stats being implanted into everyone type of data structure in Lustre. We are talking obd devices, ptlrpc service threads, ldlm pools, OSD devices, etc. Their is no uniformity so lstats_list was created for that purpose. The goal here is provide a simple simple solution for the developer to implement his stats. If the developers has to create his own Netlink handling each time for stats this approach will never be adopted. I been their before with Lustre. Its the same for the user land code. Its a one time written and done deal.

Now for the locking example which is real life. It would be very likely that various monitoring tools would be collecting the stats and
during runtime the admins for some reason take down a mount without stopping the monitoring tools. In that case the lprocfs_stats would be deleted (xa_erase) while collecting stats. The reverse is also true. A new file system could be mounted during runtime. So this is the case we need to handle.
