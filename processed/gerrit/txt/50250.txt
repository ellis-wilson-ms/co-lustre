LU-16623 lod: handle object allocation consistently

Consistently handle the various OS_STATFS_* flags that indicate
an OST or MDT is full or otherwise marked ineligible for use.

Fix lod_statfs_check() so it skips MDTs with OS_STATFS_ENOINO
for allocating dir stripes instead of only checking OST targets.

In the LOD code, ltd_active=0 indicates that the device is not
usable for new object allocations for a variety of reasons. That
includes out of space or inodes, read-only, max_create_count=0,
or disconnected export, not *only* that the OSP is disconnected
from the OST as with imp_deactive.  Targets marked ltd_active=0
will not be counted in ld_active_tgt_count, so these OSTs will
not count toward stripe_count for stripe_count=-1 files.

Set flags = LOD_USES_DEFAULT_STRIPE in lod_qos_prep_create() for
stripe_count = -1 layouts and pass it to lod_stripe_count_min()
to avoid use of *all* OSTs when free space is imbalanced or OSTs
are not available, and be happy with allocations on 3/4 of OSTs.
It looks like this functionality was missed when object allocations
transitioned from the LOV to LOD module.  Put the LOV_USES_* into
an enum and rename to LOD_USES_* for consistency with current code.

Apply the lod.*.max_stripe_count limits to PFL components as well
as plain file layouts in lod_comp_entry_stripe_count().

Rename ltd_connecting to ltd_discon, since there is no guarantee
that this target is actually *connecting*, only that it is currently
disconnected.  Use ltd_discon in places that checked ltd_active to
decide if the OSP was disconnected from the OST, which shouldn't be
skipped just because the OST is full or has creates disabled.

Patch Set 6:

I would argue that using stripe_count = 3 plus stripe_count = 6 for an 8-OST filesystem is broken, but in general it is better to accept what the user is asking for than to arbitrarily replace it.  Otherwise, someone will eventually be unhappy with this and we will need to add an "--i-really-mean-it" option to give them what they ask for, while preserving the old behavior. 

It is clear that asking for "-1" or "> ost_count" stripes isn't possible, and this should be handled on a "best effort" basis to allocate stripes. For the case of PFL files, it is _better_ to avoid re-using OSTs if possible, but if that isn't possible when the total number of stripes is larger than the OST count, then there should not be any effort to avoid already-used OSTs. 

This code for skipping already-used OSTs is most important for FLR files to avoid having the same OST on both sides of a mirror.  For PFL files (with OST balance) it is useful, but not required. For QOS PFL stripe allocations, we probably have to allow duplicate allocations. Otherwise, if we always use the most-free OSTs for the initial (small) components of a file, and then skip those most-free OSTs for the last (large) component, then it would be difficult for the large part of the file to use more space on OSTs with the most free space.

Patch Set 6:

> "It is clear that asking for "-1" or "> ost_count" stripes isn't
 > possible"
 > 
 > Andreas, can you correct or contextualize this?  I don't follow as
 > written, since in general it of course is.

What I mean is that it is possible for users to *ask* for more stripes than possible (-1 ~= infinite), but they can't actually *get* that many stripes, so (IMHO) this leaves it to the discretion of the MDS to allocate "the maximum practical number of stripes (given current operating conditions)".  This is in contrast to "user asks for N stripes, they should probably get N stripes" because they may have some specific requirement for this.

PFL files are in a bit of a middle ground, because we want the initial components (for small files) to be QOS balanced, in case there are lots of small files.  However, we don't want this to prevent the QOS balance of the later components (for large files).  So I think for QOS PFL allocation we should *not* skip OSTs used by previous components.  This isn't any worse than having a non-PFL file that is allocated on the same OST(s).

If there is RR alloc then it makes sense for later PFL components to avoid (if possible) the earlier stripes, since RR allocation should normally alloc an even number of first-component and last-component objects in the end.
