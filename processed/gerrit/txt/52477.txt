--- Commit Message ---
LU-17138 enc: prefer specific crypto engines

Some ciphers provided by external accelerators might register under
the generic cipher name. To avoid using them with Lustre, prefer the
AES-NI variant implemented directly in the CPU. And fallback to the
generic cipher if AES-NI is not available.

Introduce a new libcfs kernel module parameter named
'client_encryption_engine' to give the ability to choose the cipher.
By default its value is 'aes-ni', which makes Lustre look for the
AES-NI cipher first. This parameter can be set to 'system-default'
whic makes Lustre pick the generic cipher.

Here is test results in in-kernel QAT driver

Ubuntu22.04 (6.2.0-34-generic) in-kernel qat_4xxx driver

# fio -name=iotest-w -ioengine=sync -rw=write -blocksize=16m -iodepth=1 -direct=1 -fallocate=none -size=200g -numjobs=128 -group_reporting -directory=/lustre/encrypted -create_serialize=0 -fiename_format='f.$jobnum.$filenum' -scramble_buffers=0 -runtime=60

6.2.0-34-generic, master, no encryption				48.5GB/s
6.2.0-34-generic, master, encryption				 6.4GB/s
6.2.0-34-generic, no qat_4xxx, master, encryption		22.7GB/s
6.2.0-34-generic, master+LU-17138, encryption                   21.7GB/s
6.2.0-34-generic-noCryptStats, master, encryption		 6.4GB/s
6.2.0-34-generic-noCryptStats, master+LU-17138, encryption	48.2GB/s

Patch solved problem and the performance was back whether if qat_4xxx was loaded.
However, there is another performance problem in fscrypt. That is not related to the lustre, but linux kernel itself.
There are discussions at several places. e.g.
https://gitlab.com/redhat/centos-stream/src/kernel/centos-stream-9/-/merge_requests/2947

RHEL9 already disabled CONFIG_CRYPTO_STATS in latest security updated kernel.
 Kernel config option CONFIG_CRYPTO_STATS should be disabled until it is enhanced (BZ#2231850)
https://access.redhat.com/errata/RHSA-2023:5069

I've build custom 6.2.0-34-generic(6.2.0-34-generic-noCryptStats) with disabling CONFIG_CRYPTO_STATS in kernel config.
That changes significantly improved performance and indeed, it shows almost same of no encrypted performance for large IO size(16MB) at least.

I'm going to open a ticket to Canonical for it.
