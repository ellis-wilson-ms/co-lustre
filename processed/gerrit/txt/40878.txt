(1 comment)

I think the "real" answer is that the library version shouldn't change its major number if there is no incompatible change to the API/ABI.  That is something that I have been trying to avoid, so it isn't clear if there is a reason to change the major library version.  I think it would be reasonable to use 1:$LUSTRE_MAJOR:$LUSTRE_MINOR for this?

A better check would be to have some verification of the llapi functions (maybe part of checkpatch.pl?) to verify that the function signatures are not changing, and add a warning if there are new llapi functions added to lustreapi.h that are not being verified (which would be a good reason to bump the minor number).
