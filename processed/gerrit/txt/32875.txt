--- Message (Andreas Dilger, 2018-09-14 11:40:32.000000000) ---
Patch Set 6:

Patrick, this is how Lustre worked exclusively until 1.8, until the OSS read cache was implemented - a dedicated per-thread RDMA page list. This can reduce CPU overhead significantly. Since the read cache is disabled in this case, the RAM on the OSS will be much less used with per-thread page lists than the data cache. 

If you prefer to tune the system to have a huge number of threads, then you don't have to enable this feature.

Alex, my suggestion is to check nonrotational at startup, and use that to set the cache behavior. Setting a parameter afterward will override the default.

Patrick, one reason I'd be in favor of changing the default behavior for non-rotational storage is if this noticeably improves performance over the unpatched code.

While it is possible to say that there can be a tunable that is selected by some knowledgeable admin, I'd argue that there are already many tunables that the admin doesn't understand, and the end result is that only a very few sites (if any) would every change it from the default.

Of course, if you show me that this is a net loss for some flash devices (USB sticks notwithstanding) then I'd agree my heuristic is flawed, and try to look for a better one (e.g. only NVMe devices or similar, and not SATA SSDs).
