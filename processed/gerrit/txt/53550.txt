Patrik, did you look into lustre logs in ticket? if this is not enough, I might publish a full log (~2-4G).

Problem - we had lost UP2date bit during page reclaim/lock cancel and mkwrite send a page to write.
but parallel thread tries to read because page don't have a PG_uptodate.

We might restart an mkwrite only - but this needs in kernel loop to read page before write (or restore a up2date flag). We don't have sure parallel reading will exist, so it might be live lock.
restoring flag may cause a corruption because it might lost due lock cancel, not a local memory reclaim.

I have return -EAGAIN to restart a kernel code, but it have conflict with lustre code.

OK, all of those failures are timeouts after 95 minutes in a 10x looping run...  That's with 10000 ops, but also 10 runs, so I guess 100000 ops total.

Doing just 10000 ops 1x, ie, the non-slow version 1x, my SSD backed VM took ~400 seconds and Oleg's RAM backed janitor took 200-350 seconds.  So even Janitor would risk timeout at 10x runs, and Maloo is HDD and much slower.  A 6 minute test is very long for a non-slow test, and Maloo will be worse than that.

So I've reduced the 'slow' and non-slow versions, to 1000 ops for non-slow and 10000 ops for slow.  This means 10x runs should fit in the Maloo timeout for non-slow.
