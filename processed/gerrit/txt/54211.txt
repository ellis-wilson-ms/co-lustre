LU-17568 mdt: Append full path to RMDIR changelog record types

Changed mdt_reint_unlink to now retrieve and append full path to
RMDIR changelog record types, instead of providing only the
directory name.

This changed was intended to allow certain file system indexing
tools the capability to provide incremental updates through
leveraging the changelogs to monitor directories that need
reindexing.

--- Message (Andreas Dilger, 2024-02-29 16:51:55.000000000) ---
Patch Set 1: Code-Review-1

This seems like adding a lot of overhead for a questionable benefit. Generating the full pathname might take dozens of RPCs to other MDSes. At a very minimum, this overhead should only be enabled if such special indexing tools are in use. 

Do you have any measurement on the performance impact of this patch compared to without?  For example mdtest directory remove with a deeper tree with multiple MDTs and --max-inherit-rr=10 to spread the directories across MDTs?

 If the directory is removed, then it isn't clear what reindexing would be needed (ie. there is nothing to index anymore)?  The *parent* might be more interesting to reindex, but again if it is also deleted then there isn't anything to do except drop the index for that FID.

--- Message (Andreas Dilger, 2024-02-29 16:55:10.000000000) ---
Patch Set 1:

Just to be clear, I do appreciate that you are submitting a patch, and don't want you to feel that the contribution is unwelcome, but there isn't a lot of information about what you want to do, and this small change appears to have an outsized impact for the value that it will provide most sites. 

I still think that in the case of an RMDIR record for a tree that has been removed that you could fully resolve the full pathname in userspace by looking ahead in the changelog to find the RMDIR entries for the parent(s) of this directory.
