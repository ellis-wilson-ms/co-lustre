--- Commit Message ---
LU-14741 obdclass: Wake up entire queue of requests on close completion

Since close requests could be stuck behind normal requests and get
more slots we need to wake up entire accumulated queue waiting
for the next modrpc slot or have additional waitqueue just for
close requests.

This patch goes with the former approach.

--- Message (Neil Brown, 2021-06-18 23:29:06.000000000) ---
Patch Set 2:

This looks reasonable.

An alternate approach would be to change obd_get_mod_rpc_slot() to do a non-exclusive wait for close requests, and an exclusive wait for all others.  Then a simple "wake_up" will wake all the 'close' waiters and at most one other waiter.
I think that approach would make the intention clearer, but that's highly subjective.

I also think the wake_up() in obd_set_max_mod_rpcs_in_flight() should be a wake_up_all().  If the maximum has been substantially increased, it seems to make sense to wake everything up.  As it is, all the new slots will only be given to new requests, not requests that are alreay waiting.  This is only a mild fairness issue, not a correctness issue.

--- Message (Neil Brown, 2021-06-19 02:31:57.000000000) ---
Patch Set 2: Code-Review+1

There is certainly an inefficiency either way.

Either you wake up all waiters whenever any close completes, or you wake up all closer whenever any request completes.  I cannot guess which is more likely to be least inefficient.

It would be nice if it was easier to provide a function when waiting to test if a particular wakeup should be processed or ignored.  The core functionality is there, but you cannot do it using the wait_event macros.  It would allow us to wake up precisely the first task which could usefully respond to the wakeup.  I'm almost tempted to try it just to see how it turns out.  But we shouldn't delay this patch for that.
