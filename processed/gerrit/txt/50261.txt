LU-16633 obdclass: fix rpc slot leakage

obd_get_mod_rpc_slot() can race with obd_put_mod_rpc_slot():
finishing wait_woken() resets WQ_FLAG_WOKEN (which is set
when the corresponding thread gets a slot incrementing
cl_mod_rpcs_in_flight. then another thread execting
__wake_up_locked_key() may find that wq_entry again and call
claim_mod_rpc_function() one more time again incrementing
cl_mod_rpc_in_flight. thus it's incremented twice for a
single obd_get_mod_rpc_slot().

 #1: obd_get_mod_rpc_slot()	#2: obd_put_mod_rpc_slot()
flags &= ~WQ_FLAG_WOKEN
list_add()
wait_woken()
	schedule		claim_mod_rpc_function()
					cl_mod_rpcs_in_flight++
					wake_up()

	flags &= ~WQ_FLAG_WOKEN

				#3: obd_put_mod_rpc_slot()
				claim_mod_rpc_function()
					cl_mod_rpcs_in_flight++
					wake_up()
list_del()

the patch introduces a replacement for WQ_FLAG_WOKEN which is never
reset once set.

It would be nice if woken_wake_function() called autoremove_wake_function() instead of default_wake_function() - but it doesn't. I probably forgot this important fact when I developed the patch.

An alternate approach to fixing the problem would be to add another bool flag to 'struct mod_waiter' to record if the waiter has been given a slot, rather than depending on the WQ_FLAG_WOKEN flag meeting our need.

So claim_mod_rpc_function() would check this flag instead of WQ_FLAG_WOKEN, and would set it before calling waken_wake_function.  obd_get_mod_rpc_slot() would check this new flag before dropping the lock and calling wait_woken.

An atomic_t isn't needed - just a bool.  The value will only ever be changed or tested while holding the wait_queue_head lock - either in the waker or in the waiter - so there is no chance of a race.

I personally would prefer that approach.  Adding the list_del_init() isn't wrong, though for consistency with the latest kernels it should probably be list_del_init_careful(), which was only recently added. So that path feels a little awkward to me.
