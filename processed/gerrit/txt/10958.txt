LU-4971 ldlm: wait for the connection to be established

Consider the following race condition:

1) There are two MDTs in the system, when the client mount,
   it will trigger MDS_CONNECT request to each of the MDT.
   But the connection RPC will be handled asynchronously.
   The mount processing will go ahead when the connection
   between the client and MDT0 is established, but will
   not wait for the connection between the client and the
   MDT1 to be established.

2) After the client mount returned, the application tries
   to "stat $MOUNT/a/b", the directory "a" resides on the
   MDT0, the "b" resides on the MDT1. So the client will
   send lock_enqueue RPC to the MDT0 firstly.

3) The RPC handler on the MDT0 finds that the target object
   "b" resides on the remote MDT1, then it replies the client
   to try to talk with the MDT1.

4) The client gets the MDT0's reply, then forwards the request
   to the MDT1 via MDC1, but at that time, the connection between
   the client and the MDT1 has not been established yet.

5) For most of other RPCs, above case is not a problem, because
   the client-side RPC sender (in ptlrpc) will handle kinds of
   connection issues. But for the ldlm_enqueue request, it is
   somewhat different, because before giving the ldlm_enqueue RPC
   to the sender (or to the ptlrpc), the sponsor will check the
   export connect flags that will not be set until the connection
   between the client and the MDT has been established. So at such
   time point, the sponsor of the ldlm_enqueue request will get "0"
   flags from the export connection flags, then trigger the ASSERTION.

This patch will make the ldlm_enqueue request sponsor to wait until
the connection between the client and the server has been established.
