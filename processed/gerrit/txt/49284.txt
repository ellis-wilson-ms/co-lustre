LU-16356 hsm: store crh in rhashtable instead of list

Store coordinator restore handles in rhashtable instead of list.
Search in a list with above a million entries takes too much time
causing to wait a lot of tasks due to contention on cdt_restore_lock.
As cdt_restore_lock is not needed anymore to protect
cdt_restore_handle_list, this patch also solves the problem with
parallel restore requests(LU-15132).

Add regression test sanity-hsm 409b.

Fixes: 66b3e74bc ("LU-15132 hsm: Protect against parallel HSM restore requests")

--- Message (Neil Brown, 2023-05-23 02:01:10.000000000) ---
Patch Set 7: Code-Review-1

I think the refcounting in this patch is completely unnecessary, and confusing.

If I understand correctly, the only purpose of the rhashtable is to support cdt_restore_handle_exists().  i.e. we want to know if a handle "exists", but we never want access to that handle.  So we never increase the ref count.

All we do with entries in the table is:
1/ add a new one if nothing with the same name exists.
2/ check if a particular name exists
3/ delete an entry

None of this requires a ref count.  There are only two states: an entry is either in the table or it isn't.  This is determined by success of failure of the loop, look_insert or remove calls.
You can be certain that if rhashtable_remove_fast() succeeds, then you have a reference and can safely call rcu_free (or call_rcu).  If it fails, then you don't have a reference - someone else is freeing it.

So please remove all the refcounting.

--- Message (Nikitas Angelinas, 2023-06-01 20:22:00.000000000) ---
Patch Set 7:

(1 comment)

Thanks, Neil! I agree that the reference counting doesn't seem to be required during normal operation, fwiw. I think the only place it may be useful might be during cleanup, e.g. if the MDT device is shutting down via mdt_fini(), an HSM restore request could arrive, causing cdt_restore_handle_add() to be called. mdt_fini()->mdt_hsm_cdt_fini()->rhashtable_free_and_destroy()->crh_free_hash()->cdt_crh_put() could queue cdt_crh_free() as an RCU callback, which could race with the error-handling block for mdt_object_find_lock() in cdt_restore_handle_add() and free the memory for crh while cdt_restore_handle_add() is still trying to access the handle to also queue an RCU callback. If so, I'm not sure if using a refcount would be the best solution, but hopefully this has been fixed in the latest patch set by just using rhashtable_destroy() instead of rhashtable_free_and_destroy() in mdt_hsm_cdt_fini()
