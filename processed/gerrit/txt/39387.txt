LU-10606 hsm: store HSM xattr as a basic layout

A closer coupling between the HSM archive state and Lustre File
Level Redundancy (FLR) can make PCC and HSM management more
consistent with the management of other replicas in the file
system. Stroing HSM as a layout (FLR component of the file) can
obtain the following benefits:
- Consolidate the HSM with PFL/FLR/composite layouts to give a
  flexibility way to combine these features;
- Migration, FLR mirroring resync and HSM data movement all
  could be done through a user-space copytool;
- Allow multiple HSM copies (e.g. Tape, S3, another POSIX copy),
  which would also be useful for file versioning, flexible
  archiving;
- Allow PCC-RW mirror to be kept at the same as regular file
  mirror, by marking the regular file mirror(s) stale instead of
  releasing them;
- Possibly keep partial file components in the HSM, for limiting
  restore extents or for PFL layouts.

This patch implements the basic framework to store HSM xattr as
a basic layout component.

If there is a tool to do this, it could be done as part of LFSCK, so that it can be done when the filesystem is mounted.  Having a special-case offline tool is not very useful for users.

 > I'd prefer that it be done in-line, because we can't guarantee that
 > a customer has run the update tools on all their MDTs (and they
 > finished correctly and...).  Check for the new layout first, then
 > check for the old if it fails?  That way the cost is only paid once
 > per file, not every time.

In the distant past (i.e. at 1.0) we updated layouts from some obsolete format (layout saved in body of MDT inode) to the lov_magic_v1 xattr we still have today.  Essentially, it checked for the new format first (since all new files used the new format), but if not present it checked for the old format and then rewrote the layout in the new format.  At that point, the upgrade "tool" was to just use the filesystem for some time, or run "ls -lR" (or equivalent) for the whole filesystem.

That said, we *don't* want to force the new format xattr until there is compatibility code that would return the HSM component to userspace if the "trusted.hsm" xattr is read, for compatibility with old HSM copytools.
