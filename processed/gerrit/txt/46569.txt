Patch Set 2:

> How does not submitting pages as sync avoid this bug?  This bug is
 > saying there's no lock covering the page, so I don't see why
 > submitting them async (letting ptlrpcd handle them) would fix
 > things?

Well, if you submit page as sync, then you have to block waiting the io was finished(via cl_sync_io). If you commit page as async, then the extent will be link in tree(indicated by oe_intree), the page will be processed while the dlm lock is canceling. 
In readahead case, the readahead works in other thread, so waiting is useless(the dlm lock may be canceled before the io finished.

--- Message (Patrick Farrell, 2022-02-22 15:41:53.000000000) ---

So the reason this doesn't usually happen to normal (not async) readahead is normal readahead is done as part of a cl_io, so that cl_io has lock references.

However, readahead is expected to go beyond the I/O which started it.  So you could have a situation where readahead hits the next stripe and if there are no other references to that lock, that lock could be cancelled, just like for async readahead I think.

--- Message (Patrick Farrell, 2022-02-22 15:52:52.000000000) ---
Patch Set 2:

It seems like sync I/O should also take a reference on the dlmlocks it's using.  The mechanism async I/O uses is in osc_extent_find, it uses oi_write_osclock to find the dlmlock.

But oi_write_osclock is just set for writes - the code seems to assume there are no async submitted reads.

In fact, it kind of looks to me like we're not actually going to take a lock reference even with this change.  The extents are flushed as part of lock cancellation, but they do not have a reference on the lock.

Anyway, it looks to me like sync I/O in general should take a reference on each dlmlock used, in a manner similar to async I/O.  That will be tricky for readahead since there is no cl_lock.

Patch Set 2:

Ah, interesting - No.  Today, if readahead finds a lock, it takes a reference, but then almost immediately releases it.  (cl_io_read_ahead, cl_io_read_ahead_release, both called in ll_read_ahead_pages)

With that patch, instead of giving up, readahead will (non-blockingly) request a lock if none is present.  But it doesn't change what readahead *does* with the lock - it has that same bad reference behavior, where it references it and then releases it almost immediately.

The problem is readahead needs to take a reference which isn't released until the I/O is actually done.  Maybe that could be done with something associated with the pages...?  I *thought* they generated OSC extents which generated dlmlock references, but it turns out read extents don't generate dlmlock references, only write extents.
