--- Message (Alex Zhuravlev, 2016-01-20 17:39:41.000000000) ---
Patch Set 1:

hmm, why does lu_object_alloc() return EEXIST ?

--- Message (Fan Yong, 2016-01-21 01:08:51.000000000) ---
Patch Set 1:

According to the bug description in LU-6719, it is the osd_object_init() that returns the -EEXIST. But from the logic, the existence of the target should be normal case for object_init. It should NOT return -EEXIST. As you can see in lu_object_find_try(), we allow the low layer to establish two objects with the same FID temporary, but when add the object in the hash table, it will find that someone has already added one with the same FID, so the latter one will released at that time.

--- Message (Alex Zhuravlev, 2016-01-21 04:26:24.000000000) ---
Patch Set 1:

this is not correct, lu_object_find_try() should be able to create multiple instance of the object, then each thread lookups the object (holding a lock) and instantiate a single instance. look for the second htable_lookup() in lu_object_find_try().
so the question why osd_object_init() does return EEXIST - we create nothing new in this path, just lookup in-memory structure and reference that.

--- Message (Alex Zhuravlev, 2016-01-26 12:44:49.000000000) ---
Patch Set 1:

I think I've got a similar situation in my testing. it looks like dnode_hold_impl() returns EEXIST because dn->dn_free_txg!=0 which in turn means the object has been destroyed already.
so I guess the correct handling would be to handle this in osd_object_init() like osd_fid_lookup() returned -ENOENT.
