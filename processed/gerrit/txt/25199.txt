--- Commit Message ---
LU-8602 libcfs: call proper crypto algo when keys are passed in

In most cases keys are not passed to cfs_crypto_hash_alloc()
but if they are then crypto_ahash_setkey() will fail. Keys
are only handled by the hmac version of the algorithm requested.
If a key is passed into cfs_crypto_hash_alloc() then we should
request the hmac version of the algorithm when calling
crypto_alloc_ahash().

--- Message (Jeremy Filizetti, 2017-02-04 03:44:39.000000000) ---
Patch Set 1:

There is several LUG presentations that detailed the process but it is very obscure.

The client calls out to the lgss_keyring when finding it doesn't have a security context yet.  lgss_keyring constructs the RPC to send to the remote side and this happens via write to the init_channel file descriptor in do_nego_rpc().  When the write returns it has an updated buffer (yes this is horrible but that's the way it is).  On the kernel side where do_nego_rpc() pops out is gss_proc_write_secinit() which calls gss_do_ctx_init_rpc() to send the RPC to the remote end.  Once this returns from the remote end the return from the write syscall in do_nego_rpc now has the updated structure in param.

On the server side the data comes in (approximately) via accept() -> gss_svc_accept() -> gss_svc_accept_kr() -> gss_svc_handle_init() gss_svc_upcall_handle_init().  From here the sunrpc cache implementation ends up with the upcall using rsi_upcall() to send the data over a pipefs to the lsvcgssd process which calls handle_channel_request() -> handle_sk().  Once handle_sk() returns with the updated rawobj_t (kernel)/gss_buffer_desc (user) in out_tok handle_channel_request() writes the data back to the pipefs for the kernel using send_response() which then allows the kernel to send the RPC response pretty much through the same stack.

This of course is summarizing a lot but it should be enough to hopefully help you understand how it flows through the code base.

I haven't fully reviewed the code yet from the 4.6 changes patch but one thing that strikes me is sc_hmac was changed to sc_pad but never used why was that?  Also the new sc_hmac is an enum which is passed to the kernel.  I don't know what size an enum is from the compiler but it seems it could vary by compiler and the sizeof(sc_hmac) is used between user and kernel space.

I'll look over things this weekend more to see if I can find what's wrong.

I believe the issue (though have not verified) is because the hmac(sha*) was removed and replaced with sha*.  When gss_digest_hmac() calls crypto_ahash_setkey() it is probably failing because there is no setkey() function for not HMAC types.  See include/crypto/hash.h details about setkey():

" *          object. Not  all hashing algorithms do implement this function as it
 *          is only needed for keyed message digests. SHAx/MDx/CRCx do NOT
 *          implement this function. HMAC(MDx)/HMAC(SHAx)/CMAC(AES) do implement
 *          this function. This function must be called before any other of the:"

The kernel crypto API is sensitive to what is passed to it and the mode of operation that will be used.  As for the block size used for AES that is determined by the key provided to crypto_blkcipher_setkey().
