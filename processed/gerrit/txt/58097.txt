To be precise, I'm talking about preventing the use of the folio compat APIs [1]. Matt Wilcox called out those APIs in particular.

If we know for certain that some APIs will be replaced/removed, it's seems ideal to discourage usage as soon as possible. Eventually, the kernel itself will remove these folio compat APIs. So if we can get Lustre strictly deprecate them beforehand, it will save us a lot of effort down the line. I think a compile time check will be followed more strictly than a checkpatch warning. People writing new code with old APIs could cause a lot of wasted effort that we could avoid.

I agree that we should start with checkpatch. But I guess I don't see any downside to a stricter check?

[1] https://elixir.bootlin.com/linux/v6.14-rc3/source/mm/folio-compat.c
