LU-13566 socklnd: fix local interface binding

When a node is configured with multiple interfaces in
Multi-Rail config, socklnd was not utilizing the local interface
requested by LNet. In essence LNet was using all the NIDs in round
robin, however the socklnd module was not binding to the correct
interface. Traffic was thus sent on a subset of the interfaces.

The reason is that the route interface number was not being set.
In most cases lnet_connect() is called to create a socket. The
socket is bound to the interface provided and then
ksocknal_create_conn() is called to create the socklnd connection.
ksocknal_create_conn() calls ksocknal_associate_route_conn_locked()
at which point the route's local interface is assigned. However,
this is already too late as the socket has already been created
and bound to a local interface.

Therefore, it's important to assign the route's interface before
calling lnet_connect() to ensure socket is bound to correct local
interface.

To address this issue, the route's interface index is initialized
to the NI's interface index when it's added to the peer_ni.

Another bug fixed:
The interface index was not being initialized in the startup
routine.

Note: We're strictly assuming that there is one interface for each
NI. This is because tcp bonding will be removed from the socklnd as
it has been deprecated by LNet mutli-rail.

Patch Set 3: Code-Review-1

There is a lot that I don't understand here.

It looks to me like your patch passes the IP address of a peer - a different host to this one - to  ksocknal_ip2index().  That will always fail.  ksocknal_ip2index() expects an IP address for *this* host.

Also, why is the call to ksocknal_associate_route_conn_locked() too late -- or maybe the question is - why bothering to call it if it is too later.  If ksocknal_associate_route_conn_locked() changes the local interface (previously changed the local IP address) without causing all the TCP connections to be re-established, what is the point?

I guess I'm saying that I think there is a deeper problem here.  I don't know all the motivation behind the code so I cannot give a confident proposal about how to move forward, but things look wrong.

Maybe ksocknal_add_peer() should set route->ksnr_myiface based on the content of ni->ni_data.  That would mean ->ksnr_myiface was never -1.
Does that seem reasonable to you?

(and maybe ksocknal_associate_route_conn_locked() should be discarded?)

--- Message (Amir Shehata, 2020-06-15 15:57:51.000000000) ---
Patch Set 3:

I'm not sure I follow the concern:
ksnr_peer->ksnp_ni->ni_nid
Is the local NI NID. So it'll be the IP address in the local NI NID associated with that peer. The way a peer is looked up is by verifying that both the peer NID and the local NI match (ksocknal_find_peer_locked()) 
So the patch should work.
I think the routes concept stemmed from the fact that tcp bonding was implemented in the socklnd. So you had to maintain multiple routes over the different interfaces which were part of the bond. However, I don't think socklnd bonding is used. We kept it around in the early days of MR in case it was being used. However, I think it's time to remove it. I have an open ticket to remove tcp bonding and simplify the code around this area.
For now I think assigning the IP right before the connect makes sense. I believe it makes the intent of the code quiet clear and because removing the associate route code should be done separately as part of removing the bonding feature completely.

I think I understand why ksnr_peer->ksnp_ni->ni_nid gives a valid address now.

It's not that I think the patch will do the wrong thing, just that it doesn't seem like a good way to do it.  In particular mapping from an IP address to an interface is pointless when we already know what the interface is.
->ksnp_ni->ni_data->ksnn_interfaces[0].knsi_index
should give the appropriate interface index, and we access that in ksocknal_add_peer().  That way ksnr_myiface is always initialized properly, and we don't need to add a check whenever we create a socket.
