--- Commit Message ---
LU-5092 nodemap: users of ted_nodemap should take ref

In order to quickly map clients, the client's assigned nodemap is
saved on the export data struct. When mapping occurs, a reference to
that nodemap is used for mapping. If no reference is taken during
mapping and the nodemap is put elsewhere, then the nodemap could be
destroyed before the mapping is finished. This patch ensures that a
reference is taken before mapping.

(7 comments)

Thanks for the review. Here are my thoughts on the nodemap_ted_lock's purpose and relationship to other locks:

In order to ensure that the nodemap isn't destroyed between the NULL check and the getref call, there needs to be locking. nm_member_list_lock makes for a natural lock since any changes to ted_nodemap must be accompanied by a corresponding change to the nodemap's member list. However, functions relying on ted_nodemap can't use this lock because ted_nodemap can be NULL, and so there's the catch-22 of requiring a NULL check before acquiring the lock to do a NULL check. It's also too wide-grained a lock for mapping -- multiple clients in the same nodemap must be able to map concurrently. active_config_lock is also an option because member operations occur on the active config, however it is also too wide-grained for mapping use.

So ted_nodemap_lock's purpose is to give fine-grained protection to functions that can't use nm_member_list_lock while they acquire a reference on ted_nodemap. This leads to three basic rules:
- references to ted_nodemap can only do the NULL check/get under ted_nodemap_lock, nm_member_list_lock, or active_config_lock
- ted_nodemap can only be modified under active_config_lock, nm_member_list_lock, and ted_nodemap_lock
- the reference held by ted_nodemap can only be put after ted_nodemap is modified

I added comment to lustre_export.h explaining this, let me know if it needs tweaking.
