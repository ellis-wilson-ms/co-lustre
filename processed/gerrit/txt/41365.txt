Any thoughts on how to fix this bug?  Should we limit the size of an N-striped file to be 4KB short of the end of the first stripe in the last component.  That would mean all of the later stripes will hold no data, to avoid the 4KB hole at the end of the first stripe?  For a 1-stripe file the limit would be the same.

It seems that the problem is not only about last component.
We should be safe for case like (even though it is not a widely used case):

 lfs setstripe -E 16T -c 1 -E -1 -c 4

Otherwise, limiting file size should be ok.

However, how should we deal with files longer than new file limit which already exist?

That looks like a bug.  It is probably taking the max size from the first component, then multiplying by the stripe count, even though the second component only has 1 stripe.

I don't think we can get this 100% correct, since we don't know which OSTs the last component will be created on, but at least it could use the stripe count of the last component instead of the first one.  That will help for both when the last component has more stripes and in the unlikely case when it has fewer stripes.
