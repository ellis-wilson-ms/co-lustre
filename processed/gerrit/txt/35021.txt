LU-11549 mdt: take UPDATE lock for rename target

mdt_reint_rename() and mdt_link() take not conflicting md locks for
rename target (being destroyed) and linked object, respectively rename
takes LOOKUP|XATTR, link takes UPDATE|XATTR, having XATTR bit cleared
when one lock is taken remotely from another MDT, those lock bits do
not intersect and allow a race between rename and link, adding an
extra reference to an object scheduled to destroy, resulting in an fs
corruption of unattached inode on one MDT and a hanging cross-mdt
reference on another MDT.

The full picture of the race including
a) opening a file
b) renaming another file into the first one and deleting it,
   leaving an ophan object
c) another thread from another MDT adds a name to the file, passing
   all link checks due to insufficient locking between
   mdt_reint_link() and mdt_reint_rename()
d) file close deletes the orphan object and removes all local
   references to the object and deletes it from OI.

The fix adds UPDATE bit to the lock request from mdt_reint_rename().
