--- Commit Message ---
LU-18155 obd: remove duplicate SLAB and PAGE poisoning

The Linux kernel already has CONFIG_DEBUG_SLAB and
CONFIG_PAGE_POISONING. Developers and testers can opt into
these kernel features rather than relying on Lustre's custom
macros.

The single caller of POISON_PAGE() has been removed.

OBD_FREE_*() no longer POISON() upon the release of memory.
We rely only on CONFIG_DEBUG_SLAB now.

POISON_PTR() is updated to take advantage of
POISON_POINTER_DELTA. Per 'include/linux/poison.h'
architectures might want to move the poison pointer offset
into some well-recognized area such as 0xdead000000000000,
that is also not mappable by user-space exploits. This
mirrors the Linux kernel list poisoning.

Running a simple performance test:

fio --name=random-read \
    --ioengine=libaio \
    --iodepth=32 \
    --rw=randread \
    --bs=4k \
    --direct=1 \
    --size=1G \
    --numjobs=4 \
    --runtime=60 \
    --group_reporting

I see:

89011.68 IOPS (with poisoning)
vs.
91240.89 IOPS (without poisoning)

which represents a modest 2.5% improvement.

I don't think the poison alone adds much. If there's a use-after-free that doesn't induce a crash, the poisoning doesn't help. If there is a crash, I think the debugging would be straightforward regardless. I think something heavy-duty like DEBUG_SLAB, KASAN, UBSAN are require to find the more subtle (non-crashing) bugs.

I'm also wary of the performance hit of always-on poisoning. But I haven't taken the time to measure anything.
