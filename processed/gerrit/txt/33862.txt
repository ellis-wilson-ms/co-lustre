--- Message (James Simmons, 2019-02-12 15:01:27.000000000) ---
Patch Set 2:

I don't think spin_locks can be used as memory barriers. In fact their exist smp_mb__before_spinlock() macros. Even in that case the smp_mb() barriers only work on total order platforms like x86. For ARM / Power8 the choice for memory barriers is smp_store_*(). Also spinlocks are way more expensive then the smp_store_*() functions.

--- Message (Patrick Farrell, 2019-02-12 15:25:35.000000000) ---
Patch Set 2:

They absolutely can, in fact, they have to be.  If memory operations could be "moved" 'past' a spinlock, before or after, how could it protect anything?

All of the standard locking constructs include heavy-weight memory barriers, because they wouldn't work if not:
https://www.kernel.org/doc/Documentation/memory-barriers.txt
Look for "implicit memory barriers" (or ask Neil if you don't believe me ;) )

Memory barriers as separate constructs are available for when you can get away with a lighter-weight guarantee than those provided by actual locking.  (FWIW, I think you need a *really* good reason to use memory barriers directly.  Locking is far easier to reason about and does not require understanding memory ordering.)

--- Message (Patrick Farrell, 2019-02-12 15:28:03.000000000) ---
Patch Set 2:

OK, so as noted in the doc, locks do not use the heaviest possible memory barrier - but they're pretty close.

--- Message (Patrick Farrell, 2019-02-12 15:29:42.000000000) ---
Patch Set 2:

One more clarification...

It is occasionally necessary to have the heaviest type of memory barrier.  That's what those macros are for.  Spinlocks *do* contain memory barriers.  (For us mere mortals, it's generally safe to think of them as full barriers.)
