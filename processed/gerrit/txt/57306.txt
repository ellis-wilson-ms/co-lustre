LU-17814 utils: Enable pfind by default

Add error handling and enable pfind by default.

This requires switching find_skip_file from counting to
probability based, otherwise we'd need an expensive atomic
op to coordinate the exact skip count.  But the point of the
option is statistical rather than exact behavior.

Increase the number of files in the relevant test to get
the law of large numbers in our favor.  Some complex math
suggests the probability of failure is extremely small.

Patch Set 19: Code-Review-1

(5 comments)

Needs to add "--threads=N" to allow selecting the thread count, especially if we are undecided about an ideal value.  4 is not terrible as a v1 estimate, or maybe 8.  Consideration should be given that "lfs find" is probably *not* CPU bound, so it might not be only about the number of cores, but also the number of MDTs and max_rpcs_in_flight.

If we make the (mostly) reasonable assumption that anyone running "lfs find" wants to get results quickly, then something like "threads=MDT_COUNT * mx_rpcs_in_flight / 2" seems not unreasonable, possibly regardless of the core count (though this would be a few hundreds on a large system today).

In most cases the threads will be blocked waiting on RPC completion and not spinning on CPU (unless there is some nasty pthread contention internally) so it is probably OK to be semi-aggressive here. A single client with 8 RPCs in flight is not going to overload the MDS/MDTs, and probably not the client either.

Please update the lfs-find.1 man page for the new --threads=N option, and describe whatever is decided for the default thread count.

Note that I would be grudgingly OK to punt the "optimal" thread selection to a later patch and start with 4 or 8 as the default, but this can't go out the door with 4 hard coded and no option to change it.
