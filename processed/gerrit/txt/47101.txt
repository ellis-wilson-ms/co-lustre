LU-15759 libcfs: debugfs file_operation should have an owner

If debugfs a file is open when unloading the libcfs/lnet module, it
produces a kernel Oops (debugfs file_operations callbacks no longer
exist).

Crash generated with routerstat (/sys/kernel/debug/lnet/stats):
[ 1449.750396] IP: [<ffffffffab24e093>] SyS_lseek+0x83/0x100
[ 1449.750412] PGD 9fa14067 PUD 9fa16067 PMD d4e5d067 PTE 0
[ 1449.750428] Oops: 0000 [#1] SMP
[ 1449.750883]  [<ffffffffab7aaf92>] system_call_fastpath+0x25/0x2a
[ 1449.750897]  [<ffffffffab7aaed5>] ?
system_call_after_swapgs+0xa2/0x13a

This patch add an owner to debugfs file_operation for libcfs and
lnet_router entries (/sys/kernel/debug/lnet/*).

The following behavior is expected:
$ modprobe lustre
$ routerstat 10 > /dev/null &
$ lustre_rmmod
rmmod: ERROR: Module lnet is in use
Can't read statfile (ENODEV)
[1]+  Exit 1                  routerstat 10 > /dev/null
$ lustre_rmmod

We can't use DEFINE_DEBUGFS_ATTRIBUTE for complex output like for 'peers', 'routes', 'stats', 'nis' or 'routers'.
DEFINE_DEBUGFS_ATTRIBUTE enable to display only 1 value.
But if I follow, you would like to have 1 fops per attribute/file like elsewhere in the existing code.

So what about use the existing LDEBUGFS_SEQ_FOPS_RO and LDEBUGFS_SEQ_FOPS interfaces?

The helper functions are defined inside obdclass, so we have to move those functions in libcfs to avoid loop dependencies. And we have to rewrite all callbacks using lnet_insert_debugfs() interface.

--- Message (Neil Brown, 2022-05-05 00:17:58.000000000) ---
Patch Set 3: Code-Review-1

Why not just add 3 lines:
  .owner = THIS_MODULE,
in each of the existing file_operations structs?

This will pin libcfs while any debugfs file is open, so anything depending on libcfs will also be pinned.

As the code is in libcfs, that module must be pinned anyway.

With the patch as it is, when an debugfs file from lnet/router is open, that won't pin libcfs module, and that would be bad.

Patch Set 3:

> But the issue is that file_operation should be "const". 

So cast it to a const pointer.

> And I couldn't find an example in the kernel where the struct is defined at runtime.

lustre uses lots of little discrete modules.  Mainline linux subsystems tend to use one, or maybe two. e.g. ext4 and jbd2.  Or nfs and sunrpc (though there are nfs2 nfs3 and nfs4 now, or something like that).
You problem is largely caused by the proliferation of modules.

Longer term, it would make sense to get rid of the "struct ctl_table" and define a set of debugs attributes - which is essentially what James suggested I think.  That would be more work that seems justified to fix the problem you found, but it is worth doing eventually.
