LU-16495 libcfs: expand debug mask

Introduce debug mask expansion option with additional 32 bits to use.
This allows for more custom debug message types per subsystem.
To take advantage of the expanded mask, subsystem needs to define
its own interpretation of the expanded mask bits and manage it
via corresponding lctl option. For example:

        lctl set_param debug_lnet=+msgtrk

	CDEBUG(D_NET | (D_LNET_MSGTRK >> 32), "message");

The default behavior remains unchanged: the expanded mask is optional.

Serguei,

I do have a general question for you and Andreas:
This seems *much* more complicated than just increasing the debug mask size, is it worth the extra benefit?  It gives more flexibility, but do we expect people to add lots of new kinds of debug messages?  So many that each subsystem needs its own special subset of debug with its own command?

I worry about fragmenting debug options and making it harder to, for example, turn on all debug.  I'm not an LNet developer, but I sometimes need lnet debug, and I'd prefer it were all turned on and off the same way if practical.  (I'm open to being persuaded here, but I don't see it right now.)

Next, I think this should have *some* kind of sanity test making sure that at least some of the new debug settings work.  I recognize it would be a pain to fully test if they work, but at least something that shows we can set debug= as appropriate, and probably a quick dump and scan of a debug log to make sure something is actually printed out.

--- Message (Serguei Smirnov, 2023-04-26 20:16:49.000000000) ---
Patch Set 4:

Patrick,

The use of expanded mask is optional, so if a subsystem doesn't need it, it is not required that the expanded mask is defined for it.

The idea behind adding this for LNet specifically is focused debugging, such that "expanded" debug options are turned on only if necessary when looking at a particular part of LNet: refcount tracking, message queues, etc. Having too much debugging controlled by a single switch can create problems dealing with the output in busy systems, potentially affecting the ability to reproduce the issue for which debugging is turned on. So having to turn on all of LNet debugging, including "expanded" options, should not be common. 

I do agree with the point of needing to add tests for this.
