This patch adds WILLREAD advice to ladvise framework. OSS will
prefetch data into memory when this hint is provided. It is not
garanteed how long the cached pages will be kept in memory.

--- Message (Andreas Dilger, 2015-01-08 22:56:37.000000000) ---
Patch Set 6: Code-Review-1

(8 comments)

The patch seems pretty reasonable, but needs a few small changes.

Firstly, it isn't clear that there is a benefit to split this patch from the previous one, but it is OK if you want to keep them separate.

Secondly, it seems that osd-ldiskfs isn't doing a very good job for prefetching file data if it is not explicitly caching data for all files.  I'd think that if there is a request like this to prefetch data it should be cached for that file until read, even if read cache is off by default.  I wonder if the readcache_max_filesize handling should only evict pages from cache _after_ the data has been read, instead of during the read path?

Thirdly, it seems that osd-zfs could be improved to use dmu_prefetch() for this, instead of actually reading the data immediately into pages and then discarding it.  It might also make more sense to use ZIO_PRIORITY_ASYNC_READ for the prefetch instead of ZIO_PRIORITY_SYNC_READ.

Finally, this should have a test program which is calling the API with WILLNEED so that the code continues to work.  I'm not sure if there is any easy way to _verify_ that the code is working, but at least we know the API is functional.
