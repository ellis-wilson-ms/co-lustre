--- Message (Andreas Dilger, 2016-03-07 10:35:58.000000000) ---
Patch Set 1:

(1 comment)

Sebastien, this patch is really interesting, thanks for pushing it. I think there are a number of interesting things that can be done with nodemap a now that we have them. 

I think after this patch, I think the main gap for having strong containers is that the shared-key crypto is itself not per-nodeset, unless it was modified since the last time I looked at it.

If the shared-keys were per nodemap, then it would be possible to give different keys to different groups of nodes, and when they connect their key must match the NID for the nodemap, or possibly the nodemap can have a "wildcard NID" that allows any client with a valid key to connect.

This would allow strong client identification to the nodeset, not just generic authorization to mount the filesystem that still trusts the client NID.

--- Message (Kit Westneat, 2016-03-09 15:44:28.000000000) ---
Patch Set 2:

There's no space in the nodemap_rec at the moment for the fileset information, so it will need to be expanded. It's currently at 32B and would need to grow to at least 288B. It seems like if we are changing it, it makes sense to add space for future features, so perhaps growing it to 384 or 512 makes sense? The drawback is that the config files will be an order of magnitude larger, but on the other hand, even with 10,000s of users, it would be relatively small. On the plus side, it should make the large nodemap config test run faster :). I haven't looked but is there a global maximum to an index file's record size?

In order to add fileset, lustre_idl.h would need to be changed, as well as nodemap_storage.c and nodemap_internal.h. The major functions to change are nodemap_process_keyrec and nodemap_cluster_rec_init.

--- Message (Sebastien Buisson, 2016-03-09 16:07:06.000000000) ---
Patch Set 2:

> It seems like if we are
 > changing it, it makes sense to add space for future features, so
 > perhaps growing it to 384 or 512 makes sense?

I think I was wrong using NAME_MAX+1 as fileset array size in 'struct lu_nodemap'.
Indeed, a fileset, as specified by a client at mount, can be any valid directory path in the Lustre filesystem being mounted.
So a fileset is a string that can be PATH_MAX long. It will really make record size huge :(
