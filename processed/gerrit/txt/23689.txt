LU-8806 ofd: dt_lock object after starting transaction

For ldiskfs backend, the punch transaction may be too large to
be held in one transaction, so the transaction maybe restarted
inside ldiskfs. If we geneately lock the target to be truncated
after starting the transaction, then restart the transaction may
cause deadlock with other modifications against the same object.

To avoid such deadlock, and also make OST side transaction/dt_lock
order to be the same as MDT side, we introduce some new mechanism:

ofd_object::ofo_punch_rwsem
It is OFD internal read/write semaphore, only OFD read/write/punch
handler will touch ofd_object::ofo_punch_rwsem: the punch handler
will take write mode semaphore before the punch transaction start;
IO handler will take read mode semaphore during prepare phase and
hold it until commit phase done, that prevents the punch handler
to truncate the target during the IO process.

osd_object::oo_trans_sem
It is OSD internal read/write semaphore for OST side modification:
if some OST side modification can be triggered with concurrent
punch operation, then it needs to take read mode semaphore during
declare phase. The punch handler will take write mode semaphore
during punch decalre. Then even though the punch transaction is
restarted inside ldiskfs, no others (modification) will wait for
the dt_lock on the object being truncated but with the transaction
started already.

With these changes, both MDT side and OST side can follow the
same rule: dt_lock object after starting transaction.
