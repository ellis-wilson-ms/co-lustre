LU-15246 ptlrpc: per-device adaptive timeout parameters

When a client is mounting multiple filesystems with different
MGSes setting global parameters at_min, at_max, etc., then the
settings from one filesystem's MGS config will also apply to RPCs
sent for the OSC, MDC, and MGC devices on the other filesystem(s).
Typically the settings of the last filesystem to mount on the client
override the earlier values, and there is no way to separate them.

Moving the parameters to be per-device values allows them to be
set independently for each set of client devices, so that the
client can interact properly with each set of servers.  This allows
e.g. different timeouts for local and remote mounts, or for flash
and HDD filesystems that have different load and performance.

Add per-device adaptive timeout parameters that can optionally
replace global parameters of the same name:

    at_min     -> *.<fsname>*.at_min
    at_max     -> *.<fsname>*.at_max
    at_history -> *.<fsname>*.at_history
    ldlm_enqeue_min -> *.<fsname>*.ldlm_enqueue_min

These parameters should always be set with fsname in the device
name, rather than pure wildcard '*' device names, or it will be
be same as the global parameters in the end (settings from one
MGS will apply to devices on other filesystems).  That is a bug
in how "lctl set_param -P" works, but will be fixed separately.

Patch Set 4:

(4 comments)

> handling two existing filesystems that are setting
 > at_min/at_max, each for their own filesystem.

I think this patch is handling only half of the problem.  It adds the per-export (MDC/OSC) at_min/at_max internal variables and tunables, which is useful, but it will not fix the original problem of a client mounting two different filesystems with different "global" at_* tunables.  It will only help in the future if only the per-export "at_*" parameters are set by the filesystem config.

This also has a secondary issue that the "at_*" parameters are supposed to be the same between the clients and servers, so setting "mdc.*.at_max" on the clients would not affect the "at_max" value that is used by the servers.  The current patch passes the per-export values back to the global value when they are zero (the default), but I think that will still have the original problem reported in .

A better solution would be to save the at_* parameters by the "config instance" number, and have unset OSC/MDC parameters use that, since the OSC/MDC device configuration can determine the cfg_instance number directly.  Then, class_process_config() would pass cfg_instance as an argument, and would need to detect when at_min/at_max parameters are in the configuration log (class_match_param(..., PARAM_SYS) and check parameter name and save them in the per-cfg_instance structure (array or rhashtable?).  That would work properly because the cfg_instance value is unique per filesystem.

In fact, some of the other parameters in lustre_attrs[] would probably also benefit from being per-filesystem (e.g. "timeout", "pinger").

Since it isn't possible to mount targets *on the server* from different MGSes, it isn't possible for these global parameters to become inconsistent on the server, only the client.

I'm wondering if this patch would be less complex if it only addressed the client-side parameters, and stored the per-target parameters in obd_import (which is only used on the client) instead of obd_device (which is also used by the server targets)?

It looks like that was the case in v4 of this patch, but it changed in v5 of the patch.  I don't recall why that change was made, but it definitely increased the scope of this patch significantly.

It may be that my comments on v4 were not very clear.  What I was looking for is that "mdc.*.at_max", "mdc.*.timeout", etc. would be updated from the global "at_max" and "timeout" parameters when handling the config log for that filesystem/MGC, but *not* updated when handling the config log for a different MGC (different set servers).

That might be difficult to test with our test framework, because the MGS is usually shared even if there are two separate filesystems, and that makes it impossible to have two different "at_max" parameters.  The conf-sanity test would need multiple servers and a separate MGS for each one.  That can be done with autotest (usually two separate MDS nodes), but likely needs to be skipped for local developer testing.

Some facts:
1. struct obd_device is defined in obd.h
2. obd.h includes lustre_import.h through lustre_lib.h, lustre_net.h before struct obd_device is defined
3. some files include only lustre_import.h but need to call obd_get_at_xxx functions
4. if obd_get_at_xxx are declared in lustre_import.h as macros, they cannot dereference obd->obd_at_xxx members.
5. if obd_get_at_xxx are declared in obd.h after struct obd_device as macros, they cannot be accessed by files including only lustre_import.h

Unless we are willing to make some more significant changes, declaring these functions in lustre_import.h and implementing/exporting them in class_obd.c is a doable way. The declaring in obd.h can be removed.
