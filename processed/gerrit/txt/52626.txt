--- Commit Message ---
LU-17180 ptlrpc: don't block ptlrpcd too long

sending rpc may blocked for long time during request allocation
or checksumming. Based on HPe tests it might introduce up to 5s
delay. Lets reuse a kernel workqueue for this to avoid network
starvation.

--- Message (Patrick Farrell, 2023-10-11 18:48:38.000000000) ---
Patch Set 1:

I'd like to understand more about the problem here - ptlrpc threads always have to send RPCs, right?

So in this case, are we saying "interpret + send" is too long, so we should do interpret in one thread, then send in another thread?

In that case, a few thoughts.
0. I think this will have a performance cost.  Maybe not huge but still.
1. Isn't this basically the same as not doing osc_io_unplug at all?
2. If we need to avoid the longer no-sleep time, I think would prefer to do cond_resched() here rather than specifically always do this by queueing.  That makes this optional based on scheduler pressure.

--- Message (Alexey Lyashkov, 2023-10-14 12:38:02.000000000) ---
Patch Set 1:

> I'd like to understand more about the problem here - ptlrpc threads always have to send RPCs, right?

And recv reply - it's FSM.

> So in this case, are we saying "interpret + send" is too long, so we should do interpret in one thread, then send in another thread?

we already have a write back threads. So have no reason to block ptlrpcd for preparing rpc+bulk which have an allocation. so it might block to processing other rpc for infinity time. So ptlrpcd may don't handle any. Early replies will don't sends - so client might evicted. Lock ins't granted (as reply handled after brw).... etc.

I think it will don't have, if we talk about complex load.

osc_io_unplug might called in user application context. long time waiting is OK where.

I don't talk about sleep/no-sleep time. I talk about blocking ptlrpcd from processing other rpc in own list/queue. One of backside of it - it might caused a false timeouts.
(bulk rpc is last in queue, and executed for long time, expire_one_request called before loop and it will expire a reply where it might be processed).

I might generate many other cases related to the long reply processing on client side, cased a delay on previous brw request.
