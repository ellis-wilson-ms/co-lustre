--- Message (Andreas Dilger, 2016-06-01 09:30:35.000000000) ---
Patch Set 1: Code-Review-1

It looks like this has introduced some regression test failures where the tests are using the local lov proc to get the filesystem-wide default stripe values.

It seems that these values are also kept uptodate on the client by the MGC config log:

 [root@basemyth ~]# lfs getstripe -dv /myth
 stripe_count:   1 stripe_size:    1048576 stripe_offset:  -1
 [root@basemyth ~]# lctl get_param lov.*.stripesize
 lov.myth-clilov-ffff88046630dc00.stripesize=1048576
 [root@basemyth ~]# lfs setstripe --stripe-size=4M /myth
 [root@basemyth ~]# lfs getstripe -dv /myth
 stripe_count:   1 stripe_size:    4194304 stripe_offset:  -1
 [root@basemyth ~]# lctl get_param lov.*.stripesize
 lov.myth-clilov-ffff88046630dc00.stripesize=4194304

So it seems that there is some value to keeping these /proc values around.  Sorry for the confusion.  However, it does seem that these lov.*.stripe* /proc values they are still writable on the client, which does not actually have any effect at all and should be disabled:

 [root@basemyth ~]# lctl set_param lov.*.stripesize=1M
 lov.myth-clilov-ffff88046630dc00.stripesize=1M
 [root@basemyth ~]# lctl get_param lov.*.stripesize
 lov.myth-clilov-ffff88046630dc00.stripesize=1048576
 [root@basemyth ~]# lfs getstripe -dv /myth
 stripe_count:   1 stripe_size:    4194304 stripe_offset:  -1
 [root@basemyth ~]# lctl get_param lov.*.stripesize
 lov.myth-clilov-ffff88046630dc00.stripesize=1048576

This would be done by deleting the code only for the lov_stripe*_seq_write() functions, and then changing LPROC_SEQ_FOPS() to LPROC_SEQ_FOPS_RO().
