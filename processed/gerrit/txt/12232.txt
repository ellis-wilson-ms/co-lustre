LU-5604 tgt: return missed fail ids

OBD_FAIL_LDLM_REPLY is missing from tgt_enqueue, and it's actually
not suitable for tgt_enqueue anymore because tgt_enqueue() is a
common handler now.

This patch includes a few changes:
- tgt_enqueue sets tgt_session_info::tsi_reply_fail_id to
  OBD_FAIL_MGS/MDS/OST_LDLM_REPLY_NET based on type of target.

- rewrite test_52 of replay-single, the only reason that test_52
  can pass is because there is a typo:

  $CHECKSTAT -t file $DIR/$tfile-* which should be $DIR/$tfile

- add definitions for OBD_FAIL_LDLM_SRV_CP/BL/GL_AST and resolve
  OBD_FAIL conflictions

- OBD_FAIL_UPDATE_OBJ_NET_REP was renamed to
  OBD_FAIL_OUT_UPDATE_NET_REP but referenced with old name in tests.

- OBD_FAIL_MDS_FAIL_LOV_LOG_ADD check is obsoleted as well as tests.
  Meanwhile the OSP code was updated to fix panic in case of error.

- OBD_FAIL_TGT_LAST_REPLAY is removed along with test. It was never
  used and it seems it was even introduced by mistake.

(2 comments)

This patch needs to be resubmitted with a Test-Parameters line to run replay-dual, since that isn't currently being run for regular review sessions, but is being modified by this patch. I don't want to land a patch that is introducing regressions that we could have easily detected in advance. Patch is -1 for this reason, though the "skip" issue should also be fixed. 

That also means this patch should be rebased on your other patch http://review.whamcloud.com/12363 that fixes replay-dual.sh

A couple of interesting follow-on improvements to this fix that ensure we do not see regressions like this again in the future would be possible:
- when setting fail_loc in a subtest, check at the end of the subtest that it was actually hit (message in the logs). That seems relatively easy for many cases, since run_one() is clearing fail_loc at the end of each subtest so it could search for the message in dmesg on each node before clearing the value
- have a build check that verifies each defined FAIL_LOC appears somewhere in a .c file, or at least in another .h file if there are cases where they are used in macros or static inline functions. 
- have a build check that verifies there is a test script using each FAIL_LOC that is defined. This may be harder to do, since some FAIL_LOC values are not set directly. 

These should be separate patches and not included in this one, but I think at least the first one is easy and useful, and all of them would avoid or reduce regressions in our testing coverage.
