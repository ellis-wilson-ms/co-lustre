LU-13912 lnet: Correct the router ping interval calculation

The router ping interval is being divided by the number of local nets
which results in sending pings more frequently than defined by the
alive_router_check_interval. In addition, the current code is structured
such that we may not find a peer net in need of a ping until after
inspecting the router list multiple times. Re-work the code so that the
loop that inspects a router's peer nets will look at all of them until
it either loops back around the list or it finds one that actually
needs to be pinged.

We also move the check of LNET_PEER_RTR_DISCOVERY so that we avoid the
work of inspecting the router's peer nets if the router is already being
discovered.

Patch Set 6: Code-Review+1

Thanks for the explanations.
As you say, LNET_PEER_RTR_DISCOVERY is only set in this code, and this code is only called by lnet_monitor_thread(), which is single threaded.  So there can be no race w.r.t setting the flag, so testing it inside the lock is unnecessary.  I would prefer the commit message explained that, but I guess it is no big deal.

However - now that I understand the code a bit better and have read it again, I see that each call will only ping one local interface for each router.  So it needs to be called multiple times per ping interval is there are multiple local interfaces that connect to the same router.  Can that happen?  Might that be why the division was there before? (We have the same division in lnet_monitor_thread().  Maybe that is sufficient).

It would be clearer if lnet_check_routers() checked every local interface on every router, pinged all that were ready, and returned the time until the next one would be ready.  But that should really be a separate patch.
And ... that wouldn't work as the use of  ->lp_disc_net_id means that you can only ping one net on a router at a time..
