> I'm just wondering if the new lctl runs against an upstream kernel with the old fixed major/minor numbers? I'd guess that it should work OK, and that userspace will just accept whatever device it opens, but it is something to verify.

Yes, the new lctl does work against the old module. The only place that lctl uses the minor is when calling mknod() in the create-device-if-not-found case. If you really want to preserve the create-device-if-not-found behavior then we can get the minor (whether static or dynamic) from /proc/misc.

--- Message (Andreas Dilger, 2017-10-30 14:06:32.000000000) ---
Patch Set 1:

> If you really want to preserve the create-device-if-not-found
> behavior then we can get the minor from /proc/misc.

That should probably be handled these days by a udev rule, instead of via lctl.  I see "lustre/conf/99-lustre.rules" that looks like it is creating the /dev/obd device, but I don't see an equivalent "lnet.rules" file to create /dev/lnet.  Is there something else that is creating that device?

--- Message (John Hammond, 2017-10-30 14:20:51.000000000) ---
Patch Set 1:

> That should probably be handled these days by a udev rule, instead of via lctl.

Yes, I agree.

> I see "lustre/conf/99-lustre.rules" that looks like it is creating the /dev/obd device, but I don't see an equivalent "lnet.rules" file to create /dev/lnet.  Is there something else that is creating that device?

The entry in lustre/conf/99-lustre.rules doesn't tell udev to create the /dev/obd node. Instead this tells udev to change the mode of 0666 when the device node appears. The actual device node is created by  the kernel (since devtmpfs is used for /dev).
