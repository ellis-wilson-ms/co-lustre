--- Commit Message ---
LU-11518 ldlm: control lru_size for extent lock

We register ELC for extent locks to be canceled at enqueue time,
but it can't make positive effect to locks that have dirty pages
under it. To keep the semantics of lru_size, the client should
check how many unused locks are cached after adding a lock into
lru list. If it has already exceeded the hard limit
(ns_max_unused), the client will initiate async lock cancellation
process in batch mode (ns->ns_cancel_batch).

And a new field named new sysfs attribute named *lru_cancel_batch*
is introduced into ldlm namespace to control the batch count.

after looking at this another time, I am concerned with feature creep in the lru_resize disabled case (like that extra gap above the setting to accumulate more locks).

when lru_size is set to a non-zero number - this was supposed to be the high watermark (per import) of locks in that lru. Then lru_resize came along and added the assumption about how if we have cancelset functionality - we'll try to batch cancels together from ELC and so there's no need to cancel right away. This still held up mostly ok since you needed to get more locks (though it broke some important assumptions that later came in some ticket that I cannot find where suddenly unrelated requests are slow because lock cancel of a long untouched lock takes a long time and many of them make it even worse).

Then a later change came in and disallowed ELC for locks that have dirty pages - therefore totally breaking the lru_size logic as is correctly noted in the ticket.

Now the simple fix to that is a oneliner - we need to restore the cancel call when lru_resize is disabled, and this is all.

All the extra logic I tend to concur with Vitaly - will need to go into the pools with flexible runtime-adjusting limits and all that. Otherwise we are ending up with two places that try to sort of do the same thing by different means. Let's try to avoid it.

On a separate note, since we do have lru_resize enabled by default, we need to better understand why people seem to want to disable it and fix those usecases as much as we can instead of throwing our hands in the air.

I think that in most cases, dynamic LRU size would be useful, but it has to properly handle memory pressure and aging, which I don't think that it does currently.
