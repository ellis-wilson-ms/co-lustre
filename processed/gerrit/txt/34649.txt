LU-10026: Page buffer pool for compression

Adds a pool of contiguous buffers for compression.
- Init before Lustre runs
- Able to access and free the memory buffers by buffer pointer or page-wise
- One can request a single buffer or a bundle of buffers ("atomic" loan)

Patch Set 3:

Compression algorithms usually require source and destination buffers.
In case of lz4 those have to be contiguous. While running the system,
it becomes difficult to allocate big contiguous pages. Therefore we
decided to implement a buffer pool which will be initialized before fs runs. Data in Lustre is accessed page-wise; the pool is able to access and free the memory buffers by buffer pointer or page-wise.

My concerns about implementing this page pool are:

a) do we need to make it available for encryption as well? The question arises from the fact that we cannot encrypt data in-place in the page cache, since the cached plaintext must be preserved. Instead, we must encrypt into a temporary buffer or "bounce page", then write out the temporary buffer. The good news is fscrypt is taking care of the bounce page allocation via a page pool of its own, so we will not have to handle this page allocation at the Lustre level;

b) can we re-use what already exists with Lustre enc_pools? After looking at enc_pools implementation, it seems to be tightly coupled to GSS usage, so it would need to be made more generic. Also, I was wrong in assuming that enc_pools mechanism was allocating contiguous pages. Pages are located one by one by calling alloc_page(). So it would need other modifications to allow requesting for contiguous pages.


However, enc_pools mechanism has important features that this implementation for compression lacks in this version:

1) wait queue in case of not enough free pages. This is not really acceptable to sleep for an arbitrary amount of time, like wait-For_mem() does;

2) usage statistics. Knowing cache access/miss, grow/shrink, requests, etc would make much sense for debugging purpose and also when compression is used in production.


So, to sum up my thoughts, after examining a) and b) I would tend to think that you could go ahead with this compression specific implementation of a page pool, but that would require to get inspiration from enc_pools implementation, in particular to address remarks 1) and 2).

Points 1 and 2 are about performance optimization. It will come in the second part of my research work.
Once the first memory request is rejected, we have, I would say, three possibilities how to react.
1) just wait a bit, what we are currently doing (prioritized waiting should be possible, but not mandatory)
2) cancel compression of the particular chunk/stripe/bulk and proceed with uncompressed piece of data
3) grow the pool
We are not aiming to make the pool growable (or shrinkable), but static and preallocated before the system runs.
Since we have potentially big contiguous blocks (ZFS record size can be up to 1MiB), the chance to allocate such blocks under system load is very low.
In the thread I have linked we have already discussed how to figure out the best pool size. This has to be examined very accurately (balance client's memory for computation and compression; how to deal with additional OSTs, etc), but for the first its size will be static.
So we can choose between cancelling and waiting. In my research work I plan to introduce some smart and flexible strategies depending on our test results. But all that is future work.

Regarding enc_pool, it soudns like encryption has other demands. The pool does not offer what we need and enc does not need what we offer. Might be needles to combine them at any price. At least not yet.

--- Message (Anna Fuchs, 2019-04-26 11:48:25.000000000) ---
Patch Set 4:

The compressor requires virtually contiguous memory. We afford the luxury to have physically contiguous pages here and that, we believe, for a low price. When using only virtually contig. memory like Linux provides via vmalloc, you can be pretty sure to get what you need, even huge blocks. Therefore, you might be able to use it dynamically under system load, let the pool grow and shrink. Due to lower performance permanent vmallocing during the system run becomes more expensive. So here the decision to prepare everything from the beginning -- we hope the memory not to be very much fragmented then. I believe accessing physically contig. memory performs better than virt. 
We used alloc_pages to have a more comfortable handling when requesting a page array instead of page buffer (we have another compression algorithm which can handle page arrays to save one buffer copy). Otherwise, kvmalloc might be our choice. Once we really wanted to make the pool growable, we might go with vmalloc for the dynamic part and keep the core pool kmalloced. We might change it later when benchmarking everything.

--- Message (Andreas Dilger, 2019-04-26 23:39:22.000000000) ---
Patch Set 4:

One problem we had with large kmalloc() "at startup" is that this works if the system is newly booted, but fails if the client or server is inmounted and remounted without a reboot. That can happen during service or upgrading the RPMs.  Some very large nodes can take tens of minutes to reboot if they have large numbers of disks, CPUs, RAM, PCI cards, etc. so it is best not to require a reboot for that to work.
