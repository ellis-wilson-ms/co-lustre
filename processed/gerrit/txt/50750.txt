LU-13306 mgc: handle large NID formats

For newer versions of Lustre the MGS can send mgs_nidtbl_entry
containing NIDs of a larger format. Its also possible an old
MGS will send NIDs of the previous size. We need to handle
both cases. We reused the field of mcb_nm_cur_pass, which only
is used for nodemap, of the struct mgs_config_body to send the
NID size from the client to the MGS. Pre IPv6 clients will
by default have a zero mcb_nm_cur_pass / mcb_nid_size. When
mcb_nid_size is zero the the MGS will treat the client as
pre-IPv6 and send small NID back to the client. This avoids
needing to patch older clients. If the MGS is older then
small size NIDs will be sent back which the new MGC layer can
handle by converting those lnet_nid_t to struct lnet_nid.

To handle this new code the "swab" of the entry is split into
two parts. The "header" is "swab"ed as soon as we know the entry
is large enough for that to make sense. The content containing
NID information is swabbed later once the header has been found
to look sane.

What is the point of this patch?
For master, we want to make lustre handle large nids, not gracefully ignore them.
So is this intended to be added to older releases so they can interoperate?
If so: making that explicit would help.

I had hoped that as mgs_nidtbl_entry contains nid_type and nid_size info, that unrecognized nids would be ignored.  I now see that isn't the case.  We should just fix those problems for a backport.
So:
 - lustre_swab_mgs_nidtbl_entry() should retrun when nid_type is not 0, rather then using LASSERT.
 - mgc_apply_recover_logs should either stop reading one it finds nid_type!=0 allowing previous entries to be used, or maybe skip over unknown types, as it has the size.

I think that is sufficient.
Your patch seems to do a lot more than that.  I certainly see no need to struct mgs_nidtbl_entry in older releases.
Am I missing something?

--- Message (James Simmons, 2023-05-31 17:19:20.000000000) ---
Patch Set 8:

We send the nid size info with struct mgs_nidtbl_entry which is sent from the MGS server to the client. The reserve is not the case. So I'm sending the nid size using struct mgs_config_body from the client to the MGS server. This informs us that it is safe to send larger NID sizes. With the size info we can filter and only send NIDs that the client can handle. This is in case struct lnet_nid expands in the future.

Now we could avoid this if we send NID strings instead which should always work. Andreas, Neil are you okay with this approach or would prefer using NID strings instead.
