LU-11607 tests: create routine to get Lustre env

The Lustre tests in the test suites make repeated calls
to a small number functions that relate to the Lustre
environment. Some examples are the version of the Lustre
server or client and the file system type of the server.

Collect these calls into a routine called get_lustre_env()
in the test-framework.sh library and replace calls in
sanity with the global variables.

Patch Set 4: Code-Review-1

Jian brought out the following issues with the MDS definitions in this patch:
1.	If SINGLEMDS is defined as something other than mds1 and if the Lustre version of SINGLEMDS is different from mds1, MDS1_VERSION cannot replace $(lustre_version_code $SINGLEMDS). Andreas said the MDS server versions are allowed to have a version difference of approximately +/- 0.0.4 . In our autotesting, we never test with different MDS versions, but I can’t comment on how other sites test w.r.t server versions. Is this a difference in MDS versions that we can tolerate in our test scripts? 

2.	If SINGLEMDS is defined as something other than mds1 and if the SINGLEMDS file system type is different from mds1, mds1_FSTYPE cannot replace $(facet_fstype $SINGLEMDS). I don’t see a way around having an mds1_FSTYPE and a singlemds_FSTYPE right now. 

It is my understanding that SINGLEMDS is supposed to be used for commands that must be run on MDT0. We use “mds1” for this same purpose. Is there any reason to keep using SINGLEMDS? LU-6438 proposed removing SINGLEMDS. LU-7753 proposes renaming SINGLEMDS to ROOTMDS. The rename doesn’t seem necessary when we have and can use mds1. Any thoughts?

--- Message (Andreas Dilger, 2019-01-23 21:15:39.000000000) ---

My preference would be to just remove SINGLEMDS.  While there is some chance in the future that we would allow removing MDT0000 from mds1 (e.g. if we mirror the root directory) it seems a far way off.  Also, we already have the mechanism to move the mds1 facet to another node, so I don't see a big benefit to keeping this separate.
