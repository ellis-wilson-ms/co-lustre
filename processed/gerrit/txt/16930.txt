I thought that we agreed to have a single conversion function that only took a char to specify default units.  Can you explain why you think you still need to provide a "multiplier" option? --- Message (Giuseppe Di Natale, 2015-10-28 15:35:32.000000000) ---
Patch Set 3:

> (1 comment)
 > 
 > I thought that we agreed to have a single conversion function that
 > only took a char to specify default units.  Can you explain why you
 > think you still need to provide a "multiplier" option?  From my
 > reading of the code, it seemed pretty straight forward to handle
 > the non-units multiplier in the caller instead of in the parsing
 > function.

I provided the multiplier option because within the helper, the multiplier is accounted for when determining wrapping. I think it is safer that way instead of relying on the caller to pay attention to it.

Because I decided it was safer, I couldn't provide a unit field that was simply a char (I'd have to make up a unit for pages for instance). What I can do, is provide a simple enumeration representing default units/conversions instead of the multiplier and internally convert that to the correct conversion factor.
