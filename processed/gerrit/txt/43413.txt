LU-13883 lnet: Lookup lpni after discovery

The lpni for a nid can change as part of the discovery process (see
lnet_peer_add_nid()). As such, callers of lnet_discover_peer_locked()
need to lookup the lpni again after discovery completes to make sure
they get the correct peer.

An exception is lnet_check_routers() which doesn't do anything with
the peer or peer NI after the call to lnet_discover_peer_locked().
If the router list is changed then lnet_check_routers() will already
repeat discovery.

Hello,

We believe we have found bug with the current implementation of this patch.

We test this backport on a filesystem with 300+ OST targets (with 7 failover nodes for each). The clients (~20) access to the filesystem through a IB-IB router. Discovery and health is globally disabled on all the peers.

When mounting the filesystem on the clients (in the same time), 30% of the OST targets were not able to connect with clients. On the dk log, clients complain about credits.

We try to remove that patch from our version and it works find without.

We have 40+ patches on our 2.12.6 version. So to be sure, we tested the master branch and we saw the same symptoms.

To sum-up, I think that there is high probability to have a design flaw on the original patch.

Unfortunately, the filesystem is on the restricted site. If we can provide logs, we will create a Jira ticket for master.
