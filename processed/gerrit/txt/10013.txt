--- Commit Message ---
LU-4840 lfs: Use file lease to implement migration

Implement non-blocking migration based on exclusive open instead of
group lock. Implemented exclusive close operation to atomically put
a lease, swap two layouts and close a file. This allows race-free
migrations.

Make the caller responsible for retrying on failure (EBUSY, EAGAIN)
in non-blocking mode.

In blocking mode, allow applications to trigger layout swaps using a
grouplock they already own, to prevent race conditions between the
actual data copy and the layout swap. Updated lfs accordingly. File
leases are also taken in blocking mode, so that lfs migrate can issue
a warning if an application attempts to open a file that is being
migrated and gets blocked.

Timestamps (atime/mtime) are set from userland, after the layout swap
is performed, to prevent conflicts with the grouplock.

lli_trunc_sem is taken/released in the vvp_io layer, under the DLM
lock. This re-ordering fixes the original issue between truncate and
migrate.

--- Message (Henri Doreau, 2014-09-19 12:26:28.000000000) ---
Patch Set 7:

The deadlock comes from ll_setattr() when SWAP_LAYOUT_KEEP_*TIME flags
are used. ll_get_grouplock starts a CIT_MISC IO which ll_layout_refresh waits for.

Currently, lfs gets a grouplock to perform migration and puts it right before swapping layouts, opening a race window to concurrent processes to modify the file. That cannot work and blocking migration & layout swap have to be done under a same grouplock to be safe, I think.

A possible solution would be to call futimes() from userland, after the grouplock is put. The SWAP_LAYOUT_KEEP_*TIME flags would be useless for the ioctl then... It still looks suboptimal to me.

I'm investigating clean ways to fix the issue, meanwhile any 
hint/suggestion is welcome!

--- Message (Jinshan Xiong, 2014-10-01 18:09:37.000000000) ---
Patch Set 7:

Due to the possibility of setting timestamps by user during migration, for example, touch atime or mtime, it's really difficult to address this kind of race condition. This is why I'm okay with Henri's proposal - now that it's difficult to solve it anyway, then why not choose an easier solution.

How about this - let's copy and set the timestamps to volatile file before swapping layout, in that case, it's enough to address the race condition from file content access but leave touch case open, because:
1. if MIGRATION_BLOCKS is set, then the timestamp must be valid because nobody can access file contents due to group lock;
2. otherwise, lease will be broken and volatile file will be discarded.

Why do the application care about the jumping of atime,mtime so much? I mean backup software only do scans periodically.
