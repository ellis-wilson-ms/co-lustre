--- Commit Message ---
LU-17217 ptlrpc: service-side connection policy

Enable Lustre servers to selectively block connections
based on critereon decided by admins.

Implemented VERSION policies. Admins can provide a glob
pattern and policy (WARN, SOFT_BLOCK, HARD_BLOCK) to
control which clients can connect to a file system.

Added new lctl commands to set and display active
policies.

Create several new sanity tests to exercise this
new functionality.

Patch Set 2: Code-Review-1

(33 comments)

For version-based checking, there is already LUSTRE_VERSION_OFFSET_WARN, which will already cause clients and servers to emit a mount warning when the versions are more than 0.4.50 newer or older (eg. 2.10 clients and 2.15 servers, or vice versa).

It would seem that making LUSTRE_VERSION_OFFSET_WARN a tunable parameter instead of a compile-time constant would be less complex, but I now see that this is only checked on the server. It used to also be checked in target_handle
_connect(), which is where all of the server-side connection-time compatibility checking is done. 

I can see the benefit of the server being able to temporarily block old client versions from mounting, and maybe other criterion such as client NID, or block all clients from mounting while something like a server upgrade or LFSCK or replace_nids is being done, but I don't yet see the value of allowing clients to override this.  I think it is possible to implement that on the server without a connect flag, and just return an error to the client (unless you want it to keep retrying to mount until the service action is finished).
