--- Commit Message ---
LU-4647 nodemap: add mapping functionality

Add functions to allow the MDT to use the nodemap module
functions to map UIDs and GIDs. When a client connects to the MDT,
the NID of the client is classified by nodemap and a pointer
to the nodemap is added to the export structure.
In old_init_ucred and old_init_ucred_reint, the body ids and fs
ids are mapped according to the nodemap state (enabled, maps,
squashed ids, trust_client_ids and allow_root_access). ACLs are mapped
in mdt_getattr_internal. Squashed ACLs are discarded before the ACLs
are sent to clients. Trusted administrative nodes are allowed to
use setfacl on all users. Mapped nodes can only use setfacl for users
also mapped into the node.

Patch Set 2: Code-Review-1

(13 comments)

I'm surprised that this patch does not appear to intersect with http://review.whamcloud.com/8817 ?

I think there is a significant problem with access to the exports list and obd_devs[] array without locking, since the list traversal may go badly if clients connect or disconnect while the nodemaps are being updated.

I also think there is a real performance issue if the whole obd_devs array is traversed possibly hundreds of times.  A better solution would be to have a single global list that linked all of the MDT devices or similar.

Also, it isn't clear if the nodemaps are separate per filesystem, or if there can only be a single nodemap for the whole MDS?  I don't know if there is a good usecase for a per-filesystem nodemaps or not.

(18 comments)

While the patch has several places that need cleanup or minor fixes, there is nothing totally critical that would prevent it from landing.

That said, my understanding is that there are still at least two patches needed in order for this feature to be functional - UID/GID mapping on the OST, and some ability to load the nodemap/idmap on the MDT and OST in the first place.

Without those pieces I don't think it makes sense to land this patch, since it will add the no-op nodemap handling for every connection and every RPC, without any possibility of it actually being useful.

--- Message (Andreas Dilger, 2014-09-30 02:51:55.000000000) ---
Patch Set 28:

I would ignore the existence of rmt*acl commands for now. Those are intended to allow setting ACLs for Kerberos to contain principals that are not in the current domain (i.e. Users that are not currently mapped in the nodemap).

However, since nodemap is only dealing with numberic UID and GID it is (or should be) essentially impossible for anyone inside the nodemap to specify such an ID, so we decided to only allow specifying UID and god values that exist in the mapping. If they do not, then the setfacl should fail, or map the users in a safe manner for getfacl (i.e. squash the user gaining permission, or deny all other users if the ACL is denying access). 

Allowing setfacl and getfacl to work inside the nodemap would facilitate their use for virtual hosting and similar setup a where the clients should be isolated from each other.
