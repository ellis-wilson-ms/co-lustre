--- Commit Message ---
LU-5556 target: limit bulk transfer time

Messages lost during bulk transfer are not resent, so there is no
point in waiting for a very long time (up to at_max/600s has been
seen). This patch adds a new static timeout for the bulk transfer
(100s by default).

Having multiple layers handling resend with no coordinations is going to be a complete mess. I think the root cause of the problem i am trying to solve here is well understood (at least by us). We have fault injection code running on the router (new patches developed by Liang) which drops 1/1000 messages. In fact, using a different router in this present test isn't a guarantee of success at all.
Even if we end up with handling resend at the LNET level, there is still no point in waiting for ages for the bulk transfer to complete and to keep a service thread busy during this time.

Shadow, could you please point me at your patches (gerrit IDs) which handle resend via different routers at the LNET level? I will be delighted to give it a try on our test cluster. Thanks in advance.
