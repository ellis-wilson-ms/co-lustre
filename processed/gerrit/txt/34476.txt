--- Commit Message ---
LU-11893 o2iblnd: add secondary IP address handling

Using dev_get_by_name() in kiblnd_create_dev() means we can only
discover primary IP addresses. This breaks using network
aliasing which some people use. Move away from dev_get_by_name()
to using for_ifa() so we can detect any secondary IP addresses.

Patch Set 1: Code-Review-1

So strcmp is causing a NULL pointer reference. at 
 RIP: 0010:strcmp+0x4/0x30
That would be: 
  00:	48 83 c7 01          	add    $0x1,%rdi
  04:*	0f b6 47 ff          	movzbl -0x1(%rdi),%eax		<-- trapping instruction

So it is deferencing one of the pointers it was passed.
The "NULL" address is actually:
  BUG: unable to handle kernel NULL pointer
dereference at 0000000000000024

0x24.  The code is
 strncmp(netdev->name, ifname, strlen(netdev->name)) != 0)

So it seems rather likely that netdev is NULL, and ->name is at an offset of 0x24.  So netdev->name becomes the address 0x24, and when strcmp accesses that address, it triggers a BUG.

I don't think you can blame strcmp() for BUGing when you pass it a NULL pointer.

I get why the switch of the variables in the strcmp prevents the stack dump...
ifname can be up to 128 characters:
--------
lnet/include/uapi/linux/lnet/lnet-dlc.h:    char ni_interfaces[LNET_INTERFACES_NUM][LNET_MAX_STR_LEN];
lnet/include/uapi/linux/lnet/lnet-dlc.h:#define LNET_MAX_STR_LEN    128
--------
WHILE ifa_label is 16 bytes
--------
include/linux/inetdevice.h:    char            ifa_label[IFNAMSIZ];
include/net/dn_dev.h:    char              ifa_label[IFNAMSIZ];
include/uapi/linux/if.h:#define    IFNAMSIZ    16
--------
so if ifname was one byte over 16, then stack dump
