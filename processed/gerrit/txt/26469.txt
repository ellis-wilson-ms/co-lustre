LU-8964 llite: Make readahead async

Removed current Lustre implementation of readahead and replaced it
with in-kernel implementation. Adopted code for in-kernel readahead
usage. Perform readahead requests asynchronously from current I/O.

In combination with parallelization of generic I/O this patch allows
significantly improve read performance.

this patch (2.10.54_14):
20971520000 bytes (21 GB) copied, 7.82126 s, 2.7 GB/s

One other *enormous* benefit from this patch:
It appears to fix whatever it is that breaks in strided readahead when we read a file from multiple threads on the same node.

For example, without this patch:
Reading a file 47K at a time (on some slow-ish VMs) from one thread:
mpirun --allow-run-as-root -n 1 /shared/testcases/IOR_DIR/IOR -o /mnt/cent7s02/ior/iorfile -r -k -E -t 47K -b 47K -s 80000
read          786.22     786.22      786.22      0.00       0.21       0.21        0.21      0.00   4.67028   EXCEL

From two threads: (same line except -n = 2 and -s 4000):
read      92.98      47.00      47.00      0.001136   39.48      0.008979   39.49      0

~80% performance drop. This is typical...

With this patch, one thread:
read      835.62     47.00      47.00      0.000711   4.39       0.000501   4.39       0

Two threads:
read      829.68     47.00      47.00      0.001756   4.42       0.004775   4.43       0

~830 MB/s is close to the limit for real reads on this little VM system.  I expect a real system would show large benefits going to multiple threads, though I would have to verify.

Anyway, this is a huge benefit, and note that the I/O pattern is strided (at least, in the multi-thread case).  Still trying to find time to do a complete review.
