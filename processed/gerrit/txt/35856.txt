LU-12682 llite: fake symlink type of foreign file/dir

This patch implements a "fake symlink" specific usage of
"foreign" LOV/LMV format. It basically allows these
particular type of foreign files/dirs to behave as a
symlink from VFS point of view, by allowing to construct
a relative path from the LOV/LMV foreign content, to
complement it with a prefix, and then to expose it to
the VFS as a symlink destination. The default/internal
mechanism simply takes the full foreign free string as
the relative path, and for more complex internal formats
an upcall has been implemented to provide format's
details (presently just in terms of constant strings
and substrings positions in EA, but this can be enhanced)
to llite layer.
Using this feature, instead of real symlinks or user EA,
will permit to benefit from the special features (lock,
prefetch, caches) already implemented to handle both
LOV/LMV EAs.

You are right that I thought DAOS had its own userspace component that could interpret this.

 > You may also now better understand my own
 > question/comment concerning how you think the user-land parsing
 > should be implemented ? May be by passing EA content to some
 > user-land upcall (registered for DAOS type) to do the parsing ? But
 > since it should allow for multiple requests to run in parallel, may
 > be you could point me to some already existing piece of similar

Definitely this patch is adding a lot of "policy" to the kernel, which would be very difficult to get into the upstream client.  Having an upcall to do the symlink lookup would be much more flexible.

A similar functionality exists in the l_getidentity upcall to map a UID to a list of supplemental GIDs.  This would take the binary DAOS layout (including the magic), pass it to userspace, and userspace would return the symlink target.  That allows it to be totally flexible in changing the mountpoints for the DAOS containers (e.g. depending on the site configuration) rather than having to patch Lustre to handle changes.  Doing a pure upcall would be the most flexible for sure.  If you don't expect a large number of such symlinks to be accessed and performance is not critical.

Another option would be to have a DAOS upcall that is called only rarely for a filesystem and returns a format string that allows printing out the symlink directly in the kernel, for example "/mnt/daos/myds/%pUb/%pUb" or similar (see linux/Documentation/core-api/printk-formats.rst in the kernel sources).  There would also need to be some way to specify how the binary field offsets are mapped to the arguments.

Finally another option would be to have the llite code request a module like "llite-foreign-<type>" that has very simple functions to generate the symlink from the layout and isolate it from the core Lustre code.

I think the easiest option would be to start by moving the foreign symlink code into a separate llite-foreign.c file, would at least be a good start, and avoid sprinkling the foreign layout code around llite.
