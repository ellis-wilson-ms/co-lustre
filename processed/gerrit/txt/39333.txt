LU-13775 target: fix memory copy in tgt_pages2shortio()

tgt_pages2shortio() try to copy local pages memory to ptlrpc
inline buf.

The right logic should move page @ptr to offset + count, however,
it does this logic wrongly, this doesn't cause any problem so
far, because normally @lnb_page_offset is 0. when i tried to
play with unaligned DIO, we could hit the problem.

Anyway, fix to use right logic to handle memory.

We have to be careful here. There is both lnb_page_offset and lnb_file_offset, but I don't recall the exact difference between the two.

Long ago, there were "liblustre" clients, and they submitted unaligned IO in non-PAGE_SIZE fragments as we are trying to here.

I checked tgt_shortio2pages(), and it _looks_ correct, as I was otherwise thinking that there should be more to this patch.

It would be useful to verify e.g. sanityn.sh test_9[ab] and test_10[ab] are using short IO, since they have proper data verification.  Similarly, test_16[a-e] are doing small interleaved writes and reads fsx and have data verification and are important to verify for this kind of change.  Probably for fsx the code needs to change to *not* align the O_DIRECT to any boundary, and it would quickly expose whether there is a problem with the code.

--- Message (Patrick Farrell (personal), 2020-07-11 02:04:46.000000000) ---
Patch Set 1:

"We have to be careful here. There is both lnb_page_offset and lnb_file_offset, but I don't recall the exact difference between the two."
lnb_page_offset is the offset within the page, lnb_file_offset is the offset in the file.  I'm not sure off hand whether that's the page index or if it's a byte offset (and my home server just developed a fan issue, so the Lustre source is less accessible).

I'm pretty sure this one is not going to be possible to validate, because I'm almost certain the offset in the page is always going to be zero by the time i/o gets to the server (except for unaligned DIO, but we don't have that).

So, assuming DIO is aligned (as it must normally be), then for buffered i/o, the only case where you don't do a full page is where file size is not page aligned.  And that means you could do, say, the first 2K of page (if a file is 2K in size), but you would never do the *last* 2K of a page.  If you wrote the last 2K of a page (ie, bytes 2048-4095), the page cache pulls in the rest and writes the whole thing.

So there's this bug, but I don't think we can trigger it today.

And reads are always full pages, again unless there's a file size thing, which can only affect the ending offset in a page, not the starting offset.
