--- Commit Message ---
LU-10283 mdd: fix parent FID in changelog of striped directory

Changelog entry for the file operations such as create, rename,
link, unlink, mkdir referred to parent FID ("p=") as a shard's
FID in a striped directory. The same was true for the source's
parent FID ("sp="). This commit hides the Lustre intrinsics from
user displaying the parent's directory FID instead as expected.

An object might be in a remote MDT, in which case obtaining the parent
FID via the linkEA can be an expensive operation, so the parent FID is
cached in the mdd_object, so that the cost of the cross-MDT RPC is
amortized over the lifetime of the object.

Certain userspace tools might depend on the previous behavior of
displaying the shard's parent FID in the changelog records, so this
canp be enabled by setting mdd.*.enable_shard_pfid=1, if this is
required for compatibility.

Maybe I'm missing something, but isn't the parent FID already stored in the ".." entry of the directory stripe?  Then there is no need to do a remote getxattr for each changelog entry. 

However, there is still some compatibility concern with changing this entry, so it makes sense to have a tunable to enable/disable this option.  The main question is whether it should be on or off by default?  If on by default, it simplifies things for most users, but may break tools that expect the stripe FID to be returned.  If off by default it keeps compatibility but makes it harder to fix the situation in the long run.  

Another option would be to store the parent FID in a new record, like "ppfid=" that "aware" programs/tools can use, but it won't break old ones. 

I've added a few additional reviewers who are more familiar with HSM ChangeLog users to get their input on the best path forward.

Unless the performance impact is *really* bad, I would prefer that this functionality be enabled by default, since it makes the changelogs "correct" and that is probably more important than a small percentage slowdown for a small percentage of directories.

Have you measured how much the performance impact is for this change (if we still need the remote RPC)?  Striped directories should be pretty uncommon (only really for huge single directories), especially since DNE3 space balancing is available to handle normal MDT balance.

If this is expensive, it would also make sense to cache the pfid into the shard object in memory, so it only needs to be fetched once in the object lifetime.  In most cases, it is common to create a *lot* of entries in a striped directory rapidly, so caching the ppfid would amortize the RPC cost over many changelog entries.  It would need to be invalidated on rename, but that is an uncommon operation and (IIRC) already has to touch the shard objects.

The name of the tunable parameter should also be added at least in the patch commit message.
