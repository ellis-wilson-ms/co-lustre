LU-9679 osc: convert oe_refc to kref

oe_refc is used like a kref, so make it one.

As cl_object_put() needs an 'env', we cannot quite use
kref_put() as intended.  Maybe that can be fixed one day.

This requires kref_read() to be provided for older kernels.

The original Linux-commit also converted oe_users to a refcount_t
but that was later reverted.

Patch Set 7:

> I think there are two different uses for lu_env, and likely also some abuses as well.

Yes, the documentation talks about le_ctx to reduce stack usage, and le_ses for thread context.
I'd like to see the le_ctx usage gone but I don't think we can rush it. virtually-mapped stacks appeared in 4.9 and we've definitely had 8K stacks since then.
Support for CONFIG_4KSTACKS for x86 was removed in 2.6.27 (dcfa7262801) which was nearly 10 years ago.  So I think it might be time to start removing some le_ctx usage.

le_ses will need more investigation.  It certainly makes sense to have some thread context for kernel threads.  Whether lu_env in a good way to implement this I don't know - I won't really have an opinion until I look at the code mode.
I *think* le_ses is also used client-side outside of service threads.  I'm not at all sure that is a good idea.  Obviously I need to understand the details first, but after le_ctx is dealt with I'd be much more likely to attack client-side usages of le_ses than server-side usage.
Maybe I'll create a Jira issue...
