--- Commit Message ---
LU-13182 llite: Avoid eternel retry loops with MAP_POPULATE

Kernels 5.4+ have an infinite retry loop from MAP_POPULATE mmap
option. Use the FAULT_FLAG_RETRY_NOWAIT to instruct filemap_fault
to not drop the mmap_sem so if the call fails, we could use
the slow path and break the loop from forming.
(Idea by Neil Brown)

I think this problem is related to
Commit 6b4c9f446981 ("filemap: drop the mmap_sem for all blocking operations")

which landed in v5.1.  filemap_fault now drops mmap_sem more often and so returns VM_FAULT_RETRY more often.  I think that ll_fault0 needs to go through the slow path even when VM_FAULT_RETRY is returned.

Maybe it would need to retake the mmap_sem first?  Or maybe making sure to clear FAULT_FLAG_ALLOW_RETRY when calling filemap_fault again.  Or something.

Might be easiest just to disable fast-fault until this is properly understood.

> Also what downsides do you see to the current approach 

The main downside is that it doesn't make sense.  Why test for that particular flag?  What does the test *mean*?

I thought of something that seems to work and does seem to make sense.

When calling filemap_fault for the 'fast' case, set FAULT_FLAG_ALLOW_RETRY in vmf->flags (and then restore the ->flags value afterwards).  Also change the test of VM_FAULT_RETRY.  If that is set it is now safe to fall-through.  Only return if none of VM_FAULT_RETRY or VM_FAULT_ERROR are set.

This tells filemap_fault to *not* drop the mmap_sem - so if it cannot be fast, it fails.  ll_fault0 catches the failure and proceeds with the slow path.

This should work correctly with old and new kernels - no need to test which.

 		/* do fast fault */
+		unsigned int flags = vmf->flags;

 		fault_ret = filemap_fault(vmf);
 		ll_cl_remove(vma->vm_file, env);
+		vmf->flags = flags;
