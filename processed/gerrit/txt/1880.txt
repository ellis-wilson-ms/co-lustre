--- Commit Message ---
LU-80 tests: large xattr support

This patch adds and updates some test cases to verify the
large xattr feature. To enable this feature, the "-O large_xattr"
option needs to be set on the filesystem either with --mkfsoptions
at format time or via tune2fs.

Signed-off-by: Yu Jian <yujian@whamcloud.com>

--- Message (Andreas Dilger, 2011-12-16 17:55:20.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

Changes look pretty good. However, I don't think that we can make the test xattr size a full 65536 bytes when the "mid-sized" xattr support lands. It needs a small amount of overhead to fit the xattr_header. It would be something like 65536-64=65472 bytes. 

Instead of hard coding the maximum xattr size into all of the tests it should be a constant set by large_xattr_enabled(). Maybe instead of returning a Boolean true/false value it should be renamed large_xattr_size() and return the maximum xattr size?

The other issue is that these tests _could_ test the large xattr support, but do not.  If they are left disabled then it is unlikely that they will ever be enabled. At a minimum, once the ldiskfs patch lands some conf-sanity test should format a small large_ filesystem and always test this feature.

I added a new function max_xattr_size() to return the maximum xattr size. Since the current large xattr implementation could save a full 65536 bytes value in an external inode, I made that function still return 65536 while large xattr feature enabled. After "mid-sized" xattr support is implemented, I'd revise the function.
