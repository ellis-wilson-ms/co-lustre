--- Commit Message ---
LU-9410 ldiskfs: handle unmatched bitmap

Two fixes:

1) Read bitmap with lock
The old logic will set the bitmap buffer head as 'bitmap_uptodate'
before buffer head is really read from disk. If others also want
to read such bitmap during such interval by race, then the latter
may get uninitialized bitmap, as to some fake data corruption.

This patch makes the bitmap reader to take lock on the bitmap buffer
head before checking whether it is up-to-date or not. Because the
firstly bitmap reader will hold such lock when loading the bitmap
from disk, then it will prevent the others accessing uninitialized
bitmap. It may be not the perfect solution, but we can use it to
resolve our current trouble.

2) Set free blocks count for non-initialized group
ldiskfs_init_block_bitmap() will initialize the group block bitmap,
but the free_blocks_count in the group descriptor maybe still zero.
That will cause the subsequent ldiskfs_mb_check_ondisk_bitmap() to
report corruption by wrong.

This patch sets the free_blocks_count in the group descriptor with
the real free blocks count in the group.
