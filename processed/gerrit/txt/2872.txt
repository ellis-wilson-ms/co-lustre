to 4MB by sending 4 1MB chuncks at a time to servers from the clients.

--- Message (Andreas Dilger, 2012-05-22 17:45:41.000000000) ---
Patch Set 1:

James, I haven't had time to look at this patch yet, but two comments.

Firstly, Xyratex is reportedly working on this code as well, so it makes sense to coordinate efforts with them.  I've assigned Shadow as inspector just to inform him of this patch, but feel free to assign someone else more appropriate as needed.

Secondly, there are performance concerns with this patch that were raised in bugzilla, namely that this patch causes performance regressions in some workloads, while improving performance in other workloads.  If the patch defaulted to 1MB bulk RPCs initially, this would allow the large RPC functionality to land without hurting performance under those workloads, would allow benchmarking this functionality under different workloads, and would allow it to be enabled for those users/sites where it is a clear benefit (e.g. WAN).

Patch Set 1: I would prefer that you didn't submit this

James, there is already connect-time protocol support for negotiating the maximum BRW size for OST requests.  Please see obd_connect_data.ocd_brw_size.

It does appear that the large readdir patch included the ability to negotiate ocd_brw_size with the MDT using the OBD_CONNECT_BRW_SIZE feature, but it is entirely possible that the code doesn't expect a larger BRW size than 1MB.
