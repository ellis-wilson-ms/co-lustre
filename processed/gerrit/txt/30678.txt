--- Message (Andreas Dilger, 2018-01-02 08:45:17.000000000) ---
Patch Set 2:

(2 comments)

The description in the LU ticket explains the problem a bit better, which would be good to add here. It doesn't explain what is actually using the extra space? Is it some file that is not being deleted properly? Some directory that grows larger when the files are created, but doesn't shrink when they are deleted? An OI file that grows larger?

Instead of changing the test to adjust the record size, which affects all later tests until the filesystem is reformatted, it would be good to know what file is consuming the larger space, and we could potentially adjust the block size of that file when it is created, if that makes more sense. OI files probably shouldn't have such large leaf blocks anyway, since they are updated in random orders, and the large record size would increase write amplification.

--- Message (Hongchao Zhang, 2018-01-02 10:42:41.000000000) ---
Patch Set 2:

I have tested it in a loop locally, and found the block space usage difference is related to the recordsize of
the newly created file by dd, and it always be the times of the recordsize (I have changed the recordsize
to 32k, 64k and 128k), then it should be that the some blocks of the newly created file are not freed in time
when we call df after deletion.

Do these blocks eventually get freed if enough time passes?  It seems like this is a bug in the test, and changing the recordsize is only hiding the problem.

The test is currently calling wait_delete_completed_mds(), but it seems that it should be calling wait_delete_completed() so that it is also waiting for the OST objects to be destroyed?
