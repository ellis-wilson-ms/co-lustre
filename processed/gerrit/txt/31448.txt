(2 comments)

Style issues are minor and not a reason to refresh patch.

Would it be better to just send the RPC without the checksum if cfs_crypto_hash_init() fails?  Looking at this, I see only a few cases where it can fail:
- out of memory - would be good to send the RPC, since it may allow memory to be freed, or it will fail shortly thereafter due to other allocation failure
- unsupported checksum - something is broken with the Lustre code, since clients should verify supported checksums at startup time
- something wrong in the kernel crypto - better to send?

This doesn't need to be fixed in this patch (which is good to land separately, since it fixes a real bug), but something to think about.  The question is whether the failure in the crypto code should take the whole client offline, or if the client should do the best it can to continue?
