LU-2742 oi: handle IGIF lookup

For the MDT upgraded from 1.8 device, OI scrub will immobilize the
IGIF and insert the mapping for "IGIF <=> ino#" into the OI file.
There are three cases to be processed:

1) After the upgrading, all IGIFs have been processed and inserted
   into OI files. Then any IGIF object lookup should be processed
   as normal FID does, means via OI. If no entry in the OI file,
   then the IGIF object does not exist.

2) During the upgrading, some IGIFs may be already in OI files,
   others may be not yet. Under such case, lookup an IGIF in OI
   files may return -ENOENT, but it does not means the IGIF obj
   does not exist. Since the new immobilize IGIF is the same as
   the original one before backup/restore, OSD will generate
   local identifier from IGIF directly as old 2.x did.

3) The upgrading is paused, and backup/restore. Means upgrading
   from 1.8 may be partly processed, but some clients may hold
   some immobilized IGIFs, and use them to access related objects.
   Under such case, OSD does not know whether an given IGIF has
   been processed or to be processed, and it also cannot generate
   local ino#/gen# directly from the immobilized IGIF because of
   the backup/restore. Then force OSD to lookup the given IGIF
   in OI files, and if no entry, then ask the client to retry
   after upgrading completed. No better choice.

--- Message (Fan Yong, 2013-05-03 17:23:28.000000000) ---
Patch Set 1:

> To verify my understanding of this patch - it sets the "igif_inoi" (?) flag after a full LFSCK has completed, but only until the next reboot? The "igif_inoi" flag is also set in some cases after the initial OI scrub, but it isn't totally clear when that happens - I guess when the initial scrub finds no problems that would require a full scrub (e.g. upgraded filesystem). Having a comment to describe this would have been useful.

The "igif_inoi" will be set under the following conditions:
1) for new created filesystem, or
2) for the device restored after file-level backup, or
3) for the OI scrub completed the device upgrading.

The "set" can happen during current mount, without restart. Once "igif_inoi" is set, then it will not be cleared. Because the IGIF-in-OI is irreversible. 


> It would have been good to store the "igif_inoi" flag permanently (as OBD_FEATURE_COMPAT_NOIGIF for new filesystems or after OI svrub finishes, instead of having to decide this again every time the filesystem is mounted.

We can add OBD_FEATURE_COMPAT_NOIGIF, but it does not help the code much, because for most of the cases, the "igif_inoi" is only set when the device mount, and will not be changed then. It is more convenient/direct for the OSD to check/set "igif_inoi" internally.

On the other hand, both the "last_rcvd" and "oi_scrub" file can be crashed or lost. For "last_rcvd" lost, then "igif_inoi" will be cleared, but for the "oi_scrub" lost, "igif_inoi" will be set. The later one is our expected.
