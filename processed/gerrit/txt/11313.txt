I'm not entirely comfortable with this change.  The purpose of the ioctl changes introduced with DLC is as follows:
There are now 2 types of structures that can be sent via IOCTL.  To handle this properly, I introduced a hdr that has to be the first element of each structure, and then the common code would only deal with these hdrs and then code that needs the specific structure type would cast to the type expected and can use the VERSION vs VERSION2 for verification. 

The original code overloaded libcfs_ioctl_data quiet a bit and has to do all this funky pointer manipulation like:
	if (data->ioc_inllen1)
		data->ioc_inlbuf1 = &data->ioc_bulk[0];

	if (data->ioc_inllen2)
		data->ioc_inlbuf2 = &data->ioc_bulk[0] +
				    cfs_size_round(data->ioc_inllen1);
which is very confusing.

What I wanted was to step away from that with the new structure introduced.  And my changes would allow further structure types to be passed through ioctl as long as they have the common hdr.

This patch seems to revert all these changes.  It overloads also libcfs_ioctl_data which is what I was trying to get away from.  And I don't think the function libcfs_ioctl_getdata() will work for the new structure type as it goes and does the pointer manipulation on the structure, which is unnecessary.

LIBCFS_IOC_DATA_MAX doesn't have any reasoning for the value it has.  In the previous code the sizeof structure was used so it was very clear why the MAX data size meant.

Overall, I think the goal of making the ioctl code generic and not specific to one structure type should be maintained.
