I don't think "%p" will give you what is wanted, since newer kernels will "munge" this pointer when printing it.  At a minimum the patch should be updated to use "%pK" so that the actual pointer is printed.

I guess the expectation here is that printing the blocking lock pointer can be used to find some other message in the debug logs that shows the actual lock detail?
