--- Message (Andreas Dilger, 2018-09-20 15:23:42.000000000) ---
Patch Set 1:

(1 comment)

Does this really solve the problem, or will later OSCs in the list not be handled very often?

--- Message (Alex Zhuravlev, 2018-09-20 16:07:23.000000000) ---
Patch Set 1:

every time it scans all the OSCs registered and finds the closest (in terms of time) OSC, then it will schedule next work for that time. recently processed OSCs won't be processed because they haven't expired yet, so they won't be accounted for rpc_sent.

--- Message (Andreas Dilger, 2018-09-20 17:58:36.000000000) ---
Patch Set 1:

Does it mean that we will check and skip every OSC in the list every time?  What if there are 300 OSCs?  Then the first 100 will be processed, then the second 100 will be processed on the next call, but will the first 100 timeout before we get to the 300th or 1000th or 2000th OSC?

--- Message (Alex Zhuravlev, 2018-10-16 08:07:25.000000000) ---
Patch Set 1:

>  What if there are 300 OSCs?  Then the first 100 will be processed, then the second 100 will be processed on the next call, but will the first 100 timeout before we get to the 300th or 1000th or 2000th OSC?

osc_grant_work_handler() walks through all OSCs and schedules to process the closest one. so if some "far-on-list" OSC should schedule this work as soon as possible, essentially "now". given default 20min shrink interval, I think the case you describe is unlikely.
