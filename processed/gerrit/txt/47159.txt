LU-12514 LNet: get entropy from nid when nid set.

When the 'lustre' module is loaded, it gets a list of
net devices and uses the node ids to  add entropy
to the prng.  This means that the network interfaces need
to be configured before the module is loaded, which prevents
the module from being compiled into a monolithic kernel.

So move this entropy addition to the moment when
the interface is imported to LNet and the node id is first known.

--- Message (Andreas Dilger, 2022-04-28 00:11:29.000000000) ---
Patch Set 1: Code-Review+1

This was mainly an issue with Cray Red Storm (https://en.wikipedia.org/wiki/Red_Storm_(computing)) where 10000 diskless clients (no local RTC, HW entropy source, or /dev/random save file) booted a single-threaded monolithic kernel at the same time (Catamount, not Linux) and there was a reasonable chance that they could have *exactly* the same state in their RNG, and two nodes would generate the *same* 128-bit client UUID (that was really fun to debug).  Essentially a birthday paradox with a very constrained RNG state, so adding in the NIDs (which are always unique) to the RNG would fix that.

Whether this is a realistic problem today is unlikely, but the extra few bits of entropy doesn't hurt either, and might save us again some day in the distant future when there are a billion VMs all mounting Lustre.
