LU-17431 nodemap: allow handling a nidrange for a dynamic nm

Adding or deleting a nid range for a dynamic nodemap, by sending the
appropriate ioctl to the MDS or OSS device.
On kernel side, we need to allow handling a range for a nodemap even
if the nodemap config file is not accessible (i.e. we are not on the
MGS). This dynamic nodemap is not stored on disk.

Patch Set 5:

Would this patch be the right place to check if the dynamic range being added is a proper subset of the static nodemap range?

I think it probably makes sense to allow two different uses.  The first one is a dynamic range that is a strict subset of a static nodemap range:
- a static range covering a whole cluster (e.g. "compute_nodes") with the majority of the settings
- a dynamic range for a subset of nodes that covers nodes for a specific job (e.g. "cn_JOBID")
- the subdirectory of the dynamic nodemap should be the same or a sub-sub-directory of the static nodemap
- ID maps should be subsets of the parent mapping (if any)
- similar for other parameters

Or the dynamic nodemap does *not* overlap with any static nodemap, as a temporary (hopefully easy) way to restrict clients from mounting, forcing them read-only, etc.

What I _don't_ think is a good idea is to allow a dynamic nodemap to overlap with multiple different static nodemaps, as this more likely to be a configuration mistake and the semantics and lookup of such nodemaps would be complex.
