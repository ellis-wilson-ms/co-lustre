This seems sound, but it also occurs to me:
Weighing locks like this seems *insane*.  We'll walk all the pages under a lock - What if it's a read lock covering 20GB of data?

I guess this isn't usually a big time burden (I have never seen it be a problem), but it seems crazy that we check *every page* like this.  Again, no big deal if it's not a performance problem in practice...

Having written that comment...  I wonder about the utility of the page walk based lock weighing at all?  We should absolutely be able to start cancellation of a lock with pages locked or in writeback, since lock cancellations can come asynchronously from the server at any time, and I don't see a huge optimization just from skipping locks that are like it (if they meet the other criteria for elc).

What if we got rid of osc_lock_weight entirely and just stuck to the "don't cancel a lock if we see an i/o using it" check in osc_ldlm_weigh_ast?  Might need a new name since it's not weighing anything
