LU-16800 lustre: Improve Lustre API error reporting

- Separated log API in liblustreapi_log.c and .h-files.
- Introduced rate controlling macros LLAPI_PRINTF_LIMIT,
LLAPI_ERROR_LIMIT, LLAPI_INFO_LIMIT, LLAPI_ERROR_LIMIT_PREFIX.

- Added FILE, FUNC, LINE, PROGNAME, SEVERITY,
TIMEOFDAY, NOPREFIX, NOERRNO, STDOUT, NORATE format options
when print log messages.

--- Message (Andreas Dilger, 2024-09-06 04:14:05.000000000) ---
Patch Set 4:

Changing the library API is something that we cannot do, but I think there is a balance whether existing llapi_error() calls in our own code are changed to use the new rate-limited versions, in places where there is likely to be a high volume of messages. 

It _might_ be possible to use CPP/LD tricks keep the existing llapi_*() function interfaces in the library for binary users, while using a macro to move *new* builds to always use the rate-limited function wrappers, but I haven't investigated that.

Possibly the llapi_error() wrapper macro is in the header under "#ifndef LLAPI_INTERNAL_BUILD" or similar that redirects callers to the new functions, but the liblustreapi_log.c file that builds the existing llapi_error() and llapi_printf() functions has "#define LLAPI_INTERNAL_BUILD" before including that header, so those functions are not replaced?  That might give us the best of both worlds - replace existing API users at build time, and keep ABI compatibility for existing dynamic linkers. 

Yes, the new code should go into a new file. 

Please also add man pages for the new functions in lustre/doc/llapi_*.3 that describe the new interfaces, constants, usage, etc.
