LU-1214 ldlm: splits server specific lock handling from client

Splits/removes these server-side specific lock handlings:
1. Server-side blocking/completion/glimpse AST
2. ldlm_cancel_service
3. Resource policy functions
4. ldlm_handle_enqueue /ldlm_handle_convert etc.

Uses "HAVE_SERVER_SUPPORT" to comment out those server-side codes.

Patch Set 1:

Hi Oleg,

Let me explain why we do this kind of split/cleanup.

We want to push lustre client-side code to upstream kernel, while for some client-server shared modules like prltpc/ldlm/obdclass/fid/fld etc, there are quite a lot of server-side-only code exist. 

TBH, in the first moment, we want to add "HAVE_SERVER_SUPPORT" to protect some server-side code, and when we forking lustre-client code to submit to upstream kernel we can remove/kill those code.
But at the LSF meeting, Peng Tao got the feedback from kernel maintainers that "HAVE_SERVER_SUPPORT" is accepted for kernel as they know that lustre have both client and server side. So based on this feedback, we decided to not remove those code protected by "HAVE_SERVER_SUPPORT" for future kernel submitting.

So I think the "HAVE_SERVER_SUPPORT" is useful for kernel maintainers when they review what we submitted to upstream kernel. It means those pieces of code is only for server and will not be compiled for client, OR it may be confuse why lots of irrespective code for client is submitted to kernel.

And by this method the memory footprint can be reduced for client-side only modules, together with some server-side kernel threads need not to be started on client-side.

It is also possible for us continue doing more fully and cleanly split for client and server-side. It need large scale code restructure and we need to do it incrementally.

Do you think is it make sense?

Thanks.
