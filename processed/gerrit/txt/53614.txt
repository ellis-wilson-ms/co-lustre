LU-17250 mgs: generate a new MDT configuration by copy

The configuration for a new MDT is generated by reading the client
configuration. The MGS filter existing mdc/osc, interpret the
records and then create the corresponding osp/osc device for the MDT.

The main idea of this patch is first to convert and copy the records
from the client configuration to create the new MDT.
And then, copy the remaining record sections from an existing MDT.
So the new MDT can inherit OST pools and parameters from the existing
one.

This avoids complex compatibility checks for IPv4/v6 NID because
add_uuid records are copied without need to parse NIDs.
This also allows to copy "add failnid" section from the client.

This patch extend the usage to "add failnid" section on MDT
configurations.

Here are the steps to copy a existing MDT configuration:

1/ read client configuration and generate osp MDT/OST records for the
   new MDT
1/ find an existing MDT configuration
2/ copy and convert the remaining configuration records from the
   existing MDT configuration (parameters and OST pools)

> I'm wondering if it would be possible to simplify this code further?  I'm not totally sure it needs both the client llog and the MDT llog copy steps?  I haven't looked at this in detail, so it is an open question, but I'm wondering if you have (or could) look into whether it is possible to avoid the client llog copy step?

If we want to register lustre-MDT0003 we can copy the following osp device from the MDT0000:
 - lustre-MDT0001-osp-MDT0000
 - lustre-MDT0002-osp-MDT0000

But there is no lustre-MDT0000-osp-MDT0000 in the MDT0000 configuration, so we have to generate lustre-MDT0000-osp-MDT0003 from the client mdc device configuration.

In any cases, the tests seems to enforce that an OST can be registered before the first MDT (conf-sanity test_21e). To preserve this behavior, we have to get the osp/osc from the client because there is no MDT configuration to copy.

> Mike, please review.  I think this is quite useful/important for adding MDTs to a new filesystem.
> 
> Etienne, is there a similar issue with adding new OSTs to the filesystem, or do they copy the client log already?

This issue should not exist for OSTs, OST configuration are independent from client configuration. And pool are stored on the MDTs side.
The major difference between OST and MDS configuration is that MDS needs to connect to all OST to create/delete/update OST objects. That why MDS needs copy the OST connection configuration from the clients.
That said, we could also inherit some parameters configuration from other existing OSTs. But I am not sure about this: this will work with homogeneous OSTs, but if we have several kinds of OSTs (e.g: SSD, HDD) we do not want inherit parameters from a different kind of OST.

--- Message (Andreas Dilger, 2024-03-11 10:02:21.000000000) ---
Patch Set 9:

> And pool are stored on the MDTs side.

Yes, but there are other configurations that are stored on the OSTs, such as nodemaps or quotas?  If the parameters are stored via "set_param -P" then they are not in the per-device logs, and apply to all matching paramters (which is good).

It would be worthwhile to check whether nodemaps are stored in the "conf_param" log or the "set_param -P" log.  If the "conf_param" (per target) config log, then adding an OST *after* a nodemap or quota was configured for the filesystem may be missing those parameters.

--- Message (Etienne AUJAMES, 2024-03-11 11:13:29.000000000) ---

> It would be worthwhile to check whether nodemaps are stored in the "conf_param" log or the "set_param -P" log.  If the "conf_param" (per target) config log, then adding an OST *after* a nodemap or quota was configured for the filesystem may be missing those parameters.
