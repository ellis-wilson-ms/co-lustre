--- Message (Andreas Dilger, 2023-08-03 04:50:03.000000000) ---
Patch Set 1:

> In the long run, we may group these reparation functions into a
 > struct, and call different functions in dryrun mode which should
 > only print messages about the error to fix.

yes, one of the user complaints about LFSCK is that it doesn't print very useful information about what errors it finds and what it has fixed.   Using "printk=+lfsck" is _close_ to useful, but in some cases is much too verbose.  It would be better to have "lctl lfsck_start" run debug_daemon in the background and log LFSCK messages into a separate log file.
