--- Commit Message ---
LU-13145 lnet: increase transaction timeout

On larger clusters the current transaction timeout default of 10
seconds is too short and it causes RDMA timeouts.

Increase the timeout to 150s. With a retry count of 3 that would
bring the LND timeout to 50s, which was the initial value before
health. The original default has been around for a while and worked
well in most cases

The 150s is the timeout for the ACK/REPLY. So if we don't receive the ACK/REPLY in 150 seconds we'll fail. The LND timeout which is what lustre expects to be 50s, goes back to being 50s, since the retry count is 3 and LND timeout = transaction timeout / retry count

In effect this brings back the original LND 50s timeout. I think this should resolve the timeout issues we've been seeing in larger clusters, while still allowing for the retries. The retries can happen not only because of timeouts, but because of local failures or remote failures. In these cases retrying on different available interfaces is still important.

My question is whether there is a benefit to having LNet retry at this point rather than making Lustre aware of the timeout?  It may be for some RPCs that there is no point to retry after so long (eg. OBD_PING, or in cases where there is a mirror of a file on another OST).  If LNet is retrying up to 150s, but Lustre has given up after 50s or 100s, then we aren't getting ahead with LNet retries. 

I guess the big gap in my understanding here is which RPCs are subject to the lnet_transaction_timeout?  Are there some RPCs which do not have a fast ACK that are needed to keep LNet health happy?  Do we need to make some changes at the PtlRPC level to ensure the "lnet transactions" are completed quickly, even if the RPC service time is long?

Alternately, can LNet resend the request in a timely manner, but not "cancel" the reply buffer, so that the reply from any of the requests is acceptable?  That would improve reliability (ie. 2-3 resends of the request for redundancy, but the full amount of time for the reply), rather than reducing it (reply only has 1/3 of the time to arrive).
