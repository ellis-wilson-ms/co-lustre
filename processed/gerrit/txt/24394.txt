LU-8950 tests: do not use make_custom_file_for_progress()

Do not use make_custom_file_for_progress() in sanity-hsm in tests:
12c, 26, 27b, 28, 31b, 31c, 33, 34, 35, 36, 54, 55, 56, 60, 62, 71,
104, 200, 202, 221, 223b, 225, 251, 252, 407

Before this patch make_custom_file_for_progress() was used to create
big files (5MB to 40 MB). There were several use cases for that:
 - in combination with the --bandwidth option of lhsmtool_posix, it
   allows synchronizing HSM operations and other things;
 - to log enough "progress events", HSM operations need to last long
   enough (the --bandwidth option comes in handy too).

It needed to be removed because:
 - archiving and restoring big files at a 1MB/s rate takes too much
   time (admittedly, that was the point);
 - there are other way to have HSM operations occur concurently. For
   example: suspending copytools to delay request processing.
 - make_custom_file_for_progress() does not correctly reflect what
   the function does and should be used for. Wherever big file are
   still needed, the patch uses create_file() instead.

Removing the need to archive or restore big files at a limited rate
represents quite a speed-up on sanity-hsm.

make_custom_file_for_progress() is used to create "big" files (from 5MB to 40MB). There are several use cases for that:

- Since the copytools are started with the "--bandwidth 1" option, they cannot archive/restore a file at a faster rate than 1MB/s. So big files can be used to test the behaviour of the copytool/coordinator when HSM operations interact. For example: canceling an archive request (before it completes).

- This is also used to log enough "progress events" (from which probably comes the name). This also uses the fact that the copytool is artificially limited in throughput.

- And sometimes it is used for no good reason at all (test 104)

As for why it should be removed:
 - Archiving or restoring a 40MB file with a throughput of 1MB/s takes time... too much time.

 - There are alternatives to big files and the bandwidth option to have to have HSM operations interact (suspending the copytool process is one example, gdb another).

 - Its name is not totally adequate (I think "create_file" is clearer).

Overall, this patch speeds up tests greatly: running all thoses tests (and using loop_period=1) only took 17 seconds (mount time not included). On master this takes around 750seconds (still using loop_period=1).

I will include these informations in the commit message for the next patch. I'll try to make it lighter though =]
