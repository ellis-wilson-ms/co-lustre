--- Commit Message ---
LU-1199 build : Pass in the configure arguments to rpmbuild.

Currently the rpm spec file for lustre and ldiskfs parses
the configure arguments inside the rpm spec file instead
of the more traditional way of passing arguments with
rpmbuild. This patch is the first step to doing this by
parsing the configure options in the Makefile and then
calling rpmbuild in the Makefile with rpmbuild options
which the spec file can act upon. This patch as a first
step converts the kernel source and object options of
configure into rpmbuild options and uses them in the
spec file.

Patch Set 3: I would prefer that you didn't submit this

Commit message needs a rewrite.  The first line short-summary needs to describe what the commit does, not just be the jira ticket's subject.

The body needs to clearly state what is changed, not just "pass those configure options".  The last sentence about kernel modules and mount.lustre location is the main thing, so lead with that.

I question whether we should be supporting something so contrary to the linux Filesystem Hierarchy Standard in the spec file.  The spec file's purpose is specifically to install things in the correct locations for a given platform.

Kernel modules go in /lib/modules on RHEL systems (and pretty much any Linux system).  Should we really support Cray's bizarre installation scheme upstream?

How have they managed to install things using their method in the past?  Why can't they continue to use that method?
