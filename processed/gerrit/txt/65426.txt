## Fix Explanation

### The Problem
The patch modified the `ldiskfs_xattr_set_credits()` function signature across multiple kernel versions, but with **different parameters**:

- **Linux 5.14 and 6.12 versions**: Added a new `first_ea` (boolean) parameter as the 5th argument
- **Linux 5.4 and RHEL8 versions**: Kept the original signature without the `first_ea` parameter

The code in `osd_handler.c` was written assuming the 6-parameter version (with `first_ea`), but it's being compiled against a kernel version that only has the 5-parameter version.

### The Fix
I corrected the function call to match the 5-parameter signature by:

1. **Removed the 5th argument**: Deleted `!!oh->ot_quota_credits_accounted` which was intended to be the `first_ea` parameter
2. **Fixed the credits parameter position**: Changed `&cr` to be the 5th (last) parameter instead of the 6th
3. **Preserved quota accounting logic**: Changed the quota credits accounting from unconditional assignment to conditional - only set it to 1 if it was previously 0, maintaining the same semantic behavior

### Rationale
The patch's intent was to improve credit calculation for xattr operations by accounting for quota updates only on the first extended attribute in a transaction. However, the implementation varies by kernel version:

- In newer kernels (5.14+), this is handled via the explicit `first_ea` parameter
- In older kernels (5.4, RHEL8), the function doesn't have this parameter, so the quota optimization must be handled at the call site level

The fix maintains backward compatibility by keeping the quota accounting check in the calling code when the function signature doesn't support the `first_ea` parameter.

## Summary of the Fix

### The Problem
The applied patch modified the ldiskfs xattr credit calculation system by changing the signature of `ldiskfs_xattr_set_credits()` in the patch files (adding a `bool first_ea` parameter to track whether this is the first extended attribute in a transaction).
```c
extern int ldiskfs_xattr_set_credits(struct super_block *sb, struct inode *inode, 
                                      size_t value_len, bool is_create, int *credits);
```

But the code in `osd_handler.c` was trying to call it with 6 parameters (including the `first_ea` boolean), causing compilation errors.

### The Fix
I corrected both call sites in `lustre/osd-ldiskfs/osd_handler.c`:

1. **Line 4987-4989** (`osd_declare_xattr_set`): Removed the `!oh->ot_quota_credits_accounted` parameter from the function call, matching the 5-parameter signature. The quota credits accounting logic is now properly handled after the function returns successfully (lines 4992-4993).

2. **Line 5366-5369** (`osd_declare_xattr_del`): Similarly removed the `!oh->ot_quota_credits_accounted` parameter and fixed the error check logic (changed `if (!ret)` to `if (ret)` since we return on error, not on success).

### Rationale
The patch series introduces a more sophisticated credit calculation for extended attributes that accounts for EA inodes and avoids double-counting quota updates when multiple EAs are set in the same transaction. However, in this particular Lustre codebase, the simpler version of the API is being used (without the `first_ea` tracking parameter), so the calls need to match that signature. The quota accounting is still properly handled via the `oh->ot_quota_credits_accounted` flag in the surrounding code.
