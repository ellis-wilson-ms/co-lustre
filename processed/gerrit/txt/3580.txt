--- Message (Andreas Dilger, 2012-08-12 16:37:49.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

I like that all of these patches include the kernel commit which includes this change.  Normally when a method like this is renamed, there is a reason for the change, but I don't see any changes in the rest of the code?

Instead of having inline #ifdefs, I would prefer to have a macro in lustre_compat25.h that maps d_make_root() to d_alloc_root() for older kernels. This keeps the code cleaner, and avoids the need to change the code for upstream submission.

--- Message (Liu Xuezhao, 2012-09-05 07:16:14.000000000) ---
Patch Set 1:

"Normally when a method like this is renamed, there is a reason for the change, but I don't see any changes in the rest of the code?"

The difference bewteen d_make_root and d_alloc_root is d_make_root() drops the reference to inode if dentry allocation fails. Kernel commit adc0e91ab142abe93f5b0d7980ada8a7676231fe introduces d_make_root, and several commits after that changes to d_make_root finally commit 32991ab305ace7017c62f8eecbe5eb36dc32e13b removes d_alloc_root.

"Instead of having inline #ifdefs, I would prefer to have a macro in lustre_compat25.h that maps d_make_root() to d_alloc_root() for older kernels."

Will update it, thanks.
