--- Commit Message ---
LU-15285 mdt: fix same-dir racing rename deadlock

With LU-12125 lifting the BFL for same directory rename,
a deadlock possibility opens up since we lock source and target
of rename in the source-target order, if there are two renames
racing to rename arguments in reverse order:
mv a b &
mv b a

a lock inversion happens and a deadlock has been observed.

To avert this - instill additional order requirement:
lower PDO hash value is to be locked ahead of the higher one.

> do we really need a synthetic test for this if racer is able to
 > reproduce it so consistently though?

My main concern would be that racer fails for a lot of different reasons, and having a specific test for the rename lock ordering would help isolate this problem in the future if it would be hit again.

There was previously a "lockorder" test that I wrote many years ago to exercise a rename deadlock when the MDS always locked in inode order instead of parent/child order, but it may be a good starting point.  Alternately, I think it is possible to have a test case running racer tuned so that it *only* does renames, and limit the file count to be sure to hit it?
