Rather than disabling IDIF in this patch, it would be better if it only fixed the tests and client code (which we can easily backport to other branches), then disabled IDIF in a separate patch.

It would be great if this could be done with something like:

    lctl set_param seq.cli-cli-testfs-OST0000-osc-MDT0000.space="[0x1ffffffff-0x1ffffffff]"

but I don't think those parameters are wired up properly internally:

 seq.cli-cli-testfs-OST0000-osc-MDT0000.server=testfs-OST0000_UUID

 seq.cli-cli-testfs-OST0000-osc-MDT0001.space==[0x0 - 0x0]:0:mdt
 seq.cli-cli-testfs-OST0000-osc-MDT0001.width==4294967295
 seq.cli-cli-testfs-OST0001-osc-MDT0000.fid=[0x0:0x0:0x0]
 seq.cli-cli-testfs-OST0001-osc-MDT0000.server=testfs-OST0001_UUID

 seq.cli-cli-testfs-OST0001-osc-MDT0000.width==4294967295
 seq.cli-cli-testfs-OST0001-osc-MDT0001.fid=[0x0:0x0:0x0]
 seq.cli-cli-testfs-OST0001-osc-MDT0001.server=testfs-OST0001_UUID

 seq.cli-cli-testfs-OST0001-osc-MDT0001.width==4294967295

        do_facet ost1 $LCTL set_param fail_loc=0x00001e0

right at the start of testing?

I've been looking at the feedback,
the "space" is more like for the current seq space we can use on osp,
what we want is controlling the number of objects in one seq, so this won't help us.

The fail_loc=0x00001e0(OBD_FAIL_OFD_SET_OID) makes ofd_seq_load() report the seq is about to used up.
We can do something like this for IDIF in ofd_seq_load(),
but if the fs is already using IDIF and after the update,
osp will try to cleanup orphan by sending last used oid, on ofd we will see a large difference if we treat IDIF last oid as full.

I think it's better to keep get the rollover done from osp side, like what we are doing currently.
