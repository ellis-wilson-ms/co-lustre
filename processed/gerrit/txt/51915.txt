LU-16967 build: Add in-kernel-ko2iblnd driver

Add in-kernel-ko2iblnd.ko for users of in-kernel OFED
and only build ko2iblnd.ko if an external OFED is available

This allows for building and packaging both an external
(MOFED or OPA) o2ib driver and an in-kernel o2ib driver.

Packaging rules will be written so that only enable one
of the o2iblnd drivers can be installed.

In the case of the in-kernel-ko2iblnd.ko driver a symlink
named ko2iblnd.ko will be created to point to the in-kernel
based o2ib driver which allows for a reasonable migration path
for the majority of users.

It is useful for dist build and test to be able to build
both in-kernel IB and external OFED in the same build.

This also means there would be some install/configure
adjustments that ought to have some discussion.

Shaun,
I think allowing both drivers to be built at the same time is a huge win, so I'm definitely in support of this patch, but there are significant compatibility issues if this is changed suddenly.

It isn't only a matter of the module name changing, but also because @o2ib NIDs will no longer work for MOFED-built LNDs, since they will be assigned a different LNet network number.  Unless I'm missing some subtlety of the patch, that would break all interoperability, routing, etc. in a cluster if it isn't upgraded all at once, and all of the configurations changed to use MOFED drivers.

I would think that since the MOFED driver is by far the most common one used (at least all of the sites I'm aware of use MOFED and not in-kernel OFED), it would make sense to keep that one as ko2iblnd, and give the in-kernel driver a new name?

It probably makes sense to add support to the tools, etc. to handle a new driver in one patch, but keep the current behavior of naming either driver as ko2iblnd and using O2IBLND for the network number for a few releases, and only have separate names when both are built?  That would make for a smoother transition to the new driver name and network number.

--- Message (Andreas Dilger, 2023-08-18 21:25:24.000000000) ---

Is there some way that both OFED and MOFED could handle both the traditional O2IBLND network number as well as allowing MOFED (or OFED) to handle the new network number?

That would mean in the 99.9% of cases where only one LND is loaded that it would maintain interoperability, and only in case *both* LNDs are loaded would it need to use the new network number and @mlnx (or @kib or whatever) NIDs to route specifically to the new LND.  Allowing one driver to handle both network numbers would also simplify incremental upgrades to the new NID naming/numbering.
