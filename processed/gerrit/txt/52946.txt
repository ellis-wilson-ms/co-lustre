LU-17242 debug: CDEBUG performance testing

To ensure that performance doesn't degrade from debugging
changes, we must test performance. We do this by timing a
kernel module.

In the future, this test may be converted to use the kunit
framework. But that change is far too large for this patch.
This test was not added to the test package. I don't expect
this test to be generally useful - it serves a very narrow
goal.

Downside: the precompiled module kunit.ko doesn't seem to be available on SLES and Ubuntu. We'd need a compatibility module to run this test on those platforms.

--- Message (Andreas Dilger, 2023-11-06 22:14:02.000000000) ---
Patch Set 4:

(6 comments)

IMHO, it doesn't really look like we are getting much benefit from the kunit framework compared to e.g. just loading a module that runs the test as part of kinit() and then prints the results to the console with pr_warn() and exits with an error so the module is unloaded (as kinode.c does)?

The drawback of kunit is that it doesn't work on some distros.  That isn't an issue with this patch, since we don't expect CDEBUG() to be architecture specific, but it does mean we can't use it for more generic testing that *is* dependent on kernel version (e.g. kinode.ko, obd_test.ko, llog_test.ko).

--- Message (Andreas Dilger, 2023-11-06 22:20:59.000000000) ---

Unfortunately, there is absolutely no information printed by the module or the kunit test environment that might indicate what happened, so that is another vote against kunit and in favor of simple "load module, use printk/pr_warn messages, check dmesg for result".

--- Message (Timothy Day, 2023-11-07 01:13:22.000000000) ---
Patch Set 4:

(3 comments)

From a purely functional perspective, kunit doesn't have much advantage over "load module, use printk/pr_warn messages, check dmesg for result". It's easier to write (IMHO).

The real advantage is that all of the kunit tests we write can be easily upstream-ed alongside the client. Then when people port Lustre to new kernel APIs - they'll be less likely to break things horribly. From my understanding, when the Lustre client was in staging it suffered from these kinds of regressions often.

I think I'll be able to get kunit working on the other distros. It's only ~2-3k line module.
