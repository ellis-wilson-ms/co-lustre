--- Commit Message ---
LU-8910 osp: Add correct handling of errors to osp_statfs_interpret

MDT's statfs info could be disagreed with OST's info for a very long time.
If osp_statfs_update() is called and extends the timeout 1000*obd_timeout
into the future but then osp_statfs_interpret() hits an error it
will never reset the timeout.
Now when osp_update_statfs request fails osp_statfs_interpret causes
osp_precreate_cleanup_orphans to send new one after 10 seconds.

--- Message (Andreas Dilger, 2016-12-08 01:05:17.000000000) ---
Patch Set 1: Code-Review-1

Is there a test case for this?  LU-8910 and the commit comment doesn't really describe how this might be hit, but I'd guess if osp_statfs_update() is called and extends the timeout 1000*obd_timeout into the future but then osp_statfs_interpret() hits an error it will never reset the timeout.  That should be added to the commit comment so it is clear to understand where the problem is.

I don't think we want the MDS sending statfs RPCs as fast as it possibly can, in case there is some problem (e.g. network interface down), so it probably makes sense to set a "reasonable" timeout (e.g. tens of seconds in the future) so that the MDS does not loop trying to get statfs information when there is a persistent problem.

--- Message (Andreas Dilger, 2016-12-08 09:28:07.000000000) ---
Patch Set 1:

I don't think adding a delay to this resending is very hard.  Rather than setting timeout to 1s in the past, set it to 10s or obd_timeout in the future, or similar:

        d->opd_statfs_fresh_till = cfs_time_shift(obd_timeout);

That will avoid a stream of RPCs being sent, and give the OSS (or possibly MDS) time to recover from whatever problem was being hit.
