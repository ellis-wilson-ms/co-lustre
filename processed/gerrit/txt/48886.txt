Patch Set 2: Code-Review+1

This seems to have the side effect of removing all of the llog operations from the client version of lctl (by moving the "#ifdef HAVE_SERVER_SUPPORT/#endif"). Makes sense, considering all of the operation look to be server specific.

Question I thought of the other day (out of scope of this change): why have "#ifdef HAVE_SERVER_SUPPORT" in utils? If lctl detected whether or not it's on a server at runtime, it would be easier to output more useful messages. Right now, running server only command on client outputs something like "not a valid command", whereas it could say "this only works on MDS etc".

Also, the server/client utils could use the same binaries. That could make packaging simpler.

--- Message (Andreas Dilger, 2023-03-14 16:49:58.000000000) ---
Patch Set 2:

Tim, I've had the same thought about the utilities myself in the past. This is a discussion where James Simmons from ORNL should be included, since it is mainly on account of the upstream kernel client that the separation is done.

That said, I think it is confusing for users to list commands in the help message, only to report that they are not usable. One option that would be useful for a variety of reasons would be to change the command parser routines to have a "hidden" flag so they do not appear in the help message, but can still be processed. That would be useful for server sub-commands as well as aliases or deprecated commands. Server-only sub commands could be flagged as such so that they don't appear in help and dynamically determine if running on a client or server, but that would require some extra work and does not need to be the first step.

--- Message (James Simmons, 2023-03-22 14:11:37.000000000) ---
Patch Set 2:

The reason for the serve / client is the native Linux client support. Many of the data structures needed for the utilites are not present in the UAPI headers. You can't just include them upstream unless they have use in both the kernel and user land code. That is the rules and if I was to push such code someone else would remove it eventually. It happened in the past.
