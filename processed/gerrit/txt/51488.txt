Patch Set 6: Code-Review-1

(6 comments)

I think there is a misunderstanding here about how EC components will interact with data components, which is driving the idea that a single EC component would cover the whole file and does not need a start/end extent.

My expectation is that there will be a different EC component for each data component of a PFL file, since the geometry will be different in each case.  For PFL files with, say, 1-, 4-, 40-stripe data components, there would need to be matching EC components for each one (mirror or 2x mirror, 4+2, and 5x(8+2) respectively).

I see loi_ost_type being copied around, but I don't think it has any use in this series?  Formerly it was intended to mark EC parity stripes in a "combined" layout, but with a separate PARITY component for those stripes I don't think this field is needed.
