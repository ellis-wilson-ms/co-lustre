LU-7579 osd: move ORPHAN/DEAD flag to OSD

If a directory is unlinked while it is opened,
then it will set flags LUSTRE_DEAD_FL and
LUSTRE_ORPHAN_FL to osd object, so the remote
target can aware this status by OSP attr_get().

And also LUSTRE_ORPHAN_FL will be stored inside
LMA (converted to LMAI_ORPHAN). In the mean time,
it will remove LUSTRE_SLAVE_DEAD_FL flag for dead
stripes of striped directory, i.e. all of
dead directories (stripes) will use LMAI_ORPHAN
in LMA to indicate the status.

Also in osp_xattr, it should retrieve the return
value from the reply, which is the real length of
EA, instead of using reply buffer length.

--- Message (wangdi, 2016-01-03 01:02:43.000000000) ---
Patch Set 3:

"What confuses me about this patch us why the is a new LUSTRE_DEAD_FL being added along with a new field in the obdo at the same time the existing LUSTRE_SLAVE_DEAD_FL is being removed? Why not use LUSTRE_SLAVE_DEAD_FL to pass it around in the existing flags word? It would still be stored in the LMA by the OSD, but avoids the need for new infrastructure."

I probably did not make it clear in the comment. LUSTRE_SLAVE_DEAD_FL is a bit tricky in the original implementation.

1.  LUSTRE_SLAVE_DEAD_FL is only for striped directory, and being recorded in stripeEA, instead of LMA.

2. LUSTRE_SLAVE_DEAD_FL is using get(set)_xattr interface over the stack. mdd_mark_dead_flag()->lod_xattr_set(get)->osd_xattr_get(). But according to our discussion, I thought we prefer this flag through attr_get/set, instead of xattr(), otherwise I will use patch version 1.

The reasons I change it to LUSTRE_DEAD_FL

1. the flag will be stored in LMA as a bit in lma_incompat, so we do not need use the same value (the original LUSTRE_SLAVE_DEAD_FL in stripe EA).

2. LUSTRE_DEAD_FL is only in memory, i.e. osd_attr_get(set) will check the "dead bit" in lma_incompat, then set the flag in lu_attr. So it does not matter if we use the new flag here. plus "LUSTRE_SLAVE_DEAD_FL" is definitely an improper name here.

3.  And also la_flags in lu_attr has only 32 bits, which is equal to the flag in ldiskfs inode.  So there is no room to add another bit in la_flags, unless we reserve a bit for this in ldiskfs inode, which might be a bit tricky. Since we already decided to store this in LMA, so I figured why not just add another flag in lu_attr, and we will check and set it internally in osd_attr_get/set(). And also
I think we can just use LA_FLAGS to tell both la_flags and la_extra_flags, in attr_get, just retrieve la_flags from core inode, and get la_extra_flags from LMA. in attr_set, set la_flags to core inode, and set la_extra_flag to LMA.
