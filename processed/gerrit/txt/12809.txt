--- Commit Message ---
LU-5820 lfsck: use multiple namespace LFSCK trace files

The namespace LFSCK uses trace file to record the FID of the object
that has multiple hard links, or has remote name entry, or contains
some uncertain inconsistency, and so on. Only single namespace LFSCK
trace file may be not efficient, especially when there are millions
of FIDs to be recorded. So use multiple namespace LFSCK trace files
and per trace file based semaphore to control the concurrent access
of the trace file.

For Lustre-2.x (x <= 6), the LFSCK used LFSCK_NAMESPACE_MAGIC_V1 as
the namespace trace file magic. When downgrade to such old release,
the old LFSCK will not recognize the new LFSCK_NAMESPACE_MAGIC_V2 in
the new trace file, then it will reset the whole LFSCK, and will not
cause start failure. The similar case will happen when upgrade from
such old release.

This patch also drops some repeated FID recording in the namespace
LFSCK trace file. Related FID should have been recorded in the trace
file via lfsck_namespace_exec_oit(), it is unnecessary to do that
again when scanning the directory.

Patch Set 5:

> Is there some reason to think that the LFSCK tracing file is a source of performance problems? According to the benchmark results in LU-5820 I didn't think we needed to do anything about the link tracing.
> This patch changed from -20 lines to +150 lines in the latest version, but I don't see any data that supports the added complexity.

Except the test of 10M file set as described in LU-5820 comment, I have made more tests against more larger file set, such as the file set that contains 50M files and 5% multiplie-hardlinked files. Under such case, we need to record 50M * 5% = 2.5M FIDs in single LFSCK tracing file. Compared with the case (10M file set) described in the LU-5820 comment, the FIDs recorded in LFSCK trace file increases 4 times, but the phase2_time increases more than 20 times. So it must be related with the single LFSCK tracing file. On the other hand, caching the linkEA verification history in RAM cannot help much under such case, because a lot of RAM will be consumed by the LFSCK and cause the performance waves very much. (So the performance for caching linkEA verification history in RAM under such case may be untrustable.)

In fact, 50M file set is not very large for a large cluster, and the FIDs to be recorded in the LFSCK trace file not only for multiple-linked files, but also others, such as the object with remote name entry, and the object contains some uncertain inconsistency during the first-stage scanning, and so on.

So I think that the more suitable solution is that splitting the single LFSCK tracing file as multiple ones just like multiple OI files in OSD.

It will be interesting to see if the multiple trace files improved performance. If this resolved the performance issue then it does seem like a simpler solution. 

One of the ideas behind caching the multi-linked inodes in RAM is that they will normally be dropped from RAM when all the links have been scanned, or in cases of memory pressure they could do some out-of-order scanning using LinkEA to finish the scan. Patch Set 5:

> It will be interesting to see if the multiple trace files improved performance. If this resolved the performance issue then it does seem like a simpler solution.
> One of the ideas behind caching the multi-linked inodes in RAM is that they will normally be dropped from RAM when all the links have been scanned, or in cases of memory pressure they could do some out-of-order scanning using LinkEA to finish the scan. That means it normally won't be necessary to keep all of the links in memory at once.

The test results show that the phase2_time for 50M files with 5% multiple hard-links under the solution of multiple trace files is almost the same as the case of single trace file for 10M files with 5% multiple hard-links.

Compared with caching linkEA verification history in-RAM, the other advantages for multiple trace files are:
1) It will NOT assume a lot of RAM;
2) It will NOT be affected by the LFSCK pause/crash, the LFSCK can auto recover from the latest checkpoint;
3) It can avoid the complex iterations between LFSCK and MDD, and other layers about the hard-link changes, object pin/unpin, and so on;
4) Consider LU-5292, we will enhance the LFSCK to record the failed FIDs during the first-stage scanning and retry them before ignoring remote orphan MDT-objects handling. Then there will be more FIDs to be recorded in the LFSCK tracing file(s). Multiple trace files can help more under such cases.
