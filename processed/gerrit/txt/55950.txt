btw a vaguely related question.

We have some inconsistency with lctl snapshot commands.
they commands are lctl snapshot_xxx but the man pages are lctl-snapshot-xxx and inside they are referenced as such too, e.g.:

.TH LCTL-SNAPSHOT_UMOUNT 8 "2017-05-13" Lustre "configuration utilities"
.SH NAME
lctl-snapshot_umount \- unmount Lustre snapshot
.SH SYNOPSIS
.TP
.B lctl snapshot_umount \fR{-F | --fsname fsname} [-h | --help]
              {-n | --name ssname} [-r | --rsh remote_shell]

...

.SH AVAILABILITY
.B lctl-snapshot-umount
is part of the
.BR lustre (7)
filesystem package.
.SH SEE ALSO
.BR lctl (8),
.BR lctl-lcfg (8),
.BR lctl-barrier (8),
.BR lctl-snapshot-create (8),
.BR lctl-snapshot-destroy (8),
.BR lctl-snapshot-modify (8),
.BR lctl-snapshot-list (8),
.BR lctl-snapshot-mount (8)

so shouls all those snapshot- references be changed to snapshot_ and the files themselves renamed?

> so should all those snapshot- references be changed to snapshot_
 > and the files themselves renamed?

Honestly, my preference for "lctl snapshot_FOO" would be to create a new "lctl snapshot" command and have "FOO" be a sub-command of that, and the same for "lctl lfsck_FOO" and similar (of course keeping the old commands for compatibility).

For these ones, having a .so reference with the snapshot_FOO names would probably be best for consistency:

 lctl-snapshot_FOO.8:
 .so man1/lctl-snapshot-FOO.8

We've done that with newer commands like "lfs mirror FOO", but I don't think we knew that was possible for older commands.  That avoids breaking the git history of the existing files, but makes them accessible by either name and will work if a subcommand is added.
