--- Message (Oleg Drokin, 2011-10-31 17:35:30.000000000) ---
Patch Set 2:

I think that while having to wait on all possible changes in all tests is a pretentious goal that will fix all such problems once and for al, it's just too much overkill.

Sure waiting until end of all real bad states (like lu773) is very much needed, but corner cases like stabilizing some other minor things like number of object is something we can do at the start of a caring test since there are just a handful of them, and doing such waiting at the end of every test that does deletes is a huge overkill.

--- Message (Chris Gearing, 2011-10-31 21:10:07.000000000) ---
Patch Set 2: (1 inline comment)

As per the comment if all tests are not going to leave the file system in a predicatable state then any test that requires a particular state and is going to wait or cause that state to happen should at least contain good comments to say what state it is enforcing and why at the beginning of the test.

BTW: Oleg you may it's a pretentious goal you fix problems once and for all but I do not and I also do not consider the goal of having zero issues overkill. Maybe the technique I describe is not the best one but rather than rubbish it perhaps a proposal of a different solution that aims for zero errors would be more productive.

--- Message (Oleg Drokin, 2011-10-31 22:03:39.000000000) ---
Patch Set 2:

Oh, don't get me wrong, I am not against having zero issues.

What I think is overkill is tracking every single minuscule bit of state in every test and make sure we enter and exit subtests in the same state.
This is doable and I suspect would fix all the "Stray test change". But there are a lot of the state that might change and that we would need to wait for, and a lot of those are usually irrelevant.

Going through all the tests and seeing what state is important to be stable on the other hand might be easier, but even that is probably way too much effort.
