LU-18357 ptlrpc: Support add and del alternate filesets

This patch adds the API and functionality to add and delete alternate
filesets to a nodemap. This is implemented by using the previously
added fileset rb tree to store alternate filesets in the in-memory
nodemap struct. Both, the rb tree and nodemap IAM is synchronized.
Note, the primary fileset is handled outside the rb tree.

Consistency between both data structures is guaranteed as follows:
When adding a fileset, it is first inserted into the rb tree,
generating a free fileset ID which is then used to insert the fileset
to the nodemap IAM at the subid calculated from the fileset ID. On
reading the IAM index pages (on nodemap synchronization or restart),
the fileset ID is derived from the subid and used on inserting the
fileset into the rb tree. Therefore, free fileset IDs can be managed
by the rb tree.

The following commands are added/modified in this patch:

- (new) lctl nodemap_fileset_add [--alt] --nodemap .. --fileset '..'
- (new) lctl nodemap_fileset_del --nodemap .. --fileset '..'
- (modified) 'lctl get_param nodemap.<nm_name>.fileset' now prints
  both fileset types. Tests sanity-sec 27a and 27b were modified
  accordingly.
- (modified) 'lctl nodemap_set_fileset' is deprecated and now prints a
  corresponding warning on use.

This patch adds sanity-sec 27c which tests multiple fileset
persistence, varying fileset lengths, and the limit of 255 alternate
filesets.

Note, alternate filesets are not used yet when mounting a client.
Implementing the mounting rules based on both fileset types is done in
the next patch, including the testing of multiple filesets on client
mount.

--- Message (Marc Vef, 2024-10-15 10:08:26.000000000) ---
Patch Set 1:

To add to the previous comments, I include a summary below of some offline discussions between Sébastien and me on how setting and getting filesets could behave beyond what is proposed in this patch.

In the master branch implementation, setting a nodemap saves user/llog input into the nodemap struct as a single char*, and it is overwritten on each invocation. `nodemap_set_fileset()` was modified in this patch to still receive a single user input string that is separated by a `:` character. Since it is not practical with multiple filesets to store them as a single char*, user input is sanitized where each fileset represents an entry in a new fileset list. `nodemap_get_fileset()` returns the string by serializing the list back to how the user has set it.

The silent assumption is that each `nodemap_set_fileset()` still overwrites all previously-set filesets. An alternative to this is to append the current fileset configuration, such that `lctl set_param nodemap.<nodemap_name>.<fileset_name> ...` or `lctl nodemap_set_fileset` do not overwrite but add a single entry at a time without allowing separation via `:`.  Therefore, the fileset must be cleared first manually to remove an entry.

While it seems possible for both commands and the llog to support this, it would imply a change in what admins are used to. They could still expect each fileset `set` to overwrite all previous entries. If unnoticed by an admin that it now only adds a new entry, this could also be an issue from a security point of view. Further, it is inconsistent with how other nodemap commands work, as the default is overwriting.

Because of these reasons, the current consensus is that it would still be preferable to overwrite all filesets on each `set` as proposed by this patch.

6. Add "lctl nodemap_fileset_clear" fileset operation to wipe all filesets.
7. Add header processing for filesets in the nodemap IAM for future fileset flags.
8. Add the read-only flag for filesets.
9. Extend the "lctl nodemap_info" to show all nodemap values and not only property names.
