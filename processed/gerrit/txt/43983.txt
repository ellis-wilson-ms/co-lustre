LU-14736 utils: update leak-finder.pl for new format

Update leak-finder.pl to handle some of the newer log formats,
so that it produces more useful results. The changes include:
- add option "--by_func" to sort leak logs by function name in
  ascending order;
- add option "--summary" to print a summary report by the total
  number of leak bytes of each function in ascending order in
  YAML format,
  "- { func: <function_name>, alloc_bytes: <bytes>,
       leak_count: <count>, leak_bytes: <bytes> },"
- define LIBCFS_MEM_MSG() to print alloc/free log in a uniform
  format, as follows, also define LIBCFS_ALLOC_POST() and
  LIBCFS_FREE_PRE():
  mask:subs:cpu:epoch second.usec:?:pid:?:
  (filename:line:function_name()) alloc-type 'var_name': size at
  memory_address
- correct some alloc/free debug messages in lnet part.

Test-Parameters: trivial

Join the club.  I realized that the format of these messages is a mess in lnet (several hand-made CDEBUG() messages with a different format) are in use, and missing in other places.  It would be good (in a separate patch) to clean those up to use e.g.:

        LIBCFS_ALLOC_POST(me, sizeof(*me));

for the slab allocations (sizeof is evaluated at compile time).

Separately, I see that LIBCFS_ALLOC() and LIBCFS_FREE() are using an atomic64 counter to track the total allocations inside libcfs_kmem_inc(), which adds a non-zero overhead to this critical codepath.  It would be better to turn this into a percpu counter to avoid that, though in a separate patch.

In OBD_ALLOC_POST() and OBD_ALLOC_FREE() it is using lprocfs_counter(), which is at least a per-CPU counter, but is higher overhead than needed (we don't need min/max/stddev for allocation sizes), and it should also be moved to a percpu counter.

On a third note, a totally idle 2-MDT 4-OST 2.12.6 local client mount in my 3GB VM has allocated a whopping 600MB of memory (according to "lctl get_param *memused"), so 20%, even though the client has no cached pages or DLM locks.  That seems piggish and could probably use some better scaling of allocated resources.  On a couple of other systems (2.12/2.14) it is also about 280MB for Lustre, and 180MB for LNet on 2.12, 55MB for LNet 2.14 (I suspect it is not accounting for a number of allocations anymore due to "cleanups").

Emoly, thank you for updating this patch and getting leak_finder to work again.

Rather than hard-coding the message format in multiple places (which may break again in the future) it would be better to use only two macros to print these messages (LIBCFS_ALLOC_POST(), LIBCFS_FREE_PRE()) and then call these from wrappers (LIBCFS_ALLOC_PTR(), LIBCFS_ALLOC_SLAB(), LIBCFS_ALLOC_GFP(), ...).

Note that I don't think LIBCFS_ALLOC_POST() needs to print the verbose  "out of memory" message, since this would be handled by the kernel today.  That may also allow OBD_ALLOC_POST() to be replaced by LIBCFS_ALLOC_POST(), to further reduce duplication.
