LU-16538 llite: use lock flags to distingush OPEN ibits lock

In the original Lustre design, an OPEN ibits lock returned to a
client is mandatorily bound to a specific lock mode:
- FMODE_WRITE	CW
- FMODE_EXEC	PR
- other		CR

For DoM files, it would better to return DOM bits combined with
OPEN bits (DOM|OPEN) in an ibits lock to a client during open().
It can reduce the lock traffic for the possible subsquent read
(PR lock mode) or write (PW lock mode).
Thus, three lock flags are defined to distingush the OPEN ibits
lock with different open modes:
  open_mode	lock_flag		lock_mode
- FMODE_WRITE	LDLM_FL_OPEN_WRITE	CW or PW
- FMODE_EXEC	LDLM_FL_OPEN_EXEC	PR
- other		LDLM_FL_OPEN_READ	CR or PR

These new lock flags are for internal use on the client only and
declared in "lustre/include/lustre_dlm_flags" and store in
@ldlm_lock->l_flags.
Thus, this is a client-side patch.

--- Message (Mikhail Pershin, 2023-05-30 11:14:41.000000000) ---
Patch Set 2:

Are we sure about combining DOM|OPEN lock bits? That can give performance boost for first open, but on other side OPEN lock becomes exclusive, i.e. there will be only one client having OPEN lock on particular file. I don't think that is original intention of open lock feature as it uses CW mode but not PW

--- Message (Qian Yingjin, 2023-05-30 12:47:56.000000000) ---
Patch Set 2:

We don't grant DOM|OPEN ibits lock for every open, only grant when the client explicitly requests to obtain this kind of combined ibits locks with OPEN flags such MDS_OPEN_WILLWRITE which indicates the the file will be written soon.

The use cases include the customized mpiFileUtils/dcp or cp. We know the src file is small, and when do batch copying, we can open|create-ahead the file lists together with DOM bit, thus the subsequent open| create can be directly performed on the client locally. And the data can be written into page cache under the protection of the already obtained DOM lock.
