--- Message (Patrick Farrell, 2016-09-29 20:40:21.000000000) ---
Patch Set 1:

Frank, are you certain these two "do the same thing" here?  __l_wait_event is a really complex (and horrible) macro.

At the least, the two have different behavior with regards to signal handling.  __l_wait_event blocks all signals when it has a timeout, wait_event_timeout does not.  In this case, I think both wait in TASK_UNINTERRUPTIBLE, so the blocking of all signals should be irrelevant...

I admit, I can't find a specific difference between using the two macros in this instance, but they certainly aren't equivalent in the general case.  The LWI passed to l_wait_event allows for a really broad range of behavior.

Given that, are you sure they're equivalent here?

Sadly, Andreas, nothing has changed there.  From SLES12:
#define task_contributes_to_load(task)  \

                                 (task->flags & PF_FROZEN) == 0)

In this use case, though, if I'm reading the macro right, we're not sleeping in TASK_INTERRUPTIBLE anyway.  We're actually sleeping in TASK_UNINTERRUPTIBLE, so in this usage, we're contributing to load.
