timeout to the end of recovery:

(1 recovered, 11 in progress, and 1 evicted) to recover in 71578:57

jiffies converted to seconds.

That is done on purpose, to catch driver bugs early on instead of only after 49 days.  However, if Lustre isn't mounted in the first 5 minutes after boot, or doesn't hit this code at the right time it may not hit that particular bug.

--- Message (Andreas Dilger, 2018-12-15 18:57:08.000000000) ---
Patch Set 1:

> The complexity here is that the code handles the real time clock
 > in seconds, obd_recovery_start for exmaple, timeouts in seconds
 > and then work with jiffies due to the timer.

I did a look through the code, and I don't see where obd_recovery_start interacts directly with obd_recovery_timer?  It looks like obd_recovery_start is mostly used for printing messages to the console/lprocfs.  I'm fine with _just_ fixing the use of obd_recovery_timer in this patch to use jiffies, and leave other cleanups to other patches.

Other places I saw that need some help:
- obd_t10_performance_test() and cfs_crypto_performance_test() are using msecs_to_jiffies(MSEC_PER_SEC / 4), but msecs_to_jiffies() converts to a static inline function that directly uses HZ for kernel > 4.1.  These should be using __msecs_to_jiffies() if available or add a __msecs_to_jiffies() compat macro for older kernels.
- LNetCtl(), lnet_ping(), lnet_sock_write(), lnet_sock_read(), target_recovery_overseer(), __ldlm_namespace_free(), __lfsck_set_speed(), lfsck_start_all(), ll_kill_super(), ll_statahead_thread(), ... are also using msecs_to_jiffies() and should be fixed as above.
- brw_stats_show() is using jiffies_to_msecs(1000) / MSEC_PER_SEC, but that is wrong for multiple reasons.  I already have a patch for this.
