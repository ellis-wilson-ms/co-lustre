--- Message (Frank Zago, 2015-01-16 22:13:16.000000000) ---
Patch Set 2: Code-Review-1

The only symbols needed by liblustreapi from libcfs.a are strlcpy and strlcat. Integrating libcfs.a into liblustreapi cause 2 new issues:


* strlcpy and strlcat are now provided by linking to liblustreapi, which can create a conflict with programs also linking with libbsd, which is the regular provider of these two functions.


* every program now also has to link against libreadline, even if they don't need it. Additionally, libreadline being a GPL library, other program will have to either convert to GPL, or create a fake libreadline library. In either this is not desirable.

--- Message (James Simmons, 2015-01-16 22:55:24.000000000) ---
Patch Set 2:

I agree. I'm rethinking what I'm doing. The problem with readline is due to parser from libcfs. I should break that out into libcfs.a and keep the reset in libcfsutil.a instead. Eventually the parser will go away. True you could get a conflict due to libbsd but the autoconf system should detect strlcpy and not build those functions instead. Its one of reason I merged into libcfsutil.a since if strlcpy is present then libcfs.a ended up empty. Now liblustreapi does need the strlcpy functions which is why I directly linked it.
