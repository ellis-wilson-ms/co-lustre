(2 comments)

I'm beginning to think that this is an issue with the code and not with the test case.  The filesystem starts with the OSTs all at 1% used, so it isn't caused by QOS space balance.

The precreate counts per OST are uneven before the start of the test.  The total create counts per OST show that OST0002 and OST0004 are not precreating enough objects with MDT0003:

 - precreated_ost_obj_count lustre-OST0002-osc-MDT0003 prealloc_last_id: 16384 prealloc_next_id: 16326 count: 59
 - precreated_ost_obj_count lustre-OST0004-osc-MDT0003 prealloc_last_id: 16384 prealloc_next_id: 16266 count: 119

The balance of created objects per OST index shows that OST0002 and OST0004 were the ones reporting fewer object creations, about 400 fewer (25%) which would be the case if MDT0003 didn't create *any* objects on those OSTs:

   1622 0
   1623 1
   1212 2
   1622 3
   1270 4
   1621 5
   1620 6
   1620 7

One possibility that comes to mind is that the "force precreate" requests from all MDTs to all OSTs are overloading the ost_create threads, and the precreate is "short" because it took too long to finish.  In that case, re-running "lctl set_param osp.*.create_count=$create_count" on only those MDT-OST connections as this patch is adding _should_ fix the test issue.

It also seems possible that there is actually something wrong with the OST object precreation (e.g. the OSP connection is not working), and this test case is actually exposing a real problem?
