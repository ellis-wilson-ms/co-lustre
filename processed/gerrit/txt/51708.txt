Patch Set 18:

Neil this patch addresses a few shortcoming of parent FID handling.

Currently the fid handle for lustre supports both the file FID and parent FID but in reality the parent FID is never returned :-(

Second is we have an ioctl LL_IOC_GETPARENT that retrieves the parent FID. Well it doesn't return the true parent but the linkEA parent FID.
I believe the linkEA is for broken files under lost+found. So the linkno returns a specific linkEA parent. Currently if you call this ioctl for a normal file you don't get any parent FID since normally you don't have linkEAs for a file unless its broken.

This patch was attempting to use the fhandle API to do both above. That is why I was passing in the linkno into the fhandle. If that can't be done then we really can't remove the ioctl. I hope this makes it clear.

--- Message (Andreas Dilger, 2024-02-10 18:58:38.000000000) ---

Al regular files should have a "link" xattr that is storing one or more reverse links for the parent directory FID and the filename in that directory. I don't think this is a good idea.
This means that when a file is moved from one directory to another the handle will change.  That will confuse code that reasonably expects it not to.

The only reason we have the "parent" thing at all is for the "subtree_check" nfsd export option which I was never quite brave enough the deprecate.
As you note, it isn't accessible from name_to_handle_at() and is discouraged from nfsd.

I really think that if you want a way to get the parent FID, that should be a lustre-specific ioctl.  Don't try to overload the handle code - it isn't a good fit.
