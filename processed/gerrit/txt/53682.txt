Decrease the default value of lru_max_age from 3900s to 600s.
Otherwise, clients can accumulate too many old locks.

Poking around the code, the LRU in Lustre seems pretty simplistic. It seems like the core logic is encapsulated in ldlm_prepare_lru_list()? Patch Set 8:

While client side lock management could be improved, the real problem with lock management isn't the client side code - it's the monstrosity that is the slv code.  It buggily applies an overly complicated model of "lock volume" which has (imo) nothing to do with how locks work in Lustre.  But hey, at least it's almost incomprehensible.

Apologies for the tone - a client side LRU rework is probably worth the trouble.  While it would be more ambitious, a simplification of the server side LRU might be even more valuable.  I'd have to think a bit about what's most desirable, but I think it's probably letting clients have as many locks as they want until lru_size (on the server) is hit, or it runs low on memory.  When it does, do something simple.

So the SLV code (if that's what we're talking about) is not a performance concern.  LRU in general isn't a performance concern as long as it's vaguely reasonable (there are exceptions, but you get yourself most of the benefit for most cases with a simple reasonable-ish LRU).  The issue with SLV is that it has been persistently buggy, causing periodic issues where all locks are dropped and/or a particular client is told it can't have any locks.  Variations of these have occurred periodically since I started working on Lustre and they're still around, though rare.

In practice, SLV approximates the behavior of a much simpler thing which reacts purely to memory pressure - it just does so in a buggy and more difficult to debug fashion and without (I think) any advantage vs that simpler behavior.

So no performance concern with SLV.  For the regular LRU, I actually also don't really have a suggestion - I think the issue of frequently used locks getting pushed out is going to be hard to find a real workload for where the impact is measurable, since getting a new lock isn't too hard.  I actually think unless we can find a workload, improving it might not be the best use of time. :)

Based on your description, SLV rework seems reasonable. I'll have to dive into the code some more.

For the client, I think we'd want to retain as few unused locks as possible without degrading performance - to reduce the risk of things going haywire during recovery, network interruptions, client death, etc. I'd imagine setting lru_max_age and lru_size super low would kill performance at some point. And intuitively, implementing ARC or LFRU would counteract that. Locks that are actually used would get retained with priority.

The server should give as many locks as the client wants. But the client should be more conservative with lock caching without sacrificing its performance. The current approach seems more prone to unavailability events.
