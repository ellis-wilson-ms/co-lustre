--- Commit Message ---
LU-4367 close: combine close in unlink RPC

With open lock taken, close will cache file handle instead of
release it on MDS, but if unlink is followed by close, a real
close will be triggered before unlink, which will hurt unlink
performance.

This patch combines close into unlink RPC when SOM is disabled,
because current SOM implementation has flaws, and supporting it
will cause unexpected complexity. In the future if SOM is mainly
implemented on server side, this feature should coexist with it.

* client packs file handle in unlink reint record.
* MDT close file handle before unlinking.

It's a pretty standard procedure to piggyback additional operation in a close RPC, please take a look ll_close_inode_openhandle() -> mdc_close() and:

        if (op_data->op_bias & MDS_HSM_RELEASE) {
                req_fmt = &RQF_MDS_RELEASE_CLOSE;

In my opinion, it's not necessary to invent another mechanism for this and we can just extend it to have a unlink-on-close RPC.

It'll be not necessary to have connection flag (which we tend to invent new flags arbitrarily); if the server executed the unlink at close, it will set a flag in the reply RPC, so if it turned out the server is an old one, one extra unlink RPC will be issued followed by.

If you really like to adopt my suggestion, please keep in mind we already have release-on-close, and we're going to have swap-on-close and here we're having unlink-on-close extension here. It will be great if we can have same RPC format for all of them.

Please let me know if I understood the problem wrong.
