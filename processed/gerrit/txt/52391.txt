Unaligned DIO needs to directly access the page contents in
order to copy to the buffer.  This means it can't work with
RDMA only (non-CPU accessible) pages.

Implement that limitation.

Shadow, I'd love to see an implementation of your suggestion.

As I understand it, the server can't handle when the client to server page mapping isn't 1 to 1 - ie, when one client page can map across multiple server pages.  For example, consider if byte 100 in a client page is byte 0 in a server page.  How does that work?  When is the alignment issue handled?

Today, the alignment is the same from the source of the transfer on the client to the LNB on the server.  How would the server handle it if byte 100 in a page was byte 0 on disk?  That is what "unaligned" means in practice - things like that.  ldiskfs isn't log structured, it's block/extent based, so byte 0 in a 4K disk block must be byte 0 in a page on the client (I am ignoring large pages to make this conversation easier).

If we do RDMA from memory which is not aligned on the client, are we having the network do the aligning, so it handles the mapping when client-server don't have an aligned page mapping?  In my testing, this was slower than doing it using the CPU.  Or if not, where is the alignment achieved?

Huh!  That's really interesting.

So that's more of an Nvidia question, honestly - My understanding of this is that RDMA only pages (how we refer to GPU direct pages) are not mapped in to the CPU address space, so it's impossible for the CPU to access them, generating the crash Ihara reported recently.

Ihara I don't have the ticket where you reported the crash handy, but was that on x86-64 or ARM?

So it may be that on ARM they're CPU addressable?  That means we'd have to distinguish between the two architectures/types of pages, and "RDMA only" would no longer be an appropriate name.  If they're really CPU addressable, then we could in theory remove the restrictions and do:
- checksums
- encryption
- compression
- udio

The only question is do we really want to do all of those, since this might be defeating the point of GPU direct by pulling the data over the bus to the CPU?  Like, if we checksum all the IO, we might totally invalidate GPU direct.

Same issue for UDIO, though I recall Nvidia really wanted to do sub 64K DIO and maybe the bus travel hit is acceptable for smaller stuff.

So yeah - if the pages are CPU addressable on ARM, this would work there.  So is that the case and where was the crash seen?
