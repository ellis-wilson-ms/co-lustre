LU-11047 mdt: standardize mdt object locking

* remote object LOOKUP lock should be taken from MDT where its parent
  is located, but current mdt object locking doesn't handle this
  explicitly, instead it implies its parent is on current MDT. In
  getattr/setattr/open/unlink/rename/migrate, LOOKUP lock will be
  taken, it should check whether object is remote object, and lock
  on the correct MDT.
* mdt_object_lock_internal() shouldn't check PDO lock, instead do it
  explicitely outside in parent lock to make it more clear.
* with the above changes, mdt_object_lock_internal() becomes a simple
  wrapper: it calls mdt_remote_object_lock_try() if object is remote,
  and mdt_fid_lock() otherwise. After this, XATTR|LAYOUT|PERM locks
  can be taken remotely.
* add mdt_thread_info.mti_dist_txn to mark current operation is a
  distributed transaction, and LDLM locking functions will enforce
  COS_INCOMPAT flag if it's set.
* the MDT object locking functions are as below:
	. mdt_object_lock(): lock object, this is used in most places,
		and normally lock ibits doesn't contain LOOKUP, unless
		the caller knows it's not remote object.
	. mdt_object_check_lock(): lock object with LOOKUP and other
		ibits, it needs to check whether parent is on remote
		MDT, if so, take LOOKUP on parent MDT separately, and
		then lock other ibits on child object.
	. mdt_parent_lock(): take parent UPDATE lock with specific
		mode, if parent is local, take PDO lock by name hash,
		otherwise take regular lock.
	. mdt_object_stripes_lock(): lock object which should be
		local, and if it's a striped directory, lock its
		stripes, this is called in operations which modify
		both object and stripes.
	. mdt_object_lock_try(): lock object with trybits, the trybits
		contains optional inode lock bits that can be granted.
		This is called by getattr/open to fetch more inode
		lock bits to client, and is also called by dir
		migration to lock link parent in non-block mode to
		avoid deadlock.
	. rename/migrate source object are locked in specific
		functions, because both source parent object and
		source object may be located on remote MDTs.
