LU-18725 utils: create ll_decode_lov

Like ll_decode_linkea, create ll_decode_lov.
This allows us to directly scan the MDT for information that can
be obtained with lfs getstripe.

The LOV printing functions are moved from liblustreapi.c to a new
file liblustreapi_lovea.c. This includes both raw printing functions
(for xattr buffer decoding) and wrapper functions (with path lookups).

## Summary of the Fix

The compile errors were caused by duplicate and conflicting definitions of `lov_dump_hsm_lmm`:

### Root Cause:
1. The patch moved `lov_dump_hsm_lmm` from `liblustreapi.c` to `liblustreapi_lovea.c` with a new signature (removed `path` and `depth` parameters)
2. However, the old version with the old signature was left in `liblustreapi.c`
3. This created a conflict between the declaration in `lustreapi_internal.h` (new signature) and the definition in `liblustreapi.c` (old signature)
4. Additionally, `hsm_flags2str` was moved to `liblustreapi_lovea.c` but the old version in `liblustreapi.c` was still trying to call it

### The Fix:
**Removed the old `lov_dump_hsm_lmm` function from `liblustreapi.c`** (lines 2580-2620)

This resolves both errors:
- **Conflicting types error**: Now there's only one definition of `lov_dump_hsm_lmm` (in `liblustreapi_lovea.c`) that matches the declaration in `lustreapi_internal.h`
- **Implicit declaration of `hsm_flags2str`**: The old code that called `hsm_flags2str` has been removed. The new version in `liblustreapi_lovea.c` has `hsm_flags2str` defined in the same file as a static function

### Rationale:
The patch's intent was to move LOV printing functions to a new file `liblustreapi_lovea.c`. The `lov_dump_hsm_lmm` function was successfully moved with an updated signature (removing unused `path` and `depth` parameters), but the old version was accidentally left behind in `liblustreapi.c`, causing the conflict. Removing the duplicate resolves the issue.
