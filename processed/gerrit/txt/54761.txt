--- Commit Message ---
LU-17731 hsm: Send CANCEL requests to the correct agents

Purge and its child actions (i.e., cancel_archives) are special in
that they attempt to issue cancels without performing the normal call
to mdt_hsm_add_hall to assemble action lists.  This has an unexpected
side-effect of not revising (in the aforementioned function) the
selected UUID for the target to the agent that should be running the
command to cancel.  In this class of purge commands we must provide
that UUID up-front.

--- Message (Etienne AUJAMES, 2024-04-15 14:06:43.000000000) ---
Patch Set 1:

Thanks for the patch.

The purge path for active request does not need all the overhead check in mdt_hsm_agent_send() like: check compat, check mdt object, find agent, update on-disk data.
For "purge", we just need to get the active request info ("car") and send a cancel request to the copytool uuid. The on-disk records will be clear by mdt_cancel_all_cb() callback.

And the "purge bool" argument in mdt_hsm_agent_send() seems to be a workaround "to make it work". So I am not in favor of an additional argument for the purge path.

Maybe, we can subdivide the mdt_hsm_agent_send() into smaller/simpler parts.

e.g:
- mdt_hsm_agent_send(mti, hal, uuid): low level function, only alloc the kuc buffer, copy the hal and send it to the agent identified by the uuid.
- mti_hsm_agent_enqueue(mti, hal): find/select the agent, validate the request (compat...), declare the "car", handle the on-disk data and finally call mdt_hsm_agent_send() to send the kuc request.

With this subdivision, hsm_cancel_all_actions() should be able to call only mdt_hsm_agent_send(), a simple kuc request send.
