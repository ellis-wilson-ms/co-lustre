--- Commit Message ---
LU-16405 osd: lookup cache

MDT may need to re-lookup just checked names (after locking).
introduce a trivial tiny per-thread cache in OSD in order to
make such a repeating lookup cheap.

the original issue is that ext4_add_entry() doesn't really
check for possible duplicate (that would be expensive as
a whole 4K block must be scanned).

important: the cache is reset upon request processing completion as
we don't update iversion on a disk (due to conflict with VBR).

Fixes: 79acb9a9e7 ("LU-10235 mdt: mdt_create: check EEXIST without lock")

--- Message (Sebastien Buisson, 2023-08-04 08:57:17.000000000) ---
Patch Set 36:

> Patch Set 36:
> 
> Is there a reason that we allow encrypted filenames that are equal to 256 bytes instead of limiting them to 255 bytes?  Shouldn't the last byte would always be an encrypted NUL byte?

No, encrypted file names are not NUL terminated. And in general ldiskfs internally does not expect names to be NUL terminated, it uses a buffer and a length.

Moreover, having a limit of 255 bytes for encrypted names is not relevant. Basically file name encryption works by blocks of 16 bytes, so the length of an encrypted name is a multiple of 16. For instance, any clear text name between 225 and 240 chars will result in an encrypted name of exactly 240 bytes. Any clear text name between 241 and 255 (NAME_MAX) chars will result in an encrypted name of exactly 256 bytes.
This is how fscrypt works and how ext4 behaves with encrypted names.
