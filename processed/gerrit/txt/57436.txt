--- Message (Sebastien Buisson, 2024-12-16 17:04:59.000000000) ---
Patch Set 1: Code-Review-1

I think not being able to build the nodemap config locally is a serious error for which the target mount *should* fail. Otherwise we would end up with nodemaps "silently" ignored, which is not great from a security standpoint if nodemaps are here to control which clients can mount the file system and with which privileges.

And it is also true in case of nodemap refresh. The whole nodemap config is pushed to the targets by the MGS for every change, whether it is a new nodemap created or just a property modified on one of the existing nodemaps. We are not able to identify the scope of a nodemap change, so we reload everything every time a change occurs. So in case a reread of the nodemap config fails, it should also be considered serious, and not silently ignored, for the same security reasons.

I think a safer approach would be to make sure we do not get -ENOSPC when loading the nodemap config at target mount, as suggested by Andreas by reserving some space for internal Lustre config changes.
