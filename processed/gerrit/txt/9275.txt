LU-3606 fallocate: Implement fallocate preallocate operation

This patch adds fallocate(2) preallocate operation support
for Lustre. fallocate(2) method of the inode_operations or
file_operations is implemented and transported to the OSTs
to interface with the underlying OSD's fallocate(2) code.

In a saperate patch, a new RPC, OST_FALLOCATE has been
added and reserved for space preallocation. The fallocate
functionality (prealloc) in CLIO has been multiplexed
with CIT_SETATTR. (https://review.whamcloud.com/37277)

Lustre fsx(File system exerciser) is updated in a saperate
patch to handle fallocate calls. (https://review.whamcloud.com/37277)

Only fallocate preallocate operation is supported by this
patch for now. Other operations like, FALLOC_FL_PUNCH
(deallocate), FALLOC_FL_ZERO_RANGE, FALLOC_FL_COLLAPSE_RANGE
and FALLOC_FL_INSPECT_RANGE is not supported by this patch
and will be addressed by a separate patch.

ZFS operation is not supported by this patch. ZFS fallocate(2)
will be addressed by patch (https://review.whamcloud.com/36506/)

New test case under sanity is added to verify fallocate
call.

The part that seemed "not for landing" was all of the hard-coding of fallocate modes in fsx.  I thought the supported FALLOC_FL_* flags would be detected at fsx startup?  Is it possible that the Lustre implementation is not returning the right errors back to the caller?  

We should definitely be returning -EOPNOTSUPP from the client and/or OSD for the FALLOC_FL_* that are not supported.  In particular, we need to return an error for ZERO_RANGE, COLLAPSE_RANGE, and INSPECT_RANGE from the OST until they are handled properly.  Also, we can not support operations like COLLAPSE_RANGE for offsets that are not a multiple of stripe_count * stripe_size, otherwise it would require moving data between OSTs.  That would need to be detected on the client, and for acceptable offsets the remapped offset/range would need to be passed to the OST(s), but for now the client should _also_ return -EOPNOTSUPP for unsupported operations so that the right thing will happen when different client/server versions are in use.  A new client that supports all the modes running against an old server that does not, or an old client without COLLAPSE_RANGE running against a new server that does support additional modes can't depend on the server to return the error.

I think the only thing that is needed is to ensure that there >is enough free space in the filesystem for the fallocate() request, and >then (optionally) adjust the file size if FALLOCATE_FL_KEEP_SIZE is not >passed.

Sounds good. Additionally, For mode==0 (standard preallocate), check for available block and adjust the dataset/filesystem stat to reflect space used. Since we pretend that this mode works and actually do not mark the blocks used, this way, we also avoid possibility of other process/thread using up the blocks (space). Which, in such case would lead to preallocate eventually failing with ENOSPC. The reason is the fallocate semantics says that once preallocate'd it is guaranteed that the IO should not fail due to ENOSPC.

It is true that this is what the man page writes, but it is impossible to provide these semantics for ZFS or any copy-on-write filesystem. Even if there is guaranteed spa space for the first write, a second or third write will need extra free space that may not be available, especially if a snapshot is preventing the overwritten blocks from being freed. 

So, it seems that there may be some benefit to ensure the first copy can be written, but I wouldn't try very hard for that to work, just checking the free space is enough to start, to avoid a 1TB fallocate() to succeed when there is only 1MB of free space.
