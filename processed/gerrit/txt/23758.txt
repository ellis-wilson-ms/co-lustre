--- Message (John Hammond, 2016-11-16 17:33:17.000000000) ---
Patch Set 1:

This will mean that MDT inodes for file created by mknod() with volatile names are leaked until orhpan cleanup (which is minor but is still a bug), since volatile file destruction depends on a close RPC being received. That's also why the patch works.

This does solve the bigger problem that without HAVE_IOP_ATOMIC_OPEN volatile files are not always volatile (in the cases where ll_create_nd() is called without an intent).

--- Message (Jinshan Xiong, 2016-11-16 18:18:55.000000000) ---
Patch Set 1:

> This will mean that MDT inodes for file created by mknod() with volatile names are leaked until orhpan cleanup (which is minor but is still a bug), since volatile file destruction depends on a close RPC being received.

Yes, things are weird here. An orphan file is supposed to be an open-unlinked file, but now we have an inode under orphan directory w/o openers. If the MDT crashed during this period, the open following mknod() will fail because volatile file has been removed.
