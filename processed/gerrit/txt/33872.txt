LU-11783 build: remove lustre_user.h deprecation warning

The "lustre/lustre_user.h" header has been in use for many years.
The patch https://review.whamcloud.com/25246 "LU-6401 uapi: migrate
remaining uapi headers to uapi directory" moved the header to
"linux/lustre/lustre_user.h" and left a stub "lustre/lustre_user.h"
behind that generates a compiler warning that this header is
deprecated.

However, no window was given between the introduction of the new
header and the deprecation of the old header, which makes it harder
for applications to smoothly transition to the new header location.
Also, installing Lustre headers into the "linux/" directory before
Lustre is actually part of the kernel may potentially cause problems.

Disable the deprecation warning in the old header for several
releases, until the new header location has been available for a good
time and it is safe for applications to assume that it is available.

James, obviously I wouldn't suggest to just surprise remove the header and cause build failures. I'm suggesting in a few releases that the warning could be added back. I don't think that is a 6-year window, more like a 3-4 year window. 

Even so, the ongoing maintenance cost of having a <lustre/lustre_user.h> header that just includes <linux/lustre/lustre_user.h> header is extremely low. Also, it isolates applications from the underlying details of the headers, like <sys/types.h> includes <asm/types.h> and/or <linux/types.h> internally, and apps shouldn't include the low-level headers directly

As I previously wrote, I considered to do this myself, but lustre_user.h currently does not depend on including lustre_ver.h machinery to be usable, and I don't think that is something we really want to do either as it makes the header more fragile for no real benefit.  The userspace header "version" of the code they are using does not change by itself, so they would only get a new lustre_user.h when they upgrade Lustre anyway, so adding complexity there just to remind us is of negative value, IMHO.

It just means that we need to submit a patch in the next few years to reinstate this #warning line.  That said, I'm beginning to wonder if we even want to do that - I'm beginning to think that I'd rather have applications continue to use <lustre/lustre_user.h> which has existed for many years and have it include the gory internal per-release details from <linux/lustre/lustre_user.h>.  There are lots of headers that have a "public face" and include system-specific headers in the background (e.g. <asm/*.h> or <linux/*.h>).

This patch is trying to be as minimal as possible so we can get it into 2.12.0 (which we are trying to release this week) from the beginning.  We can discuss long term plans for 2.14 or 2.15 at our leisure, but I'd rather avoid a lot of change in this particular patch.
