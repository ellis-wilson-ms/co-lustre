LU-14555 lnet: emit CDEBUG for hops == LNET_UNDEFINED_HOPS

lnet_check_route_inconsistency() checks for inconsistency between
lr_hops and route->lr_single_hop.

The warning should not be emitted if lr_hops is LNET_UNDEFINED_HOPS
which means the admin didn't manually configure a hop count. Setting
hop count is optional.

I don't think is is appropriate. As I said before, LNET_UNDEFINED_HOPS is equivalent to manually setting hop_count == 1. In other words, the default hop count for all routes is 1. I don't think it makes sense to treat hop_count == 1 and hop_count == LNET_UNDEFINED_HOPS differently. The purpose of this warning is to tell the sysadmin that there is some misconfiguration. If we detect multi-hop route, but the hop count is undefined is the exact same situation as we detect multi-hop route but the hop count is set to 1.

Amir introduced LNET_UNDEFINED_HOPS, so maybe he can say more about the historical issues. It may be the case that we could just get rid of it now, but I would need to look at the code more closely. > If hops == LNET_UNDEFINED_HOPS is a misconfiguration, why do we allow it? 
I'm confused by your question. The purpose of the warning is to alert the following situation:
1. The router does not have an NI on the remote net, and
2. The route has a hop count of 1.

If both of those are true then there is potentially a configuration issue. Either the hop count was set incorrectly (either implicitly by taking the default or explicitly by setting the wrong value), or the router is misconfigured.

> Why not require hops to be set when creating new routes via lnetctl or module parameters?

This may be reasonable, but in general I think we should be wary about putting new requirements on sysadmins that break from historical precedent/behavior. It is not great to have to create new configuration files just because you upgrade Lustre, and LNet configuration in particular can be very complicated when you have dozens of routers/hundreds of routes.

That might have been with us.  I recall an upgrade years ago when hops suddenly started mattering in a point release.

We're just one example, but at LLNL we never set a hop count.  From a given node, any routes we define to a remote net all go through the same set of routers and thus all have the same hop count.  Therefore, at our site, lnet_compare_routes() will _always_ return 0 whether we set a hop count or not.

So for us, there has been no value in spending the effort to determine those hop counts and keep them maintained when our topology changes in some way.  That's not your problem, but I would be surprised if we were the only site that doesn't set hop counts.

That would be ideal, but I don't think it is possible. For one thing, it would require that the router be up and respond to a ping when the route is added. That won't always be the case. Additionally, we would lose the ability to detect a change in the router configuration. e.g. if a router is rebooted but one of its remote interfaces does not come up. Local peers could learn about this and issue the warning.

That said, I know how annoying and counterproductive it can be to have messages flooding the console. I take it your routes are multi-hop so you would need to set the hop count to avoid the warning message? Another tweak we could make is to only issue the warning if avoid_asym_router_failure is enabled. The check only makes sense if that feature is enabled, and that feature only makes sense for single-hop routes
