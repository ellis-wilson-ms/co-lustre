LU-10070 lod: Self-extending Layouts

This commit implements self-extending layouts (SEL), which
are described in detail in LU-10070 and the associated
design document.  Previously referred to as self-extending
progressive file layouts (SEPFL).

This feature improves behavior when low on space by making
it possible to create layouts which can adjust to use new
OSTs when a given set of OSTs is low on space.

This is done by creating layouts with components of limited
length followed by a special unitialized component, called
"extension space".  When I/O reaches the extension space,
the client send a write intent to the server and the server
adjusts the extents of the layout so the write does not
touch the extension space.

It does this by checking the OSTs of the real component
preceding the extension space & if there is sufficient
space, it increases the extent of that component (and
reduces the extent of the extension space component) so the
I/O can use that space.

If there is not enough space on the OSTs, server does not
extend the layout of the preceding component, and instead
deletes the "extension space" component and brings the
start of the next layout component down to meet the end of
the existing component.  This allows I/O to continue to the
new component, which can use a different set of OSTs.

This is referred to as "spilling over", and applies mostly
to tiering situations where there are multiple pools and
the inner tier may fill up while there is still free space
in the outer tier.

There are three further aspects to highlight:

Trailing components

Extension space components can also be of limited length or
of infinite length - This means it's possible to limit the
maximum size a component can grow to, or only stop using a
component when the OSTs run low on space.

This requires allowing layouts to continue after a -1
component.  This means we cannot specify the starting point

of layout components that come after a -1 component, we can
only specify limits on their length.  So the 'end' of these
components is used to give that limit, and when the -1
extension space preceding them is removed, we take these
trailing extents & convert them to normal extents.

Zero-length components

It is also possible to have zero-length components when
they are followed by extension space.  When I/O reaches a
zero-length component, we stripe it and check to see if
there is sufficient space on those OSTs.  If there is not,
we delete the zero-length component and spill over to the
next component.

Repeat components

If a layout consists of a normal component followed by
extension space to EOF and we report low on space when
trying to extend that layout, there is no option to spill
over to another component, because there are no further
components.  So instead we try to create a new component
using the striping information from the normal component
which is out of space as a template.  This basically
"restripes" that component, hopefully moving to other OSTs.
If there is enough space on those OSTs, we extend there,
and if there is no, we remove this new component and extend
the existing component.

Examples of usage, etc, are found in the tests and
setstripe documentation.
