LU-11130 osd-ldiskfs: create non-empty local agent symlinks

e2fsck doesn't like zero-sized symlink inodes created by
osd_create_local_agent_inode().  Store the FID of the remote
inode as the symlink target so that it is possible to debug
where this symlink comes from in case there is some problem
in the future.

It would be better to just migrate the whole symlink instead
of creating a remote symlink, in the very common case where
there is not a hard link to the symlink (which POSIX allows,
but is extremely uncommon).  For the short term, we store
keep the remote symlinks to ensure on-disk consistency with
this simple patch, until the migration code can be fixed.

--- Message (Andreas Dilger, 2018-07-18 20:40:37.000000000) ---
Patch Set 4:

> why not moving regular file inodes , with some conditions: nlink=1,
 > no data on MDS ,etc ?

One reason to not move regular file inodes automatically is because they expose the FID to clients, and clients can have locks on the file FID.  If the inode is automatically migrated to the new MDT this means some clients may have open files with locks on the wrong FID.

With symlink migration, there is no way for clients to have "open" symlinks, they are only temporarily used during pathname resolution, and they cannot be modified after creation so there is no race if a client has temporarily accessed an old or new version of the symlink.

I'd _like_ to be able to migrate regular file inodes transparently, see LU-7607 for more details.
