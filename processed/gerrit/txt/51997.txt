LU-17040 scrub: inconsistent item

When OI does not include the fid, scrub will attempt to
fix it with zero inode number. There is
low chance that fid would be found during full inode
scan. But inode scan requires an empty inconsistent
list. With repeated EINPROGRESS replies, inconsistent list is
always not empty.

Move fid with zero inode numbers to stale list.

1 scrub fix to print real OI resurect and
skip not related
2 out_handle debug for dt_locate() fid failed
3 debug for out requests when it was interrupted

Patch Set 2:

IMHO there is no need to maintain an inconsistent list, and actually osd_fid_lookup() has already done a quick inline scrub, if it can't find the mapping, it could just put the FID in stale list and return -ESTALE immediately, later if OI scrub fixed it in inode scan, it can resurrect then. Waiting for inode scan to finish doesn't help much, and it tends to cause system hung.

This may cause client see some object becomes stale, and later appears, which may be a bit strange, but it simplifies OI scrub code a lot.
