LU-19079 nodemap: allow switch to gss_id nodemap

If a client was initially part of a nodemap and gss authentication is
in place, the client should be kept as part of this nodemap when the
nodemap is switched to gssonly_identification.
To support this, add a new 'ps_nm_name' field in struct ptlrpc_sec to
store the nodemap name the import is associated with.

sanity-sec test_106 has been added to exercise this.

## Explanation

### The Fix

Changed line 478 in `lustre/ptlrpc/nodemap_member.c` from:
```c
libcfs_nid2str(nid)
```
to:
```c
libcfs_nidstr(nid)
```

### Rationale

The Lustre codebase has two different NID (Network Identifier) types and corresponding string conversion functions:

1. **Old API**: `lnet_nid_t` (a 64-bit integer type) → use `libcfs_nid2str()`
2. **New API**: `struct lnet_nid *` (a pointer to a structure) → use `libcfs_nidstr()`

In the function `nm_member_reclassify_nodemap()`, the variable `nid` is declared as `struct lnet_nid *` (a pointer), so it requires the `libcfs_nidstr()` function, not `libcfs_nid2str()`.

The patch author made a simple mistake by using the wrong conversion function. This is a common error when dealing with Lustre's dual NID APIs during the transition from the old integer-based NID representation to the new structure-based one.

The fix aligns with the rest of the codebase - for example, in `nodemap_handler.c` lines 753, 758, and many other places where `struct lnet_nid *` is converted to string, `libcfs_nidstr()` is consistently used.
