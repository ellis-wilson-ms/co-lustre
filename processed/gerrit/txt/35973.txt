LU-10467 lustre: convert most users of LWI_TIMEOUT_INTERVAL()

when l_wait_event() is called with an lwi initialised with
LWI_TIMEOUT_INTERVAL(t1, t2, NULL, NUL),
waits for a total of t1 jiffies, but wakes up every t2 jiffies
to check the condition - incase the condition changed without
triggering a wakeup.
In (nearly) every case, t2 is one second.
So this is effectively a poll loop around wait_event_timeout.
So replace with with

 seconds = t1;
 while (seconds > 0 &&
        wait_event_timeout(q, cond, cfs_time_seconds(1)) == 0)
     seconds -= 1;

Then if seconds is zero at the end, the whole loop timed out.

In the one exception ("nearly" above) if t1 is small, t2 is set to one
jiffies, so we always wait a little bit and check the condition.  For
that case, we count to "seconds >= 0" and adjust the timeout
accordingly when seconds == 0.

Note that in one case, the on_timeout function is
target_bulk_timeout() instead of NULL.  As this always returns '1', it
behaves exactly like passing NULL.

Patch Set 9: Code-Review-1

(3 comments)

I'm not a big fan of how this was converted, since it seems to me that on a busy server the one second wakeups will all be *at least* one second, and almost certainly the final wait will always be longer than what was requested.  While that doesn't look like a huge problem, I think it is possible to keep the error to a minimum without any additional complexity.

--- Message (Neil Brown, 2019-09-11 23:29:56.000000000) ---
Patch Set 9:

(2 comments)

> I'm not a big fan of how this was converted, 

I believe it is a faithful conversion - the new code behaves quite similarly to the old code.  The differences are unrelated to the concerns that you raise.

If there are no wakeups, then both old and new code call schedule_timeout(HZ) N times - which as you say could end up being longer than N seconds on a busy server.  There is no difference here.

If there are wakeups, but the condition doesn't become true, then the new code will issue a schedile_timeout() with the timeout being the remaining time returned by the previous schedule_timeout().
The old code would call schedule_timeout(HZ) again, until within one second of the final timeout - then it would schedule_timeout() the remainder.

Neither old nor new code calculate the jiffie when waiting should finish (As you suggest) and work towards that.
That possibility can be explored once this patchset lands - I've made a note to look into it.
