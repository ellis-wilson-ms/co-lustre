LU-3326 tests: sync after file deleted in run_*.sh

In run_*.sh, call sync to flush any dirty pages to OST
if MDT has not sent the object destroy request to it,
which will shrink the available disk space gotten before
running the tests and cause -EIO error.

--- Message (Hongchao Zhang, 2013-10-29 04:19:40.000000000) ---
Patch Set 1:

1, After client complete the unlink of some file, MDT will send object destroy requests to OSTs asynchronously, the delay is about several seconds

2, At client, the dirty data of the unlinked file will be flushed to OST after the unlink (these writes will also consume the disk space)

3, When OFD receive the object destroy request, one LDLM extent lock (0 ~ OBD_OBJECT_EOF) will be enqueued and it will tell clients to discard dirty data.

the main purpose of this patch is fixing the issue caused by the delay of the object destroy.

--- Message (Alex Zhuravlev, 2013-10-29 04:43:19.000000000) ---

--- Message (Hongchao Zhang, 2013-10-31 10:45:53.000000000) ---
Patch Set 1:

the cached pages of unlinked objects at Clients are not taken into account when calculating the available disk space, then the disk space would be not enough
when writing the new file.

in this case(dd in a loop), the free disk space is sometimes not big for there could be some objects without receiving the destroy request sent by MDT,
then the free disk space won't be enough to allow the total disk requirement of the new "dd" and the unflushed (cached) pages of unlinked objects.

this issue can also be fixed by modifying the run_dd.sh to wait enough disk space (free_disk_space - possible_cached_pages > 0)

Patch Set 5:

> at the moment I think it's better to wait for space just before specific test, otherwise we'd have to ensure every test should leave enough space for a subsequent one?

In run_*.sh, the operations are run in a loop to create and delete file. E.g., in run_dd.sh:

while [ ! -e "$END_RUN_FILE" ] && $CONTINUE; do
        #......
        mkdir -p $TESTDIR
        dd bs=4k count=$BLKS status=noxfer if=/dev/zero of=$TESTDIR/dd-file
        rm -rf $TESTDIR
        #......
done

So, in order to ensure $TESTDIR/dd-file is deleted, we need call wait_delete_completed after "rm -rf $TESTDIR".

I'm concerned that adding wait_delete_completed() to all of the test scripts will delay these tests running for a long time and make the . Alternately, would it be possible to just increase the OST size for these tests so that there do not need to be hacks like this in the test script?
