--- Commit Message ---
LU-9019 obd: express timeouts in timeout_t

In the latest kernels time_t has been removed since time_t has
been a 64 bit value just like time64_t so no need for it anymore.
To avoid confusion between timestamps and timeout values Lustre
has a timeout_t typedef which is in seconds and is a s32 since
timeouts are generally short. This helps to avoid errors that
has happens in the past with certain math operation between
timeouts and timestamps that lead to overflow and underflow
cases.

Patch Set 5: Code-Review-1

(28 comments)

Thank you for making this patch.  I really think it makes the code easier to understand when we distinguish between an absolute time and an elapsed time.

That said, I'm not super fond of "s32" for an elapsed time.  It has the benefit of being a standard type and not very commonly used in the Lustre code, so it is more easily seen, but we should never have anything that takes a negative amount of time to complete (if it does, please let me know as there may be money to be made... :-).

I'm a bit reluctant to add a new type for this (like time32_t, or elapsed32) but I'd really rather avoid having a signed field, since that also opens a can of worms if we ever have underflows in calculations and we need to add extra checks for negative values.  

Using "u32" in the code would be most correct for this, though it loses the visibility a bit.

-1 for a couple of actual issues to fix, and potentially a bit more to add into this patch for "at_min", "at_max", and "at_extra"?  I don't know how much more that would drag into this patch, or if it is largely overlapping already.

Actually negative times have happened in the past. In fact we had a bug at ORNL in the past in which the recovery time wasn't counting down but instead increasing which prevented recovery from ever completing.

So since we are doing math between 64 bit (time64_t) and 32 bit values we will end up with negative numbers or potential overflow conditions. So we have to watch for negative numbers. Even if we use u32 we will have problems. In that case instead of being seen as a negative number we will end up with a ridiculous time out of like 68 years. So I'm going to look at use the clamp macro to limit the potential time out range.
