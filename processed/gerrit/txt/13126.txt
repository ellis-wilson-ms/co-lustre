Patch Set 6: Code-Review+1

(4 comments)

I'm OK with this in principle, though SOM is a feature that a lot of users would like to see, to optimize "ls" performance and such.  I've been thinking for a while that having a "simplified SOM" would cover most cases and be a lot less complex than what is currently implemented.

That would only store the file size on the MDT in case of a single-client writer, and if multiple clients were involved (i.e. client didn't have full-file lock at close time and know the file size) then LFSCK could verify/repair this on the next scan.  At O_WRONLY time, the MDS would synchronously invalidate the file size, or alternately for O_RDWR (due to Fortran always using this for all files) return a write handle that the client would need to reopen if it actually wants to modify the file, like HSM already needs to do.

I'm going to bring up SOM removal as a discussion topic at the OpenSFS developer meeting next week, and if nobody chimes in with strong support for maintaining/fixing SOM then this can be removed.
