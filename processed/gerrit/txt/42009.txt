(2 comments)

Bobijam, I realized a few things here:

1. Using FUTIMES_3 will require CAP_SYS_ADMIN and this will mean that only root can expect these operations to succeed.

2. Several of the operations will cause st_blocks to change and so technically this should update st_ctime.

3. This creates ineterop issues around MDS_CTIME_SET.

So, let's go back to your solution in patch set 2. Sorry for the inconvenience.

--- Message (Andreas Dilger, 2021-03-24 18:36:06.000000000) ---
Patch Set 5:

(2 comments)

I'm not against this patch, since I definitely think the timestamps should be preserved across mirror/migrate operations.

However, I'm wondering if this is happening at the wrong level?  Should it be the MDS that is setting the timestamps on the OST objects when the layout swap or mirror extend operation is being done?  That is definitely not more expensive than having the client send extra setattr RPCs, and avoids the need for the user tools to do this correctly each time.
