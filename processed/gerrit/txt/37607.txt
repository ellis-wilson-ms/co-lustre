--- Commit Message ---
LU-9679 osc: simplify osc_extent_find()

osc_extent_find() contains some code with the same functionality as
osc_extent_merge().  So replace that code with a call to
osc_extent_merge().

This requires that we set cur->oe_grants earlier, as
osc_extent_merge() needs that.

Also:

 - fix a pre-existing bug - osc_extent_merge() should never try to
   merge two extends with different ->oe_mppr as later alignment
   checks can get confused.
 - Remove a redundant list_del_init() which is already included in
   __osc_extent_remove().

> This seems to be breaking internal working of osc

I did find a bug in my patch - grants might end up wrong by cl_grant_extent_tax.  But I don't think that causes this crash.

I think the bug was already present, but my patch must have made it easier to hit.
If I have an extent that was created when mppr was (say) 36 and this extent was full, then later when mppr is 72 we try to find an extend for a page just before this full extent.  The current code will allow that merge as oe_max_end is the same (assume the first extent was at the end of a 72-page chunk).  But that makes the eventual nr_pages in that extent larger than mppr.

In the particular bug reported the first nppr was 174 - which evenly divides the max_end+1 - 174 * 172 = 29927+1
The later nppr was, I believe 232. 232 * 129 = 29927+1.
The original extent was from 29754 to 29884.
Somehow an extent from 29696 to 29753 was merged with this.
As they had the same oe_max_end, that was allowed.  But they had different oe_mppr.

Unfortunately I cannot see how my patch makes this bug more likely to happen.
Maybe the error in the grants had some obscure effect...
