--- Commit Message ---
LU-4838 osd: object iterator in zfs to sync before iteration

The issue with dmu_object_next() is that it doesn't expose
non-committed blocks in the metadnode. Therefore the iterator
doesn't find dnodes allocated in the current (non-committed)
txg which breaks testing and may affect LFSCK. As a short-term
solution, osd_zfs_otable_it_init() ensures the current txg
is committed.

--- Message (Andreas Dilger, 2014-03-29 22:48:35.000000000) ---
Patch Set 2:

ZFS is slow enough by itself without adding a full txg sync every time an iterator is used.  This seems like a pretty major performance hit for a very marginal condition. 

The first question is whether it is necessary for LFSCK to iterate over an object that was just allocated?  I think it would be better to just skip new objects, as is done in a few places in the code already. If the sanity-lfsck or sanity-scrub test needs the sync to work properly, then this may be one of the few cases where I would advise changing the test to add a sync.  LFSCK is intended to fix long-standing problems and should not spend complexity to fix transient non-issues. 

That seems racy in any case, so it would be better to ignore such objects. If we really need to iterate this object, is it possible to only sync that object if it is being iterated?
