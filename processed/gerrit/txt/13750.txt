LU-4727 hsm: use IOC_MDC_GETFILEINFO in restore

Use IOC_MDC_GETFILEINFO rather than fstatat() to get the original file
attributes during restore. Add test_12p to sanity-hsm to check that
triggering an implicit restore from the copytool's own mount point
does not wedge the copytool.

--- Message (Andreas Dilger, 2015-03-03 03:16:11.000000000) ---
Patch Set 4:

Patrick, I think the two patches are complimentary.

This patch avoids getting the layout lock when fetching the file attributes, by using IOC_MDC_GETFILEINFO instead of fstatat() that fetches the layout lock. While in many cases using the attributes from only the MDT would be inaccurate, the MDT attributes are authoritative for a released file and that is the only case that matters here. 

Jinshan's patch is also fixing a bug on the MDS about lock revocation, so that when the file is being restored the right lock is being held.

--- Message (John Hammond, 2015-03-03 14:08:16.000000000) ---
Patch Set 4:

> Patrick, I think the two patches are complimentary.

I think that this patch should be used and Jinshan's should not.

> This patch avoids getting the layout lock when fetching the file attributes, by using IOC_MDC_GETFILEINFO instead of fstatat() that fetches the layout lock. While in many cases using the attributes from only the MDT would be inaccurate, the MDT attributes are authoritative for a released file and that is the only case that matters here.

> Jinshan's patch is also fixing a bug on the MDS about lock revocation, so that when the file is being restored the right lock is being held.

Why do we need to revoke that lock except to update LLIF_FILE_RESTORING which only exists to avoid the deadlock that is avoided by this patch? The less done in the already too complicated HSM restore path the better.
