--- Commit Message ---
LU-18498 hsm: don't lock llog at cdt

llog does not need outside locking. Only HSM data
should be protected inside callbacks. Howevere most of
it only read data. cdt_last_cookie is converted to
atomic64_t. Also fix adds synchronization between
hsm_cancel_all_actions() and mdt_coordinator(), cancels
wait ongoing work.
To be safe between llog processing vs record modification,
dt_write_lock() is added to a modification llog_osd part.

(1 comment)

llog are protected for concurrent record accesses for read, insertions and deletions.
But, I am not sure that this is still true for record in-place modifications.

llog_write() takes the loghandle->lgh_lock in write mode when calling llog_write_rec(), so this should protect us against on-disk corruptions

But, llog_process_thread() does not take the read lock. And the llog records are aligned in chunk of 8192 bytes, so an record can be split across several blocks/pages. In theory, I think that llog_process can return a corrupted record in case of a concurrent write.

Except for "purge" all the in-place modifications happened for an active request. So we can use the active request structure to lock the record for write or read.

Also, it would be great to add a test with a concurrent "purge" to verify if it works as expected.

Yes, I agree. But, as I said before, this is for new records. This (with llog bitmap) does not protect us against record in-place modification (LLOG_HDR_TAIL(llh)->lrt_index and LLOG_HDR_BITMAP(llh) are unchanged).

> There is no any problem with a modification vs reading race too. HSM modify only status 4 bytes, so the worst thing is getting old data vs new, but who cares, modification could be later or early.

mdt_agent_record_update_cb() modifies also arr_req_change (__u64). Then this can be interpreted concurrently by mdt_cdt_started_cb. So I imagine that the record can be set to "SUCCEED" and then to  "CANCEL" and emits several changelogs with different status for the same request. The occurrence is low.

After reading the HSM code, I do not see a major issue with this patch. I agree that a global lock for the llog is overkill.
But HSM code is tricky (e.g: CANCEL request), I am not confident enough (notes that https://review.whamcloud.com/26803 did not remove the locking).
So I would prefer to always trust the request cache for STARTED requests (to avoid reading the llog).
