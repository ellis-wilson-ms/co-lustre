LU-15393 lod: use killable semaphore for creation path

lod_ost_alloc_qos() function sleeps during ost failover, but object
allocation could use different OSTs. The patch changes
down_write call to down_write_killable and adds timer for a
wakeup.

The main idea of this fix is next, when OST is lost during
lod_ost_alloc_rr() and MDT does not have precreated objects for it
lod_ost_alloc_rr()->..->lod_qos_declare_object_on() would sleep while
holding a lq_rw_sem for read. Any creation thread would stuck at
lod_ost_alloc_qos() waiting lq_rw_sem for write, after statfs update.
Whith a fix sleep is limited and allocation would going through
lod_ost_alloc_rr(). For read lq_rw_sem is shared and stripe allocation
would skip OST without objects.

lod_ost_alloc_rr() refills OST pool with a lq_rw_sem for write, when
lq_rr.lqr_flags has LQ_DIRTY. This should happen only when OST is
added/removed. No need to set LQ_DIRTY for lq_rr when statfs get
error, this flag does not affect any change for pool list at
lod_qos_calc_rr().

Change behaviour for lod_check_and_reserve_ost(), it  would sleep
during object allocation for speed 2 only.

--- Message (Alex Zhuravlev, 2021-12-23 09:07:17.000000000) ---
Patch Set 2:

there are few threads serialized by the single mutex in QoS path: T1, T2 and T3.
T1 got the mutex, can't find an object, waiting now.
T2 and T3 waiting on the mutex.
finally T1 gives up, uses another OST and releases the mutex.
T2 gets the mutex, finds itself waiting for too long and avoids any waiting (say, at cost of unbalanced allocation).
T3 does the same.
this way there will be no cascading waiting.
please check also LU-13073

OK.
In your example all creation threads would stuck for a failover case, I think it is worst behavior. Even if cluster would have 1k OSTs, any failover would stop allocations. The best thing we can do is avoid sleeping on objects at alloc_qos, it is better to return EINVAL to a caller than stop creation for all clients.
In this patch I fixed a bit different situation, when alloc_rr sleeps on objects, and other threads at alloc_qos wait mutex for a write. We do see exact hang at vmcore. It is better to do alloc_rr than wait 20minutes at alloc_qos in my opinion. The stripe creation would work for clients in this case.
