--- Commit Message ---
LU-3335 scrub: rebuild LAST_ID

The /O/<seq>/LAST_ID records the last oid of the object allocated
within the sequence. The LAST_ID file can be crashed or missed as
the system running. The OI scrub can detect the LAST_ID lost case
when initial OI scrub (at mount), and can rebuild the LAST_ID via
scanning the whole device.

This functionality is also part of LU-14 live replacement of OST.

> Alex Zhuravlev		12:07 PM
> the advantage is that this recovery mechanism would apply to any backend. and again, having similar logic in the same place (LFSCK) surely makes sense. the only reason for OI scrub to be part of OSD is OI which is internal to OSD ?

I agree with that the rebuild LAST_ID should work for any backend. But if want to do that in LFSCK instead of OSD, then we need to resolve some implementation  troubles: during the LFSCK scanning, it may find the LAST_ID crashed, so it needs to update the LAST_ID if it found more higher oid in the <seq>. If each later scanned is higher than its former (for the case LAST_ID file lost), then it will cause the LAST_ID to be updated for each object scanned. That is almost cannot be accepted. So we still need some work in OSD to process such cases, which may need some new APIs or parameters for the LFSCK to notify and control OSD's behaviour.

Fan Yong, to answer one of your earlier questions - the "fid" xattr ("struct filter_fid_old" today) was first added in Lustre 1.6, and was only changed in Lustre 2.4 to be smaller and move the "self FID" into LMA.
