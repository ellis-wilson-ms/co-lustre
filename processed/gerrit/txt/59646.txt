LU-0000 mdt: fix ACTIVE_WRITER lock blocking AST crash

Fix a kernel crash that occurred when ACTIVE_WRITER locks were
cancelled via blocking AST. The crash happened because the
mdt_active_writer_dec() function tried to perform FLR state
transitions from blocking AST context, which lacks proper
Lustre environment setup.

The crash occurred in this sequence:
1. Client releases ACTIVE_WRITER lock after write operation
2. Server-side blocking AST (mdt_blocking_ast) is called
3. mdt_active_writer_dec() tries to call
   mdt_flr_state_transition_to_rdonly()
4. FLR transition calls mdt_attr_get_complex() which eventually
   calls mdd_read_lock()
5. mdd_read_lock() tries to access lu_context and crashes with
   NULL pointer dereference

The fix adds environment validation in mdt_active_writer_dec():
- Check if lu_env_find() returns a valid environment
- Check if mdt_th_info() returns valid thread info with mti_mdt
- Only attempt FLR state transition if both checks pass
- Otherwise, safely skip the transition with debug logging

Key changes:
- Enhanced blocking AST setup for ACTIVE_WRITER locks to ensure
  proper l_blocking_ast function assignment
- Added environment safety checks in mdt_active_writer_dec()
- Improved debug logging for lock lifecycle tracking

The ACTIVE_WRITER lock increment/decrement mechanism works
correctly:
- Increment: called when lock granted in mdt_intent_active_writer()
- Decrement: called when lock cancelled via blocking AST
- Lock cancellation: triggered when client calls
  ldlm_lock_decref_and_cancel()

This fix prevents crashes while maintaining the active writer
counter functionality. FLR state transitions are skipped when
called from blocking AST context but will work correctly when
called from proper thread contexts.
