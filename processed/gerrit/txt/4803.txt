LU-2473 ldiskfs: Reorganize ldiskfs kernel patches

This commit makes no changes to the patches, only moving
their location.

For some time we have supported only a single kernel per major
OS release, e.g. RHEL 6.3, or RHEL 6.2, but not both.  While
it is understandable to only support a single kernel, it is
quite another to intentionally destroy the patchset for the previous
kernel every time we upgrade.

This makes the logistics of upgrading an OS release (even a minor
one like RHEL 6.2 to 6.3, or 6.3 to 6.4) quite complicated.

We should really start a new patch series when we update support
for a new kernel.  That way we can leave the patch sets for
previous kernels in place.  We do not need to "support" them, but
at least leaving them unmolested until the users of the previous
patch set have time to upgrade would be greatly appreciated.

This commit attempts to organize the patches into subdirectories
according to the kernel/os in which they were first created.  A
later commit will then add support for RHEL6.4.

Also, fix build/confirmpatches.sh to allow checking
ldiskfs/kernel_patches (a bug prevent checking anything but the
default).

Extend both confirmpatches.sh and clearpatches.sh to work with
series files that point to patches organized into subdirectories.

I'm not against this reorganization, but I'm concerned about quickly increasing the number of patches for multiple different kernels.  The other complexity added is that a new patch for each maintenance release makes it difficult to compare the changes being added for each new kernel, and perhaps see bugs caused by just porting patches to the new kernel rather than seeing some inter relation between what caused the context change and the patch itself.

Is there any plan to deprecate the old kernel series after some time, or will it linger around forever?  The difficulty with saying that the patches will remain on the repo, but "not be supported" is that people don't read the documentation, and assume that if the patches are there that they will work.

At one time you had proposed to have a single version of ldiskfs that built against all kernels. Has that plan changed, or is this a step in that direction?

In this or the next patch (adding RHEL 6.4) please add a README that explains that patch series should re-use the patches from existing kernels instead of creating new patches for each new kernel, and that the separation into separate subdirectories is only for tracking and indication of which kernel the patch was first created for.

Jeff,
to clarify Gerrit behaviour - if you cherry-pick a patch and base later patches on it and push that branch back to Gerrit, this is essentially a "git rebase" of the cherry-picked patch since the parent git hash has changed.

If you want to submit new patches on some existing patch in Gerrit, you should use the "checkout" URL instead of the "cherry-pick" URL.  That keeps the base patch unchanged (same parent hash, same patch hash) and the base patch will not be updated when you push the branch with new patches on top of it.
