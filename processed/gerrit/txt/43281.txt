--- Commit Message ---
LU-13212 osc: fall back to vmalloc for large RPCs

For large RPC sizes (16MB+) the page array (4096 brw_page) can
become very large (128KB+ with fscrypt) and should fall back to
vmalloc() if kmalloc() fails due to memory fragmentation.

The mdc/mdt allocations are currently limited to 1MB for readdir
RPCs, but it doesn't hurt to prepare them for larger RPCs from
clients in the future if this limit is increased.

Fixes: 51b32ac2b9b8 ("LU-7990 rpc: increase bulk size")

The performance impact is "infinite", since without it the IO returns an -ENOMEM error.  With the patch, if kmalloc() fails (or is too large for the kernel) then vmalloc() will be used and the IO will succeed.

Sorry I didn't mean to imply its a bad patch. As I pointed out in an earlier email to lustre-devel using vmalloc ended up crushing our performance at ORNL when using large stripe counts (> 672 stripes) due to a global spin lock. We got around this by modifying the tools to limit max stripe count to 512. 

The problem was I never had a way to test the performance impact in our test bed since I didn't have 700+ OSTs to work with. This code suggest that if I set max_rpc_size to 16MB I can easily reproduce the vmalloc performance issues. That means I can test a potential solution to this problem.
