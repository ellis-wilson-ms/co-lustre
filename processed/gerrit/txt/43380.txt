LU-13055 mdd: per-user changelog names and mask

Allow specifying a name for newly-registered changelog users,
rather than the default "clNNN" that is otherwise used. This
allows services to register a "well-known" changelog user,
rather than having to store the changelog username in HA storage
outside of the filesystem.

Each changelog user still has a unique ID appended to it, to allow
the changelog_clear and changelog_deregister commands to be run
using only the ID if necessary/desired. User name can be used to
deregister. User name is also unique per server.

If no name is given, then default "cl" format is used.

With this new functionality, it is possible to specify the name like:
 # lctl --device testfs-MDT0000 changelog_register --user watcher
   testfs-MDT0000: Registered changelog userid 'cl13-watcher'

Per-user mask is also added to allow specific operation logging on
per-user basis. Mask can be set only during registration. Resulting
mask from per-server mask and all user masks is used for current
changelog operations.

Andreas, just want to describe how masks are handled in patch now. We have per-user masks which are stored and shown exactly in values set by user - no mask set means user mask 0, mask set means all requested bits by user during registration. If user requested mask relatively with '+options' then default mask is added as basis for that user mask. All masks are shown as is in lproc output and per-server mask is shown via .changelog_mask parameter.

All masks server+users are combined to get current_mask which is used on log generation. So server mask is just like another user which mask is set via set_param.

The situation with users without mask is treated as before - they are using per-server mask and don't apply any bits from own mask which is 0. So zero mask means 'use default server mask' it cannot mean 'no operation should be logged'.
Meanwhile if user has no mask and server has no mask - nothing will be logged and that would be strange case, though it works so right now without patch even.
And that is what we need to define clearly. If we allow per-user masks we might need to set server mask to 0 to don't interfere, then any new user without mask becomes meaningless. First intention is to treat they as having default mask, but that means we will not be able to shut them up by setting server mask value to 0 as it was possible before patch, we would need to deregister them instead. I mean that behaviour is changed and may produce non-expected results for admins and users of changelog. If they expects no changelogs when server mask is unset. Basically that is question about role of server mask - is it just another user for simple managing of changelog without registering new user, or it is some master-mask to override all users mask.
It looks like the role of per-server mask as 'another user' is meaningless - we can register just user with needed mask instead. So either server mask is not needed at all or should have some other role, e.g. overriding users mask by doing AND operation or something else

Well, I propose to start with some initial basics we can't ignore or change - for compatiblity needs we need 1) server mask to be set to DEFAULT mask and can be overwritten by "set_param -P" 2) users without mask should mean always 'server mask to be used'. This allows us to provide compatibility and keep old behavior.
That means if we have any user without mask - the server mask is applied plus all user masks. If we have no users without mask then server mask is not applied and only combined user mask is used. In the latter case the changelog logging level cannot be changed via server mask setting, so admin would need to register user with extra mask bits to be applied or just user without mask to server mask to be ON again.
