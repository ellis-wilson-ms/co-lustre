Unfortunately, after more testing I realize that the mirror number is not enough information to restart the object traversal. In fact, it looks like even without FLR mirrors the handling of PFL files is still not working properly.

I tested this on a 3-component PFL file ("-E1M -c 1 -E16M -c3 -E1G -c4" that was written up to 1G so all components initialized), and while it is showing _some_ of the stripes, it definitely is not showing all of them.

--- Message (Andreas Dilger, 2020-12-05 16:26:48.000000000) ---
Patch Set 5:

Also, in case of overstriping with a plain file, it may be that there are multiple stripes within the same component on a single OST index, so that needs to be handled also.

It may be that instead of returning the component index we need to return the stripe number across all components to make sure that the restart happens in the correct place.

For now, however, all of the PFL components are being returned in a single call, so it isn't an issue of restarting, AFAICS.

I think I've figured out where the problem is.  Lustre should be storing the stripe number with every extent, so that the old (and new) filefrag can just copy fe_device from the last returned extent into the 0th extent on a restart.

This will also fix sanity test_130e, even with old filefrag, since every extent on the same stripe will report the same fe_device.  With old kernels it would print "000x" since the stripe index is not set.  With new kernels + old filefrag it will print "1000x", then "2000y" and "3000z", ..., but it would be the same for every extent.  The new filefrag would not print the high bits, but would otherwise work the same.

Since even the old filefrag will copy fm_extent[i-1].fe_device to fm_extent[0].fe_device (which will contain both the stripe number and OST index), then the kernel should have no problem to restart even for PFL/FLR files with the old filefrag.  It just needs to skip objects until it gets to the same stripe number, and restart there.
