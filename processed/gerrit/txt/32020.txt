LU-4398 mdt: acquire an open lock for write or execute

In mdt_object_open_lock() opens for write or execute will always
acquire an open lock of the appropriate mode so that any conflicting
cached open locks on other clients will be canceled. Add a regression
test to sanityn.sh.

Patch Set 2:

John, you are right. A small correction, however: your original reproducer stopped working after LU-4367, not after LU-8019. Before LU-4367, the first execution of echo would cache the lookup lock and the second execution would use a cached dentry, go through ll_file_open() and request MDS_OPEN_LOCK. However, LU-4367 modified ll_revalidate_dentry() in such a way that revalidation fails for open requests, so the client would request open from the common (non-NFS) lookup path without MDS_OPEN_LOCK, not from ll_file_open().

Please note that I pasted a link to pastebin with the improved version of LU-4398 and updated test which we're still using for 2.5.

However, due to our migration to a later Lustre version we are also interested in LU-4398 being landed upstream. We encountered this bug only with NFS configurations in a scenario such as uploaded to pastebin.

Thank you
