LU-13485 libcfs: FIELD_SIZEOF macro removed

Linux v4.15-rc2-5-g4229a470175b introduced sizeof_field() macro
Linux v5.5-rc4-1-g1f07dcc459d5 removed FIELD_SIZEOF() macro

Provide a sizeof_field() macro in terms of FIELD_SIZEOF()

--- Message (Andreas Dilger, 2020-08-25 20:12:21.000000000) ---
Patch Set 1: Code-Review-1

(2 comments)

> I am not against this patch, but commit comment (and comment
 > inserted in the code) is misleading. FIELD_SIZEOF macro was not
 > removed in Linux 5.1 but in v5.5-rc5 (commit 1f07dcc).

Using the exact version as given by "git describe" (i.e. v4.15-rc2-5-g4229a470175b and v5.5-rc4-1-g1f07dcc459d5) is preferable (here and in the autoconf comments), to avoid the need for the reader to re-discover the commits that changed the code.

 > For future code cleanup, it would have been better to replace
 > FIELD_SIZEOF with sizeof_field in this line, and
 > #define sizeof_field to FIELD_SIZEOF in case it is not defined.

I agree that the preference is to have "forward looking" compatibility macros, so that it works without change for newer kernels, and the old compatibility code can eventually be dropped by removing the compat macros without changing the code.  That also avoids new uses of the old interface being added and increasing the amount of work that needs to be done when removing the compat macro.

I had thought of this while reviewing the patch, but for some reason I had it in my mind that sizeof_field() was GCC builtin keyword (like __attribute__) and it could cause problems #defining a duplicate macro.  I now see that this is just a convenience macro in the kernel, so adding a sizeof_field() compatibility definition is no problem.

Since there is only the one user of FIELD_SIZEOF() I'd prefer that it just be replaced in this patch.
