LU-5341 ptlrpc: rpc times out in unregistering phase

If the reply to an rpc is received before Lustre is notified
that the request has been sent, the task making the request may sleep
until the rpc times out. Both the rq_reply_unlink and rq_req_unlink
flags must be cleared before an rpc can be completed. When the reply
is received, rq_reply_unlink is reset and the  waiter on the rpc wait
queue is woken up. If rq_req_unlink has not been reset at that time,
the rpc is unregistered and the waiter goes back to sleep. When
Lustre is then notified that the request has been sent,
request_out_callback resets rq_req_unlink but it only wakes up the
waiter if the event type is an UNLINK. If the event type
is a SEND, the waiter waits until the timeout on the the wait queue
triggers.

This mod also does a bit of code cleanup: removes an unnecessary
check for a condition that has already been tested in an LASSERT;
adds the request and reply unlink flags to DEBUG_REQ messages.

--- Message (Chris Horn, 2014-10-01 17:52:26.000000000) ---
Patch Set 1:

Alexey, I don’t think there is a problem here. The ordering of request_out_callback and reply_in_callback shouldn’t matter. If we get reply_in_callback before request_out_callback then we know that the RPC was delivered and processed within the RPC timeout. We just need to wait for LNET to release its reference on the outgoing buffer.

Ideally we shouldn’t have to wait long for the request_out_callback, and most of the time it will happen very quickly (and before reply_in_callback). But I don’t think any of the LNDs can *guarantee* that request_out_callback will occur before reply_in_callback. That’s just the nature of asynchronous events.
