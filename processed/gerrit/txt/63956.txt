LU-12669 llite: fix infinite loop in ll_readpage for EC reads

During EC recovery reads (CIT_EC_RD), ll_readpage enters an
infinite AOP_TRUNCATED_PAGE loop because lcc_end_index is 0
(unset from memset).  When lcc_end_index is 0, the check
"vmpage->index >= lcc->lcc_end_index" is always true for
every page, causing every readpage to return
AOP_TRUNCATED_PAGE.  The page is never actually truncated,
so the kernel retries indefinitely.

Fix this with two changes:

1) Add "lcc_end_index > 0" guard so the check is skipped
   when lcc_end_index hasn't been explicitly set.  This
   prevents the false positive match on page index 0.

2) Set lcc_end_index in vvp_io_ec_rd_start() from
   min(ec_inner_end, i_size).  ec_inner is expanded to
   full raid-group boundaries by lov_io_set_range(), so
   it can far exceed actual file data.  Pages beyond
   i_size on deactivated stripes have no DLM lock, so
   cap the index at i_size.

3) For CIT_EC_RD pages beyond lcc_end_index, return -EIO
   instead of AOP_TRUNCATED_PAGE.  These pages have no
   DLM lock and will never become valid, so retrying
   them would loop forever.

Resolves 100% CPU hang affecting all EC
recovery tests 42a and above (bead 2-54y).
