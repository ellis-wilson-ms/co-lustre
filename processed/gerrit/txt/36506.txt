LU-14157 zfs-osd: Add fallocate functionality for ZFS

This patch adds fallocate(2) (preallocate) operation
support for ZFS backend OSD.

ZFS fallocate operation first need to verify enough
free space in the file-system is available. If this
condition fails it returns with -ENOSPC.

Preallocation is really done and blocks are
allocated and reserved only when flag
"od_fallocate_zero_blocks" is set to 1. This is the
default. Else only OST object metadata is updated.

For prealloation it calls 'dmu_prealloc'. In case where
'dmu_prealloc' really fails, we still go ahead and
update the object size and return success.

check_set_fallocate() under t-f.sh is also changed
to handle both ldiskfs and zfs OSTs.

Test-Parameters: fstype=zfs

The main fallocate() patch is pretty much ready to land, it would be good to refresh this based on my previous review so that ZFS can also be included at the same time.

After that, having FALLOC_FL_PUNCH_HOLE and FALLOC_FL_ZERO_RANGE functions would be useful as well, and likely not much more than checking the features passing the flags through for ldiskfs.  PUNCH_HOLE will need some wiring for ZFS, since it already has this functionality, and I suspect that ZERO_RANGE would end up being the same as PUNCH_HOLE for ZFS, since "preallocating blocks" doesn't make sense since they cannot be overwritten in place.

I don't think the FALLOC_FL_COLLAPSE_RANGE and FALLOC_FL_INSERT_RANGE functions will be easy implement on Lustre, unless the file has only a single stripe and we could just pass the request down to ldiskfs, or the shift is a multiple of stripe_size * stripe_count.

The FALLOC_FL_UNSHARE_RANGE function doesn't make sense, since neither ldiskfs or ZFS allow clone_file_range().
