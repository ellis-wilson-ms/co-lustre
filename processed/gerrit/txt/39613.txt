LU-10360 mgc: Use IR for client->MDS/OST connections

When a target registers with the MGS, the MGS sends an IR log
to the client to speed up recovery.

The IR log contains the updated NID information on the target
which just registered.

This patch allows clients to update their imports with the latest
NIDs for the targets reported in the IR log. It also allows clients
to create new connections for targets which were not added via
the config log.

For example if a target reboots and comes up with a new NID, then
the client can continue using it.

This functionality is disabled by default and can be enabled by
setting a new file system specific module parameter, dynamic_nids.

    lctl set_param mgc.*.dynamic_nids=1

This parameters will need to be set on the clients and the MGS

(13 comments)

I think the patch looks technically correct, but there are a bunch of minor cleanups that could be done.  It also needs a test case for the dynamic_nids=1 case, to verify it is working and continues to work in the future.

Since this would involve mounting the OST on a new IP address, there are a couple of options for how to do this in our test environment:
- make up fake IP addresses, but that has some chance of conflict
- mount underlying OST device from the MDS via the VM host, but
  this is complex and VM-specific, so not great
- copy the OST image from one OSS to another OSS or MDS or client2
  to mount it.  This may be slow if image is large, but conf-sanity
  has limited target sizes to speed up formatting, so it may be OK.
  "scp" would be the obvious choice of mechanism, but I don't think
  that is guaranteed.  Another option would be piping stdout, like:

    do_facet ost1 dd if=$(ostdevname ost1) bs=1M > $TMP/$tfile-ost1.img
