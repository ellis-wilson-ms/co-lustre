LU-2190 mountconf: revert previous commit 8e249d43f19

Revert "LU-2190 mountconf: zfs processes mount parameters"

This reverts commit 8e249d43f19a4d227c76e6def41d7eb19e6a207f.

The reverted patch is incompatible with existing ZFS based file systems,
and is arguably the wrong approach to take in general. If a single ZFS
data set parameter is desired, a patch needs to be written such that it
is backwards compatible with existing file systems. This will give sites
a "grace period" in which they can convert their file systems to use the
new configuration format.

With that said, I'd argue that using a single ZFS dataset parameter for
the Lustre configuration parameters makes it much more difficult to
administer, and is a regression on its own; regardless of the lack of
backwards compatibility.

--- Message (Brian Behlendorf, 2013-01-24 18:22:40.000000000) ---
Patch Set 1: Looks good to me, but someone else must approve

I would argue that the 'risky' thing to do here is to keep them all in a single string.

This seems like a good idea on the surface.  But we've found that in practice with ldiskfs it increases the likelyhood of an error when changing the configuration.   This is largely because all the options are always modified and it's very easy to accidentally drop one, or modify one, or add one you didn't mean too.  Keeping them separate makes it easier to spot check you got it right and limits the damage when you didn't.

Our system administrators agree that keeping them separate is far preferable from a manageability standpoint.  And this is really the perspective we need to keep in mind when developing all of the user visible interfaces.
