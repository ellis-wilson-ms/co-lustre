--- Commit Message ---
LU-20131 lnet: skip unlock-relock in ksocknal_connect

In ksocknal_connect(), one error path calls write_unlock_bh()
and then jumps to the 'failed' label, which
immediately calls write_lock_bh() on the same lock — an unnecessary
unlock-relock sequence with no work done in between.

Restructure the error labels so that paths already holding the lock
jump directly to 'failed' (which now expects the lock held), and
paths not holding the lock jump to a new 'failed_nolock' label that
acquires it first. This eliminates the redundant unlock-relock.
