I don't think the patch should be abandoned.

I was just pointing out that it has a minor problem that is easily fixed in a few lines. It is true that the current code also has one case that isn't evenly balanced (OSTCOUNT=6, stripecount=4 has 50% imbalance), but this patch has many cases where one or two OSTs allocate all of the starting objects, and the other OSTs have no starting objects.

--- Message (Andreas Dilger, 2022-01-18 16:07:35.000000000) ---
Patch Set 4:

> it is better to care about parallels creations and
 > equal object distribution for a multithreaded time. With parallels
 > creations lqr_start_idx would have precession and files would
 > located with different first stripe index. I don't agree with
 > objections about offset. It is better to improve a loop for a
 > situation when lqr->lqr_start_idx jumps between threads and loop
 > processes OSTs not one by one imho.

There are definitely many workloads that have a single thread on one node creating many files, and that can lead to OST imbalance over time if one/few OSTs have all of the starting objects, especially if the files are not large enough to cover all stripes. That is why this code was made the way it is.  Having eg. stripe_count = 4 and OSTCOUNT= n * 4 is quite common, and should be handled properly. 

If there are multiple threads/nodes creating in parallel, then lqr_start_idx will advance more quickly, but I don't think the start precession is *worse* than not having it at all.  Adding a single extra atomic access per loop is nothing compared to the overhead of all files being written to only a small subset of OSTs.
