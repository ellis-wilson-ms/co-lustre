--- Commit Message ---
LU-18912 ptlrpc: change some import bit fields to bitmap API

Often we are taking a spinlock to read a single bit field. For
ptlrpc we might run into a might_sleep issue since some of the
code is run in an atomic context. By moving to bitmap API and
using the Linux kernel bitops we can avoid needing an spinlock.
We currently only move the import bit fields that are used by
flag2str().

## Summary

**Fix Applied:**

I fixed the compile error in `lustre/ptlrpc/gss/gss_cli_upcall.c` at line 292 by replacing the direct field access `imp->imp_deactive` with the bitmap API call `test_bit(IMP_DEACTIVE, imp->imp_flags)`.

**Rationale:**

The patch converted several bit fields in the `struct obd_import` from individual boolean fields to a centralized bitmap (`imp_flags`) using the Linux kernel's bitmap API. This change:

1. **Eliminates the need for spinlocks** when reading single bit fields, as the bitmap operations are atomic
2. **Avoids potential might_sleep issues** in atomic contexts
3. **Follows Linux kernel best practices** for managing multiple boolean flags

The specific field `imp_deactive` was removed from the struct and replaced with the `IMP_DEACTIVE` bit in the `imp_flags` bitmap (as defined in the new `enum import_flags`). The file `gss_cli_upcall.c` was missed in the original patch, so it still referenced the old field name, causing the compilation error. The fix updates it to use the new `test_bit()` API to check if the `IMP_DEACTIVE` flag is set.
