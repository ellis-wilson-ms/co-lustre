LU-18357 ptlrpc: Use nodemap IAM for persisting filesets

Currently, two separate commands exist to set a fileset and both
behave differently. "lctl set_param (-P)" works as expected. On the
other hand, "lctl nodemap_set_fileset" updates the fileset only on the
local node and is not persistent. It could also be run from other
server nodes and not only from the MGS. This behavior is inconsistent
with other nodemap commands.

Ideally, we would want to use the nodemap IAM records for persisting
all nodemap properties without relying on secondary mechanisms, such
as the params llog which is currently used for filesets. This patch
adds support for persistence across restarts for "lctl
nodemap_set_fileset" commands. Filesets are split into 28 byte path
fragments which are stored alongside 4 byte fragment metadata into one
nodemap IAM record (which is 32 bytes).

For this, fileset fragments are stored in a "defined" subid range
within the NODEMAP_CLUSTER_IDX type. In anticipation of supporting
multiple filesets, the first fileset subid is reserved for a fileset
header. The subid range is set to support at maximum 256 nodemap
records for fileset fragments.  256 entries support enough fragments
to hold a PATH_MAX fileset in total. Only needed fragments are
inserted into the nodemap IAM.

To support seamless updates from fileset llog to IAM versions, a
nodemap flag was added which allows existing "lctl set_param -P
nodemap.<name>.fileset" to be applied outside the nodemap IAM as long
as no IAM records were set. Filesets from the llog are not migrated to
the IAM but are still compatbiel after an update. "lctl set_param
nodemap.<name>.fileset" is deprecated and uses the IAM once fileset
IAM records are in use.

Sanity-sec "test_27a" was modified to test fileset persistence.

Andreas, thank you for the quick review!

I was considering this at first, and it would be enough in the way filesets are handled on "master" because only "set" and "get" exist, so "lctl nodemap_set_fileset" can simply be rerouted in user space. 

It becomes more complex when supporting multiple filesets per nodemap with new commands like "lctl nodemap_{add,del}_fileset" or potentially others in the future. To use the "params" llog, we have "lctl set_param -P" available, but we cannot change how it behaves in order to be consistent with the landed version. It must therefore overwrite the existing fileset. This means that "lctl set_param" must also support multiple filesets (even if only internally). 

Consider the following future situation:
- "lctl nodemap_add_fileset --name <nm_name> --fileset /foo"
- "lctl nodemap_add_fileset --name <nm_name> --fileset /bar"

This would extend to something like this with a new separator ":" (not supported in this patch) so that each entry in the llog is a snapshot of all filesets per nodemap:
- "lctl set_param nodemap.<nm_name>.fileset='/foo:/bar'" (llog entry)

In this case, redirecting the "add" call in user space no longer works because you need all filesets to persist them. So, instead of issuing three calls from user space: add one new fileset, get all filesets, "set_param -P" with all filesets, this patch does it directly on the MGS-side. 

I don't think we gain much doing it in user space except that we don't need to forge an "lcfg" anymore, but instead, we would need more complexity to communicate through ioctl. In any case, separating nodemap and fileset commands seems to be necessary.

I don't think we can easily do it in this case. The challenge is that we cannot differentiate right now whether "lctl set_param nodemap.xx.fileset" was issued by an "lctl" of a user, by lctl set_param -P" from on another node, or by the params llog. Ultimately, what the "-P" argument does here, on a high-level, is calling on other server nodes, record that action as an lctl command to the params llog, and then upcall it as an lctl command, which just writes this action into a proc file (which you could also do manually). This upcall also occurs when the llog is read.

"lctl nodemap_*" commands operate through ioctl, and blocking "lctl set_param" for them is straightforward. In the specific case of "nodemap.nm0.squash_uid", no "seq_write" function is defined, and therefore, writing to the proc file with that action is invalid -> EIO.  

Something similar could be done for the "nodemap.xx.fileset" action, but we would need a new action for filesets then to persist them in the llog. This would also break backward compatibility because of the new name. Overall, this is really messy and I don't think we should consider it.

If there are no other ideas, I think we should pursue this in another patch if needed because it may require more extensive changes.

Thank you, Andreas and Yingjin, for your initial comments! 

Since all tests were passed, this patch is now ready for review. 

Two remarks:

First, regarding the usage of llog filesets as it might not be obvious. In this patch, llog filesets are not migrated to IAM (might not fit into IAM) and are never deleted from the llog. So, when the IAM is used for filesets (decided by the new nodemap flag), the llog entry is still loaded but no longer applied to the nodemap. 

It could be possible to upcall an "lctl set_param -d" on the nodemap fileset property as soon as the "use IAM" flag is set since it is no longer needed in the future. Nevertheless, the llog entry might still be needed in case of a downgrade, which is why it is kept in this patch. (we may want to delete the llog fileset entry when a nodemap is deleted though?)
