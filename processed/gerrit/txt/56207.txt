LU-18114 nodemap: split "lctl nodemap_*" group into subcommands

Split "lctl nodemap_*" command group into subcommands, e.g.
"lctl nodemap_add" to "lctl nodemap add". And "lctl nodemap_*"
commands in sanity-sec.sh are modified to verify this patch.

Also, the old man pages are modified to reflect the new command
format, and the new man page references for lctl-nodemap_xxx.8 are
added to match the old ones.

Patch Set 7:

I also like separating the nodemap group into subcommands. As Andreas said, I think it would make sense to apply this, e.g., on top of the multi-fileset patch series, as it adds several new commands (not all of them are in the patch series yet, though). 

I'm wondering: In the multi-fileset patch series, we decided on the following syntax since we have several new fileset commands: "lctl nodemap_fileset_{add,del,clear}" etc. This makes sense and also helps with bash auto-completion, but it is inconsistent with the other nodemap commands. 

Since this patch already changes the syntax, perhaps this patch could go one step further and do the same for the other nodemap commands, e.g., "lctl nodemap offset_{add,del}" instead of "lctl nodemap {add,del}_offset" (same for the others)?

--- Message (Andreas Dilger, 2025-01-13 21:50:09.000000000) ---
Patch Set 7:

> Since this patch already changes the syntax,

Well, it is *adding* a new alias for the same commands.  The old commands will still work.

 > perhaps this patch could go one step further and do the same for
 > the other nodemap commands, e.g., "lctl nodemap offset_{add,del}"
 > instead of "lctl nodemap {add,del}_offset" (same for the others)?

Yes, I very much prefer "object_action" command/function names, since it sorts all of the commands/functions together when trying to find (e.g. via tab completion) the operations that can be done on e.g. a nodemap.

I think it is pretty rare that you start with "hmm, I'm going to add something, but I don't know what I want to add", rather than "I want to add a nodemap, but I don't know the exact command to do this (add, create, init, etc.)".

That said, aliasing "add_offset" to "offset_add" could be done in a second patch, since I think this one is already touching a lot of code.
