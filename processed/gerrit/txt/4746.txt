LU-2378 lma: move HSM & SOM attributes to dedicated xattrs

Reduce LMA to its minimum and move HSM & SOM attributes to dedicated
on-disk extented attributes. This is more consistent with the MDS
layering (HSM, LMA & SOM are managed in different layers) and avoids
locking + read/modify/write in many cases.

As far as interoperability is concerned, the old code works just fine
if it reads a smaller LMA size and the new code retries with a
larger buffer on ERANGE error.

Patch Set 1: (4 inline comments)

Of course it will still pass if code writes and reads the same xattr. Real question is whether LMA handling will break if old (larger) LNA is too big to fit into current struct lustre_mdt_attrs?  Alternately, if new (smaller) LMAs are written, then MDS is downgraded, will this cause the old code to consider the LMA corrupted?

I'm not so worried about SOM (which is just a cache) as just normal operations. If the smaller LMA causes problems on downgrade, then we'd need an incompat LMA flag, and backport a fix to allow 2.1.4 to accept the smaller LMA (though it at least needs to contain the FID).  Likewise, the new code should not be upset to see an LMA larger than the current struct size.
