--- Commit Message ---
LU-2487 client: change resource locks type to rwlock

Client deadlock between ll_md_blocking_ast, sys_close, and
sys_open is possible. There are tree pairs of locks
resource,inode; dentry,resource and inode,dentry. The deadlock
is possible in systems that don't use dcache_lock.

This patch changes resource lock type to rwlock so two
resource locks, can't lock because acquired for reading.

--- Message (Andreas Dilger, 2012-12-21 08:01:44.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

(17 inline comments)

I'm not against the use of a rw lock for the resource, since this could potentially improve server performance for the DLM in cases where there are many locks on a single resource.  

However, I don't think the way that it is being used is necessarily safe.  Secondly, if we are adding the overhead of an rw lock, it probably makes sense to look at the other users of lock_res() and lock_res_and_lock() to see which ones could safely be changed to a read lock (i.e. they do not modify the resource).

Some style issues that should be addressed.
