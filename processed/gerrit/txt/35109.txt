--- Commit Message ---
LU-12367 llite: Disable strided readahead

The current strided readahead is partly broken for normal
use, and does not correctly recognize unaligned reads, so
it is actively harmful for the IOR hard read test.

This patch is a quick hack to disable it, which should
result in significantly fewer small read RPCs during that
test.

--- Message (Patrick Farrell, 2019-06-08 03:38:50.000000000) ---
Patch Set 1:

After looking at all of this, in particular the ria_end and RPC size stuff...

I didn't look very closely at the ria_end stuff, perhaps there is an easy fix or perhaps it is not the root of the small RPCs problem.  But assuming the ria_end modification is needed to get good RPC sizes, then I don't see how that can work with strided readahead as currently implemented.

And we have to have:
A low miss rate
The largest possible RPCs (at least the same size as the userspace i/os, no smaller except when split between stripes)

Ideally, we also want a low discard rate, so we're not wasting many pages, but that's less important.

Note that the existing readahead tests in sanity don't catch any of this because they look almost entirely at discards.  If readahead were off entirely, they would pass because discards would be 0.  Similarly, if readahead is missing a page on every read, it's not much use.  But it won't have any discards.

--- Message (Patrick Farrell, 2019-06-10 15:23:36.000000000) ---
Patch Set 1:

Now for longer thoughts...

Good!  Personally, I think for stride detection we should use the i/o, ie, the offset and count that come in from VFS, rather than try to figure out by checking every page of an i/o.  I also wonder if we should observe and store this stride info from the i/o separately.  A problem with the current readahead is that this info is sort of ... embedded in the readahead state, and inferred from that state.

This leads to the problem where tweaking the readahead window to be RPC aligned breaks strided readahead, because the state is all jammed in to one place.

It's true that moving the stride detection up to the i/o layer means we couldn't do strided readahead for mmap, but I don't think that's a big case and I think it's worth it for the simplification it allows.

Patch Set 1:

BTW, I am thinking about (and slowly working on) some improved tests for readahead, because our current ones only worry about discards.  This means that they don't notice when we miss on literally every single read - They only notice if we read ahead pages we don't use.  (Which is less serious than missing a page we needed...)

Obviously we should check for both, but misses are a fatal flaw, (continuous misses rendering readahead useless), discards are an annoyance...
