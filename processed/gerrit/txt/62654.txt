LU-19631 llite: fix EOF handling for EC parity mirrors

Parity components in EC layouts have independent sizes
on OST objects that differ from the file-level EOF.
This causes several issues:

1. File size incorrectly includes parity component
sizes
2. EOF checks prevent reading full parity components
3. Buffered reads are incorrectly constrained by EOF

This patch fixes these issues:

File size calculation:
Modify lov_attr_get_composite() to skip parity
components when calculating file size/kms. Parity
components don't represent user data and shouldn't
contribute to file size.
Blocks and timestamps are still included from all
components.

EOF handling for parity IO:
Add ci_parity_io flag to cl_io to track when IO
targets a parity mirror. This flag is set in
lov_io_mirror_init() when a parity mirror is selected
and inherited by sub-IOs.

Add ci_parity_eof field to cl_io which stores the
calculated EOF for parity components based on RAID
geometry. This is calculated in the LOV layer using
lov_io_parity_eof() based on the data file size and
parity stripe configuration.

The parity EOF calculation uses the data mirror's
stripe count, stripe size, and the parity's dstripe/
cstripe counts to compute where parity data ends.
This allows proper EOF behavior without completely
bypassing EOF checks.

Update EOF checks to use ci_parity_eof for parity IO:
- vvp_prep_size(): Use parity EOF for page checks
- ll_direct_IO_impl(): Use parity EOF for DIO checks

Testing:
Update test 5a to verify both scenarios:
- Reading from empty parity returns 0 immediately
- Reading parity after resync returns parity data
Add test 30b,c,d:
- lfs mirror read returns the correct amount of
  data for EC mirrors.

### The Root Cause

The `struct lov_stripe_md_entry` does not have a field called `lsme_data_id`. The relationship between parity components and their corresponding data components is actually tracked through the `lsme_mirror_link_id` field (a 16-bit link ID), not through a direct data mirror ID reference.

### The Fix

1. **Gets the mirror link ID** from the parity component (`parity_lsme->lsme_mirror_link_id`)
2. **Searches all mirrors** to find the data mirror by:
   - Iterating through all mirrors in the composite layout
   - Skipping parity mirrors (checking `lre->lre_parity`)
   - For each non-parity mirror, iterating through all its components
   - Finding the component whose `lsme_mirror_link_id` matches the parity's link ID
3. **Returns the matching data component's lsme** for use in parity EOF calculations

### Rationale

In Lustre's EC (Erasure Coding) implementation:
- Data and parity are stored in **separate mirrors**
- They are linked bi-directionally through the `lsme_mirror_link_id` field (as documented in the on-disk layout structure `lcme_mirror_link_id` in `lustre_user.h` line 1093)
- This link ID is set during layout creation in userspace (`llapi_layout_comp_add_ec`) and preserved through the LOD and LOV layers
- The parity mirror uses this link ID to identify which data mirror it protects, allowing the parity EOF calculation to use the correct data mirror's stripe geometry

The fix properly implements this link ID-based lookup mechanism to find the data mirror's stripe MD entry needed for calculating the parity mirror's EOF based on RAID geometry.
