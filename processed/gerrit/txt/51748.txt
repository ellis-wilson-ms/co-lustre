LU-16975 lnet: setup all available interfaces

Currently, setting up LNet with socklnd or o2iblnd
only automatically sets up the first network
interface. Unless a user knows to manually setup
the remaining interfaces, their node will experience
subpar network performance. This patch changes the
default behaviour to automatically setup all
available interfaces.

Unfortunately, there does not appear to be a way to
cleanly do this at the LND layer. NI are passed down
from LNet to the LND via lnd_startup to be setup
one-by-one. Hence, this change is in LNet itself.

As for the remaining LNDs, filnd appears to support
only a single interface. gnilnd looks like it would
need a different mechanism to grab and setup all
interfaces. Hence, both are outside the scope of
this patch.

There was some discussion on the ticket itself about whether this was the right default behavior. IMHO, I think this is less wrong than the current default. But it's an open question whether people would find this default more or less helpful than the current default. It's important to note that any user supplied config will still disable this default behavior. I'll probably leave this patch alone until I get a clearly picture of what's the best approach.

I looked into LU-10360 (I recall you mentioned it in some IPv6 presentation). It seems to solve a somewhat different problem (what server NIDs to connect to rather than how to setup client NIDs).

On a side note, it seems strange that this Ethernet/IB specific logic lives in LNet rather than the LNDs or some library. I might look at refactoring some of that LND specific logic out of LNet. It seems to me that code needed by multiple LNDs should be in some library rather than shimmed into LNet, in the style of:
