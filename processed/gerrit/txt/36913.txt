LU-13026 lnet: MR selection of gateway ni

Multi-Rail gateways can only have 1 route pointing to them. Use
the MR selection algorithm to get the best lpni on the MR
gateway to use.

Using the selection algorithm to find the gateway NI, allows us
to use the standard MR criteria: health, preference and credits.

Test-parameters: trivial

There is an overlap between this patch and the patch for LU-12919, but they are not exactly the same. The LU-12919 patch handles the restriction of the router net. This patch does the same but as part of properly MR selecting the gateway ni, in case there is no restriction of the NI.
Look at the change in peer.c and lnet_find_best_lpni_on_net(). These are required to properly do MR selection for the gateway in case you endup passing LNET_NIDNET(LNET_NID_ANY) for the net_id param in the call to lnet_find_best_lpni_on_net()
I think we'll have to settle on resolving the conflict depending on which patch lands first. I'm ok with:
LU-12919 lnet: Fix source specified route selection
LU-12756 lnet: Refactor lnet_compare_routes
LU-12756 lnet: Remove unused vars in lnet_find_route_locked
LU-12756 lnet: Avoid extra lnet_remotenet lookup
So if these land first,then I'll rebase this patch series on them and pick up those changes. Otherwise this burden will fall on you.
This will be the easiest way to handle the conflict, as I don't want to mingle both patch series and removing the overlapping code as it stands today will not make sense in this patch.
