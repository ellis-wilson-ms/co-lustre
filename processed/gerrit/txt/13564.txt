LU-6179 llite: Implement ladvise lockahead

Ladvise lockahead is a new feature allowing userspace to
request extent locks in advance of the IO which will use
them. These locks are not expanded beyond the size
requested by userspace.  They are intended to make it
possible to address lock contention between multiple
clients resulting from lock expansion.  They should allow
optimizing various IO patterns, notably strided writing.
(Further information in LU-6179)

Asynchronous glimpse locks are a speculative version of
glimpse locks, and already implement the required behavior.
Lockahead requests share this behavior.

Additionally, lockahead creates extent locks in advance
of IO, and so breaks the assumption that the holder of the
highest lock knows the current file size.

So we also modify the ofd_intent_policy code to glimpse
PW locks until it finds one it knows to be in use, taking
care to send only one glimpse to each client.

The current patch allows asynchronous non-blocking lock
ahead requests and synchronous blocking requests.  We
cannot do asynchronous blocking requests, because of
deadlocks that occur in having ptlrpcd threads handle
blocking lock requests.

Finally, this patch also adds another advice to disable
lock expansion, setting a per-file descriptor flag.  This
allows user space to control whether or not lock requests
on this file descriptor will undergo lock expansion.

This means if lockahead locks are not created ahead of IO
(due to inherent raciness) or are cancelled by a competing
IO request, the IO requests that should have used the
manually requested locks will not result in expanded locks.
This avoids lock ping-pong, and because the resulting locks
will not extend to the end of the file, future lockahead
requests can be granted.  Effectively, this means that if
lockahead usage for strided IO is interrupted by a
competing request, it can re-assert itself.

lockahead is implented via the ladvise interface from
userspace.  As lockahead results in a DLM lock request
rather than file advice, we do not use the lower levels of
the ladvise implementation.

Note this patch has one oddity:
Cray released an earlier version of lockahead without
FL_SPECULATIVE support.  That version uses
OBD_CONNECT_LOCKAHEAD_OLD, this new one uses
OBD_CONNECT_LOCKAHEAD.

The client code in this patch is interoperable with that
version, so it also advertises OBD_CONNECT_LOCKAHEAD_OLD
support, but the server version is not, so the server
advertises only OBD_CONNECT_LOCKAHEAD support.

Client support for the original lockahead is slated for
removal after the release of 2.12.  This is enforced with
a compile time version test that will remove support.

--- Message (Jinshan Xiong, 2015-02-05 19:51:07.000000000) ---
Patch Set 2:

Both glimpse and AGL lock are async on the OSC layer. The difference is that there is no process waiting for AGL lock so that the I/O can be destroyed at any time after the glimpse RPC is issued. The end result is that the OSC needs extra code to handle - this is why it has a dedicated lock callback osc_lock_upcall_agl() for AGL lock.

For the prototype of lock ahead implementation, we don't need to implement it as AGL way in the first step. Therefore, it would be simpler to wait for the lock ahead RPCs to complete before returning to user space in ioctl() call. This way will work if we make the caller of ioctl() to be a dedicated pthread of the test program.

regarding the contention handled "poorly" - depends what do you define as "poorly". No crashes or hangs or even undue long processing on the server should be happening or it becomes an easy DoS avenue.

How tolerable do we want to be to things like stat(2) returning bad file size? How about O_APPEND writes?

Keep in mind that while your carefully crafted application/library does things right, there are always external actors you do not control: impatient user doing ls (or even reading) output files, background backups/HSM stuff, some other system activities that are not immediately apparent now.

as far as the ofd_dlm - I actually feel like in line with my optimization comment, it will need to undergo a bgger change that will kill a bunch of problems by it.
Instead of processing every lock with an AST (which is somewhat stupid on surface, as I now realize), we probably should coalesce those locks as we add them to the list.
Theb the list processin is (mostly) ok.
Coalescing should be on a per-client boundary.
So basically when you have a file with locks like:

 data - Known size - lock1-client1 - lock2-client2 - lock3-client1 - lock4-client2 - lock5-client3 - lock6-client1

This in the end should produce AST list of no more than 3:
lock6, lock5, lock4
(assuming all three have the lock ahead flag whatever it would be).

Another thing you could do is you could probably send all these ASTs in parallel (at least if the number is below a certain threshold) - so there's no increasingly longer waiting the more clients you have.

--- Message (Andreas Dilger, 2016-07-25 19:50:26.000000000) ---
Patch Set 26:

I agree with Oleg that while the glimpse CB is to the specific *lock* on the client, the size data is actually retrieved from the *inode* that the lock is protecting, so it should get the same size from a callback on any lock held on that resource (excluding the fact that the size itself may be changing).  That means there is no value to sending multiple glimpses to the same client.

Similarly, there is no value sending any glimpses for extents below the actual object size on the OST, since (barring race conditions that will always be present with glimpses) we know the object size is already at least as large as anything the client could return.

So there are two O_NONBLOCK related concerns, and I'll just explain them both to be thorough.

1. (This is the big one...) O_NONBLOCK uses LDLM_FL_BLOCK_NOWAIT to get the nonblocking behavior we have for it today.  That behavior is "don't wait behind group locks, but do cancel other locks".  (Note that O_NONBLOCK group lock requests also use this behavior.)  So that means when I extended LDLM_FL_BLOCK_NOWAIT to get the desired behavior for speculative requests (always fail the request when a conflicting lock is found), I broke the expected O_NONBLOCK behavior.  I corrected this by checking for BLOCK_NOWAIT and NO_EXPANSION in the extent compat code.  (I don't think this can be fixed just on the client.)
2. This caused a second, subtler & less important issue.  Since it's possible to set NO_EXPAND on a file descriptor, that generates non-speculative lock requests with NO_EXPAND set (and not BLOCK_NOWAIT).  These requests have real i/o coming and must work as expected.  But what happens if you open that file descriptor with O_NONBLOCK?  You get NO_EXPANSION and BLOCK_NOWAIT again.  This can be worked around pretty easily on the client by just barring it - NO_EXPANSION and O_NONBLOCK is a bizarre combination, and I thought it was OK to just block it outright.  Vitaly felt no, so we added NO_EXPANSION_IO so we can distinguish between O_NONBLOCK + NO_EXPANSION and lockahead (speculative) locks.

For that matter, we are (IMHO) free to define the Lustre behaviour for O_NONBLOCK as well.  Even sending and waiting for the DLM lock RPC might be considered "blocking", so we could consider changing O_NONBLOCK to behave more like lockahead - send the RPC to the server, but don't wait for the reply, and let the client thread get back to whatever it was doing until the client actually has the lock.

The difficulty is that if O_NONBLOCK is exactly the same as lockahead then it won't cause the server to ever call back conflicting locks, and the O_NONBLOCK client may never make forward progress.  A way around that might be to have the BLOCK_NOWAIT locks send a glimpse to the client holding conflicting (if on a different export and no glimpse has recently been sent) to spur them into cancelling the lock if it is idle.  Hence your introduction of NO_EXPAND_IO.

That said, if we are going to add a third flag here, I'd prefer that the semantics be more clearly separated than NO_EXPAND implying "don't block" and NO_EXPAND_IO.  Something like:

BLOCK_NOWAIT -> send AST on conflict (maybe renamed?)
NO_EXPAND    -> don't expand lock (no other semantics)
SPECULATIVE  -> don't conflict with other locks

This makes it possible to send "expanding speculative" lock requests, for example.
