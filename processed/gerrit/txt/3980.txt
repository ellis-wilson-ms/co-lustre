--- Commit Message ---
LU-1930 build: Back end file system fixes.

Currently Lustre for ZFS requires the zfs development
rpm for its userland support to be installed on the
build machine so we can create the lustre zfs utilities.
What this patch does is allow a user to be able to build
against the zfs/spl source drops as well as the rpms.
A work around is provided so we can point the lustre
build system to were a user can temporary install the
zfs user land headers.

Building --with-[spl/zfs]=path where path is the source code tree then yes you can do 

$ZFS_DIR/lib/libspl/include -I $ZFS_DIR/include

-------------------------------------------------------
The problem is if you use the rpms. In that case the headers for the kernel modules are stored in 

$ZFS_DIR/$SPL_DIR

libzfs/libspl headers are stored in 

/usr/include/libzfs
/usr/include/libspl 

Also this patch handles the case of building lustre on a build box without the zfs-devel rpms installed. Since the user might not have super user access he/she needs to point the build system to another place to where the user land headers can be installed i.e /tmp for example.

Yes its a mess.See https://github.com/zfsonlinux/zfs/issues/453
