(8 inline comments)

the main objection to the patch is that lfsck and oi scrubber are put together, objects which aren't "scrubbed" by  OI are exposed to higher layers, which in turn requires change to the infrastructure (I don't see what this give us in return).

the design and implementation are supposed to be simple if not trivial:
1) OI scrubber is one thing - repairing OI (without which lu can't be used), running totally within OSD in a dedicated thread
2) LFSCK is another thing, running in a different thread, basically not relying on OI scrubber
3) to avoid extra IO we can synchronize both threads within OSD - for that we don't need any cache and/or fifo, just let LFSCK block till supposed inode is scanned by the scrubber and let the scrubber know the last inode returned to LFSCK by the iterator (so that the scrubber doesn't go too far)
4) OI scrubber should not be using lu infrastructure - that one is relying on a fid, which we *are* recovering
5) things like  osd_scrub_register() - why do we need this at all? 

in general, please keep the thing as simple as possible.

it would be good if we get this broken into few patches, like:
1) OI scrubber itself (should be functional without changes to other components)
2) OI verification/repair from within lookup-by-name
3) OI iterator
4) OI scrubber to run in sync with OI iterator
 
also, please rebase the patch against http://review.whamcloud.com/#change,1833 which is about to land.

According to our former discussion, there will be single dedicated thread in OSD layer for all LFSCK processing. Such thread will scanning the inode-table and feed the LFSCK components (including OI scrub) in OSD layer, and put the scanned object into some cache, which can be reused by up layer LFSCK iteration, it is equal to pre-fetching for up layer LFSCK iteration. 

Under mode case, the speed of OSD layer LFSCK thread is controlled by up layer LFSCK iteration through cache space.

But under urgent mode, like MDT restored from file-level backup, the OI files should be rebuild ASAP, then the OSD layer LFSCK thread will drive LFSCK components in OSD layer as fast as possible. Under such case, filling cache will be skipped, the up layer LFSCK iteration should scanning the device by itself.

Above mechanism is clear and agreed by all of us. Adding more OSD layer threads can help very little for the LFSCK processing, but introduce some trouble for the communication among those threads.

On the other hand, I still think OI scrub is part of LFSCK, it is one of the LFSCK components in OSD layer, although it is useless for OSD-zfs. So it should be driven under LFSCK framework.

My current patch implement a general framework for kinds of LFSCK components, OI scrub is just one of the LFSCK components in OSD layer. We allow more LFSCK components to be added into OSD layer. That is why using osd_scrub_register().

As for lu infrastructure, the OI scrub thread needs to use it, because it needs to mark related objects as "scrub" to tell MDT service threads to return "-EINPROGRESS" to client for those object accessing.
