LU-12668 lov: batch EC data reads per stripe

The EC degraded read path in lov_ec_read_data_pages
submitted one synchronous cl_io_submit_sync per page
per stripe -- ~12,000 individual round trips for a
64MB file with 4+2 EC and 1MB stripes. It also
recomputed the GF inverse matrix and ISA-L encoding
tables on every single page (~4,000 times).

Add lov_ec_read_stripe_pages which reads all pages
for one data stripe across all page positions in a
recovery group in a single cl_io_submit_sync call.
The caller invokes this once per stripe (3-4 calls)
instead of once per page per stripe (~3,000 calls
per RG). This uses cl_io_submit_sync directly, so
the page lifecycle is correct by construction.

Split lov_recover_data into lov_ec_recover_init
(once per RG) and lov_ec_recover_page (per page)
to avoid redundant GF matrix computation.

Context switches drop from ~16,500 to ~47 and
degraded read throughput goes from 10 MB/s to
722 MB/s for a 64MB file, matching normal read
speed (612 MB/s).
