--- Commit Message ---
LU-13132 osd: osd-zfs to cache dbufs for llog objects

working set for llog objects is tiny and very predictable. osd-zfs
can cache couple dbufs (first block storing the header and last
block for new records).

for sanity/60a (llog test) it gives 5939307 hits and 5776 misses
while average osd_write() goes down from 1.09 usec to 0.27 usec,
total time for sanity/60a: before - 153s, after - 101s.

this approach can be used in few other cases like last_rcvd.

(2 comments)

Based on the description in the commit comment it sounds like the dbuf is staying resident in the cache (which is good) but the dbuf hash lookup itself is slowing things down.

Normally a dbuf is held only long enough to update its contents so this is a little bit strange.  However, it looks to me like this still should be correctly handled as long as Lustre always updates this dbuf when modifing the object.  A copy of the buffer will be made when when it's written out.  The extra hold will also ensure the dbuf always stays cached.

Where this might be a minor issue long term will be with the upcoming zfs-2.2 release.  In that release we're integrating support to allow force exporting a pool which has been suspended and can't be resumed.  For that to work we need to be able to discard any inflight I/O and then make sure all the holds are dropped so we can tear everything down.
