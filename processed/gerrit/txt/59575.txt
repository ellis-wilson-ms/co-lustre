LU-0000 lov: fix immediate mirror direct write crash

Fix kernel crash that occurs when doing direct writes to immediate
mirror files. The crash was caused by an assertion failure in
lov_io_mirror_write_intent() at line 245:

  LASSERT((lov_flr_state(obj) == LCM_FL_WRITE_PENDING));

The issue was that immediate mirror files are created with a combined
FLR state of LCM_FL_RDONLY | LCM_FL_IMMEDIATE_MIRROR (0x5), but the
assertion was checking for exact equality with LCM_FL_WRITE_PENDING
(0x2).

The fix changes the logic to:
1. Check if the RDONLY bit is set using bitwise AND instead of exact
   equality, which properly handles immediate mirror files that have
   the RDONLY bit set along with other flags.
2. Update the final assertion to check only the base FLR state bits
   (0x3 mask) rather than the full flag value.

This allows immediate mirror files to properly use the write intent
path, which is the correct behavior since they are created in RDONLY
state and need write intent RPCs to transition to WRITE_PENDING state
for actual writes.

The crash was reproducible with:
  lfs mirror create -N2 --immediate test.txt
  echo "data" > test.txt  # Would cause kernel crash

After the fix, direct writes to immediate mirror files work correctly
without kernel crashes.

Do we need the IMMEDIATE_MIRROR flag to be part of the FLR mirror state, or is this "merely" a flag on the layout that tells the client whether it should be writing data to the STALE mirrors?

At the surface level, it isn't clear if that needs to be part of the FLR state machine, and as you see here this can cause clients to crash.  For an old client that doesn't understand IMMEDIATE_MIRROR, I would assume that the MDS will not get the right "lfs mirror resync" state transition on CLOSE, so it wouldn't un-STALE the mirror copies, and all is well...
