LU-4112 obdclass: no yield in cl_lock_free() to avoid deadlock

If an async brw completion is being handled (and rescheds owning
cl_loi_list_lock!) on the same core/cpu than a thread also posting
an I/O and thus requiring/spining-on cl_loi_list_lock.

This problem can only occur on versions where changes for LU-1030
(ie, "New IO Engine" known to "Reduce loi_list lock contention" with
introduction of a "per object lock") have not already been integrated.

It should also not occur on platforms where cl_loi_list_lock is a
blocking mutex (MacOS?) rather than a spin-lock (Linux).
