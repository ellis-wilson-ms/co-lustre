LU-1512 oi: reuse idle OI blocks

With the system running, some OI blocks may become empty because
related files have been removed. Originally, these empty blocks
almost cannot be reused beause it always allocates new block for
new leaf or index node. Such space waste is an important reason
for the OI file size increasing quickly.

Now, when the non-released OI leaf node becomes empty, it will be
recycled into a per-OI based idle blocks pool, which is organized
as a series of tables. For each table, the on-disk format is:

+---------+---------+---------+---------+------+---------+-------+
|  magic  |  count  |  next   |  logic  |      |  logic  | free  |
|(16 bits)|(16 bits)|  table  |  blk #  | .... |  blk #  | space |
|         |         |(32 bits)|(32 bits)|      |(32 bits)|       |
+---------+---------+---------+---------+------+---------+-------+

The logic blk# for the first table is stored as "idle_blocks" in
the OI root node. In fact, there is padding field in the OI root
node after the "dx_countlimit". We will append the "idle_blocks"
just after the "dx_countlimit" to reuse part of the padding. So
it will not cause compatibility issues with old OI files.

Adjust the strategy for OI file block allocation: reuse empty block
in its idle blocks pool with priority; if such pool is empty, then
allocate new block from system volume.

Patch Set 5: I would prefer that you didn't submit this

Sorry, this patch cannot be used as-is. To change the on-disk inode format for all ext4/ldiskfs inodes just to fix this problem in IAM is not acceptable. I believe this field is already used in newer kernels, and even if it wasn't yet, it definitely would be in the future. This would cause serious compatibility problems for ldiskfs down the road. 

Is there a field in the IAM header that could be used instead?  Would it be acceptable to just punch the free blocks from the file, making it sparse, and allocating new blocks at the end?  The concern with this would be that the file could hit the 16TB file size limit too quickly and again cause problems. Do we even have a good understanding about the rate of growth for the IAM file (e.g. 128kB/million files) to know how quickly this will be a problem on current systems?

--- Message (Fan Yong, 2012-08-04 17:22:37.000000000) ---
Patch Set 5:

> Is there a field in the IAM header that could be used instead? Would it be acceptable to just punch the free blocks from the file, making it sparse, and allocating new blocks at the end? The concern with this would be that the file could hit the 16TB file size limit too quickly and again cause problems. Do we even have a good understanding about the rate of growth for the IAM file (e.g. 128kB/million files) to know how quickly this will be a problem on current systems?

OK, I will reuse some idle space in the OI file just after the header, which make us can keep on-disk inode layout unchanged and also compatible with old 2.x releases.

As for punching free blocks from the OI file, it may be implementable, but I do not think shrinking htree will be simpler than our current solution, since ext4/ldiskfs do not support such mechanism also.

According to current OI implementation,  each htree index node can contain entries count: (4096 - sizeof(header)) / ((sizeof(FID) + sizeof(__u32))), is about 200.

The OI htree can contain at most 5 levels index node. So the total leaf nodes count: 200 ^ 5, is about 320G. 

For htree leaf node, each leaf node can contain OI mappings counts: (4096 - sizeof(header)) / (sizeof(FID) + sizeof(ino) + sizeof(gen)), is about 170.

Usually, the leaf node is in half full status, means each leaf contains about 85 OI mappings.

So for one OI file, it can contains OI mappings count: 320G * 85.

If the create speed is 50K/sec, then the OI file will be full in the time: 320G * 85 / 50K, is about 544M seconds, is about 400 years.

But consider the max file size limitation 16TB, then one OI file can contain leaf nodes count: 16TB / 4KB / 2, is about 2G. If create with the same speed as above, fill the OI file will take about 3.4M seconds, is about 2.5 years.
