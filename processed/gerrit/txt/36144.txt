--- Commit Message ---
LU-12275 sec: encryption for write path

First aspect is to make sure encryption context is properly set on
files/dirs that are created or opened/looked up.
Then encryption itself is carried out in osc_brw_prep_request(), just
before pages are added to the request to be sent. Because pages in
the page cache must hold clear text data, we have to use bounce pages
for encryption. The allocation is handled by fscrypt, and for
deallocation we call fscrypt_pullback_bio_page() and/or
fscrypt_pullback_bio_page().

The different flavor definitions are:

krb5n/skn
RPC message protection: null
Bulk data protection: checksum

krb5a/ska
RPC message protection: partial integrity (header only)
Bulk data protection: checksum

krb5i/ski
RPC message protection: integrity
Bulk data protection: integrity

krb5p/skpi
RPC message protection: privacy
Bulk data protection: privacy


So it seems there is no such thing that would allow encrypting RPC traffic and leave bulk data unencrypted.

Some minor cleanups that could be done in a later patch.

One potential issue is how to handle the addition of filename encryption in the future?  AFAIK, the LMAI_ENCRYPT flag is set on the directory inode, but in this release we don't have filename encryption so those filenames will not be encrypted.

In the next release, once filename encryption is available, all filenames will be encrypted, which will cause problems for the unencrypted filenames in directories marked with LMAI_ENCRYPT.  We will need a separate LMAI_ENCRYPT_NAMES flag to determine if the filenames in a directory are encrypted or not.

That is not specifically a problem with this patch, but something to understand before this feature is landed to 2.14 and the filename encryption is added to 2.15.  Do you already have a plan on how filename encryption will be handled with DNE?

(4 comments)

In order to handle name vs content encryption in future releases, I think the right approach is to put a restriction on the encryption policies assigned to files and directories.

With fscrypt, encryption policies are the natural place holder to specify how we want to encrypt: which alg for content, and separately which alg for file/directory names.

So what we could do here for 2.14 is to force null encryption for names in encryption policies (accompanied with a clear warning message). In 2.14, this field in the encryption policy structure is naturally ignored, because we do not implement name encryption.
In 2.15, when name encryption will be implemented, all directories and files created with 2.14 will have a void encryption alg for names, so these directory/file names will remain untouched. And directories and files on which a new encryption policy is assigned, will have their names encrypted (as long as the policy specifies that).

I will push a new patch in the series, after #37673, to force null encryption for names in encryption policies.
