--- Commit Message ---
LU-9372 ptlrpc: allow to limit number of service's rqbds

This patch provides a way to limit the number of rqbds per
service.
This should help to avoid OOM during heavy clients
requests load, like during target failover/recovery for
thousands of Clients.
This change has been required, even after first patch for
LU-9372 (ptlrpc: drain "ptlrpc_request_buffer_desc" objects)
which already allowed to drain unused rqbds previously
allocated during heavy load, but was not efficient during
too long period of load.

It isn't clear what the benefit of this patch is, if it is off by default, since most users will not even know it exists.  However, there probably isn't a good way to set a default value for the maximum number of rqbd buffers, since that depends greatly based on RAM and client load.

--- Message (Andreas Dilger, 2017-12-16 17:08:16.000000000) ---
Patch Set 3: Code-Review+1

I think this patch is OK, but doesn’t really go far enough to solve the problem for most users. Having some default limit would be an improvement. 

Reducing the rqbd memory usage by making a deceicated slab could be useful, if it gets us some efficiency. That is hard when the allocations are already so large. 

Also, it isn’t clear why the client needs so many RPCs in flight during recovery? Having 2 per client would be enough to make good progress without overloading the server.
