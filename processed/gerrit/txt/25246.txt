--- Message (Andreas Dilger, 2017-06-05 21:27:57.000000000) ---
Patch Set 4: Code-Review-1

James, is there a reason not to put all of the Lustre headers under a "lustre" subdirectory in the include/linux directory?  That would allow both old and new applications to use

 #include <lustre/lustre_user.h>

and build by adding "-I /usr/include/linux" to the compile flags, if it isn't already there, without having to change the code based on the Lustre version and/or add configure checks to see whether they should use <linux/lustre_user.h> or <lustre/lustre_user.h>.

As it stands, this patch will break all userspace applications (e.g. MPICH Lustre ADIO driver, IOR, out-of-tree HSM copytools, etc.) that currently include "lustre/lustre_user.h" and/or "lustre/lustre_idl.h".

--- Message (James Simmons, 2017-06-07 22:59:37.000000000) ---
Patch Set 4:

I asked Greg about having /usr/include/linux/lustre but he said no. So we are stuck. The way around this is to create wrapper headers like liblustreapi.h and give user land applications time to migrate to /usr/include/linux.

--- Message (Andreas Dilger, 2017-06-08 08:45:59.000000000) ---
Patch Set 4:

> I asked Greg about having /usr/include/linux/lustre but he said no.

What was the reasoning given?  I'd think that "this breaks userspace apps" would be a reasonably convincing argument, along with "then we won't have lots of Lustre headers directly in the "linux" directory.    There are already a bunch of subdirectories there (e.g. nfs/, sunrpc/, raid/, etc) and IMHO this is an improvement over putting all of the headers directly into linux/.

 > So we are stuck. The way around this is to create wrapper headers
 > like liblustreapi.h and give user land applications time to migrate
 > to /usr/include/linux.

How does that work with in-kernel clients?  We would have to install a wrapper header /usr/include/lustre/lustre_*.h for each header file, pointing to /usr/include/linux/lustre_*.h but those wrappers would only exist when building against out-of-tree clients, not in-kernel clients.

--- Message (James Simmons, 2017-06-08 20:08:59.000000000) ---
Patch Set 4:

See this thread https://www.spinics.net/lists/linux-driver-devel/msg91394.html

BTW we still need wrappers for external applications since current it is expected to see the headers in /usr/include/lustre not /usr/include/linux or /usr/include/linux/lustre if that is accepted.

--- Message (Andreas Dilger, 2017-06-09 03:30:58.000000000) ---
Patch Set 4:

> BTW we still need wrappers for external applications since current
 > it is expected to see the headers in /usr/include/lustre not
 > /usr/include/linux or /usr/include/linux/lustre if that is
 > accepted.

If the #include directives are still using <lustre/lustre_*.h> then it is relatively simple from the application POV to just add an extra "-I /usr/include/linux" to the compile options and this is totally transparent/harmless regardless of the Lustre version being built.

--- Message (Andreas Dilger, 2017-06-10 04:13:41.000000000) ---
Patch Set 4:

Reading that thread, it didn't appear that Greg was adamantly against having a separate "lustre" directory, seemed more like a suggestion.  I think if you point out that userspace applications (not under our control) are already using <lustre/lustre_*.h> and keeping this convention will avoid significant breakage of the apps.
