LU-1866 lfsck: ancillary work for namespace LFSCK

1) Share related linkEA functions with LFSCK.

2) Multiple linked objects are rare, and the objects with
   a lot of linkea entries are more rare. It is unnecessary
   to worry about performance under such cases. So drop the
   linkea entries count limitation: LINKEA_MAX_COUNT.

--- Message (Fan Yong, 2013-01-29 06:48:49.000000000) ---
Patch Set 18:

There are some trouble for pack all modifications into single transaction. We cannot know which modifications should be done before the trans stared. For example, the object has three invalid linkEA entries, then we need declare three times for the transaction, otherwise, it will trigger OSD layer assertion in the future. But during the transaction start, the invalid linkEA entries may become four, then the solution is that we need to obtain the lock on the object, and re-check how many invalid linkEA entries it has, if it does not match the old one, release the lock, stop the handle, and repeat. All the description is simple, but the code will be very ugly.

I wonder whether we can enhance the existing do_{read,write}_lock to replace the mdd rwsem introduced in my patch? Such as call "mdd_write_lock(env, obj, MOR_TGT_LFSCK);" before transaction start? Which will much simply our work.

--- Message (Alex Zhuravlev, 2013-01-29 08:29:18.000000000) ---
Patch Set 18:

as per discussion with Fan Yong: existing locking primitive should be used and either we start transaction first (cheap with ldiskfs, probably not so with zfs) or we repeat the check having transaction created and locks acquired.

sanity checks in ldiskfs-osd should not be considered as a constraint here: one declaration for big enough xattr set should be enough for arbitrary number of actual xattr sets.
