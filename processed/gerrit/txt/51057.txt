--- Commit Message ---
LU-16836 lnet: ensure dev notification on lnd startup

Look up device and link state on lnd startup so that
the initial NI state may be set properly.

Reduce code duplication by adding lnet_set_link_fatal_state() and
lnet_get_link_status() functions which are shared across LNDs.
LND-specific versions of these are removed.

This fixes the issue with adding LNet NI using an interface with
cable unplugged which results in the NI state initialized as "up".

Fixes: da230373bd ("LU-16563 lnet: use discovered ni status")

--- Message (Neil Brown, 2023-05-23 22:42:02.000000000) ---
Patch Set 2:

Thanks .... though it doesn't seem to help me as much as I hoped.

If I ignore the cosmetic changes there seem to be two things happening in this patch.
One is that the notifiers are registered (or reregistered) for every 'startup' call, instead of just the base_startup call.  I can see that sends NETDEV_REGISTER and possibly NETDEV_UP for every device.   That seems a bit heavy handed when there is only one device that should be relevant for the newly added ni.  Don't we know which device the ni is on?  Can't we just extract the up/down status and act on it?

The patch also adds a few tests for "ni_state == LNET_NI_STATE_ACTIVE" which aren't mentioned in the commit message.  Presumably they interacts with the new registration notifications in some way to ... make sure that devices don't appear "up" when there are unplugged??

Also, I don't see the point of the new kib_reg_notifiers flag.  It seems to be false precisely when kiblnd_data.kib_init == IBLND_INIT_NOTHING, so I don't see why we need it.
Same for ksocknal_data.ksnd_reg_notifiers
(and if they really are needed, I find the name confusing.  It seems to say "We need to register the notifiers" but it really mean "we have registered the notifiers", so ksnd_notifiers_registered" would work better for me).
