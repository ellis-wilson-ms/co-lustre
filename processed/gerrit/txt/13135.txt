(1 comment)

Thank you for looking at this prototype patch and the advise for limiting the testing. So to answer your questions here are the answers:

Protocol change: Yes at the LNET layer. I plan to add a new lnet_msg_type_t of LNET_MSG_PUT_CKSUM and LNET_MSG_GET_CKSUM and possible LNET_MSG_REPLY_CKSUM. The default behavior should be check sum being disabled. Interoperability will be apart of my testing.

How this interfaces with the LND checksums: It doesn't. Its meant to replace the LND driver checksum code. The LND drivers check summing work will be removed in a separate patch.

Why not LND driver checksums: The main reason to avoid this is that LND drivers do check sums for peer to peer. So in the case of client -> router -> server. The checksum is created on the client then validated at the router then sent to the server to revalidated again. In the case of our FGR set up we have layers of routing so it gets expensive to do this approach. Lastly in our setup our routers
don't support  hardware crc32c so the expense of router checking is far more costly than on the client and servers. As a note none of the LND driver support
the same set of checksum algorithms. In our case you can't do a straight gni0 -> o2ib checksum test. So not only do you have to do a checksum revalidating on each router you need to create a new checksum of a different algorithm each time. The goal here is to do it at the NI interface so in that case we create a checksum at the client and then send it over to server to revalidated using the
same checksum algorithm. 

For the last question of why another layer doing check summing. I explained above the advantage of LNET checksumming over LND. The reason to to it in the LNET layer versus ptlrpc is three fold. First ptlrpc only does check summing for bulk data. Currently it is possible to send corrupt metadata to the servers which we would never know. Second vendors are creating products that are layered on the LNET layer like Cray's DVS. This data needs to be check summed as well to avoid data corruption. Lastly currently ptlrpc only transmits 32 bit digest values. This is okay for crc32 but for algorithms like sha512 it needs to generate a value much larger than 32 bits which will not fit on the wire. If lustre is going to properly support encrypted data over the wire it will need to support check sum algorithms that are much stronger than crc32. For that case the digest created will be greater than 32 bits.
