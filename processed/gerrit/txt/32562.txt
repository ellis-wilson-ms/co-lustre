--- Commit Message ---
LU-11057 obd: check '-o network' and peer discovery conflict

"-o network=net" client mount option is not taken into account
when LNet dynamic peer discovery is active.
Check if LNet dynamic peer discovery is active on local node. If it
is, return error if "-o network=net" option is specified.

This patch will have to be reverted when the incompatibility between
"-o network=net" client mount option and LNet dynamic peer discovery
is resolved.

--- Message (Andreas Dilger, 2018-07-17 20:58:52.000000000) ---
Patch Set 1:

> Patch to add test for '-o network' client mount option is at :
 > 
 > https://review.whamcloud.com/32590

I've rebased that patch on top of this one, and removed the error_ignore calls, so that it can verify that this patch is actually fixed. 

> The problem is in the case when Dynamic Peer Discovery is not
> enabled on the client from where we are mounting, but enabled
> on the servers this client is talking to. In this case the
> '-o network' option will not work, because the servers will
> still use the NIDs they want to communicate with the client.
> This is why I would like to display a warning on the client
> every time the '-o network' option is used: the option given
> on the command line could be silently ignored by LNet.

Unfortunately, human nature is such that people will also ignore the warning message if it is printed on the console every time the client starts up with the "-o network" option, and it just makes it harder to see other messages that may be more important.

It is only useful if a message is printed when there is an actual problem.  Can this be done in some other way, being more verbose if a server sends a message to the wrong NID?

--- Message (Sebastien Buisson, 2018-08-10 12:55:03.000000000) ---
Patch Set 1:

Carrying out some more tests, I realized that the exports list as seen under /proc/fs/lustre/*/$fsname-*/exports can be misleading when LNet dynamic peer discovery is on. Indeed, the 'name' used for the export does not assume anything on the actual connection used.

So it turns out the '-o network' mount option is taken into account as soon as LNet dynamic peer discovery is deactivated on the local client node. It makes the conflict check much simpler.
