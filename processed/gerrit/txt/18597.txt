--- Commit Message ---
LU-5070 utils: rename filesysetem fsname

Introuduce new tunefs options "--rename" for renaming the existing
Lustre filesystem fsname with configuration/parameters preserved.
Usage: tunefs.lustre --fsname $NEW_FSNAME --rename $OLD_FSNAME ...

NEW_FSNAME is the new fsname, must be specified.
OLD_FSNAME is the old fsname, it is optional if the MGS and the MDT
are combined together.

It shares the functionality of fork/erase Lustre configuration logs
in kernel.

Alex, I don't think that mounting the MGS will solve any problem here.  While this would allow the config llogs rename to use the in-kernel code, it won't help for the MDT and OST filesystems, since they need to be unmounted to change the config logs, filesystem label, mountdata, and last_rcvd file.

I suppose it would be possible to mount the MGS and do the rename in the MGS config logs, and just delete the MDT/OST config logs, and only rename the mountdata and last_rcvd on the local mount, and filesystem label before or afterward.

That would reduce the complexity of the userspace tool considerably, but then it would depend on the MGS module changes to work.

--- Message (Fan Yong, 2016-04-12 03:57:00.000000000) ---
Patch Set 1:

My current idea is that we can use the existing "lctl fork_lcfg" to implement the functionality of rename the filesystem name and avoid write corruption issues as following:

1) start the MGS (if it is not started), and run "lctl fork_lcfg $old_fsname $new_fsname", then it will generate new config logs on the MGS.

2) umount the MDTs and OSTs, run "tunefs.lustre --rename" on each target (MDT or OST), the will change the filesystem label, mount data, last_rcvd file, and remove the old config logs. Please note, it will not generate new config logs.

3) mount the MDTs and OSTs, then they will fetch the new config logs from the MGS, that is equal to be renamed as new filesystem name.

4) We can run "lctl erase_lcfg $old_fsname" on the MGS to remove the old config logs if we do not want to rename it back.

With above steps, there will be no user space config logs parse. So neither code duplication nor write corruption trouble during write/rename config logs. So whether it is acceptable solution or not?
