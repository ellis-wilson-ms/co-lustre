--- Commit Message ---
LU-10406 osp: cache xattr properly

For performance consideration, the OSP may cache some xattr
for remote object. But we only can cache xattr for some cases:

1) We hold related XATTR ldlm ibits lock (MDS_INODELOCK_XATTR),
   that will guarantee that the cached xattr can be invalid if
   someone updates the xattr from other MDT (under DNE). Or

2) The xattr is for OST object. Because the OST-object's xattr
   only can be changed either by the OST itself or via the MDT
   on which its parent MDT-object resides. Currently the LFSCK
   is the main user of caching OST-object's xattr (PFID EA) on
   MDT for performance. Since the OST only can set the PFID EA
   as right value when write the OST-object for the first time,
   the cached stale (empty) PFID EA on the MDT will cause LFSCK
   to repair related PFID EA again, no other fatal side-effect.

Another MORE important facet is that: do_invalidate() cannot resolve some race cases because the bad cache control mechanism. For example:

LFSCK engine (user1) gets some object's EA from remote MDT without ldlm lock, according to your logic, before it calls do_invalidate(), such EA will be cached in OSP. At that time, another user (user2) changed the EA via remote MDT, but current MDT does not know that, still cached the stale EA. And then the third user (user3) obtains ldlm lock on current MDT, then call dt_xattr_get() for such EA. Because the LFSCK engine (user1) is still using such EA without calling do_invalidate() yet, then the user3 will get the cached but stale EA directly. That is wrong, but the user3 cannot realize that.

The key issue is that there is interval between caching EA without ldlm lock and the lockless user calling do_invalidate(). The solution is to remove such interval. My patch just does that.

It may be not the best solution, but it works well, at least better than do_invalidate(). Please do re-consider.

The important facet is that: do_invalidate() cannot resolve some race cases because the bad cache control mechanism. For example:
LFSCK engine (user1) gets some object's EA from remote MDT without ldlm lock, according to your logic, before it calls do_invalidate(), such EA will be cached in OSP. The key issue is that there is interval between caching EA without ldlm lock and the lockless user calling do_invalidate(). The solution is to remove such interval.

It only exists in theory, and more rare than the race conditions I mentioned on the local MDT. Let's analysis the operation sequence:

1) The LFSCK (user1) trigger get_xattr RPC on MDT0, such RPC arrives MDT1 and got old EA, before the RPC replied, then:

2) Another thread (user2) takes ldlm lock the target object on the MDT1 and modified the EA, then the EA given user1 become stale.

3) The third thread (user3) trigger ldlm_enqueue RPC on the MDT0 for the same target. Such RPC arrives the MDT1 and cancel the former ldlm lock taken by user2. Such cancelling will trigger OSP cache invalidation on MDT1. After done, the ldlm lock is granted to user3, and then replied to the MDT0
