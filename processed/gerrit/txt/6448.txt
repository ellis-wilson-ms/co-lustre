--- Message (Shuichi Ihara, 2013-05-25 02:20:08.000000000) ---
Patch Set 1:

OFED changed a lot. The idea is based on compat-driver project which means in general, using kernel upstream kernel, then back port to specific old kernel or distribution.

The challenge is "how avoid same funciton that distribution already defined". Even the kernel mainstream doesn't have, Linux distribution has own backport. (e.g. RHEL6, the base kernel is based on 2.6.32, but Redhat did a lot of backports from newer kernels).

This is like OFED uses mainstream kernel vs Redhat has own backport patches.

Since OFED (3.5), they support RHEL6.3 and checking few functions and macro whether these are already defined or not. 
(please see "Backport compat to RHEL6.3" on http://git.openfabrics.org/git?p=compat-rdma/compat.git;a=summary)

So, lustre doesn't need any additional define for each specific kernel or distro. Because, OFED checked itself.

However, RHEL6.4, they are more backports RHEL6.3 did.
(See "Added RHEL6.4 support" on http://git.openfabrics.org/git?p=compat-rdma/compat.git;a=summary)

And, it seems OFED made a special macro "CONFIG_COMPAT_RHEL_6_4" that is subset of all functions or macro OFED wants to ignore since RHEL6.4 kernel has.

I believe if we want to compile something (not only lustre) with OFED-3.5.1 against RHEL6.4 kernel, we have to have CONFIG_COMPAT_RHEL_6_4.

Yes, we can have this option at the compile time, like this.

# EXTRA_LNET_INCLUDE="-DCONFIG_COMPAT_RHEL_6_4" ./configure --with-o2ib=/usr/src/compat-rdma --with-linux=/usr/src/kernels/2.6.32-358.6.1.el6_lustre.x86_64

But, my patches, build checks kernel version and if it's RHEL6.4 kernel, set CONFIG_COMPAT_RHEL_6_4 for automation.
