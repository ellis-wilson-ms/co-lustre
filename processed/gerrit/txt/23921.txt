LU-9066 ldlm: don't evict client on umount if AST fails

When server is being stopped the AST send is denied and
that AST failure will lead to the unexpected client
eviction.
Patch adds extra checks in ldlm_handle_ast_error() to
distinguish that type of error and proceed without
eviction.

--- Message (Vitaly Fertman, 2016-12-01 19:33:10.000000000) ---
Patch Set 3: Code-Review+1

whereas the patch looks correct now, please clarify what is the goal of it - performance or cleanup?
regarding the performance I am not sure there will be a large gain, because:

1) only conflicted locks on a particular file are scheduled for BLAST, not all the locks on this export;

2) the export is being handled in background and it is just a matter of seconds to get rid of all of them;

3) having an export failed, there is no actual RPC send and therefore no special long delays, it is just a matter to not moving these conflicting locks to another list; i.e. not incredibly much per a lock;

having said this, if we are talking about a performance, it is worth to land such patches only after confirming by some workload we see ~5% benefits.

--- Message (Mikhail Pershin, 2016-12-02 08:33:06.000000000) ---
Patch Set 3:

(1 comment)

Vitaly, I've experienced some problems when this happens, so it is not just a message but after it something goes wrong. I see two problems here:
1) AST work processing is aborted in the middle when such error happens, so if there is several items in the list then rest of it will not be processed though they may belong to other healthy export. Now this case returns -ERESTART so it can proceed just skipping outdated lock.
2) for lock from disconnected export the class_fail_export() will be called, so export will became failed in the middle of disconnection process. This can be the reason for LU-8359 for example.

So this is more cleanup/fix patch I'd say. Correct me if I am mistaken.
