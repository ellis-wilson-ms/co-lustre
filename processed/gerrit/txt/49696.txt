LU-16494 fileset: check fileset for operations by fid

Some operations by FID, such as lfs rmfid, must be aware of
subdirectory mount (fileset) so that they do not operate on files
that are outside of the namespace currently mounted by the client.

For lfs rmfid, we first proceed to a fid2path resolution. As fid2path
is already fileset aware, it fails if a file or a link to a file is
outside of the subdirectory mount. So we carry on with rmfid only
for FIDs for which the file and all links do appear under the
current fileset.

This new behavior is enabled as soon as we detect a subdirectory mount
is done (either directly or imposed by a nodemap fileset). This means
the new behavior does not impact normal, whole-namespace client mount.

sanity test_421h is added to exercise this new capability.

--- Message (Sebastien Buisson, 2023-01-23 16:45:59.000000000) ---
Patch Set 1:

It is true that the ability to change the setting is accessible to userspace. But given that the parameter llite.*.fidops_check_fileset can be modified by root only, is it really a weakness? If a malicious user is gaining root access, then they can modify and recompile their own Lustre modules, getting rid of the fid2path check for instance.
The fact that the fidops_check_fileset tuning is per client mount makes it possible for admins to activate it only on selected subdir mounts.

Alternatively, as you suggest I could use a new nodemap property combined with an obd connect flag (because nodemaps are not exported to clients).

But I think calling fid2path for rmfid is hurting performance and more importantly preventing the original idea of a way to remove files while skipping full path traversal. So it should not be activated by default, and only enabled for very specific use cases such as subdirectory mount if admins decide this is important.

In the current implementation, if we detect that for a given fid at least one file is outside of the subdirectory mount, then we block rmfid for this fid. So even files inside the subdir are not removed. If we want to refine that behavior by proceeding to deletion of files only in the subdir, and leave the others with same fid untouched, then I have the impression that it is going to require a protocol change. Today, server side receives a fid, and carries out deletion of all files that have this fid. So client side would need to pass record numbers along with fids, so that server can filter the hard links.
