--- Commit Message ---
LU-16696 llite: Detect and adjust memcg settings

In case of memcg restrictions ensure min is non zero.
A min of 0 suggests a broken memcg configuration.
For users of cgroups-v1 the API does not provide a
method of setting min, low and high.

During buffer writes check if a memcg restriction
is enabled with a min of 0. If found adjust the settings:
    min (10% of max)
    low (20% of max) and
    high (80% of max)
to avoid Memory CGroup oom.

This does fix the oom. The correct fix is a kernel patch, imo. Since a kernel fix enables the entire memcg to avoid oom.

Running under a memcg, or not, is expected to be opaque to the filesystem. It should behave the same as booting up a VM with a small amount of ram allocated. Lustre (or any filesystem) should not have to know or care about VM vs memcg).

Performance is expected to improve with adding more memory *if* the process is memory constrained. If performance scales with memory (and it does with this hack) then that seems like a fix.

Give the above, I am sure there are things that can be done to improve running under low memory that are lustre specific ...

Unfortunately this memcg issue is present in linux from 4.18 to current (6.3) and for sites that cannot, or will not, shift to memcg-v2 where low/min/high can be modified via sysfs this is *a* workaround (it could be repackaged as a trivial kernel module to verify if this solves an issue at a particular site if a kernel/lustre fix cannot deployed quickly enough).

min,low, and high are available in cgroup-v2 api, just not in the cgroup-v1 api presumably because those values did not exist in cgroup-v1 prior to v2.

The current problem is that min,low are critical to getting sane behavior (aka memory protection) under memcg. Breaking memcg-v1 by freezing the old interface seems to only have the intent of encouraging people to move to -v2 interface, which is fine -- but almost nobody is using it and lots of places are dependent on tooling that only does -v1.

At the end of the day if you cannot set low/min (and you *have* to set low/min to enable memory protection) then you are left with a pretty limited set of options, none of them good.

Correct, none of the code outside of memcontrol.c is *intended* to know anything about the cgroup settings, ever. It was only exposed ~4.3 for performance which lets use *view* but not change (except for high, that we can view and change).

So, I think, having read over (and not fully understood, but understanding better...) the various comments...

We can't see cgroups memory pressure, but we *can* see, using the PF_MEMALLOC flag, if a thread is trying to allocate memory.  This is used in https://review.whamcloud.com/c/fs/lustre-release/+/50460 to notice we're in memory pressure (because releasepage() is only called from a thread doing an allocation if it's had to go in to memory reclaim to satisfy that allocation).

I think that does a decent job of making our response transparent to what the source of the memory limit is, RAM or cgroups, etc.

Yes, I the code in mm/vmscan and mm/memcontrol that does things like:
   mem_cgroup_calculate_protection()
   effective_protection() 
   get_scan_count()

These scan and calculate how much free memory to ensure is available to a cgroup.
If min/low are zero there is not protection (so there is no attempt to maintain a pool of memory for use when memory is running low)

By setting low/min you are telling the cgroup to do exactly what you expect it do to: enforce a limit (provide protection). Otherwise it is really just tracking usage and if you go over oom.
