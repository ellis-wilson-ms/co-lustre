ELC logic is implemented in LMV layer.

  specific fid is on remote MDT, LMV has sent async cancel request,
  and mdc_create/link/unlink/rename/setattr doesn't need cancel
  lock synchronously.

--- Message (Niu Yawei, 2011-09-13 10:19:27.000000000) ---
Patch Set 3:

This patch will miss cancelling child LOOKUP lock for unlink and rename operations.

Actually, the MF_MDC_CANCEL_FID* flags looks quite confusing to me, how about make the changes like:

- replacing MF_MDC_CANCEL_FID* to MF_LOCAL_FID*, each MF_LOCAL_FID(n) indicating if the fid(n) is a local fid; 
- by default, the MDC_LOCAL_FID* are present in op_flags, and lmv is responsible for clearing the flags in lmv_early_cancel();
- mdc operations (mdc_unlink, mdc_rename...) checks the MDC_LOCAL_FID* flags and decides if it should do sync cancel on some certain fid.

In a word, lmv should only tell mdc if the fid is local, and mdc decides if it should do sync cancel on the fid base on this information.

--- Message (Niu Yawei, 2011-09-13 16:19:04.000000000) ---
Patch Set 4:

The current logic looks a little bit twisted:
- We set all the MF_MDC_CANCEL_FID? in llite layer;
- Then clear some flags in lmv layer; (if the corresponding fid is remote fid, and it's not the fid2 of unlink or rename)
- Send sync cancel in mdc layer according to the MF_MDC_CANCEL_FID? flags;

I think it can be simplified as:

- Do nothing in llite layer;
- Set MF_REMOTE_FID? in lmv layer; (if the corresponding fid is remote fid)
- Send sync cancel in mdc layer according to the MF_REMOTE_FID? flags; (if the flag is not present or it's a _must_ sync cancel fid, for instance, fid2 of unlink or rename) 

The MF_REMOTE_FID? flags can also be used for other purpose in the future.
