LU-10472 osc: add T10PI support for RPC checksum

T10 Protection Information (T10 PI), previously known as Data
Integrity Field (DIF), is a standard for end-to-end data integrity
validation. T10 PI prevents silent data corruption, ensuring that
incomplete and incorrect data cannot overwrite good data.

Lustre file system already supports RPC level checksum which
validates the data in bulk RPCs when writing/reading data to/from
objects on OSTs. RPC level checksum can detect data corruption that
happens during RPC being transferred over the wire. However, it is
not capable to prevent silent data corruption happening in other
conditions, for example, memory corruption when data is cached in
page cache. And by using the existing checksum mechanism, only
disjoint protection coverage is provided. Thus, in order to provide
end-to-end data protection, T10PI support for Lustre should be added.

In order to provide end-to-end data integrity validation, the T10 PI
checksum of data in a sector need to be calculated on Lustre client
side and validated later on the Lustre OSS side. The T10 protection
information should be sent together with the data in the RPC.
However, in order to avoid significant performance degradation,
instead of sending all original guard tags for all sectors in a bulk
RPC, the existing checksum feature of bulk RPC will be integrated
together with the new T10PI feature.

When OST starts, necessary T10PI information will be extracted from
storage, i.e. the T10PI DIF type and sector size. The DIF type could
be one of TYPE1_IP, TYPE1_CRC, TYPE3_IP and TYPE3_CRC. And sector
size could be either 512 or 4K bytes.

When an OSC is connecting to OST, OSC and OST will negotiate about
the checksum types. New checksum types are added for T10PI support
including OBD_CKSUM_T10IP512, OBD_CKSUM_T10IP4K, OBD_CKSUM_T10CRC512,
and OBD_CKSUM_T10CRC4K. If the OST storage has T10PI suppoort, the
only selectable T10PI checksum type would have the same type with the
T10PI type of the hardware. The other existing checksum types (crc32,
crc32c, adler32) are still valid options for the RPC checksum type.

When calculating RPC checksum of T10PI, the T10PI checksums of all
sectors will be calculated first using the T10PI chekcsum type, i.e.
16-bit crc or IP checksum. And then RPC checksum will be calculated on
all of the T10PI checksums. The RPC checksum type used in this step is
always alder32. Considering that the checksum-of-checksums is only
computed on a * 4KB chunk of GRD tags for a 1MB RPC for 512B sectors,
or 16KB of GRD tags for 16MB of 4KB sectors, this is only 1/256 or
1/1024 of the total data being checksummed, so the checksum type used
here should not affect overall system performance noticeably.

obdfilter.*.enforce_t10pi_cksum can be used to tune whether to enforce
T10-PI checksum or not.

If the OST supports T10-PI feature and T10-PI chekcsum is enforced, clients
will have no other choice for RPC checksum type other than using the T10PI
chekcsum type. This is useful for enforcing end-to-end integrity in the
whole system.

If the OST doesn't support T10-PI feature and T10-PI chekcsum is enforced,
together with other checksums with reasonably good speeds (e.g. crc32,
crc32c, adler, etc.), all the T10-PI checksum types (t10ip512, t10ip4K,
t10crc512, t10crc4K) will be added to the available checksum types,
regardless of the speeds of T10-PI chekcsums. This is useful for testing
T10-PI checksums of RPC.

If the OST supports T10-PI feature and T10-PI chekcsum is NOT enforced,
the corresponding T10-PI checksum type will be added to the checksum type
list, regardless of the speed of the T10-PI chekcsum. This provide the
clients to flexibility to choose whether to enable end-to-end integrity
or not.

If the OST does NOT supports T10-PI feature and T10-PI chekcsum is NOT
enforced, together with other checksums with reasonably good speeds,
all the T10-PI checksum types with good speeds will be added into the
checksum type list. Note that a T10-PI checksum type with a speed worse
than half of Alder will NOT be added as a option. In this circumstance,
T10-PI checksum types has the same behavior like other normal checksum
types.

The clients that has no T10-PI RPC checksum support will not be affected
by the above-mentioned logic. And that logic will only be enforced to the
newly connected clients after changing obdfilter.*.enforce_t10pi_cksum on
an OST.

Following are the speeds of different checksum types on a server with CPU
of Intel(R) Xeon(R) E5-2650 @ 2.00GHz:

crc: 1575 MB/s
crc32c: 9763 MB/s
adler: 1255 MB/s
t10ip512: 6151 MB/s
t10ip4k: 7935 MB/s
t10crc512: 1119 MB/s
t10crc4k: 1531 MB/s

Patch Set 9: Code-Review-1

(22 comments)

Thanks for submitting this patch, even though there is clearly a bunch of debugging going on (I guess it is still under development).

It is much easier to provide feedback on the patch earlier, before you have spent a lot of time to fix every detail up.

As you can see from the inline comments, I think that this should be better integrated into the existing checksum mechanism. Here comes the question, which kind of checksum should be choosen. I am not sure whether CRC32 is supported and optimized by all platforms. It might be good enough for most of cases. And in theory, the checksum type is should not have large impact on the performance, because T10PI only need it for calculating the checksum of T10PI guard tags. But still, we want to do benchmarks against other combinations (T10PI + Adler, T10PI + CRC32C etc.). If we loose the option to change checksum type, we loose the chance to optimize. And even the performance impact is only 5%, it is still significant, becasue it affects all kinds of read and write.

And as you've mentioned once, we might want to change the 32 bits RPC checksum to 64 bits (or 128 bits) for stronger checksum. If that becomes true, assuming we have fixed checksum type for T10PI, we will need to rework the T10PI codes again for 64 bit support. But with the current implemetation, nothing needs to be changed at all on T10PI side. And as we can imagine, 64 bits or 128 bits checksum has higher impact on performance, and checksum type used by T10PI becomes more important. And at that time, once again, we will need to run benchmarks to choose the best checksum type.

That avoids all of the extra tunables, compatibility, etc. that is needed for handling a different kind of checksum.

For users who have T10-PI hardwares and want to use T10-PI support for RPC, there will only be two options in /proc/fs/lustre/osc/XXX/t10pi_type, one is "disable", the other is the T10PI type of the hardware. For the users who have no T10-PI hardware and want to run tests on the T10PI RPC support, they will need to learn new things of /proc/fs/lustre/osc/XXX/t10pi_type, but interface of /proc/fs/lustre/osc/XXX/checksum_type will be kept the same. I feel this will be even simpler to keep compatibility, especially for simple scripts that parse  /proc/fs/lustre/osc/XXX/checksum_type for supported checksum types.

> I am not sure using underlying T10-PI checksum is proper. The
> underlying T10-PI checksum (crc_t10dif() or ip_compute_csum())
> returns 16 bit checksum, which is not enough for 32 bit RPC
> checksum.

That is fine.  I agree the IP checksum is very weak, and only 16
bits, so I'm happy if it is not used for the checksum-of-checksums.

> Here comes the question, which kind of checksum should
> be chosen. I am not sure whether CRC32 is supported and optimized
> by all platforms.

There have been CPU-optimized versions of CRC32C for some time now
(both assembly and C-unrolled-loop versions).  There is also CPU
acceleration support for these checksums for several years.  I don't
think you are going to see systems that have T10-PI-capable disks,
but do not have CRC32C optimization.

> It might be good enough for most of cases. And in theory, checksum
> type should not have large impact on the performance, because T10PI
> only need it for calculating the checksum of T10PI guard tags.

Consider that the checksum-of-checksums is only computing CRC32 on
a 4KB chunk of GRD tags for a 1MB RPC for 512B sectors (only 1KB to
checksum for 4KB sectors), or 16KB of GRD tags for 16MB of 4KB sectors
(I assume most modern systems using 16MB RPCs also have 4KB sectors).
This is only 1/256 or 1/1024 of the total data being checksummed (only
0.25% or 0.1% of the total data), so it should not affect overall
system performance noticeably.

> But still, we want to do benchmarks against other combinations
> (T10PI + Adler, T10PI + CRC32C etc.).  If we loose the option to
> change checksum type, we loose the chance to optimize. And even
> the performance impact is only 5%, it is still significant, because
> it affects all kinds of read and write.

You could do that checksum benchmarking now, before the patch is landed.
I expect you will see that the top-level checksum type does not affect
performance in any meaningful way.  I think the only question would be
whether it is better to use CRC32 or CRC32C.

> And as you've mentioned once, we might want to change the 32 bits RPC
> checksum to 64 bits (or 128 bits) for stronger checksum. If that
> becomes true, assuming we have fixed checksum type for T10PI, we will
> need to rework the T10PI codes again for 64 bit support. But with the
> current implementation, nothing needs to be changed at all on T10PI
> side. And as we can imagine, 64 bits or 128 bits checksum has higher
> impact on performance, and checksum type used by T10PI becomes more
> important. And at that time, once again, we will need to run benchmarks
> to choose the best checksum type.

No, this feature would be for ZFS, so that the client can compute the
ZFS on-disk checksums, and then send the checksum-of-checksums in the RPC
to the OSS in the same way that this patch allows clients to calculate
T10-PI GRD tags.  There is no need to change existing 32-bit checksums.
The only difference would be that there are some extra fields in the RPC
for the larger ZFS checksum value.

> And I don't think the current /proc/fs/lustre/osc/XXX/t10pi_type adds
> too many tunables comparing to /proc/fs/lustre/osc/XXX/checksum_type.
> Some extra interfaces will need to be added any way. For users that
> don't care about T10-PI, they can just ignore the existence of
> /proc/fs/lustre/osc/XXX/t10pi_type, becasue the T10-PI support for RPC
> will disabled by default. For users who have T10-PI hardwares and want
> to use T10-PI support for RPC, there will only be two options in
> /proc/fs/lustre/osc/XXX/t10pi_type, one is "disable", the other is the
> T10PI type of the hardware. For the users who have no T10-PI hardware
> and want to run tests on the T10PI RPC support, they will need to learn
> new things of /proc/fs/lustre/osc/XXX/t10pi_type, but interface of
> /proc/fs/lustre/osc/XXX/checksum_type will be kept the same. I feel
> this will be even simpler to keep compatibility, especially for simple
> scripts that parse /proc/fs/lustre/osc/XXX/checksum_type for supported
> checksum types.

What happens if a user wants to change the overall checksum type to e.g.
"adler" if T10-PI is enabled (if this is faster)?  From what I can see,
setting "lctl set_param osc.*.checksum_type=adler" (which is what their
scripts had done in the past) would only change the checksum-of-checksum
type, but not disable the T10-PI checksums?  They would also have to
disable the T10-PI checksums separately, which is confusing, IMHO.

What is the difference between type1 and type3 T10-PI checksums, and where
will this be documented?  This is even a question that I don't know the
answer to myself, even though I've checked online.  The docs I found point
to the difference in the SCSI command set (10-bit, 12-bit, 16-bit, or
32-bit SCSI commands) and where the REF and APP tags are passed, but I
don't see why this complexity needs to be exported to the Lustre client?

I think the more different options that are available, the more likely it
is that the user will get it wrong.  Having a single "osc.*.checksum_types"
parameter that controls which checksum is used for the RPCs is simplest.
If the OSS supports T10PI checksums, it can return this in the list of
supported checksum types as any other checksum type today ("adler", "crc32",
"crc32c", "t10crc4k, "t10crc512", "t10ip4k", "t10ip512") then the same code
and userspace interface that selects the RPC checksum will continue to work.


There is a separate question as to which checksum type the client should use,
if the OSS offers the T10PI checksums even if there is not hardware support
for them?  Currently the client just picks the fastest checksum type based
on the list it is offered and the local CPU performance.  However, if the
server can support T10PI in hardware, it makes sense to only offer valid T10PI
choices to the client, so that it has to select one of the T10PI options.
Since the client sends the OSS the list of supported checksums, there would
only be a valid T10PI choice if the client supports this checksum type.
Otherwise, the server would need to do two checksums (RPC checksum and T10PI),
even if one of the non-T10PI options are faster on the client.

That implies that there should be a /proc tunable on the *server* that decides
whether T10PI should be ADDED to the supported checksum types (for testing, or
if the underlying hardware does not support T10PI), or if it should be the ONLY
type returned if the both the client and the underlying hardware supports it.
Maybe something like "obdfilter.*.checksum_t10pi_enforce" or similar?

This patch breaks local testing by the means of sh llmount.sh because it tries to use crc_t10dif symbol that is typically a module.

Since llmount.sh insmods modules, the dependencies do not load automatically, so we need to update framework to modprobe crc_t10dif similar to how we do it for e.g. mbcache and exportfs
