LU-17412 mgs: Fix write barrier failed state

Creating a snapshot after mounting another snapshot fails and
causes the barrier to get into a "failed" state. The following
error is seen:

-> lctl snapshot_create -F testfs -n snap1
-> lctl snapshot_mount -F testfs -n snap1
mounted the snapshot snap1 with fsname 7a47062d
-> lctl snapshot_create -F testfs -n snap2
Fail to freeze barrier for testfs: Object is remote
Can't create the snapshot snap2

Due to this subsequent operations to lustre snapshots fail unless
barrier failed state is cleared. This is because the current
implementation assumes that mgs_barrier_glimpse_lock() returning
rc=0 means the barrier is done. However, this is not always the
case, as mgs_barrier_done() can still be 0. This fix ensures the
operation retries until the barrier is actually complete.

Added the above testcase to the existing sanity-lsnapshot.sh
(test_1c) to reliably reproduce the above issue and prevent
future occurrences.

The Lustre autotest framework is now running every modified subtest in a loop for 30 minutes to try and catch new/modified test cases that are introducing intermittent test failures.

For this patch it means sanity-lsnapshot test_1c and sanity test_801a will be run repeatedly to confirm proper functionality.

This is done using the "ONLY=NNN ONLY_MINUTES=30" options in test-framework.sh (which you can also run locally).  This will automatically delete files with the $DIR/$tdir* and $DIR/$tfile* filename prefixes between each loop, but the subtest should otherwise leave the filesystem in a usable state when it is completed.

It looks like test_1c is creating the snapshot named lss_1c_0, but it is not being deleted at the end, which may also cause problems for later subtests.  It probably is best to register a stack_trap when the snapshot is first created, so it is always deleted when the subtest exits (whether successfully or via lss_err).
