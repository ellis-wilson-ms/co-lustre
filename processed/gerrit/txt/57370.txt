OK, so I guess replacing "reformat" with "formatall" is touching a ton of subtests, and they are all being run by the "subtest-change" session in a loop and failing for one reason or another.  That will make this transition more difficult than expected by just moving the subtests as-is to another file.

Let's revert that change from this patch (keep the reformat() wrapper for now), and just move the helper functions to conf-sanity-framework.sh without touching any of the subtests.

To defeat the "ONLY_MINUTES=30" setting that autotest is setting in the subtest-change session, please add a line at the start of conf-sanity-framework.sh:

 export ONLY_REPEAT=${ONLY_MINUTES:+1}

which means "if ONLY_MINUTES is set, then set ONLY_REPEAT=1, which should limit the subtest-change session to only a single iteration of each subtest.  This line must be removed at the end of the patch series, but doing so will not trigger the subtests to be run in a loop.

--- Message (Andreas Dilger, 2025-05-23 11:11:52.000000000) ---
Patch Set 10:

> I just don’t quite understand why such a minor change (replacing
 > one function with another (identical one)) has such a negative
 > effect.

It's not that the change is breaking the tests, so much as *any* change to a test_XXX function will cause Autotest to run this subtest in a loop for 30 minutes to try and avoid landing tests that only pass intermittently.

The problem is that most of these tests were written *long* before this new test-in-loop policy was implemented, so they may have bugs when run in this mode (e.g. leave files or configuration state behind, or depend implicitly on some previous state).  That isn't great, but I don't think this patch should have to fix 50 subtests in order to land.

So my proposal is to temporarily force conf-sanity-framework.sh to run each "modified" (relocated) subtest only once, so this patch can land, and then remove the workaround so that ONLY_REPEAT and ONLY_MINUTES will work properly again.

You are of course welcome to try and debug/fix the subtests that cannot run in a loop, but I suspect your time would be better spent to fix the intermittent subtest failures where they can't even run consistently *once*... :-/
