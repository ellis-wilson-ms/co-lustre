I reproduced sanity.sh 183 failure with upstream client that uses the same atomic_open code as master. The reason for the failure is when atomic_open passes in hashed negative dentry that is created by other node, we leak the inode and do not instantiate the dentry even if server returns DISP_LOOKUP_POS and DISP_OPEN_OPEN.

sanity.sh 183 fails the same path because occasionally it causes client to create false negative dentry in dcache after doing error sim testing and resetting fail_loc.
