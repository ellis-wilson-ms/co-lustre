LU-5153 fld: lsf_lock in update_from_controller

The lsf_lock mutex must be held for calls to
fld_index_create, and is currently taken before calls to
fld_insert_entry.

This patch adds a comment and an assertion to
fld_insert_entry, and takes the mutex for the call in
fld_update_from_controller.

Also adds subsystem description from Di Wang to header file.

>Could you give a few sentence explanation of what the fld subsystem is for? I've inferred some from reading the code, but it remains a bit mysterious.

FLD is FID Location Database, which stores the information of where FIDs are located. And the database is basically a record file, each record consists FID sequence range, MDT/OST index and flags.  The FLD for the whole FS is only stored on the sequence controller(MDT0) right now, but each target also have its local FLD, which only store the local sequence.

The FLD subsystem usually has two tasks
1. maintain the database, i.e. when sequence controller allocates new sequence ranges to some nodes, it will call FLD API to insert the location information <sequence_range, node_index> in FLDB

2. Handle the request from other nodes, i.e. if client needs to where the FID is located, if it can not find the information in the local cache, it will send FLD lookup RPC to FLD service, and FLD service will lookup the FLDB and return the location information to client.
