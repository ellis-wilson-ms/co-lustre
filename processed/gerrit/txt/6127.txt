--- Commit Message ---
LU-3219 ost: Ensure dirty flushed on fiemap ioctl

Data corruption is possible if cp(coreutils) uses FIEMAP to obtain
data holes in shared file, since there could be dirty cache on
other clients which hasn't been flushed back.

To ensure all the dirty on remote clients being flushed back on
fiemap ioctl, we'd acquire ldlm lock on server side for fiemap,
unless the local client (which invoke fiemap) has cached lock.

I like this patch a lot better, but it still suffers from the same problem that it will always revoke the locks from the calling client, because there is no data beyond [i_size, EOF]. It would also always cause a lock revocation for sparse files.

Since the calling client has flushed it's cache already due to FIEMAP_FLAG_SYNC, it should be possible to have the OST skip locking on extents that are locked by the calling client?  That would avoid any lock contention on the common "calling client holds [0, EOF] lock" case, but still handle the distributed writers/FIEMAP case correctly.

Two problems with this patch:
- in osc_get_info() it needs to check for overflow in the l_extend.end calculation. If fm_length = ~0 then the "round up" will overflow to zero. It needs to check:

        if (OBD_OBJECT_EOF - fm_key->fiemap.fm_length <=
            fm_key->fiemap.fm_start + CFS_PAGE_SIZE - 1)
                policy.l_extent.end = OBD_OBJECT_EOF;
        else
                policy.l_extent.end = ...;

Secondly, in ost_get_info() it should only do the server-side locking if the client sends OBD_FL_SRVLCK, otherwise it will be canceling the locks held by the client, and may deadlock if the client holds the write lock already.

Finally, as a minor style issue, there are many "in-between" lines that could be converted to tabs.
