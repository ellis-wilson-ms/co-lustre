LU-15546 mdt: keep history of mdt_reint_open() lock

The open path doesn't know in advance if a read or write lock should
be used for the parent directory (LCK_PR to open an existing file
without cancelling other locks, or LCK_PW to create a new file).

Instead of always taking a write lock or looking up the file in
advance (both add measurable overhead), keep a history of which lock
was needed.  If the majority of recent locks are the same (create
or lookup heavy workload) then predict the next lock will also be
the same.  If prediction is incorrect (LCK_PR, but file does not
exist) then the operation will be retried with the right lock type
(LCK_PW for existing file does not need a retry, but is expensive).
This can be racy even if open does a pre-lookup to check whether
name exists or not before taking the lock, so there is no danger
to using the wrong lock mode initially.

If a mixed workload of open and create operations are done recently
on the same parent (i.e. history not beyond thresholds), then lookup
is done on name first to avoid needlessly using the wrong lock mode
(e.g. many Fortran threads opening same file with O_CREAT, LU-10262).

Hi Andreas.
While I appreciate this will avoid doing an extra lookup on benchmark style loads and help raising scores, I'm afraid it might have negative impacts on production: we've seen some real mess after just a few dozens of PW locks on fs root when there are tens of thousands of clients each owning a PR lock on it and actively trying to get it back.
With this kind of patch on mixed workloads we're bound to have some new clients (compute nodes are put back online all the time, by batch as well!) opening / while a job is doing file-creating intensive workload and having these clients open turn into PW lock taking, and admins will have literally no way of understanding what happened.

At the very least there should be some debug log after the fact that we took a bigger lock than required, or make the speculative counter per JOBID...

Honestly if you just want to improve benchmarks I like my idea better:
Do one lookup before taking the lock first and trust it. If after trusting it it turns out you were raced (mdd lookup/object create fails because of ENOENT/resp EEXIST) then potientially upgrade lock and try again.
This only has one lookup in the nominal case (e.g. all benchmarks) so performance will be back, and has no risk of screwing up normal workloads.

Sorry, I was a bit optimistic in my worst worst case (there are two races window). Let's try again.

Original 2.12 code:
 - PW lock, lookup, open or create

Current master:
 - PR lock, lookup, 
 - (if file didn't exist) PW lock + 2nd lookup
 - open or create based on last lookup

Étienne's patch:
 - lookup, PR or PW lock, lookup
 - (if we got raced and had PR lock) PW lock + 3rd lookup
 - open or create based on last lookup

This patch:
 - guess lock type (or lookup if can't decide)
 - PR or PW lock, lookup
 - (if raced and had PR lock) PW lock + 2nd (or 3rd) lookup
 - open or create based on last lookup

My suggestion:
 - lookup
 - PR lock + open, or PW lock + create
 - (was open case, if ENOENT) PW lock, create
 - (was create case, if EEXIST) open

So the worst case here is actually 1 lookup, 2 locks, 2 open and 1 create because we could be re-raced when upgrading lock, but we can't be reraced when opening after create failed since we can keep opening within PW lock.
"Common" optimal case is the same 1/1/1 back to original 2.12 performance (except sometimes PR instead of PW), and my conclusion hasn't changed (I believe complexity is still simpler with optimistic open/create and odd of races low enough), but I wanted to clarify that.

I think the main idea is good. But it could be simplified:

We don't really need to have the history between LCK_PW *n and LCK_PR *n (because there are only 2 values).

And this history is not really a lock history, but more a history about file existence when creating a file.

We can the history interval around 0 and chose a value resolution.
So the initial value is 0.

For a file creation
- If the file existed, we add the value resolution to the history.
- If the file did not exist, we subtract the value to the history.

static void mdt_open_create_history_add(struct mdt_object *parent,
				     bool file_exist)
{
        int val = file_exist ? 1 : -1;

	parent->mot_open_lock_history += ( val * MDT_OPEN_HIST_RES -
		parent->mot_open_lock_history) / MDT_OPEN_HIST_COUNT;
}

If we keep the actual resolution of: 2 * MDT_OPEN_HIST_COUNT * MDT_OPEN_HIST_COUNT
This could simplify to:

static void mdt_open_create_history_add(struct mdt_object *parent,
				     bool file_exist)
{
        int val = file_exist ? 1 : -1;

	parent->mot_open_lock_history += 2 * val * MDT_OPEN_HIST_COUNT -
		parent->mot_open_lock_history / MDT_OPEN_HIST_COUNT;
}

Thresholds could be more easily determine (in percentage of the value).
e.g:
/* 10% of the global history interval to set LCK_PW directly */
#define MDT_OPEN_HIST_THRESH_PW (- 80 * MDT_OPEN_HIST_RES / 100)
/* 5% of the global history interval to set LCK_PR directly */
#define MDT_OPEN_HIST_THRESH_PR (90 * MDT_OPEN_HIST_RES / 100)
