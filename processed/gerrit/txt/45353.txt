--- Message (Andreas Dilger, 2022-03-22 00:16:15.000000000) ---
Patch Set 12: Code-Review-1

In commit v4.6-rc3-27-g9902af79c01a ("parallel lookups: actual switch to rwsem" and a few earlier patches, e.g. in sles15, el8, and later) there is already a change to use i_rwsem instead of i_mutex, and addition of inode_lock_shared().  It makes sense to stick with that naming convention and locking.

Using mmap_sem is inconsistent with the VFS locking and could lead to problems in some cases if some other code is doing a lookup here (e.g. other VFS code that is using inode_lock()).  It probably makes sense to just add a wrapper for el7 servers like (with appropriate configure check):

 #ifndef HAVE_INODE_LOCK_SHARED
 #define inode_lock_shared(parent) inode_lock(parent)
 #define inode_unlock_shared(parent) inode_unlock(parent)
 #endif

(i.e. it is a no-op for older kernels) and use the shared locking only for newer kernels?

Another alternative (that would need significantly more work, maybe not worthwhile?) is to change object lookup to use pdirops, which would probably be even better than mmap_sem since there are often millions of entries in each OST object directory for large OSTs, and hundreds or thousands of OST threads doing concurrent operations here.

Neil Brown is also working to get parallel directory operations (not just lookups) into the upstream kernel, but that will likely be a couple of years before it gets into vendor kernels that we use.
