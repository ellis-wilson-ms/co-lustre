LU-13779 lnet: Correct asymmetric route detection

Failure to lookup the remote net for LNET_NIDNET(src_nid) indicates an
asymmetric route, but we do not drop the message in this case. Another
problem with this code is that there is no guarantee that we'll have a
route->lr_lnet that matches the net of ni->ni_nid.

We can move the asymmetric route detection to after we have looked up
the lpni of from_nid. Then, we can look at just the routes associated
with the gateway that owns the lpni. If one of those routes has
lr_net == LNET_NIDNET(src_nid), then the route is symmetrical.

--- Message (Chris Horn, 2020-07-28 16:06:12.000000000) ---
Patch Set 2:

> (1 comment)
 > 
 > Have you considered a 2-stage LNet routing configuration? In such a
 > scenario, I am worried the route would be incorrectly declared as
 > asymmetrical, because the loop over the routes associated with the
 > gateway that owns the lpni would not find any matching lr_net.

If the gateway does not have a route where lr_net matches the net of the message originator (src_net), then that route is, by definition, asymmetrical and the message _should_ be dropped. This is true regardless of the number of hops a route takes.

Consider the following multi-hop config.

PeerA: tcp1
Router 1: tcp1 <-> tcp2
Router 2: tcp2 <-> tcp3
PeerB: tcp3

PeerA Route: router1@tcp1 -> tcp3
Router 1 Route: router2@tcp2 -> tcp3
Router 2 Route: router1@tcp2 -> tcp1
PeerB Route: router2@tcp3 -> tcp1

The lr_net for all routes in the system are either tcp1 or tcp3, and the source net for any forwarded message should be either tcp1 or tcp3.

 > 
 > Also, do you think it could be possible to add a test for
 > 'lnet_drop_asym_route' option, now that auster can do lnet testing?

Yes, it is possible to construct a test case for this with a 4 node cluster.
