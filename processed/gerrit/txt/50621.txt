LU-17433 osc: Make unaligned DIO async

Because unaligned DIO is using a copy of the data from
userspace, we can make the writes async on the OST side.
Because DIO expects that data be 'safe', we cannot just
go entirely async and not wait for RPC completion, but we
can wait only for RPC completion and not force a commit
sync on the server.

This is because having our own copy of the data lets us
replay the RPC after write() has completed, which is not
possible for regular DIO.

This reduces the DIO 'O_SYNC' guarantee slightly for
unaligned DIO (and other small DIO if we change this).
Before this patch, after a DIO write(), the data is fully
committed to disk and cannot be lost even if the client and
server crash.

With this patch, after write() returns, if the server
crashes, the client can replay the RPC, and if the client
crashes, the server will finish writing out the data.
However, if the client and server both crash, the data will
be lost.

For this reason, we make this behavior tunable:
llite.*.dio_full_sync

With the default to '0'.

> After thinking about this (per comments in LU-13805) I _think_ this is allowed for O_DIRECT, but we might want it to be configurable anyway ("llite.*.dio_is_sync" tunable?), in case applications have this O_DIRECT == O_SYNC expectation and cannot be modified.

Yeah.  Per our discussion in LU-13805, I think this represents a *reduction* in the O_SYNC expectation - it's still much stronger guarantee than buffered I/O provides, since buffered I/O can lose data after write() completes with only a client crash, whereas this would be client + server crash - and the tradeoff is that you get better performance.  So it should absolutely be tunable for folks who *really* don't want to run the risk.

By the way, I expect the biggest perf difference is probably in scaling to multiple clients, since we won't be generating a sort of 'sync storm' on the OST as everyone does small DIO.

Yeah, what Yingjin said.  This has the same behavior as buffered IO.  The same problem - memory starvation - applies right now with buffered IO.  Without the new patch he linked, buffered I/O can make infinite uncommitted/unstable pages.  With the new patch, we take care of buffered or DIO uncommitted pages.

RE: sync
This is just 'journal commit async'.  I can probably improve the message to describe it better.  But data is still forced to disk on the server, and we have replay (because this DIO is using a buffer).  So it is not 100% safe, but data loss happens after write() complete only if client + server crash.  If client crashes or server crashes, no data loss.

We are also going to put a tunable so people who are very concerned about data integrity can turn this off. :)
