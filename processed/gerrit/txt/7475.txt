--- Commit Message ---
LU-3544 llite: simplify dentry revalidate

Lustre client dentry validation is protected by LDLM lock, so
any time a dentry is found, it's valid and no need to revalidate
from MDS, and even it does, there is race that it may be
invalidated after revalidation is finished.

--- Message (Oleg Drokin, 2013-08-30 03:26:09.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

This patch works from an assumption, that if the dentry is there in place, it must have lock, but this might not actually true, because the dentry might have been reinstantiated without a lock by e.g. an open of some sort and revalidate still must make sure the lock is there.

This simple patch shows the problem when you run sanityn.sh:
--- a/lustre/llite/dcache.c
+++ b/lustre/llite/dcache.c
@@ -394,6 +394,15 @@ int ll_revalidate_nd(struct dentry *dentry, struct nameidat
                RETURN(0);
 #endif
 
+       {
+       u64 ibits = MDS_INODELOCK_LOOKUP;
+
+       LASSERT(!d_lustre_invalid(dentry));
+
+       if (dentry->d_inode && !ll_have_md_lock(dentry->d_inode, &ibits, LCK_MIN
+               LBUG();
+       }
+
        RETURN(1);
 }

Though it dies because the lock is already cancelled, but we did not get to the dentry removal yet, so in the end we returned "Valid" for a dentry that just got invalidated.

Hm... this actually might be a valid race that could happen with old code too (after we release the revalidated lock before returning).

Additionally it used to be that you could not return false from revalidate for the open, because that totally failed the open, was this changed in the kernel API?

--- Message (Fan Yong, 2013-09-03 06:22:09.000000000) ---
Patch Set 1: I would prefer that you didn't submit this

Removing statahead interaction from revalidate will misguid the statahead thread. For example:

1) The User1 performs "ls -l $DIR" for the first time, then there is no cache on the client-side, so both statahead thread and the "ls" thread will work well.

2) After the first time "ls -l $DIR", some of the $DIR dentries will be cached on the client-side, some may be not.

3) The User1 perform "ls -l $DIR" for the second time, then the first dentry in the $DIR may be cached already. Then it will not trigger statahead thread according to current statahead mechanism.

4) Even if the first dentry of $DIR is not cached, and the statahead thread is triggered as normal case, but because some dentries are cached on the client already and caused revalidate will not interact with the statahead thread, then the statahead thread local dcache will become full quickly, and then cause the statahead thread to wait "ls" thread to interact with it. But when the "ls" trigger some lookup and interact with the statahead thread, the "ls" thread may be ahead of the statahead thread, and cannot found related statahead entry in its local dcache. Under such case, the statahead thread will be stopped unexpectedly.
