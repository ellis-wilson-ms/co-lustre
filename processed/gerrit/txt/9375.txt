--- Message (Andreas Dilger, 2014-03-03 05:55:10.000000000) ---
Patch Set 2:

In b2_4 this is checked when the OST is first mounted:

 int ofd_server_data_init()
 {
        :
        :
        if (lsd->lsd_feature_incompat & ~OFD_INCOMPAT_SUPP) {
                CERROR("%s: unsupported incompat filesystem feature(s) %x\n",
                       obd->obd_name,
                       lsd->lsd_feature_incompat & ~OFD_INCOMPAT_SUPP);
                GOTO(err_fsd, rc = -EINVAL);
        }
 }

So if the OST is mounted but it was modified by a newer version of Lustre to have a feature that it does not understand, then it will refuse to mount the OST.

This was being hit on 2.5 when a 2.6-mounted OST was downgraded to 2.5 again, per the original bug.  That behaviour was _almost_ correct on the part of 2.5 (refusing to mount an OST with a filesystem that it didn't understand), with the exception that 2.5 DID know how to handle the OBD_INCOMPAT_FID feature already, and that 2.6 shouldn't have set this feature if it hadn't yet started using FID-on-OST (i.e. mounted in DNE mode).

This patch is half of the fix - allowing the 2.4/2.5/master OST to be mounted with OBD_INCOMPAT_FID on the OST.  The other half is that the OST should get OBD_INCOMPAT_FID set when it starts creating FID_SEQ_NORMAL objects on the OST for DNE.  That will ensure that anyone upgrading to 2.4/2.5 and starting DNE will not be able to downgrade to 2.1 OSTs.
