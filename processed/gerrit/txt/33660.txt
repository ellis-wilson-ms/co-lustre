--- Commit Message ---
LU-11670 osc: glimpse - search for active lock

When there are lock-ahead write locks on a file, the server
sends one glimpse AST RPC to each client having such (it
may have many) locks. This callback is sent to the lock
having the highest offset.

Client's glimpse callback goes up to the clio layers and
gets the global (not lock-specific) view of size.  The clio
layers are connected to the extent lock through the
l_ast_data (which points to the OSC object).

Speculative locks (AGL, lockahead) do not have l_ast_data
initialised until an IO happens under the lock. Thus, some
speculative locks may not have l_ast_data initialized.

It is possible for the client to do a write using one lock
(changing file size), but for the glimpse AST to be sent to
another lock without l_ast_data initialized.  Currently, a
lock with no l_ast_data set returns ELDLM_NO_LOCK_DATA to
the server.  In this case, this means we do not return the
updated size.

The solution is to search the granted lock tree for any lock
with initialized l_ast_data (it points to the OSC object
which is the same for all the extent locks) and to reach the
clio layers for the size through this lock instead.

--- Message (Patrick Farrell, 2018-11-16 15:56:49.000000000) ---
Patch Set 6:

Andreas,

Few responses here...
The "check only the last valid one" is correct, but it's more of a server side thing.  The server starts at the highest lock, and works its way down - And if an LDLM lock is not NO_EXPANSION, it's guaranteed to be valid, so it sends that glimpse and stops.  For NO_EXPANSION locks, it sends a glimpse to each client that holds one.  (All of this is only for locks above the size the server has locally – Once that size is reached, it stops.  [This is why the problem is timing dependent and how it escaped testing previously: if the current size has reached the server, we don’t see an issue.])

Then, on the client:
The client receives a glimpse to a particular LDLM lock, but it does not actually get the size from the LDLM lock - It gets it from the OSC object/clio.  The issue is that invalid/inactive (not used for I/O yet) lockahead locks don't have an OSC object stored in their l_ast_data pointer (because they are created without associated I/O, so the OSC object used for their creation often disappears).

So if the lock that receives the glimpse has no OSC object pointer in l_ast_data, we search on the client for any write/group lock on the resource with l_ast_data set (if the client has only a read lock meeting the criteria, then it doesn’t have updated size info anyway).  Once we find one, we stop - there's only one OSC object per resource, so any pointer will do.  If we don't find one, then either there are no active/valid locks or the OSC object is being torn down – Either way, the client doesn’t have updated size info.

And I believe the iteration of the extent tree so it will start at zero (on the theory that earlier locks are more likely to have been used, since lock ahead requests locks ahead of I/O).  So the only case where we'd scan a bunch of locks is when the client had requested a ton of lockahead locks but not used them yet.

So if you requested a bunch of locks on a client that are beyond current file size, did no I/O, and then did stat, that could end up walking the whole list on each client.  It’s a risk but we normally only request a few hundred locks, at most, in advance of need.  And actually doing any I/O will make this go away.  (For example, if you wrote 1 byte to the first lock you asked for, the scan would find it immediately.)

Apologies for the long explanation.  I’ll see if I can improve the commit message to make some of that clearer.
