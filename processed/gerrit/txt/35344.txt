--- Message (Andreas Dilger, 2019-07-06 17:15:04.000000000) ---
Patch Set 3: Code-Review-1

It doesn't make sense to me to remove checks from eg. 3.1 kernels before removing all the checks for older kernels first?  That breaks 3.1 compatibility (whether that is important in itself is not important) while keeping compatibility for a whole bunch of configure checks and conditional code that is now totally useless. 

If we are going to remove compatibility for older kernels it makes sense to remove support for the oldest kernel first and work our way forward.

Andreas,

Yes, I agree about the ideal order of things. --- Message (Andreas Dilger, 2019-07-15 15:49:20.000000000) ---
Patch Set 3:

> What if we expanded the patch to enable removal of
 > lustre_patchless_compat.h. Would that make this acceptable?

I don't think making the patch larger really addresses my concerns. Also, lustre_patchless_compat.h is a generic header that holds the compat macros for a range of different kernels, up to 4.0 and 4.2, so it isn't necessary to remove this file. 

Essentially, breaking functionality for 3.1 kernels while keeping checks for 2.6.32 kernels leaves a bunch of dead code behind. Since this patch isn't critical for any other patch (AFAIK), there is no rush to land this before the other checks and compat code have been removed.
